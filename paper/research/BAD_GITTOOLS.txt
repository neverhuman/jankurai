\section{Inexcusable Git Tooling Anti-Patterns}
\label{sec:bad_gittools}

The Git tooling bad-behavior source corpus converges on one operating rule:
local hooks are useful feedback, but they are not authority. They depend on
clone-local installation, executable bits, local toolchains, and developer
cooperation. Any policy that matters to shared history must be repeated in CI,
protected branches, repository rulesets, or server-side hooks.

The inexcusable patterns are not ordinary absence of polish. They are checked-in
automation that teaches contributors to bypass verification, disables hooks as
routine policy, writes invisible per-clone hook files, destructively mutates Git
state from a hook, or stages broad unrelated work. These behaviors undermine
review and make the repository less trustworthy for agents and humans.

High-confidence detectors should therefore stay close to executable evidence:
\texttt{--no-verify} in hook or release automation, \texttt{core.hooksPath}
pointing to \texttt{/dev/null}, team installers mutating \texttt{.git/hooks},
destructive Git commands inside hook surfaces, broad \texttt{git add .} or
\texttt{git add -A}, and lint-staged tasks that manually restage the project.
Best-practice gaps such as missing branch protection, missing hosted rulesets,
missing onboarding docs, slow hook suites, unpinned hook references, or missing
repository hygiene files remain advisory unless repository-local evidence shows
an actual unsafe policy.

A good default stack keeps commit hooks fast and staged-file focused, moves
slower checks to pre-push or explicit local commands, runs full checks in CI,
and relies on host or server policy for non-bypassable enforcement. The repair
path is to remove destructive or bypass-normalizing hook behavior, make hook
configuration versioned and documented, and preserve remote enforcement as the
source of truth.
