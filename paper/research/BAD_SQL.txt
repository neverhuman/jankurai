\section{Inexcusable SQL Anti-Patterns}
\label{sec:bad_sql}

The following section categorizes the most severe, inexcusable SQL anti-patterns observed in production environments and AI-generated code. These patterns compromise safety, maintainability, and performance.\

\subsection{Known Best Practices}
To counteract the anti-patterns, ensure adherence to these core SQL best practices:
\begin{itemize}
  \item Parameterize all inputs: Never concatenate strings to build SQL queries.
  \item Normalize schemas carefully: Start with 3NF and denormalize only with measured performance justification.
  \item Enforce data integrity: Use \texttt{NOT NULL}, \texttt{UNIQUE}, \texttt{CHECK} constraints, and foreign keys.
  \item Manage connections efficiently: Use connection pooling and configure timeouts.
  \item Version control all schema changes: Use migration tools and review query plans for all DDL operations.
\end{itemize}

\subsection{1. Immediate rejection behaviors}
\begin{itemize}
  \item **Building SQL by string concatenating untrusted input.** This is the classic SQL injection failure: user-controlled data is interpreted as SQL code. Use parameters/bind variables; do not “sanitize later.” OWASP recommends prepared statements/parameterized queries, and MITRE CWE-89 describes the core failure as constructing SQL from externally influenced input without neutralizing special elements.
  \item **Relying on deny-lists, regex filters, or manual escaping as the primary SQL injection defense.** MITRE explicitly warns deny-lists are incomplete and that manual escaping/stored procedures can still fail when dynamic SQL is built unsafely.
  \item **Letting an application connect as a PostgreSQL superuser, table owner, or migration owner for normal runtime work.** Runtime code should use least-privilege roles. MITRE lists least privilege as a SQL injection mitigation, and PostgreSQL privileges can be granted or revoked per object and role.
  \item **Using \texttt{trust} authentication over TCP/IP in production.** The PostgreSQL wiki calls out that \texttt{host all all 0.0.0.0/0 trust} lets anyone who can reach the server authenticate as any PostgreSQL user, including superusers.
  \item **Shipping code that depends on app-only data integrity when the database can enforce it.** Primary keys, foreign keys, unique constraints, \texttt{NOT NULL}, and \texttt{CHECK} constraints exist to protect data regardless of which app, script, migration, or admin session writes to the database. PostgreSQL documentation explicitly frames constraints as the mechanism for controlling allowed data and raising errors on violations.
  \item **No tested restore path.** A backup that has not been restored is an assumption, not a recovery strategy. A GitHub SQL anti-pattern list calls out “not testing restores,” and GitLab’s 2017 database incident is a real-world reminder that backup mechanisms can silently fail.
  \item **Running destructive production migrations without a compatibility, rollback, lock, and data-safety plan.** GitLab’s migration guide treats avoiding downtime as a first-order requirement and documents multi-step approaches for destructive changes such as dropping columns.
  \item **Accepting AI/vibe-coded SQL without inspecting the generated SQL, constraints, query plans, migrations, and rollback behavior.** Current PostgreSQL/AI community work notes that AI tools often produce SQL that mixes dialects, misses modern PostgreSQL features, or omits constraints and indexes.
\end{itemize}

\subsection{1. SQL injection and unsafe dynamic SQL}
\begin{itemize}
  \item Concatenate, interpolate, template, or format untrusted input into SQL.
  \item Treat “internal” request fields as trusted: headers, cookies, JWT claims, admin forms, queue messages, CSV imports, webhooks, LLM output, or feature flags.
  \item Build dynamic \texttt{WHERE}, \texttt{ORDER BY}, \texttt{LIMIT}, \texttt{OFFSET}, table names, schema names, column names, operators, or direction strings directly from user input.
  \item Use escaping or regex sanitization as the main injection defense.
  \item Assume stored procedures are safe while they build SQL strings internally.
  \item Use PL/pgSQL \texttt{EXECUTE} with unsafely concatenated values or identifiers.
  \item Use ORM “raw SQL” APIs with string interpolation.
  \item Build \texttt{IN (...)} lists by joining user-provided strings.
  \item Accept user-provided SQL fragments for filtering, reporting, search, sorting, dashboards, or “advanced query” features.
  \item Expose detailed SQL errors, constraint names, query text, schema names, or stack traces to users.
  \item Log fully expanded SQL with secrets, passwords, tokens, session IDs, or PII.
  \item Assume client-side parameterization is enough if the final server query is still a string.
  \item Use dynamic identifiers without a strict allow-list mapping from user-facing names to known safe identifiers.
  \item Use “AI-generated safe SQL” as a substitute for bind parameters.
  \item Allow analytics, BI, or admin tools to run arbitrary SQL as an application owner role.
  \item Treat escaping quotes as enough for non-string contexts such as identifiers, operators, JSON paths, \texttt{ORDER BY}, or function names.
  \item Reuse a database connection after changing session state, role, \texttt{search\_path}, time zone, or temp objects without resetting it.
  \item Concatenating user input into SQL strings.
  \item Interpolating request parameters into \texttt{WHERE}, \texttt{ORDER BY}, \texttt{LIMIT}, \texttt{OFFSET}, \texttt{JOIN}, or \texttt{RETURNING} clauses.
  \item Treating “we escaped quotes” as a SQL injection defense.
  \item Building raw SQL from JSON/API payloads without strict allowlists.
  \item Letting users supply column names, table names, sort directions, operators, or function names without allowlisting.
  \item Using bind parameters for values but forgetting that identifiers still need allowlisting and safe quoting.
  \item Building PL/pgSQL dynamic SQL with raw variables instead of \texttt{format()}, \texttt{\%I}, \texttt{\%L}, \texttt{quote\_ident}, \texttt{quote\_literal}, \texttt{quote\_nullable}, or \texttt{USING}.
  \item Dollar-quoting dynamic variable content in PL/pgSQL and pretending it is safe.
  \item Using ORM “raw SQL” fragments with string interpolation.
  \item Passing unreviewed GraphQL/filter-builder input directly into SQL.
  \item Letting frontend state define arbitrary SQL predicates.
  \item Accepting AI-generated SQL that contains interpolated strings.
  \item Using “internal admin tool” as an excuse for injectable SQL.
  \item Relying on WAFs, API gateways, or input validation instead of parameterization.
  \item Exposing SQL error details to end users.
  \item Logging full SQL with secrets, tokens, passwords, session IDs, or PII.
  \item Using \texttt{EXECUTE} in stored procedures without proving every interpolated part is safe.
  \item Building \texttt{IN (...)} lists by joining raw strings instead of binding values safely.
  \item Using “trusted users” as the security model for SQL construction.
  \item Copy-pasting SQL from dashboards into code without parameter review.
\end{itemize}

\subsection{10. JSON/document behavior that is never excusable in Postgres design}
\begin{itemize}
  \item Putting core relational entities in JSON because “migrations are annoying.”
  \item Storing required fields in JSON with no constraints.
  \item Storing foreign keys inside JSON with no enforceable referential integrity.
  \item Storing values in JSON that must be indexed, joined, validated, and reported constantly.
  \item Allowing multiple incompatible JSON shapes in the same column without versioning.
  \item Querying deeply nested JSON in hot paths without indexes and statistics strategy.
  \item Using JSON as an EAV system.
  \item Using JSON for data that must participate in uniqueness constraints.
  \item Storing timestamps, money, booleans, and numbers as JSON strings.
  \item Letting application versions write different JSON shapes without compatibility tests.
  \item Having no migration plan from JSON to typed columns once fields become core.
  \item Treating JSON schema validation only in application code as enough for critical data.
\end{itemize}

\subsection{10. Migrations and DDL release behavior}
\begin{itemize}
  \item Make manual production schema changes outside version control.
  \item Run migrations no one reviewed.
  \item Run migrations generated by an ORM or AI without reading the SQL.
  \item Run migrations that require application downtime unless the system is intentionally offline and approved.
  \item Combine destructive schema changes with application deployment in one irreversible step.
  \item Make backward-incompatible changes without expand/contract rollout.
  \item Rename or drop columns before all running app versions stop using them.
  \item Drop tables before proving no code, jobs, reports, views, functions, or replicas depend on them.
  \item Add \texttt{NOT NULL}, defaults, generated columns, foreign keys, unique constraints, or indexes to large live tables without checking locks, rewrites, validation, and version-specific behavior.
  \item Add multiple foreign keys in one risky migration.
  \item Add foreign keys and leave them unvalidated forever.
  \item Add constraints without cleaning existing bad data.
  \item Create unique indexes without duplicate cleanup.
  \item Use non-concurrent index creation on large hot tables without a maintenance window.
  \item Assume concurrent index creation is safe without monitoring and failure cleanup.
  \item Run \texttt{ALTER TABLE} casually on large tables.
  \item Run broad \texttt{UPDATE} / \texttt{DELETE} migrations without batching.
  \item Write batch migrations that cannot resume.
  \item Write batch migrations without an upper bound, progress table, or break condition.
  \item Write batch migrations that can infinite-loop.
  \item Backfill data without measuring WAL, bloat, locks, replication lag, and job duration.
  \item Depend on current application code inside migrations that may be rerun later.
  \item Make migrations non-idempotent when deployment tooling may retry.
  \item Omit \texttt{lock\_timeout} and \texttt{statement\_timeout} for production migrations.
  \item Run DDL while long transactions are open.
  \item Run \texttt{VACUUM FULL}, \texttt{CLUSTER}, blocking \texttt{REINDEX}, or table rewrites during normal traffic without an explicit maintenance plan.
  \item Modify RLS policies, privileges, or security-definer functions without security review.
  \item Modify financial, identity, tenant, or audit tables without extra validation.
  \item Fail to test migration rollback or roll-forward.
  \item Fail to test migrations on production-like data volume.
  \item Treat staging success on tiny data as production proof.
  \item Run migrations without a fresh backup / PITR confidence.
  \item Ignore migration observability: locks, waits, rows processed, lag, errors, and rollback state.
  \item Leave schema drift between environments.
  \item Leave old columns, old indexes, old triggers, and old functions forever because cleanup is scary.
\end{itemize}

\subsection{10. Migrations, DDL, and backfill recklessness}
\begin{itemize}
  \item Changing production schema by hand outside versioned migrations.
  \item Running unreviewed SQL in production consoles.
  \item Running migrations generated by AI without reading every line.
  \item Running migrations without knowing table size.
  \item Running migrations without knowing lock level.
  \item Running migrations without \texttt{lock\_timeout}.
  \item Running migrations without \texttt{statement\_timeout}.
  \item Running migrations without a rollback or forward-fix plan.
  \item Running destructive migrations without fresh backups.
  \item Running destructive migrations without restore confidence.
  \item Combining schema change, code change, and data rewrite in one risky deploy.
  \item Dropping a column in the same deploy that stops using it.
  \item Renaming a column/table in one step while old code may still run.
  \item Changing column type on a large hot table without rewrite/lock analysis.
  \item Adding \texttt{NOT NULL} to an existing large table in one step.
  \item Adding a column with an expensive volatile default to a large table without understanding rewrite/lock behavior.
  \item Adding a foreign key to a large table without considering validation strategy.
  \item Adding a check constraint to a large table without considering validation strategy.
  \item Not using staged \texttt{NOT VALID} / \texttt{VALIDATE CONSTRAINT} patterns when appropriate.
  \item Creating a non-concurrent index on a large live table without a maintenance window.
  \item Dropping a large index in a way that creates avoidable blocking.
  \item Running huge backfills in one transaction.
  \item Running huge deletes in one transaction.
  \item Updating every row when only changed rows need updates.
  \item Backfilling without batching.
  \item Backfilling without progress logging.
  \item Backfilling without pause/resume.
  \item Backfilling without idempotency.
  \item Backfilling without throttling.
  \item Backfilling without monitoring replication lag.
  \item Backfilling without monitoring locks and vacuum impact.
  \item Backfilling without an index supporting the batch predicate.
  \item Running backfills through application code that may change before the migration finishes.
  \item Using ORM models inside migrations when model definitions can drift.
  \item Not testing migrations on production-like volume.
  \item Not replaying migrations from scratch in CI.
  \item Not testing rollback/forward-fix paths.
  \item Using \texttt{DROP CASCADE} because the database complained.
  \item Dropping constraints to make deploys pass.
  \item Leaving half-applied migrations unresolved.
  \item Not cleaning up invalid indexes after failed concurrent index creation.
  \item No release choreography for expand/contract changes.
  \item No compatibility window for rolling deploys.
  \item No migration owner watching production during risky changes.
\end{itemize}

\subsection{10. Postgres-specific footguns that are never justified to ignore}
\begin{itemize}
  \item Using app roles as superusers.
  \item Using schema-owner roles for app traffic.
  \item Forgetting default privileges for future tables, sequences, and functions.
  \item Forgetting sequence privileges.
  \item Assuming \texttt{SERIAL}/sequence increments roll back.
  \item Assuming sequence values prove creation order under concurrency.
  \item Assuming \texttt{now} string literals in stored definitions behave like \texttt{now()} calls.
  \item Using \texttt{'tomorrow'::date}, \texttt{'today'::date}, or \texttt{'now'::timestamp} in stored definitions where a function should be used.
  \item Marking table-reading functions \texttt{IMMUTABLE}.
  \item Marking time-dependent functions \texttt{IMMUTABLE}.
  \item Marking random or side-effect functions non-volatile.
  \item Using \texttt{SECURITY DEFINER} without fixed \texttt{search\_path}.
  \item Relying on RLS while app role owns the table or bypasses RLS.
  \item Writing RLS policies without \texttt{WITH CHECK}.
  \item Forgetting that \texttt{TRUNCATE} is not subject to row-level security.
  \item Leaving invalid indexes after failed concurrent index creation.
  \item Creating foreign keys and forgetting supporting indexes.
  \item Creating partial indexes and assuming all queries can use them.
  \item Creating multicolumn B-tree indexes while ignoring leading-column rules.
  \item Creating indexes concurrently inside a transaction block.
  \item Running ordinary \texttt{CREATE INDEX} on a write-heavy production table.
  \item Ignoring \texttt{ANALYZE} after large data changes.
  \item Disabling autovacuum.
  \item Letting long transactions prevent vacuum.
  \item Ignoring wraparound warnings.
  \item Relying on \texttt{search\_path} for object resolution in production-critical SQL.
  \item Using \texttt{ctid} as a permanent identifier.
  \item Using physical tuple order as business order.
  \item Using table inheritance as partitioning without understanding constraint and FK implications.
  \item Using JSONB for frequently updated large documents that could be normalized.
  \item Using GIN indexes casually on write-heavy JSONB workloads.
  \item Using \texttt{text} for everything because Postgres can store it.
  \item Using \texttt{varchar(n)} limits that have no business meaning.
  \item Using \texttt{money}/locale-sensitive formatting without understanding behavior.
  \item Using \texttt{timestamp without time zone} for cross-timezone event instants.
  \item Assuming \texttt{timestamptz} preserves the original timezone name.
  \item Assuming \texttt{time with time zone} solves scheduling.
  \item Using advisory locks as security controls.
  \item Creating extensions in production migrations without privilege, version, and environment checks.
  \item Depending on a local extension existing in production without migration-managed installation.
  \item Ignoring logical replication identity when tables are replicated and updated/deleted.
  \item Running bulk jobs on primary without \texttt{statement\_timeout}, batching, and monitoring.
  \item Letting \texttt{idle in transaction} sessions persist.
  \item Ignoring lock waits during migrations.
  \item Using \texttt{DROP CASCADE} instead of understanding dependencies.
\end{itemize}

\subsection{11. Backup, restore, and disaster-recovery malpractice}
\begin{itemize}
  \item No backups.
  \item “The replica is our backup.”
  \item “The dump probably works.”
  \item Backups with no restore test.
  \item Backups with no owner.
  \item Backups with no monitoring.
  \item Backups with no retention policy.
  \item Backups stored only on the same host.
  \item Backups stored only in the same account/project with the same blast radius.
  \item Backups not encrypted.
  \item Backup keys unavailable during disaster.
  \item Backup keys accessible to too many people.
  \item No point-in-time recovery plan for critical systems.
  \item No WAL archiving where RPO requires it.
  \item No documented RPO.
  \item No documented RTO.
  \item No restore runbook.
  \item No regular restore drill.
  \item No test of restoring into a clean environment.
  \item No test of restoring after accidental table drop.
  \item No test of restoring after bad migration.
  \item No logical dump strategy where logical restore is needed.
  \item No physical backup strategy where fast full restore is needed.
  \item Filesystem-level copies taken without a safe database backup method.
  \item Backups that silently exclude extensions, roles, privileges, schemas, or large objects.
  \item Backups that include secrets and PII but are treated as harmless files.
  \item Copying production backups into insecure dev laptops.
  \item No anonymization/synthetic-data strategy for lower environments.
  \item No verification of backup size/completeness.
  \item No alert on failed WAL archiving.
  \item No alert on replica lag if replicas are part of recovery.
  \item No plan for restoring across major Postgres versions.
\end{itemize}

\subsection{11. Backup, restore, and operational safety}
\begin{itemize}
  \item Run production without backups.
  \item Run production without testing restores.
  \item Assume a backup works because a job says “success.”
  \item Store backups in the same failure domain as the primary database.
  \item Store unencrypted backups containing sensitive data.
  \item Give broad backup access to people or systems that should not have production data.
  \item Run destructive migrations before confirming restore/PITR posture.
  \item Omit WAL archiving/PITR for systems where point-in-time recovery is required.
  \item Fail to define RPO and RTO.
  \item Fail to monitor disk space, WAL growth, replication slots, replication lag, locks, long transactions, autovacuum, bloat, errors, and slow queries.
  \item Let replication slots fill disks.
  \item Let logs fill disks.
  \item Use production as the only copy of critical data.
  \item Restore production data into development without masking or access controls.
  \item Keep data longer than policy permits because backups are unmanaged.
  \item Fail to test failover.
  \item Fail to test minor/major version upgrades.
  \item Run unsupported PostgreSQL versions.
  \item Ignore extension compatibility during upgrades.
  \item Ignore collation/version drift.
  \item Change time zone, locale, encoding, or collation assumptions without impact review.
  \item Let cron jobs, ETL, BI, and admin scripts mutate production without the same controls as application code.
  \item Have no runbook for corruption, accidental delete, bad migration, bad deploy, credential leak, or disk-full events.
  \item Have no owner for database health.
\end{itemize}

  \item Raw string SQL built from user input.
  \item No bind parameters for values.
  \item Dynamic identifiers without allow-lists.

\subsection{13. ORM and application-layer database behavior}
\begin{itemize}

  \item Accept ORM-generated migrations blindly.
  \item Accept ORM-default column types blindly.

\subsection{18. The “feature is okay, abuse is not” list}
\begin{itemize}
  \item **Raw SQL** is fine; raw SQL with interpolation is not.
  \item **Dynamic SQL** is fine for controlled administrative code; dynamic SQL from untrusted input is not.
  \item **JSONB** is fine for semi-structured data; JSONB as a substitute for relational integrity is not.
  \item **Soft delete** is fine for some audit/lifecycle models; soft delete that breaks uniqueness, FKs, privacy deletion, or query correctness is not.
  \item **Denormalization** is fine when measured; denormalization without consistency rules is not.
  \item **Triggers** are fine for carefully documented database-side behavior; invisible business logic nobody tests is not.
  \item **Stored procedures** are fine; procedures that concatenate unsafe SQL or bypass review are not.
  \item **Materialized views** are fine; stale materialized views used as truth without refresh semantics are not.
  \item **Caches** are fine; caches as the only source of truth are not.
  \item **Read replicas** are fine; ignoring lag and consistency is not.
  \item **Partitioning** is fine; table-per-month/customer hacks are not.
  \item **UUIDs** are fine; using IDs as secrets is not.
  \item **ORMs** are fine; assuming the ORM replaces schema design is not.
  \item **AI assistance** is fine; AI autonomy over production data is not.
\end{itemize}

\subsection{2. Authentication, roles, privileges, and RLS}
\begin{itemize}
  \item Use \texttt{trust} authentication over TCP/IP in production.
  \item Expose Postgres directly to the internet with permissive \texttt{pg\_hba.conf} rules.
  \item Let the application connect as \texttt{postgres}, a superuser, database owner, schema owner, migration role, or replication role.
  \item Give the runtime app role DDL privileges.
  \item Give read-only services write privileges.
  \item Give BI/reporting users write, DDL, owner, or broad production privileges.
  \item Use one shared database credential across apps, humans, jobs, CI, and migrations.
  \item Leave default or stale credentials active.
  \item Store database passwords in repos, prompts, tickets, logs, screenshots, shell history, or notebooks.
  \item Grant broad privileges to \texttt{PUBLIC}.
  \item Leave function execution broadly available when the function performs privileged operations.
  \item Depend on network location alone as the security boundary.
  \item Rely only on application-side tenant filters for sensitive multi-tenant data.
  \item Give an app role \texttt{BYPASSRLS}.
  \item Disable RLS “temporarily” in production without a controlled maintenance window and verification.
  \item Create permissive RLS policies that accidentally allow all rows.
  \item Use \texttt{SECURITY DEFINER} functions without a locked-down \texttt{search\_path}, schema-qualified object references, and restricted \texttt{EXECUTE} grants. PostgreSQL specifically warns to exclude writable schemas from \texttt{search\_path}, put \texttt{pg\_temp} last, and revoke public execution where needed.
  \item Put security-sensitive functions in writable schemas.
  \item Let untrusted users create objects in schemas that privileged code searches.
  \item Use ownership as an access-control model for applications.
  \item Run migrations as the same role used by the web app.
  \item Let AI tools, background jobs, or ETL systems share the web app’s credential.
  \item Fail to audit DDL, privilege changes, security policy changes, and mass writes.
  \item Ignore TLS/encryption requirements for untrusted networks.
  \item Treat backups, replicas, logs, query samples, and data exports as less sensitive than the primary database.
\end{itemize}

\subsection{20. “Not automatically bad” tools that become inexcusable when cargo-culted}
\begin{itemize}
  \item JSONB is fine for flexible/semi-structured data; it is bad when used to avoid schema design for core relational facts.
  \item Soft deletes are fine with a full lifecycle, uniqueness, FK, retention, and query strategy; they are bad as a random \texttt{deleted\_at} column slapped everywhere.
  \item Denormalization is fine for measured read performance; it is bad without source-of-truth, invalidation, and reconciliation.
  \item Materialized views are fine; they are bad without refresh semantics and staleness guarantees.
  \item Triggers are fine for database-owned invariants/auditing; they are bad as hidden, undocumented business logic.
  \item Stored procedures are fine; they are bad when they hide privilege escalation, unsafe dynamic SQL, or untestable behavior.
  \item Raw SQL is fine; it is bad when unparameterized, unreviewed, or dialect-misunderstood.
  \item ORMs are fine; they are bad when treated as a substitute for understanding SQL.
  \item UUIDs are fine; they are bad when write locality/index/storage consequences are ignored.
  \item Natural keys are fine for truly stable identifiers; they are bad when mutable/PII values become primary keys by accident.
  \item Enums are fine for stable finite domains; they are bad for fast-changing product taxonomy without migration planning.
  \item Partitioning is fine; it is bad when added before query/retention/access patterns are understood.
  \item Read replicas are fine; they are bad when used where fresh reads are required.
  \item Caching is fine; it is bad when it becomes the source of truth.
  \item \texttt{SELECT *} is fine for ad hoc exploration; it is bad in durable application contracts.
  \item Temporary tables are fine; they are bad when used to smuggle unbounded production work into request paths.
  \item Index hints/workarounds are sometimes useful in other databases; in Postgres, fighting the planner without statistics/query evidence is bad engineering.
  \item “NoSQL-style” flexible attributes are fine for genuinely unknowable sparse metadata; they are bad for known business entities.
  \item Application validation is fine; it is bad when it replaces database constraints for durable facts.
  \item Background jobs are fine; they are bad when they create race conditions or inconsistent invariants.
  \item Manual admin scripts are fine; they are bad without review, dry run, transaction strategy, and audit trail.
  \item Concatenating, interpolating, f-stringing, template-stringing, or string-formatting untrusted values into SQL.
  \item Treating “it came from the frontend,” “it came from an admin,” “it came from a webhook,” or “it came from an LLM” as trusted input.
  \item Building \texttt{WHERE}, \texttt{ORDER BY}, \texttt{LIMIT}, \texttt{OFFSET}, column names, table names, sort directions, operators, schema names, or function names directly from request parameters.
  \item Using escaping as the primary defense instead of bind parameters for values.
  \item Quoting bind parameters inside SQL text, for example \texttt{WHERE id = '\$1'}, which turns a parameter into text rather than a parameter.
  \item Allowing users to submit raw SQL fragments, raw filters, raw expressions, raw JSONPath/regex predicates, or arbitrary “advanced search syntax” that compiles directly into SQL.
  \item Letting generated SQL from an AI tool run against a real database before review, parameterization, tests, and an explicit migration/query plan.
  \item Allowing multiple SQL statements in a user-driven query path when only one statement is needed.
  \item Dynamically choosing identifiers without an allow-list. Parameterization protects values; identifiers require strict allow-listing and identifier quoting.
  \item Writing dynamic SQL inside PL/pgSQL with unsafe \texttt{EXECUTE} string concatenation.
  \item Using \texttt{SECURITY DEFINER} functions without a secure \texttt{search\_path}.
  \item Logging fully interpolated SQL statements that include passwords, tokens, session IDs, personal data, or secrets.
  \item “Fixing” injection risk by regex-stripping bad characters.
  \item “Fixing” injection risk by blacklisting keywords like \texttt{DROP}, \texttt{UNION}, or \texttt{--}.
  \item Assuming an ORM automatically protects raw SQL fragments.
  \item Assuming GraphQL, REST, RPC, or internal admin routes are not injection surfaces.
  \item Passing comma-joined ID lists into SQL instead of using proper array parameters or temporary/staging tables.
  \item Using \texttt{LIKE '\%user\_input\%'} without escaping wildcard semantics when the user input is meant to be literal text.
  \item Using database superuser credentials in a query path that also accepts external input.
  \item Using \texttt{trust} authentication over TCP/IP in production.
  \item Allowing application code to connect as \texttt{postgres}, a superuser, table owner, migration owner, or DBA role.
  \item Giving the application role \texttt{CREATE}, \texttt{DROP}, \texttt{ALTER}, \texttt{TRUNCATE}, \texttt{BYPASSRLS}, \texttt{SUPERUSER}, \texttt{CREATEDB}, \texttt{CREATEROLE}, or replication privileges without a documented need.
  \item Using one shared database account for humans, services, jobs, analytics, and migrations.
  \item Reusing the same database credentials across dev, staging, test, and production.
  \item Storing database credentials in source code, migrations, notebooks, committed \texttt{.env} files, Docker images, Slack messages, tickets, or SQL comments.
  \item Letting app logs, query logs, or error reports expose credentials, tokens, API keys, session IDs, reset tokens, or PII.
  \item Connecting to the database without TLS where the network is not fully trusted.
  \item Using weak or deprecated password authentication methods when stronger methods are available.
  \item Leaving default accounts, default passwords, unused roles, or stale contractor/employee access active.
  \item Granting broad \texttt{SELECT} on all tables to app roles, BI tools, notebooks, or support tooling.
  \item Granting write permissions to analytics/reporting users.
  \item Making the app role own the schema it queries.
  \item Failing to revoke unnecessary privileges from \texttt{PUBLIC}.
  \item Installing arbitrary extensions or procedural languages in production without security review.
  \item Treating “internal network” as authorization.
  \item Exposing Postgres directly to the public internet.
  \item Using RLS but connecting as a table owner/superuser or a role with \texttt{BYPASSRLS}, thereby bypassing the policy.
  \item Using RLS without tests that prove cross-tenant reads and writes fail.
  \item Assuming views, functions, triggers, or ORMs automatically enforce least privilege.
  \item Running migrations with production data from a developer laptop using personal superuser credentials.
  \item Storing passwords in plaintext.
  \item Storing passwords with reversible encryption so support staff or developers can “recover” them.
  \item Hashing passwords with MD5, SHA-1, SHA-256, SHA-512, or any fast general-purpose hash.
  \item Hashing passwords without unique salts.
  \item Using one global salt.
  \item Storing password reset tokens, magic login links, API tokens, OAuth tokens, MFA backup codes, or session tokens in plaintext.
  \item Keeping expired reset/session tokens forever.
  \item Letting application users query password/token columns unnecessarily.
  \item Replicating sensitive columns into analytics databases without masking or purpose limitation.
  \item Including sensitive values in audit logs, SQL logs, dead-letter queues, exception telemetry, or BI extracts.
  \item Using production personal data in dev/test without masking and access controls.
  \item Designing tables with no retention/deletion/anonymization strategy for regulated or sensitive data.
  \item “Soft deleting” sensitive data while continuing to expose it through joins, search indexes, materialized views, caches, or replicas.
  \item Tables without primary keys.
  \item Tables with duplicate rows that cannot be distinguished.
  \item Business-unique facts without \texttt{UNIQUE} constraints.
  \item Required attributes without \texttt{NOT NULL}.
  \item Domain rules enforced only in application code when the database can enforce them.
  \item Missing \texttt{CHECK} constraints for obvious invariants: positive quantities, valid percentages, nonnegative balances where required, start date before end date, nonempty strings, allowed ranges, valid status transitions where modelable.
  \item Missing foreign keys for real relationships.
  \item Using “foreign key by convention” names such as \texttt{user\_id} without an actual FK.
  \item Allowing orphan rows.
  \item Allowing child rows to point to deleted, hidden, wrong-tenant, or wrong-type parents.
  \item Using nullable foreign keys when the relationship is mandatory.
  \item Using \texttt{ON DELETE CASCADE} without explicit business approval of every cascaded deletion.
  \item Using \texttt{ON DELETE SET NULL} on data that must retain accountability.
  \item Using app code, cron jobs, or “cleanup scripts” to restore referential integrity after the fact.
  \item Creating polymorphic associations like \texttt{(entity\_type, entity\_id)} where the database cannot enforce valid parents.
  \item Using generic relationship tables that can link anything to anything without constraints.
  \item Encoding multiple states with contradictory booleans, for example \texttt{is\_active}, \texttt{is\_deleted}, \texttt{is\_archived}, \texttt{is\_pending}, \texttt{is\_cancelled}.
  \item Allowing impossible status combinations because no state machine or constraint exists.
  \item Using magic sentinel values such as \texttt{0}, \texttt{-1}, \texttt{9999-12-31}, \texttt{'unknown'}, or empty string instead of a clearly modeled \texttt{NULL}, separate status, or proper domain value.
  \item Treating \texttt{NULL}, empty string, zero, missing JSON key, and “unknown” as interchangeable.
  \item Duplicating the same fact in multiple tables without a declared source of truth and reconciliation mechanism.
  \item Storing derived totals as authoritative values without constraints, refresh logic, or consistency checks.
  \item Designing schemas where insert/update/delete anomalies are inevitable.
  \item Storing comma-separated IDs in a text column.
  \item Storing pipe-separated, JSON-stringified, or newline-separated lists in scalar columns.
  \item Storing multiple values in columns like \texttt{phone1}, \texttt{phone2}, \texttt{phone3}.
  \item Creating columns like \texttt{jan\_sales}, \texttt{feb\_sales}, \texttt{mar\_sales} instead of rows.
  \item Creating one table per month, year, customer, tenant, status, or region instead of using rows, partitioning, or a clear physical design.
  \item Encoding metadata in table names or column names.
  \item Using Entity-Attribute-Value for known attributes.
  \item Using JSONB as an excuse to avoid modeling stable entities and relationships.
  \item Using arrays of foreign keys where relationship rows are required.
  \item Using a spreadsheet import as the final schema.
  \item Designing one “god table” with hundreds of unrelated columns.
  \item Designing one “god JSONB” column with no constraints, generated columns, indexes, or schema validation.
  \item Designing generic \texttt{objects}, \texttt{attributes}, \texttt{relationships}, or \texttt{events} tables for everything when concrete entities are known.
  \item Mixing multiple entity types in one table without a discriminator, constraints, and type-specific validity rules.

  \item **Community signal:** Reddit discussions are useful for spotting recurring pain points, but I would not use Reddit or X as normative authority for a permanent engineering standard. I treated them as anecdotal signal and relied on official docs, engineering guides, and publications for the rules above.
\end{itemize}

\subsection{3. Data integrity and constraints}
\begin{itemize}
  \item Create persistent application tables with no primary key or candidate key.
  \item Use a surrogate \texttt{id} while failing to enforce the real business uniqueness rule.
  \item Enforce uniqueness only in application code.
  \item Enforce foreign-key relationships only in application code.
  \item Store relationship IDs in JSON, arrays, comma-separated strings, or polymorphic pairs when the database must enforce referential integrity.
  \item Use nullable columns for values that are required by the business.
  \item Use \texttt{NULL} as a magic status value when the domain has real states.
  \item Allow invalid enum/status/state transitions without constraints or checked transition logic.
  \item Omit \texttt{CHECK} constraints for values with hard domain limits: positive quantities, valid percentages, non-empty codes, valid date ranges, sane money amounts, allowed units.
  \item Allow duplicated emails, usernames, external IDs, order numbers, invoice numbers, idempotency keys, or natural keys when uniqueness matters.
  \item Disable constraints or triggers for imports and forget to re-enable and validate them.
  \item Leave \texttt{NOT VALID} constraints unvalidated indefinitely.
  \item Add constraints that do not actually match the business invariant.
  \item Use triggers as a substitute for basic \texttt{NOT NULL}, \texttt{UNIQUE}, \texttt{CHECK}, and \texttt{FOREIGN KEY} constraints.
  \item Use application tests as a substitute for database constraints.
  \item Let failed imports partially commit without a quarantine/reconciliation path.
  \item Use soft deletes in a way that breaks uniqueness, foreign keys, retention, or restore semantics.
  \item Hard-delete data that is legally or operationally required for audit.
  \item Soft-delete data that is legally required to be purged or anonymized.
  \item Use mismatched types between foreign-key columns and referenced keys.
  \item Use \texttt{ON DELETE CASCADE} without understanding the blast radius.
  \item Omit \texttt{ON DELETE} / \texttt{ON UPDATE} behavior decisions and rely on defaults by accident.
  \item Store amounts without currency.
  \item Store measurements without units.
  \item Store local times without location/time-zone semantics when they represent real-world events.
  \item Store “latest state” only when history, auditability, or reconciliation is required.
  \item Let background jobs repair integrity that the database could have prevented.
  \item Use \texttt{SELECT DISTINCT} or cleanup jobs to hide duplicate data that should be impossible.
  \item Accept orphaned rows as “normal.”
  \item Allow cross-tenant foreign keys or references without tenant scoping.
  \item Fail to test constraints with bad data, not just happy-path inserts.
  \item Use \texttt{CHECK} constraints for cross-row or cross-table invariants that require \texttt{UNIQUE}, \texttt{EXCLUDE}, \texttt{FOREIGN KEY}, or carefully designed triggers; PostgreSQL notes that \texttt{CHECK} constraints are intended for immutable row-level checks.
\end{itemize}

\subsection{3. Data-integrity and schema-design behavior that is never justified}
\begin{itemize}
  \item Creating a persistent, mutable business table with no way to uniquely identify a row.
  \item Omitting a primary key or equivalent unique, non-null identifier because “the app knows.”
  \item Allowing duplicate logical rows when the business domain says duplicates are impossible.
  \item Relying on application code alone for uniqueness.
  \item Relying on application code alone for referential integrity.
  \item Omitting foreign keys for real relationships without another explicit, tested integrity mechanism.
  \item Allowing orphan rows accidentally.
  \item Creating foreign-key-looking columns such as \texttt{user\_id}, \texttt{account\_id}, or \texttt{tenant\_id} that do not actually reference anything.
  \item Using \texttt{text} or \texttt{integer} for relationship fields that should be foreign keys.
  \item Using polymorphic references like \texttt{(entity\_type, entity\_id)} without enforceable integrity rules.
  \item Creating many-to-many relationships without a junction table or equivalent relational structure.
  \item Storing comma-separated IDs in one column.
  \item Storing arrays of IDs as a substitute for a relationship table when those IDs must be joined, constrained, searched, or deleted safely.
  \item Storing stable, queryable business attributes only inside unvalidated JSONB.
  \item Using JSONB as a dumping ground because schema design is hard.
  \item Using EAV for stable, known attributes that need constraints, joins, indexes, and readable queries.
  \item Creating \texttt{field1}, \texttt{field2}, \texttt{field3}, \texttt{phone\_1}, \texttt{phone\_2}, \texttt{tag\_1}, \texttt{tag\_2} instead of a child table.
  \item Creating one column per month, year, status, region, or category instead of modeling the value as data.
  \item Creating one table per customer, tenant, month, year, or status without a real partitioning/sharding design.
  \item Encoding data in object names: \texttt{orders\_2025}, \texttt{orders\_2026}, \texttt{paid\_users}, \texttt{unpaid\_users}, \texttt{tenant\_123\_orders}.
  \item Using nullable columns where the value is required for a valid row.
  \item Using \texttt{NOT NULL} where missingness is meaningful and should be represented.
  \item Using sentinel values such as \texttt{0}, \texttt{-1}, \texttt{9999-12-31}, \texttt{'N/A'}, \texttt{'unknown'}, or empty string instead of modeling NULL or a domain value correctly.
  \item Treating NULL as false, zero, empty string, or “not applicable” without distinction.
  \item Creating nullable booleans when the domain is only two-state.
  \item Creating non-null booleans when the domain actually has “unknown,” “not asked,” or “not applicable.”
  \item Failing to add \texttt{CHECK} constraints for obvious ranges: nonnegative prices, valid percentages, valid time intervals, valid quantities.
  \item Using \texttt{float}, \texttt{real}, or \texttt{double precision} for money or exact accounting values.
  \item Storing dates as strings.
  \item Storing prices as strings.
  \item Storing numeric values with currency symbols in the same column.
  \item Splitting dates into \texttt{year}, \texttt{month}, \texttt{day} integers when a date type is required.
  \item Storing absolute instants as \texttt{timestamp without time zone} without a strong reason.
  \item Assuming \texttt{timestamptz} stores the original timezone name. It stores an instant and displays it according to session timezone.
  \item Using \texttt{time with time zone} for real-world scheduling without understanding that timezones need dates for DST rules.
  \item Using quoted mixed-case identifiers everywhere.
  \item Naming columns with reserved words like \texttt{user}, \texttt{order}, \texttt{select}, or \texttt{group}.
  \item Using generic columns like \texttt{data}, \texttt{info}, \texttt{value}, \texttt{type}, \texttt{status}, or \texttt{metadata} without constraints or documented meaning.
  \item Using free-text status fields where only a controlled set of statuses is valid.
  \item Using an enum for business values that change frequently, without a migration plan.
  \item Duplicating derived data without declaring the source of truth.
  \item Denormalizing without tests or constraints that detect drift.
  \item Maintaining totals, counters, balances, or inventory quantities without transactional correctness.
  \item Storing balances as mutable facts instead of deriving from or reconciling with ledger-like events where correctness matters.
  \item Updating audit/history tables in app code only, with no transactional guarantee.
  \item Soft-deleting rows without deciding what happens to uniqueness, foreign keys, cascades, restores, retention, and legal deletion.
  \item Hard-deleting rows where audit, compliance, restore, or downstream references require a retention model.
  \item Cascading deletes without knowing the blast radius.
  \item Refusing cascade/delete rules and then leaving manual cleanup to hope.
  \item Treating lookup/reference data as magic constants scattered through application code.
  \item Creating tenant-scoped tables without including tenant identity in keys, uniqueness rules, RLS, indexes, and tests.
  \item Allowing cross-tenant joins or references unless they are explicitly part of the model.
  \item Using \texttt{ctid}, \texttt{xmin}, physical row order, sequence gaps, or insertion order as durable business identity.
  \item Assuming sequence values are gapless or rollback on transaction abort.
  \item Relying on implicit casts for important semantics.
  \item Allowing invalid states temporarily and never cleaning them up.
  \item Designing the schema only around one screen or one API response instead of the underlying facts.
  \item Allowing “schema later” for data that already has known relational structure.
  \item Creating wide sparse tables because “joins are bad.”
  \item Using database comments, docs, and diagrams nowhere for a complex domain.
\end{itemize}

  \item Letting support tooling bypass tenant boundaries.
  \item Relying only on application code for tenant isolation in a high-risk multi-tenant system.
  \item Missing \texttt{tenant\_id} from tenant-owned tables.
  \item Having \texttt{tenant\_id} columns but no tenant-scoped foreign keys or unique constraints.
  \item Joining tenant-owned tables without tenant predicates.
  \item Using globally unique IDs as an excuse to skip tenant constraints.
  \item Disabling row-level security “temporarily” and forgetting to restore it.

  \item Treating “internal admin” as authorization.
  \item Hiding tenant filters in app helper functions instead of enforcing boundaries where possible.
  \item No tests that prove cross-tenant reads/writes fail.
\end{itemize}

\subsection{3. Schema and data-modeling bad behavior}
\begin{itemize}
  \item **Tables without primary keys.** PostgreSQL docs describe primary keys as unique and non-null, and state that every table should usually have one.
  \item **Transactional relationships without foreign keys.** App-only referential integrity is not enough when data can be written by migrations, jobs, imports, consoles, multiple apps, or future services.
  \item **Using “we enforce it in the app” as a reason not to add \texttt{UNIQUE}.**
  \item **Using “we validate it in the app” as a reason not to add \texttt{NOT NULL}.**
  \item **Using “we validate it in the app” as a reason not to add \texttt{CHECK} constraints for simple domain rules.**
  \item **Nullable-by-default schemas.** Unknown/optional values are real, but “everything nullable” creates silent invalid states.
  \item **Boolean columns that allow \texttt{NULL} without a defined third meaning.**
  \item **Status columns with arbitrary text and no allowed-value constraint, enum, lookup table, or domain rule.**
  \item **No uniqueness constraint for natural uniqueness: emails, slugs, external IDs, provider IDs, idempotency keys, usernames, etc.**
  \item **Uniqueness enforced by \texttt{SELECT} then \texttt{INSERT}.** That is a race condition; use a unique constraint and \texttt{INSERT ... ON CONFLICT}.
  \item **Missing composite uniqueness for scoped uniqueness.** Example: \texttt{slug} unique globally when it should be unique per tenant, or not unique per tenant when it must be.
  \item **Foreign keys that point only to \texttt{id} while ignoring tenant scope in multi-tenant systems.**
  \item **Foreign keys with the wrong \texttt{ON DELETE} behavior.** PostgreSQL docs note \texttt{CASCADE} fits component/dependent rows, while independent objects often need \texttt{RESTRICT}/\texttt{NO ACTION}; choosing this casually can delete or orphan important data.
  \item **Using \texttt{ON DELETE CASCADE} because cleanup is annoying.**
  \item **Using \texttt{ON DELETE SET NULL} when the business relationship is actually mandatory.**
  \item **Soft deletes everywhere without uniqueness, FK, partial-index, and retention semantics.**
  \item **Soft-deleted rows that still block legitimate unique values forever.**
  \item **Soft-deleted parent rows with live child rows and no policy for visibility, restore, or purge.**
  \item **Many-to-many relationships encoded as comma-separated IDs, JSON arrays, or text blobs when referential integrity/queryability matters.** PostgreSQL docs show many-to-many relationships modeled with a join table.
  \item **The “Jaywalking” anti-pattern: storing multiple foreign keys in one delimited string.** This is one of the classic SQL antipatterns cataloged in *SQL Antipatterns*.
  \item **Entity-Attribute-Value for normal structured business entities.** EAV can be appropriate for sparse/extensible metadata, but not as a replacement for a schema when fields are known, constrained, indexed, and queried.
  \item **One true lookup table for unrelated concepts.** Community discussions repeatedly flag this because it weakens foreign keys and domain meaning.
  \item **One giant \texttt{metadata JSONB} column for core relational data.**
  \item **Using JSONB to avoid migrations.**
  \item **Using JSONB to avoid foreign keys.**
  \item **Using JSONB to avoid type checks.**
  \item **Using JSONB to hide an unstable product model rather than designing extension points.**
  \item **Creating a new table per tenant, customer, month, status, event type, or feature flag when a normal relational model or partitioning should be used.**
  \item **Encoding hierarchy as fragile strings without constraints when recursive queries, closure tables, materialized paths, or adjacency lists are appropriate.**
  \item **No source-of-truth distinction between raw facts, derived facts, cached facts, and denormalized facts.**
  \item **Storing derived values with no invalidation, recomputation, reconciliation, or ownership.**
  \item **Duplicating the same business fact in multiple tables without a rule for which copy wins.**
  \item **Using floating point for money.**
  \item **Blindly using PostgreSQL \texttt{money}.** The PostgreSQL wiki advises against \texttt{money}; \texttt{numeric} plus an explicit currency column is usually the safer model.
  \item **Currency amounts without currency codes.**
  \item **Measurements without units.**
  \item **Timestamps without a clear time-zone policy.**
  \item **Using \texttt{timestamp without time zone} for real-world points in time.** The PostgreSQL wiki says to use \texttt{timestamptz} for points in time.
  \item **Storing UTC in \texttt{timestamp without time zone}.** The PostgreSQL wiki explicitly warns against this because it hides the semantic meaning.
  \item **Using \texttt{timetz}.** The PostgreSQL wiki says not to use it.
  \item **Using \texttt{CURRENT\_TIME}.** The PostgreSQL wiki says not to use it.
  \item **Using \texttt{timestamp(0)} or \texttt{timestamptz(0)} to “drop milliseconds.”** PostgreSQL rounds, not truncates; the wiki says not to use these.
  \item **Using text offsets like \texttt{-07:00} as time-zone names.** Store actual instants and named zones where the named civil zone matters.
  \item **Using \texttt{char(n)}.** The PostgreSQL wiki says \texttt{char(n)} is almost never useful and recommends \texttt{text} plus constraints/domains where needed.
  \item **Using arbitrary \texttt{varchar(n)} lengths as fake validation.** The PostgreSQL wiki recommends \texttt{text} or unconstrained \texttt{varchar} unless the limit is a real business rule.
  \item **Using \texttt{serial} for new PostgreSQL designs.** The PostgreSQL wiki recommends identity columns for new applications.
  \item **Using SQL\_ASCII for normal text.** The PostgreSQL wiki warns that mixed encodings can create unrecoverable messes.
  \item **Quoted mixed-case identifiers.** \texttt{"UserProfile"} and \texttt{"createdAt"} force quoting forever and cause confusion; the PostgreSQL wiki recommends avoiding uppercase table and column names.
  \item **Reserved words as table or column names.**
  \item **Columns named \texttt{type}, \texttt{date}, \texttt{user}, \texttt{order}, \texttt{group}, or \texttt{desc} without understanding quoting and ambiguity costs.**
  \item **\texttt{id} columns whose meaning changes across tables and APIs without table-qualified names.**
  \item **No naming convention for PKs, FKs, unique constraints, indexes, and check constraints.**
  \item **Schema objects created manually in prod but missing from migrations.**
  \item **Migrations that do not represent the real schema state.**
  \item **Using database inheritance for new designs without a strong PostgreSQL-specific reason.** The PostgreSQL wiki warns against table inheritance for ordinary use and notes native partitioning replaced many historical inheritance uses.
  \item **Using PostgreSQL rules instead of triggers for ordinary behavior.** The PostgreSQL wiki says not to use rules and recommends triggers instead.
  \item **Modeling audit history by overwriting rows and hoping logs are enough.**
  \item **Audit tables without actor, time, source, old/new value, request/job ID, or retention policy.**
  \item **Using production tables as scratchpads for imports, one-off scripts, or temporary transformations.**
  \item **No ownership metadata for business-critical rows when ownership matters.**
  \item **No lifecycle columns where lifecycle matters: created, updated, archived, deleted, effective dates, validity windows.**
  \item **Validity windows with overlapping rows and no exclusion constraint or conflict prevention.**
  \item **Temporal data without a rule for “as of,” “effective at,” “recorded at,” and “corrected at.”**
  \item **Analytics facts with mutable dimensions and no slowly-changing-dimension policy.**
  \item **Event tables without idempotency keys or natural deduplication.**
  \item **Inbox/outbox tables without uniqueness, delivery state, retry limits, and cleanup policy.**
  \item **Polymorphic foreign keys that cannot be enforced.**
  \item **Generic \texttt{object\_type} + \texttt{object\_id} references for critical data without triggers, constraints, or redesign.**
  \item **A god table with hundreds of unrelated columns because joins are considered scary.** GitLab’s database docs warn that wide tables increase write amplification, WAL, vacuum overhead, and indexing costs.
  \item **High-frequency updates to a single counter/config row.** GitLab warns that high-frequency updates to a single row can create lock queues, pool saturation, vacuum pressure, and WAL pressure.
\end{itemize}

\subsection{4. Migration and DDL behavior that is never justified}
\begin{itemize}
  \item Making manual production schema changes that are not captured in version-controlled migrations.
  \item Editing an already-applied migration as if history can be rewritten.
  \item Letting environments drift with different migration histories.
  \item Auto-syncing ORM models to production schema.
  \item Running destructive DDL without a backup and restore plan.
  \item Running destructive DDL without knowing all application versions that still read or write the old schema.
  \item Dropping a table, column, index, constraint, enum value, or trigger because “nothing seems to use it.”
  \item Renaming a table or column in one step while old code may still be deployed.
  \item Changing column meaning without changing column name, constraints, documentation, and dependent code.
  \item Adding a \texttt{NOT NULL} constraint to a large existing table without a backfill/validation/lock plan.
  \item Adding a foreign key to a large existing table without a validation and locking strategy.
  \item Adding a unique constraint to dirty data without finding and resolving duplicates first.
  \item Creating a normal index on a large production table when writes cannot be blocked.
  \item Forgetting to clean up invalid indexes after failed \texttt{CREATE INDEX CONCURRENTLY}.
  \item Creating indexes during peak traffic without understanding extra CPU and I/O load.
  \item Backfilling a large table in one transaction.
  \item Updating millions of rows without batching, throttling, monitoring, and retry.
  \item Deleting millions of rows in one transaction when batching or partition drop is appropriate.
  \item Migrating data without idempotency.
  \item Migrating data without recording progress.
  \item Migrating data without safe retry behavior.
  \item Disabling triggers or constraints for a migration and not revalidating.
  \item Disabling foreign keys because the import is “trusted.”
  \item Using \texttt{DROP CASCADE} casually.
  \item Running \texttt{TRUNCATE} on production tables without understanding locks, foreign keys, replicas, and audit requirements.
  \item Running \texttt{VACUUM FULL}, \texttt{CLUSTER}, or table rewrites on production without understanding blocking.
  \item Changing column types on large tables without checking whether it rewrites the table.
  \item Adding defaults, generated columns, or expressions without understanding rewrite and lock behavior.
  \item Adding volatile defaults during large migrations without cost analysis.
  \item Creating enum changes with no downgrade path.
  \item Shipping app code that writes new columns before the column exists everywhere.
  \item Dropping old columns before all deployed code has stopped reading them.
  \item Adding required columns before all deployed code writes them.
  \item Failing to use expand/contract migrations for incompatible app/schema changes.
  \item Running migrations without \texttt{lock\_timeout} and \texttt{statement\_timeout} policies.
  \item Running migrations from a laptop against production.
  \item Running migrations without CI on a clean database and an upgraded realistic database.
  \item Running migrations without rollback output or roll-forward remediation.
  \item Running migrations without reviewing generated SQL.
  \item Running migrations without query plans for new nontrivial queries.
  \item Running migrations without database-owner review for high-risk schema changes.
  \item Hiding data migrations inside app startup.
  \item Hiding schema mutations inside request paths.
  \item Creating triggers/functions/views in production manually because “the migration tool is annoying.”
  \item Using migrations to repair data silently without audit.
  \item Treating staging success on tiny data as proof a production migration is safe.
  \item Applying migrations out of order.
  \item Skipping migrations in one environment and “catching up” manually.
  \item Making irreversible migrations for convenience when a reversible design is practical.
  \item Forgetting replicas, logical replication, downstream warehouses, CDC, and backup tooling when changing schema.
\end{itemize}

  \item Removing foreign keys to make tests or imports easier.
  \item Creating orphan rows as normal behavior.

  \item Using \texttt{ON DELETE CASCADE} on important business data without explicit domain approval.
  \item Using \texttt{ON DELETE SET NULL} where null would create invalid state.

  \item Trusting a migration succeeded without checking constraint validity.

\subsection{4. Query correctness bad behavior}
\begin{itemize}
  \item **\texttt{SELECT *} in application code, APIs, migrations, views, exports, or long-lived jobs.** It creates hidden coupling to column order and future schema changes; GitHub’s anti-pattern list and GitLab’s query guidelines both flag implicit/ambiguous columns as dangerous.
  \item **\texttt{INSERT INTO table VALUES (...)} without an explicit column list.**
  \item **Relying on column order instead of column names.**
  \item **Unqualified column names in joins.** GitLab documents deploy-time failures from ambiguous or changed column selection and recommends explicit table-qualified columns in relevant cases.
  \item **Using \texttt{DISTINCT} to hide duplicate rows caused by a broken join.**
  \item **Using \texttt{GROUP BY} to hide broken cardinality.**
  \item **Using \texttt{LIMIT} without \texttt{ORDER BY} when deterministic results matter.**
  \item **Using \texttt{ORDER BY created\_at} alone when ties can change pagination; add a stable tiebreaker such as the primary key.**
  \item **Offset pagination for deep or mutable production lists when keyset/cursor pagination is required.**
  \item **Using \texttt{NOT IN} when the subquery/list can contain \texttt{NULL}.** PostgreSQL’s wiki warns that \texttt{NOT IN} behaves surprisingly with nulls and recommends \texttt{NOT EXISTS} in many cases.
  \item **Using \texttt{BETWEEN} for timestamp windows.** PostgreSQL’s wiki recommends half-open timestamp ranges, for example \texttt{>= start AND < end}.
  \item **Comparing to \texttt{NULL} with \texttt{=} or \texttt{<>} instead of \texttt{IS NULL} / \texttt{IS NOT NULL}.**
  \item **Forgetting SQL’s three-valued logic.** Reddit discussions about SQL null behavior are a useful community reminder, but the real issue is correctness: \texttt{TRUE}, \texttt{FALSE}, and \texttt{UNKNOWN} must be modeled deliberately.
  \item **Using \texttt{COUNT(*)} to test existence when \texttt{EXISTS} or \texttt{LIMIT 1} is the actual question.**
  \item **Using \texttt{COUNT(DISTINCT ...)} casually on large joins without plan inspection.**
  \item **Using \texttt{LEFT JOIN} and then filtering the right table in \texttt{WHERE}, accidentally turning it into an inner join.**
  \item **Moving predicates between \texttt{ON} and \texttt{WHERE} without understanding outer join semantics.**
  \item **Using \texttt{UNION} when \texttt{UNION ALL} is correct and deduplication is unnecessary.**
  \item **Using \texttt{UNION ALL} when duplicates are semantically wrong.**
  \item **Changing one branch of a \texttt{UNION} without keeping column count, order, and types stable.** GitLab documents deployment failures from inconsistent \texttt{SELECT} columns in \texttt{UNION} queries after schema changes.
  \item **Sending giant \texttt{IN (...)} lists from application memory.** GitLab warns about queries with huge integer lists, accidental full scans, and large query text; prefer joins, subqueries, temp tables, \texttt{unnest}, or server-side sets when appropriate.
  \item **Plucking IDs into the app just to send them back to the database.** GitLab explicitly warns against moving query logic into application memory when PostgreSQL can optimize the full query with subqueries.
  \item **N+1 queries hidden behind ORM relationships.**
  \item **ORM scopes that look harmless but generate unbounded joins, subqueries, or scans.**
  \item **Using leading-wildcard \texttt{LIKE} / \texttt{ILIKE} on large tables without trigram/full-text support.** GitLab notes wildcard searches at the start of a pattern cannot use ordinary indexes and points to trigram indexes as a possible solution.
  \item **Using \texttt{LIKE '\%term\%'} as a search engine.** Use full-text search, trigram indexes, or an external search system when requirements exceed simple indexed lookup.
  \item **Using \texttt{ORDER BY random()} on large tables.**
  \item **Applying functions to indexed columns in predicates without a matching expression index.** Example: \texttt{lower(email) = ...} without an index on \texttt{lower(email)}.
  \item **Implicit type casts on indexed columns because app parameters use the wrong type.**
  \item **String comparison for numeric, date, UUID, boolean, or enum values.**
  \item **Storing dates as text and then sorting lexicographically.**
  \item **Using local time arithmetic for global instants.**
  \item **Using \texttt{now()} / \texttt{clock\_timestamp()} inconsistently without knowing transaction timestamp semantics.**
  \item **Using \texttt{COALESCE} in predicates to paper over bad null modeling and defeat indexes.**
  \item **Using \texttt{OR} filters that defeat indexes without rewriting, partial indexes, \texttt{UNION}, or plan validation.**
  \item **Unbounded \texttt{UPDATE} or \texttt{DELETE} without a \texttt{WHERE} clause, batch limit, transaction boundary, and explicit review.**
  \item **Using a CTE or join in an update/delete where the intended scope does not actually constrain the target.** GitLab documents a case where a CTE plus \texttt{update\_all} risked updating an entire table and recommends bounded/batched updates where volume is uncertain.
  \item **Assuming CTEs, subqueries, joins, and views are performance-equivalent without checking the generated plan.**
  \item **Using views as a dumping ground for business logic that nobody tests.**
  \item **Using triggers to hide behavior that application developers cannot discover.** Triggers can be correct, but invisible side effects without docs/tests are not.
  \item **Recursive queries without cycle prevention or depth bounds.**
  \item **Window functions without deterministic ordering.**
  \item **Aggregates without a correct grouping key.**
  \item **Time-bucket queries that ignore time zones, daylight-saving transitions, or boundary semantics.**
  \item **Reporting queries that silently drop rows because of inner joins to optional dimensions.**
  \item **Backfills that compute values differently from production writes.**
  \item **Migrations that update data with application logic that has since changed.**
  \item **Using \texttt{NULL} and empty string interchangeably.**
  \item **Using magic sentinel values such as \texttt{0}, \texttt{-1}, \texttt{'unknown'}, or \texttt{1970-01-01} instead of a modeled state.**
  \item **Using \texttt{SELECT ... FOR UPDATE} without understanding what rows are actually locked.**

  \item **Relying on implicit transaction behavior in client libraries.**
  \item **Letting generated SQL differ silently between development, staging, and production.**
  \item **Not testing query behavior against realistic data distributions.**
\end{itemize}

\subsection{4. Schema design and modeling smells}
\begin{itemize}
  \item Store comma-separated lists in a column instead of a junction table.
  \item Store multiple values in \texttt{phone1}, \texttt{phone2}, \texttt{phone3} columns instead of a child table.
  \item Use entity-attribute-value tables for core domain data simply to avoid migrations.
  \item Use JSON/JSONB as a dumping ground because you do not want to design tables.
  \item Use \texttt{owner\_type} / \texttt{owner\_id} polymorphic associations where the database cannot enforce the referenced row.
  \item Create one “god table” for unrelated concepts.
  \item Create table-per-customer, table-per-tenant, table-per-month, table-per-status, or table-per-type designs where partitioning or normalization is the right tool.
  \item Encode metadata in table names or column names.
  \item Mix unrelated lifecycles in the same table.
  \item Use one generic \texttt{data} table for many business entities.
  \item Use meaningless columns like \texttt{value}, \texttt{type}, \texttt{flag}, \texttt{data}, \texttt{status2}, or \texttt{misc} without a documented domain.
  \item Create boolean-flag explosions where combinations represent hidden states.
  \item Store derived data as the only source of truth without a recomputation strategy.
  \item Denormalize without proving the read benefit and documenting invalidation/refresh rules.
  \item Normalize so poorly that every query needs application-side joins or repeated lookups.
  \item Create circular dependencies without a clear ownership/lifecycle model.
  \item Use surrogate IDs everywhere while ignoring natural uniqueness.
  \item Clone schemas for every customer instead of designing tenancy intentionally.
  \item Create “future proof” columns such as \texttt{custom1}, \texttt{custom2}, \texttt{extra}, \texttt{reserved}, or \texttt{spare}.
  \item Mix OLTP event writes, mutable current state, analytics rollups, and audit data in one table without clear boundaries.
  \item Put large, cold, rarely used blobs in hot transactional rows.
  \item Store files in the database when the system actually requires object-store semantics, or store file paths only when transactional consistency with metadata is required.
  \item Fail to document non-obvious tables, columns, functions, triggers, policies, and invariants.
  \item Use quoted mixed-case identifiers, reserved words, spaces, or punctuation in identifiers.
  \item Design from ORM defaults without reviewing PostgreSQL types, indexes, constraints, and migrations.
  \item Treat “we can change it later” as an excuse for a schema that cannot safely migrate later.
\end{itemize}

\subsection{5. Performance and physical-design bad behavior}
\begin{itemize}
  \item **Merging performance-sensitive SQL without \texttt{EXPLAIN}.** PostgreSQL’s \texttt{EXPLAIN} exists to show the planner’s chosen plan, and PostgreSQL docs emphasize that plan choice is critical for performance.
  \item **Using \texttt{EXPLAIN ANALYZE} on side-effecting statements outside a transaction/rollback wrapper.** PostgreSQL docs note \texttt{EXPLAIN ANALYZE} actually executes the statement.
  \item **Testing only on empty or toy data.**
  \item **Ignoring row-count estimates, selectivity, and skew.**
  \item **No statistics maintenance.** PostgreSQL uses \texttt{ANALYZE} statistics for planning, and autovacuum/autoanalyze do not remove every need for manual awareness.
  \item **Disabling autovacuum globally.** PostgreSQL docs say routine vacuuming is required and that disabling autovacuum is unwise except in very narrow, predictable workloads.
  \item **Ignoring dead tuples, bloat, wraparound risk, and table/index growth.**
  \item **Running long transactions that hold old row versions and block cleanup.**
  \item **Leaving sessions \texttt{idle in transaction}.**
  \item **No \texttt{statement\_timeout} for application roles.**
  \item **No \texttt{lock\_timeout} for migrations.**
  \item **No protection against runaway analytical queries on the primary.**
  \item **No \texttt{pg\_stat\_statements} or equivalent query observability.** PostgreSQL’s \texttt{pg\_stat\_statements} tracks planning and execution statistics for SQL statements.
  \item **Index shotgun: indexing every column.** Indexes can speed retrieval but add write and maintenance overhead; PostgreSQL docs advise using them sensibly.
  \item **Missing indexes for critical foreign keys, joins, filters, and uniqueness checks.** PostgreSQL docs note foreign key declarations do not automatically create indexes on referencing columns, although such indexes are often useful.
  \item **Adding redundant indexes that differ only trivially.**
  \item **Keeping unused indexes forever.**
  \item **Adding indexes to large hot tables without concurrent/index-build strategy.**
  \item **Adding an index without proving it matches a real query shape.** PostgreSQL docs recommend checking index usage against the real workload with \texttt{EXPLAIN}.
  \item **Using the wrong index type or operator class: B-tree when trigram, GIN, GiST, BRIN, hash, partial, expression, or composite indexes are the actual fit.**
  \item **Composite indexes with columns in the wrong order for the workload.**
  \item **Partial indexes whose predicate does not match application queries.**
  \item **Expression indexes whose expression does not exactly match query expressions.**
  \item **Case-insensitive lookups without a case-insensitive strategy: normalized column, expression index, \texttt{citext}, or appropriate collation.**
  \item **Creating indexes to rescue bad data modeling without fixing the model.**
  \item **Ignoring write amplification from indexes.**
  \item **Bulk updates/deletes on large tables without batching.**
  \item **Large backfills with no pause/resume, progress tracking, throttling, or retry.**
  \item **High-frequency updates to a single row or narrow set of rows.** GitLab explicitly warns this can create lock queues, pool saturation, vacuum pressure, and WAL pressure.
  \item **Using the primary database as an unbounded job queue without queue semantics.**
  \item **Queues without \texttt{SKIP LOCKED}, lease timeouts, retry limits, visibility state, dead-letter handling, and cleanup.**
  \item **Unbounded result sets returned to application memory.**
  \item **No pagination on user-facing list endpoints.**
  \item **Export queries that run on the primary during peak traffic.**
  \item **Expensive reports coupled to OLTP tables without replicas, snapshots, summaries, or workload isolation.**
  \item **Frequent \texttt{VACUUM FULL} on live tables as routine maintenance.** PostgreSQL docs note \texttt{VACUUM FULL} requires an \texttt{ACCESS EXCLUSIVE} lock and cannot run in parallel with normal use of the table.
  \item **Assuming partitioning automatically improves performance.**
  \item **Partitioning without pruning-friendly predicates.**
  \item **Too many partitions without planning for maintenance overhead.**
  \item **No retention policy for event, audit, log, notification, or job tables.**
  \item **No archiving policy for cold data.**
  \item **Wide hot tables with many cold columns.** GitLab warns that wide tables increase overhead and recommends splitting cold or frequently updated columns in appropriate cases.
  \item **Storing large blobs in the main OLTP table when object storage or separate blob tables are the better operational design.**
  \item **Creating materialized views without refresh strategy, locking strategy, and staleness semantics.**
  \item **Refreshing materialized views synchronously on user requests.**
  \item **Relying on planner luck instead of stable query/index design.**
  \item **Ignoring connection count.**
  \item **Opening a new DB connection per request/job without pooling.**
  \item **Letting worker concurrency exceed DB capacity.**
  \item **No backpressure when the database is saturated.**
  \item **No load test for the database path.**
  \item **No slow-query budget.**
  \item **No plan-regression checks for critical queries.**
  \item **Treating the database as infinitely scalable because “Postgres is fast.”**
\end{itemize}

\subsection{6. Transaction and concurrency bad behavior}

  \item **Retrying failed jobs that are not safe to retry.**

\subsection{A compact merge-gate checklist}
\begin{itemize}
  \item What invariant is this enforcing, and is it enforced in the database where possible?
  \item What data volume exists today, and what volume is expected later?
  \item What locks will this take?
  \item What is the rollback or forward-fix?
  \item What happens if it fails halfway?
  \item What is the query plan on realistic data?
  \item What indexes support the query and constraints?
  \item What credentials/roles can run this?
  \item What sensitive data is read, written, logged, exported, or retained?
  \item What concurrency race would break this?
  \item What monitoring proves it is safe after deploy?
  \item What restore path exists if the worst case happens?

\subsection{Anti-relational designs}
\begin{itemize}
  \item Comma-separated IDs in a column.
  \item JSON arrays of foreign keys instead of a join table.
  \item Repeating columns like \texttt{phone1}, \texttt{phone2}, \texttt{phone3}.
  \item Repeating attribute columns like \texttt{tag\_1}, \texttt{tag\_2}, \texttt{tag\_3}.
  \item Table-per-month/year/customer/tenant when native partitioning or a tenant key is the correct model.
  \item One “god table” for unrelated entity types.
  \item One “god lookup table” for unrelated domains where foreign keys become meaningless.
  \item Entity-Attribute-Value as the default schema because the model was not understood. Bill Karwin’s *SQL Antipatterns* explicitly groups dangerous antipatterns across logical design, physical design, queries, and application development, and EAV is one of the classic patterns surfaced repeatedly in SQL anti-pattern discussions.
  \item JSONB as an excuse to avoid modeling stable, queryable, relational data.
  \item Storing structured data as unvalidated text blobs.
  \item Storing numbers, booleans, timestamps, or money as text.
  \item Storing multiple meanings in one column.
  \item Storing one concept in multiple columns with inconsistent rules.
  \item Duplicating data as if both copies are authoritative.
  \item Denormalizing before proving the normalized model cannot meet requirements.
  \item Normalizing into unusability without understanding access patterns.
  \item Creating a schema directly from UI forms rather than domain entities and invariants.
  \item Letting ORM defaults define the data model without database review.
  \item “We can migrate later” as an excuse for a schema that cannot enforce today’s truth.
  \item No ERD, relationship map, or written invariants for a nontrivial schema.
\end{itemize}

\subsection{Anti-relational schema design}
\begin{itemize}
  \item Storing comma-separated IDs in a text column.
  \item Storing pipe-delimited, JSON-delimited, or array-packed foreign keys instead of using a join table.
  \item Storing multiple logical values in one scalar column.
  \item Creating “list of IDs” columns that cannot be constrained with foreign keys.
  \item Creating many-to-many relationships without a junction table and appropriate uniqueness.
  \item Using entity-attribute-value design for core business entities just to avoid migrations.
  \item Using one giant \texttt{jsonb} column for relational data that needs constraints, joins, reporting, or indexing.
  \item Creating one table per customer, tenant, month, status, region, or type without a partitioning or lifecycle design.
  \item Creating a “god table” with hundreds of nullable unrelated columns.
  \item Creating polymorphic associations where the database cannot enforce the referenced target.
  \item Using a generic \texttt{object\_id} that can point to anything.
  \item Using a generic \texttt{type} plus \texttt{id} pair instead of enforceable foreign keys.
  \item Encoding values in table names or column names instead of rows.
  \item Creating a column per day, per month, per feature flag, per metric, or per user.
  \item Using arrays or JSON to hide normalization problems.
  \item Repeating groups instead of child tables.
  \item Duplicating data across tables without a single source of truth, synchronization rule, or reconciliation job.
  \item Creating “shadow tables” manually maintained by application code with no consistency guarantees.
\end{itemize}

\subsection{Bad keys and identity design}
\begin{itemize}
  \item Using a surrogate \texttt{id} while allowing duplicate real-world business records.
  \item Failing to define the natural/business uniqueness of an entity.
  \item Using mutable external provider IDs as the only identity.
  \item Using email as a primary key when email changes are allowed.
  \item Using names, titles, slugs, or display labels as permanent identifiers.
  \item Using globally meaningful IDs without tenant scoping in multitenant systems.
  \item Forgetting tenant ID in unique constraints, foreign keys, indexes, and row-level policies.
  \item Letting two tenants accidentally reference each other’s rows.
  \item Using UUIDs as a substitute for authorization or tenant isolation.
  \item Treating sequence gaps as errors.
  \item Exposing internal sequential IDs in places where enumeration is a security risk without separate public identifiers.
\end{itemize}

\subsection{Broken authentication and authorization}
\begin{itemize}
  \item Using \texttt{trust} authentication over TCP/IP.
  \item Letting application code connect as \texttt{postgres}, a superuser, or the schema owner.
  \item Sharing one powerful database account across humans, services, migrations, and dashboards.
  \item Granting broad privileges to \texttt{PUBLIC} without deliberate review.
  \item Granting \texttt{ALL} because the exact needed privileges are inconvenient to identify.
  \item Letting web applications run migrations with the same role they use for normal traffic.
  \item Giving read-only tools write privileges.
  \item Giving background jobs DDL privileges.
  \item Running ad hoc production queries as a superuser.
  \item Using \texttt{SECURITY DEFINER} functions without a locked-down \texttt{search\_path} and privilege review.
  \item Granting \texttt{BYPASSRLS} casually.
  \item Enabling row-level security but forgetting that table owners and superusers can bypass it unless configured carefully.
  \item Depending only on application-side tenant filters when the database can enforce tenant isolation.
  \item Leaving default privileges unreviewed after creating new schemas, tables, functions, or extensions.
  \item Using hard-coded credentials in application code, migration scripts, notebooks, shell history, CI logs, or test fixtures.
\end{itemize}

\subsection{Dangerous production index operations}
\begin{itemize}
  \item Creating a regular index on a large hot table in production without understanding locks.
  \item Reindexing large production tables casually.
  \item Creating multiple heavyweight indexes at once during peak traffic.
  \item Running index builds with no \texttt{lock\_timeout} or \texttt{statement\_timeout}.
  \item Ignoring failed \texttt{CREATE INDEX CONCURRENTLY} artifacts.
  \item Leaving invalid indexes behind.
  \item Assuming an index is used because it exists.
  \item Not checking \texttt{pg\_stat\_user\_indexes} or query plans for index usage.
  \item Not measuring index impact on writes.
  \item Treating indexes as a substitute for correct schema design.
\end{itemize}

\subsection{Encoding and collation mistakes}
\begin{itemize}
  \item Using \texttt{SQL\_ASCII} for a new database.
  \item Mixing encodings in one database.
  \item Assuming case-insensitive uniqueness without enforcing it.
  \item Using \texttt{lower(email)} in queries without a matching unique expression index or \texttt{citext}/collation plan.
  \item Ignoring collation behavior in sorting, uniqueness, and upgrades.
  \item Letting application code normalize strings differently from the database.
\end{itemize}

\subsection{Expand/contract failures}
\begin{itemize}
  \item Deploying code that requires a new column before the column exists everywhere.
  \item Dropping a column while old app versions still read it.
  \item Reusing a column for a different meaning.
  \item Reusing a table for a different entity.
  \item Reusing enum values for different semantics.
  \item Changing a constraint and assuming all services understand the new rule.
  \item Making incompatible changes to views consumed by downstream systems.
  \item Breaking analytics, exports, jobs, or replicas because only the web app was checked.
  \item Skipping deprecation periods for public database contracts.
\end{itemize}

\subsection{Incorrect SQL semantics}
\begin{itemize}
  \item Using \texttt{NOT IN} when the subquery can return \texttt{NULL}.
  \item Using \texttt{NULL = NULL} logic as if it were true.
  \item Treating \texttt{NULL}, empty string, zero, empty array, and missing JSON key as interchangeable.
  \item Using \texttt{WHERE column = NULL} instead of \texttt{IS NULL}.
  \item Using \texttt{COUNT(column)} when \texttt{COUNT(*)} or a different expression is intended.
  \item Accidentally turning a \texttt{LEFT JOIN} into an inner join by putting right-table filters in the \texttt{WHERE} clause.
  \item Writing joins without explicit join predicates.
  \item Hiding accidental duplicate rows with \texttt{DISTINCT}.
  \item Using \texttt{GROUP BY} to mask a broken join.
  \item Aggregating after joining tables at different granularities and double-counting.
  \item Using \texttt{UNION} when \texttt{UNION ALL} is intended.
  \item Using \texttt{ORDER BY random()} on large tables.
  \item Using window functions without deterministic ordering.
  \item Comparing floating-point values for exact equality.
  \item Using implicit casts that change semantics or defeat indexes.
  \item Assuming JSON missing key, JSON null, SQL null, and empty string are the same thing.
  \item Assuming \texttt{LIKE} case behavior without checking collation and operator semantics.
\end{itemize}

\subsection{Index behavior that is not excusable}
\begin{itemize}
  \item Adding no index for a hot lookup.
  \item Adding an index without knowing which query it supports.
  \item Adding indexes until performance improves without measuring write cost, storage, bloat, and planner usage.
  \item Creating redundant indexes that differ only trivially.
  \item Indexing every column.
  \item Missing composite indexes for multi-column filters.
  \item Composite indexes in the wrong column order.
  \item Assuming \texttt{(a, b)} supports efficient lookup on \texttt{b} alone.
  \item Missing partial indexes for hot selective subsets.
  \item Using a partial index that cannot support the actual predicate.
  \item Forgetting that foreign-key referencing columns are not automatically indexed.
  \item Using a partial index to support cascading deletes on a foreign key. GitLab explicitly says partial indexes cannot serve that purpose.
  \item Adding a unique index after duplicates already exist, with no cleanup plan.
  \item Adding non-unique indexes when the real requirement is uniqueness.
  \item Failing to verify index validity after \texttt{CREATE INDEX CONCURRENTLY}. GitLab’s process explicitly includes checking that asynchronous/concurrent indexes exist and are not invalid.
  \item Leaving invalid indexes around.
  \item Creating huge indexes during peak traffic.
  \item Dropping indexes without checking query dependencies.
  \item Dropping indexes that support constraints or foreign keys.
  \item Ignoring index bloat and vacuum behavior.
  \item Using GIN/GiST/BRIN because they sound powerful without understanding operator classes and workload.
  \item Creating expression indexes where the application query does not exactly match the expression.
  \item Ignoring collation/case-sensitivity requirements.
  \item Adding indexes in the same release as code that depends on them when the index may not exist before the code runs. GitLab recommends splitting index creation and dependent application changes when needed.
\end{itemize}

\subsection{Integrity and keys}
\begin{itemize}
  \item Tables with no primary key or stable unique identifier.
  \item Tables where the “real” uniqueness rule exists only in application code.
  \item Missing \texttt{NOT NULL} on required fields. PostgreSQL’s docs note that most database designs should mark the majority of columns not null.
  \item Nullable columns used to dodge modeling decisions.
  \item Nullable columns in unique constraints when the business rule means “only one,” but the designer forgot that PostgreSQL treats nulls as distinct unless \texttt{NULLS NOT DISTINCT} is used.
  \item No foreign keys for real relationships inside the same database, followed by hand-written cleanup jobs to chase orphans.
  \item Foreign keys with no supporting index on the referencing side when deletes/updates can happen. PostgreSQL notes that referenced columns are indexed, but referencing columns are not automatically indexed; GitLab requires indexes for foreign keys because missing ones have caused timeout incidents.
  \item Polymorphic foreign keys such as \texttt{(target\_type, target\_id)} that can point to many tables with no real referential integrity.
  \item Dual-purpose foreign keys where one column can reference different tables depending on another column.
  \item “Soft foreign keys” documented in comments but not enforced, monitored, or repaired.
  \item Cascading deletes added casually without proving the child rows are true components of the parent.
  \item Missing \texttt{ON DELETE} decision. Every relationship should choose \texttt{RESTRICT}, \texttt{NO ACTION}, \texttt{CASCADE}, \texttt{SET NULL}, or \texttt{SET DEFAULT} intentionally.
  \item \texttt{ON DELETE CASCADE} on relationships where the child is independently meaningful.
  \item \texttt{ON DELETE SET NULL} on columns that should never be null.
  \item Business invariants implemented only in background jobs.
  \item Relying on API validation for constraints when imports, admin scripts, migrations, queues, or future services can bypass the API.
  \item “We trust the app” as a reason to avoid constraints.
  \item Fixing duplicate rows periodically instead of enforcing uniqueness.
  \item Allowing orphan rows because “the UI won’t show them.”
  \item Adding constraints only after launch with no plan to clean existing bad data.
\end{itemize}

\subsection{Migration and DDL behavior that is not excusable}
\begin{itemize}
  \item Production migrations with no peer review.
  \item Production migrations with no tested staging run.
  \item Production migrations with no estimate of affected rows.
  \item Production migrations with no lock analysis.
  \item Production migrations with no query-plan analysis for validation/backfill queries.
  \item Production migrations with no rollback, forward-fix, or restore plan.
  \item Production migrations with no monitoring plan.
  \item Running schema changes manually instead of through versioned migrations.
  \item Editing old migrations after they have shipped.
  \item Relying on ORM auto-migrate in production without review.
  \item Combining risky schema changes and risky data changes in one opaque migration.
  \item Long-running migrations inside one giant transaction.
  \item Adding a column with a volatile default to a huge table casually.
  \item Rewriting a huge table without knowing it will rewrite.
  \item Adding \texttt{NOT NULL} to a populated table without backfill and validation strategy.
  \item Adding foreign keys to large tables without low-lock validation strategy.
  \item Dropping a column while old application versions may still reference it.
  \item Renaming a column/table during rolling deployments without compatibility shims.
  \item Changing a column type without checking casts, indexes, constraints, and rewrite cost.
  \item Changing enum/domain/check values without checking old application versions.
  \item Adding a constraint \texttt{NOT VALID} and never validating it.
  \item Backfilling all rows in one statement when batching is required.
  \item Deleting data in a migration without a data-deletion label, business approval, and recovery plan. GitLab’s database review guidance treats data migrations as inherently risky and requires extra reversal/recovery detail for potentially destructive changes.
  \item Migration code that calls live application models whose behavior may change later.
  \item Migration code depending on external APIs.
  \item Migration code depending on current wall-clock behavior unless that is explicitly intended.
  \item \texttt{DROP ... CASCADE} without enumerating what will be dropped.
  \item Running migrations during peak traffic when the operation is known to contend.
  \item Ignoring failed partial migrations.
  \item Re-running \texttt{CREATE INDEX CONCURRENTLY IF NOT EXISTS} blindly without checking whether a previous invalid index exists.
  \item No \texttt{disable\_ddl\_transaction!} where Postgres requires operations outside a transaction, such as \texttt{CREATE INDEX CONCURRENTLY}. GitLab’s migration style guide calls this out directly.
  \item Treating “the migration succeeded” as proof the app is safe.
\end{itemize}

\subsection{Missing database-enforced integrity}
\begin{itemize}
  \item Creating production tables without primary keys.
  \item Treating primary keys as “optional because the ORM has IDs.”
  \item Referring to another table without a foreign key when both tables live in the same database and the relationship is real.
  \item Relying only on application code for referential integrity.
  \item Relying only on “check before insert” code for uniqueness.
  \item Missing \texttt{UNIQUE} constraints for business identities that must be unique.
  \item Missing \texttt{NOT NULL} constraints for required attributes.
  \item Missing \texttt{CHECK} constraints for domain rules the database can enforce.
  \item Allowing invalid enum/status values because “the frontend validates it.”
  \item Allowing negative quantities, impossible dates, invalid percentages, or out-of-range scores when simple constraints can prevent them.
  \item Allowing orphan rows and planning to “clean them up later.”
  \item Disabling or dropping constraints to make imports easier, then not restoring and validating them.
  \item Marking constraints \texttt{NOT VALID} and never validating them.
  \item Creating nullable foreign keys to avoid modeling the actual lifecycle of the relationship.
  \item Using triggers to fake constraints when native constraints would work.
\end{itemize}

\subsection{Missing or wrong indexes}
\begin{itemize}
  \item Shipping a new high-traffic query without checking its plan.
  \item Adding a foreign key without an index strategy for the referencing column.
  \item Creating composite indexes with the foreign-key column not usable as the leftmost prefix when cascade/delete checks need it.
  \item Using a partial index where a full foreign-key-supporting index is required.
  \item Adding indexes only after production incidents.
  \item Assuming a primary key index covers every query.
  \item Indexing the wrong side of a relationship.
  \item Indexing low-cardinality columns blindly.
  \item Creating duplicate indexes.
  \item Creating indexes that differ only trivially.
  \item Creating unused indexes and never removing them.
  \item Creating every possible index “just in case.”
  \item Ignoring index bloat.
  \item Ignoring write amplification from indexes.
  \item Creating wide covering indexes without checking write cost.
  \item Using the wrong index type for the operator.
  \item Using \texttt{ILIKE '\%term\%'} without trigram/full-text/search infrastructure.
  \item Using JSONB queries without appropriate GIN/expression indexes.
  \item Using expression predicates without expression indexes.
  \item Creating indexes on expressions and never running \texttt{ANALYZE}.
  \item Forgetting that partial indexes only help when the predicate matches.
\end{itemize}

\subsection{Non-sargable and planner-hostile queries}
\begin{itemize}

  \item Wrapping indexed columns in functions in predicates without expression indexes.
  \item \texttt{WHERE lower(email) = ...} without a matching expression index or case-insensitive type strategy.
  \item \texttt{WHERE date(created\_at) = ...} instead of a timestamp range.

  \item Correlated subqueries that run once per row when a join or pre-aggregation is needed.

  \item Query hints, planner toggles, or disabled join methods used as permanent fixes without root-cause analysis.
  \item Relying on ORM-generated SQL without inspecting the actual SQL.
  \item Merging a query that has never been run with \texttt{EXPLAIN}.
  \item Merging a query that has only been tested on tiny seed data.
  \item Running \texttt{EXPLAIN ANALYZE} on destructive statements outside a transaction/rollback guard.
\end{itemize}

\subsection{Postgres-specific type/design mistakes}
\begin{itemize}
  \item Creating new Postgres databases with \texttt{SQL\_ASCII}. The PostgreSQL wiki warns that \texttt{SQL\_ASCII} means “no conversions” and can leave mixed encodings that cannot be recovered reliably.
  \item Using quoted mixed-case table/column names in normal application schemas.
  \item Using reserved words as identifiers.
  \item Using names that require quoting everywhere.
  \item Inconsistent singular/plural naming that hides relationships.
  \item Ambiguous columns like \texttt{status}, \texttt{type}, \texttt{data}, \texttt{value}, \texttt{flag}, \texttt{source}, \texttt{kind} without clear domain constraints.
  \item Using \texttt{char(n)} by habit.
  \item Using \texttt{varchar(255)} by habit when no actual 255-character business rule exists.
  \item Using \texttt{money} for monetary systems without understanding locale, rounding, and currency issues. The PostgreSQL wiki recommends \texttt{numeric} or sometimes integers instead.
  \item Storing money as floating point.
  \item Storing currency amount without currency code when multi-currency is possible.
  \item Storing “UTC” in \texttt{timestamp without time zone} for real instants. The PostgreSQL wiki recommends \texttt{timestamptz} for instants in time.
  \item Storing offsets like \texttt{-07:00} as if they were time zones.
  \item Using \texttt{timetz} or \texttt{CURRENT\_TIME} in ordinary application schemas; the PostgreSQL wiki says these are generally not useful.
  \item Using timestamp precision like \texttt{timestamptz(0)} without realizing it rounds, not truncates.
  \item Using \texttt{serial} for new schemas when identity columns are available, unless you are intentionally supporting old PostgreSQL versions or a special sequence-sharing case.
  \item Assuming sequence values are gapless, meaningful, rollback-safe, or chronological. PostgreSQL documents that sequence changes are visible immediately and are not rolled back on transaction abort.
  \item Using table inheritance for ordinary partitioning or object modeling. The PostgreSQL wiki says to avoid table inheritance now that native partitioning exists.
  \item Using PostgreSQL rules for business logic instead of triggers or explicit SQL. The PostgreSQL wiki says rules rewrite queries and “don’t do what they look like they do.”
  \item Using \texttt{bytea}, JSON, or text to hide data that needs constraints, indexes, joins, or permissions.
\end{itemize}

\subsection{Query-writing behavior that is not excusable}
\begin{itemize}
  \item Writing queries whose correctness depends on implicit row order. Use \texttt{ORDER BY}.
  \item Using \texttt{LIMIT} without deterministic \texttt{ORDER BY}.
  \item Paginating without a stable tie-breaker.
  \item Deep \texttt{OFFSET} pagination on large hot paths without considering keyset pagination.
  \item \texttt{SELECT *} in application contracts, APIs, joins, migrations, or long-lived queries where schema changes can break behavior or pull unnecessary data.
  \item \texttt{SELECT *} from wide tables with sensitive columns.
  \item Joining large tables without understanding cardinality.
  \item Using \texttt{DISTINCT} to hide duplicate rows caused by a wrong join.
  \item Using \texttt{GROUP BY} to mask a modeling bug.
  \item Cross joins by accident.
  \item Unbounded queries in request paths.
  \item Unbounded \texttt{DELETE} or \texttt{UPDATE} without guardrails.
  \item \texttt{DELETE FROM table} when \texttt{TRUNCATE}, partition drop, archival, or batched delete is the real operation.
  \item \texttt{TRUNCATE} without understanding locks, triggers, identity reset behavior, and referential impact.
  \item \texttt{NOT IN (subquery)} where nulls are possible. The PostgreSQL wiki warns that \texttt{NOT IN} behaves unexpectedly with nulls and can produce very bad plans.
  \item \texttt{BETWEEN} on timestamps for date ranges. The PostgreSQL wiki recommends half-open ranges like \texttt{>= start AND < end}.
  \item Using local time arithmetic for global events.
  \item Comparing timestamps across time zones without explicit semantics.
  \item Casting indexed columns in predicates instead of casting parameters.
  \item Wrapping indexed columns in functions in hot predicates without matching expression indexes.
  \item \texttt{LOWER(email) = LOWER(\$1)} on a huge table without a functional index or \texttt{citext}-style decision.
  \item Leading-wildcard \texttt{LIKE '\%term\%'} on large tables without a trigram/search design.
  \item \texttt{ORDER BY random()} on large tables.
  \item \texttt{COUNT(*)} on huge filtered result sets in every page render without a plan.
  \item Re-querying the same row repeatedly in a loop.
  \item N+1 queries hidden behind an ORM.
  \item Per-row queries in migrations instead of set-based or batched operations.
  \item Fetching entire result sets into application memory.
  \item Relying on ORM-generated SQL without looking at the emitted SQL.
  \item Not using \texttt{EXPLAIN}/\texttt{EXPLAIN ANALYZE} for important queries.
  \item Comparing query performance only on tiny dev data.
  \item Assuming a query is safe because it is fast locally.
  \item Ignoring row estimates, sequential scans, sort spill, hash spill, nested loop explosions, or temp files on hot queries.
  \item Ignoring \texttt{NULL} semantics in joins and filters.
  \item Using \texttt{=} when \texttt{IS NOT DISTINCT FROM} is required for null-aware equality.
  \item Using \texttt{!=}/\texttt{<>} and expecting nulls to match.
  \item Building dynamic SQL in stored procedures without bind variables. OWASP notes that stored procedures can still introduce SQL injection if dynamic SQL is constructed unsafely.
  \item Treating stored procedures as automatically safer than parameterized SQL.
  \item Returning sensitive columns because “the frontend won’t display them.”
  \item Using read replicas for read-after-write flows without handling replica lag.
\end{itemize}

  \item Having no emergency credential revocation path.
  \item Letting developers connect to production as the application role.

  \item **Concatenating identifiers into SQL strings without safe identifier quoting.** In PostgreSQL dynamic SQL, identifiers and literals must be handled differently; use parameters for values and safe quoting/formatting such as \texttt{quote\_ident}, \texttt{quote\_literal}, \texttt{quote\_nullable}, or \texttt{format()} where dynamic identifiers are unavoidable.

  \item **Making production databases reachable from the public internet without a very narrow, audited, encrypted access path.**
  \item **Disabling TLS for client-server database traffic that crosses untrusted networks.** PostgreSQL has native SSL/TLS support for encrypting client-server communication.

  \item **Letting AI agents, code generators, BI tools, notebooks, or admin dashboards run unrestricted production DML/DDL.**
  \item **Copy-pasting SQL from tickets, Slack, Reddit, Stack Overflow, or an LLM into production without review and a transaction/rollback plan.**
  \item **Using production data in dev/test without masking, minimization, and access control.**

  \item **Failing open on permission errors.** Security checks should deny by default.
  \item **Using database comments, enum labels, logs, or error strings to store secrets.**

  \item Concatenating user input into SQL strings.
  \item Using string interpolation, f-strings, template literals, \texttt{format()}, or ORM raw fragments with untrusted values.
  \item Treating escaping as the primary SQL injection defense instead of parameterized queries.

  \item Running normal app traffic as the schema owner.

\subsection{The highest-severity red lines}
\begin{itemize}
  \item **Concatenating user input into SQL.** Use server-side parameters or prepared statements. Escaping is not an acceptable primary defense. OWASP explicitly says to stop writing dynamic queries with string concatenation, and GitHub CodeQL flags user-controlled query construction as a high-severity SQL injection pattern.
  \item **Concatenating user-controlled identifiers, sort columns, table names, \texttt{ORDER BY}, \texttt{LIMIT}, or \texttt{WHERE} fragments without an allowlist.** Parameters protect values, not arbitrary SQL structure.
  \item **Running AI-generated SQL directly against production.** Generated SQL must be reviewed like hand-written destructive code: threat model, transaction behavior, locks, rollback, plan, data volume, and blast radius.
  \item **Using a production application role with superuser, owner, \texttt{CREATEDB}, \texttt{CREATEROLE}, broad DDL, or unrestricted write privileges.** PostgreSQL’s role system exists to control object ownership and privileges; using one omnipotent role for everything defeats that control.
  \item **Using \texttt{trust} authentication over TCP/IP in production.** The PostgreSQL wiki explicitly says not to use \texttt{trust} over TCP/IP in production and calls \texttt{host all all 0.0.0.0/0 trust} especially dangerous.
  \item **Hardcoding database passwords, API keys, encryption keys, or connection strings in source, migrations, notebooks, shell history, Dockerfiles, SQL files, or CI logs.** OWASP identifies database credentials as secrets and recommends centralized, auditable, rotatable secrets management with least privilege.
  \item **Storing plaintext passwords, reversible-encrypted passwords, or fast hashes such as MD5/SHA-1 for passwords.** Passwords require purpose-built adaptive hashing such as Argon2id, bcrypt, scrypt, or PBKDF2 with proper salts/work factors; OWASP also says passwords should not be stored using reversible encryption.
  \item **Storing sensitive data unnecessarily.** Data that is not retained cannot be stolen; OWASP explicitly recommends minimizing sensitive storage and encrypting sensitive data at rest and in transit.
  \item **Turning off \texttt{fsync} or \texttt{full\_page\_writes} for persistent production data.** PostgreSQL warns that disabling \texttt{fsync} can cause unrecoverable corruption after a crash, and disabling \texttt{full\_page\_writes} can lead to unrecoverable or silent corruption.
  \item **Operating without tested backups and point-in-time recovery for important data.** PostgreSQL’s WAL-based backup/PITR design exists so you can restore a consistent database state to a chosen time; having “backups” you have never restored is not a recovery plan.
  \item **Running destructive DDL/DML without a rollback or recovery plan.** \texttt{DROP}, \texttt{TRUNCATE}, bulk \texttt{DELETE}, bulk \texttt{UPDATE}, \texttt{ALTER TABLE}, and backfills must have a tested undo/recovery story.
  \item **Deploying migrations to production without understanding locks.** PostgreSQL DDL takes locks; migration-safety tools and GitLab’s own guidelines exist because unsafe migrations can block traffic or corrupt application behavior.
  \item **Creating indexes non-concurrently on large hot production tables.** Standard \texttt{CREATE INDEX} blocks writes; PostgreSQL documents \texttt{CREATE INDEX CONCURRENTLY} for avoiding write-blocking, and GitLab requires special care, testing, and asynchronous creation for very large tables.
  \item **Adding foreign keys, \texttt{NOT NULL}, large defaults, or validation constraints to big tables in one naïve migration.** These can scan/rewrite/lock large tables; safe migration helpers split, validate, retry, and time-limit operations.
  \item **Doing bulk data migrations in a normal request/transaction path.** GitLab recommends batched background migrations when a data migration exceeds time limits or touches high-traffic/large tables.
  \item **Assuming the application alone will preserve integrity when the database can enforce it.** PostgreSQL constraints exist to prevent invalid data regardless of which client writes it; constraints reject invalid values even if the value came from a default.
  \item **Using a multi-tenant database without database-enforced tenant isolation where the blast radius is high.** PostgreSQL Row-Level Security can restrict rows per user/role; by default, if RLS is enabled but no policy exists, access is default-deny.
  \item **Disabling constraints, triggers, RLS, permissions, or auditing “temporarily” without a tracked, time-boxed, reviewed recovery step.** Temporary safety removal becomes permanent corruption.
  \item **Manual production fixes that are not captured in versioned migrations or audited runbooks.** “I fixed it in psql” is how systems become unreproducible.
  \item **Treating a database as disposable cache when it is actually the source of truth.** If customers, money, permissions, orders, ledgers, inventory, or audit events depend on it, it needs durability, integrity, observability, and recovery.
\end{itemize}

\subsection{Time and timezone mistakes}
\begin{itemize}
  \item Using \texttt{timestamp without time zone} for real-world instants.
  \item “Storing UTC” in a naive timestamp column and expecting everyone to remember that forever.
  \item Storing timezone offsets as text instead of modeling time correctly.
  \item Using \texttt{timetz}.
  \item Using \texttt{current\_time}.
  \item Using \texttt{timestamp(0)} and silently rounding time.
  \item Using \texttt{BETWEEN} for timestamp ranges where the correct logic is a half-open interval.
  \item Comparing timestamps in application-local time without an explicit timezone policy.
  \item Mixing server timezone, database timezone, user timezone, and business timezone accidentally.
  \item Using local wall-clock time for ordering globally distributed events.
  \item Building daily/monthly reports without defining which timezone defines the day or month.
  \item Ignoring daylight-saving transitions in scheduling and recurrence logic.
\end{itemize}

\subsection{Transaction and concurrency behavior that is not excusable}
\begin{itemize}
  \item Assuming \texttt{READ COMMITTED} means “the whole transaction sees one consistent world.” PostgreSQL documents the different isolation levels and which anomalies are possible.
  \item Assuming \texttt{READ UNCOMMITTED} gives dirty reads in PostgreSQL; PostgreSQL maps it to \texttt{READ COMMITTED}.
  \item Read-modify-write cycles outside a transaction.
  \item \texttt{SELECT} then \texttt{INSERT} race conditions instead of \texttt{INSERT ... ON CONFLICT}, constraints, or locks.
  \item Checking “does row exist?” in application code without a unique constraint.
  \item Implementing counters by reading a value, adding one in app code, and writing it back without atomic SQL or locking.
  \item Not handling deadlocks.
  \item Not retrying serialization failures when using serializable isolation.
  \item Holding transactions open while making network calls.
  \item Holding transactions open while waiting for user input.
  \item “Idle in transaction” sessions in production.
  \item Running long reports in the primary OLTP transaction context without limits.
  \item Mixing OLTP writes and long analytical reads without isolation/resource planning.
  \item No \texttt{statement\_timeout}.
  \item No \texttt{lock\_timeout} for migrations or risky maintenance.
  \item No \texttt{idle\_in\_transaction\_session\_timeout}.
  \item Acquiring locks in inconsistent order across code paths.
  \item Using table locks as application mutexes.
  \item Using advisory locks without a documented key namespace and failure behavior.
  \item Assuming advisory locks protect data from sessions that do not use the same advisory lock.
  \item Swallowing database errors and continuing the transaction.
  \item Retrying non-idempotent writes blindly.
  \item No idempotency key for externally retried financial/order/job operations.
  \item Generating IDs in the app and assuming collisions cannot happen.
  \item Using timestamp order as a correctness mechanism.
  \item Assuming two SQL statements are atomic because they are adjacent in code.
\end{itemize}

\subsection{Unsafe and unstable query shape}
\begin{itemize}
  \item \texttt{SELECT *} in production APIs, views, unions, ETL contracts, application queries, or migrations.
  \item Relying on implicit column order.
  \item \texttt{INSERT INTO table VALUES (...)} without a column list.
  \item Unqualified column names in joins.
  \item Creating views or functions whose behavior changes when a column is added.
  \item Using \texttt{SELECT *} on both sides of a \texttt{UNION}.
  \item Adding columns to a table without checking code that uses \texttt{SELECT *}.
  \item Returning more columns than the caller needs by default.
  \item Pulling entire rows just to check existence.
  \item Fetching large result sets into application memory instead of streaming or paginating.
  \item Using \texttt{LIMIT} without deterministic \texttt{ORDER BY}.
  \item Paginating by non-unique sort keys without a tie-breaker.
  \item Assuming \texttt{created\_at} is unique enough for stable pagination.
  \item Relying on default row order.
  \item Using offset pagination for large, frequently changing datasets where keyset pagination is needed.
\end{itemize}

\subsection{Unsafe production migrations}
\begin{itemize}
  \item Running hand-written production DDL outside version control.
  \item Running migrations without review.
  \item Running migrations without knowing lock behavior.
  \item Running migrations without \texttt{lock\_timeout} and \texttt{statement\_timeout}.
  \item Running migrations that require downtime when an online path exists.
  \item Mixing risky schema changes and large data backfills in one transaction.
  \item Backfilling millions of rows synchronously in a deploy migration.
  \item Running large data migrations inside web requests.
  \item Adding a required column, constraint, or index without checking existing data.
  \item Adding \texttt{NOT NULL} without a safe validation/backfill path.
  \item Adding unique constraints without deduplicating existing data first.
  \item Adding foreign keys without validating existing references.
  \item Dropping columns before all old code paths stop reading them.
  \item Renaming columns or tables in one deploy without compatibility handling.
  \item Changing column types on hot large tables without testing rewrite/lock behavior.
  \item Creating indexes non-concurrently on large live tables.
  \item Running multiple heavyweight migrations in one deploy.
  \item Having no rollback plan for destructive migrations.
  \item Writing migrations that depend on application models that may change later.
  \item Using irreversible migrations casually.
  \item Failing to test migrations on production-like data volume.
  \item Failing to test rollback or forward-fix paths.
  \item Failing to coordinate application deploy order with schema compatibility.
  \item Treating ORM-generated migrations as automatically safe.
  \item Accepting an LLM-generated migration that has not been lock-tested, reviewed, and rehearsed.
\end{itemize}

\subsection{Wrong data types}
\begin{itemize}
  \item Storing dates as text.
  \item Storing timestamps as text.
  \item Storing numbers as text.
  \item Storing booleans as \texttt{'true'}, \texttt{'false'}, \texttt{'Y'}, \texttt{'N'}, \texttt{0}, or \texttt{1} when a boolean type is appropriate.
  \item Storing UUIDs as text when \texttt{uuid} is available.
  \item Storing IP addresses as text when \texttt{inet}/\texttt{cidr} would be appropriate.
  \item Storing money in floating-point columns.
  \item Using approximate floats for exact quantities such as money, inventory, votes, counts, or billing.
  \item Using \texttt{real}/\texttt{double precision} for equality joins or exact grouping.
  \item Using \texttt{numeric} with no thought to precision, scale, and performance.
  \item Using \texttt{varchar(255)} everywhere by habit.
  \item Using \texttt{char(n)} for ordinary text.
  \item Using \texttt{text} for small enumerated domains without a constraint or lookup table.
  \item Using Postgres \texttt{money} casually instead of a deliberate currency model.
  \item Mixing currencies in one numeric column without a currency column and constraint.
  \item Mixing units in one column without a unit column and constraint.
  \item Using magic sentinel values such as \texttt{-1}, \texttt{0}, \texttt{'unknown'}, \texttt{'N/A'}, \texttt{'1970-01-01'}, or \texttt{'9999-12-31'} instead of modeling the state correctly.
  \item Using JSON strings to represent typed values that PostgreSQL can natively validate.
\end{itemize}
