\section{Inexcusable CI Anti-Patterns}
\label{sec:bad_ci}

The following section categorizes the most severe, inexcusable CI anti-patterns observed in production environments and AI-generated code. These patterns compromise safety, maintainability, and performance.\

\subsection{Known Best Practices}
To counteract the anti-patterns, ensure adherence to these core CI best practices:
\begin{itemize}
  \item Treat CI as source of truth: Local 'works on my machine' is irrelevant; the CI pipeline is the definitive test.
  \item Keep pipelines fast: Parallelize independent jobs, utilize caching aggressively, and fail fast.
  \item Run comprehensive checks: Include linting, formatting, type-checking, unit tests, and security scanning.
  \item Protect the main branch: Require passing CI checks and approvals before any merge.
  \item Use ephemeral and isolated environments: Build artifacts once and promote them through environments.
  \item Pin pipeline dependencies: Explicitly version CI/CD actions and runners to prevent sudden breakages.
\end{itemize}

\subsection{A. Bad CI runner/agent behavior}
\begin{itemize}
  \item Executes code from an untrusted branch with privileged credentials.
  \item Executes untrusted code on a persistent host.
  \item Reuses workspace state across trust boundaries.
  \item Leaves secrets in files after job completion.
  \item Leaves containers, images, volumes, or processes running after jobs.
  \item Allows one job to read another job’s workspace.
  \item Allows one repository to read another repository’s checkout.
  \item Runs as root unnecessarily.
  \item Mounts host paths unnecessarily.
  \item Has access to Docker socket.
  \item Has access to Kubernetes cluster credentials.
  \item Has access to production network segments.
  \item Has access to internal package registries with write permissions.
  \item Has access to cloud metadata credentials.
  \item Uses broad service-account credentials.
  \item Does not enforce timeout/resource limits.
  \item Does not enforce network egress restrictions.
  \item Does not attest runner image or configuration.
  \item Does not log job-to-runner assignment.
  \item Does not clean caches safely.
  \item Does not separate protected and unprotected jobs.
  \item Does not separate public and private repositories.
  \item Does not separate build, test, publish, and deploy trust zones.
  \item Accepts arbitrary labels from jobs.
  \item Accepts arbitrary scripts from comments or issue events.
  \item Auto-registers runners without approval.
  \item Runs obsolete runner versions.
  \item Runs unpatched host OS images.
  \item Stores SSH keys on disk.
  \item Stores cloud keys on disk.
  \item Shares credentials between runners.
  \item Can be reached inbound from the internet.
  \item Can initiate arbitrary outbound connections.
  \item Can access secrets manager without job identity constraints.
  \item Can deploy without human approval.
  \item Fails open when a security scanner fails.
  \item Retries suspicious jobs automatically without quarantine.
  \item Provides no forensic evidence after compromise.
  \item Cannot be rapidly reimaged.
  \item Cannot revoke its own credentials.
\end{itemize}

\subsection{A. Bad identity, permissions, and privilege behavior}
\begin{itemize}
  \item **Running all CI jobs with write permissions by default.**
  \item **Sharing one powerful token across unrelated steps.**
  \item **Allowing CI agents to approve, merge, or create privileged PRs.**
  \item **Using long-lived personal access tokens instead of short-lived scoped credentials.**
  \item **Using personal SSH keys in CI.**
  \item **Giving untrusted contributors access to protected variables or protected runners.**
  \item **Disabling token allowlists or broadening job-token access.**
  \item **Letting CI tokens push code without strong controls.**
  \item **Using global credentials instead of scoped credentials.**
  \item **Allowing anyone with write access to change CI behavior without review.**
\end{itemize}

\subsection{A. Bad pipeline structure}
\begin{itemize}
  \item **Long serial pipelines with no DAG.**
  \item **No parallelization for independent tests.**
  \item **Over-parallelization that starves runners.**
  \item **No path filters.**
  \item **No change-based test selection.**
  \item **No smoke tests before expensive tests.**
  \item **Running slow integration/e2e tests before fast unit/lint checks.**
  \item **No fail-fast behavior.**
  \item **Failing late because quality gates are placed after expensive jobs.**
  \item **Rebuilding the same artifact in every job.**
  \item **Downloading dependencies separately in every job.**
  \item **No dependency cache, or a cache key that never hits.**
  \item **Cache keys that are too broad and restore stale/incompatible dependencies.**
  \item **Cache keys that are too narrow and create a new cache every run.**
  \item **Caching huge directories with little reuse value.**
  \item **Uploading/download large artifacts unnecessarily.**
  \item **Using artifacts where caches should be used, or caches where artifacts should be used.**
  \item **No timeout on jobs or steps.**
  \item **Timeouts set so low that normal work flakes.**
  \item **No cancellation of superseded runs.**
  \item **No interruptible/cancel behavior for obsolete branches.**
  \item **Running full CI on every push to a branch while a PR is already open.**
  \item **Duplicated pipelines for push and PR/MR events.**
  \item **Scheduled jobs running during peak developer hours.**
  \item **Cron storms.**
\end{itemize}

\subsection{A. Trigger and pipeline-explosion problems}
\begin{itemize}
  \item Running full CI on every push, every PR, every tag, every schedule, and every file change.
  \item Running the same pipeline twice for both branch push and merge request/pull request.
  \item Running expensive integration or end-to-end suites for documentation-only changes.
  \item Running deployment preparation on every branch.
  \item Creating broad schedules that overlap with normal push pipelines.
  \item Triggering downstream pipelines recursively.
  \item Creating child pipelines from every job without guardrails.
  \item Running all language/runtime/platform combinations on every change.
  \item Running large matrix jobs without risk-based pruning.
  \item No concurrency groups or auto-cancel for superseded commits.
  \item No \texttt{interruptible}/cancel-safe marking for non-deployment jobs.
  \item Retrying entire pipelines instead of failed jobs.
  \item Retrying flaky jobs indefinitely.
  \item Allowing bot commits to trigger bot commits to trigger more pipelines.
  \item Letting an AI agent repeatedly push tiny commits that each trigger full CI.
  \item No maximum runtime per job.
  \item No maximum number of parallel jobs per branch/project.
  \item No queue/backpressure controls.
  \item No separate lightweight pre-merge and heavier post-merge pipelines.
  \item No skip logic for generated files, docs, comments, or non-runtime assets.
\end{itemize}

\subsection{A. Trust-boundary violations}
\begin{itemize}
  \item Running **untrusted pull request code** with production secrets available.
  \item Running untrusted PR code with a write-capable repository token.
  \item Running forked PRs on self-hosted runners that contain long-lived credentials.
  \item Treating branch names, PR titles, commit messages, issue comments, release names, tags, and artifact names as trusted shell input.
  \item Using GitHub \texttt{pull\_request\_target} and then checking out or executing code from the untrusted PR head.
  \item Triggering privileged jobs from comments such as \texttt{/test}, \texttt{/deploy}, \texttt{/approve}, or \texttt{/retest} without robust authorization and immutable commit binding.
  \item Approving a PR once, then allowing the author to push new commits that run under the earlier approval.
  \item Running old PRs against old, vulnerable workflow definitions after a workflow security fix.
  \item Running build/test scripts supplied by the PR before reviewing changes to package scripts, Makefiles, Gradle/Maven plugins, shell scripts, test harnesses, or CI YAML.
  \item Allowing generated pipeline definitions from an untrusted branch to create child pipelines or dynamic jobs.
  \item Treating test output, coverage output, linter output, or AI-agent feedback as trusted instructions.
  \item Letting an AI/automation agent read malicious PR content and then follow embedded instructions such as “print environment variables,” “disable security checks,” or “upload logs.”
  \item Running \texttt{npm install}, \texttt{pip install}, \texttt{bundle install}, \texttt{go generate}, \texttt{mvn}, Gradle plugins, or similar commands with secrets present when package lifecycle scripts may execute arbitrary code.
  \item Using privileged deploy workflows to process artifacts produced by an unprivileged PR workflow without validation.
  \item Letting untrusted code write cache entries consumed later by privileged workflows.
  \item Allowing the CI agent to fetch and run scripts from arbitrary URLs.
  \item Allowing branch or tag names to select deployment targets.
  \item Allowing a build matrix or job definition to be generated from untrusted user input.
  \item Treating “tests passed” as proof that the CI YAML itself is safe.
  \item Allowing pipeline configuration changes to merge without special review.
\end{itemize}

\subsection{B. Bad job graph design}
\begin{itemize}
  \item Everything is placed in one giant job.
  \item Everything is placed in one giant stage.
  \item Fast checks wait behind slow checks.
  \item Lint/unit tests wait behind image builds.
  \item Integration tests wait for unrelated jobs.
  \item Deployment jobs rebuild artifacts instead of depending on build outputs.
  \item Jobs use implicit dependencies rather than explicit \texttt{needs}.
  \item A job downloads all prior artifacts by default.
  \item Fan-in jobs gather huge artifacts from every parallel job.
  \item Fan-out jobs each repeat identical setup.
  \item Database migrations run before basic compile or unit tests.
  \item Security scans run after deployment packaging rather than early enough to fail fast.
  \item Every job installs the same dependencies from scratch.
  \item Every job rebuilds the same Docker image.
  \item Every job pulls large base images.
  \item No separation between fast smoke tests and slow full suites.
  \item No test sharding.
  \item No fail-fast behavior for matrix jobs.
  \item No dependency graph visibility.
  \item Hidden dependencies encoded in global \texttt{before\_script}.
\end{itemize}

  \item **Persistent runners accumulating stale state.**
  \item **Cold-starting large images repeatedly.**
