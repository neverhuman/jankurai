\section{Inexcusable PYTHON Anti-Patterns}
\label{sec:bad_python}

The following section categorizes the most severe, inexcusable PYTHON anti-patterns observed in production environments and AI-generated code. These patterns compromise safety, maintainability, and performance.\

\subsection{Known Best Practices}
To counteract the anti-patterns, ensure adherence to these core PYTHON best practices:
\begin{itemize}
  \item Use type hints everywhere: Annotate function signatures and use \texttt{mypy} or \texttt{pyright} in strict mode.
  \item Manage dependencies deterministically: Use modern tools (Poetry, uv, pip-tools) with lockfiles.
  \item Handle exceptions specifically: Catch narrow exception types and avoid bare \texttt{except:} blocks.
  \item Follow PEP 8 and use formatters: Enforce \texttt{black}, \texttt{ruff}, and \texttt{isort} via pre-commit hooks.
  \item Favor composition and pure functions: Keep classes small and avoid deep inheritance trees.
\end{itemize}

\subsection{1. Arbitrary code execution}
\begin{itemize}
  \item \texttt{eval()} on anything not hardcoded and fully controlled.
  \item \texttt{exec()} on anything not hardcoded and fully controlled.
  \item \texttt{compile(..., "exec")} or \texttt{compile(..., "eval")} on dynamic strings.
  \item Building Python source with f-strings or string concatenation, then executing it.
  \item Accepting “safe because it came from the LLM” generated code and running it.
  \item Letting config files be Python code when the config author is outside the trust boundary.
  \item Importing user-controlled modules, plugin names, class names, or dotted paths without a strict allowlist.
  \item Using \texttt{globals()[name]}, \texttt{locals()[name]}, or \texttt{getattr(module, name)} as a user-controlled router.
  \item Using \texttt{\_\_import\_\_}, \texttt{importlib.import\_module}, entry-point discovery, or plugin loading on untrusted names.
  \item Using \texttt{base64.b64decode(...); exec(...)}, \texttt{marshal.loads(...); exec(...)}, or any obfuscated execute pattern.
  \item Treating \texttt{ast.parse()} as a sandbox. Parsing code is not the same as safely running code.
  \item Letting notebooks, markdown, README snippets, GitHub issue text, PR comments, or agent messages become executable Python.
  \item Letting an agent “fix an error” by adding an \texttt{eval}, \texttt{exec}, dynamic import, or generated-code runner.
  \item Using Python as a templating language for untrusted templates.
  \item Shipping self-modifying code, hidden import hooks, or runtime monkeypatching that changes security behavior.
  \item Creating “admin consoles,” “debug shells,” or “quick REPL endpoints” inside production apps.
\end{itemize}

\subsection{1. Code execution and command execution sins}
\begin{itemize}
  \item Never execute user input with \texttt{eval()}.
  \item Never execute user input with \texttt{exec()}.
  \item Never compile and run untrusted strings with \texttt{compile()}.
  \item Never pass untrusted data into \texttt{getattr()}, \texttt{globals()}, \texttt{locals()}, or \texttt{\_\_import\_\_()} to select executable behavior.
  \item Never dynamically import modules based on unsanitized user input.
  \item Never load plugins from writable, user-controlled, temp, downloaded, or current-working-directory paths without verification.
  \item Never run generated Python code in production without sandboxing and review.
  \item Never run notebook cells from untrusted notebooks.
  \item Never treat a string from an LLM, database, config file, queue, webhook, or API request as executable Python.
  \item Never use \texttt{os.system()} with variable input.
  \item Never use \texttt{subprocess(..., shell=True)} with any tainted input.
  \item Never build shell commands using f-strings, \texttt{\%}, \texttt{.format()}, concatenation, or template substitution with untrusted data.
  \item Never pass a command string when an argument list is available.
  \item Never rely on quoting as your primary defense when avoiding the shell is possible.
  \item Never invoke shell metacharacters, pipes, redirects, globs, or command substitution using data that came from a user, file, database, LLM, webhook, or environment variable.
  \item Never call external commands without checking exit status.
  \item Never ignore \texttt{subprocess} return codes.
  \item Never run commands without timeouts in services, CI, workers, or agents.
  \item Never let a web request directly trigger arbitrary command execution.
  \item Never let an agent choose arbitrary shell commands against a real machine without an allowlist and sandbox.
  \item Never invoke package managers, cloud CLIs, Docker, Kubernetes, Terraform, SSH, or database CLIs from Python using untrusted arguments.
  \item Never run shell commands as root unless the command is static, audited, and unavoidable.
  \item Never let untrusted input control \texttt{PATH}, \texttt{PYTHONPATH}, working directory, interpreter path, or command name.
  \item Never hide command execution inside “helper” functions that look harmless.
  \item Never accept “it’s internal” as a justification for command injection.
\end{itemize}

\subsection{10. Dependencies, packaging, and supply chain}
\begin{itemize}
  \item Unpinned dependencies in production builds.
  \item \texttt{pip install package} with no lockfile, no hashes, and no review.
  \item Mutable dependency ranges for deploys.
  \item Docker base images pinned to \texttt{latest}.
  \item Installing hallucinated packages from LLM output.
  \item Installing packages because an agent suggested them without checking provenance.
  \item Typosquatted or slopsquatted package names.
  \item Installing from random GitHub repos, gists, raw URLs, or HTTP indexes.
  \item Runtime \texttt{pip install} based on user input.
  \item Running untrusted package setup/build hooks on privileged machines.
  \item Building source distributions from unknown packages in CI with secrets present.
  \item Ignoring dependency advisories.
  \item Ignoring \texttt{pip-audit}, Dependabot, GitLab dependency scanning, or equivalent alerts.
  \item No review of transitive dependency changes.
  \item Vendored code with unknown origin.
  \item Vendored code with incompatible licenses.
  \item Generated code with no provenance.
  \item Binary blobs in the repo with no source or checksum.
  \item Private package names not reserved publicly where dependency confusion is possible.
  \item Mixing public and private indexes unsafely.
  \item Publishing packages with secrets in metadata, wheels, sdists, notebooks, tests, or docs.
  \item Trusting package import names that shadow standard library modules.
  \item A local \texttt{random.py}, \texttt{email.py}, \texttt{jwt.py}, \texttt{typing.py}, \texttt{logging.py}, or \texttt{asyncio.py} that breaks imports.
  \item Skipping reproducible builds for critical services.
  \item No SBOM or artifact provenance for high-risk systems.
  \item Disabling dependency scanning because “it’s noisy.”
\end{itemize}

\subsection{10. File, path, temp-file, and filesystem sins}
\begin{itemize}
  \item Never join paths with untrusted filenames and assume the result is safe.
  \item Never allow \texttt{../} traversal.
  \item Never trust normalized-looking paths without resolving and checking the base directory.
  \item Never write files using user-supplied absolute paths.
  \item Never delete files using user-supplied paths without strict allowlists and invariant checks.
  \item Never recursively delete a path that was not constructed and verified by your code.
  \item Never follow symlinks unexpectedly in upload, extraction, cleanup, or temp directories.
  \item Never create world-writable files containing sensitive data.
  \item Never create world-writable directories for application state.
  \item Never use \texttt{tempfile.mktemp()}.
  \item Never hand-roll temporary filenames.
  \item Never write temp files into predictable locations.
  \item Never assume \texttt{/tmp} is private.
  \item Never leave sensitive temp files behind.
  \item Never read entire untrusted files into memory without size limits.
  \item Never process user uploads without quota, type validation, malware policy where relevant, and cleanup.
  \item Never trust file extensions.
  \item Never trust image libraries to be safe from decompression bombs without limits.
  \item Never let an agent write outside the repository or workspace unless explicitly allowed.
  \item Never run code that assumes the current working directory is safe.
  \item Never rely on relative paths in privileged scripts.
  \item Never put writable directories ahead of trusted directories in import or executable search paths.
  \item Never name project files \texttt{json.py}, \texttt{typing.py}, \texttt{email.py}, \texttt{requests.py}, \texttt{logging.py}, or other stdlib/package-shadowing names.
  \item Never run Python scripts from untrusted directories without considering import-path hijacking.
  \item Never chmod broadly, such as \texttt{chmod -R 777}, to “fix permissions.”
\end{itemize}

\subsection{11. Agent-hostile Python behavior}
\begin{itemize}
  \item Import-time side effects: network calls, DB writes, filesystem mutation, cloud calls, subprocesses, logging reconfiguration, environment mutation, thread/process startup.
  \item Code that behaves differently depending on the current directory.
  \item Code that behaves differently depending on hidden environment variables.
  \item Code that silently reads credentials from the developer’s machine.
  \item Code that silently writes outside the repo/workspace.
  \item Code that silently modifies global process state.
  \item Monkeypatching builtins, stdlib modules, TLS verification, auth checks, serializers, or logging globally.
  \item “Convenience” global clients initialized at import time with production credentials.
  \item Hidden singletons with mutable state.
  \item Mutable default arguments.
  \item Bare module-level code that performs real work.
  \item Scripts that lack \texttt{if \_\_name\_\_ == "\_\_main\_\_":}.
  \item Functions that mix parsing, auth, IO, business logic, retries, logging, and mutation in one block.
  \item Huge untestable files named \texttt{utils.py}, \texttt{helpers.py}, or \texttt{misc.py} containing security-critical behavior.
  \item Dynamic imports for normal control flow.
  \item Reflection-based routing instead of explicit allowlists.
  \item Magic decorators that change authorization, serialization, or transaction boundaries invisibly.
  \item Runtime patching to “make tests pass.”
  \item Generated code that no human can explain.
  \item Code comments that contradict behavior.
  \item Dead code paths that still contain dangerous behavior.
  \item Commented-out secrets, commands, credentials, or vulnerable snippets.
  \item Relying on agent memory instead of explicit repo documentation.
  \item Requiring a specific local machine setup that is not declared.
  \item Hiding required services, credentials, or commands outside README/config.
  \item Non-deterministic tests due to time, randomness, network, or external APIs.
  \item No schema at boundaries.
  \item No typed boundary objects for agent-edited APIs.
  \item Security decisions spread across unrelated files.
  \item “Temporary” bypasses without issue, owner, and expiry.
\end{itemize}

\subsection{11. Network, HTTP, API, and SSRF sins}
\begin{itemize}
  \item Never call \texttt{requests.get()} or similar network functions without timeouts in services, workers, CLIs, or CI.
  \item Never let user input choose arbitrary outbound URLs.
  \item Never fetch URLs from webhooks, uploaded files, metadata, OpenGraph tags, PDFs, XML, or LLM output without SSRF defenses.
  \item Never allow server-side fetches to internal IP ranges unless explicitly intended.
  \item Never allow server-side fetches to cloud metadata IPs.
  \item Never follow redirects blindly for user-controlled URLs.
  \item Never forward internal headers, cookies, or tokens to user-controlled URLs.
  \item Never disable TLS verification.
  \item Never send credentials over plaintext HTTP.
  \item Never expose debug servers to public interfaces.
  \item Never run Flask/Django/FastAPI debug mode in production.
  \item Never expose Werkzeug/Django debug consoles.
  \item Never bind development servers to \texttt{0.0.0.0} on untrusted networks.
  \item Never trust inbound webhook payloads without verifying signatures.
  \item Never trust IP allowlists alone for webhook security.
  \item Never parse API responses without schema validation.
  \item Never assume third-party APIs always return expected fields.
  \item Never log full request/response bodies when they may contain secrets or PII.
  \item Never implement clients without backoff, timeout, and rate-limit handling.
  \item Never let an agent browse, scrape, or call APIs with privileged credentials without domain allowlists.
\end{itemize}

\begin{itemize}

  \item Run apps with write access to their own source code.
  \item Mount Docker socket into app containers.
  \item Give app containers cloud-admin IAM roles.
  \item Use one environment for dev/staging/prod.
  \item Use production databases for local testing.
  \item Run migrations automatically on every container start without coordination.
  \item Deploy without health checks.
  \item Deploy without rollback.
  \item Deploy without timeouts.
  \item Deploy without resource limits.
  \item Deploy without dependency vulnerability visibility.
  \item Store local state in containers that are expected to be ephemeral.
  \item Use SQLite for multi-writer production workloads where it cannot meet concurrency/recovery requirements.
  \item Ignore backups and restore tests.
  \item Encrypt backups with keys stored beside the backups.
  \item Keep logs forever without retention and privacy controls.
  \item Disable audit trails because they cost money.
  \item Expose internal services to the public internet for convenience.
  \item Leave default passwords or default admin accounts.
  \item Use shared admin accounts.
  \item Disable MFA on admin, cloud, GitHub, GitLab, PyPI, or CI accounts.
  \item Use long-lived deploy keys with broad write access.
  \item Let CI/CD deploy from unprotected branches.
  \item Let untrusted PRs run privileged CI.
\end{itemize}

\subsection{12. Data integrity and privacy failures}
\begin{itemize}
  \item Trust client timestamps, prices, balances, permissions, quotas, or IDs.
  \item Trust data simply because it came from your frontend.
  \item Trust data simply because it came from your queue.
  \item Process messages without schema validation.
  \item Ignore duplicate events.
  \item Ignore idempotency for payments, webhooks, order creation, email sending, and job processing.
  \item Store PII you do not need.
  \item Log PII by default.
  \item Put PII in metrics labels.
  \item Put secrets or PII in exception tracking breadcrumbs.
  \item Export customer data without authorization checks.
  \item Build admin bulk actions without dry-run, confirmation, and audit logs.
  \item Mix tenants in caches without tenant-scoped keys.
  \item Cache personalized responses publicly.
  \item Use user-controlled cache keys without normalization.
  \item Use pickle-backed caches for untrusted data.
  \item Use stale authorization data indefinitely.
  \item Fail open when policy services are unreachable.
  \item Delete data without soft-delete/recovery where business risk requires recoverability.
  \item Build “temporary” data exports with no retention policy.
\end{itemize}

  \item Never shadow stdlib module names with local filenames.
  \item Never rely on \texttt{assert} for required runtime behavior.
  \item Never ignore timezone awareness in datetime code.
  \item Never store local naive datetimes for cross-timezone systems.
  \item Never compare naive and aware datetimes casually.
  \item Never use floating-point money.
  \item Never use binary floats for exact decimal, billing, or accounting logic.
  \item Never mutate a collection while iterating over it unless deliberately coded and tested.
  \item Never rely on object finalizers for critical cleanup.
  \item Never forget context managers for files, locks, DB sessions, temp dirs, and network clients.
  \item Never use \texttt{except:} to catch \texttt{KeyboardInterrupt} and \texttt{SystemExit} accidentally.
  \item Never use \texttt{from x import *} in production modules.
  \item Never use hidden module-level side effects as configuration.
  \item Never use circular imports as an architecture pattern.
  \item Never write async code that blocks the event loop with CPU or synchronous I/O.
  \item Never forget \texttt{await} and ignore coroutine warnings.
  \item Never share non-thread-safe objects across threads without protection.
  \item Never assume the GIL makes compound operations logically safe.
  \item Never use multiprocessing without understanding serialization and process boundaries.
  \item Never rely on object \texttt{\_\_del\_\_} for releasing security-sensitive resources.
  \item Never use metaprogramming to avoid straightforward code in security-sensitive paths.
\end{itemize}

\subsection{17. Database and migration sins}
\begin{itemize}
  \item Never run raw SQL with interpolated user input.

\subsection{Never execute dynamic code from untrusted or uncontrolled input}

  \item No retry limits.

\end{itemize}

\subsection{Never hide failures}
\begin{itemize}

\begin{itemize}

  \item **Unpickling untrusted data** with \texttt{pickle}, \texttt{dill}, \texttt{shelve}, \texttt{joblib}, \texttt{pandas.read\_pickle}, model checkpoints, or pickle-backed loaders. Python explicitly says pickle is not secure and malicious pickle data can execute arbitrary code.
