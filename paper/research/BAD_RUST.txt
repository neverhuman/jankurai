\section{Inexcusable RUST Anti-Patterns}
\label{sec:bad_rust}

The following section categorizes the most severe, inexcusable RUST anti-patterns observed in production environments and AI-generated code. These patterns compromise safety, maintainability, and performance.\

\subsection{Known Best Practices}
To counteract the anti-patterns, ensure adherence to these core RUST best practices:
\begin{itemize}
  \item Embrace the borrow checker: Prefer references and lifetimes over cloning or \texttt{Rc}/\texttt{Arc} when ownership is clear.
  \item Use \texttt{Result} and \texttt{Option} strictly: Avoid \texttt{.unwrap()} and \texttt{.expect()} in production code. Handle errors gracefully using \texttt{?}.
  \item Leverage the type system: Use newtypes to encapsulate and validate domain invariants.
  \item Keep \texttt{unsafe} minimal and isolated: Wrap unsafe blocks in safe abstractions with rigorous testing.
  \item Practice modular architecture: Split large crates into workspace members with clear boundaries.
\end{itemize}

\subsection{Borrow-checker bypasses that lie}
\begin{itemize}
  \item \texttt{mem::transmute} to extend a lifetime.
  \item Turning a borrowed reference into \texttt{'static} because an API “wanted static.”
  \item Using \texttt{Box::leak} as a lazy lifetime fix in long-running code.
  \item Using raw pointers to create aliasing that references would reject.
  \item Using \texttt{Rc<RefCell<\_>>} or \texttt{Arc<Mutex<\_>>} merely to dodge ownership design.
  \item Creating self-referential structs with raw pointers without a proven pinning/drop story.
  \item Storing references into containers while pretending moves cannot happen.
  \item Using \texttt{unsafe} because “the borrow checker is too strict” without proving the compiler-rejected case is actually valid.
\end{itemize}

\subsection{Crypto}
\begin{itemize}
  \item Rolling your own crypto.
  \item Using non-cryptographic RNGs for keys, nonces, salts, IVs, session IDs, reset tokens, CSRF tokens, or capability URLs.
  \item Reusing nonces in AEAD modes.
  \item Fixed IVs/nonces.
  \item Hardcoded keys.
  \item Unauthenticated encryption.
  \item Disabling TLS certificate validation.
  \item Accepting invalid certificates in production.
  \item Inventing password hashing.
  \item Using fast hashes for passwords.
  \item Using deprecated algorithms because the code compiled.
  \item Treating crypto examples as production architecture.
  \item Calling \texttt{unwrap} on cryptographic operations involving attacker-controlled ciphertext, because that can turn malformed input into denial of service.
\end{itemize}

\subsection{False thread-safety}
\begin{itemize}
  \item \texttt{unsafe impl Send} or \texttt{unsafe impl Sync} because “it seems fine.”
  \item Sharing non-thread-safe FFI handles across threads.
  \item Wrapping a raw pointer in a newtype and declaring it \texttt{Send}/\texttt{Sync} without proving the pointee, ownership, and foreign library are thread-safe.
  \item Using atomics as decoration without a memory-ordering argument.
  \item Mixing atomics and non-atomic access to the same memory.
  \item Assuming “single-threaded today” makes a type safe to mark \texttt{Send}.
  \item Ignoring reentrancy, callbacks, signal handlers, or interrupt contexts.
  \item \texttt{unsafe} used as a compiler-silencer.
  \item \texttt{unsafe} blocks with no safety comment.
  \item Safety comments that merely restate the code: “SAFETY: this is safe.”
  \item Safety comments that say “tested,” “works on my machine,” or “AI generated this.”
  \item Huge unsafe blocks that mix unsafe operations with ordinary logic.
  \item Unsafe code whose invariants are spread across unrelated modules with no documentation.
  \item Unsafe code in public APIs without a \texttt{\# Safety} section.
  \item \texttt{unsafe fn} bodies that perform unsafe operations without explicit local unsafe blocks.
  \item \texttt{unsafe trait} implementations without proving the trait contract.
  \item Hiding unsafe inside macros, build scripts, proc macros, or generated files.
  \item Adding \texttt{\#![allow(unsafe\_code)]} or broad lint suppressions instead of reviewing each unsafe site.
  \item Copying unsafe snippets from StackOverflow, GitHub, Reddit, or an LLM without reading the docs for the exact API and version.
  \item Treating Miri, tests, fuzzing, or “it ran once” as proof of soundness. They are evidence, not proof.
  \item Accepting unsafe code from a dependency without checking whether the crate has active maintenance, advisories, tests, and a credible safety story.
  \item Wrong \texttt{extern} ABI.
  \item Calling C functions with the wrong signature.
  \item Passing Rust references to C that can store them beyond their lifetime.
  \item Passing \texttt{\&mut T} to C while Rust aliases still exist.
  \item Passing Rust structs across FFI without \texttt{\#[repr(C)]} or an explicit layout guarantee.
  \item Passing enums, trait objects, slices, \texttt{String}, \texttt{Vec}, \texttt{Option<T>}, \texttt{Result<T, E>}, or Rust-owned generics across FFI as if they were C ABI types.
  \item Freeing memory on the wrong side of the FFI boundary.
  \item Using \texttt{CString::from\_raw} on memory not created by \texttt{CString::into\_raw}.
  \item Letting C mutate memory that Rust believes is immutable.
  \item Accepting null pointers without checking them.
  \item Ignoring alignment, length, ownership, encoding, and initialization of foreign buffers.
  \item Treating foreign \texttt{char *} as UTF-8 without validation.
  \item Returning borrowed pointers to temporary Rust data.
  \item Allowing callbacks after the Rust object they reference has been dropped.
  \item Assuming a foreign library is thread-safe, async-signal-safe, reentrant, or fork-safe without documentation.
  \item Letting panics cross an FFI boundary with the wrong ABI.
  \item Using \texttt{extern "C"} where unwinding is possible and should be modeled as \texttt{C-unwind}, or using \texttt{C-unwind} casually without understanding the other language’s exception/unwind settings.
  \item Using \texttt{\#[no\_mangle]}, \texttt{\#[export\_name]}, or \texttt{\#[link\_section]} without understanding the global symbol/linker safety requirements. Rust 2024 requires marking these as unsafe attributes because the compiler cannot verify their soundness requirements.
  \item \texttt{unwrap()} or \texttt{expect()} on user input, network input, filesystem input, environment variables, database rows, CLI args, IPC, untrusted JSON, decrypted data, or anything attacker-influenced.
  \item \texttt{panic!} as normal error handling in libraries.
  \item Panicking in public APIs without documenting the contract.
  \item Panicking in \texttt{Drop}.
  \item Panicking across FFI.
  \item Panicking inside async tasks where the \texttt{JoinHandle} is ignored.
  \item Treating malformed input as impossible.
  \item Discarding \texttt{Result} with \texttt{let \_ = ...}.
  \item Calling \texttt{.ok()} or \texttt{.err()} just to throw information away.
  \item \texttt{map\_err(|\_| Error::Failed)} that destroys critical context.
  \item Logging an error and continuing as if the operation succeeded.
  \item Retrying forever without backoff, limit, or cancellation.
  \item Converting all errors to strings too early.
  \item Using \texttt{anyhow::Error} as a public library API when callers need structured recovery.
  \item Returning \texttt{Option} for a failure that needs an error explanation.
  \item Returning \texttt{bool} for a failure that has multiple causes.
  \item Swallowing \texttt{JoinError}, channel-close errors, task failures, flush failures, fsync failures, or serialization failures.
  \item Calling \texttt{todo!()}, \texttt{unimplemented!()}, or \texttt{unreachable!()} in reachable production paths.
  \item Depending on \texttt{debug\_assert!} for security or memory safety.
  \item Holding a \texttt{std::sync::MutexGuard}, \texttt{RwLockGuard}, \texttt{RefCell} borrow, or other blocking guard across \texttt{.await}.
  \item Blocking an async runtime thread with \texttt{std::thread::sleep}, blocking filesystem/network calls, blocking crypto, CPU-heavy loops, or synchronous clients.
  \item Spawning tasks and ignoring their \texttt{JoinHandle} when task failure matters.
  \item Fire-and-forget background tasks with no shutdown path.
  \item Unbounded channels fed by external input.
  \item No backpressure on network, queue, WebSocket, streaming, logging, or telemetry paths.
  \item \texttt{tokio::select!} code that is not cancellation-safe.
  \item Assuming dropping a future rolls back partial side effects.
  \item Creating a new runtime inside an existing runtime.
  \item Mixing async runtimes accidentally.
  \item Using sleeps as synchronization.
  \item Relying on timing to make tests pass.
  \item Holding locks while calling user code, callbacks, logging hooks, or async code.
  \item Lock-order inversions.
  \item \texttt{Arc<Mutex<\_>>} as default architecture for shared mutable state.
  \item Shared global mutable state without a concurrency model.
  \item Using \texttt{static mut}.
  \item Using atomics without documenting the invariant.
  \item Using \texttt{Relaxed} because it is faster, not because it is correct.
  \item Busy-spinning without yielding, parking, backoff, or power considerations.
  \item Ignoring poisoning or pretending poisoning is recovery.
  \item Assuming code is single-threaded because today’s executor happens to run it that way.
  \item Blindly adding a crate because an LLM suggested it.
  \item Adding a crate whose name, repository, docs, ownership, maintenance, license, and advisory status were not checked.
  \item Trusting a hallucinated crate name.
  \item Depending on random Git branches for production.
  \item Wildcard dependency versions.
  \item Unpinned git dependencies.
  \item Ignoring RustSec advisories.
  \item Ignoring yanked crates.
  \item Ignoring unmaintained crates in security-sensitive paths.
  \item Ignoring duplicate versions of critical crates, especially crypto, TLS, parsers, allocators, async runtimes, and serialization crates.
  \item Allowing build scripts from untrusted crates without review.
  \item Letting \texttt{build.rs} download code, run shell commands, depend on network state, or inspect host secrets unexpectedly.
  \item Pulling in default features without checking what they enable.
  \item Accidentally enabling OpenSSL/native TLS, vendored C, proc macros, networking, compression, or filesystem features.
  \item Non-additive features: features that remove APIs, change semantics, or conflict with other features.
  \item Changing default features casually.
  \item Publishing a library without testing feature combinations.
  \item Shipping an application without a \texttt{Cargo.lock}.
  \item Treating \texttt{cargo update} as harmless in a release branch.
  \item Failing to check licenses and source provenance.
  \item Relying on abandoned crates for core safety/security behavior.
  \item Vendoring C/C++ code without tracking its CVEs and compiler flags.
  \item Letting proc macros from unknown crates execute in your build pipeline.
  \item Accepting dependency bloat because “Cargo makes it easy.”
\end{itemize}

\subsection{Invalid initialization and destruction}
\begin{itemize}
  \item \texttt{MaybeUninit::assume\_init()} before every byte and invariant is initialized.
  \item \texttt{mem::zeroed()} for types where all-zero is invalid.
  \item \texttt{mem::uninitialized()} or equivalent patterns.
  \item \texttt{Vec::set\_len()} before elements are fully initialized.
  \item Double-dropping values.
  \item Forgetting values to paper over ownership bugs.
  \item \texttt{ManuallyDrop} used without a precise drop protocol.
  \item \texttt{ptr::read}, \texttt{ptr::copy}, or \texttt{ptr::copy\_nonoverlapping} with wrong ownership, overlap, alignment, or initialization assumptions.
  \item \texttt{Box::from\_raw}, \texttt{Vec::from\_raw\_parts}, \texttt{CString::from\_raw}, or allocator APIs with pointers not produced by the matching owner/allocator.
  \item Creating references to packed fields that may be unaligned.
  \item Creating \texttt{\&T} or \texttt{\&mut T} from raw pointers without proving non-null, aligned, initialized, dereferenceable, valid, and correctly aliased.
\end{itemize}

\subsection{Pointers, references, aliasing, and validity}
\begin{itemize}
  \item **Dereferencing null, dangling, misaligned, or invalid raw pointers.**
  \item **Creating a Rust reference from a raw pointer before proving validity, alignment, provenance, aliasing, and lifetime.**
  \item **Creating \texttt{\&mut T} when any other live reference or pointer can read or write the same memory in violation of aliasing rules.**
  \item **Creating multiple mutable references to the same object through raw pointers.**
  \item **Mutating through \texttt{\&T} except through valid interior mutability such as \texttt{UnsafeCell}.**
  \item **Producing invalid values for Rust types: invalid \texttt{bool}, invalid \texttt{char}, invalid enum discriminants, invalid \texttt{NonZero*}, invalid \texttt{NonNull}, invalid references, invalid \texttt{String}, invalid \texttt{Vec}, or invalid niche-optimized types.**
  \item **Assuming all bit patterns are valid for a type.**
  \item **Reading uninitialized memory as a typed value.** The Rustonomicon is blunt: interpreting uninitialized memory as a value causes UB.
  \item **Calling \texttt{MaybeUninit::assume\_init} before every byte that matters is initialized to a valid value.**
  \item **Dropping uninitialized memory.**
  \item **Double-dropping through \texttt{ptr::read}, \texttt{ManuallyDrop}, \texttt{MaybeUninit}, \texttt{mem::forget}, or custom drop glue.**
  \item **Using \texttt{mem::zeroed} on types where the all-zero bit pattern is invalid.**
  \item **Using \texttt{transmute} as a normal conversion tool.** The standard library says both source and destination must be valid at their types, violating that is UB, and \texttt{transmute} should be an absolute last resort.
  \item **Transmuting lifetimes to \texttt{'static} unless the value truly lives for \texttt{'static}.**
  \item **Transmuting between pointer and integer types without understanding the memory model and provenance.**
  \item **Transmuting containers like \texttt{Vec<T>} to \texttt{Vec<U>} by assuming internal layout.**
  \item **Relying on \texttt{repr(Rust)} layout for FFI, disk formats, network formats, or cross-version compatibility.**
  \item **Using \texttt{slice::from\_raw\_parts} or \texttt{from\_raw\_parts\_mut} without proving pointer validity, alignment, length, initialized memory, and aliasing.**
  \item **Using \texttt{get\_unchecked} when the index can be influenced by input, races, or stale validation.**
  \item **Using \texttt{unreachable\_unchecked} for a branch that is merely “unlikely.”**
  \item **Using \texttt{assume}, unchecked math, unchecked indexing, or unchecked UTF-8 as performance tricks without proof.**
\end{itemize}

\subsection{Secrets}
\begin{itemize}
  \item Hardcoded API keys, tokens, passwords, private keys, salts, or seed phrases.
  \item Secrets in tests that are also valid in production.
  \item Secrets in logs, panics, telemetry, metrics labels, traces, \texttt{Debug}, \texttt{Display}, or error messages.
  \item Deriving \texttt{Debug}, \texttt{Clone}, \texttt{Serialize}, or \texttt{Deserialize} on secret-holding types without thinking.
  \item Storing passwords as plaintext.
  \item Comparing secrets with ordinary equality when constant-time comparison is needed.
  \item Keeping sensitive data in memory longer than necessary.
  \item Assuming \texttt{drop} zeroes memory.
  \item Using \texttt{String}/\texttt{Vec<u8>} for secrets without a zeroization story in high-risk code.
  \item Sending secrets to an LLM or external tool.
\end{itemize}

\subsection{The core rule}

  \item **Creates public APIs that hide invariants, errors, panics, blocking, or unsafety.**
  \item **Breaks dependency, feature, or semver trust.**
  \item **Treats “it compiles” as evidence of correctness.**
  \item Accepting AI-written Rust because it compiles, without understanding ownership, lifetimes, trait bounds, error behavior, concurrency behavior, and safety invariants.
  \item Asking an AI to “fix the borrow checker” and accepting the first version that adds \texttt{clone()}, \texttt{Arc<Mutex<\_>>}, \texttt{Box::leak}, \texttt{Rc<RefCell<\_>>}, \texttt{static mut}, \texttt{unsafe}, or \texttt{transmute}.

\subsection{The highest-priority “never justified” Rust behaviors}
\begin{itemize}
  \item **Creating undefined behavior and calling it acceptable.** This includes data races, dangling or misaligned pointer access, invalid values, broken aliasing, mutating immutable bytes, invalid lifetimes, and wrong assumptions inside \texttt{unsafe}. The Rust Reference’s UB list is explicitly serious and also warns it is not exhaustive.
  \item **Publishing a safe API that safe callers can use to trigger UB.** That is unsound Rust. This is worse than merely “using unsafe,” because it leaks unsafety into code that appears safe. Real ecosystem bugs include safe Rust triggering UB through incorrect \texttt{Send}/\texttt{Sync} bounds in \texttt{futures-rs}, and GitLab advisories for crates with invalid bit patterns, out-of-bounds iterator advances, and uninitialized \texttt{MaybeUninit::assume\_init} usage.
  \item **Writing \texttt{unsafe} without a real safety proof.** “Seems fine,” “the AI said so,” “copied from StackOverflow,” “Miri passed once,” or “the tests pass” is not a proof. The Rust Book recommends keeping unsafe blocks small, wrapping unsafe code in safe abstractions, and using Miri to check unsafe code.
  \item **Suppressing lints, warnings, tests, or audits just to get green CI.** Rust and Clippy expose mechanisms to \texttt{allow}, \texttt{warn}, \texttt{deny}, and \texttt{forbid} lints; using \texttt{allow} to hide a real problem without a reason is a review smell. Clippy also warns that restriction lints should be enabled case-by-case, not blindly.
  \item **Treating “it compiles” as “it is correct.”** Rust prevents many memory-safety bugs in safe code, but it does not prove business logic, cancellation safety, panic policy, input validation, resource limits, cryptography, SQL safety, authorization, protocol compatibility, or operational resilience.
  \item **Accepting generated Rust without understanding every invariant it relies on.**
  \item **Letting an AI introduce \texttt{unsafe}, FFI, \texttt{Send}, \texttt{Sync}, \texttt{Pin}, \texttt{transmute}, atomics, \texttt{static mut}, \texttt{no\_mangle}, \texttt{export\_name}, or lifetime tricks without human review.**
  \item **Using the compiler as a random patch oracle: adding \texttt{clone}, \texttt{Arc}, \texttt{Mutex}, \texttt{'static}, \texttt{Box::leak}, \texttt{unsafe}, \texttt{RefCell}, \texttt{Rc}, or \texttt{unwrap} until errors disappear.**
  \item **Satisfying the borrow checker by changing the program’s ownership model without checking semantics.**
  \item **Adding \texttt{Arc<Mutex<T>>} everywhere to avoid thinking about ownership.**
  \item **Adding \texttt{.clone()} everywhere to avoid thinking about borrowing, especially on large data, secret data, locked data, or hot paths.**
  \item **Adding \texttt{'static} bounds until async code compiles, without proving the data really can live that long.**
  \item **Using \texttt{Box::leak}, \texttt{mem::forget}, global singletons, or lazy statics to silence lifetime errors.**
  \item **Replacing precise error types with \texttt{anyhow::Error} or strings everywhere in a library API because the generated code was easier.**
  \item **Turning recoverable errors into panics because the AI could not thread \texttt{Result} through the call graph.**
  \item **Turning panics into \texttt{Result} while silently losing the invariant violation that the panic was meant to expose.**
  \item **Deleting failing tests because generated code changed behavior.**
  \item **Weakening tests to match generated output.**
  \item **Accepting generated benchmarks that do not measure release builds, realistic data, or optimized-away work.**
  \item **Accepting generated documentation that overclaims safety, complexity, performance, or security.**
  \item **Shipping generated code that uses crates the team has not reviewed.**
  \item **Letting an AI choose dependencies by popularity/name alone.**
  \item **Copying code from Reddit, X, StackOverflow, GitHub issues, or an LLM into \texttt{unsafe} without going back to primary docs.**
  \item **Inventing a \texttt{SAFETY:} comment after the fact to appease Clippy.**
  \item **Saying “the compiler would catch it” for code involving \texttt{unsafe}, FFI, logic, concurrency, IO, or external inputs.**
  \item **Trusting AI-generated FFI signatures.**
  \item **Trusting AI-generated \texttt{repr(C)} structs without checking C headers, padding, alignment, ABI, ownership, and platform differences.**
  \item **Trusting AI-generated atomic orderings.**
  \item **Trusting AI-generated \texttt{Pin} projections.**
  \item **Trusting AI-generated parsers for untrusted input without fuzz/property tests.**
\end{itemize}

\subsection{The master rule}
\begin{itemize}
  \item **Lies to the compiler** about lifetimes, aliasing, initialization, layout, thread-safety, or ownership.
  \item **Lies to users** by exposing a safe API that secretly requires unsafe preconditions.
  \item **Lies to reviewers** by hiding risk behind \texttt{unsafe}, broad \texttt{allow}, vague comments, generated code, or “it compiles.”
  \item **Turns expected failure into process failure** without a documented contract.
  \item **Treats AI output as authority** rather than as untrusted draft code.
  \item **Ships code whose safety/security story the author cannot explain.**
\end{itemize}

\subsection{Undefined behavior}
\begin{itemize}
  \item Data races.
  \item Dereferencing null, dangling, or misaligned pointers.
  \item Out-of-bounds pointer arithmetic or slice access.
  \item Breaking Rust aliasing rules, especially creating multiple active mutable aliases.
  \item Mutating memory through a shared reference except through \texttt{UnsafeCell}.
  \item Producing invalid values such as invalid \texttt{bool}, invalid \texttt{char}, invalid enum discriminants, invalid references, uninitialized integers, or invalid wide-pointer metadata.
  \item Calling a function with the wrong ABI.
  \item Unwinding through a stack frame that does not permit it.
  \item Incorrect inline assembly.
  \item Running code compiled for unsupported CPU target features.
  \item Violating Rust runtime assumptions, including stack-frame/destructor assumptions around foreign \texttt{longjmp}-style behavior.
\end{itemize}

\subsection{Unsafe blocks and functions}
\begin{itemize}
  \item **Using \texttt{unsafe} as a way to bypass the borrow checker rather than encode the real invariant.**
  \item **Writing an unsafe block without a nearby \texttt{SAFETY:} explanation.** Clippy’s \texttt{undocumented\_unsafe\_blocks} lint exists because undocumented unsafe blocks and impls are harder to maintain and can hide unsoundness.
  \item **Writing a \texttt{SAFETY:} comment that restates the code instead of proving the preconditions.**
  \item **Writing a false or unverifiable \texttt{SAFETY:} comment.**
  \item **Putting many unrelated unsafe operations into one block so reviewers cannot audit each obligation separately.** Clippy has a \texttt{multiple\_unsafe\_ops\_per\_block} restriction lint for this exact auditability reason.
  \item **Assuming \texttt{unsafe fn} means the whole body should be implicitly unsafe.** Rust 2024 warns by default for unsafe operations inside unsafe functions without explicit unsafe blocks, because conflating “caller has obligations” with “body may perform unsafe operations” was considered too risky.
  \item **Making a function safe when the caller must uphold memory-safety preconditions.**
  \item **Making a trait safe when implementors must uphold safety invariants that unsafe code relies on.**
  \item **Hiding unsafe behavior inside a macro so the callsite looks safe.** Clippy explicitly calls out macros that hide unsafe blocks and allow unsafe operations in seemingly safe code.
  \item **Adding \texttt{\#![allow(unsafe\_code)]} or \texttt{\#[allow(...)]} around unsafe merely to suppress review.**
  \item **Using \texttt{unsafe} in generated code without marking generated files as such and without a review process.**
  \item **Relying on a private invariant from another crate unless that invariant is part of its public, documented contract.**
  \item **Depending on debug-only checks to justify release-mode unsafe code.**
\end{itemize}

\subsection{Unsafe traits and marker traits}
\begin{itemize}
  \item **Incorrectly implementing \texttt{Send} or \texttt{Sync}.** The Rustonomicon says these are unsafe traits, unsafe code can rely on them being correct, and incorrect implementations can cause UB.
  \item **Marking a type \texttt{Send} because “it compiles except for this raw pointer.”**
  \item **Marking a type \texttt{Sync} while it contains unsynchronized interior mutability.**
  \item **Treating \texttt{Rc}, \texttt{RefCell}, \texttt{Cell}, raw pointers, or non-thread-safe C handles as thread-safe by wrapping them in newtypes with unsafe impls.**
  \item **Implementing unsafe traits like \texttt{TrustedLen}, allocator traits, or FFI-related traits without satisfying every invariant.**
  \item **Deriving or implementing \texttt{Copy}, \texttt{Clone}, \texttt{Default}, \texttt{Deref}, \texttt{Drop}, \texttt{Send}, or \texttt{Sync} in a way that violates ownership or safety invariants.**
\end{itemize}

\subsection{Unsound safe APIs}
\begin{itemize}
  \item \texttt{pub fn from\_raw(ptr: *mut T) -> Self} as a safe function when the pointer must be valid, aligned, non-null, uniquely owned, or allocator-compatible.
  \item A safe wrapper around C that assumes the C pointer is valid.
  \item A safe function that accepts any \texttt{usize} and internally calls \texttt{get\_unchecked}.
  \item A safe constructor that can produce invalid internal state.
  \item A public field that lets users break invariants relied on by unsafe code.
  \item A macro that expands to unsafe behavior while presenting a safe API.
  \item A safe \texttt{Iterator}, \texttt{Read}, \texttt{Future}, or callback API whose implementation violates hidden invariants.
\end{itemize}
