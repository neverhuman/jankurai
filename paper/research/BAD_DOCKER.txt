\section{Inexcusable DOCKER Anti-Patterns}
\label{sec:bad_docker}

The following section categorizes the most severe, inexcusable DOCKER anti-patterns observed in production environments and AI-generated code. These patterns compromise safety, maintainability, and performance.\

\subsection{Known Best Practices}
To counteract the anti-patterns, ensure adherence to these core DOCKER best practices:
\begin{itemize}
  \item Keep images small and secure: Use minimal base images (e.g., Alpine or Distroless) and multi-stage builds.
  \item Run as non-root: Always define a \texttt{USER} instruction to limit privilege escalation risks.
  \item Avoid embedding secrets: Pass secrets at runtime or build-time via safe mechanisms, never store them in layers.
  \item Pin base image tags: Use specific versions or SHA digests instead of \texttt{latest}.
  \item Optimize build caching: Order Dockerfile instructions to maximize layer caching (copy dependency files before source code).
\end{itemize}

\subsection{10.1 Dockerfile policy}
\begin{itemize}
  \item Trusted base image.
  \item Specific tag or digest.
  \item \texttt{.dockerignore}.
  \item Lockfile copied before source.
  \item Dependency install before app copy.
  \item Multi-stage build where applicable.
  \item No secrets in layers.

\subsection{2.1 Base image mistakes}
\begin{itemize}
  \item Using random images from unknown publishers.
  \item Using abandoned base images.
  \item Using unnecessarily large bases such as full Ubuntu/Debian when a slim or distroless runtime would work.
  \item Using mutable tags like \texttt{latest}, \texttt{stable}, \texttt{dev}, \texttt{main}, or unqualified tags.
  \item Using digest pinning with no update automation, causing images to silently age.
  \item Using images from public registries without verifying publisher, signature, digest, or provenance.
  \item Pulling from untrusted registries over insecure transport.
  \item Mixing package ecosystems from multiple distributions.
  \item Using EOL distributions.
  \item Using language runtime images that include compilers, shells, package managers, test tooling, and build credentials in the final production image.
\end{itemize}

\subsection{2.11 Bad multi-stage build behavior}
\begin{itemize}
  \item Compiler remains in final image.
  \item Source code remains in final image.
  \item Build cache and module cache may remain.
  \item Secrets used during build may remain.
  \item Final image is larger than necessary.
  \item Attack surface is larger.
  \item Startup can be slower if build happens at container start.
  \item Scanning finds more irrelevant CVEs.
  \item Final image may include package managers and shells.
  \item Runtime and build concerns are mixed.
\end{itemize}

\subsection{2.12 Bad ENTRYPOINT/CMD behavior}
\begin{itemize}
  \item Shell-form commands can mishandle signals.
  \item PID 1 may not reap zombie processes.
  \item \texttt{tail -f /dev/null} hides process failure.
  \item Startup scripts can swallow exit codes.
  \item Multiple daemons in one container complicate shutdown and health.
  \item App cannot receive SIGTERM cleanly.
  \item \texttt{docker stop} can hang or kill ungracefully.
  \item Logs may go to files instead of stdout/stderr.
  \item Shell expansion can create injection risks.
  \item Debug wrappers become production behavior.
\end{itemize}

\subsection{2.13 Healthcheck mistakes}
\begin{itemize}
  \item No healthcheck for long-running services.
  \item Healthcheck only checks process existence.
  \item Healthcheck requires internet access.
  \item Healthcheck mutates state.
  \item Healthcheck authenticates with a real privileged credential.
  \item Healthcheck logs secrets.
  \item Healthcheck interval is too aggressive.
  \item Healthcheck timeout is too short for cold starts.
  \item Healthcheck uses tools missing from minimal image.
  \item Compose \texttt{depends\_on} used as a replacement for readiness.
\end{itemize}

\subsection{2.4 Bad package-manager behavior}
\begin{itemize}
  \item \texttt{apt-get update} and \texttt{apt-get install} in separate layers can use stale package indexes because of Docker cache behavior.
  \item Installing without \texttt{--no-install-recommends} pulls unnecessary packages.
  \item Leaving \texttt{/var/lib/apt/lists/*} bloats the image.
  \item Installing broad toolsets increases attack surface.
  \item Running \texttt{apt-get upgrade} or \texttt{dist-upgrade} inside app images can make builds less predictable.
  \item Not pinning package versions can make rebuilds non-reproducible.
  \item Installing compilers and build tools into the final runtime image increases exploitability.
  \item Installing from random curl scripts bypasses package verification.
  \item Installing from default package repos without checking EOL or security support.
  \item Installing debug tools “temporarily” and never removing them.
\end{itemize}

\subsection{2.5 `curl | bash` and unverified downloads}
\begin{itemize}
  \item No checksum verification.
  \item No signature verification.
  \item No pinned version.
  \item No auditable artifact source.
  \item Installer script can change without notice.
  \item Build becomes non-reproducible.
  \item Build can execute compromised installer content.
  \item Downloaded artifact can be replaced server-side.
  \item Redirects can fetch unexpected content.
  \item Scripts often install extra dependencies.
\end{itemize}

\subsection{2.6 Misusing `ADD`}
\begin{itemize}
  \item \texttt{ADD} has extra behavior that can surprise reviewers.
  \item Remote URL downloads are harder to verify.
  \item Local tar auto-extraction can hide unexpected file structure.
  \item Cache behavior can be confusing.
  \item Security review is harder than with explicit \texttt{COPY}, \texttt{curl}, checksum, and \texttt{tar}.
\end{itemize}

\subsection{2.7 Bad file permissions}
\begin{itemize}
  \item World-writable directories invite tampering.
  \item Recursive permission changes create huge layers.
  \item Setuid/setgid binaries increase privilege-escalation risk.
  \item Root-owned app files can force runtime root.
  \item Writable code directories let attackers modify application code after compromise.
  \item Writable config directories can let attackers alter runtime behavior.
  \item Overly broad permissions hide the real ownership problem.
  \item Recursive \texttt{chown} after \texttt{COPY} is slow on large contexts.
  \item Permission mistakes often trigger agents to add \texttt{privileged: true}.
  \item Permissions may behave differently with bind mounts across host OSes.
\end{itemize}

\subsection{2.8 No `.dockerignore`}
\begin{itemize}
  \item Larger build context.
  \item Slower local and CI builds.
  \item Cache invalidations from irrelevant files.
  \item Secret leakage.
  \item Source-control history leakage.
  \item Test artifacts in production image.
  \item Host dependencies copied into image.
  \item Non-reproducible builds.
  \item Unnecessary Docker daemon upload time.
  \item More vulnerability scanner noise.
\end{itemize}

\subsection{3.3 Unstable build inputs}
\begin{itemize}
  \item Unpinned branch.
  \item Unpinned package version.
  \item Unpinned artifact.
  \item No lockfile.
  \item Build output changes over time.
  \item Cache becomes misleading.
  \item Rollbacks become difficult.
  \item Vulnerability triage becomes impossible.
  \item SBOMs are less useful.
  \item “Works on my machine” becomes likely.
  \item Use lockfiles: \texttt{package-lock.json}, \texttt{pnpm-lock.yaml}, \texttt{poetry.lock}, \texttt{Pipfile.lock}, \texttt{go.sum}, \texttt{Cargo.lock}.
  \item Use deterministic installers: \texttt{npm ci}, \texttt{pnpm install --frozen-lockfile}, \texttt{pip install --require-hashes}, \texttt{poetry install --sync}, \texttt{go mod download}.
  \item Pin Git SHAs, not branches.
  \item Verify checksums for downloaded artifacts.
  \item Emit SBOM/provenance.
\end{itemize}

\subsection{4.6 Startup ordering, health, and reliability mistakes}
\begin{itemize}
  \item Using short-form \texttt{depends\_on} and assuming the dependency is ready.
  \item No healthcheck on databases, brokers, or APIs.
  \item Healthchecks that only check “process exists” instead of “service is usable.”
  \item Healthchecks that require network tools not installed in the image.
  \item Healthchecks that mutate state.
  \item Healthchecks that are too frequent and create load.
  \item Healthchecks that leak credentials in command output.
  \item Disabling healthchecks inherited from images.
  \item \texttt{depends\_on} cycles.
  \item Migration jobs racing from multiple replicas.
  \item Long-running services depend on one-shot migration jobs but do not check success.
  \item \texttt{restart: always} on one-shot jobs.
  \item No \texttt{restart} policy for production services that should recover.
  \item \texttt{restart: always} on services that should fail visibly.
  \item No graceful shutdown settings.
  \item No \texttt{stop\_grace\_period} for apps that need time to drain.
  \item No \texttt{init: true} for processes that spawn children and do not reap them.
  \item Entrypoint scripts swallowing signals.
  \item Containers kept alive with \texttt{sleep infinity} or \texttt{tail -f /dev/null}.
  \item Services write critical data to ephemeral container filesystem.
\end{itemize}

\subsection{4.9 Data persistence mistakes}
\begin{itemize}
  \item Database data stored in container writable layer.
  \item No named volume for stateful services.
  \item Named volumes with no backup.
  \item Backups exist but restores are never tested.
  \item \texttt{docker compose down -v} in README cleanup command without warning.
  \item \texttt{docker volume prune} in automated cleanup.
  \item Mixing dev and prod project names so volumes collide.
  \item Using the same volume for incompatible app versions.
  \item Mounting a single writable volume into multiple containers that are not designed for it.
  \item Mounting host directories with wrong ownership, then running container as root.
  \item Mounting NFS/SMB volumes without checking database compatibility.
  \item No data migration plan.
  \item No initialization idempotency.
  \item \texttt{depends\_on} used instead of migration orchestration.
  \item Secret files stored in persistent app volumes.
  \item Logs stored in app data volumes indefinitely.
  \item Unbounded caches stored in named volumes.
  \item Anonymous volumes created accidentally and forgotten.
\end{itemize}

\subsection{5.1 Privileged Docker-in-Docker everywhere}
\begin{itemize}
  \item docker:dind
  \item Privileged CI containers weaken host isolation.
  \item Untrusted PRs can attack the runner.
  \item Registry credentials can be stolen.
  \item Build secrets can be exfiltrated.
  \item Cache poisoning becomes easier.
  \item Host breakout blast radius increases.
  \item Secrets from other jobs may be reachable on persistent runners.
  \item Cloud metadata services may be reachable.
  \item Agent “fixes” build errors by further widening permissions.
  \item CI becomes a high-value attack surface.
  \item Use rootless builders where practical.
  \item Use restricted privileged mode only for explicitly allowed rootless DinD images.
  \item Use isolated ephemeral runners for privileged builds.
  \item Never run untrusted PRs with privileged mode and secrets.
  \item Pin Docker CLI/DinD images instead of using \texttt{docker:latest}.
\end{itemize}

\subsection{7.10 Scan, lint, and enforce policy}
\begin{itemize}
  \item **Dockerfile lint:** Hadolint parses Dockerfiles and integrates ShellCheck for shell in \texttt{RUN} instructions.
  \item **Image vulnerability scanning:** Trivy, Grype, Docker Scout, Clair, and similar tools.

\subsection{8.1 A strong Dockerfile pattern}
\begin{itemize}
  \item A trusted, minimal, pinned base image.
  \item A \texttt{.dockerignore}.
  \item Dependency manifests copied before application source.
  \item Lockfile-based deterministic dependency install.
  \item Multi-stage separation: dependency, test, build, runtime.
  \item BuildKit cache mounts for package managers.
  \item BuildKit secrets/SSH mounts for private dependencies.
  \item A minimal final runtime image.
  \item No package manager, compiler, VCS, or shell in final image unless required.
  \item A non-root runtime user.
  \item Exec-form \texttt{CMD}/\texttt{ENTRYPOINT}.
  \item No secrets in layers, args, env, labels, or logs.
  \item Metadata labels for source/revision/license when non-sensitive.
  \item Rebuild/update/scanning process.
\end{itemize}

\subsection{A. Secure, fast Dockerfile standard}
\begin{itemize}
  \item Dependency manifests are copied before source, so dependency layers remain cached when app code changes.
  \item The image uses a multi-stage pattern to keep runtime smaller.
  \item Packages are installed with \texttt{apt-get update \&\& apt-get install} in the same layer.
  \item The apt lists are removed to shrink the image.

\end{itemize}

\subsection{Bad user, permission, and filesystem choices}
\begin{itemize}
  \item No \texttt{USER} instruction in final image.
  \item \texttt{USER root} in final image “because permissions are hard.”
  \item Creates a non-root user but leaves app files owned by root and writable only by root.
  \item Fixes permissions with \texttt{chmod -R 777}.
  \item Runs package managers at container startup.
  \item Leaves \texttt{sudo} installed in runtime image.
  \item Leaves SSH server installed in runtime image.
  \item Leaves compilers and linkers installed in runtime image.
  \item Leaves setuid/setgid binaries unnecessarily.
  \item Makes application source writable at runtime when it only needs read access.
  \item Stores runtime data inside the container writable layer instead of a volume.
  \item Uses root-owned volumes and then switches service back to root.
  \item Uses unpredictable user names rather than stable numeric UID/GID for production.
  \item Uses \texttt{chown -R} during startup, slowing every container start.
  \item Gives the container write access to configuration files that should be immutable.
  \item Assumes non-root automatically fixes all privilege problems. It helps, but capabilities, mounts, devices, and Docker socket access can still dominate risk.
\end{itemize}

\subsection{Runtime hardening best practices}
\begin{itemize}
  \item **Default to no privilege.** No \texttt{privileged}, no host namespaces, no host devices, no docker socket, no broad bind mounts.
  \item **Drop Linux capabilities.** Start with \texttt{cap\_drop: ["ALL"]}; add back only what the process truly needs. OWASP explicitly recommends this model.
  \item **Use \texttt{no-new-privileges}.** In Compose: \texttt{security\_opt: ["no-new-privileges:true"]}.
  \item **Keep default seccomp/AppArmor/SELinux unless you have a narrow, reviewed need.**
  \item **Use read-only root filesystems for stateless services.** Add tmpfs or named volumes only for specific writable paths.
  \item **Make bind mounts read-only whenever possible.** Docker says bind mounts are writable by default and can modify host files.
  \item **Set resource limits.** At minimum: memory, CPU, PIDs, file descriptors, and log rotation.
  \item **Do not disable the OOM killer casually.** Docker warns this can be dangerous without memory limits.
  \item **Use rootless Docker or user namespace remapping where feasible.** OWASP recommends evaluating rootless mode for strong isolation needs.
  \item **Keep host kernel, Docker Engine, Compose, BuildKit, and runtimes updated.** OWASP notes containers share the host kernel, so host/kernel vulnerabilities affect containers.
\end{itemize}
