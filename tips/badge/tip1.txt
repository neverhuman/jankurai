I studied the repo. The best path is **not** to bolt on an external badge service first. Jankurai already has most of the right pieces: local audit receipts, canonical score artifacts, versioned schemas, CI templates, and an early Phase 12 public badge generator. The missing product layer is: **a first-class README badge command + managed README block + CI write-back mode.**

## Recommendation

Make this a Jankurai standard surface:

```md
<!-- jankurai-badge:start -->
[![Jankurai score](agent/jankurai-badge.svg)](agent/repo-score.md)
<!-- jankurai-badge:end -->
```

That gives every adopted project a tiny, portable, repo-local badge that shows the Jankurai logo and score, links to the latest human-readable score report, and can be regenerated automatically by CI.

GitHub supports relative links and relative image paths in Markdown, and specifically recommends relative paths for images stored in the repository, so a repo-local `agent/jankurai-badge.svg` is the most robust default for public, private, forked, and cloned repos. ([GitHub Docs][1])

## What I found in Jankurai

Jankurai’s README already frames the tool as a local, receipt-backed conformance/audit system: `jankurai audit` writes JSON and Markdown score reports, plus score history, under predictable paths. The “Try Safely” flow writes `repo-score.json`, `repo-score.md`, and `score-history.jsonl`; the broader control-plane surfaces include audit, witness, score diff/trend, repair queues, and publish evidence. ([GitHub][2])

There is already a Phase 12 badge concept in the code. The CLI has `publish --badge-json` and `publish --badge-svg`, and `publish.rs` builds a `CertificationBadge` with fields like label, message, color, score, conformance level, standard version, and artifact link. It also renders a static SVG badge.  The artifact contract docs list `target/jankurai/public/jankurai-badge.json` and `target/jankurai/public/jankurai-badge.svg` as outputs generated by `jankurai publish`. ([GitHub][3])

The gap is that this badge is currently tied to **Phase 12 certification/publish**, not the ordinary adoption path. Most users will start with `audit`, `init`, and CI, not a full public evidence bundle. The normal score report already has the data needed for a README badge: score, raw score, policy/minimum score, decision, hard findings, conformance decision, standard version, auditor version, schema version, and report paths. 

## The standard I would add

Jankurai should have two related badges:

| Badge                  | Source                          | Meaning                                                                  | Default path                                                     |
| ---------------------- | ------------------------------- | ------------------------------------------------------------------------ | ---------------------------------------------------------------- |
| **Jankurai Score**     | `agent/repo-score.json`         | Current audit score for an adopted repo                                  | `agent/jankurai-badge.svg`                                       |
| **Jankurai Certified** | Phase 12 public evidence bundle | Release/certification claim after benchmark + certify + govern + publish | `docs/jankurai/public/jankurai-certified.svg` or hosted artifact |

The everyday badge should say something like:

```text
jankurai | score 86
jankurai | HL1 score 91
jankurai | blocked 64
jankurai | stale
```

The certification badge should be reserved for repos that pass the Phase 12 publishability checks. The existing publish code already evaluates whether a bundle is publishable by looking at score threshold, critical/high findings, hard caps, and benchmark failures/inconclusive tasks.  That is good: “certified” should mean more than “ran audit once.”

## New CLI surface

Add a new command:

```bash
jankurai badge . \
  --from agent/repo-score.json \
  --svg agent/jankurai-badge.svg \
  --json agent/jankurai-badge.json \
  --readme README.md \
  --link agent/repo-score.md \
  --yes
```

Suggested aliases:

```bash
jankurai badge update .
jankurai badge install .
jankurai badge check .
```

Behavior:

```text
jankurai badge install
  Inserts the managed README block if missing.

jankurai badge update
  Reads the current score/certification artifact, writes SVG + JSON, and refreshes only the managed README block.

jankurai badge check
  Fails or warns if the README badge is missing, stale, manually edited, or inconsistent with the latest score artifact.
```

I would **not** make ordinary `jankurai audit` mutate `README.md` by default. Jankurai’s adoption model is intentionally explicit: observe first, then opt into scaffolds, CI, hooks, and ratchets when the team chooses. ([GitHub][2]) The badge should become automatic after `init --yes` or `ci install --badge`, not during a read-only audit.

## README update rules

The README updater should be tiny and conservative:

```md
<!-- jankurai-badge:start -->
[![Jankurai score](agent/jankurai-badge.svg)](agent/repo-score.md)
<!-- jankurai-badge:end -->
```

Rules:

1. If the markers exist, replace only the content between them.
2. If the markers do not exist, insert after the first H1 and existing badge block.
3. If there is no README, create a minimal one with the badge and project title.
4. Never rewrite unrelated README prose.
5. Add a `--check` mode for CI that reports drift without writing.
6. Keep the generated SVG at height 20, with an accessible `role="img"` and `aria-label`.

This makes the badge safe enough to standardize across all Jankurai-adopted repos.

## Logo + score implementation

Extract the current badge rendering out of `commands/publish.rs` into a reusable module:

```text
crates/jankurai/src/report/badge.rs
```

Then both of these can use the same renderer:

```text
jankurai badge update
jankurai publish --badge-svg
```

Add a small embedded logo mark:

```text
assets/jankurai-mark.svg
```

Then embed that mark directly into the generated badge SVG. Do not load an external logo image from the badge. A self-contained SVG is faster, works offline, works in private repos, and avoids broken external image requests.

The default SVG should be one badge, not two separate images:

```text
[logo] jankurai | score 86
```

Color recommendation:

| State                                         | Color                |
| --------------------------------------------- | -------------------- |
| Score passes policy and no hard blockers      | brightgreen          |
| Score passes but has advisory concerns        | green or yellowgreen |
| Below score floor                             | yellow/orange        |
| Hard findings, hard caps, or blocked decision | red                  |
| Missing/stale/unverified score                | lightgrey            |

The current `publish` badge already uses named badge colors and maps them into SVG hex colors, so this is a natural extension rather than a new rendering system. 

## CI automation

Extend the generated GitHub workflow so adopted repos can choose badge write-back.

The current scaffold already generates a `.github/workflows/jankurai.yml` with audit, artifact upload, and `contents: read`.  For badge auto-updates, add a separate push-only job with `contents: write`; GitHub Actions supports defining `GITHUB_TOKEN` scopes such as `contents: read` or `contents: write` in the `permissions` key. ([GitHub Docs][4])

Example generated workflow section:

```yaml
name: jankurai

on:
  pull_request:
  push:
    branches: [main]

jobs:
  audit:
    runs-on: ubuntu-latest
    permissions:
      contents: read
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - name: Install jankurai
        run: cargo install jankurai --locked
      - name: jankurai audit
        run: |
          jankurai audit . \
            --mode advisory \
            --json agent/repo-score.json \
            --md agent/repo-score.md \
            --score-history agent/score-history.jsonl \
            --score-history-csv agent/score-history.csv

  badge:
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
      - name: Install jankurai
        run: cargo install jankurai --locked
      - name: Refresh Jankurai badge
        run: |
          jankurai audit . \
            --mode advisory \
            --json agent/repo-score.json \
            --md agent/repo-score.md \
            --score-history agent/score-history.jsonl \
            --score-history-csv agent/score-history.csv

          jankurai badge update . \
            --from agent/repo-score.json \
            --svg agent/jankurai-badge.svg \
            --json agent/jankurai-badge.json \
            --readme README.md \
            --link agent/repo-score.md \
            --yes
      - name: Commit badge update
        run: |
          if git diff --quiet -- README.md agent/repo-score.json agent/repo-score.md agent/score-history.jsonl agent/score-history.csv agent/jankurai-badge.svg agent/jankurai-badge.json; then
            exit 0
          fi

          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add README.md agent/repo-score.json agent/repo-score.md agent/score-history.jsonl agent/score-history.csv agent/jankurai-badge.svg agent/jankurai-badge.json
          git commit -m "chore(jankurai): refresh score badge [skip ci]"
          git push
```

For protected branches, use a PR mode instead:

```bash
jankurai ci install --github --mode advisory --badge readme --badge-write-mode pr
```

Suggested write modes:

```text
artifact  Upload badge as CI artifact only.
commit    Commit badge updates to main.
pr        Open/update a badge refresh PR.
check     Fail/warn when badge is stale, but do not write.
```

## Integration with `init` and `update`

For new adopters:

```bash
jankurai init . --level score --yes
```

should install:

```text
agent/repo-score.json
agent/repo-score.md
agent/jankurai-badge.svg
agent/jankurai-badge.json
README.md managed badge block
.github/workflows/jankurai.yml with badge mode if requested
```

For existing adopters:

```bash
jankurai update . --apply --yes
```

should add the badge block only if the repo has opted into the new badge surface. If you want this to become the default “badge of honor,” make it default for `--level score` and `--level full`, but not for `adopt --mode observe`.

This fits the current profile system: init profiles already declare generated paths, merge policies, CI templates, docs, and validation commands.  The current Rust/TS/Postgres profile generates `README-jankurai-scaffold.md`, Jankurai control-plane files, and the GitHub workflow, but not a managed badge in the real `README.md`; the badge should be added as a special README merge operation rather than a normal static template. 

## Shields.io option

I would keep Shields.io as an optional public-repo enhancement, not the default.

Shields endpoint badges expect a JSON endpoint with `schemaVersion: 1`, `label`, `message`, and `color`, and they also support custom `logoSvg`. ([shields.io][5]) Jankurai’s existing certification badge schema is richer and uses Jankurai-specific fields such as score, conformance levels, standard version, and blockers. 

So add a separate output:

```bash
jankurai badge update . \
  --shields-json docs/jankurai/shields.json
```

Do **not** replace the richer Jankurai badge JSON with Shields JSON. Use both:

```text
agent/jankurai-badge.json        Rich Jankurai metadata
docs/jankurai/shields.json       Optional Shields-compatible endpoint
agent/jankurai-badge.svg         Default static README image
```

## The product stance

Use this language in docs and README:

```text
Jankurai Score badge
Shows the latest tracked Jankurai audit score for this repository.

Jankurai Certified badge
Shows a publishable Phase 12 evidence bundle: audit, benchmark, governance, and attestation have passed.
```

That distinction matters. “Score” can become widespread quickly. “Certified” should be harder to earn.

My suggested rollout:

1. **v0.8:** Add `jankurai badge update/install/check`, static SVG, JSON schema, README managed block.
2. **v0.9:** Add `ci install --badge readme`, write-back modes, doctor drift checks.
3. **v1.0:** Make README score badge part of the default full/score adoption standard.
4. **Later:** Add public Shields/GitHub Pages badge endpoints and a gallery of repos wearing the badge.

The main design principle: **the badge should be generated evidence, not decoration.** It should always link to the score report or public evidence bundle that proves why the repo earned it.
