## The moonshot: Jankurai becomes the operating system for agent-native software

**Jankurai should not be “a linter.” It should become the open, versioned, machine-enforced constitution for building software with agents.**

The mission:

> **Jankurai makes every repository a proof machine.**
> Every file has an owner. Every boundary has a contract. Every generated artifact is declared. Every change is routed to the smallest sufficient proof lane. Every UI is rendered, measured, and regression-tested. Every dependency is scanned. Every exception expires. Every agent action is constrained, receipted, and auditable. Every merge must prove that it made the system better, not just bigger.

The destroyer line is this:

> **End vibe coding by making unprovable code unmergeable.**

That is the real opportunity here. The bottleneck is no longer “can an AI or human produce code?” The bottleneck is **can the system prove the code is correct, secure, maintainable, fast, cheap, understandable, and repairable?** The Jankurai paper already points in exactly this direction: it frames the repo as a verification interface and says the scarce act is a trustworthy merge, not the first draft. ([GitHub][1])

Jankurai should become the **default install/init of every serious software project**:

```bash
jankurai init
jankurai audit
jankurai proof
jankurai repair
jankurai ship
```

Not because it owns creativity. Because it kills ambiguity.

I inspected both the public repo and the uploaded source archive. The uploaded archive is source-only, so I read the paper/source materials rather than a rendered PDF. The current project already has the bones of something real:

**Jankurai v0.2.0 already defines itself as an open-source standard, paper, and audit CLI for agent-native engineering.** Its stated default target stack is Rust core, TypeScript/React/Vite, PostgreSQL, generated contracts, and bounded Python. ([GitHub][2])

It already has a **Vibe-Coding Risk Model** with explicit categories: security, business truth, contracts/data, verification, entropy, context/setup, and repair. That is exactly the right framing: the danger is not “bad style,” it is unverifiable entropy. ([GitHub][2])

It already requires the right machine-readable artifacts for higher conformance: `AGENTS.md`, `agent/JANKURAI_STANDARD.md`, `owner-map.json`, `test-map.json`, `generated-zones.toml`, `standard-version.toml`, `repo-score.json`, proof commands, and CI audit. ([GitHub][3])

It already has a strict weighted rubric around ownership, contracts, proof routing, security, code shape, data truth, observability, context economy, Python containment, and build speed. ([GitHub][1])

It already treats **rendered UX QA** as first-class: Storybook, Playwright, visual regression, accessibility, Lighthouse/Web Vitals, MSW, design tokens, and deterministic geometry checks. That is huge, because CEOs complain about “pixels” and layout because normal engineering systems still treat rendered UX as vibes instead of proof. ([GitHub][2])

The current included `repo-score.json` reports a score of **72**. That is honest and useful. The repo is already above chaos, but below its own intended target floor of 85. The weak areas I saw locally are exactly where the moonshot should focus next: real security execution instead of placeholders, stronger contract/data truth, observability/repair, build-speed evidence, semantic analysis, and a complete `init` product.

# The real vision: the Cold Standard

The ambition should be larger than “lint this repo.”

The ambition should be:

## **The Cold Standard for Agent-Native Software**

A global standard where every project is classified as one of three things:

1. **Jankurai-native** — built to the standard from day zero.
2. **Jankurai-contained** — legacy or alternate-stack code wrapped with contracts, proof lanes, generated boundaries, and migration plans.
3. **Jankurai-rejected** — unowned, unprovable, insecure, drift-prone, or agent-hostile code that cannot merge or deploy.

I would not frame this as “kill Java” or “kill all old software” literally. That makes the standard ideological and easy to dismiss. The winning move is colder:

> **No language, framework, or dependency gets moral immunity. Everything must justify itself against the rubric.**

The current docs already point toward a target-stack doctrine and a layer ownership model, including Rust, TypeScript/React/Vite, PostgreSQL, generated contracts, and bounded Python. ([GitHub][4]) The next step is to make that doctrine enforceable:

* New greenfield work defaults to the Jankurai stack.
* Legacy stacks enter through exception files.
* Exceptions require owners, expiry dates, migration paths, and proof lanes.
* Unsafe stacks are not debated endlessly; they are quarantined, wrapped, and retired.
* The standard wins by making the best path the fastest path.

The end state is not “humans stop mattering.” The end state is:

> Humans define intent, taste, ethics, constraints, product truth, and ambition.
> Jankurai destroys the waste between intent and trustworthy software.

# The doctrine: “perfect code” means provable code

“Perfect code” cannot just mean elegant code. It has to mean:

1. **Correct by construction where possible.**
2. **Contract-bound at every boundary.**
3. **Generated where repetition is dangerous.**
4. **Minimal but not cryptic.**
5. **Fast by default.**
6. **Memory-efficient by design.**
7. **Concurrency-aware.**
8. **Observable at runtime.**
9. **Repairable by agents.**
10. **Auditable by machines.**
11. **Understandable to future humans.**
12. **Hostile to drift.**

Jankurai should make this the merge law:

> **No proof, no merge. No owner, no merge. No contract, no boundary. No generated-zone declaration, no generated code. No security evidence, no deploy. No UI proof, no UI release. No exception without expiry.**

# The conformance ladder should become much more ambitious

The current standard already defines HL0 through HL5, from no standard to autonomous agent-safe. ([GitHub][3]) I would extend that into a moonshot ladder:

## HL0 — Chaos

No standard. Human memory required. Agent-hostile.

## HL1 — Documented

Some instructions exist, but they are mostly prose.

## HL2 — Auditable

Jankurai can score the repo and produce findings.

## HL3 — CI-enforced

Jankurai runs on every PR. Required maps exist. Score gates merge.

## HL4 — Agent-operable

Agents can navigate, test, repair, and avoid generated zones without human babysitting.

## HL5 — Autonomous-safe

Agents can make bounded changes with proof receipts, no broad permissions, and reliable repair loops.

## HL6 — Contract-complete

Every external and internal boundary has generated types, schema validation, compatibility checks, and negative tests.

## HL7 — UX-proofed

Rendered UI, accessibility, visual regression, layout geometry, focus behavior, responsive behavior, and performance budgets are merge-gated.

## HL8 — Runtime-verifiable

Observability, SLOs, error taxonomies, trace IDs, audit logs, incident runbooks, and repair signals are standardized.

## HL9 — Organization-native

Jankurai governs an entire engineering org: repo templates, dependency policies, reuse registry, exception inventory, component library, agent permissions, and compliance evidence.

## HL10 — Cold Standard

The repo is not merely “good.” It is **machine-governed software infrastructure**: optimized, bounded, observable, secure, reusable, and continuously upgraded as the standard evolves.

# Current repo: what I would fix immediately

This repo is a great start, but to become the Cold Standard it needs to get much stricter.

## 1. Replace security placeholders with real scanners

The current workflow and Justfile document security tools, but the lane is mostly placeholder/echo behavior. That is fine for v0.2.0, but not for the mission.

Add real execution for:

* CodeQL,
* Semgrep,
* gitleaks,
* Trivy,
* Syft,
* Grype,
* cargo-audit,
* cargo-deny,
* actionlint,
* zizmor,
* OpenSSF Scorecard.

Then feed those results into `repo-score.json`.

## 2. Add SARIF output

Jankurai should emit:

* JSON,
* Markdown,
* SARIF,
* GitHub annotations,
* JUnit where useful,
* signed conformance receipts.

SARIF makes Jankurai integrate with code scanning and security dashboards.

## 3. Add schema validation for every agent artifact

Validate:

* `owner-map.json`,
* `test-map.json`,
* `repo-score.json`,
* `generated-zones.toml`,
* `proof-lanes.toml`,
* `audit-policy.toml`,
* `standard-version.toml`.

Do not merely check that the file exists. Check structure, paths, commands, owners, expiry dates, and proof mapping.

## 4. Move from text heuristics to semantic analysis

Today much of the audit is text/pattern based. That is acceptable for an early standard, but the moonshot needs AST and graph analysis.

Add:

* Rust AST / `syn` layer,
* TypeScript AST via OXC or SWC,
* SQL parser,
* OpenAPI parser,
* Protobuf parser,
* TOML/JSON schema validation,
* dependency graph,
* call graph where possible,
* ownership graph,
* generated artifact graph.

## 5. Make `jankurai init` the killer feature

Right now the repo defines the standard. The world-changing version must generate the standard.

`jankurai init` should produce a full working project with:

* backend,
* frontend,
* contracts,
* database,
* UX QA,
* security CI,
* observability,
* agent instructions,
* generated maps,
* proof lanes,
* docs,
* runbooks,
* compliance evidence shell.

## 6. Create a public conformance registry

A public standard should expose:

* rule IDs,
* rule docs,
* severity,
* examples,
* autofix availability,
* standard version,
* deprecation policy,
* exception policy,
* migration recipes.

## 7. Make every finding repairable

Every finding should include:

* rule ID,
* exact location,
* owner,
* reason code,
* severity,
* why it matters,
* safest fix,
* proof command,
* docs link,
* whether an agent may fix it,
* whether human review is required.

The standard already defines agent-friendly error fields in spirit; this should become universal across all Jankurai findings. ([GitHub][3])

# The “no sprawl” rule

The full suite is only acceptable if Jankurai makes it feel simple.

The developer experience should be:

```bash
just fast
just proof
just full
jankurai audit
```

Everything else is routed.

No human should need to remember whether a UI change requires Playwright, Storybook, axe, Lighthouse, screenshots, ARIA snapshots, or design-token checks. Jankurai should infer that from changed paths and the proof map.

No CEO should need to ask whether the app is secure. Jankurai should produce:

```text
Security: pass
Supply chain: pass
Secrets: pass
Contracts: pass
UX QA: pass
Performance budget: pass
SOC evidence: ready
Exceptions: 2 active, 0 expired
```

# The rule architecture

Jankurai should evolve from “scan and score” to a full rule engine.

Each rule should have:

```text
id: HLT-SEC-006
name: no-unpinned-third-party-actions
severity: high
tlr_category: Security
applies_to:
  - .github/workflows/**
evidence:
  - workflow_ast
  - action_reference
fails_when:
  - uses_third_party_action_without_sha_pin
autofix:
  available: partial
  agent_allowed: true
proof:
  command: just security
  artifact: artifacts/security/workflow-policy.json
docs: docs/rules/HLT-SEC-006.md
```

This is how you make it transparent. This is how you make it global. No hidden taste. No “because the linter says so.” Every rule has a reason.

* agents cannot silently invent architecture,
* agents cannot edit generated code,
* agents cannot hand-wave tests,
* agents cannot hide behind TODOs,
* agents cannot expand scope without receipts,
* agents cannot merge unowned files,
* agents cannot ship unrendered UI,
* agents cannot create security drift,
* agents cannot turn product truth into scattered code.

Jankurai should not merely detect vibes.

It should make vibes structurally impossible.

---

# Final vision statement

**Jankurai is the Cold Standard for agent-native software engineering.**

It is the default initialization system, audit engine, proof router, repair planner, security harness, UX QA system, contract enforcer, performance budgeter, compliance evidence generator, and reuse registry for modern software.

It exists to end the era where every team reinvents the same broken stack, repeats the same security mistakes, writes the same fragile tests, hand-rolls the same contracts, ships the same layout bugs, forgets the same observability, and teaches every new agent a different messy repo.

Jankurai does not replace the human hero.

It removes the sludge around them.

It lets humans chase vision while the system enforces truth.

The project should aim for this:

> **Every serious codebase starts with Jankurai. Every agent reads Jankurai. Every merge proves itself through Jankurai. Every exception expires through Jankurai. Every reusable primitive improves through Jankurai. Every substandard pattern becomes either migrated, contained, or rejected.**

That is the moonshot.

Not a linter.

**The end of ungoverned software.**
