# The moonshot

## Jankurai becomes the **Agent-Native Software Operating Standard**

Not вҖңa linter.вҖқ

Not вҖңanother dev tool.вҖқ

Not вҖңa style guide.вҖқ

**Jankurai becomes the default control plane for building software with agents.**

Its mission:

> **End vibe coding by making every repository self-describing, self-auditing, self-testing, self-securing, self-optimizing, and agent-operable from the first commit.**

The project should become the thing every serious team installs before writing code:

```bash
jankurai init
jankurai prove
jankurai repair
jankurai release
```

The goal is not to win a debate about languages. The goal is to make the old mess economically irrational. Jankurai should make substandard software feel like shipping without tests, shipping without auth, shipping without observability, shipping without a compiler, and shipping without brakes.

The existing standard is already pointed in this direction. Its own mission says the new standard should make wrong code easy to **reject, localize, prove, audit, and repair**, and that agents need enforceable paths rather than human-comfort-oriented conventions. ([GitHub][2])

# The name of the vision: **The Cold Star Standard**

A вҖңcold starвҖқ is the fixed point every agent, developer, CEO, repo, CI system, and architecture decision orbits around.

It says:

> There is one highest-default way to build software: typed, bounded, observable, secure, measurable, contract-driven, rendered, tested, and repairable by agents.

This is how Jankurai вҖңkillsвҖқ bad software in practice: not by coercion, but by making every deviation visible, expensive, and temporary. Legacy stacks can exist, but they become migration targets. Exceptions can exist, but they expire. Alternative stacks can exist, but they must prove themselves against the same rubric.

The current standard already defines conformance levels **HL0 through HL5**, hard gates, stable rule IDs, ownership boundaries, generated zones, proof lanes, CI outputs, and finding schemas. That is the seed of a global standard. ([GitHub][3])

# Redefine вҖңperfect codeвҖқ

вҖңPerfect codeвҖқ cannot mean mystical code that never fails under unspecified requirements.

Jankurai should define perfect code as:

> **Code that is optimal against its declared specification, constraints, tests, threat model, performance budget, memory budget, UX contract, observability contract, and change policy.**

That makes perfection measurable.

A Jankurai-perfect repo has:

1. **A declared stack.**
2. **A declared architecture.**
3. **A declared owner map.**
4. **A declared contract map.**
5. **A declared proof map.**
6. **A declared UX proof map.**
7. **A declared security model.**
8. **A declared performance and memory budget.**
9. **A declared exception system.**
10. **A declared agent operating protocol.**

No hidden lore. No mystery folders. No вҖңask Dave.вҖқ No undocumented generated code. No random Python scripts under `tools/old2/final_final_really.py`. No UI that вҖңpasses testsвҖқ while visually breaking. No agent making broad edits because the repo did not tell it where the boundaries were.

The white paperвҖҷs core claim is already close to this: AI coding moves the bottleneck from вҖңtyping plausible codeвҖқ to вҖңgetting trustworthy changes merged,вҖқ and Jankurai acts as a control plane for that merge trust. ([GitHub][4])

# The non-negotiable stack doctrine

The repo already names the target profile:

> **Rust core + TypeScript/React/Vite + PostgreSQL + generated contracts + bounded Python.** ([GitHub][1])

I would make that doctrine even sharper:

## Default stack

| Layer                |                                            Default | Why                                                                                                                                                                                                                                                   |
| -------------------- | -------------------------------------------------: | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Core/domain logic    |                                           **Rust** | Memory safety, concurrency discipline, high performance, strong compilation boundary. RustвҖҷs ownership model is explicitly designed to catch memory and concurrency mistakes statically. ([Rust Documentation][5])                                    |
| Web UI               |                      **TypeScript + React + Vite** | Strict typing, componentized UI, fast dev loop, mature ecosystem. TypeScript strict mode strengthens correctness checks; Vite is built for fast frontend development; React is a component UI library. ([TypeScript][6])                              |
| Database             |                                     **PostgreSQL** | Constraints, foreign keys, transactional integrity, row-level security, mature query planning. PostgreSQL foreign keys and RLS policies are native tools for data integrity and access control. ([PostgreSQL][7])                                     |
| Contracts            | **Generated OpenAPI/Protobuf/JSON Schema clients** | No handwritten DTO drift. No вҖңsame shape in six files.вҖқ                                                                                                                                                                                               |
| AI/data escape hatch |                            **Bounded Python only** | Python allowed where it wins: ML, notebooks, data science, evaluation harnesses. It does not own product truth.                                                                                                                                       |
| Browser proof        |       **Playwright + Storybook + Jankurai UX QA** | Screenshots, traces, accessibility checks, deterministic geometry, visual baseline review. Playwright supports screenshots, visual comparisons, and trace debugging; Storybook supports visual testing and accessibility workflows. ([Playwright][8]) |

## But the doctrine must be empirical

Jankurai should not say вҖңRust because vibes.вҖқ It should say:

> Here is the rubric. Here is the score. Here is the measured tradeoff. Beat the default if you can.

The repoвҖҷs mission docs already define a 100-point stack rubric across correctness, security, performance, concurrency, boundaries, simplicity, ecosystem, observability, data integrity, and product velocity. ([GitHub][2])

That is the move: **make stack choice auditable**.

Java, PHP, Ruby, Go, Python, Kotlin, Scala, Swift, C#, or anything else should not be вҖңbanned by emotion.вҖқ They should be forced through the same scorecard. If they fail, they are migration targets. If they win in a domain, Jankurai can permit them through a versioned exception or domain pack.

## 2. `jankurai select`: stack and architecture selection

Before code exists, Jankurai should generate an architecture decision packet:

```bash
jankurai select --type b2b-saas --traffic 100k-rps --latency p95=80ms --compliance soc2 --ai-agent-heavy
```

Output:

```text
Recommended stack: rust-ts-vite-react-postgres-bounded-python
Architecture: modular monolith with explicit domain crates
Reason: lower agent confusion, lower memory risk, stronger generated contracts, simpler proof lanes
Rejected: Java/Spring, Python/FastAPI-only, Node-only
Migration risk: low
Proof lanes: fast, contract, db, web, e2e, security, perf, release
```

## 3. `jankurai prove`: evidence-driven development

Agents should not say, вҖңI think I fixed it.вҖқ

They should say:

```json
{
  "claim": "checkout tax calculation fixed",
  "changed_paths": ["crates/domain/src/tax/*.rs"],
  "proof_lanes": ["fast", "contract", "property", "db"],
  "evidence": [
    "target/jankurai/evidence/unit.json",
    "target/jankurai/evidence/property.json",
    "target/jankurai/evidence/contract.json"
  ],
  "residual_risk": "none declared"
}
```

The repo already has the seed: proof lanes like fast, contracts, security, DB, UI, full, and audit are part of the mission docs. ([GitHub][2])

## 4. `jankurai qa`: deterministic rendered product proof

This is the answer to the CEO asking whether AI can replace human QA testers for pixels and layout.

The answer:

The repoвҖҷs UX QA package is already excellent directionally: it combines Playwright screenshots, ARIA snapshots, viewport artifacts, deterministic geometry checks, visual baseline review, and review-only AI/VLM opinions unless backed by deterministic evidence. ([GitHub][1])

* route screenshots,

* responsive breakpoints,
* ARIA snapshots,
* keyboard navigation,
* focus visibility,
* tap target size,
* text clipping,
* horizontal overflow,
* sticky obstruction,

* button wrapping,
* form labels,
* contrast,
* layout shift,
* Lighthouse/Web Vitals,
* visual baselines,
* trace artifacts,
* design token drift,
* screenshot-to-issue localization.

## 5. `jankurai repair`: agent-ready fixes, not vague warnings

A finding should not be:

```text
Bad code found.
```

It should be:

```json
{
  "rule_id": "HLT-014",
  "severity": "high",
  "owner": "domain",
  "path": "crates/domain/src/billing/tax.rs",
  "problem": "duplicate tax calculation exists in two ownership zones",
  "why_it_matters": "agents may update one path and leave business truth divergent",
  "safe_fix": "move calculation into domain service and regenerate API contract",
  "proof_required": ["unit", "property", "contract"],
  "agent_prompt": "Refactor duplicate tax calculation into crates/domain/src/billing/tax_policy.rs without changing public API. Run just fast and just contract."
}
```

The standard already says findings should include severity, category, path, problem, agent fix, evidence, and version bindings. ([GitHub][2])

## 6. `jankurai migrate`: destroy legacy by making migration mechanical

This is the biggest commercial weapon.

Jankurai should be able to inspect a legacy app and produce:

* stack score,
* architecture map,
* domain extraction plan,
* DTO drift map,
* duplicate logic map,
* security gap map,
* test gap map,
* migration slices,
* risk-ranked rewrite plan,
* agent prompts,
* proof lanes for each slice.

Example:

```bash
jankurai migrate analyze ./legacy-java-service
jankurai migrate plan --target rust-ts-vite-react-postgres
jankurai migrate slice billing-tax
jankurai migrate prove billing-tax
```

The vision is not вҖңrewrite everything blindly.вҖқ The vision is **measured strangler migration**:

1. Discover.
2. Score.
3. Extract contracts.
4. Generate test harness.
5. Port one domain slice.
6. Prove equivalence.
7. Cut over.
8. Retire old code.

Every standard dies if it becomes stale.

* UX rules,

A repo should not just pass once. It should stay alive.

# The Cold Star rubric

The current repo has two related score systems: mission-level stack scoring and repo audit scoring. I would unify them into one public rubric:

## Jankurai Cold Star Score: 100 points

| Dimension               | Weight | Meaning                                                                                  |
| ----------------------- | -----: | ---------------------------------------------------------------------------------------- |
| Correctness proof       |     12 | Unit, integration, property, contract, DB, and release proof.                            |
| Security                |     12 | SAST, secrets, vuln scans, dependency review, supply chain, threat model.                |
| Agent operability       |     12 | Owner maps, generated zones, agent docs, repair receipts, no contradictory instructions. |
| Contract integrity      |     10 | Generated clients, schemas, API drift detection, no handwritten DTO divergence.          |
| Architecture boundaries |     10 | Domain/application/adapters separation, dependency direction, no вҖңutils soup.вҖқ           |
| Performance             |      8 | Benchmarks, latency budgets, bundle budgets, query budgets.                              |
| Memory efficiency       |      7 | Heap budgets, allocation checks, unsafe ledger, leak checks.                             |
| Concurrency safety      |      7 | Thread/task discipline, race tests, async boundaries, lock budgets.                      |
| UX proof                |      7 | Rendered browser evidence, accessibility, visual baselines, responsive states.           |
| Data integrity          |      6 | Migrations, constraints, RLS, backup/restore, seed validation.                           |
| Observability           |      5 | Traces, metrics, logs, correlation IDs, error catalog.                                   |
| Simplicity/minimality   |      4 | LOC caps, duplication control, dead-code control, no overengineering.                    |

Hard caps should remain brutal. A repo with no security lane, no root agent instructions, no generated-zone discipline, disabled tests, contract drift, silent fallbacks, direct DB misuse, or destructive migrations should be capped no matter how good the rest looks. The standard already defines hard gates around these kinds of failures. ([GitHub][3])

No drift. No prompt fork. No tool-specific mythology.

# The вҖңend vibe codingвҖқ rule

A Jankurai repo should make vibe coding structurally impossible.

Blocked patterns should include:

* duplicate business logic,
* handwritten DTOs when generated contracts exist,
* silent catch-all errors,
* fallback soup,
* disabled tests,
* fake green tests,
* giant files,
* giant functions,
* direct DB access from the wrong layer,
* generated file mutation,
* magic globals,
* TODO-driven design,
* unclear ownership,
* ambiguous prompts,
* unbounded Python,
* visual QA absence,
* security lane absence,
* dependency drift,
* undocumented exceptions,
* contradictory agent instructions.

The mission docs already define vibe coding as plausible code that lacks ownership, proof, or repairability, and they list violations like duplicate logic, handwritten DTOs, silent fallbacks, mega files/functions, direct DB access, TODO-driven design, inconsistent naming, and conflicting instructions. ([GitHub][2])

# Roadmap

## Phase 0 вҖ” Current seed

Already present:

* Rust audit CLI.
* Agent standard.
* Owner map.
* Test map.
* Generated zones.
* Repo scoring.
* Paper.
* CI pattern.
* UX QA package.
* Playwright-based rendered artifacts.
* HL conformance concept.
* Rule IDs and hard gates.

The current repo map confirms the major pieces: `crates/jankurai`, `packages/ux-qa`, `agent`, `docs`, `paper`, `tips`, and `reference`. ([GitHub][1])

## Phase 1 вҖ” Make it installable as the default repo spine

Target:

```bash
cargo install jankurai
jankurai init
```

Deliver:

* `jankurai init`.
* Template generator.
* CI generator.
* Agent files generator.
* Proof lanes generator.
* Security lane generator.
* UX QA setup.
* SARIF output.
* GitHub Action.
* Standard lockfile.
* Exception registry.
* Version ratchet.

Success metric:

> A founder can run one command and get an agent-native repo with CI, QA, security, contracts, docs, and proof lanes.

## Phase 2 вҖ” Replace regex-heavy audit with semantic audit

The current scanner is a powerful start, but the next version should add AST and graph intelligence.

Deliver:

* Rust AST rules.
* TypeScript AST rules.
* SQL migration rules.
* Python containment rules.
* Dependency graph rules.
* Contract drift rules.
* Generated-code mutation rules.
* Semantic duplicate detection.
* Layer-boundary graph.
* Changed-path proof selection.

Success metric:

> Jankurai can tell not just that something looks wrong, but which architectural invariant it violates.

## Phase 3 вҖ” Build the reference product platform

Deliver:

```bash
jankurai init fullstack-saas
```

Creates:

* Rust API.
* Rust domain crates.
* PostgreSQL migrations.
* TypeScript React/Vite frontend.
* Generated API client.
* Storybook.
* Playwright.
* UX QA.
* Auth cell.
* Org/team cell.
* Audit-log cell.
* Observability.
* Security lanes.
* Release evidence.

Success metric:

> A serious SaaS product can start with Jankurai and not immediately rebuild the same boring infrastructure.

## Phase 4 вҖ” Certified reusable cells

Deliver a registry:

```bash
jankurai cell search
jankurai cell add auth
jankurai cell add billing
jankurai cell add notifications
jankurai cell prove
```

Each cell has:

* contracts,
* migrations,
* tests,
* UX states,
* security policy,
* performance budgets,
* docs,
* repair recipes.

Success metric:

> Common software stops being rebuilt badly.

## Phase 5 вҖ” Migration engine

Deliver:

```bash
jankurai migrate analyze
jankurai migrate plan
jankurai migrate slice
jankurai migrate prove
```

Support migration from:

* Java/Spring,
* Node/Express,
* Python/FastAPI,
* Rails,
* PHP/Laravel,
* legacy monoliths,
* random internal tools.

Success metric:

> Legacy stacks become measurable migration queues instead of permanent prisons.

## Phase 6 вҖ” HL5 certification

Deliver:

* public conformance badges,
* benchmark corpus,
* certified templates,
* certified cells,
* rule-pack governance,
* reproducible audits,
* signed evidence,
* release attestations,
* third-party review path.

Success metric:

> вҖңHL5вҖқ becomes a meaningful signal: this repo is agent-operable, secure by default, proof-driven, and migration-safe.

## Phase 7 вҖ” Autonomous repair and optimization

Deliver:

```bash
jankurai repair --auto-pr
jankurai optimize --budget latency:p95=80ms
jankurai reduce --tokens
jankurai refactor --proof-preserving
```

Capabilities:

* agent-generated PRs,
* proof-preserving refactors,
* duplication collapse,
* contract regeneration,
* dead code removal,
* performance tuning,
* memory optimization,
* UI regression repair,
* dependency cleanup,
* exception expiration.

Success metric:

> The repo improves itself faster than humans can manually maintain it.

# The killer feature: token minimization

Agent coding burns tokens because repos are ambiguous.

Jankurai should cut token waste by generating:

* `repo.intent.json`,
* `symbol-map.json`,
* `owner-map.json`,
* `test-map.json`,
* `contract-map.json`,
* `generated-map.json`,
* `route-map.json`,
* `error-map.json`,
* `proof-map.json`,
* `dependency-map.json`,
* `change-impact.json`.

Then an agent does not need to read the whole repo.

For any task:

```bash
jankurai context "fix checkout tax bug"
```

Output:

```json
{
  "read_first": [
    "crates/domain/src/billing/tax_policy.rs",
    "contracts/schemas/billing.schema.json",
    "tests/property/billing_tax.rs"
  ],
  "do_not_edit": [
    "contracts/generated/**",
    "apps/web/src/generated/**"
  ],
  "owner": "billing-domain",
  "proof_lanes": ["fast", "property", "contract"],
  "likely_rules": ["HLT-004", "HLT-009", "HLT-014"],
  "max_context_files": 7
}
```

This is how you stop agents from wandering.

# The governance model

Jankurai should become a living standard, not one personвҖҷs preference.

Create:

* Jankurai Foundation or standards group.
* Public rule registry.
* Public benchmark corpus.
* Versioned rule packs.
* Public conformance levels.
* Public certified templates.
* Public certified cells.
* Public migration reports.
* Public вҖңbad patternвҖқ museum.
* Public reproducibility harness.
* RFC process.
* Security advisory process.
* Deprecation calendar.

The standard must be opinionated enough to be useful and empirical enough to stay honest.

# The final vision statement

> **Jankurai is the Cold Star Standard for agent-native software: a universal, machine-verifiable control plane that turns every repo into a self-describing, self-testing, self-securing, self-optimizing system. It ends vibe coding by replacing guesswork with ownership, contracts, proof lanes, rendered UX evidence, security gates, performance budgets, and agent-readable repair receipts. It gives humans the thing they actually want: the ability to describe what should exist, while the machine enforces how it is built, proven, secured, optimized, and evolved.**

The highest ambition is not вҖңbetter linting.вҖқ

The highest ambition is:

> **No more software chaos by default. No more agent drift by default. No more rebuilding the same broken foundations by default. No more unprovable code by default.**

Jankurai should become the first thing installed in every serious repo because it makes the alternative feel insane.
