# 1. What jankurai already is

The current repo is already unusually strong because it is not trying to be generic. It has an opinionated target stack: **Rust core, TypeScript/React/Vite product surface, PostgreSQL durable truth, generated contracts, and bounded Python for AI/data work**. The README explicitly positions the project as an open-source standard, paper, and audit CLI for agent-native engineering, focused on helping coding agents reject wrong code, localize repair, run the right proof lane, and leave evidence. ([GitHub][2])

The audit rubric is also correctly strict. It asks whether an agent can reject, localize, prove, audit, and repair without turning shortcuts into production behavior. The current required shape separates web, API edge, domain logic, application commands, adapters, database constraints, bounded Python, and ops responsibilities. That is the right instinct: agents need hard architectural boundaries, not vibes. ([GitHub][3])

The paper’s rule set is already going in the right direction: ownership maps, proof lanes, generated zones, security gates, version binding, rendered UX QA, repair receipts, and machine-readable artifacts. ([GitHub][1])

The rendered UX QA package is especially important. It already defines deterministic browser rules such as edge clearance, target size, interactive overlap, text clipping, button wrapping, horizontal overflow, sticky obstruction, focus visibility, form labels, and nested scrollbars. That is exactly the kind of “AI can replace some QA pixels/layout checks” system a CEO is asking about. ([GitHub][4])

The release plan is also correctly thinking beyond a repo: paper, standard, auditor, agent-ready artifacts, CI integrations, templates, benchmark suite, badges, version channels, adoption phases, and repair loops. ([GitHub][5])

So the foundation is real.

But the moonshot is bigger.

# 4. The ideal stack

Jankurai should be brutally opinionated by default.

Not because there are no other good tools. Because agents need low-entropy environments. Human teams need fewer branches of confusion. Optimization needs global concentration.

## Default “COLD Stack”

### Runtime core

**Rust** for domain logic, services, background workers, critical APIs, systems code, parsers, data integrity, performance-critical paths, and anything that must be safe, fast, memory-efficient, and auditable.

### Product surface

**TypeScript + React + Vite** for web UI, with strict typing, generated clients, design tokens, deterministic layout QA, accessibility checks, and no business truth in the UI.

### Durable truth

**PostgreSQL** for durable state, constraints, transactions, idempotency, audit trails, and relational truth.

### Contracts

**TypeSpec / OpenAPI / Protobuf / JSON Schema**, generated into Rust and TypeScript clients. TypeSpec is especially aligned with this because it is designed to eliminate manually maintained API description files and emit standards-compliant schemas such as OpenAPI, JSON Schema, and Protobuf. ([TypeSpec][6])

### RPC and API generation

**Buf + ConnectRPC** where protobuf/RPC is appropriate. Buf provides standardized Protobuf generation, and ConnectRPC’s docs show generated web clients through Buf plugins and runtime packages. ([Buf][7])

### REST client generation

**Orval** where OpenAPI/REST is appropriate, because it generates type-safe clients, mocks, validators, React Query/SWR hooks, and MSW handlers from API specifications. ([Orval][8])

### Bounded Python

**Python only where it is best**: AI experimentation, notebooks, data pipelines, model wrappers, evaluation harnesses, and scientific/data tooling. No scattered product truth. No direct production database logic unless explicitly granted by policy.

### CI and proof

**GitHub Actions initially**, but Jankurai’s abstraction should be CI-portable. The standard should describe proof lanes; GitHub Actions is just the first implementation.

### Observability

**OpenTelemetry by default**. Traces, metrics, logs, structured errors, and repair receipts should be first-class, not post-launch glue.

### Security and supply chain

**Default security evidence**, not “run a scanner sometimes.” CodeQL, Semgrep, Gitleaks, dependency review, cargo-deny, Syft, Grype, Trivy, Scorecard, SLSA provenance, and Cosign should become orchestrated proof lanes, not disconnected tools. CodeQL produces code scanning alerts for vulnerabilities and code errors; GitHub dependency review can fail pull requests when dependency changes introduce vulnerable versions; Gitleaks detects secrets in repos and files; Syft generates SBOMs and Grype scans them; SLSA defines supply-chain integrity levels and provenance concepts; Cosign supports keyless signing through OIDC. ([GitHub Docs][9])

# 5. The thing to “kill”

The thing to kill is not people, not learning, not exploration, and not healthy engineering judgment.

The thing to kill is **unproven entropy**.

Jankurai should make the following impossible by default:

* Handwritten API clients when contracts exist
* Business truth in React components
* Raw SQL in handlers
* Secrets in repo
* Untested generated code
* Unowned files
* Missing proof lanes
* “TODO later” production paths
* Silent fallbacks
* Magic retries
* False-green tests
* Unbounded agent authority
* Prompt-injection-sensitive agent instructions
* Generated files edited by hand
* Python product truth leakage
* UX regressions with no screenshots, geometry, or accessibility proof
* CI that passes without proving the affected surface
* Legacy stack introduction without an exception and migration plan
* Security tools that emit noise without machine-readable ownership and remediation

That is the real revolution.

Not “ban Java” as a slogan. Instead:

**Make non-standard stacks fail the default audit unless they carry an explicit, time-boxed, evidence-backed exception with a migration path.**

That gives Jankurai legitimacy. It is strict without becoming unserious.

# 7. The Jankurai product surface

Jankurai should become a suite.

## Core commands

```bash
jankurai init
```

Creates the repo constitution: stack, boundaries, maps, proof lanes, workflows, generated zones, docs, and agent adapters.

```bash
jankurai audit
```

Scores the repo against the current standard and emits JSON, Markdown, SARIF, PR comments, and repair queues.

```bash
jankurai lane
```

Given a diff, returns the minimal sufficient proof lane: fast, contracts, DB, security, UI, UX geometry, full, release.

```bash
jankurai prove
```

Runs the selected proof lanes locally or in CI and emits evidence receipts.

```bash
jankurai repair-plan
```

Converts findings into agent-sized repair tasks with owners, commands, expected artifacts, and stop conditions.

```bash
jankurai migrate
```

Generates a migration plan from legacy architecture into the COLD stack.

```bash
jankurai explain
```

Produces executive-readable status: what is safe, what is risky, what is blocked, what must be fixed, what can ship.

```bash
jankurai context-pack
```

Produces a token-minimized packet for agents: only the files, rules, owners, tests, contracts, and examples needed for the current task.

```bash
jankurai attest
```

Signs or records evidence: SBOMs, provenance, test receipts, UX artifacts, security scans, migration approvals.

```bash
jankurai doctor
```

Validates local developer/agent environment: tool versions, lockfiles, database availability, browser install, cache, CI parity.

```bash
jankurai registry
```

Installs canonical reusable building blocks: auth, RBAC, CRUD, billing, webhooks, queues, observability, admin tables, audit logs, file upload, search, and more.

# 8. The proof model

Jankurai should make proof cheaper, not heavier.

The repo should never run everything blindly. It should route each change to the smallest proof lane that actually covers risk.

## Proof lanes

| Lane            | Purpose                                                                  |
| --------------- | ------------------------------------------------------------------------ |
| `setup`         | Validate toolchain, versions, lockfiles, generated artifacts             |
| `fast`          | Format, lint, typecheck, unit tests, changed-path checks                 |
| `contracts`     | TypeSpec/OpenAPI/Protobuf compatibility, generated clients, schema drift |
| `domain`        | Rust domain unit/property tests                                          |
| `application`   | command handlers, authz, idempotency, workflow tests                     |
| `db`            | migrations, constraints, rollback safety, query checks                   |
| `api`           | API integration, contract tests, auth boundaries                         |
| `web`           | TypeScript, React, component tests, Storybook stories                    |
| `ux`            | Playwright geometry, screenshots, accessibility, visual diffs            |
| `security`      | secrets, SAST, SCA, SBOM, provenance, CI hardening                       |
| `perf`          | benchmarks, memory, concurrency, bundle size, latency budgets            |
| `observability` | traces, logs, metrics, error taxonomy, repair receipts                   |
| `release`       | full evidence pack, signatures, changelog, deployment readiness          |
| `full`          | complete proof for high-risk changes                                     |

The current release plan already describes a staged CI adoption model: observe, advisory, guardrails, standard gate, repair loop, and release contract. It also names minimum lanes like audit, fast, contracts, security, DB, UI, and full. That should become the central Jankurai execution model. ([GitHub][5])

# 9. The anti-sprawl rule

The tooling list below is large, but the installed experience must not feel large.

Jankurai should enforce a rule:

**A tool is allowed only if it emits unique, machine-readable evidence mapped to a Jankurai rule, lane, owner, and repair action.**

No random tools. No dashboards for dashboards’ sake. No “enterprise theater.”

Every tool must satisfy:

1. It catches a real class of defects.
2. It has a deterministic or clearly advisory decision mode.
3. It runs locally and in CI.
4. It emits parseable output.
5. It maps to a Jankurai rule ID.
6. It has an owner and repair path.
7. It has a removal condition if superseded.

That is how this becomes massive without becoming bloated.

# 12. The Jankurai standard levels

Jankurai needs a ladder.

## Level 0 — Uncontrolled

Vibe-coded repo. No reliable ownership, proof, or boundaries.

## Level 1 — Observable

Jankurai audit installed. Findings visible. No blocking yet.

## Level 2 — Routed

Changed files map to proof lanes. Agents know what to run.

## Level 3 — Guarded

Generated zones, security scans, and critical architectural boundaries block.

## Level 4 — Agent-native

Agents can safely repair most findings using owner maps, test maps, proof lanes, and receipts.

## Level 5 — COLD compliant

Repo follows default stack or has approved exceptions, all critical proof lanes active, evidence is machine-readable, and release receipts are complete.

## Level 6 — Autonomous repair

Agents generate PRs for most defects, dependency updates, UX regressions, and policy drift under constrained authority.

## Level 7 — Self-optimizing

Repo continuously benchmarks build speed, runtime performance, context size, agent success rate, duplicate code, dependency risk, and proof cost — then proposes simplifications.

That is not evidence of a code failure; it means the next hardening step should include a reproducible dev environment and/or vendored/cache-friendly CI setup.

## Phase 1 — `jankurai init`

Turn Jankurai from an auditor into an installer.

```bash
jankurai init --profile web-api-db
```

Generates:

* `AGENTS.md`
* `jankurai.toml`
* proof lanes
* owner map
* test map
* generated-zone map
* GitHub workflows
* Justfile
* Rust workspace
* Vite app
* PostgreSQL migrations
* TypeSpec/OpenAPI/Buf setup
* Playwright/Storybook/UX QA
* security lanes
* docs skeleton
* runbooks
* CI policy
* exception policy

This is the moment Jankurai becomes a default project starter.

## Phase 2 — Agent adapters

Ship adapters for every major coding-agent environment:

* Codex
* Claude Code
* Cursor
* GitHub Copilot
* Aider
* Continue
* OpenHands-style agents
* CI repair bots

Each adapter should be generated from the same standard, not hand-maintained.

## Phase 3 — Repair queues

The current repo already has the right concept: findings should become repairable tasks.

Make it real:

```bash
jankurai repair-plan --from repo-score.json
```

Output:

* owner
* failing rule
* expected fix
* files likely involved
* commands to run
* proof lane
* stop conditions
* example patch shape
* risk level

Then:

```bash
jankurai repair --agent codex --max-risk medium
```

The agent repairs only within allowed authority.

## Phase 4 — Reuse registry

Create canonical building blocks for the things humans rebuild endlessly.

Start with:

1. Auth/session
2. RBAC
3. CRUD table/form
4. Audit log
5. Webhook receiver
6. Background job
7. File upload
8. Billing/subscription
9. Notification/email
10. Admin dashboard
11. Search
12. Feature flags
13. API key management
14. Data export/delete
15. Observability pack

Every block must be:

* Contract-first
* Rust-first for truth
* Generated into UI
* Tested
* UX-audited
* Security-scanned
* Observable
* Agent-readable

## Phase 5 — Benchmark the end of vibe coding

Jankurai should publish a benchmark suite.

Measure:

* Agent success rate
* Wrong-file edit rate
* Token usage per accepted PR
* Time to trustworthy merge
* CI false-green rate
* Security findings per KLOC
* Generated-zone violations
* Duplicate primitive rebuilds
* Runtime performance
* Memory usage
* Build time
* UX regressions caught before review
* Migration time from legacy stacks

The current release plan already points toward public examples, repair-agent readiness, templates, compliance badges, rule packs, and benchmark packs. Push that much harder. ([GitHub][5])

## Phase 6 — Organization control plane

For companies:

* Org-wide stack policy
* Approved exceptions
* Migration plans
* Repo scores
* Security posture
* Agent efficiency metrics
* Reuse opportunities
* Technical debt heatmaps
* Compliance evidence
* Executive status

The CEO view:

> “Here are the product surfaces. Here is what is safe. Here is what is blocked. Here is the cost of non-standard code. Here are the reusable modules you should not rebuild. Here is the fastest path to ship.”

## Phase 7 — Global standard

This is the giant move.

Jankurai becomes:

* A paper
* A standard
* A CLI
* A GitHub Action
* A package suite
* A starter template
* A benchmark
* A compliance badge
* A registry
* A migration engine
* An agent protocol
* A proof ledger
* A reuse economy

At that point, Jankurai is no longer a repo.

It is the default grammar of agent-native software.

# 16. The immediate “level-up this repo” punch list

If I were driving this as the next leap, I would do these first:

1. **Ship `jankurai init` before adding twenty more audit rules.**
   The project becomes huge when it creates the standard repo, not just scores one.

2. **Turn security from placeholder to real proof lane.**
   The current direction is right; now make the lane actually run scanners, produce receipts, and fail correctly.

3. **Add SARIF and PR comments.**
   Findings must show up where developers and agents work.

4. **Create the canonical “perfect repo” example.**
   One example should show the full COLD stack: Rust, TS/React/Vite, PostgreSQL, generated contracts, UX QA, security, observability, repair receipts.

5. **Create the “bad vibe repo” example.**
   Show chaos. Then show Jankurai repair plan. Make the before/after emotionally obvious.

6. **Build the reusable primitive registry.**
   Start with auth, CRUD, audit logs, jobs, webhooks, and admin tables.

7. **Make UX QA a first-class gate.**
   Add screenshot baselines, cross-browser profiles, ARIA snapshots, accessibility, geometry, and artifact-backed failure receipts.

8. **Add AST-aware analysis.**
   String scanning is a good start, but Jankurai should graduate to tree-sitter, Rust AST where useful, TypeScript AST, import graphs, SQL schema graphs, and call-boundary checks.

9. **Add agent benchmark metrics.**
   This is how you prove “end vibe coding”: lower token use, fewer wrong edits, faster accepted PRs, fewer regressions.

10. **Define exception economics.**
    A non-standard stack can exist, but it must carry a visible cost: exception owner, expiration date, migration plan, proof burden, and executive visibility.

**Jankurai is the COLD standard for agent-native engineering: a global, versioned, machine-readable software constitution that makes the best stack, safest architecture, fastest proof lanes, strictest security posture, cleanest UX checks, and most reusable product primitives the default starting point for every repo. It ends vibe coding by replacing human memory and agent guesswork with executable standards, generated contracts, proof receipts, repair queues, and ruthless context economy. Humans define vision. Agents build inside constraints. Jankurai proves the result.**

That is the project.

Not a linter.
