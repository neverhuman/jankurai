# Mission statement

**Jankurai exists to make software development agent-native by default.**

Every repository should begin with a verified stack, machine-readable architecture, explicit ownership boundaries, generated contracts, proof-routed CI, deterministic UX checks, security and supply-chain evidence, observability hooks, token-budget discipline, and repair instructions that agents can execute safely.

Humans should spend their attention on **what must exist**.

Agents and automated systems should handle **how it is implemented, validated, secured, optimized, documented, and kept from drifting into entropy**.

The final state is not “AI helps developers code faster.”

The final state is:

# The cold standard

You called it the “COLD standard.” I would make that literal:

## COLD = Constrained, Observable, Localized, Deterministic

**Constrained:**
Agents cannot freely spray code across the repo. Every file has ownership, allowed dependencies, test lanes, generated-zone rules, security expectations, and exception policies.

**Observable:**
Every important behavior emits traces, metrics, logs, screenshots, accessibility evidence, contract evidence, performance evidence, and supply-chain receipts. OpenTelemetry is the right default backbone here because it is vendor-neutral and designed around traces, metrics, and logs. ([OpenTelemetry][2])

**Localized:**
Every change should have a narrow blast radius. Ownership maps, dependency direction, module boundaries, and contract maps tell agents where a change belongs and where it does not.

**Deterministic:**
No vague “looks good.” No “probably secure.” No “tests passed” when the wrong tests ran. The repo should produce machine-readable proof: SARIF, JSON reports, JUnit, screenshots, traces, SBOMs, provenance, coverage, mutation results, benchmark deltas, and audit receipts.

That is the standard.

Not “pretty code.”

**Proven code.**

# What `jankurai` already is

The current project already contains the skeleton of this system:

1. **A standard and paper**, not only source code. The repo includes an agent-native engineering standard, a white paper, audit docs, testing docs, and policy files.

2. **A Rust audit CLI** that scores repositories and emits JSON/Markdown reports. The README describes a score interpretation where below 70 is not agent-operable, 70–84 is advisory, and 85+ is the target floor. ([GitHub][1])

3. **Machine-readable agent artifacts**, including `owner-map.json`, `test-map.json`, `generated-zones.toml`, `proof-lanes.toml`, `audit-policy.toml`, and `standard-version.toml`. The standard also describes required adoption files and conformance levels. ([GitHub][1])

4. **A specific preferred stack**, currently centered on Rust, TypeScript/React/Vite, PostgreSQL, generated contracts, and bounded Python. ([GitHub][1])

5. **A deterministic UX QA package** for Playwright that checks things humans often judge visually: target size, edge clearance, overlap, clipping, button wrapping, horizontal overflow, sticky obstruction, focus visibility, form labels, z-index ceilings, nested scrollbars, and screenshots/artifacts. ([GitHub][1])

6. **A repo-shape philosophy**: AGENTS instructions, application boundaries, generated zones, documentation, contracts, scripts, and test routing. The standard already describes an ideal repo layout, ownership boundaries, dependency direction, generated zones, and test lanes. ([GitHub][3])

That is a serious start.

But the current system is still mostly an **audit layer**.

The moonshot is to make it a **software creation layer**.

# The north star

## `jankurai` should make every repo answer seven questions instantly

1. **What is this system?**
   Machine-readable product purpose, architecture, domain map, boundaries, public contracts, and runtime topology.

2. **Where may an agent change code?**
   Ownership maps, generated zones, locked zones, approval zones, dependency direction, and risk-based edit permissions.

3. **How is correctness proven?**
   Test lanes, property tests, contract tests, integration tests, snapshot tests, visual checks, security scans, performance checks, and migration checks.

4. **How is safety proven?**
   Secrets scanning, SAST, dependency audit, SBOM, provenance, container scanning, IaC scanning, least-privilege checks, prompt-injection checks, and compliance evidence.

5. **How is UX proven?**
   DOM geometry, screenshots, accessibility, visual regression, layout stability, keyboard navigation, responsive viewport matrix, and design-token compliance.

6. **How is performance proven?**
   Benchmarks, flamegraphs, memory ceilings, concurrency tests, load tests, bundle budgets, query plans, and runtime telemetry.

7. **How does the system repair itself?**
   Agent-readable findings, fix queues, minimal patches, test rerouting, exception governance, regression prevention, and migration plans.

That is the end of vibe coding.

# The ideal leading stack

The current target stack is already good: Rust core, TypeScript/React/Vite, PostgreSQL, generated contracts, bounded Python. I would sharpen it into a reference stack, not because no other stack can ever exist, but because agents need fewer degrees of freedom.

## Default reference stack

| Layer                 | Standard choice                                   | Why                                                                                                                               |
| --------------------- | ------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| Core backend          | **Rust**                                          | Memory safety, performance, strong typing, good concurrency, strong agent-legibility once patterns are standardized.              |
| API framework         | **Axum + Tower**                                  | Composable, typed, async Rust web stack.                                                                                          |
| Database              | **PostgreSQL**                                    | Durable, proven, expressive, strong relational constraints, good operational ecosystem.                                           |
| DB access             | **SQLx**                                          | Compile-time query checking, explicit SQL, avoids ORM magic.                                                                      |
| Frontend              | **TypeScript + React + Vite**                     | Broad ecosystem, typed UI, fast dev loop.                                                                                         |
| UI runtime validation | **Zod / Valibot**                                 | Runtime boundary validation for generated contracts.                                                                              |
| Testing               | **Vitest + Playwright + jankurai UX QA**         | Unit, integration, browser, and deterministic layout evidence.                                                                    |
| Contracts             | **OpenAPI / JSON Schema / Protobuf where needed** | Generated clients, boundary truth, drift detection.                                                                               |
| Observability         | **OpenTelemetry + tracing**                       | Vendor-neutral telemetry spine.                                                                                                   |
| AI/Python             | **Bounded Python only**                           | Use Python where it dominates: ML, data science, model evals, notebooks, certain AI workloads. Keep product truth outside Python. |
| Dev task runner       | **Justfile**                                      | Simple, readable, agent-friendly command surface.                                                                                 |
| Package policy        | **Lockfiles required**                            | Deterministic builds.                                                                                                             |
| CI                    | **GitHub Actions first, portable spec second**    | Common default, but the standard should not be trapped in one CI vendor.                                                          |

## Stack doctrine

The standard should not say “never use anything else.”

It should say:

> **The approved stack is the default. Deviations require a machine-readable exception with reason, owner, expiration, risk, and migration plan.**

The existing standard already has an exception concept, including structured fields like purpose, risk, owner, expiry, and related tests. That should become central. ([GitHub][3])

## Phase 3 — Proof router

Goal: every change automatically maps to the right proof.

A changed file should produce an answer like:

```text
Changed:
  crates/billing/src/invoice.rs
  contracts/openapi/billing.yaml
  apps/web/src/routes/billing/InvoicePage.tsx

Required proof:
  cargo test -p billing
  cargo nextest run -p billing
  cargo sqlx prepare --check
  openapi-diff previous current
  pnpm test InvoicePage
  playwright billing.spec.ts
  jankurai ux-qa --route /billing/invoices
  semgrep billing rules
  gitleaks
  cargo audit
  cargo deny
```

No more wrong tests.

No more false green.

No more “the agent changed the UI but only ran backend unit tests.”

The existing standard already describes test lanes and coverage expectations; this should become executable routing. ([GitHub][3])

# The rubric: Agent-Native Engineering Index

The current repo already has a weighted technical-liability model. I would evolve it into a public score called the **Agent-Native Engineering Index**, or ANEI.

## Proposed score

| Category                                        |  Weight |
| ----------------------------------------------- | ------: |
| Ownership, boundaries, and architecture clarity |      15 |
| Proof routing and test integrity                |      15 |
| Security and supply-chain integrity             |      15 |
| Contract and data truth                         |      12 |
| UX and accessibility evidence                   |      10 |
| Performance, memory, and concurrency discipline |      10 |
| Agent context economy                           |       8 |
| Observability and repairability                 |       8 |
| Stack alignment and migration posture           |       5 |
| Reuse and anti-duplication                      |       2 |
| **Total**                                       | **100** |

## Hard gates

Some problems should cap the score no matter how many points the repo earns elsewhere:

```text
No AGENTS.md                              cap 75

No one-command local validation           cap 70

```

The standard already defines conformance concepts and hard gates; the next step is making them richer, executable, and publicly comparable. ([GitHub][3])

# The anti-vibe-coding doctrine

Vibe coding fails because agents are given too much ambiguity.

The fix is not “better prompts.”

The fix is **less freedom in the repo**.

Every agent task should get a generated brief:

```text
Task:
  Add invoice export.

Allowed files:
  crates/billing/**
  apps/web/src/routes/billing/**
  contracts/billing/**
  tests/billing/**

Forbidden files:
  crates/auth/**
  db/migrations/** unless migration requested
  generated/** unless generated by command

Required commands:
  cargo nextest run -p billing
  pnpm test billing
  pnpm playwright test billing
  jankurai ux-qa --route /billing
  jankurai audit --changed

Expected artifacts:
  contract diff
  screenshot
  UX report
  test report
  security scan receipt
```

This is what kills vibe coding:

```text
bounded context
bounded edit authority
bounded proof requirements
bounded exceptions
bounded generated code
bounded language choice
bounded architecture drift
```

# The most important command set

```bash
jankurai init
jankurai doctor
jankurai audit
jankurai audit --changed
jankurai prove
jankurai route-tests
jankurai ux-qa
jankurai security
jankurai stack-score
jankurai explain
jankurai repair
jankurai migrate
jankurai brief
jankurai evidence
jankurai exceptions
jankurai registry update
```

## What each command should do

| Command           | Purpose                                                              |
| ----------------- | -------------------------------------------------------------------- |
| `init`            | Create the standard repo skeleton.                                   |
| `doctor`          | Verify local machine, dependencies, versions, and required tools.    |
| `audit`           | Score the repo and detect drift.                                     |
| `audit --changed` | Score only change impact for PR use.                                 |
| `prove`           | Run the correct proof lanes for the current change.                  |
| `route-tests`     | Explain exactly which tests/checks are required and why.             |
| `ux-qa`           | Run deterministic UI geometry, accessibility, and screenshot checks. |
| `security`        | Run the security/supply-chain lane and produce receipts.             |
| `stack-score`     | Score the stack against the current approved standard.               |
| `explain`         | Explain a file, rule, finding, or architecture boundary.             |
| `repair`          | Generate an agent-ready fix plan or patch.                           |
| `migrate`         | Plan or execute migration from legacy stack to target stack.         |
| `brief`           | Produce a compressed agent task brief.                               |
| `evidence`        | Validate all evidence artifacts.                                     |
| `exceptions`      | List expired, risky, or unjustified exceptions.                      |
| `registry update` | Update rulepacks and tool recommendations.                           |

# Concrete next implementation steps

If I were turning this repo into the monster version, I would do this first:

1. **Split the Rust CLI into core crates** so rules, output formats, schemas, and commands can evolve cleanly.
2. **Add SARIF output** immediately.
3. **Replace placeholder security lane** with real `gitleaks`, `osv-scanner`, `syft`, `grype`, `cargo audit`, `cargo deny`, `zizmor`, and `actionlint`.
4. **Add schema validation** for every file under `agent/`.
5. **Add `jankurai init`** with one profile: `rust-ts-postgres`.
6. **Add `jankurai doctor`** to check local tools.
7. **Add `jankurai route-tests --changed`** as the first killer agent feature.
8. **Add UX evidence output standard** for screenshots, ARIA snapshots, traces, and geometry.
9. **Add benchmark fixtures** for vibe-coding failure modes.
10. **Make the repo score 85+ under its own rules**, or document why tool repos use a separate rubric.

I also attempted to run the local Rust tests from the uploaded source archive, but the sandbox could not resolve `index.crates.io`, so dependency fetch failed. `cargo fmt --all -- --check` did succeed locally.
