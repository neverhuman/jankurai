I reviewed the current Jankurai repo, the uploaded AFCF paper source, and the nine idea files in the tarball. My conclusion: **yes, there are several things worth building into Jankurai now**, but they should be framed as **developer-useful control-plane + generator primitives**, not as “the future model-training system” yet.

The winning move is this: **make Jankurai the tool that turns ordinary repos into canonical, proof-carrying, cell-shaped repos today.** That gives coders value now and creates the substrate for the future translated training set.

## The short answer

Build **canonical behavior cells** now.

Not tiny utility functions. Not a bigger linter. Not a pure style guide. Build a first-class Jankurai layer that lets teams say:

> “This repo contains a standard resource lifecycle, standard auth boundary, standard generated contract, standard migration lane, standard external adapter, and standard proof lane.”

That aligns with what Jankurai already is: a local audit/control-plane CLI for AI-assisted merges, with proof lanes, ownership maps, generated zones, security boundaries, repair queues, and local evidence receipts. The README describes the core loop as “find vibe artifacts, prove the merge, repair the repo,” and says it checks many rule families while producing JSON/Markdown reports, score history, proof artifacts, and repair queues. ([GitHub][1])

## What to build into Jankurai now

### 1. `jankurai cell`: canonical behavior-cell registry

This is the biggest immediate item.

Add a registry and CLI for standard reusable cells:

```text
jankurai cell list
jankurai cell add resource --name invoice
jankurai cell add policy --name tenant_access
jankurai cell add adapter --kind stripe
jankurai cell verify
jankurai cell evidence
```

A cell should be a **behavior unit with generated code, tests, ownership, proof lanes, and receipts**, not just a template. Start with these cells:

| Cell                   | Why it matters today                                                |
| ---------------------- | ------------------------------------------------------------------- |
| `ResourceLifecycle<T>` | CRUD/list/search/archive/audit is everywhere.                       |
| `GeneratedContract`    | Schema → Rust types → TS clients → fixtures → docs → tests.         |
| `PolicyGuard`          | Authz, tenant boundaries, roles, permissions.                       |
| `MigrationPlan`        | Forward, rollback, invariant checks, seed/replay.                   |
| `ExternalAdapter`      | Stripe, GitHub, Slack, S3, email, LLM providers behind typed ports. |
| `ErrorEnvelope`        | Stable reason codes, common fixes, correlation IDs.                 |
| `EventOutbox`          | Transactional events, webhooks, retries, idempotency.               |
| `JobQueue`             | Background work, dedupe, retry, dead-letter, cancellation.          |
| `UiResourceView`       | List/detail/form/table/loading/error/empty states.                  |
| `ProofLane`            | Build/test/security/migration/e2e/release evidence.                 |
| `BehaviorDossier`      | Provenance, accepted exceptions, tests, diffs, confidence score.    |

This is the bridge between “useful today” and “training set later.” Developers get scaffolds and guardrails now. The future canonical corpus gets named, typed, behavior-preserving units.

Jankurai already has the right surfaces for this: adoption/drift commands, guard/write enforcement, kickoff/context-pack, proof/evidence, audit/routing, repair/expiry, and public evidence/registry concepts are already part of the README’s “Control-Plane Surfaces.” ([GitHub][1])

### 2. `jankurai contract`: contract-first source of truth

This should be the second major build.

The tarball repeatedly converges on the same point: **contracts/schemas are the biggest practical collapse after behavior cells.** Jankurai should make this boring and enforceable:

```text
jankurai contract init
jankurai contract generate
jankurai contract diff
jankurai contract verify
jankurai contract fixtures
```

It should generate:

```text
contracts/source/*
contracts/generated/openapi/*
apps/web/src/generated/*
apps/api/src/generated/*
crates/*/src/generated/*
tests/fixtures/generated/*
```

And it should enforce:

```text
Change source contracts, not generated outputs.
Public API changes must regenerate clients.
Generated clients must be stale-checked in CI.
Contract changes must route to contract proof lanes.
```

This is already conceptually present in the agent-native standard: the standard defines generated zones, requires generated identity/stamps, tells agents to change source contracts instead of generated outputs, and says CI must fail when generated output is stale. ([GitHub][2])

So the product opportunity is not inventing the rule. It is making the rule delightful and automatic.

### 3. `jankurai profile canonical-product-v1`

The repo already has a strong Rust/TypeScript/React/Vite/PostgreSQL posture, but it needs to become a **named profile** with generators, verification, and migration paths.

I would define:

```text
canonical-product-v1
  Rust backend/core/tooling
  TypeScript product/client surface
  React/Vite browser shell
  PostgreSQL/SQL durable truth
  schema-first contracts
  strict generated zones
  Docker/IaC deploy envelope
  proof lanes
  behavior dossiers
```

This matches the existing standard, which declares a target stack of Rust core, TypeScript/React/Vite product surface, PostgreSQL truth, generated contracts, and exception-only Python AI/data service. ([GitHub][2])

But do **not** make this sound like “everything must become Rust/React.” Make it the **primary product profile**, with future secondary profiles for kernels, compilers, databases, browsers, GPU, embedded, native mobile, and HPC.

### 4. `jankurai migrate --to canonical-product-v1`

This is where adoption becomes real.

Today, Jankurai adoption is staged: scan without writes → adoption plan → dry-run control plane → observe CI → accepted baseline → ratchet. ([GitHub][3]) That is good. But for the AFCF thesis, Jankurai also needs a **migration slicer**:

```text
jankurai migrate . --to canonical-product-v1 --slices
```

Output:

```text
target/jankurai/migration/slices.md
target/jankurai/migration/equivalence-plan.json
target/jankurai/migration/cell-map.json
target/jankurai/migration/generated-boundary-plan.md
```

It should not rewrite everything automatically. It should identify:

| Existing repo smell          | Canonical target                    |
| ---------------------------- | ----------------------------------- |
| Handwritten DTO/client forks | `GeneratedContract`                 |
| Scattered SQL/ORM calls      | `PostgresAdapter` + `MigrationPlan` |
| Repeated CRUD flows          | `ResourceLifecycle<T>`              |
| Middleware auth sprawl       | `PolicyGuard`                       |
| Copy-pasted vendor calls     | `ExternalAdapter`                   |
| Ad hoc errors                | `ErrorEnvelope`                     |
| Worker/retry spaghetti       | `JobQueue` + `EventOutbox`          |
| UI table/form repetition     | `UiResourceView`                    |

This would immediately help developers while also generating messy→canonical mapping data for future training.

### 5. `jankurai primitive catalog`: dependency and adapter governance

The uploaded tips consistently identify dependency drift as a huge entropy source. Build a catalog that says:

```text
For email, use this adapter shape.
For object storage, use this adapter shape.
For OAuth/OIDC, use this policy cell.
For payments/webhooks, use this external adapter pattern.
For queues, use this job/outbox cell.
```

This should not require Jankurai to own every library. It should own the **interface shape, proof obligations, failure fixtures, and upgrade lane**.

Jankurai already tracks tool adoption and evidence-producing tools, including security, UX QA, DB migration analysis, contract drift, copy-code, web-security checks, and repo-rot checks. ([GitHub][1]) A primitive catalog would turn that from “audit what exists” into “guide what should exist.”

### 6. `jankurai behavior-dossier`: corpus-ready evidence records

For the future training set, every canonicalized artifact needs a durable record:

```json
{
  "behavior_id": "...",
  "source_provenance": "...",
  "canonical_profile": "canonical-product-v1",
  "cells": ["ResourceLifecycle", "PolicyGuard", "GeneratedContract"],
  "accepted_differences": [],
  "proof_lanes": ["fast", "contract", "db", "security"],
  "generated_zones": [],
  "fixtures": [],
  "negative_tests": [],
  "confidence": 0.92
}
```

But this is also useful today: it gives teams a clean audit trail for why a change is safe.

This fits Jankurai’s existing philosophy: it treats agent behavior as repository policy, not chat convention, and records evidence through receipts, score history, repair queues, proof lanes, owner maps, generated-zone manifests, and security controls. ([GitHub][1])

### 7. `jankurai cell bench`: prove that cells actually help agents

This is essential. The AFCF thesis must not remain philosophical.

Add a benchmark harness:

```text
jankurai cell bench --task add-field-to-resource
jankurai cell bench --task add-tenant-policy
jankurai cell bench --task migrate-column
jankurai cell bench --task add-webhook
```

Measure:

| Metric                       | Why                     |
| ---------------------------- | ----------------------- |
| raw vs canonical token count | source compression      |
| path search count            | repo navigation entropy |
| invalid edit attempts        | action-space reduction  |
| proof-lane pass rate         | reliability             |
| time to accepted change      | developer value         |
| changed files touched        | blast-radius control    |
| generated drift incidents    | boundary correctness    |
| cost per correct change      | the real thesis metric  |

This should be built now because it answers the only question that matters: **does Jankurai make correct changes cheaper?**

## What belongs later, after the standard is stronger

These are important, but I would not lead with them.

| Later item                                     | Why not now                                                                                                                                         |
| ---------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| Full messy→canonical corpus translation        | Too expensive before cells, contracts, profiles, and behavior dossiers are stable.                                                                  |
| Training a canonical foundational coding model | Needs measured canonical corpora first.                                                                                                             |
| 90% smaller model claims                       | Treat as hypothesis until paired raw/canonical evals exist.                                                                                         |
| Universal software IR                          | Valuable, but premature unless built from real cells and migration traces.                                                                          |
| Massive behavior-equivalence mining            | Needs a stable `behavior_id` / dossier format first.                                                                                                |
| Cross-language automatic porting at scale      | Dangerous without proof lanes, contract generation, and equivalence fixtures.                                                                       |
| Public certification/badge registry            | Jankurai’s README lists public evidence registry/badge policy as a known gap; build it after the cell/profile evidence model is real. ([GitHub][1]) |
| Secondary profiles for kernel/GPU/mobile/HPC   | Important, but should follow product-profile success.                                                                                               |
| Canonical MoE/model-routing architecture       | Downstream of corpus evidence, not a Jankurai adoption hook.                                                                                        |

## The priority order I would actually ship

### Phase 1: make Jankurai useful to normal repos

Ship these first:

1. `jankurai cell`
2. `jankurai contract`
3. `canonical-product-v1` profile
4. `jankurai migrate --slices`
5. `jankurai cell verify`
6. `jankurai cell bench`

This is the minimum viable “gold standard now.”

### Phase 2: make adoption addictive

Add:

1. `jankurai cell add resource`
2. `jankurai cell add policy`
3. `jankurai cell add adapter`
4. `jankurai cell add ui-resource`
5. generated docs + tests + fixtures
6. PR comments showing “this change preserved cell boundaries”

This gives developers a reason to use Jankurai beyond compliance.

### Phase 3: make the future dataset inevitable

Add:

1. `behavior-dossier.json`
2. `messy-to-canonical.diff`
3. `cell-map.json`
4. `equivalence-fixtures/`
5. `accepted-differences.md`
6. `canonical-training-record.jsonl`

At that point, Jankurai becomes the pipeline that produces canonical training examples.

## The main thing I would not build

I would **not** start with a giant “canonical code formatter” or “rename all code into the standard” tool.

That gives the illusion of progress but misses the thesis. The big win is not names. It is:

> same behavior, fewer valid implementations, fewer editable paths, fewer generated-file mistakes, fewer proof rituals, fewer duplicated contracts, fewer invalid agent repairs.

The agent-native standard already says repos should optimize for agents to reject wrong code quickly, localize failures, repair narrow scopes, and leave auditable evidence. ([GitHub][2]) The next layer should make that operational through reusable behavior cells.

## My final recommendation

Put this into Jankurai now:

```text
Jankurai Canonical Cells v0
```

With these first ten cells:

1. `GeneratedContract`
2. `ResourceLifecycle`
3. `PolicyGuard`
4. `MigrationPlan`
5. `ExternalAdapter`
6. `ErrorEnvelope`
7. `EventOutbox`
8. `JobQueue`
9. `UiResourceView`
10. `ProofLane`

That is the best near-term bridge between:

* value for coders today,
* stricter agent performance now,
* measurable Jankurai adoption,
* future messy→canonical translation,
* and eventually a canonical-code training corpus.

The slogan I’d use internally:

> **Don’t standardize code first. Standardize behavior cells, generated boundaries, and proof receipts. Code follows.**
