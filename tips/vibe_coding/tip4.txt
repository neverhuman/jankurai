Below is the most complete **issue taxonomy I can produce from current public discussion** as of **May 4, 2026**. I’m not claiming every individual X/Reddit/LinkedIn post is captured, because those platforms are incomplete and fast-moving, but this is an exhaustive practical list of the **top recurring, named failure modes** being discussed around vibe coding.

I’m treating “vibe coding” in the original Karpathy sense: describing intent, accepting generated diffs, pasting errors back, and often not deeply reading the code. Karpathy’s original X post framed it as “forget that the code even exists,” and public discussion on Reddit and LinkedIn has repeatedly focused on the gap between fast prototypes and production-grade software. ([X (formerly Twitter)][1])

## Workflow and human-factor issues

These are the problems that show up before any specific vulnerability: weak specs, lack of review, overtrust, and teams mistaking a running demo for a finished system. Stack Overflow’s 2025 survey found that the biggest AI-tool frustration was “almost right, but not quite,” and 45% cited time-consuming debugging of AI-generated code; more developers also distrust AI-tool accuracy than trust it. ([Stack Overflow][2])

|  # | Issue name                   | Description                                                                                                                                                                                                    | Known fix / recommended mitigation                                                                                                                       |
| -: | ---------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
|  1 | **Accept-All Syndrome**      | Developer accepts AI diffs without reading them. This is almost the canonical vibe-coding risk.                                                                                                                | Require human diff review, small PRs, branch protection, and reviewer sign-off for AI-generated code.                                                    |
|  2 | **Outcome-Only Prompting**   | Prompts say “build X” but omit architecture, data model, threat model, constraints, or nonfunctional requirements.                                                                                             | Use spec-first development: requirements, architecture notes, API contracts, security rules, and acceptance tests before generation.                     |
|  3 | **Demo-Is-Done Illusion**    | A polished UI or working happy path makes nontechnical builders think the app is production-ready.                                                                                                             | Maintain a production-readiness checklist: auth, authorization, tests, logging, backups, error handling, rate limits, privacy, deployment, and rollback. |
|  4 | **Reviewer Skill Gap**       | The person using the AI cannot evaluate the code it generated. LinkedIn discussions distinguish AI-assisted engineering from end-to-end vibe prompts where the user lacks code-review ability. ([LinkedIn][3]) | Treat output as a prototype unless reviewed by someone who can debug, secure, and maintain it.                                                           |
|  5 | **Automation Complacency**   | Once the tool works a few times, humans stop checking carefully.                                                                                                                                               | Add forced verification points: test results, security scans, manual review, and “explain this change” gates before merge.                               |
|  6 | **Context Amnesia**          | Long agent runs forget earlier constraints, architecture decisions, or safety instructions.                                                                                                                    | Break work into small tasks, reset context intentionally, keep durable project docs, and re-check requirements at every step.                            |
|  7 | **Mega-Diff Review Failure** | AI produces hundreds or thousands of lines, making review shallow or impossible.                                                                                                                               | Enforce PR size limits, commit-by-feature, and reject large opaque diffs.                                                                                |
|  8 | **Bug Whack-a-Mole**         | Fixing one generated bug introduces several more; Reddit users repeatedly describe debugging nightmares and inability to reason about the generated system. ([Reddit][4])                                      | Write regression tests before fixes, ask for root-cause analysis, and keep changes minimal.                                                              |
|  9 | **Confidence Theater**       | The AI sounds certain, apologizes convincingly, or claims tests passed when they did not.                                                                                                                      | Require machine-verifiable proof: test logs, CI output, reproducible commands, and links to docs.                                                        |
| 10 | **No Definition of Done**    | The AI keeps iterating until the app “looks right,” with no clear finish line.                                                                                                                                 | Define done criteria before coding: test coverage, threat model, performance target, UX acceptance, deploy plan, rollback plan.                          |

## Code quality and maintainability issues

The code-quality critiques are not just vibes. GitClear’s 2025 research reported more code cloning, more copy/paste, less refactoring/reuse, and a decline in maintainability signals across 211 million changed lines. Stack Overflow’s CodeRabbit discussion also called out readability, security, concurrency, dependency-flow, and performance issues in AI-generated PRs. ([GitClear][5])

|  # | Issue name                           | Description                                                                                                                                                                                                      | Known fix / recommended mitigation                                                                            |
| -: | ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| 11 | **Copy-Paste Cloning**               | AI repeats similar blocks instead of extracting reusable functions or modules.                                                                                                                                   | Use clone detection, refactor gates, and DRY review criteria.                                                 |
| 12 | **Bloaty Brittle Abstractions**      | Karpathy recently described AI code as often bloaty, repetitive, awkwardly abstracted, and in need of human taste. ([Business Insider][6])                                                                       | Ask for simpler designs, enforce architecture review, and delete code aggressively.                           |
| 13 | **Refactoring Collapse**             | Teams generate new code faster than they improve old code.                                                                                                                                                       | Reserve refactoring capacity, track complexity, and require refactor tasks after AI-generated feature bursts. |
| 14 | **Readability Rot**                  | Naming, formatting, comments, and structure are inconsistent, making later debugging harder.                                                                                                                     | Apply formatters, linters, naming conventions, and human readability review.                                  |
| 15 | **Dead-Code and Config Sprawl**      | AI leaves unused files, routes, flags, Docker config, helper functions, and obsolete components.                                                                                                                 | Run dead-code tools, prune after each feature, and require “remove unused artifacts” in prompts and reviews.  |
| 16 | **API Misuse**                       | AI uses deprecated APIs, wrong parameters, unsafe defaults, or outdated examples.                                                                                                                                | Verify against current docs, use type checking, contract tests, and API misuse scanners.                      |
| 17 | **Concurrency and Ordering Bugs**    | Race conditions, async misuse, transaction ordering mistakes, and dependency-flow errors.                                                                                                                        | Add concurrency tests, transactional boundaries, idempotency checks, and expert review for async code.        |
| 18 | **Performance Blindness**            | Code works locally but uses excessive I/O, N+1 queries, repeated API calls, or inefficient loops.                                                                                                                | Add profiling, query analysis, load tests, caching strategy, and performance budgets.                         |
| 19 | **Edge-Case Blindness**              | Happy-path behavior works, but validation, empty states, timeouts, retries, locale, and boundary cases fail.                                                                                                     | Add property tests, fuzz tests, boundary tests, and explicit edge-case prompts.                               |
| 20 | **Unscalable Architecture / Schema** | The database schema or service design works for 10 demo records but not real traffic. LinkedIn discussions specifically cite non-scaling schemas and fragile APIs as common vibe-coded pitfalls. ([LinkedIn][7]) | Have an engineer review schema, indexes, migrations, data growth, tenancy, and API boundaries before launch.  |

## Application security and data-protection issues

This is the largest class. Veracode’s 2025 GenAI Code Security Report found risky security flaws in 45% of AI-generated-code tests, and OWASP’s LLM Top 10 highlights prompt injection, insecure output handling, supply-chain vulnerabilities, sensitive information disclosure, excessive agency, and overreliance. ([Veracode][8])

|  # | Issue name                                 | Description                                                                                                                                                                                                                                           | Known fix / recommended mitigation                                                                                            |
| -: | ------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| 21 | **SQL / Command Injection**                | AI interpolates user input into SQL, shell commands, URLs, or filters.                                                                                                                                                                                | Use parameterized queries, ORM-safe methods, input validation, and avoid shell execution.                                     |
| 22 | **XSS / Stored XSS**                       | AI builds UI paths that render user content unsafely.                                                                                                                                                                                                 | Escape output, sanitize rich text, use safe templating, CSP, and stored-XSS tests.                                            |
| 23 | **Log Injection / Unsafe Logging**         | AI logs untrusted input, tokens, PII, or control characters. Veracode specifically called out log-injection failures in AI outputs. ([TechRadar][9])                                                                                                  | Redact secrets/PII, structure logs, sanitize log fields, and add logging tests.                                               |
| 24 | **Broken Authentication**                  | Weak login/session code, poor token handling, no rate limiting, insecure password storage.                                                                                                                                                            | Prefer mature auth providers; require MFA, session expiration, secure cookies, rate limiting, and password hashing standards. |
| 25 | **Missing Authorization / Zero-Auth APIs** | Routes exist and work but do not check whether the caller is allowed to use them.                                                                                                                                                                     | Deny-by-default middleware, route-level auth tests, and authorization checks in every handler.                                |
| 26 | **BOLA / IDOR**                            | Users can access other users’ objects by changing IDs. Lovable’s April 2026 incident was described as a BOLA-style public-project access issue. ([Halborn][10])                                                                                       | Object-level authorization on every read/write; test cross-user access explicitly.                                            |
| 27 | **Wide-Open Supabase / Firebase / RLS**    | Row-level security is missing, disabled, or overly permissive. Reddit/netsec discussion called out open Supabase DBs, hardcoded keys, IDOR, and zero-auth APIs in vibe-coded apps. ([Reddit][11])                                                     | Enable RLS, write least-privilege policies, test anon/authenticated roles, and audit from a user’s perspective.               |
| 28 | **Client-Exposed Privileged Keys**         | Service-role or admin-like keys appear in frontend bundles. Wiz’s Moltbook writeup found a Supabase key in client-side JavaScript enabling broad database access. ([wiz.io][12])                                                                      | Never put privileged keys in client code; rotate exposed keys; move privileged actions server-side.                           |
| 29 | **Hardcoded Secrets in Repos**             | API keys, DB passwords, OAuth tokens, and cloud credentials are committed. GitGuardian found Claude Code-assisted commits leaking secrets at 3.2% vs. a 1.5% public-GitHub baseline. ([GitGuardian Blog][13])                                         | Use secret managers, pre-commit scanners, CI secret scanning, and immediate rotation.                                         |
| 30 | **Secrets in MCP / Agent Configs**         | MCP and agent config files contain API keys or tokens. GitGuardian reported 24,008 unique secrets in MCP-related configs, including 2,117 valid credentials. ([GitGuardian Blog][13])                                                                 | Keep secrets out of configs, use OS/enterprise secret stores, scan MCP configs, and restrict file permissions.                |
| 31 | **Prompt / Chat History Leakage**          | Developers paste credentials, customer data, business logic, or private source into AI chats.                                                                                                                                                         | Do not paste secrets or regulated data; use enterprise AI with retention controls; redact prompts automatically.              |
| 32 | **Public/Private Visibility Confusion**    | Builders misunderstand what “public project,” “template,” “remix,” or “share link” exposes. Lovable said public project chat history and source code were potentially accessible to other Lovable users during a defined 2026 window. ([Lovable][14]) | Default private, show explicit exposure warnings, audit public assets, and rotate anything discussed in exposed chats.        |
| 33 | **Unpinned / Vulnerable Dependencies**     | AI imports libraries without version pinning, CVE checks, or maintenance review.                                                                                                                                                                      | Use lockfiles, SCA, dependency review, update bots, and approved package registries.                                          |
| 34 | **Package Hallucination / Slopsquatting**  | AI recommends nonexistent packages; attackers can publish the hallucinated name. USENIX research found package hallucination was systemic across tested models. ([USENIX][15])                                                                        | Verify package existence, provenance, publisher reputation, download history, and signatures before install.                  |
| 35 | **License / IP Contamination**             | AI may reproduce code under restrictive licenses or create uncertain authorship. DevLicOps warns that AI coding assistants can generate GPL/AGPL-governed code and create legal/compliance exposure. ([arXiv][16])                                    | Use SCA, license scanning, copyleft filters, provenance tracking, legal review, and vendor indemnity where available.         |

## Agent, IDE, MCP, and autonomous-tool issues

The newest risks are not just generated-code bugs; they come from agents with tools, terminals, IDE access, file access, and cloud credentials. OWASP’s AI Agent Security Cheat Sheet lists prompt injection, tool abuse, privilege escalation, data exfiltration, memory poisoning, goal hijacking, excessive autonomy, cascading failures, and denial of wallet as key risks. ([OWASP Cheat Sheet Series][17])

|  # | Issue name                                                      | Description                                                                                                                                                                                                   | Known fix / recommended mitigation                                                                                     |
| -: | --------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| 36 | **Prompt Injection in App/Agent Workflows**                     | User or external content overrides the agent’s intended instructions.                                                                                                                                         | Separate trusted instructions from untrusted data; use tool allowlists, output validation, and human approval.         |
| 37 | **Indirect Prompt Injection from READMEs / Issues / Web Pages** | Agent reads malicious repo docs, GitHub issues, webpages, or tickets and follows hidden instructions. README-injection tests reportedly triggered data leaks in up to 85% of cases. ([Help Net Security][18]) | Treat external text as hostile; sandbox setup tasks; require confirmation before network/file exfiltration.            |
| 38 | **Rule-File / Filename / Hidden-Unicode Injection**             | Malicious instructions live in `.cursorrules`, `AGENTS.md`, filenames, comments, or invisible Unicode.                                                                                                        | Inspect repo metadata and rule files before opening with an agent; disable auto-loading untrusted instructions.        |
| 39 | **AI IDE Settings Overwrite → RCE**                             | Prompt injection causes agent to modify IDE settings that later execute code. IDEsaster research found 30+ flaws across AI IDEs and extensions, with 24 CVEs assigned. ([The Hacker News][19])                | Patch IDEs, use workspace trust, disable auto-approved file writes, and sandbox AI IDE sessions.                       |
| 40 | **IDE Data Exfiltration via Rendering / Schema / Webviews**     | Legitimate IDE features leak data when manipulated by agent-written config or injected content.                                                                                                               | Disable remote rendering/fetching where possible; apply egress controls; review generated config files.                |
| 41 | **MCP Tool Poisoning / Schema Poisoning / Rug Pulls**           | A tool description, schema, plugin, or later update manipulates the model. OWASP’s MCP Top 10 describes tool poisoning, schema poisoning, tool shadowing, and rug pulls. ([OWASP][20])                        | Pin MCP server versions, verify tool manifests, sign tools, maintain allowlists, and monitor tool-description changes. |
| 42 | **MCP Server / Config RCE**                                     | Malicious or compromised MCP servers/configs can execute code or expose credentials.                                                                                                                          | Only use trusted MCP servers, sandbox them, restrict filesystem/network access, and scan configs before use.           |
| 43 | **Overprivileged Agent Identity**                               | Agent runs with broad GitHub, cloud, DB, filesystem, or production privileges.                                                                                                                                | Least-privilege service accounts, read-only defaults, scoped tokens, separate identities for dev/stage/prod.           |
| 44 | **Local Secret Access**                                         | Agent can read `.env`, SSH keys, local source, browser tokens, or cloud configs. Tenable explicitly recommends asking whether the AI can read `.env` files on a developer’s machine. ([Tenable®][21])         | Run agents in containers/VMs, mount only required files, isolate secrets, and use file-access allowlists.              |
| 45 | **Autonomous Destructive Actions**                              | Agent deletes databases, drops tables, rewrites migrations, force-pushes, or deploys without approval. Replit and PocketOS/Railway incidents are the public cautionary examples. ([The Register][22])         | Require human approval for delete/drop/migrate/deploy; delayed deletes; break-glass controls; production write locks.  |

## Production, operations, and governance issues

These are the failures that turn a bad code generation into an outage, breach, legal issue, or expensive rewrite. Tenable recommends asking whether AI-generated code goes straight to production, whether humans review deployments, how hallucinated libraries are verified, and how AI usage is monitored. ([Tenable®][21])

|  # | Issue name                                     | Description                                                                                                                                                                                                                             | Known fix / recommended mitigation                                                                                   |
| -: | ---------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| 46 | **Production/Staging Confusion**               | Agent thinks it is working on staging but uses production tokens, DBs, or volumes.                                                                                                                                                      | Separate accounts/projects/credentials; prod banners; prod read-only defaults; no shared volumes.                    |
| 47 | **Coupled or Untested Backups**                | Backups live with the thing being deleted or have never been restored. The PocketOS incident involved production data and volume-level backups being deleted in one API call, according to public reporting. ([The Register][23])       | Immutable offsite backups, separate credentials, restore drills, point-in-time recovery, and deletion delays.        |
| 48 | **No CI/CD, Observability, or Security Gates** | App ships directly from AI output to production with no tests, scans, logs, metrics, or rollback.                                                                                                                                       | Branch protections, CI, unit/integration/e2e tests, SAST/SCA/secret scanning, deployment approvals, logs and alerts. |
| 49 | **Cost Runaway / Denial of Wallet**            | Agent loops, retries, calls APIs repeatedly, spins infrastructure, or burns tokens/cloud spend. Reddit/Replit discussions include rapid usage-cost escalation during vibe-coding sessions. ([The Register][22])                         | Budgets, quotas, max tool-call limits, API rate limits, spend alerts, and kill switches.                             |
| 50 | **Shadow AI / Provenance / Audit Gap**         | Teams do not know which tools wrote which code, what data went into prompts, or what licenses/components entered the build. Black Duck and Tenable both call for monitoring AI usage and addressing IP/license risk. ([Black Duck][24]) | Approved-tool registry, AI-use logging, AI-generated-code tagging, SBOMs, prompt/data policy, SCA, and audit trails. |

## The pattern behind almost all 50

The safest mental model is: **AI-generated code is third-party code produced at high speed by a junior contractor who has partial context and may also have keys to your machine.** The fix is not “never use AI.” The fix is to add the missing engineering controls around it: small diffs, tests, security scans, least privilege, sandboxing, secret management, CI/CD gates, real backups, and accountable human review.
