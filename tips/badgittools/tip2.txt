I interpreted “each of these” as: every hook, hook manager, check type, repo file/config, Git config, and enforcement layer in your writeup. “Always bad” below means “bad in any real shared/team/security-sensitive repo.” In a disposable scratch repo, some of these are merely inconvenient rather than dangerous.

I found strong alignment across primary Git docs, GitHub/GitLab/Bitbucket docs, tool docs, GitHub issues, and Reddit engineering discussions: **local hooks are good feedback, but bad authority**. Git’s own docs say client-side hooks are not copied on clone, several local hooks can be bypassed with `--no-verify`, and server-side hooks are the place to enforce push policy. GitHub/GitLab/Bitbucket enforcement features cover the same idea with protected branches, required checks, approvals, CODEOWNERS, code scanning, and merge checks. ([Git][1])

I used official docs as the authority, GitHub issues for real tool failure modes, and Reddit for practitioner consensus. X/Twitter-style sources were too noisy to treat as load-bearing evidence, so I would not anchor policy on them.

# 0. Universal rules

| Area               | BAD behavior — always bad                                                                               | GOOD behavior — always good                                                                                      |
| ------------------ | ------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| Authority          | Treating local hooks as the final gate. They can be absent, skipped, or misconfigured.                  | Use hooks for fast feedback, then enforce in CI, protected branches, rulesets, or server-side hooks.             |
| Bypass culture     | Teaching people to use `--no-verify` as normal workflow.                                                | Treat bypasses as rare exceptions, with follow-up fixes to slow/flaky hooks.                                     |
| Speed              | Running huge, slow, flaky checks in `pre-commit`.                                                       | Keep `pre-commit` fast; put heavier checks in `pre-push` or CI.                                                  |
| Determinism        | Hooks that behave differently per machine because tool versions are not pinned.                         | Pin versions through lockfiles, pre-commit revisions, package managers, or containers.                           |
| Staged-file safety | Modifying unstaged files, committing unstaged changes accidentally, or hiding partial-staging behavior. | Run checks against the staged snapshot or clearly restage only intended files.                                   |
| Secrets            | Letting secrets enter Git history.                                                                      | Scan locally, scan in CI, use push protection, and rotate secrets if anything leaks.                             |
| CI                 | Having local hooks that are stricter than CI, or CI that does not run the same critical checks.         | Mirror important local checks in CI and make them required before merge.                                         |
| Reviews            | Rubber-stamp reviews, self-approval, or stale approvals after major changes.                            | Use meaningful review, CODEOWNERS for sensitive paths, and stale-review dismissal where appropriate.             |
| Generated files    | Committing `dist/`, `build/`, vendored blobs, or generated files without a policy.                      | Either regenerate reproducibly in CI or commit generated files only when the repo explicitly requires it.        |
| Workflow security  | Running untrusted PR code in privileged workflows with secrets or write tokens.                         | Use least privilege, avoid dangerous `pull_request_target` patterns, and pin trusted actions. ([GitHub Docs][2]) |

# 1. Native Git hooks

Git’s official hook docs distinguish hooks that can abort an operation from notification-only hooks. A core bad pattern is using a post-event hook as if it can block the event. Another is relying on client-side hooks for policy enforcement when Git explicitly warns that client hooks are not copied when cloning. ([Git][1])

| Hook                    | BAD behavior — always bad                                                                                                                       | GOOD behavior — always good                                                                                                |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| `pre-commit`            | Using it as the only protection; making it slow/flaky; silently changing unrelated files; relying on it for secrets/security gates without CI.  | Run fast checks on staged content: format, lint, conflict markers, obvious secrets, large-file checks, debug-print checks. |
| `prepare-commit-msg`    | Using it as a policy validator or blocker for bad messages. Git says it is not suppressed by `--no-verify`, so bad scripts here can trap users. | Auto-fill ticket IDs, branch names, templates, or metadata; keep it deterministic and lightweight.                         |
| `commit-msg`            | Enforcing unclear or arbitrary message rules locally only; blocking useful commits because the regex is too brittle.                            | Enforce Conventional Commits, DCO `Signed-off-by`, ticket IDs, and message length, then mirror in CI or repo rules.        |
| `pre-merge-commit`      | Assuming it catches every merge problem, especially manual conflict resolution mistakes.                                                        | Run the same fast checks as `pre-commit` before a clean merge commit is created.                                           |
| `post-commit`           | Expecting it to block a commit. It cannot.                                                                                                      | Use for notifications, metrics, reminders, or local automation after the commit exists.                                    |
| `pre-rebase`            | Blanket-banning all rebases and preventing normal cleanup on private branches.                                                                  | Prevent rebasing protected, public, release, or shared branches.                                                           |
| `post-checkout`         | Mutating tracked files silently or failing checkout because local tools are missing.                                                            | Restore local-only toolchains, warn about environment drift, refresh caches.                                               |
| `post-merge`            | Using it as an enforcement hook. It runs after merge and cannot undo the policy failure cleanly.                                                | Reinstall dependencies, warn after lockfile changes, regenerate local-only artifacts.                                      |
| `post-rewrite`          | Assuming every history rewrite tool triggers it. Git docs note full-history tools may not call it.                                              | Warn after amend/rebase, update local metadata, remind developers to rerun checks.                                         |
| `post-index-change`     | Doing heavy work on every index change. That makes Git feel broken.                                                                             | Lightweight cache invalidation, editor integration, or status refresh.                                                     |
| `pre-push`              | Treating it as authoritative, or running a huge test suite that makes pushing painful.                                                          | Run fuller but still reasonable checks: typecheck, build, targeted tests, secret scan.                                     |
| `pre-receive`           | Slow, flaky, opaque rejection scripts that make pushes fail without actionable messages.                                                        | Central hard gate across pushed refs: secrets, branch policy, commit identity, file-size limits, forbidden paths.          |
| `update`                | Duplicating all global policy per ref with inconsistent behavior.                                                                               | Per-ref enforcement: fast-forward-only branches, protected tags, tag rewrite blocking.                                     |
| `proc-receive`          | Using advanced reference policy casually without strong tests or rollback.                                                                      | Advanced self-hosted Git policy when ordinary `pre-receive`/`update` is not enough.                                        |
| `post-receive`          | Expecting it to reject a bad push. It runs after acceptance.                                                                                    | Deploy, notify, mirror, or trigger indexing after refs are updated.                                                        |
| `post-update`           | Using it for blocking policy. It cannot stop the accepted push.                                                                                 | Lightweight notification/indexing after refs update.                                                                       |
| `reference-transaction` | Unsafe side effects inside ref transactions; slow scripts that risk ref-update reliability.                                                     | Audit or enforce ref-transaction policy carefully, with deterministic behavior.                                            |
| `push-to-checkout`      | Treating push-to-deploy as safe without deployment controls.                                                                                    | Controlled push-to-deploy worktree update checks on self-hosted repos.                                                     |
| `pre-auto-gc`           | Routinely disabling garbage collection and letting repos degrade.                                                                               | Defer automatic GC only for known local cases, such as avoiding conflict with another process.                             |
| `fsmonitor-watchman`    | Returning incomplete file-change data, causing stale Git status.                                                                                | Conservative, complete file-monitor output; fall back safely on error.                                                     |
| `applypatch-msg`        | Ignoring patch/email workflows while claiming commit policy is universal.                                                                       | Validate or normalize patch commit messages before `git am` commits them.                                                  |
| `pre-applypatch`        | Running huge checks on every patch apply.                                                                                                       | Inspect the resulting tree before the patch commit is finalized.                                                           |
| `post-applypatch`       | Expecting it to block a patch. It cannot.                                                                                                       | Notify or log after patch application.                                                                                     |
| `sendemail-validate`    | Sending patch series with broken metadata, MIME, or applyability.                                                                               | Validate patch series before `git send-email`.                                                                             |
| `p4-changelist`         | Applying Git-p4 assumptions to normal Git repos.                                                                                                | Validate Git-p4 changelist text in Git-p4 workflows only.                                                                  |
| `p4-prepare-changelist` | Using it as a general commit-template hook.                                                                                                     | Prepare changelist text for Perforce submission workflows.                                                                 |
| `p4-post-changelist`    | Expecting it to block anything.                                                                                                                 | Notification/logging after Git-p4 changelist creation.                                                                     |
| `p4-pre-submit`         | Treating it as a normal Git push gate.                                                                                                          | Validate Git-p4 submission before sending to Perforce.                                                                     |

# 2. Hook distribution tools

Git defaults hooks to `.git/hooks`, but `core.hooksPath` can redirect hooks to a shared directory. That makes versioned hook directories possible, but it does **not** magically enforce policy for everyone unless setup is installed and backed by CI/server rules. ([Git][3])

| Tool/setup                                 | BAD behavior — always bad                                                                                                             | GOOD behavior — always good                                                                                                                                                                                   |
| ------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Raw `.git/hooks/*`                         | Using unversioned personal hooks as team policy. They are not cloned.                                                                 | Use for one-off personal automation only.                                                                                                                                                                     |
| Checked-in `.githooks/` + `core.hooksPath` | Assuming every clone automatically uses it; hiding surprising scripts in the repo.                                                    | Commit simple, readable hooks; provide an install command; mirror critical checks in CI.                                                                                                                      |
| `init.templateDir`                         | Assuming it updates existing repos or other people’s clones.                                                                          | Use to seed your own new repos with preferred personal hooks/templates.                                                                                                                                       |
| `pre-commit`                               | Unpinned hook revisions, giant slow hooks, or installing only one hook type while claiming all hooks are covered.                     | Use `.pre-commit-config.yaml`, pin hook revisions, install needed hook types, and run `pre-commit run --all-files` in CI. The tool is explicitly designed as a multi-language hook manager. ([Pre-Commit][4]) |
| Husky                                      | Using it in non-Node repos just because it is popular; putting complex logic directly in shell hooks.                                 | Best in Node/JS repos; commit `.husky/`, call package scripts, keep hooks minimal. Husky uses Git’s `core.hooksPath` and supports client-side hooks. ([Typicode][5])                                          |
| `lint-staged`                              | Misunderstanding partial staging, stash behavior, or using `--no-stash` in ways that can commit unintended unstaged changes.          | Use for staged-file format/lint checks; understand its stash/backup behavior; leave global typecheck/build/test to `pre-push` or CI. ([GitHub][6])                                                            |
| Lefthook                                   | Parallelizing jobs that mutate the same files or share unsafe state.                                                                  | Use for fast polyglot hook orchestration with explicit `lefthook.yml`; validate config and keep jobs deterministic. ([GitHub][7])                                                                             |
| Overcommit                                 | Forcing Ruby-based tooling into repos where contributors do not have the runtime, or where branch-specific Ruby versions break hooks. | Good for Ruby-heavy repos that already standardize Ruby tooling; store hook config in source control. ([GitHub][8])                                                                                           |
| CaptainHook                                | Adding PHP/Composer hook tooling to non-PHP workflows without reason.                                                                 | Good for PHP/Composer repos that want commit-message validation, quality checks, tests, and post-pull automation. ([GitHub][9])                                                                               |

# 3. Commit-quality checks

The rule here is simple: **fast local checks, authoritative CI checks**. Reddit practitioner discussions line up with this: hooks are useful when they prevent wasted CI cycles, but they become counterproductive when slow, flaky, or used as a substitute for CI. ([Reddit][10])

| Check category         | BAD behavior — always bad                                                                                                 | GOOD behavior — always good                                                                                                                                                                                   |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Formatting             | Fighting style manually in review; letting CI reformat after merge; using different formatter versions locally and in CI. | Run deterministic formatters with pinned versions; use separate formatting commits and add mass-format commits to `.git-blame-ignore-revs`.                                                                   |
| Linting                | Flaky or environment-dependent lint; lint warnings ignored forever.                                                       | Pin linter versions, fail CI on agreed rules, keep local lint fast.                                                                                                                                           |
| Type checks            | Running only staged-file checks when type systems require whole-project context.                                          | Run quick local checks where possible, but require full typecheck in `pre-push` or CI.                                                                                                                        |
| Tests                  | Slow/flaky full suites in `pre-commit`; tests that developers habitually skip.                                            | Fast targeted tests locally; full test matrix in required CI.                                                                                                                                                 |
| Secret scanning        | Committing API keys, passwords, tokens, private keys, or cloud credentials.                                               | Use tools like Gitleaks, git-secrets, TruffleHog, detect-secrets, plus host push protection; rotate anything that leaks. ([GitHub][11])                                                                       |
| Commit messages        | Accepting `wip`, `fix stuff`, `changes`, or missing legal/compliance trailers when required.                              | Enforce Conventional Commits, DCO trailers, ticket IDs, and clear subject length. Conventional Commits define a machine-readable structure that supports changelogs and SemVer automation. ([Commitlint][12]) |
| Huge files             | Committing large binaries directly into Git history.                                                                      | Use Git LFS for large assets, and commit LFS patterns in `.gitattributes`. Git LFS stores large content externally while Git stores pointers. ([Git Large File Storage][13])                                  |
| Generated/vendor files | Accidentally committing `dist/`, `build/`, `.next/`, `node_modules/`, vendored blobs, or generated code with no policy.   | Define which generated files are committed, which are reproducible, and which are ignored.                                                                                                                    |
| Debug code             | Committing `debugger`, `pdb.set_trace`, `console.log` spam, `dbg!`, or temporary bypasses.                                | Block obvious debug markers with allowlists for tests/docs.                                                                                                                                                   |
| Conflict markers       | Committing `<<<<<<<`, `=======`, `>>>>>>>`.                                                                               | Always scan staged files and CI for conflict markers.                                                                                                                                                         |
| Permissions            | Accidental executable-bit changes.                                                                                        | Review and block unexpected mode changes.                                                                                                                                                                     |
| Line endings           | Cross-platform repos with uncontrolled CRLF/LF churn.                                                                     | Use `.gitattributes` normalization and `core.safecrlf`. Git can warn or reject irreversible CRLF conversion. ([Git][14])                                                                                      |
| Whitespace             | Trailing whitespace, mixed tabs/spaces where prohibited, blank-at-EOF churn.                                              | Use `core.whitespace`, formatter rules, and `apply.whitespace=error` where appropriate. ([Git][3])                                                                                                            |
| Dependency changes     | Manifest changes without lockfile updates, or lockfile changes without explanation.                                       | Use lockfile checks, dependency review, OSV scanning, Dependabot or Renovate. ([GitHub Docs][15])                                                                                                             |
| Docs                   | Broken Markdown, broken links, stale generated docs.                                                                      | Run Markdown lint, link checks, and docs builds in CI.                                                                                                                                                        |
| Migrations             | Unsafe migrations without rollback/compatibility review.                                                                  | Migration linting, backward-compatible deploy checks, and tests against representative DB state.                                                                                                              |
| API/schema drift       | Breaking OpenAPI, GraphQL, protobuf, or DB schema compatibility silently.                                                 | Run compatibility checks and require owner review for public contracts.                                                                                                                                       |
| IaC                    | Terraform/Kubernetes changes with no validation or security scan.                                                         | `terraform fmt`, `terraform validate`, `tflint`, Checkov/tfsec-style scanning in CI.                                                                                                                          |
| Containers             | Insecure Dockerfiles, unpinned bases, privileged images, no image scan.                                                   | Dockerfile linting, image scanning, minimal base images, reproducible builds.                                                                                                                                 |
| License/SBOM           | Shipping unknown dependencies/licenses.                                                                                   | Generate SBOMs with tools like Syft and scan dependency vulnerabilities with tools like OSV-Scanner. ([GitHub][16])                                                                                           |

# 4. Repo-level Git setup files

| File/config                              | BAD behavior — always bad                                                                                                                    | GOOD behavior — always good                                                                                                                                                                 |
| ---------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `.gitignore`                             | Using `.gitignore` as if it protects already tracked secrets or files. Git explicitly says ignore rules do not affect already tracked files. | Commit shared ignore rules for intentionally untracked files; use `.git/info/exclude` or global ignores for personal/local-only patterns. ([Git][17])                                       |
| `.gitattributes`                         | Omitting it in cross-platform repos; misclassifying binary files as text; using unsafe merge behavior blindly.                               | Normalize line endings, mark binary files, configure diff/merge behavior, and store LFS patterns. ([Git][14])                                                                               |
| `.editorconfig`                          | Letting it conflict with formatters, causing editor-vs-CI churn.                                                                             | Use it for shared indentation, newline, charset, and trim rules across editors. EditorConfig is designed to maintain consistent coding styles across editors and IDEs. ([EditorConfig][18]) |
| `.pre-commit-config.yaml`                | Unpinned remote hooks; hooks that mutate too much; no CI mirror.                                                                             | Version hook config, pin revisions, document install, run in CI.                                                                                                                            |
| `.husky/`                                | Committing opaque, complex shell logic; relying on missing execute bits or local-only scripts.                                               | Commit minimal hook scripts that call documented package scripts.                                                                                                                           |
| `lefthook.yml`                           | Race-prone parallel jobs or undocumented commands.                                                                                           | Declarative hook config, validated locally and in CI.                                                                                                                                       |
| `.lintstagedrc` / `lint-staged` config   | Running whole-project tools and pretending they only check staged files; partial-staging surprises.                                          | Run path-aware staged-file tools only: formatters, targeted linters, simple scans.                                                                                                          |
| `commitlint.config.*`                    | Arbitrary bureaucracy that blocks useful commits without helping automation.                                                                 | Keep commit rules small, documented, and aligned with release/changelog automation.                                                                                                         |
| `.gitmessage`                            | Huge templates that people delete every time.                                                                                                | Short template that reminds users of subject format, body style, ticket/DCO rules. Git supports `commit.template`. ([Git][3])                                                               |
| `.mailmap`                               | Rewriting history just to fix author display names.                                                                                          | Canonicalize author/committer names and emails without rewriting history. ([Git][19])                                                                                                       |
| `.git-blame-ignore-revs`                 | Hiding real functional changes from blame.                                                                                                   | List pure mass-formatting/mechanical commits; configure blame to ignore them. ([Git][20])                                                                                                   |
| `.lfsconfig` + `.gitattributes`          | Relying on personal/global LFS config so fresh clones or forks behave differently.                                                           | Commit LFS patterns and config needed for the repo. ([GitHub Docs][21])                                                                                                                     |
| `CODEOWNERS`                             | Stale owners, individual-only owners, or paths so broad that reviews become meaningless.                                                     | Use teams, require owner review for sensitive paths, and keep ownership current. GitHub can require CODEOWNERS review through branch protection. ([GitHub Docs][22])                        |
| `.github/workflows/*`                    | Broad write tokens, unpinned third-party actions, secrets exposed to untrusted PR code.                                                      | Least privilege, pinned actions, OIDC for cloud auth, restricted permissions, protected environments. ([GitHub Docs][2])                                                                    |
| `.gitlab-ci.yml`                         | Optional pipelines that can be ignored before merge.                                                                                         | Required GitLab pipelines/status checks, approvals, Code Owners, and protected branches. ([GitLab Docs][23])                                                                                |
| `bitbucket-pipelines.yml`                | Pipelines that run but do not gate merges.                                                                                                   | Bitbucket branch restrictions, merge checks, CODEOWNERS/custom merge checks where available. ([Atlassian Support][24])                                                                      |
| `renovate.json` / `dependabot.yml`       | Unreviewed auto-merges, PR spam, or major updates with no test gate.                                                                         | Scheduled, grouped dependency PRs with lockfile updates and required CI. ([GitHub Docs][15])                                                                                                |
| `SECURITY.md`                            | No vulnerability reporting path.                                                                                                             | Clear disclosure channel, supported versions, and response expectations.                                                                                                                    |
| `CONTRIBUTING.md`                        | Hidden rules that only CI reveals after failure.                                                                                             | Document setup, hooks, branch rules, commit rules, tests, and release process.                                                                                                              |
| `LICENSE`                                | No license in public/shared code, creating legal ambiguity.                                                                                  | Explicit license that matches intended reuse.                                                                                                                                               |
| `Makefile` / `justfile` / `Taskfile.yml` | Destructive or surprising default targets.                                                                                                   | One-command local quality entry points: `fmt`, `lint`, `test`, `typecheck`, `check`.                                                                                                        |

# 5. Git config hardening

| Config                     | BAD behavior — always bad                                                                                                                                                                                    | GOOD behavior — always good                                                                                                                  |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------- |
| `core.hooksPath .githooks` | Assuming clones automatically enable it. It is config, not repository magic.                                                                                                                                 | Use it with an onboarding/bootstrap command, and keep hooks versioned in the repo. ([Git][3])                                                |
| `commit.template`          | Giant templates that make every commit harder.                                                                                                                                                               | Concise template for subject/body/footer reminders. ([Git][3])                                                                               |
| `commit.gpgSign true`      | Forcing signing before contributors have keys, agents, or hardware-token support; breaking rebases with repeated prompts. Git docs note signing all commits can cause many passphrase prompts during rebase. | Require signed commits only where identity assurance matters, and document setup. ([Git][3])                                                 |
| `push.default=simple`      | Using old `matching` behavior in shared central repos. Git docs warn `matching` is not suitable for shared central workflows.                                                                                | Use `simple` as a safe default. Git documents it as the default since Git 2.0 and safest for beginners. ([Git][3])                           |
| `push.default=nothing`     | Enabling it for beginners without explaining explicit refspecs.                                                                                                                                              | Use in high-risk repos where accidental push must be impossible. Git says it makes `git push` error unless a refspec is explicit. ([Git][3]) |
| `core.safecrlf warn/true`  | Ignoring warnings or using it without `.gitattributes` clarity.                                                                                                                                              | Use with `.gitattributes` to catch irreversible CRLF conversion. ([Git][3])                                                                  |
| `core.whitespace`          | Enforcing whitespace rules that conflict with formatters or language norms.                                                                                                                                  | Highlight common whitespace problems consistently. ([Git][3])                                                                                |
| `apply.whitespace=error`   | Breaking patch workflows before the team has normalized whitespace policy.                                                                                                                                   | Enforce once rules are documented and formatter-compatible. ([Git][3])                                                                       |

# 6. Server-side and repo-host enforcement

This is where prevention becomes real. GitHub rulesets/status checks, GitHub branch protection, GitLab approval rules/external status checks, and Bitbucket merge checks all exist to move enforcement out of optional local tooling. ([GitHub Docs][25])

| Enforcement layer       | BAD behavior — always bad                                                                                               | GOOD behavior — always good                                                                                                                                                         |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Protected branches      | Protecting only `main` while `release/*`, `prod/*`, or deployment branches remain open; letting admins bypass casually. | No direct pushes, no deletion, no force-pushes on critical branches.                                                                                                                |
| Required PR reviews     | Rubber stamps, self-approval, stale approvals after major changes.                                                      | Require meaningful review; dismiss stale approvals when relevant; require conversation resolution.                                                                                  |
| CODEOWNERS              | Owners that are stale, absent, or too broad to be useful.                                                               | Required review from active domain-owner teams for critical paths.                                                                                                                  |
| Required status checks  | Optional CI, flaky checks, or checks that can be skipped by renaming jobs.                                              | Required exact checks from trusted CI: tests, lint, typecheck, build, security scans, docs.                                                                                         |
| Required signed commits | Requiring signatures but not verifying contributor onboarding or allowed identities.                                    | Use where commit identity matters; document key setup and recovery.                                                                                                                 |
| Linear history          | Mandating it without teaching squash/rebase workflow, causing repeated conflicts.                                       | Use if the repo values clean history; pair with clear merge strategy.                                                                                                               |
| Merge queue             | Not using it on busy repos where stale green builds race into `main`.                                                   | Use on high-throughput repos to test the merge result before landing.                                                                                                               |
| Secret push protection  | Allowing routine unreviewed bypasses.                                                                                   | Block supported secrets before they enter the repo; review bypass requests; rotate exposed secrets. GitHub documents push-protection bypass review flows. ([GitHub Docs][26])       |
| Code scanning gate      | Treating serious alerts as optional warnings.                                                                           | Block merges when required tools report alerts above threshold, when analysis is pending, or when required tools are missing. ([GitHub Docs][27])                                   |
| Server-side hooks       | Slow, opaque, untested rejection logic that developers cannot fix.                                                      | Deterministic `pre-receive`, `update`, or `proc-receive` hooks with clear messages, logs, and tests. Git’s server-side hooks can reject pushes before refs are updated. ([Git][28]) |

# 7. Recommended setups: bad vs good

| Setup                      | BAD behavior — always bad                                                                                                                                      | GOOD behavior — always good                                                                                                                        |
| -------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| Minimal solo repo          | No `.gitignore`, no `.gitattributes`, no formatter, no secret scan, and “I’ll remember not to commit bad stuff.”                                               | `.gitignore`, `.gitattributes`, `.editorconfig`, formatter, secret scan, commit template, and a one-command `check`.                               |
| Serious team repo          | Local hooks only; optional CI; unprotected `main`; no review requirements; no ownership for sensitive paths.                                                   | Local hooks for speed, required CI for authority, protected branches, PR reviews, CODEOWNERS, secret scanning, dependency scanning, code scanning. |
| High-risk / regulated repo | Direct production pushes, force-pushes to protected history, unsigned/unreviewed commits where identity matters, manual security bypasses with no audit trail. | Server-side enforcement, no force-pushes, signed commits where required, mandatory security review, SBOM/provenance, audit logging, merge queue.   |

# 8. Practical “always bad / always good” policy template

Use this as a team rule set:

| Policy          | BAD                                            | GOOD                                                                                     |
| --------------- | ---------------------------------------------- | ---------------------------------------------------------------------------------------- |
| Local hooks     | “The hook blocked it locally, so we are safe.” | “The hook gives fast feedback; CI/rulesets enforce.”                                     |
| CI              | “CI is advisory.”                              | “Required checks must pass before merge.”                                                |
| Secrets         | “Delete the secret commit and move on.”        | “Rotate the secret, remove from history if needed, add detection to prevent recurrence.” |
| Formatting      | “Reviewers argue about style.”                 | “Formatter decides; reviewers discuss design.”                                           |
| Commit messages | “Anything is fine.”                            | “Readable, searchable, machine-parseable when release automation needs it.”              |
| Reviews         | “Approval means someone clicked approve.”      | “Approval means ownership, correctness, security, and maintainability were considered.”  |
| Bypasses        | “Bypass whenever the tool is annoying.”        | “Bypass only for documented emergencies; fix the tool afterward.”                        |
| Generated files | “Whatever changed gets committed.”             | “Generated files have an explicit include/exclude policy.”                               |
| Dependencies    | “Update randomly.”                             | “Automated update PRs, lockfiles, vulnerability checks, tests.”                          |
| Workflows       | “Actions can do anything by default.”          | “Least privilege, pinned dependencies, no untrusted privileged execution.”               |

The core conclusion is: **bad behavior is relying on one layer, especially a local one. Good behavior is layered, fast locally, authoritative remotely, deterministic, documented, and boring.**
