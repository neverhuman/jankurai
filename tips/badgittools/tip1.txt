Below, **“always bad”** means “a pattern that should be avoided in normal professional repos because it creates risk, bypasses policy, wastes developer time, or causes unclear failures.” **“Always good”** means “a broadly safe best practice that remains good even when the exact tools differ.”

The strongest source alignment is this: **local hooks are fast feedback, not authority**. Git’s own docs say client hooks are not copied when a repo is cloned, and hooks like `pre-commit`, `commit-msg`, and `pre-merge-commit` can be bypassed with `--no-verify`; Git also documents server-side hooks as the place to enforce push policy. Community discussions on Reddit, X/Twitter-style posts, and Hacker News consistently echo the same practical lesson: slow or fragile local hooks get skipped, while CI/protected branches/server hooks are the reliable gate. ([Git][1])

## 1. Native Git hooks

Git supports many client-side and server-side hooks; the key distinction is that client hooks guide developers, while server-side hooks can reject pushes for everyone. Git documents the bypass behavior for local hooks and the blocking behavior for `pre-receive`, `update`, and related server-side hooks. ([Git][2])

| Hook                                                                            | Always-bad behavior                                                                                                                                                                                          | Always-good behavior                                                                                                                                                                                                             |
| ------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `pre-commit`                                                                    | Treating it as the only enforcement layer; running the full slow test suite; changing unrelated unstaged files; producing vague failures; blocking commits for subjective style nitpicks not enforced in CI. | Keep it fast, deterministic, and staged-file-focused. Run formatters, basic lint, secret checks, conflict-marker checks, large-file checks, and obvious debug-code checks. Mirror important checks in CI.                        |
| `prepare-commit-msg`                                                            | Silently rewriting what the author meant; using it for policy enforcement; making commits fail for reasons unrelated to message preparation.                                                                 | Auto-insert helpful context such as ticket IDs, branch names, template sections, or trailers while leaving the author in control.                                                                                                |
| `commit-msg`                                                                    | Enforcing cryptic regexes with no examples; allowing bad messages locally but failing later in CI; requiring fake verbosity; accepting fake DCO sign-offs.                                                   | Enforce clear commit-message standards: Conventional Commits, DCO `Signed-off-by` where required, ticket IDs, and useful subject length. Show examples on failure.                                                               |
| `pre-merge-commit`                                                              | Assuming it protects all merges or all conflict resolutions; running checks so heavy that normal merges become painful.                                                                                      | Run the same quick checks as `pre-commit` before Git creates an automatic merge commit.                                                                                                                                          |
| `post-commit`                                                                   | Trying to enforce policy after the commit already exists; running destructive scripts; doing slow network operations that make committing feel broken.                                                       | Use for non-blocking notifications, metrics, reminders, or local convenience tasks.                                                                                                                                              |
| `pre-rebase`                                                                    | Blocking all rebases because “rebasing is bad”; interfering with private local workflows.                                                                                                                    | Prevent rebasing protected/public branches, warn before rewriting shared history, and guide safer alternatives.                                                                                                                  |
| `post-checkout`                                                                 | Mutating tracked source files unexpectedly; deleting local work; doing heavyweight setup every branch switch.                                                                                                | Restore local toolchains, refresh caches, warn about missing environment setup, or run idempotent lightweight setup.                                                                                                             |
| `post-merge`                                                                    | Treating it as enforcement; auto-changing source files without telling the developer; hiding dependency changes.                                                                                             | Reinstall dependencies, refresh generated local-only artifacts, or remind developers after lockfile/toolchain changes.                                                                                                           |
| `post-rewrite`                                                                  | Assuming it can block rewrites after they already happened; hiding amend/rebase consequences.                                                                                                                | Warn after amend/rebase, update local metadata, or remind developers about force-push implications.                                                                                                                              |
| `post-index-change`                                                             | Running expensive scans every time the index changes; causing `git add`/`git status` workflows to lag.                                                                                                       | Use only for lightweight cache invalidation or tooling that truly needs index-change awareness.                                                                                                                                  |
| `pre-push`                                                                      | Running giant flaky suites that developers routinely bypass; treating it as a replacement for CI; failing with unclear remote-ref context.                                                                   | Run medium-cost checks before code leaves the machine: type checks, builds, fast integration tests, and secret scans. Keep full enforcement in CI/protected branches.                                                            |
| `pre-receive`                                                                   | Slow, opaque, global checks that block everyone; duplicating CI poorly; returning unhelpful “rejected” messages; depending on unavailable network services.                                                  | Hard-block universal repo policy: protected refs, forbidden secrets, unsigned commits where required, non-fast-forward updates, invalid commit messages, blocked file types, or tag rewrites. Keep it fast and explain failures. |
| `update`                                                                        | Implementing all global policy per ref in inconsistent ways; forgetting multi-ref atomicity concerns.                                                                                                        | Enforce per-branch/per-tag rules: fast-forward-only, no tag rewrites, protected branch policy, branch naming, or ref-specific permissions.                                                                                       |
| `proc-receive`                                                                  | Using it for simple policies that normal branch protection or `pre-receive` could handle; deploying poorly tested custom receive flows.                                                                      | Use only for advanced server-side reference workflows, virtual refs, or custom receive policy that truly needs it.                                                                                                               |
| `post-receive`                                                                  | Trying to reject pushes after refs were already accepted; doing fragile synchronous production deployments with no rollback.                                                                                 | Notify, mirror, index, trigger CI/deploy pipelines, or update external systems after successful push acceptance.                                                                                                                 |
| `post-update`                                                                   | Treating it as a blocking policy hook; doing heavy work synchronously.                                                                                                                                       | Run lightweight post-ref-update notifications or repository indexing.                                                                                                                                                            |
| `reference-transaction`                                                         | Doing slow or deadlock-prone work inside ref transactions; making ref updates brittle.                                                                                                                       | Audit or enforce ref-transaction invariants where low-level ref consistency matters.                                                                                                                                             |
| `push-to-checkout`                                                              | Turning branch pushes into unsafe production deploys; overwriting a dirty worktree.                                                                                                                          | Use only for controlled push-to-deploy setups with clean worktree checks, permissions, rollback, and logging.                                                                                                                    |
| `pre-auto-gc`                                                                   | Disabling garbage collection forever because one tool once had a problem.                                                                                                                                    | Defer or coordinate GC in constrained environments, large repos, or maintenance windows.                                                                                                                                         |
| `fsmonitor-watchman`                                                            | Returning incomplete file-change data, which can make Git believe dirty files are clean.                                                                                                                     | Be conservative: report all possibly changed files and fall back to full verification on errors.                                                                                                                                 |
| `applypatch-msg`                                                                | Ignoring commit-message policy in email/patch workflows.                                                                                                                                                     | Validate patch-supplied commit messages before applying emailed patches.                                                                                                                                                         |
| `pre-applypatch`                                                                | Applying unreviewed patches without basic checks; running destructive changes.                                                                                                                               | Run snapshot checks, lint, or safety checks before accepting patch-based changes.                                                                                                                                                |
| `post-applypatch`                                                               | Trying to block after the patch was already applied.                                                                                                                                                         | Notify, log, or remind maintainers after patch application.                                                                                                                                                                      |
| `sendemail-validate`                                                            | Sending malformed patches, wrong recipients, private data, or broken series.                                                                                                                                 | Validate patch format, recipients, headers, and project-specific email workflow rules before `git send-email`.                                                                                                                   |
| `p4-changelist`, `p4-prepare-changelist`, `p4-post-changelist`, `p4-pre-submit` | Using Git-P4 hooks outside a Git-P4 workflow; duplicating policies inconsistently between Git and Perforce.                                                                                                  | Enforce Perforce changelist and submit hygiene only for teams actually using Git-P4.                                                                                                                                             |

## 2. Hook distribution tools

The universal anti-pattern is **manual, invisible, per-clone hook setup**. Git supports `core.hooksPath`, and hook managers such as pre-commit, Husky, lint-staged, Lefthook, Overcommit, and CaptainHook exist to make hooks versioned and repeatable; several of these docs also emphasize staged-file checks, multiple hook types, or project-specific configuration. ([Git][3])

| Tool/setup                                 | Always-bad behavior                                                                                                                                                                           | Always-good behavior                                                                                                                   |
| ------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| Raw `.git/hooks/*`                         | Relying on it for team policy; assuming it exists in every clone; storing important checks only in unversioned local files.                                                                   | Use for personal experiments or one-off local convenience only.                                                                        |
| Checked-in `.githooks/` + `core.hooksPath` | Forgetting executable bits; writing shell that breaks on Windows/macOS/Linux; assuming users cannot disable it.                                                                               | Good lightweight repo-native setup. Commit hooks, document installation, and mirror important checks in CI.                            |
| `init.templateDir`                         | Treating seeded hooks as enforcement; letting templates become stale; assuming it affects already-cloned repos.                                                                               | Good personal/org bootstrap for new repos. Use it to seed helpful defaults, not to enforce policy.                                     |
| `pre-commit`                               | Using unpinned hook revisions; running slow full-project checks in `pre-commit`; never running the same hooks in CI; relying on `--no-verify` culture.                                        | Pin hook versions, keep checks fast, run `pre-commit run --all-files` in CI, and use targeted `SKIP=` only for intentional exceptions. |
| Husky                                      | Installing heavy Node-only hooks in non-Node repos; hiding surprising behavior in npm lifecycle scripts; relying on Husky as server enforcement.                                              | Excellent for Node/JS repos. Keep `.husky/` scripts small and call normal package scripts that CI also runs.                           |
| lint-staged                                | Running project-wide tools that ignore file arguments; adding `git add` manually in tasks; using global tools not declared in the repo; formatting files outside the staged set unexpectedly. | Run formatters/linters only on staged files, use local project dependencies, and configure tool-level ignores carefully.               |
| Lefthook                                   | Parallelizing jobs that mutate the same files and race; making YAML too magical to debug.                                                                                                     | Use for fast polyglot teams, parallel independent checks, explicit globs, and clear staged-file commands.                              |
| Overcommit                                 | Requiring a full Ruby app environment for every contributor when the repo is not Ruby-heavy; leaving hook gem versions floating; putting side effects in pre-commit hooks.                    | Good in Ruby-heavy repos with a minimal locked hook Gemfile, explicit configuration, and documented `SKIP`/`ONLY` escape hatches.      |
| CaptainHook                                | Assuming Composer-installed hooks are authoritative; skipping backend/CI checks; hiding complex PHP scripts in hook actions.                                                                  | Good for PHP/Composer repos. Use it for shared local feedback, then enforce important policy in CI or server-side checks.              |

## 3. Commit-quality checks

For checks, the bad behavior is usually not “using the tool”; it is **using the tool in the wrong layer**. Fast, deterministic checks belong in local hooks; full enforcement belongs in CI, protected branches, rulesets, or server hooks.

| Check category         | Always-bad behavior                                                                                                                                                     | Always-good behavior                                                                                                                                                                                                                                                                        |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Formatters             | Arguing about style manually in review; mixing formatting with feature changes; using different formatter versions locally and in CI.                                   | Use one formatter per language, pin versions, run locally and in CI, and isolate mass-formatting commits.                                                                                                                                                                                   |
| Linters                | Enabling noisy rules nobody fixes; allowing broad suppressions without review; failing locally but not in CI.                                                           | Use actionable rules, keep local lint fast, run full lint in CI, and review suppressions.                                                                                                                                                                                                   |
| Type checks            | Checking only changed files when the type system needs whole-project context; skipping type checks because tests pass.                                                  | Run fast type checks in `pre-push` and full type checks in CI.                                                                                                                                                                                                                              |
| Tests                  | Running the entire slow suite on every commit; accepting flaky tests as normal; letting developers bypass tests without CI catching failures.                           | Run fast unit tests locally, broader tests in `pre-push`, and full reliable test suites in CI.                                                                                                                                                                                              |
| Secret scans           | Committing real tokens, passwords, API keys, `.env` files, private keys, or cloud credentials; relying on `.gitignore` as security; treating a local scanner as enough. | Use local scanners plus CI/server/push protection. Rotate exposed secrets immediately and remove them from history where required. Gitleaks, git-secrets, TruffleHog, detect-secrets, and GitHub push protection all address secret detection/prevention in different layers. ([GitHub][4]) |
| Commit messages        | `wip`, `fix stuff`, `changes`, fake ticket IDs, fake sign-offs, or messages that hide breaking changes.                                                                 | Use Conventional Commits where useful, enforce DCO sign-off where the project requires it, and include ticket/breaking-change metadata consistently. ([Conventional Commits][5])                                                                                                            |
| Huge files             | Committing large binaries, archives, datasets, build outputs, or media directly to normal Git history.                                                                  | Enforce size limits and use Git LFS, artifact stores, package registries, or object storage for large assets.                                                                                                                                                                               |
| Generated/vendor files | Committing `node_modules`, `.next`, `dist`, `build`, generated blobs, or vendored dependencies without policy.                                                          | Ignore generated outputs by default; commit generated/vendor files only when intentional, reproducible, reviewed, and documented.                                                                                                                                                           |
| Debug code             | Shipping `debugger`, `pdb.set_trace`, `console.log`, `dbg!`, `println!`, temporary flags, or “TODO remove” markers in production paths.                                 | Block obvious debug markers with allowlists for tests, examples, or intentional logging.                                                                                                                                                                                                    |
| Conflict markers       | Committing `<<<<<<<`, `=======`, or `>>>>>>>`.                                                                                                                          | Always block merge-conflict markers before commit and in CI.                                                                                                                                                                                                                                |
| Permissions            | Accidental executable-bit changes, mass `chmod` changes, or platform-specific permission churn.                                                                         | Flag permission changes and require explicit review for intentional executable files.                                                                                                                                                                                                       |
| Line endings           | Mixed CRLF/LF, binary files treated as text, or cross-platform churn.                                                                                                   | Use `.gitattributes`, `core.safecrlf`, and editor/formatter settings.                                                                                                                                                                                                                       |
| Whitespace             | Trailing whitespace, tab/space drift, blank-at-EOL, or patch whitespace errors.                                                                                         | Use Git whitespace settings, formatters, and `apply.whitespace=error` for patch workflows.                                                                                                                                                                                                  |
| Dependency changes     | Updating manifests without lockfiles; updating lockfiles without manifests; accepting major upgrades without review; ignoring security advisories.                      | Enforce manifest/lockfile consistency, audit checks, license checks, and reviewed dependency automation.                                                                                                                                                                                    |
| Docs                   | Changing behavior/API without docs; broken Markdown links; docs that no longer build.                                                                                   | Run Markdown lint, link checks, and docs builds in CI.                                                                                                                                                                                                                                      |
| Migrations             | Destructive migrations with no rollback/compatibility plan; breaking old app versions during rolling deploys.                                                           | Use migration linters, backward-compatible migrations, rollback plans, and staged deploy checks.                                                                                                                                                                                            |
| API/schema drift       | Breaking OpenAPI, GraphQL, protobuf, database, or event schemas without versioning.                                                                                     | Run compatibility checks and require explicit breaking-change review.                                                                                                                                                                                                                       |
| IaC                    | Committing unformatted Terraform, invalid plans, open security groups, hardcoded secrets, or unreviewed infra drift.                                                    | Run `terraform fmt`, `terraform validate`, `tflint`, `checkov`, `tfsec`, and plan review where appropriate.                                                                                                                                                                                 |
| Containers             | Shipping unchecked Dockerfiles, huge images, root-by-default containers, or vulnerable base images.                                                                     | Lint Dockerfiles, scan images, pin bases intentionally, and minimize runtime privileges.                                                                                                                                                                                                    |
| License/SBOM           | Adding dependencies with unknown licenses; failing to produce SBOM/provenance where required.                                                                           | Run license checks, vulnerability scans, and SBOM tooling such as `syft`, `osv-scanner`, `cargo-deny`, or equivalent tools.                                                                                                                                                                 |

## 4. Repo-level Git setup files

These files make good behavior easy and repeatable. Git’s docs cover `.gitignore`, `.gitattributes`, push defaults, CRLF safety, whitespace handling, commit signing/templates, and blame ignore files; these are not hooks, but they prevent many bad commits before review. ([Git][6])

| File/config                              | Always-bad behavior                                                                                                    | Always-good behavior                                                                                          |
| ---------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| `.gitignore`                             | Treating it as security; assuming it removes already-tracked files; ignoring files that should be reproducible source. | Ignore local-only, generated, cache, build, and secret-like files before they are tracked.                    |
| `.gitattributes`                         | Overbroad text/binary rules that corrupt files; inconsistent EOL rules across platforms.                               | Normalize line endings, mark binaries, configure diff/merge drivers, and document special path behavior.      |
| `.editorconfig`                          | Conflicting with the project formatter; setting rules nobody’s editor respects.                                        | Keep basic indentation, final newline, charset, and trim rules aligned with formatters.                       |
| `.pre-commit-config.yaml`                | Floating hook versions; stale hooks; local-only use with no CI equivalent.                                             | Commit pinned hook config, update deliberately, and run the same hooks in CI.                                 |
| `.husky/`                                | Large shell scripts full of business logic; hooks that only work on one developer’s machine.                           | Keep hook files tiny and delegate to package scripts that work locally and in CI.                             |
| `lefthook.yml`                           | Race-prone parallel jobs; unclear glob patterns; hidden file mutations.                                                | Use explicit staged-file globs, independent parallel jobs, and predictable output.                            |
| `.lintstagedrc` / lint-staged config     | Running tools that do not support file lists; manually adding `git add`; depending on globally installed binaries.     | Run local project tools on staged files only and let lint-staged manage staged updates.                       |
| `commitlint.config.*`                    | Enforcing arbitrary message rules that do not map to release, review, or audit needs.                                  | Enforce a small, documented convention with examples and CI/server parity.                                    |
| `.gitmessage`                            | Huge templates that create noisy commits; required sections that authors leave blank.                                  | Provide concise prompts for subject, body, issue ID, DCO, and breaking-change footer.                         |
| `.mailmap`                               | Using it to obscure authorship or rewrite credit socially; making unreviewed identity changes.                         | Canonicalize names/emails for clean logs, stats, and attribution.                                             |
| `.git-blame-ignore-revs`                 | Hiding real feature, bug, or behavior-changing commits from blame.                                                     | Add only mechanical mass-formatting or code-move commits.                                                     |
| `.lfsconfig` + `.gitattributes`          | Putting normal source/text in LFS unnecessarily; forgetting to track required large-file patterns.                     | Store large binary assets in LFS with reviewed path patterns.                                                 |
| `CODEOWNERS`                             | Stale owners, dead teams, overbroad ownership, or no branch rule requiring owner review.                               | Use living ownership for sensitive paths and require CODEOWNER review on protected branches.                  |
| `.github/workflows/*`                    | Unpinned actions, overbroad tokens, jobs that do not run on PRs, or optional checks for critical paths.                | Run required lint/test/build/security checks on PRs and protected branches with least privilege.              |
| `.gitlab-ci.yml`                         | CI jobs that can be skipped silently; different commands than local hooks; no required pipeline gate.                  | Use reproducible jobs, required pipeline checks, and the same quality commands developers can run locally.    |
| `bitbucket-pipelines.yml`                | Pipelines that do not gate merges; checks that differ from branch restrictions.                                        | Align pipelines with merge checks and protected branch rules.                                                 |
| `renovate.json` / `dependabot.yml`       | Blind auto-merge of risky updates; noisy PR floods nobody reviews.                                                     | Automate dependency updates with grouping, labels, required CI, and review rules for majors/security updates. |
| `SECURITY.md`                            | Empty or unmonitored security contact; telling reporters to open public issues for secrets/vulnerabilities.            | Provide a clear private reporting path, response expectations, and security ownership.                        |
| `CONTRIBUTING.md`                        | Rules that differ from actual hooks/CI; undocumented setup steps.                                                      | Document install, commit style, tests, review expectations, and one-command quality checks.                   |
| `LICENSE`                                | Missing or conflicting license files; copying incompatible code without clarity.                                       | Include the correct project license and keep dependency/license policy consistent.                            |
| `Makefile` / `justfile` / `Taskfile.yml` | Local commands that diverge from hooks and CI.                                                                         | Provide a single source of truth such as `make check`, `just check`, or `task check` that hooks and CI reuse. |

## 5. Git config hardening

| Config                   | Always-bad behavior                                                                                                                                                                                                     | Always-good behavior                                                                                           |
| ------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| `core.hooksPath`         | Pointing it to `/dev/null` or a personal path in a shared repo; silently disabling project hooks. Git documents that `core.hooksPath` can redirect hooks and can even disable hooks when set to `/dev/null`. ([Git][3]) | Use a committed `.githooks/` path or a hook manager path, document installation, and keep CI authoritative.    |
| `init.templateDir`       | Assuming templates enforce policy for all clones; never updating seeded hooks.                                                                                                                                          | Use it to bootstrap personal/org defaults for newly created repos.                                             |
| `commit.template`        | Forcing long boilerplate that authors ignore; making every commit message noisy.                                                                                                                                        | Use concise guidance: subject format, body reminder, ticket, sign-off, breaking-change footer.                 |
| `commit.gpgSign`         | Mandating signed commits without onboarding contributors, bots, and CI; treating signature alone as code review.                                                                                                        | Require signed commits where provenance matters, document setup, and combine with reviews and required checks. |
| `push.default=simple`    | Using unsafe broad push defaults in shared central repos. Git documents `simple` as a safer/default push mode and `nothing` as the strict mode that errors without an explicit refspec. ([Git][3])                      | Use `simple` for normal safety or `nothing` for strict teams that want explicit push refspecs.                 |
| `push.default=nothing`   | Surprising beginners without documentation; causing avoidable friction in simple repos.                                                                                                                                 | Good for high-control environments where every push target should be explicit.                                 |
| `core.safecrlf`          | Disabling CRLF safety in cross-platform repos and then accepting line-ending churn.                                                                                                                                     | Use `warn` or stricter settings together with `.gitattributes`.                                                |
| `core.whitespace`        | Letting whitespace junk create noisy diffs and patch failures.                                                                                                                                                          | Highlight trailing spaces, blank-at-EOL, and tab/space problems.                                               |
| `apply.whitespace=error` | Applying patches with known whitespace errors in projects where patch cleanliness matters.                                                                                                                              | Reject whitespace-damaged patches in email/patch workflows.                                                    |

## 6. Server-side and repo-host enforcement

This is the layer that actually prevents bad code from landing. GitHub, GitLab, Bitbucket, and OpenSSF guidance all support protected branches, required status checks, reviews, CODEOWNERS, signed commits, code scanning, and merge gates as durable enforcement mechanisms. ([GitHub Docs][7])

| Enforcement layer       | Always-bad behavior                                                                                         | Always-good behavior                                                                                                                                                                                                                                                                                                                           |
| ----------------------- | ----------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Protected branches      | Direct pushes to `main`, `release/*`, or `prod/*`; force-pushes to shared protected branches.               | Protect important branches, require PRs/MRs, and restrict who can push.                                                                                                                                                                                                                                                                        |
| Required PR reviews     | Rubber-stamp approvals; self-approval; stale approvals after major changes.                                 | Require meaningful review, use stale-review dismissal where useful, and require extra review for risky paths.                                                                                                                                                                                                                                  |
| CODEOWNERS              | Stale teams, overbroad ownership, owners with no time to review, or ownership not enforced by branch rules. | Require domain-owner review for security, infra, billing, auth, migrations, public APIs, and release paths.                                                                                                                                                                                                                                    |
| Required status checks  | Optional CI; merge allowed despite failed tests; required checks that do not cover real risk.               | Require lint, tests, type checks, build, security scans, docs, and relevant compatibility checks.                                                                                                                                                                                                                                              |
| Required signed commits | Blocking contributors without setup support; treating signatures as a substitute for review.                | Use for provenance-sensitive repos with documented setup and bot/service-account support.                                                                                                                                                                                                                                                      |
| Linear history          | Enforcing it blindly where merge commits are valuable for context or release trains.                        | Use when the repo wants clean rebase/squash history and the team understands the workflow.                                                                                                                                                                                                                                                     |
| Merge queue             | Letting many individually green PRs race into a broken `main`.                                              | Use merge queues on busy protected branches so changes are tested together before landing.                                                                                                                                                                                                                                                     |
| Secret push protection  | Relying only on post-merge alerts after secrets are already in history.                                     | Enable push protection and still run local/CI secret scans for defense in depth.                                                                                                                                                                                                                                                               |
| Code scanning gate      | Blocking every low-confidence alert forever; allowing untriaged critical alerts to merge.                   | Gate on severity/confidence thresholds, require triage, and block new high-risk findings.                                                                                                                                                                                                                                                      |
| Server-side hooks       | Slow, opaque hooks that block everyone; network-dependent scripts; duplicating host features badly.         | Use fast `pre-receive`/`update` hooks for policies the host cannot express: custom ref rules, legal checks, signed-off-by enforcement, forbidden paths, or regulated workflows. GitHub Enterprise’s pre-receive hook docs describe these as push-accept/reject scripts and warn that they affect workflows and performance. ([GitHub Docs][8]) |

## 7. Tool-specific “always bad” and “always good” patterns

| Tool/check                                                               | Always-bad behavior                                                                                                   | Always-good behavior                                                                                 |
| ------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| Prettier / Black / Ruff format / rustfmt / gofmt / clang-format / shfmt  | Manual formatting debates; mixed versions; formatting only some files with no policy.                                 | Pin versions and run the formatter automatically locally and in CI.                                  |
| ESLint / Ruff / Clippy / ShellCheck / markdownlint / yamllint / hadolint | Noisy warnings nobody fixes; lint suppressions without review; local rules different from CI.                         | Keep rules actionable, fail CI on agreed violations, and review suppressions.                        |
| `tsc --noEmit` / Pyright / mypy / `cargo check` / `go test ./...`        | Skipping type/build checks because unit tests pass; running checks only on touched files when global context matters. | Run in `pre-push` or CI, with faster subsets locally.                                                |
| Gitleaks / git-secrets / TruffleHog / detect-secrets                     | Treating scanners as perfect; committing baselines without reviewing findings; leaving leaked secrets active.         | Combine local, CI, and push-protection scanning; triage false positives; rotate exposed credentials. |
| commitlint / Conventional Commits                                        | Enforcing a convention nobody uses for changelogs/releases; cryptic failure messages.                                 | Use a convention that drives changelogs, releases, or review clarity; show examples.                 |
| DCO checks                                                               | Allowing fake or missing `Signed-off-by`; adding sign-offs on behalf of others without authorization.                 | Require accurate per-commit sign-off where the project has a DCO policy.                             |
| Git LFS policy                                                           | Adding massive blobs to Git history and then trying to ignore them later.                                             | Track large binary patterns up front and reject oversized normal Git blobs.                          |
| Terraform / IaC tools                                                    | Merging infra changes with no format, validate, plan, or security review.                                             | Require format, validate, lint, security checks, and reviewed plans for sensitive environments.      |
| Docker/image scanning                                                    | Shipping unscanned images or vulnerable bases by habit.                                                               | Lint Dockerfiles and scan images as part of required CI.                                             |
| License/SBOM tools                                                       | Discovering license conflicts only during release.                                                                    | Check licenses and generate SBOM/provenance continuously.                                            |

## 8. The practical default policy

For most repos, this is the clean split:

| Layer              | Bad behavior                                                                | Good behavior                                                                                                    |
| ------------------ | --------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| Local commit layer | Slow, fragile, bypass-prone hooks pretending to be authority.               | Fast staged-file checks: format, lint, secrets, conflict markers, file size, debug markers, commit message.      |
| Local push layer   | Huge flaky suites that make developers hate pushing.                        | Medium checks: type check, build, fast integration tests, broader secret scan.                                   |
| CI layer           | Optional checks, different commands than local, green builds that go stale. | Required status checks using the same commands as local tooling plus full tests/build/security/docs.             |
| Repo-host layer    | Unprotected `main`, direct pushes, no review, no owner gates.               | Protected branches, PR/MR review, CODEOWNERS, required checks, signed commits where needed, merge queue.         |
| Server-side layer  | Slow custom hooks replacing CI.                                             | Fast hard policy that must never be bypassed: refs, signatures, secrets, tags, forbidden files, regulated rules. |

The most robust rule is: **make the right thing easy locally, then make the unsafe thing impossible at the merge/push boundary.**
