Here’s the consolidated policy matrix. I’m using **“always bad”** to mean **bad for shared/team repos, protected branches, production code, or shared history**. A few things can be acceptable in a private scratch branch, but they should not be allowed to land in shared history.

The central rule is: **local hooks are guardrails, not gates**. Git’s own docs say client-side hooks are not copied when a repo is cloned, and several hooks, including `pre-commit`, `commit-msg`, and `pre-push`, can be bypassed with `--no-verify`; hard enforcement belongs in CI, protected branches/rulesets, or server-side hooks such as `pre-receive` and `update`. ([Git][1]) Community discussions on Reddit/X generally align with that: hooks are useful for fast feedback, but developers complain when they are slow, surprising, or treated as the only enforcement layer. ([Reddit][2])

## 1. Native Git hooks

| Hook / hook family                                                              | Always bad behavior                                                                                                                                                                                                       | Always good / best-practice behavior                                                                                                                                                                                                                    |
| ------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `pre-commit`                                                                    | Treating it as authoritative policy; running the whole slow test suite; doing network-heavy scans; silently modifying unstaged files; blocking normal small WIP commits.                                                  | Use it for fast, deterministic checks on staged content: formatting, linting, conflict markers, large files, secrets, debug prints, whitespace. Mirror important checks in CI.                                                                          |
| `prepare-commit-msg`                                                            | Using it as an unbypassable validator; overwriting the author’s message; inserting noisy boilerplate that people delete.                                                                                                  | Auto-prefill useful context: branch ticket ID, commit template, co-author hints, issue metadata. Do validation in `commit-msg`.                                                                                                                         |
| `commit-msg`                                                                    | Allowing vague shared-history messages like `wip`, `stuff`, `fix bug`; enforcing a private convention that CI does not enforce; blocking legitimate `revert`, `fixup!`, or `squash!` commits when the workflow uses them. | Enforce Conventional Commits, ticket IDs, DCO `Signed-off-by`, subject length, and breaking-change footers locally and in CI/server policy. Conventional Commits and DCO both define machine-readable conventions for this. ([Conventional Commits][3]) |
| `pre-merge-commit`                                                              | Assuming it catches all merges; relying on it instead of protected branches; running expensive checks that make local merges painful.                                                                                     | Run quick sanity checks before merge commits. Keep authoritative merge policy in CI/branch protection. Git notes this hook can abort merge commits but is still local and bypassable. ([Git][4])                                                        |
| `post-commit`                                                                   | Using it for blocking rules. It runs after the commit already exists.                                                                                                                                                     | Notifications, metrics, reminders, lightweight local automation.                                                                                                                                                                                        |
| `pre-rebase`                                                                    | Allowing people to rebase protected/public branches without warning; or blocking all rebases so developers cannot clean local branches.                                                                                   | Block or warn before rewriting public/protected history, especially `main`, `release/*`, and shared feature branches.                                                                                                                                   |
| `post-checkout`                                                                 | Destructive setup scripts; automatic dependency installs that alter the worktree without clear output.                                                                                                                    | Safe reminders: “toolchain changed,” “run install,” “env file missing,” “lockfile changed.”                                                                                                                                                             |
| `post-merge`                                                                    | Treating it as a merge gate; doing hidden destructive cleanup after pulls. Git says post-merge cannot affect the merge outcome. ([Git][4])                                                                                | Reinstall dependencies, restore permissions, warn about regenerated artifacts, or remind after lockfile changes.                                                                                                                                        |
| `post-rewrite`                                                                  | Assuming it sees every history rewrite; silently changing metadata after amend/rebase.                                                                                                                                    | Warn after amend/rebase, update review metadata, or refresh local caches.                                                                                                                                                                               |
| `post-index-change`                                                             | Running heavy jobs every time the index changes; creating noisy or recursive index edits.                                                                                                                                 | Very lightweight reactions to staging-area changes, such as refreshing status caches.                                                                                                                                                                   |
| `pre-push`                                                                      | Running hour-long suites; prompting for credentials; relying on it as the only protection against bad pushes.                                                                                                             | Run medium-weight checks before sharing: typecheck, build, fast tests, secret scan, package audit. Git says nonzero exit from `pre-push` aborts the push, but it remains local. ([Git][4])                                                              |
| `pre-receive`                                                                   | Flaky, slow, opaque server rejections; rules that differ from CI; scanning entire repo history on every small push without caching.                                                                                       | Hard-block organization-wide rules before refs update: protected patterns, secrets, unsigned commits, banned files, tag policy. Git says nonzero exit rejects all refs in the push. ([Git][4])                                                          |
| `update`                                                                        | Inconsistent branch-by-branch policy; allowing tag rewrites or non-fast-forward updates accidentally.                                                                                                                     | Per-ref enforcement: fast-forward-only, no tag rewrites, branch-specific permissions. Git explicitly calls out fast-forward enforcement as a use case. ([Git][4])                                                                                       |
| `proc-receive`                                                                  | Using it for ordinary checks when `pre-receive`/`update` is enough; hard-to-debug custom ref behavior.                                                                                                                    | Advanced server-side reference workflows, pseudo-refs, or custom centralized receive policy.                                                                                                                                                            |
| `reference-transaction`                                                         | Blocking broad ref transactions without clear messages; writing fragile hooks that corrupt normal ref maintenance.                                                                                                        | Auditing or enforcing atomic ref updates where you truly need transaction-level policy. Git allows rejection in preparing/prepared states. ([Git][4])                                                                                                   |
| `push-to-checkout`                                                              | Treating push-to-deploy as a safe production deployment model without isolation or rollback.                                                                                                                              | Carefully controlled push-to-worktree setups for staging/internal deploys only.                                                                                                                                                                         |
| `pre-auto-gc`                                                                   | Disabling Git garbage collection globally because it is inconvenient.                                                                                                                                                     | Rarely used; only suppress auto-GC for a clear maintenance reason.                                                                                                                                                                                      |
| `fsmonitor-watchman`                                                            | Returning an incomplete changed-file list, causing Git to miss modifications.                                                                                                                                             | Return an inclusive changed-file list; Git says including unchanged files is okay, but omitting changed files is not. ([Git][4])                                                                                                                        |
| `applypatch-msg`, `pre-applypatch`, `post-applypatch`                           | Ignoring email/patch workflows while enforcing PR workflows; using post hooks to block after the fact.                                                                                                                    | Validate patch messages, formatting, authorship, and tests for email-based contribution flows.                                                                                                                                                          |
| `sendemail-validate`                                                            | Sending malformed, wrongly targeted, or policy-violating patch series.                                                                                                                                                    | Validate recipients, subject format, patch series metadata, and sensitive content before `git send-email`.                                                                                                                                              |
| `p4-changelist`, `p4-prepare-changelist`, `p4-post-changelist`, `p4-pre-submit` | Letting Git-p4 policies drift from Perforce policy; using them in repos that do not use Git-p4.                                                                                                                           | Keep Git-p4 changelist/submission checks aligned with the authoritative Perforce workflow.                                                                                                                                                              |

## 2. Hook distribution tools

| Tool / setup                               | Always bad behavior                                                                                                                                                                                                                                         | Always good / best-practice behavior                                                                                                                                                                                                                                                                                          |
| ------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Raw `.git/hooks/*`                         | Treating manually installed hooks as team policy. They are not versioned and are not copied on clone. ([Git][1])                                                                                                                                            | Use only for personal one-off local automation.                                                                                                                                                                                                                                                                               |
| Checked-in `.githooks/` + `core.hooksPath` | Hardcoding absolute paths; changing `core.hooksPath` without onboarding docs; assuming every clone has installed it. Git allows `core.hooksPath`, and it can even be pointed at `/dev/null` to disable hooks, so do not treat it as enforcement. ([Git][5]) | Version the hook scripts, provide a bootstrap command, keep hooks portable, and enforce the same rules in CI/server policy.                                                                                                                                                                                                   |
| `init.templateDir`                         | Assuming it affects existing clones or enforces org policy.                                                                                                                                                                                                 | Good personal or enterprise bootstrap for new repos; still pair with CI/server checks.                                                                                                                                                                                                                                        |
| `pre-commit`                               | Unpinned hook versions; slow all-files checks on every commit; no CI mirror; using `--no-verify` as the normal escape hatch.                                                                                                                                | Pin `rev`s, run fast staged checks locally, use `pre-commit run --all-files` in CI, install needed hook types, and use `SKIP=` for intentional local skips. Pre-commit’s docs recommend config via `.pre-commit-config.yaml`, pinning `rev`, and note that always-running hooks can be slow and surprising. ([Pre-Commit][6]) |
| Husky                                      | Node lifecycle surprises; brittle shell scripts; assuming hooks are installed when dependencies are not; using it as the only gate.                                                                                                                         | Best for Node/JS repos. Keep scripts tiny, call package scripts, pair with `lint-staged`, and enforce in CI. Husky is explicitly built around Git’s `core.hooksPath`. ([Typicode][7])                                                                                                                                         |
| `lint-staged`                              | Overlapping globs that run mutating tasks concurrently; whole-project commands that ignore staged-file arguments; relying on globally installed binaries.                                                                                                   | Run formatters/linters only on staged files, use arrays for sequential tasks, avoid mutating races, and configure monorepos carefully. Its docs warn that overlapping globs with mutating tasks can race and that tasks run concurrently by default. ([GitHub][8])                                                            |
| Lefthook                                   | Parallel jobs that race on the same files; unvalidated config; using language-specific assumptions in a polyglot repo.                                                                                                                                      | Fast polyglot hook runner, especially for monorepos. Use parallelism for independent jobs, define explicit file scopes, and validate config. Lefthook documents parallel jobs, flexible file selection, and a checked-in `lefthook.yml`. ([GitHub][9])                                                                        |
| Overcommit                                 | Ruby runtime/version friction in non-Ruby teams; hooks with hidden side effects; not pinning dependencies.                                                                                                                                                  | Good in Ruby-heavy repos. Version `.overcommit.yml`, pin via Gemfile when needed, and understand that post hooks cannot block outcomes. ([GitHub][10])                                                                                                                                                                        |
| CaptainHook                                | PHP/Composer dependency mismatch; treating local PHP hooks as the only policy layer.                                                                                                                                                                        | Good for PHP/Composer repos: commit-message validation, coding-standard checks, tests, and shared project hooks. ([GitHub][11])                                                                                                                                                                                               |

## 3. Commit-quality checks

| Area                   | Always bad behavior                                                                                                        | Always good / best-practice behavior                                                                                                                                                                                             |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Formatting             | Merging code that churns formatting randomly; formatting unstaged files behind the user’s back.                            | Use standard formatters per language and run a non-mutating format check in CI.                                                                                                                                                  |
| Linting                | Blanket-disabling lint rules to “make green”; merging known lint failures.                                                 | Use language-specific linters, scoped suppressions, and CI enforcement.                                                                                                                                                          |
| Type checks            | Merging known type errors; making typecheck optional for shared branches.                                                  | Run typecheck in `pre-push` or CI; keep generated types and schemas current.                                                                                                                                                     |
| Tests                  | Committing broken tests; running the entire slow suite in `pre-commit` so people bypass it.                                | Fast unit/smoke tests locally; full suite in CI; flaky tests quarantined or fixed, not ignored.                                                                                                                                  |
| Secrets                | Committing hardcoded tokens, passwords, private keys, or cloud credentials; assuming local scanning is enough.             | Use local scanners plus server/host push protection. GitHub says push protection can block supported hardcoded secrets before they reach the repo, and detected leaked credentials should be rotated immediately. ([GitHub][12]) |
| Commit messages        | Shared history full of `wip`, `stuff`, `changes`, `fix bug`; missing signoffs where required.                              | Conventional Commits, DCO when required, ticket IDs when useful, and clear imperative subjects. ([Conventional Commits][3])                                                                                                      |
| Huge files             | Accidentally committing binaries, archives, dumps, videos, or model files directly into Git history.                       | Enforce max file size and use Git LFS intentionally for large assets. Git LFS replaces large files with pointer files and stores the large contents remotely. ([Git Large File Storage][13])                                     |
| Generated/vendor files | Accidentally committing `node_modules/`, `.next/`, `dist/`, `build/`, coverage, caches, or vendored blobs.                 | Ignore generated output by default; allow only intentional generated artifacts with clear ownership.                                                                                                                             |
| Debug code             | Landing `console.log`, `debugger`, `pdb.set_trace`, `dbg!`, temporary prints, or “TODO remove” markers in production code. | Use lint rules or hooks to block known debug patterns, with test-only exceptions.                                                                                                                                                |
| Conflict markers       | Landing `<<<<<<<`, `=======`, `>>>>>>>`.                                                                                   | Always block in local hooks and CI. This is one of the clearest “always bad” checks.                                                                                                                                             |
| Permissions            | Accidental executable-bit changes, especially on source files.                                                             | Review mode changes; only scripts/binaries should be executable.                                                                                                                                                                 |
| Line endings           | Mixed CRLF/LF across platforms; relying on every developer’s editor settings.                                              | Normalize with `.gitattributes`, use `.editorconfig`, and use `core.safecrlf` to detect unsafe conversions. Git documents `.gitattributes` text normalization and `core.safecrlf` checks. ([Git][14])                            |
| Whitespace             | Trailing spaces, space-before-tab drift, blank-at-EOL noise in shared diffs.                                               | Use `core.whitespace`, editor trimming, and `apply.whitespace=error` for patches. Git documents both highlighting and erroring on whitespace issues. ([Git][5])                                                                  |
| Dependencies           | Changing manifests without lockfiles; ignoring audit/license failures; surprise major upgrades.                            | Require lockfile updates, dependency review, audits, and controlled Renovate/Dependabot PRs. Dependabot and Renovate both support automated dependency-update PR workflows. ([GitHub Docs][15])                                  |
| Docs                   | Broken links, stale generated docs, examples that no longer build.                                                         | Markdown lint, link checks, docs build in CI.                                                                                                                                                                                    |
| Migrations             | Irreversible or unreviewed schema changes; migrations that fail on existing production-like data.                          | Migration linting, backwards compatibility checks, rollback/forward strategy.                                                                                                                                                    |
| API/schema drift       | Changing OpenAPI, GraphQL, protobuf, or DB schema without compatibility review.                                            | Schema diff checks, generated client updates, compatibility gates.                                                                                                                                                               |
| IaC                    | Unformatted Terraform, unvalidated plans, insecure defaults.                                                               | `terraform fmt/validate`, `tflint`, `checkov`/`tfsec`, plan review in CI.                                                                                                                                                        |
| Containers             | Dockerfiles with obvious lint/security issues; image builds that only work locally.                                        | Dockerfile linting, reproducible builds, image scanning.                                                                                                                                                                         |
| License/SBOM           | Adding dependencies without license or vulnerability visibility.                                                           | License checks, SBOM generation, vulnerability scanning, provenance where required.                                                                                                                                              |

## 4. Repo-level files and configs

| File/config                              | Always bad behavior                                                                                                    | Always good / best-practice behavior                                                                                                                                                                             |
| ---------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `.gitignore`                             | Ignoring files that are already tracked and assuming Git will forget them; hiding important source/config by accident. | Keep project-specific untracked noise out of Git. Git says `.gitignore` is for intentionally untracked files and does not affect files already tracked. ([Git][16])                                              |
| `.gitattributes`                         | No line-ending or binary policy; bad diff/merge behavior for generated or binary files.                                | Normalize text, mark binaries, define custom diff/merge behavior by path. ([Git][14])                                                                                                                            |
| `.editorconfig`                          | Letting every editor choose indentation, charset, final newline, and trim rules differently.                           | Use it for shared editor behavior across IDEs/editors. EditorConfig is designed to maintain consistent coding styles across editors. ([EditorConfig][17])                                                        |
| `.pre-commit-config.yaml`                | Floating/unpinned hook versions; slow hooks everywhere; config not run in CI.                                          | Pin hook versions, scope files, keep hooks fast, and run all-files in CI periodically. ([Pre-Commit][6])                                                                                                         |
| `.husky/`                                | Large shell scripts, hidden install assumptions, Node-only tooling forced on non-Node contributors.                    | Small scripts that call documented package commands; pair with CI. ([Typicode][7])                                                                                                                               |
| `lefthook.yml`                           | Parallel mutating jobs racing on the same files; undocumented local dependencies.                                      | Explicit file scopes, independent parallel jobs, portable commands. ([GitHub][9])                                                                                                                                |
| `.lintstagedrc` / lint-staged config     | Overlapping mutating globs; commands that lint the entire repo while pretending to lint staged files.                  | Staged-file-only commands, sequential arrays for mutating tasks, monorepo-aware configs. ([GitHub][8])                                                                                                           |
| `commitlint.config.*`                    | Rules that reject normal workflow commits like `revert:` or `fixup!`; rules not documented in `CONTRIBUTING.md`.       | Conventional Commit rules aligned with release automation and documented exceptions.                                                                                                                             |
| `.gitmessage`                            | Giant templates people ignore; templates containing sensitive internal info.                                           | Short message template: subject style, body guidance, issue footer, DCO reminder. Git supports commit templates via `commit.template`. ([Git][5])                                                                |
| `.mailmap`                               | Letting author identities fragment across old emails and aliases.                                                      | Canonicalize names/emails for logs, blame, and contribution summaries. Git documents `.mailmap` for canonical author/committer identities. ([Git][18])                                                           |
| `.git-blame-ignore-revs`                 | Blaming mass-formatting commits forever; hiding functional changes in the ignore list.                                 | Add only mechanical reformat/move commits; configure blame to ignore them. Git blame supports ignored revision files. ([Git][19])                                                                                |
| `.lfsconfig` + `.gitattributes`          | Tracking large files after they already polluted history; using LFS without documenting install/auth.                  | Track large assets before commit, document LFS setup, and migrate existing large blobs when needed. ([Git Large File Storage][13])                                                                               |
| `CODEOWNERS`                             | Stale owners, broad wildcard ownership, missing sensitive paths.                                                       | Require domain-owner review for critical paths. GitHub uses CODEOWNERS on the base branch to request review for matching paths. ([GitHub Docs][20])                                                              |
| `.github/workflows/*`                    | Optional CI, ambiguous job names, secrets in logs, workflows that only run after merge.                                | Required CI jobs with unique names, PR triggers, least-privilege tokens, caches, and clear artifacts. GitHub Actions defines repo workflows for automation/CI/CD. ([GitHub Docs][21])                            |
| `.gitlab-ci.yml`                         | Pipelines that do not run on merge requests; jobs that only pass on one developer’s machine.                           | Define stages/jobs/scripts in `.gitlab-ci.yml`, run MR pipelines, and make required jobs authoritative. GitLab documents `.gitlab-ci.yml` as the file that defines CI/CD jobs and pipelines. ([GitLab Docs][22]) |
| `bitbucket-pipelines.yml`                | Pipelines that are not required by merge checks; no reproducible build container.                                      | Use Bitbucket Pipelines for build/test/deploy automation and enforce merge checks. Atlassian documents `bitbucket-pipelines.yml` as the repository pipeline config file. ([Atlassian Support][23])               |
| `renovate.json` / `dependabot.yml`       | No dependency automation; unreviewed auto-merges of risky major updates.                                               | Scheduled dependency PRs, grouping rules, lockfile maintenance, security update policy. ([GitHub Docs][15])                                                                                                      |
| `SECURITY.md`                            | No vulnerability reporting path; asking reporters to open public issues for secrets/vulns.                             | Clear private reporting instructions and supported versions. GitHub documents `SECURITY.md` for vulnerability reporting instructions. ([GitHub Docs][24])                                                        |
| `CONTRIBUTING.md`                        | Hidden contribution rules enforced only by hooks or tribal knowledge.                                                  | Explain setup, commit format, tests, branch policy, review expectations, and security rules.                                                                                                                     |
| `LICENSE`                                | No license in public/open-source repo; inconsistent license headers.                                                   | Clear legal terms and compatible dependency policy.                                                                                                                                                              |
| `Makefile` / `justfile` / `Taskfile.yml` | Many undocumented commands that differ from CI.                                                                        | One-command local parity: `format`, `lint`, `test`, `typecheck`, `security`, `ci`.                                                                                                                               |

## 5. Git config hardening

| Config                   | Always bad behavior                                                                                                          | Always good / best-practice behavior                                                                                                                                                                                           |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `core.hooksPath`         | Silently overriding hooks globally; pointing to an unversioned path; disabling hooks via `/dev/null` in shared environments. | Use a checked-in path such as `.githooks` for local feedback, with bootstrap docs and CI/server enforcement. Git documents `core.hooksPath` and notes it can redirect hook lookup. ([Git][5])                                  |
| `commit.template`        | Global template that conflicts with repo conventions; huge template noise.                                                   | Short project-specific template aligned with commitlint/release rules. ([Git][5])                                                                                                                                              |
| `commit.gpgSign`         | Requiring signing without onboarding keys, bots, rebases, or CI automation.                                                  | Require signed commits only where identity verification matters, and provide key/bot setup. Git supports signing all commits with `commit.gpgSign`. ([Git][5])                                                                 |
| `push.default=simple`    | Using unsafe broad defaults that push more branches than intended.                                                           | `simple` is Git’s safer default for normal use; `nothing` is stricter and forces explicit refspecs. Git documents `simple` as the default and safest for beginners, and `nothing` as refusing push without refspec. ([Git][5]) |
| `push.default=matching`  | Bad in shared central repos because it can push every matching branch, including unfinished local branches.                  | Avoid for teams unless the workflow is intentionally designed around it. Git’s docs warn `matching` is generally unsuitable for central shared repositories. ([Git][5])                                                        |
| `core.safecrlf`          | Disabling CRLF safety while contributors use different OS/editor settings.                                                   | Use `.gitattributes` first, then `core.safecrlf=warn` or `true` depending on strictness. ([Git][5])                                                                                                                            |
| `core.whitespace`        | Letting whitespace noise pollute diffs.                                                                                      | Highlight trailing spaces, space-before-tab, and blank-at-EOL issues. ([Git][5])                                                                                                                                               |
| `apply.whitespace=error` | Accepting patches with obvious whitespace errors.                                                                            | Reject whitespace-bad patches when applying mail/patch workflows. ([Git][5])                                                                                                                                                   |

## 6. Server-side and repo-host enforcement

| Layer                                        | Always bad behavior                                                                                                     | Always good / best-practice behavior                                                                                                                                                                                                                                                                                        |
| -------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Protected branches                           | Direct pushes to `main`, `release/*`, or production branches; allowing deletion or force-pushes casually.               | Protect important branches, block direct pushes, block deletes/force-pushes, and apply rules to admins unless there is a documented break-glass path. GitHub documents protected-branch settings for PRs, checks, signed commits, linear history, merge queue, deployments, force pushes, and deletion. ([GitHub Docs][25]) |
| Required PR reviews                          | Self-approval; stale approvals after major changes; no review for sensitive paths.                                      | Require at least one or two approvals, dismiss stale approvals when appropriate, and require CODEOWNER approval for critical areas. ([GitHub Docs][25])                                                                                                                                                                     |
| CODEOWNERS                                   | Missing ownership for auth, crypto, billing, infrastructure, migrations, or deployment paths.                           | Map sensitive paths to accountable maintainers and require their review.                                                                                                                                                                                                                                                    |
| Required status checks                       | Optional CI; ambiguous duplicate job names; allowing any app/user to spoof a status.                                    | Require uniquely named checks from expected sources, and require them before merge. GitHub explicitly recommends unique job names and supports expected-source status checks in rulesets. ([GitHub Docs][25])                                                                                                               |
| Required signed commits                      | Requiring signatures but leaving bots, squash merges, or rebase workflows broken.                                       | Require verified signatures on protected branches only after tooling and bot identities are ready. ([GitHub Docs][25])                                                                                                                                                                                                      |
| Linear history                               | Forcing linear history without teaching rebase/squash workflows; destroying useful merge context in repos that need it. | Use when the repo values clean revertability and simple bisect history. GitHub notes linear history can make reverting changes easier. ([GitHub Docs][25])                                                                                                                                                                  |
| Merge queue                                  | Allowing stale green PRs to race into a busy `main`.                                                                    | Use on high-traffic protected branches so the branch stays green after integration. GitHub says merge queues help ensure the branch is never broken by incompatible changes. ([GitHub Docs][25])                                                                                                                            |
| Secret push protection                       | Disabled push protection; easy bypass without audit; treating alerts as enough after secrets land.                      | Enable push protection, custom patterns where needed, delegated bypass, alerting, audit logs, and immediate credential rotation. ([GitHub Docs][26])                                                                                                                                                                        |
| Code scanning gate                           | Security tools that only produce ignored warnings; merge allowed while analysis is pending.                             | Block merges for required severities/tools and require analysis to complete. GitHub code-scanning rulesets can prevent merges when alerts exist, analysis is pending, or required tools are missing. ([GitHub Docs][27])                                                                                                    |
| Server-side hooks                            | Long, flaky scripts on every push; unclear rejection messages; policy that differs from CI.                             | Use `pre-receive`, `update`, or `proc-receive` for fast, deterministic hard policy on self-hosted Git/GitLab/GHES. GitLab and GitHub Enterprise both document server-side hooks for custom push policy. ([GitLab Docs][28])                                                                                                 |
| GitLab approval rules / Code Owners          | Merge requests that bypass domain experts or external checks.                                                           | Use approval rules, protected branches, Code Owners, and external status checks for sensitive paths. ([GitLab Docs][29])                                                                                                                                                                                                    |
| Bitbucket branch restrictions / merge checks | Allowing merges with failed builds, unresolved tasks, or no approvals.                                                  | Use required approvals, required builds/checks, unresolved-task gates, CODEOWNERS/custom merge checks where available. ([Atlassian Support][30])                                                                                                                                                                            |

## 7. Strong “always bad” rules worth writing into policy

These are the clearest universal bad behaviors:

| Always bad                                                 | Correct policy                                                           |
| ---------------------------------------------------------- | ------------------------------------------------------------------------ |
| Secrets committed to Git history                           | Block locally and server-side; rotate immediately if detected.           |
| Conflict markers in committed files                        | Block everywhere.                                                        |
| Direct pushes to protected production branches             | Require PR/MR, checks, review, and audit trail.                          |
| Force-pushing or deleting protected branches/tags casually | Disable by default; require explicit admin break-glass.                  |
| Treating local hooks as security enforcement               | Mirror in CI/server.                                                     |
| Slow hooks that make `--no-verify` normal                  | Keep local hooks fast; move slow checks to `pre-push` or CI.             |
| Generated/vendor/large files committed accidentally        | Ignore, size-check, or use LFS intentionally.                            |
| Unreviewed auth/security/infra/migration changes           | CODEOWNERS + required review + required CI.                              |
| Optional CI on shared branches                             | Required status checks.                                                  |
| Vague shared-history commits                               | Commit-message policy, especially when releases/changelogs depend on it. |

## 8. Strong “always good” defaults

For most serious repos, the consistently good behavior is:

```text
Local fast feedback:
- pre-commit or Lefthook
- format/lint/secret scan/conflict-marker/large-file checks
- commit-msg validation

Before push:
- typecheck
- build
- fast tests
- dependency/security checks when reasonable

Authoritative enforcement:
- protected main/release branches
- required PR/MR reviews
- CODEOWNERS for sensitive paths
- required status checks
- secret push protection
- code scanning gate where available
- merge queue for busy repos
- server-side hooks for self-hosted/high-risk repos

Repo hygiene:
- .gitignore
- .gitattributes
- .editorconfig
- .git-blame-ignore-revs
- LFS policy
- SECURITY.md
- CONTRIBUTING.md
- dependency update automation
```

The best mental model is: **hooks prevent wasted time; CI and branch/server policy prevent bad code from landing.**
