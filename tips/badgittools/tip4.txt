I’m interpreting **“BAD behavior”** as anti-patterns in how the hook/tool/config is used, and **“GOOD behavior”** as best-practice usage. “Always good” here means “good when you choose to use that layer,” not “every repo must use every tool.”

The sources mostly align on one big rule: **local hooks are for fast feedback, not final enforcement**. Git’s own docs say client-side hooks are not copied when a repository is cloned, and hooks such as `pre-commit`, `commit-msg`, and `pre-push` can be bypassed with `--no-verify`; Git’s server-side hooks and repo-host controls are the enforcement layer. ([Git][1]) Reddit/X/community discussions echo the same operational lesson: slow or surprising hooks get bypassed, while CI/server-side checks are where policy should stick. ([Reddit][2])

## 1. Universal rules across all Git quality layers

| Area                 | BAD behavior — always bad                                                                                 | GOOD behavior — best practice                                                                                        |
| -------------------- | --------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| Enforcement model    | Relying only on local hooks to protect `main` or releases.                                                | Use hooks for fast feedback, then enforce the same policy in CI, protected branches, rulesets, or server-side hooks. |
| Bypass culture       | Treating `--no-verify` as a normal workflow shortcut.                                                     | Allow bypass only for exceptional cases, and make CI catch whatever was skipped locally.                             |
| Hook speed           | Running huge test suites, long integration tests, network-heavy scanners, or full builds on every commit. | Keep `pre-commit` fast; move heavier checks to `pre-push` or CI.                                                     |
| Developer experience | Hooks that change unstaged files, hide errors, delete commit messages, or require obscure local setup.    | Hooks should be deterministic, clear, fast, and easy to install.                                                     |
| Tool drift           | Different developers run different formatter/linter versions.                                             | Pin tool versions in config, lockfiles, or hook manager config.                                                      |
| “Fix later” commits  | Letting secrets, conflict markers, broken tests, or generated junk land because “CI will catch it.”       | Prevent known-bad patterns locally and block them authoritatively before merge.                                      |
| Policy visibility    | Enforcing hidden rules that are not documented.                                                           | Put expectations in `CONTRIBUTING.md`, hook configs, CI, and branch/ruleset settings.                                |

## 2. Native Git hooks

Git’s official hook docs are the source of truth for what each hook can block. Hooks that run before an action can usually abort it with a non-zero exit; post-action hooks generally cannot affect the outcome. ([Git][3])

| Hook                    | BAD behavior — always bad                                                                                                                        | GOOD behavior — best practice                                                                                           |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------- |
| `pre-commit`            | Running slow full builds, hitting external services, checking unstaged work by accident, or treating it as authoritative.                        | Check only staged content for quick issues: formatting, lint, secrets, conflict markers, large files, debug statements. |
| `prepare-commit-msg`    | Using it as a replacement for `pre-commit` or silently rewriting user intent. Git explicitly says it should not replace `pre-commit`. ([Git][3]) | Insert helpful metadata such as ticket IDs or branch names while preserving the author’s message.                       |
| `commit-msg`            | Rejecting valid Git-generated messages without exceptions, or enforcing vague rules nobody understands.                                          | Enforce a documented convention: Conventional Commits, DCO trailers, ticket IDs, subject length, no junk messages.      |
| `pre-merge-commit`      | Assuming it runs for all merge scenarios. Git notes it does not run when conflicts are resolved and committed separately. ([Git][3])             | Reuse fast `pre-commit` checks before automatic merge commits.                                                          |
| `post-commit`           | Trying to block a commit after it already happened.                                                                                              | Use for notifications, local metrics, reminders, or non-blocking automation.                                            |
| `pre-rebase`            | Blocking all rebases blindly, including safe local cleanup.                                                                                      | Prevent rebasing protected/public branches; warn before rewriting shared history.                                       |
| `post-checkout`         | Modifying tracked files unexpectedly after checkout.                                                                                             | Restore local toolchains, warn about missing environment, refresh non-tracked caches.                                   |
| `post-merge`            | Assuming it can block a bad merge.                                                                                                               | Reinstall dependencies, regenerate local artifacts, warn after lockfile changes.                                        |
| `post-rewrite`          | Assuming it catches all history rewrites. Git notes full-history rewrite tools may not call it. ([Git][3])                                       | Warn after amend/rebase, update local metadata, refresh notes or caches.                                                |
| `post-index-change`     | Running expensive scans every time the index changes.                                                                                            | Use sparingly for lightweight index-related reactions.                                                                  |
| `pre-push`              | Running flaky, very long, or network-dependent jobs that make people avoid pushing.                                                              | Run medium-weight checks: typecheck, targeted tests, build smoke test, secret scan.                                     |
| `pre-receive`           | Using vague rejection messages or fragile scripts that block legitimate emergency work without an escape process.                                | Hard-block server-wide violations: secrets, unsigned commits, forbidden paths, invalid refs, protected branch policy.   |
| `update`                | Duplicating complex global logic per ref and creating inconsistent branch behavior.                                                              | Enforce per-ref policy: fast-forward-only, no tag rewrites, branch naming, release branch restrictions.                 |
| `proc-receive`          | Using it casually for simple rules that branch protection could handle.                                                                          | Use only for advanced server-side reference workflows.                                                                  |
| `post-receive`          | Treating it as a gate. It runs after accepted updates.                                                                                           | Deploy, notify, mirror, index, or trigger follow-up systems after a successful push.                                    |
| `post-update`           | Using it when you need old and new object IDs; Git says `post-receive` is better for that. ([Git][3])                                            | Use for simple after-update notifications or `update-server-info` style tasks.                                          |
| `reference-transaction` | Ignoring transaction states and accidentally blocking committed/aborted states incorrectly.                                                      | Use for auditable ref policy; only block in preparing/prepared states when appropriate.                                 |
| `push-to-checkout`      | Mutating a deploy worktree unsafely or ignoring dirty worktrees.                                                                                 | Make push-to-deploy updates explicit, safe, and reversible.                                                             |
| `pre-auto-gc`           | Blocking garbage collection forever.                                                                                                             | Delay GC only for a known temporary reason.                                                                             |
| `fsmonitor-watchman`    | Missing changed files; that can make Git believe the worktree is cleaner than it is.                                                             | Be conservative: include all possibly changed paths; false positives are safer than false negatives.                    |
| `applypatch-msg`        | Letting email patches bypass commit-message policy.                                                                                              | Reuse `commit-msg` rules for patch workflows.                                                                           |
| `pre-applypatch`        | Applying patches without the same quality checks as normal commits.                                                                              | Reuse `pre-commit` checks after patch application and before commit creation.                                           |
| `post-applypatch`       | Trying to reject an already-created patch commit.                                                                                                | Notify or update local metadata only.                                                                                   |
| `sendemail-validate`    | Sending patch series that cannot apply cleanly or violate basic policy.                                                                          | Validate recipients, metadata, and applyability before `git send-email`.                                                |
| `p4-changelist`         | Letting Git-p4 changelist messages bypass project policy.                                                                                        | Normalize or reject bad P4 changelist text.                                                                             |
| `p4-prepare-changelist` | Silently replacing user-authored changelist content.                                                                                             | Add helpful template metadata before editing.                                                                           |
| `p4-post-changelist`    | Trying to block a P4 submit after it occurred.                                                                                                   | Notify or sync metadata after successful submit.                                                                        |
| `p4-pre-submit`         | Skipping quality checks before submitting to Perforce.                                                                                           | Block known-bad P4 submits before launch.                                                                               |

## 3. Hook distribution tools

| Tool/setup                                 | BAD behavior — always bad                                                                                                                                | GOOD behavior — best practice                                                                                                                                                                                                                                                            |
| ------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Raw `.git/hooks/*`                         | Assuming the scripts are shared with clones. Git hooks live in `.git/hooks` by default and are not copied on clone. ([Git][1])                           | Use only for personal one-off automation.                                                                                                                                                                                                                                                |
| Checked-in `.githooks/` + `core.hooksPath` | Forgetting that every clone still needs `git config core.hooksPath .githooks`, or breaking user/global hooks without warning.                            | Commit the hook directory, document setup, make scripts executable, keep them portable. Git supports changing the hook directory through `core.hooksPath`. ([Git][3])                                                                                                                    |
| `init.templateDir`                         | Thinking it updates existing repos or enforces team policy. Templates seed hooks at init/clone time; they are not a central enforcement mechanism.       | Use for your personal defaults or org bootstrap templates.                                                                                                                                                                                                                               |
| `pre-commit`                               | Not running `pre-commit install`; pinning nothing; putting slow jobs in `pre-commit`; failing to mirror checks in CI.                                    | Use `.pre-commit-config.yaml`, pin hook versions, install needed hook types, run `pre-commit run --all-files` in CI or pre-commit.ci. pre-commit’s docs say every clone should run `pre-commit install`, and `run --all-files` checks the repository manually/CI-wide. ([Pre-Commit][4]) |
| Husky                                      | Installing Husky but not committing `.husky/` or not wiring package-manager install scripts; making Node-only hooks mandatory for non-Node contributors. | Best for JS/TS repos; keep hooks thin and call package scripts. Husky uses Git `core.hooksPath` and supports client-side Git hooks. ([Typicode][5])                                                                                                                                      |
| lint-staged                                | Assuming it checks the whole repo; misconfiguring it so unstaged changes get committed; running tasks that are not file-list aware.                      | Use it for staged-file format/lint only, then run full checks in CI. lint-staged is designed to run shell tasks against staged files filtered by globs. ([GitHub][6])                                                                                                                    |
| Lefthook                                   | Enabling parallel jobs that race on the same files; treating speed as a substitute for enforcement.                                                      | Use for fast polyglot repos/monorepos, especially when independent checks can run in parallel. Lefthook is a Git hook manager that can run commands in parallel. ([GitHub][7])                                                                                                           |
| Overcommit                                 | Adding Ruby dependency friction to non-Ruby repos, or failing when GUI clients do not have the right Ruby environment.                                   | Good for Ruby-heavy repos that want configurable, source-controlled hook behavior. Overcommit manages and configures Git hooks and supports repository-specific hooks stored in source control. ([GitHub][8])                                                                            |
| CaptainHook                                | Using it outside a PHP/Composer ecosystem where nobody has the runtime.                                                                                  | Good for PHP teams that want Composer-friendly hook management for messages, tests, and quality checks. CaptainHook documents commit-message validation, tests, code quality checks, and post-pull tasks. ([GitHub][9])                                                                  |

## 4. Checks worth adding

| Check                  | BAD behavior — always bad                                                                                     | GOOD behavior — best practice                                                                                                                                                                                                                                                              |
| ---------------------- | ------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Formatters             | Arguing about formatting in review; allowing formatter drift between machines.                                | Auto-format locally; check formatting in CI.                                                                                                                                                                                                                                               |
| Linters                | Blocking on thousands of legacy warnings without a baseline; disabling rules silently.                        | Start with high-signal rules; baseline legacy debt; fail only on new violations.                                                                                                                                                                                                           |
| Type checks            | Skipping type checks because tests pass.                                                                      | Run fast type checks in `pre-push` or CI; optionally staged/light checks in `pre-commit`.                                                                                                                                                                                                  |
| Tests                  | Running the entire slow suite on every commit; allowing flaky tests to block randomly.                        | Fast unit/smoke tests locally; full deterministic suite in CI.                                                                                                                                                                                                                             |
| Secret scanning        | Treating a scanner as cleanup after secrets leak; committing real `.env`, tokens, keys, or cloud credentials. | Scan before commit, before push, and in CI/host push protection. Gitleaks detects secrets like passwords/API keys/tokens; git-secrets prevents committing sensitive information; TruffleHog finds/verifies leaked credentials; detect-secrets supports a baseline workflow. ([GitHub][10]) |
| Commit messages        | Accepting `wip`, `fix stuff`, blank messages, or misleading messages.                                         | Use a clear convention. Conventional Commits defines `<type>[optional scope]: <description>` and supports SemVer/changelog automation; commitlint checks commit messages against that format. ([Conventional Commits][11])                                                                 |
| DCO / sign-off         | Faking or copying `Signed-off-by` lines; requiring DCO but not checking it.                                   | Require a DCO trailer only when the project uses DCO, and enforce it consistently. The DCO is a contributor certification that they have the right to submit the contribution. ([Developer Certificate][12])                                                                               |
| Huge files             | Committing binaries, datasets, videos, archives, or model files directly into normal Git history.             | Use Git LFS or external artifact storage. Git LFS replaces large files with pointer files while storing content on a remote server. ([Git Large File Storage][13])                                                                                                                         |
| Generated/vendor files | Committing `node_modules`, `.next`, `dist`, vendored blobs, or generated code unintentionally.                | Ignore generated output unless it is intentionally versioned and reviewed.                                                                                                                                                                                                                 |
| Debug code             | Committing `debugger`, `pdb.set_trace`, noisy `console.log`, temporary flags, or test-only credentials.       | Block obvious debug patterns; allow documented exceptions.                                                                                                                                                                                                                                 |
| Conflict markers       | Committing `<<<<<<<`, `=======`, `>>>>>>>`.                                                                   | Always block conflict markers locally and in CI.                                                                                                                                                                                                                                           |
| Executable bits        | Accidentally making source files executable or scripts non-executable.                                        | Check permission changes and require intent.                                                                                                                                                                                                                                               |
| Line endings           | Letting CRLF/LF churn dominate diffs.                                                                         | Normalize with `.gitattributes`; Git attributes can normalize text line endings to LF in the index. ([Git][14])                                                                                                                                                                            |
| Whitespace             | Committing trailing whitespace, tabs/spaces drift, blank-at-EOF churn.                                        | Use `.editorconfig`, formatter, Git whitespace settings, and CI checks.                                                                                                                                                                                                                    |
| Dependency changes     | Changing dependency manifests without lockfiles; committing lockfile changes without review.                  | Require manifest + lockfile consistency, audit checks, and security review for risky updates.                                                                                                                                                                                              |
| Docs                   | Merging broken docs links, invalid Markdown, or docs that do not build.                                       | Lint Markdown and build docs in CI.                                                                                                                                                                                                                                                        |
| Migrations             | Merging irreversible or unsafe migrations without review.                                                     | Lint migrations; require backwards-compatible migration checks for production systems.                                                                                                                                                                                                     |
| API/schema drift       | Changing OpenAPI, GraphQL, protobuf, DB schemas, or public APIs without compatibility checks.                 | Gate breaking schema/API changes and require explicit `BREAKING CHANGE` metadata.                                                                                                                                                                                                          |
| IaC                    | Merging unformatted or risky Terraform/Kubernetes/cloud config.                                               | Run `terraform fmt/validate`, policy checks, and security scans.                                                                                                                                                                                                                           |
| Containers             | Shipping unlinted Dockerfiles, root containers, floating base images, or known CVEs.                          | Lint Dockerfiles and scan images.                                                                                                                                                                                                                                                          |
| License/SBOM           | Adding dependencies without license/security visibility.                                                      | Generate SBOMs and check licenses/vulnerabilities for release paths.                                                                                                                                                                                                                       |

## 5. Repo-level files and configs

| File/config                              | BAD behavior — always bad                                                                                                                                                                                                 | GOOD behavior — best practice                                                                                                                                                                            |
| ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `.gitignore`                             | Using `.gitignore` to hide files that are already tracked; ignoring source files accidentally; putting personal editor junk in the shared ignore. Git notes ignore rules do not affect already tracked files. ([Git][15]) | Commit shared project ignores; use global ignore or `.git/info/exclude` for personal junk. GitHub says committed `.gitignore` shares ignore rules with everyone who clones the repo. ([GitHub Docs][16]) |
| `.gitattributes`                         | Omitting it in cross-platform repos; misclassifying binary files as text; causing line-ending churn.                                                                                                                      | Normalize text, mark binaries, configure diff/merge behavior, and commit LFS tracking rules.                                                                                                             |
| `.editorconfig`                          | Treating editor settings as personal taste in team repos.                                                                                                                                                                 | Use it for basic indentation, charset, final newline, and trim rules. EditorConfig is designed to maintain consistent coding styles across editors/IDEs. ([EditorConfig][17])                            |
| `.pre-commit-config.yaml`                | Floating `rev: main`; adding slow/flaky hooks; not running it in CI.                                                                                                                                                      | Pin versions, keep hooks fast, and mirror them in CI.                                                                                                                                                    |
| `.husky/`                                | Committing hooks that assume a single shell/OS or missing package manager setup.                                                                                                                                          | Keep scripts tiny and call `npm/yarn/pnpm` scripts.                                                                                                                                                      |
| `lefthook.yml`                           | Racing parallel fixers on the same file.                                                                                                                                                                                  | Use parallelism for independent checks; serialize mutating fixers.                                                                                                                                       |
| `.lintstagedrc`                          | Believing staged-file checks replace full-repo checks.                                                                                                                                                                    | Use staged checks for speed; full checks in CI.                                                                                                                                                          |
| `commitlint.config.*`                    | Enforcing types/scopes not documented anywhere.                                                                                                                                                                           | Document allowed types/scopes and keep CI + `commit-msg` aligned.                                                                                                                                        |
| `.gitmessage`                            | Making templates so long that nobody reads them.                                                                                                                                                                          | Provide a short helpful template; Git supports commit templates via `commit.template`. ([Git][18])                                                                                                       |
| `.mailmap`                               | Rewriting identity carelessly or hiding authorship confusion.                                                                                                                                                             | Canonicalize known author/email variants.                                                                                                                                                                |
| `.git-blame-ignore-revs`                 | Hiding real logic changes from blame.                                                                                                                                                                                     | Put only mechanical formatting/mass-move commits in it.                                                                                                                                                  |
| `.lfsconfig` + `.gitattributes`          | Tracking large files inconsistently across clones.                                                                                                                                                                        | Commit `.gitattributes`; GitHub’s LFS docs strongly suggest committing the file that stores LFS patterns. ([GitHub Docs][19])                                                                            |
| `CODEOWNERS`                             | Assigning dead teams, missing critical paths, or assuming CODEOWNERS works without branch protection.                                                                                                                     | Keep owners current and require code-owner review for sensitive paths. GitHub requires the CODEOWNERS file on the base branch for review requests. ([GitHub Docs][20])                                   |
| `.github/workflows/*`                    | Letting PRs edit their own required checks without owner review; using unpinned third-party actions in high-risk repos.                                                                                                   | Define required CI workflows, protect workflow files, pin permissions, and minimize secrets exposure. GitHub Actions workflows are YAML-defined automated jobs. ([GitHub Docs][21])                      |
| `.gitlab-ci.yml`                         | Running non-required pipelines that can fail without blocking merge.                                                                                                                                                      | Make required pipelines/status checks block merge.                                                                                                                                                       |
| `bitbucket-pipelines.yml`                | Treating pipeline results as informational only.                                                                                                                                                                          | Combine pipelines with required merge checks.                                                                                                                                                            |
| `renovate.json` / `dependabot.yml`       | Auto-merging dependency updates without tests/security policy.                                                                                                                                                            | Automate updates but require CI, changelog review, and security rules.                                                                                                                                   |
| `SECURITY.md`                            | Missing a private vulnerability reporting path.                                                                                                                                                                           | Provide clear reporting and disclosure process.                                                                                                                                                          |
| `CONTRIBUTING.md`                        | Letting hidden norms live only in senior engineers’ heads.                                                                                                                                                                | Document branch, commit, review, hook, CI, and release rules.                                                                                                                                            |
| `LICENSE`                                | Shipping code with unclear legal terms.                                                                                                                                                                                   | Commit a clear license before external distribution.                                                                                                                                                     |
| `Makefile` / `justfile` / `Taskfile.yml` | Having hooks call commands that humans cannot run manually.                                                                                                                                                               | Provide one-command local quality gates like `make check`, `just ci`, or `task test`.                                                                                                                    |

## 6. Git config hardening

| Config                   | BAD behavior — always bad                                                                                                                                                                                                               | GOOD behavior — best practice                                                       |
| ------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| `core.hooksPath`         | Setting it globally in a way that unexpectedly disables repo-specific hooks.                                                                                                                                                            | Use repo-local `core.hooksPath` when a project intentionally ships shared hooks.    |
| `commit.template`        | A template that encourages boilerplate but no substance.                                                                                                                                                                                | Short template with subject/body/footer reminders.                                  |
| `commit.gpgSign`         | Requiring signing but not helping developers configure keys; signing rewritten commits without an agent can create friction. Git notes signing during rebase may require repeated passphrase entry unless an agent is used. ([Git][18]) | Use commit signing when identity assurance matters; document GPG/SSH signing setup. |
| `push.default=simple`    | Using dangerous broad push behavior by default.                                                                                                                                                                                         | `simple` is a safer default for most users.                                         |
| `push.default=nothing`   | Mandating it without teaching explicit refspecs.                                                                                                                                                                                        | Good for high-risk users/repos where accidental pushes are costly.                  |
| `core.safecrlf`          | Ignoring CRLF warnings in cross-platform repos.                                                                                                                                                                                         | Warn or fail on unsafe conversions, backed by `.gitattributes`.                     |
| `core.whitespace`        | Letting whitespace churn pollute reviews.                                                                                                                                                                                               | Highlight trailing spaces, blank-at-EOL, and tab issues.                            |
| `apply.whitespace=error` | Applying patches with known whitespace errors.                                                                                                                                                                                          | Reject patch whitespace errors when the project requires clean diffs.               |

## 7. Server-side and repo-host enforcement

This is the layer that actually prevents bad commits from landing on protected branches. GitHub branch protection can require status checks, reviews, signed commits, linear history, and control force-push/delete behavior. ([GitHub Docs][22]) GitHub rulesets can control interactions with branches/tags and allow explicitly defined bypassers; GitHub push protection blocks supported hardcoded secrets before they reach the repository. ([GitHub Docs][23]) GitLab supports approval rules/code owners and external status checks that can block merges; Bitbucket supports required merge checks/custom merge checks. ([GitLab Docs][24])

| Enforcement layer       | BAD behavior — always bad                                                               | GOOD behavior — best practice                                                                                                                                                                         |
| ----------------------- | --------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Protected branches      | Allowing direct pushes to `main`, `release/*`, or production branches.                  | Require PR/MR flow, block deletion/force-push, and apply rules to admins unless there is a controlled break-glass path.                                                                               |
| Required PR reviews     | Letting authors self-approve risky changes; treating rubber-stamp review as compliance. | Require meaningful review; increase approvals for risky code.                                                                                                                                         |
| CODEOWNERS              | Owners who are unavailable, too broad, or stale.                                        | Require domain-owner review for security, infra, auth, billing, migrations, releases, and CI files.                                                                                                   |
| Required status checks  | Checks that are optional, flaky, or do not run on the exact merge commit.               | Require deterministic lint/test/type/security/build checks on the mergeable state.                                                                                                                    |
| Required signed commits | Requiring signing but allowing unsigned merge/squash paths.                             | Enforce consistently across all merge methods used by the repo.                                                                                                                                       |
| Linear history          | Enabling it without agreeing on merge strategy.                                         | Use linear history only when the team wants rebase/squash workflows.                                                                                                                                  |
| Merge queue             | Merging stale green PRs that break when combined.                                       | Use merge queue for busy repos where multiple PRs race to `main`.                                                                                                                                     |
| Secret push protection  | Relying only on after-the-fact alerts.                                                  | Enable push protection plus local/CI secret scanning.                                                                                                                                                 |
| Code scanning gate      | Treating security findings as advisory on protected branches.                           | Block merges for required high-confidence alerts or pending required analysis. GitHub code scanning merge protection can prevent PRs with potential vulnerabilities from merging. ([GitHub Docs][25]) |
| Server-side hooks       | Writing opaque scripts that reject pushes without actionable messages.                  | Use for self-hosted Git hard policy: no secrets, no non-fast-forward updates, no forbidden refs/tags, no invalid commit metadata.                                                                     |

## 8. Commit behaviors themselves

| Commit behavior  | BAD behavior — always bad                                                                                         | GOOD behavior — best practice                                                                                               |
| ---------------- | ----------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| Commit size      | Huge “everything” commits mixing refactor, feature, formatting, dependency updates, and migrations.               | Small cohesive commits with one purpose.                                                                                    |
| Commit message   | `wip`, `stuff`, `changes`, `fix`, blank messages, misleading summaries.                                           | Imperative, specific subject; body explains why; footer carries tickets/breaking changes/DCO.                               |
| Formatting       | Formatting mixed with logic changes without separation.                                                           | Separate mechanical formatting from behavior changes, then add formatter commit to `.git-blame-ignore-revs` if appropriate. |
| Secrets          | Any real credential, token, private key, `.env`, production URL with credentials, or cloud secret in Git history. | Use secret managers; commit templates like `.env.example`; rotate immediately if leaked.                                    |
| Generated output | Generated blobs committed accidentally.                                                                           | Commit generated output only when the repo intentionally versions it and CI verifies it is current.                         |
| Dependencies     | Dependency update hidden inside feature work.                                                                     | Separate dependency changes and include lockfile/test results.                                                              |
| Public history   | Force-pushing shared branches casually.                                                                           | Rewrite only private/local history; protect shared branches.                                                                |
| Emergency fixes  | Bypassing all checks silently.                                                                                    | Use a documented break-glass path and create follow-up remediation.                                                         |
| Review           | “Looks good” without reading risky diffs.                                                                         | Review tests, security impact, migrations, rollback, and ownership-sensitive files.                                         |

## 9. A clean default policy

For most serious repos, the aligned best-practice stack is:

```text
Local fast feedback:
  pre-commit or Lefthook
  formatter
  linter
  secret scan
  conflict-marker / large-file / debug-code checks
  commit-msg validation

Before push:
  typecheck
  targeted tests
  build smoke test

CI / repo host enforcement:
  full tests
  lint/type/build/docs/security scans
  protected main
  required status checks
  required PR review
  CODEOWNERS for sensitive paths
  secret push protection
  merge queue for busy repos
```

The core rule is simple: **hooks should make the right thing easy; CI and protected branches should make the wrong thing impossible to merge.**
