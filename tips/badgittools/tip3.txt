Below is a “bad behavior / good behavior” matrix for the Git layers you listed. I treated official docs as authoritative, and Reddit/GitHub discussions as community signals: they strongly align on the same theme—**local hooks are useful guardrails, but bad as the only gate because they can be missing, slow, or bypassed; real enforcement belongs in CI, branch protection/rulesets, or server-side hooks.** Git’s own book says client-side hooks are not copied on clone and recommends server-side enforcement for policy; it also notes `pre-commit` can be bypassed with `--no-verify`. ([Git][1]) Reddit and GitHub discussions echo the same failure mode: slow hooks lead people to bypass them, and repeating checks in CI is the safer pattern. ([Reddit][2])

## 0. Universal rule

| Layer                   | Always bad behavior                                                                                      | Always good behavior                                                                                                                   |
| ----------------------- | -------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| Local hooks             | Treating local hooks as authoritative enforcement. They are developer feedback, not a security boundary. | Use local hooks for fast feedback, then repeat critical checks in CI or server-side controls.                                          |
| CI                      | Making CI optional, flaky, or easy to ignore.                                                            | Require CI status checks before merging protected branches.                                                                            |
| Server/repo host policy | Allowing direct pushes, force-pushes, or admin bypasses on protected production branches without audit.  | Enforce protected branches, required reviews, required status checks, signed commits where needed, and merge queues for busy branches. |
| Exceptions              | Normalizing `--no-verify`, blanket allowlists, “temporary” disabling of checks, or unreviewed bypasses.  | Make exceptions explicit, narrow, reviewed, logged, time-bound, and followed by remediation.                                           |
| Developer experience    | Hooks that take so long or fail so opaquely that developers avoid committing frequently.                 | Keep pre-commit fast, staged-only, deterministic, and actionable; move slower checks to pre-push or CI.                                |

GitHub branch protection and rulesets can require PR reviews, status checks, conversation resolution, signed commits, linear history, merge queues, deployments, restricted pushes, and force-push blocking. ([GitHub Docs][3])

## 1. Native Git hooks

Git stores hooks in `$GIT_DIR/hooks` by default, and `core.hooksPath` can point Git at another hook directory. Git also distinguishes hooks that can block an action from post-action hooks that are only useful for notification or follow-up. ([Git][4])

| Hook                                                                            | Always bad behavior                                                                                                                                                                          | Always good behavior                                                                                                                                                                                                                                          |
| ------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `pre-commit`                                                                    | Running huge full-project suites, scanning the unstaged worktree instead of the staged snapshot, auto-staging unreviewed changes, hiding failures, or relying on it as the only policy gate. | Run fast staged-file checks: format, lint, conflict markers, secret scan, large-file check, debug-print check. Repeat critical checks in CI.                                                                                                                  |
| `prepare-commit-msg`                                                            | Using it as a substitute for `pre-commit` or `commit-msg`, overwriting human-written messages, or silently changing meaning.                                                                 | Insert helpful metadata such as branch ticket ID, template text, or merge/squash context without deleting user intent. Git specifically says it is for editing the message file and “should not be used as a replacement for the pre-commit hook.” ([Git][4]) |
| `commit-msg`                                                                    | Allowing `wip`, `fix stuff`, missing DCO, missing issue ID, or inconsistent release-driving messages; enforcing only locally when the rule matters.                                          | Enforce Conventional Commits, DCO `Signed-off-by`, issue IDs, and breaking-change footers locally and in CI/server checks. Git says this hook can validate or normalize the message and abort nonzero. ([Git][4])                                             |
| `pre-merge-commit`                                                              | Assuming it covers conflicted manual merges. Git notes it is not run when automatic merge fails and conflicts must be committed separately.                                                  | Run the same fast checks before automatic merge commits; rely on normal `pre-commit` after manual conflict resolution. ([Git][4])                                                                                                                             |
| `post-commit`                                                                   | Putting blocking logic here or modifying committed content after the fact.                                                                                                                   | Notifications, metrics, reminders, or local indexing only. Git states it cannot affect the commit outcome. ([Git][4])                                                                                                                                         |
| `pre-rebase`                                                                    | Blocking all rebases blindly, or allowing public/protected history to be rewritten without warning.                                                                                          | Prevent rebasing protected branches or commits already pushed to shared branches.                                                                                                                                                                             |
| `post-checkout`                                                                 | Running surprise destructive setup, changing source files silently, or performing slow installs on every branch switch.                                                                      | Lightweight environment checks, dependency reminders, generated-file hints, or safe local metadata restore.                                                                                                                                                   |
| `post-merge`                                                                    | Expecting it to block a bad merge, or auto-committing changes after pull.                                                                                                                    | Non-blocking dependency reinstall reminders, permission restore, generated-file refresh, or local setup. Git says it cannot affect merge outcome. ([Git][4])                                                                                                  |
| `post-rewrite`                                                                  | Rewriting history again, losing metadata, or doing network side effects without idempotency.                                                                                                 | Update notes, metadata, caches, or warnings after amend/rebase.                                                                                                                                                                                               |
| `post-index-change`                                                             | Expensive recursive scans every time the index changes, or recursively mutating the index from the hook.                                                                                     | Tiny cache refreshes or editor/tool integrations that react safely to index changes.                                                                                                                                                                          |
| `pre-push`                                                                      | Treating it as final enforcement, running overly slow suites on every push, ignoring the refs provided on stdin, or failing without clear remediation.                                       | Run moderate checks on outgoing commits: typecheck, focused tests, build smoke test, secret scan. CI must repeat anything important. Git says nonzero aborts the push before anything is pushed. ([Git][4])                                                   |
| `pre-receive`                                                                   | Slow, flaky, vague, or stateful rejection logic that blocks everyone unpredictably.                                                                                                          | Hard-block unacceptable pushes across refs: secrets, protected ref updates, unsigned commits, forbidden files, branch naming, commit-message rules. Git says nonzero prevents all refs from being updated. ([Git][4])                                         |
| `update`                                                                        | Implementing whole-push policy in a per-ref hook and missing cross-ref invariants.                                                                                                           | Per-branch or per-tag rules: fast-forward-only, block tag rewrites, restrict specific refs. Git says it runs once per ref and can enforce fast-forward-only policy. ([Git][4])                                                                                |
| `proc-receive`                                                                  | Using it for simple policies that branch protections or `pre-receive` can handle.                                                                                                            | Advanced receive workflows that need custom reference handling and careful protocol testing.                                                                                                                                                                  |
| `post-receive`                                                                  | Expecting it to reject a bad push after acceptance, or deploying untested code.                                                                                                              | Notifications, mirroring, CI triggers, deployment only after all required upstream gates. Git says it does not affect receive-pack outcome. ([Git][4])                                                                                                        |
| `post-update`                                                                   | Using it for old/new revision auditing; Git says it lacks original and updated values and recommends `post-receive` for that.                                                                | Simple notifications or `update-server-info` for dumb HTTP transports. ([Git][4])                                                                                                                                                                             |
| `reference-transaction`                                                         | Assuming post-commit phases can block, or performing irreversible side effects before the transaction is certain.                                                                            | Enforce/audit ref transactions during `preparing` or `prepared`; use later phases only for audit. Git says only `preparing` and `prepared` nonzero exits abort. ([Git][4])                                                                                    |
| `push-to-checkout`                                                              | Treating push-to-deploy as a production deployment system without review, CI, rollback, or audit.                                                                                            | Use only for controlled low-risk push-to-deploy workflows with clear worktree policy.                                                                                                                                                                         |
| `pre-auto-gc`                                                                   | Disabling GC forever or hiding repository corruption/performance problems.                                                                                                                   | Delay GC only during sensitive local operations, or notify that automatic GC is happening.                                                                                                                                                                    |
| `fsmonitor-watchman`                                                            | Returning stale or incomplete path lists; this can make Git miss changed files.                                                                                                              | Be conservative: include all possibly changed, new, deleted, and renamed paths. Git explicitly says it is OK to include files that have not actually changed. ([Git][4])                                                                                      |
| `applypatch-msg`                                                                | Accepting emailed patches with invalid messages or silently changing meaning.                                                                                                                | Normalize or reject patch commit messages before applying email patches.                                                                                                                                                                                      |
| `pre-applypatch`                                                                | Skipping tests or validation for `git am` workflows.                                                                                                                                         | Inspect the applied patch before commit; abort if it fails checks.                                                                                                                                                                                            |
| `post-applypatch`                                                               | Expecting it to block the patch.                                                                                                                                                             | Notifications only; Git says it cannot affect the outcome. ([Git][4])                                                                                                                                                                                         |
| `sendemail-validate`                                                            | Sending patch series that do not apply, omit required trailers, or leak secrets.                                                                                                             | Validate patch headers and applyability before `git send-email`; Git’s sample checks patch application against upstream. ([Git][4])                                                                                                                           |
| `p4-changelist`, `p4-prepare-changelist`, `p4-post-changelist`, `p4-pre-submit` | Bypassing enterprise review or letting Git/P4 metadata diverge.                                                                                                                              | Validate changelist text and pre-submit policy; use post hooks only for notifications. ([Git][4])                                                                                                                                                             |

## 2. Hook distribution tools

| Tool/setup                                 | Always bad behavior                                                                                                                                    | Always good behavior                                                                                                                                                                                                                                                                |
| ------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Raw `.git/hooks/*`                         | Relying on manual per-clone files for team policy. They are not cloned and are easy to miss.                                                           | Use for personal one-off local automation only.                                                                                                                                                                                                                                     |
| Checked-in `.githooks/` + `core.hooksPath` | Forgetting onboarding/install steps, not making files executable, or believing this is server enforcement.                                             | Version `.githooks/`, document `git config core.hooksPath .githooks`, and duplicate important checks in CI. Git documents `core.hooksPath` as a centralized alternative to per-repo hooks. ([Git][5])                                                                               |
| `init.templateDir`                         | Assuming it updates existing clones or guarantees policy after clone.                                                                                  | Seed personal or organization hooks for new repos only.                                                                                                                                                                                                                             |
| `pre-commit`                               | Leaving hook revisions unpinned, installing only `pre-commit` while expecting `commit-msg` or `pre-push`, or using `--no-verify` for one failing hook. | Keep `.pre-commit-config.yaml` versioned, pin/update `rev`, install all required hook types, use `SKIP=<hook>` instead of skipping everything, and run `pre-commit run --all-files` in CI. The docs recommend considering CI too and support multiple hook types. ([Pre-Commit][6]) |
| Husky                                      | Using Node-specific install assumptions in non-Node repos, hiding complex shell in hooks, or mixing several hook systems.                              | Good for Node/JS repos; keep `.husky/*` short, call package scripts, combine with `lint-staged`, and avoid duplicate hook managers. Husky uses `core.hooksPath` and supports client-side hooks. ([Typicode][7])                                                                     |
| `lint-staged`                              | Using it for checks that require the entire repo, or assuming staged-only checks replace full CI.                                                      | Run formatters/linters only on staged files to keep commits fast; run whole-project checks in CI. Its own README says whole-project tasks can be slow and staged-file filtering avoids irrelevant results. ([GitHub][8])                                                            |
| Lefthook                                   | Running non-thread-safe tasks in parallel, hiding required install steps, or letting local hooks diverge from CI.                                      | Good polyglot runner; commit `lefthook.yml`, install via the project package manager, use parallelism carefully, and mirror required checks in CI. Lefthook installs configured hooks into `.git/hooks/`. ([Lefthook][9])                                                           |
| Overcommit                                 | Adding Ruby startup delay to non-Ruby repos, disabling signature verification casually, or running side-effecting pre-commit hooks.                    | Good for Ruby-heavy repos; sign config, use a small hook-specific Gemfile, document dependencies, and keep hooks fast. Overcommit itself warns about side effects and notes dependency/startup-delay tradeoffs. ([GitHub][10])                                                      |
| CaptainHook                                | Forcing PHP/Composer workflows onto non-PHP teams, or putting destructive setup in hooks.                                                              | Good for PHP projects; share JSON config, validate commit messages, run quality checks/tests, and automate activation. ([GitHub][11])                                                                                                                                               |

## 3. Checks worth adding

| Check family                                                                  | Always bad behavior                                                                                                                         | Always good behavior                                                                                                                                                                                                                                                               |
| ----------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Formatters: Prettier, Black, Ruff format, rustfmt, gofmt, clang-format, shfmt | Committing unformatted code, using different formatter versions per developer, or reformatting unrelated files in feature commits.          | Pin versions, run staged-only locally, run whole-project format check in CI, and put mass-format commits in `.git-blame-ignore-revs`.                                                                                                                                              |
| Linters: ESLint, Ruff, Clippy, ShellCheck, markdownlint, yamllint, hadolint   | Disabling rules without explanation, treating warnings as invisible, or having local-only lint rules.                                       | Commit linter config, fail on real errors, document warning policy, and require CI status checks.                                                                                                                                                                                  |
| Type checks: `tsc --noEmit`, Pyright, mypy, `cargo check`, `go test`          | Ignoring type errors, making generated types stale, or allowing typecheck to be optional on PRs.                                            | Run quick type checks pre-push and required full checks in CI.                                                                                                                                                                                                                     |
| Tests                                                                         | Committing broken tests, flaky tests, or tests that only pass on one developer’s machine.                                                   | Fast focused tests locally, full deterministic test suite in CI, owner/quarantine process for flakes.                                                                                                                                                                              |
| Secret scans: Gitleaks, git-secrets, TruffleHog, detect-secrets               | Committing secrets, assuming deletion from the latest commit removes the leak, blanket allowlisting, or never revoking exposed credentials. | Scan locally, in CI, and at push-protection/server level; rotate any leaked credential. Gitleaks detects passwords/API keys/tokens, git-secrets rejects configured secret patterns, and GitHub push protection blocks supported secrets before they reach the repo. ([GitHub][12]) |
| Commit messages                                                               | `wip`, `stuff`, `changes`, missing issue IDs, missing DCO, or misleading types.                                                             | Conventional Commits, DCO where required, issue IDs, and breaking-change footers. Conventional Commits define `<type>[optional scope]: <description>`, and DCO requires a per-commit `Signed-off-by` line. ([Conventional Commits][13])                                            |
| Huge files / Git LFS                                                          | Committing videos, datasets, archives, model weights, or binaries directly into normal Git history.                                         | Enforce max file size, use Git LFS for approved large assets, and reject accidental blobs in CI/server hooks.                                                                                                                                                                      |
| Generated/vendor files                                                        | Committing `dist/`, `build/`, `.next/`, `node_modules/`, generated clients, or vendored blobs without policy.                               | Either ignore generated output or commit it only when the repo explicitly requires it and regeneration is reproducible.                                                                                                                                                            |
| Debug code                                                                    | Shipping `debugger`, `pdb.set_trace`, `console.log` in production paths, `dbg!`, temporary credentials, or “TODO remove” code.              | Block known debug patterns in production paths; allow intentional logging through structured logging.                                                                                                                                                                              |
| Conflict markers                                                              | Committing `<<<<<<<`, `=======`, `>>>>>>>`.                                                                                                 | Always block conflict markers before commit and in CI.                                                                                                                                                                                                                             |
| Permissions                                                                   | Accidental executable-bit changes on source files or scripts missing executable bit.                                                        | Review mode changes and enforce executable policy by path.                                                                                                                                                                                                                         |
| Line endings                                                                  | Mixed CRLF/LF drift or relying on every developer’s editor settings.                                                                        | Use `.gitattributes` for normalization and `core.safecrlf` to warn or reject irreversible conversions. Git documents that mixed line endings and accidental text conversion can corrupt binary files. ([Git][14])                                                                  |
| Whitespace                                                                    | Unintended trailing spaces, blank-at-EOF churn, or whitespace-only noise mixed into logic changes.                                          | Highlight/fix whitespace issues, while allowing intentional cases such as Markdown hard breaks. Git’s `core.whitespace` controls common whitespace problems noticed by diff/apply. ([Git][5])                                                                                      |
| Dependencies                                                                  | Changing manifests without lockfiles, ignoring vulnerability/license alerts, or committing unreviewed transitive upgrades.                  | Lockfile consistency checks, dependency audit, license policy, Renovate/Dependabot PRs, and required CI.                                                                                                                                                                           |
| Docs                                                                          | Broken docs builds, stale examples, or broken links in docs-only PRs.                                                                       | Markdown lint, link check, docs build, and examples tested where practical.                                                                                                                                                                                                        |
| Migrations                                                                    | Irreversible, non-backward-compatible migrations deployed with app code that cannot tolerate old/new schema states.                         | Migration lint, expand/contract pattern, rollback plan, and deploy-order tests.                                                                                                                                                                                                    |
| API/schema                                                                    | Breaking OpenAPI, GraphQL, protobuf, database, or event schema compatibility without detection.                                             | Contract compatibility checks and explicit versioning/breaking-change review.                                                                                                                                                                                                      |
| IaC                                                                           | Unformatted or invalid Terraform/Kubernetes/cloud config, or unchecked public exposure.                                                     | `terraform fmt/validate`, `tflint`, Checkov/tfsec, policy-as-code in CI.                                                                                                                                                                                                           |
| Containers                                                                    | Broken Dockerfiles, latest-tag drift, root containers without reason, or vulnerable base images.                                            | Dockerfile lint, reproducible tags/digests, image scans, minimal privileges.                                                                                                                                                                                                       |
| License/SBOM                                                                  | Unknown dependency licenses, missing notices, or no artifact provenance.                                                                    | License checks, SBOM generation, provenance/signing for releases.                                                                                                                                                                                                                  |

## 4. Repo-level Git setup files

| File/config                                                        | Always bad behavior                                                                                                         | Always good behavior                                                                                                                                                                                                                             |
| ------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `.gitignore`                                                       | Ignoring files that should be tracked, using `.gitignore` to hide secrets, or expecting it to affect already-tracked files. | Track shared project ignore rules; use `.git/info/exclude` for local-only repo ignores and global excludes for editor/OS junk. Git says tracked files are not affected and explains the right location for shared vs local patterns. ([Git][15]) |
| `.gitattributes`                                                   | Letting line endings, binary handling, or diff/merge behavior depend on every developer’s machine.                          | Normalize text files, mark binaries, configure diff/merge per path, and commit attributes meant for all developers. ([Git][14])                                                                                                                  |
| `.editorconfig`                                                    | Allowing editor-specific indentation/newline drift.                                                                         | Define indentation, charset, final newline, trimming rules consistently across editors.                                                                                                                                                          |
| `.pre-commit-config.yaml`                                          | Unpinned hooks, stale hooks, or config that nobody installs/runs.                                                           | Version the config, pin hook revisions, run `pre-commit autoupdate` deliberately, and run it in CI.                                                                                                                                              |
| `.husky/`                                                          | Large shell scripts, duplicated hook systems, or hooks that depend on uninstalled global tools.                             | Keep scripts tiny and call package-manager scripts.                                                                                                                                                                                              |
| `lefthook.yml`                                                     | Parallel race conditions or undocumented installation.                                                                      | Use declarative jobs, glob filters, and package-manager installation.                                                                                                                                                                            |
| `.lintstagedrc` / `lint-staged` config                             | Running whole-project checks through staged-file plumbing, or mutating unstaged files.                                      | Use for staged-file format/lint only.                                                                                                                                                                                                            |
| `commitlint.config.*`                                              | Rules that disagree with release automation or are enforced only on some machines.                                          | Enforce the same commit convention in local `commit-msg` and CI. commitlint checks whether messages meet Conventional Commit format. ([GitHub][16])                                                                                              |
| `.gitmessage`                                                      | A template so verbose people ignore it, or one that conflicts with commitlint.                                              | Short template showing exact required format, trailers, and examples. Git supports `commit.template`. ([Git][5])                                                                                                                                 |
| `.mailmap`                                                         | Rewriting identity expectations incorrectly or hiding authorship confusion instead of fixing Git config.                    | Canonicalize known author/committer name/email variants. Git says `.mailmap` maps names and emails to canonical forms. ([Git][17])                                                                                                               |
| `.git-blame-ignore-revs`                                           | Hiding real logic changes inside “formatting” commits.                                                                      | List only mechanical mass-format commits; configure blame to use it. Git supports `--ignore-revs-file` and `blame.ignoreRevsFile`. ([Git][18])                                                                                                   |
| `.lfsconfig` + `.gitattributes`                                    | Putting normal source or small text files into LFS, or forgetting CI/fork access implications.                              | Use LFS only for approved large binary assets and enforce with size checks.                                                                                                                                                                      |
| `CODEOWNERS`                                                       | Stale owners, patterns that do not match intended paths, files over platform limits, or code owners without write access.   | Keep ownership precise, reviewable, and paired with required code-owner review on protected branches. GitHub uses CODEOWNERS from the PR base branch and requires valid syntax/access. ([GitHub Docs][19])                                       |
| `.github/workflows/*`, `.gitlab-ci.yml`, `bitbucket-pipelines.yml` | Optional CI, duplicate ambiguous jobs, or checks that do not run on PRs/merge queues.                                       | Required, uniquely named, deterministic jobs that run on the right events. GitHub warns duplicate job names can create ambiguous status checks. ([GitHub Docs][3])                                                                               |
| `renovate.json` / `dependabot.yml`                                 | Auto-merging risky updates without tests or review.                                                                         | Small dependency PRs with changelog, tests, vulnerability metadata, and owner review.                                                                                                                                                            |
| `SECURITY.md`                                                      | No vulnerability reporting path, or a dead contact.                                                                         | Clear supported versions, reporting channel, SLA expectations, and disclosure policy.                                                                                                                                                            |
| `CONTRIBUTING.md`                                                  | Hidden rules enforced only by tribal knowledge.                                                                             | Explain branch naming, commits, tests, hooks, PR requirements, and release process.                                                                                                                                                              |
| `LICENSE`                                                          | Missing or conflicting license terms.                                                                                       | Clear license at repo root, with notices for dependencies where required.                                                                                                                                                                        |
| `Makefile` / `justfile` / `Taskfile.yml`                           | One-off commands that differ from CI.                                                                                       | `make check` / `just check` runs the same core quality gates developers and CI use.                                                                                                                                                              |

## 5. Git config hardening

| Config                         | Always bad behavior                                                                                              | Always good behavior                                                                                                                                                                  |
| ------------------------------ | ---------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `core.hooksPath .githooks`     | Setting it locally but never documenting it; assuming it cannot be bypassed; pointing it at a missing directory. | Use for versioned shared hooks, with onboarding and CI backup. Git documents absolute/relative hook paths and central hook configuration. ([Git][5])                                  |
| `commit.template`              | Template conflicts with commitlint or encourages giant boilerplate messages.                                     | Use a short template that matches the enforced commit format. ([Git][5])                                                                                                              |
| `commit.gpgSign true`          | Enabling signing without key setup, agent setup, or server enforcement; sharing signing keys.                    | Enable signing where identity assurance matters and pair it with protected-branch/ruleset enforcement. GitHub can block commits that are not signed and verified. ([GitHub Docs][20]) |
| `push.default=simple`          | Using `matching` in a shared central repo and accidentally pushing multiple unfinished branches.                 | Use `simple` for normal central workflows; Git calls it the safest beginner option. Use `nothing` where explicit refspecs are required. ([Git][5])                                    |
| `push.default=nothing`         | Setting it globally for novices without explaining why pushes now fail.                                          | Good for high-caution environments where every push target must be explicit.                                                                                                          |
| `core.safecrlf=warn` or `true` | Ignoring irreversible conversion warnings.                                                                       | Warn or reject CRLF conversions and encode normalization in `.gitattributes`. ([Git][5])                                                                                              |
| `core.whitespace`              | Treating every whitespace pattern as universally illegal without language/docs exceptions.                       | Highlight common whitespace errors and tune exceptions intentionally. ([Git][5])                                                                                                      |
| `apply.whitespace=error`       | Applying external patches while ignoring whitespace damage.                                                      | Reject whitespace errors when applying patches, especially in patch/email workflows.                                                                                                  |

## 6. Server-side and repo-host enforcement

This is the authoritative layer. Git’s server-side pre hooks can reject pushes; GitHub/GitLab/Bitbucket provide hosted equivalents through branch protections, rulesets, approvals, status checks, and merge checks. ([Git][1])

| Enforcement layer                   | Always bad behavior                                                                                                      | Always good behavior                                                                                                                                                                                                               |
| ----------------------------------- | ------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Protected branches                  | Direct pushes to `main`, `release/*`, or `prod/*`; allowing deletion/force-push by default; admin bypass without audit.  | Protect important branches, restrict pushes, block deletions and force-pushes, and apply restrictions to admins where compliance requires it.                                                                                      |
| Required PR reviews                 | Rubber-stamp approvals, self-approval, stale approvals after major changes, or emergency merges with no audit.           | Require meaningful review count, dismiss stale approvals when needed, and document emergency paths.                                                                                                                                |
| CODEOWNERS                          | Owners who are inactive, too broad, or not required by branch protection.                                                | Require domain-owner review for sensitive paths. GitLab specifically requires protected target branches and code-owner approval enabled for enforcement. ([GitLab Docs][21])                                                       |
| Required status checks              | Optional checks, ambiguous job names, checks that run on the wrong commit, or checks accepted from any untrusted source. | Require unique, trusted, app-scoped status checks before merge. GitHub says all required checks must pass and lets rulesets restrict the expected source. ([GitHub Docs][22])                                                      |
| Required signed commits             | Local signing without branch enforcement; shared keys; accepting “verified” as a substitute for code review.             | Require signed and verified commits where identity assurance matters, while still requiring review/tests.                                                                                                                          |
| Linear history                      | Enforcing linear history without team agreement or forcing risky rebases on shared branches.                             | Use when simpler reverts/bisects matter; require squash/rebase merges consistently. GitHub notes linear history prevents merge commits on targeted branches. ([GitHub Docs][22])                                                   |
| Merge queue                         | Enabling it without CI listening to merge-group events, or manually jumping queues constantly.                           | Use on busy branches to test PRs against the latest target plus queued changes. GitHub says merge queue ensures changes pass required checks when applied to the latest branch and queued PRs. ([GitHub Docs][23])                 |
| Secret push protection              | Bypassing alerts casually, ignoring unsupported secret types, or leaving leaked credentials active.                      | Enable push protection, review bypass requests, and rotate exposed credentials. GitHub says push protection blocks supported secrets before they reach the repository. ([GitHub Docs][24])                                         |
| Code scanning gate                  | Treating security scans as advisory only for high-risk repos.                                                            | Use rulesets to block merges when required tools find alerts, analysis is pending, or required tools are missing. ([GitHub Docs][25])                                                                                              |
| Server-side hooks                   | Long-running, flaky, network-dependent hooks with vague rejection messages.                                              | Fast deterministic `pre-receive`/`update` hooks with clear error output and tests.                                                                                                                                                 |
| GitLab approvals/protected branches | Allowing “push and merge” users to bypass MR approvals and Code Owners unintentionally.                                  | Use protected branches, approval rules, Code Owners, signed commits, status checks, and push rules. GitLab warns users allowed to push to protected branches can skip MR approval rules and Code Owners. ([GitLab Docs][26])       |
| Bitbucket merge checks              | Optional checks where compliance needs hard gates, or duplicated inherited checks that confuse teams.                    | Require builds, approvals, resolved tasks, and custom merge checks where needed. Bitbucket supports required builds/approvals/tasks and premium required custom checks at workspace/project/repo levels. ([Atlassian Support][27]) |

## 7. Best-practice default by repo maturity

| Repo type                | Bad behavior to avoid                                                                                                           | Good behavior to standardize                                                                                                                     |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| Solo repo                | No `.gitignore`, no formatter, committing secrets, giant generated blobs, no one-command check.                                 | `.gitignore`, `.gitattributes`, `.editorconfig`, formatter, secret scan, large-file check, `make check` / `just check`.                          |
| Small team               | Hooks only on some machines, CI advisory only, direct pushes to `main`, no review ownership.                                    | Hook manager, required CI, protected `main`, at least one review, CODEOWNERS for risky paths.                                                    |
| Serious product repo     | Slow local hooks, flaky CI, force-push access, no secret push protection, no dependency/security scanning.                      | Fast local hooks, full required CI, branch protection/rulesets, secret scanning/push protection, dependency scanning, code scanning.             |
| Regulated/high-risk repo | Unreviewed bypasses, unsigned/unverified commits where identity matters, no audit trail, production deploys from direct pushes. | Required signed commits, no direct pushes, no force-pushes, CODEOWNERS, merge queue, server-side hooks/rulesets, SBOM/provenance, audit logging. |

## 8. The clean default stack

For most modern repos, the balanced setup is:

```text
Local:
  pre-commit or Lefthook
  commitlint + Conventional Commits
  Gitleaks or equivalent
  staged-only format/lint
  typecheck and focused tests in pre-push

Repo:
  .gitignore
  .gitattributes
  .editorconfig
  .git-blame-ignore-revs
  CODEOWNERS
  Makefile / justfile / Taskfile.yml

Server/CI:
  protected main
  required PR review
  required status checks
  secret push protection
  code/dependency scanning
  merge queue for busy branches
```

The behavior to stamp out is not “bad commits happen locally.” The truly bad behavior is letting locally bypassable or optional checks be the last line of defense.
