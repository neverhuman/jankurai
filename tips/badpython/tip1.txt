Below is a **redline checklist** for Python “vibe coding”: behavior that should be treated as **never acceptable in production, shared repositories, deployed services, CI/CD, agent-written code, or code that handles real users/data**. The only sane exceptions are deliberately quarantined security test fixtures, educational examples clearly labeled as unsafe, or exploit reproduction in an isolated lab.

I treated social/forum material as signal, not authority. The hard rules below are grounded mainly in Python’s own security notes, OWASP, Bandit, Semgrep, CodeQL/GitHub, GitLab, PyPA/pip, and recent AI-generated-code studies. Python’s own docs flag modules such as `pickle`, `subprocess`, `random`, `tempfile`, `ssl`, `xml`, and `zipfile` as having security considerations; OWASP’s Top 10 centers the same families of risk: broken access control, cryptographic failures, injection, insecure design/configuration, vulnerable components, auth failures, integrity failures, logging failures, and SSRF. ([Python documentation][1])

## 0. The meta-rule

**Never ship code you do not understand, cannot test, cannot explain, cannot monitor, and cannot safely roll back.** This is especially true for AI/agent-generated Python. Recent studies and reports on AI-generated code repeatedly find correctness, maintainability, and security problems: one Copilot study found security weaknesses in 32.8% of sampled Python snippets, and a 2026 large-scale study found AI-introduced issues including unsafe subprocess usage, path traversal, SQL injection, and complexity/security debt surviving into repository heads. ([arXiv][2])

# 1. Code execution and injection redlines

## Never execute dynamic code from untrusted or uncontrolled input

Unacceptable behavior:

* Using `eval()`, `exec()`, `compile()`, `execfile`-style helpers, dynamic `import`, `__import__`, `getattr` dispatch, or plugin loaders on strings influenced by users, files, tickets, prompts, webpages, issue comments, environment variables, config, database rows, or model output.
* “Sandboxing” `eval` with `{}` and pretending it is safe.
* Letting an agent generate Python and immediately run it against real data, production credentials, or a developer machine with broad filesystem access.
* Building Python expressions with f-strings and evaluating them.
* Using `ast.literal_eval` as a magic security blanket without bounding input size and type.
* Loading “rules,” “formulas,” “templates,” “workflows,” or “business logic” from editable text and interpreting it as Python.
* Accepting user-supplied module names, class names, function names, or dotted paths and importing/calling them directly.
* Using dynamic deserialization or metaprogramming to “make the tests pass” instead of designing a real interface.

Semgrep explicitly warns that dynamic `exec()` content controlled by a user can become code injection, and OWASP’s secure coding guide says not to pass user-supplied data to dynamic execution functions. ([Semgrep][3])

## Never construct shell commands with untrusted data

Unacceptable behavior:

* `subprocess.run("cmd " + user_input, shell=True)`.
* `os.system`, `os.popen`, `commands.*`, `pexpect`, `sh -c`, `bash -c`, or `subprocess(..., shell=True)` with any untrusted component.
* Passing a single command string when an argv list is possible.
* Manually quoting shell strings as the main defense.
* Concatenating file paths, branch names, URLs, package names, issue titles, PR titles, or model-generated text into shell commands.
* Running package manager commands generated by an agent without review.
* Invoking shell pipelines for simple operations that Python APIs can do safely.
* Letting user input influence `cwd`, `env`, executable path, or `PATH`.

Python’s `subprocess` docs warn to read security considerations before `shell=True` and state that when a shell is invoked, the application is responsible for quoting metacharacters to avoid shell injection. Semgrep similarly recommends internal APIs first, argv arrays over command strings, and avoiding user-controlled input. ([Python documentation][4])

## Never build SQL, NoSQL, LDAP, GraphQL, or search queries by string concatenation

Unacceptable behavior:

* `f"SELECT * FROM users WHERE id = {user_id}"`.
* `"WHERE name = '%s'" % name`.
* `.format()` or f-string SQL, even when “it’s just an integer.”
* SQLAlchemy `text()` with untrusted interpolated text.
* Mongo/NoSQL queries built from raw request JSON without schema validation and allow-lists.
* Dynamically choosing table, column, sort, filter, operator, or index names from user strings.
* Letting agents “fix” SQL errors by making queries more permissive.
* Falling back to raw SQL because the ORM query was inconvenient.
* Swallowing SQL errors and retrying with broader queries.

OWASP’s secure coding checklist says to use strongly typed parameterized queries and not to run database commands when validation/encoding fails; CodeQL flags SQL/NoSQL query construction from user-provided data as an injection risk. ([OWASP Foundation][5])

# 2. Deserialization, parsing, and data-loading redlines

## Never deserialize untrusted Python objects

Unacceptable behavior:

* `pickle.load(s)`, `pickle.loads(s)`, `dill`, `cloudpickle`, `shelve`, `jsonpickle`, `marshal`, `joblib.load`, `pandas.read_pickle`, `torch.load`, or similar on files from users, uploads, URLs, caches, queues, model outputs, public buckets, emails, CI artifacts, or “trusted” internal services without provenance guarantees.
* Treating signed pickle as safe when the signing key is broadly accessible.
* Using pickle for API payloads.
* Accepting `.pkl`, `.pickle`, `.pt`, `.pth`, `.joblib`, `.sav`, `.dill`, or model artifacts from users and loading them in-process.
* Allowing plugins/extensions to ship as pickled Python objects.
* Using `marshal` for application data.
* “It came from S3” or “it came from our DB” as a trust argument after any user or compromised service can write there.

Python’s pickle docs are explicit: pickle is not secure, malicious pickle data can execute arbitrary code during unpickling, and only trusted data should be unpickled. Bandit also blacklists pickle-like calls including `dill`, `shelve`, `jsonpickle`, and `pandas.read_pickle` as unsafe for untrusted deserialization. ([Python documentation][6])

## Never use unsafe YAML/object loaders

Unacceptable behavior:

* `yaml.load(data)` without a safe loader.
* Accepting YAML from users or repos and letting it instantiate arbitrary Python objects.
* Assuming YAML is “just config.”
* Letting agents add YAML-based “automation” without schema validation.
* Loading CI, plugin, pipeline, or workflow YAML with code-capable constructors.

Bandit’s `B506` warns that PyYAML’s `yaml.load` can construct arbitrary Python objects and recommends `yaml.safe_load` for simple types. ([Bandit][7])

## Never parse untrusted XML casually

Unacceptable behavior:

* Parsing untrusted XML without understanding entity expansion, decompression bombs, external resources, parser version, and size limits.
* Accepting XML uploads without strict schema, depth, and size limits.
* Enabling DTDs or external entity resolution unless explicitly required and isolated.
* Treating SAML, SOAP, SVG, DOCX/XLSX, or XML-in-zip as harmless.
* Feeding XML from emails, webhooks, uploaded documents, or agents directly into business logic.

Python’s XML docs warn that attacker-controlled XML can cause denial of service, local file access, network connections, or firewall bypasses; they specifically discuss entity expansion, decompression bombs, and large-token DoS issues. ([Python documentation][8])

## Never unpack archives without containment

Unacceptable behavior:

* Extracting zip/tar files from users without path normalization, allow-lists, file count limits, total uncompressed size limits, symlink handling, and destination containment.
* Trusting archive filenames.
* Allowing `../`, absolute paths, device files, symlinks, hardlinks, or permission bits from archives.
* Extracting into application directories, web roots, repo roots, `$HOME`, or `/tmp` shared locations.
* Ignoring zip bombs and decompression bombs.
* Processing nested archives recursively without limits.

Python flags `zipfile` as security-sensitive because malicious zip files can cause disk exhaustion, and its zipfile documentation calls out decompression pitfalls and resource limitations. ([Python documentation][1])

# 3. Secrets, credentials, and sensitive data redlines

## Never put secrets in source code, prompts, logs, notebooks, tests, or Git history

Unacceptable behavior:

* Hardcoding API keys, passwords, tokens, private keys, OAuth secrets, database URLs, cloud credentials, JWT signing keys, encryption keys, webhook secrets, SSH keys, service account JSON, or basic-auth strings.
* Committing `.env`, `.pem`, `.key`, `.p12`, `.kube/config`, `service-account.json`, SQLite prod dumps, test fixtures with real data, or notebook outputs containing credentials.
* Including secrets in AI prompts, issue comments, PR descriptions, screenshots, stack traces, telemetry, debug logs, or generated code.
* “Removing the secret in the next commit” and treating it as fixed.
* Sharing one token across dev, CI, staging, and prod.
* Long-lived human tokens for automation.
* Storing secrets in client-side code, mobile apps, browser JS, or downloadable config.
* Default credentials, sample credentials, `changeme`, `password`, or copied tutorial keys.
* Printing `os.environ`, full config objects, request headers, or database URLs.

GitHub secret scanning looks across entire Git history and repository surfaces for hardcoded credentials, and GitHub advises immediate rotation when a leak is detected. OWASP says secrets are commonly hardcoded in source/config and recommends centralized storage, access control, rotation, and lifecycle management. ([GitHub Docs][9])

## Never log or expose sensitive data

Unacceptable behavior:

* Logging passwords, tokens, session IDs, cookies, Authorization headers, PII, PHI, card data, reset links, signed URLs, private prompts, embeddings of sensitive text, full request bodies, or full database records.
* Returning stack traces, SQL errors, file paths, environment variables, dependency versions, or secrets to users.
* Logging untrusted strings without preventing log injection/control characters.
* Keeping debug logs enabled in production.
* Sending secrets to analytics, crash reporting, LLM telemetry, or third-party observability without explicit review.
* Treating logs as low-sensitivity.
* Using print debugging with secrets and forgetting it.

OWASP’s logging guidance emphasizes security event logging but also cautions that logs need deliberate design and must avoid collecting the wrong amount/type of data; OWASP secure coding guidance also says not to store sensitive information in clear text or insecure client-side forms. ([OWASP Cheat Sheet Series][10])

# 4. Authentication, authorization, and access-control redlines

## Never trust the client for identity, role, ownership, or authorization

Unacceptable behavior:

* Taking `user_id`, `is_admin`, `role`, `tenant_id`, `organization_id`, or `owner_id` from a request body/query/header and trusting it.
* Checking authentication but not object-level authorization.
* “Admins only” enforced only in frontend JavaScript.
* Hidden routes, obscured IDs, UUIDs, or “unguessable” URLs used as authorization.
* Missing tenant isolation in queries.
* Allowing users to access files by path or key without ownership checks.
* Trusting JWT claims without verifying signature, issuer, audience, algorithm, expiration, and key rotation.
* Accepting `alg=none` or algorithm confusion in JWT-like systems.
* Using one service account for all users and skipping per-user authorization.
* Letting an agent “simplify” auth by bypassing decorators/middleware/tests.
* Returning different login error messages that disclose whether a user exists.

OWASP identifies broken access control as the top web-app risk in the 2021 Top 10 and says authentication/access controls must be enforced on trusted systems, fail securely, and use least privilege. ([OWASP Foundation][11])

## Never store passwords incorrectly

Unacceptable behavior:

* Plaintext passwords.
* Reversible encrypted passwords for normal login.
* Fast hashes like MD5/SHA1/SHA256 alone.
* Unsalted hashes.
* One global salt.
* Homegrown password hashing.
* Storing reset tokens as plaintext.
* No rate limiting or lockout/backoff on login/reset.
* Password reset links that never expire.
* Logging password reset tokens.
* Comparing secrets with normal equality where timing attacks matter.

OWASP recommends modern adaptive one-way password hashing such as Argon2id, scrypt, bcrypt, or PBKDF2 with appropriate work factors, and explicitly says passwords should be hashed rather than encrypted or stored in plaintext. Python’s `secrets` docs also state passwords should not be stored recoverably and should be salted and hashed with a strong one-way hash. ([OWASP Cheat Sheet Series][12])

# 5. Cryptography and TLS redlines

## Never invent cryptography

Unacceptable behavior:

* Custom encryption algorithms, custom MACs, custom token formats, custom password hashing, custom signing, custom key exchange.
* MD5, SHA1, DES, RC4, ECB mode, static IVs/nonces, reused nonces, predictable IVs, or unauthenticated encryption for security.
* Using `random` for tokens, passwords, reset links, invite codes, CSRF tokens, session IDs, API keys, salts, nonces, or crypto keys.
* Hardcoded crypto keys.
* Reusing keys across environments or purposes.
* Failing to rotate keys.
* Disabling certificate validation.
* `requests.get(..., verify=False)`.
* `ssl._create_unverified_context()`.
* Turning off hostname checks.
* Allowing TLS fallback to insecure connections.
* Building “encryption” with base64, XOR, rot13, Fernet keys in code, or “obfuscation.”
* Signing without binding audience, issuer, expiration, and purpose.
* Comparing HMACs/tokens with `==` when constant-time comparison is needed.

Python’s `secrets` module is specifically for cryptographically strong randomness and should be preferred over `random` for passwords/tokens; Python’s SSL docs state SSLv2/SSLv3 are insecure and that `PROTOCOL_TLS_CLIENT` implies certificate validation and hostname checks by default. Bandit also flags insecure MD5/SHA1 use for security-sensitive hashing. ([Python documentation][13])

# 6. Filesystem, paths, uploads, and temporary-file redlines

## Never trust paths from users, archives, repos, or model output

Unacceptable behavior:

* Joining `base_dir / user_supplied_path` without resolving and verifying containment.
* Allowing `../`, absolute paths, drive letters, UNC paths, symlinks, hardlinks, URL-encoded traversal, null bytes, Unicode confusables, or alternate separators.
* Letting users choose filenames used on disk.
* Serving files by raw path.
* Deleting paths from user input.
* Running cleanup scripts over paths generated by an agent without dry-run and containment checks.
* Writing uploads into executable or served directories.
* Trusting file extensions or MIME types.
* Using uploaded filename as storage key.
* Failing to scan or quarantine risky uploaded files.
* Making files or directories `777`.
* Relying on current working directory.

OWASP’s file-management guidance says to authenticate uploads, limit file types, validate file headers rather than extensions, avoid saving uploads in the web context, disable execution in upload directories, use allow-lists, avoid passing directory/file paths directly, and never send absolute paths to clients. ([OWASP Foundation][5])

## Never use insecure temporary-file patterns

Unacceptable behavior:

* `tempfile.mktemp()`.
* Predictable temp filenames.
* Writing sensitive data to `/tmp/foo`.
* Opening temp files with broad permissions.
* Creating a name first and opening it later.
* Ignoring symlink races.
* Leaving temp files with secrets behind.
* Using shared temp dirs for privileged operations without safe primitives.

Python’s `tempfile.mktemp()` docs warn it is insecure because another process can create the file before you use the returned name, creating a security hole; Python recommends primitives that create the file immediately. ([Python documentation][14])

# 7. Network, HTTP, SSRF, and API redlines

## Never fetch arbitrary URLs from user input

Unacceptable behavior:

* Server-side `requests.get(user_url)` without allow-listing scheme, host, port, DNS behavior, redirect behavior, IP ranges, and response limits.
* Allowing access to `localhost`, RFC1918/private ranges, link-local metadata services, Kubernetes service IPs, cloud metadata endpoints, Unix sockets, internal DNS, or admin panels.
* Following redirects from allowed external hosts to internal hosts.
* Fetching files from `file://`, `ftp://`, `gopher://`, custom schemes, or internal services.
* Returning fetched content blindly to users.
* No timeout.
* No max response size.
* No content-type validation.
* No retry limits.
* No rate limiting.
* Using user-controlled webhook URLs without validation.

OWASP’s Top 10 includes SSRF as a major category, and its input-validation guidance says to classify sources as trusted/untrusted, validate all untrusted data, and prefer allow-lists over deny-lists. ([OWASP Foundation][11])

## Never make outbound or inbound HTTP unsafe by default

Unacceptable behavior:

* No timeouts on `requests`, `httpx`, DB, Redis, S3, SMTP, or RPC calls.
* Infinite retries.
* Retrying non-idempotent operations blindly.
* No circuit breaker/backoff.
* No request body size limit.
* No upload size limit.
* No concurrency limit.
* No rate limit on expensive endpoints.
* No authentication on internal APIs because “only internal services call it.”
* Trusting `X-Forwarded-For`, `Host`, or proxy headers without trusted proxy config.
* Putting tokens in query strings.
* CORS `*` with credentials.
* CSRF disabled for cookie-authenticated state-changing requests.
* Returning overly broad error details.

# 8. Web framework configuration redlines

## Never run development/debug servers in production

Unacceptable behavior:

* Flask `debug=True` in production.
* Django `DEBUG=True` on any public server.
* `python -m http.server` for production.
* Werkzeug dev server as production.
* Auto-reloaders in production.
* Public interactive debuggers.
* Stack traces visible to users.
* Django `ALLOWED_HOSTS = ["*"]` without explicit host validation.
* Committed Django `SECRET_KEY`.
* Missing secure cookie flags for session/auth cookies.
* Missing HTTPS enforcement where auth/sensitive data is involved.
* Disabling CSRF middleware to “make frontend work.”
* Broad CORS to unblock local development and forgetting it.

Flask’s docs warn that the debugger allows arbitrary Python code execution from the browser and must not be run in production. Django’s deployment docs require appropriate `ALLOWED_HOSTS` when `DEBUG=False` to protect against Host-header-related attacks. ([Flask Documentation][15])

# 9. Dependency and supply-chain redlines

## Never install or upgrade dependencies casually

Unacceptable behavior:

* `pip install package` from an agent suggestion without checking name, source, maintainer, release history, license, vulnerabilities, and whether it is actually needed.
* Copying install commands from random READMEs, Stack Overflow, Reddit, X, issue comments, or model output.
* Installing typo-squatted, dependency-confusion, or hallucinated packages.
* Allowing unpinned broad ranges in deployable apps without lockfiles.
* No lockfile for applications.
* No hash checking for high-integrity builds.
* Installing from `main`, random Git SHAs, forks, gists, pastebins, or URLs without review.
* `curl | python`, `curl | bash`, or running arbitrary setup scripts.
* Ignoring package install-time code execution risk.
* Depending on abandoned packages with known CVEs.
* Vendoring code without provenance.
* Removing dependency pins to “fix conflicts.”
* Editing lockfiles by hand.
* Disabling dependency scanners because they are noisy.
* Failing to separate dev/test/prod dependencies.
* Installing build tools/compilers in production images unnecessarily.
* Not reviewing transitive dependency changes in PRs.
* Not using private index controls for internal package names.

pip’s secure-install guidance states that, by default, pip does not check against remote tampering and may run arbitrary code from distributions; it recommends hash-checking mode and disallowing source distributions for more secure installs. GitHub recommends vulnerability scanning, dependency audits, CI integration, lockfiles, dependency pinning, and tools that block vulnerable dependencies from merging. ([Pip][16])

## Never ignore known vulnerable dependencies

Unacceptable behavior:

* Merging with open high/critical dependency alerts without documented risk acceptance.
* “It’s transitive so it’s not our problem.”
* “It’s only a dev dependency” without proving it cannot affect build/release/CI.
* Pinning a vulnerable version forever.
* Using unmaintained packages for authentication, crypto, parsing, XML, YAML, markdown, templating, HTTP, JWT, SAML, OAuth, or file processing.
* No Dependabot/pip-audit/Safety/GitLab dependency scanning/equivalent.
* No owner assigned to dependency alerts.
* No update tests.

GitHub’s Dependabot docs say vulnerable dependencies can make a project a target and that alerts help upgrade to secure versions; dependency review is designed to catch insecure dependencies before they enter the environment and can block PRs when vulnerable packages are introduced. ([GitHub Docs][17])

# 10. Git, GitHub/GitLab, and CI/CD redlines

## Never let CI run untrusted code with secrets or write permissions

Unacceptable behavior:

* `pull_request_target` checking out and running fork code.
* Workflows triggered by untrusted PRs with write-scoped `GITHUB_TOKEN`.
* Passing PR title/body/branch name/commit message into shell commands.
* Running untrusted tests on self-hosted runners with network/secrets.
* Sharing caches across trust boundaries.
* Exposing secrets to forks.
* Printing secrets in CI.
* Long-lived cloud credentials in CI instead of short-lived OIDC.
* Default write-all permissions.
* Release jobs triggered by unreviewed code.
* Deployment from branches without protection.
* CI that modifies source code and commits back without review.
* “Fixing” CI by skipping tests, scanners, or permission checks.
* No branch protection, CODEOWNERS, required reviews, or required checks for protected branches.

OpenSSF’s GitHub workflow guidance calls out common attack vectors including privileged workflows running untrusted code, code injections, vulnerable actions, malicious releases, tag-renaming attacks, imposter commits, and unsafe caches. GitHub’s Actions security guidance recommends least privilege for workflow credentials and read-only default `GITHUB_TOKEN` permissions where possible. ([OpenSSF][18])

## Never use mutable CI dependencies for sensitive build/release paths

Unacceptable behavior:

* GitHub Actions pinned only to tags like `@v4` for high-integrity release workflows.
* Docker base images like `latest`.
* Unpinned setup scripts.
* Unpinned build tools.
* Unverified artifact downloads.
* Release artifacts built outside CI with unknown local state.
* No provenance/attestation for published packages.
* No reproducible build process.
* Publishing to PyPI/npm/container registries from developer laptops.

GitHub recommends pinning workflow dependencies to a specific commit SHA to limit compromised-dependency impact, and OpenSSF Scorecard explains that pinned dependencies support reproducibility and help mitigate compromised dependencies and dependency-confusion attacks. ([The GitHub Blog][19])

# 11. Error handling and logging redlines

## Never hide failures

Unacceptable behavior:

* Bare `except:`.
* `except Exception: pass`.
* `try: ... except: return None`.
* Catching everything and continuing.
* Catching `BaseException`.
* Swallowing `KeyboardInterrupt`, `SystemExit`, cancellation, or timeout errors.
* Logging an error and pretending success.
* Replacing specific exceptions with generic falsey values.
* Broad retry loops that hide systemic failure.
* “Best effort” security checks.
* Ignoring return codes from subprocesses.
* Ignoring failed writes, failed deletes, failed auth, failed encryption, failed validation, failed migrations, failed dependency scans, or failed tests.
* Catching exceptions around a large block instead of the precise operation.
* Losing exception context with `raise NewError()` instead of `raise NewError(...) from exc` when appropriate.
* Printing exceptions instead of structured logging and error propagation.

Python’s tutorial recommends catching specific exception types and allowing unexpected exceptions to propagate; Ruff’s `E722` rule explains that a bare `except` catches `BaseException`, can make programs hard to interrupt, and can disguise problems. ([Python documentation][20])

## Never use `assert` for runtime validation, security, permissions, or business rules

Unacceptable behavior:

* `assert user.is_admin`.
* `assert amount > 0`.
* `assert signature_valid`.
* `assert request.user.id == object.owner_id`.
* `assert path.startswith(base)`.
* `assert isinstance(data, ExpectedType)` for external input.
* Asserting auth, authz, tenant checks, balance checks, limits, or invariants that must hold in production.

Python removes assert statements when optimization is requested, and Bandit warns that using `assert` for interface/protection checks can remove protections in optimized bytecode. ([Python documentation][21])

# 12. Input validation, data modeling, and schema redlines

## Never process untrusted data without validation

Unacceptable behavior:

* Trusting request JSON shape.
* Trusting CSV/Excel headers/types.
* Trusting webhook payloads without signature validation.
* Trusting queue messages because “internal.”
* Trusting database values that originally came from users.
* Trusting LLM/agent output as structured data without schema validation.
* Using deny-lists for dangerous characters as the only defense.
* Accepting unlimited string lengths, list sizes, nesting depth, numeric ranges, regex complexity, file counts, or object graphs.
* Allowing unknown fields silently in security-sensitive APIs.
* Mixing validation, coercion, authorization, and persistence in one blob of code.
* Validating only in the frontend.
* “Sanitizing” by stripping quotes.
* Treating type hints as runtime validation.

OWASP’s input-validation checklist says to validate all untrusted sources, conduct validation server-side, use centralized validation, reject validation failures, validate expected types/ranges/lengths, and prefer allow-lists over deny-lists. ([OWASP Foundation][5])

# 13. Resource exhaustion and denial-of-service redlines

Unacceptable behavior:

* No timeout on network calls, subprocesses, DB queries, locks, queues, or futures.
* No request body limit.
* No upload limit.
* No decompressed-size limit.
* No pagination.
* Returning unbounded query results.
* Loading entire files into memory unnecessarily.
* Unbounded recursion.
* Unbounded regex on attacker-controlled input.
* Unbounded task/thread/process spawning.
* Unbounded async gather over user-controlled lists.
* Infinite retries or retry storms.
* No backpressure on queues/streams.
* No cancellation handling.
* No rate limiting on expensive endpoints.
* No cache limits/TTL.
* User-controlled image/video/PDF processing without CPU/memory/time sandboxing.
* Running model inference or data analysis on user files without quotas.

# 14. Python-specific maintainability redlines

These are not always “CVEs,” but in vibe-coded Python they create hidden state, unreviewable behavior, or production bugs.

## Never create hidden shared state accidentally

Unacceptable behavior:

* Mutable default arguments for state: `def f(x=[]): ...`.
* Global mutable registries without lifecycle/reset.
* Module-level work that opens sockets, reads env, starts threads, writes files, or calls services at import time.
* Singleton clients initialized before config is loaded.
* Monkeypatching builtins, stdlib modules, third-party APIs, or global settings outside isolated tests.
* `from module import *`.
* Shadowing builtins: `list`, `dict`, `id`, `file`, `input`, `type`, `open`.
* Circular imports “fixed” by moving imports into random functions without design.
* Runtime mutation of `sys.path` to make imports work.
* Importing from parent directories or relying on cwd.
* “Works when run from my IDE” path logic.

## Never make code unreviewable

Unacceptable behavior:

* Giant functions.
* Multi-thousand-line files.
* Deeply nested conditionals.
* Copy-paste branches with tiny differences.
* Dead code kept “just in case.”
* Commented-out code.
* Unused imports, unused variables, unreachable branches.
* Boolean parameters that radically change behavior.
* Magic strings/numbers scattered everywhere.
* Clever one-liners for critical logic.
* Metaclasses/decorators/descriptors/dynamic attributes where simple code works.
* Framework magic without tests.
* Overloaded functions that accept anything.
* `Any` everywhere to silence type errors.
* No docstring/comment for non-obvious security-sensitive logic.
* Comments that lie or describe old behavior.

PEP 8 frames readability as central because code is read more often than written; Ruff’s rule set exists to catch a broad range of common Python errors and quality problems early. ([Python Enhancement Proposals (PEPs)][22])

# 15. Testing redlines

Unacceptable behavior:

* No tests for new behavior.
* Deleting or weakening tests to pass.
* Marking failing tests as skipped/xfail without a tracked reason.
* Changing assertions to match broken behavior.
* Snapshot-updating without inspecting diffs.
* Mocking the code under test.
* Only testing happy paths.
* No tests for authz, validation, boundaries, malformed input, concurrency, timeouts, and failure modes.
* Tests that depend on order, wall-clock time, timezone, network, external APIs, local files, or developer machine state.
* Tests that use production services or production credentials.
* Tests that pass only when run individually.
* Fixtures with real secrets or real customer data.
* No regression test for a bug fix.
* No migration tests for database changes.
* No property/fuzz tests for parsers and validators where applicable.
* “Manual tested” as a substitute for automated tests.
* CI allowed to go red for long periods.

# 16. Type, lint, and scanner suppression redlines

Unacceptable behavior:

* `# type: ignore` without a precise code and justification.
* `# noqa` without a precise rule and justification.
* `# nosec` / `bandit: skip` without a security ticket or documented false positive.
* Disabling mypy/pyright/ruff/bandit/semgrep/CodeQL for whole files because fixing is hard.
* Lowering scanner severity thresholds after findings appear.
* Turning off dependency scanning because alerts are noisy.
* Treating type hints as documentation only while runtime data is unvalidated.
* Hiding `Any` in public interfaces.
* Blanket `cast()` to silence type failures.
* `object`/`dict[str, Any]` for everything.
* Ignoring nullability/optional handling.
* Letting generated code bypass lint/security gates.

GitHub describes CodeQL as a static analysis engine for finding vulnerabilities and errors in code; GitLab’s SAST and dependency scanning docs similarly describe automated detection of source vulnerabilities and dependency vulnerabilities before production/merge. ([GitHub Docs][23])

# 17. Database and migration redlines

Unacceptable behavior:

* Destructive migrations without backup/rollback plan.
* Data migrations that cannot be rerun safely.
* Non-idempotent scripts.
* No transaction around multi-step data changes.
* Long locks on production tables without planning.
* Adding nullable/constraint/index changes blindly on large tables.
* Dropping columns/tables in the same deploy that stops writing them.
* Using admin DB credentials in the app.
* App connects as schema owner/root.
* No separate read/write/admin users.
* Hardcoded connection strings.
* Logging full queries with sensitive parameters.
* No connection timeout.
* No pool limits.
* Leaking connections/cursors.
* Creating N+1 queries that become DoS.
* “Fixing” authorization by filtering in Python after fetching all rows.

OWASP’s database guidance calls for parameterized queries, least-privilege DB access, secure credentials, non-hardcoded connection strings, and closing connections promptly. ([OWASP Foundation][5])

---

# 18. Concurrency, async, and state redlines

Unacceptable behavior:

* Shared mutable state across threads/tasks without locks or immutability.
* Assuming the GIL makes compound operations safe.
* Blocking calls inside async request handlers.
* Fire-and-forget tasks with no ownership, cancellation, logging, or error handling.
* Race-prone check-then-act filesystem or DB logic.
* No idempotency keys for retried external operations.
* No transaction isolation thought.
* Global caches with no invalidation or bounds.
* Background workers that can process the same job twice unsafely.
* Distributed locks without expiry/fencing.
* Unbounded task queues.
* Swallowing cancellation.
* Doing CPU-bound work in event loop.
* Using multiprocessing with pickled untrusted objects.

Python’s security notes specifically call out that `multiprocessing.Connection.recv()` uses pickle, and OWASP general coding guidance calls for locking/synchronization to prevent race conditions and protection of shared variables/resources. ([Python documentation][1])

---

# 19. Packaging, project structure, and runtime redlines

Unacceptable behavior:

* No `pyproject.toml`/clear dependency declaration.
* App depends on globally installed packages.
* Works only with the developer’s venv.
* No Python version pin/constraint.
* No reproducible install instructions.
* Mixing app code, scripts, notebooks, generated files, and secrets in one directory.
* Running modules by path hacks instead of package entry points.
* Shipping dev dependencies/tools in production images unnecessarily.
* Running as root in containers when not required.
* Writable application code directory in production.
* No health checks.
* No graceful shutdown.
* No config validation at startup.
* Silent fallback to insecure defaults when config is missing.
* Auto-creating admin users/passwords in production.
* Running migrations implicitly on app startup without controls.

# 20. Agent-friendly / vibe-coding redlines

These are the behaviors that make AI-generated Python dangerous even when individual lines look “fine.”

## Never let an agent optimize for “green” over “correct”

Unacceptable behavior:

* Deleting tests.
* Weakening assertions.
* Broadening mocks.
* Skipping security checks.
* Disabling type/lint/security tools.
* Making code less strict to pass tests.
* Catching exceptions instead of fixing the cause.
* Returning dummy data on failure.
* Replacing real authorization with TODOs.
* Making migrations optional because they fail.
* “Temporary” bypasses with no ticket/expiry.
* Fixing flakes by removing code paths.
* Suppressing warnings without understanding them.

## Never let an agent make broad, hidden, or unrelated changes

Unacceptable behavior:

* Reformatting entire repositories during a feature change.
* Renaming public APIs without migration.
* Touching lockfiles without dependency intent.
* Editing generated files manually.
* Changing config defaults silently.
* Changing tests unrelated to the task.
* Moving files to satisfy imports instead of fixing package structure.
* Adding new frameworks/libraries for small tasks.
* Replacing stable code with a rewrite because it is easier for the model.
* Adding abstraction layers not required by the problem.
* Copy-pasting large chunks from old code and creating drift.

## Never let an agent operate with unnecessary authority

Unacceptable behavior:

* Agent has production secrets.
* Agent can push to protected branches.
* Agent can deploy.
* Agent can publish packages.
* Agent can approve its own PR.
* Agent can run arbitrary shell commands on a developer laptop with SSH keys/cloud tokens.
* Agent can access private data unrelated to the task.
* Agent can browse issue comments/webpages and execute instructions found there.
* Agent can modify CI permissions.
* Agent can write to `$HOME`, global package caches, shell startup files, SSH config, or cloud config.
* Agent can install packages globally.

## Never accept agent output without human-grade review

Unacceptable behavior:

* No diff review.
* No threat-model review for auth, parsing, subprocess, file, network, dependency, or secrets changes.
* No tests run locally/CI.
* No dependency review.
* No migration review.
* No rollback plan.
* No observability for new behavior.
* No documentation for changed interfaces.
* No owner for generated code.
* “The AI said it is secure” as evidence.

Georgia Tech’s 2026 writeup on “vibe coding” says researchers found AI-coded vulnerabilities that lead to breaches and specifically frames missing authentication as a design flaw rather than a typo; this aligns with the redline that agent output must be reviewed for design, not just syntax. ([Georgia Tech Research][24])

# 21. “Fixes” that are themselves never acceptable

These deserve their own list because they are common in rushed Python/vibe-coded patches:

* Fix TLS errors with `verify=False`.
* Fix auth errors by removing auth.
* Fix CSRF errors by disabling CSRF.
* Fix CORS by allowing all origins with credentials.
* Fix SQL errors by using string concatenation.
* Fix pickle/YAML errors by using more permissive loaders.
* Fix path errors by using absolute user-provided paths.
* Fix permission errors with `chmod -R 777`.
* Fix import errors by mutating `sys.path` randomly.
* Fix race conditions with `time.sleep`.
* Fix flaky tests by increasing sleeps.
* Fix timeouts by removing timeouts.
* Fix dependency conflicts by unpinning everything.
* Fix scanner noise by disabling scanners.
* Fix failing tests by deleting tests.
* Fix type errors with blanket `Any`.
* Fix lint errors with blanket `# noqa`.
* Fix exceptions with `except Exception: pass`.
* Fix production debug need by turning debug on.
* Fix performance by caching sensitive data forever.
* Fix “not found” by broadening access.
* Fix “invalid input” by accepting anything.

# 22. Compact policy version

A Python change should be blocked if it does any of the following:

1. Executes strings as code.
2. Runs shell commands from uncontrolled input.
3. Builds queries by string interpolation.
4. Deserializes untrusted objects.
5. Parses untrusted XML/YAML/archives without safe loaders and limits.
6. Commits, logs, prints, prompts, or exposes secrets.
7. Disables TLS verification, auth, CSRF, validation, scanners, tests, or logging.
8. Uses weak crypto or non-crypto randomness for security.
9. Trusts the client for identity, role, tenant, ownership, or path.
10. Handles files/uploads/archives without containment and limits.
11. Fetches arbitrary URLs server-side.
12. Runs debug/dev servers in production.
13. Uses broad exception swallowing or `assert` for production checks.
14. Ships without tests for new behavior and failure modes.
15. Adds dependencies without review, pinning/locking strategy, and vulnerability scanning.
16. Runs CI workflows with secrets/write permissions on untrusted code.
17. Lets an agent change broad/unrelated code or weaken gates to get green.
18. Produces code that cannot be read, reviewed, reproduced, monitored, or rolled back.

That is the “never justified” bar for Python under a security- and agent-friendly standard.
