Below is the release-governance checklist I would use for a serious engineering team. **“Best practice”** here means enforceable, auditable, automatable, and understandable by both developers and operators. **“Inexcusable”** means a preventable mistake that can break production, destroy traceability, leak secrets, bypass controls, or damage user trust.

I weighted official specifications and platform docs highest: Git, GitHub, GitLab, SemVer, Conventional Commits, Keep a Changelog, SLSA, NIST SSDF, OWASP, DORA, and Google SRE. I also checked community/social sources such as GitHub discussions and LinkedIn posts, but treated them as anecdotal signals rather than standards; they mostly reinforced the same themes: protect important branches, require review, use tags, run CI, write meaningful commits, and avoid sloppy history. ([Git][1])

# 1. The gold standard: what excellent Git + release practice looks like

## 1.1 Core principles

Every release should answer these questions without detective work: **what changed, who approved it, what commit produced it, what build produced the artifact, what tests passed, what security checks ran, what version was shipped, where it was deployed, how it can be rolled back, and what users need to know.** Google’s SRE guidance frames release engineering as the discipline of building and delivering software through repeatable, automated processes rather than “unique snowflakes.” ([Google SRE][2])

The healthiest teams optimize for **small changes, fast feedback, protected paths to production, automation, traceability, and reversibility.** DORA’s current software-delivery metrics focus on change lead time, deployment frequency, failed-deployment recovery time, change fail rate, and deployment rework rate; DORA also warns against gaming metrics and emphasizes improvement over competition. ([Dora][3])

A release is not just a Git tag. A proper release is a **complete, permanent, documented snapshot**: source revision, version, artifacts, notes, evidence, approvals, and operational context. GitLab describes releases as combining code, binaries, documentation, and release notes into a complete snapshot, with audit-ready evidence. GitHub releases similarly support release notes, contributors, and binary assets. ([GitLab Docs][4])

# 2. Standards you should adopt

## 2.1 Versioning

Use **Semantic Versioning** for libraries, APIs, SDKs, packages, shared internal modules, and anything consumed by other systems. SemVer’s core rule is `MAJOR.MINOR.PATCH`: increment MAJOR for incompatible API changes, MINOR for backward-compatible functionality, and PATCH for backward-compatible bug fixes. ([Semantic Versioning][5])

Use **Calendar Versioning** only when product/date cadence matters more than API compatibility, and document it explicitly. Do not mix SemVer-looking versions with non-SemVer behavior; that is worse than choosing a simpler scheme honestly.

Use pre-release identifiers for candidates and betas: `1.4.0-alpha.1`, `1.4.0-beta.2`, `1.4.0-rc.1`. SemVer defines pre-release precedence below the associated normal version, for example `1.0.0-alpha < 1.0.0`. ([Semantic Versioning][5])

## 2.2 Commit convention

Use **Conventional Commits** when you want automated changelog generation, release-note drafts, and semantic version bumping. The standard format is:

```text
<type>[optional scope]: <description>

[optional body]

[optional footer(s)]
```

`fix:` maps to PATCH, `feat:` maps to MINOR, and `BREAKING CHANGE:` or `!` maps to MAJOR under SemVer. ([Conventional Commits][6])

Good examples:

```text
fix(auth): reject expired refresh tokens

feat(billing): add annual invoice export

feat(api)!: remove legacy v1 payment endpoint

BREAKING CHANGE: clients must use /v2/payments.
```

Bad examples:

```text
updates
misc fixes
final changes
wip
fix stuff
```

## 2.3 Changelog standard

Maintain a human-readable changelog. Keep a Changelog defines a changelog as a curated, chronologically ordered list of notable changes and emphasizes that changelogs are for humans, not machines. It also recommends grouping changes, linking versions, showing release dates, and clearly listing breaking changes, deprecations, removals, fixes, and security changes. ([Keep a Changelog][7])

A good changelog has:

```markdown
## [1.8.0] - 2026-05-05

### Added
- Added SSO enforcement for enterprise accounts.

### Changed
- Improved import performance for large CSV files.

### Deprecated
- Deprecated API v1 token creation endpoint.

### Fixed
- Fixed incorrect timezone handling in billing exports.

### Security
- Patched dependency CVE-XXXX-YYYY.
```

Never dump raw Git logs as release notes. Keep a Changelog explicitly calls commit-log diffs noisy and unsuitable for communicating meaningful user-facing change. ([Keep a Changelog][7])

## 2.4 Tagging standard

Use **annotated tags** for real releases. Git’s own documentation says annotated tags are meant for releases, while lightweight tags are meant for private or temporary labels; annotated tags include tagger, date, message, and optional cryptographic signature. ([Git][1])

Recommended exact release tag format:

```text
v1.2.3
v1.2.4
v2.0.0-rc.1
```

Acceptable component tag format in monorepos:

```text
api/v1.2.3
web/v3.4.0
worker/v0.9.1
```

Prefer signed tags and signed release commits for high-trust projects. GitHub supports signing commits and tags with GPG, SSH, or S/MIME and marks verifiable signatures as verified or partially verified. ([GitHub Docs][8])

## 2.5 Branch/review protection standard

Protect `main`, `master`, `trunk`, `release/*`, `hotfix/*`, and release tags. GitHub rulesets can restrict branch/tag creation, updates, and deletion; require pull requests; require signed commits; require status checks; require deployments; require linear history; block force pushes; and require code scanning or code quality results. ([GitHub Docs][9])

At minimum, protected branches should require:

1. Pull request before merge.
2. Required approvals.
3. CODEOWNERS approval for owned files.
4. Required CI status checks.
5. No force pushes.
6. No deletion.
7. Dismiss stale approvals or require approval of the most recent push.
8. Conversation resolution before merge.
9. Merge queue or merge train for busy repos.
10. Admin bypasses disabled or tightly audited.

GitHub CODEOWNERS can automatically request owners and can be required by branch protection; GitHub notes that if multiple owners are listed, approval from any one of them is sufficient unless stricter rulesets are configured. ([GitHub Docs][10])

## 2.6 CI/CD security standard

Treat CI/CD as production infrastructure. NSA and CISA explicitly warn that CI/CD pipelines are key DevSecOps infrastructure and recommend hardening authentication, access control, development environments, tools, and the development process. ([NSA][11])

For GitHub Actions, follow least privilege for workflow credentials and secrets. GitHub warns that write-access users can read repository secrets through workflows, so credentials used in workflows should have the least privileges required. GitHub also recommends pinning workflow dependencies to a specific commit SHA to limit compromised dependency impact. ([GitHub Docs][12])

For GitLab, use protected environments and protected variables. GitLab protected environments restrict who can deploy to sensitive environments, and protected variables are only passed to pipelines running on protected branches or protected tags. ([GitLab Docs][13])

## 2.7 Supply-chain security standard

Use SLSA, NIST SSDF, and OWASP as the high-level security frame. SLSA v1.2 describes itself as an industry-consensus specification for incrementally improving supply-chain security, with levels and tracks plus recommended attestation formats including provenance. SLSA’s public overview describes it as a checklist of standards and controls to prevent tampering, improve integrity, and secure packages and infrastructure. ([SLSA][14])

NIST SSDF is a set of secure software development practices designed to reduce vulnerabilities in released software and address root causes; its groups include preparing the organization, protecting software, producing well-secured software, and responding to vulnerabilities. ([NIST Computer Security Resource Center][15])

OWASP SCVS focuses on identifying activities, controls, and best practices to reduce software supply-chain risk, while OWASP SAMM’s Secure Build practice emphasizes standardized, repeatable builds using secure components and automated security checks. ([OWASP][16])

# 3. Best practices checklist

## 3.1 Repository governance

Best practices:

1. Every repository has clear owners.
2. Every important path has CODEOWNERS.
3. Default branch is protected.
4. Release branches are protected.
5. Release tags are protected.
6. Admin bypasses are rare, logged, and reviewed.
7. Nobody force-pushes to shared protected branches.
8. Nobody deletes release tags.
9. Repository settings are documented or managed as code.
10. Access is least privilege: read, triage, write, maintain, admin.
11. Production deployment permission is separate from ordinary merge permission.
12. CI/CD service accounts have minimal scopes.
13. Bot credentials are rotated and owned.
14. Secrets are never stored in Git.
15. Secret scanning is enabled where available.
16. Security alerts have owners and SLAs.
17. Archived repositories are read-only and clearly marked.
18. Abandoned branches are deleted or marked stale.
19. Fork and external-contributor workflows are locked down.
20. Merge rights are not handed out casually.

Inexcusable mistakes:

1. Unprotected `main`.
2. Everyone can push directly to `main`.
3. Everyone can create or move release tags.
4. Everyone can deploy production.
5. Shared admin accounts.
6. Personal access tokens used as permanent deployment credentials.
7. Secrets in `.env`, source files, test fixtures, release archives, or Docker images.
8. No owner for a production repository.
9. No audit trail for who approved and shipped a release.
10. Emergency bypasses with no retroactive review.

## 3.2 Branching strategy

Best practices:

1. Pick one workflow and document it.
2. Prefer trunk-based development, GitHub Flow, or GitLab Flow for continuous delivery.
3. Use GitFlow only when you truly have release trains, parallel supported versions, or packaged software that cannot be continuously deployed.
4. Keep feature branches short-lived.
5. Keep `main` green and releasable.
6. Branch from `main` unless your documented workflow says otherwise.
7. Use descriptive branch names: `feature/customer-export`, `fix/login-timeout`, `hotfix/v1.8.2-token-refresh`.
8. Delete merged branches.
9. Keep release branches narrowly focused on stabilization, backports, and fixes.
10. Do not use permanent environment branches unless your deployment model explicitly requires them and the flow is automated.
11. Backport fixes intentionally; do not let branches drift.
12. Merge or cherry-pick hotfixes back into the active development line.
13. Avoid long-running “integration” branches that become dumping grounds.
14. Use feature flags or branch-by-abstraction for incomplete work.
15. Document how release branches are created, frozen, patched, and retired.

GitHub Flow is a lightweight branch-based workflow; GitLab Flow is a simpler alternative to GitFlow that keeps work centered on `main` while supporting production/stable branches when needed. GitLab’s own best-practice list includes testing all commits, code review before merging, automated deployments based on branches or tags, releases based on tags, and everyone starting from and targeting `main`. ([GitHub Docs][17])

GitFlow is not evil, but it is often overused. Atlassian summarizes GitFlow as a release-based workflow with `develop`, `release`, `feature`, and `hotfix` branches, and says it is good for release-based software workflows. That does not make it ideal for teams practicing frequent continuous delivery. ([Atlassian][18])

Inexcusable mistakes:

1. No documented branching model.
2. A fake workflow where every exception becomes normal.
3. Long-lived feature branches lasting weeks or months.
4. `main` is often broken.
5. `develop` is broken but “that’s normal.”
6. Release branches accept random features after freeze.
7. Hotfixes are applied to production but never merged back.
8. Multiple branches claim to represent production.
9. Environment branches drift from each other with manual cherry-picks.
10. Developers are afraid to merge because integration happens too late.

## 3.3 Commit hygiene

Best practices:

1. Make commits atomic.
2. Each commit should explain why, not only what.
3. Separate refactors from behavior changes.
4. Separate formatting-only changes from logic changes.
5. Use Conventional Commits if automation depends on commit history.
6. Reference tickets, issues, incidents, or design docs.
7. Avoid committing generated noise unless required.
8. Do not commit local debug changes.
9. Do not commit commented-out experiments.
10. Do not commit secrets.
11. Do not rewrite public/shared history unless the team explicitly approves.
12. Squash messy WIP commits before merging if your workflow uses squash merges.
13. Preserve useful history when reviewing complex changes.
14. Keep authorship accurate.
15. Use signed commits for sensitive or high-trust repos.

Inexcusable mistakes:

1. Commit messages like `stuff`, `changes`, `fix`, `final`, `really final`.
2. One giant commit mixing unrelated changes.
3. Rewriting protected branch history.
4. Force-pushing over teammates’ work.
5. Committing secrets and treating deletion as sufficient remediation.
6. Committing build artifacts or dependency folders that should not be versioned.
7. Committing broken code directly to shared branches.
8. Mislabeling a breaking change as a patch.
9. Hiding risky changes inside “refactor” commits.
10. Using someone else’s identity or shared credentials.

## 3.4 Pull requests / merge requests

Best practices:

1. PRs are small enough to review well.
2. PR title is meaningful and release-note friendly.
3. PR description explains intent, risk, testing, rollout, rollback, and screenshots where useful.
4. PR links to issue/ticket/design.
5. PR includes tests or explains why not.
6. PR updates docs when behavior changes.
7. PR updates changelog or release-note metadata when user-facing.
8. PR includes migration notes for schema/API/config changes.
9. PR includes observability updates for new production paths.
10. PR includes security notes for auth, permissions, crypto, data handling, or secrets.
11. CODEOWNERS are required for critical areas.
12. Stale reviews are dismissed or the most recent push needs approval.
13. All review comments are resolved before merge.
14. CI must pass before merge.
15. Merge method is consistent: squash, merge commit, or rebase.
16. Merge queues or merge trains are used for high-volume repos.
17. Large changes are split behind flags.
18. Risky changes get staged rollout plans.
19. Breaking changes require explicit approval.
20. Emergency merges are documented.

GitHub merge queue is designed to automate merges into busy branches while ensuring incompatible changes do not break the target branch; GitLab merge trains similarly queue merge requests and test combined changes before merging. ([GitHub Docs][19])

Inexcusable mistakes:

1. Self-approving risky production changes.
2. Merging without required checks.
3. Ignoring failing tests.
4. Rubber-stamp approvals on massive PRs.
5. Sneaking unrelated changes into a PR.
6. Merging after approvals are stale.
7. Bypassing CODEOWNERS because “it is urgent.”
8. No test evidence for production code.
9. No rollback notes for high-risk changes.
10. Review comments unresolved but merged anyway.

## 3.5 CI quality gates

Best practices:

1. CI runs on every PR/MR.
2. CI runs on every push to protected branches.
3. CI runs on every release tag.
4. Required checks block merges.
5. Required checks block release creation or deployment.
6. Tests include unit, integration, contract, migration, and end-to-end where appropriate.
7. Flaky tests are treated as defects.
8. Slow tests are parallelized or split intelligently.
9. Build artifacts are produced by CI, not by laptops.
10. The same build path is used for release candidates and final releases.
11. CI configuration is reviewed like production code.
12. CI dependencies are pinned.
13. Third-party actions are pinned to immutable references.
14. Workflow permissions default to read-only.
15. Secrets are scoped by environment and branch/tag.
16. Pull requests from forks do not receive production secrets.
17. Self-hosted runners are isolated and ephemeral where possible.
18. Caches are scoped safely.
19. Release jobs cannot be triggered by untrusted code.
20. CI logs do not leak secrets.
21. Security scans are part of the pipeline.
22. Dependency review blocks newly introduced vulnerable dependencies.
23. Code scanning runs on relevant languages.
24. Container/IaC scans run where relevant.
25. SBOM/provenance generation happens during build.

GitHub Dependabot alerts help find vulnerable dependencies, and dependency review is intended to prevent introducing vulnerable dependencies rather than fixing them later. ([GitHub Docs][20])

Inexcusable mistakes:

1. CI only runs after merge.
2. Release job ignores test failures.
3. Production deploy job can run from any branch.
4. Untrusted fork PRs can access secrets.
5. Actions or build plugins use floating branches like `main`.
6. Workflow has broad write/admin permissions by default.
7. CI secrets are available to all branches.
8. A red build is normal.
9. Flaky tests are ignored for months.
10. Local-only “works on my machine” release process.

## 3.6 Tagging and release identity

Best practices:

1. Every release has exactly one immutable version identity.
2. Exact release tags are immutable: `v1.2.3` must always mean the same commit.
3. Use annotated tags for releases.
4. Sign release tags where practical.
5. Protect release tags from creation, update, and deletion by unauthorized users.
6. Tag only commits that passed release CI.
7. Tag from `main` or an approved release branch.
8. Use pre-release tags for candidates: `v1.2.3-rc.1`.
9. Keep Git tag, package version, container tag, release title, and changelog version aligned.
10. Include commit SHA in release metadata.
11. Include artifact checksums.
12. Use immutable container image digests for deployment.
13. Do not depend on mutable tags like `latest` for production rollouts.
14. If you use moving channel tags such as `v1` or `stable`, document that they are channels, not release records.
15. Never reuse version numbers.
16. Never delete a bad release silently; mark it yanked, superseded, or deprecated.
17. Publish a patch version for fixes.
18. Keep old release tags available.
19. Use signed provenance/attestation for high-trust releases.
20. Make release comparison links easy to find.

Inexcusable mistakes:

1. Moving `v1.2.3` to a different commit.
2. Deleting and recreating release tags to hide mistakes.
3. Reusing a version number for different artifacts.
4. Creating a release from an unreviewed local commit.
5. Tagging a dirty working tree.
6. Tagging a commit that never passed CI.
7. Publishing binaries that do not match the tagged source.
8. `latest` is the only production version.
9. “Version 2” in docs, `1.9.0` in package, `v1.8.7` in Git.
10. No way to know which commit is running in production.

## 3.7 Build and artifact management

Best practices:

1. Build once, promote the same artifact across environments.
2. Do not rebuild separately for staging and production.
3. Build in controlled CI/CD.
4. Use reproducible or at least repeatable builds.
5. Capture build metadata: commit SHA, tag, builder, timestamp, dependency lockfiles, compiler/runtime versions.
6. Generate SBOMs.
7. Generate provenance/attestations.
8. Sign artifacts or checksums.
9. Store artifacts in a registry with retention.
10. Scan artifacts before release.
11. Scan containers before promotion.
12. Use immutable digests in deployment manifests.
13. Keep source archives, binaries, SBOMs, checksums, and release notes together.
14. Make artifact promotion auditable.
15. Separate build credentials from deployment credentials.
16. Do not embed secrets in artifacts.
17. Do not build from mutable dependencies without lockfiles.
18. Do not let release artifacts expire before your support window ends.
19. Document how to reproduce a release.
20. Verify that published package contents match expectations.

Inexcusable mistakes:

1. Building releases on a developer laptop.
2. Uploading a manually zipped folder as the official release.
3. Rebuilding in production.
4. No lockfile for production dependencies.
5. No artifact checksum.
6. No artifact retention.
7. No way to rebuild or verify a release.
8. Production artifact differs from tested artifact.
9. Container image overwritten under the same tag.
10. Release artifacts contain `.env`, private keys, test data, or internal credentials.

## 3.8 Release notes and changelog

Best practices:

1. Release notes are written for users.
2. Changelog is maintained continuously.
3. Use an `Unreleased` section.
4. Group changes: Added, Changed, Deprecated, Removed, Fixed, Security.
5. Clearly label breaking changes.
6. Clearly label deprecations.
7. Include migration instructions.
8. Include config changes.
9. Include database migration notes.
10. Include known issues.
11. Include rollback caveats.
12. Include security fixes and CVEs where disclosure policy allows.
13. Include contributors where appropriate.
14. Include comparison links.
15. Use auto-generated notes as a draft, not as the final truth.
16. Exclude internal noise.
17. Avoid jargon users cannot act on.
18. Update docs before or with release.
19. Make release notes discoverable from the release page.
20. Keep release notes immutable except for correction notices.

GitHub can generate release notes, but its own docs instruct maintainers to check generated notes to ensure they include all and only the desired information. ([GitHub Docs][21])

Inexcusable mistakes:

1. No release notes.
2. Raw Git log as release notes.
3. Breaking change hidden under “minor fixes.”
4. Security fix shipped silently when policy requires disclosure.
5. Deprecation not documented.
6. Migration required but not mentioned.
7. “Various improvements” as the whole release note.
8. Release notes contradict actual shipped artifact.
9. Changelog updated after users complain.
10. No public/internal record of what changed.

## 3.9 Deployment and rollout

Best practices:

1. Separate **deployment** from **release** where possible.
2. Use feature flags for incomplete or risky features.
3. Use canary, blue/green, rolling, or progressive delivery for risky systems.
4. Deploy small changes frequently.
5. Automate deployments.
6. Protect production environments.
7. Require approvals for high-risk production deploys.
8. Use environment-specific secret scoping.
9. Run smoke tests after deploy.
10. Monitor golden signals after deploy.
11. Define automatic or manual rollback criteria.
12. Keep rollback commands tested.
13. Prefer roll-forward for simple defects, rollback for dangerous changes.
14. Use backward-compatible database migrations.
15. Use expand/migrate/contract for schema changes.
16. Make config changes auditable.
17. Use maintenance windows only when genuinely necessary.
18. Communicate customer-impacting changes before release.
19. Track deployment status centrally.
20. Keep deployment runbooks current.

Google SRE recommends canarying to reduce deployment risk by testing a change on a small portion of traffic before full rollout; Martin Fowler’s feature-toggle guidance explains how release toggles let teams merge incomplete work into a shared branch while keeping it inactive in production. ([Google SRE][22])

Inexcusable mistakes:

1. No rollback plan.
2. Rollback plan never tested.
3. One-click production deployment available to everyone.
4. Manual SSH changes to production.
5. Database migration cannot be rolled forward or backward safely.
6. Feature flag permanently left behind with no owner.
7. No monitoring after deploy.
8. No alerting for the newly released path.
9. Big-bang release with no staged rollout for high-risk change.
10. Deployment success means “job exited 0,” not “service is healthy.”

## 3.10 Hotfixes and incident releases

Best practices:

1. Start from the affected production tag or release branch.
2. Keep the hotfix minimal.
3. Reproduce the bug with a test.
4. Fix the bug.
5. Run full relevant CI.
6. Tag a patch release.
7. Deploy through the same controlled release path.
8. Merge or cherry-pick the fix back to `main`.
9. Backport to all supported release branches.
10. Update changelog and release notes.
11. Notify affected users if needed.
12. Write an incident review or postmortem for severe issues.
13. Record affected versions.
14. Record fixed versions.
15. Remove temporary mitigations after the permanent fix.

Inexcusable mistakes:

1. Patching production directly.
2. Hotfix not committed to Git.
3. Hotfix committed only to a dead release branch.
4. No test proving the fix.
5. No patch version.
6. No note that the release fixes an incident.
7. No backport to active branches.
8. Same incident recurs because no postmortem action was taken.
9. Emergency bypass becomes routine.
10. “We will clean it up later” with no ticket, owner, or deadline.

## 3.11 Dependencies and third-party code

Best practices:

1. Use lockfiles.
2. Review dependency changes.
3. Enable vulnerability alerts.
4. Enable dependency update automation.
5. Block known critical vulnerabilities unless explicitly risk-accepted.
6. Track licenses.
7. Pin CI actions and plugins.
8. Avoid unmaintained dependencies.
9. Avoid dependencies with suspicious ownership changes.
10. Maintain an SBOM.
11. Audit transitive dependencies for critical systems.
12. Keep base images updated.
13. Use private registries carefully.
14. Verify package provenance where possible.
15. Remove unused dependencies.
16. Separate dev dependencies from production dependencies.
17. Use minimal runtime images.
18. Watch for typosquatting.
19. Document approved package sources.
20. Have a process for emergency dependency patches.

Inexcusable mistakes:

1. Floating dependency versions in production.
2. No lockfile.
3. Ignoring known exploited vulnerabilities.
4. Pulling build scripts from random URLs.
5. Running third-party install scripts with production credentials.
6. Using abandoned packages in critical paths without review.
7. No license awareness.
8. Dependency updates merged without tests.
9. CI actions pinned to mutable branches.
10. No owner for dependency hygiene.

## 3.12 Database and schema releases

Best practices:

1. Treat schema changes as release changes.
2. Prefer backward-compatible migrations.
3. Use expand/migrate/contract.
4. Separate destructive changes from additive changes.
5. Test migrations on production-like data volume.
6. Estimate migration time.
7. Make migrations idempotent where possible.
8. Back up before risky migrations.
9. Monitor migration progress.
10. Avoid long locks.
11. Include rollback/roll-forward strategy.
12. Coordinate app and DB deploy order.
13. Avoid deploying code that requires a schema not yet present.
14. Avoid dropping columns still read by old code.
15. Version migration scripts.
16. Review data migrations carefully.
17. Keep PII handling explicit.
18. Include migration notes in release notes.
19. Have a manual stop plan.
20. Validate after migration.

Inexcusable mistakes:

1. Destructive migration with no backup.
2. Migration tested only on empty local DB.
3. App deploy and DB migration order guessed manually.
4. Rollback impossible but not disclosed.
5. Long table lock during peak traffic.
6. Dropping fields before all services stop using them.
7. No owner watching the migration.
8. No metric/log for migration status.
9. Data correction performed outside version control.
10. Breaking customer data without an incident process.

## 3.13 Feature flags

Best practices:

1. Flags have names, owners, creation dates, and expiry dates.
2. Flags are categorized: release, experiment, ops, permission.
3. Defaults are safe.
4. Flag state is observable.
5. Critical flag changes are audited.
6. Test important flag combinations.
7. Remove release flags after rollout.
8. Avoid nested flag spaghetti.
9. Avoid long-lived flags unless they are true product permissions.
10. Document operational kill switches.
11. Avoid evaluating remote flags in hot loops without caching.
12. Ensure disabled code paths are safe.
13. Use flags to reduce branch lifetime, not to hide unfinished chaos forever.
14. Include flag cleanup in definition of done.
15. Include flag states in incident debugging.

Martin Fowler warns that toggles introduce complexity and should be categorized and managed deliberately; release toggles are useful for continuous delivery but should not become permanent clutter. ([martinfowler.com][23])

Inexcusable mistakes:

1. Feature flag with no owner.
2. Flag left forever after release.
3. Flag default exposes unfinished feature.
4. Production incident because nobody knows current flag state.
5. Flag removed without checking users still depend on it.
6. Flag changes not audited.
7. Flag combinations never tested.
8. Flag names are vague: `newFlow`, `test2`, `temp`.
9. Kill switch cannot be used without deploy.
10. Flag service outage takes down core product.

## 3.14 Observability and post-release verification

Best practices:

1. Define success metrics before release.
2. Define failure metrics before release.
3. Add dashboards before rollout.
4. Add logs/traces for new paths.
5. Add alerts for critical regressions.
6. Watch error rate, latency, saturation, and business KPIs.
7. Compare canary against control.
8. Monitor background jobs and queues.
9. Monitor dependency calls.
10. Monitor database load.
11. Monitor customer-impacting workflows.
12. Verify release version visible at runtime.
13. Record deploy events in observability tools.
14. Have on-call aware of high-risk releases.
15. Keep release owner available during rollout.
16. Create incident quickly if user impact appears.
17. Feed release failures into postmortems.
18. Track change failure rate.
19. Track failed deployment recovery time.
20. Improve the release process after every serious miss.

Inexcusable mistakes:

1. Release shipped and nobody watches it.
2. No way to tell which version is running.
3. No dashboards for new feature.
4. No alert for obvious failure mode.
5. Release declared successful before smoke tests.
6. Customer reports are the first monitoring signal.
7. No deploy markers in logs/metrics.
8. On-call unaware of risky release.
9. Same release failure repeats.
10. Metrics improve on paper because failures are hidden.

# 4. The “inexcusable mistakes” master list

These are the ones I would treat as serious process violations.

## 4.1 Git history and branch control

1. Direct pushes to protected production branches.
2. Force-pushing shared protected branches.
3. Deleting protected branches without approval.
4. Moving exact release tags.
5. Deleting exact release tags.
6. Reusing version numbers.
7. Rewriting public history to hide mistakes.

# 5. Recommended reference workflow

A strong default workflow for most modern product teams:

1. Developer branches from `main`.
2. Developer makes small commits using Conventional Commits.
3. Developer opens PR.
4. PR template requires summary, test evidence, risk, rollout, rollback, and release-note impact.
5. CODEOWNERS are automatically requested.
6. CI runs tests, lint, build, security scans, dependency review, and migration checks.
7. Required reviewers approve.
8. Required checks pass.
9. Merge queue validates the PR against latest `main`.
10. PR merges to `main`.
11. Main branch build creates immutable artifact.
12. Artifact is scanned, signed, and stored.
13. Deployment to staging uses the same artifact.
14. Staging smoke tests pass.
15. Release candidate tag is created if needed: `v1.4.0-rc.1`.
16. Production deployment is approved if risk requires it.
17. Production deploy uses progressive rollout.
18. Observability confirms health.
19. Final tag/release is created or promoted: `v1.4.0`.
20. Release notes and changelog are published.
21. Old feature flags are cleaned up.
22. DORA/release metrics are reviewed for improvement.

For libraries/packages, reverse steps 18–20 slightly: tag first from the approved release commit, let CI build from the tag, publish immutable package artifacts, then publish release notes.

# 7. Policy defaults I would enforce immediately

For a professional team, these should be non-negotiable:

1. Protect `main`.
2. Protect `release/*`.
3. Protect `v*` release tags.
4. Require PRs.
5. Require at least one approval; two for critical services.
6. Require CODEOWNERS for critical paths.
7. Require CI status checks.
8. Require security/dependency checks for production systems.
9. Block force pushes.
10. Block branch/tag deletion except release admins.
11. Require signed commits or tags for sensitive repositories.
12. Build releases only in CI.
13. Build once, promote artifact.
14. Pin CI actions/plugins.
15. Scope secrets to protected branches/tags/environments.
16. Maintain changelog.
17. Use SemVer where API compatibility matters.
18. Never move exact release tags.
19. Record release evidence.
20. Have a tested rollback or roll-forward plan.

# 8. The highest-signal maturity test

Ask the team to pick any production version from six months ago and answer in under five minutes:

1. What commit was it built from?
2. Who approved it?
3. What CI run built it?
4. What tests passed?
5. What security checks passed?
6. What artifact was deployed?
7. What dependencies were inside it?
8. What changed from the previous release?
9. What users were told?
10. How would we roll it back or patch it?

If the team cannot answer those quickly, the release process is not mature yet.
