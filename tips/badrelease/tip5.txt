Below is a **standards-grade Git and release checklist** you can turn directly into an engineering policy. I weighted official/primary sources highest: Git/GitHub docs, SemVer, DORA, NIST SSDF, OWASP, OpenSSF/SLSA, and Google SRE. Practitioner discussions on GitHub, LinkedIn, X, Quora-style Q&A, and blogs mostly reinforced the same themes: short-lived branches, protected main/release branches, immutable tags, automated CI/CD, explicit changelogs, secrets discipline, and rollback readiness.

## 1. The core standard

A mature Git and release process should do four things extremely well:

1. **Protect the source of truth.** The default branch, release branches, release tags, and release artifacts must be protected from accidental or unauthorized change. GitHub branch protections and rulesets support required reviews, status checks, signed commits, linear history, merge queues, tag restrictions, and blocking force pushes/deletions. ([GitHub Docs][1])

2. **Integrate continuously.** High-performing teams keep branches short-lived, merge to trunk frequently, avoid late integration phases, and keep the mainline deployable. DORA explicitly ties trunk-based development to continuous integration and warns that long-lived branches cause complex merges, code freezes, and stabilization phases. Martin Fowler’s CI guidance says every developer should integrate at least daily and every integration should be verified by automated build/tests. ([Dora][2])

3. **Release from automation, not heroics.** Releases should be produced by a versioned, repeatable pipeline after successful build/test/security checks. Google SRE describes release engineering as defining consistent, repeatable release methodology, including build steps, flags, build IDs, canaries, and rollback strategy. ([Google SRE][3])

4. **Make every release traceable, reversible, and explainable.** A release should map to an exact commit/tag, version, changelog, artifacts, checksums/signatures, SBOM/provenance where appropriate, deployment record, and rollback path. GitHub releases are based on Git tags, and immutable releases can prevent assets or associated tags from being changed after publication. ([GitHub Docs][4])

# 2. Best practices: the complete list

## A. Repository hygiene

### Must-have practices

1. **One canonical repository and default branch.** Everyone knows where the authoritative source lives.

2. **Protect the default branch.** No direct pushes. Require pull requests, required checks, required reviews, resolved conversations, and no force pushes/deletions. Apply protections to admins too, or require explicit break-glass logging. GitHub notes that branch protections can restrict force pushes, require reviews/status checks, require signed commits, require linear history, and block deletion; protections do not apply to admins unless configured. ([GitHub Docs][1])

3. **Use rulesets for branch and tag governance.** Protect `main`, `release/*`, `hotfix/*`, and release tags such as `v*`. Rulesets can restrict branch/tag creation, updates, deletion, require pull requests, require status checks, block force pushes, require signed commits, and enforce code scanning or quality gates. ([GitHub Docs][5])

4. **Keep generated files, local config, secrets, caches, and build outputs out of Git.** Commit a project-level `.gitignore`; GitHub’s docs recommend committing `.gitignore` so all clones share the same ignore rules. If a file is already tracked, remove it from tracking before expecting `.gitignore` to work. ([GitHub Docs][6])

5. **Keep repositories small and source-focused.** Do not use Git as a binary artifact store. GitHub recommends keeping repos ideally under 1 GB and strongly recommends staying under 5 GB; large deliverables should be distributed through releases or appropriate artifact/package storage. ([GitHub Docs][7])

6. **Use package managers and registries for dependencies.** Do not vendor enormous dependency trees unless there is a clear, documented reason.

7. **Use Git LFS only for genuine large source assets.** For example: media, model files, design assets, game assets. Do not put release builds in Git LFS as a substitute for a release registry.

8. **Require clear ownership files.** Use `CODEOWNERS` for automatic reviewer assignment and ownership enforcement. GitHub supports requiring code owner approval before merge. ([GitHub Docs][8])

9. **Have standard repo documents.** At minimum: `README`, `CONTRIBUTING`, `SECURITY`, `LICENSE`, `CHANGELOG`, `CODEOWNERS`, issue/PR templates, release process docs, and support policy.

10. **Make local development reproducible.** Provide setup scripts, dev containers, lockfiles, pinned tool versions, or documented environment requirements.

## B. Commit discipline

### Must-have practices

1. **Make commits atomic.** One logical change per commit. A bug fix, refactor, dependency bump, formatting change, and feature should not be mixed unless inseparable.

2. **Use meaningful commit messages.** The subject should explain what changed; the body should explain why when context matters.

3. **Prefer Conventional Commits when automation matters.** Conventional Commits defines a lightweight message structure such as `feat:`, `fix:`, and `BREAKING CHANGE`, which helps automate versioning and changelog generation and aligns with SemVer. ([Conventional Commits][9])

4. **Do not leave `WIP`, `fix typo`, `oops`, `changes`, or `final final` commits on protected branches.** Squash, reword, or clean them before merge.

5. **Do not rewrite shared history casually.** The Git book’s guidance is clear: rewriting local history before sharing is fine, but once work is pushed, treat it as final unless there is a strong reason. Amending a pushed commit changes its identity and can disrupt collaborators. ([Git][10])

6. **Use signed commits or signed tags for high-trust projects.** At minimum, release tags should be signed or otherwise verified.

7. **Reference issues, tickets, incidents, or design docs where useful.** Traceability matters during audits, incident reviews, and release notes generation.

## C. Branching strategy

### Preferred default: trunk-based or GitHub Flow

For most SaaS, web, API, platform, internal tools, and continuously delivered products, the default should be:

1. **`main` is always releasable.**
2. **Feature branches are short-lived.**
3. **Merge to `main` frequently.**
4. **Use feature flags, branch-by-abstraction, and backward-compatible changes for incomplete work.**
5. **Release from `main` or from a release tag created from `main`.**

DORA describes trunk-based development as essential for continuous integration, with developers integrating small batches frequently and avoiding long-lived branches, code freezes, and integration phases. ([Dora][2])

### When Gitflow is acceptable

Gitflow is not inherently wrong. It can fit products with **scheduled releases**, **multiple supported versions**, **on-prem/customer-installed software**, or **separate maintenance lines**. Atlassian describes Gitflow as a workflow well suited to release-based development with dedicated feature, release, and hotfix branches. ([Atlassian][11])

But Gitflow becomes harmful when teams use it by habit while claiming to do continuous delivery. Long-lived `develop`, `release/*`, and feature branches often create late integration risk, duplicated fixes, and painful stabilization.

### Branching rules

1. **Use clear branch names.** Examples:
   `feature/payment-retry`, `fix/login-timeout`, `hotfix/v2.4.1-token-refresh`, `release/2.5`.

2. **Keep feature branches short-lived.** Prefer hours or days, not weeks.

3. **Rebase local branches before opening or updating a PR when appropriate.** Do not rebase branches that others are actively using unless the team explicitly coordinates it.

4. **Do not share long-lived personal branches.**

5. **Use release branches only for real release/support needs.** Examples: maintaining `release/1.8` while `main` advances to `2.0`.

6. **Lock release branches.** Require PRs, CI, approvals, and backport labels.

7. **Backport intentionally.** Fix the bug on the mainline first, then cherry-pick/backport to supported release branches unless the emergency demands the reverse.

8. **Delete merged branches.** Keep branch lists clean.

9. **Do not use branches as environments.** `dev`, `qa`, `staging`, and `prod` branches usually become drift factories. Environments should be deployment targets, not divergent source histories.

## D. Pull request standards

### Required PR rules

1. **Every non-trivial change goes through a PR.**

2. **PRs must be small enough to review well.** A good PR has one purpose. Large PRs should be split by behavior, not by arbitrary files.

3. **Use PR templates.** Include:

   * What changed
   * Why it changed
   * Screenshots or API examples when useful
   * Test evidence
   * Migration steps
   * Rollback notes
   * Security/privacy impact
   * Breaking changes
   * Observability changes

4. **Require review approval.** GitHub supports requiring a set number of approving reviews before merge. ([GitHub Docs][12])

5. **Require CODEOWNER approval for sensitive areas.** Examples: auth, billing, infra, crypto, release pipeline, database migrations, public APIs, compliance code.

6. **Dismiss stale approvals after meaningful changes.**

7. **Require all conversations resolved.**

8. **Do not allow authors to approve their own PRs.**

9. **Use draft PRs for early collaboration, not as a bypass.**

10. **Treat review as engineering work, not ceremony.** Review correctness, maintainability, security, tests, observability, performance, and release risk.

11. **Use a consistent merge strategy.** Common policies:

* **Squash merge** for clean product history.
* **Rebase merge** for linear commit history.
* **Merge commits** when preserving branch topology matters.

The worst choice is letting every team member use a different strategy randomly.

12. **Use merge queues for busy repos.** GitHub’s merge queue ensures changes pass required checks against the latest target branch plus queued changes before merging. ([GitHub Docs][1])

## E. CI standards

### Required CI behavior

1. **CI runs on every PR and every push to protected branches.** Fowler’s CI guidance emphasizes automated builds/tests on integration and fixing broken builds immediately. ([martinfowler.com][13])

2. **Required checks block merge.** Lint, formatting, type checks, unit tests, build, integration tests, security checks, and package validation should be required where applicable.

3. **CI must test the merge result, not only the branch head.** This prevents “green PR, red main.”

4. **Main must stay green.** A broken mainline is an all-hands priority.

5. **Fix flaky tests, do not normalize them.** Quarantining may be temporary, but flaky required tests are a release-risk smell.

6. **Use clean, reproducible environments.** CI should not depend on a developer’s laptop, local cache, or hidden global state.

7. **Pin toolchain versions.** Language runtime, package manager, compiler, Docker base image, and test dependencies should be controlled.

8. **Cache carefully.** Caches speed builds but can create correctness and security problems. Release and publishing workflows should be especially conservative.

9. **Separate validation from deployment.** Build/test should be possible without production credentials.

10. **Run security checks early.** Include dependency scanning, secret scanning, SAST, container scanning, IaC checks, and license checks where relevant.

11. **Build once, promote the same artifact.** Do not rebuild separately for staging and production. Rebuilding can produce different artifacts and destroys confidence.

12. **Store CI configuration in version control.** Build and release behavior should be reviewed like code.

## F. Versioning standards

### Use SemVer when you expose an API or contract

Semantic Versioning defines `MAJOR.MINOR.PATCH`: increment major for incompatible API changes, minor for backward-compatible functionality, and patch for backward-compatible fixes. It also defines pre-release and build metadata labels. ([Semantic Versioning][14])

Required rules:

1. **Declare what your public API or compatibility contract is.** SemVer requires a public API to be declared; this may be code API, HTTP API, CLI behavior, file format, infrastructure module interface, plugin API, or package contract. ([Semantic Versioning][14])

2. **Never modify released contents under the same version.** SemVer says once a versioned package is released, its contents must not be modified; changes must be released as a new version. ([Semantic Versioning][14])

3. **Use pre-release identifiers for unstable releases.** Examples: `1.4.0-alpha.1`, `1.4.0-beta.2`, `1.4.0-rc.1`.

4. **Use build metadata for traceability.** Example: `1.4.0+sha.abc1234`. Remember build metadata does not change precedence under SemVer.

5. **Do not label breaking changes as patch or minor.**

6. **Do not use `latest` for prerelease builds unless the ecosystem explicitly expects that.**

7. **For monorepos, define versioning mode.** Either:

   * one repo-wide version, or
   * independent package versions.

   Do not mix both informally.

8. **Use a consistent tag convention.** Common:

   * `v1.2.3` for single-package repos
   * `service-a/v1.2.3` for monorepos
   * `frontend@1.2.3` for package-oriented repos

   SemVer notes that `v1.2.3` is a common tag name, while the semantic version itself is `1.2.3`. ([Semantic Versioning][14])

## G. Tags and GitHub Releases

### Required release object rules

1. **Every production release has a Git tag.** GitHub releases are based on Git tags that mark a point in repository history. ([GitHub Docs][4])

2. **Release tags are immutable.** Do not move, delete, or recreate a published release tag. GitHub immutable releases can lock assets and the associated Git tag, preventing tag movement/deletion and asset modification/deletion while the release exists. ([GitHub Docs][15])

3. **Prefer annotated and signed tags for releases.**

4. **Tag from the exact commit that produced the artifact.**

5. **Release artifacts must identify their source.** Include version, commit SHA, build ID, build timestamp, and build environment metadata where applicable.

6. **Attach binaries, checksums, signatures, SBOMs, provenance, migration notes, and docs to the release.** GitHub releases can include release notes and links to binary files. ([GitHub Docs][16])

7. **Create draft releases when assets must be attached before publication.** GitHub recommends draft releases when immutable releases are enabled so all assets can be attached before publishing. ([GitHub Docs][16])

8. **Review generated release notes before publishing.** GitHub can auto-generate release notes, but its docs explicitly advise checking that generated notes include all and only the desired information. ([GitHub Docs][17])

9. **Security releases should include an advisory path.** GitHub’s release docs point users to security advisories when releases fix vulnerabilities. ([GitHub Docs][4])

## H. Changelog and release notes

### Required standards

1. **Maintain a human-readable changelog.** Keep a Changelog explicitly warns against dumping Git logs into changelogs and recommends documenting notable changes for humans. ([Keep a Changelog][18])

2. **Use standard categories.** Recommended:

   * Added
   * Changed
   * Deprecated
   * Removed
   * Fixed
   * Security
   * Breaking Changes
   * Migration Required
   * Known Issues

3. **Separate internal noise from user impact.** “Refactored controller” is rarely useful to customers unless it affects behavior, performance, or risk.

4. **Call out breaking changes loudly.**

5. **Include migration instructions.**

6. **Mention database migrations, config changes, API changes, permission changes, dependency/runtime changes, and deprecations.**

7. **Document known issues and workarounds.**

8. **For security releases, avoid exploit-enabling detail until users have a reasonable patch window.** Provide CVE/advisory links and mitigation guidance when appropriate.

9. **Credit contributors when appropriate.**

10. **Release notes are part of the release, not an afterthought.**

## I. Release automation

### Required release-pipeline rules

1. **Releases are created by CI/CD, not by a developer laptop.** Semantic-release, for example, automates version determination, changelog generation, and package publishing based on formal commit conventions after a successful CI build. ([GitHub][19])

2. **The release pipeline must be version-controlled.**

3. **The release pipeline must run from protected source.** Usually protected `main`, protected release branch, or protected release tag.

4. **Use least-privilege credentials.** GitHub Actions lets workflows or jobs set `GITHUB_TOKEN` permissions; grant only what the job needs. ([GitHub Docs][20])

5. **Prefer OIDC over long-lived cloud secrets.** GitHub Actions supports OpenID Connect so workflows can authenticate to cloud providers without storing long-lived secrets. ([GitHub Docs][21])

6. **Pin third-party GitHub Actions to full-length commit SHAs.** GitHub says pinning to a full-length commit SHA is the only way to use an action as an immutable release. Tags are convenient but depend on trusting the action maintainer not to move or compromise the tag. ([GitHub Docs][22])

7. **Release jobs must be separate from untrusted PR jobs.** Never expose publishing credentials to code from forks or untrusted branches.

8. **Release pipelines should produce:**

   * version
   * tag
   * changelog
   * release notes
   * build artifact
   * checksums
   * signatures where applicable
   * SBOM where applicable
   * provenance/attestation where applicable
   * deployment manifest
   * rollback reference

9. **Use environment protections for production.** Require approval for production deploys when business risk demands it.

10. **Keep manual steps explicit and auditable.** Manual approval can be fine; manual build/upload from a laptop is not.

## J. Supply-chain and security standards

### Required security practices

1. **No secrets in Git. Ever.** OWASP’s secrets guidance highlights hardcoded source-code secrets and plaintext config secrets as a core problem and recommends centralized storage, provisioning, auditing, rotation, and lifecycle management. ([OWASP Cheat Sheet Series][23])

2. **Enable secret scanning and push protection.** GitHub push protection can block supported secrets before they reach the repository through command line, web UI, uploads, and REST API. ([GitHub Docs][24])

3. **Rotate leaked credentials immediately.** GitHub’s secret scanning docs say that when a credential is leaked, the first step is to rotate it; removing it from history is time-intensive and often unnecessary once revoked. ([GitHub Docs][25])

4. **Protect CI/CD as production infrastructure.** OWASP warns that CI/CD integrations and plugins expand the attack surface and should be vetted, least-privileged, maintained, updated, and removed when unnecessary. ([OWASP Cheat Sheet Series][26])

5. **Adopt NIST SSDF practices.** NIST’s Secure Software Development Framework groups practices around preparing the organization, protecting software, producing well-secured software, and responding to vulnerabilities. ([NIST Computer Security Resource Center][27])

6. **Use SLSA-style provenance for important artifacts.** SLSA is a supply-chain security framework focused on preventing tampering and improving artifact integrity. ([SLSA][28])

7. **Generate artifact attestations where possible.** GitHub artifact attestations create signed provenance/integrity claims that can include workflow, repository, environment, commit SHA, triggering event, and optionally SBOM data. ([GitHub Docs][29])

8. **Generate SBOMs for distributed or high-risk software.** CycloneDX is an international BOM standard, including SBOM support, for software supply-chain transparency. ([CycloneDX][30])

9. **Scan dependencies and containers.** Use Dependabot/Renovate, SCA, container scanning, and license checks.

10. **Do not publish unsigned or unverifiable artifacts for critical software.**

11. **Review release permissions regularly.** Remove stale deploy keys, tokens, bot accounts, and admin access.

12. **Use separate credentials per environment.** Staging credentials should not deploy to production.

13. **Never let untrusted PR code access secrets.** This is one of the most serious CI/CD mistakes.

---

## K. Deployment and release rollout

### Required deployment practices

1. **Use progressive delivery where risk warrants it.** Canary, blue-green, rolling deploys, region-by-region, tenant-by-tenant, or percentage rollout.

2. **Canary before full rollout.** Google SRE describes canarying as exposing a change to a small portion of traffic to detect defects before full production rollout. ([Google SRE][31])

3. **Monitor during deployment.** Watch SLOs, error rate, latency, saturation, logs, traces, business metrics, queue depth, and customer-impact indicators.

4. **Rollback first when production is clearly unhealthy.** Google SRE guidance emphasizes rolling back first and diagnosing after unexpected production behavior to minimize recovery time. ([Google SRE][32])

5. **Have a documented rollback path before release.**

6. **Test rollbacks periodically.** A rollback plan that has never been exercised is a hope, not a plan.

7. **Use feature flags for risky or incomplete behavior.** But every flag needs an owner, purpose, creation date, and removal date.

8. **Use backward-compatible database migrations.** Prefer expand/contract:

   * add nullable/new columns first
   * deploy code that can read old and new
   * backfill
   * switch reads/writes
   * remove old schema later

9. **Avoid lockstep deploys across many services.** Design APIs/events to tolerate version skew.

10. **Use release windows only when necessary.** The better goal is safe releases during normal business hours with monitoring and rollback.

---

## L. Release governance and metrics

### Required governance practices

1. **Define release criteria.** A release is eligible only when tests, security checks, docs, migrations, observability, and rollback plan meet the standard.

2. **Define release ownership.** Every release has an accountable release owner.

3. **Define support policy.** Which versions are supported? How long? What gets security fixes?

4. **Define hotfix policy.** How emergency fixes are branched, reviewed, tested, released, and backported.

5. **Track DORA metrics.** DORA’s current software delivery metrics include change lead time, deployment frequency, failed deployment recovery time, change fail rate, and deployment rework rate. ([Dora][33])

6. **Use metrics for improvement, not punishment.**

7. **Review failed releases.** Blameless postmortems should produce process/code/pipeline improvements.

8. **Keep an audit trail.** Who approved, what commit, what artifact, what environment, what time, what checks passed, what rollback target exists.

9. **Document exceptions.** A break-glass release without written reason, owner, and follow-up is process debt.

# 3. Inexcusable mistakes and worst practices

These are the practices I would classify as **severe**, **inexcusable**, or **requiring immediate corrective action** in a professional engineering environment.

## A. Source-control mistakes

1. **Direct pushes to `main` or release branches.**
2. **No branch protection on important branches.**
3. **Admins bypassing branch protection as normal practice.**
4. **Force-pushing protected/shared branches.**
5. **Deleting or recreating release branches/tags without a formal incident process.**
6. **Rewriting public history after others have based work on it.**
7. **Rebasing shared branches casually.**
8. **Merging unreviewed code.**
9. **Allowing self-approval.**
10. **Ignoring required reviewers/CODEOWNERS.**
11. **Merging with failing checks.**
12. **Turning off CI “temporarily” to get a release out.**
13. **Keeping long-lived feature branches that diverge for weeks.**
14. **Using branches as permanent environment states: `dev`, `qa`, `prod`.**
15. **Leaving stale branches forever.**
16. **Committing generated build artifacts, dependency folders, logs, local configs, or IDE state.**
17. **Letting repository size grow uncontrollably with binaries.**
18. **No `.gitignore`, or `.gitignore` added after files are already tracked with no cleanup.**
19. **Meaningless commits on main: `fix`, `oops`, `stuff`, `final`, `real final`.**
20. **Huge PRs mixing formatting, refactoring, feature work, dependency bumps, and behavior changes.**

## B. Release-versioning mistakes

1. **Reusing a version number for different contents.**
2. **Changing a released artifact without changing the version.**
3. **Moving a published release tag.**
4. **Deleting and recreating release tags.**
5. **Publishing a patch release with breaking changes.**
6. **Publishing a minor release with breaking changes while claiming SemVer.**
7. **Marking a prerelease as the latest stable release by accident.**
8. **Producing artifacts that do not include version or commit SHA.**
9. **Tagging a commit that did not produce the artifact.**
10. **Manually editing release artifacts after CI produced them.**
11. **Publishing release notes that omit breaking changes.**
12. **Publishing release notes that are just raw commit logs.**
13. **No changelog for customer-facing or API-facing software.**
14. **No migration instructions for breaking changes.**
15. **No known-issues section when known issues exist.**

## C. CI/CD mistakes

1. **Releasing from a developer laptop.**
2. **Building locally and uploading binaries manually.**
3. **Deploying uncommitted code.**
4. **Deploying from a dirty working tree.**
5. **No reproducible build process.**
6. **No automated test gate before release.**
7. **No clean build environment.**
8. **Different artifact built for staging and production.**
9. **No required status checks.**
10. **CI configuration not version-controlled.**
11. **Flaky tests ignored as “normal.”**
12. **Manual release checklist existing only in someone’s head.**
13. **Release credentials stored in a developer’s local machine.**
14. **Using personal access tokens for production releases when a bot/OIDC/workload identity should be used.**
15. **Over-privileged CI tokens.**
16. **Unpinned third-party GitHub Actions in release workflows.**
17. **Running untrusted PR code with access to secrets.**
18. **Using CI cache in a way that lets low-privilege jobs poison privileged release jobs.**
19. **No separation between validation jobs and publishing jobs.**
20. **No audit trail for who approved production deployment.**

## D. Security mistakes

1. **Committing secrets.**
2. **Failing to rotate a leaked secret immediately.**
3. **Relying only on “we removed it from Git history” instead of revoking the credential.**
4. **No secret scanning.**
5. **No push protection.**
6. **Long-lived cloud keys stored as repository secrets when OIDC is available.**
7. **Release process depends on one human’s admin token.**
8. **No review for CI/CD workflow changes.**
9. **No CODEOWNER protection over deployment workflows.**
10. **No dependency scanning.**
11. **No vulnerability response process.**
12. **No SBOM/provenance/signature strategy for high-risk distributed software.**
13. **Publishing artifacts that consumers cannot verify.**
14. **Installing CI plugins/actions without review.**
15. **Leaving stale deploy keys, bot accounts, or admin users active.**

## E. Deployment mistakes

1. **No rollback plan.**
2. **Rollback plan never tested.**
3. **Irreversible database migration deployed together with risky application code.**
4. **Application version cannot run against previous database version.**
5. **No monitoring during rollout.**
6. **No health checks.**
7. **No owner watching the release.**
8. **No canary/progressive rollout for high-risk changes.**
9. **Diagnosing production failure for an hour instead of rolling back a clearly bad release.**
10. **No kill switch for risky user-facing features.**
11. **Permanent feature flags with no cleanup.**
12. **Deploying right before a known low-staff period without rollback/on-call coverage.**
13. **No incident communication path.**
14. **No postmortem after a failed release.**
15. **Hotfix made directly in production and never committed back to source control.**

## F. Governance mistakes

1. **No written release process.**
2. **No definition of done for release readiness.**
3. **No support policy for old versions.**
4. **No hotfix policy.**
5. **No backport policy.**
6. **No release owner.**
7. **No approval/audit trail for production changes.**
8. **No release calendar or release train when the business requires coordination.**
9. **Treating release management as bureaucracy instead of risk control.**
10. **Using DORA metrics to punish teams instead of improving systems.**

# 4. Minimum “gold standard” policy

A strong baseline policy would look like this:

## Git policy

* `main` is protected.
* No direct pushes to `main`.
* No force pushes or branch deletion on protected branches.
* Pull request required.
* At least one approval required; two for high-risk repos.
* CODEOWNER approval required for sensitive paths.
* Required CI checks must pass.
* Stale reviews are dismissed after meaningful changes.
* Conversations must be resolved.
* Admin bypass is disabled or logged as break-glass.
* Release tags are protected.
* Release tags are signed or otherwise verified.
* Public/shared history is not rewritten.

## Branch policy

* Prefer trunk-based development.
* Feature branches should be short-lived.
* Release branches exist only for active supported versions or release stabilization.
* Hotfix branches are temporary.
* Merged branches are deleted.
* Long-running changes use feature flags or branch-by-abstraction.

## Commit policy

* Commits are atomic.
* Commit messages are meaningful.
* Conventional Commits are used when automated release/versioning is desired.
* No secrets, generated outputs, local configs, or large binaries.
* No WIP/fixup commits on protected branches.

## CI policy

* CI runs on every PR and protected-branch push.
* Required checks include build and tests.
* Security checks run for relevant repos.
* CI config is version-controlled.
* Release workflows use least privilege.
* Third-party Actions are pinned to full commit SHA.
* Production credentials are unavailable to untrusted PR code.
* Main must stay green.

## Release policy

* Every release has a version.
* Every release has an immutable tag.
* Every release has release notes.
* Every release maps to an exact commit.
* Every artifact is produced by CI.
* Every artifact includes version and commit metadata.
* Release artifacts are not modified after publication.
* Breaking changes are explicitly documented.
* Security fixes follow advisory process.
* Rollback plan exists before production deployment.

## Deployment policy

* Deploy the same artifact that was tested.
* Use progressive rollout for meaningful risk.
* Monitor during and after deployment.
* Roll back first when the release is clearly unhealthy.
* Database migrations are backward-compatible.
* Feature flags have owners and expiration dates.
* Every production release has an accountable owner.

# 5. Practical release checklist

Use this before publishing any release:

## Before merge

* PR is small and focused.
* Required reviewers approved.
* CODEOWNERS approved sensitive areas.
* CI is green.
* Tests were added or updated.
* Security/privacy impact considered.
* Observability/logging/metrics updated where needed.
* Migration plan included.
* Rollback plan included.
* Breaking changes documented.

## Before tagging

* Main or release branch is green.
* Version number is correct.
* Changelog is updated.
* Release notes are drafted.
* Dependency/security checks passed.
* No secrets detected.
* Artifact build is reproducible through CI.
* Release commit is known.

## While releasing

* Tag is created from the correct commit.
* Artifact is built by CI.
* Checksums/signatures are generated where applicable.
* SBOM/provenance/attestation generated where applicable.
* GitHub release/package registry entry is created.
* Release notes are reviewed.
* Prerelease/stable/latest setting is correct.
* Deployment approval recorded if required.

## During deployment

* Rollout starts small.
* Dashboards are watched.
* Error budget/SLO impact is monitored.
* Logs/traces/alerts are active.
* Rollback command/path is ready.
* Release owner is present.

## After deployment

* Confirm production health.
* Confirm business metrics.
* Announce release.
* Close milestone.
* Backport hotfixes if needed.
* Remove temporary flags when safe.
* Record incidents or lessons learned.
* Update support/deprecation docs.

# 6. The simplest executive summary

The best teams treat Git and releases as a **controlled production system**:

* **Protected main.**
* **Short-lived branches.**
* **Small reviewed PRs.**
* **Automated tests.**
* **Automated releases.**
* **Immutable tags and artifacts.**
* **Human-readable changelogs.**
* **SemVer discipline.**
* **No secrets.**
* **Pinned CI dependencies.**
* **Least-privilege release credentials.**
* **Traceable artifacts.**
* **Progressive rollout.**
* **Fast rollback.**
* **Measured improvement.**

The worst teams do the opposite:

* mutable tags,
* direct pushes,
* force pushes,
* manual laptop releases,
* unreviewed changes,
* failing CI,
* secrets in Git,
* unclear versions,
* missing changelogs,
* no rollback,
* no ownership,
* and no audit trail.

That is the line between “we use Git” and “we have professional release engineering.”
