## Executive summary

For Git and releases, the gold standard is this:

**A release must be an immutable, auditable, reproducible snapshot of source code, build inputs, tests, artifacts, approvals, version number, changelog, deployment record, and rollback path.** Anything less eventually becomes “works on my machine” at organizational scale.

I weighted official and standards-grade sources highest: Git, GitHub, GitLab, SemVer, Conventional Commits, Keep a Changelog, Google SRE, DORA, OWASP, NIST SSDF, and SLSA. LinkedIn/GitHub community discussions were useful for repeated field mistakes such as retagging, tagging the wrong commit, and forgetting to push tags; Quora was not accessible through my browsing, and X results were too noisy to treat as authoritative.

# 1. The standards and conventions worth adopting

## Core release standards

**Semantic Versioning:** Use `MAJOR.MINOR.PATCH` when compatibility matters. `MAJOR` means incompatible API changes, `MINOR` means backward-compatible functionality, and `PATCH` means backward-compatible fixes. Pre-release labels like `1.2.0-rc.1` are valid and sort lower than the final release. SemVer also requires you to define a clear public API, otherwise version numbers become vibes instead of contracts. ([Semantic Versioning][1])

**Conventional Commits:** Use machine-readable commit messages such as `fix:`, `feat:`, and `feat!:` or `BREAKING CHANGE:`. This maps naturally to SemVer automation: `fix` → patch, `feat` → minor, breaking change → major. ([Conventional Commits][2])

**Keep a Changelog:** Changelogs are for humans, not machines. Every version should have an entry; changes should be grouped; latest version should appear first; versions and sections should be linkable; release dates should be shown; and the changelog should state whether SemVer is followed. ([Keep a Changelog][3])

**Git annotated or signed tags for releases:** Git’s own docs say annotated tags are intended for releases, while lightweight tags are more like temporary/private labels. Annotated tags include metadata such as tagger, date, message, and optionally a cryptographic signature. ([Git][4])

**Protected branches and protected tags:** Important branches should block force pushes and deletion, require PRs, require reviews, require status checks, and optionally require signed commits or linear history. GitHub rulesets can target both branches and tags; GitLab similarly supports protected branches/tags and deployment approvals. ([GitHub Docs][5])

**Immutable releases:** Published release tags and artifacts should not be mutable. GitHub immutable releases lock release assets and the associated tag after publication, preventing tag movement, tag deletion while the release exists, or asset replacement. ([GitHub Docs][6])

**Secure CI/CD:** OWASP treats CI/CD as a high-value attack surface and highlights risks such as inadequate IAM, dependency-chain abuse, poisoned pipeline execution, poor credential hygiene, improper artifact validation, and insufficient logging. NIST SSDF frames secure development as something that must be integrated into the SDLC, not bolted on later. ([OWASP Cheat Sheet Series][7])

**SLSA, SBOM, signing, provenance:** SLSA is a supply-chain security framework for preventing tampering and improving artifact integrity. SBOM standards such as CycloneDX and SPDX help communicate software component, license, security, and provenance information. ([SLSA][8])

**Release engineering discipline:** Google SRE emphasizes automation, frequent smaller releases, hermetic/repeatable builds, code review, gated release operations, release reports, release-branch testing, signed packages, build IDs, canaries, and rollout automation. ([Google SRE][9])

# 2. The ideal Git-to-release flow

A mature release flow looks like this:

1. **Work starts from a tracked issue, ticket, incident, RFC, or release objective.**
2. **Developer creates a short-lived branch from the current trunk/default branch.**
3. **Commits are small, coherent, and written using a team convention.**
4. **A pull request is opened early, reviewed by the correct owners, and kept small.**
5. **CI runs linting, tests, build, dependency checks, code scanning, secret scanning, and policy checks.**
6. **The PR cannot merge until required reviews and required checks pass.**
7. **The default branch remains releasable.**
8. **Release candidate is cut from a known green commit.**
9. **Release branch is used only when needed for stabilization, backports, or multi-version support.**
10. **Final release is tagged with an annotated/signed, protected tag.**
11. **Build happens once from the release commit/tag.**
12. **The built artifact is promoted through environments; it is not rebuilt per environment.**
13. **Release assets are signed, hashed, SBOM’d, and tied to a build ID and commit SHA.**
14. **Release notes and changelog are published.**
15. **Deployment is staged: dev/test → staging → canary → production.**
16. **Monitoring, health checks, and rollback criteria are explicit.**
17. **Post-release verification is recorded.**
18. **Hotfixes are cherry-picked deliberately, tested on the release branch, tagged as patch releases, and merged back to main.**

This matches the release-engineering pattern in Google SRE: automated builds/releases, small frequent changes, hermetic builds, release branches from exact revisions, cherry-pick approvals, tests on the actual release branch, package labels for canary/production, and logged release reports. ([Google SRE][9])

# 3. Best practices: exhaustive practical list

## A. Repository governance

**Protect `main`, `master`, `trunk`, `develop`, `release/*`, `hotfix/*`, and production GitOps/config branches.** Required protections should include no deletion, no force push, required PR, required CI checks, required reviews, and restricted bypass. GitHub and GitLab both provide controls for this. ([GitHub Docs][5])

**Use rulesets, not tribal memory.** The repo should enforce policy mechanically. A standard policy should prevent unsafe branch/tag creation, updates, deletion, large files, unsafe file paths, missing status checks, unsigned commits where required, and unauthorized release tags. ([GitHub Docs][10])

**Apply protections to admins too, unless you have a narrowly logged emergency process.** A bypass that is casually used is not a control; it is a suggestion.

**Use CODEOWNERS for critical paths.** Own `.github/workflows/`, `.gitlab-ci.yml`, infrastructure, auth, payment, database migrations, public API schemas, security-sensitive code, release scripts, package manifests, and deployment manifests. GitHub can automatically request code-owner review and optionally require it before merge. ([GitHub Docs][11])

**Restrict who can create release tags.** Tags matching `v*`, `release-*`, or production deployment patterns should be protected. GitHub rulesets can target tags; GitLab protected tags can use wildcards and control who creates protected tags. ([GitHub Docs][10])

**Never use Git as access control theater.** If a repo contains sensitive code or secrets, assume everyone with repo read access can inspect history. Split repositories or use real access controls where needed.

---

## B. Branching strategy

**Default to trunk-based development for modern CI/CD.** Developers merge small, frequent updates into a core trunk/default branch. Atlassian describes trunk-based development as a required practice for CI/CD, because long-lived branches delay integration and increase conflicts. ([Atlassian][12])

**Use short-lived feature branches.** A branch should live hours to a few days, not weeks. Long-lived feature branches are where integration problems hide.

**Use feature flags for incomplete work.** Feature toggles allow unfinished code paths to be integrated into trunk while keeping production behavior safe. This supports continuous delivery without hiding work in long-running branches. ([martinfowler.com][13])

**Use GitFlow only when it truly fits.** GitFlow can make sense for packaged products, mobile/desktop apps, SDKs, embedded systems, on-prem software, or products supporting multiple maintained versions. It is usually overkill for a single continuously deployed web service.

**Never let release branches become alternate realities.** Release branches should be stabilization/backport lanes, not places where new product development happens.

**Do not use environment branches as a substitute for deployment configuration.** `dev`, `qa`, `staging`, and `prod` branches often drift. Prefer one source branch plus environment-specific config, promotion metadata, or GitOps overlays.

**Branch names should be predictable.** Examples:
`feature/PROJ-123-short-name`, `fix/PROJ-456-null-check`, `hotfix/1.4.3-payment-timeout`, `release/2.7.0`.

---

## C. Commit discipline

**Make commits atomic.** One logical change per commit. Avoid “misc fixes,” “WIP,” “stuff,” and commits that combine refactor + behavior change + formatting + dependency upgrade.

**Use Conventional Commits where release automation matters.** `feat:`, `fix:`, `perf:`, `refactor:`, `docs:`, `test:`, `build:`, `ci:`, and `chore:` are common. Mark breaking changes explicitly. ([Conventional Commits][2])

**Explain the why, not only the what.** The diff already shows what changed. The commit body should explain context, tradeoffs, risk, and migration notes when relevant.

**Reference issues or tickets.** Every non-trivial change should be traceable to a bug, feature, incident, dependency update, security advisory, or operational need.

**Use signed commits where risk justifies it.** For regulated, enterprise, release, infrastructure, or security-sensitive repos, require signed/verified commits or at least signed release tags. GitHub rulesets can require verified signatures. ([GitHub Docs][14])

**Do not rewrite public history.** For public/shared branches, use `git revert` to create a new commit that reverses earlier changes rather than rewriting history. Git’s `revert` records new commits that reverse earlier commits. ([Git][15])

**Use `--force-with-lease`, not blind `--force`, on personal branches only.** Never force-push shared branches or release branches.

---

## D. Pull request standards

**Every production-bound change goes through PR/MR review.** Exceptions should be rare, logged, and post-reviewed.

**Keep PRs small.** Small PRs are easier to review, easier to test, easier to revert, and less likely to hide risky changes.

**Require the right reviewers, not just any reviewer.** Use CODEOWNERS, domain owners, security owners, database owners, and platform owners where appropriate.

**Dismiss stale approvals after material changes.** If the diff changed after review, it needs review again. GitHub branch/ruleset settings support stale-review dismissal and requiring approval from someone other than the last pusher. ([GitHub Docs][14])

**Resolve all review conversations before merge.** Unresolved review threads are unfinished work.

**For risky changes, require a rollout plan in the PR.** Include migrations, feature flags, monitoring, rollback, affected services, and customer impact.

**Protect CI/CD workflow changes.** Any change to `.github/workflows`, `.gitlab-ci.yml`, deployment scripts, IaC, release scripts, or package publishing config should require platform/security review. GitHub explicitly recommends CODEOWNERS for workflow files. ([GitHub Docs][16])

---

## E. Merge strategy

**Pick one strategy and document it.** Merge commits, squash merges, and rebase merges are all valid in different contexts. The mistake is inconsistency.

**For product repos, squash merge often works well.** It keeps main history cleaner when PRs are the review unit.

**For libraries or highly traceable engineering repos, merge commits can preserve PR branch context.**

**For linear-history teams, enforce linear history.** GitHub rulesets can require linear history, preventing merge commits in targeted branches/tags. ([GitHub Docs][14])

**Never merge red.** A failing required check should block merge. If CI is flaky, fix CI; do not normalize bypassing it.

---

## F. CI quality gates

**CI must run on every PR and on the release commit/tag.** Testing only a branch before it is rebased/merged is not enough.

**Required checks should include:** formatting, linting, type checks, unit tests, integration tests, build, package validation, dependency review, secret scanning, SAST, IaC validation, container scanning, migration tests, and artifact generation where applicable.

**Required checks must be unambiguous.** GitHub warns that duplicate job names across workflows can make required status checks ambiguous and block PRs. ([GitHub Docs][5])

**Run release-gating tests against the same commit that will be released.** Google SRE recommends release test targets correspond to continuous build test targets and recommends creating releases from the last continuous test build that successfully completed all tests. ([Google SRE][9])

**Treat flaky tests as production defects.** A flaky release gate trains people to bypass release gates.

**Do not let tests depend on local developer machines.** Release builds should be hermetic/repeatable. Google SRE’s release engineering guidance expects identical builds from the same revision, independent of machine-local libraries/tools. ([Google SRE][9])

---

## G. CI/CD security

**Set CI tokens to least privilege.** GitHub recommends setting `GITHUB_TOKEN` to read-only by default and increasing permissions only per job as needed. ([GitHub Docs][16])

**Prefer OIDC and short-lived cloud credentials over long-lived secrets.** GitHub recommends OIDC for cloud-provider authentication to avoid storing long-lived cloud credentials in GitHub. ([GitHub Docs][16])

**Never expose secrets to untrusted fork PRs.** Fork PRs and external contributions need strict isolation.

**Pin third-party GitHub Actions to full-length commit SHAs for high-security pipelines.** GitHub says pinning to a full-length commit SHA is the only way to use an action as an immutable release; tags can move or be deleted. ([GitHub Docs][16])

**Audit third-party actions and workflows.** A compromised action can access repository secrets and use the CI token. ([GitHub Docs][16])

**Use secret scanning and block committed secrets.** A committed secret must be considered exposed even if later deleted, because Git history can retain it.

**Separate build, test, publish, and deploy identities.** The identity that builds should not automatically have production deploy rights unless deliberately scoped.

**Protect package publishing credentials.** Publishing to npm, PyPI, Maven, NuGet, Docker registries, app stores, or internal artifact repositories must require protected branches/tags and controlled CI contexts.

---

## H. Versioning

**Choose one versioning system and document it.** SemVer is best for libraries, SDKs, APIs, packages, and anything consumed as a dependency. CalVer is best when the release date/support window matters more than API compatibility, such as some platforms, distros, or large applications. CalVer is explicitly calendar-based rather than arbitrary-number-based. ([Semantic Versioning][1])

**Do not mix SemVer and CalVer casually.** `2026.05.1` and `2.4.7` communicate different things. Pick one per product/package.

**Define what “breaking” means.** Include API signatures, behavior, configuration, CLI flags, database schema, events, SDK contracts, serialized formats, auth scopes, performance assumptions, and operational requirements.

**Use pre-release versions properly.** Examples: `2.0.0-alpha.1`, `2.0.0-beta.2`, `2.0.0-rc.1`. SemVer defines pre-release precedence lower than the final release. ([Semantic Versioning][1])

**Use build metadata for non-ordering metadata.** Examples: `1.4.2+build.1847` or `1.4.2+sha.abc123`. SemVer build metadata is ignored for precedence. ([Semantic Versioning][1])

**Use `v` prefix consistently for Git tags if you choose it.** `v1.2.3` is common for Git tags, but the SemVer version itself is `1.2.3`.

**For monorepos, choose fixed or independent versioning.** Fixed versioning means all packages share one version. Independent versioning means each package/service versions separately. Tools like Changesets and Lerna exist specifically to manage multi-package versioning and changelogs. ([GitHub][17])

---

## I. Tags and releases

**Use annotated or signed tags for releases.** Lightweight tags are not enough for serious releases. ([Git][18])

**Tag the exact commit that ships.** Not “roughly the branch head.” Not “the commit after a quick fix.” The tag must point to the reviewed, tested, released commit.

**Protect release tag patterns.** Examples: `v*`, `release-*`, `prod-*`.

**Never move a published release tag.** Community discussions and social posts repeatedly flag reusing/moving tags as a major source of broken builds and confusion; GitHub immutable releases now provide a platform-level way to prevent it. ([linkedin.com][19])

**Do not reuse tags.** If `v1.2.3` is wrong, publish `v1.2.4` or yank/deprecate `v1.2.3` according to ecosystem norms. Do not silently replace it.

**Push tags intentionally.** Creating a local tag is not enough. GitHub community answers repeatedly point out that tags must be pushed to the remote for team/release visibility. ([GitHub][20])

**Use GitHub/GitLab Releases, not only Git tags.** A proper release object should include release notes, assets, source snapshot, artifacts, and links to evidence. GitLab describes releases as combining code, binaries, documentation, release notes, and evidence into a complete snapshot. ([GitLab Docs][21])

**Create immutable GitHub releases as drafts first, attach all assets, then publish.** GitHub recommends this because immutable releases lock assets after publication. ([GitHub Docs][22])

---

## J. Changelog and release notes

**Maintain both internal and external release notes when needed.** Internal notes can include risk, deploy details, flags, metrics, and operational caveats. External notes should focus on user impact.

**Do not dump raw Git logs.** Keep a Changelog’s guiding principle is that changelogs are for humans, not machines. ([Keep a Changelog][3])

**Group entries consistently.** Common groups: Added, Changed, Deprecated, Removed, Fixed, Security, Breaking Changes, Migration Notes.

**Always include breaking changes and migration steps.** Hidden breaking changes are one of the fastest ways to destroy trust in versioning.

**Include release date, version, links, and compare range.** Example: `v2.4.0 — 2026-05-05 — compare v2.3.1...v2.4.0`.

**Document hotfixes.** A hotfix without release notes becomes institutional amnesia.

**Mention security fixes carefully.** Provide enough information for users to upgrade, but coordinate with security policy and embargo rules when appropriate.

---

## K. Build artifacts

**Build once, promote many.** The artifact tested in staging should be the artifact deployed to production. Do not rebuild separately for staging and production.

**Every artifact should know its source.** Embed commit SHA, version, build ID, build timestamp, and dirty-tree status where appropriate.

**Never release from a dirty working tree.** A release should be reconstructable from Git and build metadata.

**Sign release artifacts.** Packages, containers, binaries, installers, Helm charts, and archives should be signed where ecosystem tooling supports it.

**Publish checksums.** At minimum, publish SHA-256 checksums for downloadable artifacts.

**Generate SBOMs.** Use SPDX or CycloneDX, especially for enterprise, regulated, security-sensitive, or externally distributed software. SPDX is an open SBOM standard covering provenance, license, security, and related information; CycloneDX is an OWASP full-stack BOM standard for supply-chain risk reduction. ([SPDX][23])

**Generate provenance/attestations.** SLSA focuses on preventing tampering and improving artifact integrity across the software supply chain. ([SLSA][8])

**Never publish only `latest`.** Use immutable tags/digests like `myapp:1.8.3`, `myapp:1.8`, `myapp:sha-abc123`, and the image digest. `latest` can exist as a convenience pointer, not as the release identity.

---

## L. Deployment and rollout

**Separate release from deployment.** A release is a versioned artifact. A deployment is putting that artifact into an environment. A feature launch may be a third thing controlled by flags.

**Use environment promotion.** Dev/test → staging → canary → production.

**Use protected environments and approvals for production.** GitLab deployment approvals block deployments until required approvals are granted; GitHub environments can protect secrets until approval. ([GitLab Docs][24])

**Use canaries, blue/green, rolling deploys, or progressive delivery.** Do not make “all users, all at once” the default for risky changes.

**Define automatic and manual rollback criteria.** Example: error rate > X%, p95 latency > Y, checkout conversion drops, queue lag exceeds threshold, crash rate increases, canary health check fails.

**Test rollback before you need it.** An untested rollback plan is a hope document.

**Track DORA-style delivery metrics.** Current DORA guidance focuses on throughput and instability: change lead time, deployment frequency, failed-deployment recovery time, change fail rate, and deployment rework rate. ([Dora][25])

---

## M. Database and schema changes

**Use expand-and-contract migrations.** Add backward-compatible schema first, deploy code that supports both old and new, migrate data, then remove old schema later.

**Never ship destructive migrations without rollback/backup.** Dropping columns, rewriting large tables, or changing constraints must have a plan.

**Make migrations idempotent where possible.**

**Test migrations against production-like data size.** A migration that passes on a toy database can still lock production.

**Separate schema deployment from feature activation.** Use feature flags and compatibility windows.

**Document data backfill ownership.** Who runs it, monitors it, pauses it, and verifies it?

---

## N. API and contract releases

**Version public contracts deliberately.** APIs, SDKs, events, schemas, CLI flags, webhooks, and file formats are contracts.

**Do not break consumers in patch/minor releases under SemVer.** Breaking changes require major versions or a clearly documented compatibility exception. ([Semantic Versioning][1])

**Provide deprecation windows.** Mark deprecated behavior, notify users, provide migration guides, and define removal dates.

**Use contract tests.** Consumer-driven contract tests are excellent for services with many consumers.

**Version events and schemas.** Message queues and event streams need compatibility discipline just like REST/GraphQL/gRPC APIs.

---

## O. Hotfixes and backports

**Hotfix from the released version, not from random main.** Start from the release tag or release branch.

**Cherry-pick the smallest safe fix.** Do not opportunistically include unrelated refactors.

**Run full release tests on the hotfix branch.** Google SRE explicitly notes that cherry-picked release branches can contain a version of code that does not exist on main, so tests must pass in the actual release context. ([Google SRE][9])

**Tag hotfixes as patch releases.** Example: `v2.3.1`.

**Merge the fix back to main.** The “hotfix fixed prod but not main” mistake guarantees regression later.

**Document the incident link, root cause, customer impact, and release note.**

---

## P. Documentation and governance

**Create a release checklist.** It should include version, scope, approvals, CI status, artifact links, SBOM, signatures, migrations, rollback, comms, monitoring, and post-release verification.

**Name a release owner.** Every release needs a directly responsible person.

**Have a release calendar or release train if coordination matters.**

**Define release freeze rules.** What changes are allowed after branch cut? Who approves exceptions?

**Keep a support matrix.** Which versions are supported? Which receive security patches? What is EOL?

**Archive release evidence.** GitLab release evidence captures release-related snapshots such as test artifacts, linked milestones, and matching packages for audit processes. ([GitLab Docs][26])

# 4. The “inexcusable” mistakes list

These are the ones I would treat as unacceptable in a professional team.

| Inexcusable mistake                                      | Why it is severe                           | Correct practice                                      |
| -------------------------------------------------------- | ------------------------------------------ | ----------------------------------------------------- |
| Direct pushing to `main`/production branches             | Bypasses review, tests, traceability       | Require PR/MR, reviews, status checks                 |
| Force-pushing a shared branch                            | Can remove others’ work and corrupt PRs    | Block force pushes on protected branches              |
| Moving a published release tag                           | Destroys release integrity                 | Create a new version; protect/immutabilize tags       |
| Rebuilding the same version with different contents      | Makes debugging and compliance impossible  | Build once; new contents require new version          |
| Publishing mutable release assets                        | Users cannot know what they installed      | Immutable releases, checksums, signatures             |
| Tagging the wrong commit                                 | Release does not match tested source       | Tag the exact green release commit                    |
| Forgetting to push the release tag                       | CI/CD and teammates cannot see the release | Push the tag intentionally; automate release creation |
| Using lightweight tags for official releases             | Missing metadata/signature                 | Use annotated or signed tags                          |
| Reusing `latest` as the only artifact identity           | No rollback or auditability                | Use immutable version tags and digests                |
| Merging with failing CI                                  | Normalizes broken safety systems           | Fix CI or fix code; do not bypass                     |
| Ignoring flaky tests                                     | Makes release gates meaningless            | Quarantine/fix flaky tests immediately                |
| Disabling branch protection “temporarily” and forgetting | Turns controls into theater                | Emergency bypass must be logged and restored          |
| Committing secrets                                       | Secret is exposed through history          | Block secrets, rotate immediately if exposed          |
| Deleting a secret commit without rotating the secret     | Git history may still expose it            | Rotate first; purge history only as cleanup           |
| Using long-lived admin PATs in CI                        | High blast radius                          | Use least privilege, OIDC, short-lived tokens         |
| Allowing unreviewed CI workflow changes                  | Attackers can alter the pipeline           | CODEOWNERS + required security/platform review        |
| Running production deploys from laptops                  | Not reproducible or auditable              | Deploy from controlled CI/CD                          |
| Pulling unpinned third-party actions/scripts             | Supply-chain risk                          | Pin to SHA or trusted immutable source                |
| Giving fork PRs access to secrets                        | External code can exfiltrate credentials   | Isolate untrusted PRs                                 |
| Self-approving high-risk production deployments          | No separation of duties                    | Require independent approval                          |
| Shipping breaking changes as patch/minor SemVer releases | Violates user contract                     | Major bump or compatibility layer                     |
| No changelog for user-visible changes                    | Users cannot assess risk                   | Maintain human-readable changelog                     |
| Raw Git log as release notes                             | Not useful to customers/operators          | Curated release notes                                 |
| Hotfix not merged back to main                           | Fix disappears in future release           | Back-merge/cherry-pick to main                        |
| No rollback plan                                         | Incident response starts too late          | Define rollback before deploy                         |
| Untested rollback                                        | “Rollback” may fail when needed            | Practice rollback                                     |
| Destructive DB migration without backup/compatibility    | Can cause irreversible outage/data loss    | Expand-contract, backup, test                         |
| Deploying all users at once by default                   | Maximizes blast radius                     | Canary/progressive rollout                            |
| No post-deploy monitoring                                | Failures are found by customers            | Health checks, alerts, dashboards                     |
| Release from dirty local state                           | Cannot reproduce artifact                  | Release from clean tagged commit                      |
| Missing artifact checksum/signature                      | Integrity cannot be verified               | Publish checksums and signatures                      |
| No artifact-to-source mapping                            | Cannot answer “what code is running?”      | Embed SHA/version/build ID                            |
| Manual version edits with no validation                  | Easy to mismatch package/release/tag       | Automate versioning where possible                    |
| Inconsistent tag naming                                  | Automation and humans get confused         | Standardize `vMAJOR.MINOR.PATCH` or documented scheme |
| Tag/branch name collisions                               | Checkout ambiguity and operational risk    | Prevent tags with branch names                        |
| Publishing packages from unprotected branches            | Anyone with write access may publish       | Publish only from protected branch/tag                |
| Dependency upgrades without review                       | Can introduce vulnerabilities/breakage     | Dependency review, tests, SBOM                        |
| Ignoring license changes                                 | Legal/compliance risk                      | License scanning and review                           |
| No release owner                                         | Nobody owns go/no-go                       | Assign DRI/release manager                            |
| No incident/postmortem after bad release                 | Repeats same failure                       | Blameless postmortem and corrective actions           |

GitHub’s own ruleset documentation calls out why force pushes are dangerous: they can remove commits that collaborators based work on, corrupt PRs, or point branches/tags to commits not approved in a PR. Community posts echo the same practical mistakes around retagging, wrong branch head, and forgotten tag pushes. ([GitHub Docs][14])

# 5. Recommended baseline policy

For a serious team, I would set this as the minimum.

## GitHub/GitLab repository settings

Use:

* Protected default branch.
* Protected `release/*` and `hotfix/*`.
* Protected tag pattern `v*`.
* Required PR/MR before merge.
* Required status checks.
* Required code-owner review for critical paths.
* Stale approval dismissal.
* No branch deletion.
* No force push.
* Restricted bypass.
* Optional signed commits; stronger requirement for release, infra, regulated, or security repos.
* Immutable releases where available.
* Deployment approvals for production.
* CODEOWNERS for workflows, infra, release scripts, migrations, security-sensitive code, and package manifests.

GitHub rulesets support controls such as required PRs, required status checks, signed commits, linear history, deployment success before merge, blocked force pushes, and branch/tag restrictions. GitLab protected branches/tags and deployment approvals cover the same class of governance controls. ([GitHub Docs][14])

## CI/CD baseline

Use:

* Read-only default CI token.
* Per-job permissions.
* OIDC for cloud deploy credentials.
* No long-lived deploy secrets where OIDC is available.
* Secret scanning.
* Dependency review.
* SAST.
* IaC scanning.
* Container scanning.
* SBOM generation.
* Artifact signing.
* Provenance/attestations.
* Pinned third-party actions.
* CODEOWNERS review for CI config.
* Production deploy only from protected tags/branches.

GitHub’s security guidance specifically recommends least privilege for secrets/tokens, pinning third-party actions to full-length SHAs for immutability, CODEOWNERS for workflow changes, OIDC to avoid long-lived cloud secrets, and controls to prevent workflows from creating/approving PRs without oversight. ([GitHub Docs][16])

# 6. Release checklist

## Before cutting release

* Scope finalized.
* All release PRs merged.
* CI green on release candidate commit.
* Version bump correct.
* Changelog drafted.
* Breaking changes documented.
* Migration guide written if needed.
* Database migration tested.
* Security scans complete.
* Dependency/license review complete.
* Rollback plan ready.
* Monitoring dashboard ready.
* Release owner assigned.
* Stakeholders notified.
* Support/on-call aware.

## Cutting release

* Create release from clean, reviewed, green commit.
* Create annotated/signed tag.
* Push tag.
* Build once in CI.
* Generate artifacts.
* Generate SBOM.
* Sign artifacts.
* Publish checksums.
* Attach release notes/assets.
* Publish immutable release.

## Deploying release

* Deploy to staging.
* Run smoke tests.
* Run migration checks.
* Deploy canary.
* Monitor error rate, latency, logs, business metrics.
* Continue progressive rollout.
* Confirm health.
* Announce release.

## After release

* Verify version in production.
* Close release issue/ticket.
* Merge hotfix/release branch back to main if applicable.
* Delete merged branches.
* Keep tags and release artifacts.
* Record evidence.
* Update docs.
* Run postmortem for incidents or near misses.

# 7. The highest-value principles

**Small releases beat heroic releases.** Google SRE explicitly notes that frequent releases mean fewer changes between versions, making testing and troubleshooting easier. ([Google SRE][9])

**Traceability beats memory.** Every production artifact should answer: what source, what commit, what tag, what build, what dependencies, what tests, who approved, what changed, where deployed, and how to roll back.

**Immutability beats cleanup.** Once users can consume a version, do not mutate it. Publish a new version.

**Automation beats checklists, but checklists still matter.** Automate the gates; use checklists for human judgment, communications, and operational readiness.

**Security is part of release engineering.** CI/CD is a privileged path to production, so it deserves the same seriousness as production infrastructure. OWASP, NIST SSDF, SLSA, and GitHub/GitLab controls all point in the same direction: protect source, protect pipelines, protect artifacts, and preserve evidence. ([OWASP Cheat Sheet Series][7])
