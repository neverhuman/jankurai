Great question — this is exactly the kind of topic where teams either become **boringly reliable** or **dangerously lucky**.

I treated “best practices” as what a serious production team should enforce, and “inexcusable mistakes” as things that should trigger a postmortem, policy change, or access restriction. I weighted official docs and standards much higher than social posts. Community/social sources were useful as signal, not authority: LinkedIn and GitHub community discussions broadly echoed frequent meaningful commits, PRs, release tags, SemVer, release notes, and “do not reuse tags”; Quora was not reliably accessible from this browsing environment, so I did not rely on it as evidence. # Executive standard

A professional Git release should be:

**Traceable** — anyone can identify the exact commit, tag, pipeline run, artifacts, changelog, approvers, and deployment record.

**Immutable** — once released, the version, Git tag, artifact, checksum, SBOM, and provenance should not change. Publish a new version instead.

**Automated** — builds, tests, scans, packaging, signing, and publishing should happen through CI/CD, not a developer laptop.

**Protected** — main branches, release branches, and release tags should be protected with reviews, status checks, limited write access, and no force-pushes.

**Communicated** — users and operators should know what changed, what broke compatibility, how to migrate, and how to roll back.

**Auditable** — every important action should leave a durable record: who approved, what ran, what commit built, what artifact deployed, and what checks passed.

GitHub’s release model is based on Git tags marking points in repository history; Git’s own docs distinguish annotated tags as appropriate for releases; SemVer says versioned package contents must not be modified after release; GitHub’s immutable releases prevent tag and asset modification after publishing. ([GitHub Docs][2])

# The standards and conventions to adopt

## 1. Semantic Versioning

Use **Semantic Versioning 2.0.0** for public APIs, libraries, SDKs, CLIs, services with external consumers, and products where compatibility matters.

Use:

```text
MAJOR.MINOR.PATCH
```

Meaning:

```text
MAJOR = incompatible API or contract change
MINOR = backward-compatible functionality
PATCH = backward-compatible bug fix
```

Pre-release versions such as `1.4.0-rc.1`, `2.0.0-beta.3`, and build metadata such as `1.4.0+build.52` are valid SemVer concepts. SemVer also explicitly says released package contents must not be modified; any change requires a new version. ([Semantic Versioning][3])

Common Git tag format:

```text
v1.2.3
```

The `v` prefix is common for Git tags, but the semantic version itself is `1.2.3`, not `v1.2.3`. ([Semantic Versioning][3])

## 2. Annotated or signed Git tags for releases

Use annotated tags for releases, preferably signed tags for important software. Git describes annotated tags as full objects with metadata and an optional cryptographic signature, while lightweight tags are simple pointers and are better suited for private or temporary labels. ([Git][4])

Good:

```bash
git tag -s v1.8.0 -m "Release v1.8.0"
git tag -v v1.8.0
git push origin v1.8.0
```

Acceptable when signing is not yet available:

```bash
git tag -a v1.8.0 -m "Release v1.8.0"
git push origin v1.8.0
```

Avoid for official releases:

```bash
git tag v1.8.0
```

That creates a lightweight tag.

## 3. Keep a Changelog

Use a human-readable changelog. “Keep a Changelog” recommends organizing notable changes by version, date, and categories such as **Added**, **Changed**, **Deprecated**, **Removed**, **Fixed**, and **Security**. It also explicitly warns against dumping raw Git logs into changelogs. ([Keep a Changelog][5])

Minimum release notes should include:

```text
Version
Date
Summary
Added
Changed
Fixed
Security
Breaking changes
Migration steps
Known issues
Contributors
Artifact checksums or links
```

## 4. Conventional Commits

Use **Conventional Commits** when you want automated release notes, automated version bumps, or clearer commit history. The convention maps `fix` to patch-level changes, `feat` to minor-level changes, and `BREAKING CHANGE` or `!` to major-level changes. ([Conventional Commits][6])

Examples:

```text
fix(auth): reject expired refresh tokens
feat(api): add customer export endpoint
feat!: remove legacy v1 invoice endpoint
docs(release): document v2 migration path
```

## 5. Protected branches and protected tags

Protect `main`, `master`, `trunk`, release branches, and release tags.

At minimum, require:

```text
Pull request before merge
Required reviews
Required status checks
No force pushes
No deletion
Code owner approval for sensitive areas
Signed commits or signed tags where appropriate
Restricted who can create release tags
Restricted who can bypass rules
```

GitHub branch/ruleset controls can require pull requests, status checks, signed commits, linear history, successful deployments, and code scanning; they can also block force pushes and branch/tag deletion. GitLab protected tags similarly control who can create tags and prevent accidental tag updates or deletion. ([GitHub Docs][7])

## 6. Trunk-based development as the default

Prefer **trunk-based development** unless you have a strong reason not to. That means small, frequent integrations into `main` or `trunk`, with incomplete work hidden behind feature flags or branch-by-abstraction.

DORA and Google Cloud’s DevOps research associate trunk-based practices, frequent merging, CI, and deployment automation with higher software delivery performance. Martin Fowler’s CI guidance also emphasizes frequent integration, automated builds/tests, fixing broken builds immediately, and making deployment automation routine. ([Dora][8])

GitFlow-style branching can still be valid for release-based products, mobile apps, embedded software, on-premise software, or products that must maintain several supported versions. But release branches should be created close to release time, kept short-lived when possible, and hotfixes must be merged back into all affected active lines. ([Atlassian][9])

## 7. Supply-chain security baseline

For serious software, especially public packages, containers, CLIs, enterprise software, or regulated systems, adopt:

```text
Signed tags
Signed artifacts
Checksums
SBOMs
Artifact attestations
Dependency review
Secret scanning
Code scanning
Pinned CI actions
Least-privilege CI tokens
OIDC instead of long-lived cloud keys
Immutable releases
Protected environments
```

GitHub’s Actions security guidance recommends least-privilege workflow permissions, OIDC short-lived credentials, repository rules, Dependabot, pinning actions to SHAs, avoiding mutable dependencies, and being careful with untrusted pull requests and self-hosted runners. SLSA provides a supply-chain integrity framework, NIST SSDF defines secure software development practices, and SBOM guidance from U.S. security agencies emphasizes dependency transparency and supply-chain risk management. ([Well-Architected][10])

13. Practice incident response.
14. Review permissions periodically.
15. Review dependencies periodically.
16. Review CI security periodically.
17. Review artifact retention periodically.
18. Review release metrics periodically.
19. Archive unsupported versions clearly.
20. Celebrate uneventful releases.

# Inexcusable mistakes and worst practices

Treat these as serious process failures in any professional production environment.

## A. Identity and traceability failures

1. Releasing without a Git tag.
2. Releasing without knowing the exact commit SHA.
3. Releasing without knowing what artifact was deployed.
4. Releasing from uncommitted local changes.
5. Releasing from a developer laptop.
6. Releasing from someone’s personal branch without review.
7. Releasing from a branch that was not tested.
8. Releasing an artifact that was not built by CI/CD.
9. Releasing a different artifact than the one tested.
10. Having production run code that does not exist in Git.
11. Manually patching production and not committing the change.
12. Not being able to answer: “What version is in production?”
13. Not being able to answer: “What commit is in production?”
14. Not being able to answer: “Who approved this release?”
15. Not being able to answer: “What changed since the last release?”

## B. Tagging failures

1. Moving a public release tag.
2. Deleting a public release tag.
3. Reusing a release tag.
4. Replacing an artifact under the same tag.
5. Creating a release tag before tests pass.
6. Creating a release tag from the wrong branch.
7. Creating a release tag from a dirty working tree.
8. Creating a stable tag for a beta or RC.
9. Using lightweight tags for important official releases when annotated or signed tags are expected.
10. Not protecting release tags.
11. Allowing everyone to create production release tags.
12. Allowing everyone to delete production release tags.
13. Using ambiguous tags such as `latest`, `stable`, or `final` as the only release identity.
14. Accidentally pushing unrelated local tags with `git push --tags`.
15. Having a branch and tag with the same name for release-critical workflows.

These are especially serious because both GitHub and GitLab provide mechanisms to protect tags, and GitHub immutable releases can prevent post-publication tag and asset mutation. ([GitHub Docs][7])

## C. Branch protection failures

1. Force-pushing `main`.
2. Force-pushing a release branch.
3. Force-pushing a protected branch.
4. Disabling branch protection to “just get something in.”
5. Directly committing to `main` when policy requires PRs.
6. Directly committing to release branches when policy requires PRs.
7. Letting admins bypass protections casually.
8. Merging without required reviews.
9. Merging with failing checks.
10. Merging after manually disabling checks.
11. Merging unreviewed CI/CD workflow changes.
12. Merging unreviewed deployment script changes.
13. Merging unreviewed dependency changes.
14. Merging stale branches without rerunning tests.
15. Leaving branch protections weaker than the release risk.

GitHub explicitly notes that force pushes can remove commits others depend on, corrupt pull requests, and point branches at unapproved commits. ([GitHub Docs][14])

## D. CI/CD failures

1. Skipping tests to hit a release date.
2. Treating flaky tests as ignorable forever.
3. Running only manual tests for production releases.
4. Building in staging and rebuilding separately for production.
5. Deploying an artifact that never passed CI.
6. Allowing release jobs to pass despite failed required checks.
7. Marking critical checks as optional.
8. Letting CI scripts mutate release contents after tagging.
9. Running release jobs with excessive permissions.
10. Giving every job write access to the repository.
11. Giving every job access to production secrets.
12. Using long-lived cloud keys when OIDC is available.
13. Using mutable CI dependencies.
14. Using third-party actions pinned only to branches or tags for critical release workflows.
15. Running untrusted pull request code with secrets.
16. Using self-hosted runners for untrusted public PRs without isolation.
17. Not reviewing workflow changes.
18. Not logging release pipeline runs.
19. Not preserving release artifacts.
20. Not preserving release test results.

GitHub states that third-party actions can compromise jobs, secrets, and tokens, and that a full commit SHA is the only immutable way to pin an action. ([GitHub Docs][15])

## E. Versioning failures

1. Publishing two different builds as the same version.
2. Modifying a released package without changing the version.
3. Publishing breaking changes as a patch version.
4. Publishing breaking changes as a minor version.
5. Hiding breaking changes in vague release notes.
6. Changing public APIs without migration guidance.
7. Removing deprecated features without warning when warning was feasible.
8. Marking a pre-release as stable accidentally.
9. Publishing package manifests that disagree with the Git tag.
10. Publishing docs for one version and artifacts for another.
11. Releasing `v1.2.3` while package metadata says `1.2.2`.
12. Releasing multiple packages with incompatible internal dependency versions.
13. Using date/version schemes nobody understands.
14. Failing to define what compatibility means.
15. Failing to define supported versions.

SemVer is very direct here: once a versioned package has been released, its contents must not be modified; changes must be released as a new version. ([Semantic Versioning][3])

## F. Artifact and supply-chain failures

1. Publishing public binaries without checksums.
2. Publishing important artifacts without signatures.
3. Publishing containers without recording digests.
4. Publishing packages without provenance when your risk level requires it.
5. Publishing without an SBOM when customers, regulators, or internal policy require one.
6. Publishing artifacts from unknown build inputs.
7. Publishing artifacts built from mutable dependencies.
8. Publishing artifacts with unknown dependency vulnerabilities.
9. Ignoring critical dependency alerts.
10. Ignoring secret scanning alerts.
11. Ignoring code scanning alerts for release-critical issues.
12. Deleting artifacts needed for rollback.
13. Overwriting artifacts in place.
14. Rebuilding old versions without clear provenance.
15. Depending on `latest` as a production release identity.
16. Failing to retain old images or packages.
17. Failing to verify artifact integrity before deploy.
18. Allowing manual artifact uploads without review.
19. Allowing release assets to be edited after publication.
20. Not being able to prove where an artifact came from.

SLSA, GitHub artifact attestations, SBOM guidance, and NIST SSDF all point toward the same direction: release integrity depends on provenance, protected build systems, dependency transparency, and tamper resistance. ([SLSA][16])

## G. Changelog and communication failures

1. Releasing with no release notes.
2. Releasing with no changelog.
3. Dumping raw commit logs as the changelog.
4. Omitting breaking changes.
5. Omitting migration steps.
6. Omitting security fixes.
7. Omitting known issues.
8. Saying only “bug fixes and improvements.”
9. Publishing release notes that do not match the released commit.
10. Publishing release notes before the release content is final.
11. Announcing a release before artifacts are available.
12. Announcing security details before users can patch.
13. Not telling support or operations what changed.
14. Not documenting config changes.
15. Not documenting database migrations.

## H. Hotfix failures

1. Hotfixing production manually.
2. Hotfixing production without committing to Git.
3. Hotfixing one release branch and forgetting trunk.
4. Hotfixing one supported version and forgetting other affected versions.
5. Shipping a hotfix without tests.
6. Shipping a hotfix without a tag.
7. Shipping a hotfix without release notes.
8. Shipping a hotfix with unrelated changes.
9. Treating emergency hotfix process as the normal process.
10. Not doing a postmortem after a serious emergency release.

## I. Deployment failures

1. Deploying without a rollback or roll-forward plan.
2. Deploying without knowing the previous good version.
3. Deploying without retaining the previous artifact.
4. Deploying without health checks.
5. Deploying without monitoring.
6. Deploying irreversible migrations without backups.
7. Deploying destructive schema changes casually.
8. Deploying during risky periods without business approval.
9. Deploying with manual undocumented steps.
10. Deploying different steps depending on who is on call.
11. Deploying directly from a local machine.
12. SSH-ing into production and editing files.
13. Skipping staging for high-risk changes.
14. Ignoring failed canaries.
15. Ignoring post-deployment errors.

## J. Security-release failures

1. Silently fixing severe vulnerabilities without advisory when users need to know.
2. Publishing exploit details before fixes are available.
3. Not identifying affected versions.
4. Not identifying fixed versions.
5. Not giving mitigation guidance.
6. Not patching supported old versions.
7. Not revoking leaked credentials.
8. Not rotating compromised secrets.
9. Not invalidating compromised artifacts.
10. Leaving vulnerable artifacts marked as latest or recommended.
11. Hiding security fixes under “miscellaneous bug fixes.”
12. Not coordinating with customers or downstream maintainers when needed.

# Minimum release policy I would enforce

For a serious production repository, I would make this the minimum.

## Protected branches

```text
main/trunk:
  - PR required
  - Required status checks
  - Required review
  - CODEOWNER review for sensitive paths
  - No force push
  - No deletion
  - No direct commits
  - Admin bypass restricted and audited
```

## Protected tags

```text
v*:
  - Only release automation or release maintainers can create
  - No updates
  - No deletion
  - Annotated tags required
  - Signed tags preferred
```

## Required release inputs

```text
Version
Git tag
Commit SHA
Clean CI run
Changelog entry
Release notes
Artifact list
Checksums
Signatures where applicable
SBOM where applicable
Provenance/attestation where applicable
Rollback or roll-forward plan
Approver
```

## Required CI gates

```text
Build
Unit tests
Integration tests
Lint
Type check
Security scan
Dependency review
Secret scan
License check where needed
Package/install test
Migration test where needed
Artifact signing/checksum generation
```

## Required release process

```text
1. Merge approved PRs into protected main/trunk.
2. Confirm CI is green.
3. Update version, changelog, and release notes.
4. Create signed or annotated release tag.
5. CI builds from the tag.
6. CI runs release validation.
7. CI creates immutable artifacts.
8. CI generates checksums, SBOM, and provenance where required.
9. CI publishes release draft with all assets.
10. Release owner reviews final release.
11. Publish release.
12. Deploy immutable artifact through protected environment.
13. Monitor health.
14. Record release status.
```

GitHub’s immutable release workflow specifically recommends creating a draft, attaching all assets, and then publishing, because after publication the associated tag and assets cannot be changed. ([GitHub Docs][13])

# The shortest possible rule

A release is acceptable only when you can answer all of these:

```text
What version is it?
What Git tag is it?
What commit is it?
Who approved it?
What tests passed?
What changed?
What artifact was built?
Where is the checksum/signature/provenance?
How do we roll back or roll forward?
Is the release immutable?
```

When any of those answers is missing, the release process is not mature yet.
