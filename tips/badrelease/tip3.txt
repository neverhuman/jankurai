# Git and release management: best practices, standards, and inexcusable mistakes

I treated social/forum content as **weak signal** and formal specs/docs as **strong signal**. The strongest references are SemVer, Conventional Commits, Keep a Changelog, Git/GitHub docs, DORA, Martin Fowler’s branching/release patterns, OpenSSF/SLSA/Scorecard, NIST SSDF, SPDX/CycloneDX, and release automation projects such as `semantic-release`, Release Please, and Changesets. Quora access was blocked, and public X/LinkedIn results mostly repeated the same themes: force-push risk, tag hygiene, “latest” tag risk, and the need for clean release notes.

The executive standard is simple:

> **A release must be immutable, traceable, tested, documented, reproducible or at least verifiable, securely built, and reversible.**

Everything below supports that.

---

## 1. Non-negotiable release principles

A good release process answers these questions instantly:

| Question                   | Required answer                                                              |
| -------------------------- | ---------------------------------------------------------------------------- |
| What code was released?    | Exact commit SHA and immutable tag.                                          |
| What changed?              | Human-readable changelog/release notes.                                      |
| Why was it changed?        | Linked issues, PRs, tickets, or design notes.                                |
| Who approved it?           | PR reviews, CODEOWNERS, release approver, audit trail.                       |
| Was it tested?             | CI result, required checks, staging/canary status.                           |
| What artifact was shipped? | Artifact hash, package digest, container digest, checksum.                   |
| Where was it built?        | CI run, workflow, build environment, provenance/attestation.                 |
| How do we roll back?       | Previous known-good version and documented rollback path.                    |
| How do users migrate?      | Breaking-change notes, deprecation notes, upgrade steps.                     |
| Is it secure?              | Dependency scan, secrets scan, SBOM, signed artifacts/tags where applicable. |

Git tags are meant to mark important points in history, commonly release points, and GitHub releases are built on Git tags; GitHub defines releases as deployable software iterations packaged for users. ([Git][1])

## 2. Standards to adopt

### A. Versioning standard: Semantic Versioning

Use **SemVer** for libraries, APIs, packages, SDKs, CLIs, shared services, and any software consumed by other systems.

Use:

```text
MAJOR.MINOR.PATCH
```

Where:

```text
MAJOR = incompatible / breaking change
MINOR = backward-compatible feature
PATCH = backward-compatible bug fix
```

SemVer also states that once a versioned package has been released, the contents of that version must not be modified; any modification requires a new version. That one rule alone eliminates many release disasters. ([Semantic Versioning][2])

Best practices:

1. Define the **public API** explicitly. For a library, it is exported functions/classes. For a service, it is endpoints, events, schemas, behavior, SLAs, CLI flags, database contracts, file formats, etc.
2. Treat breaking behavior as breaking, not just breaking types.
3. Do not hide breaking changes in patch/minor releases.
4. Use pre-releases for unstable work:

```text
1.4.0-alpha.1
1.4.0-beta.1
1.4.0-rc.1
```

5. Use build metadata when needed:

```text
1.4.0+build.20260505.shaabc123
```

6. For internal apps without public consumers, you may use calendar versioning, build numbers, or deployment IDs, but still tag releases immutably.

---

### B. Commit standard: Conventional Commits

Adopt **Conventional Commits** or another machine-readable commit/PR-title convention. The key mappings are:

```text
fix:     patch
feat:    minor
feat!:   major
BREAKING CHANGE: major
```

The Conventional Commits spec maps `fix` to PATCH, `feat` to MINOR, and `BREAKING CHANGE` or `!` to MAJOR in SemVer. ([Conventional Commits][3])

Recommended commit types:

```text
feat:       user-visible feature
fix:        bug fix
docs:       documentation only
test:       tests only
refactor:   behavior-preserving code restructuring
perf:       performance improvement
build:      build system or dependency changes
ci:         CI/CD configuration
chore:      maintenance
revert:     revert a previous change
security:   security fix, if your tooling supports it
```

Best practices:

1. Make commits atomic: one logical change per commit.
2. Write commit messages in the imperative mood:

```text
fix(api): reject expired tokens
```

3. Add body text when context matters.
4. Use footers for issue links:

```text
Fixes: #1234
Refs: PROJ-456
```

5. Use `BREAKING CHANGE:` with migration guidance, not just “breaking”.
6. Enforce commit or PR-title format in CI.
7. Do not allow meaningless release commits like:

```text
update
fix stuff
changes
misc
wip
final
final-final
```

---

### C. Changelog standard: Keep a Changelog

Use **Keep a Changelog** style for human-readable release history. Its principles include: changelogs are for humans, every version should have an entry, changes should be grouped, latest version comes first, release dates should be displayed, and SemVer usage should be mentioned. ([Keep a Changelog][4])

Recommended sections:

```text
Added
Changed
Deprecated
Removed
Fixed
Security
```

Best practices:

1. Keep an `[Unreleased]` section.
2. Write from the user/operator perspective.
3. Group changes by impact, not by commit order.
4. Include breaking changes prominently.
5. Include migration steps.
6. Include security notes and CVE references when public.
7. Link versions to GitHub/GitLab compare views.
8. Do not dump raw `git log` as a changelog.
9. Do not make release notes only readable by developers.
10. Do not omit operational changes: config, infra, migrations, permissions, env vars.

GitHub can generate release notes from merged PRs, contributors, and full changelog links; it also lets teams categorize release notes with labels. Generated notes are useful, but they still need human review. ([GitHub Docs][5])

## 3. Branching and workflow best practices

There is no universal Git workflow; teams should choose a workflow that fits their culture, scale, release schedule, and ability to undo mistakes. Atlassian explicitly frames Git workflows as guidelines rather than rigid rules. ([Atlassian][6])

### Preferred for modern CI/CD: trunk-based development

Use **trunk-based development** when you want frequent, low-risk releases. DORA emphasizes frequent integration of small batches to avoid long stabilization phases and big merges. ([Dora][7])

Best practices:

1. Keep `main` always releasable.
2. Use short-lived branches.
3. Merge small PRs frequently.
4. Require CI to be green before merge.
5. Hide incomplete work behind feature flags.
6. Separate **deploy** from **release**.
7. Revert quickly when `main` breaks.
8. Avoid long-lived feature branches.
9. Avoid “integration hell” before release.
10. Prefer many small releases over rare massive releases.

Feature toggles are a common way to separate feature release from code deployment. ([martinfowler.com][8])

For large rewrites, use **branch by abstraction** rather than holding a giant branch open for weeks or months. Martin Fowler describes branch by abstraction as a way to make large-scale changes gradually while continuing to release regularly. ([martinfowler.com][9])

---

### Acceptable for scheduled/versioned products: Gitflow or release branches

Gitflow can work for products with scheduled versioned releases, on-prem deployments, mobile apps, embedded software, or support windows. Gitflow uses `main`, `develop`, feature branches, release branches, and hotfix branches. ([Atlassian][10])

But use it carefully:

1. Keep release branches short-lived.
2. Create release branches as close to the release as possible.
3. Back-merge or forward-port every hotfix.
4. Do not allow release branches to become parallel development universes.
5. Automate cherry-pick/backport tracking.
6. Delete release branches when no longer supported.
7. Use maintenance branches only for supported versions.
8. Document which branches receive security fixes.
9. Do not run multiple undocumented release trains.
10. Do not use Gitflow by default just because it is familiar.

Release branches create duplicate work and make it easy to forget merging changes to both the release branch and mainline; if used, create them close to release time. ([Atlassian][11])

## 4. Branch protection and code review standards

Use branch protection or repository rulesets for `main`, release branches, deployment branches, and protected tags. GitHub rulesets can restrict creation, updates, deletions, require signed commits, require PRs, require status checks, block force pushes, require code scanning, and more. ([GitHub Docs][12])

Minimum rules for `main`:

1. No direct pushes.
2. Pull request required.
3. At least one approval required.
4. CODEOWNERS approval required for critical paths.
5. Required status checks must pass.
6. Required branch up to date or merge queue enabled.
7. Dismiss stale approvals after changes.
8. Require approval from someone other than the last pusher.
9. Require conversation resolution.
10. Block force pushes.
11. Block deletions.
12. Require signed commits where feasible.
13. Require deployment to staging before production merge if your workflow supports it.
14. Restrict who can bypass rules.
15. Audit bypasses.

GitHub CODEOWNERS can require designated owners to review changes before merging; CODEOWNERS is especially important for CI/CD workflows, package manifests, infra, deployment scripts, security-sensitive modules, and release automation. ([GitHub Docs][13])

Critical paths that should have CODEOWNERS:

```text
.github/workflows/
.github/actions/
Dockerfile
docker-compose.yml
kubernetes/
helm/
terraform/
pulumi/
package.json
package-lock.json
pnpm-lock.yaml
yarn.lock
requirements.txt
poetry.lock
pyproject.toml
go.mod
go.sum
Cargo.toml
Cargo.lock
pom.xml
build.gradle
gradle.lockfile
release scripts
deployment scripts
security policy
CODEOWNERS itself
```

## 5. Pull request best practices

1. Keep PRs small and reviewable.
2. One purpose per PR.
3. Include problem, solution, risk, test evidence, rollback notes.
4. Link issue/ticket/design doc.
5. Require screenshots or recordings for UI changes.
6. Require migration notes for database/schema/API changes.
7. Label PRs for release note generation.
8. Label breaking changes explicitly.
9. Require tests with behavior changes.
10. Require docs with user-facing changes.
11. Require security review for auth, crypto, permissions, dependencies, deserialization, secrets, workflows, infrastructure.
12. Do not merge stale PRs without rebasing/updating.
13. Do not let bots auto-merge major dependency upgrades without tests.
14. Do not merge generated code without verifying source.
15. Do not approve your own critical PR.
16. Use a merge queue for busy repos.
17. Squash merge if your release tooling depends on clean PR-level commit messages.
18. Preserve merge commits if you rely on branch topology.
19. Pick one merge strategy and document it.
20. Do not mix unrelated fixes, refactors, and features in one PR.

Release Please specifically recommends squash merges for a linear history because it makes changelog control easier when commit messages inside a PR are not individually meaningful after merge. ([GitHub][14])

## 6. CI/CD standards

Minimum CI gates:

1. Build.
2. Unit tests.
3. Integration tests where applicable.
4. Lint/static analysis.
5. Type checks.
6. Formatting check.
7. Dependency vulnerability scan.
8. Secret scan.
9. License check where required.
10. SBOM generation for production artifacts.
11. Artifact signing or attestation for release artifacts.
12. Container image scan if containers are produced.
13. Infrastructure validation for IaC.
14. Migration validation for database changes.
15. Smoke tests for deployed environments.

Rules:

1. `main` must stay green.
2. Failed CI blocks merge.
3. Flaky tests are treated as production risk, not ignored.
4. CI config changes require code owner review.
5. Release jobs run only from protected branches/tags.
6. Release credentials are unavailable to untrusted PRs.
7. Release jobs use least privilege.
8. Release jobs produce immutable artifacts.
9. Release jobs publish hashes/checksums.
10. Release jobs attach provenance or attestations.

GitHub recommends minimum required `GITHUB_TOKEN` permissions and warns that actions can access the token through context even when not explicitly passed, so token permissions should be limited. ([GitHub Docs][15])

## 7. Version bumping and release automation

Manual versioning is error-prone. Automate it when possible.

Good options:

| Tool                  | Best for                                                   |
| --------------------- | ---------------------------------------------------------- |
| `semantic-release`    | Fully automated releases from commits.                     |
| Release Please        | Release PRs, changelog, version bump, GitHub release.      |
| Changesets            | Monorepos and multi-package repositories.                  |
| language-native tools | Maven Release Plugin, npm, Poetry, Cargo, GoReleaser, etc. |

`semantic-release` automates version determination, release notes, package publishing, Git tagging, and publishing after successful CI. ([GitHub][16])

Release Please automates changelog generation, GitHub releases, version bumps, and release PRs by parsing Conventional Commit messages. ([GitHub][14])

Changesets lets contributors declare release intent and then automates version updates, changelogs, and publishing, especially for multi-package repositories. ([GitHub][17])

Best practices:

1. Release only from CI, not laptops.
2. CI determines or verifies the version.
3. Version in code, tag, package, changelog, and artifact metadata must match.
4. Release automation must fail if uncommitted changes remain.
5. Release automation must fail if the tag already exists.
6. Release automation must fail if artifacts are missing.
7. Release automation must fail if required checks are skipped.
8. Release automation must create release notes.
9. Release automation must publish artifacts from the same commit being tagged.
10. Release automation must not use broad personal access tokens when GitHub App tokens/OIDC/scoped tokens work.

## 8. Tagging standards

Use release tags consistently.

Recommended tag format:

```text
v1.2.3
v1.2.3-rc.1
v1.2.3-beta.1
```

Best practices:

1. Use annotated tags for releases.
2. Sign release tags where feasible.
3. Tag the exact commit that produced the artifact.
4. Protect release tags.
5. Do not move release tags.
6. Do not delete release tags.
7. Do not reuse version numbers.
8. Do not tag from dirty local worktrees.
9. Do not tag commits that have not passed release CI.
10. Use a consistent prefix, usually `v`.
11. Use SemVer sorting.
12. Push only the intended tag, not every local tag:

```bash
git push origin v1.2.3
```

Prefer:

```bash
git push --follow-tags
```

Avoid careless:

```bash
git push --tags
```

13. Keep maintenance tags clear:

```text
v1.2.4
v1.3.8
v2.0.1
```

14. For monorepos, choose either repo-wide tags:

```text
v1.2.3
```

or package-specific tags:

```text
api-v1.2.3
web-v4.5.6
sdk-js-v2.1.0
```

Do not mix both without a documented policy.

## 9. Immutable releases and artifacts

A release should be immutable after publication. GitHub immutable releases protect assets and associated tags from being changed after publication; they also generate release attestations. ([GitHub Docs][18])

Best practices:

1. Enable immutable releases where available.
2. Create releases as drafts first.
3. Attach all assets before publishing.
4. Publish once.
5. If something is wrong, publish a new version.
6. Do not replace binaries under the same version.
7. Do not mutate a release because “nobody downloaded it yet”.
8. Preserve checksums.
9. Preserve SBOMs.
10. Preserve signatures/attestations.
11. Make release assets verifiable outside the hosting platform.

GitHub’s release docs recommend creating a draft first when immutable releases are enabled so all assets are attached before the release becomes immutable. ([GitHub Docs][19])

## 10. Artifact standards

For every production release, produce:

1. Source tag.
2. Build artifact.
3. Artifact hash.
4. Signature or attestation.
5. SBOM.
6. Release notes.
7. Deployment manifest.
8. Container digest, if using containers.
9. Package registry URL/version, if publishing packages.
10. Provenance linking artifact to commit and workflow.

Recommended artifact metadata:

```text
version
commit_sha
tag
build_timestamp
builder
workflow_run_id
source_repository
dependency_lockfile_hash
artifact_sha256
sbom_sha256
```

Never rely only on mutable names:

```text
latest
stable
prod
main
master
```

For production deployment, use immutable identifiers:

```text
ghcr.io/org/app@sha256:...
npm package version 1.2.3
Maven coordinate 1.2.3
Git tag v1.2.3
commit SHA abc123...
```

GitHub artifact attestations provide cryptographically signed provenance claims that include repository, organization, environment, commit SHA, workflow, and triggering event. ([GitHub Docs][20])

## 11. SBOM and supply chain standards

For serious software, generate an SBOM for release artifacts. SPDX and CycloneDX are the two most common standards. SPDX is an open SBOM standard and is ISO/IEC 5962:2021; CycloneDX is an OWASP/Ecma bill-of-materials standard with supply-chain risk capabilities. ([SPDX][21])

Best practices:

1. Generate SBOM at build time.
2. Include direct and transitive dependencies.
3. Include versions and package identifiers.
4. Include license data.
5. Include hashes where possible.
6. Attach SBOM to release.
7. Sign or attest SBOM.
8. Scan SBOM for vulnerabilities.
9. Keep SBOM for every released version.
10. Make SBOM accessible to customers/users where appropriate.

Use SLSA as the security maturity model for build provenance and tamper resistance. OpenSSF describes SLSA as an incrementally adoptable software supply-chain security framework useful for producers and consumers. ([OpenSSF][22])

Use OpenSSF Scorecard to evaluate repository security posture; Scorecard checks for risks such as build system compromise, source compromise, malicious packages, and other supply-chain practices. ([undefined][23])

NIST SSDF is also a strong baseline: it organizes secure software development into preparing the organization, protecting software, producing well-secured software, and responding to vulnerabilities. ([NIST Computer Security Resource Center][24])

## 12. GitHub Actions / CI security standards

Minimum security posture:

1. Set default workflow permissions to read-only.
2. Grant write permission only at job level where needed.
3. Pin third-party actions to full-length commit SHAs.
4. Audit third-party actions before use.
5. Use trusted publishers where available.
6. Use OIDC for cloud credentials.
7. Avoid long-lived cloud secrets.
8. Do not expose release secrets to fork PRs.
9. Do not use self-hosted runners for untrusted public PRs.
10. Require CODEOWNERS review for workflow changes.
11. Treat workflow files as production code.
12. Rotate secrets.
13. Mask sensitive output.
14. Avoid inline shell interpolation of untrusted PR fields.
15. Use environment protection rules for production deploys.
16. Require manual approval for production secrets where appropriate.
17. Use Dependabot/Renovate for actions and dependencies.
18. Scan workflows for script injection and risky token permissions.
19. Log enough for audit, but never log secrets.
20. Separate build and deploy credentials.

GitHub says pinning actions to a full-length commit SHA is the only way to use an action as an immutable release, and warns that tags can be moved or deleted if an action repository is compromised. ([GitHub Docs][25])

GitHub also recommends OIDC so workflows can request short-lived cloud tokens instead of storing long-lived cloud credentials as GitHub secrets. ([GitHub Docs][26])

## 13. Dependency management standards

1. Use lockfiles.
2. Commit lockfiles for applications.
3. For libraries, follow ecosystem norms, but test against locked and allowed ranges.
4. Use Dependabot/Renovate.
5. Review dependency updates.
6. Auto-merge only low-risk updates after CI.
7. Never auto-merge major updates without review.
8. Block known critical vulnerabilities.
9. Run dependency review on PRs.
10. Monitor transitive dependencies.
11. Pin Docker base images by digest for production builds.
12. Avoid unmaintained dependencies.
13. Vendor only with policy.
14. Keep dependency update PRs small and frequent.
15. Document dependency support windows.

GitHub dependency review can catch insecure dependencies in PRs before merge, and Dependabot can scan for vulnerabilities, create alerts, and open update PRs. ([GitHub Docs][27])

## 14. Release notes best practices

Every release note should include:

1. Version.
2. Date.
3. Summary.
4. Breaking changes.
5. New features.
6. Fixes.
7. Security fixes.
8. Deprecations.
9. Removals.
10. Migration steps.
11. Known issues.
12. Contributors.
13. Checksums/signature instructions.
14. Compatibility notes.
15. Links to full diff, issues, PRs.
16. Upgrade instructions.
17. Rollback caveats.
18. Database migration notes.
19. Config changes.
20. Operational impact.

Bad release note:

```text
Bug fixes and improvements.
```

Good release note:

```text
## v2.4.0 - 2026-05-05

### Added
- Add SAML group mapping for enterprise accounts.

### Changed
- Increase default API timeout from 5s to 15s.

### Fixed
- Fix token refresh failure after clock skew greater than 60s.

### Security
- Upgrade openssl from 3.2.1 to 3.2.3.

### Breaking Changes
- Remove deprecated /v1/session endpoint. Use /v2/session.

### Migration
- Run `app migrate --to 20260505` before deploying workers.
```

## 15. Deployment versus release

These are not the same:

| Term         | Meaning                               |
| ------------ | ------------------------------------- |
| Build        | Compile/package software.             |
| Release      | Make a version available or approved. |
| Deploy       | Put software into an environment.     |
| Enable       | Turn on a feature for users.          |
| Rollout      | Gradually expand exposure.            |
| Rollback     | Return to prior version/behavior.     |
| Roll forward | Ship a fix as a newer version.        |

Best practices:

1. Build once; promote the same artifact across environments.
2. Do not rebuild separately for staging and production.
3. Use environment-specific config, not environment-specific binaries.
4. Use feature flags for risky or incomplete features.
5. Use canaries or phased rollouts for high-risk services.
6. Keep deploys boring and reversible.
7. Separate database migration rollout from application rollout when needed.
8. Make migrations backward compatible.
9. Monitor after deploy.
10. Define automatic rollback criteria.

---

## 16. Database and schema release standards

1. Prefer expand-migrate-contract.
2. Make migrations backward compatible.
3. Deploy additive schema changes before code depends on them.
4. Backfill data separately for large tables.
5. Avoid locking migrations during peak traffic.
6. Test migrations on realistic data volume.
7. Version database migrations.
8. Include migration rollback or roll-forward plan.
9. Do not couple irreversible migrations to risky app releases.
10. Never drop columns/tables in the same release where code stops using them.
11. Include migration timing in release notes.
12. Ensure old and new app versions can run during rolling deploys.
13. Treat schema changes as breaking changes if external consumers depend on them.
14. Monitor migration progress.
15. Document manual intervention steps.

---

## 17. Hotfix standards

A hotfix is not an excuse to abandon process.

Best practices:

1. Start from the affected release tag or current mainline, depending on your model.
2. Make the smallest safe change.
3. Run the same relevant CI gates.
4. Create a patch version:

```text
1.2.3 -> 1.2.4
```

5. Tag immutably.
6. Publish release notes.
7. Backport/forward-port to all supported branches.
8. Verify the fix in production.
9. Add regression tests.
10. Do a brief postmortem if it was urgent.

Martin Fowler notes that continuous delivery can allow hotfixes to go through the normal release process rather than sidestepping it. ([martinfowler.com][28])

---

## 18. Rollback and recovery standards

Before release, know:

1. Previous known-good version.
2. Exact artifact/digest to restore.
3. Whether database changes are backward compatible.
4. Whether config changed.
5. Whether queues/events are compatible.
6. Whether clients can tolerate rollback.
7. Whether feature flags can disable the feature.
8. Who can authorize rollback.
9. How to communicate rollback.
10. How to verify recovery.

Rollback options:

| Situation               | Preferred action                                             |
| ----------------------- | ------------------------------------------------------------ |
| Simple app regression   | Roll back artifact.                                          |
| Bad config              | Roll back config.                                            |
| Bad feature behavior    | Disable feature flag.                                        |
| Data migration issue    | Pause rollout, roll forward with fix, or restore if safe.    |
| Security release broken | Roll forward rapidly if rollback reintroduces vulnerability. |

## 19. Multi-package / monorepo standards

Choose one versioning model:

### Fixed/locked versioning

All packages share one version:

```text
repo v3.4.0
```

Good for tightly coupled systems.

### Independent versioning

Each package has its own version:

```text
api-client v2.1.0
ui-kit v5.8.2
core v1.9.0
```

Good for libraries/packages with independent consumers.

Best practices:

1. Define ownership per package.
2. Use package-specific changelogs or clear sections.
3. Use package-specific tags if independently versioned.
4. Track internal dependency bumps.
5. Use Changesets or equivalent for release intent.
6. Avoid accidental releases of every package.
7. Validate cross-package compatibility.
8. Keep generated release notes package-scoped.
9. Document support matrix.
10. Avoid version drift between code, package metadata, and docs.

# Common “inexcusable” mistakes

“Inexcusable” here means **preventable through basic policy, automation, or documented procedure**.

## A. Git history and branch mistakes

1. **Force-pushing to `main` or a release branch.**
2. **Rewriting public history that others may have based work on.** The Git book’s rebase warning is blunt: do not rebase commits that exist outside your repository and that others may have based work on. ([Git][29])
3. Deleting `main`.
4. Disabling branch protection “temporarily” and forgetting to restore it.
5. Allowing admins to bypass rules without audit.
6. Direct commits to protected branches.
7. Merging without review.
8. Merging with failed CI.
9. Merging stale PRs without updating.
10. Keeping long-lived feature branches for weeks/months.
11. Letting release branches diverge permanently.
12. Forgetting to merge hotfixes back to `main`.
13. Cherry-picking without tracking what was backported.
14. Using multiple branching models at once with no documentation.
15. Naming branches ambiguously:

```text
fix
new
test
release
john
```

16. Committing generated garbage or local IDE files.
17. Not using `.gitignore`.
18. Not using `.gitattributes` for line endings in cross-platform teams.
19. Storing huge binaries in Git instead of LFS/artifact storage.
20. Using submodules without pinning and documentation.

---

## B. Tag and release mistakes

1. **Moving a published release tag.**
2. **Deleting a release tag.**
3. **Reusing a version number.**
4. **Replacing a release artifact without changing the version.**
5. Publishing a release from the wrong commit.
6. Tagging before CI passes.
7. Tagging from a dirty local working tree.
8. Creating lightweight tags for official releases when annotated/signed tags are expected.
9. Pushing all local tags accidentally.
10. Having tag version, package version, Docker tag, and changelog disagree.
11. Marking a beta/RC as latest.
12. Publishing `latest` as the only production identifier.
13. Using mutable Docker tags in production without digest pinning.
14. Failing to attach checksums.
15. Failing to attach release notes.
16. Failing to attach migration notes.
17. Publishing artifacts from a developer laptop.
18. Publishing artifacts that cannot be traced to source.
19. Publishing artifacts that cannot be reproduced or verified.
20. Publishing release assets and later editing them silently.

SemVer explicitly says released contents must not be modified and modifications must be released as a new version. ([Semantic Versioning][2])

---

## C. Versioning mistakes

1. **Breaking APIs in a patch release.**
2. **Breaking APIs in a minor release without marking it breaking.**
3. Calling everything `1.0.0` forever.
4. Never leaving `0.x` to avoid compatibility promises.
5. Using date versions for libraries consumed by dependency managers without clear compatibility rules.
6. Having no documented public API.
7. Treating config changes as non-breaking when they break deploys.
8. Treating schema/event changes as non-breaking when consumers break.
9. Hiding removals under “refactor”.
10. Forgetting to update package metadata.
11. Forgetting to update lockfiles.
12. Forgetting to update docs.
13. Versioning code and docs separately with no compatibility matrix.
14. Releasing multiple artifacts with the same version but different contents.
15. Manually bumping versions and getting them wrong.

1. **No changelog.**

3. “Bug fixes and improvements” for every release.
4. Omitting breaking changes.
5. Omitting migration steps.
6. Omitting deprecations.
7. Omitting removals.
8. Omitting security fixes after disclosure.
9. Publishing security details before users can patch.
10. Not identifying known issues.
11. Not saying whether the release is production-ready.
12. Not linking PRs/issues.
13. Not crediting contributors where appropriate.
14. Not documenting config changes.
15. Not documenting database migrations.
16. Not documenting operational impact.
17. Not documenting rollback caveats.
18. Release notes written only for the author.
19. Release notes written only for machines.
20. Changelog order oldest-first with no clear latest release.

Keep a Changelog’s explicit position is that changelogs are for humans, not machines. ([Keep a Changelog][4])

## E. CI/CD and automation mistakes

1. **Release pipeline can be triggered from unprotected branches.**
2. **Release pipeline has broad write/admin credentials.**
3. **CI failures can be bypassed casually.**
4. Release job does not verify version/tag consistency.
5. Release job does not verify clean build state.
6. Release job does not run tests.
7. Release job builds from one commit and tags another.
8. Release job publishes artifacts before tests finish.
9. Release job uploads partial artifacts.
10. Release job can overwrite existing artifacts.
11. Release job uses personal access tokens instead of scoped automation.
12. CI secrets available to untrusted PRs.
13. Fork PRs can execute privileged workflows.
14. Third-party actions are not pinned.
15. CI workflow changes do not require review.
16. Production deployment is one unreviewed button.
17. No concurrency control; two releases race.
18. No idempotency; retrying corrupts release state.
19. No audit logs for who released.
20. No notification on failed release.

GitHub warns that third-party actions can compromise secrets and repository tokens, and recommends pinning actions to full commit SHAs, auditing action source, and limiting token permissions. ([GitHub Docs][25])

# The minimum “gold standard” policy

Here is the policy I would actually put into an engineering handbook.

## Repository policy

1. `main` is always releasable.
2. `main` is protected.
3. Direct pushes to `main` are blocked.
4. Force pushes to `main` and release branches are blocked.
5. PR required.
6. Required CI checks must pass.
7. CODEOWNERS required for critical paths.
8. Workflow/deployment changes require platform/security review.
9. Secrets are never committed.
10. Dependencies are scanned continuously.

## Commit and PR policy

1. PR title or squash commit follows Conventional Commits.
2. Breaking changes use `!` or `BREAKING CHANGE:`.
3. PRs include test evidence.
4. PRs include docs/release note labels when user-facing.
5. PRs are small.
6. PRs link issues/tickets.
7. Security-sensitive PRs get security review.
8. Database PRs include migration/rollback notes.

## Release policy

1. Releases are created by CI only.
2. Version follows SemVer unless documented otherwise.
3. Release tag is annotated and preferably signed.
4. Release tag is protected.
5. Release is immutable after publication.
6. Artifact is built from the tagged commit.
7. Artifact has checksum/digest.
8. Artifact has provenance/attestation where feasible.
9. SBOM is generated for production/customer artifacts.
10. Release notes follow Keep a Changelog.
11. Breaking changes and migrations are prominent.
12. Hotfixes create new patch versions.
13. Published artifacts are never overwritten.
14. Rollback path is known before release.
15. Post-release monitoring is required.

## CI/CD security policy

1. `GITHUB_TOKEN` default is read-only.
2. Job permissions are explicit and minimal.
3. Third-party actions are pinned to full commit SHA.
4. OIDC is used for cloud deployments where supported.
5. Long-lived credentials are avoided.
6. Production secrets require protected environments.
7. Self-hosted runners do not run untrusted public PR code.
8. Workflow changes require CODEOWNERS.
9. Dependabot/Renovate keeps actions and dependencies current.
10. Scorecard/SLSA-style controls are tracked over time.

# The shortest possible checklist before pressing “release”

Use this every time:

```text
[ ] Correct version?
[ ] Correct branch?
[ ] Correct commit SHA?
[ ] All required CI green?
[ ] Required reviews complete?
[ ] Changelog updated?
[ ] Breaking changes documented?
[ ] Migration steps documented?
[ ] Rollback/roll-forward path known?
[ ] Tag protected/signed?
[ ] Artifact built by CI?
[ ] Artifact version matches tag?
[ ] Checksums/digests generated?
[ ] SBOM generated, if required?
[ ] Attestation/signature generated, if required?
[ ] Release notes reviewed by a human?
[ ] Pre-release/latest flag correct?
[ ] Deployment plan ready?
[ ] Monitoring ready?
[ ] Owner available after release?
```

If any answer is “no” for a production release, do not publish unless it is an emergency hotfix and the exception is written down.
