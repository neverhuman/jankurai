Absolutely — here is a rigorous, production-grade checklist of **Git and release best practices**, followed by the **worst practices that should be treated as inexcusable** in a serious engineering team.

I treated “best” as meaning: **traceable, reproducible, secure, reversible, automatable, auditable, and understandable by humans**. I prioritized official/spec sources first: Git, SemVer, Keep a Changelog, Conventional Commits, GitHub, GitLab, DORA, OWASP, OpenSSF/SLSA, and NIST-aligned guidance. I also checked community discussions from places like Stack Overflow, Reddit, LinkedIn-style posts, and GitOps commentary, but I weighted those lower than standards and official documentation. Quora was not accessible in a way I could reliably inspect here, so I did not rely on it.

## The north-star standard

A professional release should be:

**Identifiable**: every release has a unique version and a unique Git reference.

**Immutable**: once published, the release tag and release artifacts are not changed.

**Reproducible**: the release can be rebuilt or at least traced back to source, commit, dependencies, build environment, and pipeline.

**Verifiable**: artifacts have checksums, signatures, provenance, and ideally SBOMs.

**Documented**: humans can understand what changed, what broke, what was fixed, and how to upgrade.

**Recoverable**: rollback, roll-forward, hotfix, and incident paths are known before release.

**Governed**: protected branches, protected tags, required checks, approvals, and audit logs are in place.

Git’s own documentation distinguishes annotated tags from lightweight tags and says annotated tags carry metadata such as tagger, date, message, and optional cryptographic signature; lightweight tags are more like simple pointers. The Git book also describes tagging as the normal way to mark a release so it can be recreated later. ([Git][1])

# 1. The release object: what a release must contain

A serious release should contain all of the following:

1. **A unique version identifier**

   * Usually SemVer: `MAJOR.MINOR.PATCH`.
   * Sometimes CalVer: `YYYY.MM.DD`, `YYYY.MM`, or similar.
   * Sometimes a product-specific version scheme.
   * But it must be unique, ordered, documented, and never reused.

2. **A Git tag**

   * Prefer `v1.2.3` or another consistent tag convention.
   * Use **annotated tags** for releases.
   * Use **signed tags** where possible.
   * Protect release tag patterns such as `v*`.

3. **A source commit**

   * The release must map to exactly one commit SHA.
   * Release notes, artifacts, images, packages, and deployment metadata should all point back to that SHA.

4. **Release notes**

   * Human-readable summary.
   * Breaking changes.
   * Migration instructions.
   * Security fixes.
   * Known issues.
   * Contributors, when appropriate.

5. **A changelog entry**

   * Ideally maintained in `CHANGELOG.md`.
   * Should not be merely a raw dump of commit messages.

6. **Built artifacts**

   * Packages, binaries, containers, installers, archives, charts, SDKs, docs, or whatever the product ships.

7. **Integrity metadata**

   * Checksums.
   * Signatures.
   * Provenance or attestations.
   * SBOM, especially for public, enterprise, regulated, or security-sensitive software.

8. **Deployment record**

   * Environment deployed to.
   * Time.
   * Actor or automation identity.
   * Artifact version.
   * Commit SHA.
   * Approval record.
   * Rollback/roll-forward status.

GitHub describes releases as deployable software iterations based on Git tags, with release notes and downloadable archives; GitLab similarly treats a release as a combination of code snapshot, artifacts, docs, and release metadata. OpenSSF’s baseline guidance also calls for unique release identifiers, release logs, and signed releases or signed manifests with artifact hashes. ([GitHub Docs][2])

# 2. Versioning best practices

## Use a documented versioning scheme

The most common standard is **Semantic Versioning**:

`MAJOR.MINOR.PATCH`

Use it like this:

* Increment **MAJOR** for incompatible API or contract changes.
* Increment **MINOR** for backward-compatible functionality.
* Increment **PATCH** for backward-compatible bug fixes.
* Use pre-release labels such as `1.4.0-rc.1`, `1.4.0-beta.2`, or `2.0.0-alpha.1`.
* Use build metadata only for metadata, not compatibility meaning.

SemVer explicitly says released contents must not be modified after release. Any change requires a new version. That rule is one of the most important release-management principles. ([Semantic Versioning][3])

## Best practices

1. **Document the versioning policy**

   * Put it in `CONTRIBUTING.md`, `RELEASE.md`, or engineering docs.
   * Explain what counts as breaking for your project.
   * Include API, CLI, config, database schema, file format, event schema, dependency, and behavioral compatibility rules.

2. **Treat compatibility broadly**

   * Breaking changes are not only API signature changes.
   * They can include changed defaults, removed config keys, changed CLI output, changed error codes, changed database assumptions, changed event schemas, removed environment variables, changed auth scopes, or changed deployment requirements.

3. **Never reuse a version**

   * `1.2.3` must always mean the same source and artifacts.
   * If a release is bad, publish `1.2.4`, `1.2.3+fix`, or an officially documented replacement depending on ecosystem rules — but do not mutate the old release.

4. **Do not publish “final” versions from untested code**

   * Use `alpha`, `beta`, `rc`, `preview`, or `nightly` labels.

5. **Align every surface**

   * Git tag: `v1.2.3`
   * Package version: `1.2.3`
   * Container image tag: `1.2.3`
   * Release title: `1.2.3`
   * Docs version: `1.2.3`
   * Changelog section: `1.2.3`

6. **Use monotonic versioning**

   * Versions should move forward.
   * Avoid strange resets, hidden forks, duplicate release lines, or undocumented parallel version schemes.

7. **Use pre-release versions for release candidates**

   * `2.0.0-rc.1`
   * `2.0.0-rc.2`
   * `2.0.0`

8. **Make breaking changes painfully visible**

   * Put them in release notes.
   * Put them in changelog.
   * Put them in migration docs.
   * Mention them in PR labels.
   * Consider a `BREAKING CHANGE:` footer if using Conventional Commits.

## Inexcusable versioning mistakes

1. **Shipping a breaking change as a patch release.**
2. **Changing the contents of an already published version.**
3. **Reusing a version number for different code.**
4. **Having no documented versioning policy.**
5. **Using “latest” as the only meaningful release identifier.**
6. **Publishing packages, images, and docs with mismatched versions.**
7. **Calling something stable while it is actually experimental.**
8. **Forgetting to remove `-rc`, `-beta`, or `SNAPSHOT` labels from a final release.**
9. **Using dates, build numbers, or commit hashes inconsistently without explaining precedence.**
10. **Treating internal-only version bumps as public releases without release notes or traceability.**

# 3. Git tag best practices

## Use annotated, signed tags for releases

A release tag should normally be:

```bash
git tag -s v1.2.3 -m "Release v1.2.3"
git push origin v1.2.3
```

If signing is not available:

```bash
git tag -a v1.2.3 -m "Release v1.2.3"
git push origin v1.2.3
```

Annotated tags carry release metadata. Signed tags add cryptographic verification. Lightweight tags are fine for personal bookmarks, temporary markers, or local experiments, but they are a poor choice for official releases. Git’s documentation explicitly describes annotated tags as full objects with tagger, date, message, and optional signature, while lightweight tags are just direct references. ([Git][1])

## Best practices

1. **Protect release tag patterns**

   * Protect `v*`, `release-*`, or whatever pattern your project uses.
   * Restrict who can create, update, or delete release tags.

2. **Never move a release tag**

   * A tag named `v1.2.3` should always point to the same commit.

3. **Never delete a published release tag casually**

   * Deleting a tag can break package consumers, deployment tools, audit trails, and reproducibility.

4. **Sign tags**

   * Use GPG, SSH signing, Sigstore, or your platform’s supported signing model.
   * Enforce signed tags where your hosting platform allows it.

5. **Tag the exact source used to build the release**

   * Build from the tag.
   * Do not tag after manually modifying artifacts.

6. **Use consistent names**

   * Good: `v1.2.3`
   * Good: `my-service-v1.2.3` in monorepos.
   * Bad: `release`, `stable`, `prod`, `latest`, `final-final`, `new-prod`.

7. **Avoid branch/tag name collisions**

   * Do not create a branch and tag with the same name.
   * GitLab specifically warns that tags with the same name as branches can cause confusing checkout behavior. ([GitLab Docs][4])

8. **Tag from automation when possible**

   * Humans can approve.
   * Automation should perform the mechanical release operation.

9. **Put release metadata in the release object, not only in the tag**

   * Tag message.
   * GitHub/GitLab release notes.
   * Changelog.
   * Artifact checksums.
   * Provenance.

10. **Verify the tag before publishing**

* Confirm SHA.
* Confirm branch.
* Confirm CI status.
* Confirm version files.
* Confirm changelog.

GitHub rulesets can restrict branch and tag creation, updates, and deletion; GitLab protected tags similarly allow teams to control who can create release tags and prevent accidental or unauthorized tag changes. ([GitHub Docs][5])

## Inexcusable tag mistakes

1. **Force-moving a published release tag.**
2. **Deleting a public release tag without a documented emergency reason.**
3. **Using lightweight tags for important production releases.**
4. **Creating release tags from a developer laptop without verification.**
5. **Tagging the wrong commit.**
6. **Tagging before CI has passed.**
7. **Using mutable names like `latest`, `stable`, or `prod` as the only release reference.**
8. **Allowing everyone to create or delete release tags.**
9. **Having branch names and tag names collide.**
10. **Publishing artifacts that were not built from the tagged commit.**

# 4. Branching best practices

## Prefer trunk-based development unless you have a strong reason not to

For many modern teams, the default should be:

* Short-lived branches.
* Frequent merges into `main`.
* CI always green.
* Feature flags for incomplete work.
* Release branches only when necessary.

DORA’s guidance says high-performing teams tend to keep fewer active branches, merge to trunk frequently, and avoid long code-freeze or integration phases. Atlassian’s trunk-based guidance similarly emphasizes small, frequent changes merged into the main branch, with CI and feature flags reducing the risks of long-lived branches. ([Dora][6])

## When release branches are appropriate

Release branches are useful when:

* You support multiple versions at once.
* You ship scheduled enterprise releases.
* You maintain LTS versions.
* You need stabilization after feature cutoff.
* You need hotfixes for older supported versions.

A common model:

```text
main
  ├── release/1.4
  ├── release/1.5
  └── release/2.0
```

Release branches should receive only:

* Bug fixes.
* Security fixes.
* Documentation corrections.
* Release-specific configuration or metadata.
* Version/changelog updates.

Atlassian’s Gitflow documentation describes release branches as places for bug fixes, documentation, and release-oriented tasks — not new feature development — and hotfix branches as starting from production/main and merging back into the main development line. ([Atlassian][7])

## Best practices

1. **Keep `main` releasable**

   * The main branch should be healthy, tested, and deployable.

2. **Use short-lived feature branches**

   * Hours or days, not weeks or months.
   * Large work should be split behind flags.

3. **Use pull requests or merge requests**

   * Even for senior engineers.
   * Especially for release-critical changes.

4. **Avoid long-lived integration branches**

   * They hide risk.
   * They create merge hell.
   * They delay feedback.

5. **Use release branches only for stabilization or maintenance**

   * Do not turn them into parallel development universes.

6. **Fix bugs on trunk first**

   * Then cherry-pick/backport to release branches.
   * Trunk-based development guidance recommends reproducing and fixing bugs on trunk first, verifying with CI, and then cherry-picking to the release branch. ([Trunk Based Development][8])

7. **Require documented cherry-picks**

   * Link to original PR.
   * Link to issue.
   * Keep the same test.
   * Note conflicts or deviations.

8. **Use branch naming conventions**

   * `feature/...`
   * `fix/...`
   * `hotfix/...`
   * `release/...`
   * `docs/...`
   * `security/...`

9. **Delete merged feature branches**

   * Reduces clutter.
   * Reduces accidental reuse.

10. **Protect long-lived branches**

* `main`
* `develop`, if used
* `release/*`
* `hotfix/*`
* LTS branches

## Inexcusable branching mistakes

1. **No protected `main` branch.**
2. **Long-lived feature branches with weeks or months of unmerged work.**
3. **Merging unreviewed code into production branches.**
4. **Fixing bugs only on a release branch and forgetting trunk.**
5. **Undocumented cherry-picks.**
6. **Letting release branches accept new features after cutoff.**
7. **Using branches as permanent environments without controlling drift.**
8. **Keeping dead branches forever.**
9. **Having no naming convention.**
10. **Treating merge conflicts as a release-day surprise.**

# 5. Commit and history best practices

## Keep commits small, meaningful, and reviewable

Good commits are:

* Focused.
* Tested.
* Revertable.
* Well-named.
* Linked to issue or PR context.
* Not mixed with unrelated formatting churn.

Microsoft’s Git guidance emphasizes small commits, avoiding mixing whitespace and functional changes, committing complete and tested work, and writing good commit messages. ([Microsoft GitHub][9])

## Use Conventional Commits when automation matters

Conventional Commits is useful when you want automated changelogs, release notes, and SemVer version bumps.

Examples:

```text
feat(auth): add SAML login
fix(api): handle expired refresh tokens
docs(release): document rollback procedure
perf(search): reduce index lookup time
refactor(payments): isolate invoice calculation
test(cli): add regression test for config parsing
chore(deps): update build tooling
```

Breaking change example:

```text
feat(api)!: remove v1 invoice endpoint

BREAKING CHANGE: /v1/invoices has been removed. Use /v2/invoices.
```

Conventional Commits defines a structured commit convention that maps naturally to SemVer: `fix` corresponds to patch-level changes, `feat` to minor-level changes, and breaking-change markers to major changes. ([Conventional Commits][10])

## Best practices

1. **Write imperative commit subjects**

   * Good: `fix cache invalidation for expired tokens`
   * Bad: `fixed stuff`

2. **Explain why, not just what**

   * The diff shows what changed.
   * The message should explain intent, context, and risk.

3. **Keep commits atomic**

   * One logical change per commit.

4. **Do not mix formatting with behavior**

   * Formatting-only PRs should be separate.

5. **Squash noisy work when appropriate**

   * Especially “fix typo”, “oops”, “try again” commits.
   * But preserve useful history for complex work.

6. **Do not rewrite public history**

   * Rebasing your private branch is fine.
   * Rebasing shared/public history is dangerous.
   * Atlassian’s Git rebase guidance plainly warns not to rebase public history because it replaces commits and can make existing shared history disappear from other contributors’ perspective. ([Atlassian][11])

7. **Use signed commits for sensitive projects**

   * Especially for release branches, security-sensitive repos, and regulated environments.

8. **Reference issues**

   * `Fixes #123`
   * `Refs #456`
   * Links release work to planning and audit trails.

9. **Keep generated files policy clear**

   * Either commit generated files intentionally or never commit them.
   * Avoid inconsistency.

10. **Never commit secrets**

* Use secret scanning.
* Rotate immediately if exposure happens.

## Inexcusable commit/history mistakes

1. **Force-pushing shared branches without coordination.**
2. **Rewriting release history.**
3. **Committing secrets, keys, tokens, or credentials.**
4. **Committing generated build artifacts accidentally.**
5. **Making giant unreviewable commits.**
6. **Using meaningless messages like `stuff`, `changes`, `fix`, `wip`, `asdf`.**
7. **Mixing refactor, formatting, feature, and bug fix in one huge commit.**
8. **Committing directly to protected branches.**
9. **Changing code after review without re-review.**
10. **Squashing away important security, migration, or authorship context.**

# 6. Pull request and merge best practices

## Protect critical branches

At minimum, protect:

* `main`
* `release/*`
* `hotfix/*`
* LTS branches
* deployment branches, if they exist

Require:

* Passing CI.
* Required status checks.
* At least one review.
* CODEOWNERS review for sensitive areas.
* Resolved conversations.
* Up-to-date branch or merge queue.
* No direct pushes.
* No force pushes.
* No deletions.
* Signed commits where appropriate.

GitHub branch protection supports required pull request reviews, required status checks, signed commits, linear history, merge queues, and deployment-related requirements. GitHub also notes branch protection can block deletion and force pushes by default. ([GitHub Docs][12])

## Best practices

1. **Require PRs for release-bound changes**

   * Even “tiny” release changes can break production.

2. **Use CODEOWNERS**

   * Security owns auth-sensitive files.
   * DB owners review migrations.
   * Platform owns CI/CD.
   * Docs owns public docs.

3. **Require status checks**

   * Build.
   * Unit tests.
   * Integration tests.
   * Lint.
   * Typecheck.
   * Security scans.
   * License checks.
   * Compatibility checks.

4. **Use merge queues for busy repos**

   * Prevents “passed alone, failed together” problems.

5. **Make PRs small**

   * Smaller PRs are reviewed better and released safer.

6. **Label PRs**

   * `breaking-change`
   * `security`
   * `bug`
   * `feature`
   * `docs`
   * `release-note-needed`
   * `no-release-note`
   * `backport-needed`

7. **Require release note labels**

   * Every user-visible change should be labeled.

8. **Require tests for fixes**

   * A bug fix without a regression test should be suspicious.

9. **Use draft PRs early**

   * Drafts give early visibility without pretending work is ready.

10. **Block merge if release metadata is missing**

* Version bump.
* Changelog.
* Migration note.
* Feature flag plan.
* Rollback plan.

## Inexcusable PR/merge mistakes

1. **Merging failing CI.**
2. **Bypassing required reviews.**
3. **Approving PRs nobody actually read.**
4. **Letting authors self-approve production changes.**
5. **Merging huge PRs at the last minute before release.**
6. **Ignoring unresolved review comments.**
7. **Changing release-critical code after approval without re-review.**
8. **Treating flaky tests as acceptable background noise.**
9. **Disabling branch protection to “get the release out.”**
10. **Using admin override as a normal workflow.**

# 7. Changelog and release-note best practices

## Maintain a human-readable changelog

A changelog is not the same as Git history. Git history is for developers. A changelog is for users, operators, support, security teams, auditors, and future maintainers.

Keep a Changelog recommends human-readable changelogs, an `Unreleased` section, and common categories such as Added, Changed, Deprecated, Removed, Fixed, and Security. ([Keep a Changelog][13])

## Best practices

1. **Keep `CHANGELOG.md` in the repo**

   * Or maintain equivalent release notes in a durable release system.

2. **Use an `Unreleased` section**

   * Accumulate changes before the release.
   * Move them under the version at release time.

3. **Use consistent categories**

   * Added.
   * Changed.
   * Deprecated.
   * Removed.
   * Fixed.
   * Security.

4. **Call out breaking changes separately**

   * Make them impossible to miss.

5. **Call out security fixes carefully**

   * Give enough information to upgrade.
   * Avoid exploit-enabling detail before disclosure is appropriate.

6. **Include migration instructions**

   * Config changes.
   * DB migrations.
   * API replacements.
   * CLI changes.
   * Deprecation timelines.

7. **Include known issues**

   * Especially if users might hit them immediately.

8. **Use generated notes as a starting point, not the final product**

   * GitHub can generate release notes from merged PRs and contributors, and can be customized with labels and exclusions, but generated notes should still be checked before publication. ([GitHub Docs][14])

9. **Use labels to drive release notes**

   * `release-note`
   * `no-release-note`
   * `breaking-change`
   * `security`
   * `deprecation`

10. **Make release notes audience-aware**

* Developers need API changes.
* Operators need deployment notes.
* Security teams need CVE/security impact.
* Users need behavior changes.

## Inexcusable changelog/release-note mistakes

1. **No release notes for a production release.**
2. **Release notes that only say “bug fixes and improvements.”**
3. **Hiding breaking changes.**
4. **Failing to mention required migrations.**
5. **Failing to mention security fixes.**
6. **Dumping raw commit logs as “release notes.”**
7. **Forgetting known issues.**
8. **Publishing notes that do not match shipped code.**
9. **Not distinguishing user-visible changes from internal chores.**
10. **No upgrade path for removed or deprecated behavior.**

# 8. CI/CD and build best practices

## Build from clean, controlled automation

A release should not be built from a developer’s laptop. It should be built by CI from a known commit or tag in a clean environment.

Best practice:

```text
tag v1.2.3
   ↓
CI checks out tag
   ↓
CI builds artifact
   ↓
CI tests artifact
   ↓
CI signs/checksums/attests artifact
   ↓
CI publishes release
   ↓
deployment uses same artifact
```

OWASP describes CI/CD systems as critical, high-value targets because they have access to code, secrets, build systems, deployment credentials, and automation identities. That means CI/CD must be treated as production infrastructure, not convenience scripting. ([OWASP Cheat Sheet Series][15])

## Best practices

1. **Build once, promote the same artifact**

   * Do not rebuild separately for staging and production.
   * Build once.
   * Promote through environments.

2. **Use immutable artifacts**

   * Container digest, not only tag.
   * Package checksum.
   * Signed archive.
   * Immutable release asset.

3. **Pin dependencies and build actions**

   * Avoid floating versions where possible.
   * Pin GitHub Actions by SHA for high-security repos.
   * Pin base images by digest when practical.

4. **Use hermetic or reproducible builds where possible**

   * Lockfiles.
   * Containerized build environments.
   * Toolchain version pinning.
   * Deterministic build inputs.

5. **Separate build and deploy permissions**

   * Build job should not automatically have unrestricted production deploy rights.

6. **Use least-privilege tokens**

   * Short-lived credentials.
   * Scoped permissions.
   * No personal access tokens for production releases.

7. **Require CI for release branches and tags**

   * Do not let tags bypass tests.

8. **Run security checks**

   * SAST.
   * dependency scanning.
   * secret scanning.
   * container scanning.
   * license scanning.
   * IaC scanning where relevant.

9. **Archive logs and metadata**

   * Build logs.
   * Test results.
   * SBOM.
   * provenance.
   * approvals.
   * deployment records.

10. **Make release pipelines idempotent**

* Re-running a job should not create a different release unexpectedly.
* Failed releases should be recoverable.

Microsoft’s continuous-delivery playbook frames release strategy as including ownership, asset and configuration strategy, environments, approval processes, deployment configuration, external systems, and disaster recovery. ([Microsoft GitHub][16])

## Inexcusable CI/CD mistakes

1. **Building production artifacts on a developer laptop.**
2. **Deploying code that did not pass CI.**
3. **Rebuilding different artifacts for different environments.**
4. **Using mutable dependencies without locks.**
5. **Using personal credentials in release automation.**
6. **Giving release bots broad admin permissions.**
7. **Storing secrets in repo, logs, artifacts, or release notes.**
8. **Skipping tests because the release is “urgent.”**
9. **Not retaining build logs or release evidence.**
10. **Letting untrusted pull requests access privileged release secrets.**

# 9. Artifact, package, and container best practices

## Artifacts should be immutable and verifiable

A release artifact is the thing users or production actually consume. The Git tag alone is not enough.

For each artifact, you should know:

* Version.
* Commit SHA.
* Build job.
* Build time.
* Source repo.
* Builder identity.
* Dependencies.
* Checksum.
* Signature.
* SBOM.
* Provenance.

GitHub supports immutable releases, which lock release assets and the Git tag after publication, preventing modifications to assets or tags after release. GitHub also supports artifact attestations that provide signed provenance claims such as the workflow, repository, environment, and commit SHA associated with an artifact. ([GitHub Docs][17])

## Best practices

1. **Attach checksums**

   * `SHA256SUMS`
   * `SHA256SUMS.sig`

2. **Sign artifacts**

   * GPG.
   * Sigstore/cosign.
   * Platform-native signing.
   * Code-signing certificates where required.

3. **Publish SBOMs**

   * SPDX or CycloneDX are common formats.
   * OWASP recommends generating SBOMs during build, publishing machine-readable SBOMs per release, signing SBOMs/artifacts, and storing them for audit and incident response. ([OWASP Cheat Sheet Series][18])

4. **Publish provenance**

   * SLSA-style provenance.
   * GitHub artifact attestations.
   * in-toto/Sigstore where appropriate.

5. **Use immutable package registries where possible**

   * Prevent overwrites.
   * Prevent silent replacement.

6. **Use container digests**

   * Good: `myapp@sha256:...`
   * Useful: `myapp:1.2.3`
   * Dangerous alone: `myapp:latest`

7. **Do not mutate `latest`**

   * It can exist as a convenience pointer.
   * It must not be the source of truth.

8. **Scan artifacts after build**

   * Scan what you actually ship, not only source code.

9. **Keep artifacts available**

   * Do not rely on ephemeral CI output as the only copy.

10. **Record artifact-to-source mapping**

* Every artifact should map back to repo, commit, tag, workflow, and version.

OpenSSF’s SLSA framework is designed to improve software supply-chain integrity and reduce tampering risk, while OpenSSF baseline guidance specifically calls for signed releases or signed manifests with hashes for release assets. ([SLSA][19])

## Inexcusable artifact mistakes

1. **Replacing a release binary without changing the version.**
2. **Publishing artifacts with no checksum.**
3. **Publishing unsigned artifacts for security-sensitive software.**
4. **Publishing packages that do not correspond to the Git tag.**
5. **Using `latest` as the only deployment artifact reference.**
6. **Rebuilding in production instead of promoting the tested artifact.**
7. **Shipping artifacts with unknown dependencies.**
8. **No SBOM for enterprise, public, regulated, or security-sensitive releases.**
9. **No provenance for critical software.**
10. **Deleting old artifacts needed for rollback or audit.**

# 10. Security-release best practices

Security releases need a stricter process than normal feature releases.

## Best practices

1. **Have a private vulnerability intake path**

   * Security policy.
   * Private reporting.
   * Security contact.
   * Coordinated disclosure process.

2. **Use private advisories when needed**

   * Prepare fix before public disclosure.
   * Coordinate with downstream consumers.

3. **Avoid leaking vulnerability details early**

   * Branch names, PR titles, commit messages, and CI logs can leak.
   * Avoid `fix-critical-auth-bypass` in public before disclosure.

4. **Patch supported versions**

   * Mainline.
   * LTS.
   * Recently supported release branches.
   * Customer-specific supported versions, if applicable.

5. **Issue clear upgrade guidance**

   * Affected versions.
   * Fixed versions.
   * Workarounds.
   * Severity.
   * CVE, if applicable.

6. **Sign and verify security releases**

   * Tags.
   * Commits.
   * Artifacts.
   * SBOMs.
   * Advisories.

7. **Rotate secrets if exposure occurred**

   * Do not merely remove the secret from Git.
   * Consider it compromised.

8. **Audit release permissions**

   * Especially after an incident.

9. **Monitor adoption**

   * Know whether users/customers upgraded.

10. **Preserve evidence**

* Advisory timeline.
* Fix commits.
* release artifacts.
* approvals.
* communications.

GitHub’s release documentation explicitly notes that when a release fixes a security vulnerability, teams should publish a security advisory. ([GitHub Docs][2])

## Inexcusable security-release mistakes

1. **Publicly exposing a vulnerability before a fix is ready, without a disclosure plan.**
2. **Silently fixing a serious vulnerability with no advisory or upgrade guidance.**
3. **Only patching `main` while supported release branches remain vulnerable.**
4. **Committing secrets and merely deleting them without rotation.**
5. **Publishing security artifacts without signatures or checksums.**
6. **Letting untrusted CI jobs access production secrets.**
7. **Using broad, long-lived credentials in release pipelines.**
8. **Not knowing which released versions are affected.**
9. **No SBOM or dependency inventory when responding to a supply-chain issue.**
10. **No audit trail for who approved and published the fix.**

# 11. Deployment and environment best practices

## Keep code, config, and deployment traceable

A release is not finished when the tag is pushed. It is finished when the right artifact is safely deployed, observed, and either accepted or rolled back.

## Best practices

1. **Promote the same artifact through environments**

   * Dev.
   * Test.
   * Staging.
   * Production.

2. **Protect production environments**

   * Require approvals.
   * Restrict who can deploy.
   * Use deployment logs.
   * Use environment-specific secrets.

GitLab protected environments can restrict who deploys and can require approvals for deployment jobs. GitHub branch protection and deployment controls can also require checks and approvals before changes reach protected targets. ([GitLab Docs][20])

3. **Use environment-specific config outside the artifact**

   * The binary/container should be the same.
   * Config changes should be versioned, reviewed, and auditable.

4. **Do not manually patch production**

   * Emergency manual changes must be backported into Git immediately.

5. **Use progressive delivery where appropriate**

   * Canary.
   * Blue/green.
   * Rolling deploy.
   * Feature flags.
   * Ring deployments.

6. **Have smoke tests**

   * Health check.
   * Login/auth.
   * Core transaction.
   * API compatibility.
   * Migration verification.

7. **Have observability ready**

   * Logs.
   * Metrics.
   * Traces.
   * Error rates.
   * SLOs.
   * Dashboards.
   * Alerts.

8. **Define rollback and roll-forward**

   * Know when rollback is safe.
   * Know when database migrations prevent rollback.
   * Know how to disable a feature flag quickly.

9. **Record deployment metadata**

   * Who approved.
   * What version.
   * What commit.
   * What artifact digest.
   * What environment.
   * Start/end time.
   * Result.

10. **Separate release from deployment when useful**

* Releasing an artifact and deploying it to production are not always the same event.
* This distinction is crucial for libraries, packages, mobile apps, SaaS, and regulated environments.

## Inexcusable deployment mistakes

1. **Deploying a different artifact than the one tested.**
2. **No rollback or roll-forward plan.**
3. **No smoke tests after production deployment.**
4. **No monitoring during or after release.**
5. **Manual production changes outside Git and CI/CD.**
6. **Production deploys using personal credentials.**
7. **No approval trail for production releases.**
8. **No way to know what version is running.**
9. **No backup/restore plan before risky migrations.**
10. **Deploying on Friday afternoon with no coverage, no monitoring, and no rollback path.**

# 12. Database and migration best practices

Database changes are often where releases fail. Treat migrations as first-class release artifacts.

## Best practices

1. **Use backward-compatible migrations**

   * Add columns before using them.
   * Dual-write if needed.
   * Backfill safely.
   * Read from new structure only after data is ready.
   * Remove old structure in a later release.

2. **Separate schema deployment from application behavior**

   * Especially for large systems.

3. **Make migrations idempotent where possible**

   * Re-running should not destroy data.

4. **Test migrations on realistic data**

   * Size matters.
   * Index creation, locks, and backfills can behave differently at production scale.

5. **Have rollback strategy**

   * Some migrations cannot be rolled back safely.
   * Document that explicitly.

6. **Back up before destructive changes**

   * Verify restore, not just backup.

7. **Use expand-and-contract**

   * Release 1: expand schema.
   * Release 2: migrate usage.
   * Release 3: contract/remove old schema.

8. **Avoid long locks**

   * Use online migration tools where necessary.

9. **Version migrations**

   * Migrations should map to release versions or commits.

10. **Coordinate app and DB compatibility**

* New app with old DB?
* Old app with new DB?
* Rolling deploy compatibility?

## Inexcusable migration mistakes

1. **Dropping data without backup and approval.**
2. **Assuming rollback is possible when it is not.**
3. **Untested production-scale migration.**
4. **Long locking migration during peak traffic.**
5. **Deploying app code that requires a schema not yet deployed.**
6. **No migration ownership.**
7. **Manual DB changes not represented in Git.**
8. **No verification after migration.**
9. **No plan for partial migration failure.**
10. **Mixing risky schema changes with unrelated feature releases.**

# 13. Hotfix and patch-release best practices

Hotfixes are where process discipline matters most because everyone is under pressure.

## Best practices

1. **Start from the affected production version**

   * Identify the tag or release branch.

2. **Reproduce the bug**

   * Add regression test if possible.

3. **Fix on trunk first when practical**

   * Then cherry-pick/backport to release branch.
   * This prevents the fix from being lost.

4. **Keep hotfix scope tiny**

   * Fix the incident.
   * Avoid opportunistic cleanup.

5. **Use a new patch version**

   * `1.2.3` → `1.2.4`

6. **Create release notes**

   * Even for small hotfixes.

7. **Run required CI**

   * Do not skip tests just because it is urgent.

8. **Tag and sign**

   * Same discipline as normal releases.

9. **Merge back**

   * Ensure main/trunk and supported release branches contain the fix.

10. **Post-incident review**

* What failed?
* What test was missing?
* What release control should improve?

## Inexcusable hotfix mistakes

1. **Patching production manually and never committing the fix.**
2. **Skipping version bump.**
3. **Skipping CI.**
4. **Skipping release notes.**
5. **Forgetting to merge the hotfix back to trunk.**
6. **Bundling unrelated changes into an emergency hotfix.**
7. **Force-moving the old release tag instead of creating a new patch release.**
8. **Not adding a regression test.**
9. **Not documenting affected versions.**
10. **Leaving customers unable to identify whether they have the fix.**

# 15. Monorepo release best practices

Monorepos need extra clarity because not every change releases every component.

## Best practices

1. **Choose a versioning model**

   * Fixed version: all packages share one version.
   * Independent version: each package versions separately.
   * Hybrid: groups of packages version together.

2. **Use clear tag names**

   * Fixed version: `v1.2.3`
   * Independent: `api-v1.2.3`, `web-v3.4.1`, `sdk-js-v2.0.0`

3. **Generate component-specific changelogs**

   * Users of `package-a` should not read irrelevant changes for `package-z`.

4. **Track dependency graph**

   * If package A changes, know whether package B must release.

5. **Avoid accidental global releases**

   * Do not bump everything just because one package changed, unless using fixed versioning intentionally.

6. **Use release tooling**

   * Changesets.
   * semantic-release.
   * Lerna/Nx/Turborepo-style tooling where appropriate.
   * Custom tooling if necessary, but document it.

7. **Protect shared infrastructure**

   * Build scripts.
   * Dependency lockfiles.
   * CI templates.
   * Release tooling.

8. **Use ownership boundaries**

   * CODEOWNERS per package or directory.

9. **Test affected packages**

   * Plus integration tests for dependents.

10. **Publish in dependency order**

* Internal libraries before consumers.

## Inexcusable monorepo mistakes

1. **No clear answer to “what version of this component shipped?”**
2. **Tag names that do not identify the component.**
3. **Publishing packages in the wrong dependency order.**
4. **Global version bumps with no rationale.**
5. **Component changelogs polluted with unrelated changes.**
6. **No ownership boundaries.**
7. **Breaking internal dependents unknowingly.**
8. **No affected-test strategy.**
9. **Inconsistent package versions and Git tags.**
10. **Release tooling only one person understands.**

# 18. The “gold standard” release flow

A strong release process looks like this:

1. **Work lands through reviewed PRs**

   * Small changes.
   * Required checks.
   * Required reviews.
   * CODEOWNERS for sensitive areas.

2. **Main branch remains releasable**

   * CI green.
   * Feature flags for incomplete work.
   * No hidden integration branch.

3. **Release candidate is created**

   * Version bump.
   * Changelog finalized.
   * Migration docs drafted.
   * Known issues reviewed.

4. **Release branch is created only if needed**

   * `release/1.4`
   * Bug fixes only.
   * No new features.

5. **CI validates the release candidate**

   * Build.
   * tests.
   * security scans.
   * license checks.
   * compatibility checks.
   * migration tests.

6. **Annotated signed tag is created**

   * `v1.4.0`
   * Points to exact release commit.

7. **CI builds from the tag**

   * Clean environment.
   * Locked dependencies.
   * No laptop builds.

8. **Artifacts are produced once**

   * Packages.
   * containers.
   * binaries.
   * docs.
   * charts.
   * installers.

9. **Artifacts are secured**

   * Checksums.
   * signatures.
   * SBOM.
   * provenance/attestations.

10. **Draft release is prepared**

* Attach assets.
* Review notes.
* Verify links.
* Verify checksums.
* Verify docs.

11. **Release is published immutably**

* Tag locked.
* assets locked.
* package immutable.

12. **Same artifact is promoted**

* Staging.
* canary.
* production.

13. **Deployment is observed**

* Smoke tests.
* metrics.
* logs.
* alerts.
* SLOs.

14. **Post-release record is completed**

* Version deployed.
* environment.
* approver.
* result.
* rollback status.
* incident notes, if any.

# 20. Minimum enforceable release policy

This is the compact policy I would actually enforce in a serious repo:

1. **All production-bound code must enter through PR/MR.**
2. **`main`, `release/*`, and release tags must be protected.**
3. **Required CI checks must pass before merge.**
4. **Release tags must be annotated and preferably signed.**
5. **Published release tags and artifacts must never be moved or replaced.**
6. **Each release must have a unique version.**
7. **SemVer or another documented versioning scheme must be followed.**
8. **Every user-visible release must have release notes.**
9. **Breaking changes require explicit migration instructions.**
10. **CI must build release artifacts from the release tag.**
11. **The same artifact must be promoted across environments.**
12. **Artifacts must have checksums; important artifacts must be signed.**
13. **Security-sensitive releases must include SBOM and provenance.**
14. **Production deployment must use protected environments and auditable approval.**
15. **Hotfixes must use new versions, not moved tags.**
16. **Hotfixes must be merged or backported so trunk and supported branches stay consistent.**
17. **Secrets must never be committed; exposed secrets must be rotated.**
18. **Rollback or roll-forward instructions must exist before release.**
19. **Release evidence must be retained.**
20. **Release failures must trigger process improvement, not blame-only cleanup.**

# 21. The practical hierarchy

If you want the most important items first, prioritize in this order:

1. **Protect `main`, release branches, and release tags.**
2. **Use unique immutable versions.**
3. **Use annotated/signed release tags.**
4. **Build from clean CI, not laptops.**
5. **Build once, promote the same artifact.**
6. **Never move published release tags.**
7. **Never overwrite published artifacts.**
8. **Require CI and code review.**
9. **Maintain real release notes and changelog.**
10. **Document breaking changes and migrations.**
11. **Use checksums, signatures, SBOMs, and provenance according to risk.**
12. **Have rollback, hotfix, and security-release procedures.**
13. **Track DORA-style delivery and reliability metrics.**
14. **Continuously remove manual, fragile release steps.**

The healthiest release systems are boring in the best possible way: the version is obvious, the tag is protected, the artifact is immutable, the notes are clear, the deployment is traceable, and nobody has to ask, “What exactly did we ship?”
