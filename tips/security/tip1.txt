Here is the security review. Jankurai is already impressively strong as an **agent-native governance project**: it has a root agent instruction surface, owner/test maps, generated-zone rules, deterministic proof lanes, audit artifacts, security reporting policy, and CI that runs the Jankurai audit in ratchet mode. The big missing layer is this: **many security controls are documented or hinted at, but not yet enforced as hard Git/GitHub/CI/supply-chain gates with normalized evidence and scoring**.

Scope note: I reviewed the public repository files and GitHub metadata available through GitHub. I could not verify private repository settings such as branch protection/rulesets, GitHub Advanced Security toggles, secret scanning enablement, push protection, org 2FA, or bypass permissions. Those should become explicit audit checks instead of assumptions.

## 1. Current security posture: what is already good

Jankurai’s core thesis is security-positive: “No proof, no merge; no receipt, no trust,” and the README frames the project around repository conformance, local audit evidence, advisory/strict modes, and agent-risk controls such as prompt injection, broad writes, generated drift, and security regressions. ([GitHub][1])

The repository has a real CI workflow that runs the Jankurai audit, produces JSON/Markdown/SARIF/repair-queue outputs, runs a security lane, runs UX QA, builds the paper, and uploads artifacts. That is a strong foundation for evidence-based merge governance. 

It has a `SECURITY.md` with private vulnerability reporting, expected report contents, security proof lanes, `just security-strict`, and AI-agent boundary language that says trusted policy must not be rewritten by untrusted context, generated outputs, repair plans, or model responses. 

It has owner and proof routing. `agent/owner-map.json` maps important paths such as `.github/`, `agent/`, `SECURITY.md`, lockfiles, tools, docs, generated/reference areas, and package manifests to owners.  `agent/test-map.json` maps security-sensitive files such as `.gitleaksignore`, `SECURITY.md`, `.github/`, package manifests, lockfiles, schemas, generated surfaces, and tools to proof commands. 

It has a security policy and security lane, but the current policy only enables `gitleaks`, `cargo audit`, and `npm audit`; only `gitleaks` is policy-required, while `cargo audit` and `npm audit` are advisory in `agent/security-policy.toml`.  The shell lane does run `gitleaks`, `cargo audit`, and `npm audit`, and it treats `syft` and `zizmor` as advisory. 

It has an audit rubric that already recognizes many of the right security/agent risks: secret scanning, dependency review, SBOM/provenance, workflow linting, prompt injection, overbroad agency, agent-tool supply-chain gaps, release-readiness gaps, cost-budget gaps, and human-review evidence gaps. 

## 2. Main verdict: what is missing or under-enforced

The largest gap is **not awareness**. Jankurai clearly knows what good looks like. The gap is **hard enforcement with machine-verifiable evidence**.

The security tool matrix says CI should require `gitleaks`, `cargo audit`, `npm audit`, and `zizmor`, recommends Semgrep/CodeQL and `cargo deny`, and says release should require Syft, Grype/Trivy, OpenSSF Scorecard, and SLSA/Cosign.  But the actual policy and workflow currently enforce only a subset, and several important controls are advisory, absent, or only string-detected.

The current security analyzer gives score credit mostly by detecting the presence of lockfiles or strings such as `gitleaks`, `audit`, `syft`, `slsa`, `cosign`, `actionlint`, and `zizmor`; it does not prove that those tools ran successfully, uploaded code-scanning results, found zero high-risk issues, or met severity thresholds.  The security runner records steps and has fields for `finding_count`, `highest_severity`, and `normalized_decision`, but those fields are not actually populated from tool output yet. 

The CI workflow generates a SARIF file, but I did not see a `github/codeql-action/upload-sarif` step, so the SARIF is uploaded as an artifact rather than becoming GitHub code-scanning evidence.  GitHub’s SARIF upload flow requires a workflow step with appropriate permissions such as `security-events: write`. ([GitHub Docs][2])

I did not find visible repo files for `CODEOWNERS`, Dependabot, CodeQL, dependency-review, OpenSSF Scorecard, release provenance, or a separate hardened security workflow. The `.github` directory listing shows issue templates, instructions, workflows, a PR template, and Copilot instructions; the workflows directory shows only `jankurai.yml`. ([GitHub][3])

## 3. The mandatory security baseline for every agent-first coding project

Every agent-first coding project should be measured against four baselines at once:

First, use **secure software development lifecycle controls**. NIST’s Secure Software Development Framework groups secure development work into preparing the organization, protecting software, producing well-secured software, and responding to vulnerabilities. That maps very cleanly to Jankurai’s “proof, receipts, repair” model. ([NIST Computer Security Resource Center][4])

Second, use **GitHub/Git controls** as hard gates, not tribal knowledge. Branch protection and rulesets can require PR reviews, status checks, conversation resolution, signed commits, linear history, deployment gates, lock/delete restrictions, and bypass limits. ([GitHub Docs][5])

Third, use **supply-chain controls**. GitHub dependency review catches vulnerable dependencies before they are introduced in a PR and can fail checks based on severity and license policy. ([GitHub Docs][6]) CodeQL supports code scanning for JavaScript/TypeScript and Rust public preview, with default setup scanning PRs, default/protected branches, and schedules. ([GitHub Docs][7]) OpenSSF Scorecard measures repository security posture across checks such as branch protection, dependency-update tooling, SAST, token permissions, signed releases, and vulnerabilities. ([GitHub][8])

Fourth, use **agent-specific controls**. OWASP’s LLM and agentic security work highlights risks such as prompt injection, sensitive information disclosure, supply-chain compromise, improper output handling, excessive agency, and agentic application threats. ([OWASP Gen AI Security Project][9]) These are not optional in an agent-first repo; they need first-class policy, tests, and scoring.

Add a `jankurai github verify` command that queries GitHub branch protection/rulesets and emits `target/jankurai/github-policy-evidence.json`.

### C. GitHub Actions least privilege

**Missing or under-enforced:** the current workflow does not declare top-level or job-level `permissions`.  GitHub recommends giving `GITHUB_TOKEN` the least required access, setting default read access, and raising permissions only per job. Actions can access `github.token` even when it is not explicitly passed, so this matters a lot in agent-first repos. ([GitHub Docs][10])

**Implement:**

At the top of `.github/workflows/jankurai.yml`:

```yaml
permissions: read-all

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true
```

Then set job-specific permissions:

```yaml
jobs:
  audit:
    permissions:
      contents: read
      security-events: write

  ux-qa:
    permissions:
      contents: read

  paper:
    permissions:
      contents: read
```

Use `security-events: write` only for SARIF upload jobs. Use `id-token: write` only for OIDC/provenance/Scorecard publishing jobs. OIDC is preferred for cloud credentials because workflows can exchange short-lived tokens instead of storing long-lived secrets. ([GitHub Docs][11])

**Enforce:**

`zizmor` and `actionlint` should be required, not advisory. Fail if any workflow lacks `permissions`, uses broad `write-all`, or gives write permissions to jobs that only read/build/test.

**Audit/scoring:**

```toml
[caps]
workflow_missing_permissions = 70
workflow_write_all_token = 60
unnecessary_write_token = 74
oidc_missing_for_cloud_release = 78
```

### E. Harden checkout and PR execution

**Missing or under-enforced:** `actions/checkout` does not set `persist-credentials: false`.  This should be the default for jobs that do not push.

**Implement:**

    persist-credentials: false

For release/provenance jobs, use `fetch-depth: 0` only where needed. Do not use `pull_request_target` for workflows that checkout or execute untrusted PR code. Do not expose secrets to fork PRs.

```text
SEC-CI-PR-001: fail if pull_request_target runs checkout of head SHA

```

**Audit/scoring:**

```toml

### J. Secret scanning, push protection, and redaction

**Missing or under-enforced:** Jankurai runs `gitleaks detect --redact`, which is great.  But the repo should also require GitHub secret scanning, push protection, transcript/artifact scanning, and exception governance. GitHub push protection can block supported secrets from being pushed through the CLI, web UI, uploads, REST API, and public GitHub MCP server, while repository-level push protection must be enabled by admins. ([GitHub Docs][12]) Secret scanning scans Git history and branches for hardcoded credentials. ([GitHub Docs][13])

The current `.gitleaksignore` contains narrow hash/path/rule ignores, but no owner, reason, or expiration metadata. 

**Implement:**

Add `agent/secret-policy.toml`:

schema_version = "1.0.0"

github_push_protection_required = true
gitleaks_required = true
scan_git_history = true
scan_artifacts = true
scan_agent_transcripts = true
scan_vector_stores = true

[exceptions]
manifest = "docs/exceptions/secrets.toml"
max_days = 30
```

Replace bare `.gitleaksignore` entries with a reviewed exception manifest:

```toml
[[exception]]
id = "SEC-SECRET-0001"
tool = "gitleaks"

```

**Enforce:**

Fail if a secret is detected outside an approved non-expired exception. Fail if `.gitleaksignore` changes without `just security-strict`. Fail if CI logs contain secret-like values, environment dumps, API keys, tokens, cookies, private prompt transcripts, or Playwright traces with credentials.

### K. SBOM, artifact vulnerability scanning, and release provenance

**Missing or under-enforced:** the shell security lane can run Syft as advisory, but the generated SBOM is not included in the workflow artifact list I saw.   The security matrix says release should require Syft, Grype/Trivy, OpenSSF Scorecard, and SLSA/Cosign, but those are not hard release gates. 

SLSA’s build track defines increasing provenance guarantees: Build L1 requires provenance, L2 requires signed provenance from a hosted build platform, and L3 requires a hardened build platform. ([SLSA][14]) The current SLSA specification is maintained as a versioned supply-chain security standard. ([SLSA][15])

**Implement:**

For every release build:

```text
syft . -o spdx-json=target/jankurai/sbom.spdx.json
grype sbom:target/jankurai/sbom.spdx.json --fail-on high
cosign sign-blob --yes --output-signature target/jankurai/release.sig <artifact>
cosign attest --yes --predicate target/jankurai/sbom.spdx.json <artifact>
```

Use SLSA GitHub generator for release provenance where applicable, and verify the provenance before publishing artifacts.

**Enforce:**

Release job must fail if:

```text
SBOM missing
SBOM invalid
artifact has high/critical vulnerability
artifact unsigned
provenance missing
provenance subject digest does not match artifact
release tag not protected
release job not run from protected ref/environment
```

**Audit/scoring:**

```toml
[caps]
no_release_sbom = 82

unsigned_release_artifact = 80
missing_slsa_provenance = 80
provenance_not_verified = 74
```

## 5. Concrete workflow changes I would make first

Here is the hardened shape I would use for the existing `jankurai.yml`.

```yaml
name: jankurai

on:
  pull_request:
  push:
    branches: [main]

permissions: read-all

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  audit:
    runs-on: ubuntu-latest
    timeout-minutes: 30
    permissions:
      contents: read
      security-events: write
    steps:
      - uses: actions/checkout@<sha>
        with:
          persist-credentials: false

      - uses: dtolnay/rust-toolchain@<sha>

      - name: Check audit versions
        run: cargo run -p jankurai -- versions

      - name: Jankurai audit
        run: >
          cargo run -p jankurai -- audit .
          --mode ratchet
          --json agent/repo-score.json
          --md agent/repo-score.md
          --sarif target/jankurai/jankurai.sarif
          --github-step-summary target/jankurai/summary.md
          --repair-queue-jsonl target/jankurai/repair-queue.jsonl

      - name: Upload Jankurai SARIF
        if: always()
        uses: github/codeql-action/upload-sarif@<sha>
        with:
          sarif_file: target/jankurai/jankurai.sarif
          category: jankurai

      - name: Publish Jankurai summary
        run: cat target/jankurai/summary.md >> "$GITHUB_STEP_SUMMARY"

      - name: Security lane strict
        run: cargo run -p jankurai -- security run . --strict --out target/jankurai/security/evidence.json

      - uses: actions/upload-artifact@<sha>
        if: always()
        with:
          name: jankurai-score
          retention-days: 14
          path: |
            agent/repo-score.json
            agent/repo-score.md
            target/jankurai/jankurai.sarif
            target/jankurai/repair-queue.jsonl
            target/jankurai/security/evidence.json
            target/jankurai/security/*.log
            target/jankurai/sbom.spdx.json
```

Then split dedicated workflows for `codeql`, `dependency-review`, `scorecard`, and `release-provenance`. Keeping them separate makes required checks easier to reason about.

## 7. Upgrade the security lane script

Current `tools/security-lane.sh` should become strict by default in CI, and advisory only locally. It should also emit machine-parseable JSON for every tool.

Add required commands:

```bash
run_required "actionlint" "actionlint"
run_required "zizmor" "zizmor --format sarif .github/workflows > target/jankurai/security/zizmor.sarif"
run_required "gitleaks" "gitleaks detect --source . --redact --no-banner --report-format json --report-path target/jankurai/security/gitleaks.json"
run_required "cargo" "cargo audit --json > target/jankurai/security/cargo-audit.json"
run_required "cargo" "cargo deny check --format json > target/jankurai/security/cargo-deny.json"
run_required "npm" "npm audit --json --audit-level=high > target/jankurai/security/npm-audit.json"

```

Also fix the tool-name mismatch: the policy says `cargo audit`, but the script checks `command -v cargo-audit`. It should check `command -v cargo` and then run `cargo audit`, or standardize on the actual executable expectation.  

## 8. Upgrade the Jankurai audit evidence model

The current security runner already has fields for `finding_count`, `highest_severity`, and `normalized_decision`, but they are not populated.  That is the exact place to make Jankurai excellent.

Every tool result should normalize into this shape:

```json
{
  "control_id": "SEC-SCA-001",
  "tool": "cargo-audit",
  "status": "fail",
  "blocking": true,
  "finding_count": 2,
  "highest_severity": "high",
  "normalized_decision": "block",
  "evidence": [
    {
      "path": "target/jankurai/security/cargo-audit.json",
      "package": "example",
      "advisory": "RUSTSEC-...",
      "severity": "high"
    }
  ],
  "owner": "security",
  "agent_fix": "upgrade affected crate or add a dated advisory exception with compensating control"
}
```

Then security score becomes evidence-based, not string-presence-based.

## 9. Proposed 100-point security scoring model

Use this as a new `Security and agent-supply-chain posture` score:

| Area                                | Weight | Required evidence                                                                                           |
| ----------------------------------- | -----: | ----------------------------------------------------------------------------------------------------------- |
| GitHub governance                   |     12 | branch protection/ruleset, CODEOWNERS, required checks, signed commits/tags, no force-push/delete           |
| CI hardening                        |     12 | least-privilege permissions, pinned actions, safe checkout, timeouts, concurrency, no unsafe PR triggers    |
| Secrets and data leakage            |     12 | GitHub secret scanning, push protection, gitleaks, artifact/log/transcript scanning, exception expiry       |
| Dependency and package supply chain |     12 | Dependabot/Renovate, dependency review, cargo audit/deny/vet, npm audit, OSV, license policy                |
| SAST and secure code checks         |     10 | CodeQL/Semgrep, clippy, shellcheck, actionlint, unsafe ledger, input-boundary tests                         |
| Agent/tool/MCP security             |     12 | capability manifest, prompt-injection fixtures, tool pinning, MCP scope review, untrusted-context isolation |
| SBOM/provenance/release signing     |     10 | Syft, Grype/Trivy, SLSA provenance, Cosign signatures, protected release environment                        |
| Privacy/artifact governance         |      8 | retention, redaction, screenshot/trace policy, no raw prompts/secrets/PII in artifacts                      |
| Vulnerability response              |      6 | disclosure SLA, severity model, incident/rotation runbooks, advisory process                                |
| Audit evidence quality              |      6 | normalized tool results, SARIF/code-scanning upload, repair queue, owner/test-map linkage                   |

Suggested gates:

```text
Pass: score >= 90 and no high/critical finding
Warn: score 85-89 and no high/critical finding
Fail: score < 85, any high/critical finding, or any hard cap below threshold
Release: score >= 95, provenance/signing/SBOM complete, no unresolved medium+ release finding
```

## Security Review

- [ ] Dependency changes reviewed; lockfile diff is expected.
- [ ] New dependencies have owner, reason, license, and advisory status.

- [ ] Agent instructions/tool permissions did not broaden unexpectedly.

## 12. Prioritized implementation plan

**P0: next security-hardening PR**

Add CODEOWNERS, Dependabot, dependency-review, workflow `permissions: read-all`, job-level permissions, `persist-credentials: false`, workflow timeouts, concurrency, SARIF upload, actionlint, strict zizmor, and strict security lane. This closes the biggest GitHub/CI gaps fast.

**P1: supply-chain and SAST PR**

Add CodeQL, cargo-deny, OSV Scanner, SBOM upload, Grype/Trivy scanning, Scorecard, dependency exception manifests, and license policy.

**P2: agent-specific security PR**

Add `agent/capabilities.toml`, `agent/tool-supply-chain.toml`, prompt-injection fixtures, MCP/tool scope review, transcript/artifact redaction policy, and untrusted-context conformance tests.

**P3: release/provenance PR**

Add SLSA provenance, Cosign signing/verification, protected release environment, signed tags, SBOM attestation, checksums, and release gate scoring.

**P4: scoring engine PR**

Change security scoring from string-presence detection to normalized evidence parsing. Populate `finding_count`, `highest_severity`, and `normalized_decision` from tool outputs. Fail based on policy thresholds, not whether tool names appear in files.

## 13. The north-star rule

For an agent-first coding project, the standard should be:

**No agent can change code, policy, dependencies, workflows, release artifacts, secrets, generated zones, or tool permissions unless the repo can prove who owns the change, what lane allowed it, what tools ran, what they found, what artifact records it, and which human or policy accepted the residual risk.**

Jankurai is very close to being the right system for that. The next leap is turning its excellent written standard into hard GitHub-native gates, strict security tooling, normalized evidence, and score caps that make security regressions impossible to merge quietly.
