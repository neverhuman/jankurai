## Full security addendum for Jankurai and every agent-first coding project

I’m treating “missing” broadly here: absent from the visible repository, documented but not enforced, advisory instead of blocking, or present in prose but not yet normalized into audit evidence and scoring. I could inspect the public repo files, but I cannot verify GitHub-side settings that live outside the repository, such as branch protection, push protection, private vulnerability reporting enablement, environments, rulesets, or organization policies. Those should become explicit audit inputs.

The big conclusion: **Jankurai already has the right philosophy**: “no proof, no merge,” local audit artifacts, proof lanes, generated-zone controls, agent boundaries, ratchets, and a security lane. The missing layer is turning the best security ideas into **hard, machine-verifiable gates** with normalized evidence, GitHub settings evidence, score caps, exception expiry, and release provenance. The repo’s README describes Jankurai as a local audit CLI for auditable AI-assisted merge, with ownership maps, proof lanes, generated zones, boundaries, rolling scores, merge witnesses, and repair queues. It also already calls out AI-agent risks such as generated-zone drift, prompt-injection via root agent instructions, security runs, and deterministic reviewable agent guidance. ([GitHub][1])

Jankurai also already has a `SECURITY.md` with private vulnerability reporting guidance, evidence expectations, strict security proof lanes, and a statement that agent boundaries should not be rewritten by untrusted context. ([GitHub][2]) The tool matrix already names many of the right scanners: Gitleaks, cargo audit, npm audit, Zizmor, Semgrep, CodeQL, cargo-deny, Syft, Grype/Trivy, OpenSSF Scorecard, SLSA, and Cosign. ([GitHub][3]) The agent-native standard is also strong: it includes rule IDs for secret sprawl, prompt injection, overbroad agency, supply-chain drift, CI hardening, authorization isolation, input boundaries, agent tool supply, release readiness, cost budget, and human review evidence. ([GitHub][4])

The highest-priority security issue is not that Jankurai lacks security thinking. It is that several controls are still **not strict enough, not fully implemented in CI, not normalized into findings, or not auditable from Git alone**.

# 1. Highest-priority gaps to fix first

## 1. CI security lane is not strict by default

The visible workflow runs `just security-lane` in CI, but it does not appear to force strict mode for every PR and push. The script has strict behavior controlled by `JANKURAI_SECURITY_STRICT`, and the policy file marks only Gitleaks as required while dependency tools are advisory. That means a project can appear security-aware while still letting missing tools or advisory dependency results pass. ([GitHub][5])

**Fix:** CI must run:

```bash
JANKURAI_SECURITY_STRICT=1 just security-lane
```

and the Jankurai CLI should fail if the project is in CI and strict mode is not enabled.

**Audit rule:** `SECURITY_CI_STRICT_REQUIRED`

**Scoring:** cap score at **70** if CI security lane is not strict; cap at **50** if missing tools are silently skipped.

---

## 2. Security policy and security-lane script disagree on tool names

The policy uses names such as `cargo audit` and `npm audit`, while the script appears to emit or check tool names such as `cargo-audit` and `npm`. That can cause required/advisory classification drift.

**Fix:** create canonical tool IDs:

```toml
[tools.gitleaks]
id = "gitleaks"
command = "gitleaks detect --redact --no-git"

[tools.cargo_audit]
id = "cargo-audit"
command = "cargo audit --json"

[tools.npm_audit]
id = "npm-audit"
command = "npm audit --json"
```

**Audit rule:** `SECURITY_TOOL_ID_CANONICAL`

**Scoring:** cap at **80** if policy and executed evidence use mismatched tool identifiers.

---

## 3. Scanner output is not normalized into enforceable findings

The Rust evidence model already has fields such as `finding_count`, `highest_severity`, and `normalized_decision`, but the current enrichment path sets them to `None` rather than parsing scanner outputs into normalized decisions. ([GitHub][6])

That is a major gap. A security lane should not only know that a command ran; it should know:

```json
{
  "tool": "cargo-audit",
  "version": "0.x",
  "finding_count": 2,
  "highest_severity": "high",
  "decision": "block",
  "findings": [
    {
      "id": "RUSTSEC-...",
      "severity": "high",
      "package": "example",
      "fixed_version": "1.2.3",
      "source": "cargo-audit"
    }
  ]
}
```

**Fix:** normalize all scanners into one finding schema.

**Audit rule:** `SECURITY_FINDINGS_NORMALIZED`

**Scoring:** cap at **80** if tools run but findings are not machine-readable; cap at **60** if severity thresholds are not enforceable.

---

## 4. GitHub Actions permissions are not explicitly minimized

The visible workflow does not show an explicit top-level `permissions:` block. GitHub’s own hardening guidance recommends setting the `GITHUB_TOKEN` to the minimum permissions required, with read-only contents by default where possible. ([GitHub Docs][7])

**Fix:** every workflow must declare permissions at the top level and narrow them per job.

```yaml
permissions:
  contents: read
```

Only jobs that truly need more should request it:

```yaml
permissions:
  contents: read
  security-events: write   # SARIF upload only
```

```yaml
permissions:
  contents: read
  id-token: write
  attestations: write      # release provenance only
```

**Audit rule:** `GITHUB_ACTIONS_MINIMUM_PERMISSIONS`

**Scoring:** cap at **70** if a workflow omits `permissions`; cap at **50** if broad write permissions are used without justification.

---

## 5. GitHub Actions are version-tagged, not pinned to immutable SHAs

The visible workflow uses actions such as `actions/checkout@v4`, `actions/setup-node@v4`, and `actions/upload-artifact@v4`. Tags are mutable references. For high-trust agent-first projects, third-party actions should be pinned to full commit SHAs, then updated by Dependabot or Renovate with review. ([GitHub][5])

**Fix:**

```yaml
- uses: actions/checkout@<full-commit-sha>
```

Also add a policy that only approved action owners are allowed.

**Audit rule:** `GITHUB_ACTIONS_PINNED_TO_SHA`

**Scoring:** cap at **72** if any third-party action is not pinned; cap at **60** if release or deployment workflows use unpinned actions.

---

## 6. No visible CodeQL hard gate

The security matrix recommends CodeQL, but the visible workflow does not show CodeQL as a required PR or protected-branch gate. GitHub CodeQL can be configured to scan code on default and protected branches and PRs. ([GitHub Docs][8])

**Fix:** add CodeQL default or advanced setup and require it in branch protection.

**Audit rule:** `SAST_CODEQL_REQUIRED`

**Scoring:** cap at **78** if no SAST is required; cap at **65** if SAST exists but is advisory only.

---

## 7. No dependency-review gate for PRs

The repo has dependency scanning ideas, but a PR-level dependency review gate should block vulnerable or policy-violating dependency changes before merge. GitHub’s dependency-review action is specifically designed to scan dependency changes in pull requests and fail when vulnerabilities or invalid licenses are introduced. ([GitHub][9])

**Fix:** add a dedicated PR-only dependency-review workflow.

**Audit rule:** `DEPENDENCY_REVIEW_PR_GATE`

**Scoring:** cap at **80** if dependency scanning exists only after merge; cap at **70** if dependency changes are not PR-gated.

---

## 8. No visible secret-scanning push-protection evidence

The repo uses Gitleaks locally and in its security lane, which is good, but GitHub-side secret scanning and push protection should also be enabled and audited. GitHub secret scanning push protection blocks pushes that contain supported secrets. ([GitHub Docs][10])

**Fix:** add repo-settings evidence proving:

```json
{
  "secret_scanning": "enabled",
  "secret_scanning_push_protection": "enabled",
  "valid_until": "2026-06-04"
}
```

**Audit rule:** `SECRET_PUSH_PROTECTION_ENABLED`

**Scoring:** cap at **75** if no push-protection evidence exists; hard block if a real secret is detected.

---

## 9. Release provenance is documented but not enforced

The matrix names Syft, Grype/Trivy, OpenSSF Scorecard, SLSA, and Cosign for release, but visible CI does not yet show a fully enforced release-provenance lane. GitHub artifact attestations can create provenance and integrity guarantees for artifacts, and GitHub documents their use for improving supply-chain security. ([GitHub Docs][11])

**Fix:** release jobs must produce:

1. SBOM.
2. Vulnerability scan.
3. Signed artifact.
4. Artifact attestation.
5. Provenance verification step.
6. Release score threshold.

**Audit rule:** `RELEASE_PROVENANCE_REQUIRED`

**Scoring:** block release if missing; cap repository security score at **75** if release artifacts are unsigned or unattested.

---

## 10. Repo settings are not first-class evidence

Several crucial controls are not visible in Git files:

* branch protection,
* required reviews,
* CODEOWNERS enforcement,
* signed commits,
* signed tags,
* required status checks,
* rulesets,
* secret scanning,
* private vulnerability reporting,
* environment reviewers,
* deployment protection rules.

**Fix:** add a `jankurai repo-settings snapshot` command that queries GitHub and writes a redacted evidence artifact.

**Audit rule:** `REPO_SETTINGS_EVIDENCE_REQUIRED`

**Scoring:** cap at **70** if repo settings are unknown.

# 2. What should become mandatory for every agent-first coding project

An agent-first project has extra risk because the “developer” may be an autonomous or semi-autonomous system that can edit files, call tools, follow prompt context, run commands, propose dependencies, generate migrations, alter CI, and touch deployment paths. OWASP’s LLM Top 10 calls out prompt injection as a risk that can cause unauthorized actions, data exposure, or compromised decisions, and OWASP’s agentic work treats skills/tools as the execution layer that grants access to resources and workflows. ([OWASP Foundation][12])

Therefore, every agent-first coding project should make the following mandatory:

1. **No proof, no merge.**
2. **No normalized security evidence, no score.**
3. **No branch protection evidence, no trust.**
4. **No strict security lane in CI, no merge.**
5. **No secret scanning and push protection, no merge.**
6. **No dependency review, no merge for dependency-changing PRs.**
7. **No SAST, no merge.**
8. **No action pinning and least-privilege workflow permissions, no merge.**
9. **No CODEOWNERS and sensitive-path review, no merge for sensitive paths.**
10. **No agent tool allowlist, no autonomous tool execution.**
11. **No prompt-injection regression tests, no agentic feature release.**
12. **No SBOM, vulnerability scan, signing, and attestation, no release.**
13. **No expiring exceptions, no waivers.**
14. **No evidence freshness, no score.**
15. **No human approval for high-risk changes, no merge.**

NIST’s Secure Software Development Framework is a good anchor for this because it focuses on reducing vulnerabilities, reducing impact, and addressing root causes across the software lifecycle. ([NIST Computer Security Resource Center][13])

OpenAI’s current function-calling documentation emphasizes that Structured Outputs with strict schemas constrain function-call arguments to the supplied JSON Schema, which is the right pattern for agent tool boundaries. ([OpenAI Help Center][15])

AWS’s agentic AI security guidance also emphasizes that user inputs are a primary attack vector and recommends prompt validation suites, logging with metrics, and layered sanitization. ([AWS Documentation][16])

# 5. Secure GitHub Actions baseline

Every workflow should start from least privilege:

```yaml
name: security

on:
  pull_request:
  push:
    branches: [main]

permissions:
  contents: read

concurrency:
  group: security-${{ github.ref }}
  cancel-in-progress: true

jobs:
  security:
    runs-on: ubuntu-latest
    timeout-minutes: 30

    steps:
      - name: Checkout
        uses: actions/checkout@<full-commit-sha>
        with:
          persist-credentials: false

      - name: Run security lane strictly
        env:
          JANKURAI_SECURITY_STRICT: "1"
          CI: "true"
        run: |
          set -euo pipefail
          just security-lane

      - name: Run audit
        run: |
          set -euo pipefail
          cargo run -p jankurai -- audit --strict
```

For SARIF upload:

```yaml
permissions:
  contents: read
  security-events: write
```

For release provenance only:

```yaml
permissions:
  contents: read
  id-token: write
  attestations: write
```

For package publishing only:

```yaml
permissions:
  contents: read
  id-token: write
  packages: write
```

Never give the whole workflow broad write permissions just because one release job needs them.

# 7. Security evidence schema that Jankurai should require

Jankurai’s current evidence model has the right direction, but it needs to become stricter. The normalized evidence file should contain:

```json
{
  "schema_version": "2.0.0",
  "repo": "owner/name",
  "commit": "abc123",
  "tree_hash": "sha256:...",
  "policy_hash": "sha256:...",
  "generated_at": "2026-05-05T00:00:00Z",
  "expires_at": "2026-05-19T00:00:00Z",
  "strict": true,
  "environment": {
    "ci": true,
    "runner": "github-hosted",
    "os": "ubuntu-latest"
  },
  "steps": [
    {
      "tool": "gitleaks",
      "tool_version": "x.y.z",
      "command_hash": "sha256:...",
      "input_hash": "sha256:...",
      "output_hash": "sha256:...",
      "exit_code": 0,
      "duration_ms": 1000,
      "finding_count": 0,
      "highest_severity": "none",
      "decision": "pass",
      "raw_output_path": ".jankurai/security/gitleaks.json",
      "sarif_path": ".jankurai/security/gitleaks.sarif"
    }
  ],
  "findings": [],
  "exceptions_applied": [],
  "redaction_scan": {
    "passed": true,
    "tool": "gitleaks"
  },
  "decision": "pass"
}
```

Hard requirements:

* `strict` must be true in CI.
* Every required tool must have a step.
* Every step must include tool version.
* Every scanner must produce parseable output.
* Every finding must have severity, rule ID, source, location, and decision.
* Every exception must have owner, reason, expiry, and finding ID.
* Evidence containing secrets must be rejected.

# 8. Proposed security scoring model

Use a 100-point model with hard gates and caps. Do not let a repo “average out” a catastrophic issue.

## Category weights

| Category                                 |  Points |
| ---------------------------------------- | ------: |
| Git/repo governance                      |      15 |
| CI/CD hardening                          |      15 |
| Secrets and sensitive data               |      12 |
| Dependency and supply chain              |      14 |
| SAST and secure code                     |      12 |
| Agent-first runtime/tool/prompt security |      14 |
| Evidence quality and audit integrity     |       8 |
| Release and incident readiness           |      10 |
| **Total**                                | **100** |

## Hard blocking conditions

These should produce **no merge** regardless of numeric score:

| Finding                                                                 | Result        |
| ----------------------------------------------------------------------- | ------------- |
| Real secret committed or present in artifact/log                        | Block         |
| Critical/high unexceptioned vulnerability                               | Block         |
| Security lane missing on PR                                             | Block         |
| Security lane not strict in CI                                          | Block         |
| Missing required scanner in strict CI                                   | Block         |
| Unsafe `pull_request_target` with untrusted checkout or secret exposure | Block         |
| Release artifact unsigned/unattested                                    | Block release |
| Protected branch disabled or bypassed for merge                         | Block         |
| Agent changed security/CI/release policy without owner review           | Block         |
| Expired exception for high/critical issue                               | Block         |
| Evidence tampering or missing commit/tree hash                          | Block         |
| Agent had raw secret value in context                                   | Block         |
| Autonomous deployment without protected environment approval            | Block         |

## Score caps

| Condition                                        | Maximum score |
| ------------------------------------------------ | ------------: |
| Default branch protection unknown                |            70 |
| Default branch unprotected                       |            50 |
| No explicit workflow permissions                 |            70 |
| Broad workflow write token without justification |            50 |
| Unpinned third-party actions                     |            72 |
| Unpinned release/deploy action                   |            60 |
| No strict security lane in CI                    |            70 |
| Required tools skipped when missing              |            50 |
| No normalized security findings                  |            80 |
| No CodeQL/Semgrep-equivalent SAST                |            78 |
| No dependency review PR gate                     |            80 |
| No secret scanning push-protection evidence      |            75 |
| No CODEOWNERS for sensitive paths                |            78 |
| No agent tool allowlist                          |            60 |
| No prompt-injection regression suite             |            78 |
| No SBOM for release                              |            75 |
| No artifact signing/attestation for release      |            70 |
| No repo-settings evidence                        |            70 |
| No exception expiry enforcement                  |            80 |
| No incident response/secret rotation plan        |            85 |

## Finding penalties

After caps are applied:

```text
critical = block and -25
high     = block unless exception and -12
medium   = -5
low      = -2
info     = -0.5
```

Positive credit should require fresh evidence, not prose.

## Evidence freshness

| Evidence type          |                                                     Maximum age |
| ---------------------- | --------------------------------------------------------------: |
| Fast test lane         |                                                          7 days |
| Security lane          |                                                         14 days |
| Repo settings snapshot |                                                         30 days |
| SBOM/release scan      |                                                     per release |
| Threat model           |                                90 days, or any high-risk change |
| Agent tool policy      |                              30 days, or any tool/policy change |
| Exceptions             | must have explicit expiry, max 30–90 days depending on severity |

# 10. What to change in the current Jankurai code

## `tools/security-lane.sh`

Change behavior from “skip missing required tool unless strict” to:

* local non-strict: warn and mark advisory,
* CI: always strict,
* release: stricter than strict,
* missing required tool in CI: hard fail,
* every tool emits JSON or SARIF,
* every raw output is saved and hashed,
* every scanner has a parser.

Required behavior:

```bash
if [ "${CI:-}" = "true" ] && [ "${JANKURAI_SECURITY_STRICT:-}" != "1" ]; then
  echo "security lane must run strict in CI" >&2
  exit 2
fi
```

Also require canonical names:

```bash
tool_id="cargo-audit"
```

not mixed names like `cargo audit`, `cargo-audit`, and `npm`.

---

## `crates/jankurai/src/commands/security.rs`

Implement:

1. Tool version detection.
2. JSON/SARIF parser registry.
3. Normalized severity mapping.
4. Required/advisory classification after canonicalization.
5. Evidence expiry.
6. Redaction scan.
7. CI strict detection.
8. Policy hash and tree hash.
9. Finding exceptions.
10. Score caps.

The existing optional fields should become required in strict mode:

```rust
finding_count: u64,
highest_severity: Severity,
normalized_decision: Decision,
```

In non-strict local mode they can be nullable, but strict CI should reject nulls.

---

## `crates/jankurai/src/audit/security_artifact.rs`

The audit summarizer should not only summarize command status. It should enforce:

```text
- all required tools ran
- all required tools produced parseable output
- no high/critical findings without valid exception
- strict mode was enabled in CI
- evidence is fresh
- evidence commit matches current commit
- evidence policy hash matches current policy
- evidence has passed redaction scan
```

The current summarization path records useful envelope data such as exit codes, elapsed time, strict flag, commands ran, failed/skipped commands, and timestamps, but it needs to become a true security decision engine. ([GitHub][17])


Every agent-first repo should have a threat model with at least these sections:

```text
1. Assets
   - source code
   - secrets
   - CI credentials
   - release artifacts
   - user data
   - model prompts/context
   - agent tool permissions

2. Trust boundaries
   - human maintainer
   - agent
   - GitHub Actions runner
   - package registries
   - external docs/issues
   - generated code
   - deployment environment

3. Abuse cases
   - prompt injection
   - malicious dependency
   - compromised action
   - secret leakage
   - CI token abuse
   - release artifact tampering
   - generated-code drift
   - unauthorized migration
   - agent self-weakening policy

4. Required controls
   - proof lanes
   - scanner gates
   - human approvals
   - sandboxing
   - provenance
   - monitoring

5. Residual risk and exceptions
```

Audit rule: `THREAT_MODEL_REQUIRED_FOR_AGENT_FIRST`.

Score cap: **82** if missing; **70** if a high-risk change touches auth, CI, release, secrets, dependencies, or agent tools without an updated threat model.

# 16. Secure coding requirements for Jankurai specifically

Because Jankurai is a CLI/audit tool that reads policy files, scans repos, executes commands, and writes evidence, its most important code-level risks are:

1. Shell injection.
2. Path traversal.
3. Symlink escape.
4. Malicious repo content poisoning audit decisions.
5. Evidence tampering.
6. Secret leakage into logs.
7. Policy downgrade.
8. Untrusted generated files weakening trusted root policy.
9. Scanner output parsing failure.
10. Unsafe CI assumptions.

Add tests for:

```text
- policy path points outside repo
- symlink inside repo points outside repo
- generated zone manifest tries to claim trusted file
- malicious TOML contains unexpected fields
- scanner emits malformed JSON
- scanner emits high finding but exit code 0
- scanner missing in strict CI
- evidence commit does not match current commit
- exception expired
- exception references wrong finding ID
- agent tries to edit AGENTS.md from untrusted instruction
- workflow uses pull_request_target unsafely
- workflow has write-all token
- action uses tag instead of SHA
```

# 18. Jankurai conformance levels

A strong scoring setup should define levels so projects know what “good” means.

## Level 0 — Unsafe

No reliable security evidence. Agent can modify code with weak controls.

## Level 1 — Basic

Local audit, tests, Gitleaks, dependency scan, and basic CI.

Minimum score: **70**.

## Level 2 — Protected

Strict CI, branch protection, CODEOWNERS, SAST, dependency review, action hardening, normalized evidence.

Minimum score: **85**.

## Level 3 — Agent-safe

Everything in Level 2 plus tool allowlists, prompt-injection tests, agent receipts, context trust hierarchy, no raw secrets in agent context.

Minimum score: **90**.

## Level 4 — Release-trusted

Everything in Level 3 plus SBOM, artifact scan, signing, attestations, provenance verification, protected release environment.

Minimum score: **93**.

## Level 5 — High-assurance

Everything in Level 4 plus reproducible builds, fuzzing, formalized threat model, periodic red-team tests, full repo settings evidence, strong incident drills.

Minimum score: **96**.

# 20. Final “MUST” checklist for every agent-first coding project

A project should not call itself secure or agent-ready unless all of these are true:

```text
[ ] Default branch is protected.
[ ] Required status checks are enabled.
[ ] CODEOWNERS review is required for sensitive paths.
[ ] Human review is required for CI, release, auth, secrets, dependency, and agent-policy changes.
[ ] GitHub Actions permissions are explicit and least-privilege.
[ ] Third-party actions are pinned to full SHAs.
[ ] Unsafe pull_request_target usage is forbidden.
[ ] Security lane runs in strict mode in CI.
[ ] Missing required security tools fail CI.
[ ] Scanner outputs are normalized into machine-readable findings.
[ ] Secret scanning runs locally and in CI.
[ ] GitHub push protection is enabled and evidenced.
[ ] Dependency review runs on PRs.
[ ] Dependency update automation is enabled.
[ ] License policy is enforced.
[ ] SAST runs on PRs and protected branches.
[ ] Workflow security scanning runs on every workflow change.
[ ] Agent tool permissions are allowlisted.
[ ] Agent tool calls produce receipts.
[ ] Agents cannot access raw secrets.
[ ] Agents cannot autonomously deploy.
[ ] Agents cannot weaken their own policies without owner review.
[ ] Untrusted context is quarantined.
[ ] Prompt-injection regression tests exist.
[ ] Security-sensitive changes require threat-model updates.
[ ] Release artifacts include SBOMs.
[ ] Release artifacts are scanned.
[ ] Release artifacts are signed.
[ ] Release artifacts have provenance attestations.
[ ] Evidence includes commit, tree, policy, tool, input, and output hashes.
[ ] Evidence expires.
[ ] Exceptions expire.
[ ] Security score has hard caps.
[ ] Critical/high findings block merge unless explicitly and temporarily accepted.
[ ] Evidence is redacted before upload.
[ ] Incident response and secret rotation runbooks exist.
```

The strongest version of Jankurai would make these not just recommendations, but **machine-checkable contract rules**: every PR gets a security decision, every decision has evidence, every exception expires, every high-risk agent action has a receipt, and every release can prove where it came from.
