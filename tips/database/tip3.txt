## Executive verdict

Jankurai is already pointed in the right direction: it treats agentic coding as a governed merge system with proof lanes, owner maps, generated zones, security boundaries, score history, repair queues, and local JSON/Markdown receipts. The README’s core rule is exactly the right north star: “no proof, no merge; no receipt, no trust.” It also already names database and migration analysis as part of the source tree and control plane. ([GitHub][1])

The biggest gap is that **database safety and security are still too advisory**. The repo has `db/` placeholders and a destructive-migration scanner, but not a full production-grade database control system. The current audit rubric gives “Data truth and workflow safety” only 8 points, while the security posture dimension is broad and not yet fully backed by enforced tool coverage. For every agent-first coding project, I would promote DB/data/migration safety into a first-class, proof-backed, hard-cap-heavy standard. ([GitHub][2])

---

## What Jankurai already has

Jankurai’s required shape is strong: web has no secrets or direct DB access, API handlers have no raw SQL, domain has no I/O, adapters own PostgreSQL, `db/` owns migrations/constraints/seeds/indexes/RLS, and Python is bounded away from product truth and production DB ownership. That is an excellent agent-first boundary model. ([GitHub][2])

It also has a DB root that says `db/` is reserved for PostgreSQL migrations, constraints, and durable-truth policy; `db/migrations/` should contain versioned schema changes and rollback/backfill notes; and `db/constraints/` should contain named constraints, RLS, and invariant notes. ([GitHub][3])

It has a testing page with a `db` lane, a `db-migration-analyze` lane, a rule ID for destructive migration risk, and a route that sends `db/migrations/` changes to `jankurai migrate --analyze`. However, the current migration scanner is line-oriented, can be suppressed by safety language or a marker, and explicitly says the marker “does not prove the SQL is safe.” ([GitHub][4])

The audit rubric already says PostgreSQL proof should include migration apply/rollback where possible, constraint tests, and seed validation; adapters should have DB integration tests; and ops/security should include secret scan, dependency/SBOM scan, workflow lint, and CI audit scoring. That is exactly the right minimum, but it needs stronger machine enforcement. ([GitHub][2])

# Mandatory database, migration, and testing standard for every agent-first project

## 1. Database ownership and topology

Every repo with durable state must have a machine-readable database manifest.

**Required files**

```text
agent/db-policy.toml
agent/data-classification.toml
agent/db-environments.toml
db/README.md
db/migrations/
db/constraints/
db/seeds/
db/fixtures/
db/schema/snapshot.sql
db/schema/checksums.json
docs/db-runbook.md
docs/restore-drill.md
```

**What `agent/db-policy.toml` must define**

```toml
schema_version = "1.0.0"
dialect = "postgresql"
migration_roots = ["db/migrations"]
constraint_roots = ["db/constraints"]
seed_roots = ["db/seeds"]

[ownership]
db_owner = "data-platform"
migration_reviewers = ["backend", "security", "data-platform"]
requires_codeowners = true

[agent_sandbox]
agents_may_run_prod_migrations = false
agents_may_connect_to_shared_dev_db = false
required_db = "ephemeral-container"
forbidden_commands = ["dropdb", "createdb --template prod", "psql $PROD_DATABASE_URL"]

[timeouts]
required_lock_timeout_ms = 5000
required_statement_timeout_ms = 60000
required_idle_transaction_timeout_ms = 60000

[migrations]
append_only_after_merge = true
require_checksum_registry = true
require_expand_contract_for_breaking_changes = true
require_rollback_or_forward_fix = true
require_backfill_plan = true
require_lock_analysis = true
require_rehearsal = true
require_backup_proof_for_high_risk = true
require_restore_proof_for_destructive = true

[security]
require_rls_for_multitenant_tables = true
require_force_rls_where_app_owner_bypasses = true
forbid_app_superuser = true
forbid_bypassrls_for_app_roles = true
require_parameterized_queries = true
require_data_classification = true

[exceptions]
require_owner = true
require_expiry_days = 30
require_human_approval = true
require_compensating_control = true
```

**Audit enforcement**

A repo scores zero for DB lifecycle if it has persistent data but lacks `agent/db-policy.toml`. A DB exception cannot be a comment; it must be an expiring, owner-signed, machine-readable exception with linked proof.

---

## 2. Migration file contract

Every migration must have a structured header. Agents are very good at generating plausible SQL; the header makes them generate a safety case before the SQL exists.

```sql
-- jankurai:migration v1
-- id: 20260505_143000_add_order_status
-- owner: backend-data
-- reviewer_required: backend,security,data-platform
-- risk: medium
-- phase: expand
-- tables: orders
-- data_classification: customer_operational
-- compatible_app_versions: ">=2026.05.05"
-- forward_plan: "add nullable status, backfill, validate, enforce not null later"
-- rollback_plan: "drop column before app uses it; after use, forward-fix only"
-- down_migration: "db/migrations/down/20260505_143000_add_order_status.sql"
-- backfill_plan: "batched by id, 5k rows/batch, retryable, idempotent"
-- lock_timeout: "5s"
-- statement_timeout: "60s"
-- expected_locks: "ACCESS EXCLUSIVE under 100ms for ADD COLUMN nullable"
-- estimated_rows_touched: 0
-- backup_required: false
-- restore_drill_required: false
-- rls_impact: none
-- feature_flag: order_status_v2
-- proof_required:
--   - db.ephemeral_apply
--   - db.schema_drift
--   - db.constraint_tests
--   - app.adapter_integration
```

**Audit enforcement**

A migration without this header is not “partially compliant”; it is unreviewable by an agent and should trigger a hard cap. The audit should parse the header into `target/jankurai/db/migration-manifest.json`.

---

## 3. Expand/contract migration discipline

Breaking schema changes must be staged:

1. **Expand:** add nullable columns, new tables, new indexes, new constraints as `NOT VALID` when appropriate, new code that can read both shapes.
2. **Backfill:** idempotent batched job with cursoring, retry receipts, row counts, and timeout budgets.
3. **Dual-read/dual-write:** if needed, prove old and new paths produce equivalent durable truth.
4. **Validate:** validate constraints, compare counts/checksums, prove no drift.
5. **Contract:** remove old column/table/path only after all app versions depending on it are gone.
6. **Cleanup:** delete feature flag and old code with tests proving no reachable dependency remains.

**Audit enforcement**

Any `DROP`, type narrowing, `SET NOT NULL`, column rename, table rewrite, unbounded `UPDATE`, unbounded `DELETE`, `TRUNCATE`, RLS disable, trigger disable, or privilege broadening outside an expand/contract plan triggers a hard cap.

---

## 4. Lock and timeout safety

For PostgreSQL, migration proof should require `lock_timeout`, `statement_timeout`, and, for long-running work, a batching strategy. PostgreSQL documents `statement_timeout` as aborting statements exceeding a specified duration, and `lock_timeout` as aborting statements that wait too long for table/index/row/object locks. ([PostgreSQL][11])

**Required implementation**

```sql
SET lock_timeout = '5s';
SET statement_timeout = '60s';
SET idle_in_transaction_session_timeout = '60s';
```

For index additions on large/hot tables, prefer production-safe online patterns. PostgreSQL documents that normal index builds can block writes, while `CREATE INDEX CONCURRENTLY` allows normal operations to continue, with caveats such as invalid indexes after failure and no transaction-block use. ([PostgreSQL][12])

**Audit enforcement**

The migration analyzer should detect:

```text
CREATE INDEX without CONCURRENTLY on non-empty/hot table
ALTER TABLE ... SET NOT NULL without prevalidated check
ALTER TABLE ... ADD COLUMN ... NOT NULL without DEFAULT safety analysis
ALTER TABLE ... ALTER TYPE / TYPE narrowing
DROP TABLE / DROP COLUMN / TRUNCATE
DELETE FROM without WHERE
UPDATE ... without WHERE or bounded batch predicate
DISABLE TRIGGER
ALTER TABLE ... DISABLE ROW LEVEL SECURITY
GRANT ... TO PUBLIC
GRANT superuser/BYPASSRLS-like capabilities
```

Each finding must require proof, not a comment.

---

## 5. RLS, tenant isolation, and authorization proof

PostgreSQL row-level security can restrict which rows are returned or modified per user; when RLS is enabled and no policy exists, PostgreSQL applies default deny. It also notes that superusers and roles with `BYPASSRLS` bypass RLS, so app roles must not be superuser or `BYPASSRLS`. ([PostgreSQL][13])

**Required tests**

For every multitenant table:

```text
owner can select own rows
owner cannot select other tenant rows
owner cannot update other tenant rows
owner cannot delete other tenant rows
insert rejects wrong tenant_id
service role scope is documented and minimal
anonymous/public role sees only explicitly public data
RLS is enabled and FORCE RLS is used where table owner bypass matters
```

**Audit enforcement**

Add `DB-011-RLS-TENANT-ISOLATION-GAP`. It should fail if a table has `tenant_id`, `org_id`, `account_id`, `workspace_id`, or equivalent but no RLS/authorization proof. It should also fail if RLS exists but no owner/non-owner negative tests exist.

---

## 6. Parameterized query proof

OWASP’s SQL Injection Prevention Cheat Sheet says prepared statements with parameterized queries force SQL code to be defined separately from user-supplied parameters, so the database distinguishes code from data regardless of input. ([OWASP Cheat Sheet Series][14])

**Required implementation**

For Rust/PostgreSQL:

```rust
// Good: parameters are bound, not concatenated.
sqlx::query!(
    "select id, email from users where tenant_id = $1 and email = $2",
    tenant_id,
    email
)
.fetch_optional(pool)
.await?;
```

**Forbidden**

```rust
format!("select * from users where email = '{}'", email)
```

**Audit enforcement**

Add language-specific detectors for SQL string concatenation, unsafe `format!`, `+` SQL construction, template strings containing user input, raw query escape hatches, and ORM “raw” calls. Allow raw SQL only in adapters and migrations, never in handlers/UI/domain.

---

## 7. Backup, restore, and PITR proof

Rollback SQL is not enough. Some migrations are not safely reversible after application writes begin. For destructive or high-risk migrations, the real safety net is verified backup and restore.

**Required proof artifacts**

```text
target/jankurai/db/backup-proof.json
target/jankurai/db/restore-drill.json
target/jankurai/db/pitr-window.json
target/jankurai/db/destructive-migration-approval.md
```

Each must include:

```json
{
  "environment": "staging",
  "migration_id": "20260505_143000_add_order_status",
  "backup_snapshot_id": "snap-...",
  "backup_started_at": "...",
  "backup_completed_at": "...",
  "restore_tested_at": "...",
  "restore_duration_seconds": 412,
  "rpo_seconds": 300,
  "rto_seconds": 1800,
  "verified_by": "ci",
  "human_reviewer": "data-platform"
}
```

**Audit enforcement**

A destructive migration without backup/restore proof should cap at **62**. A production-bound destructive migration without human approval should cap at **55**.

---

## 8. Schema drift detection

Agents often edit code and forget durable schema truth. Every PR touching DB, adapters, contracts, or generated clients should prove that:

```text
migrations apply from empty DB
schema snapshot after migrations matches committed snapshot
generated DB query metadata is current
ORM/schema model matches real DB
OpenAPI/protobuf/JSON Schema contracts still match DB-backed API behavior
no direct manual DB drift is required
```

**Audit enforcement**

Add `DB-009-SCHEMA-DRIFT-GAP`. Required artifact:

```text
target/jankurai/db/schema-drift.json
```

It must show `expected_schema_hash`, `actual_schema_hash`, `migration_count`, `pending_migrations`, and `drift_status`.

---

## 9. Seed, fixture, and synthetic data controls

Seeds and fixtures must be deterministic, minimal, non-secret, non-production, and classified.

**Required controls**

```text
db/seeds/README.md defines purpose and ownership
fixtures cannot contain real PII, tokens, passwords, customer text, private prompts, or production IDs
seed scripts are idempotent
seed validation runs in CI
fixture sanitizer has tests
large fixture files are scanned for secrets and PII-like patterns
```

**Audit enforcement**

Add `DB-015-SEED-FIXTURE-SAFETY-GAP`. Cap at **76** for missing seed policy, **60** for secret-like fixture data, **55** for production data copied into tests.

---

## 10. DB observability and runtime safety

Every DB-backed service must emit structured evidence for:

```text
migration id
schema version
query timeout
lock wait timeout
transaction duration
rows touched by writes/backfills
slow query events
connection pool saturation
deadlocks/retry count
idempotency key
correlation/request/job id
```

Jankurai already values tracing, request IDs, structured diagnostics, repair receipts, and agent-friendly exceptions; DB work should plug directly into that model. ([GitHub][2])

**Audit enforcement**

A DB-backed app without structured DB telemetry gets a medium finding; a write-heavy app without lock/deadlock/backfill telemetry gets a high finding.

# Security best practices missing or only partially enforced

Below is the security baseline I would require for every agent-first project. Some items are mentioned in Jankurai prose, but they need to become **detector-backed, proof-lane-backed, and scoring-backed**.

## A. Governance and secure-development controls

| Rule                                 | Required implementation                                                                                                                              | Audit/scoring                                                                |
| ------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| `SEC-001-THREAT-MODEL-GAP`           | Every new auth, data, DB, AI/tool, file, network, payment, admin, or background-job surface needs a threat-model note.                               | Cap **80** if missing on medium-risk change; cap **65** on high-risk change. |
| `SEC-002-NIST-SSDF-MAPPING-GAP`      | Map controls to NIST SSDF practices: prepare org, protect software, produce well-secured software, respond to vulnerabilities.                       | Advisory at first; high if repo claims compliance without mapping.           |

| `SEC-051-UNPINNED-ACTION-GAP`        | Pin third-party actions to full-length commit SHA. GitHub says this is the only immutable way to use an action. ([GitHub Docs][9])                                                        | Cap **72** for tag-only action refs.                   |
| `SEC-052-WORKFLOW-INJECTION-GAP`     | Treat PR titles, branch names, issue text, commit messages, and tool outputs as untrusted; pass via env vars or action inputs, not inline shell interpolation.                            | Cap **65** if unsafe.                                  |
| `SEC-053-FORK-PR-SECRET-GAP`         | No secrets on untrusted fork PR workflows; use `pull_request` safely, avoid dangerous `pull_request_target` patterns.                                                                     | Cap **55** if unsafe.                                  |
| `SEC-054-NO-TIMEOUT-CONCURRENCY`     | Add `timeout-minutes` to every job and `concurrency` for branch/PR.                                                                                                                       | Cap **84** if missing; **76** for deployment workflow. |
| `SEC-055-ARTIFACT-RETENTION-GAP`     | Set artifact retention, redact artifacts, scan uploaded evidence.                                                                                                                         | Cap **82** if missing in security-sensitive repo.      |
| `SEC-056-ENVIRONMENT-PROTECTION-GAP` | Protected deploy environments, required reviewers, OIDC short-lived cloud credentials. GitHub recommends OIDC for cloud provider access instead of long-lived secrets. ([GitHub Docs][9]) | Cap **70** if production deploy lacks protection.      |

## G. Agentic AI, tools, prompts, and RAG security

OWASP’s 2025 GenAI risks are directly relevant here: prompt injection can alter an LLM’s behavior and cause unauthorized access or command execution, while excessive agency is rooted in excessive functionality, permissions, or autonomy. ([OWASP Gen AI Security Project][17])

| Rule                                    | Required implementation                                                                                                                                                                                                                                                   | Audit/scoring                                                                 |
| --------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| `SEC-060-PROMPT-INJECTION-GAP`          | Source hierarchy: system/developer/repo policy outrank untrusted issue/page/tool content; prompt-injection regression tests for docs, issues, PR comments, web pages, PDFs, tool output.                                                                                  | Cap **78** if weak; **65** if untrusted context can change policy.            |
| `SEC-061-OVERBROAD-AGENT-AGENCY`        | Least-privilege tool profiles: read-only, edit-local, test-local, network-off, db-ephemeral-only, release-human-approved.                                                                                                                                                 | Cap **65** for broad write/network/shell/db access without gates.             |
| `SEC-062-AGENT-DB-SANDBOX-GAP`          | Agents may run only ephemeral DBs; no production/shared dev DB credentials in agent contexts.                                                                                                                                                                             | Cap **60** if agents can touch prod/shared DB.                                |
| `SEC-063-LLM-OUTPUT-HANDLING-GAP`       | LLM output must be schema-validated before parsing, rendering, storing, executing, querying, or using as policy. OWASP LLM01 recommends defining and validating expected output formats. ([OWASP Gen AI Security Project][17])                                            | Cap **70** if unvalidated output drives tools/code/SQL.                       |
| `SEC-064-TOOL-CALL-MEDIATION-GAP`       | Deterministic policy engine approves tool calls; LLM cannot self-authorize high-impact actions. OWASP LLM06 recommends complete mediation and human approval for high-impact actions. ([OWASP Gen AI Security Project][18])                                               | Cap **65** if missing.                                                        |
| `SEC-065-MODEL-PROMPT-EVAL-DRIFT`       | Version prompts, models, tools, MCP servers, eval data, embedding models; run golden evals before merge.                                                                                                                                                                  | Cap **78** if drift is untracked.                                             |
| `SEC-066-RAG-VECTOR-STORE-GAP`          | Tenant filters at retrieval and storage, embedding data classification, poisoning tests, deletion propagation, retrieval evals, source attribution. OWASP LLM Top 10 includes vector/embedding weaknesses and data/model poisoning. ([OWASP Gen AI Security Project][18]) | Cap **74** for RAG app without vector security proof.                         |
| `SEC-067-UNBOUNDED-CONSUMPTION-GAP`     | Model/tool-call budgets, rate limits, spend alerts, recursion limits, kill switches.                                                                                                                                                                                      | Cap **82** if missing; **70** if public endpoint can trigger unbounded spend. |
| `SEC-068-PROMPT-TRANSCRIPT-PRIVACY-GAP` | Prompt logs/transcripts classified, redacted, retention-limited, secret-scanned, and excluded from public artifacts.                                                                                                                                                      | Cap **70** if missing; **55** if sensitive transcript leaked.                 |

## H. Runtime, infrastructure, and incident response

| Rule                              | Required implementation                                                                                          | Audit/scoring                                       |
| --------------------------------- | ---------------------------------------------------------------------------------------------------------------- | --------------------------------------------------- |
| `SEC-070-CONTAINER-HARDENING-GAP` | Non-root containers, read-only filesystem where possible, minimal base image, vulnerability scan, pinned digest. | Cap **78** if missing in containerized prod.        |
| `SEC-071-IAC-SCANNING-GAP`        | Terraform/Kubernetes/cloud config scanning, least-privilege IAM, public exposure checks.                         | Cap **78** if missing.                              |
| `SEC-072-EGRESS-CONTROL-GAP`      | Default-deny egress for workers/agents where possible; allowlist external APIs.                                  | Cap **80** if high-risk agent/tool has open egress. |
| `SEC-073-RATE-LIMIT-ABUSE-GAP`    | Rate limits, quotas, anti-automation checks, abuse logs, IP/user/device throttles.                               | Cap **78** if public API lacks it.                  |
| `SEC-074-VULNERABILITY-SLA-GAP`   | Severity rubric, patch SLAs, advisory process, private reporting, CVE/GHSA path.                                 | Cap **84** if missing.                              |
| `SEC-075-INCIDENT-RUNBOOK-GAP`    | Incident roles, revoke/rotate steps, forensic artifact policy, customer comms template, postmortem process.      | Cap **82** if missing; **70** for production app.   |
| `SEC-076-SECURITY-EVIDENCE-GAP`   | CI uploads raw logs, SARIF, SBOM, scan JSON, proof receipts, and reviewer receipts with retention/redaction.     | Cap **76** if claims cannot be replayed.            |

# Proposed audit/scoring setup

## Revised 100-point score

I would adjust the scoring model so DB/security cannot be hidden inside a small “data truth” bucket.

| Dimension                                             | Weight |
| ----------------------------------------------------- | -----: |
| Agent ownership, navigation, and context economy      |      8 |
| Contract and boundary integrity                       |     10 |
| **Database lifecycle and data governance**            | **12** |
| **Migration safety and recoverability**               | **12** |
| Proof lanes, testing, and false-green prevention      |     14 |
| Application security controls                         |     12 |
| Supply chain, CI/CD, and release provenance           |     12 |
| Agentic AI/tool security                              |      8 |
| Observability, incident response, and repair evidence |      8 |
| Build speed and developer ergonomics                  |      4 |

For repos with no durable database, DB points can be marked N/A and redistributed. For any repo with durable state, DB and migration controls should be mandatory.

| Missing schema drift check                                                |                       78 |
| Missing constraint tests                                                  |                       80 |
| Missing tenant/RLS isolation tests                                        |                       68 |
| Missing seed/fixture sanitization                                         |                       76 |
| Secret/PII in fixtures                                                    |                       55 |

## Final recommendation

Jankurai should keep its current philosophy, but make the DB and security layers much less “advisory.” The strongest next version would add:

1. **A DB policy manifest.**
2. **Structured migration headers.**
3. **Append-only migration registry and checksums.**
4. **Ephemeral apply/rollback/schema-drift proof.**
5. **Dedicated constraint and RLS lanes.**
6. **Backup/restore/PITR proof for destructive changes.**
7. **A security policy that actually matches the security matrix.**
8. **Hardened CI with SHA-pinned actions and least-privilege permissions.**
9. **NIST SSDF + OWASP ASVS + OWASP GenAI mappings.**
10. **Hard caps that fail unsafe DB/security changes before score averaging can hide them.**

The core idea is simple: **agents may propose database and security changes, but only deterministic proof, bounded permissions, durable receipts, and human-reviewed exceptions can let those changes merge.**
