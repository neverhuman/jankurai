I studied Jankurai closely. My read is: **Jankurai already has an unusually strong agent-first conformance spine**, but its database and security posture is still mostly **policy-aware, not proof-complete**. The missing work is to turn the current docs, placeholders, scanners, and lanes into **executable, blocking, artifact-backed controls** that an agent cannot hand-wave past.

I’m treating “missing” as: absent, only documented, only advisory, not represented in scoring, not backed by CI artifacts, or not strong enough for real production database/security risk.

## 1. What Jankurai already does well

Jankurai’s core premise is excellent for agent-first work: **“no proof, no merge; no receipt, no trust.”** The repo already emphasizes audit lanes, proof routing, generated-zone boundaries, score history, repair queues, security evidence, and agent-readable maps. The README explicitly frames Jankurai as a repository conformance standard and local audit CLI for making AI-generated code safe, reviewable, and reproducible. ([GitHub][1])

The repo already has key control-plane files: `agent/audit-policy.toml`, `agent/boundaries.toml`, `agent/proof-lanes.toml`, `agent/security-policy.toml`, `agent/test-map.json`, generated-zone policy, owner maps, score history, and standard docs. ([GitHub][2])

For databases, it already recognizes `db/`, `db/migrations/`, and `db/constraints/` as first-class surfaces. The DB README says the area is reserved for PostgreSQL migrations, constraints, durable-truth policy, rollback/backfill notes, lock behavior, and transaction proof. ([GitHub][3]) The standard also says `db/` should own migrations, constraints, indexes, RLS, seeds, and extension policy, while durable invariants must not live only in application code. ([GitHub][4])

For migration risk, Jankurai already has `HLT-021-DESTRUCTIVE-MIGRATION`, a `db-migration-analyze` lane, and route mapping from `db/migrations/` to migration-liability analysis.   Its tests show the scanner catches `DROP TABLE`, `TRUNCATE`, and unbounded `DELETE`, and allows explicit safety evidence such as rollback comments or `jankurai:migration-safe`.  The docs are honest that the migration scanner is mostly line-oriented and can be confused by pathological SQL, which is exactly why executable DB proof lanes need to be added. ([GitHub][5])

For security, Jankurai already has private reporting guidance, `just security`, `just security-strict`, a security evidence artifact, and explicit AI-agent boundaries: trusted policy must not be rewritten by untrusted context, generated outputs, repair plans, or model responses.  The security tool matrix also names the right direction: local secret scanning, CI dependency/security/workflow checks, and release-time SBOM, vulnerability scanning, OpenSSF Scorecard, SLSA, and Cosign. 

The gap: **many of these ideas are described, but not yet enforced deeply enough.**

---

## 2. The most important missing pieces

The biggest missing pieces are:

1. **No executable database harness.**
   `db/` currently contains policy/readme scaffolding more than runnable DB proof. The test map routes `db/migrations/` to analysis, not to an actual disposable PostgreSQL apply/revert/constraint/RLS/performance proof. 

2. **Migration safety is static, not proven.**
   The destructive migration scanner is valuable, but it is not enough. It does not prove that migrations apply cleanly, rollback safely, preserve data, avoid long locks, maintain compatibility, or keep tenant isolation intact. The docs already acknowledge this limitation. ([GitHub][5])

3. **Security tooling is named but not consistently enforced.**
   `agent/security-policy.toml` lists `gitleaks` as required and `cargo audit` / `npm audit` as advisory.  But `tools/security-lane.sh` treats `cargo audit` and `npm audit` as required, and separately treats `syft` and `zizmor` as advisory.  That mismatch should be fixed immediately because an agent or reviewer needs a single source of truth.

4. **The GitHub workflow needs hardening.**
   The current workflow runs audit/security/UX/paper lanes, which is great, but it does not show explicit least-privilege `permissions:`, SHA-pinned actions, checkout credential hardening, CodeQL/Semgrep, Scorecard, provenance signing, SBOM upload, or release attestations.  GitHub and OWASP both recommend minimizing token permissions, pinning actions to commit SHAs, using CODEOWNERS for sensitive workflow changes, and treating CI/CD as a high-value attack surface. ([GitHub Docs][6])

5. **Agent-specific security is under-modeled.**
   Jankurai names prompt injection, overbroad agency, generated-zone risk, and supply-chain drift, but it needs concrete red-team fixtures, tool-permission profiles, MCP/tool manifest pinning, transcript/data-leak scanning, and approval receipts. OWASP’s LLM Top 10 explicitly includes prompt injection, insecure output handling, supply-chain vulnerabilities, sensitive information disclosure, insecure plugins/tools, and excessive agency. ([OWASP][7])

# 3. Database management, migration, and testing controls every agent-first project must have

## A. Database ownership and source-of-truth controls

### DB-001: Canonical database policy file

Add:

```text

```

It should declare:

```toml
schema_version = "1.0.0"

minimum_postgres_version = "16"

[ownership]
schema_owner = "platform-db"
migration_owner = "platform-db"
application_write_owner = "crates/adapters"
read_model_owner = "crates/adapters"

[allowed_paths]
migrations = ["db/migrations"]
constraints = ["db/constraints"]
roles = ["db/roles"]
seeds = ["db/seeds"]
fixtures = ["db/fixtures"]
schema_snapshots = ["db/schema"]

[prohibited]
app_only_durable_invariants = true
raw_sql_in_api_handlers = true
raw_sql_in_domain = true
raw_sql_in_web = true
production_pii_in_fixtures = true

[required_proof]
apply_migrations = true
rollback_or_roll_forward = true
schema_drift = true
constraint_negative_tests = true
tenant_isolation_tests = "required_if_multitenant"
backup_restore = "required_for_release"

```

**Audit enforcement:**
If `db/policy.toml` is missing in a repo with `db/migrations/`, apply a new hard cap:

```text
HLT-028-DB-POLICY-GAP
severity = medium
max_score = 82
```

If the repo has production DB code but no DB ownership policy:

```text
max_score = 76
```

### DB-002: Canonical schema snapshot

Add:

```text
db/schema/current.sql
db/schema/current.sha256
```

Every migration PR should:

1. Start disposable PostgreSQL.
2. Apply all migrations.
3. Dump normalized schema.
4. Compare against committed `db/schema/current.sql`.
5. Fail if drift exists and the schema snapshot was not updated.

Command:

```bash
just db-schema-drift
```

Suggested implementation:

```bash
pg_dump \
  --schema-only \
  --no-owner \
  --no-privileges \
  --no-comments \
  "$DATABASE_URL" \
  | sed '/^--/d' \
  > target/jankurai/db/schema.actual.sql

diff -u db/schema/current.sql target/jankurai/db/schema.actual.sql
```

**Audit enforcement:**

```text
HLT-031-SCHEMA-DRIFT-UNPROVED
severity = high
max_score = 76
trigger = db/migrations changed but no schema drift artifact
required_artifact = target/jankurai/db/schema-drift.json
```

---

### DB-003: DB role and privilege model

Add:

```text
db/roles/001_roles.sql
db/roles/README.md
```

Required roles:

```text
app_rw          normal app read/write role
app_ro          read-only app/reporting role
migrator        migration-only role
ci_migrator     disposable CI migration role
admin_breakglass emergency role, never used by app
analyst_ro      restricted analytics role
```

Rules:

* Application runtime must not use owner/superuser credentials.
* Migrations must not run as the same role as the app runtime.
* Read-only jobs must not have write grants.
* Agents must never receive production database credentials.
* CI database credentials must point only to disposable DBs.
* `PUBLIC` privileges must be revoked where appropriate.
* Extensions must be allowlisted.

Example:

```sql
REVOKE ALL ON SCHEMA public FROM PUBLIC;

CREATE ROLE app_rw NOLOGIN;
CREATE ROLE app_ro NOLOGIN;
CREATE ROLE migrator NOLOGIN;

GRANT USAGE ON SCHEMA public TO app_rw, app_ro;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO app_ro;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_rw;
```

**Audit enforcement:**

```text
HLT-035-DB-ROLE-PRIVILEGE-GAP
severity = high
max_score = 70
```

OWASP’s database security guidance emphasizes secure DB configuration, and OWASP’s SQL injection guidance also recommends defense in depth such as parameterization and least-privilege access. ([OWASP Cheat Sheet Series][8])

## B. Migration authoring controls

### DB-004: Required migration metadata block

Every migration must begin with a machine-readable block:

```sql
-- jankurai:migration
-- id: 202605050930_add_accounts_last_login
-- owner: platform-db
-- reviewer: security-db
-- change_type: expand
-- safety: online
-- transaction: required
-- lock_timeout: 5s
-- statement_timeout: 60s
-- estimated_rows: 1200000
-- deploy_phase: expand
-- app_compatibility: old_app_new_db,new_app_old_db
-- rollback: db/rollbacks/202605050930_add_accounts_last_login.sql
-- backfill: none
-- data_classification: internal
-- tenant_impact: none
-- tests:
--   - db-apply
--   - db-rollback
--   - db-schema-drift
--   - db-query-compile
```

Then every migration should set timeouts:

```sql
SET lock_timeout = '5s';
SET statement_timeout = '60s';
```

**Audit enforcement:**

```text
HLT-029-MIGRATION-METADATA-GAP
severity = medium
max_score = 78
```

If destructive SQL lacks metadata:

```text
severity = high
max_score = 68
```

---

### DB-005: Expand-contract migration policy

Agent-generated migrations must default to **expand-contract**:

1. **Expand:** add nullable column/table/index/trigger/RLS policy.
2. **Dual-write or backfill:** maintain old and new shapes.
3. **Read switch:** app reads new shape after compatibility proof.
4. **Contract:** remove old shape only after old app versions are gone.
5. **Cleanup:** remove temporary compatibility code and prove no dead paths.

Disallowed without explicit exception:

```sql
DROP TABLE ...
DROP COLUMN ...
ALTER COLUMN ... SET NOT NULL
ALTER COLUMN ... TYPE ...
RENAME COLUMN ...
TRUNCATE ...
DELETE FROM table;
```

These are not always wrong, but they are **agent-dangerous** and must require a proof receipt.

**Audit enforcement:**

```text
HLT-033-DB-COMPATIBILITY-BREAK
severity = critical
max_score = 68
required_artifact = target/jankurai/db/compatibility.json
```

---

### DB-006: Online DDL and lock-safety proof

Every migration must be checked for:

* `CREATE INDEX` vs `CREATE INDEX CONCURRENTLY`.
* `ALTER TABLE ... ADD COLUMN ... DEFAULT ... NOT NULL`.
* Full table rewrites.
* Long exclusive locks.
* Missing `lock_timeout`.
* Missing `statement_timeout`.
* Backfill batch size.
* Trigger overhead.
* FK validation strategy.
* `VALIDATE CONSTRAINT` timing.
* Transaction compatibility, especially for `CONCURRENTLY`.

**Audit enforcement:**

```text
HLT-032-ONLINE-DDL-LOCK-RISK
severity = high
max_score = 72
```

Required artifact:

```text
target/jankurai/db/lock-risk.json
```

Example artifact fields:

```json
{
  "migration": "202605050930_add_index.sql",
  "risk": "create_index_without_concurrently",
  "table": "events",
  "estimated_rows": 120000000,
  "lock_timeout_present": true,
  "statement_timeout_present": true,
  "decision": "block"
}
```

---

### DB-007: Down migration or roll-forward proof

A rollback comment should not be enough to suppress destructive risk. Jankurai currently allows rollback comments to suppress `HLT-021`.  That is useful for early adoption, but production projects should require executable proof.

Allowed rollback models:

1. **Executable down migration.**
2. **Roll-forward repair migration.**
3. **Point-in-time restore runbook plus rehearsal receipt.**
4. **Manual exception with owner, expiry, blast radius, and approval.**

Required files:

```text
db/rollbacks/<migration_id>.sql
db/runbooks/rollback/<migration_id>.md
```

**Audit enforcement:**

```text
HLT-030-DB-APPLY-ROLLBACK-UNPROVED
severity = high
max_score = 70
required_artifact = target/jankurai/db/rollback-proof.json
```

## D. Required database tests

### DB-008: Constraint negative tests

Every DB constraint must have a negative test.

Example:

```sql
-- should fail: blank email
INSERT INTO accounts (id, email, role)
VALUES ('acct_bad_email', '', 'member');
```

The test runner should assert this fails with the expected constraint name.

Required artifact:

```json
{
  "constraint": "accounts_email_check",
  "negative_case": "blank_email",
  "expected_error": "check_violation",
  "result": "passed"
}
```

**Audit enforcement:**

```text
HLT-038-DB-CONSTRAINT-TEST-GAP
severity = medium
max_score = 80
```

---

### DB-009: Tenant isolation and RLS tests

The Jankurai standard already says tenant isolation should be tested when multi-tenant. ([GitHub][4]) The example scaffold currently documents append-only audit behavior as application-layer policy and recommends RLS or triggers for production.  That should become mandatory for real production projects.

Required tests:

* Owner can read own resource.
* Owner cannot read another tenant’s resource.
* Admin cannot cross tenant unless explicitly allowed.
* Read-only role cannot write.
* Background job role cannot bypass tenant policy.
* API adapter cannot disable RLS.
* Report/analytics role cannot see restricted PII.
* Service role exceptions are named, owned, and tested.

Example:

```sql
ALTER TABLE resources ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_resources_policy
ON resources
USING (organization_id = current_setting('app.organization_id')::text);
```

Test:

```sql
SET app.organization_id = 'org_a';
SELECT * FROM resources WHERE organization_id = 'org_b';
-- expect 0 rows
```

**Audit enforcement:**

```text
HLT-034-RLS-TENANT-ISOLATION-GAP
severity = critical
max_score = 66
```

---

### DB-010: Audit log immutability

Audit/event tables must be append-only at the database layer, not just the application layer.

Required controls:

* No `UPDATE`/`DELETE` grant for app role.
* Optional trigger blocking mutation.
* Separate retention/archive policy.
* Tamper-evident hash chain for high-risk audit logs.
* Tests proving app role cannot update/delete audit records.

Example:

```sql
CREATE FUNCTION prevent_audit_event_mutation()
RETURNS trigger AS $$
BEGIN
  RAISE EXCEPTION 'audit_events are append-only';
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER audit_events_no_update
BEFORE UPDATE OR DELETE ON audit_events
FOR EACH ROW EXECUTE FUNCTION prevent_audit_event_mutation();
```

**Audit enforcement:**

```text
HLT-040-AUDIT-IMMUTABILITY-GAP
severity = high
max_score = 72
```

### DB-013: Backup and restore proof

Every production-bound DB project must prove restore, not just backup existence.

Required artifacts:

```text

```

Minimum restore proof:

# 4. Exhaustive security best practices missing or only partially enforced

## A. CI/CD hardening

Jankurai’s current GitHub workflow is a good start, but it should be hardened as a protected production system. OWASP warns that CI/CD compromise can cause secret exfiltration, write-token compromise, artifact poisoning, and denial-of-wallet incidents. ([OWASP Cheat Sheet Series][9])

Must add:

1. Explicit top-level permissions:

```yaml
permissions:
  contents: read
```

2. Narrow permissions per job:

```yaml
permissions:
  contents: read
  security-events: write
```

Only the SARIF upload job should get `security-events: write`.

3. Checkout hardening:

```yaml
- uses: actions/checkout@<full_commit_sha>
  with:
    persist-credentials: false
```

GitHub recommends minimum `GITHUB_TOKEN` permissions and pinning actions to full-length commit SHAs for stronger supply-chain integrity. ([GitHub Docs][6])

4. SHA-pin all actions:

```yaml
- uses: actions/checkout@<sha>
- uses: actions/setup-node@<sha>
- uses: dtolnay/rust-toolchain@<sha>
- uses: actions/upload-artifact@<sha>
```

5. Add CODEOWNERS:

```text
.github/workflows/* @security-team @platform-team
agent/security-policy.toml @security-team
agent/proof-lanes.toml @platform-team
db/** @db-team @security-team
tools/security-lane.sh @security-team
```

GitHub recommends CODEOWNERS for monitoring changes to workflows and other sensitive areas. ([GitHub Docs][6])

6. Add workflow linting:

```bash
zizmor .github/workflows
actionlint
```

7. Add Scorecard:

```bash
scorecard --repo=. --format=json
```

GitHub specifically recommends OpenSSF Scorecard for evaluating repository security posture. ([GitHub Docs][6])

**Audit rules:**

```text
HLT-042-CI-PERMISSIONS-OVERBROAD     max_score = 72
HLT-043-ACTIONS-NOT-SHA-PINNED       max_score = 78
HLT-044-CHECKOUT-CREDENTIAL-RISK     max_score = 80
HLT-046-WORKFLOW-LINT-MISSING        max_score = 78
```

## D. SAST, CodeQL, Semgrep, and unsafe-code checks

Missing:

1. CodeQL for Rust/JavaScript/TypeScript where supported.
2. Semgrep rules for:

   * SQL injection,
   * command injection,
   * path traversal,
   * XSS,
   * SSRF,
   * unsafe deserialization,
   * insecure crypto,
   * missing authz checks,
   * logging secrets,
   * prompt/tool injection patterns.
3. Rust unsafe policy:

   * forbid `unsafe` by default,
   * allow only with `SAFETY:` comment,
   * require reviewer,
   * require Miri or dedicated test when relevant.
4. TypeScript strictness:

   * no `any` at trust boundaries,
   * no unchecked JSON parsing,
   * schema validation at external inputs.
5. API schema validation:

   * OpenAPI request/response validation,
   * generated clients only,
   * no handwritten DTO drift.


## E. Application security controls

Every agent-first project should require these as audit-visible controls:

1. Authentication:

   * no custom password crypto,
   * secure session cookies,
   * rotation/invalidation,
   * MFA for admin surfaces,
   * no auth state in frontend-only logic.

2. Authorization:

   * role matrix,
   * owner/non-owner tests,
   * tenant isolation tests,
   * deny-by-default policy,
   * API and DB-level enforcement for high-risk data.

3. Input validation:

   * schema validation at every external boundary,
   * reject unknown fields where appropriate,
   * canonicalize paths/URLs,
   * strict upload limits,
   * content-type validation.

4. SQL injection prevention:

   * parameterized queries only,
   * no string-built SQL with user input,
   * allowlisted dynamic identifiers,
   * query compile checks.

OWASP states that prepared statements/parameterized queries force developers to define SQL code first and pass parameters separately, which prevents attackers from changing query intent. ([OWASP Cheat Sheet Series][13])

5. XSS prevention:

   * output encoding,
   * no unsafe HTML rendering without sanitizer,
   * CSP for web apps,
   * test malicious fixtures.

6. SSRF prevention:

   * URL allowlists,
   * block private IP ranges,
   * DNS rebinding protection,
   * egress restrictions.

7. File safety:

   * path traversal tests,
   * MIME sniffing policy,
   * upload scanning,
   * size and decompression limits.

8. Crypto:

   * no homegrown crypto,
   * modern password hashing,
   * key rotation,
   * KMS-backed secrets,
   * envelope encryption for sensitive data.

9. Logging:

   * no secrets/PII in logs,
   * correlation IDs,
   * structured error codes,
   * redaction tests.

10. Abuse controls:

* rate limits,
* quotas,
* bot protection,
* idempotency keys,
* replay protection.

**Audit rules:**

```text
HLT-022-AUTHZ-ISOLATION-GAP          existing
HLT-023-INPUT-BOUNDARY-GAP           existing
HLT-061-LOG-REDACTION-GAP            max_score = 76
HLT-062-IDEMPOTENCY-GAP              max_score = 78
HLT-063-ABUSE-CONTROL-GAP            max_score = 80
```

# 6. Audit and scoring setup

## Recommended score model

Jankurai’s current scoring already has security and data truth dimensions, but DB safety is underweighted for production agent-first projects. The current audit rubric gives “Data truth and workflow safety” 8 points and “Security and supply-chain posture” 14 points.  I would rebalance production profiles like this:

```text
Ownership and navigation surface             10
Contract and boundary integrity              10
Proof lanes and test routing                 10
Database change safety                       14
Security and supply-chain posture            18
Agent agency and prompt/tool safety          10
Application security and input boundaries    10
CI/CD and release provenance                 10
Observability, repair, and evidence           8
Total                                       100
```

For non-DB projects, redistribute the 14 DB points across application security, testing, and release readiness.

---

## Blocking severity model

Use this policy:

```toml
minimum_score = 85
fail_on = ["critical", "high"]

[release]
minimum_score = 92
fail_on = ["critical", "high", "release_blocking"]

[pull_request]
ratchet = true
cannot_lower_score = true
```

Rules:

* Critical finding: merge blocked.
* High finding: merge blocked unless time-bounded exception exists.
* Medium finding: score impact and repair queue.
* Low finding: advisory unless repeated or stale.
* Expired exception: becomes high.
* Missing required artifact: lane fails, not advisory.
* Generated evidence without raw command/log receipt: cap score.

## Evidence schema all lanes should emit

Every proof lane should emit JSON like:

```json
{
  "schema_version": "1.0.0",
  "lane": "db-apply",
  "repo": "owner/repo",
  "git_head": "abc123",
  "changed_paths": ["db/migrations/202605050930_add_index.sql"],
  "started_at": "2026-05-05T12:34:56Z",
  "duration_ms": 42110,
  "decision": "pass",
  "required": true,
  "tools": [
    {
      "id": "postgres",
      "version": "16.3",
      "command": "just db-apply",
      "exit_code": 0
    }
  ],
  "artifacts": [
    {
      "path": "target/jankurai/db/apply.log",
      "sha256": "..."
    }
  ],
  "controls": [
    "HLT-030-DB-APPLY-ROLLBACK-UNPROVED"
  ],
  "finding_counts": {
    "critical": 0,
    "high": 0,
    "medium": 0,
    "low": 0
  },
  "exceptions": []
}
```

A lane should not be considered “green” unless:

* command exited successfully,
* required artifacts exist,
* artifacts validate against schema,
* artifact hashes are recorded,
* highest severity is below threshold,
* no expired exception was used,
* changed files are covered by `agent/test-map.json`.

## Final recommendation

Jankurai should evolve from:

> “This repo knows where DB/security risk is.”

to:

> “This repo refuses to merge unless DB/security risk has executable, replayable, artifact-backed proof.”

The strongest next move is to make **database proof** a peer of the existing audit/security lanes:

```bash
just db-proof
just security-strict
just agent-redteam
just release-proof
```

Then wire those into `agent/test-map.json`, `agent/proof-lanes.toml`, `agent/audit-policy.toml`, CI, SARIF, score history, and repair queues. That turns the standard into a true agent-first safety harness: every agent change has a smallest valid proof lane, every risky database/security change produces receipts, and every missing control becomes a scored, actionable finding rather than a reviewer’s memory burden.
