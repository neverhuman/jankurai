## Phase 02 verdict: treat it as **complete enough to exist, not hardened enough to trust blindly**

Phase 02’s stated goal is exactly the right load-bearing surface: move the audit from mostly heuristic scanning into a versioned rule engine with semantic understanding, while keeping the exit state realistic: not a compiler-grade analyzer, but a clean architecture that can support AST and graph checks incrementally without breaking the audit. ([GitHub][1])

The important contradiction is that the Phase 02 header says `Status: complete`, but its own receipt says the phase is only a “partial rule engine and semantic oracle,” with “analyzer split remains partial” and “AST/import pilot remain gated follow-on work.” The Phase 00 router explicitly says that when a phase header and receipt disagree, the receipt controls. So I would not call 02 hardened yet. I would mark it **complete-with-residual** or **partial-hardening-required** until the items below are closed. ([GitHub][2]) ([GitHub][1])

The target architecture is already documented as:

```text
inventory -> analyzers -> rule registry -> findings -> caps -> score -> reports
```

and Phase 02 names the core modules: `audit/rules.rs`, `audit/analyzers/`, `boundaries/`, `audit/finding_builder.rs`, and `audit/caps.rs`. It also says rule metadata must include stable ID, category, TLR, severity, lane, owner hint, docs URL, evidence kind, repair eligibility, confidence rules, and cap impact. ([GitHub][1])

My plan below is to finish 02 by making that architecture true in code, then hardening it with fixtures, compatibility guards, performance proof, and receipts.

# Definition of “100% done and hardened” for Phase 02

Phase 02 is done only when all of these are true:

1. **Every emitted high or critical finding has a stable rule ID**, lane, owner or owner hint, docs link, evidence kind, confidence, exact repair instruction, and rerun command. This matches the standard’s audit finding requirements: findings must include `rule_id` when stable, severity, category, path, line when available, problem, evidence, `agent_fix`, owner, validation lane, and docs link. ([GitHub][3])

2. **Every cap maps to one or more stable rules.** Caps can stay legacy-stable, but they cannot be a parallel hidden policy universe. A cap should be explained by rule metadata.

3. **Every analyzer returns structured rule hits, not just dimension scores.** The current architecture has analyzer modules, but the hardening target is that analyzers feed the rule registry and finding builder directly, not a giant hand-built switchboard.

4. **Boundary checks are policy-driven from `agent/boundaries.toml`.** Phase 02 explicitly calls for loading `agent/boundaries.toml` as first-class input and normalizing Rust, TypeScript, Python, SQL, and streaming policies. ([GitHub][1])

5. **Generated-zone checks are source-oriented.** Findings must tell agents to edit source contracts and regenerate outputs, not patch generated files. Phase 02 explicitly requires generated-zone findings to direct repair toward source, command, and ownership. ([GitHub][1])

6. **The AST/import graph pilot emits all applicable semantic violations, degrades safely, and is performance-bounded.** Phase 02 says parser failures must become advisory diagnostics, not audit crashes, and performance must remain compatible with `just fast`. ([GitHub][1])

7. **Findings are actionable without reading the whole standard.** Phase 02’s finding-quality section says each finding must answer what failed, why it matters, where to fix, and how to prove, with stable fingerprints and deterministic repair queue ordering. ([GitHub][1])

8. **The phase receipt is reconciled.** Since the receipt currently says analyzer split and AST/import pilot remain follow-on work, the hardening pass must finish those or explicitly move remaining work into a new named residual phase with proof. ([GitHub][1])

The repo’s own mission is a control plane for agent-native repositories: ownership, proof, generated zones, and repair evidence are made machine-readable so bad code is easier to reject than merge. The moonshot loop is `intent -> bounded agents -> proof lanes -> evidence -> expired exceptions -> reusable primitives`. ([GitHub][4])

```rust
    pub required_header: Option<String>,
    pub checksum: Option<String>,
    pub advisory_until_reproducible: bool,
}
```

Analyzer behavior:

```text
1. Parse `agent/generated-zones.toml`.
2. Validate every zone has non-empty path, source, command.
3. Build a path matcher for zones.
4. For files inside generated zones:
   - require generated/do-not-edit marker where policy applies
   - forbid direct repair instruction that patches the generated file
   - route fix to source and command
5. For contract source changes:
   - require generated output diff or proof artifact
   - if reproducibility data is absent, emit advisory/high depending on policy
6. For suspected hand edits:
   - if checksum exists and mismatch is unexplained, emit high
   - if checksum does not exist, emit advisory or high based on path criticality
```

Finding text standard:

```text
problem:
"generated file appears edited outside declared source regeneration"

reason:
"generated zones are not product truth; patching output hides drift and makes future regeneration unsafe"

agent_fix:
"edit `{source}`, run `{command}`, then commit regenerated output and proof artifact"
```

Acceptance criteria:

```text
- Generated findings always point to source and command.
- Generated-zone manifest parse failures are routed to HLT-002 or manifest observability rule.
- Hand-edit checks stay advisory until reproducibility metadata exists, unless the repo declares checksums.
- Public API contract changes without generated output/proof produce a contract/generation finding.
```

```bash

cargo test -p jankurai --test contract_source_smoke
```

```rust
pub fn build_dependency_graph(ctx: &AuditContext) -> AnalyzerResult<DependencyGraph> {
    let mut graph = DependencyGraph::default();
    let mut diagnostics = Vec::new();

    for file in product_code_files(ctx) {
        match file.suffix.as_str() {
            ".rs" => parse_rust_imports(&file.rel_path, &file.text, &mut graph, &mut diagnostics),
            ".ts" | ".tsx" => parse_typescript_imports(&file.rel_path, &file.text, &mut graph, &mut diagnostics),
            _ => {}
        }
    }

    AnalyzerResult { value: graph, diagnostics }
}
```

```rust
let graph_result = ast::build_dependency_graph(ctx);

output.rule_hits.extend(boundary::rust_domain_hits(ctx, manifest, &graph_result.value));
output.rule_hits.extend(boundary::typescript_web_hits(ctx, manifest, &graph_result.value));
```

```text
Option A: Keep HLT-006 for DB wrong-layer only, add new HLT rule for boundary import impurity.
Option B: Use HLT-007 for handwritten/contract boundary drift and HLT-006 only for direct DB.
Option C: Add HLT-022-BOUNDARY-IMPORT-IMPURITY with docs and tests.
```

# The highest-leverage next changes

The first five changes I would make are:

```text
1. Add RuleSpec.cap_impacts, RuleSpec.confidence, and RuleSpec.detector.
2. Add RuleHit and AnalyzerOutput.
3. Change FindingBuilder to add_rule_hit(RuleHit) and make hard findings require rule metadata.
4. Expand BoundaryManifest to parse Rust/TypeScript/Python policy sections.
5. Refactor AST pilot to build graph once and emit all boundary import hits.
```

That sequence closes the real architectural hole: today Phase 02 has a registry, analyzers, caps, scans, and a finding builder, but they are not yet a single hardened rule engine. Once those five are done, generated-zone, fixture, and compatibility hardening become straightforward rather than messy.
