## Phase 02 readout: what is already there, and what is missing

Phase 02 is **real but not finished**. The repo already has the right bones for a rule engine: a Rust audit CLI, a `rules.rs` registry, analyzer modules, scan helpers, finding enrichment, repair queue generation, JSON/Markdown/SARIF/JUnit-style outputs, and tests that prove some smoke-level behavior. The project layout confirms that `crates/jankurai/` owns the Rust audit CLI and proof/router logic, `packages/ux-qa/` owns rendered UX geometry/accessibility checks, and `agent/` owns control-plane manifests like owner map, test map, generated zones, proof lanes, and version bindings. ([GitHub][1])

But Phase 02 is not yet “hardened.” Today, much of the semantic layer is still pattern-based, fixture coverage is smoke-oriented, fingerprints are not sufficiently deterministic, owner routing has nondeterministic edge cases, rule metadata is not rich enough to be the single operational source of truth, and the AST pilot is narrow. The correct goal for Phase 02 is not “add more regexes.” The goal is to turn the audit engine into a **deterministic, versioned, rule-backed semantic oracle** that emits stable findings, explains why each finding exists, routes each finding to the smallest credible proof lane, and feeds repair safely.

The load-bearing contract should be:

> Every emitted finding must be traceable to a registered rule, a standard clause, a semantic fact, an evidence source, an owner, a proof lane, a repair policy, and a deterministic fingerprint.

That is the finish line.

# 1. Current Phase 02 inventory

## 1.1 Rule registry exists, but it is metadata-thin

The current registry defines `RuleSpec` with fields like `id`, `name`, `category`, `tlr`, `lane`, `docs_url`, `owner_hint`, `evidence_kind`, `severity`, `repairable`, `repair_eligibility`, `repair_risk`, and `repair_reason`. The current rules cover `HLT-001` through `HLT-021`, including future-hostile markers, generated mutation risk, ownerless paths, unmapped proof, Python product truth, direct DB access, handwritten contracts, false-green risk, generated security, secret sprawl, prompt injection, overbroad agency, rendered UX gaps, a11y gaps, context setup, supply-chain drift, opaque observability, perf/concurrency drift, streaming runtime drift, CI hardening, and destructive migrations. ([GitHub][2])

That is a good foundation. The missing piece is that the registry is not yet a full executable contract. Each rule needs enough metadata to drive docs, schemas, test fixtures, proof routing, repair eligibility, suppression policy, SARIF metadata, and CI gates.

## 1.2 Analyzer pipeline exists, but semantic depth is uneven

The analyzer layer currently includes dimensions for ownership, contracts, proof, security, shape, data, observability, context, Python, and speed. UX readiness is also inferred from web-surface indicators, Storybook/Playwright/a11y/layout artifacts, API mocks, design tokens, and proof artifacts. ([GitHub][3])

This is the right shape for Phase 02. The issue is that many checks still operate as broad heuristics. For example, the shape analyzer scores code size, duplicate blocks, TODO/stub/fallback/future-hostile markers, weak naming, and domain I/O hits. ([GitHub][4]) That is useful, but a hardened semantic oracle needs to know **what layer a file belongs to**, **what symbol or import caused the violation**, **whether an exception exists**, **whether the exception is expired**, and **which proof lane must be run**.

## 1.3 Finding enrichment exists, but needs determinism and stronger provenance

`FindingBuilder` already enriches findings with severity, category, path, problem, fix, evidence, rule ID, TLR, lane, docs URL, owner, line, rerun command, evidence kind, confidence, and hardness. It also maps categories to default TLRs and proof lanes. ([GitHub][5])

This is one of the best parts of the current implementation. The hardening problems are:

1. **Fingerprinting uses `DefaultHasher`**, which is not the right long-term choice for stable, cross-run, cross-version report identity.
2. Owner matching reads `owner-map.json` into a `HashMap`, then iterates entries. That makes overlapping prefixes nondeterministic unless the map is explicitly normalized and sorted.
3. `matched_term` and `reason` exist in the model, but the current builder path does not consistently populate them.
4. Findings can fall back to `HLT-000-SCORE-DIMENSION`, but that rule is not part of the visible registry. Either register it or stop emitting it as a rule-backed finding.
5. There is not yet a formal distinction between “pattern hit,” “semantic confirmation,” “policy violation,” “manifest violation,” and “proof coverage failure.”

## 1.4 Scan layer exists, but it is mostly text-first

The scan layer includes helpers for pattern hits, TODO/fallback/future-hostile markers, secrets, duplicate blocks, direct DB markers, manifest parse errors, CI hardening, and other repository-shape findings. It uses Aho-Corasick and regex-based matching in several places. ([GitHub][6])

That is fine as the **fast prefilter**. It should not be the final semantic oracle. The hardened architecture should keep text scanning for speed, then validate high-risk findings with typed facts: AST imports, dependency graph edges, manifest declarations, generated-zone provenance, SQL operation classification, workflow permissions, proof-lane coverage, and exception state.

## 1.5 AST pilot exists, but it is too narrow

There is already an AST pilot that parses Rust and TypeScript imports and builds dependency edges. The tests show that it can catch domain impurity such as Rust domain code importing forbidden I/O/DB modules, and TypeScript UI code importing backend modules. ([GitHub][7])

That is exactly the right direction. Phase 02 needs to expand this pilot into the real semantic oracle.

## 1.6 Tests exist, but they are mostly smoke-level

There are useful tests for rule registry uniqueness, rule metadata presence, specific HLT registrations, report serialization, low-dimension findings, owner/test-map authority, self-audit behavior, streaming runtime drift, registered rule IDs, and AST pilot behavior. ([GitHub][8])

The missing piece is a **rule corpus**: positive, negative, edge, exception, generated/vendor/docs allowlist, changed-scope, and self-audit fixtures for every rule.

# 2. Definition of “100% done” for Phase 02

Phase 02 is finished when the rule engine and semantic oracle satisfy all of these:

## 2.1 Rule completeness

Every rule has:

| Requirement        | Done means                                                                          |
| ------------------ | ----------------------------------------------------------------------------------- |
| Registered ID      | Rule appears in the canonical registry.                                             |
| Standard clause    | Rule maps to a clause in `agent-native-standard.md` / `agent/JANKURAI_STANDARD.md`. |
| Category           | Category is stable and documented.                                                  |
| TLR                | TLR is explicit and not inferred silently.                                          |
| Minimal proof lane | Rule declares the smallest credible lane.                                           |
| Strict proof lane  | Rule declares escalation lane if fast proof is insufficient.                        |
| Evidence kind      | Evidence type is explicit.                                                          |
| Severity           | Severity is calibrated and tested.                                                  |
| Confidence model   | Rule defines how confidence is assigned.                                            |
| Repair policy      | Auto-safe, agent-assisted, human-required, or never-auto.                           |
| Repair risk        | Low, medium, high, critical.                                                        |
| Suppression policy | Whether suppressions are allowed, how scoped, and expiration rules.                 |
| Fixture coverage   | Positive, negative, edge, exception, generated/vendor/docs, changed-scope.          |
| SARIF metadata     | Rule title, docs URL, help text, precision, tags.                                   |
| Report metadata    | Rule version, standard version, evidence, owner, proof lane.                        |

## 2.2 Finding completeness

Every emitted finding must include:

| Field                     | Why it matters                                                        |
| ------------------------- | --------------------------------------------------------------------- |
| `rule_id`                 | No unregistered findings.                                             |
| `rule_version`            | Supports rule evolution.                                              |
| `standard_clause`         | Connects to Phase 01.                                                 |
| `path`                    | Localizes the issue.                                                  |
| `line` / `span`           | Makes it actionable.                                                  |
| `semantic_subject`        | E.g. import edge, SQL operation, workflow permission, generated file. |
| `trigger_kind`            | Pattern, AST, manifest, graph, proof, artifact, policy.               |
| `matched_term` or fact ID | Shows why it fired.                                                   |
| `reason`                  | Human-readable causal explanation.                                    |
| `evidence_kind`           | Explains evidence source.                                             |
| `owner`                   | Routes work.                                                          |
| `lane`                    | Routes proof.                                                         |
| `rerun_command`           | Tells agent/human what to run.                                        |
| `repair_eligibility`      | Controls automation.                                                  |
| `repair_risk`             | Controls blast radius.                                                |
| `fingerprint`             | Stable across runs unless the issue materially changes.               |
| `confidence`              | Quantifies uncertainty.                                               |
| `blocking`                | Whether it gates pass/fail.                                           |

## 2.3 Semantic oracle completeness

The oracle must be able to answer:

1. What files exist?
2. Which files are authored, generated, vendored, docs-only, test-only, or self-audit internal?
3. What owner owns each path?
4. What proof lane covers each path?
5. What tests cover each path?
6. What generated zone produced each file?
7. What imports cross layer boundaries?
8. What runtime clients are used, and in which layer?
9. What SQL operations exist, and are they destructive?
10. What workflows exist, and are permissions hardened?
11. What secrets, tokens, or secret-like assignments exist?
12. What exceptions exist, who owns them, when do they expire, and what residual risk remains?
13. What proof receipt proves or fails to prove the finding?
14. What repair is allowed, and what repair is forbidden?

## 2.4 CI completeness

The normal verification path should include at least:

```bash
cargo fmt --check
cargo clippy -p jankurai --all-targets -- -D warnings
cargo test -p jankurai
cargo test -p jankurai --test rule_corpus
cargo run -p jankurai -- rules verify
cargo run -p jankurai -- audit . --json target/jankurai/score.json --md target/jankurai/score.md
cargo run -p jankurai -- compat
just fast
just score
just security-strict
just ux-qa
```

The repo already has `just fast`, `just score`, `just ux-qa`, `just check`, `just compat`, `just self-audit`, and security-related commands, so Phase 02 should wire into those rather than inventing a disconnected verification surface. ([GitHub][9])

# 3. The Phase 02 finish plan

## Workstream A — Turn the rule registry into the executable contract

Current `RuleSpec` is useful, but it needs to become the canonical source for everything Phase 02 emits.

### A1. Extend `RuleSpec`

Add fields like:

```rust
pub struct RuleSpec {
    pub id: &'static str,
    pub version: &'static str,
    pub status: RuleStatus,
    pub phase: &'static str,

    pub name: &'static str,
    pub category: &'static str,
    pub standard_clause: &'static str,
    pub tlr: &'static str,

    pub severity: &'static str,
    pub confidence_floor: f32,
    pub precision: RulePrecision,

    pub minimal_lane: &'static str,
    pub escalation_lane: Option<&'static str>,
    pub evidence_kind: &'static str,
    pub expected_artifacts: &'static [&'static str],

    pub owner_hint: &'static str,
    pub docs_url: &'static str,

    pub repairable: bool,
    pub repair_eligibility: RepairEligibility,
    pub repair_risk: RepairRisk,
    pub repair_reason: &'static str,
    pub repair_template: Option<&'static str>,

    pub suppression: SuppressionPolicy,
    pub fixture_group: &'static str,
    pub tags: &'static [&'static str],
}
```

Add enums:

```rust
pub enum RuleStatus {
    Active,
    Experimental,
    Deprecated,
}

pub enum RulePrecision {
    Exact,
    High,
    Medium,
    Heuristic,
}

pub enum SuppressionPolicy {
    Forbidden,
    ManifestOnlyExpiring,
    InlineAllowedForLowRisk,
}
```

### A2. Add a registry verification command

Add:

```bash
jankurai rules verify
```

It should fail if:

* Any rule ID is duplicated.
* Any rule is missing docs.
* Any rule is missing a standard clause.
* Any rule has an unknown lane.
* Any rule has an unknown category.
* Any rule has an unknown TLR.
* Any high/critical rule allows unsafe auto-repair.
* Any rule lacks positive/negative fixtures.
* Any rule has a docs URL but no generated docs page.
* Any emitted finding references an unregistered rule.
* Any registered rule is unreachable from the engine.

### A3. Generate docs from the registry

Generate:

```text
docs/rules/HLT-001-DEAD-MARKER.md
docs/rules/HLT-002-GENERATED-MUTATION.md
...
docs/rules/index.md
```

Each page should include:

* Rule ID
* Name
* Standard clause
* Why it matters
* What triggers it
* What does not trigger it
* Examples
* Minimal proof lane
* Repair policy
* Suppression policy
* Fixtures
* SARIF metadata

This makes the registry the operational source of truth, not a Rust-only lookup table.

### A4. Fix `HLT-000`

Today, `FindingBuilder` can fall back to `HLT-000-SCORE-DIMENSION`. That must be resolved.

Pick one:

**Option 1: Register it.**

```rust
RuleSpec {
    id: "HLT-000-SCORE-DIMENSION",
    name: "Low scoring audit dimension",
    category: "audit",
    ...
}
```

**Option 2: Do not treat it as a rule.**

Make score-dimension findings explicitly non-rule findings with a separate field:

```rust
finding.source = FindingSource::DimensionScore;
finding.rule_id = None;
```

For hardening, I would pick **Option 1** only if score-dimension findings are meant to drive repair and proof routing. Otherwise pick **Option 2** and require all hard findings to be rule-backed.

## Workstream B — Make finding identity deterministic and stable

### B1. Replace `DefaultHasher` with SHA-256

Current finding fingerprints are not strong enough for long-lived evidence. Use SHA-256 and canonical JSON.

Canonical fingerprint input:

```json
{
  "schema": "jankurai.finding-fingerprint.v1",
  "rule_id": "HLT-006-DIRECT-DB-WRONG-LAYER",
  "rule_version": "1.0.0",
  "path": "src/domain/order.rs",
  "span": {
    "start_line": 12,
    "end_line": 12
  },
  "semantic_subject": "import:sqlx",
  "trigger_kind": "ast.import",
  "problem_code": "db_client_in_domain_layer"
}
```

Then:

```rust
sha256(canonical_json).hex()
```

Do **not** include volatile fields like generated timestamp, evidence ordering, absolute repo root, elapsed time, dirty worktree state, or natural-language problem text.

### B2. Add source spans

Move from:

```rust
line: Option<usize>
```

to:

```rust
pub struct SourceSpan {
    pub start_line: usize,
    pub start_col: Option<usize>,
    pub end_line: usize,
    pub end_col: Option<usize>,
}
```

Keep `line` for backward compatibility if needed, but add `span` for hardened reports.

### B3. Populate `matched_term` and `reason`

The scan layer already has `matched_term` in `FindingHit`. The builder path should preserve it.

For example:

```json
{
  "matched_term": "sqlx",
  "reason": "Domain-layer file imports a DB client; DB clients are only allowed in adapter/repository layers."
}
```

This turns findings from “the scanner says no” into “the oracle can explain causality.”

### B4. Make ordering deterministic

Before output:

* Sort findings by severity rank, rule ID, path, line, fingerprint.
* Sort evidence lists.
* Sort owner/test/proof mappings before resolution.
* Sort manifest-derived facts.
* Sort repair queue items by deterministic tuple.

This matters for stable reports, stable diffs, and agent trust.

## Workstream C — Fix owner/test/proof map resolution

### C1. Longest-prefix owner matching

Current owner resolution should not depend on map iteration order. Implement:

```rust
let mut entries = owner_map.entries();
entries.sort_by(|a, b| {
    b.prefix.len()
        .cmp(&a.prefix.len())
        .then(a.prefix.cmp(&b.prefix))
});
```

Then match longest prefix first.

### C2. Normalize all paths

Every manifest path and file path should go through one canonical normalizer:

* Convert backslashes to slashes.
* Remove `./`.
* Reject `../`.
* Normalize trailing slash semantics.
* Treat `src` and `src/` consistently.
* Reject absolute paths in manifests unless explicitly allowed.
* Reject duplicate normalized entries.

### C3. Validate overlapping owner prefixes

Overlaps are fine only if deterministic.

Example:

```json
{
  "src/": "app-team",
  "src/payments/": "payments-team"
}
```

This is valid if longest-prefix matching is documented.

But this should warn or fail:

```json
{
  "src/payments": "payments-team",
  "src/payments/": "other-payments-team"
}
```

### C4. Apply the same logic to test maps and proof lanes

Owner routing, test routing, and proof routing should use one shared path-matching module.

Create:

```rust
crates/jankurai/src/audit/routing.rs
```

With:

```rust
pub struct RouteMatch<'a> {
    pub prefix: &'a str,
    pub value: &'a str,
    pub specificity: usize,
}

pub fn longest_prefix_match<'a, T>(
    path: &str,
    routes: &'a [Route<T>],
) -> Option<RouteMatch<'a>>;
```

This avoids three different routing semantics across owner map, test map, and proof lanes.

### D4. Make rules consume facts, not raw files

Define:

```rust
pub trait RuleExecutor {
    fn spec(&self) -> &'static RuleSpec;
    fn evaluate(&self, ctx: &OracleContext) -> Vec<FindingDraft>;
}
```

Then each rule becomes:

```rust
pub struct Hlt006DirectDbWrongLayer;

impl RuleExecutor for Hlt006DirectDbWrongLayer {
    fn spec(&self) -> &'static RuleSpec {
        rules::lookup("HLT-006-DIRECT-DB-WRONG-LAYER").unwrap()
    }

    fn evaluate(&self, ctx: &OracleContext) -> Vec<FindingDraft> {
        // Use ImportFact, SqlOperationFact, FileFact.layer, exceptions.
    }
}
```

This separates **fact extraction** from **rule judgment**.

# 4. Per-rule hardening plan

## HLT-001 — Dead marker / future-hostile language

### Current likely behavior

Text scanning detects terms like TODO, FIXME, fallback, placeholder, temporary, hack, shim, old, legacy, and related terms. ([GitHub][6])

### Problems

* Too easy to false-positive product copy.
* Too easy to false-positive legitimate docs.
* Too hard to distinguish “legacy migration table” from “we left legacy garbage.”
* Matched terms need context and allowlist provenance.

### Finish work

Implement:

1. Word-boundary and case-aware matching.
2. Context windows around matches.
3. Role-aware filtering:

   * Product code: strict.
   * Docs: mostly ignored unless docs are operational instructions.
   * Test fixtures: lower severity.
   * Product copy/i18n: ignored unless it appears in code comments or control files.
4. Explicit exception support:

   * `agent/exceptions.toml`
   * owner
   * reason
   * expiration
   * scope
5. Confidence scoring:

   * `TODO` in product code comment: medium.
   * `temporary workaround` in runtime code: high.
   * `legacy` in table name: low unless paired with risk words.
6. Positive/negative fixtures:

   * Positive: `// TODO remove this fallback later`
   * Negative: `LegacyCustomer` domain type
   * Negative: docs explaining legacy migration policy
   * Edge: i18n copy containing “temporary password”

### Done when

* No unactionable future-hostile findings in docs/product copy.
* Every emitted finding includes matched term and reason.
* Fixtures prove false-positive controls.

## HLT-002 — Generated zone mutation risk

### Finish work

Generated-zone checks should move from “path looks generated” to manifest-backed provenance.

Add facts:

```rust
GeneratedZoneFact {
    path_prefix,
    generator,
    source,
    allow_manual_edits,
}
```

Rules:

1. If authored changes touch generated zone and `allow_manual_edits = false`, emit `HLT-002`.
2. If generated files exist without a manifest entry, emit generated-zone manifest gap.
3. If generated-zone source is missing, emit source/provenance gap.
4. If generated-zone artifacts are stale relative to source hash, emit drift finding.
5. If generated security artifacts are manually edited, escalate to `HLT-009`.

### Done when

* Generated files are not only detected; they are explainable.
* The finding says which generator owns the file.
* The proof route points to the generation or contract lane.

---

## HLT-003 — Ownerless path

### Current behavior

The audit can detect ownerless paths and tests assert that owner/test maps are authoritative. ([GitHub][10])

### Finish work

1. Use deterministic longest-prefix matching.
2. Require all product, config, workflow, manifest, generated-source, and migration paths to have owners.
3. Allow docs-only paths to have lower severity unless the doc is a standard/control-plane doc.
4. Fail overlapping ambiguous owner map entries.
5. Add changed-scope mode:

   * If only changed files are audited, every changed path must resolve to owner.
6. Add owner validity:

   * Owner must match known owner registry or declared owner string pattern.
   * No empty owner.
   * No `unknown`, `todo`, `tbd`.

### Done when

* Owner routing is deterministic.
* Every finding has owner or explicit “owner missing because HLT-003.”
* Fixture corpus covers overlapping prefixes and path normalization.

---

## HLT-004 — Unmapped proof

### Current behavior

The audit already treats proof lanes/test routing as core repo quality and emits findings when proof mapping is missing. ([GitHub][11])

### Finish work

This is where Phase 02 must connect directly to Phase 03.

For each path:

1. Resolve owner.
2. Resolve test map.
3. Resolve proof lane.
4. Verify lane command exists.
5. Verify lane command is runnable from repo root.
6. Verify lane has expected artifacts.
7. Verify lane can appear in `ProofReceipt.rules_covered`.
8. Verify proof lane is the smallest credible lane for the associated rule.

Add a rule-lane matrix:

```toml
[rules.HLT-004-UNMAPPED-PROOF]
minimal_lane = "fast"
strict_lane = "check"
expected_artifacts = ["json-report", "md-report"]
```

### Done when

* No hard finding can exist without a proof lane.
* Every high/critical finding has a rerun command.
* Proof receipts can claim rule coverage and the audit can validate that claim.

## HLT-005 — Python product truth

### Finish work

Detect Python product-truth risk with more nuance.

Rules:

1. Python under `tools/`, `scripts/`, `experiments/`, or `ai/` may be allowed with owner and proof.
2. Python under product domain, API, DB, or business-logic paths is high risk unless standard explicitly allows it.
3. Python code importing DB clients, HTTP servers, model/business packages, or production config should escalate.
4. Python notebooks are allowed only under experiments/docs with no product-truth authority.
5. Python-generated artifacts must map to generator/source.

Facts needed:

* Python import facts.
* Python file role.
* Python path layer.
* Python package metadata.
* Python execution lane.

### Done when

* Python as tooling is allowed.
* Python as hidden product truth is blocked.
* Fixtures cover scripts, notebooks, generated code, and product logic.

## HLT-006 — Direct DB wrong layer

### Current behavior

There are text-based DB marker checks for SQL keywords and DB libraries in suspicious layers. ([GitHub][6])

### Finish work

Move this to semantic detection.

Facts:

* Import facts for DB clients:

  * Rust: `sqlx`, `diesel`, `tokio_postgres`, `rusqlite`
  * TS/JS: `pg`, `mysql2`, `prisma`, `drizzle`, `knex`
  * Python: `psycopg`, `sqlite3`, `sqlalchemy`
* Raw SQL string facts.
* File layer facts.
* Adapter/repository exceptions.

Rules:

1. DB clients are allowed only in `adapter`, `repository`, `migration`, or explicitly declared data-access layers.
2. UI/domain/application layers cannot call DB directly.
3. Raw SQL in product code must be classified:

   * Read query
   * Write query
   * Destructive query
   * Migration operation
4. Query construction through string concatenation escalates severity.
5. Direct DB calls in tests are allowed only in integration test lanes.

### Done when

* The finding says exactly which import/query caused the violation.
* False positives on docs and migrations are gone.
* DB layer rules are tested across Rust, TS, Python, and SQL fixtures.

## HLT-007 — Handwritten contract

### Current behavior

The audit detects contract/boundary integrity gaps and generated contract drift risks. ([GitHub][11])

### Finish work

Add a contract oracle.

Supported contract types:

* OpenAPI
* GraphQL schema
* Protobuf
* JSON Schema
* TypeScript generated clients
* Rust generated API bindings
* DB migration schema outputs
* Event schemas

Rules:

1. Public API contract exists.
2. Generated clients are derived from contract.
3. Generated clients are in generated zones.
4. Manual edits to generated clients are forbidden.
5. Drift proof exists:

   * contract source hash
   * generated artifact hash
   * proof command
6. Runtime API code does not define hidden contract truth outside canonical schema.

### Done when

* Contract drift is based on source/artifact provenance, not just path presence.
* Public boundary changes route to contract lane.
* Handwritten contract truth is blocked.

## HLT-008 — False-green risk

### Finish work

False-green risk should detect tests that appear to pass while proving little.

Detect:

* `#[ignore]`, `#[should_panic]` misuse, empty tests.

## 5.2 Evaluation flow

The hardened flow should be:

```text
inventory repo
  ↓

  ↓

  ↓

  ↓

  ↓

  ↓

TextMatchFact {

    surrounding_context_hash,
}
```

Rules then decide whether a text match matters.

This avoids the common mistake where a scanner directly emits a high-severity finding without semantic validation.

# 6. Proof-routing integration for Phase 02

Phase 02 cannot be finished independently of Phase 03. The rule engine must produce proof-routing inputs.

Every rule should declare:

```rust
pub struct ProofRequirement {
    pub minimal_lane: &'static str,
    pub escalation_lane: Option<&'static str>,
    pub expected_commands: &'static [&'static str],
    pub expected_artifacts: &'static [&'static str],
    pub rules_covered_key: &'static str,
}
```

Every finding should include:

```json
{
  "lane": "security",
  "rerun_command": "just security",
  "proof_requirement": {
    "minimal_lane": "security",
    "expected_artifacts": ["security-evidence.json", "jankurai-report.json"]
  }
}
```

Every proof receipt should be validated against:

* Lane exists.
* Command matches allowed command or declared equivalent.
* Exit code is acceptable.
* Artifact paths exist.
* Artifact hashes match.
* `rules_covered` includes the finding rule ID.
* Changed paths match or dominate finding path.
* Receipt is fresh enough.
* Receipt was generated under matching standard version.

The existing model already has `ProofReceipt` fields like lane, command, exit code, elapsed time, artifacts, changed paths, owner, skipped reason, residual risk, log path, receipt path, generated time, repo root, git head, run ID, plan path, rules covered, retryable, and stdout/stderr byte count. ([GitHub][13]) Phase 02 should lean into that instead of treating proof routing as separate post-processing.

# 7. Exception and suppression hardening

No high-risk rule should be suppressible through casual inline comments.

## 7.1 Add structured exceptions

Use something like:

```toml
[[exceptions]]
id = "EXC-2026-001"
rule_id = "HLT-019-STREAMING-RUNTIME-DRIFT"
path = "src/runtime/kafka.rs"
owner = "platform"
reason = "Temporary runtime adapter while moving event publishing behind adapter boundary"
expires = "2026-06-30"
residual_risk = "Domain still indirectly depends on Kafka event schema"
proof_lane = "db"
migration_path = "docs/migrations/kafka-adapter-plan.md"
```

## 7.2 Exception rules

* Critical rules: suppressions forbidden unless rule explicitly permits.
* High rules: manifest-only, expiring, owner-required.
* Medium/low rules: may allow scoped inline suppression if standard permits.
* Expired exceptions become findings.
* Exceptions must be fingerprinted.
* Exceptions must show in reports.
* Exception scope must be narrower than or equal to finding scope.
* Exception owner must match owner map or approved override.

## 7.3 No silent suppression

Every suppressed finding should produce a suppressed-finding record:

```json
{
  "rule_id": "HLT-019-STREAMING-RUNTIME-DRIFT",
  "fingerprint": "...",
  "suppressed": true,
  "exception_id": "EXC-2026-001",
  "expires": "2026-06-30"
}
```

This is crucial for agent trust.

# 8. Repair queue hardening

The rule engine feeds repair. Therefore findings must be safe for automated action.

## 8.1 Repair ranking

Rank repair items by:

1. Blocking status.
2. Severity.
3. Repair eligibility.
4. Repair risk.
5. Confidence.
6. Blast radius.
7. Owner.
8. Proof lane cost.
9. Dependency order.

Example priority:

```text
critical never-auto finding
  → create human task, do not patch

high agent-assisted owner/test/proof map gap
  → agent may draft manifest update

medium context gap
  → agent may patch docs or AGENTS.md with bounded template

low style/context warning
  → optional repair
```

## 8.2 Repair templates

Every repairable rule should point to a repair template.

Examples:

```text
repair_templates/
  HLT-003-ownerless-path.md
  HLT-004-unmapped-proof.md
  HLT-015-context-setup-gap.md
  HLT-016-supply-chain-drift.md
```

Templates should include:

* What can be changed.
* What cannot be changed.
* Required proof command.
* Rollback guidance.
* Human review trigger.
* Example patch shape.

## 8.3 Never auto-repair these by default

* Secrets
* Destructive migrations
* Prompt-injection policy
* Overbroad agency
* Generated security
* DB layer migrations
* Contract truth rewrites
* Anything with critical risk

The current registry already distinguishes auto-safe, agent-assisted, human-required, and never-auto repair eligibility. ([GitHub][2]) Harden Phase 02 by making that policy enforceable, not merely descriptive.

1. Every emitted hard finding has a registered `HLT-*` rule.
2. Every registered rule has standard clause metadata.
3. Every registered rule has proof-lane metadata.
4. Every registered rule has repair policy metadata.
5. Every registered rule has suppression policy metadata.
6. Every registered rule has positive and negative fixtures.
7. Every high/critical rule has exception fixtures.
8. Every finding has stable SHA-256 fingerprint.
9. Every finding has deterministic owner routing.
10. Every finding has deterministic proof-lane routing.
11. Every finding has evidence kind and trigger kind.
12. Every finding has a reason, not just a problem string.
13. Every finding has source span where possible.
14. Every secret finding redacts sensitive content.
15. Every generated-zone finding references generator/source provenance.
16. Every DB-layer finding references an import/query/layer fact.
17. Every CI finding references workflow/job/permission/action.
18. Every migration finding classifies the destructive operation.
19. Every UX/a11y finding references missing or present artifacts.
20. Every exception is owner-scoped and expiring.
21. Expired exceptions fail.
22. Suppressed findings remain visible in report.
23. Repair queue respects never-auto/human-required policy.
24. SARIF contains rule metadata and source spans.
25. JSON schema validates.

27. Report output is stable across repeated runs.
28. Rule corpus passes.
29. Self-audit passes.

# 15. The core hardening principle

Phase 02 should stop thinking in terms of “find bad strings” and start thinking in terms of **facts → rules → findings → proof → repair**.

The final architecture should look like this:

```text
facts:
  src/domain/order.rs imports sqlx
  src/domain/order.rs is domain layer
  sqlx is DB client
  domain layer cannot import DB client

rule:
  HLT-006-DIRECT-DB-WRONG-LAYER

finding:
  domain file imports DB client
  severity high
  owner platform
  lane db
  repair human-required
  fingerprint sha256(...)

proof:
  run db lane
  collect receipt
  verify rules_covered includes HLT-006

repair:
  move DB access to adapter/repository
  update proof map if needed
  rerun lane
```

That is the 100% bar for Phase 02. Not “more checks.” Not “more warnings.” A deterministic semantic oracle that can explain, route, prove, and safely feed repair.
