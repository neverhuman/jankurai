I could not complete a normal `git clone` in the container because DNS resolution to GitHub failed, so I inspected the repository through GitHub’s rendered/raw files and downloaded key raw source files for local formatting/review. My read: **Phase 02 is already useful and partially complete, but it is not yet hardened to “100% done.”** The path to finish it is to convert the current centralized heuristic scanner into a registry-driven semantic rule pipeline with fixture-backed proof for every rule, every cap, and every emitted finding.

## Phase 02 verdict

**Current status:** the phase router calls Phase 02 complete, and the feedback ledger says the current architecture satisfies the initial feedback because the repo already has `audit/rules.rs`, finding builders, boundary checks, and registry tests. However, the Phase 02 receipt itself still says the analyzer split remains partial and AST/import support is a pilot. That means this phase should be treated as **complete enough to proceed**, but **not hardened enough to freeze**. ([GitHub][1])

**What “done” should mean:** Phase 02’s objective is not to build a compiler-grade static analyzer. The stated target is a versioned rule engine with semantic understanding, where inventory flows through analyzers into a rule registry, findings, caps, score, and reports. The acceptance criteria explicitly call for stable HLT rules, semantic boundary checks, generated-zone checks, an AST/import pilot, richer finding metadata, tests, and docs/rule index coverage. ([GitHub][2])

So the goal should be:

> **Finish Phase 02 by making every rule, finding, cap, boundary decision, generated-zone decision, and semantic import decision traceable, fixture-tested, registry-backed, and schema-compatible.**

# 2. The hardening gaps to close

These are the concrete blockers preventing Phase 02 from being “100% done.”

## Gap A — The rule registry does not fully own caps

Right now the rule registry owns HLT rule metadata, while caps and score effects are still substantially separate. The Phase 02 target architecture says rules should feed findings, findings should feed caps/score, and the report should flow from that. ([GitHub][2])

**Problem:** a cap can still exist as a scoring/report concept without being first-class registry metadata.

**Finish condition:** every hard cap has a stable rule/cap mapping, and no high-impact report decision can happen without a registry-backed explanation.

Add to `RuleSpec` or a sibling `CapSpec`:

```rust
pub struct RuleSpec {
    pub id: &'static str,
    pub name: &'static str,
    pub category: &'static str,
    pub tlr: &'static str,
    pub lane: &'static str,
    pub docs_url: &'static str,
    pub owner_hint: &'static str,
    pub evidence_kind: EvidenceKind,
    pub severity: Severity,
    pub repairable: bool,
    pub repair_eligibility: RepairEligibility,
    pub repair_risk: RepairRisk,
    pub repair_reason: &'static str,

    // Add these:
    pub status: RuleStatus,              // Stable, Experimental, Deprecated
    pub standard_section: &'static str,  // e.g. "Test Lanes", "Boundaries"
    pub cap: Option<CapSpec>,
    pub confidence_policy: ConfidencePolicy,
    pub analyzer: &'static str,
}
```

And:

```rust
pub struct CapSpec {
    pub id: &'static str,
    pub max_score: Option<u16>,
    pub hardness: Hardness,
    pub reason: &'static str,
}
```

Acceptance criteria:

* Every emitted `Finding.rule_id` exists in `RULES`.
* Every hard cap in report output maps to either a `RuleSpec.cap` or an explicit legacy compatibility exception.
* Every `RuleSpec` has a docs target.
* Every rule has a fixture.
* Every cap has a fixture.
* Deprecated rules remain resolvable for old reports but are not emitted by new analyzers.

## Gap B — Finding evidence is being normalized too aggressively

The standard requires findings to carry rule ID, category, severity, owner, path, line, matched term, reason, problem, repair guidance, and proof lane context. ([GitHub][14]) The current `FindingBuilder` enriches many metadata fields, but the builder path loses some raw evidence fields such as matched term and reason by default. ([GitHub][6])

**Problem:** the report is good for routing, but not yet strong enough for forensic-quality repair because some analyzer evidence is flattened.

**Finish condition:** every finding is created from a typed draft that preserves evidence, source span, matched term, reason, and semantic context.

Introduce:

```rust
pub struct FindingDraft {
    pub rule_id: &'static str,
    pub check_id: &'static str,
    pub severity_override: Option<Severity>,
    pub path: Option<PathBuf>,
    pub line: Option<u32>,
    pub evidence: Vec<FindingEvidence>,
    pub problem: String,
    pub repair: String,
    pub analyzer: &'static str,
}

pub struct FindingEvidence {
    pub kind: EvidenceKind,
    pub matched_term: Option<String>,
    pub reason: Option<String>,
    pub source_span: Option<SourceSpan>,
    pub snippet_hash: Option<String>,
    pub semantic_context: Option<String>,
}
```

Then make `FindingBuilder` materialize a `Finding` from `FindingDraft + RuleSpec`.

Acceptance criteria:

* High/critical findings always include owner, lane, rerun command, docs URL, evidence kind, repair eligibility, repair risk, and agent fix metadata.
* Pattern findings preserve matched term and reason.
* Semantic findings preserve import edge, boundary edge, generated-zone edge, or migration edge.
* Fingerprints are stable across irrelevant line shifts when possible.
* Fingerprints do not depend on volatile prose ordering.

## Gap C — Analyzer architecture is still too centralized

The Phase 02 target architecture names a clean flow: inventory → analyzers → rule registry → findings → caps → score → reports. ([GitHub][2]) Current code has that shape at a high level, but `audit/mod.rs` still centrally wires many checks in `build_findings`, while `scan.rs` holds many unrelated detectors. ([GitHub][7])

**Problem:** central wiring makes it harder to prove coverage, test individual rules, add analyzers, or reason about false positives.

**Finish condition:** each analyzer owns its rules and emits `FindingDraft`s; the central audit pipeline only orchestrates.

Add a trait like:

```rust
pub trait RuleAnalyzer {
    fn name(&self) -> &'static str;
    fn rules(&self) -> &'static [&'static str];
    fn analyze(&self, ctx: &AuditContext, inventory: &Inventory) -> Vec<FindingDraft>;
}
```

Create analyzer modules:

```text
crates/jankurai/src/audit/analyzers/
  rule.rs
  boundary_oracle.rs
  generated_zone_oracle.rs
  import_graph.rs
  contract_oracle.rs
  migration_oracle.rs
  ci_oracle.rs
  proof_oracle.rs
  prompt_oracle.rs
  security_oracle.rs
  shape_oracle.rs
  python_oracle.rs
```

Refactor in this order:

1. Add `FindingDraft` without changing output.
2. Wrap existing `scan.rs` functions in analyzer modules.
3. Move one rule family at a time out of `build_findings`.
4. Keep old report shape stable.
5. Delete or shrink central `build_findings` only after parity tests pass.

Acceptance criteria:

* `run_audit_with_options` no longer knows individual rule checks.
* It only runs registered analyzers, materializes findings, applies caps, scores, and reports.
* Each analyzer declares the rule IDs it can emit.
* A test asserts that every stable rule has at least one analyzer or an explicit non-emitted/manual status.
* A test asserts that no analyzer emits undeclared rules.

## Gap D — Boundary oracle is not manifest-driven enough

The boundary oracle doc says violations should map to owner, proof lane, rerun command, evidence kind, and repair queue item. ([GitHub][15]) The code has boundary modules for Rust, TypeScript, Python, SQL, and streaming, with static forbidden markers and import policies. ([GitHub][16])

The repo also has a richer `agent/boundaries.toml` with Rust, TypeScript, Python, queues, DB, and streaming policy. ([GitHub][12])

**Problem:** some boundary policy exists in TOML, but parts of the analyzer still rely on hard-coded constants and partial manifest parsing.

**Finish condition:** `agent/boundaries.toml` becomes the source of truth for boundary rules, with static code defaults only used as fallback.

Extend `BoundaryManifest`:

```rust
pub struct BoundaryManifest {
    pub stack: StackBoundary,
    pub rust: Option<RustBoundary>,
    pub typescript: Option<TypeScriptBoundary>,
    pub python: Option<PythonBoundary>,
    pub queues: QueueBoundary,
    pub db: DbBoundary,
    pub streaming_exception: Option<StreamingException>,
}
```

Add typed subcontracts:

```rust
pub struct RustBoundary {
    pub domain_roots: Vec<String>,
    pub forbidden_domain_imports: Vec<String>,
    pub allowed_adapter_roots: Vec<String>,
}

pub struct TypeScriptBoundary {
    pub web_roots: Vec<String>,
    pub forbidden_web_imports: Vec<String>,
    pub generated_contract_roots: Vec<String>,
    pub handwritten_contract_roots: Vec<String>,
}

pub struct PythonBoundary {
    pub allowed_roots: Vec<String>,
    pub forbidden_truth_markers: Vec<String>,
}
```

Boundary checks to harden:

| Check                  | Required behavior                                                                             |
| ---------------------- | --------------------------------------------------------------------------------------------- |
| Rust domain IO         | Domain paths cannot import DB/network/filesystem/HTTP adapters unless manifest allows it.     |
| TypeScript web imports | Web code cannot import DB/backend/adapters directly.                                          |
| Contract source        | Handwritten public API DTOs must be rejected unless generated-zone relationship is declared.  |
| Python truth leakage   | Python cannot own product truth unless path is allowlisted.                                   |
| DB wrong layer         | DB access must route through declared DB roots or adapters.                                   |
| Streaming exception    | Runtime drift is allowed only with owner, reason, classification, expiry, and migration path. |
| Queue/event contracts  | Queue clients must map to event contract paths.                                               |
| Expired exceptions     | Expired exceptions become hard findings.                                                      |

Acceptance criteria:

* Boundary manifest schema covers all parsed fields.
* Unknown fields fail in strict validation mode.
* Expired exceptions fail deterministically.
* All boundary findings preserve boundary edge evidence.
* All boundary findings include owner, lane, rerun command, and repair queue metadata.
* Fixture tests cover positive, negative, allowlisted, generated, vendor/reference, and expired-exception cases.

## Gap E — Generated-zone oracle needs source/output semantics

The standard and artifact contracts treat generated surfaces as durable repo contracts, with generated artifacts, schemas, owner maps, test maps, proof receipts, and audit outputs all expected to stay compatible. ([GitHub][17]) Existing scans detect generated-zone metadata issues, missing headers, missing files, orphan contract sources, and related contract problems. ([GitHub][8])

**Problem:** the oracle checks symptoms, but not a complete source → command → output relationship.

**Finish condition:** every generated zone has a declared source, generator command, output set, edit policy, and reproducibility proof.

Add a generated-zone graph:

```rust
pub struct GeneratedZone {
    pub id: String,
    pub owner: String,
    pub source_paths: Vec<PathBuf>,
    pub output_paths: Vec<PathBuf>,
    pub generator_command: String,
    pub proof_lane: String,
    pub edit_policy: EditPolicy,
}
```

Generated-zone checks:

1. Declared outputs exist.
2. Declared outputs contain required generated header.
3. Generated files are not manually modified in changed-scope mode.
4. Source changes route to generator/proof lane.
5. Public API contract changes require generated output proof.
6. Orphan generated outputs are flagged.
7. Orphan handwritten public contract sources are flagged.
8. Generator command is present and maps to test lane.
9. Optional: output content hash is stored or reproducibility proof is attached.

Acceptance criteria:

* A change to a contract source without regenerated output creates a finding.
* A generated file edited by hand creates a finding.
* A missing generated header creates a finding.
* A generated output with no declared source creates a finding.
* A declared output missing from disk creates a finding.
* A declared generated zone with an invalid command creates a finding.
* All findings route to the smallest credible lane.

## Gap F — AST/import graph is still a pilot

The AST pilot currently uses regexes to parse Rust and TypeScript imports and checks a small set of forbidden import relationships. ([GitHub][9]) That is enough for a pilot, but not enough to call the semantic oracle hardened.

**Problem:** regex-only import parsing will miss common forms and can overflag or underflag.

**Finish condition:** the import graph is parser-light, deterministic, cached, and good enough to support boundary decisions without pretending to be a full compiler.

Phase the AST/import hardening carefully.

### F1. First pass: robust parser-light graph

Support:

**Rust**

* `use foo::bar`
* `use foo::{bar, baz}`
* `use crate::...`
* `use super::...`
* `mod x`
* `pub mod x`
* package names from `Cargo.toml`

**TypeScript**

* `import x from "..."`
* `import { x } from "..."`
* `export { x } from "..."`
* `export * from "..."`
* `require("...")`
* `import("...")`
* tsconfig path aliases
* relative path normalization

**Python**

* `import x`
* `from x import y`
* DB/HTTP/product-truth marker linkage
* allowlisted roots from boundary manifest

**SQL**

* destructive operation markers
* migration transaction/rollback hints
* tenant/RLS/constraint hints where applicable

### F2. Cache the semantic graph

Create:

```rust
pub struct SemanticGraph {
    pub imports: Vec<ImportEdge>,
    pub db_edges: Vec<DbEdge>,
    pub generated_edges: Vec<GeneratedEdge>,
    pub boundary_edges: Vec<BoundaryEdge>,
    pub parse_errors: Vec<ParseError>,
}
```

Run it once per audit, then pass it to analyzers.

This also fixes the current inefficiency where the AST pilot path can be called more than once in the central finding build.

### F3. Parser failures must not crash the audit

Rules:

* Parse failure in one file becomes advisory evidence.
* Parse failure does not hide simpler pattern checks.
* Analyzer confidence drops when semantic parse is partial.
* Report includes parse coverage summary.

Acceptance criteria:

* Import graph tests cover Rust nested imports, TypeScript re-exports, dynamic import, require, path aliases, Python imports, and malformed files.
* Boundary oracle consumes graph edges, not raw regex hits.
* Semantic parse coverage appears in audit diagnostics.
* No new dependency is added until timing impact is measured.
* If a parser crate is added later, it must be behind measured performance and compatibility tests.

# 5. Merge order

Do not merge the semantic pieces before the contract pieces. The safest order is:

1. **Baseline + tests only**
2. **Registry metadata expansion**
3. **FindingDraft/evidence preservation**
4. **Analyzer trait with no behavior change**
5. **Boundary manifest parser**
6. **Boundary oracle behavior**
7. **Generated-zone oracle**
8. **Import graph upgrade**
9. **Cap/score provenance**
10. **Fixture matrix completion**
11. **Docs/rule index/final receipt**

This avoids the dangerous failure mode where analyzers start emitting new semantic findings before the report, registry, schema, and repair contracts can explain them.

# 9. The main risks

## Risk 1 — Breaking report compatibility

The audit report is a durable artifact. The repo already has artifact contracts and compatibility guards for reports, schemas, proof receipts, standard status, audit boundaries, security, UX, and related outputs. ([GitHub][17])

Mitigation:

* Add new fields as optional first.
* Preserve existing field names.
* Keep legacy cap IDs.
* Run `report_compatibility_guard`.
* Run `schema_contracts`.
* Only tighten schema after compatibility tests prove consumers are safe.

## Risk 2 — Overbuilding AST

The Phase 02 doc explicitly says the target is not compiler-grade static analysis. ([GitHub][2])

Mitigation:

* Build a parser-light semantic graph first.
* Add real parser dependencies only if false positives/negatives justify them.
* Measure audit time before and after.
* Treat parse failures as advisory.

## Risk 3 — Too many false positives

Boundary and generated-zone checks can become noisy if docs/reference/vendor/generated paths are not excluded correctly.

Mitigation:

* Add allowlist fixtures.
* Add generated/vendor/reference fixtures.
* Sort findings deterministically.
* Emit top-N findings per rule with summary counts rather than flooding.

## Risk 4 — Central scan code remains forever

If the analyzer split is not completed, Phase 02 will continue to look done but behave like a pile of checks.

Mitigation:

* Require every stable rule to name an analyzer.
* Require every analyzer to declare emitted rules.
* Add a test that fails if new direct rule emission appears outside analyzer modules or the builder.

## Risk 5 — Proof routing drift

Phase 02 findings must route to the smallest credible proof lane. That depends on the Phase 03 proof router and the standard’s lane contract. The standard requires path changes to route through the test map and agents to run the smallest mapped lane. ([GitHub][18])

Mitigation:

* Do not invent lanes inside analyzers.
* Lanes come from `RuleSpec`.
* Rerun commands come from lane mapping.
* Tests assert lane/rerun on every high/critical finding.

# 10. Final “100% done” definition

I would not mark Phase 02 hardened until this exact statement is true:

> For every stable HLT rule, Jankurai has registry metadata, analyzer ownership, fixture coverage, docs coverage, finding evidence preservation, repair metadata, proof-lane routing, cap/score provenance where applicable, and compatibility-tested report output. Boundary decisions are manifest-driven. Generated-zone decisions are source/output/generator-driven. Import decisions come from a cached semantic graph. All of it passes focused, crate-wide, compatibility, and lane validation.

That is the finish line. Current Phase 02 is a solid base; the hardening work is primarily about **making the rule engine provable, not merely functional**.
