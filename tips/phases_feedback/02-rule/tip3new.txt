## Phase 02 readout: treat it as “complete baseline, not hardened”

I was not able to do a full local `git clone` in the container because GitHub DNS resolution failed, but I inspected the repository through GitHub/raw file views. Based on that, Phase 02 is already meaningfully started: the repo has `crates/jankurai/src/audit/`, `audit/rules.rs`, `audit/finding_builder.rs`, `audit/caps.rs`, `audit/scan.rs`, an `audit/analyzers/` tree, and a `boundaries/` tree. The repo README positions Jankurai as an agent-native control plane with the loop `intent -> bounded agents -> proof lanes -> evidence -> expired exceptions -> reusable primitives`, and the Rust audit CLI is explicitly the audit/proof/router core. ([GitHub][1])

The Phase 02 document says the objective is to move from mostly heuristic scanning to a versioned rule engine with semantic understanding, where the audit can explain why a pattern violates ownership, boundary, generated-zone, contract, DB, security, UX, or agent-context policy. It also says the intended pipeline is `inventory -> analyzers -> rule registry -> findings -> caps -> score -> reports`. Importantly, the phase receipt says there is still residual work: analyzer split and AST/import pilot remain follow-on work, even though the phase header says “complete.” ([GitHub][2])

So my recommendation is: **do not call Phase 02 hardened yet. Call it “functional baseline,” then land a dedicated hardening wave that locks correctness, rule coverage, semantic evidence, report compatibility, and proof routing integration.**

The target is **not** to build a perfect compiler-grade verifier. The Phase 02 doc explicitly says the exit state is not compiler-grade; the goal is a clean rule architecture that supports AST and graph checks incrementally without breaking current audit behavior. ([GitHub][2])

# What “100% done and hardened” should mean for Phase 02

Phase 02 is finished only when these are all true:

1. **Every emitted finding is rule-backed.** No finding may leave the audit with an unknown `rule_id`, missing lane, missing rerun command, missing owner hint where ownership can be inferred, or vague repair text.

2. **The rule registry is the source of truth.** `HLT-*` metadata should define rule identity, category, TLR, severity, proof lane, docs URL, evidence kind, repair eligibility, repair risk, cap impact, fingerprint behavior, and proof expectations.

3. **Heuristics become named analyzers.** `audit/scan.rs` can remain as a low-level detector library, but the audit flow should not be “giant scanner soup.” Each semantic family should have its own analyzer with fixtures and acceptance tests.

4. **Semantic oracles must explain boundary violations.** Wrong-layer DB access, generated-zone hand edits, Python product-truth leakage, Rust domain impurity, TypeScript UI/backend imports, streaming runtime leakage, destructive migration risk, security/prompt risks, and false-green tests must produce explainable, stable findings.

5. **Proof Router compatibility is locked.** Since Phase 03 depends on Phase 02, every high/critical finding must route to the smallest credible proof lane: `fast`, `contract`, `db`, `db-migration-analyze`, `web`, `e2e`, `security`, `observability`, `audit`, or `release`. The standard defines those proof lanes and requires audit outputs with score fields, caps, dimensions, findings, and ordered fix queue. ([GitHub][3])

6. **The standard and registry cannot drift.** `agent/JANKURAI_STANDARD.md`, `docs/agent-native-standard.md`, `audit/rules.rs`, schemas, rendered docs, and tests must agree on rule IDs, proof lanes, output shape, and hard blocks. The docs require core manifests such as `AGENTS.md`, owner map, test map, generated zones, standard version, repo score JSON, validation commands, and CI audit on PR. ([GitHub][4])

7. **The report contract is backward-compatible.** Current tests already validate repo-score schemas, SARIF/JUnit/JSONL exports, and semantic report compatibility. Harden Phase 02 without breaking those surfaces. ([GitHub][5])

# Current code reality

The good news: Phase 02 already has the right skeleton.

`audit/rules.rs` has a real rule registry with `RuleSpec`, repair eligibility, repair risk, docs URLs, categories, TLR values, lanes, and registered rules from `HLT-001` through `HLT-021`. `audit/finding_builder.rs` already enriches findings with metadata from the registry: rule lookup, lane selection, owner inference, evidence kind, stable fingerprint, rerun command, docs URL, severity, category, and TLR. That is exactly the correct choke point. ([GitHub][7])

The analyzer module currently returns dimension results through `all_dimensions(ctx)`, while semantic rule-hit production still appears partially split across `scan.rs`, `finding_builder.rs`, and the AST pilot. The boundaries layer has useful pieces: `agent/boundaries.toml` ingestion, stack/queue/DB/streaming exception fields, Rust forbidden imports, TypeScript forbidden imports, Python product-truth path logic, SQL destructive markers, and streaming client markers. The tests are also nontrivial: there are audit smoke tests, boundary manifest tests, generated-zone tests, contract-source tests, migration safety tests, proof surface tests, report compatibility guards, schema contract tests, security/UX ingest tests, and rule registry smoke tests. Rule registry tests currently check uniqueness, specific registration for `HLT-014-A11Y-GAP` and `HLT-021-DESTRUCTIVE-MIGRATION`, and repair policy metadata. ([GitHub][5])

## 4. Harden the Boundary Oracle

This is the most important Phase 02 workstream.

The boundary oracle should answer:

```text
What boundary was crossed?
What policy says it is forbidden?
Where exactly is the crossing?
Who owns the path?
Which proof lane proves the repair?
What is the smallest safe repair?
Is there a valid exception?
When does that exception expire?
```

### 4.1 Boundary manifest hardening

Current `BoundaryManifest` supports stack, queues, DB, and streaming exceptions. Expand it into a normalized policy model. ([GitHub][11])

Recommended `agent/boundaries.toml` shape:

```toml
[stack]
id = "rust-web-postgres"
version = "0.1.0"

[db]
root_paths = ["crates/adapters/db/", "db/"]
migration_paths = ["db/migrations/", "migrations/"]
constraint_paths = ["db/constraints/"]

[web]
ui_paths = ["apps/web/", "frontend/"]
forbidden_imports = ["pg", "postgres", "better-sqlite3", "mysql2", "@aws-sdk/client-s3"]
allowed_api_clients = ["apps/web/src/api/", "packages/api-client/"]

[rust]
domain_paths = ["crates/domain/", "domain/"]
adapter_paths = ["crates/adapters/", "adapters/"]
forbidden_domain_imports = ["std::fs", "std::env", "std::net", "sqlx", "diesel", "reqwest", "rdkafka"]

[python]
product_truth_allowed_paths = ["python/ai-service/", "tools/"]
product_truth_forbidden_paths = ["apps/", "frontend/", "docs/marketing/"]

[queues]
adapter_paths = ["crates/adapters/queue/"]
event_contract_paths = ["contracts/events/"]
generated_type_paths = ["crates/generated/events/"]
client_markers = ["rdkafka", "kafkajs", "nats", "redis::streams"]

[[streaming_exception]]
runtime = "kafka"
classification = "brownfield"
reason = "legacy runtime under migration"
owner = "platform"
expires = "2026-06-01"
migration_path = "docs/migrations/kafka-to-adapter.md"
```

### 4.3 Rust boundary oracle

Current Rust logic detects forbidden imports such as `std::fs`, `std::env`, `std::net`, `std::time`, `rand`, `sqlx`, `diesel`, `reqwest`, `rdkafka`, `tracing`, and `log` for domain contexts. ([GitHub][12])

Harden it:

```text
Domain path imports forbidden IO/DB/runtime module -> HLT-006 or boundary-specific rule
Domain path calls env/fs/net/time/random directly -> boundary violation
Domain path imports tracing/log directly -> observability boundary violation, unless standard allows instrumentation
Adapter path imports are allowed
Test path imports are allowed only if test-map lane includes integration/e2e
```

Add cases:

```rust
use std::{fs, env};
use std::fs::{File, read_to_string};
use sqlx::PgPool;
use diesel::prelude::*;
use reqwest::Client;
use rdkafka::producer::FutureProducer;
```

Make the evidence precise:

```json
{
  "rule_id": "HLT-006-DIRECT-DB-WRONG-LAYER",
  "path": "crates/domain/src/order.rs",
  "line": 3,
  "matched_term": "sqlx::PgPool",
  "reason": "domain path crates/domain/ imports DB runtime sqlx; DB access is only allowed in crates/adapters/db/",
  "agent_fix": "move SQL access to crates/adapters/db and inject a repository trait into the domain layer",
  "lane": "db"
}
```

### 4.4 TypeScript boundary oracle

Current TypeScript logic has forbidden web imports such as `pg`, `postgres`, `better-sqlite3`, `mysql2`, and `@aws-sdk/client-s3`, plus backend import checks. ([GitHub][13])

Harden it:

```text

frontend imports adapters/db
UI imports DB clients
UI imports server-only env/config
UI mutates product truth directly instead of API client
UI imports generated server types from wrong generated zone
```

Do not build a full TypeScript compiler. Start with shallow alias extraction.

### 4.6 SQL/destructive migration oracle

Current SQL logic detects destructive markers such as drop table/column/database/schema/index/constraint, truncate, delete from, and alter table patterns. ([GitHub][15])

Harden it into two levels:

**Level A: static SQL detector**

Detect:

```sql

DROP COLUMN
DROP DATABASE
DROP SCHEMA
TRUNCATE
DELETE FROM table;
ALTER TABLE table DROP ...
UPDATE table SET ...; -- without WHERE for risky tables
```

**Level B: migration safety evidence**

Require at least two of:

```text

backfill plan
lock timeout
statement timeout
advisory lock
expand-contract staging
preflight row-count query
foreign-key/index safety note

```

For destructive migration findings, lane must be `db-migration-analyze`, matching the standard’s proof-lane set and the current `HLT-021-DESTRUCTIVE-MIGRATION` lane. ([GitHub][3])

### 4.7 Streaming runtime oracle

Current streaming detection looks for runtime markers and has support for Kafka brownfield exceptions with classification, reason, owner, and migration path. ([GitHub][9])

Harden exception validation:

```text
Exception must include runtime
classification must be brownfield/migration/temporary
reason must be non-empty
owner must map to owner-map
expires must be valid date
expires must not be in the past
migration_path must exist
migration_path must mention target adapter or decommission plan
```

Finding behavior:

```text
No exception -> high
Malformed exception -> high
Expired exception -> critical or high, depending policy
Valid exception -> warning/advisory and cap note, not hard fail
```

## 5. Harden the Generated Zone Oracle

Generated-zone handling is a Phase 02 must-have because generated code is one of the standard’s hard blocks. The standard treats generated hand edits and public API drift as hard problems, and the Phase 02 doc calls out generated-zone metadata, headers, checksum/drift proof, and advisory behavior where data is incomplete. ([GitHub][3])

### Required manifest structure

Make `agent/generated-zones.toml` strict:

```toml
[[zone]]
path = "crates/generated/"
owner = "platform"
generator = "tools/generate-contracts"

header = "@generated by jankurai"
checksum = "sha256:..."
proof_lane = "contract"
drift_command = "just contract"
allow_manual_edits = false
```

### Detection rules

Detect:

```text
generated file missing required header
generated file modified but source contract unchanged
source contract changed but generated output unchanged
generator command missing
checksum missing
checksum mismatch
public API contract changed without generated output drift proof
manual edit inside generated zone
generated path has no owner
```

Generated findings need more than “file is generated.” Include:

```json
{
  "zone": "crates/generated/",
  "source": "contracts/openapi.yaml",
  "generator": "tools/generate-contracts",
  "expected_header": "@generated by jankurai",
  "actual_header": "missing",
  "checksum_status": "missing|mismatch|ok",

```

### Acceptance tests

Fixtures:

```text

generated-source-changed-output-missing/

```

```rust
pub struct DependencyGraph {
```

## 7. Harden finding quality

The finding is the product. For agent-native repair, every finding must answer:

```text
What failed?
Why does it matter?
```

## 9. Harden security, prompt, and agency rules

The standard includes hard blocks for prompt injection, overbroad agency, skipped/focused/tautological tests, secret-like content, and CI hardening. ([GitHub][3])

### Secret detector

Current secret detection already has pattern logic and exclusions. Harden it with:

```text
context-aware redaction
test fixture safe tokens
allowlisted docs/reference/vendor paths
SARIF-safe evidence snippets
critical severity for likely live secret
high severity for ambiguous token
never-auto repair eligibility
```

A finding must never print the full secret. Evidence should be redacted:

```text
AWS_ACCESS_KEY_ID=AKIA************
```

### Prompt-injection and overbroad agency

Scan only high-risk instruction surfaces:

```text
AGENTS.md
agent/*.md
.github/copilot-instructions.md
.github/workflows/*
prompt templates
agent policy files
```

Detect:

```text
ignore previous instructions
disable safety checks
edit anything
run destructive commands without approval
bypass tests
do not tell the user
```

Require:

```text
path
line
matched phrase
why this weakens agent boundaries
safe replacement guidance
security or audit lane
```

### False-green test oracle

Detect:

```text
it.skip
describe.skip
test.skip
.only
fit/fdescribe
assert true
expect(true).toBe(true)
snapshot-only tests for behavior-critical surfaces
pytest.mark.skip without reason
cargo ignored tests without reason
```

But avoid false positives in docs/reference/vendor.

Add a distinction:

```text
intentional skip with owner + expiry + reason -> advisory
unbounded skip/focus/tautology -> high
snapshot-only behavior proof -> medium/high depending surface
```

## 10. Harden ownership and proof-lane integration

Phase 02 becomes useful to Phase 03 only if findings route cleanly.

### Ownership

For every finding with a path:

1. Try `agent/owner-map.json`.
2. Try generated-zone owner.
3. Try boundary manifest owner.
4. Try rule owner hint.
5. If none, emit or attach `HLT-003-OWNERLESS-PATH`.

The standard requires owner/test/generator/version manifests as part of the start ritual and treats ownerless paths as a hard block. ([GitHub][3])

### Proof lane mapping

Make `lane_for_rule(rule_id)` registry-based only. Avoid category fallbacks except for legacy soft findings.

Proof lane acceptance:

```rust
#[test]
fn all_registered_lanes_exist_in_proof_lanes_manifest_or_standard() {}

#[test]
fn every_high_critical_finding_routes_to_smallest_lane() {}
```

Example smallest-lane mapping:

```text
HLT-001 future-hostile marker       -> fast
HLT-002 generated mutation          -> contract
HLT-003 ownerless path              -> fast
HLT-004 unmapped proof              -> fast
HLT-005 Python product truth        -> contract
HLT-006 direct DB wrong layer       -> db
HLT-007 handwritten contract drift  -> contract
HLT-008 false green test risk       -> fast
HLT-009 generated security gap      -> security
HLT-010 secret-like content         -> security
HLT-011 prompt injection risk       -> security
HLT-012 overbroad agency            -> audit
HLT-013 UX proof gap                -> web
HLT-014 accessibility proof gap     -> web
HLT-015 context/setup gap           -> fast
HLT-016 supply-chain drift          -> security
HLT-017 opaque observability        -> observability
HLT-018 performance/concurrency     -> fast or e2e, depending evidence
HLT-019 streaming runtime drift     -> contract or db, depending manifest; prefer contract for adapter boundary
HLT-020 CI workflow hardening       -> security
HLT-021 destructive migration       -> db-migration-analyze
```

```text

proof lane not declared
```
