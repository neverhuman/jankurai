I inspected the repo through GitHub rather than a local clone because the runtime could not resolve GitHub for direct `git clone`. The code and docs were still accessible enough to build a concrete Phase 02 finish plan.

## What Phase 02 needs to become

**Phase 02 – Rule Engine & Semantic Oracle** should not be “more pattern scans.” It should become the deterministic decision layer that turns the standard into trusted, versioned, explainable findings.

The repo’s own operating loop is:

```text
intent -> bounded agents -> proof lanes -> evidence -> expired exceptions -> reusable primitives
```

and the README frames Jankurai as a control plane that makes ownership, proof, generated zones, and repair evidence machine-readable so wrong code is easier to reject than merge. That makes Phase 02 the quality gate between “repo facts” and every downstream action: proof routing, repair routing, audit scoring, exceptions, and no-proof-no-merge governance. ([GitHub][1])

The standard currently says the repository is optimized for agents to reject wrong code quickly, localize failures, repair narrow scopes, and leave auditable evidence; it also says agent-native repos must make ownership, generated files, proof commands, actionable failures, size limits, dependencies, fallbacks, and duplicates machine-obvious. ([GitHub][2]) Phase 02 is the implementation of that promise.

My recommended target is:

> **A typed, versioned rule engine backed by a semantic repo oracle, with one canonical rule registry, deterministic evidence, rule-specific fixtures, redaction, drift checks, and CI gates proving every stable rule is implemented, documented, tested, and routed to the correct proof lane.**

No scanner can guarantee absence of every possible semantic bug, so “100% done” should mean **contract-complete against the Jankurai standard**: every stable rule has a detector or explicit exception policy, every detector has positive/negative/adversarial fixtures, every finding is evidence-backed and repair-routable, and code/docs/schema cannot drift silently.

## Current state: strong start, not yet hardened

### What is already good

The repo already has the right skeleton:

1. **A Rust audit CLI and proof/router surface** live under `crates/jankurai/`; `agent/` contains owner map, test map, generated-zone manifest, proof lanes, and version bindings; `schemas/` contains the machine contracts; and the canonical audit lane emits JSON and Markdown. ([GitHub][1])

2. **The standard defines HL3+ adoption requirements**, including `AGENTS.md`, `agent/JANKURAI_STANDARD.md`, owner map, test map, generated-zone manifest, standard-version binding, CI-produced repo score, fast/full validation commands, and audit on every PR. ([GitHub][2])

3. **The standard has stable rule IDs HLT-001 through HLT-019** covering dead markers, generated mutation, ownerless paths, unmapped proof, Python product truth, wrong-layer DB access, handwritten contracts, false-green tests, generated security, secrets, prompt injection, overbroad agency, rendered UX, a11y, setup/context, supply-chain drift, observability, perf/concurrency, and streaming runtime drift. ([GitHub][2])

4. **The code already has a `RuleSpec` registry** with fields for rule ID, name, category, top-level risk, lane, docs URL, owner hint, evidence kind, severity, repairability, repair eligibility, repair risk, and repair reason. That is exactly the right direction. ([GitHub][3])

5. **The rule registry currently includes HLT-020 and HLT-021** for CI hardening and destructive migrations, in addition to HLT-001 through HLT-019. That is useful, but it creates an immediate code/standard drift issue because the public standard’s stable rule table stops at HLT-019. ([GitHub][3])

6. **The analyzer layout is already dimension-oriented**: ownership, contracts, proof, security, shape, data, observability, context, Python, and speed analyzers are wired into `all_dimensions`. This maps cleanly to the audit rubric’s scoring dimensions. ([GitHub][4])

7. **There are already meaningful scanner functions** for secret-like content, prompt-injection patterns, overbroad agency patterns, false-green tests, generated-zone metadata, generated-file markers, wrong-layer DB hints, duplicate blocks, future-hostile language, destructive SQL migrations, and streaming runtime drift. ([GitHub][5])

8. **There is already a good test surface**, including audit smoke tests, proof surface smoke tests, rule registry smoke tests, schema contract tests, security evidence tests, UX QA tests, migration safety tests, and Phase 13 repair-safety tests. ([GitHub][6])

### Biggest gaps to close

The main gap is not that Phase 02 lacks ideas. The gap is that it is not yet **closed, canonical, adversarially tested, and semantically precise**.

The hard gaps are:

| Gap                                                                           | Why it matters                                                                                                                                                                                                                                                                                                                                                                               |
| ----------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Code/standard drift**: registry has HLT-020/021, standard lists HLT-001–019 | Phase 02 cannot be trusted if the rule catalog is not the single source of truth.                                                                                                                                                                                                                                                                                                            |
| **Finding schema is still too thin**                                          | Current finding schema requires severity, category, path, problem, agent_fix, evidence, check_id, hardness, confidence, evidence_kind, rerun_command, and fingerprint, with optional rule_id/owner/lane; for hardening, rule ID, owner, lane, docs, detector version, redaction state, exception linkage, and evidence spans should become mandatory for stable-rule findings. # The architecture to land

## Layer 1: Inventory

Purpose: collect files and raw manifest inputs.

Outputs:

```text
FileInventory
ManifestInventory
GitDiffInventory
CommandInventory
```

Hardening requirements:

* Normalize all paths to forward-slash repo-relative paths.
* Sort deterministically.
* Ignore `.git`, build outputs, vendored deps, and generated artifacts according to explicit policy, not hidden conventions.
* Track why a file was ignored.
* Support full-repo and changed-path modes.
* Never read huge/binary files as text without limits.

## Layer 2: Semantic Oracle

Purpose: compute facts once.

This is the “Semantic Oracle” proper. It should answer questions like:

```text
Who owns this path?
Which proof lane proves it?
Is it generated?
What source regenerates it?
Is this product/runtime code?
Is this a test?
Is this an adapter?
Does this import DB, HTTP, queues, env, filesystem, unsafe, secret material?
Does this define public contract surface?
Does this test assert behavior or merely execute?
Does this UI path have rendered evidence?
Does this migration have rollback/backfill/lock evidence?
Is this exception expired?
```

The oracle should not output “findings.” It outputs facts. The rule engine turns facts into findings.

## Layer 3: Rule Engine

Purpose: apply stable rule specs to oracle facts.

Recommended trait:

```rust
pub trait RuleDetector {
    fn id(&self) -> &'static str;
    fn version(&self) -> &'static str;
    fn rule(&self) -> RuleId;
    fn detect(&self, facts: &RepoFacts) -> Vec<RuleEvidence>;
}
```

Then a central evaluator converts evidence into findings:

```rust
pub struct RuleEvidence {
    pub rule_id: RuleId,
    pub detector_id: String,
    pub detector_version: String,
    pub subject: EvidenceSubject,
    pub reason: String,
    pub confidence: f32,
    pub matched: Option<MatchedValue>,
    pub related_paths: Vec<String>,
    pub raw_evidence: Vec<EvidenceItem>,
}
```

## Layer 4: Finding Builder

Purpose: attach rule metadata, owner, lane, docs, repair text, severity, cap, fingerprint, and exception status.

This layer should be boring and deterministic.

## Layer 5: Report Builder

Purpose: score dimensions, hard caps, fix queue, receipts, Markdown, JSON, and future SARIF.

The standard requires audit JSON to include standard version, auditor version, schema version, paper edition, target stack ID, raw score, final score, hard caps, dimensions, findings, and ordered `agent_fix_queue`. ([GitHub][2])

## P1 — Harden the finding contract

**Goal:** findings become safe, complete, machine-routable evidence objects.

The current finding schema has the right base fields, but it should be upgraded so stable-rule findings cannot omit critical routing information. ([GitHub][7])

### Add or require these fields

```json
{
  "rule_id": "HLT-010-SECRET-SPRAWL",
  "rule_version": "0.4.1",
  "detector_id": "secret.lexical.v2",
  "detector_version": "2.0.0",
  "top_level_risk": "Security, secrets, agency",
  "owner": "ops",
  "lane": "security",
  "docs_url": "docs/audit-rubric.md#top-level-risk-mapping",
  "line": 42,
  "column": 10,
  "span": {
    "start_line": 42,
    "start_column": 10,
    "end_line": 42,
    "end_column": 31
  },
  "redaction_state": "redacted",
  "exception_status": "none",
  "related_paths": [],
  "input_fingerprint": "...",
  "finding_fingerprint": "..."
}
```

### Redaction policy

For secrets and token-like content:

```text
Never store full token.
Never emit full token in Markdown.
Never put full token in fingerprint.
Fingerprint should use redacted token class + path + span + salted hash.
Markdown evidence should show class, not value.
JSON evidence should show redacted preview only.
```

Example:

```text
BAD:
evidence: ["GITHUB_TOKEN=ghp_abcd..."]

GOOD:
evidence: ["secret-like material: github_pat/ghp token at .env:12; value redacted"]
```

### Acceptance gate

* Every high/critical finding includes stable `rule_id`, owner, lane, docs URL, evidence kind, rerun command, and fingerprint.
* Secret-like findings never expose raw secret material.
* Schema contract tests fail if required fields disappear.

Perf/concurrency drift is naturally harder. Start with advisory + high-confidence cases:

blocking calls in async context
unbounded retry
unbounded channel/queue
cache eviction changes
N+1 query smell
timeouts removed

mutex/lock changes

Require:

```text

backpressure policy

```

### Family E: maintainability, context, observability

Rules:

```text
HLT-001-DEAD-MARKER
HLT-015-CONTEXT-SETUP-GAP
HLT-017-OPAQUE-OBSERVABILITY
```

#### HLT-001 finish tasks

Current future-hostile detection scans product code and uses allowlisted prefixes/product-copy path parts. ([GitHub][5])

Harden it:

* Use token-boundary matching, not raw substring matching.
* Avoid false positives for substrings such as `template`, `older`, `gold`, etc.
* Classify terms:

```text
deadness: legacy, deprecated, old
temporariness: temporary, workaround, shim
proof debt: TODO, FIXME, HACK, placeholder, stub, dummy
fallback risk: fallback, best effort, try again
```

* Require owner + expiration for allowed TODO-like terms.
* Allow quoted product copy/docs/generated/vendor.
* Emit matched term, reason class, and exact fix.
* Add fixtures for product code, generated code, docs, and product copy.

#### HLT-015 finish tasks

Context/setup detector should validate:

```text
AGENTS.md exists and is concise
agent/JANKURAI_STANDARD.md exists
standard-version exists and matches expected schema
one fast command exists
one full command exists
owner map parses
test map parses
generated zones parse
proof lanes parse
local agent rules do not contradict root standard
```

#### HLT-017 finish tasks

The standard’s agent-friendly exception contract requires stable fields such as name, purpose, reason code, message, common fixes, docs URL, owner, retryable, severity, correlation ID, source, and contract version, and forbids string-only errors, catch-all reason loss, production panic/unwrap/expect without invariant, Python bare except, TypeScript throw string, and HTTP 500 without stable payload. ([GitHub][2])

Detector should scan for:

```text
Rust unwrap/expect/panic in product path
Rust anyhow/string errors crossing boundary
TypeScript throw string
catch (e) { ... } without reason mapping
Python bare except
HTTP 500 generic response
console/log-only failure
retry without reason_code
missing correlation/request ID
```

# Fixture and test strategy

The existing test directory already has smoke and contract coverage, but Phase 02 needs rule-completeness fixtures. ([GitHub][6])

Create:

```text
crates/jankurai/tests/fixtures/phase_02/
  HLT-001-DEAD-MARKER/
    positive/
    negative/
    allowlisted/
    adversarial/
    golden.json
  HLT-002-GENERATED-MUTATION/
  ...
  HLT-021-DESTRUCTIVE-MIGRATION/
```

## Required fixture types per rule

### Positive fixture

Must trigger exactly the intended rule.

Example:

```text
apps/web/src/page.tsx contains direct DB URL
=> HLT-006 only
```

### Negative fixture

Similar code that should not trigger.

Example:

```text
crates/adapters/src/db/orders.rs uses sqlx
=> no HLT-006
```

### Allowlisted fixture

A valid exception should suppress or lower severity.

Example:

```text
docs/exceptions/kafka-brownfield.toml
=> HLT-019 suppressed until expiry
```

### Adversarial fixture

Designed to catch false negatives.

Example:

```text
multi-line ALTER TABLE ... DROP COLUMN
=> HLT-021
```

## Golden report tests

Each rule should have a locked expected report:

```text
rule_id
severity
path
line
problem
evidence
owner
lane
docs_url
agent_fix
fingerprint pattern
```

Normalize timestamps and paths so goldens are deterministic.

## Property tests

Add property tests for:

```text
path normalization
glob resolution
owner-map precedence
test-map precedence
finding fingerprint stability
redaction never leaking raw value
duplicate detection stability
rule registry uniqueness
rule ID format
```

## Metamorphic tests

These are especially useful for semantic oracle hardening:

| Mutation                                     | Expected result                                         |
| -------------------------------------------- | ------------------------------------------------------- |
| Move DB code from adapter to UI              | HLT-006 appears                                         |
| Move DB code back to adapter                 | HLT-006 disappears                                      |
| Add generated header/source/command          | HLT-002 disappears                                      |
| Remove test-map lane                         | HLT-004 appears                                         |
| Add valid exception with future expiry       | finding suppressed or downgraded                        |
| Change exception expiry to past              | finding reappears                                       |
| Add semantic assertion to snapshot-only test | HLT-008 disappears                                      |
| Add rollback/backfill evidence to migration  | HLT-021 disappears only if structured evidence is valid |

## Mutation tests

Build a tiny mutator that injects known violations into clean fixtures:

```text
insert TODO
insert direct DB import
insert test.skip
insert fake secret
insert prompt-injection text
insert workflow permissions: write-all
insert generated file without marker
insert destructive SQL
```

Every injected violation must be caught.

## Redaction tests

For secret-like values:

```text
JSON output must not contain raw token.
Markdown output must not contain raw token.
Fingerprint must not contain raw token.
Problem text must not contain raw token.
Evidence must contain class and location only.
```

# Specific hardening traps I would fix immediately

## 1. Registry/standard mismatch

The Rust registry includes HLT-020 and HLT-021, while the standard’s stable list ends at HLT-019. ([GitHub][3])

**Fix:** either mark 020/021 advisory and outside stable list, or add them to the standard. I recommend adding them.

## 2. Secret scanning skip policy

The current secret scanner skips docs, paper, reference, and tips. ([GitHub][5]) But the standard says secret-like values, env dumps, fixtures, and transcripts are part of secret-sprawl risk. ([GitHub][2])

**Fix:** scan docs/artifacts with different classification and stronger allowlists, not blanket skip.

## 3. Generated-zone parse failures

Generated-zone metadata parsing currently returns no findings on read/parse failure. ([GitHub][5])

**Fix:** parse failure must be a finding because malformed generated-zone policy is equivalent to missing policy.

## 4. Wrong-layer DB is too path/string-based

Current wrong-layer DB detection uses selected prefixes and simple text markers. ([GitHub][5])

**Fix:** route through boundary facts and AST/import facts.

## 5. Future-hostile matching can be noisy

Current future-hostile detection uses lowercase substring containment. ([GitHub][5])

**Fix:** token-aware matching, term classes, and allowlisted contexts.

## 6. Destructive SQL needs statement assembly

Current destructive migration detection is line-oriented, though it has good early safety markers and lookahead for `DELETE ... WHERE`. ([GitHub][5])

**Fix:** assemble SQL statements and validate structured migration receipts, not keyword-only safety evidence.

# Final Phase 02 acceptance checklist

Phase 02 is done when this checklist is all green:

```text
[ ] Rule catalog is canonical.
[ ] HLT-001 through HLT-021 are either all stable or advisory status is explicit.
[ ] docs/agent-native-standard.md, docs/audit-rubric.md, schemas, and Rust registry cannot drift.
[ ] Every stable rule has detector(s).
[ ] Every stable rule has positive, negative, allowlisted, and adversarial fixtures.
[ ] Every stable rule has golden JSON/Markdown output.
[ ] Every finding includes rule_id, severity, category, path, problem, evidence, agent_fix, owner, lane, docs_url, rerun command, and fingerprint.
[ ] Secret-like evidence is redacted everywhere.
[ ] Oracle can explain owner, lane, generated status, boundary cell, language, imports, and rule-relevant facts for any path.
[ ] Owner-map, test-map, generated-zones, proof-lanes, security policy, boundary policy, and exceptions are schema-validated.
[ ] Parse failures in control-plane files become findings.
[ ] False-positive allowlists require owner, reason, and expiry.
[ ] Expired exceptions re-enable findings.
[ ] Rule engine output is deterministic across runs.
[ ] `just phase-02` exists.
[ ] CI blocks registry/schema/docs drift.
[ ] CI blocks unredacted secret evidence.
[ ] CI blocks high/critical findings without stable rule ID.
[ ] Self-audit emits Phase 02 receipt.
[ ] Phase 03 Proof Router can consume owner/proof/oracle facts without duplicating rule logic.
```

The core move is to treat Phase 02 as **compiler infrastructure for the standard**: parse the repo into facts, apply stable rules, emit safe evidence, and prove the rule engine with fixtures. Once that is done, Phase 03 can trust the oracle instead of re-deriving ownership, risk, and proof lanes from scratch.
