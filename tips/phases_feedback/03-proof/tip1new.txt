# Phase 03: Proof Router — current state, gap analysis, and finish plan

## 0. Executive verdict

The repo already has the load-bearing pieces for Phase 03:

`jankurai lane` / `jankurai proof` build plans, `jankurai prove` can execute either a persisted plan or `--changed` / `--changed-from`, receipts and logs are written under `target/jankurai/`, an evidence index is produced, commands are allowlisted against `agent/proof-lanes.toml` plus `agent/test-map.json`, and release-mode audit can require proof receipts. The repo’s own Phase 03 document marks the phase as complete/hardened and lists validation as 93 score, 0 findings, and 13 `proof_surface_smoke` tests passing.  

The main thing left is **not basic functionality**. The remaining work is **adversarial hardening**:

1. Make path routing exact and segment-aware.
2. Move lane semantics out of command-string matching and into explicit metadata.
3. Make schemas strict enough to catch malformed plans/receipts.
4. Make evidence tamper-evident with digests for every linked artifact.
5. Add a first-class `proof verify` / gate step so release mode checks **coverage**, not merely “some receipts exist.”
6. Add property/adversarial tests for routing, stale evidence, path escapes, sibling-prefix collisions, and receipt mutation.
7. Wire CI so a PR cannot pass with stale, partial, or mismatched proof.

That is the path to declaring Phase 03 truly “100% done.”

---

# 1. What Phase 03 must guarantee

The standard says path changes must route through `agent/test-map.json`, agents must run the smallest mapped lane, and skipped lanes must be reported with reasons. It also defines the proof lane vocabulary: `fast`, `contract`, `db`, `web`, `e2e`, `security`, `observability`, `audit`, and `release`. 

For this repo’s moonshot claim, Phase 03 should guarantee this invariant:

> Given a changed scope, Jankurai deterministically selects the smallest credible proof lane set, executes only approved commands, records receipts and evidence, and can later verify that the evidence still proves the exact changed scope at the exact git revision.

That means the router is not finished merely when it emits a plan. It is finished when it can answer all of these, mechanically:

| Question                                       | Required answer                                                                             |
| ---------------------------------------------- | ------------------------------------------------------------------------------------------- |
| Which changed paths are in scope?              | Normalized, repo-relative, no traversal/root ambiguity.                                     |
| Who owns each path?                            | Exact owner route and owner name.                                                           |
| Which route matched each path?                 | Test-map route, proof lane, command, specificity score, and reason.                         |
| Why is this the smallest credible proof?       | Required lanes, skipped lanes, risk escalations, and human-approval requirements.           |
| Did the command that ran match the plan?       | Receipt command hash/ID matches plan command hash/ID.                                       |
| Did the command run at the right git revision? | Receipt/evidence `git_head` matches expected head.                                          |
| Did the artifacts remain unchanged?            | SHA-256 digest chain from plan → receipt → logs/artifacts.                                  |
| Can release mode trust it?                     | A verifier returns pass/fail based on coverage, freshness, success, and artifact integrity. |

# 2. Current implementation map

## 2.1 Planner and runner

The core Phase 03 code lives in `crates/jankurai/src/commands/proof.rs`. It defines `ProofPlanArgs`, `ProveArgs`, `ProofPlan`, `PlannedRun`, `SkippedLaneEntry`, and `ProofEvidenceIndex`; it builds proof plans from changed paths; and it executes planned runs into receipts/logs/evidence indexes. 

Important current behavior:

* `run_lane` and `run_proof` build a plan.
* `run_prove` accepts either `--plan` or `--changed` / `--changed-from`, but rejects mixing both modes.
* Changed-mode `prove` writes `proof-plan.json` and `proof-plan.md`, reloads the persisted plan, then executes it.
* The proof runner writes receipts, logs, failed receipt paths, skipped lanes, risk notes, human approval requirements, and companion artifact paths into the evidence index.
* If there are no runnable proof commands, it still writes repairable plan/evidence artifacts and then fails.
* Command execution is allowlisted unless both `--allow-unsigned-commands` and `JANKURAI_ALLOW_UNSIGNED_PROOF_COMMANDS=1` are present. 

## 2.2 Control-plane inputs

`RepoCatalog` loads the routing data from:

* `agent/owner-map.json`
* `agent/test-map.json`
* `agent/generated-zones.toml`
* `agent/proof-lanes.toml`

It exposes owner lookup, test-route lookup, proof-lane lookup, generated-zone lookup, and the proof command allowlist. 

The repo’s actual `test-map.json` already routes many important surfaces: `docs/` to `just score`, `schemas/` to Rust tests plus UX build, `crates/jankurai/` to Rust tests plus version checks, `packages/ux-qa/` to `just ux-qa`, and `db/migrations/` to the migration analysis lane. 

The actual `proof-lanes.toml` defines live lanes including `fast`, `audit`, `paper`, `security`, `full`, `adapters`, `versions-check`, `doctor`, `test-cli`, `db-migration-analyze`, `build-schemas`, `test-ux-ci`, `versions`, and `ux-qa`. 

## 2.3 Schemas and validation

There are schemas for proof plan, proof receipt, and evidence index, and the Rust validation layer has `ArtifactSchema::ProofPlan`, `ProofReceipt`, and `EvidenceIndex`, with `validation::write_json` validating artifacts before writing them.    

Current schema posture is intentionally permissive: the proof plan schema allows `planned_runs` as loose objects and allows additional properties; proof receipt and evidence index schemas also allow additional properties. That is compatibility-friendly, but not fully hardened.   

## 2.4 Tests already present

The existing `proof_surface_smoke.rs` test file is strong for a Phase 03 baseline. It covers plan equivalence, unmapped-path risk, unsigned command rejection, receipt/log writing, continue-on-error, `prove --changed`, `prove --changed-from`, invalid argument combinations, no-runnable-route behavior, and named-lane `rules_covered` behavior. 

This is excellent smoke coverage. What is missing is adversarial/property coverage and integrity verification.

# 3. The hardening gaps that matter most

## Gap 1 — path matching is too permissive

Current route matching uses this shape:

```rust
path == prefix || path.starts_with(prefix) || prefix.starts_with(path)
```

That means matching is broad and symmetric. It helps with directory-ish inputs, but it risks sibling-prefix false positives and ambiguous broad routes. For a load-bearing proof router, this is the first thing I would harden. 

Examples to defend against:

* `docs2/file.md` must not match `docs/`.
* `agentic/foo.md` must not match `agent/`.
* `crates/jankurai-extra/file.rs` must not match `crates/jankurai/`.
* Passing repo root, `.`, `./`, or an absolute path outside the repo must never create a broad proof pass.
* Passing a directory must either expand deterministically to files or be represented as a broad scope with explicit residual risk.

## Gap 2 — proof lane identity is still command-string-centered

The planner maps a test route to a named lane by looking up the test command in `proof-lanes.toml`. This works, but it makes command strings the de facto identity. That is fragile: whitespace normalization helps, but a hardened router should use stable lane IDs and command IDs.  

A mature Phase 03 should route like this:

```text
changed path -> test-map route -> lane_id -> command_spec -> receipt.command_id
```

not:

```text
changed path -> command string -> maybe lane name
```

## Gap 3 — `rules_covered` is hard-coded in Rust

`rules_covered_for_run` maps lane names like `audit`, `security`, `db`, `web`, and `ux-qa` to hard-coded HLT rule IDs, and filters IDs through `rules::lookup`. That is safe compared with guessing from command text, but it is not yet a declarative source-of-truth contract. 

The next step is to put rule coverage metadata in `proof-lanes.toml` or a related schema, then have Rust validate it against the rule registry.

## Gap 4 — schemas are too loose for “100% hardened”

The schemas currently validate the basic shape, but they allow extra fields and leave some important objects loose. For example, `planned_runs` accepts any object with additional properties, and receipt/evidence schemas permit unknown extra data.   

That is fine during rapid iteration, but the hardening bar should be:

* typed `planned_runs`
* typed `route_decisions`
* strict known fields
* explicit version bump
* negative schema tests
* migration compatibility for older artifacts

## Gap 5 — evidence index does not yet provide a full digest chain

The evidence index records important links and has a digest for `ux_qa_report_path`, but most receipts, logs, companion artifacts, and the plan itself are linked by path only. A malicious or accidental post-run mutation can currently change logs/artifacts without the evidence index proving the mutation.  

A hardened ledger needs SHA-256 for:

* canonical proof plan
* each receipt
* each log
* every expected artifact
* companion reports: UX, security, repo score, SARIF, GitHub summary, repair queue, boundaries
* control manifests: owner map, test map, proof lanes, generated zones, audit policy, standard version

## Gap 6 — audit release mode checks receipt presence, not full coverage

Release mode currently adds a high finding if no proof receipts are supplied. That is a good start, but the true gate must verify that receipts cover the exact plan, changed paths, required lanes, commands, git head, and artifact digests.  

The missing primitive is a first-class verifier:

```bash
jankurai proof verify --plan ... --evidence-index ...
```

or:

```bash
jankurai prove ... --verify
```

## Gap 7 — doctor validates only the default proof ledger shape

`doctor` validates proof receipts and evidence index when present under default `target/jankurai/proof-receipts` and `target/jankurai/evidence-index.json`, and it warns on stale git heads. That is useful, but hardened proof runs often use custom output directories. Those custom runs should also be discoverable and verifiable. 

## Gap 8 — `bash -lc` execution should become a compatibility path, not the ideal path

`execute_run` currently invokes `bash -lc <command>`. The allowlist makes this much safer, but 100% hardening should move toward structured commands: `argv`, command ID, fixed cwd, scrubbed env, timeout, and declared side-effect policy.

# 4. The target “100% done” definition

I would declare Phase 03 complete/hardened only when all of these are true:

1. **Route correctness:** every changed path has an exact, segment-aware owner/test route or an explicit blocking residual risk.
2. **Smallest credible lane:** the planner chooses the minimal lane set that covers all changed paths plus risk escalators.
3. **Deterministic plan:** the same repo state and changed scope always produces byte-stable canonical plan content except for documented volatile fields.
4. **Strict plan schema:** proof plans, runs, route decisions, skipped lanes, and expected artifacts are typed and schema-validated.
5. **Command safety:** proof commands resolve from lane IDs/command IDs, not raw ad hoc strings; shell execution is legacy-compatible but not the preferred representation.
6. **Execution safety:** commands have timeout, cwd, env policy, output capture, non-destructive defaults, and structured failure metadata.
7. **Tamper-evident evidence:** plan, receipts, logs, and artifacts have digest links.
8. **Verification gate:** a verifier can prove that evidence matches the plan, command policy, changed paths, and git head.
9. **Audit integration:** release/ratchet mode requires verified coverage, not just receipt presence.
10. **Adversarial tests:** routing/property/schema/integrity tests fail on sibling-prefix matches, stale receipts, mutated logs, invalid commands, missing artifacts, and unmapped paths.
11. **CI enforcement:** PRs produce, verify, and archive proof evidence; stale or partial evidence fails.
12. **Docs/phase receipt:** standard, testing docs, and Phase 03 receipt describe the real behavior and exact validation commands.

## Workstream A — Harden path normalization and matching

### Goal

Make it impossible for the proof router to accidentally treat a sibling path, root path, path traversal, or broad directory input as proven by the wrong route.

### Code changes

Primary files:

* `crates/jankurai/src/commands/context_data.rs`
* `crates/jankurai/src/commands/proof.rs`
* new optional module: `crates/jankurai/src/commands/proof_route.rs`
* `crates/jankurai/tests/proof_route_matching.rs`
* `crates/jankurai/tests/proof_surface_smoke.rs`

Replace the current symmetric `path_matches` logic with segment-aware matching. Current behavior lives in `context_data.rs` and uses plain `starts_with` in both directions. 

Target behavior:

```rust
enum RoutePatternKind {
    ExactFile,
    DirectoryPrefix,
}

struct RoutePattern {
    raw: String,
    normalized: String,
    kind: RoutePatternKind,
}

struct RouteMatch {
    pattern: String,
    matched_path: String,
    specificity: usize,
    kind: RoutePatternKind,
}
```

Rules:

* `docs/` matches `docs/readme.md`.
* `docs/` matches `docs` only if the input is explicitly treated as a directory scope.
* `docs/` does **not** match `docs2/readme.md`.
* `docs` as an exact file route matches only `docs`.
* `agent/MASTER_PLAN.md` beats `agent/`.
* Equal-specificity conflicting routes are a hard diagnostic.
* Absolute paths must resolve inside repo or fail.
* `..` components must fail unless canonicalization proves the final target is inside repo.
* Root-equivalent paths remain rejected.

### Add a route decision record

Add this to `ProofPlan`:

```rust
pub struct RouteDecision {
    pub changed_path: String,
    pub owner: Option<String>,
    pub owner_pattern: Option<String>,
    pub test_pattern: Option<String>,
    pub lane: Option<String>,
    pub command_id: Option<String>,
    pub command: Option<String>,
    pub specificity: usize,
    pub decision: String, // routed | unmapped-owner | unmapped-test | blocked
    pub reason: String,
    pub risk_notes: Vec<String>,
}
```

This is the “explainability spine” for the Proof Router. It lets downstream audit/repair prove exactly why a lane was selected.

### Tests to add

Add focused tests:

```text
route_does_not_match_sibling_prefix
route_file_specificity_beats_directory_prefix
route_rejects_repo_root
route_rejects_parent_traversal
route_accepts_absolute_path_inside_repo
route_rejects_absolute_path_outside_repo
route_directory_input_requires_explicit_scope_or_expands
route_conflicting_equal_specificity_fails
```

### Acceptance criteria

* No `starts_with` route can match a sibling prefix.
* Every `changed_path` gets exactly one route decision.
* Every unmapped path is represented in `risk_notes` and `human_approval_requirements`.
* Existing `proof_surface_smoke` tests still pass.

## Workstream C — Add risk escalators to prove “smallest credible lane”

### Goal

The current planner picks lanes from test-map and records residual risk for unmapped paths. The hardened router should also escalate when the **type of change** requires more than the base test-map route.

Examples:

| Change type                                                             | Required escalation                 |
| ----------------------------------------------------------------------- | ----------------------------------- |
| `Cargo.lock`, `package-lock.json`, `.github/workflows`                  | security/supply-chain lane          |
| `db/migrations/**`                                                      | DB migration analyze lane           |
| `contracts/**`, schema/source API contract changes                      | contract/build-schemas lane         |
| `packages/ux-qa/**`, UI/story/route files                               | UX/rendered proof lane              |
| `agent/test-map.json`, `agent/proof-lanes.toml`, `agent/owner-map.json` | audit + route-self-test             |
| generated-zone source changes                                           | generated reproduction lane         |
| generated-zone output changes                                           | source-regeneration proof or block  |
| release-critical paths                                                  | release/full lane or human approval |

The standard already makes proof lanes and risk classes central to conformance; the router should embody that policy instead of leaving it as prose. 

### Code changes

Add a planner stage:

```rust
fn risk_escalations_for_path(path: &str, catalog: &RepoCatalog) -> Vec<RequiredLane>
```

Then compute:

```text
base lanes from test-map
+ escalated lanes from risk rules
- duplicate lanes
= required lanes
```

### Acceptance criteria

* `Cargo.lock` plans include security/supply-chain proof.
* `db/migrations/foo.sql` plans include migration analysis.
* changing `agent/test-map.json` forces proof-router self-validation.
* generated output changed without source route creates blocking risk.
* docs-only changes still avoid unnecessary UX/security/full lanes unless routed.

# 6. Recommended implementation order

## Wave 1 — Router correctness first

Do this before evidence and CI changes.

1. Replace path matching with segment-aware route matching.
2. Add route decision records.
3. Add route-specific tests.
4. Preserve current plan output compatibility where possible.

**Why first:** if routing is wrong, every receipt downstream can be beautifully structured and still prove the wrong thing.

## Wave 2 — Lane metadata and plan schema

1. Add explicit lane IDs and command IDs.
2. Add `rules_covered` metadata to lanes.
3. Add typed `planned_runs` and `route_decisions`.
4. Canonicalize plan digest.
5. Add schema negative tests.

**Why second:** once route decisions are precise, lock the contract.

## Wave 3 — Runner safety

1. Add command specs.
2. Add timeout/env/cwd/destructive policy.
3. Add richer receipt fields.
4. Keep shell-command compatibility for current lanes.

**Why third:** the runner must not become the attack surface.

## Wave 4 — Evidence integrity

1. Add plan, receipt, log, and artifact digests.
2. Store proof runs under run IDs.
3. Add latest pointer.
4. Add evidence index v1.3 or v2.0.
5. Add mutation tests.

**Why fourth:** evidence must become durable and tamper-evident.

## Wave 5 — Verification gate

1. Add `proof verify`.
2. Validate coverage, freshness, commands, success, artifacts, and digests.
3. Emit verification artifact.
4. Add audit integration.

**Why fifth:** this turns evidence into a merge decision.

## Wave 6 — CI and docs

1. Update workflows.
2. Upload evidence artifacts.
3. Update docs and phase receipt.
4. Run full validation.

**Why last:** CI should enforce the final model, not an intermediate contract.

# 9. Non-negotiable acceptance criteria

Phase 03 is 100% done when these all pass:

1. **Path routing**

   * Sibling-prefix false matches are impossible.
   * Root/path traversal escapes fail.
   * Route specificity is deterministic.
   * Equal-specificity conflicts fail.

2. **Lane selection**

   * Every route resolves to lane ID or blocking risk.
   * Required lanes include risk escalators.
   * Optional/skipped lanes have reasons.
   * The plan explains why selected lanes are sufficient.

3. **Execution**

   * Commands are allowlisted by ID/spec.
   * Shell execution is bounded and compatible, not uncontrolled.
   * Timeouts and failures produce receipts.
   * Unsigned command hatch is impossible to trigger accidentally.

4. **Evidence**

   * Plan, receipts, logs, and artifacts have digests.
   * Evidence survives CI archiving.
   * Mutation after run is detectable.
   * Custom proof runs are discoverable.

5. **Verification**

   * `proof verify` passes only for fresh, successful, complete, digest-valid evidence.
   * Release audit requires verified proof coverage.
   * Missing/stale/partial evidence produces hard findings.

6. **Tests**

   * Existing proof smoke tests pass.
   * New adversarial tests pass.
   * Negative schema tests prove schemas are strict.
   * Evidence mutation tests prove integrity checks work.

7. **Docs/CI**

   * Standard and testing docs match actual CLI behavior.
   * CI runs proof, verifies it, audits release mode, and uploads artifacts.
   * Phase receipt records exact validation and residual risk.

# 10. The “smallest credible lane” algorithm I would implement

This is the core algorithm that should live behind Phase 03.

```text
Input:
  repo root
  changed paths or changed-from ref
  owner map
  test map
  proof lanes
  generated zones
  standard/rule metadata
  audit/risk metadata

Step 1: Normalize scope
  - Convert to repo-relative POSIX paths.
  - Reject root, traversal, and outside-repo paths.
  - Expand or explicitly mark directory scopes.

Step 2: Route each path
  - Match owner route by segment-aware specificity.
  - Match test route by segment-aware specificity.
  - Emit RouteDecision for every path.
  - Block or flag missing owner/test route.

Step 3: Resolve lane identity
  - Prefer test-map lane ID.
  - Fall back to command ID.
  - Fall back to legacy command string.
  - Validate lane metadata.

Step 4: Apply risk escalators
  - Generated zones
  - Security/supply-chain
  - DB migration
  - Contract/schema
  - UI/rendered UX
  - Standard/control-plane changes
  - Release-critical changes

Step 5: Build minimal lane set
  - Deduplicate by lane ID.
  - Sort by deterministic priority.
  - Preserve all changed path coverage.
  - Mark skipped lanes with reasons.

Step 6: Emit canonical plan
  - Route decisions
  - Required lanes
  - Planned runs
  - Expected artifacts
  - Manifest fingerprints
  - Plan digest

Step 7: Execute safely
  - Validate command specs.
  - Run with timeout/env/cwd policy.
  - Capture logs and receipts.

Step 8: Index evidence
  - Link every artifact.
  - Digest every artifact.
  - Record git head and manifest fingerprints.

Step 9: Verify
  - Validate plan/evidence/receipts.
  - Check coverage/freshness/digests.
  - Return merge decision.
```

That algorithm is what makes the Proof Router the “load-bearing wall.”
