## Phase 03 diagnosis: implemented, but not yet “load-bearing-wall hardened”

I’d treat **03 – Proof Router** as **implemented baseline, not finished**. The repo’s own phase note currently marks Phase 03 “complete” and says `jankurai lane` / `jankurai proof` build a normalized proof plan, while `jankurai prove` executes planned commands, writes receipts/logs/evidence, and enforces the command allowlist unless the explicit unsigned-command escape hatch is enabled. That is real, useful infrastructure. But “100% done and hardened” is a higher bar: the router must be able to prove, mechanically, that it selected the **smallest credible lane**, that the evidence is untampered, that receipts cover the current change/head, and that release/audit mode can fail closed when the proof is stale, incomplete, overbroad, or false-green. 

The current core lives where expected: CLI wiring in `main.rs`, planner/runner/evidence logic in `crates/jankurai/src/commands/proof.rs`, catalog loading in `context_data.rs`, route control files in `agent/owner-map.json`, `agent/test-map.json`, and `agent/proof-lanes.toml`, and smoke coverage in `proof_surface_smoke.rs`. `prove` already supports `--plan`, `--changed`, `--changed-from`, `--plan-out`, `--plan-md`, `--out-dir`, `--evidence-index`, `--continue-on-error`, and `--allow-unsigned-commands`; it rejects ambiguous input such as `--plan` plus `--changed`, persists changed-mode plans, validates plans/receipts/evidence against schemas, and writes logs plus proof receipts.   

The biggest gap is conceptual: the repo currently **routes** proof, but it does not yet fully **prove the routing decision**. The plan shows selected lanes, commands, skipped lanes, risk notes, and planned runs, but it does not carry a rigorous route trace explaining why each candidate lane was included, why broader lanes were skipped, what lane superseded what, what artifacts must exist for the lane to count, or why the selected lane is the smallest credible proof rather than merely the first matching command. The standard says agents must run the smallest mapped lane and report skipped lanes with reason, but the proof plan schema does not yet encode enough structure to make that property enforceable.  

## The target state

When Phase 03 is truly finished, the proof router should answer this question with machine-checkable evidence:

> For this exact git head, base ref, changed path set, owner map, test map, proof-lane manifest, policy version, and rule registry, these are the minimal required lanes; these commands were allowed; these artifacts prove the lanes; these receipts match the plan; no required proof is missing, stale, failed, overbroad, or unsigned.

That means Phase 03 is done only when these are true:

1. **Route selection is deterministic.** Same repo state plus same changed paths always produces the same plan, independent of CLI path ordering, duplicate paths, platform separators, or absolute/relative spelling.

2. **Minimality is explicit.** Every required, optional, skipped, and superseded lane has a reason. “Skipped” cannot mean “ignored”; it must mean “not required because X covers the changed scope” or “blocked because Y risk remains.”

3. **Command execution is safe by construction.** The executor runs only catalog-defined commands by stable command identity or digest, with timeout, cwd, environment policy, log capture, and no silent unsigned-command path.

4. **Receipts are tamper-evident.** A receipt must bind together plan fingerprint, command fingerprint, lane manifest fingerprint, git head, changed paths, log digest, artifact digests, exit code, and verdict.

5. **Release/audit can verify coverage.** `audit --mode release --proof-receipts ...` should not merely ingest receipts. It should prove that all required lanes in the current plan succeeded against the current head and changed scope.

6. **False-green risk is handled.** A command that exits `0` is not sufficient if expected artifacts are missing, if artifacts are stale, if the route required UX/security/schema evidence and no such evidence exists, or if the receipt covers a different path/head/plan.

7. **The schema is strict enough to be a contract.** The current schemas are intentionally permissive in places; hardening requires stricter shape, required fields, and compatibility migration rather than open-ended `additionalProperties` for core proof objects.

# Critical findings from the current code

## 1. The current router is path-prefix based and needs sharper directory semantics

`RepoCatalog` loads owner, test, generated-zone, and proof-lane manifests, then chooses the longest matching owner/test prefix. The current matcher returns true when `path == prefix`, `path.starts_with(prefix)`, **or** `prefix.starts_with(path)`. That third branch can be useful for broad directory inputs, but it also creates overmatching risk: a changed parent path can match a more specific child prefix without a separate “directory expansion” rule. This needs to be made explicit so “changed `db`” cannot accidentally prove as if only `db/migrations/` changed, and “changed `agent`” cannot silently route as a child prefix unless the router knows it is a directory scope. 

**Hardening move:** replace `path_matches(path, prefix)` with a typed matcher:

```text
ExactFile(path, key)
DirectoryContains(path_dir, key_dir)
DirectoryInputCoversPrefix(input_dir, key_dir) only when input was explicitly a directory
GlobMatch(pattern, path) only for future manifest patterns
```

A changed parent directory should expand to inventory/git-diff files where possible; only if expansion is impossible should it become a broad-scope route with elevated residual risk.

## 2. “Smallest credible lane” is not yet a first-class algorithm

`build_proof_plan` finds a test-map route, maps command to a proof lane if the command matches a named lane, and deduplicates planned runs by command. This is good, but it does not encode lane weight, lane kind, supersession, merge gate, required artifacts, or proof strength. The live `agent/proof-lanes.toml` contains `name`, `command`, and `purpose` only, so the router cannot yet prove that `audit` is smaller than `full`, that `db-migration-analyze` is sufficient for SQL migrations, or that `security` must be included on a dependency/security-sensitive path.  

**Hardening move:** turn proof lanes into typed route capabilities. Keep backward compatibility, but add fields such as:

```toml
[[lane]]
name = "audit"
kind = "audit"
command_id = "jankurai.audit.score"
command = "just score"
purpose = "write agent/repo-score.json and agent/repo-score.md"
risk_level = "standard"
cost = 20
merge_gate = true
supersedes = ["fast"]
proves_rules = ["HLT-003-OWNERLESS-PATH", "HLT-004-UNMAPPED-PROOF"]
required_artifacts = ["agent/repo-score.json", "agent/repo-score.md"]
timeout_seconds = 180
```

Then the router can solve a minimal covering set instead of relying on command equivalence.

## 3. The schemas validate existence, not proof integrity

The proof plan schema currently requires top-level fields such as changed paths, matched maps, lanes, commands, artifacts, and risk notes, but `planned_runs` is just an array of arbitrary objects and top-level additional properties are allowed. The proof receipt schema requires only `lane`, `command`, `exit_code`, `elapsed_ms`, and `artifacts`; evidence-index schema requires paths/lists but not digests, plan fingerprints, receipt fingerprints, or a final coverage verdict.   

**Hardening move:** introduce `proof-plan.schema.json` v1.1 or v2.0, `proof-receipt.schema.json` v1.1 or v2.0, and `evidence-index.schema.json` v1.3 or v2.0 with strict internal objects. Do not break old artifacts abruptly; support old read compatibility, but write the new strict schema.

## 4. Receipts are useful, but not yet tamper-evident enough

`execute_run` writes logs and receipts with command, lane, exit code, elapsed time, changed paths, owner, residual risk, plan path, run ID, git head, and optional rule coverage. That is a strong start. But the receipt does not yet include the proof-plan digest, lane manifest digest, test-map digest, owner-map digest, command digest, log digest, artifact digest list, artifact existence result, or final proof verdict.  

**Hardening move:** every receipt should include:

```json
{
  "schema_version": "1.1.0",
  "verdict": "pass | fail | blocked | stale | incomplete",
  "plan_fingerprint": "sha256:...",
  "route_fingerprint": "sha256:...",
  "catalog_fingerprints": {
    "owner_map": "sha256:...",
    "test_map": "sha256:...",
    "proof_lanes": "sha256:...",
    "standard_version": "sha256:..."
  },
  "command_fingerprint": "sha256:...",
  "log_digest": "sha256:...",
  "artifact_digests": [
    { "path": "target/jankurai/...", "sha256": "sha256:...", "required": true, "status": "present" }
  ]
}
```

## 5. Command allowlisting exists, but execution should be hardened

The code already prevents execution of commands not present in `agent/proof-lanes.toml` or `agent/test-map.json`, unless both the CLI flag and `JANKURAI_ALLOW_UNSIGNED_PROOF_COMMANDS=1` are present. That is a good guardrail. However, commands are still shell strings executed through `bash -lc`, so the hardened target should move toward structured command descriptors or command IDs, timeouts, constrained environment variables, redaction, and output-size limits.  

**Hardening move:** keep shell-string compatibility for now, but add a canonical command object internally:

```rust
struct CommandSpec {
    id: String,
    argv: Vec<String>,
    shell: Option<String>,
    cwd: PathBuf,
    timeout_seconds: u64,
    env_policy: EnvPolicy,
    output_limit_bytes: u64,
}
```

Then write receipts using the command ID and digest, not just the human command string.

## 6. Audit release mode needs to verify proof coverage, not just receive receipts

The CLI already has `--proof-receipts`, and release mode adds a high finding when no proof receipts are supplied. The audit report model also carries proof receipts. But the hardening goal is stricter: audit should compare the current changed scope to the current proof plan and receipts, ensuring every required lane succeeded for this exact head and scope.  

**Hardening move:** add a `verify_proof_coverage` module that emits findings for:

```text
missing-plan
stale-plan-head
stale-receipt-head
missing-required-lane
failed-required-lane
missing-required-artifact
receipt-plan-fingerprint-mismatch
command-digest-mismatch
unsigned-command
changed-path-not-covered
rule-coverage-claim-invalid
```

That is where Phase 03 becomes the load-bearing wall for Phases 07 and 10–13.

# Full plan to finish and harden Phase 03

## Workstream A — Freeze the contract and define “smallest credible lane”

**Goal:** Make the routing objective precise before touching more code.

### A1. Add a normative definition to `docs/agent-native-standard.md`

Add a subsection under “Test Lanes” or immediately after it:

```md
### Smallest Credible Proof Lane

For a changed path set `C`, the proof router MUST select the lowest-cost set of lanes `L`
such that:

1. every path in `C` has an owner-map route;
2. every path in `C` has at least one test-map route;
3. every selected route maps to an allowed proof lane or explicitly documented custom command;
4. every stable rule implicated by the path category is covered by at least one selected lane;
5. every required artifact for the selected lanes is expected in the proof plan;
6. any broader lane that is skipped has a machine-readable reason;
7. any unmapped, ambiguous, generated, security-sensitive, DB, UX, or contract-sensitive path raises residual risk or escalates to the appropriate lane.
```

The current standard already says path changes must route through `agent/test-map.json` and agents must run the smallest mapped lane locally; this change turns that sentence into an auditable algorithm.

```text
pass        all required lanes passed and required artifacts exist
fail        at least one required lane ran and failed
blocked     route is incomplete, unmapped, unsigned, unsafe, or requires human approval
stale       proof was valid for another git head/catalog/plan
incomplete  commands passed but required receipts/artifacts are missing
advisory    non-blocking lane only
```

Use this same vocabulary in proof plan, proof receipt, evidence index, doctor diagnostics, and audit findings.

### A3. Declare Phase 03 invariants

```text
route_is_order_invariant
```

**Done when:** the standard, testing doc, and phase doc all use the same terms: route, candidate, selected lane, skipped lane, superseded lane, residual risk, proof verdict, and evidence ledger.

## Workstream C — Fix path semantics and changed-scope normalization

**Goal:** Eliminate overmatching, path traversal, symlink ambiguity, and root-scope surprises.

### C1. Replace the current prefix matcher

Current matcher:

```rust
path == prefix || path.starts_with(prefix) || prefix.starts_with(path)
```

Replace with explicit behavior:

```rust
enum MatchKind {
    ExactFile,
    DirectoryContains,
    DirectoryInputCoversPrefix,
    NoMatch,
}
```

Rules:

```text
"docs/moonshot.md" matches "docs/" as DirectoryContains.
"docs/" matches "docs/" as ExactDirectory.
"docs" normalizes to "docs/" only if filesystem/git says it is a directory.
"db" does not silently match "db/migrations/" unless treated as directory scope.
"db/migrations/001.sql" chooses "db/migrations/" over "db/".
```

### C2. Harden `normalize_changed_paths`

Add checks for:

```text
empty path
"."
repo root
path escaping repo root
absolute path outside repo
Windows separator normalization
duplicate paths
trailing slash consistency
deleted files from git diff
renamed files from git diff
submodule paths
symlink paths
```

### C3. Distinguish changed file, changed directory, deleted path, renamed path

`--changed-from` should preserve enough status from git diff to know whether a file was added, modified, deleted, or renamed. A deleted contract or deleted test route may need stricter proof than a modified doc.

Suggested structure:

```rust
pub struct ChangedPath {
    pub path: String,
    pub status: ChangeStatus, // Added, Modified, Deleted, Renamed, Unknown
    pub source: ChangeSource, // Cli, GitDiff
    pub is_directory_scope: bool,
}
```

### C4. Test path normalization aggressively

Add table tests:

```text
./docs/a.md -> docs/a.md
docs//a.md -> docs/a.md
absolute path inside repo -> repo-relative path
absolute path outside repo -> reject
. -> reject
repo root -> reject
db -> directory-scope only when known directory
db/migrations/x.sql -> most specific db/migrations route
agent -> route as directory only when known
```

**Done when:** the route engine cannot accidentally over-route a parent path to a child proof route without marking the scope as broad and risky.

```rust
    pub passed_lanes: Vec<String>,
    pub missing_lanes: Vec<String>,
    pub stale_receipts: Vec<String>,
    pub failed_receipts: Vec<String>,
    pub missing_artifacts: Vec<String>,
    pub findings: Vec<Finding>,
}
```

### I2. Integrate with `audit --mode release`

Current release mode fails when no proof receipts are supplied. Harden it so release mode fails when any required proof is not valid for the audited scope. 

Release mode should:

1. Rebuild the proof plan for the audit changed scope.
2. Load proof receipts and evidence index.
3. Verify plan fingerprint or recompute compatible route fingerprint.
4. Verify all required lanes passed.
5. Verify changed paths in receipts cover changed paths in plan.
6. Verify git head/base ref compatibility.
7. Verify required artifacts exist and digest-match.
8. Emit findings for every gap.

### I3. Add proof findings

Suggested finding rules:

```text
HLT-004-UNMAPPED-PROOF        missing route, missing required lane, no receipt
HLT-008-FALSE-GREEN-RISK      command passed but artifacts missing or route mismatch
HLT-012-OVERBROAD-AGENCY      unsigned/broad command or execution outside lane scope
HLT-016-SUPPLY-CHAIN-DRIFT    dependency/lockfile change missing security receipt
HLT-020-CI-HARDENING-GAP      workflow/CI change missing doctor/security proof
```

### I4. Keep advisory mode permissive

Advisory mode can ingest and display proof receipts without failing. Release mode should fail closed.

**Done when:** a green command cannot satisfy release mode unless it proves the exact changed scope with current, matching receipts.

## Workstream J — Make rule coverage evidence-backed

**Goal:** `rules_covered` should not overclaim.

The current code maps lane names to stable rule IDs and filters unknown IDs through `audit::rules::lookup`. That is a good compatibility-preserving start, and tests assert security lane coverage uses registered rule IDs.  

### J1. Split rule claims from rule coverage

Use:

```json
"rules_claimed": [
  { "rule_id": "HLT-010-SECRET-SPRAWL", "source": "proof-lanes.toml" }
],
"rules_covered": [
  { "rule_id": "HLT-010-SECRET-SPRAWL", "status": "covered", "evidence": "target/jankurai/security/evidence.json" }
]
```

A lane may claim to cover security rules, but coverage should be `covered` only if required artifacts exist and pass validation.

### J2. Move mapping from hard-coded Rust to lane metadata

Short term: keep current hard-coded mapping as fallback.

Long term: `proof-lanes.toml` owns `proves_rules`, and the route engine validates those IDs against `rules.rs`.

### J3. Add statuses

```text
claimed
covered
missing-artifact
failed
advisory
not-applicable
```

**Done when:** receipts cannot claim rule coverage solely because a lane name was present; the claim must be tied to passing evidence.

### K2. Property tests

Use `proptest` or table-driven equivalents:

```text
input order invariance:
  route(paths) == route(shuffle(paths))

duplicate invariance:
  route([a, a, b]) == route([a, b])

specificity:
  if "db/migrations/" and "db/" both match, db/migrations wins for db/migrations/x.sql

monotonicity:
  adding a changed path cannot remove proof for an existing path unless a superseding lane is selected and recorded

allowlist:
  every planned command must normalize to an allowed catalog command

skip completeness:
  every lane not selected has a skipped/superseded/not-applicable reason
```
