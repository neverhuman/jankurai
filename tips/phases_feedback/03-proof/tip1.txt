## Phase name

**Phase 03: Proof Router And Evidence Ledger**

I would treat this as the final hardening slice for Phase 03, not a new phase. The moonshot requires the repo to route intent into bounded agents, proof lanes, durable evidence, expiring exceptions, and reusable primitives; it also says every changed path should map to ownership and proof, and evidence should be durable, machine-readable, and easy to audit. ([GitHub][1]) The repo README describes Jankurai as the control plane that makes ownership, proof, generated zones, and repair evidence machine-readable, with the operating loop `intent -> bounded agents -> proof lanes -> evidence -> expired exceptions -> reusable primitives`. ([GitHub][2])

The Phase 03 doc already names the exact acceptance target: changed paths should route deterministically through owners, lanes, commands, artifacts, and merge decision; changed-mode `prove` should persist JSON and Markdown proof plans, execute allowlisted commands, write receipts, write an evidence index, and populate deterministic rule coverage only where linkage is explicit. ([GitHub][3]) The current source is close, but the inspected `proof.rs` still builds changed-mode plans in memory, records `plan_path` as `"in-memory"`, and leaves `rules_covered` empty. ([GitHub][4]) The current `main.rs` also exposes `--plan`, `--changed`, and `--changed-from`, but not the required `--plan-out` / `--plan-md` controls for persisted changed-mode plans. ([GitHub][5])

Below is the complete patch I would apply. It keeps schemas unchanged, preserves `prove --plan`, adds persisted plan artifacts for `prove --changed` / `--changed-from`, rejects ambiguous command input, rejects root-equivalent changed paths, prevents no-command proof plans from succeeding silently, and emits schema-valid rich `rules_covered` objects using only registered rule IDs. I did not run validation in this environment; the commands to run are included at the end.

### Why this gets Phase 03 to 100%

The critical product behavior is that `prove --changed` is no longer a shortcut that quietly constructs a throwaway in-memory plan. It now becomes a first-class evidence-producing path: plan JSON, plan Markdown, receipts, logs, and the evidence index all point at the same persisted proof plan. That is the difference between “the command passed” and “the repo can explain what changed, why this was the smallest credible proof, and where the next agent can inspect the evidence.”

The rule-coverage change is intentionally conservative. It does **not** inspect shell command strings like `just score` or `cargo test` and pretend to know what rules were proven. It maps only named lane contracts to registered HLT rule IDs. That matches the standard’s emphasis on stable rule IDs and proof lanes, while avoiding the moonshot’s rejected pattern of proof that passes without actually covering changed behavior. ([GitHub][6])

The no-runnable-proof failure matters. Without it, an unmapped path can generate a plan with risk notes, execute zero commands, write an evidence index, and exit successfully. That is false green evidence. The patch still writes the plan and evidence index for repair, but it fails the proof command so a maintainer or agent must update `agent/test-map.json`, add ownership/proof routing, or provide a valid persisted plan.
