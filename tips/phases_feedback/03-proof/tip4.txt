# Phase name

**Phase 03: Proof Router And Evidence Ledger**

This is the phase where this feedback and patch should be considered.

The core moonshot requirement is that the repo should make a fresh agent able to “find the owner, choose the proof lane, run the checks, and leave evidence without learning the whole codebase first.” That means Phase 03 cannot stop at “the command ran.” It needs a durable chain: changed paths → owner/test-map route → planned lane → executed command → logs/receipts → evidence index → merge decision. The current repo is very close, but there is still one important integrity gap: current `prove --changed` builds a plan in memory and records `"in-memory"` as the plan path, while the phase criteria require changed-mode `prove` to persist JSON and Markdown plan artifacts before execution. Current `execute_run` also leaves `rules_covered` empty, even though Phase 03’s residual hardening plan explicitly calls for deterministic rule linkage where lane-to-rule contracts are known. ([GitHub][1])

Below is one copy-pasteable solution. It keeps schemas stable, keeps command allowlisting intact, avoids command-text inference, and finishes the Phase 03 hardening without stepping into Phase 05/06/07/08 ownership.

---

## What this completes

This patch completes the remaining Phase 03 hardening slice:

1. **`jankurai prove --changed` and `--changed-from` now write file-backed proof-plan artifacts before executing commands.**
2. **Evidence index `plan_path` now points to the persisted plan path, not `"in-memory"`.**
3. **`--plan` mode remains compatible.**
4. **`--plan` cannot be combined with `--changed` or `--changed-from`.**
5. **Changed-mode proof refuses `--plan-out -` because a proof run needs an auditable file-backed plan.**
6. **Receipts include deterministic `rules_covered` only for explicit lane-to-rule contracts.**
7. **Unknown/custom lanes stay empty rather than making false coverage claims.**
8. **Root-equivalent changed paths such as `.` are rejected so the router does not treat “the whole repo” as a concrete changed path.**
9. **Tests assert changed-mode artifacts, `changed-from` git behavior, bad argument failures, and rule coverage.**
10. **Docs explain exactly how to use `prove` and where artifacts land.**

This is intentionally additive. It does **not** change proof-plan, proof-receipt, or evidence-index schemas, which matches the Phase 03 guidance that the current data model is sufficient. ([GitHub][2])

## Why this is the right Phase 03 finish

The proof router already has the important structure: `agent/test-map.json` maps paths to commands, `agent/proof-lanes.toml` defines named command lanes, proof plans include `planned_runs` and `skipped_lane_entries`, execution writes receipts/logs/evidence index, and command execution is allowlisted unless the explicit dual emergency hatch is used. ([GitHub][2])

The remaining issue was that changed-mode execution violated the evidence chain. The code path built a plan, serialized it to a string, and used `"in-memory"` as the plan path when `args.plan` was absent. That meant a later repair agent could find the receipt and evidence index, but not the exact proof plan that produced them. ([GitHub][3])

This patch makes `prove --changed` and `prove --changed-from` behave like a real evidence-producing proof run:

```text
changed paths
  -> build plan
  -> write proof-plan.json
  -> write proof-plan.md
  -> validate/allowlist planned commands
  -> execute commands
  -> write logs
  -> write receipts
  -> write evidence-index.json pointing at the persisted plan
```

That directly matches the Phase 03 target: “Changed paths -> owners -> lanes -> commands -> artifacts -> merge decision.” ([GitHub][2])

The `rules_covered` mapping is deliberately conservative. It does **not** parse shell command text or guess that `true` means security coverage. It only looks at the planned lane name and then filters every candidate through the registered rule lookup. That honors Phase 03’s warning that unknown/custom route coverage must remain empty rather than misleading. ([GitHub][2])
