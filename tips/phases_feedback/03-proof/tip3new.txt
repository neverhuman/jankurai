# Phase 03: Proof Router And Evidence Ledger — hardening plan

## 1. What Phase 03 must prove when it is truly finished

The moonshot claim is:

```text
Changed paths -> owners -> lanes -> commands -> artifacts -> merge decision
```

That is also exactly how the Phase 03 doc states the goal: make Jankurai answer which proof covers a change, deterministically, with owners, lanes, commands, artifacts, and merge decision evidence. 

For Phase 03 to be “100% done and hardened,” the repo must guarantee these invariants:

1. **Every changed path is routed or explicitly blocked.**
   No changed path can silently disappear, normalize away, or match the wrong prefix.

2. **Every route is deterministic.**
   Overlapping owner/test/proof mappings always resolve the same way, with the selected route and skipped alternatives recorded.

3. **Every proof command is bounded.**
   Commands come only from `agent/proof-lanes.toml` or `agent/test-map.json`, are normalized, are tied to a digest, and execute with timeouts, log limits, and minimal environment.

4. **Every receipt is tied to the exact plan and manifest state.**
   A receipt must prove the command that was planned, from the same git head, same map fingerprints, same lane, same command digest, and same changed-path coverage.

5. **Release audit validates coverage, not just receipt existence.**
   Release mode must fail if receipts are missing, stale, failed, tampered, too broad, too narrow, or from a different plan.

6. **Doctor can audit the evidence ledger deeply.**
   Doctor must validate schemas, stale heads, digest mismatches, missing artifact paths, missing logs, receipt/plan mismatch, and uncovered required lanes.

7. **Tests attack the router like an adversary.**
   The test suite must cover overlapping prefixes, broad path inputs, outside-repo paths, duplicate commands, unsigned commands, failed commands, stale receipts, tampered plans, timeout behavior, and evidence digest mismatch.

The repo is already close. The Phase 03 file currently says the router/evidence ledger is implemented and hardened; it specifically lists `jankurai lane`, `jankurai proof`, `jankurai prove`, `planned_runs`, `skipped_lane_entries`, receipts, logs, evidence index `1.2.0`, command allowlisting, proof-receipt audit ingestion, and doctor validation.  The work below is the final hardening layer that makes Phase 03 production-grade.

# 3. The critical gaps to close

## Gap A — path matching is too permissive

Current matching logic in `context_data.rs` is:

```rust
path == prefix || path.starts_with(prefix) || prefix.starts_with(path)
```

That third clause, `prefix.starts_with(path)`, is risky. It lets a broad changed input match a narrower configured prefix. For example, a changed path like `docs` can match `docs/` or potentially a narrower configured route. The intent may be helpful for directory inputs, but it also creates ambiguity.

**Finish criteria:** replace this with segment-aware matching and explicit broad-path handling.

A hardened route match should distinguish:

```text
ExactFile
PathUnderPrefix
DirectoryContainsMappedPrefix
NoMatch
OutsideRepo
RootEquivalent
```

The router should not pretend those are the same.

## Gap B — broad paths need explicit expansion or blocking

`jankurai lane --changed crates` and `jankurai lane --changed docs` are not the same as a file-level diff. A directory input should either:

1. expand to tracked changed files under that directory, or
2. route as a directory only if the manifest explicitly supports directory-level routes, or
3. fail with a clear diagnostic requiring explicit files.

**Finish criteria:** no broad input can quietly under-prove a set of files.

## Gap C — schemas are too loose for a hardened contract

`proof-plan.schema.json` currently requires top-level fields, but `planned_runs` items are just objects with `additionalProperties: true`.  `proof-receipt.schema.json` validates core receipt shape, but not enough integrity metadata.  `evidence-index.schema.json` includes the current ledger shape but does not require artifact digests or plan/receipt cross-link integrity. 

**Finish criteria:** schemas must enforce the fields that make proof auditably meaningful.

## Gap D — proof-lane manifest lacks execution policy

`proof-lanes.schema.json` currently requires only `name`, `command`, and `purpose`.  That is not enough for a hardened command runner.

Each lane should include:

```toml
name = "test-cli"
command = "cargo test -p jankurai"
purpose = "run Rust unit tests"
risk = "low"
timeout_ms = 120000
max_log_bytes = 1048576
destructive = false
network = "none"
env_allowlist = ["PATH", "HOME", "CARGO_HOME", "RUSTUP_HOME"]
expected_artifacts = ["target/jankurai/logs/*.log"]
rules_covered = ["HLT-003-OWNERLESS-PATH", "HLT-004-UNMAPPED-PROOF"]
```

Backwards compatibility can keep `command` as a string, but hardening should introduce typed metadata.

## Gap E — shell execution is too powerful

`execute_run` uses:

```rust
Command::new("bash")
    .arg("-lc")
    .arg(&run.command)
```

That matches current manifest style, but a hardened router should avoid shell interpretation wherever possible.

**Finish criteria:** introduce a typed command runner:

```rust
argv: ["cargo", "test", "-p", "jankurai"]
shell: false
cwd: "."
timeout_ms: 120000
max_log_bytes: 1048576
env_policy: "minimal"
```

Keep shell mode temporarily for compatibility, but require it to be explicitly marked and treated as higher-risk.

## Gap F — no plan/receipt tamper evidence

Receipts currently cite `plan_path` and `run_id`, but a hardened ledger needs:

* `plan_id`
* `plan_digest`
* `command_digest`
* `manifest_fingerprints`
* `route_decision_ids`
* artifact digests
* log digest
* receipt digest in evidence index

**Finish criteria:** changing a plan, command, manifest, receipt, or log after the fact must be detectable.

## Gap G — release audit only requires some proof receipts

`main.rs` makes release mode fail when no proof receipts are supplied. That is good, but not enough.  Release mode must fail if receipts exist but do not cover the expected plan.

**Finish criteria:** release mode validates receipt coverage against the changed scope.

The question should not be:

> “Did you provide receipts?”

It should be:

> “Do successful, fresh, untampered receipts cover every required lane for every changed path according to the current owner/test/proof manifests?”

## Gap H — doctor validates shape, not full integrity

Doctor validates receipt schema and stale `git_head`, and validates the evidence index schema.  It should also validate:

* every evidence-index receipt path exists
* every evidence-index log path exists
* every receipt’s `receipt_path` matches its actual path
* every receipt’s `log_path` exists
* every log digest matches
* every receipt digest matches evidence index
* every receipt command is still allowlisted
* every receipt command digest matches the plan
* every receipt `plan_id` exists and matches the referenced plan
* every receipt changed path is included in the plan
* every required planned run has a successful receipt
* every failed/skipped lane is reflected in risk/human approval data

# 4. Definition of done

Phase 03 should be considered **100% finished and hardened** only when all of this is true.

## Router completeness

* Every changed path has an owner route or creates a blocking `HLT-003-OWNERLESS-PATH` risk.
* Every changed path has a test/proof route or creates a blocking `HLT-004-UNMAPPED-PROOF` risk.
* Overlapping prefixes are resolved by longest segment-aware match.
* Directory-level inputs are either expanded or explicitly treated as broad routes with risk.
* Absolute paths outside the repo are rejected, not silently ignored.
* Root-equivalent inputs are rejected unless a deliberate full-repo proof mode is used.

## Plan integrity

* Every plan has a stable `plan_id`.
* Every plan includes manifest fingerprints for owner map, test map, proof lanes, generated zones, boundaries, and standard version.
* Every path has a `route_decision` record.
* Every run has a stable `run_id` or `planned_run_id`.
* Every run has `command_digest`, timeout, max log bytes, execution mode, expected artifacts, rule coverage, and risk tier.

## Runner safety

* Commands are allowlisted by digest, not only by normalized string.
* Shell execution is explicit and marked as higher risk.
* Commands have timeouts.
* Logs have max size.
* Logs are redacted for obvious secret patterns before writing.
* Failed commands still write receipts and evidence.
* No destructive command can run without explicit lane metadata and approval requirements.

## Receipt integrity

* Every receipt includes `plan_id`, `plan_digest`, `planned_run_id`, `command_digest`, manifest fingerprints, artifact digests, log digest, git head, and exit code.
* Every receipt validates against a strict schema.
* Failed receipts are first-class evidence, not missing evidence.

## Evidence index integrity

* Evidence index includes plan digest, receipt digests, log digests, artifact descriptors, failed/skipped lane details, and coverage summary.
* Evidence index validates every referenced artifact path.
* Evidence index can be archived by CI and later used to verify the proof without rerunning.

## Audit integration

* `audit --mode release --proof-receipts ...` fails if receipts are missing, stale, failed, incomplete, mismatched, or tampered.
* Advisory/standard modes can warn, but release mode must block.
* Report output names the missing lane/path/run and gives the exact rerun command.

## Doctor integration

* `jankurai doctor --fail-on medium` catches malformed plans, receipts, logs, evidence indexes, stale heads, missing artifacts, digest mismatches, and coverage gaps.

## Test coverage

* Tests include happy paths, adversarial routing, tampering, stale evidence, broad paths, outside paths, command timeout, log truncation, duplicate commands, and release audit coverage.
* `cargo test -p jankurai`, `just fast`, `just score`, and a real `jankurai prove` smoke all pass.

## Workstream 1 — make route matching exact and explicit

### Files

* `crates/jankurai/src/commands/context_data.rs`
* `crates/jankurai/src/commands/proof.rs`
* `crates/jankurai/tests/proof_surface_smoke.rs`
* new: `crates/jankurai/tests/proof_routing_golden.rs`

### Change

Replace the current boolean `path_matches` helper with a typed matcher.

Suggested model:

```rust
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum RouteMatchKind {
    Exact,
    UnderPrefix,
    DirectoryContainsMappedPrefix,
}

#[derive(Debug, Clone)]
pub struct RouteMatch {
    pub prefix: String,
    pub kind: RouteMatchKind,
    pub specificity: usize,
}
```

Segment-aware logic:

```rust
fn path_matches_prefix(path: &str, prefix: &str) -> Option<RouteMatchKind> {
    let path = normalize_route_path(path);
    let prefix = normalize_route_path(prefix);

    if path == prefix {
        return Some(RouteMatchKind::Exact);
    }

    if prefix.ends_with('/') && path.starts_with(&prefix) {
        return Some(RouteMatchKind::UnderPrefix);
    }

    None
}
```

Then handle broad directory inputs separately:

```rust
fn directory_contains_mapped_prefix(path: &str, prefix: &str) -> bool {
    let path = normalize_dir(path);
    let prefix = normalize_dir(prefix);
    prefix.starts_with(&path) && prefix != path
}
```

### Why

The proof router’s credibility depends on not overmatching. The current `prefix.starts_with(path)` behavior is useful but too implicit. It should become an explicit broad-path condition with a risk note or expansion path. 

### Acceptance tests

Add tests for:

* `docs/testing.md` matches `docs/`
* `docs-testing.md` does **not** match `docs/`
* `crates/jankurai/src/main.rs` matches `crates/jankurai/`
* `crates/jankurai-other/file.rs` does **not** match `crates/jankurai/`
* `docs` as a changed input is either expanded or marked broad
* `.` and repo root are rejected
* absolute path inside repo normalizes
* absolute path outside repo is rejected

* For each required planned run, require a matching successful receipt.

  * same `plan_id` or same `plan_digest`
  * same `planned_run_id`
  * same `lane`
  * same `command_digest`
  * same manifest fingerprints
  * exit code `0`
  * no timeout
  * git head same, or explicitly accepted base/head relation
  * changed path coverage includes the route decisions

### Why

This is the load-bearing hardening. Without this, someone can provide receipts that exist but do not prove the current change.

* Release audit fails with no receipts.
* Release audit fails with stale receipts.
* Release audit fails with failed receipts.
* Release audit fails with tampered command digest.
* Release audit fails with receipt from a different path.
* Release audit fails with receipt from a previous plan.

# 6. Exact code-shaped guidance

## 6.1 Replace `path_matches`

Current:

```rust
fn path_matches(path: &str, prefix: &str) -> bool {
    path == prefix || path.starts_with(prefix) || prefix.starts_with(path)
}
```

Replace with:

```rust
fn normalize_route_path(value: &str) -> String {
    value
        .replace('\\', "/")
        .trim_start_matches("./")
        .trim_end_matches('/')
        .to_string()
}

fn normalize_route_prefix(value: &str) -> String {
    let mut value = normalize_route_path(value);
    if !value.is_empty() && !value.ends_with('/') && value != "." {
        // File routes remain file routes; directory routes should be stored with trailing slash.
        // Do not add slash blindly here. Instead preserve manifest intent.
    }
    value
}

fn path_is_under_prefix(path: &str, prefix: &str) -> bool {
    let path = normalize_route_path(path);
    let prefix = prefix.replace('\\', "/").trim_start_matches("./").to_string();

    if path == prefix.trim_end_matches('/') {
        return true;
    }

    if prefix.ends_with('/') {
        return path.starts_with(&prefix);
    }

    false
}
```

Then make broad route matching explicit:

```rust
fn changed_dir_contains_prefix(changed: &str, prefix: &str) -> bool {
    let changed = normalize_route_path(changed);
    let prefix = normalize_route_path(prefix);
    !changed.is_empty()
        && prefix != changed
        && prefix.starts_with(&(changed + "/"))
}
```

Do **not** bury broad matching inside the same boolean as normal proof routing.

## 6.2 Add plan digest helper

```rust
fn digest_json_value(mut value: serde_json::Value, pending_field: &str) -> String {
    if let Some(object) = value.as_object_mut() {
        object.insert(
            pending_field.to_string(),
            serde_json::Value::String("sha256:pending".to_string()),
        );
    }
    let bytes = serde_json::to_vec(&value).unwrap_or_default();
    format!("sha256:{:x}", Sha256::digest(bytes))
}
```

Plan flow:

1. Build plan with `plan_digest = "sha256:pending"`.
2. Serialize to value.
3. Compute digest.
4. Set `plan_digest`.
5. Derive `plan_id`.

## 6.3 Add command digest helper

```rust
fn command_digest(command: &str) -> String {
    let normalized = RepoCatalog::normalize_proof_command(command);
    format!("sha256:{:x}", Sha256::digest(normalized.as_bytes()))
}
```

Use this in:

* planned run
* receipt
* coverage validator

## 6.4 Coverage validator pseudocode

```rust
fn validate_receipts_cover_plan(
    plan: &ProofPlan,
    receipts: &[ProofReceipt],
) -> ProofCoverageSummary {
    let mut missing = Vec::new();
    let mut failed = Vec::new();
    let mut stale = Vec::new();
    let mut mismatched = Vec::new();

    for run in &plan.planned_runs {
        let matching = receipts.iter().find(|receipt| {
            receipt.plan_id.as_deref() == Some(plan.plan_id.as_str())
                && receipt.planned_run_id.as_deref() == Some(run.planned_run_id.as_str())
                && receipt.command_digest.as_deref() == Some(run.command_digest.as_str())
                && receipt.lane == run.lane
        });

        match matching {
            None => missing.push(run.planned_run_id.clone()),
            Some(receipt) if receipt.exit_code != 0 => {
                failed.push(receipt.receipt_path.clone().unwrap_or_default())
            }
            Some(receipt) if receipt.git_head.as_deref() != Some(plan.git_head.as_str()) => {
                stale.push(receipt.receipt_path.clone().unwrap_or_default())
            }
            Some(_) => {}
        }
    }

    ProofCoverageSummary { ... }
}
```

# 11. Highest-leverage priority order

If you only do five things, do these:

1. **Segment-aware routing + per-path route decisions.**
   This prevents the router from selecting the wrong proof lane.

2. **Plan/receipt identity with digests and manifest fingerprints.**
   This prevents stale or tampered proof from looking valid.

3. **Release coverage validator.**
   This turns receipts into a real merge gate instead of an attachment.

4. **Bounded runner with timeout/log limits.**
   This makes proof lanes safe enough to run consistently.

5. **Doctor deep ledger validation.**
   This gives every future agent a one-command way to detect broken evidence.

That set is the true Phase 03 hardening wall.
