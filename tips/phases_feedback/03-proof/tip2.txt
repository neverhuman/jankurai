# Phase name: Phase 03 — Proof Router + Evidence Ledger

Consider the following response as the Phase 03 closeout package for `tips/phases/03-proof-router-evidence-ledger.md`.

The right interpretation of “100% finished” here is not to rebuild the proof router from scratch. The phase file already says the repo has `jankurai lane`, `jankurai proof`, and `jankurai prove`, plus normalized receipts, logs, evidence index writing, command allowlisting, and changed-path entry points. The remaining hardening it explicitly calls out is: persist generated changed-mode proof plans as durable artifacts, populate `rules_covered` only where the route metadata is deterministic, and strengthen evidence cross-links as future artifact types become defined. ([GitHub][1])

That also matches the moonshot: every path should map to ownership/proof, generated truth and evidence should be machine-readable and durable, exceptions should be explicit, and proof should not “pass” without covering the behavior or route it claims to cover. ([GitHub][2]) The implementation below closes Phase 03 without inventing speculative behavioral coverage. It only claims `rules_covered` for deterministic routing facts: owner-map coverage and test/proof-map coverage. It intentionally does **not** claim broader rules like false-green or generated-security coverage until a future explicit lane-to-rule contract exists.

The current code path for `prove --changed` / `--changed-from` builds a plan in memory and records the plan as `"in-memory"` in receipts/evidence; the current execution receipt writes `rules_covered: vec![]`. ([GitHub][3]) The patch below fixes that by making changed-mode `prove` write the generated plan JSON and Markdown before execution, then reuses that plan path in receipts and the evidence index. It also uses the existing `RuleCoverage` model, and it does not require a schema bump: the proof receipt schema already supports rich `rules_covered` objects, and the proof plan schema allows extra planned-run properties. ([GitHub][4]) Command safety stays on the existing allowlist model: planned commands still have to come from `agent/proof-lanes.toml` or `agent/test-map.json`, unless the existing explicit escape hatch is used. ([GitHub][5])

## What this completes

This completes Phase 03 by making the proof-router output durable from the moment a changed-path proof is executed. Before this patch, `prove --changed` was useful as a shortcut, but the generated plan was not a durable evidence artifact. That left receipts pointing at `"in-memory"`, which is bad for later agents because they cannot replay the exact proof plan from the evidence index.

After this patch, the changed-mode flow is:

```text
changed paths
  -> owner/test-map routing
  -> generated proof plan JSON + Markdown
  -> allowlisted command execution
  -> command logs
  -> proof receipts
  -> evidence index with durable plan/log/receipt cross-links
````

That is the evidence-ledger behavior the moonshot wants: a future agent can open the evidence index, follow `plan_path`, inspect planned runs, inspect command logs, inspect receipts, and understand why each command was selected without reconstructing the whole repo context.

## Why `rules_covered` is deliberately narrow

The important design choice is to avoid false proof claims. It would be tempting to map `fast` to `HLT-008-FALSE-GREEN-RISK`, `security` to `HLT-010-SECRET-SPRAWL`, or `ux-qa` to UI rules. That would be unsafe without an explicit lane-to-rule contract proving that a lane actually checks those rules.

So this Phase 03 closeout only writes:

```json
{
  "rule_id": "HLT-003-OWNERLESS-PATH",
  "status": "route-evidence"
}
```

when the changed path matched the owner map, and:

```json
{
  "rule_id": "HLT-004-UNMAPPED-PROOF",
  "status": "route-evidence"
}
```

when the changed path matched the test/proof map.

That means the receipt is saying: “this proof run was deterministically routed through the durable ownership/proof metadata for this changed path.” It is **not** saying: “this command proved all semantic behavior touched by the change.” That distinction keeps Phase 03 aligned with the standard instead of creating a new false-green risk.

## Expected output after validation

After running the changed-mode validation command, these artifacts should exist:

```text
target/jankurai/phase03-proof-plan.json
target/jankurai/phase03-proof-plan.md
target/jankurai/phase03-proof-receipts/*.json
target/jankurai/logs/*.log
target/jankurai/phase03-evidence-index.json
```

The generated evidence index should contain:

```json
{
  "plan_path": "target/jankurai/phase03-proof-plan.json",
  "receipt_dir": "target/jankurai/phase03-proof-receipts",
  "log_dir": "target/jankurai/logs"
}
```

Each receipt produced from a changed-path generated plan should contain a durable plan link:

```json
{
  "plan_path": "target/jankurai/phase03-proof-plan.json"
}
```

And for owner/test-map-routed paths, the receipt should include conservative route evidence:

```json
{
  "rules_covered": [
    {
      "rule_id": "HLT-003-OWNERLESS-PATH",
      "status": "route-evidence"
    },
    {
      "rule_id": "HLT-004-UNMAPPED-PROOF",
      "status": "route-evidence"
    }
  ]
}
```

Existing `prove --plan` behavior remains intact. A hand-written or legacy plan that does not contain deterministic route metadata should still produce receipts with no `rules_covered` field, which is the correct safe behavior.
