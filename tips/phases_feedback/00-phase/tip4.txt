## Moonshot Operating Contract

The moonshot loop is:

```text
intent -> bounded agents -> proof lanes -> evidence -> expired exceptions -> reusable primitives
```

Phase work is valid only when it strengthens that loop.

This index translates the loop into execution law:

| Moonshot demand                          | Routing meaning                                                                                               | Primary phase families     | Evidence expected                                                                  |
| ---------------------------------------- | ------------------------------------------------------------------------------------------------------------- | -------------------------- | ---------------------------------------------------------------------------------- |
| Intent is explicit                       | Every phase starts from an objective, non-goals, dependency gates, and completion criteria.                   | 00, 01, 09, 11, 12         | Phase file objective, active log start row, handoff receipt                        |
| Agents are bounded                       | Owned paths, forbidden paths, generated zones, and permission profile expectations are declared before edits. | 00, 08, 13                 | Owner/test-map coverage, context pack, repair plan, permission profile notes       |
| Proof lanes are smallest credible        | Changed paths route to the least expensive lane that covers the risk.                                         | 03, 05, 06, 07, 12         | Proof plan, proof receipts, UX/security/boundary evidence, certification artifacts |
| Evidence is durable and machine-readable | Reports and receipts use schemas or documented exemptions; volatile proof stays under `target/jankurai/`.     | 01, 03, 05, 06, 07, 11, 12 | JSON/Markdown/SARIF/JUnit summaries, evidence index, phase logs                    |
| Exceptions expire                        | Temporary deviations require owner, risk, expiry, and migration or repair path.                               | 06, 07, 12, 13             | Exception inventory, governance policy, expired-exception repair path              |
| Repeated fixes become primitives         | High-repeat patterns become profiles, golden examples, cells, or migration slices.                            | 04, 09, 10, 11             | Init profiles, golden platform, cell registry, migration plans                     |

A phase is not "done" merely because code changed. It is done when the relevant claims are routed, proven, logged, and safe for the next agent to reuse.

## Status Taxonomy

Use these terms consistently:

| Router state             | Meaning                                                                                                                               | Default selection behavior                                                                           |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| `complete`               | The phase has delivered its required surface and has a completion receipt. Residual risk is non-material or deferred to later phases. | Do not reopen unless user names it or downstream evidence proves a gap.                              |

## Parallel MCP Work Model

Parallel workers are allowed only when write scopes are disjoint and merge order is explicit.

Safe patterns:

* One worker owns schemas while another owns docs that will cite them after schema lock.
* One worker owns Rust command internals while another owns tests for that command, if the function signatures are already locked.
* One worker owns UX runtime changes while another owns Rust ingestion after the UX report schema locks.
* One worker owns security evidence normalization while another owns compliance docs.
* One worker owns benchmark corpus fixtures while another owns report rendering.
* One worker owns generated template source while another owns init command logic after generator contract locks.

Unsafe patterns:

* Multiple workers editing the same schema without a locked contract.
* Multiple workers changing report JSON shape independently.
* Multiple workers editing `agent/test-map.json` independently.
* Multiple workers editing generated outputs directly.
* Multiple workers changing phase status or receipts without shared evidence.
* Multiple workers changing paper content while phase implementation is still moving.
* A worker broadening permissions or validation scope to hide uncertainty.

Merge order:

```text
1. shared contracts
2. implementation
3. tests
4. generated outputs through source/generator only
5. docs that cite the implementation
6. validation
7. phase log finish row and receipt
```

## Universal Stop Conditions

Stop and ask for human/maintainer input, or switch to a prerequisite phase, when any of these occur:

* Target path has no owner-map route.
* Target path has no test-map route.
* Required generated-zone metadata is missing and the phase is not allowed to edit it.
* The implementation would require hand-editing generated output.
* The change would require editing `reference/`.
* The change would require paper edits not explicitly in scope.
* Public report/schema compatibility would break without a migration note.
* A proof command would need unsigned/unallowlisted execution outside the phase policy.
* Security or compliance wording would claim more than the evidence proves.
* Autonomous repair would need destructive migration, secret rotation, infrastructure credential changes, broad architecture rewrite, or auto-merge.
* The worker cannot produce a finish log row with validation and residual risk.

## Cleanup And Deprecation Policy

* Do not update the paper until paper work is explicitly in scope.
* Do not remove phase docs, receipts, or logs to make the roadmap look cleaner.
* Do not rename phase files after implementation work has started unless the rename is itself a logged migration.
* If a phase deprecates a command, schema, rule ID, report field, artifact path, profile, or cell, it must provide compatibility notes and a migration path.
* Generated artifacts are never cleaned up by hand. Change the source or generator and run the recorded command.
* Remove obsolete docs only when replacements are linked, routed, and covered by validation.
* Existing dirty worktree changes are project state, not clutter to clean during unrelated phase work.
