## Purpose

This file is the canonical execution router for the Jankurai moonshot phases.

It is not an implementation phase by itself. It is the living control index that tells a fresh agent, a parallel worker, or a maintainer:

- what the moonshot requires
- which phase owns which outcome
- which phase is complete, partial, hardened, or blocked
- which dependencies must be respected
- which files may be edited
- which files must not be edited
- which proof lane is the smallest credible lane for a change
- which receipt closes a phase
- what the next safe work is

For compact agent routing, `MASTER_PLAN` means:

1. `agent/JANKURAI_STANDARD.md`
2. `agent/MASTER_PLAN.md`
3. this file
4. the active `tips/phases/*.md` file
5. the matching append-only log under `tips/phases/logs/`

This index must not become a marketing roadmap. It is an execution contract. If a phase says “complete,” that means there is a receipt, proof, and a named residual-risk story. If a phase says “partial,” the next implementer must be able to start without rediscovering the whole repository.

## Moonshot Contract

Jankurai’s operating loop is:

```text
intent -> bounded agents -> proof lanes -> evidence -> expired exceptions -> reusable primitives
````

The phase system exists to make that loop executable. Every phase must move at least one part of the loop from aspiration into machine-readable repo behavior.

The moonshot demands these guarantees:

| Moonshot demand                                | Required repo mechanism                                                               | Phase-index consequence                                                      |
| ---------------------------------------------- | ------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| Intent is explicit before code moves           | phase objective, active workstream, read-first ritual, stop conditions                | no implementation starts from vague “cleanup” language                       |
| Agents get bounded authority                   | owned paths, forbidden paths, permission profiles, generated/read-only zones          | every phase declares write scope before edits                                |
| Every changed path maps to ownership and proof | `agent/owner-map.json`, `agent/test-map.json`, `agent/proof-lanes.toml`, proof router | new durable paths must be routed before or with the change                   |
| Generated truth is declared                    | `agent/generated-zones.toml`, generated headers, source/command metadata              | generated outputs are never patched by hand                                  |
| Evidence is durable and auditable              | JSON/Markdown reports, proof receipts, evidence index, phase logs                     | phase completion requires a receipt and validation result                    |
| Exceptions expire                              | exception policy, owner, expiry, migration path                                       | permanent architecture-by-exception is not allowed                           |
| Repeated repairs become primitives             | init profiles, reference platform, certified cells, migration slices, repair packets  | later phases must collapse repeated work into reusable cells/templates/plans |

## Status Vocabulary

Use these exact terms in this file and in phase receipts:

| Status       | Meaning                                                                                              | What must exist                                           |
| ------------ | ---------------------------------------------------------------------------------------------------- | --------------------------------------------------------- |
| `planned`    | phase exists as a plan, but implementation has not started                                           | phase file, dependencies, intended validation             |
| `partial`    | at least one implementation slice landed, but material acceptance criteria remain open               | receipt for the slice, residual risks, next work          |
| `complete`   | the phase’s intended v0 surface is implemented or intentionally bounded, with validation and receipt | phase receipt, validation result, residual-risk statement |
| `hardened`   | a completed or partial phase has an additional production-quality slice beyond the first completion  | receipt, exact hardened surfaces, validation              |
| `blocked`    | dependencies or unsafe ambiguity prevent responsible progress                                        | blocker, owner, unblock condition                         |
| `superseded` | phase work moved to another phase or external plan with a migration note                             | replacement location, compatibility note                  |

A phase can be `complete` while still listing future enhancements. It cannot be `complete` if a material acceptance criterion is silently missing. If the header and receipt disagree, the receipt controls for execution, and this index must state the operational interpretation.

## Phase Selection Algorithm

When asked to “work the plan,” “continue the moonshot,” or “pick pending work,” use this algorithm:

1. If the user names a phase, use that phase unless a dependency gate makes progress unsafe.
2. Otherwise, choose the earliest phase whose operational status is `partial`, `blocked`, or materially inconsistent.
3. If multiple partial phases exist, prefer the one that most directly improves the moonshot loop in this order:

   1. ownership/proof/evidence correctness
   2. generated truth and contract boundaries
   3. exception expiry and repair safety
   4. reusable primitive certification
   5. public certification or governance
4. Before editing, append a start row to the matching log in `tips/phases/logs/`.
5. Declare owned paths, forbidden paths, and generated/read-only paths.
6. Check owner/test-map coverage for every durable path that will be added or changed.
7. Use `jankurai lane` or `jankurai proof` to select the smallest credible proof lane when code or policy changes are involved.
8. Make the smallest coherent slice.
9. Run focused validation first, then broader validation required by the touched paths.
10. Append a finish log row with changed paths, validation, artifacts, current git SHA, and residual risk.
11. Update the phase receipt only after validation facts exist.
12. Never hand-edit generated artifacts. Fix the source and regenerate.

## Parallel MCP Work Model

Use parallel workers only when write scopes are disjoint and the merge path is clear.

Safe parallel patterns:

* one worker owns Rust CLI internals while another owns docs
* one worker owns schemas while another owns docs that cite the schema after it locks
* one worker owns UX QA runtime while another owns Rust ingestion after the UX schema locks
* one worker owns security evidence normalization while another owns compliance docs
* one worker owns benchmark fixtures while another owns report rendering
* one worker owns generated template source while another owns init command logic after generator contract locks

Unsafe parallel patterns:

* multiple workers editing the same rule registry schema before contract lock
* multiple workers changing report JSON shape without compatibility plan
* multiple workers editing `agent/test-map.json` independently
* multiple workers editing generated outputs directly
* multiple workers changing `paper/` while phase plans are unresolved
* one worker adding a generated path while another changes generated-zone policy without coordination
* one worker adding suppressions while another changes rule severity/caps

Merge order:

1. schemas and public contracts
2. routing maps and proof lanes
3. implementation
4. generated artifacts produced by command, if any
5. docs that cite actual behavior
6. validation
7. phase logs and receipts

Every packet must be small enough that a worker can finish without broad repo rediscovery. Give exact files, commands, and acceptance evidence. Do not give a worker permission to hand-edit generated files or broaden agent permissions.

## Phase Log Routing

Tracked phase logs live under:

```text
tips/phases/logs/
```

Use one file per phase. Append start, progress, and finish entries for every attempt.

Log row format:

```text
timestamp_utc | actor/tool | phase | action | changed_paths | validation | artifacts | git_sha | residual_risk
```

Use explicit values:

* `none`
* `pending`
* `not-run`
* `unknown`
* `not-applicable`

Do not leave blank fields.

Volatile command output, screenshots, generated proof plans, proof receipts, SARIF, UX reports, security evidence, and score artifacts stay under:

```text
target/jankurai/
```

Canonical cross-agent history stays under:

```text
tips/phases/logs/
```

## Receipt Requirements

A phase receipt must include:

```md
## Phase Status Receipt

- Phase status:
- Files changed:
- Schemas changed:
- Public interfaces changed:
- Generated artifacts:
- Routing maps changed:
- Validation commands:
- Results:
- Skipped validation:
- Exceptions created:
- Residual risk:
- Follow-up phases:
```

Rules:

* Do not invent validation. Say `not-run` if not run.
* Do not mark `passed` without actual results.
* Do not claim generated artifacts were produced if they were only planned.
* Do not claim compliance certification from readiness evidence.
* Do not hide skipped security/UX/paper validation if touched paths require it.
* Do not shorten or erase older receipts. Add a new slice receipt or clarify the operational state.

## Compatibility And Deprecation Policy

* Report JSON field removals require migration notes.
* Schema version changes require compatibility notes and tests.
* Rule ID removals require deprecation windows.
* Generated path changes require generated-zone updates.
* Proof lane command changes require test-map/proof-lane review.
* Public CLI command removals require docs and migration guidance.
* Phase file renames require this index, logs, and any references to be updated together.
* Obsolete docs may be removed only when the replacement is linked, routed, and covered by validation.
* Existing dirty worktree changes are project state, not clutter to clean during unrelated work.

## Cleanup Policy

Do not perform unrelated cleanup inside a phase.

Allowed cleanup:

* deleting a duplicate only when replacement is linked and validated
* moving durable detail out of root docs into routed docs
* removing obsolete generated output only by changing source/generator and rerunning the generator
* deleting stale local volatile artifacts under `target/jankurai/` only if the active task owns those artifacts

Forbidden cleanup:

* editing `reference/`
* hand-editing generated outputs
* deleting phase receipts
* narrowing phase status to hide incomplete work
* removing test-map or owner-map entries without replacement
* removing compatibility fields from reports without migration notes

### Why this gets Phase 00 to “100%”

The current `00-phase-index.md` already has the right bones: it identifies itself as the canonical router, gives a read-first ritual, names repository constraints, includes the dependency graph, defines phase readiness gates, includes parallel-work guidance, and requires phase logs/receipts. :contentReference[oaicite:3]{index=3}

The missing piece is that it does not fully behave like a **live control plane** after later phases have landed. Several downstream phase files now record real implementation slices: for example, Phase 03 has `prove --changed`, proof receipts, and evidence index support; Phase 05 has UX report validation/ingest; Phase 06 has security evidence; Phase 07 is explicitly partial with remaining contract/DB/generated-boundary work; Phases 10, 12, and 13 are also partial planner/dry-run surfaces. :contentReference[oaicite:4]{index=4}

This replacement fixes that by making Phase 00 the single place that says:

```text
Earliest material pending phase: Phase 07
Downstream partial phases: 10, 12, 13
Do not overclaim complete phases with residual hardening
Use receipts and logs as the source of truth
````

It also directly honors the standard’s hard blocks: every path must have owner/test coverage, generated files must not be hand-edited, public API/schema changes require contract regeneration, high-risk changes require security proof, destructive migrations require safety evidence, and UI changes require rendered UX proof. ([GitHub][4])

The most important steering change is this: **Phase 00 should no longer merely say “Phase 01 can start.” It should continuously answer “what is the safest next moonshot work, and what proof closes it?”** That is exactly what the replacement does.
