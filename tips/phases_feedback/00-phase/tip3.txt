## State Vocabulary

Use this vocabulary consistently in phase plans, logs, receipts, and handoffs.

| State         | Meaning                                                                                 | Completion rule                                              |
| ------------- | --------------------------------------------------------------------------------------- | ------------------------------------------------------------ |
| `not-started` | Phase exists but no implementation receipt exists.                                      | Start only if dependencies are satisfied.                    |
| `planned`     | Worker-ready plan exists, implementation not landed.                                    | Do not claim implemented behavior.                           |
| `partial`     | Some implementation exists, but required surfaces or receipts remain incomplete.        | Name the missing slices and next safe work.                  |
| `complete`    | Phase goal has an implementation receipt and validation evidence for the claimed scope. | Follow-on improvements are allowed, but gaps must be honest. |
| `hardened`    | Complete plus additional proof, fixtures, docs, or regression coverage.                 | Still not immune to later hardening.                         |
| `blocked`     | Cannot progress without a dependency, decision, or human approval.                      | Log blocker and route to the dependency.                     |
| `deprecated`  | Replaced by another phase, path, command, or schema.                                    | Must include migration path and compatibility note.          |

Never downgrade or erase a prior receipt. If the status heading and receipt disagree, the receipt and log win. If the receipt is ambiguous, mark the phase `partial` and name the unresolved proof gap.

## Dependency Types

Use three dependency classes:

| Type                | Definition                                                              | May work start?                                                              |
| ------------------- | ----------------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| Hard dependency     | Required before implementation can safely proceed.                      | No, unless the work is a docs-only unblocker for that dependency.            |
| Soft dependency     | Improves quality or reduces rework, but does not block a bounded slice. | Yes, with residual risk logged.                                              |
| Evidence dependency | Prior phase behavior exists, but proof artifact or receipt is missing.  | Only for audit/hardening slices that produce or verify the missing evidence. |

## Phase Selection Algorithm

When the user names a phase, work on that phase unless a hard dependency makes progress unsafe. If blocked, route to the earliest blocker and say why.

When the user asks generally to “work the plan,” “pick pending work,” or “continue the moonshot,” choose work this way:

1. Read `agent/MASTER_PLAN.md`, this index, the latest relevant phase file, and the matching log.
2. Identify the earliest phase whose state is not `complete` or `hardened`.
3. Check whether that phase’s hard dependencies are satisfied.
4. If a hard dependency is missing, move to the dependency.
5. If multiple incomplete phases are available, pick the one with the highest moonshot leverage:

   * proof, contract, owner, or generated-zone correctness beats cosmetic docs
   * evidence-producing work beats roadmap wording
   * reusable primitives beat one-off fixes only after proof lanes are strong
6. Before edits, append a `start` entry to the phase log.
7. Bound owned paths and forbidden paths.
8. Run or plan the smallest credible proof lane.
9. Append `progress` entries when scope, validation, or residual risk changes.
10. Append a `finish` entry at handoff with changed paths, validation, artifacts, git SHA, and residual risk.

Default next-work queue after this Phase 00 hardening:

1. Phase 02 residual semantic analyzer work, if downstream contract/repair work needs deeper rule metadata or analyzer split.
2. Phase 07 contracts, DB, and generated boundary closure, because durable truth and generated boundaries are central to the moonshot.
3. Phase 10 registry/certified-cell implementation, after Phase 07 has enough contract/DB enforcement to keep cells safe.
4. Phase 12 benchmark/certification governance, after enough real surfaces exist to justify public evidence.
5. Phase 13 autonomous repair/optimization, only after proof, context, registry, migration, and governance are mature enough to prevent unsafe autonomy.

If a later phase is already active in the worktree, continue it only if its hard blockers are satisfied and the log makes the merge path clear.

```text
Stop conditions:
Merge order:
Handoff expectations:
Residual risk:
```

Safe parallel patterns:

* one worker owns schemas while another owns docs that cite the schemas after they lock
* one worker owns Rust internals while another owns fixtures
* one worker owns UX runtime while another owns Rust ingestion after the UX receipt schema locks
* one worker owns security evidence normalization while another owns compliance wording
* one worker owns generated template source while another owns init logic after generator contract locks
* one worker owns benchmark fixtures while another owns report rendering after schemas lock

Unsafe parallel patterns:

* multiple workers editing the same schema without a locked contract
* multiple workers changing report JSON shape independently
* multiple workers changing `agent/test-map.json` independently
* multiple workers editing generated outputs directly
* multiple workers changing permission profiles while patch execution is being enabled
* multiple workers updating paper content while phase implementation receipts are incomplete

Merge order:

1. contracts before implementations
2. implementations before generated outputs
3. generated outputs before docs that cite them
4. docs before receipts
5. receipts before status upgrades
6. final validation before handoff

## Phase Log Routing

Tracked phase logs live under `tips/phases/logs/`. Use one file per phase.

Required log format:

```text
timestamp_utc | actor/tool | phase | action | changed_paths | validation | artifacts | git_sha | residual_risk
```

Required actions:

* `start`: before edits
* `progress`: when scope, validation choice, artifact path, or residual risk changes
* `finish`: at handoff

Rules:

* append only
* use `none`, `pending`, `not-run`, or `unknown` explicitly
* do not leave blank fields
* do not move phase history to `target/jankurai/`
* cite volatile artifacts by path when they matter
* do not rewrite prior receipts or old log entries
* if a phase status changes, the log entry must say why

## Receipt Rules

Every phase completion or status upgrade must include:

```text

Skipped validation:

Residual risk:

```

If validation is not run, the phase cannot be marked complete for that slice. It may be marked `planned`, `partial`, or `pending-validation`.

## Cleanup And Deprecation Policy

* Do not update the paper until paper work is explicitly in scope.
* Do not rename phase files after implementation work starts unless the rename is planned and logged.
* Do not remove old receipts.
* Do not delete docs unless the replacement is linked, routed, and validated.
* Do not deprecate command/schema/rule/artifact paths without compatibility notes.
* Generated artifacts are never cleaned by hand; update sources and run the generator.
* Existing dirty worktree changes are project state unless the active task owns cleanup.
