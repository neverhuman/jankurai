## Phase State Model

Use this vocabulary exactly:

| State           | Meaning                                                                      | Merge implication                                                                      |
| --------------- | ---------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| `planned`       | Phase exists as a plan, with little or no implementation.                    | May start if dependencies are satisfied.                                               |
| `partial`       | Some implementation exists, but material acceptance criteria remain.         | Selectable as active work.                                                             |
| `complete-core` | The dependency-unblocking slice is complete, but residual hardening remains. | Downstream phases may proceed; residual work is selectable when requested or blocking. |
| `complete`      | Phase acceptance criteria are met and receipt-backed.                        | Do not reopen unless regression, hardening, or user request.                           |
| `hardened`      | Phase is complete and has an additional quality/hardening slice.             | Treat as closed unless future phases need changes.                                     |
| `planner-only`  | Commands/schemas emit plans but not full execution/certification.            | Selectable when the moonshot requires evidence beyond planning.                        |
| `blocked`       | Dependencies or policy prevent safe work.                                    | Do not implement until blocker is resolved.                                            |
| `deferred`      | Explicitly postponed with reason.                                            | Do not select by default.                                                              |

A phase can have two statuses:

* `header_status`: the literal status in the phase file header.
* `effective_status`: this index's reconciled state after reading the latest receipt.

Default work selection uses `effective_status`.

## Phase Selection Algorithm

Use this algorithm when the user does not name a phase:

```text
1. Read MASTER_PLAN, this index, the phase ledger, and phase logs.
2. Filter phases whose dependencies are blocked.
3. Prefer the earliest phase with effective_status in:
   partial, partial planner-only, partial dry-run, blocked-but-advanceable.
4. Ignore residual hardening in complete-core phases unless:
   a. the user asked for that phase,
   b. the hardening blocks the earliest active phase,
   c. a regression or validation failure points there.
5. Select the smallest slice that can leave proof evidence in one session.
6. Declare owned paths, forbidden paths, validation, artifacts, and stop conditions.
7. Append start log.
8. Implement.
9. Run mapped validation.
10. Append finish log and update phase receipt.
```

If the user names a phase, use that phase unless dependencies make progress unsafe. If the user names a file, route through the phase that owns that file and state the phase name in the handoff.

## Parallel MCP Work Model

Use parallel workers only when write scopes are disjoint and the merge order is clear.

Safe patterns:

* one worker owns schemas while another owns docs,
* one worker owns Rust CLI internals while another owns report docs,
* one worker owns UX runtime while another owns Rust ingest after the UX receipt schema locks,
* one worker owns security evidence normalization while another owns compliance docs,
* one worker owns benchmark fixtures while another owns report rendering,
* one worker owns generated template sources while another owns init command logic after the generator contract locks.

Unsafe patterns:

* multiple workers editing the same schema without a locked contract,
* multiple workers changing report JSON shape without compatibility notes,
* multiple workers changing `agent/test-map.json` independently,
* multiple workers editing generated outputs directly,
* multiple workers changing paper content while implementation evidence is still moving,
* patch execution and permission model changes running in parallel before permission profiles are locked.

Merge order:

```text
contracts before implementations
implementations before generated outputs
generated outputs before docs that cite them
docs before final validation receipts
```
