## Phase 01 target: move from “complete” to **hardened**

My read: Phase 01 is already marked **complete** in the phase ledger, but not **hardened**. The phase index explicitly distinguishes `complete` from `hardened`, and the current ledger says “01 | complete | standard stabilized,” while later phases like 07, 10, 11, 12, and 13 are already marked hardened. So the right goal is not “implement Phase 01 from scratch”; it is to **promote Phase 01 from complete → hardened** by eliminating every remaining drift path between the standard, schemas, report output, proof lanes, doctor/security evidence, CI, and release receipts. 

Phase 01 is the foundation contract. The phase file says its job is to stabilize v0.4 foundations, clarify schema/report contracts, add compatibility guard expectations, verify doctor/security lane reality, separate implemented behavior from roadmap claims, and define release evidence conventions.  The repo has already shipped a lot of that: the standard files, schema validation, report compatibility tests, doctor, security evidence, artifact contract docs, CI workflow, and phase feedback reconciliation are all present.    

The hardening work is about making those contracts impossible to drift.

# 3. The remaining Phase 01 risks I would close

## Risk A — Standard/rule drift

`agent/JANKURAI_STANDARD.md` lists stable rule IDs through `HLT-021`, including CI hardening and destructive migration.  The fetched full standard’s stable rule table stops earlier, at `HLT-019`, while later sections discuss many related gates.  That is a classic Phase 01 hardening problem: agents reading different standard files may form different expectations.

**Fix:** create a single machine-readable rule registry and make both standard docs derive from or get tested against it.

## Risk B — Proof lane vocabulary drift

The standard brief says required lanes include `fast`, `contract`, `db`, `db-migration-analyze`, `web`, `e2e`, `security`, `observability`, `audit`, and `release`.  `docs/testing.md` lists a similar but not identical set and uses `full`.  `agent/proof-lanes.toml` says the **live minimal lane set** is `fast`, `audit`, `paper`, `security`, `full`, plus repo-specific helper lanes. 

That is not necessarily wrong, but it must be explicit: live lanes, aliases, target lanes, and roadmap lanes need a single compatibility story.

**Fix:** define lane classes: `canonical`, `alias`, `repo-local`, and `roadmap`. Then test `agent/test-map.json` commands against `agent/proof-lanes.toml`.

## Risk C — Version constants can drift

`model.rs` hardcodes `STANDARD_VERSION`, `AUDITOR_VERSION`, `SCHEMA_VERSION`, `PAPER_EDITION`, and target stack constants.  `agent/standard-version.toml` contains the same version bindings and artifact bindings.  The compatibility guard already checks emitted report fields against the manifest, which is excellent. 

But hardcoding means future drift is still possible until the check catches it. Hardened Phase 01 should either load the manifest into the report path or generate the constants from it.

**Fix:** make `agent/standard-version.toml` the source of truth for report versions, with fallback only in tests/fixtures.

## Risk D — Artifact contract index can drift

`ArtifactSchema` is in Rust code, while `docs/artifact-contracts.md` is prose.   Today, humans have to remember to update both.

**Fix:** add a machine-readable artifact catalog or a test that checks every `ArtifactSchema` variant appears in the artifact index or a documented exemption list.

## Risk E — Doctor receipts are too thin for hardened status

`doctor` checks many important things, validates control-plane files, validates proof/security/UX/migration artifacts when present, writes receipts, and distinguishes missing security tools as low/advisory.  But `doctor-receipt.schema.json` currently only requires `check_id`, `severity`, `path`, and `message` per diagnostic. 

For hardened Phase 01, doctor diagnostics should carry enough structure for an agent to repair or skip safely: category, class, retryability, owner, lane, docs URL, install hint, expected artifact, and whether the issue is a repo defect or local environment condition.

**Fix:** expand doctor diagnostics and receipt schema without breaking compatibility.

## Risk F — Security per-tool evidence depends on `python3`

The security wrapper parses `jankurai-security-step=` JSON lines into per-tool evidence.  But the shell script emits those JSON lines only when `python3` is available; otherwise the envelope remains valid but falls back to wrapper-level evidence.  The docs acknowledge this. 

For hardened Phase 01, per-tool security evidence should not silently degrade because Python is missing.

**Fix:** move per-tool orchestration into Rust or emit JSON from shell without Python dependency.

## Risk G — Audit output examples and target stack identity need tightening

`model.rs` distinguishes `target_stack_id` from a human-readable `target_stack`.  `agent/standard-version.toml` uses `target_stack = "rust-ts-vite-react-postgres-bounded-python"`.  `docs/release-plan.md` shows an example where `target_stack` is also the stack ID string. 

This is small, but Phase 01 is exactly where small contract inconsistencies should die.

**Fix:** standardize `target_stack_id` as the slug and `target_stack` as the human-readable label everywhere, or explicitly document that manifest `target_stack` maps to report `target_stack_id`.

## Risk H — CLI policy override should be verified

`main.rs` exposes `--policy` on `AuditArgs`.  In the inspected `run_audit_and_write` path, the main flow shows mode, baseline, fail-under, and fail-on handling, but I did not see the policy path being applied in the visible section. Because the fetched file was truncated near the later helper code, this should be verified directly before changing it.

**Fix:** either wire `--policy` to load an alternate audit policy, add a compatibility test for it, or deprecate/remove the argument with a migration note.

let manifest = standard_version::load_standard_version_manifest(root)?;

report.auditor_version = manifest.auditor_version;
report.schema_version = manifest.schema_version;
report.paper_edition = manifest.paper_edition;

Given the repo is v0.4, Option B is safer unless a schema migration is already planned.

for schema in ArtifactSchema::ALL {

# 11. Workstream 7 — harden security evidence

## Objective

Security lane evidence should be durable, per-tool, parseable, and independent of incidental local Python availability.

## Owned paths

```text

crates/jankurai/tests/security_evidence_smoke.rs

Justfile

```

## Current state

`jankurai security run` runs a shell script, writes a log, parses `jankurai-security-step=` JSON lines into `commands[]`, validates `target/jankurai/security/evidence.json`, and fails if the wrapper exits nonzero.  The shell script treats `gitleaks`, `cargo-audit`, and `npm` as required in strict mode, and `syft`/`zizmor` as advisory. 

## Implementation steps

### Step 7.1 — remove Python dependency for step JSON

Best fix: move tool orchestration into Rust. Add a policy-driven tool list:

```rust
struct SecurityTool {
    label: &'static str,
    tool: &'static str,
    command: &'static str,
    required_in_strict: bool,
    advisory: bool,
}
```

Then `security run` itself checks `command -v`, runs available tools, captures exit codes/stderr excerpts, and writes evidence.

If keeping shell, replace Python JSON emission with a robust shell-safe encoder or a small Rust helper. The current shell only emits per-step JSON when `python3` exists. 

### Step 7.2 — extend `SecurityLaneStep`

Current fields include label, command, tool, status, exit code, advisory, stderr excerpt, finding count, highest severity, normalized decision.  Use them fully.

Set:

```text
status = ran | skipped | failed
normalized_decision = pass | advisory-skip | fail | blocked
highest_severity = none | low | medium | high | critical
finding_count = integer when parser can determine it, otherwise null
```

### Step 7.3 — make strict/advisory semantics exact

Rules:

* Missing required tool:

  * non-strict: `skipped`, advisory, decision `advisory-skip`, wrapper exits 0
  * strict: `failed`, non-advisory, decision `blocked`, wrapper exits 1
* Failing required tool:

  * always failed
* Missing advisory tool:

  * skipped, advisory, wrapper exits 0
* Failing advisory tool:

  * choose policy: either advisory failure exits 0 with decision `review`, or exits nonzero. The current shell exits on advisory command failure.  Document the chosen behavior.

### Step 7.4 — add fixture tests

Test:

1. fake all tools present and success
2. fake required tool missing non-strict
3. fake required tool missing strict
4. fake advisory tool missing
5. malformed `jankurai-security-step=` line ignored with warning
6. no step lines still produces valid wrapper-level evidence
7. evidence validates against schema

### Step 7.5 — improve audit ingest summary

`Report` already has `security_evidence` readiness summary.  Ensure ingest summary includes:

* commands ran/skipped/failed
* strict flag
* generated_at
* git_head
* highest normalized decision
* missing required tools count

If adding fields, update `repo-score.schema.json` and compatibility tests.

## Acceptance criteria

* `just security` always writes schema-valid evidence.
* Evidence contains one row per expected security tool.
* Missing local tools are distinguishable from security failures.
* Strict mode blocks missing required tools.
* Audit report ingests security evidence summary.

## Validation

# 18. Merge order

1. **Packet A: version/rule/lane contract**
   This removes ambiguity from every later change.

2. **Packet B: artifact schema registry**
   This locks machine-readable surfaces.

3. **Packet C: report compatibility**
   This ensures output remains stable after schema/rule updates.

4. **Packet D: doctor/security evidence**
   This makes environment and security evidence repairable.
