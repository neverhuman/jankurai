I inspected the repo surfaces that matter for Phase 01. My verdict: **Phase 01 is functionally complete, but not yet “100% hardened.”** The right move is not to rebuild it. The right move is a targeted **Phase 01 Hardening Pass** that turns the standard from “documented and mostly guarded” into a **self-verifying contract**: standard text, brief, rule catalog, schemas, report output, version manifest, doctor/security evidence, CI, and phase receipts all fail fast when they drift.

The key reason this matters: Phase 03 Proof Router and Phase 02 Rule Engine both depend on Phase 01 being immovable. If the standard is even slightly inconsistent, proof routing can select the wrong lane, rules can emit findings that docs do not define, repair agents can trust stale schemas, and downstream “hardened” phases become built on sand.

---

## Current Phase 01 state

Phase 01 is currently marked **complete**, not hardened. The Phase 00 ledger says Phase 01 is complete, while its status taxonomy reserves **hardened** for “complete plus an additional validated production-quality slice.” 

The Phase 01 plan already captures the correct intent: stabilize v0.4 foundations, clarify schema and report contracts, add compatibility guard expectations, verify doctor/security lane reality, separate implemented behavior from moonshot roadmap docs, and define release evidence conventions. It also says Phase 01 can touch `crates/jankurai/`, `schemas/`, `docs/`, `agent/`, and `Justfile`, while avoiding `paper/`, `reference/`, generated hand-edits, new moonshot features, report-field renames without migration notes, and new mandatory external tools in `just fast`. 

A lot is genuinely done. The Phase 01 log shows completed work for repo-score schema parity, control-plane schemas, report compatibility guards, security evidence wiring, Markdown section guards, finding schema coverage, and parsing of `jankurai-security-step=` security log rows. 

So the hardening plan should be framed as:

> **Upgrade Phase 01 from `complete` to `hardened` by making every Phase 01 contract machine-checked, drift-resistant, and release-evidence-backed.**

# The most important gaps to close

## P0 gap 1 — Standard/rule catalog drift

This is the biggest Phase 01 hardening gap I found.

`agent/JANKURAI_STANDARD.md` lists stable rules through `HLT-021-DESTRUCTIVE-MIGRATION`, including `HLT-020-CI-HARDENING-GAP` and `HLT-021-DESTRUCTIVE-MIGRATION`. 

The Rust auditor’s canonical `RULES` table also includes `HLT-020-CI-HARDENING-GAP` and `HLT-021-DESTRUCTIVE-MIGRATION`. 

But the full `docs/agent-native-standard.md` stable-rule table currently stops at `HLT-019-STREAMING-RUNTIME-DRIFT`. 

That means the repo has a **real source-of-truth inconsistency**: the agent brief and auditor know about rules that the full standard’s stable table does not fully define. Phase 01 cannot be called hardened while that drift can exist.

### Fix

Add a machine guard that fails when any rule ID in `audit::rules::all()` is absent from:

* `docs/agent-native-standard.md`
* `agent/JANKURAI_STANDARD.md`
* `docs/audit-rubric.md`
* optionally `docs/testing.md` when the rule is proof-lane-specific

Then reconcile the docs by adding `HLT-020` and `HLT-021` to the full standard’s stable-rule table and ensuring their category, lane, TLR, docs link, severity, and repair posture match the code.

---

## P0 gap 2 — Versions are checked, but still too hardcoded

The version manifest is clearly intended to be canonical: `agent/standard-version.toml` pins `standard_version = "0.4.0"`, `auditor_version = "0.4.0"`, `schema_version = "1.2.0"`, `paper_edition = "2026.05-ed3"`, and the target stack ID. 

But `crates/jankurai/src/model.rs` hardcodes the same constants, and the audit report is built from those constants. 

There is already a `versions` check that compares manifest values to hardcoded constants and scans several docs/files for expected version strings. 

That is good, but not fully hardened. A hardened Phase 01 should make the manifest the actual source used by report generation, or at minimum add stronger negative tests proving drift fails every relevant command.

### Fix

Prefer this design:

* Add a `StandardVersions` struct.
* Parse and validate `agent/standard-version.toml` at audit/report/security-evidence generation time.
* Use the parsed manifest values in emitted reports.
* Keep constants only as fallback defaults for non-repo contexts or compile-time package metadata.
* Make `jankurai versions` fail if any of these drift:

  * `VERSION`
  * `crates/jankurai/Cargo.toml`
  * `packages/ux-qa/package.json`
  * `agent/standard-version.toml`
  * `agent/audit-policy.toml`
  * `docs/agent-native-standard.md`
  * `agent/JANKURAI_STANDARD.md`
  * emitted `repo-score.json`
  * `security-evidence.json`

Acceptance test: create a fixture with `standard-version.toml` changed to `0.4.1` and prove `versions` fails unless every bound surface also changes.

## P0 gap 3 — Schema validation is not yet strong enough

The repo has a custom validator in `crates/jankurai/src/validation.rs`. It handles core pieces like `$ref`, `const`, `enum`, `pattern`, `type`, `required`, `properties`, `additionalProperties`, arrays, `allOf`, `anyOf`, and `oneOf`. 

But the schemas use JSON Schema keywords such as `minimum`. For example, `repo-score.schema.json` uses `minimum: 0` for fields like `elapsed_ms`, counts, and other numeric summaries. 

The custom validator shown does not enforce `minimum`, `maximum`, `maxLength`, `format`, `uniqueItems`, `patternProperties`, `propertyNames`, or other draft 2020-12 keywords. That means an artifact can “validate” even though it violates declared schema constraints. The current dependencies also do not include a full JSON Schema validator crate. 

Phase 01 cannot be considered 100% hardened while schema validation is weaker than the schema language the repo claims to use.

### Fix

Adopt one of two approaches.

The stronger path:

* Add a real JSON Schema validator dependency.
* Validate all `schemas/*.schema.json` using draft 2020-12 semantics.
* Keep the current validator only as a lightweight fallback or remove it once the new validator is stable.

The acceptable fallback path:

* Keep the custom validator.
* Add a `schema_keyword_coverage` test that scans every schema and fails if it finds a keyword not implemented by the validator.
* Implement all currently used keywords, especially:

  * `minimum`
  * `maximum`
  * `minLength`
  * `maxLength`
  * `uniqueItems`
  * `format`
  * `patternProperties`
  * `propertyNames`
  * nested `$defs`
  * `additionalProperties` object forms
  * `items` array/object forms

Acceptance test: an object with `"elapsed_ms": -1` must fail against schemas that declare `"minimum": 0`.

## P0 gap 4 — Doctor receipt diagnostics are too thin

`doctor.rs` already performs many valuable checks: required control files, manifest schemas, lockfiles, stale score, local path leaks, echo-only workflow proof, missing security tools, UX artifacts, proof receipts, evidence index, security evidence, and other JSON artifacts. 

But the diagnostic shape is only:

```text
check_id
severity
path
message
```

The doctor receipt schema matches that thin shape and forbids additional properties. 

That is not hardened enough for Phase 01’s stated gap: “doctor and security lane diagnostics need to distinguish missing tools from broken repo state.” 

Right now, a missing tool is mostly distinguished by message text and low severity. For a repair agent, that is not enough. It needs a structured diagnostic kind.

### Fix

Extend doctor diagnostics to include:

```rust
#[serde(rename_all = "kebab-case")]
enum DiagnosticKind {
    RepoDefect,
    MissingTool,
    MissingControlFile,
    SchemaFailure,
    StaleEvidence,
    EnvironmentNotice,
    GeneratedArtifactDrift,
    SecurityEvidenceGap,
}
```

Add fields:

```text
kind
environment_sensitive
strictly_blocking
common_fixes
docs_url
```

Then update `schemas/doctor-receipt.schema.json`.

Missing tools should emit:

```json
{
  "kind": "missing-tool",
  "severity": "low",
  "environment_sensitive": true,
  "strictly_blocking": false,
  "common_fixes": [
    "install the tool locally",
    "rerun just security",
    "use strict mode only in release validation"
  ]
}
```

Repo defects should emit:

```json
{
  "kind": "repo-defect",
  "severity": "high",
  "environment_sensitive": false,
  "strictly_blocking": true
}
```

Acceptance test: missing `gitleaks` in normal mode is not treated as a repo defect; missing `AGENTS.md` is.

## P0 gap 5 — Security policy, shell lane, and evidence schema disagree on strictness

The security shell script treats `gitleaks`, `cargo-audit`, and `npm` as required in its `required_tool_names`, and treats `syft` and `zizmor` as advisory. 

But `agent/security-policy.toml` says only `gitleaks` is required, while `cargo audit` and `npm audit` are advisory. 

The security evidence schema permits command statuses `ran`, `skipped`, and `failed`, but it does not encode whether a skipped command was missing-tool advisory, policy-optional, or release-blocking. 

That creates a subtle but important Phase 01 issue: the **policy**, **runner**, and **evidence envelope** are not fully aligned.

### Fix

Make `agent/security-policy.toml` the source of truth for tool strictness.

Concrete changes:

1. `jankurai security run` loads `agent/security-policy.toml`.
2. The runner passes required/advisory tool status into the script or replaces the hardcoded arrays in `tools/security-lane.sh`.
3. `SecurityLaneStep.status` expands to:

   * `ran`
   * `skipped-missing-tool`
   * `skipped-policy-optional`
   * `failed`
4. Add `required_by_policy: boolean`.
5. Add `blocking: boolean`.
6. Strict mode fails if any `required_by_policy && skipped/failed`.
7. Normal mode records missing tools as environment-sensitive evidence, not as repo defects.

Acceptance test: with `gitleaks` required and missing, `just security-strict` fails; with `cargo audit` advisory and missing, normal `just security` passes but records a skipped advisory step.

# Target hardened definition

* Every stable rule ID in the Rust rule catalog appears in the full standard and brief.

pub struct StandardVersions {

# Hard parts and how to avoid mistakes

## Hard part 1 — Do not create a new “source of truth” accidentally

The repo already has several source-like surfaces:

* full standard
* agent brief
* rule table
* version manifest
* audit policy
* schemas
* artifact-contract docs

The hardening pass should not add yet another manual catalog unless it is generated or guarded.

Best pattern:

```text
Code rule table -> tested against docs
Standard version manifest -> used by emitted reports
Schemas -> tested against artifacts
Artifact contracts doc -> tested against schemas/ArtifactSchema
```

## Hard part 2 — JSON Schema migration can break many tests

If you add a real validator, previously “valid” outputs may fail because the custom validator was permissive.

Handle this safely:

1. Add the validator.
2. Run existing tests.
3. Fix schemas or outputs, not the validator.
4. Only relax schema constraints when the contract was truly over-specified.
5. Add migration notes for any output shape change.

## Hard part 3 — Security lane must remain useful locally

Do not make normal `just security` impossible on machines without every tool.

Correct posture:

* normal mode: missing optional tools become structured advisory evidence
* strict mode: missing required tools block
* release CI: install required tools, then use strict mode

## Hard part 4 — Phase logs are append-only

Do not rewrite prior Phase 01 log entries. Add new entries only. The logs already show prior Phase 01 completions; the new work should be labeled hardening, not retroactive completion. 

## Hard part 5 — Do not edit generated artifacts by hand

If `agent/repo-score.json` or `agent/repo-score.md` changes, regenerate through:

```bash
cargo run -p jankurai -- . --json agent/repo-score.json --md agent/repo-score.md
```

That is the generated-zone command already declared.
