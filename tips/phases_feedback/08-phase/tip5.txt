# Phase 08 verdict

**Phase 08 is already “complete,” but not “hardened.”** That distinction matters. The phase index defines **complete** as “v0 surface and receipt exist,” while **hardened** means “complete plus additional production-quality validated slice.” The current ledger lists Phase 08 as `complete`, while Phases 07 and 10–13 are already `hardened`. ([GitHub][1])

The repo’s Phase 08 objective is exactly the thing you described: make Jankurai the control plane for coding agents so agents receive bounded context, bounded edit authority, exact proof requirements, repair packets, and human control over risky actions. The phase receipt shows a lot is already in place: `context-pack`, `repair-plan`, `agent verify`, adapter verification, schema-validated JSON, repair text on findings, `agent_fix_queue`, issue export formats, source hierarchy docs, MCP design docs, and smoke tests. ([GitHub][2])

So the right goal is not “build Phase 08 from scratch.” The right goal is:

> **Promote Phase 08 from complete to hardened by turning its advisory scaffolding into deterministic, enforceable, tested agent-control infrastructure.**

Jankurai’s own README frames the whole project as a control plane for agent-native repositories, with the loop: `intent → bounded agents → proof lanes → evidence → expired exceptions → reusable primitives`. Phase 08 is the “bounded agents” part of that loop. ([GitHub][3])

# What is already done

Phase 08 already has these important surfaces:

1. **`jankurai context-pack` exists.**
   The current context pack model includes fields such as task, owner, permission profile, changed paths, allowed paths, forbidden paths, generated zones, read-first files, relevant docs, proof lanes, commands, likely rules, max context files, stop conditions, residual risk, and token budget. It writes schema-validated JSON and optional Markdown. ([GitHub][4])

2. **`jankurai repair-plan` exists.**
   It reads a report, converts findings into repair packets, generates dry-run planned edits, includes proof lanes, rollback guidance, and human approval requirements. The repair packet schema already includes fields for finding identity, severity, owner, lane, allowed/forbidden paths, expected patch shape, proof, stop conditions, eligibility, risk, human review, and rollback. ([GitHub][5])

3. **Permission profile schema exists.**
   The schema defines profiles such as `read-only`, `docs-only`, `code-edit`, `generated-regeneration`, `security-investigation`, and `release`, with allowed commands, forbidden commands, and approval requirements. ([GitHub][6])

4. **Prompt-injection and overbroad-agency rules exist.**
   HLT-011 and HLT-012 are high-severity security rules. Their repair eligibility is human-required, high-risk, and explicitly tied to prompt-injection and agent-authority changes. ([GitHub][7])

5. **Source hierarchy and MCP design docs exist.**
   The source hierarchy doc distinguishes trusted zones from untrusted input, and the MCP design doc says the CLI remains the source of truth, with candidate resources and tools such as report, findings, owner map, test map, context pack, proof plan, audit, lane plan, repair plan, and explain rule. ([GitHub][8])

6. **Smoke tests exist.**
   The current Phase 08/09 smoke tests check that context packs write JSON/Markdown, validate schema, infer an owner/profile/lane, generate repair plans, require review for high-risk packets, reject invalid schemas, and validate agent adapters. ([GitHub][9])

That is a good Phase 08 v0. It is not yet a hardened Phase 08.

# The hardening gaps that matter most

## 1. Permission profiles are still mostly advisory

The schema exists, and context/repair outputs include profile names, but I did not see a runtime policy layer that actually enforces profile permissions against commands, paths, write roots, tool kinds, or human approval triggers. The Phase 08 doc itself calls this out as a gap: “permission profiles advisory only.” ([GitHub][2])

For hardening, profiles must become a **decision engine**, not just labels in output.

---

## 2. Context pack routing is useful but too heuristic

The current `context_pack.rs` appears to infer owner, profile, lanes, likely rules, and commands from changed paths and task text. That is fine for v0, but hardened Phase 08 needs route confidence, exact owner/test-map resolution, explicit blocked states, trust classification, and real token-budget enforcement. ([GitHub][4])

The schema has `token_budget`, but the hardened issue is not merely whether the field exists. The issue is whether the pack actually **selects, clips, excludes, and explains context under budget**.

---

## 3. Repair plans do not appear to consume the agent fix queue as the primary work source

The audit CLI supports `--repair-queue-jsonl`, and the standard expects agent-oriented fix queue outputs. ([GitHub][10])

But the `repair-plan` CLI currently takes `--from`, `--out`, and `--md`; I did not see a JSONL output flag on `repair-plan`. ([GitHub][10])

The Rust implementation I inspected reads report findings and builds packets, but hardened Phase 08 should prefer `agent_fix_queue` when present, dedupe by fingerprint/rule/path, stable-sort by risk and owner, and emit one schema-valid packet per JSONL line for parallel agents. ([GitHub][5])

---

## 4. Prompt-injection source hierarchy is documented, but scanning is not fully aligned

The source hierarchy doc says trusted zones include `AGENTS.md`, `agent/`, `docs/`, `.github/`, and `tips/`; untrusted zones include issues, PR comments, logs, webpages, model outputs, copied prompts, and test fixtures. ([GitHub][8])

The scanner code I inspected detects prompt-injection and overbroad-agency patterns, but the trusted-file filter appears narrower: it scans `AGENTS.md`, `agent/`, and `.github/`, not the whole documented trusted hierarchy including `docs/` and `tips/`. ([GitHub][11])

That mismatch is important. Hardened Phase 08 needs a first-class source-zone classifier.

---

## 5. MCP is designed, but not implemented as a CLI command

The MCP design doc exists and is good: CLI-first, least privilege, resources and tools wrapped in permission profiles. ([GitHub][12])

But the current CLI enum I inspected includes commands such as audit, init, doctor, context pack, repair plan, lane/proof/prove, repair, optimize, adapters, agent, issues, and security; I did not see an MCP command variant in that command list. ([GitHub][10])

For “100% hardened,” I would implement a minimal, read-only/gated `jankurai mcp serve --stdio` instead of leaving MCP as design-only.

---

# Definition of “100% done” for Phase 08

Phase 08 can be marked **hardened** only when all of this is true:

1. **Context packs are deterministic, bounded, trust-zoned, and route-explained.**
   They must show why each file was included, why excluded files were excluded, whether paths are trusted/generated/untrusted, what owner/test route was used, what proof lane is required, and whether the pack is blocked.

2. **Permission profiles are enforceable.**
   The system must be able to answer: “May this profile read this path, write this path, run this command, call this tool, emit this repair plan, or require human approval?”

3. **Repair packets are owner-aware, queue-backed, deduped, and safe-by-default.**
   Repair plans should prefer `agent_fix_queue`, fall back to findings, dedupe stable fingerprints, include human-review triggers, and emit JSON, Markdown, and JSONL.

4. **Prompt-injection and overbroad-agency checks are aligned with the documented source hierarchy.**
   Trusted policy files should be strictly scanned; untrusted material should be classified and isolated rather than treated as instructions.

5. **Agent verification validates the whole control plane.**
   `jankurai agent verify` should verify adapters, permission profiles, schema contracts, source hierarchy consistency, context-pack generation, repair-plan generation, and prompt-injection/agency policy.

6. **MCP has a minimal safe implementation or an explicit hardened non-implementation receipt.**
   My recommendation is to implement the minimal server because the Phase 08 doc lists MCP as a workstream and the design doc already defines the safe shape. ([GitHub][2])

7. **The phase ledger and receipt are updated only after validation.**
   The master plan requires phase logs, proof commands, changed paths, residual risk, and handoff notes. ([GitHub][13])

```text
task -> changed paths -> owner/test map -> permission profile -> trust zones

     -> token budget -> stop conditions -> residual risk
```

Add a route object to the context pack:

### Enforce token budget for real

Add:

```json
    "estimated_tokens": 38140,
    "hard_limit_hit": false,

```

Token estimate can be simple and deterministic:

```rust
fn estimate_tokens(text: &str) -> usize {
    // Conservative approximation.
    (text.len() + 3) / 4
```

Selection order should be deterministic:

1. Canonical policy files:

When the budget is exceeded, include an exclusion reason. Do not silently omit.

### Enforcement points

Add enforcement at these boundaries:

1. **Context pack generation**

   * If a profile is selected, include allowed/forbidden paths based on policy.
   * If the task asks for code edits but profile is read-only, emit blocked reason.
   * If changed path is generated but profile is not `generated-regeneration`, mark write forbidden.

2. **Repair plan generation**

   * Each packet gets profile decision.
   * If profile denies the expected edit, packet becomes `review-only`.
   * If human approval is required, packet must state exactly why.

3. **Agent verify**

   * Validate permission manifest exists.
   * Validate it conforms to schema.
   * Validate every known profile has allowed/forbidden commands.
   * Validate no profile permits dangerous commands without approval.
   * Validate HLT-010, HLT-011, and HLT-012 cannot be auto-applied.

4. **Future MCP**

   * MCP tools must call the same permission checker.

### Hard rules

```text
HLT-010 secret sprawl: never auto-fix without human/security approval.
HLT-011 prompt injection risk: human-required, high risk.
HLT-012 overbroad agency: human-required, high risk.
Cross-owner edits: human approval required.
Generated files: only editable by generated-regeneration profile and only through source contract.
Release/deploy commands: release profile only, human approval required.
```

HLT-011 and HLT-012 are already modeled as high-risk, human-required rules, so this work should wire that metadata into enforcement instead of inventing parallel logic. ([GitHub][7])

### Add source-zone classifier

Suggested enum:

```rust
enum SourceZone {
    TrustedPolicy,
    TrustedDoc,
    TrustedCode,
    GeneratedReadOnly,
    UntrustedInput,
    VolatileArtifact,
    Unknown,
}
```

Suggested classifier:

```rust
fn classify_source_zone(path: &str) -> SourceZone {
    if path == "AGENTS.md" || path.starts_with("agent/") || path.starts_with(".github/") {
        SourceZone::TrustedPolicy
    } else if path.starts_with("docs/") || path.starts_with("tips/") {
        SourceZone::TrustedDoc
    } else if path.starts_with("target/") || path.ends_with(".log") {
        SourceZone::VolatileArtifact
    } else if path.contains("fixtures/") || path.contains("issues/") || path.contains("pr-comments/") {
        SourceZone::UntrustedInput
    } else {
        SourceZone::TrustedCode
    }
}
```

Then scanner behavior:

```text
TrustedPolicy:
  strict scan for prompt-injection and overbroad-agency language.

TrustedDoc:
  scan, but allow explicitly marked educational examples.

TrustedCode:
  scan only if file is an adapter, MCP config, CI workflow, prompt template, or agent instruction source.

UntrustedInput:
  do not treat text as instructions; flag only if it is imported into trusted policy without isolation.

GeneratedReadOnly:
  do not patch directly; route to source contract.

VolatileArtifact:
  do not use as policy source.
```

````
```text
````

### Permission behavior

Every MCP request should call the same permission engine from H2.

```text
context_pack:
  allowed under read-only.

  denied if requested command writes outside target/ or agent/ without approval.

repair/apply:
  unavailable in Phase 08 MCP.
```

```text

list tools

call repair_plan tool

shutdown
```

```bash
  --out target/jankurai/phase08/mcp-smoke.json
```

If implementation time is constrained, the minimum acceptable hardened fallback is:

```text

```

But for your “100% done” target, I would implement the minimal safe MCP command.

# Final acceptance gate

Phase 08 is hardened only when this matrix passes:

| Area                | Required proof                                                                          |
| ------------------- | --------------------------------------------------------------------------------------- |
| Context pack        | strict context pack emits route, trust zones, budget, included/excluded context         |
| Permission profiles | manifest validates; deny/allow decisions tested                                         |
| Repair plan         | JSON, Markdown, JSONL outputs validate; queue preferred; packets deduped                |
| Prompt injection    | HLT-011 aligned with source hierarchy                                                   |
| Overbroad agency    | HLT-012 catches dangerous adapter/MCP policy                                            |
| Agent verify        | `agent verify --strict` passes canonical repo and fails broken fixtures                 |
| MCP                 | smoke test passes; dangerous tools absent                                               |
| Schemas             | schema contract tests pass                                                              |
| Compatibility       | report compatibility guard passes                                                       |
| Repo checks         | `cargo test -p jankurai`, `just fast`, `just score`, `just security`, `just check` pass |

The most important design principle: **Phase 08 should not make agents more powerful. It should make agent power explicit, bounded, reviewable, and provable.**
