## Phase 08 — Agent Context & Repair: hardening plan to finish it to 100%

I inspected the repo through GitHub. I could not clone it locally because the container could not resolve `github.com`, so I could not run the test suite. The plan below is based on repository/code inspection rather than local execution.

My verdict: **Phase 08 has a strong scaffold, but it is not yet “100% hardened.”** The phase file marks it complete, and the repo already has `context-pack`, `repair-plan`, adapter verification, schemas, agent docs, and smoke tests. However, the same Phase 08 file still names hardening gaps: richer owner-aware repair scoping, advisory-only permission enforcement, sharper source hierarchy for prompt-injection defense, no MCP/server surface, and schema refinements that were only partially addressed in the receipt. ([GitHub][1])

Jankurai’s core loop is meant to route intent into bounded agents, proof lanes, evidence, expiring exceptions, and reusable primitives. Phase 08 must therefore become the **control-plane layer that prevents agents from receiving too much context, editing too broadly, trusting hostile instructions, or proving with the wrong lane**. ([GitHub][2])

# 2. Current hardening gaps I would treat as blockers

## 2.1 Context packs are useful, but still too heuristic

`context-pack.rs` currently builds allowed paths from changed paths, inferred roots, owner prefixes, and task-keyword heuristics. It also hard-codes `max_context_files: 12` and `token_budget: 100000`. ([GitHub][5])

That means Phase 08 does not yet have a real bounded-context guarantee. The output says “token budget,” but the system does not appear to calculate actual included bytes/tokens, enforce truncation, record omitted files, or fail when the task cannot be safely represented within budget.

## 2.2 Path matching is too permissive

The repo catalog uses a `path_matches(path, prefix)` helper that returns true when `path == prefix`, `path.starts_with(prefix)`, **or `prefix.starts_with(path)`**. ([GitHub][4])

That last branch is dangerous for agent scoping. It can cause short paths to match broader prefixes than intended. For a hardened agent system, path matching needs to be segment-aware and directionally explicit:

* “Does this changed path fall under this owner/test/generated prefix?”
* “Does this allowed prefix explicitly include this file?”
* “Is this requested edit exactly inside the allowed write set?”

Those are different questions and should not share a symmetric matcher.

## 2.3 Permission profiles exist, but enforcement is still weak

Phase 08 explicitly notes that permission profiles exist as schemas, but lane enforcement is advisory only. ([GitHub][1])

The code currently infers a profile such as `docs-only`, `read-only`, `code-edit`, `generated-regeneration`, `security-investigation`, or `release`, but a profile string is not enough. A hardened system must enforce:

* allowed file globs;
* forbidden file globs;
* allowed commands;
* forbidden commands;
* whether network access is allowed;
* whether secrets/env vars are allowed;
* whether generated files can be edited;
* whether `git` operations are allowed;
* whether human approval is required;
* max changed files;
* max diff size;
* allowed proof lanes.

Without enforcement, agents can still receive good advice and then do the wrong thing.

## 2.4 Repair plans are structured, but not yet safely executable

`RepairPlan` has the right shape, and `RepairPacket` carries strong fields: finding fingerprint, path, rule, severity, owner, lane, problem, permission profile, allowed paths, forbidden paths, expected patch shape, required proof, stop conditions, repair eligibility, risk, human-review requirement, and rollback. ([GitHub][6])

However, the current `planned_edit_from_packet` path appears to emit planned edits with `apply_strategy: "none"` and empty patch primitives. ([GitHub][6])

That is safe, but it means Phase 08 is mostly a planner, not a hardened repair execution layer. To finish Phase 08, repair plans must either:

1. explicitly remain **planner-only** and make that contract impossible to misread; or
2. support a restricted set of safe, deterministic, machine-checkable edits with strict preconditions.

The best hardened design is both: planner-only by default, with narrowly eligible safe-auto repairs for low-risk, known-rule cases.

## 2.5 Prompt-injection defense needs to become machine-enforced

The standard says agent instructions should minimize tokens, use owner/test maps, exclude build/generated output, and separate tool-specific instructions from canonical docs. It also lists prompt injection and overbroad agency as stable rule IDs: `HLT-011-PROMPT-INJECTION` and `HLT-012-OVERBROAD-AGENCY`. ([GitHub][8])

The Phase 08 receipt says `docs/source-hierarchy.md` was touched, and the phase notes call for sharper source hierarchy. ([GitHub][1])

For 100% hardening, source hierarchy cannot just be a doc. It needs to be a parsed policy that determines which files may provide instructions, which files are untrusted evidence, and which files must be quoted but never obeyed.

## 2.6 MCP/server support is not done

Phase 08 names MCP/server surface as a target command, but marks it optional/later. The gap list says there is no MCP/server tool surface yet, while the receipt says an MCP design doc was added. ([GitHub][1])

For “100% done,” MCP can be minimal, but it needs to exist as a real read-only control-plane interface or be explicitly deferred out of Phase 08. My recommendation is to implement a small read-only MCP server and keep write/apply tools disabled by default.

# 3. The hardened end state

Phase 08 is complete only when this statement is true:

> Given a task, changed path set, audit finding, or repair request, Jankurai can deterministically produce a schema-valid context pack and repair plan that tells an agent exactly what it may read, what it may edit, what it must not trust, what proof lane it must run, when it must stop, and when a human must approve. The same restrictions are enforced by CLI commands, adapters, repair apply, doctor checks, and MCP.

The desired behavior should be:

1. **No agent receives unbounded context.**
2. **No agent gets write access outside its owner/path/rule scope.**
3. **No agent can treat untrusted repo content as policy.**
4. **No repair plan can invent proof commands.**
5. **No generated artifact is edited without tracing to its source contract.**
6. **No high-risk security, migration, prompt-injection, release, or reference-data change auto-applies.**
7. **Every output validates against schemas.**
8. **Every repair has stop conditions, rollback guidance, and proof requirements.**
9. **Every adapter is generated from the same canonical contract.**
10. **Every Phase 08 artifact can be consumed by machines, not just humans.**

## Workstream B — Make path scoping precise and safe

### Goal

Eliminate broad or symmetric path matching.

### Current concern

`path_matches` currently treats `prefix.starts_with(path)` as a match. ([GitHub][4])

That is not safe enough for edit authority.

### Required changes

Create separate functions for separate questions:

```rust
fn path_under_prefix(path: &RepoPath, prefix: &RepoPrefix) -> bool;
fn prefix_under_path(prefix: &RepoPrefix, path: &RepoPath) -> bool;
fn exact_or_child(path: &RepoPath, dir: &RepoPath) -> bool;
fn edit_allowed(path: &RepoPath, allowed: &[RepoPrefix], forbidden: &[RepoPrefix]) -> bool;
```

Use segment-aware matching only. For example:

* `agent/context-pack.rs` should match `agent/`.
* `agentic/foo.rs` should **not** match `agent/`.
* `docs` should match `docs/agent-native-standard.md` only if `docs/` is explicitly a directory prefix.
* `.` or empty path should never imply full-repo edit authority.
* `reference/`, `target/`, generated zones, build output, and vendored/generated artifacts should always lose to deny rules.

### Tests to add

Add tests in a dedicated path module test file:

```text
tests/path_scope_hardening.rs
```

Cases:

| Case                                           | Expected result                                            |
| ---------------------------------------------- | ---------------------------------------------------------- |
| `agent/foo.rs` under `agent/`                  | true                                                       |
| `agentic/foo.rs` under `agent/`                | false                                                      |
| `docs2/file.md` under `docs/`                  | false                                                      |
| `docs/file.md` under `docs`                    | true only after normalized directory handling              |
| `reference/foo` under allowed owner prefix     | forbidden wins                                             |
| generated zone path under allowed owner prefix | forbidden unless regeneration profile with source contract |
| empty changed path                             | no write authority                                         |

### Acceptance criteria

* All owner, test, generated, allowed, and forbidden matching uses typed path helpers.
* No call site uses symmetric matching for edit permission.
* Tests prove forbidden paths override allowed paths.

## Workstream C — Harden context-pack generation

### Goal

Make `jankurai context-pack` deterministic, bounded, owner-aware, proof-routed, and injection-safe.

The current `ContextPack` already carries many of the right fields, including owner, permission profile, changed paths, allowed paths, forbidden paths, generated zones, read-first files, relevant docs, proof lanes, commands, likely rules, max context files, stop conditions, residual risk, and token budget. ([GitHub][5])

Now those fields need to become stricter.

---

## C1. Replace heuristic owner resolution with explicit owner confidence

### Current issue

Owner inference currently uses changed paths, allowed paths, and keyword heuristics such as `agent`, `context`, `repair`, `adapter`, `ux`, `security`, `paper`, and `docs`. ([GitHub][5])

### Required design

Add structured owner resolution:

```json
{
  "owner": "agent",
  "owner_resolution": {
    "source": "owner-map",
    "confidence": "high",
    "matched_prefixes": ["crates/jankurai/src/commands/context_pack.rs"],
    "ambiguous_owners": [],
    "requires_split": false
  }
}
```

Possible `source` values:

```text
owner-map
changed-path
finding-path
task-keyword
manual
unknown
```

Possible `confidence` values:

```text
high
medium
low
unknown
```

### Rule

Only `high` or explicit `manual` ownership may grant write authority.

If owner confidence is `medium`, `low`, `unknown`, or mixed:

* generate a read-only or planner-only pack;
* recommend split packs;
* do not give broad allowed paths.

### Acceptance criteria

* Mixed-owner tasks produce either multiple packs or a single planner-only pack.
* Task-keyword ownership never grants write authority by itself.
* The context pack tells the agent why the owner was chosen.

## C2. Enforce real token/file/byte budgets

### Current issue

`token_budget` exists, but the implementation appears to use a fixed value rather than computing actual budget use. ([GitHub][5])

### Required schema fields

Extend `context-pack.schema.json`:

```json
{
  "budget": {
    "max_files": 12,
    "max_bytes": 120000,
    "max_tokens": 30000,
    "reserved_response_tokens": 4000,
    "actual_files": 9,
    "actual_bytes": 84231,
    "estimated_tokens": 21058,
    "truncated": false,
    "omitted": [
      {
        "path": "docs/large-file.md",
        "reason": "budget-exceeded",
        "bytes": 420000
      }
    ]
  }
}
```

Keep `token_budget` only as a compatibility alias if needed, but make `budget` the real contract.

### Required behavior

`context-pack` must:

1. sort candidate context files deterministically;
2. always include canonical policy files first;
3. exclude generated/build/reference/forbidden files unless explicitly read-only evidence;
4. calculate actual byte count;
5. estimate tokens deterministically;
6. truncate or omit files with recorded reasons;
7. fail or downgrade to planner-only if mandatory context cannot fit.

### Priority order

1. `AGENTS.md`
2. `agent/JANKURAI_STANDARD.md`
3. `agent/MASTER_PLAN.md`
4. selected phase file
5. relevant source file
6. owner map slice
7. test map slice
8. proof lane slice
9. relevant docs
10. audit finding or report slice
11. optional surrounding files
12. untrusted evidence, clearly labeled

The standard already emphasizes token minimization, tool-specific instructions as short pointers, owner/test maps, and excluding build/generated artifacts. ([GitHub][8])

### Acceptance criteria

* A context pack cannot exceed its own budget.
* Omitted files are recorded with reasons.
* Generated or forbidden files are never silently included as trusted instructions.
* The output is stable across repeated runs.

## C3. Make permission profile selection deterministic

### Current issue

`context-pack` currently infers profiles with broad rules. For example, security/ops maps to `security-investigation`, docs-like paths map to `docs-only`, generated or `.toml` paths can map to `generated-regeneration`, and empty changes map to `read-only`. ([GitHub][5])

That is too coarse. A `.toml` change is not automatically a generated-regeneration task.

### Required design

Create a proper `PermissionProfile` enum and resolver:

```rust
enum PermissionProfileKind {
    ReadOnly,
    DocsOnly,
    CodeEdit,
    GeneratedRegeneration,
    SecurityInvestigation,
    Release,
    PlannerOnly,
}
```

Resolution inputs:

```text
task intent
changed paths
owner confidence
finding severity
rule id
generated-zone status
proof lane
source hierarchy
human approval requirement
```

### Required profile output

```json
{
  "permission_profile": {
    "name": "code-edit",
    "source": "resolver",
    "confidence": "high",
    "allowed_path_globs": ["crates/jankurai/src/commands/context_pack.rs"],
    "denied_path_globs": ["reference/**", "target/**", "agent/generated/**"],
    "allowed_commands": ["cargo test -p jankurai phase_08_09_smoke"],
    "denied_commands": ["rm -rf *", "git push", "curl *"],
    "network": "deny",
    "secrets": "deny",
    "git_write": "deny",
    "max_files_changed": 3,
    "max_diff_lines": 300,
    "human_approval_required": false
  }
}
```

### Acceptance criteria

* Profile is an object, not only a string.
* Existing string field may remain for backward compatibility, but the policy object is authoritative.
* Profile resolution explains itself.
* Unknown or mixed authority becomes `planner-only` or `read-only`.

## C4. Route proof commands only through Phase 03 contracts

### Current issue

The repo catalog can derive commands from test maps and proof lanes, and it has an allowed proof command list. ([GitHub][4])

But Phase 08 should not independently guess proof. It should call or share the same proof-router logic hardened in Phase 03.

### Required design

Context packs should contain:

```json
{
  "proof": {
    "router_version": "phase-03",
    "selected_lane": "fast",
    "selected_commands": ["just fast"],
    "why_this_lane": "changed path maps to Rust CLI and schema contract tests",
    "minimum_credible": true,
    "fallback_used": false,
    "unmapped_paths": []
  }
}
```

### Hard rule

If a path cannot be mapped to a credible proof lane:

* do not silently fall back to `just fast`;
* mark proof as unmapped;
* add stop condition;
* require human review or proof-router update.

### Acceptance criteria

* Every command in a context pack is in the proof allowlist.
* No arbitrary shell command is emitted from task text.
* Unmapped proof is visible and blocks safe-auto repair.

# 5. Harden repair plans and repair packets

## D1. Split repair planning from repair execution

### Goal

Make it impossible for an agent to confuse a plan with permission to edit.

### Required `plan_mode` values

```text
planner-only
safe-auto-candidate
human-required
apply-approved
```

### Behavior

* Default: `planner-only`.
* If any packet is high-risk, security-sensitive, prompt-injection-related, reference-path-related, generated-without-source, release-related, destructive-migration-related, or unknown-rule: `human-required`.
* Only low-risk, known, local, deterministic repairs may become `safe-auto-candidate`.
* `apply-approved` requires an explicit approval receipt.

The current repair logic already marks human review for critical severity, human-required/never-auto eligibility, high/critical risk, high-risk rule IDs, reference paths, or security profiles. ([GitHub][6])

### Acceptance criteria

* `repair-plan` cannot imply auto-apply unless `plan_mode` allows it.
* Human-required packets block automatic apply.
* The reason for every mode is recorded.

---

## D2. Make owner-aware repair scoping exact

### Current issue

Repair allowed paths are derived from owner prefixes, parent path, and finding path. ([GitHub][6])

That can be too broad if an owner owns many prefixes.

### Required design

Each packet should have:

```json
{
  "edit_scope": {
    "primary_path": "crates/jankurai/src/commands/context_pack.rs",
    "allowed_write_paths": [
      "crates/jankurai/src/commands/context_pack.rs"
    ],
    "allowed_read_paths": [
      "agent/owner-map.json",
      "agent/test-map.json",
      "agent/proof-lanes.toml",
      "docs/agent-native-standard.md"
    ],
    "owner_prefixes_considered": [
      "crates/jankurai/src/commands/"
    ],
    "excluded_owner_prefixes": [
      {
        "prefix": "docs/",
        "reason": "same owner but not required for this finding"
      }
    ]
  }
}
```

### Rule

Same-owner does not automatically mean same-repair-scope.

### Acceptance criteria

* A repair packet for one file does not grant write access to all files owned by that owner.
* Broad owner prefixes become read context unless explicitly needed.
* Mixed-owner repair plans split into separate repair groups.

---

## D3. Add safe deterministic patch primitives

### Current issue

Planned edits currently appear to carry `apply_strategy: "none"` and empty patch fields. ([GitHub][6])

### Required design

Allow only a small set of apply strategies:

```text
none
replace_exact
append_exact
create_file
delete_exact_generated_only
regenerate_from_source
```

Each executable strategy must carry preconditions:

```json
{
  "apply_strategy": "replace_exact",
  "path": "agent/MASTER_PLAN.md",
  "match": "old exact text",
  "replacement": "new exact text",
  "preconditions": {
    "file_sha256": "abc123...",
    "match_count": 1,
    "path_must_exist": true,
    "path_must_be_allowed": true
  }
}
```

### Hard restrictions

Never auto-apply:

* security findings;
* secrets;
* prompt injection;
* overbroad agency;
* destructive migrations;
* reference-data mutation;
* generated output without source contract;
* release config;
* dependency updates;
* unknown rules;
* critical/high severity;
* multi-file architectural changes.

### Acceptance criteria

* A repair apply command refuses `apply_strategy: none`.
* A repair apply command refuses if `match_count != 1`.
* A repair apply command refuses if file hash changed.
* A repair apply command refuses if path is outside allowed write scope.
* A repair apply command emits a receipt.

### Acceptance criteria

* Every stable rule ID has repair metadata.
* Unknown rule IDs default to `human-required`.
* Rule metadata is validated by schema tests.
* Repair packets inherit risk and proof requirements from the rule table.

# 6. Harden permission profiles into an enforcement layer

## E1. Create a policy engine

### Files to add or refactor

* `crates/jankurai/src/permissions.rs`
* `crates/jankurai/src/commands/repair_apply.rs`
* `crates/jankurai/src/commands/context_pack.rs`
* `crates/jankurai/src/commands/repair_plan.rs`
* `crates/jankurai/src/commands/agent.rs`
* `schemas/permission-profile.schema.json`

### Required API

```rust
struct PermissionDecision {
    allowed: bool,
    reason: String,
    profile: PermissionProfile,
    violations: Vec<PermissionViolation>,
}

fn check_read(profile: &PermissionProfile, path: &RepoPath) -> PermissionDecision;
fn check_write(profile: &PermissionProfile, path: &RepoPath) -> PermissionDecision;
fn check_command(profile: &PermissionProfile, command: &ProofCommand) -> PermissionDecision;
fn check_network(profile: &PermissionProfile) -> PermissionDecision;
fn check_generated(profile: &PermissionProfile, path: &RepoPath) -> PermissionDecision;
```

### Required enforcement points

| Command/surface | Enforcement                                                   |
| --------------- | ------------------------------------------------------------- |
| `context-pack`  | Must not output write authority outside profile.              |
| `repair-plan`   | Must not produce safe-auto plan if profile forbids it.        |
| `repair apply`  | Must reject write/command violations.                         |
| `agent verify`  | Must check adapter instructions do not exceed allowed agency. |
| `doctor`        | Must validate saved context packs and repair plans.           |
| MCP tools       | Must call the same policy engine.                             |

### Acceptance criteria

* Permission profile violations fail closed.
* Permission checks are tested independently of CLI.
* The same permission engine is reused everywhere.

## E2. Add command allowlist enforcement

The repo catalog already has logic for normalized proof commands and allowed proof commands from proof lanes and test maps. ([GitHub][4])

Phase 08 should use that list everywhere an agent-facing command is emitted or executed.

### Rules

* `planned_commands` must be normalized.
* Commands must come from `agent/proof-lanes.toml`, `agent/test-map.json`, or Phase 03 proof-router output.
* Any command not in the allowlist requires human approval.
* Shell metacharacters should be rejected unless the command is explicitly declared in a lane.

### Acceptance criteria

* No command generated from free-form task text.
* No repair packet emits arbitrary shell.
* Tests cover command injection attempts.

# 7. Harden prompt-injection defense

## F1. Make source hierarchy machine-readable

### Add file

```toml

[[source]]

trust_level = "tool-generated"
instruction_authority = "none"

[[source]]
glob = "target/**"
trust_level = "generated-output"
instruction_authority = "none"

[[source]]
glob = "reference/**"
trust_level = "reference"
instruction_authority = "none"

[[source]]
glob = "**/*.log"
trust_level = "untrusted-evidence"
instruction_authority = "none"
```

### Required behavior

Context packs and repair plans must label every context item with source hierarchy metadata.

### Acceptance criteria

* Source hierarchy is parsed by code.
* `docs/source-hierarchy.md` explains the policy, but TOML/JSON is authoritative for enforcement.
* Untrusted sources cannot grant permissions or proof commands.

# 8. Harden adapter canonicalization

## G1. Make every adapter generated from one canonical template

The current phase already names adapter verification and sync as existing surfaces. ([GitHub][1])

agent/adapter-template.md

or:

agent/adapters.toml

Generated from agent/adapter-template.md by `jankurai adapters sync`.
Do not edit this file directly.

* `agent/JANKURAI_STANDARD.md`

* Adapter files do not contain broad authority such as “edit anything necessary.”
* Adapter files do not contain conflicting instructions.
* Adapter files stay under a configured token/line budget.

# 15. Final Definition of Done

Phase 08 is 100% hardened when all of this is true:

| Area             | Done condition                                                                                                     |
| ---------------- | ------------------------------------------------------------------------------------------------------------------ |
| Contract         | Phase 08-H invariants are documented and mapped to code/tests.                                                     |
| Schemas          | Context pack, repair packet, repair plan, and permission profile schemas validate all emitted artifacts.           |
| Context packs    | Bounded by real budget, deterministic, owner-aware, source-labeled, and proof-routed.                              |
| Path scoping     | Segment-aware, deny-overrides-allow, no symmetric overmatch.                                                       |
| Permissions      | Enforced by code, not advisory strings.                                                                            |
| Proof routing    | Commands only come from proof router, proof lanes, or test map allowlist.                                          |
| Repair plans     | Planner-only by default, exact edit scopes, human gates, rollback, safe patch primitives only where eligible.      |
| Prompt injection | Source hierarchy is machine-readable; untrusted content never becomes policy.                                      |
| Adapters         | Generated/synced from canonical source; verify fails on drift or broad agency.                                     |
| MCP              | Read-only MCP exists or is explicitly removed from Phase 08 scope; no hidden write bypass.                         |
| Doctor           | `doctor phase-08` validates context packs, repair plans, permissions, proof commands, source labels, and adapters. |
| Tests            | Unit, golden, schema, smoke, property, and CLI tests cover the unsafe cases.                                       |
| Evidence         | Final receipt records commands run, artifacts produced, and proof result.                                          |

---

# 16. The most important hardening rule

Do **not** let Phase 08 become “agent convenience.” It must remain “agent containment.”

The current implementation already gives agents helpful context and repair packets. The hardening work is to ensure those packets are **bounded, enforceable, proof-routed, injection-resistant, and auditable**. Once the path matcher, permission engine, context budget, source hierarchy, repair modes, and doctor checks are in place, Phase 08 becomes the control plane the moonshot loop needs.
