I wasn’t able to `git clone` in the sandbox because GitHub DNS resolution failed, so I inspected the repository through GitHub source views. My read: **Phase 08 is “complete” as v0 scaffolding, but it is not yet “100% hardened.”** The phase doc itself says Phase 08’s objective is bounded context, bounded edit authority, exact proof requirements, and repair packets that preserve human control, while its own gap list calls out advisory-only permission enforcement, sharper prompt-injection/source hierarchy needs, richer repair scoping, and no MCP/server surface yet. ([GitHub][1])

## Phase 08 target state

Treat Phase 08 as finished only when a fresh coding agent can be handed a Jankurai-generated packet and safely know:

1. **What it may read.**
2. **What it may edit.**
3. **What it must never touch.**
4. **Which proof lane is the smallest credible lane.**
5. **Which commands are allowed.**
6. **Which commands or file changes require human approval.**
7. **Which content is trusted policy versus untrusted user, issue, log, webpage, prompt, or fixture text.**
8. **How to stop before widening authority.**
9. **How to produce durable evidence.**
10. **How another agent can resume the repair without re-deriving the whole repo.**

Right now, the repo has the command surfaces and schemas: `jankurai context-pack`, `jankurai repair-plan`, `jankurai agent verify`, adapter verification, context-pack schema, repair-plan schema, repair-packet schema, and permission-profile schema. The context pack command builds a heuristic pack from owner/test/generated/proof data and validates JSON output; the repair-plan command turns findings into packets and validates JSON output. ([GitHub][1])

The hardening plan below moves Phase 08 from **“complete scaffolding”** to **“hardened control plane.”**

# Definition of done for Phase 08

Phase 08 is **100% done** when these gates pass:

| Gate                     | Done state                                                                                                                                                                               |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Context pack contract    | Every pack is deterministic, path-scoped, owner-scoped, proof-routed, token-budgeted, source-hierarchy aware, schema-validated, and reproducible.                                        |
| Repair packet contract   | Every packet has owner, lane, permission profile, allowed paths, forbidden paths, required proof, stop conditions, rollback, risk, repair eligibility, and human-review decision.        |
| Permission profiles      | Profiles are not just schema text. They are manifest-backed, rule/lane mapped, and enforced against generated commands and packet authority.                                             |
| Prompt-injection defense | Trusted and untrusted sources are classified. Untrusted content can never alter policy, tool authority, proof requirements, or stop conditions.                                          |
| Agent verify             | `jankurai agent verify --strict` validates adapters, permission profiles, source hierarchy, context-pack fixtures, repair-plan fixtures, proof command allowlists, and schema contracts. |
| MCP surface              | At minimum, a read-only local MCP/stdout-compatible surface or explicitly documented CLI-backed server contract exists, with CLI remaining source of truth.                              |
| Tests                    | Golden, fixture, schema, property-style, and negative tests cover the failure modes.                                                                                                     |
| Evidence                 | Phase log and receipt show commands run, artifacts generated, skipped validation, residual risk, and no unapproved exceptions.                                                           |

I would **not** update the router ledger to “hardened” until all gates above are green. The repo’s phase taxonomy distinguishes “complete” from “hardened”: complete means the required v0 surface and receipt exist, while hardened means the production-quality slice is validated. ([GitHub][2])

# Current Phase 08 state

## What is already good

Phase 08 already has a strong skeleton:

* The phase doc says adapters, `jankurai adapters verify`, `adapters sync`, `jankurai agent verify`, `jankurai context-pack`, and `jankurai repair-plan` exist, and that JSON outputs for context-pack and repair-plan are schema-validated. ([GitHub][1])
* Shared schemas exist for context packs, repair packets, repair plans, and permission profiles. ([GitHub][3])
* Context packs currently include task, owner, permission profile, changed paths, allowed paths, forbidden paths, generated zones, read-first files, relevant docs, proof lanes, commands, likely rules, max context files, stop conditions, residual risk, and token budget. ([GitHub][4])
* Repair plans currently include packets, planned edits, planned commands, proof lanes, rollback guidance, and human approval requirements. ([GitHub][5])
* Adapter verification is already useful: it checks that adapter files point to canonical docs, phase index/logs, and planner protocol, rejects contradictory “ignore AGENTS.md”-style language, and enforces a short-adapter rule. ([GitHub][6])
* The catalog layer already loads `owner-map.json`, `test-map.json`, `generated-zones.toml`, and `proof-lanes.toml`, and it can derive allowed proof commands from proof lanes plus test-map commands. ([GitHub][7])

## What is not hardened yet

The main weakness is that Phase 08 currently **generates useful guidance**, but it does not yet fully enforce the agent contract.

The context pack builder still relies heavily on task-string heuristics and a static `token_budget: 100000`; it emits path lists and commands, but not a detailed included-context ledger with token estimates, source trust, file digests, line ranges, exclusion reasons, or proof receipt expectations. ([GitHub][4])

The repair planner is still mostly a dry-run expansion layer. It builds packets and planned edits, but `planned_edit_from_packet` currently emits `apply_strategy: "none"` and no `match_text`, `replacement_text`, `append_text`, or `create_text`, even when the schema supports those fields. ([GitHub][5])

`jankurai agent verify` currently delegates only to adapter verification. That means it does not yet validate the full Phase 08 control plane: permission manifests, context-pack samples, repair-plan samples, source hierarchy fixtures, proof-command allowlists, or conventional artifacts under `target/jankurai`. ([GitHub][8])

Permission profiles exist as a schema with names like `read-only`, `docs-only`, `code-edit`, `generated-regeneration`, `security-investigation`, and `release`, but the phase doc still identifies lane enforcement as advisory-only. ([GitHub][9])

## 4. Make context pack generation deterministic and budgeted

Current context pack generation infers roots from task keywords, derives owner, profile, docs, proof lanes, commands, likely rules, stop conditions, and emits a static `max_context_files: 12` and `token_budget: 100000`. ([GitHub][4])

Replace that with a deterministic planner.

### Inputs

Support these CLI flags:

```bash
jankurai context-pack <repo> \
  --task "..." \
  --changed pathA \
  --changed pathB \
  --from-report agent/repo-score.json \
  --from-finding sha256:... \
  --owner agent \
  --profile code-edit \
  --max-tokens 60000 \
  --max-files 12 \
  --strict \
  --explain-pruning \
  --out target/jankurai/context-pack.json \
  --md target/jankurai/context-pack.md
```

### Ranking algorithm

Build a candidate list from:

1. Current task.
2. `--changed` paths.
3. Matching audit finding, if provided.
4. Owner map.
5. Test map.
6. Generated zones.
7. Proof lanes.
8. Phase docs.
9. Standard docs.
10. Schemas touched by the task.

Rank context candidates in this order:

| Rank | Include reason                                                        |
| ---: | --------------------------------------------------------------------- |
|    1 | `AGENTS.md` and root adapter pointers                                 |
|    2 | `agent/JANKURAI_STANDARD.md`                                          |
|    3 | `agent/MASTER_PLAN.md` if phase/planning work                         |
|    4 | active phase file                                                     |
|    5 | active phase log                                                      |
|    6 | `agent/owner-map.json`                                                |
|    7 | `agent/test-map.json`                                                 |
|    8 | `agent/generated-zones.toml`                                          |
|    9 | `agent/proof-lanes.toml`                                              |
|   10 | directly changed source files                                         |
|   11 | directly related schemas                                              |
|   12 | directly related tests                                                |
|   13 | docs required by likely rules                                         |
|   14 | generated artifacts only as read-only metadata, never as edit targets |
|   15 | volatile reports only as input references, not pasted context         |

### Token budget

Implement an approximate tokenizer:

```rust
pub fn estimate_tokens(text: &str) -> usize {
    // Conservative English/code approximation.
    // Round up so packs are not under-budgeted.
    (text.len() + 3) / 4
}
```

Then enforce:

```rust
pub struct TokenBudget {
    pub max_input_tokens: usize,
    pub reserved_output_tokens: usize,
    pub max_context_files: usize,
    pub per_file_soft_limit: usize,
    pub estimated_input_tokens: usize,
}
```

Rules:

* Always include canonical policy pointers, but summarize if too large.
* Never include `target/`, vendored deps, generated build artifacts, large logs, or raw untrusted issue bodies unless explicitly marked as untrusted snippets.
* If a file would exceed budget, add it to `excluded_context` with reason.
* If a changed path is forbidden, fail in `--strict` mode.
* If an owner or test route is missing, emit stop condition and fail in `--strict` mode.

### Output invariant

Every emitted command must be traceable to:

* `agent/test-map.json`, or
* `agent/proof-lanes.toml`, or
* a documented fallback lane that is marked `requires_approval`.

The catalog already has the machinery to derive allowed proof commands from proof lanes and test-map commands, so reuse that instead of introducing another allowlist. ([GitHub][7])

Add:


# High-risk hard parts

## Hard part 1: “Bounded context” must be real

The context pack cannot just say “max context files: 12.” It needs to prove what it included, why it included it, what it excluded, why it excluded it, and how many tokens the pack costs.

Required invariant:

```text
Every included context file has path, role, trust, owner, include reason, token estimate, and digest.
```

## Hard part 2: Permissions must not be advisory

The current phase doc explicitly calls out advisory-only permission profiles as a gap. ([GitHub][1]) Hardened Phase 08 must make permission profile violations validation failures.

Required invariant:

```text
No context pack or repair packet may ask for a forbidden command, forbidden write path, or broader profile than the selected lane/rule requires.
```

## Hard part 3: Prompt-injection defense must distinguish source trust

Trusted policy with bypass language is a failure. Untrusted issue text with bypass language is evidence to isolate, not policy to obey.

Required invariant:

```text
Untrusted content can never define policy, permission profile, proof lane, stop condition, or tool authority.
```

## Hard part 4: Repair packets must be executable, but not autonomous

Phase 08 is not Phase 13. Do not implement broad auto-repair here. Emit exact safe payloads only where they are low-risk and mechanically bounded.

Required invariant:

```text
High-risk and human-required packets are review-only, even if an obvious edit exists.
```

## Hard part 5: MCP must not become a backdoor

The MCP surface should expose the same bounded objects as the CLI. It should not gain shell execution, mutation, or authority expansion.

Required invariant:

```text
MCP is transport. CLI policy is source of truth.
```

# The shortest path to “100% hardened”

The critical path is:

```text
1. Permission profiles manifest + enforcement
2. Context-pack v1.1 with context_files, source trust, token budget, proof plan
3. Repair-plan v1.1 with dedupe, packet groups, proof plan, JSONL, meaningful planned edits
4. Prompt-injection/source hierarchy scanner shared by audit/adapters/agent verify
5. agent verify --strict as whole-control-plane gate
6. Read-only MCP surface or explicit hardened MCP non-goal
7. Golden + negative fixture tests
8. Full validation receipt
```

That finishes Phase 08 in the spirit of the moonshot: **bounded agents, smallest credible proof, durable evidence, explicit stop conditions, and no hidden authority expansion.**
