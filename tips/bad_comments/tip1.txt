## Low-false-positive regex patterns

Use these case-insensitively and only against comment text.

### AI/scaffold residue

```regex
(?i)\b(as an ai language model|generated by (chatgpt|copilot|claude|ai)|ai-generated|for demonstration purposes|for demo purposes|in a real (app|application)|not production ready|do not use in production|omitted for brevity|left as an exercise|pseudo-?code)\b
```

### Placeholder implementation

```regex
(?i)\b(actual|real)?\s*(implementation|logic)\s+(goes here|would go here)\b|\b(add|insert)\s+(your\s+)?(logic|business logic|implementation)\s+here\b
```

### “Replace with real thing” scaffold

```regex
(?i)\b(replace|swap)\s+(this\s+)?with\s+(your\s+)?(actual|real|own)\s+(api|endpoint|database|implementation|logic|secret|token|password|key|credentials?)\b
```

### Dangerous TODO/FIXME

```regex
(?i)\b(todo|fixme|xxx)\b.{0,80}\b(auth|authentication|authorization|permission|access control|validation|sanitize|escape|encrypt|secret|token|password|api key|credential|csrf|xss|sql injection|tenant|ownership|rate limit|error handling|exception|rollback|transaction|audit|pii)\b
```

### Security bypass

```regex
(?i)\b(skip|bypass|disable|turn off)\b.{0,50}\b(auth|authentication|authorization|permission|validation|csrf|ssl|tls|certificate|cert|signature|jwt|rate limit|access control)\b
```

### Trusting unsafe input

```regex
(?i)\b(trust|assume)\b.{0,50}\b(client input|user input|request input|frontend|front-end|user is authenticated|user is authorized|user is admin|same tenant|owner)\b
```

### Hardcoded secrets

```regex
(?i)\b(hard-?coded|hardcoded)\b.{0,50}\b(secret|token|password|api key|apikey|credential|private key|access key|client secret)\b
```

### Error swallowing

```regex
(?i)\b(ignore|swallow|silently ignore|catch and ignore)\b.{0,50}\b(error|errors|exception|exceptions|failure|failures|rejection|promise rejection|panic)\b
```

### Bad fallback

```regex
(?i)\b(fallback|default)\b.{0,60}\b(admin|root|password|token|secret|credential|auth|authenticated|authorized|true|false|null|empty|dummy|fake|mock)\b
```

### TypeScript suppression smell

```regex
(?i)\b(shut typescript up|make typescript happy|ignore type error|temporary any|any for now|cast to any|force cast|todo remove ts-ignore|todo remove ts-expect-error|todo fix types)\b
```

### React dependency/hydration smell

```regex
(?i)\b(disable|ignore|suppress)\b.{0,50}\b(exhaustive deps|dependency array|hydration|hydration mismatch)\b|\b(todo|fixme)\b.{0,50}\b(dependency array|hydration|suspense|rerender|re-render)\b
```

### Rust unsafe/unwrap smell

```regex
(?i)\b(unwrap|expect|panic|unsafe|clone)\b.{0,50}\b(for now|temporar(y|ily)|trust me|appease|borrow checker|safe because|never fails)\b
```

### Docker/deploy smell

```regex
(?i)\b(use latest for now|latest is fine|run as root for now|todo pin (version|image)|todo healthcheck|disable healthcheck|expose everything|open all ports|self-signed for now)\b
```

## Practical scoring rule

For minimal false positives, I’d use a severity model:

**Block merge immediately** when a comment mentions:

```text
auth
authorization
permissions
validation
sanitize
encrypt
secret
token
password
certificate
ssl
tls
csrf
xss
sql injection
tenant
ownership
pii
```

combined with:

```text
todo
skip
bypass
disable
ignore
hardcoded
temporary
for now
replace
placeholder
fake
dummy
mock
```

That catches the genuinely dangerous cases without banning normal comments like:

```text
// Fallback for older browsers
// TODO: improve copy
// Mock service worker setup
// Default theme value
```

The best blacklist is not `TODO`; it is:

```text
TODO + security/correctness noun
FALLBACK + privileged/default value
IGNORE + error/security noun
DISABLE/SKIP/BYPASS + protection noun
HARDCODED + credential/sensitive noun
PLACEHOLDER/DEMO/AI phrase + implementation noun
```
