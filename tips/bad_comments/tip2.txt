Below is a high-signal brainstorm for **production source code comments**. I would not ban plain `TODO` or `FALLBACK` alone if you care about false positives. Instead, ban phrases where the comment itself admits something unsafe, fake, unfinished, or AI/tutorial-generated.

## Good regex-style patterns

These are better than banning single words like `TODO`, `FALLBACK`, `HACK`, or `MOCK`.

```regex
(?i)\b(do\s+not|don't|dont)\s+(ship|deploy|release)\b
```

```regex
(?i)\b(remove|delete)\s+(before|prior\s+to)\s+(prod|production|deploy|deployment|release)\b
```

```regex
(?i)\b(not\s+for\s+(prod|production)|do\s+not\s+use\s+in\s+(prod|production))\b
```

```regex
(?i)\b(fake|mock|dummy|stub|placeholder)\s+(implementation|response|data|client|service|api|backend)\b
```

```regex
(?i)\b(replace\s+with\s+(real|actual|production)|implement\s+(real|actual|production)\s+logic)\b
```

```regex
(?i)\b(bypass|skip|disable|turn\s+off)\s+(auth|authentication|authorization|permission|permissions)\b
```

```regex
(?i)\b(ignore|skip|disable|turn\s+off)\s+(ssl|tls|certificate|cert)\s+(verification|validation|check|checks|errors?)\b
```

```regex
(?i)\b(trust|accept)\s+all\s+(certs|certificates)\b
```

```regex
(?i)\b(hardcoded|hard-coded)\s+(password|secret|token|api[-_\s]?key|credential|credentials)\b
```

```regex
(?i)\b(skip|disable|no)\s+(input\s+)?(validation|sanitization|escaping)\b
```

```regex
(?i)\b(validate|sanitize|escape|encrypt|redact)\s+(later|for\s+now)\b
```

```regex
(?i)\b(ignore|swallow|silently\s+ignore|silently\s+swallow)\s+(errors?|exceptions?|failures?)\b
```

```regex
(?i)\b(todo|fixme|xxx)\b.*\b(auth|security|secret|password|token|validation|sanitize|sanitization|encrypt|encryption|redact|error|exception|race|data\s+loss|prod|production)\b
```

```regex
(?i)\b(generated\s+by\s+(chatgpt|claude|copilot|ai|llm)|ai[-\s]?generated|llm[-\s]?generated)\b
```

```regex
(?i)\b(as\s+an\s+ai|as\s+an\s+ai\s+language\s+model|i\s+(do\s+not|don't)\s+have\s+access)\b
```

```regex
(?i)\b(in\s+a\s+(real|production)\s+(app|application|project)|in\s+production,?\s+you\s+would)\b
```

```regex
(?i)\b(for\s+demonstration\s+purposes|simplified\s+for\s+demo|this\s+is\s+a\s+simplified\s+example)\b
```

```regex
(?i)\b(i\s+(do\s+not|don't)\s+know\s+why|not\s+sure\s+why|don't\s+ask\s+why|hope\s+this\s+works|works\s+on\s+my\s+machine)\b
```

```regex
(?i)\b(good\s+enough\s+for\s+now|fine\s+for\s+now|fix\s+later|clean\s*up\s+later|remove\s+later|hack\s+for\s+now)\b
```

```regex
(?i)\b(@ts-nocheck|@ts-ignore|any\s+for\s+now|cast\s+to\s+any|force\s+cast|type\s+hack|fix\s+types\s+later)\b
```


## Words I would not ban alone

These are useful smells, but they create false positives unless combined with dangerous context.

```txt
TODO
FIXME
XXX
HACK
TEMP
TEMPORARY
WORKAROUND
FALLBACK
LEGACY

STUB

PLACEHOLDER

```

For example, `fallback` can be totally valid: fallback fonts, fallback routes, fallback image loaders, fallback UI, fallback locales. But these are high-signal:

## The most upsetting comments overall

If I had to make a “never ignore this” top tier, it would be:

```txt
DO NOT SHIP
REMOVE BEFORE PRODUCTION
DO NOT USE IN PRODUCTION
BYPASS AUTH
AUTH DISABLED
TRUST ALL CERTS
IGNORE CERTIFICATE ERRORS
HARDCODED PASSWORD
HARDCODED SECRET
SKIP VALIDATION
SANITIZE LATER
IGNORE ERRORS
SWALLOW EXCEPTIONS
PLACEHOLDER IMPLEMENTATION
REPLACE WITH REAL IMPLEMENTATION
MOCK API
FAKE AUTH
GENERATED BY CHATGPT
IN A REAL APP
FOR DEMONSTRATION PURPOSES ONLY
I DON'T KNOW WHY THIS WORKS
HOPE THIS WORKS
GOOD ENOUGH FOR NOW
TODO SECURITY
TODO AUTH
TODO ENCRYPT
TODO VALIDATE
TODO HANDLE ERRORS
UNSOUND
UNDEFINED BEHAVIOR
DATA RACE
CHMOD 777
RUN AS ROOT FOR NOW
```

The safest scanning philosophy is: **don’t block on vague words; block on confessions.** A phrase like `TODO` says “there is work.” A phrase like `TODO SECURITY`, `REMOVE BEFORE PRODUCTION`, or `BYPASS AUTH` says “this code is knowingly unsafe or incomplete.”
