Yes — there are genuinely smart tools for this, but **the best answer is not one magic scanner**. The strongest setup is a layered coverage-audit system:

1. **Coverage maps** show what code ran.
2. **Mutation testing** shows whether tests would actually fail if code were wrong.
3. **API schema/traffic coverage** shows which endpoints, parameters, status codes, and workflows are untested.
4. **User-flow/E2E tools** show which real product paths are missing.
5. **DB/container/static scanners** catch untested migrations, infra assumptions, security paths, and config gaps.

A key warning: **100% line coverage is not the same as “fully tested.”** A large ICSE study of 31,000 test suites found that coverage has only low-to-moderate correlation with test-suite effectiveness once suite size is controlled, so coverage should be used as a gap-finding signal, not the final quality target. 

## The best overall stack I’d use

### 1. Coverage dashboard and CI enforcement: **Codecov**

Use **Codecov** as the central coverage control plane. The most valuable feature for your stack is **coverage flags**: split coverage by Rust unit tests, TypeScript/Vitest tests, Playwright E2E, Storybook/component tests, API tests, and migration tests. That lets you see, for example, that backend code is well covered while API contract paths or UI flows are not. Codecov also supports PR status checks and patch coverage, so CI can fail when new code is not covered. ([Codecov][1])

**Use it for:** coverage aggregation, PR gates, changed-line coverage, monorepo visibility, and “which test suite covers this code?” analysis.

**Recommended flags:** `rust-unit`, `rust-integration`, `ts-unit`, `react-component`, `playwright-e2e`, `api-contract`, `api-fuzz`, `db-migration`.

---

### 2. Rust coverage and test quality: **cargo-llvm-cov + cargo-mutants + proptest + cargo-fuzz**

For Rust, make **cargo-llvm-cov** the primary coverage tool. It uses LLVM source-based coverage and supports Rust test workflows including unit tests, doctests, integration tests, nextest, and other cargo commands; it can also generate LCOV output for Codecov. ([GitHub][2])

Pair that with **cargo-mutants**. Coverage tells you that a line ran; mutation testing tells you whether your assertions would catch a bug there. cargo-mutants works by inserting plausible code changes and checking whether the test suite fails, which is exactly the “do we really test this behavior?” question. ([GitHub][3])

Add **proptest** for property-based testing and **cargo-fuzz** for fuzzing. proptest generates randomized inputs and shrinks failures to small cases; cargo-fuzz integrates Rust with libFuzzer and uses coverage-guided fuzzing to explore input spaces. ([GitHub][4])

For unsafe Rust or critical invariants, add **Kani** or **Miri**. Kani is useful for model checking panics, overflows, assertions, contracts, and some classes of undefined behavior; Miri is useful for detecting undefined behavior in Rust code. ([model-checking.github.io][5])

**Best Rust stack:**

* **cargo-llvm-cov** — primary line/region/branch coverage.
* **cargo-nextest** — faster, more scalable test runner.
* **cargo-mutants** — mutation score and weak-test detection.
* **proptest** — property-based tests for domain logic.
* **cargo-fuzz** — fuzz parsers, APIs, serialization, auth/input paths.
* **Kani/Miri** — critical correctness and unsafe-code checks.

I would treat **Tarpaulin** as secondary for this stack. It is still useful, but the current Rust community trend and tooling ergonomics strongly favor cargo-llvm-cov for accurate LLVM-based coverage; Reddit Rust users also report cargo-llvm-cov as easier and more accurate in practice, though that is anecdotal community signal rather than proof. ([GitHub][6])

---

### 3. Vite/TypeScript coverage and mutation testing: **Vitest + StrykerJS**

For Vite/TypeScript, use **Vitest coverage** as the default coverage engine. Vitest supports V8 and Istanbul providers, and its docs note that since Vitest 3.2.0, V8 coverage remapping is designed to be as accurate as Istanbul while being faster and lower-memory in many cases. Vitest also supports strict thresholds, including a `100` shortcut, per-file thresholds, and include/exclude configuration. ([Vitest][7])

Important: configure `coverage.include`. Otherwise, many JS/TS coverage tools only count files imported during tests, which can hide completely untested files. Vitest explicitly documents include/exclude behavior for controlling which files appear in coverage. ([Vitest][8])

Then add **StrykerJS** for TypeScript mutation testing. StrykerJS supports TypeScript and major JS frameworks, and it can use TypeScript type checking to avoid invalid mutants. It also has coverage-aware/per-test optimization so it can run only the tests that cover a given mutant, which matters in larger repos. ([Stryker Mutator][9])

**Best TS/Vite stack:**

* **Vitest coverage** — unit/component coverage.
* **StrykerJS** — mutation testing for weak assertions.
* **TypeScript strict mode + ESLint** — not coverage, but catches untested type holes.
* **Codecov flags** — separate frontend unit coverage from E2E coverage.

---

### 4. API coverage, contract coverage, and fuzzing: **Schemathesis + TraceCov + RESTler**

For APIs, the best “smart” coverage tools are not normal code coverage tools. You want tools that read your **OpenAPI/GraphQL schema**, generate requests, exercise edge cases, and measure which operations, parameters, schemas, examples, responses, and workflows were actually tested.

My top pick is **Schemathesis**. It performs property-based testing for OpenAPI and GraphQL, generates test cases from the API schema, adapts to observed responses, and can chain operations into stateful workflows. It is designed to catch problems like 500s, schema mismatches, validation bypasses, and undocumented behavior. ([GitHub][10])

Pair Schemathesis with **TraceCov**, which measures API coverage beyond “endpoint hit/not hit.” It tracks operations, parameters, JSON Schema keywords, examples, responses, and uncovered parts of the API contract. That is extremely close to what you asked for: “catch a lack of coverage in API.” ([Schemathesis][11])

Add **RESTler** for deeper stateful REST API fuzzing. Microsoft’s RESTler analyzes an OpenAPI spec, infers producer-consumer dependencies between requests, and generates sequences that reach deeper service states. It is especially useful for APIs where bugs require “create → update → attach → delete → retry” sequences. ([GitHub][12])

For measurement/research-grade API coverage, look at **Restats**. It is a newer tool that derives black-box REST API test coverage metrics from OpenAPI specifications plus observed HTTP requests/responses, which is useful when you want a formal API coverage audit rather than only test execution. ([arXiv][13])

**Best API stack:**

* **Schemathesis** — schema-driven API fuzzing/property tests.
* **TraceCov** — API coverage measurement from schema/test traces.
* **RESTler** — stateful API fuzzing.
* **Spectral** — OpenAPI linting and style/consistency checks.
* **42Crunch** — API security audit against OpenAPI.
* **Pact, Specmatic, or Microcks** — contract testing when frontend/backend or services must stay in sync. ([Stoplight][14])

A Reddit API-testing thread had a useful caveat: tools like Schemathesis, Pact, Dredd, EvoMaster, and RESTler help, but meaningful tests still require knowing the intended behavior. I agree with that: generated API tests are excellent at finding untested paths and protocol violations, but business correctness still needs human-defined assertions. ([Reddit][15])

---

### 5. User-flow coverage: **Playwright + Meticulous + Storybook/Chromatic**

For actual user flows, use **Playwright** as the core E2E runner. It supports Chromium, Firefox, and WebKit, has auto-waiting, assertions, tracing, parallelism, and a code generator that can record interactions and create locators. Its trace viewer is especially good for debugging failed flows. ([Playwright][16])

Playwright can also gather JavaScript/CSS coverage from the browser, although the coverage API is Chromium-oriented. That helps identify frontend code that no user flow reaches, but it should not be your only UI coverage metric. ([Playwright][17])

For “smart” user-flow discovery, **Meticulous** is one of the most interesting tools. It records real user interactions, tracks covered code branches, and curates visual E2E tests intended to cover lines, edge cases, and user flows. Treat it as a strong accelerator for discovering missed flows, then keep the most important flows as explicit Playwright tests in your repo. ([Meticulous AI][18])

For component states, add **Storybook’s Vitest addon/test tooling** plus **Chromatic**. Storybook can turn stories into executable tests and supports interaction, accessibility, and visual testing; Chromatic is useful for catching visual regressions across browsers, viewports, and themes. ([Storybook][19])

Add **axe-core with Playwright** for accessibility checks. The important caveat is that automated accessibility testing cannot catch every WCAG issue, but it is still excellent for catching common violations quickly. ([Playwright][20])

**Best user-flow stack:**

* **Playwright** — canonical E2E and flow tests.
* **Meticulous** — smart discovery from real sessions and visual E2E generation.
* **Storybook + Vitest addon** — component-state coverage.
* **Chromatic** — visual regression coverage.
* **axe-core** — automated accessibility coverage.
* **Checkly** — run Playwright flows as production/API monitoring if you want pre-prod and prod coverage continuity. ([Checkly][21])

---

### 6. Postgres and database coverage: **Testcontainers + pgTAP + Atlas**

For Postgres, the main mistake is testing against mocks. Use **Testcontainers** to run real Postgres in integration tests. Testcontainers is designed for throwaway Docker-based dependencies such as databases, message brokers, browsers, and anything else that can run in a container; it supports both Node and Rust ecosystems. ([Testcontainers][22])

Use **pgTAP** when you have database behavior that deserves direct tests: functions, triggers, views, constraints, RLS policies, generated columns, stored procedures, and migration invariants. pgTAP is a PostgreSQL unit testing framework with TAP-style assertions. ([pgTAP][23])

Use **Atlas** for schema and migration testing/linting. Atlas can test database schemas and migrations, and its migration linting is aimed at catching destructive changes, backward incompatibilities, table locks, and constraint problems before they hit production. ([AtlasGo][24])

If you use Sqitch, its `verify` flow can also validate that planned/deployed database changes satisfy verification scripts. ([Sqitch][25])

**Best Postgres stack:**

* **Testcontainers** — real Postgres integration tests.
* **pgTAP** — database-native unit tests.
* **Atlas** — migration linting and schema tests.
* **Sqitch verify** — useful if Sqitch is already your migration system.
* **Schemathesis/RESTler + Testcontainers** — API tests against real DB state.

---

### 7. Docker, container, and infrastructure coverage: **Trivy + Hadolint + Grype/Docker Scout**

For Docker and infra, use **Trivy** as the primary scanner. It scans container images, filesystems, Git repositories, Kubernetes configs, IaC, SBOMs, vulnerabilities, secrets, misconfigurations, and licenses. It also supports config scanning for Dockerfiles, Kubernetes, Terraform, CloudFormation, and related IaC. ([GitHub][26])

Add **Hadolint** for Dockerfile linting. It parses Dockerfiles into an AST and uses ShellCheck for shell snippets inside `RUN`, which catches a different class of Dockerfile quality issues than vulnerability scanners. ([GitHub][27])

Add **Grype** or **Docker Scout** if you want a second opinion on container/image vulnerabilities and SBOM-based analysis. Grype scans images, filesystems, and SBOMs and supports OS and language ecosystems including JavaScript and Rust; Docker Scout uses SBOMs and vulnerability data to analyze images. ([GitHub][28])

**Best Docker stack:**

* **Trivy** — vulnerabilities, IaC, secrets, licenses, misconfigurations.
* **Hadolint** — Dockerfile correctness and maintainability.
* **Grype** — second vulnerability scanner/SBOM path.
* **Docker Scout** — Docker-native SBOM/vulnerability workflow.

---

### 8. Static analysis and security coverage: **CodeQL + Semgrep + RustSec/cargo-deny**

Use **CodeQL** for semantic security and correctness analysis. GitHub describes CodeQL as treating code as data and querying it for vulnerabilities and errors; it supports JavaScript/TypeScript and has Rust support in public preview. ([CodeQL][29])

Use **Semgrep** for fast, custom static checks. Semgrep supports many languages, runs in IDE/pre-commit/CI workflows, and uses code-like rules. The big value for coverage auditing is that you can write repo-specific rules such as:

* “Every new API route must have a matching test file.”
* “Every new migration must have an Atlas/pgTAP verification.”
* “Every unsafe Rust block must have nearby tests or a justification.”
* “Every exported service method must be referenced by unit, integration, or API tests.” ([GitHub][30])

For Rust dependency risk, add **cargo-audit** and **cargo-deny**. cargo-audit checks `Cargo.lock` against RustSec advisories, while cargo-deny can enforce dependency, license, source, and duplicate-version policies. ([RustSec][31])

---

### 9. AI-assisted test generation: useful, but do not make it the foundation

There are some promising AI-ish tools, but I would use them as accelerators, not as the source of truth.

**Keploy** is useful for API/service test generation from real traffic, OpenAPI, Postman collections, and curl commands. It records and replays API behavior and can help produce regression tests from real calls. ([Keploy][32])

**Momentic** is interesting for AI-assisted E2E test creation and maintenance; it can identify important user flows, generate tests, and update tests as the app changes. ([Momentic][33])

**Qodo** has current products for AI-assisted test generation in several languages including JavaScript/TypeScript, but I would not base your Rust strategy on it. Also, the older open-source `qodo-cover` repository is no longer maintained, so avoid treating that repo as a core long-term tool. ([Qodo Documentation][34])

For your Rust + Vite + TypeScript + Postgres + Docker stack, I’d implement this exact core set:

## The audit process I’d run

First, generate baseline reports: `cargo-llvm-cov` for Rust, Vitest coverage for TypeScript, Playwright coverage for E2E flows, Schemathesis/TraceCov for API coverage, Atlas/pgTAP for DB coverage, and Trivy/Hadolint for Docker. Upload all of those to Codecov using separate flags so gaps are visible by layer.

Second, run mutation testing: `cargo-mutants` for Rust and StrykerJS for TypeScript. This is where you find the embarrassing gaps: code that is “covered” but where tests do not actually assert the behavior.

Third, build an API coverage matrix from OpenAPI: every operation, required parameter, optional parameter, auth mode, response code family, error case, pagination path, and schema boundary should be either covered by deterministic tests or fuzzed by Schemathesis/RESTler.

Fourth, create a user-flow inventory: login, signup, onboarding, CRUD flows, billing/admin flows, permission boundaries, failure states, empty states, mobile/responsive paths, and destructive actions. Use Playwright for explicit flows and Meticulous to discover flows real users hit that your test suite misses.

Fifth, add repo-specific Semgrep rules so future gaps are caught at PR time: new route without API test, new component without Storybook story, new migration without Atlas/pgTAP verification, new public Rust function without tests, new Dockerfile without Trivy/Hadolint pass.

## The realistic definition of “full coverage”

I would define “full coverage” as this, not merely 100% line coverage:

* **High patch coverage:** new code is covered before merge.
* **High branch coverage:** important conditionals are tested both ways.
* **Mutation score tracked:** covered code has meaningful assertions.
* **API contract coverage:** every documented operation, parameter class, schema boundary, auth path, and major response class is exercised.
* **User-flow coverage:** every critical product workflow has at least one deterministic E2E test.
* **DB coverage:** migrations, RLS, triggers, functions, constraints, and critical queries are tested against real Postgres.
* **Container coverage:** every image, Dockerfile, and IaC artifact is scanned in CI.
* **Static/security coverage:** CodeQL/Semgrep/RustSec checks run on every PR.

The highest-leverage first install would be: **Codecov + cargo-llvm-cov + Vitest coverage + Playwright + Schemathesis/TraceCov + Testcontainers + Trivy + cargo-mutants + StrykerJS**. That combination will catch the most coverage gaps the fastest across Rust, Vite/TypeScript, Postgres, Docker, APIs, and real user flows.
