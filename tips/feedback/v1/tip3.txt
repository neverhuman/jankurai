## Overall verdict

The draft has the right instinct and several strong raw materials: the “languages as bottleneck compression” framing, the 100-point rubric, the winner-stack argument, repository routing artifacts, generated zones, proof lanes, and the jankurai audit idea. But it is **not yet a landmark standard**. Right now it reads like a forceful position paper with an opinionated stack recommendation. To become the standard you want, it needs to become a **specification plus evidence-backed manifesto**: clearer definitions, stronger empirical grounding, reproducible scoring, a richer treatment of vibe-coding failure modes, a full testing/QA model, and a much more concrete jankurai adoption protocol. I reviewed the full 12-page draft, including the abstract, Sections I–XII, tables, appendices, and references. 

The core upgrade is this: **stop arguing that one stack is best because it feels right; show that agent-era repositories need verification density, context economy, and repair routing, then prove why the proposed stack gives the highest default verification density per unit of agent effort.**

---

## The biggest problem: the paper is still more “strong opinion” than “new standard”

The draft’s strongest claim is that AI makes code generation cheap and trustworthy merging scarce. That claim is excellent. It aligns with DORA’s finding that AI amplifies the strengths and weaknesses of an organization rather than magically fixing broken systems, and it is reinforced by security evidence showing that AI-generated code can be syntactically plausible while still containing significant flaws. ([Dora][1])

But the paper currently jumps too quickly from that claim to a stack verdict. The missing bridge is a formal standard model:

**Define the unit of the new standard.** Is it a language stack standard, a repository standard, a CI standard, an agent-instruction standard, an audit standard, or all of them? Right now it is all of them, but the paper does not name the layers clearly enough.

**Define “agent-native.”** The paper uses the idea well, but it needs a crisp definition: an agent-native repository is one whose structure, contracts, tests, observability, and exception model allow an autonomous coding agent to localize ownership, make a bounded change, run the correct proof lane, and produce repair evidence without relying on hidden human context.

**Separate MUST / SHOULD / MAY.** Landmark standards are enforceable. The draft should introduce “jankurai Standard v0.1” with normative language. Example: “Generated client code MUST be read-only unless the source contract or generator changes.” “Every path prefix MUST map to an owner cell and proof lane.” “Agent-specific instruction files MAY exist, but MUST mirror the canonical root policy.”

**Make the stack ranking reproducible.** Table IV gives scores, but not enough score decomposition. A skeptical CTO will ask, “Why is Rust/TS/Postgres 94 and Go/TS/Postgres 90?” The paper must show the sub-scores, scoring rubric, evidence basis, uncertainty band, and sensitivity analysis.

---

## Priority rewrite: make the paper take the reader on a journey

The introduction should not start by defending the paper’s provocation. It should start with the historical shift:

1. **Code was once expensive to write.** Languages compressed hardware, syntax, memory management, deployment, and team coordination.
2. **Frameworks made product scaffolding cheap.** Teams optimized around human fluency and review culture.
3. **AI made first drafts cheap across every mainstream language.** The bottleneck moved from typing to verification.
4. **Old repositories are badly shaped for this world.** They encode tribal knowledge, informal boundaries, stale exceptions, and fragile tests.
5. **Therefore the new standard is not “prompt better.”** The new standard is repository-as-verification-interface.

This journey is more compelling than the current opening. It will also make the “winner stack” argument feel earned rather than declared.

GitHub’s 2025 Octoverse data gives you a strong public-data anchor: TypeScript became the most-used language on GitHub by monthly contributors in August 2025, with GitHub explicitly connecting typed languages and AI-assisted development to stronger guardrails. ([The GitHub Blog][2]) That supports your TypeScript product-surface claim. CISA/NSA memory-safe-language guidance supports your Rust-core claim, but you need to connect it directly to agent-generated code: memory-safe defaults reduce the number of catastrophic errors that cannot be caught by prompt discipline or human review. ([U.S. Department of War][3])

## Abstract feedback

The abstract is directionally strong, but it tries to do too much and does not yet sell the standard. It names the stack, rubric, audit, and end state, but it does not clearly state the paper’s central contribution as a standard.

What to improve:

**Make the first sentence sharper.** The first sentence should say that AI coding changes the scarce resource from code production to code rejection, localization, and repair.

**Define the contribution in one sentence.** Right now the abstract lists many artifacts. Convert that list into a single named standard: “We propose jankurai: an agent-native repository and CI standard for scoring, constraining, testing, and repairing AI-generated code.”

**Tone down unsupported certainty in the abstract.** The stack can be the default winner under the rubric, but the abstract should acknowledge scope: durable product systems, web/product surfaces, AI/data services, and general-purpose backend cores. Otherwise readers will attack edge cases: embedded, mobile-native, hard realtime, HPC, regulated JVM/.NET estates.

**Add evaluation intent.** The abstract should say the rubric is a v0 standard requiring empirical calibration. That does not weaken the paper; it makes it credible.

A stronger abstract shape:

> AI-assisted development changes the scarce resource in software engineering. First drafts are no longer the bottleneck; trustworthy merge is. Modern agents can generate plausible code across languages, but repositories built around human memory, informal review, and local taste cannot reliably reject bad generated code, localize defects, prove repairs, or preserve security posture. This paper proposes jankurai, an agent-native repository and CI standard for durable product systems. The standard combines a stack rubric, a default winning architecture, machine-readable owner and proof-lane maps, generated contract zones, agent-scoped instruction files, exception records, visual QA, and an audit model for detecting vibe-coding drift. Under the proposed rubric, Rust core, TypeScript/React/Vite product surface, PostgreSQL durable truth, generated contracts, and bounded Python AI/data services form the strongest default stack because they maximize compile-time rejection, contract enforcement, data integrity, security gating, observability, and repair routing. The goal is not human-free engineering; it is human-directed automation where the repository becomes the verification interface for agents.

## Section-by-section feedback

### I. Introduction

The introduction has the right thesis but the wrong first move. It should not spend early space explaining its own rhetorical posture. Open with a concrete failure narrative:

An agent changes a checkout flow. The UI looks right. The generated API client is stale. A fallback path bypasses an idempotency key. A test passes because it mocks the wrong layer. A secret lands in an environment file. The PR looks productive but the repository cannot prove safety. That is the new failure mode.

Then generalize. This would make the reader feel the problem before seeing the standard.

Also add a paragraph on **why human expertise does not disappear**. The paper currently risks sounding like human code knowledge no longer matters. That is too crude. The better claim is: human expertise moves from syntax recall and manual patching to architecture, policy, threat modeling, review of proof evidence, and standard evolution. This aligns better with DORA’s “AI as amplifier” finding. ([Dora][1])

### II. Languages as Compression Technology

This is one of the best conceptual sections. Keep it. Expand it.

The phrase “language as compression package” is strong because it reframes language choice away from taste. But the paper needs more examples of **what each generation compressed**:

Machine code compressed nothing. Assembly compressed numeric operations into symbolic labels. C compressed portable systems access. Java compressed memory management and enterprise deployment. Python compressed exploration. JavaScript compressed browser access. TypeScript compressed large-team browser maintenance. Rust compresses memory safety and ownership discipline into the type system.

The next step is: **agent-native stacks compress verification work.** That should become the section’s conclusion.

### II-A. Why Languages Fracture / Hype and Niche Projects

This needs a major upgrade. The current section is too short and risks sounding dismissive. Do not call Julia dead. Julia is still a serious language in scientific computing; the better argument is that Julia solved an important inner-loop problem but did not become the universal product-engineering default. Julia’s own founding paper framed the language around high-level numerical computing and the two-language problem, while more recent discussion of Julia for scientific machine learning still treats adoption and ecosystem issues as active topics. ([SIAM][4])

Rename the section to something like **“The Hype-to-Niche Pattern.”** The lesson is not “these projects failed.” The lesson is: technical elegance does not automatically create default-stack gravity.

Add a table with columns:

| Case              | Real breakthrough                             | Why it did not become the default product stack                                                         | Agent-era lesson                                |
| ----------------- | --------------------------------------------- | ------------------------------------------------------------------------------------------------------- | ----------------------------------------------- |
| Julia             | High-level numerical speed, multiple dispatch | Scientific strength did not automatically solve product ops, hiring, web ecosystem, deployment defaults | Domain excellence is not stack universality     |
| Elm               | Reliable frontend architecture                | Interop and ecosystem constraints                                                                       | Correctness must coexist with ecosystem gravity |
| D / Nim / Crystal | Ergonomic native/system alternatives          | Could not overcome C/C++/Go/Rust gravity                                                                | Small corpora weaken agent repair reliability   |
| Scala / Haskell   | Powerful type and abstraction systems         | Complexity, training, uneven conventions                                                                | Proof power must remain readable to agents      |
| Clojure / F#      | High-leverage functional niches               | Smaller mainstream product footprint                                                                    | Expert abstraction can become repair opacity    |
| Elixir/Phoenix    | Fault tolerance and realtime                  | Specialist fit, not universal backend default                                                           | Specialist override, not default standard       |

Use multiple adoption signals, not one ranking. TIOBE currently places Julia outside the mainstream top tier, but TIOBE’s methodology is only a proxy, so pair it with GitHub contributor trends, package maturity, job postings, cloud docs, and public examples. ([TIOBE][5])

### III. The Harsh AI Reality

This section is necessary but underpowered. It should become the paper’s evidence spine.

Add three subsections:

**A. First-draft abundance.** Agents can produce plausible code across languages, so syntax fluency is no longer the main bottleneck.

**B. Plausibility is dangerous.** Veracode’s 2025 GenAI Code Security Report says AI-generated code introduced risky security flaws in 45% of tests; GitGuardian reports 28.65 million new hardcoded secrets added to public GitHub commits in 2025, with the report framing AI-assisted coding as part of the acceleration pressure. ([Veracode][6])

**C. Agents fail at environment, context, and proof.** SWE-bench evaluates real GitHub issue resolution; SWE-agent shows agent-computer interface design matters; SetupBench shows agents struggle to bootstrap environments; ContextBench shows context retrieval remains a bottleneck; UTBoost shows that patches can pass insufficient tests without truly solving the issue. ([arXiv][7])

This section should end with the key standard claim: **the repository must become an adversarial verifier of agent output.**

### IV. Method: 100-Point Stack Rubric

The rubric is a good start, but it is not yet defensible.

Problems:

**The weights look invented.** That is acceptable for v0, but the paper must admit it and provide a calibration plan.

**There is no scoring protocol.** For each criterion, define 0, 25%, 50%, 75%, and 100% credit. Example: “Security, memory safety, supply chain: 0 = no structural controls; 4 = SCA only; 8 = SCA + lockfiles + secret scanning; 12 = memory-safe core + SCA + secret scanning + SBOM; 16 = all prior plus provenance, unsafe ledger, policy gates.”

**No sensitivity analysis.** Show what happens if product velocity weight increases, or if enterprise integration weight increases, or if AI/data ecosystem weight increases. If Rust still wins under multiple weight profiles, the claim becomes much stronger.

**Missing criteria.** Add:

* Agent context economy.
* Tooling stability and LTS.
* Upgrade/migration friction.
* Visual QA compatibility.
* Accessibility/test automation compatibility.
* Agent-instruction portability.
* Brownfield migration viability.

### V. Top-Five Stack Ranking

The ranking needs more rigor and more humility.

Rust/TypeScript/PostgreSQL is a credible default. But the paper must address obvious alternatives directly: Next.js/Supabase-style TypeScript stacks, Python/FastAPI/React/Postgres, Rails/Postgres, Laravel/Postgres, Go/React/Postgres, .NET/React/Postgres, Elixir/Phoenix/Postgres, and Java/Kotlin Spring stacks.

Right now the paper ranks five stacks but does not explain why other common vibe-coding stacks are absent. That is a gap. The vibe-coding world is full of TypeScript-heavy, Supabase-heavy, Firebase-heavy, and Python-heavy projects. If this paper is meant to “cover every corner of vibe coding,” those stacks must be scored or at least placed in a “why they are not the default” table.

Also, the phrase “winner-only architecture” is strategically useful but academically fragile. Better formulation:

**“One default, explicit exceptions.”**

The standard can say: for durable product systems, the default target is Rust core + TypeScript product surface + PostgreSQL truth + generated contracts + bounded Python. Exceptions are allowed only when named, scoped, scored, tested, and time-bounded.

### VI. Winner-Only Architecture

The architecture section is useful but needs more operational detail.

Add:

**Dependency direction rules.** Example: web can depend on generated clients; generated clients depend on contracts; application depends on domain; adapters depend on domain/application interfaces; domain depends on nothing external.

**Forbidden edges.** Direct SQL from web. Product truth in Python. Business invariants in handlers. Handwritten DTO mirrors. Generated zone edits. Cross-service auth logic duplication.

**Patch routing examples.** Give concrete examples:

* “Add refund eligibility rule” → Rust domain + property tests + Postgres constraint if durable.
* “Change checkout form copy” → TypeScript route/component + visual lane.
* “Add embedding provider” → bounded Python AI service + contract tests.
* “Add payment retry” → Rust application/workers + idempotency tests + telemetry.
* “Add account deletion” → application + adapters + DB migrations + Playwright journey + privacy runbook.

This makes the repository tree feel like a standard rather than a diagram.

### VII. Agent-First Repository Design

This section is important but needs to become one of the paper’s centerpieces.

The paper should distinguish **agent-agnostic controls** from **tool-specific adapters**.

Agent-agnostic controls:

* Short root instruction file.
* Local path-level instruction files.
* `owner-map.json`.
* `test-map.json`.
* `proof-lanes.toml`.
* `generated-zones.toml`.
* Exception records.
* Repair receipts.
* Filtered command wrappers.
* Deterministic local setup.
* Read-only generated outputs.
* Structured failure output.

Tool-specific adapters:

* Codex can use `AGENTS.md` as project guidance, so the root file should be short and canonical. ([OpenAI Developers][8])
* Cursor supports persistent Project, Team, User Rules, plus `AGENTS.md`; this is where the paper should discuss Cursor’s strength in scoped style and project-rule enforcement. ([Cursor][9])
* Google Antigravity is explicitly positioned as an agentic development platform where agents can plan, execute, verify, use browser/editor/terminal surfaces, and operate across workspaces; the paper should connect that to proof artifacts, screenshot evidence, and manager-style orchestration. ([Google Antigravity][10])
* Claude Code’s current memory documentation appears to emphasize project memory through `MEMORY.md` and related hierarchy, so the paper should avoid hard-coding stale tool assumptions unless it verifies current behavior. ([Claude][11])
* GitHub Copilot repository/path instructions should be treated as a mirror of the canonical standard, not a competing source of architecture truth.

The key line to add: **agent instruction files are not documentation; they are policy adapters.**

### VIII. Testing and Automated QA

This section is too short for the ambition of the paper. Testing is where the standard lives or dies.

Add subsections:

**A. Proof lanes, not test piles.** The paper already hints at this. Make it formal: fast, contract, DB, security, web, visual, accessibility, AI-eval, full release.

**B. Generated tests are suspect.** AI can generate many tests, but test volume is not test signal. UTBoost’s SWE-bench analysis found insufficient test cases and erroneous patches labeled as passed, which supports the paper’s claim that passing a weak test suite is not proof. ([ACL Anthology][12])

**C. Test provenance.** Every generated test should carry provenance: generated_by, prompt/task ID, behavior under test, source requirement, mutation expectation, and whether it failed before the fix.

**D. Negative proof.** Require red/green evidence for bug fixes: the new test fails on the old code and passes on the new code.

**E. Flake policy.** No silent retries. Flaky tests go to quarantine with owner, expiration, and risk impact. A flaky CI lane is a human bottleneck disguised as automation.

**F. Coverage model.** Move beyond line coverage. Require layer-weighted coverage: domain invariant coverage, contract coverage, DB constraint coverage, critical journey coverage, visual state coverage, accessibility coverage, and security gate coverage.

### Pixel QA and app-step automation

This needs a full dedicated section. The paper currently mentions Playwright, but it does not yet cover pixel QA deeply enough.

Automated pixel QA should be treated as a first-class proof lane for agent-generated UI. Playwright supports screenshot comparison through `toHaveScreenshot`, and its page assertion waits until screenshots stabilize before comparing; Storybook/Chromatic turns component stories into visual tests; Applitools provides visual AI workflows for web/app/doc testing. ([Playwright][13])

The standard should require:

| Layer                   | What to test                                       | Tooling direction                           | Jankurai rule                                  |
| ----------------------- | -------------------------------------------------- | ------------------------------------------- | ----------------------------------------------- |
| Component visual states | Buttons, forms, modals, empty/loading/error states | Storybook + Chromatic or equivalent         | Every shared component has canonical stories    |
| Route screenshots       | Critical pages at defined breakpoints              | Playwright screenshots                      | UI PRs touching routes must run visual lane     |
| Journey evidence        | Checkout, auth, onboarding, destructive actions    | Playwright video/trace/screenshot artifacts | Agent must attach artifacts to repair receipt   |
| Dynamic content         | Dates, ads, personalization, async content         | Mask regions, freeze clocks, seed data      | No baseline churn from nondeterminism           |
| Accessibility           | Keyboard, focus, labels, contrast, landmarks       | axe-core + Playwright + WCAG 2.2 checklist  | Visual lane must not replace accessibility lane |

WCAG 2.2 should be cited because pixel-perfect UI can still be inaccessible; axe-core can automate part of the accessibility signal inside functional test environments. ([W3C][14])

Hard rule to add: **agents may not update visual baselines unless the PR declares a visual-intent change.** Otherwise agents will “fix” visual tests by accepting regressions.

### IX. Agent-Friendly Exceptions

This is one of the better sections. Make it more concrete with examples.

Add a full example:

```text
code: PAYMENT_IDEMPOTENCY_KEY_MISSING
owner: application/payments
protected_invariant: one external charge attempt per logical checkout command
safe_user_message: "We could not safely retry this payment."
common_agent_fixes:
  - ensure command creates or loads idempotency key before provider call
  - add integration test for duplicate retry
  - verify OpenTelemetry span includes payment.command_id
forbidden_fixes:
  - retry provider call without idempotency key
  - suppress error at route handler
proof_lanes:
  - fast
  - application
  - security
  - payment-contract
```

This turns the exception model into a repair interface.

Also add RFC 9457 and OpenTelemetry as standards anchors: RFC 9457 defines machine-readable HTTP problem details, while OpenTelemetry provides vendor-neutral telemetry and exception attributes. ([RFC Editor][15])

### X. The Jankurai Audit

This should become the central enforcement section, not just a late-stage audit.

The paper should state: **jankurai belongs in every coding project, not only projects using the winner stack.** For non-target stacks, it should still score boundary clarity, proof lanes, generated zones, security gates, exceptions, and vibe-coding drift. The target stack gets the highest ceiling; other stacks can still be audited.

Add two modes:

**Advisory mode:** report-only, no CI failure, first 30 days.

**Enforced mode:** hard gates on secrets, generated-zone mutation, missing proof lane, unsafe dependency, direct DB from wrong layer, undocumented exception, security scanner disabled.

**Ratchet mode:** once the repo reaches a score, it cannot regress without an exception.

The “track closer to the center” idea should become a metric:

```text
center_distance =
  boundary_drift
+ generated_zone_drift
+ proof_lane_gap
+ exception_debt
+ security_gate_gap
+ context_noise
+ dead_language_density
+ ownership_ambiguity
```

Every PR should produce:

* Raw jankurai score.
* Capped score.
* Score delta.
* Center-distance delta.
* Findings by severity.
* Agent fix queue.
* Required proof lanes.
* Exceptions touched.
* Visual artifacts required.
* Security artifacts required.

### XI. Migration, Versioning, Future Research

Good section, but it needs an adoption plan.

Add:

**30 days:** run jankurai advisory, create root instructions, owner map, test map, generated-zone inventory.

**60 days:** enforce hard gates for secrets, generated-zone mutation, missing proof lanes, direct DB violations, and security scans.

**90 days:** move to ratchet mode, publish exception register, add visual QA for critical flows, require repair receipts.

**180 days:** stack migration roadmap, Rust core extraction for invariants, DB constraint hardening, Python containment, generated contracts.

Also add governance: standard version, changelog, compatibility matrix, deprecation policy, reference implementation, sample repos, and public leaderboard.

### XII. Conclusion

The conclusion is strong but too absolute. Keep the force, but avoid easy dismissal by skeptics.

Better final claim: **the future is not “AI writes code.” The future is “repositories reject, route, and prove code fast enough that agents can safely operate at full speed.”**

## The jankurai section should be upgraded into a full standard

The paper should define jankurai as both a **static audit** and a **behavioral CI contract**.

### Required jankurai artifacts

```text
agent/
  standard-version.toml
  owner-map.json
  dependency-policy.json
  test-map.json
  proof-lanes.toml
  generated-zones.toml
  exception-policy.toml
  visual-qa.toml
  security-policy.toml
  repo-score.json
  repo-score.md
  repair-queue.json
```

### Required jankurai output

```json
{
  "standard_version": "0.1.0",
  "target_stack": "rust-ts-vite-postgres-bounded-python",
  "raw_score": 86,
  "capped_score": 78,
  "center_distance": 22,
  "score_delta": -4,
  "hard_gates_failed": [
    "generated_zone_mutation",
    "missing_visual_lane_for_ui_change"
  ],
  "findings": [],
  "agent_fix_queue": [],
  "required_proof_lanes": ["fast", "contract", "web", "visual", "security"]
}
```

### Hard caps the paper should specify

A repo cannot score above:

* **40** if secret scanning or dependency scanning is absent.
* **50** if there is no owner map.
* **55** if there is no test map.
* **60** if generated zones are not declared.
* **65** if direct DB access crosses ownership boundaries.
* **70** if critical user journeys lack e2e proof.
* **75** if runtime code contains undocumented banned markers.
* **80** if exceptions are not machine-readable.
* **85** if visual QA is absent for UI product surfaces.
* **90** if no provenance/SBOM/release evidence exists.

OpenSSF Scorecard and SLSA are useful external anchors for the supply-chain portion of the audit because they already frame automated security-health checks and artifact integrity. ([undefined][24])

## Stack rubric improvements

The current 100-point rubric is directionally good. I would revise it like this:

| Dimension                            | Weight | Why                                      |
| ------------------------------------ | -----: | ---------------------------------------- |
| Compile/type/proof rejection speed   |     15 | Agent output must fail fast              |
| Security and memory safety           |     15 | Generated code expands attack surface    |
| Boundary and contract integrity      |     12 | Agents need stable interfaces            |
| Data truth and workflow safety       |     10 | Durable invariants must survive bad code |
| Test routing and proof-lane maturity |     10 | Test volume without routing is noise     |

## Layout, formatting, and LaTeX improvements

The current body has dense tables and cramped code/tree blocks. The paper needs more white space and stronger typographic hierarchy.

Concrete upgrades:

**Use fewer giant tables in the main body.** Keep the most important table in each section and push expanded tables to appendices.

**Use `booktabs` tables.** Avoid heavy borders. Use `\toprule`, `\midrule`, `\bottomrule`.

**Use `tabularx` or `longtable` for wide tables.** Several tables are too dense for two-column layout.

**Use a professional font stack.** For a polished research feel, use Libertinus or STIX Two with `microtype`. Example:

```latex
\usepackage[T1]{fontenc}
\usepackage{libertinus}
\usepackage{microtype}
```

**Use `siunitx` for numbers and scores.**

```latex
\usepackage{siunitx}
\sisetup{detect-weight=true,detect-family=true}
```

**Use `cleveref` for references.**

```latex
\usepackage[nameinlink,noabbrev]{cleveref}
```

**Use `xurl` and proper BibTeX/BibLaTeX fields** so long URLs do not destroy the references section.

**Use `tcolorbox` or `fvextra` for repository trees.** The appendix tree is too small and visually exhausting. Break it into three boxes: product tree, agent-control tree, CI/security tree.

**Fix extracted-text artifacts.** Some PDF-extracted words show line-break artifacts. Even if the visual PDF looks fine, a landmark paper should copy/paste cleanly for search, indexing, and citation.

**Use sentence-case headings if not locked to IEEE.** The all-caps headings make the paper feel more like a conference template than a new standard document.

**Add a one-page “standard summary.”** After the introduction, include a compact page with definitions, required artifacts, hard gates, and the target stack. This makes it feel like a standard, not just an essay.
