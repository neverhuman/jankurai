I reviewed the full draft, including the abstract, sections I–XII, tables, appendices, and references. The draft is directionally strong: it already has the right ingredients—agent-native repository shape, stack rubric, winner stack, proof lanes, generated zones, exceptions, and a jankurai audit. But it is not yet a landmark standard. It currently reads like a forceful manifesto with a standard embedded inside it. The next version needs to read like a standard-setting research paper with a manifesto’s urgency. 

## Core verdict

The central thesis is right: AI coding moves the bottleneck from **writing plausible code** to **rejecting, localizing, repairing, and proving generated code**. That is the strongest idea in the paper. The problem is that the draft often asserts this before it has fully earned it.

The paper should become four things at once:

1. **A historical argument**: programming languages evolved as bottleneck-compression systems.
2. **A technical standard**: agent-first repositories must expose machine-readable ownership, proof lanes, generated zones, exceptions, and audit signals.
3. **A stack decision framework**: choose the stack that maximizes verification speed, security, repair locality, and agent routing—not human comfort.
4. **A conformance program**: jankurai becomes the enforcement mechanism every repo runs, with scores, hard caps, drift detection, repair queues, and CI policy.

Right now, the paper has all four pieces, but the order and rigor need work. The strongest upgrade is to stop making the paper merely argue for a preferred stack and instead make it launch **a normative engineering standard**: what every project must contain, how it is scored, how deviations are justified, and how agents are safely allowed to operate.

External evidence supports the direction, but the draft should use it more strategically. DORA’s 2025 research frames AI as an amplifier of existing organizational capability, not a magic productivity switch. GitHub’s 2025 Octoverse says TypeScript became the most-used language on GitHub and connects typed languages to agent-assisted production reliability. Veracode reports that AI-generated code introduced risky security flaws in 45% of tests. GitGuardian reports 28.65 million new hardcoded secrets in public GitHub commits in 2025 and an 81% surge in AI-service leaks. Sonar reports a verification gap: 96% of developers do not fully trust AI-generated code, yet only 48% always check AI-assisted code before committing. These facts should form the empirical spine of the paper. ([Dora][1])

## The paper’s goal should be rewritten as a sharper standard

The paper should explicitly say something like this early:

> The purpose of this standard is to make repositories safe for high-speed agentic software production. It does not assume generated code is trustworthy. It assumes generated code is cheap, plausible, and dangerous until routed through ownership boundaries, generated contracts, deterministic proof lanes, security gates, visual verification, and repair-oriented audit output.

That is much stronger than simply saying the paper proposes a stack. The standard is not “use Rust, TypeScript, and PostgreSQL.” The standard is:

> Every repository must become a verification interface for coding agents.

The stack is the recommended default because it best supports that interface.

## Recommended new structure

I would restructure the paper around this flow:

1. **The bottleneck moved**
   AI did not remove software engineering. It moved the scarce resource from code production to trustworthy merge.

2. **Languages as bottleneck compression**
   Walk the reader through machine code, assembly, Fortran, COBOL, C, Java, Python, JavaScript, TypeScript, Rust.

3. **Why technically exciting stacks fail to become defaults**
   This is where Julia, Elm, Haskell, Scala, D, Nim, Crystal, Reason/OCaml frontend, and Elixir belong. Do not call Julia dead. Say: “technical success is not the same as default-stack success.”

4. **Definitions**
   Define vibe coding, agent-first, agent-native, proof lane, generated zone, repair receipt, exception, centerline drift, jankurai.

5. **Threat model**
   Agents can edit files, run commands, browse, call tools, mutate tests, leak secrets, follow malicious instructions, and overfit to local green checks.

6. **Why stack choice is now proof choice**
   Language familiarity matters less. Rejection speed, contract strength, security posture, repair locality, and corpus/tooling maturity matter more.

7. **The 100-point stack rubric**
   Keep the rubric, but justify weights and include sensitivity analysis.

8. **The winner default and exception policy**
   Declare the preferred stack, but allow explicit, scored, temporary exceptions.

9. **Agent-first repository design**
   Universal repo rules first; tool-specific adapters second.

10. **jankurai conformance and code audit**
    This should be a main pillar, not a later enforcement detail.

11. **Testing, pixel QA, and future CI**
    This needs a much larger section.

12. **Migration and governance**
    How teams adopt the standard, version it, and avoid private dialects.

13. **Conclusion**
    End with the repo-as-verification-interface argument, not just the stack.

## Abstract feedback

The current abstract is strong but overloaded. It tries to introduce the problem, preferred stack, rubric, audit markers, and philosophical endpoint all at once. It should do less but land harder.

Problems:

* It names too many implementation details before the reader understands the standard.
* It frames the stack as the main contribution, but the real contribution should be the **agent-native conformance system**.
* It does not define “vibe coding” or “agent-native” clearly enough.
* It underplays jankurai. For this paper’s goal, jankurai is not a side audit. It is the enforcement layer of the standard.
* It says the end state is not human-free engineering, which is good, but it needs a stronger framing: humans move from typing and informal review to setting invariants, policy, risk budgets, and proof thresholds.

A stronger abstract would be:

> AI coding has made plausible code cheap. That changes the failure mode of software organizations: the bottleneck is no longer producing code, but rejecting unsafe code, localizing defects, proving repairs, preserving security posture, and teaching the next agent what changed. This paper proposes an agent-native engineering standard for repositories designed to be maintained by humans and coding agents together. The standard treats the repository as a verification interface: every path has an owner, every change has a proof lane, every generated file has a source of truth, every exception is scoped and expiring, and every merge is audited for drift away from the standard.
>
> The paper contributes a stack-scoring rubric, a default winning architecture, a repository layout, proof-lane model, generated-zone policy, exception schema, visual QA strategy, CI model, and jankurai audit. The recommended default stack is Rust for core invariants, TypeScript/React/Vite for product surface, PostgreSQL for durable truth, generated contracts for boundaries, and bounded Python for AI/data service work. This stack is not chosen for author preference. It is chosen because it maximizes compile-time rejection, contract enforcement, database integrity, security scanning, deterministic testing, observability, and repair routing.
>
> The goal is not to remove humans from engineering. It is to move humans to the control plane: defining product intent, invariants, risk policy, review thresholds, and exception governance while agents perform increasingly automated implementation and repair.

## Introduction feedback

The introduction has energy, but it starts too internally. Remove the meta-opening and start with the bottleneck shift. The reader should immediately feel the historical arc.

A better opening would be:

> Software history is often told as a story of languages. That is too shallow. Software history is a story of bottleneck compression. Assembly compressed machine instruction recall. Fortran compressed numerical expression. COBOL compressed business records. C compressed portable systems access. Java compressed memory management and deployment portability. Python compressed exploration. JavaScript compressed browser programmability. TypeScript compressed large-team JavaScript maintenance. Rust compresses memory-safety and concurrency review into the compiler.
>
> AI coding changes the next bottleneck. Syntax, boilerplate, and first-draft implementation are no longer scarce. Plausible code is cheap. Trustworthy code is not.
>
> The repository must therefore change roles. It can no longer be a loose archive of human-written files plus tribal review conventions. It must become a control surface for agents: a machine-readable system that routes edits, rejects invalid states, proves repairs, records exceptions, and prevents local convenience from becoming long-term architecture.

That gives the paper the “journey of code evolution” you want.

The introduction also needs one very clear “claims and contributions” paragraph. Right now, the claims exist, but they are not framed like a landmark paper. Add:

> This paper makes three normative claims and one operational contribution. The normative claims are: first, agentic software engineering requires repositories optimized for verification rather than memory; second, stack choice should be scored by rejection speed and repair locality rather than developer preference; third, autonomous coding requires a default winner stack plus explicit exception governance. The operational contribution is jankurai: a repo audit and CI standard that scores conformance, detects vibe-coding drift, and emits machine-readable repair queues for agents.

## Section-by-section critique

### Abstract

Keep the central bottleneck argument. Cut some implementation density. Move the banned-marker examples into the jankurai section. The abstract should not spend precious space listing “legacy,” “deprecated,” “temporary,” and similar markers. It should say jankurai detects unmanaged temporary architecture and dead-language markers, then leave the list for the audit section.

### Index Terms

Good, but add: **agent-first repositories**, **visual regression testing**, **CI policy**, **software supply chain**, **agent security**, **conformance audit**.

### I. Introduction

The current section has the right claims, but it needs more narrative discipline. It should not feel like it begins in the middle of an argument. The reader needs to be taken from “language history” to “AI changes bottlenecks” to “repositories must become verification systems.”

Add a short “What changes in the agent era?” list:

* Code generation is cheap.
* Verification is expensive.
* Context is limited.
* Agents overfit to visible patterns.
* Agents need path ownership and exact proof commands.
* Security mistakes scale with generation speed.
* CI becomes the merge authority.

That list sets up the entire paper.

### II. Languages as Compression Technology

This is one of the best ideas in the draft. Expand it. It should become the conceptual foundation of the paper.

Add a formal sentence:

> A programming language compresses human burden into syntax, type systems, runtimes, libraries, deployment conventions, and tooling. In the agent era, the value of that compression is measured by how much ambiguity is removed before code reaches production.

Also add a sharper distinction:

* **Human-era value**: readability, familiarity, hiring, library access.
* **Agent-era value**: invalid-state rejection, proof locality, boundary enforcement, context economy, repairability.

### II.A Why Languages Fracture

This section is directionally right but too compressed. It needs the “hype stack autopsy” the user is asking for.

Do not frame Julia as dead. That will get attacked immediately by technical readers. Julia is a serious scientific-computing language designed to address the two-language problem, and the original SIAM Review paper explicitly presents it as a fresh approach to numerical computing. The correct lesson is not “Julia failed.” The lesson is: “A language can solve a real domain problem and still fail to become the default company-wide product stack.” ([SIAM][2])

Add a table with columns:

| Ecosystem             |                       Real technical win |                                        Why it did not become universal default |                                   Agent-era lesson |
| --------------------- | ---------------------------------------: | -----------------------------------------------------------------------------: | -------------------------------------------------: |
| Julia                 | Scientific productivity plus performance | Product/cloud/security/deployment defaults weaker than dominant product stacks |        Domain success is not default-stack success |
| Elm                   |                     Frontend reliability |                                              Ecosystem and interop constraints | Correctness is not enough without adoption gravity |
| Haskell               |                        Type-system power |                                         Expert abstraction and hiring friction |     Proof power must be operationally approachable |
| Scala                 |                JVM plus functional power |                                                Complexity and style divergence |      Agents struggle when local idioms vary wildly |
| D/Nim/Crystal         |     Better ergonomics/native compilation |                        Smaller ecosystems and fewer boring production patterns |       Thin corpus weakens agent repair reliability |
| Reason/OCaml frontend |                      Typed functional UI |                                                Tooling/community fragmentation |          Elegant syntax loses to ecosystem gravity |
| Elixir/Phoenix        |                 Fault tolerance/realtime |                                  Specialist fit, not universal product default |            Excellent override, not general default |

Also add a rule:

> The standard should respect niche winners while refusing to let every niche become a default architecture.

RedMonk’s 2026 ranking still shows mainstream inertia—JavaScript, Python, Java, C#, TypeScript, C++, Go, Kotlin, Scala, and Rust appear in the top 20, while many technically interesting languages remain outside the default adoption tier. That supports the paper’s point that ecosystem gravity matters, but it also means the paper must be careful not to confuse “not universal” with “dead.” ([RedMonk][3])

### III. The Harsh AI Reality

Good section, but it should be harsher and more concrete.

Add the following failure model:

> AI coding does not remove engineering debt. It accelerates debt creation unless the repository rejects ambiguity faster than agents create it.

This section should cite the trust and security evidence directly. Stack Overflow reports that 84% of developers use or plan to use AI tools, but 46% do not trust the accuracy of AI-tool output; it also defines vibe coding as generating software from LLM prompts and notes that nearly 77% said vibe coding is not part of their professional development work. Sonar’s 2026 survey shows a stronger verification gap: 96% do not fully trust AI-generated code, but only 48% always check AI-assisted code before committing. ([Stack Overflow][4])

Add a direct bridge:

> Therefore, the standard cannot depend on trust, seniority, or review heroics. It must depend on machine-enforced rejection.

Also include agent security, not only generated-code quality. OWASP identifies prompt injection as a top LLM risk, and its AI Agent Security Cheat Sheet emphasizes that agents introduce risks beyond normal LLM output because they can reason, use tools, keep memory, and take actions. MCP security guidance also calls out protocol-specific risks and security considerations for tool-connected agents. ([OWASP Gen AI Security Project][5])

### IV. Method: 100-Point Stack Rubric

This is crucial but currently too subjective. A standard-setting paper cannot just present weights; it must explain why those weights exist.

Add:

* **Weight rationale**: why correctness/security outrank velocity.
* **Scoring instructions**: who scores, what evidence counts, what does not count.
* **Hard caps**: no repo can score above certain thresholds if it lacks key controls.
* **Sensitivity analysis**: show whether Rust still wins if security, ecosystem, or simplicity weights shift.
* **Workload modifiers**: embedded, mobile, regulated enterprise, realtime collaboration, AI research, and brownfield migration.

Example hard caps:

| Missing control                           | Maximum score |
| ----------------------------------------- | ------------: |
| No deterministic CI proof lane            |            55 |
| No secret scanning/push protection        |            60 |
| No generated boundary contracts           |            70 |
| No path-to-owner map                      |            75 |
| No security/SCA/SBOM lane                 |            80 |
| No visual QA for user-facing app          |            85 |
| No jankurai audit in CI                  |            85 |
| No exception registry                     |            88 |
| No repair receipts / agent evidence trail |            90 |

This makes the rubric enforceable.

### V. Top-Five Stack Ranking

The ranking is plausible, but the paper currently makes it look more settled than it is. The score differences between ranks two through five are small. That is fine, but the paper needs to admit uncertainty without weakening the standard.

Add:

> The ranking is not a claim that every workload should be rewritten immediately. It is a claim about the default target for new durable product systems where agentic maintenance, security, and repairability are first-class requirements.

The current winner is defensible:

* Rust for domain/core invariants and memory-safe systems work.
* TypeScript/React/Vite for product surface.
* PostgreSQL for durable truth.
* Generated contracts for cross-language boundaries.
* Python boxed into AI/data service work.

But the paper must include direct “why not” subsections:

* **Why not Go as winner?** Excellent simplicity, cloud fit, and agent readability, but weaker type-level invariant density than Rust.
* **Why not C#/.NET as winner?** Strong enterprise platform, tooling, and regulated fit, but more platform identity and runtime surface.
* **Why not TypeScript everywhere?** Fastest product velocity and massive corpus, but weaker durable backend safety if allowed to own product truth.
* **Why not Python everywhere?** Dominates AI/data exploration, but too weak as product truth without heavy external guardrails.
* **Why not JVM as target?** Mature and durable, but broad framework surface and legacy gravity create more agent confusion.

Also add “specialist overrides”:

* Elixir/Phoenix for realtime collaboration and fault-tolerant coordination.
* Swift/Kotlin for native mobile.
* C/C++ only for constrained legacy/hardware cases, with a memory-safety roadmap.
* Existing Kafka only when brownfield/event workload demands it.

CISA/NSA memory-safety guidance supports the paper’s preference for memory-safe defaults, but the paper should cite this as part of the security argument, not as a blanket dismissal of all legacy systems. ([U.S. Department of War][6])

### VI. Winner-Only Architecture

This is conceptually strong. The paper should sharpen the principle:

> “Winner-only” does not mean no exceptions. It means one default architecture and no unowned alternatives.

Add a concrete example:

> A payment authorization change must not land in a UI component, Python prompt wrapper, or background job fallback. It must land in Rust domain/application logic, PostgreSQL constraints/policies where appropriate, generated contracts, and proof lanes that exercise authorization, idempotency, and rollback.

That makes the architecture feel real.

Also strengthen PostgreSQL’s role. PostgreSQL constraints can reject invalid data below application code, and row-level security can restrict which rows are visible or mutable based on policies. Those are exactly the kinds of lower-layer controls agents need when application code is fallible. ([PostgreSQL][7])

### VII. Agent-First Repository Design

This section is important but too short. It should become one of the paper’s flagship sections.

Split it into two subsections:

#### A. Universal agent-first design

Every AI coding tool benefits from:

* `AGENTS.md` or equivalent short root router.
* Local instructions near owned code.
* `owner-map.json`.
* `test-map.json`.
* `proof-lanes.toml`.
* `generated-zones.toml`.
* `audit-policy.toml`.
* `docs/exceptions/`.
* `docs/repair/`.
* deterministic setup command.
* command wrappers that summarize evidence and preserve raw logs.
* explicit forbidden edits.
* exact generated-code regeneration commands.
* CI that emits JSON for agents and Markdown for humans.

#### B. Tool-specific adapters

The paper should compare current agent surfaces and explain what each is good at:

| Tool surface       | Strength to call out                                                                                                                                      | Standard requirement                                                              |
| ------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| OpenAI Codex       | Reads `AGENTS.md` before work; useful for repo-level conventions and validation commands                                                                  | Generate `AGENTS.md` from canonical jankurai config                              |
| Cursor             | Project/team/user rules and scoped rule files are strong for style, architecture, and path-specific conventions                                           | Use Cursor rules as mirrors, not separate truth                                   |
| Claude Code        | Project memory and terminal-oriented workflows are useful for long repair sessions                                                                        | Memory must not override canonical repo policy                                    |
| GitHub Copilot     | Repository and path-specific custom instructions fit PR review and cloud-agent workflows                                                                  | Use `.github/copilot-instructions.md` and path instructions as generated adapters |
| Google Antigravity | Manager surface, multi-agent orchestration, browser/editor/terminal work, and artifacts such as plans, screenshots, and recordings fit end-to-end UI work | Treat artifacts as required proof evidence, not optional summaries                |

Codex’s AGENTS.md docs, Cursor’s rules docs, Claude Code’s memory docs, GitHub Copilot’s custom-instruction docs, and Google’s Antigravity announcement all support the need for a canonical instruction source that gets mirrored into tool-specific formats. ([OpenAI Developers][8])

### VIII. Testing and Automated QA

This section needs a major expansion. It is currently too conventional for the ambition of the paper.

Add a new model:

> Agent-native testing is not a test pyramid. It is a proof-routing system.

Minimum proof lanes:

| Lane          | Purpose                                                            |
| ------------- | ------------------------------------------------------------------ |
| `fast`        | changed-file proof, typecheck, lint, cheap unit tests              |
| `domain`      | Rust unit/property tests for invariants and state machines         |
| `application` | authz, transaction, idempotency, workflow tests                    |
| `contract`    | generated clients, OpenAPI/Protobuf/JSON Schema drift              |
| `db`          | migrations, constraints, RLS, schema snapshots                     |
| `web`         | TypeScript, component tests, critical Playwright journeys          |
| `visual`      | screenshot/pixel/perceptual diffs for design-critical screens      |
| `security`    | secrets, SCA, SBOM, provenance, policy                             |
| `agent`       | verifies instruction consistency, generated zones, repair receipts |
| `release`     | full merge confidence lane                                         |

Use the research evidence: SWE-bench evaluates real GitHub issue resolution, SWE-agent shows agent-computer interface design matters, ContextBench shows context retrieval is a bottleneck, and SetupBench shows agents struggle with environment bootstrapping. That supports the paper’s claim that repo interfaces, setup, context, and proof commands are not secondary—they determine whether agents can actually work. ([arXiv][9])

### Pixel QA and step QA need their own subsection

This is a critical missing piece.

The paper should say:

> For user-facing software, functional correctness is insufficient. Agents can produce a screen that passes route tests while breaking layout, spacing, z-index, responsive behavior, truncation, focus states, dark mode, or accessibility. Agent-native CI therefore needs visual proof lanes.

Concrete requirements:

* Use Playwright for critical user journeys and route-level screenshots.
* Use component-level screenshot tests for design-system primitives.
* Use full-page visual checks only for stable pages.
* Use masked regions for timestamps, avatars, ads, randomized content, and dynamic charts.
* Freeze time, seed data, disable animations, pin fonts, set viewport/DPR, and use deterministic test users.
* Require baseline updates to be explicit PR artifacts.
* Never allow an agent to silently update visual baselines.
* Store visual diffs as reviewable artifacts.
* Add accessibility checks alongside visual checks.
* Track visual flake rate separately from functional flake rate.

Playwright supports screenshot comparison through `toHaveScreenshot()`, and visual testing systems such as Applitools describe baseline/checkpoint comparison as a regression-testing method for catching unexpected screen changes. ([Playwright][10])

Add a “step QA” concept:

> Step QA is sequence-level proof. The agent must show that a user can complete the journey, not merely that a page renders.

For each critical journey, CI should store:

* test name.
* route.
* viewport.
* test user role.
* steps executed.
* assertions.
* screenshots.
* video or trace where useful.
* console/network errors.
* accessibility violations.
* visual diff summary.
* repair receipt if the test was created to fix a prior failure.

This is where Antigravity-like artifacts become important: screenshots, walkthroughs, and recordings should become formal proof artifacts rather than nice-to-have review aids. Google explicitly describes Antigravity artifacts such as task lists, implementation plans, screenshots, and browser recordings as a way to verify agent work without reading raw logs. ([Google Developers Blog][11])

### IX. Agent-Friendly Exceptions

Good section. Needs stricter schema.

Every exception should include:

* stable ID.
* owner.
* owning cell.
* exact scope.
* permitted deviation.
* reason.
* risk accepted.
* expiration condition.
* proof lane.
* monitoring requirement.
* repair strategy.
* forbidden widening.
* audit behavior.
* allowed terms or paths.
* responsible human approver.

Also distinguish:

* **Product exception**: real domain language uses words like “legacy” or “temporary.”
* **Migration exception**: temporary path during planned refactor.
* **Vendor exception**: external API requires unusual pattern.
* **Security exception**: must be rare, separately approved.
* **Generated exception**: generator emits known undesirable pattern.

HTTP boundary errors should use machine-readable structures. RFC 9457 defines Problem Details for HTTP APIs, which supports the paper’s push toward standard, parseable error responses for agents and clients. OpenTelemetry exception attributes also support consistent error capture across traces/logs. ([RFC Editor][12])

### X. The jankurai Audit

This should move earlier or become a full main section with subsections. It is the enforcement layer of the entire paper.

jankurai should be framed as:

> jankurai is the conformance test for agent-native engineering.

Required outputs:

* `repo-score.json`
* `repo-score.md`
* SARIF for code scanning integrations
* PR annotations
* `agent_fix_queue`
* trend report
* centerline drift report
* exception debt report
* generated-zone mutation report
* proof-lane coverage report
* visual QA coverage report
* security posture report

Required exit codes:

| Exit code | Meaning                 |
| --------: | ----------------------- |
|         0 | pass                    |
|         1 | score below floor       |
|         2 | hard rule violation     |
|         3 | generated-zone mutation |
|         4 | missing proof lane      |
|         5 | security gate failure   |
|         6 | expired exception       |
|         7 | invalid audit config    |

Add “centerline drift”:

> Each repo has a target centerline: ownership clarity, proof routing, contract integrity, security posture, visual QA, exception hygiene, generated-zone integrity, context economy, and repair evidence. jankurai measures drift from that centerline over time.

That gives the paper a powerful governance concept.

### X.A Vibe-Coding Hard Rules

The banned-term list is useful, but it risks feeling simplistic if not presented carefully. The paper should say:

> The terms are not forbidden because words are magic. They are forbidden because unmanaged temporary language is a reliable proxy for architectural debt unless scoped, owned, and expiring.

Also add categories:

* dead-path markers: `unused`, `stale`, `dead code`
* time-debt markers: `temporary`, `cleanup later`, `remove later`
* uncertainty markers: `best effort`, `probably`, `maybe`
* compatibility debt: `shim`, `compat`, `backcompat`
* fake implementation: `stub`, `placeholder`, `dummy`
* review bypass: `hack`, `todo`, `fixme`, `xxx`
* unowned lifecycle: `legacy`, `deprecated`, `obsolete`, misspellings included

The allowlist must be machine-readable and expiring.

### XI. Migration, Versioning, and Future Research

Good section, but it needs a conformance maturity model.

Add:

| Level | Name            | Meaning                                                                 |
| ----: | --------------- | ----------------------------------------------------------------------- |
|     0 | Observed        | jankurai runs but does not block                                       |
|     1 | Routed          | owner/test maps and proof lanes exist                                   |
|     2 | Guarded         | generated zones, security scans, exceptions enforced                    |
|     3 | Agent-ready     | CI emits repair queues and agents can safely patch                      |
|     4 | Autonomous-safe | high-confidence changes can be agent-authored with bounded human review |
|     5 | Certified       | external audit, reproducible evidence, version-pinned standard          |

This is what lets companies adopt without pretending they can rewrite everything instantly.

### XII. Conclusion

The conclusion is good but should end with the standard, not only the stack.

End with something like:

> The future standard is not a language preference. It is a merge discipline. Repositories that cannot route, prove, audit, and repair generated code will drown in plausible output. Repositories that become verification interfaces will compound. The purpose of this standard is to make that second path explicit, enforceable, and fast.

## What the jankurai section must add

The paper should make jankurai mandatory for every coding project using the standard.

Add this language:

> A repository is not agent-native because it has an AI instruction file. It is agent-native only if jankurai can score it, route repair work, detect drift, and block unsafe merges.

jankurai should inspect:

* repo layout.
* ownership cells.
* allowed and denied dependency edges.
* generated-zone mutations.
* contract generation drift.
* proof-lane existence and runtime.
* security scans.
* visual QA coverage.
* test routing.
* dead-language markers.
* exception records.
* file/function size.
* duplicate logic.
* direct DB access from wrong layer.
* Python product-truth leakage.
* handwritten DTOs.
* broad catch blocks.
* weak names like `utils`, `helpers`, `common`, `misc`.
* missing docs near critical ownership cells.
* agent instruction conflicts.
* stale repair receipts.
* expired exceptions.
* missing CI artifacts.

The audit should produce a prioritized `agent_fix_queue` like:

```json
{
  "priority": 1,
  "path": "apps/api/src/routes/payments.rs",
  "finding": "route performs object access without authz proof lane",
  "owner_cell": "apps/api",
  "required_repair": "add application-layer authorization check and integration test",
  "proof_lane": "just test-authz",
  "blocking": true
}
```

That is the kind of detail that turns the paper into a standard.

## Pixel QA section to add

Add a dedicated section titled something like **Automated UI, Pixel, and Step QA**.

Core argument:

> UI correctness is not only behavior. It is rendered state. Agentic UI coding needs visual proof because generated code frequently changes layout, spacing, overflow, stacking, focus behavior, and responsive states while preserving nominal functionality.

Required layers:

| Layer                  | What it catches                                 | Tooling pattern                  |
| ---------------------- | ----------------------------------------------- | -------------------------------- |
| Component visual tests | Button, modal, form, card, table regressions    | Storybook/Playwright screenshots |
| Route screenshots      | Layout, spacing, dark mode, responsive breakage | Playwright `toHaveScreenshot()`  |
| Step QA                | Real user journey completion                    | Playwright traces/videos         |
| Accessibility QA       | keyboard, ARIA, contrast, focus                 | axe/Playwright integration       |
| Console/network QA     | JS errors, failed requests                      | Playwright event capture         |
| Design-token QA        | spacing/color/font drift                        | token snapshot diff              |
| Baseline governance    | prevents snapshot laundering                    | human-approved baseline PR       |

Mitigations for visual flake:

* pin browser versions.
* pin fonts.
* freeze clock.
* disable animations.
* seed data.
* use deterministic accounts.
* mask dynamic regions.
* set viewport and DPR explicitly.
* test both light/dark only where product-critical.
* do not run every viewport on every PR.
* run risk-weighted visual suites.
* require baseline diffs as PR artifacts.
* prohibit automatic baseline updates by agents.

The paper should say visual testing is not a replacement for functional tests. It is a separate proof class.

## Future CI and testing section to add

The future of CI is not “more tests.” It is **risk-routed proof generation**.

Good outcomes of AI-generated tests:

* more edge cases.
* faster regression capture.
* better property tests.
* richer visual coverage.
* generated failure repros.
* automatic contract-drift tests.
* repair receipts tied to prior incidents.

Bad outcomes:

* test explosion.
* flaky suites.
* tests that only verify the current implementation.
* slow CI.
* impossible review burden.
* snapshot churn.
* agents patching tests instead of code.
* false confidence from line coverage.
* compute cost blowups.
* local green / production red failures.

The standard should require these CI metrics:

| Metric                        | Why it matters                                 |
| ----------------------------- | ---------------------------------------------- |
| proof-lane runtime            | keeps agents from defaulting to full slow runs |
| failure localization time     | measures repairability                         |
| flake rate                    | prevents agents from ignoring CI               |
| mutation score                | checks whether tests actually catch faults     |
| contract drift count          | measures boundary discipline                   |
| visual diff approval rate     | detects baseline churn                         |
| exception debt                | measures unmanaged deviation                   |
| generated-zone mutation count | detects agent misuse                           |
| security gate bypasses        | detects policy erosion                         |
| repair receipt closure        | measures learning loop                         |

## LaTeX, layout, and formatting upgrades

The current draft has the bones of a serious paper, but the production quality needs to improve.

Critical fixes:

* Fix visible encoding/hyphenation artifacts in the extracted text, such as split words with odd characters. This makes the paper look unpolished and hurts search/copy quality.
* Use `microtype` for better justification and fewer ugly line breaks.
* Use `booktabs` for all tables; avoid dense grid-like tables.
* Move very large tables to appendices or use `table*` across both columns.
* Add short “standard rule” boxes using MUST/SHOULD/MAY language.
* Add a definitions box early.
* Add a conformance-level table.
* Add a threat-model table.
* Add a jankurai output example.
* Add a CI pipeline diagram.
* Add a pixel QA matrix.
* Use consistent capitalization in references: protect names like `{GitHub}`, `{TypeScript}`, `{PostgreSQL}`, `{OpenTelemetry}`, `{OpenAI}`, `{DORA}` in BibTeX.
* Clean reference metadata. The current reference list has inconsistent casing and access-date style.
* Use `cleveref` for cross-references.
* Use `siunitx` for scores and percentages.
* Use `tabularx` or `array` for controlled column widths.
* Use `enumitem` to reduce list spacing.
* Use `tcolorbox` for normative callouts and schemas.
* Use `fvextra` or `listings` for code/config blocks; avoid huge raw monospace blocks that dominate pages.
* Prefer one highly readable serif family plus a clean monospaced font. For pdfLaTeX, `newtxtext,newtxmath` with `inconsolata` is safe. For LuaLaTeX, `Libertinus` plus `Inconsolata` or `IBM Plex Mono` gives a more modern research look.
* Add a “Reproducibility and Data Sources” appendix for scoring data, plot data, and jankurai sample outputs.
* Add a “Normative Requirements” appendix that can become the standard spec.
