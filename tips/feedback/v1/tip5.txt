## Executive judgment

The paper already has a strong core: it correctly identifies that AI coding shifts the bottleneck from **code production** to **verification, routing, rejection, repair, and audit**. The current manuscript proposes a stack rubric, stack ranking, repository architecture, agent-control files, proof lanes, exceptions, and a jankurai audit layer, which is the right direction for a standard-launch paper. 

The hard truth: **it does not yet read like the launch of a new engineering standard.** It reads like a sharp manifesto plus several useful tables. To become a landmark, it needs to become three things at once:

1. **A persuasive historical argument**: how code evolved from human memory compression to agent-verifiable systems.
2. **A normative standard**: clear MUST / SHOULD / MAY requirements that teams can adopt.
3. **An empirical operating model**: public data, score calibration, failure taxonomy, CI artifacts, pixel QA, and jankurai enforcement.

Right now, the paper’s strongest idea is “repositories must become verification interfaces for agents.” That should be the organizing sentence of the whole work. Everything else should serve that.

---

## The biggest missing upgrade: define the standard before defending the stack

The paper jumps too quickly from “AI changes coding” to “here is the winning stack.” That is emotionally forceful but structurally risky. Skeptical readers will ask: **Why this stack? Why now? Why this rubric? Why not Go? Why not .NET? Why not Python plus stronger CI?**

The paper should first define the **Agent-Native Engineering Standard** as a set of requirements independent of any particular language:

**An agent-native codebase must:**

* route every change to an owned layer;
* reject invalid code through type, contract, database, security, and test gates;
* produce machine-readable repair evidence;
* minimize agent context waste;
* forbid ambiguous generated edits;
* isolate high-risk tools, secrets, browser access, and production credentials;
* require visual, behavioral, and accessibility proof for user-facing software;
* track repository drift through jankurai on every project.

Only after that should the paper say: **given these requirements, the best default stack is Rust core, TypeScript/React/Vite product surface, PostgreSQL durable truth, generated contracts, and bounded Python for AI/data work.**

That ordering makes the paper harder to dismiss as “Rust maximalism.” It turns the stack into a consequence of the standard.

## Abstract feedback

The abstract is clear, punchy, and correctly frames the bottleneck as rejection and repair rather than typing. It also correctly lists the contribution: rubric, ranking, architecture, standard, exception pattern, test model, and audit rubric. 

But it is too compressed and too declarative. It needs to introduce **jankurai** as a mandatory enforcement mechanism, not as one item in a contribution list. It also needs to name the actual problem class: “vibe coding” should be defined as **code merged without structural proof**, not merely “AI-assisted coding.”

A stronger abstract shape would be:

> AI-assisted programming has made code generation cheap, but it has not made software correctness cheap. The new failure mode is false plausibility: code that compiles, looks idiomatic, and satisfies a prompt while violating security, contracts, data invariants, user flows, or maintainability constraints. This paper proposes an agent-native engineering standard in which repositories become verification interfaces for coding agents. The standard defines ownership cells, generated contracts, proof lanes, visual QA, exception records, security gates, and a mandatory jankurai audit that measures repository drift from an agent-verifiable center. We apply a 100-point stack rubric and argue for a narrow default stack: Rust core, TypeScript/React/Vite product surface, PostgreSQL durable truth, generated contracts, and bounded Python for AI/data work. The result is not human-free engineering; it is human-directed, agent-executed engineering with machine-checkable proof.

That abstract makes the paper feel less like an opinion and more like a standard proposal.

## Introduction feedback

The introduction has the right claim: the old repository model was built around human memory, human review, and informal taste. But it needs a stronger journey.

The first body paragraph should not spend energy defending the rhetoric. It should start with the technical failure mode:

> The first age of programming optimized for expressing instructions to machines. The second optimized for expressing intent to humans. The agent era must optimize for expressing constraints to automated maintainers.

That would immediately set up the evolution story.

The introduction also needs stronger data anchoring. DORA’s 2025 report says AI acts as an amplifier of organizational strengths and weaknesses, not an automatic productivity guarantee. ([Dora][1]) Stack Overflow’s 2025 survey found that 84% of respondents use or plan to use AI tools, while the top frustration was “almost right” AI output, cited by 66% of developers; 45.2% also cited more time debugging AI-generated code. ([Stack Overflow][2]) That is the introduction’s empirical backbone: **AI is already mainstream, but the failure mode is false confidence.**

The introduction should then give the reader the full roadmap:

1. Languages evolved by compressing human bottlenecks.
2. AI removes syntax familiarity as the critical bottleneck.
3. The new scarce resource is trustworthy merge.
4. Trustworthy merge requires an agent-native standard.
5. The standard requires a default stack and mandatory audits.
6. jankurai is the compliance layer.

The current introduction makes these claims, but it does not yet make the reader feel the inevitability of the standard.

## Section II: language evolution and “compression technology”

This is one of the paper’s best conceptual sections. Keep it, but deepen it.

The section should become the emotional and intellectual journey of the paper. The current table is useful, but the prose should be expanded into a narrative:

* **Machine code / assembly** compressed almost nothing except symbolic names.
* **Fortran / COBOL** compressed domain expression.
* **C** compressed portable systems control.
* **Java / .NET** compressed memory management, enterprise libraries, and deployment shape.
* **Python / JavaScript** compressed iteration speed.
* **TypeScript** compressed large-team JavaScript maintenance.
* **Rust** compresses memory safety, concurrency discipline, and invalid-state rejection.
* **Agent-native repositories** compress human review and repair routing into machine-readable proof surfaces.

The paper should explicitly say:

> The agent era does not end programming-language choice. It changes what languages are chosen for. A language is no longer primarily selected because humans can write it fastest. It is selected because agents can be constrained, rejected, repaired, and audited through it fastest.

That is a landmark-level sentence.

The section should also avoid implying that technically elegant languages “died.” For example, Julia is very much alive in scientific computing, and its original promise was serious: easy high-level numerical programming with performance and multiple dispatch. ([SIAM][3]) The sharper argument is that Julia did not become a universal company-product default. TIOBE’s April 2026 index, while not a measure of “best language,” places Julia at #32 and TypeScript at #34, with Python, C, C++, Java, C#, and JavaScript occupying the top six, showing how adoption metrics often diverge from technical elegance. ([TIOBE][4])

So the section should frame “hype language failure” as **default-stack failure**, not death. A better subsection name would be:

> Why Excellent Languages Fail to Become Default Stacks

Then classify failures:

| Failure type                        | What happens                                                                                    | Examples to discuss        |
| ----------------------------------- | ----------------------------------------------------------------------------------------------- | -------------------------- |
| Inner-loop win, outer-loop weakness | Language solves coding elegance but not deployment, hiring, security, package maturity, and ops | Julia, Nim, Crystal        |
| Reliability win, interop loss       | Language improves correctness but makes ecosystem integration harder                            | Elm, Reason/OCaml frontend |
| Expert abstraction trap             | Powerful abstractions become opaque to general agents and broad teams                           | Haskell, Scala, Clojure    |
| Platform niche lock                 | Great for a workload but not universal product architecture                                     | Elixir/Phoenix, F#, R      |
| Ecosystem gravity loss              | Technically good but cannot beat entrenched tooling and corpus scale                            | D, OCaml frontend variants |

This would satisfy the “hype projects that died and why” goal while staying accurate and credible.

## Section III: harsh AI reality

This section is directionally right, but it needs sharper evidence and a more direct thesis.

The strongest version:

> AI makes bad architecture compound faster. Weak tests become faster false greens. Weak security becomes faster vulnerability production. Weak ownership becomes faster cross-layer drift. Weak documentation becomes faster hallucinated conventions.

DORA supports the “AI amplifies the system” framing. ([Dora][1]) Veracode reports that AI-generated code introduced risky security flaws in 45% of tested tasks and that larger/newer models did not automatically improve security. ([Veracode][5]) GitGuardian reports 28,649,024 new secrets detected in public GitHub commits in 2025, 34% year-over-year growth, and 81% year-over-year growth in AI-service secrets. ([GitGuardian][6])

The paper should include those data points in the main body, not just references. They prove the central claim: **speed without gates is not progress; it is accelerated defect creation.**

The section also needs to define “second place.” Right now “second place is more expensive” is rhetically strong but underspecified. It should mean:

* second-best language for rejection speed;
* second-best contract discipline;
* second-best visual QA;
* second-best secret hygiene;
* second-best generated-zone discipline;
* second-best CI routing.

The point is not that every team must rewrite tomorrow. The point is that every deviation from the agent-native center must be visible, owned, justified, and temporary.

## Section IV: the 100-point stack rubric

The rubric is necessary, but it currently looks asserted rather than measured. For a landmark paper, that is the largest credibility gap.

The paper must explain:

1. **How each category is scored.**
2. **What evidence earns full credit.**
3. **What evidence caps the score.**
4. **How weights were chosen.**
5. **How sensitive the ranking is to weight changes.**
6. **How future versions will recalibrate the rubric.**

Add a formal equation:

> ANSS(stack) = Σ weightᵢ × scoreᵢ(stack), where each scoreᵢ is normalized from 0 to 1 using observable repository, ecosystem, security, deployment, and agent-repair criteria.

Then add hard caps. Example:

| Hard cap                                        | Maximum score |
| ----------------------------------------------- | ------------: |
| No secret scanning or push protection           |            70 |
| No generated contract boundary                  |            75 |
| No deterministic CI proof lanes                 |            70 |
| Product truth spread across UI/Python/scripts   |            65 |
| No visual/e2e proof for user-facing product     |            80 |
| No dependency/SBOM/provenance policy            |            75 |
| No machine-readable owner/test map              |            80 |
| Mutable generated zones                         |            75 |
| Direct production DB access from unowned layers |            65 |

This makes the rubric enforceable.

Also add **evidence levels**:

* **Level 0**: claimed in README.
* **Level 1**: manually followed convention.
* **Level 2**: lint/audit detects violation.
* **Level 3**: CI blocks violation.
* **Level 4**: violation is structurally impossible or generated away.

A standard should reward Level 3 and Level 4, not “we usually review that.”

## Section V: stack ranking

The ranking is the most controversial part. It must be more rigorous.

The current scores are close: 94, 90, 89, 88, 87. That is a narrow range, but the paper treats the winner as obvious. That mismatch will invite criticism. The paper should include:

* a per-category score table for each stack;
* a sensitivity analysis varying weights ±20%;
* an uncertainty band;
* workload-specific overrides;
* a clear distinction between **default standard** and **valid exception**.

GitHub Octoverse gives useful support for TypeScript’s importance: by GitHub contributor counts, TypeScript became the most-used language on GitHub in August 2025, surpassing Python by about 42,000 contributors, with GitHub linking the rise to typed JavaScript and AI-assisted development benefiting from stricter type systems. ([The GitHub Blog][7]) But Stack Overflow’s 2025 survey still shows JavaScript, HTML/CSS, SQL, Python, Bash/Shell, and TypeScript as the most used languages among respondents, meaning the paper should not rely on one ecosystem metric alone. ([Stack Overflow][8])

The Rust argument is strong, but it needs nuance. NSA and CISA explicitly encourage adopting memory-safe languages to improve software security and reduce memory-related vulnerabilities. ([NSA][9]) But TIOBE’s April 2026 commentary also notes Rust’s broader mainstream adoption remains challenged by difficulty for non-experts. ([TIOBE][4]) The paper’s answer should be:

> Rust’s human learning curve mattered more when humans typed most core code. In an agent-native workflow, the decisive question is not whether every developer writes Rust quickly from memory. It is whether the repository can reject unsafe ownership, invalid state, concurrency mistakes, and boundary drift before production.

That is the best defense of Rust as the core.

Also: be careful with “human knowledge of code is not critical path.” That is too broad. The better claim is:

> Human syntax fluency is no longer the critical path. Human judgment about product intent, threat models, invariants, and exceptions remains critical.

That keeps the paper credible.

## Section VI: winner-only architecture

This section is good but underdeveloped. It needs examples.

The filesystem-as-routing-table idea is excellent. The paper should show “wrong patch vs right patch” examples:

| Change request                  | Wrong agent behavior                           | Correct routing                                                         |
| ------------------------------- | ---------------------------------------------- | ----------------------------------------------------------------------- |
| “Add discount eligibility”      | Add conditional in React checkout component    | Rust domain invariant + application workflow + DB constraint if durable |
| “Add admin-only export”         | Hide button in UI                              | Rust authz policy + API middleware + Playwright negative path           |
| “Add model summarization”       | Python service writes production rows directly | Python returns typed result; Rust application owns persistence          |
| “Add field to customer profile” | Hand-edit DTOs in UI and API                   | Update schema/contract source; regenerate clients; run drift lane       |
| “Fix flaky loading state”       | Add arbitrary sleeps                           | Use user-facing locators, web-first assertions, deterministic fixtures  |

This would make the standard operational.

Also add dependency direction rules:

* UI imports generated clients only.
* API edge calls application layer.
* Application owns transactions.
* Domain owns invariants and has no I/O.
* Adapters own external systems and must not encode domain rules.
* Python AI service owns inference/evals only.
* DB owns durable constraints, indexes, migrations, and row policies where needed.

This section should be the first place where the reader can imagine building a repo from the paper.

## Section VII: agent-first repository design

This section needs to become much more specific and tool-aware.

The general principle is correct: instructions matter, but deterministic control surfaces matter more. SWE-agent’s research supports the idea that agent-computer interface design affects agent performance. ([arXiv][10]) ContextBench also shows that coding agents still struggle with context retrieval precision and useful context use across repositories. ([arXiv][11])

The paper should split this section into two parts.

### General agent-first rules that work across all AI coding tools

Every project should have:

| Artifact                          | Purpose                                       | Hard rule                                        |
| --------------------------------- | --------------------------------------------- | ------------------------------------------------ |
| `AGENTS.md`                       | Short root router                             | No long tutorials; point to maps and proof lanes |
| `owner-map.json`                  | Path ownership and dependency direction       | CI fails on forbidden edges                      |
| `test-map.json`                   | Changed path → proof lane                     | Agents never guess validation commands           |
| `proof-lanes.toml`                | Fast, contract, DB, web, security, full lanes | Each command deterministic and CI-runnable       |
| `generated-zones.toml`            | Generated outputs and sources                 | Generated files read-only unless source changes  |
| `jankurai-policy.toml`           | Audit rules, caps, exceptions                 | Every repo pins standard version                 |
| `docs/exceptions/*.md`            | Temporary deviations                          | Must have owner, scope, expiry, proof            |
| `repair/*.md`                     | Known repair recipes                          | Must be updated after major incidents            |
| `visual-baselines/` or equivalent | Approved UI baselines                         | Agent cannot self-approve baseline changes       |

### Tool-specific strengths to exploit

| Tool class                          | Strength to call out                                                                                                                                                                                 | Risk to control                                                                                                                                                                                                     | Standard requirement                                                                                    |
| ----------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------- |
| **Codex-style AGENTS.md workflows** | Codex reads layered `AGENTS.md` guidance before work, with project and nested instruction discovery. ([OpenAI Developers][12])                                                                       | Conflicting or oversized instructions.                                                                                                                                                                              | Root file must be short; nested files must mirror canonical policy.                                     |
| **Cursor**                          | Cursor supports persistent rules, and Bugbot can review PR diffs, use PR comments as context, apply team/repo/project rules, and propose fixes. ([Cursor][13])                                       | Learned rules drift into private tribal knowledge.                                                                                                                                                                  | Cursor rules must be generated from canonical jankurai policy; no extra permissions.                   |
| **Google Antigravity**              | Antigravity emphasizes task-level agent orchestration across editor, terminal, and browser, and uses artifacts such as plans, screenshots, and recordings for review. ([Google Developers Blog][14]) | Browser/tool access expands prompt-injection and destructive-action risk. Antigravity’s codelab explicitly calls browser access both powerful and vulnerable and recommends URL allowlists. ([Google Codelabs][15]) | Browser allowlist, command approval policy, artifact capture, and no self-approval of visual baselines. |
| **Claude Code**                     | Auto memory can preserve build commands, debugging insights, architecture notes, and workflow habits across sessions. ([Claude][16])                                                                 | Stale or conflicting memory can silently bias future work; docs note vague/conflicting instructions reduce compliance. ([Claude][16])                                                                               | Memory must be auditable, plain-text, bounded, and periodically linted.                                 |
| **GitHub Copilot**                  | Repository-wide, path-specific, and agent instructions can guide build/test/validation behavior. ([GitHub Docs][17])                                                                                 | Instruction dialect fragmentation.                                                                                                                                                                                  | `.github` instructions, `AGENTS.md`, and local rules must mirror one canonical standard.                |

This would make the paper immediately practical for real teams.

## Section VIII: testing and automated QA

This is currently too thin for the ambition of the paper. It needs to become one of the most important sections.

The paper should reject the old “test pyramid” as insufficient and introduce a **proof lattice**:

| Proof type              | What it catches                                  | Required lane |
| ----------------------- | ------------------------------------------------ | ------------- |
| Type/compile proof      | syntax, ownership, type errors, missing variants | fast          |
| Unit proof              | pure local behavior                              | fast          |
| Property/fuzz proof     | invariant classes and edge cases                 | fast/full     |
| Contract proof          | API/schema drift                                 | contract      |
| DB proof                | constraints, migrations, RLS, indexes            | db            |
| Integration proof       | adapters, external APIs, transactions            | integration   |
| E2E behavioral proof    | critical user journeys                           | web/e2e       |
| Pixel proof             | visual regressions                               | visual        |
| Accessibility proof     | ARIA/semantic regressions                        | a11y          |
| Security proof          | secrets, dependencies, SAST, policy              | security      |
| Observability proof     | traces, correlation IDs, error attributes        | audit         |
| Production repair proof | incident replay, regression capture              | release/full  |

The future of CI is not “more tests.” It is **routed proof**.

### The test explosion: good and bad

Good:

* Agents can generate broad regression coverage faster than humans.
* Every bug fix can become a regression test.
* Property tests and fuzzing can expand edge-case discovery.
* Visual and accessibility snapshots can catch failures invisible to unit tests.
* Production incidents can become replay tests.

Bad:

* Agents can generate shallow tests that only prove their own implementation.
* Snapshot baselines can become rubber stamps.
* CI cost and runtime can explode.
* Flaky tests can train teams to ignore red builds.
* E2E tests can become brittle if they use CSS selectors, sleeps, or implementation details.
* Generated tests can encode wrong product behavior as permanent truth.

Playwright’s official guidance supports resilient locators based on user-facing attributes and web-first assertions instead of brittle DOM/CSS assumptions and manual polling. ([Playwright][18]) That should be a core rule of the paper.

## Required new section: automated pixel QA and step-level app validation

This needs a full standalone section. User-facing apps fail in ways that code review, unit tests, and even normal e2e tests miss: spacing, overlap, broken responsive layouts, invisible buttons, wrong theme, clipped modals, z-index bugs, disabled states, loading skeletons, and “looks fine in code, unusable in browser” failures.

The paper should define **Pixel QA** as:

> A CI-controlled visual proof lane that captures approved user journeys step by step, compares rendered states against baselines, records browser traces and screenshots, and prevents agents from approving visual changes without human or policy approval.

Minimum standard:

| Requirement               | Rule                                                                                                                           |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| Step screenshots          | Capture every critical transition: initial, loading, error, success, empty, permission-denied, mobile, dark mode if supported. |
| Semantic assertions first | Assert visible behavior with user-facing locators before pixel comparison.                                                     |
| Visual baseline second    | Use screenshot comparison only after behavior is stable.                                                                       |
| Deterministic data        | Freeze clocks, seed data, mock external services, disable animations, mask dynamic regions.                                    |
| Browser matrix            | At minimum Chromium desktop + one mobile viewport; expand by product risk.                                                     |
| Accessibility snapshot    | Include ARIA/role structure for key screens.                                                                                   |
| Trace retention           | Store screenshot, video, network, console, and DOM trace for failed runs.                                                      |
| Baseline governance       | Agents may propose baseline updates but cannot approve them alone.                                                             |
| Diff classification       | Diffs must be classified as intentional design change, product bug, test flake, or baseline drift.                             |
| Pixel debt tracking       | Uncovered critical journeys count against jankurai score.                                                                     |

Playwright supports screenshot comparison through `toHaveScreenshot()`, warns that rendering can vary by OS/settings/hardware, and allows options such as `maxDiffPixels` and stylesheet masking for determinism. ([Playwright][19]) Playwright also supports ARIA snapshots, but its docs warn against over-reliance and large snapshots that are hard to interpret. ([Playwright][20]) Percy’s workflow captures and compares screenshots in CI and routes visual changes through review and approval. ([BrowserStack][21]) Applitools adds DOM/CSS root-cause analysis for visual mismatches and perceptual visual comparison rather than raw pixel-only matching. ([Applitools][22])

The paper should also say what can go wrong:

| Pixel QA risk                         | Mitigation                                                                |
| ------------------------------------- | ------------------------------------------------------------------------- |
| False positives from fonts/rendering  | Run in pinned containers; same OS/browser versions; deterministic fonts.  |
| False negatives from loose thresholds | Keep thresholds low for critical UI; classify allowed regions explicitly. |
| Baseline rubber-stamping              | Require human/design approval or policy-reviewed artifact.                |
| Dynamic content noise                 | Mask ads, timestamps, avatars, randomized IDs.                            |
| Test flake                            | Use web-first assertions before screenshots; remove sleeps.               |
| Visual-only confidence                | Pair pixel tests with behavioral, accessibility, and API assertions.      |
| Cost explosion                        | Run full visual matrix nightly; changed-route visual lane per PR.         |

This section is essential because agent-built UI will otherwise look plausible while being wrong.

## Section IX: agent-friendly exceptions

This section is strong and should stay. It needs concrete examples.

Add examples for Rust, TypeScript, and API boundary errors. The section should distinguish:

* **domain errors**: typed, stable, exhaustive;
* **application errors**: authz, idempotency, transaction failure;
* **adapter errors**: provider/network/database failure;
* **boundary errors**: RFC 9457-style problem response;
* **agent repair hints**: safe fixes and forbidden fixes.

A good standard rule:

> Every error that can reach a log, API boundary, test failure, queue retry, or trace must include a stable code, owner cell, correlation ID, safe message, protected invariant, and repair documentation link.

The key improvement: exceptions should not just help users. They should help **future agents repair safely**.

Add this required shape:

```json
{
  "error_code": "BILLING_DISCOUNT_INELIGIBLE",
  "owner_cell": "crates/domain",
  "invariant": "A discount cannot be applied after invoice finalization.",
  "safe_user_message": "This invoice can no longer be discounted.",
  "repair_doc": "docs/repair/billing-discounts.md",
  "allowed_repairs": ["change eligibility rule in domain", "add migration if durable state changes"],
  "forbidden_repairs": ["bypass in UI", "override in adapter", "direct DB update"]
}
```

That is much more agent-native than ordinary exception prose.

## Section X: jankurai audit

This should become the centerpiece of the paper.

Right now jankurai appears as an audit mechanism. It needs to be framed as the **mandatory compliance layer** for every coding project. Without jankurai, the paper is just advice. With jankurai, it becomes a standard.

### Reframe jankurai as drift control

The key concept should be:

> jankurai measures how far a repository has drifted from the agent-native center.

The score should not only be a single number. It should be a vector:

```json
{
  "standard": "jankurai-agent-native-0.1.0",
  "repo_score": 82,
  "center_distance": 18,
  "caps": ["NO_VISUAL_QA", "UNOWNED_DB_ACCESS"],
  "dimensions": {
    "ownership": 0.81,
    "contracts": 0.74,
    "proof_lanes": 0.69,
    "security": 0.88,
    "data_truth": 0.72,
    "observability": 0.63,
    "context_economy": 0.79,
    "pixel_qa": 0.31,
    "polyglot_hygiene": 0.86
  },
  "agent_fix_queue": [
    {
      "rank": 1,
      "finding": "No Playwright visual proof for checkout route",
      "owner_cell": "apps/web",
      "proof_lane": "web.visual",
      "suggested_fix": "Add checkout happy/error/permission-denied screenshots"
    }
  ]
}
```

### Required jankurai outputs

Every CI run should produce:

* `jankurai.json` for agents;
* `jankurai.md` for reviewers;
* `jankurai.sarif` for code scanning systems;
* `jankurai-trend.csv` for longitudinal drift;
* `agent_fix_queue.json` for automated repair routing;
* `exceptions-expiring.md` for governance.

### Hard gates

The paper should define mandatory blockers:

| Finding                                                        | Severity     |
| -------------------------------------------------------------- | ------------ |
| Secret in repo, logs, fixtures, artifacts, or agent transcript | Block        |
| Generated zone edited without source/generator change          | Block        |
| Direct DB access from UI/Python/unowned layer                  | Block        |
| Authz change without negative test                             | Block        |
| Contract source changed without regenerated clients/stubs      | Block        |
| Payment/billing/security invariant changed without domain test | Block        |
| Browser automation route changed without e2e proof             | Block        |
| Visual baseline updated by agent without approval              | Block        |
| New production dependency without rationale/SCA/license scan   | Block        |
| New TODO/fallback/stub in runtime code without exception       | Block or cap |

GitHub push protection is relevant here because it blocks detected hardcoded credentials before they reach a repository rather than merely alerting after exposure. ([GitHub Docs][23]) OSV-Scanner is relevant because it connects project dependencies to known vulnerabilities and can run in CI. ([Google GitHub][24]) SLSA is relevant because it provides a framework of controls for preventing tampering and improving artifact integrity across the software supply chain. ([SLSA][25])

## Future CI and testing section to add

The paper should have a dedicated section called something like:

> From CI to Continuous Proof

Core argument:

> CI used to answer “does this build pass?” Agent-native CI must answer “which proof obligation did this patch create, did the repository run the smallest credible proof, and what repair evidence was produced?”

Required future CI features:

| CI capability         | Agent-native requirement                                                 |
| --------------------- | ------------------------------------------------------------------------ |
| Change classification | Diff maps to ownership cell and proof lane                               |
| Targeted proof        | Run smallest credible lane first, not random full suite                  |
| Full release proof    | Full lane before merge/release                                           |
| Security proof        | Secrets, SCA, SAST, SBOM, provenance                                     |
| Contract proof        | Regenerate and diff API clients/schemas                                  |
| DB proof              | Apply migrations, verify constraints/RLS/drift                           |
| Visual proof          | Screenshot/trace critical user journeys                                  |
| Test quality proof    | Detect shallow generated tests and missing assertions                    |
| Flake accounting      | Flaky tests reduce repo score                                            |
| Repair receipts       | CI emits machine-readable failure causes                                 |
| Cost governance       | Track minutes, retries, artifact size, and agent token cost              |
| Human approval gates  | Required for destructive actions, baseline approvals, exception widening |

The good future is massive coverage. The bad future is massive meaningless coverage. The standard must separate the two.

## Layout, formatting, and LaTeX upgrades

The current body is dense. It has too many large tables fighting the two-column format. For a landmark standard paper, readability matters.

### Structural layout

Use one of two formats:

1. **Research report / standard format**: single-column, 11pt, wider margins, polished figures, spec callouts.
2. **Academic format**: two-column only after the draft is tightened, with large tables moved to full-width floats or appendices.

For this paper, I strongly recommend **single-column report style** for the launch version. The paper is trying to define a standard, not squeeze into a conference page limit.

### LaTeX recommendations

Use:

```latex
\usepackage{microtype}
\usepackage{booktabs}
\usepackage{tabularx}
\usepackage{longtable}
\usepackage{threeparttable}
\usepackage{siunitx}
\usepackage{enumitem}
\usepackage{xcolor}
\usepackage{tcolorbox}
\usepackage{cleveref}
\usepackage{hyperref}
```

For code/config snippets:

```latex
\usepackage{listings}
```

or `minted` if shell-escape is acceptable.

### Table improvements

Current tables are information-rich but visually heavy. Convert them into:

* short main-body tables;
* expanded appendix tables;
* scorecards with icons or status marks;
* full-width tables for the rubric;
* `booktabs` instead of dense gridlines;
* `tabularx` for wrapped text;
* `siunitx` for aligned numeric scores.

### Figure improvements

The ANSS bar plot should not rely only on a cropped x-axis. If the x-axis starts at 85, include either:

* a full-scale inset; or
* uncertainty bars; or
* a sensitivity plot.

Otherwise skeptical readers will say the figure exaggerates a small difference.

### Typography

Use a premium but sober stack:

* Body: Libertinus Serif, STIX Two Text, or TeX Gyre Termes.
* Sans: Source Sans 3 or IBM Plex Sans.
* Mono: JetBrains Mono, IBM Plex Mono, or Inconsolata.
* Math: Libertinus Math or STIX Two Math.

Avoid cramped all-caps section headings. The paper needs authority, not shouting. Use strong section names and clean hierarchy.

### Standard callout boxes

Add visually distinct boxes:

```latex
\newtcolorbox{standardbox}{
  colback=gray!4,
  colframe=black!60,
  title=Agent-Native Requirement,
  sharp corners,
  boxrule=0.5pt
}
```

Then use boxes for:

* MUST requirements;
* banned patterns;
* proof lane requirements;
* jankurai gates;
* exception template;
* pixel QA rules.

That will make the paper feel like a serious standard.

## 30/60/90-day adoption plan to add

The paper needs to tell teams how to adopt the standard without a rewrite fantasy.

### First 30 days: visibility

* Add jankurai advisory mode.
* Add `AGENTS.md`, `owner-map.json`, `test-map.json`, `proof-lanes.toml`.
* Enable secret scanning and dependency scanning.
* Identify generated zones.
* Record current score and center distance.
* Add exception records for known deviations.

### Days 31–60: enforcement

* Turn on blocking for secrets, generated-zone mutation, missing proof lanes, and banned runtime placeholders.
* Add contract generation and drift checks.
* Add DB migration/constraint proof.
* Add Playwright critical-path tests.
* Add repair receipts for CI failures.

### Days 61–90: agent-native acceleration

* Add pixel QA for critical flows.
* Add property tests for core domain invariants.
* Move durable truth into Rust/domain/application and PostgreSQL constraints.
* Box Python into AI/data service boundaries.
* Start agent-driven remediation from `agent_fix_queue.json`.
* Require non-regression of jankurai score per merge.

This makes the standard adoptable.

## Final critical recommendations

The paper should become less of a claim and more of a **machine-checkable operating system for agentic software engineering**.

The strongest upgrades are:

1. **Put jankurai at the center.** Every repo, every CI run, every agent repair queue.
2. **Define vibe coding precisely.** It is not AI usage; it is proof-free plausibility.
3. **Make the stack ranking auditable.** Show scoring inputs, caps, sensitivity, and exceptions.
4. **Add a serious pixel QA section.** This is currently a major gap.
5. **Add future CI as continuous proof.** The test explosion must be governed, not celebrated blindly.
6. **Expand tool-specific agent design.** Antigravity artifacts, Cursor rules/review, Codex AGENTS, Claude memory, Copilot path instructions each need strengths and risks.
7. **Treat language hype carefully.** Julia and similar tools are not “dead”; they are examples of failing to become universal default stacks despite real technical merit.
8. **Use public data plots.** Stack Overflow, GitHub Octoverse, GitGuardian, Veracode, SetupBench, ContextBench, and NSA/CISA make the paper feel evidence-backed.
9. **Upgrade typography and layout.** The body should feel like a polished standard from a serious lab, not a compressed conference draft.
10. **Use normative language.** MUST, SHOULD, MAY, hard caps, exception expiry, versioned policy.

The paper’s landmark sentence should be:

> Agent-native engineering is the discipline of making repositories into verification interfaces for autonomous code generation.

Everything in the paper should be organized around proving and operationalizing that sentence.
