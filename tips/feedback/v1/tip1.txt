I reviewed the 12-page manuscript as a proposed engineering standard, not just as an essay. My direct read: the core is strong, provocative, and close to a useful standard, but it is not yet a landmark paper. Right now it is a manifesto plus a promising scaffold. To become the standard you want, it needs a stronger evidence spine, a clearer conformance model, a better introduction journey, a much deeper jankurai section, a full vibe-coding failure taxonomy, a tool-specific agent-design section, and a real treatment of UI/pixel QA and future CI. The existing manuscript already has several standardizable assets: the 100-point stack rubric, top-five stack table, winner-only architecture, repository map, proof lanes, exception pattern, and codebase audit dimensions. Those should become the operational heart of the paper. 

## 1. Overall critical assessment

The paper’s strongest claim is correct: AI coding changes the bottleneck from “can a human write code?” to “can the repo reject, localize, prove, and repair generated code?” That is the right center. DORA’s 2025 report supports the “AI as amplifier” framing: AI magnifies existing strengths and weaknesses rather than automatically fixing delivery systems, and Google’s summary specifically notes that increased change volume can expose downstream instability without automated testing, fast feedback, and strong controls. ([Dora][1])

The paper’s weakest point is that it asserts a winner stack before sufficiently proving why that stack wins under the proposed rubric. A 94 vs. 90 vs. 89 score looks precise, but the paper does not yet show how those points were assigned, who scored them, how much uncertainty exists, or which criteria are hard caps versus preferences. If this paper is meant to force a standard, the ranking must become auditable, reproducible, and falsifiable.

The second weakness is scope imbalance. The manuscript spends real energy on stack choice and repository layout, but the most important future-facing sections are still underdeveloped: vibe-coding failure modes, jankurai as mandatory CI control, pixel QA, agent-specific workflows, and the future test explosion. Those are not side topics. They are the mechanism by which the paper becomes a standard instead of an opinion piece.

The third weakness is rhetorical credibility. “Only standard” energy is useful for rallying people, but the published argument needs to earn that stance through conformance levels, adoption path, governance, and data. The better formulation is: this is the default agent-native profile; deviations are allowed only when scored, documented, tested, and time-bounded.

## 2. The paper’s goal should be sharpened

The goal should be stated as:

> Define an agent-native software engineering standard where repositories are designed as verification interfaces for AI agents: one preferred stack, one proof-routing architecture, one audit standard, one exception protocol, and one CI evidence model.

That is stronger than “AI writes code faster.” The core should be “maximum speed through minimum ambiguity.”

The paper should explicitly separate five ideas:

| Layer            | What the paper should standardize                                                                 | Why it matters                                                          |
| ---------------- | ------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| Stack            | Rust core, TypeScript/React/Vite surface, PostgreSQL truth, generated contracts, bounded Python   | Gives agents strong compile, contract, and data-truth feedback loops    |
| Repository shape | Ownership cells, dependency direction, generated zones, proof lanes                               | Makes the next edit routable instead of guess-based                     |
| Audit            | jankurai score, hard caps, findings, repair queue                                                | Turns standards into CI enforcement                                     |
| Agent interface  | AGENTS.md, CLAUDE.md, Cursor rules, Copilot instructions, Antigravity artifacts, tool mirrors     | Prevents each AI tool from receiving contradictory project instructions |
| QA/CI            | Unit, property, integration, contract, DB, security, e2e, visual, accessibility, eval, provenance | Keeps generated velocity from becoming generated entropy                |

The manuscript already points in this direction, but it needs to say this explicitly near the beginning and then organize the rest of the paper around these five layers. 

## 3. Abstract feedback

The abstract is good but not yet standard-launch quality.

What works: it clearly says AI changes the primary failure mode; it names the proposed stack; it lists operational contributions; and it ends by clarifying that the desired future is human-directed, not human-free. That is a strong foundation.

What needs improvement:

First, the abstract should define “agent-native” more concretely. Right now it says the repo becomes a verification interface, but it should name the required artifacts: owner map, test map, proof lanes, generated-zone ledger, exception registry, and jankurai audit output.

Second, the abstract should make the stack ranking less nakedly assertive. A reader will ask: why this stack, why these weights, why these scores? Add one sentence saying the ranking is based on structural rejection speed, security posture, contract integrity, data truth, test determinism, observability, and repair routing.

Third, jankurai should be elevated. The audit currently appears as one contribution among many. It should be framed as the enforcement mechanism that makes the standard real.

Fourth, the abstract should name pixel QA and CI evidence. If the goal is 100% code automation, UI verification cannot be buried. The abstract should signal that the paper treats browser behavior, visual regression, and test routing as first-class proof surfaces.

Suggested abstract shape:

> AI coding shifts the bottleneck of software engineering from code production to trustworthy merge. This paper defines an agent-native engineering standard in which repositories are designed as verification interfaces: generated code must be routed to an owner, checked by a proof lane, constrained by contracts, rejected by database truth, scanned for security, observed in production, and repaired through machine-readable evidence. The proposed default stack is Rust core, TypeScript/React/Vite product surface, PostgreSQL durable truth, generated contracts, and bounded Python for AI/data services. The paper contributes a 100-point stack rubric, a conformance model, a winner-stack architecture, required agent-control artifacts, a jankurai CI audit, a vibe-coding failure taxonomy, an exception protocol, and a future testing model covering property tests, contract tests, security gates, browser journeys, visual regression, and AI-service evals. The goal is not human-free engineering. The goal is human-directed automation where the repository proves or rejects agent work faster than humans can manually inspect it.

## 4. Section-by-section feedback

### Section I: Introduction

The introduction has the right thesis, but it should be rewritten as a journey through code evolution. The current opening moves too quickly into provocation and evidence. A landmark paper needs to take the reader from historical bottlenecks to the new bottleneck.

Recommended structure:

Start with the bottleneck inversion:

> For seventy years, software engineering optimized around a scarce resource: human ability to express intent as code. AI has made first-draft code abundant. The scarce resource is now trustworthy change.

Then walk the reader through compression:

Machine code compressed nothing. Assembly compressed opcodes. C compressed portable systems programming. Java and .NET compressed memory management and enterprise deployment. Python compressed exploration. JavaScript compressed browser programmability. TypeScript compressed large-scale frontend maintenance. Rust compresses memory-safety and concurrency discipline into the compiler. The agent era compresses syntax itself, which means repositories must now compress judgment into structure.

Then state the new failure mode:

> The agent can produce plausible code. The repo must decide whether the code is allowed to exist.

Then cite the external pressure: GitHub’s Octoverse reports TypeScript became the most-used language on GitHub by contributor counts in August 2025; DORA says AI amplifies organizational systems; Veracode reports AI-generated code introduced risky security flaws in 45% of tests; GitGuardian reports 28.65 million new hardcoded secrets in public GitHub commits in 2025, with AI-service secrets growing 81% year over year. ([The GitHub Blog][2])

The introduction also needs a short definition box:

> **Agent-native engineering:** repository design that gives coding agents explicit ownership maps, bounded edit zones, deterministic proof lanes, generated contracts, security gates, observable failures, and machine-readable repair queues.

### Section II: Languages as compression technology

This is one of the best conceptual sections. Keep it, but make it more rigorous.

Add a table with three columns:

1. Human bottleneck compressed.
2. Agent-era bottleneck exposed.
3. Structural control needed.

For example:

| Era            | Compression             | Agent-era control                                            |
| -------------- | ----------------------- | ------------------------------------------------------------ |
| C              | Hardware control        | Memory-safe replacement or unsafe ledger                     |
| Python         | Exploration             | Bounded runtime role, typed contracts, eval gates            |
| TypeScript     | Browser/product surface | Strict types, generated clients, visual QA                   |
| Rust           | Safe systems core       | Ownership, exhaustive matching, typed errors, property tests |
| SQL/PostgreSQL | Durable truth           | Constraints, migrations, RLS, transactional proof            |

The section should also explicitly state that syntax is no longer the scarce asset, but semantic compression still matters. Agents can imitate syntax; they do not automatically know product invariants, ownership boundaries, compliance obligations, or migration intent.

### Section II.A: Why languages fracture

This needs to become a major section, not a subsection. The user-facing goal is to explain why technically exciting projects often fail to become company-wide defaults.

Be careful with Julia. Do not call it dead. Julia is active: its official site says it has been downloaded over 100 million times and has more than 12,000 registered packages. Its original promise—high-level numerical computing with performance through multiple dispatch and JIT compilation—is real. ([The Julia Programming Language][3])

The stronger framing is:

> Some languages do not die; they fail to become the default operating system of product engineering.

That lets you discuss Julia honestly. It is a serious scientific/numerical language, but it did not become the universal product stack because a language’s inner-loop elegance is not the same as deployment gravity, security tooling, hiring surface, cloud defaults, package maturity, contract discipline, and agent corpus density.

Add a case-study table:

| Ecosystem      | Real promise                                          | Why it did not become default product infrastructure                                              | Agent-era lesson                                              |
| -------------- | ----------------------------------------------------- | ------------------------------------------------------------------------------------------------- | ------------------------------------------------------------- |
| Julia          | Scientific performance plus high-level expressiveness | Product/backend/cloud/security/hiring defaults remained thinner than Python/TS/Go/.NET ecosystems | Great niche strength does not equal universal repo standard   |
| Elm            | Strong frontend correctness                           | Interop and ecosystem constraints limited adoption                                                | Correctness must coexist with ecosystem gravity               |
| D              | Better C/C++ ergonomics                               | Could not displace entrenched C/C++ tooling and libraries                                         | Technical superiority does not beat migration gravity         |
| Nim/Crystal    | Scripting ergonomics plus native compilation          | Smaller production corpus and fewer boring enterprise patterns                                    | Agents need abundant canonical examples                       |
| Haskell/Scala  | Powerful abstraction and types                        | Complexity, training, and local convention burden                                                 | Proof power fails when repair agents cannot infer conventions |
| Clojure/F#     | High-leverage functional niches                       | Smaller mainstream footprint despite mature hosts                                                 | Expert productivity can be opaque to general agents           |
| Elixir/Phoenix | Fault tolerance and realtime systems                  | Specialist fit rather than universal default                                                      | Some stacks deserve workload overrides, not default status    |

Also add a plot suggestion: “technical promise vs. adoption gravity.” Use GitHub Octoverse, Stack Overflow survey data, RedMonk rankings, package counts, job postings, and cloud/runtime support as axes. Stack Overflow’s 2025 survey has more than 49,000 responses and includes admired/desired language data; RedMonk explicitly warns that numerical language rankings should be taken with caution, which is exactly the kind of caveat this paper needs. ([Stack Overflow][4])

### Section III: The harsh AI reality

This section is directionally right but under-evidenced. It should become the bridge from “AI changes coding” to “therefore we must pick a stack and enforce repo structure.”

Strengthen it with three claims:

1. **AI accelerates change volume.** DORA reports broad AI adoption and says the returns depend on platform quality, workflow clarity, and team alignment. ([Google Cloud][5])
2. **AI code is not automatically safe.** Veracode reports risky flaws in 45% of generated-code tests and says larger/newer models did not automatically improve security. ([Veracode][6])
3. **AI increases machine-identity and secret risk.** GitGuardian reports 28.65 million new hardcoded secrets in public GitHub commits in 2025, AI-related secret growth of 81%, and leaked secrets in AI-assisted commits at about 2× baseline in one analyzed category. ([GitGuardian][7])

Also fix one overstatement: human knowledge is not irrelevant. Human syntax fluency is less decisive; human judgment about domain invariants, risk, product intent, and legal/security obligations is still critical. The paper should say: “human knowledge is no longer the critical path for first-draft syntax; it remains the critical path for goals, boundaries, and accountability.”

### Section IV: 100-point stack rubric

This section needs the biggest methodological upgrade.

The rubric is useful, but right now it reads like a scorecard the author already agreed with. To make it credible:

1. Define each criterion with measurable signals.
2. Add scoring examples.
3. Add hard caps.
4. Add confidence intervals.
5. Add calibration data.
6. Add a “who scores this?” procedure.
7. Publish the scoring sheet as an appendix.

For example, “Agent-verifiable correctness loop” should not just be a weighted criterion. It should break down into:

| Subcriterion                          | Points | Evidence                                  |
| ------------------------------------- | -----: | ----------------------------------------- |
| Fast local compile/typecheck          |      4 | Command under time budget                 |
| Deterministic changed-file proof lane |      4 | `test-map.json` + CI                      |
| Generated contract drift check        |      4 | OpenAPI/Protobuf/JSON Schema regeneration |
| Property/fuzz tests for invariants    |      3 | Domain proof                              |
| Repair-localizing failure output      |      3 | Jankurai/CI report                       |

The paper should also distinguish “score” from “merge eligibility.” A stack can score well but fail a hard cap. Example hard caps:

| Failure                                        | Maximum score |
| ---------------------------------------------- | ------------: |
| No secret scanning                             |            70 |
| No dependency/SCA scan                         |            75 |
| No generated contract boundary                 |            80 |
| No test map                                    |            82 |
| No owner map                                   |            82 |
| Direct UI-to-database access                   |            65 |
| Python owns product truth outside an exception |            70 |
| Generated files hand-edited                    |            75 |
| No browser proof for critical UI               |            80 |

This cap system will make the paper feel like a standard, not an essay.

### Section V: Top-five stack ranking

The ranking is compelling but currently not defensible enough.

The difference between 94, 90, 89, 88, and 87 looks too exact. If you keep numeric scores, add uncertainty bands: Rust stack 94 ± 3, Go stack 90 ± 4, .NET 89 ± 4, and so on. That would admit that the exact difference is less important than the stack’s conformance to the standard.

The paper should also explain why Rust wins beyond “memory safety.” Go, Java, C#, Swift, Python, and others are also memory-safe in important senses. The Rust argument should be:

* Low runtime overhead.
* No garbage collector in the core path.
* Strong ownership and borrowing model.
* Exhaustive enums and pattern matching.
* Explicit error handling.
* Strong package tooling.
* Good fit for domain invariants, concurrency, and systems boundaries.
* Strong rejection before runtime.

CISA and NSA’s 2025 guidance supports adoption of memory-safe languages to reduce risk, but the paper must not imply Rust is the only memory-safe option. ([U.S. Department of War][8])

The Kafka paragraph is too speculative. “Rust-native replacement” may be directionally plausible, but the paper needs evidence or should soften it. Say: “event streaming is workload-specific; the standard should define how to add streaming without letting it become stack identity.”

Add a “workload override” table:

| Workload                        | Default                   | Allowed override       | Required proof                                    |
| ------------------------------- | ------------------------- | ---------------------- | ------------------------------------------------- |
| Realtime presence/collaboration | Rust/TS/PG                | Elixir/Phoenix         | Presence correctness tests, ops runbook           |
| Enterprise Microsoft estate     | Rust/TS/PG                | .NET/TS/PG             | Azure identity, compliance, SCA gates             |
| Simple cloud services           | Rust/TS/PG                | Go/TS/PG               | Race tests, contract tests, security lane         |
| Existing JVM estate             | Rust/TS/PG target         | Kotlin/Java transition | Migration plan, dependency cap, contract boundary |
| Scientific compute              | Bounded Python, Rust core | Julia module/service   | Contract boundary, deployment proof, package risk |

This prevents the standard from sounding naive while still enforcing a winner profile.

### Section VI: Winner-only architecture

This is one of the stronger sections. The repository-as-routing-table idea is excellent.

Upgrade it by adding:

1. A dependency-direction diagram.
2. A “forbidden edge” table.
3. An example patch route.
4. A migration route from messy repo to standard repo.
5. A generated-zone policy.
6. A database truth policy.

Example forbidden edges:

| Forbidden edge                                  | Reason                                          |
| ----------------------------------------------- | ----------------------------------------------- |
| `apps/web` → direct SQL                         | UI must not own durable truth                   |
| `python/ai-service` → production billing tables | AI/data service must not own product invariants |
| `adapters` → domain decision logic              | Prevents business rules from hiding in I/O      |
| generated client → hand edits                   | Contract drift                                  |
| route handler → complex invariant               | Domain/application boundary violation           |
| tests only → new behavior without product code  | Test theater                                    |

Also add a “patch routing example”:

> A pricing rule change touches Rust domain invariant, Rust application transaction tests, PostgreSQL constraint/migration, generated API contract if exposed, and Playwright checkout journey. It must not be implemented only in React form validation or Python prompt logic.

That kind of concrete example will make the standard adoptable.

### Section VII: Agent-first repository design

This section is good but incomplete. It discusses Codex, Claude Code, Cursor, and Copilot, but the paper needs a tool-specific strength matrix and must include agent-first environments such as Google Antigravity.

Codex supports AGENTS.md discovery with layered global and project guidance; Claude Code uses CLAUDE.md and path-scoped rules but treats instructions as context rather than hard enforcement; GitHub Copilot supports repository-wide, path-specific, and agent instructions; Google Antigravity emphasizes agent orchestration across editor, terminal, and browser with artifacts such as plans, screenshots, and browser recordings. ([OpenAI Developers][9])

Add this table:

| Tool                       | Strength to call out                                                              | Standard adapter                                                                          |
| -------------------------- | --------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| Codex                      | AGENTS.md hierarchy, repo-level task execution, custom instructions, hooks/config | Root `AGENTS.md`, local `AGENTS.md`, proof-lane commands                                  |
| Claude Code                | Strong planning, memory, CLAUDE.md, path-scoped rules, long refactors             | `CLAUDE.md` imports canonical standard; `.claude/rules` mirrors owner/test maps           |
| Cursor                     | Fast local iteration, style absorption, project/team/user rules, inline edits     | `.cursor/rules/*.mdc` generated from canonical rules; scoped style rules                  |
| GitHub Copilot             | Organization rollout, GitHub-native PR/review flow, repo/path instructions        | `.github/copilot-instructions.md`, `.github/instructions/*`, AGENTS mirrors               |
| Antigravity                | Multi-agent orchestration, browser validation, artifacts, task-level delegation   | Require sandbox, permission policy, artifact capture, visual proof, terminal restrictions |
| Aider/terminal-first tools | Explicit Git diff loop and command-line control                                   | Proof-lane wrapper, no direct destructive shell without allowlist                         |

The rule should be: tool files are generated mirrors, not independent policy. If Cursor rules, Claude rules, and AGENTS.md disagree, the repo has failed.

### Section VIII: Testing and automated QA

This section is too short for the paper’s ambition. It should be split into three sections:

1. Proof-lane testing.
2. Pixel/browser QA.
3. Future CI and coverage explosion.

The current Playwright discussion is good but insufficient. Playwright supports user-visible locators, web-first assertions, and visual comparisons through screenshot baselines; those are exactly the right primitives for agent repair because they test behavior and visible output rather than implementation details. ([Playwright][10])

Add a required test matrix:

| Layer            | Required proof                                                    |
| ---------------- | ----------------------------------------------------------------- |
| Rust domain      | Unit + property tests for invariants/state machines               |
| Rust application | Authz, idempotency, transaction, workflow integration tests       |
| Rust adapters    | DB, external API, queue, failure injection                        |
| PostgreSQL       | Migration apply/rollback, constraints, indexes, RLS, schema drift |
| Contracts        | Generate, diff, compatibility, consumer tests                     |
| TypeScript UI    | Typecheck, component tests, critical Playwright journeys          |
| Pixel QA         | Visual baselines by route/state/breakpoint/theme                  |
| Accessibility    | Keyboard, role/name, contrast, screen-reader-critical flows       |
| Security         | Secret scan, SCA, SAST, dependency review, SBOM/provenance        |
| AI/data service  | Contract tests, eval suites, prompt/version drift tests           |
| Observability    | Trace/log/error receipt tests                                     |
| Performance      | Latency, memory, cold start, concurrency, DB query budgets        |

### Section IX: Agent-friendly exceptions

This is a very strong section. It should become more concrete.

Add one full example for each language:

* Rust domain error enum.
* TypeScript boundary error.
* Python AI-service error wrapper.
* RFC 9457 HTTP problem detail response.
* OpenTelemetry exception attributes.
* Redacted log event.

OpenTelemetry’s semantic conventions and exception attributes support the paper’s claim that standardized error metadata can make production failures more repairable. ([OpenTelemetry][11])

Add “exception receipts”:

```json
{
  "error_code": "PAYMENT_IDEMPOTENCY_CONFLICT",
  "stable_name": "PaymentIdempotencyConflict",
  "protected_invariant": "A payment command may be applied at most once per idempotency key.",
  "safe_user_message": "This payment request was already processed.",
  "repair_doc": "docs/exceptions/payment-idempotency.md",
  "correlation_id": "req_...",
  "allowed_repairs": [
    "check idempotency key generation",
    "verify transaction boundary",
    "replay workflow test lane"
  ],
  "required_proof_lane": "application"
}
```

That is exactly the kind of artifact agents can use.

### Section X: The jankurai audit

This should be the centerpiece of the paper.

Right now the audit section is promising, but it needs to move from “audit idea” to “mandatory standard control.”

The paper should say:

> Every serious repo must install jankurai. A repo without jankurai has no machine-readable centerline and therefore cannot be agent-native.

Define jankurai as more than a linter:

> jankurai is a repository alignment auditor. It measures whether the codebase is moving toward or away from the agent-native centerline.

Minimum required outputs:

```json
{
  "standard_version": "0.2.0",
  "target_profile": "rust-ts-pg-python-ai",
  "score_raw": 86,
  "score_capped": 72,
  "centerline_delta": -6.4,
  "caps_applied": [
    "generated_zone_mutated",
    "missing_secret_push_protection"
  ],
  "dimension_scores": {
    "ownership": 82,
    "contracts": 77,
    "proof_lanes": 69,
    "security": 60,
    "code_shape": 84,
    "data_truth": 72,
    "observability": 70,
    "context_economy": 88,
    "python_containment": 90
  },
  "agent_fix_queue": [
    {
      "priority": 1,
      "finding": "Generated client was hand-edited",
      "path": "apps/web/src/generated/api.ts",
      "repair": "Regenerate from contracts/openapi/api.yaml",
      "proof_lane": "contract"
    }
  ]
}
```

Also require SARIF output so jankurai can surface in GitHub code scanning. GitHub supports uploading third-party SARIF files and displaying alerts in code scanning. ([GitHub Docs][12])

Hard rules should include:

| Category           | jankurai check                                                       |
| ------------------ | --------------------------------------------------------------------- |
| Ownership          | path owner exists; dependency edge allowed                            |
| Proof routing      | touched file maps to proof lane                                       |
| Generated zones    | generated outputs not hand-edited                                     |
| Contracts          | OpenAPI/Protobuf/JSON Schema drift checked                            |
| Data truth         | DB access only in allowed layer                                       |
| Python containment | Python cannot own billing/authz/product truth without exception       |
| Dead language      | banned markers blocked unless allowlisted                             |
| Security           | secret scanning, SCA, lockfiles, SBOM/provenance                      |
| Tests              | critical path coverage, property tests, DB tests, Playwright journeys |
| Pixel QA           | visual baseline coverage for critical UI                              |
| Observability      | structured errors, request IDs, trace propagation                     |
| Agent docs         | canonical rules and tool mirrors match                                |
| Exceptions         | exceptions named, scoped, owned, expiring                             |
| Context economy    | root instructions short; local instructions scoped                    |

GitHub push protection can block hardcoded credentials before they reach a repository, OSV-Scanner maps dependencies to known vulnerabilities, and SLSA provides supply-chain controls for artifact integrity; those should be standard jankurai security integrations rather than optional nice-to-haves. ([GitHub Docs][13])

### Section XI: Migration, versioning, future research

This section is good but should be more actionable.

Add a maturity model:

| Level | Name               | Meaning                                                                    |
| ----- | ------------------ | -------------------------------------------------------------------------- |
| HL0   | Uncontrolled       | No maps, no proof lanes, no audit                                          |
| HL1   | Advisory           | jankurai runs but does not block                                          |
| HL2   | Routed             | owner/test/generated-zone maps exist                                       |
| HL3   | Gated              | CI blocks hard caps and severe drift                                       |
| HL4   | Agent-native       | agents receive repair queues and deterministic proof lanes                 |
| HL5   | Regulated/critical | provenance, SBOM, security approvals, release evidence, compliance mapping |

Add a migration order:

1. Install jankurai in advisory mode.
2. Add owner map, test map, generated-zone map.
3. Add proof lanes.
4. Add contract generation.
5. Move durable truth into PostgreSQL constraints and Rust domain/application.
6. Box Python.
7. Add Playwright and pixel QA for critical paths.
8. Add security/provenance gates.
9. Enable hard caps.
10. Publish score trend per PR.

### Section XII: Conclusion

The conclusion has the right force. To make it land harder, end with a concrete adoption command:

> Install the audit. Pin the standard version. Generate the agent files. Map owners. Map tests. Lock generated zones. Run the proof lanes. Publish the score. Block drift.

The conclusion should not only persuade; it should tell teams what to do Monday morning.

## 7. Jankurai must be positioned as mandatory

The paper should make jankurai the enforcement layer for all coding projects, not just the project associated with the paper.

A strong claim:

> Agent-first engineering without jankurai is unenforced preference. jankurai is the centerline. Every PR either moves the repo closer to the center or farther away.

Add these concepts:

### Centerline score

A repo has a target profile. Every PR changes the repo’s distance from that profile.

Metrics:

| Metric              | Meaning                                                |
| ------------------- | ------------------------------------------------------ |
| Centerline score    | Current conformance score                              |
| Centerline delta    | Change in score from previous commit/PR                |
| Drift velocity      | Score loss per PR or per 1,000 changed lines           |
| Proof adequacy      | Percent of touched files mapped to passing proof lanes |
| Boundary entropy    | Number and severity of forbidden dependency edges      |
| Dead-marker density | Banned markers per KLOC                                |
| Generated drift     | Hand-edited generated output count                     |
| Contract drift      | Schema/client/server mismatch count                    |
| Pixel coverage      | Critical route-state-breakpoint coverage               |
| Secret posture      | Secret scan/push protection/SCA status                 |
| Exception debt      | Open exceptions weighted by age and severity           |

### Mandatory jankurai files

Every repo should include:

```text
jankurai.toml

agent/owner-map.json
agent/test-map.json
agent/proof-lanes.toml
agent/generated-zones.toml
agent/audit-policy.toml

```

### Required outputs

```text
jankurai.audit.json

jankurai.audit.sarif
jankurai.centerline.json
jankurai.fix-queue.json
```

### CI modes

## 8. Pixel QA needs a full section

Pixel QA is not cosmetic. It is one of the most important verification surfaces for agent-coded apps.

AI can produce a page that compiles, passes typecheck, and still has broken layout, hidden buttons, unreadable contrast, missing focus states, clipped modals, wrong spacing, mobile overflow, dark-mode failure, or localization breakage. DOM tests often miss this.

Add a section called something like “Browser and Pixel Proof.”

### Required pixel QA model

| Coverage dimension | Examples                                             |
| ------------------ | ---------------------------------------------------- |
| Route              | login, dashboard, checkout, settings, admin          |
| State              | loading, empty, populated, error, permission denied  |
| Breakpoint         | mobile, tablet, desktop, wide                        |
| Theme              | light, dark, high contrast                           |
| Locale             | English, long-string locale, RTL where relevant      |
| Data density       | few rows, many rows, extreme names                   |
| Interaction        | hover, focus, open menu, modal, toast                |
| Accessibility      | keyboard, role/name, visible focus, ARIA correctness |
| Critical journey   | signup, purchase, invite, password reset, export     |

### Implementation standard

1. Use Playwright for critical journeys.
2. Use user-visible locators, not brittle CSS selectors.
3. Use web-first assertions.
4. Capture trace/video/screenshot on failure.
5. Add screenshot baselines for critical states.
6. Mask dynamic regions.
7. Freeze time/randomness/test data.
8. Disable animations in visual mode.
9. Run consistent browser/OS/container versions.
10. Store visual diffs as PR artifacts.
11. Require approval for baseline changes.
12. Do not let agents update baselines without an explicit “visual baseline update” proof lane.

Playwright’s docs support screenshot comparisons through `toHaveScreenshot`, resilient locators such as role-based selectors, and web-first assertions that wait for expected states. ([Playwright][17])

### Pixel QA failure modes and mitigations

| Failure                                         | Mitigation                                                            |
| ----------------------------------------------- | --------------------------------------------------------------------- |
| False positives from font/rendering differences | Containerized browser, deterministic fonts, CSS-pixel mode, stable OS |
| Dynamic content noise                           | Mask regions, freeze data/time/randomness                             |
| Animation/transitions                           | Disable animations in test CSS                                        |
| Baseline spam                                   | Require human approval for baseline updates                           |
| Screenshot passes while UX fails                | Pair visual checks with semantic assertions and accessibility tests   |
| Agent accepts bad diff                          | Human approval or separate visual-review lane                         |
| Mobile overflow missed                          | Required breakpoint matrix                                            |
| Hidden focus/keyboard bug                       | Keyboard journey tests                                                |
| Localization break                              | Long-string and RTL snapshots                                         |
| Dark mode break                                 | Theme-specific baselines                                              |

### Future pixel QA

The paper should predict that agents will create UI state maps automatically:

```text
route × state × viewport × theme × locale
```

Then the CI system will select only the affected cells for each PR, while nightly CI runs the full visual matrix. This is the right future: massive coverage, routed intelligently.

## 9. Future CI and testing section

The paper should make a major argument: CI will stop being a single pipeline and become a proof router.

### Future CI model

| Lane               | Trigger                   | Purpose                                   |
| ------------------ | ------------------------- | ----------------------------------------- |
| Fast lane          | Every PR                  | Compile, typecheck, changed-file tests    |
| Contract lane      | API/schema changes        | Generate and diff contracts               |
| DB lane            | Migration/model changes   | Apply migrations, constraints, RLS, drift |
| Security lane      | Every PR + nightly        | Secrets, SCA, SAST, SBOM/provenance       |
| Browser lane       | UI changes                | Playwright critical journeys              |
| Pixel lane         | UI/design changes         | Visual baselines                          |
| Property/fuzz lane | Domain changes            | Invariant exploration                     |
| Performance lane   | Hot-path changes          | Latency, memory, query budgets            |
| AI eval lane       | Prompt/model/tool changes | Eval suite and tool contract tests        |
| Full release lane  | Release candidates        | All lanes + provenance + evidence bundle  |

### The good side of test explosion

* Agents can generate more edge-case tests.
* Property tests can cover invariant space.
* Browser tests can cover more journeys.
* Visual tests can catch UI regressions.
* AI-service evals can track prompt/model drift.
* CI can produce repair artifacts for the next agent.

### The bad side

* Flaky tests multiply.
* Generated tests can assert nonsense.
* Visual baselines can become rubber stamps.
* CI cost can explode.
* Slow pipelines push teams to bypass gates.
* Agents can optimize for tests rather than correctness.
* Coverage can become performative.

### Required controls

| Control                | Why                                               |
| ---------------------- | ------------------------------------------------- |
| test-map.json          | Agents should never guess proof commands          |
| Flake registry         | Flaky tests must be owned and time-boxed          |
| Mutation testing       | Detects tests that do not prove behavior          |
| Test budget            | Prevents endless generated test bloat             |
| Failure receipts       | Every failure should route to repair              |
| Lane timing budget     | Keeps fast feedback fast                          |
| Critical-path registry | Ensures business-critical flows are always tested |
| Evidence retention     | Makes release proof auditable                     |

## 10. Agent-specific design section

Add a section that separates tool-agnostic principles from tool-specific adapters.

### Tool-agnostic principles

These apply to every AI coding tool:

1. Keep root instructions short.
2. Put local rules near local code.
3. Use machine-readable maps.
4. Never duplicate contradictory rules.
5. Give exact commands.
6. Make generated zones read-only.
7. Prefer contracts over prose.
8. Emit repair receipts.
9. Require proof before merge.
10. Treat agent output as untrusted until verified.

### Tool-specific additions

| Tool            | What the paper should say                                                                                                                                                                                         |
| --------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Codex           | Best when AGENTS.md and proof commands are clear. Use layered instructions and exact validation commands.                                                                                                         |
| Claude Code     | Strong for planning/refactor/debug loops. Keep CLAUDE.md concise; import AGENTS.md to avoid divergent policy; use path-scoped rules.                                                                              |
| Cursor          | Strong for local style, inline changes, refactors, and day-to-day codebase iteration. Generate Cursor rules from canonical policy; do not let style rules override architecture.                                  |
| GitHub Copilot  | Strong for organization-scale GitHub workflows, repo/path instructions, reviews, and PR integration. Use repository instructions as a rollout mechanism.                                                          |
| Antigravity     | Strong for task-level orchestration, browser validation, artifacts, screenshots, and multi-agent work. Requires stricter sandboxing and artifact review because it operates across editor, terminal, and browser. |
| Terminal agents | Strong for explicit diff loops and deterministic commands. Require destructive-command controls.                                                                                                                  |

Google describes Antigravity as an agentic development platform where agents can plan, execute, and verify tasks across editor, terminal, and browser, and produce artifacts such as task lists, plans, screenshots, and browser recordings. That is directly relevant to the paper’s pixel QA and evidence model. ([Google Developers Blog][21])

## 11. The “hype projects” section should be reframed

The manuscript’s current “dead-hype” framing is useful but risky. Some ecosystems listed are not dead; many are thriving in niches. The paper should not sound dismissive. The better section is:

> Technical Promise vs. Standard Gravity

Main argument:

> A language becomes an agent-native default only when it combines technical strength with deployment gravity, tooling gravity, security gravity, corpus gravity, hiring gravity, and repair gravity.

Add this framework:

| Gravity            | Question                                                                     |
| ------------------ | ---------------------------------------------------------------------------- |
| Deployment gravity | Can ordinary teams deploy it boringly?                                       |
| Security gravity   | Are scanners, SBOM, SCA, policies, and secure defaults mature?               |
| Corpus gravity     | Are there enough public examples for agents to learn common patterns?        |
| Contract gravity   | Are API/schema/client generation patterns mature?                            |
| Hiring gravity     | Can teams staff and review it?                                               |
| Ops gravity        | Are logs, metrics, tracing, migrations, queues, and runtime behavior boring? |
| Repair gravity     | Can agents localize and fix failures quickly?                                |
| Migration gravity  | Can existing systems adopt it incrementally?                                 |

Then Julia becomes a valuable lesson, not a punchline. It solved a real numerical-computing problem; it did not become the universal default for product engineering because universal defaults require a full operating ecosystem, not only elegant language design.

## 14. Bottom-line verdict

The manuscript has the right center: agent-native engineering must be verification-first, not prompt-first. The proposed stack is plausible. The repo layout and audit concepts are strong. But to become a landmark, the paper must move from declaration to enforcement.

The most important upgrade is this:

> The standard is not the stack. The standard is the stack plus the repo shape plus jankurai plus proof lanes plus generated contracts plus pixel QA plus security gates plus repair receipts.

That is the version that can force alignment.
