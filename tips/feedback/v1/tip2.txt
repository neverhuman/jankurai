I reviewed the manuscript as a standards-launch paper, not as a casual essay. The current draft has a strong skeleton: it already contains the core thesis, a 100-point stack rubric, a ranked stack table, an ownership model, proof lanes, an exceptions model, jankurai audit rules, and machine-readable repo artifacts. But it is not yet landmark-grade. Right now it reads like a forceful manifesto plus a useful draft spec. To become the standard you want, it needs a cleaner argument arc, harder evidence, clearer compliance mechanics, a deeper vibe-coding failure taxonomy, a much stronger testing/pixel-QA section, and a more defensible stack-ranking methodology. 

## Overall verdict

The manuscript’s best idea is this: **the repository becomes the verification interface for agents**. That idea is strong enough to anchor a real standard.

The biggest weakness is that the draft often **asserts** the standard instead of **proving why the standard is inevitable**. The paper says the winning stack matters, but the score gap between the top five stacks is small. It says the audit matters, but the audit is not yet specified tightly enough to become a mandatory compliance layer. It says vibe coding is dangerous, but it does not yet give the reader a comprehensive, ranked failure model with concrete detection rules and repair gates.

The paper should stop trying to win by rhetorical force and win by **operational inevitability**: every claim should lead to a required repo artifact, a CI gate, a score cap, a repair queue item, or a migration path.

The standard should be framed as:

> A compliant agent-native repository must route every change to an owner, constrain every generated edit by boundary rules, prove every modified behavior through deterministic proof lanes, preserve security and provenance by default, and emit machine-readable repair evidence for the next agent.

That is much stronger than simply saying teams should adopt one stack.

## The core upgrade: separate manifesto, standard, and evidence

The current draft mixes three genres:

1. A manifesto about the end of human-centered coding.
2. A technical standard for agent-first repositories.
3. A research-style argument for one full-stack architecture.

That mixture weakens the paper. A landmark version should use this structure:

| Part                                 | Purpose                                                                                        | What it must contain                                                                                            |
| ------------------------------------ | ---------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| **Part I: Why the old model breaks** | Take the reader through software evolution and the collapse of human typing as the bottleneck. | Language evolution, hype-stack lessons, AI-code risk evidence, failure modes of vibe coding.                    |
| **Part II: The standard**            | Define what every agent-native repo must contain.                                              | Required artifacts, proof lanes, jankurai audit, score caps, repo layout, exception model, pixel QA, CI gates. |
| **Part III: The stack decision**     | Justify the winning stack and rank alternatives.                                               | Rubric, hard filters, sensitivity analysis, score matrix, migration paths, exceptions.                          |
| **Part IV: Adoption and governance** | Make this usable as a living standard.                                                         | Versioning, compliance levels, rollout plan, reference implementation, governance model, future research.       |

The paper should not merely say “this is the standard.” It should make noncompliance look technically irresponsible.

## Abstract feedback

The abstract is clear but overloaded. It currently tries to do too much: problem statement, stack selection, audit terms, architecture, and end-state philosophy. The result is dense and less memorable than it should be.

What works:

* The bottleneck shift is well stated.
* The phrase “reject wrong generated code quickly” is excellent.
* The stack is named directly.
* The repository-as-verification-interface idea is strong.

What needs improvement:

* The abstract should introduce **jankurai** as the enforcement mechanism, not just one component in a list.
* It should not list every contribution in one sentence; it should group them.
* It should define “agent-native” earlier.
* It should distinguish **100% code automation** from **zero human responsibility**. The stronger claim is: humans define goals, risk, and policy; agents generate, repair, test, and migrate code under enforced repo constraints.
* It should not bury the audit’s role near the end. The audit is the standard’s control plane.

A stronger abstract would be:

> AI coding changes the critical path of software engineering. The scarce skill is no longer typing syntactically plausible code; agents can already generate plausible implementations across mainstream languages. The scarce capability is verification: a repository must reject unsafe changes, localize defects, prove repairs, preserve security posture, and provide machine-readable context for the next agent.
>
> This paper proposes an agent-native engineering standard for repositories maintained primarily through automated code generation and repair. The standard defines a target stack, mandatory repository control surfaces, generated-contract boundaries, proof lanes, exception records, pixel-level QA, security gates, and a jankurai audit that converts vague code-quality concerns into enforceable CI findings and agent repair queues.
>
> The recommended default stack is Rust for durable core logic, TypeScript/React/Vite for product surface, PostgreSQL for durable truth, generated contracts for boundaries, and bounded Python for AI/data service work. The stack is chosen not for author preference but for rejection speed, proof locality, security posture, observability, and repair routing. The goal is not less engineering discipline. The goal is to make full code automation safe enough to become the default.

## Introduction: the paper needs a journey, not just a claim

The introduction currently starts too defensively and jumps too quickly into supporting reports. It should take the reader through the evolution of software bottlenecks:

1. Early programming bottleneck: machine detail.
2. Language bottleneck: human memory and expression.
3. Framework bottleneck: repeated product patterns.
4. Cloud bottleneck: deployment and scaling.
5. AI bottleneck: verification, not generation.
6. New standard: repos must become agent-operable proof systems.

The intro should make readers feel the historical transition. Right now, it states the transition.

A stronger opening arc:

> Software engineering has always been a history of moving burden out of human heads and into machinery. Assembly removed the need to remember raw opcodes. High-level languages removed much of the need to manually manage machine detail. Frameworks compressed repeated product patterns. Cloud platforms compressed deployment and scaling. Static types, linters, contracts, and CI compressed parts of review.
>
> AI coding compresses a different bottleneck: code production itself. A developer no longer needs deep fluency in every syntax, framework, or library to produce a plausible first implementation. That changes the failure mode. The dangerous repository is no longer the one where code is slow to write. The dangerous repository is the one where generated code is easy to merge, hard to localize, weakly tested, poorly observed, and impossible for the next agent to repair without rediscovering tribal knowledge.
>
> This shift does not make engineering discipline obsolete. It makes discipline mandatory. DORA’s 2025 research frames AI as an amplifier of existing organizational strengths and weaknesses, not an automatic productivity guarantee. Veracode reports that AI-generated code introduced risky security flaws in 45% of tests. GitGuardian reports 28.65 million new hardcoded secrets in public GitHub commits in 2025, with AI-service secrets growing 81% year over year and AI-assisted commits leaking secrets at roughly twice the baseline. ([Dora][1])
>
> The correct response is not to slow agents until old review processes survive. The correct response is to redesign repositories so that agents operate inside explicit boundaries, deterministic proof lanes, generated contracts, security gates, and repairable exception records.

That introduction would give the paper a real journey.

## Section-by-section feedback

### I. Introduction

**Current problem:** The section has a strong claim but not enough narrative. It cites good external evidence, but the evidence feels bolted on.

**Upgrade:**

* Start with the evolution of bottlenecks.
* Define “agent-native engineering” in the first page of text.
* State the standard’s non-negotiables: owner map, proof lanes, generated contracts, jankurai audit, pixel QA, security/provenance, exception records.
* Add a paragraph that says the paper is proposing a **compliance standard**, not just an opinionated stack.

**Missing sentence:**
“An agent-native repository is one where the default path for an automated change is not exploration, guessing, and patching, but routing, constrained editing, deterministic proof, and machine-readable repair.”

### II. Languages as compression technology

This is one of the better conceptual sections. The language-as-compression framing is strong and should stay.

**Current problem:** It is too compressed. It needs more connection to agentic coding.

**Upgrade:**

Add a subsection called **“What AI changes about language choice.”** The key point:

* Pre-AI language choice optimized for human familiarity, hiring, syntax comfort, ecosystem, and runtime.
* Agent-era language choice optimizes for rejection speed, static feedback, boundary clarity, deterministic tooling, and repair locality.

GitHub’s 2025 Octoverse data is useful here, but the paper must be precise: TypeScript became the most-used language on GitHub by monthly contributors in August 2025, with GitHub explicitly connecting typed systems to earlier detection of AI-generated errors. That supports TypeScript for product surfaces, not every backend choice. ([The GitHub Blog][2])

### II.A. Why languages fracture

This section should become a major bridge into the “hype projects that did not become defaults” discussion.

**Current problem:** The table is good but too short and too polite. The paper needs a sharper pattern language for why technically exciting ecosystems fail to become universal defaults.

Do **not** frame these ecosystems as stupid or dead. That will alienate serious engineers and make the paper look unserious. Frame them as:

> serious ideas that solved one bottleneck but failed to become the default because they did not solve the whole operating system of company-scale software.

For Julia specifically, the paper should say: Julia attacked the two-language problem in numerical computing; that is real and important. But solving scientific inner-loop performance does not automatically solve product engineering defaults: hiring, packaging, cloud deployment, security tooling, observability, browser integration, CI norms, and agent corpus density. Julia’s own research framing emphasizes the two-language problem; the paper should use that as the case study, not as a cheap dismissal. ([The Julia Programming Language][3])

Add a table like this:

| Ecosystem                       | Real promise                                                                 | Why it did not become universal default                                                                         | Agent-era lesson                                                                               |
| ------------------------------- | ---------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| Julia                           | High-level numerical code with performance; attack the two-language problem. | Excellent scientific niche, weaker default product/cloud/security/hiring surface than Python/TS/Rust/Go stacks. | Solving inner-loop expression is not enough; agents need boring deployment and broad examples. |
| Elm                             | Strong frontend correctness and no-runtime-exception ambition.               | Interop and ecosystem constraints limited default adoption.                                                     | Great safety lessons; too narrow as universal product surface.                                 |
| CoffeeScript                    | Cleaner JS syntax during a painful era.                                      | Syntax improvement lost to TypeScript’s stronger type/contract value.                                           | Agent-era abstractions must improve verification, not just aesthetics.                         |
| Scala/Haskell/F#                | Powerful abstractions and type systems.                                      | Expert-friendly abstractions can exceed average team and agent repair bandwidth.                                | Proof power must be paired with common conventions and simple repair paths.                    |
| D/Nim/Crystal                   | Ergonomic native compilation alternatives.                                   | Could not overcome ecosystem gravity and tooling maturity.                                                      | Small corpora hurt both human onboarding and agent reliability.                                |
| Elixir/Phoenix                  | Fault tolerance, supervision, realtime collaboration.                        | Specialist fit rather than default full-stack choice.                                                           | Specialist overrides should be explicit, not confused with standard stack identity.            |
| Meteor/Rails-style magic stacks | Very fast product creation.                                                  | Hidden boundaries and framework magic become painful under scale and automated repair.                          | Speed without boundary proof becomes vibe debt.                                                |

### III. The harsh AI reality

This section is directionally correct but underpowered.

**Current problem:** It says second place is more expensive, but does not prove why. It should distinguish “second place for humans” from “second place for agents.”

**Upgrade:**

Add a subsection called **“The false plausibility problem.”**

Core claim:

* AI makes code plausible before it makes code correct.
* Syntax success is no longer evidence.
* The stronger the generation loop, the more dangerous weak verification becomes.
* A repository that lacks proof lanes will accumulate “vibe debt” faster than a human team can review it.

Use Veracode’s 45% AI-generated security flaw figure here, not only in the intro. Use GitGuardian’s secrets-sprawl data to show that speed increases identity risk. Use DORA to show why AI amplifies weak orgs instead of fixing them. ([Dora][1])

Also add a paragraph on **iterative degradation**: a 2025 study found a 37.6% increase in critical vulnerabilities after five rounds of iterative AI code “improvement.” That is directly relevant to vibe coding because vibe coding often proceeds through repeated prompt nudges rather than disciplined design/test cycles. ([arXiv][4])

### IV. Method: 100-point stack rubric

This is necessary but not yet defensible enough.

**Current problem:** The weights look author-chosen. That is acceptable for version 0.1 of a standard, but the paper must say so and add calibration plans.

**Upgrade the method section with:**

1. **Hard filters before scoring.** Some failures should cap the score regardless of other strengths.
2. **Sensitivity analysis.** Show how rankings change if weights shift toward enterprise, hiring, performance, security, or product velocity.
3. **Dimension-level score matrix.** Do not only show final ANSS scores.
4. **Explicit scoring evidence.** Each stack score should show why it gained/lost points.
5. **Review panel plan.** The standard should say future versions will calibrate weights across real repos and incident data.

The current rubric says “product velocity” is only 2 points. That may be strategically right, but the paper needs to justify it harder. Otherwise product teams will dismiss the rubric as unrealistic.

Add hard filters like:

| Hard filter                                                     | Cap if missing |
| --------------------------------------------------------------- | -------------: |
| No deterministic local proof lane                               |         Max 55 |
| No CI jankurai audit                                           |         Max 60 |
| No secret scanning and push protection                          |         Max 60 |
| No generated contract boundary for cross-language APIs          |         Max 70 |
| No owner/test map                                               |         Max 72 |
| No DB migration/schema drift gate                               |         Max 75 |
| No pixel/UI journey proof for user-facing app                   |         Max 80 |
| No SBOM/provenance path                                         |         Max 82 |
| Unbounded agent terminal/browser permissions in normal workflow |         Max 65 |

This is where the paper starts to become a real standard.

**Upgrade:**

| ----------------------------- | ----------------: | ------------: | -----------: | --------: | -------: | --------: | ------: | --------------: | ----: |

Then add **why Rust wins despite Go being close**:

* Rust gives stronger local rejection through ownership, exhaustive matching, algebraic data types, explicit error handling, and compile-time constraints.
* Go gives excellent simplicity and operational maturity, but more correctness is left to runtime tests and review.
* .NET is excellent for enterprise/regulatory contexts but carries platform assumptions.

* JVM stacks remain powerful but are not the standard’s target because they preserve more legacy/runtime surface than necessary.

The NSA/CISA memory-safe language guidance supports the direction toward memory-safe defaults, but it does not prove Rust alone. The paper must explicitly say that Rust wins because of **memory safety plus ownership plus compile-time repair locality**, not just because it is memory safe. ### VIII. Testing and automated QA

This section is far too small for the role testing must play in the standard.

**Current problem:** It names proof lanes but does not yet explain the future of CI, generated tests, pixel QA, or the bad side of test explosion.

The paper needs a major section called:

## Automated proof: from tests to repair routing

It should cover:

1. **Proof lanes by risk, not by convenience.**
2. **Generated tests as hypotheses, not truth.**
3. **Property tests for invariants.**
4. **Contract tests for boundaries.**
5. **Migration and RLS tests for PostgreSQL.**
6. **Pixel QA and visual regression tests for apps.**
7. **Accessibility snapshots as semantic UI proof.**
8. **Trace/video artifacts for agent repair.**
9. **Flake budgets and quarantine rules.**
10. **Mutation testing or fault injection for critical logic.**
11. **CI output as machine-readable repair input.**

Playwright’s own best practices support user-visible behavior, resilient locators, and web-first assertions, which fits the paper’s goal: agents repair better when tests describe what users see rather than brittle implementation details. ([Playwright][11])

### Pixel QA needs a full subsection

The paper should be explicit: user-facing apps cannot be agent-native without automated visual and interaction proof.

Add a section like this:

> **Pixel QA is not cosmetic.** Agent-generated UI code can pass type checks and functional tests while breaking spacing, hierarchy, responsiveness, overflow, loading states, dark mode, localization, accessibility, and brand-critical flows. Human reviewers are bad at repeatedly inspecting every screen after high-velocity agent edits. Therefore, agent-native applications need automated pixel and journey proof.

Required pixel-QA layers:

| Layer                        | Purpose                                                       | Required control                                                                           |
| ---------------------------- | ------------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| Component visual snapshots   | Catch visual regressions in isolated UI states.               | Storybook/Chromatic or Playwright component screenshot baselines.                          |
| Route-level screenshots      | Catch layout, spacing, overflow, responsive breakage.         | `toHaveScreenshot()` with deterministic viewport, font, clock, and masked dynamic regions. |
| Critical journey video/trace | Prove checkout, signup, auth, upload, settings, billing, etc. | Playwright trace/video retained as CI artifact and repair evidence.                        |
| ARIA snapshots               | Catch semantic/accessibility regressions invisible to pixels. | `toMatchAriaSnapshot()` for core flows.                                                    |
| Cross-browser/device matrix  | Catch rendering differences.                                  | Chromium/WebKit/Firefox for core flows; mobile viewport for high-use journeys.             |
| Visual diff governance       | Avoid approving accidental drift.                             | Baseline updates require explicit PR label, reviewer, and design intent note.              |
| Dynamic content masking      | Reduce flake.                                                 | Mask timestamps, random IDs, ads, skeleton shimmer, cursor, animations.                    |
| Design-token enforcement     | Prevent arbitrary agent styling.                              | Agents must use tokens; raw colors/spacing outside token files fail audit.                 |

Playwright’s visual comparison docs support screenshot baselines and comparison on later runs; its ARIA snapshot docs support accessibility-tree snapshots for structure and semantic consistency. ([Playwright][12])

### Future CI and coverage

The paper should say coverage will explode, and that is both good and dangerous.

**Good:**

* Agents can generate unit tests, property tests, integration tests, and UI journeys faster than humans.
* Every bug fix can produce a regression test.
* Every production incident can create a repair receipt and replay test.
* CI can become an agent repair router rather than a pass/fail wall.

**Bad:**

* Agents can generate tests that encode the bug.
* Coverage numbers can rise while real risk remains uncovered.
* Screenshot tests can become flaky and expensive.
* Test suites can become too large for fast repair loops.
* Duplicate tests can bury signal.
* Overbroad mocks can prove fake systems.
* Long CI can push teams back into bypass behavior.

Required controls:

* `test-map.json` routes changed files to the smallest credible proof lane.
* CI must emit machine-readable failures.
* Every test must declare risk covered: invariant, boundary, workflow, visual, security, data, performance.
* Flaky tests have an owner and expiration.
* Generated tests require mutation/fault sanity checks for critical code.
* Coverage is not a score by itself; coverage must be paired with mutation score, path risk, and incident history.

## jankurai must become the “centerline” mechanism

You mention the need to keep projects from drifting off course. Make that precise.

Define the **agent-native centerline** as:

* Target stack compliance.
* Correct ownership cell.
* No unauthorized dependency edges.
* Generated contracts only.
* Proof lane mapped to every changed path.
* No unjustified dead-language markers.
* Security gates active.
* Pixel QA active for user-facing surfaces.
* Structured exceptions and observability.
* Bounded Python and bounded polyglot surface.
* Machine-readable repair evidence.

Then define a **centerline drift score**:

| Drift category      | Example                                                  | Measurement                              |
| ------------------- | -------------------------------------------------------- | ---------------------------------------- |
| Ownership drift     | SQL in UI, authz in handler, invariant in adapter.       | Forbidden import/access edges.           |
| Contract drift      | Handwritten DTO, fetch wrapper, generated file mutation. | Generated-zone diff and schema mismatch. |
| Proof drift         | Changed path has no mapped proof lane.                   | `test-map.json` coverage.                |
| Security drift      | Missing secret/SCA/SBOM/provenance gate.                 | CI policy check.                         |
| Language drift      | New runtime or Python owns product truth.                | Runtime inventory.                       |
| Semantic drift      | TODO/fallback/stub/temporary markers.                    | Marker scan with exception matching.     |
| UI drift            | No screenshot/ARIA proof for changed app route.          | Pixel-QA map.                            |
| Observability drift | Error lacks code/correlation/trace.                      | Error schema scan and trace tests.       |

The paper should say: **jankurai is not a style linter. It is the steering system for automated software creation.**

## Agent-first design needs examples

The paper currently tells readers what should exist, but it needs small examples that work across AI coding tools.

Add examples like:

### Example 1: wrong-layer repair

Bad agent patch:

```ts
// apps/web/src/routes/Checkout.tsx
if (cart.total > user.creditLimit) {
  throw new Error("Payment blocked");
}
```

jankurai finding:

```text
HL-OWNERSHIP-004: Durable payment invariant found in UI layer.
Owner cell: crates/domain or crates/application.
Required proof: domain property test + application authz workflow test.
```

Correct repair:

* Move invariant to Rust domain/application.
* Expose result through generated contract.
* Add UI display only.
* Add DB constraint if durable data invariant exists.

### Example 2: dead-language marker

Bad:

```rust
// temporary fallback until billing is fixed
```

jankurai:

```text
HL-DEADLANG-002: Runtime code contains future-hostile marker "temporary fallback".
Required action: delete, implement, or create scoped exception with expiration.
```

### Example 3: pixel QA

Bad:

```text
Agent changes pricing-card spacing and button hierarchy.
Typecheck passes. E2E click path passes.
Visual diff fails on mobile viewport.
```

Required repair:

* Use design tokens.
* Update component state.
* Re-run route screenshot and ARIA snapshot.
* Attach trace/video artifact.

These examples will make the standard feel real.

## References and evidence concerns

The paper uses many good sources, but it needs more careful wording around them.

Critical fixes:

* Do not say GitHub proves TypeScript is globally the most-used language. Say GitHub reports TypeScript became the most-used language on GitHub by monthly contributors in August 2025. ([The GitHub Blog][2])
* Do not use Veracode’s GenAI security data to claim Rust is secure unless Rust was included in that evaluation. Use it to prove generated code is risky generally. ([Veracode][15])
* Do not use NSA/CISA memory-safe guidance as Rust-only evidence. It supports memory-safe languages broadly; Rust needs additional argument. ([NSA][5])
* Do not lean too heavily on tool version announcements like TypeScript 7 or Vite 8. Those age quickly. The standard should depend on structural properties, not release hype.
* Archive volatile tool docs. Agent-tool docs change fast.
* Add DOIs where possible.
* Fix capitalization in references: GitHub, TypeScript, OpenAPI, PostgreSQL, OpenTelemetry, GitGuardian, DORA, Playwright.

## The conclusion needs more force but less slogan

The current conclusion is good, but it should end with compliance language.

Stronger ending:

> Agent-native engineering is not a request for developers to trust AI more. It is a demand that repositories become strict enough for automation to be safe. The future stack is not chosen by taste, familiarity, or local heroics. It is chosen by how quickly it rejects false plausibility, routes repairs, protects durable truth, preserves security, and proves user-visible behavior.
>
> The standard proposed here makes the repository the control surface: ownership maps, generated contracts, proof lanes, pixel QA, exception records, observability, provenance, and jankurai audit. Once those controls exist, agents can generate and repair code at speeds human-centered repositories cannot absorb.
>
> The mandate is simple: automate code production, but mechanize verification. Anything less is unmanaged vibe debt.

That is the landmark claim.
