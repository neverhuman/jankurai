## Overall verdict

The paper has the raw ingredients for a much stronger submission, but it is not yet shaped like a tier-1 paper. Right now it is **too broad, too table-heavy, too rhetorically repetitive, and too float-constrained**. The strongest version is not “a giant paper about vibe coding, language history, UX QA, security, waivers, language rankings, public repo scoring, and toolchain bad behavior.” The strongest version is:

**Jankurai: a repository-local conformance model that binds changed paths, owners, proof lanes, receipts, missing evidence, waivers, and repair queues into a reproducible merge witness for AI-assisted pull requests.**

Everything that does not directly prove that sentence should either be shortened, moved to appendix/supplement, or turned into implementation documentation.

3. **Reduce the main paper to 6–8 core visuals/tables.** The current draft has far too many tables for a research paper.

## The biggest content problem: the paper is over-scoped

The core paper wants to prove one thing: **a repository can carry merge evidence for AI-assisted code.** That is strong. But the current paper also tries to be:

* a history of programming-language evolution,
* a critique of vibe coding,
* a conformance standard,
* a security standard,
* a UX evidence standard,
* a waiver governance manual,
* a public-repository scoring report,
* a language/toolchain bad-behavior catalog,
* a Rust/TypeScript/PostgreSQL reference-profile argument,
* and a CLI operations manual.

That breadth is why the paper feels clunky. A tier-1 reviewer will ask: “What is the central research contribution, and what evidence supports it?” The answer is present, but buried.

### Keep this as the main contribution

The strongest contribution is the **merge witness**:

> changed paths + owner route + required proof receipts + observed evidence + missing evidence + artifact digests + commit/tool identity + decision.

This is crisp, novel enough to motivate a paper, and concrete enough to evaluate. The abstract already states it, and Figure 2 shows it, but the draft delays the technical center too long.

### Move these out of the main paper

The following sections should become appendix, companion standard, or supplement:

* most of **Section XIV–XVIII** on language proof-cost, standard gravity, reference-profile scoring, profile comparison, and reference architecture;
* most of **Section XIX** on toolchain bad behavior;
* the full **Appendix F public repo table**;
* the full **Appendix G language matrix**;
* most waiver templates and command maps;
* large registry tables that belong in the standard, not the research narrative.

The stack/ranking material is especially risky. The paper repeatedly says Jankurai is stack-neutral, but then gives score bands where “Rust core + TypeScript/React/Vite + PostgreSQL” scores 91–97 and Rails/Python/FastAPI/Elixir profiles score lower. That will make reviewers argue about the scoring model rather than the merge-witness idea. Keep the reference profile in a **non-normative implementation appendix**, not in the main argument.

## Abstract feedback

The abstract has the right ingredients but is too long and tries to do too much. It currently contains problem framing, definitions, standard claims, artifact claims, scope limitations, and contribution list. It reads more like an executive summary than a research abstract.

### What works

The first sentence is strong:

> “AI-assisted coding makes plausible code cheap and vibe-code drift expensive.”

That is memorable. The phrase “evidence debt” is also good. The “proof means repository-local evidence receipts, not formal proof” caveat is necessary and should stay.

### What weakens it

The abstract repeats several terms before the reader knows why they matter: “vibe artifacts,” “evidence receipts,” “repair queues,” “merge witness,” “proof lanes,” “generated-zone policy,” “bounded authority.” The density is impressive but not digestible.

### Suggested abstract shape

Use a structured abstract in one paragraph:

1. **Problem:** AI-generated PRs are plausible but often lack auditable merge evidence.
2. **Approach:** Jankurai models the repository as the alignment layer.
3. **Artifact:** The merge witness binds changed paths to owners, proof lanes, receipts, digests, and decisions.
4. **Implementation/evidence:** Reference auditor, schema bundle, seed suite, public readiness scan.
5. **Limitations:** Not full semantic proof; stack-neutral; scores are not approval.

A tighter version could be:

> AI-assisted coding reduces the cost of plausible edits but increases the cost of determining whether a pull request is safe to merge. We propose Jankurai, a repository-local conformance model for proof-carrying pull requests. Jankurai maps changed paths to owners, generated-boundary policy, proof lanes, evidence receipts, waivers, and repair queues, then binds those artifacts into a versioned merge witness. The witness records the current commit, required and observed receipts, missing evidence, artifact digests, tool identity, score deltas, and a closed merge decision. We implement an early reference auditor and report conformance-oriented evidence from schema-valid artifacts, a seed fixture suite, and an advisory scan of public repositories. Jankurai does not prove full program semantics and does not require a particular implementation stack; it defines the repository interface needed to turn plausible AI-assisted changes into auditable merge decisions.

That is more tier-1: specific, scoped, and evidence-oriented.

## Raw LaTeX/source fixes

### 1. Remove `\raggedbottom`

The frontmatter source uses `\raggedbottom`. That tells LaTeX not to stretch pages/columns to equal height, which makes float-heavy IEEE layouts look uneven. For this paper, it amplifies the “clunky white area” feeling. ([GitHub][5])

Use:

```tex
\flushbottom
```

Also consider:

```tex
\clubpenalty=10000
\widowpenalty=10000
\displaywidowpenalty=10000
```

### 2. Reduce `\FloatBarrier`

The main source controller uses repeated `\FloatBarrier`s between sections. That is a primary reason pages get stranded with white space: LaTeX is forced to flush pending floats before it can continue. ([GitHub][6])

Do not place barriers after routine sections. Use them only:

* before appendices,
* before bibliography if absolutely necessary,
* after a major part boundary where float leakage would be confusing.

Suggested controller shape:

```tex
\input{paper/tex/frontmatter}

\input{paper/tex/sections/01_introduction}
\input{paper/tex/sections/02_model}
\input{paper/tex/sections/03_implementation}
\input{paper/tex/sections/04_evaluation}
\input{paper/tex/sections/05_discussion}
\input{paper/tex/sections/06_related_work}
\input{paper/tex/sections/07_limitations_conclusion}

\FloatBarrier
\bibliographystyle{IEEEtran}
\bibliography{paper/references}

\clearpage
\appendices
\onecolumn
\input{paper/tex/appendices/a_rule_ids}
...
```

For a conference submission, I would remove appendices from the submitted PDF and provide them as supplemental material.

### 3. Stop using so many full-width floats

The paper uses many `table*` and `figure*` blocks. In two-column LaTeX, full-width floats usually appear only at the top of a later page. If you combine that with `\FloatBarrier`, you get exactly the white areas visible on pages 9–12 and 25–26.

Use `table*` only for:

* merge witness schema excerpt,
* one major evaluation table,
* one major related-work/comparison table if needed.

Everything else should be:

* a one-column `table`,
* a shortened table,
* a figure,
* or appendix material.

### 4. Relax float placement

Many floats are `[t]`. Use `[tbp]` or `[!t]` selectively. For float-heavy papers:

```tex
\renewcommand{\topfraction}{0.95}
\renewcommand{\dbltopfraction}{0.95}
\renewcommand{\textfraction}{0.03}
\renewcommand{\floatpagefraction}{0.80}
\renewcommand{\dblfloatpagefraction}{0.80}
```

This gives LaTeX more legal placements and reduces float-only/half-empty pages.

### 5. Move references before appendices

The current order forces appendices into one column, then clears the page and switches back to two columns for references. That creates the page 35 blank. If the venue allows it, place references before appendices. If the venue requires references last, move appendices to supplementary material.

### 6. Reformat source files

The raw source is very hard to review because many files are effectively compressed into very long lines. The controller file is a single line, and section files pack long prose/tables into dense lines. That makes diffs, comments, and layout debugging painful. ([GitHub][6])

Run `latexindent` and enforce:

* one sentence or clause per source line,
* one table per file for very large tables,
* one `\input` per line,
* macros in `paper/tex/macros.tex`,
* styling in `paper/tex/style.tex`,
* generated tables isolated under `paper/tex/generated/`.

That will not directly reduce PDF white space, but it will make the paper much easier to edit into shape.

* adapter drift fail,
* clean waiver pass/review case.

### Add false-positive validation for public scans

For the public repository scan, manually label a random sample of findings:

* true missing evidence under Jankurai,
* false positive due heuristic,
* not applicable to repo,
* locally acceptable tradeoff,
* needs owner context.

Even 200–300 manually labeled findings would dramatically improve credibility.

### Separate adoption from quality

Public scoring should distinguish:

1. **Jankurai adoption missing** — no owner map, no test map, no witness.
2. **Generic evidence missing** — no security scan, no generated-boundary evidence.
3. **Potential code issue** — secret-like value, destructive migration, unsafe boundary.

Otherwise, the top-level score will look like it punishes projects for not already using your standard.

## Tone and writing style

The writing is energetic, which is good, but it sometimes sounds like a manifesto. Tier-1 papers need more measured claims.

### Words/phrases to reduce

* “vibe” repeated too often,
* “not an accident,”
* “the correct response,”
* “insult into audit surface,”
* “weak implementation agent,”
* “No proof, no merge. No receipt, no trust.”

Keep “vibe artifact” as the named concept, but once defined, use more precise terms:

* “unauditable change,”
* “missing evidence,”
* “unrouted proof,”
* “unreceipted claim,”
* “evidence gap.”

### Replace prescriptive language

Instead of:

> “The correct response is to redesign the repository as the alignment layer.”

Use:

> “We argue that the repository should serve as the alignment layer.”

Instead of:

> “A patch is still a vibe.”

Use:

> “The change remains unauditable under the repository’s conformance policy.”

You can keep one or two punchy lines, but not every section should end with a slogan.

## Proposed new paper structure

Here is the structure I would use for a tier-1 submission.

### 1. Introduction

* Start with checkout PR.
* Show why normal review/CI is insufficient.
* Define evidence debt.
* State contributions.
* State non-goals.

### 2. Model: continuous repository alignment

* Changed paths.
* Owner route.
* Proof lanes.
* Generated zones.
* Receipts.
* Merge witness.
* Decision enum.

### 3. Reference implementation

* CLI/auditor.
* Schemas.
* Artifact outputs.
* Receipt freshness algorithm.
* Repair queue.

### 4. Evaluation

* Conformance fixtures with observed results.
* Self-audit.
* Public readiness scan aggregate.
* Runtime/overhead.
* False-positive analysis.

### 5. Case study: checkout PR

Either keep the checkout example as a running example throughout or give it a dedicated case-study section showing blocked→repaired→pass.

### 6. Discussion

* Agent adapters.
* UX evidence.
* Security/permissions.
* Waivers.
* Adoption.

### 7. Related work

* Technical debt/refactoring.
* Agent benchmarks.
* Supply chain/provenance.
* Policy-as-code.
* UI/accessibility evidence.
* Standards/conformance.

### 8. Limitations and conclusion

* Scope.
* Threats to validity.
* Future work.
* Final claim.

### Appendices/supplement

* Full rule registry.
* Full glossary.
* Full public repo rankings.
* Reference profile.
* Toolchain bad-behavior matrix.
* Waiver templates.
* Command map.
* File tree diagrams.

## A practical “next draft” target

Aim for:

* **10–12 pages main paper** if targeting a conference-style venue;
* **18–22 pages** if targeting arXiv/technical report;
* **6–8 main visuals/tables**;
* appendices as separate supplement;
* observed fixture results, not expected-only results;
* public repo scan as aggregate readiness evidence, not named quality ranking;
* one running example from beginning to end.

The exciting thing is that the paper does have a real spine: **merge witnesses for proof-carrying AI pull requests**. If you cut the stack debates, registry bulk, and standards-manual tables from the main body, the paper will read much more like a serious research artifact and much less like a draft spec with a paper wrapped around it.
