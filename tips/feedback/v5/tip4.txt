I reviewed the uploaded 37-page PDF and the GitHub paper source. The core idea is genuinely strong: **“merge evidence” as the bottleneck for AI-assisted PRs** is a compelling thesis, and the “merge witness” is the clearest candidate for a tier-1 contribution. The current draft, though, reads like **three documents collapsed into one**: a research paper, a standards specification, and an adoption/manual/reference-profile report. That is the main reason it feels clunky and why the white space is so noticeable. The paper has a lot of material, but not yet the right kind of concentrated data story for a top venue. 

The GitHub source confirms this: `paper/jankurai.tex` is a wrapper that pulls many section files, inserts repeated `\FloatBarrier`s, switches to `\onecolumn` for appendices, then switches back to `\twocolumn` for references; the front matter defines a full title page, custom footer, table styling, logo/header block, and uses `\raggedbottom`. Those source choices are directly responsible for much of the accidental white space. ([GitHub][1])

## Overall verdict

This should become a much stronger paper, but the current draft is not yet tier-1 shaped.

The **best paper hiding inside this draft** is:

> **Jankurai: Repository-Local Evidence for Proof-Carrying Pull Requests in AI-Assisted Software Development**

The current version instead tries to prove all of these at once:

1. AI coding creates a new “evidence debt” problem.
2. Jankurai is a standard for repository conformance.
3. Jankurai has a rule taxonomy.
4. Jankurai has a reference implementation.
5. Jankurai has a public repository scoring system.
6. Jankurai recommends a Rust/TypeScript/PostgreSQL reference stack.
7. Jankurai defines agent adapters, UX evidence, security gates, waivers, repair receipts, governance, command paths, and bad-behavior matrices.

That is too much for one tier-1 paper. The concept is exciting, but reviewers will struggle to identify the single scientific contribution.

The paper should become:

**Main paper:** one crisp research artifact: repository-local merge evidence and merge witnesses.

**Supplement/spec:** rule registry, command maps, reference profile, public score tables, waiver templates, language/toolchain matrices.

**Website/standard docs:** all operational standard details.

That separation alone will fix a huge amount of the white space, visual clutter, and reader fatigue.

## What is already strong

The strongest parts are worth protecting.

**The “merge witness” is the central contribution.** It gives the paper a concrete artifact, not just a philosophy. The witness binds changed paths, ownership, proof receipts, missing evidence, artifact digests, tool identity, commit identity, score delta, and decision. That is the most publishable idea in the draft.

**The phrase “score is posture; witness is decision; conformance is pass/fail” is excellent.** It should be elevated into a boxed principle or formal invariant. This line clarifies a major distinction that many governance tools blur.

**The threat model table is valuable.** Table IV on page 5 is one of the more mature parts of the draft because it separates false plausibility, stale claims, adapter drift, flaky proof lanes, emergency bypass, evidence leakage, CI downgrade, malicious tools, generated-test self-confirmation, and review rubber-stamping.

**The running checkout example is a good anchor.** The checkout PR example on page 4 gives readers a concrete change: UI, generated client, and API contract. That should be introduced even earlier, before the long language-history section. The source explicitly has a separate `02_running_example.tex` with this checkout case and a table describing blocked-to-pass evidence flow. ([GitHub][2])

**The evaluation honesty is good.** The paper correctly says the current seed conformance lane validates fixture inventory and expected JSON presence, not observed per-fixture witness decisions. That honesty is important, but the paper must then avoid sounding like the evaluation is stronger than it is. ([GitHub][3])

## Biggest blockers to tier-1 quality

### 1. The paper is overstuffed

The draft reaches Table LII by page 35. That gives the reader the feeling of reading a standards catalog, not a focused research paper. Many tables are useful, but they do not belong in the main line.

A tier-1 paper needs a crisp spine:

1. Problem: AI-assisted PRs make plausible edits cheap but merge evidence expensive.
2. Model: changed paths → owner routes → proof lanes → receipts → witness.
3. Implementation: auditor, maps, generated zones, receipts, queue.
4. Evaluation: fixtures, case study, public scan, repair/evidence metrics.
5. Limitations and related work.

Everything else should support that spine or move out.

### 2. The paper has too many “policy claims” and too little measured evidence

There is a lot of data-shaped material, but much of it is **policy**, not empirical evidence. RPN scores, reference profile scores, hard caps, and stack rankings are policy judgments. They are useful for a standard, but tier-1 reviewers will ask: what was measured, what was compared, what improved, and against which baseline?

The source says the RPN is a Jankurai policy priority, not an empirical frequency claim, and the detector maturity table distinguishes deterministic, evidence-routed, governance-backed, and research rows. That framing is good; the paper should lean into it and stop presenting policy tables like experimental findings. ([GitHub][4])

### 3. The evaluation section currently undercuts the paper’s strongest claim

The draft says the seed suite has 10 fixture directories and 12 expected JSON files, but the command currently validates inventory and expected JSON presence rather than observed per-fixture witness decisions. That is not enough for a paper whose main artifact is conformance and witness decision-making. ([GitHub][3])

Before submission, add a real per-fixture runner and report:

| Fixture                      | Expected decision | Observed decision | Primary rule | Receipt present? | Pass/fail |
| ---------------------------- | ----------------: | ----------------: | ------------ | ---------------: | --------: |
| hl3-pass-minimal             |              pass |              pass | none         |              yes |      pass |
| generated-zone-mutation-fail |             block |             block | HLT-002      |               no |      pass |
| ownerless-path-fail          |             block |             block | HLT-003      |               no |      pass |

That one table would be much stronger than several current policy tables.

### 4. The stack-ranking/reference-profile sections distract from stack-neutrality

The paper repeatedly says Jankurai Core is stack-neutral, but then spends pages ranking Rust/TypeScript/PostgreSQL, Go, .NET, JVM, Rails, Python, and Elixir profiles. This creates a contradiction in reader perception even when the text says the ranking is non-normative.

For a tier-1 paper, move Sections XIV–XVIII into a supplement or separate position paper. Keep only one paragraph:

> Jankurai is stack-neutral. We include a non-normative reference profile in the artifact to demonstrate one possible implementation.

That is enough.

### 5. The “vibe” branding is memorable but academically risky

“Vibe coding” is a great hook, but “Anti-Vibe Coding Standard” may read as manifesto language. For a tier-1 venue, make the technical contribution lead and let “vibe” be the motivating term.

Better title options:

| Current                                                             | Stronger tier-1 version                                              |
| ------------------------------------------------------------------- | -------------------------------------------------------------------- |
| Anti-Vibe Coding Standard                                           | Jankurai: Repository-Local Evidence for Proof-Carrying Pull Requests |
| Continuous Repository Alignment for Proof-Carrying AI Pull Requests | Continuous Repository Alignment for AI-Assisted Software Development |
| Find the vibe. Prove the merge. Repair the repo.                    | Good as a tagline, not the main academic claim                       |

The phrase “vibe artifact” can stay, but define it once as **an evidence gap in a repository state**.

## Internal consistency issues to fix

These are important because tier-1 reviewers notice them quickly.

### HLT rule range mismatch

Page 6 says stable rule identifiers are **HLT-001 through HLT-027**, but later tables and appendices use **HLT-028 through HLT-035**, including Rust, SQL, TypeScript, Docker, Python, CI, and Git bad-behavior rules. Fix this everywhere.

Use one of these:

> “HLT-001 through HLT-035 are stable rule families in v0.7.0.”

or:

> “HLT-001 through HLT-027 are Core families; HLT-028 through HLT-035 are reference-profile detector families.”

Right now it reads like an accidental version drift.

### Merge witness example has a receipt contradiction

Figure 2 shows:

```json
"required_receipts": ["web", "contract", "security"],
"present_receipts": ["contract"],
"missing_evidence": [
  "contract_receipt",
  "rendered_ux_receipt"
]
```

That is confusing: if `contract` is present, why is `contract_receipt` missing? The example should distinguish **lanes** from **receipts**.

Better:

```json
"required_lanes": ["web", "contract", "security"],
"present_receipts": {
  "contract": "target/jankurai/receipts/contract.json"
},
"missing_receipts": ["web", "security"],
"missing_evidence": ["rendered_ux_receipt", "security_receipt"]
```

Or, if the contract receipt is meant to be missing, then `present_receipts` should not include `contract`.

### “Proof” still risks being misunderstood

The abstract says proof means repository-local evidence receipts, not formal proof. That is necessary, but the title phrase “Proof-Carrying AI Pull Requests” still evokes proof-carrying code and formal methods. Consider “Evidence-Carrying Pull Requests” unless you want to keep the formal-methods association.

### Public scoring median is statistically awkward

The aggregate table says “median score (upper middle) 34” for 30 repositories. With an even number of samples, standard median is usually the average of the 15th and 16th ordered values. In the full table, those appear to be 34 and 32, so the conventional median would be 33. Report either:

> median = 33.0

or:

> lower/upper middle = 32/34

Avoid “upper middle” unless the methodology section defines why.

### Public repository scoring needs commit hashes

The paper says the May 5, 2026 run scanned 30 public repositories, succeeded on all 30, top score was 48, average was 33.4, and emitted 15,391 findings. That is interesting, but for a research paper each repo row needs commit SHA, scan command, policy version, auditor version, and false-positive sampling method. ([GitHub][5])

Otherwise reviewers may treat the public scan as anecdotal.

## Content restructuring recommendation

The main paper should be much shorter and sharper. I would restructure it like this.

### Proposed main paper outline

**Abstract**
One paragraph, 180–230 words. No long list of every artifact.

**1. Introduction**
Start with checkout PR. Then generalize. Cut most language-history material or move it later.

**2. Problem and Threat Model**
Define evidence debt, vibe artifact, proof receipt, merge witness. Include a compact threat model.

Formalize the witness:

```text
W = (Δ, O, G, P_req, R_valid, E_missing, D, I_commit, I_tool, waiver_state, decision)
```

* `Δ` = changed paths
* `O` = owner route
* `G` = generated-zone status

3. Public repository advisory scan with methodology.
4. One agent repair case study.
5. Performance/runtime overhead.
6. False-positive sample.

**6. Discussion**
Where Jankurai helps, where it fails, why it is stack-neutral.

**7. Related Work**

**8. Limitations and Conclusion**

### Move these to supplement/spec

* Full HLT registry.
* Full public repo score table.
* Language bad-behavior matrix.
* Waiver templates.
* File tree diagrams.
* Golden command path.
* Reference profile score/ranking.
* Detailed governance process.
* Long stack-comparison sections.

This would make the paper feel serious immediately.

## Revised abstract direction

The current abstract is too long and too list-heavy. It reads like a standards document summary. Here is a tighter direction:

> AI coding assistants make plausible repository patches cheap; they do not make ownership, generated boundaries, proof freshness, security evidence, or review claims auditable. We present Jankurai, a repository-local conformance model for evidence-carrying pull requests. Jankurai maps changed paths to owner routes, generated-zone policy, proof lanes, receipts, waivers, and a merge witness: a versioned artifact binding paths, required evidence, observed receipts, digests, tool and commit identity, missing-evidence decisions, and a closed merge decision. We define rule families for common evidence gaps, including ownerless paths, unmapped proof, hand-edited generated outputs, false-green tests, secret sprawl, overbroad agent permissions, unproven UI changes, stale waivers, and unreceipted review. An early reference implementation emits JSON/Markdown reports, proof plans, repair queues, and seed conformance fixtures. We evaluate the current artifact as a conformance proposal, not a semantic verifier: current fixture validation is inventory-level, while a 30-repository advisory scan suggests conventional CI often lacks merge-witness surfaces. Jankurai does not prove full program correctness; it makes merge evidence explicit, replayable, and repairable.

That abstract is less flashy but much more reviewer-friendly.

## Layout and white-space diagnosis

The white space is not just aesthetic. It is caused by structural LaTeX choices.

### Root cause 1: full custom title page

Page 1 has a logo, metadata block, long abstract, and then a huge lower blank area. The source uses a `titlepage`, custom header graphic, minipage abstract, multiple `\vspace`s, and `\vfill`. ([GitHub][6])

For a tier-1 paper, remove the title page. Use the normal conference first-page layout. The first page should contain title, authors, abstract, and the beginning of the introduction.

Fix:

```tex
% Do not use:
\begin{titlepage}
...
\vfill
\end{titlepage}

% Prefer standard conference flow:
\title{Jankurai: Repository-Local Evidence for Proof-Carrying Pull Requests}
\author{...}
\maketitle
\begin{abstract}
...
\end{abstract}
```

Also remove the mascot/logo from the paper submission. It is fine for a project website or technical report, but it makes the paper look like a branded whitepaper.

### Root cause 2: `\raggedbottom`

The source sets `\raggedbottom`, which lets pages end at uneven heights. That can be useful while drafting, but here it preserves visible dead space at page bottoms. ([GitHub][6])

For camera-ready style, test:

```tex
\flushbottom
```

This will not solve everything, but it reduces the “random blank bottom” feeling.

### Root cause 3: repeated `\FloatBarrier`

The main wrapper includes repeated `\FloatBarrier`s. These force LaTeX to dump pending floats before continuing. In a table-heavy two-column paper, that creates half-empty pages and float-only pages. ([GitHub][7])

Current pattern:

```tex
\input{...}
\FloatBarrier
\input{...}
\FloatBarrier
```

Better:

```tex
% Use FloatBarrier only before major structural transitions:
% - before appendices
% - before references, maybe
% - before a section only when absolutely necessary
```

Most body sections should not be separated by float barriers.

### Root cause 4: too many `table*` and `figure*` floats

The source uses full-width floats for tables and figures, including the alignment loop, the taxonomy table, the RPN chart, the expected fixture table, and metrics table. Full-width floats in two-column papers are hard to place; they often float to the top of the next page and leave holes. The intro source uses a full-width `figure*` for the alignment loop, and the evaluation source uses full-width `table*` for fixture decisions and metrics. ([GitHub][8])

Use full-width floats only for the few figures that truly need the full page width.

### Root cause 5: appendices are forced into `onecolumn`

The wrapper switches to `\onecolumn` for appendices and later back to `\twocolumn` for references. That makes pages 30–35 feel like a manual pasted into the paper. It also contributes to big blank areas before references. ([GitHub][7])

Move most appendices to a supplement. Keep the main paper in one consistent format.

```tex
\setlength{\textfloatsep}{8pt plus 2pt minus 2pt}
\setlength{\dbltextfloatsep}{8pt plus 2pt minus 2pt}
```

The source is not just a rendering problem. It also needs to be made easier to review.

### Hard-wrap the TeX source

Many source files put whole paragraphs, tables, or sections on very long lines. For example, `01_new_bottleneck.tex` has dense section content and table code inline across long lines, and the running-example source similarly embeds the full table in a compact block. ([GitHub][8])

Use:

```bash
latexindent -w paper/tex/**/*.tex
```

Or manually format:

```tex
\section{Running Example: Checkout PR}

The checkout-flow pull request is intentionally ordinary.
It changes a page a customer can see, a generated client the page imports,
and the API contract that should own the generated shape.

\begin{table}[t]
  \caption{Running checkout example from blocked witness to pass shape.}
  ...
\end{table}
```

This makes diffs reviewable and aligns with the paper’s own philosophy: the source should be agent-readable and repairable.

### Split generated tables from prose

Create:

```text
paper/tex/tables/
  conformance_levels.tex
  artifact_model.tex
  public_scan_top.tex
  fixture_results.tex

paper/tex/figures/
  alignment_loop.tex
  rpn_distribution.tex
```

Then section files become readable.

### Rename “exception” files to “waiver”

The body carefully distinguishes runtime exceptions from standards waivers, but source names such as `c_exception_template` and `14_exceptions_repair` undermine that distinction. Rename to:

```text
14_waivers_repair.tex
c_waiver_template.tex
```

This small source cleanup improves conceptual consistency.

## Specific content edits by section

### Abstract

Cut by 35–45%. It currently tries to name every artifact. A strong abstract should establish:

1. Problem.
2. Model.
3. Artifact.
4. Implementation.
5. Evaluation status.
6. Limitation.

Do not include long lists of artifacts in the abstract.

### Section I: From Language Chaos to Verified Merge

This section is thoughtful, but too long before the reader knows what Jankurai actually does.

Keep only:

* AI makes plausible edits cheap.
* Repository intent remains hidden.
* Merge evidence is the new bottleneck.
* Checkout PR example.

Move most language-history discussion to related work or background.

### Section II: Running Example

This should become the opening hook. The checkout PR is clear and practical. Put it on page 1 or early page 2.

### Section III: Definitions and Threat Model

Table III is too large. Keep only six terms in the main paper:

* Vibe artifact.
* Evidence debt.
* Proof receipt.
* Generated zone.
* Repair queue.
* Merge witness.

Move the rest to a glossary.

### Section IV: Jankurai Core Standard and Conformance

This is core. Strengthen it by adding formal pseudocode:

```text
Algorithm 1: BuildMergeWitness(change Δ, policy P)
1. paths ← changedPaths(Δ)
2. owners ← routeOwners(paths, P.ownerMap)
3. zones ← detectGeneratedTouches(paths, P.generatedZones)
4. lanes ← requiredProofLanes(paths, P.testMap, zones)
5. receipts ← validReceipts(lanes, Δ.head, P)
6. missing ← lanes - receipts
7. decision ← gate(missing, findings, waivers, ratchet, releasePolicy)
8. return witness(paths, owners, zones, lanes, receipts, missing, decision)
```

That would make the standard feel like a research contribution.

### Section V: Taxonomy

Move the full Table XII to appendix. In the main paper, show the top 8–10 rule families and one chart.

Also clarify:

> “RPN is a prioritization prior used to order implementation work; it is not calibrated incident frequency.”

### Section VI: Evaluation

This should be the strongest section. Right now it is honest but too preliminary. Finish the per-fixture runner before submission. The paper’s own source says the current validation is inventory-only and observed decision comparison is future work. ([GitHub][3]) That is acceptable for a draft, but not for a tier-1 claim.

### Section VII: Public Repository Scoring

Keep it, but make it exploratory and methodologically careful. The most defensible claim is:

> Conventional CI and popularity do not imply merge-witness-ready evidence surfaces.

Do not imply those repositories are low quality.

### Sections XIV–XVIII: Language/reference profile

Move almost all of this out. It is interesting, but it shifts the paper from “new merge-evidence standard” into “stack philosophy.” That weakens the central contribution.

### Section XIX: Toolchain bad behavior

Move to appendix/spec. In the main paper, keep a short paragraph:

> Escape hatches in Rust, SQL, TypeScript, Python, Docker, CI, and Git are not rejected because of taste; they are rejected when they move trust from machine-checkable boundaries into unrecorded assumptions.

That sentence captures the whole section.

### Related Work

The related-work table is helpful, but the paper needs prose synthesis. Add stronger comparisons to:

* CODEOWNERS-style ownership.
* CI/CD pipelines.
* OpenSSF Scorecard.
* SLSA/SPDX/provenance.
* SARIF/JUnit reports.
* Proof-carrying code.
* Technical debt/refactoring.
* SWE-bench/SWE-agent/OpenHands/aider.

The key distinction should be:

> Existing tools produce signals; Jankurai binds signals to a changed-path merge decision.

That is the novelty.

## Visual design improvements

### Reduce color shading

The light blue table headers are readable, but the paper uses a lot of colored table shading. In print or grayscale, this can feel heavy. Use fewer shaded rows. For main paper tables, prefer `booktabs` with minimal shading.

### Reduce custom footer

The footer repeats:

> Jankurai 0.7.0 public_draft • schema 1.5.0 • paper 2026.05-ed7 • p. X

That is useful for a standard draft, but it makes the paper look nonstandard. For submission, remove it or move version details to an artifact appendix.

### Avoid code boxes in the body unless essential

The JSON witness excerpt is useful, but several later code boxes are operational documentation. Keep one compact witness excerpt in the body. Move command examples and templates to supplement.

### Make diagrams do more work

Right now the paper leans too heavily on tables. Add 3–4 strong figures:

1. Jankurai merge-witness pipeline.
2. Witness data model.
3. Evaluation result dashboard.
4. Public scan score distribution.

Good figures will make the paper feel less clunky while also reducing table overload.

## The version I would aim for

A strong tier-1 version would be around **12–14 pages main text** with a supplement. It would have:

* 1 sharp title.
* 1 short abstract.
* 1 running example.
* 1 formal witness model.
* 1 algorithm.
* 4–5 main tables.
* 4 strong figures.
* 1 real evaluation section.
* A clear artifact package.

The current draft has the raw material. The big move is to stop making the paper carry the whole standard. Let the paper sell the central idea; let the standard carry the operational detail.

The highest-return edit is this: **delete or move 35–45% of the current main-body material, then replace the recovered space with real evaluation figures and observed witness results.** That will simultaneously fix the white space, improve the read, and make the paper feel much closer to a tier-1 submission.
