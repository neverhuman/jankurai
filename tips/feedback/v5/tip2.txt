I studied the uploaded PDF and checked the GitHub paper source. The big picture is exciting: **“repository-local merge evidence for AI-assisted PRs” is a strong, timely thesis.** The current draft has the bones of a serious paper, especially the merge-witness concept, receipt freshness, proof lanes, repair queues, and the honest limitation that this is not full semantic proof. But right now it reads like **three documents fused together**: a research paper, a standards manual, and a stack/opinion whitepaper. That fusion is the source of both the content clunkiness and the ugly white-space behavior. 

The source confirms the layout problem is not random. The paper directory contains a generated PDF plus TeX source and modular section files; the wrapper `paper/jankurai.tex` is a one-line file that inputs many sections, inserts several `\FloatBarrier`s, then switches to `\onecolumn` for appendices. ([GitHub][1]) The frontmatter also uses a custom `titlepage`, `\vfill`, and `\raggedbottom`, which directly contributes to the title-page void and uneven page bottoms. ([GitHub][2])

# Executive verdict

**This should become two artifacts:**

1. **Tier-1 research paper:** focused, data-heavy, 10-12 pages, centered on the question:
   **Can repository-local evidence artifacts make AI-assisted pull requests auditable and repairable?**

2. **Jankurai standard/specification:** longer, normative, table-heavy, with full HLT registry, waiver templates, source manifests, stack guidance, CLI examples, and conformance details.

Right now the paper is trying to be both. That makes the draft feel crowded in content but strangely empty on the page. The main paper has too many large policy tables and too little experimentally validated evidence. The layout has too much white space because large floats are being forced into awkward pages instead of being cut, moved, split, or integrated with prose.

**Do this instead:**

Use a normal IEEE-style title/abstract block or a compact custom top block, then let Section I start on page 1. Keep the logo only if it is extremely small or move it to an artifact badge/appendix. The paper should not open like a product brochure.

## 2. Remove most `\FloatBarrier`s

The wrapper currently inserts `\FloatBarrier` between major section inputs. That is almost certainly causing several float-only or half-empty pages. `\FloatBarrier` is useful when you have one local float that must not escape a section; it is destructive when used as a global page-flow strategy.

**Bad current pattern:**

```tex
\input{...}
\FloatBarrier
\input{...}
\FloatBarrier
```

**Better pattern:**

Use barriers only before a major conceptual reset, and only after all floats in that cluster have been placed. For the main paper, remove most of them. Let LaTeX place floats naturally at the top of the next column/page.

## 3. Cut the main paper tables by at least 50%

The paper has great ingredients, but too many tables are not “data”; they are specification material. A tier-1 paper should keep the tables that support the scientific claim and push the rest to appendices or the standard.

**Move to appendix/spec:**

Full HLT registry, full language/toolchain bad-behavior matrix, stack reference scoring, version manifesto, waiver templates, full public repo table, expanded file tree diagrams, command path, and most conformance checklist tables.

## 4. Move the stack-ranking/reference-profile material out of the main paper

Sections XIV-XVIII are the biggest content distraction. The paper repeatedly says Jankurai Core is stack-neutral, then spends several pages ranking Rust/TypeScript/PostgreSQL and other stacks. Even if the scoring is useful for your project, reviewers will read this as subjective and potentially ideological.

Keep only one paragraph in the main paper:

> Jankurai Core is stack-neutral. The artifact boundary is owner routing, proof receipts, generated-zone evidence, security/UX evidence, and merge witnesses. A non-normative reference profile is provided in the companion standard to illustrate one implementation path, but conformance is not tied to Rust, TypeScript, PostgreSQL, or any specific stack.

Move the score bands and reference architecture cells to a companion standard or appendix.

## Title and framing

## Abstract

The abstract is currently too long and a little repetitive. It says the same core idea several times: plausible AI code is cheap; repository evidence is missing; Jankurai binds ownership, proof lanes, receipts, and merge decisions.

A tier-1 abstract should be shorter, sharper, and more empirical. It needs: problem, method, artifact, evaluation, result, limitation.

Suggested replacement:

> AI-assisted coding makes plausible pull requests cheap, but leaves reviewers with a harder question: what evidence makes a change safe to merge? We introduce Jankurai, a repository-local conformance model for AI-assisted software changes. Jankurai maps changed paths to owners, generated-boundary policy, proof lanes, evidence receipts, waivers, repair queues, and a merge witness: a versioned artifact binding changed paths, required receipts, observed evidence, missing-evidence decisions, artifact digests, tool identity, and commit identity. The standard is stack-neutral and treats “proof” as auditable repository-local evidence, not full semantic verification. We implement an early reference auditor and evaluate it on a seed conformance suite plus an advisory scan of public repositories. The results show that conventional CI and repository popularity do not necessarily expose merge-witness-ready evidence surfaces. Jankurai’s contribution is a concrete interface for turning plausible AI- or human-authored patches into auditable, repairable merge decisions. It does not prove full program correctness and does not require a specific language stack.

That is still strong, but less manifesto-like.

## Introduction

The introduction has a compelling historical arc, but it takes too long to reach the actual contribution. Page 2 spends a lot of space on language history: assembly, C, managed runtimes, scripting, JavaScript, TypeScript, Julia, Elm, F#, Clojure, Haskell, Scala, Elixir, Nim, Crystal, and D. That material is interesting, but it delays the core research problem.

> An AI agent opens a pull request that changes `Checkout.tsx`, a generated client, and an OpenAPI contract. The diff compiles. The tests are green. Should it merge?

Then ask:

> What repository-local evidence should be required before merge?

After that, introduce the thesis:

> Jankurai treats the repository as the alignment layer for human- and agent-authored code.

The language-history material can be compressed to one paragraph or moved to related work/discussion.

## Evaluation

This is the biggest gap.

The paper currently says Jankurai is best understood as a standards proposal with an early reference implementation, not a completed empirical benchmark. That is honest, but a tier-1 paper needs stronger evidence before the main claim lands. The PDF itself says the seed suite currently validates inventory and expected JSON presence and does not yet emit observed per-fixture witness decisions. 

Minimum evaluation upgrade:

1. **Per-fixture conformance runner**

   * Run all seed fixtures.
   * Emit actual witness JSON for each.
   * Compare expected vs observed.
   * Report pass/fail.

2. **Manual validation sample**

   * Take public scan findings.
   * Randomly sample, for example, 300 findings across rule families.
   * Label true positive, false positive, policy-only, requires project context.
   * Report precision by detector maturity.

3. **Agent repair experiment**

   * Give agents the same failing fixture with and without Jankurai repair queue.

4. **Ablation study**

   * Remove owner map.
   * Remove test map.
   * Remove generated-zone policy.
   * Remove witness freshness.
   * Measure missed evidence or wrong decisions.

5. **Case study**

   * Use the checkout PR from the paper.
   * Show initial block, required receipts, repair queue, regenerated client, rendered UX evidence, final pass.

# Specific LaTeX fixes

## Restore page balance

Current frontmatter uses `\raggedbottom`. For a camera-ready two-column paper, switch back to flush bottoms after fixing floats:

```tex
% Remove or comment this:
% \raggedbottom

% Prefer:
\flushbottom
```

If some pages become over-stretched, fix the float placement rather than leaving pages ragged.

## Reduce float spacing

Add conservative spacing controls:

```tex
\setlength{\textfloatsep}{8pt plus 1pt minus 2pt}
\setlength{\floatsep}{6pt plus 1pt minus 2pt}
\setlength{\intextsep}{6pt plus 1pt minus 2pt}
\setlength{\dbltextfloatsep}{8pt plus 1pt minus 2pt}
\setlength{\dblfloatsep}{6pt plus 1pt minus 2pt}
```

## Let floats share pages with text

```tex
\renewcommand{\topfraction}{0.92}
\renewcommand{\dbltopfraction}{0.92}
\renewcommand{\textfraction}{0.05}
\renewcommand{\floatpagefraction}{0.75}
\renewcommand{\dblfloatpagefraction}{0.75}
```

## Top-align unavoidable float pages

This helps pages like the current Figure 3 page, where the float is vertically centered:

```tex
\makeatletter
\setlength{\@fptop}{0pt}
\setlength{\@fpsep}{8pt plus 1fil}
\setlength{\@fpbot}{0pt plus 1fil}
\makeatother
```

This is not a substitute for cutting floats, but it makes remaining float pages less ugly.

## Use tighter table macros

Your tables are visually nice, but too tall. Current `\arraystretch` is `1.13`. For dense IEEE tables, use local tightening:

```tex
\newcommand{\JKTightTable}{%
  \footnotesize
  \setlength{\tabcolsep}{3.2pt}
  \renewcommand{\arraystretch}{1.04}
}
```

Then:

```tex
\begin{table}[!t]
\caption{...}
\JKTightTable
\begin{tabularx}{\columnwidth}{...}
...
\end{tabularx}
\end{table}
```

Do not globally shrink everything to unreadable text. Use local table tightening only.

## Stop floating appendix tables

In one-column appendices, big tables should not float above headings. Use `longtable` or plain non-floating tabular blocks after the appendix heading:

```tex
\appendix
\section{Rule IDs and Conformance Evidence}

\begin{longtable}{p{0.12\textwidth}p{0.24\textwidth}p{0.16\textwidth}p{0.18\textwidth}p{0.2\textwidth}}
...
\end{longtable}
```

This fixes the current issue where the compact rule metadata registry appears before the Appendix A heading.

## Split source files into readable lines

Both the TeX wrapper and frontmatter are hard to review because major source content is compressed into extremely long lines. The raw wrapper is one line containing all section inputs and layout transitions. ([GitHub][4]) Break it into normal lines:

```tex
\input{paper/tex/frontmatter}

\input{paper/tex/sections/01_intro}
\input{paper/tex/sections/02_model}
\input{paper/tex/sections/03_standard}
\input{paper/tex/sections/04_implementation}
\input{paper/tex/sections/05_evaluation}
\input{paper/tex/sections/06_security}
\input{paper/tex/sections/07_related_work}
\input{paper/tex/sections/08_limitations_conclusion}

\clearpage
\appendices
\input{paper/tex/appendices/a_rule_ids}
...
```

Also rename section files so numbers match paper order. Right now the source order jumps from evaluation/public scoring to agent controls, then later back to language compression and stack rubric. That is manageable for LaTeX, but bad for maintainability and agent editing.

The concept “vibe artifact” is memorable. Keep it. But the draft uses “vibe” so often that the academic reader may start to feel the paper is arguing by slogan.

Use this pattern:

* Introduce **vibe artifact** once.
* Define it operationally.
* Then mostly use **evidence gap**, **missing receipt**, **unmapped proof**, **generated-zone drift**, and **witness failure**.

Rename these:

| Current                                          | Better                                         |
| ------------------------------------------------ | ---------------------------------------------- |
| Anti-Vibe Coding Standard                        | Repository-Local Evidence Standard             |
| Vibe Coding Bad Behavior Across Toolchains       | Toolchain Escape Hatches and Required Evidence |
| Find the vibe. Prove the merge. Repair the repo. | Detect artifacts. Prove merges. Route repairs. |
| bad behavior                                     | non-defensible escape hatch                    |
| vibe-code drift                                  | evidence drift                                 |

You can still use the memorable language in the intro and conclusion.
