Reviewed the uploaded paper and the current public repo/README/materials. The thesis is strong and timely: current evidence supports the idea that AI is an amplifier of existing engineering systems, not an automatic quality guarantee, and that generated code/security/secrets risks require structural controls rather than confidence alone.  ([Dora][1])

The highest-leverage problem is not the paper’s idea; it is credibility alignment. The paper reads like a nearly mature standard, while the repo currently looks like an early prototype: 4 commits, no releases, no packages, no visible adoption, no real security lane, and several claimed outputs/features that are planned or partial. ([GitHub][2])

## Highest-priority feedback

### 1. Severity: Critical

**Location:** Paper Sections VI–VIII; README; audit CLI
**Issue:** The paper blurs two standards: “agent-native repository controls” and “preferred stack conformance.” The README says the audit works on any repo, but the auditor hard-caps “non-optimal product language” and “too much Python,” which makes the tool feel like a Rust/TS/Postgres enforcement tool rather than a general agent-native standard. ([GitHub][2])
**Why it matters:** A skeptical Go, .NET, JVM, Rails, or Python team will dismiss the whole standard as author preference before engaging with the more valuable control-plane ideas.
**Proposed fix:** Split conformance into two modes: `jankurai controls` and `jankurai target-stack profile`. The base standard should grade owner maps, proof lanes, generated zones, repair receipts, security, UX proof, and observability across stacks. The Rust/TS/Postgres recommendation should be an optional profile, not a universal hard cap.
**Type:** Needs internal product building

### 2. Severity: Critical

**Location:** Paper Section VIII; Tables VI–IX; Figure 2
**Issue:** The stack ranking is too precise for the evidence. Scores like 94, 90, 89, 88, 87 imply calibrated measurement, but the paper admits the weights are policy choices and Table IX’s uncertainty bands overlap. 
**Why it matters:** Reviewers will challenge the ranking as preordained. The close scores and overlapping bands actually support “clustered recommendations,” not a clean ordinal victory.
**Proposed fix:** Add a scoring workbook appendix with per-criterion justifications, sensitivity analysis, and alternative weight profiles: startup CRUD, regulated enterprise, cloud infra, AI/data platform, realtime collaboration, brownfield monolith. Present the top five as tiers unless the sensitivity analysis shows a stable winner.
**Type:** Needs writing-editing

### 3. Severity: Critical

**Location:** README “Rendered UX QA”; `packages/ux-qa/src/page-analyzer.ts`
**Issue:** The README claims reports include screenshot, crop, and ARIA snapshot artifact paths when requested, but the current analyzer initializes `artifacts: []` and does not appear to capture screenshots, crops, traces, or ARIA snapshots. ([GitHub][2])
**Why it matters:** This is the biggest product credibility gap because rendered UX QA is one of the paper’s most original and valuable claims. A practitioner will try it, inspect the JSON, and conclude the standard overpromises.
**Proposed fix:** Build artifact capture now: screenshot path, crop path per violation, ARIA snapshot path, optional Playwright trace path, viewport/browser metadata, route/story ID, and stable rule IDs. Then add a sample `ux-qa.json` and failing demo page.
**Type:** Needs internal product building

### 4. Severity: Critical

**Location:** `.github/workflows/jankurai-audit.yml`; Section XIII security claims
**Issue:** The CI “security lane” currently echoes commands instead of running scanners. The paper says secret scanning, SCA, SBOM/provenance, and workflow linting are core controls, but the repo has placeholder commands for `gitleaks`, `syft`, `grype`, and `zizmor`. ([GitHub][3])
**Why it matters:** Security is the top TLR bucket. If the project’s own security lane is simulated, the standard looks performative.
**Proposed fix:** Run at least one real secret scanner, one dependency/SCA scanner, one workflow linter, and upload artifacts. For example: `gitleaks`, `osv-scanner` or `npm audit --audit-level`, `zizmor`, and SBOM generation. Mark unsupported scanners as future work only after at least one real gate exists.
**Type:** Needs internal product building

### 5. Severity: Critical

**Location:** `jankurai/cli.py`, `tools/check_versions.py`, `packages/ux-qa/src/*.ts`
**Issue:** Major source files are compressed into one or a few extremely long physical lines. That conflicts with the project’s own “small files/functions, repair locality, agent-readable source” philosophy. ([GitHub][4])
**Why it matters:** A serious open-source adopter will question maintainability, reviewability, formatter discipline, and whether the tool exempts itself from its own standard.
**Proposed fix:** Reformat all source and docs with normal line structure. Add formatter/lint gates. Then run jankurai against itself with no self-exemption for maintainability rules, or explicitly document why tool internals are exempt.
**Type:** Needs internal product building

### 6. Severity: High

**Location:** Abstract; Sections I–II; Conclusion
**Issue:** The thesis is clear but sometimes rhetorically stronger than the evidence. “Anything less is just faster vibe coding” and repeated “winner” framing make the argument sound more absolute than the data supports. 
**Why it matters:** The paper is trying to be a standards-oriented engineering proposal, not only a manifesto. The rhetoric may alienate readers who agree with the controls but reject the totalizing language.
**Proposed fix:** Keep the bluntness, but add a “claim ladder” early: evidence supports AI risk; repo controls are a proposed mitigation; jankurai is a versioned policy; stack ranking is a default profile awaiting empirical calibration. Replace absolute language with “without comparable controls” or “under this target profile.”
**Type:** Needs writing-editing

### 7. Severity: High

**Location:** Section II definitions; Section V taxonomy
**Issue:** The distinction between AI-assisted engineering and unsafe vibe coding is mostly there, but it should be made unmissable in one compact definition box.
**Why it matters:** Readers may think the paper is anti-AI coding unless the safe/unsafe distinction is repeated in operational terms.
**Proposed fix:** Add a boxed definition: “AI-assisted engineering is acceptable when generated code has explicit owner, legal edit surface, proof lane, security/UX gates as applicable, and repair evidence. Vibe coding is generated or human-written code accepted without those controls.”
**Type:** Needs writing-editing

### 8. Severity: High

**Location:** Section V; Table IV; Figure 1
**Issue:** The taxonomy is broad and useful, but the RPN/TLR pie can look like empirical incident-frequency data even though the caption says it is policy-weighted. 
**Why it matters:** A reviewer will ask: where did 32%, 20%, 13% come from, and why should I trust them?
**Proposed fix:** Add a small RPN methodology table: impact scale, likelihood scale, detection-difficulty scale, example row calculation, and sensitivity results. Consider labeling Figure 1 “policy priority allocation” rather than “risk share.”
**Type:** Needs writing-editing

### 9. Severity: High

**Location:** Table IV “Ranked Vibe-Coding Fault Taxonomy”
**Issue:** The fault rows are strong, but the paper lacks the requested “fault, impact, why it happens, mitigation” explanation for each major category.
**Why it matters:** Practitioners need to recognize failures in their repo, not just see a ranked list.
**Proposed fix:** Add a second table or appendix with one row per TLR bucket: concrete example, how agents cause/amplify it, escaped-defect symptom, detector, and repair receipt.
**Type:** Needs writing-editing

### 10. Severity: High

**Location:** Section V risk taxonomy; Table XV security controls
**Issue:** Missing or underweighted risks: MCP/tool-server compromise, agent workspace compromise, CI token abuse, license/IP provenance, model/provider deprecation, vector-store poisoning, context poisoning via docs/issues, flaky-test quarantine abuse, visual-baseline rubber-stamping, cost/resource explosions, and artifact PII leakage. OWASP’s LLM Top 10 includes prompt injection, sensitive information disclosure, supply-chain vulnerabilities, improper output handling, and excessive agency, so the paper is directionally right but should extend the taxonomy to agent tooling and artifact workflows. ([OWASP][5])
**Why it matters:** The standard is strongest when it anticipates how coding agents actually operate: shell, browser, MCP, credentials, CI, package managers, artifacts, and context stores.
**Proposed fix:** Add `HLT-019-TOOL-SERVER-RISK`, `HLT-020-ARTIFACT-PII-LEAK`, `HLT-021-LICENSE-PROVENANCE`, and `HLT-022-FLAKY-BASELINE-DRIFT`, or explicitly fold them into existing rules.
**Type:** Needs online research

### 11. Severity: High

**Location:** Section XII rendered UX; Table XIII
**Issue:** The rendered UX section is a standout, but it needs more evidence that deterministic geometry catches real defects that ordinary Playwright, Storybook, axe, and screenshots miss.
**Why it matters:** “World-class case” requires examples, not just tool inventory.
**Proposed fix:** Add a mini evaluation: 20 synthetic UI regressions across clipping, overlap, target size, sticky obstruction, focus visibility, long locale text, mobile viewport, and horizontal overflow. Show which are caught by typecheck, component tests, axe, visual diff, ARIA snapshot, geometry, and human review.
**Type:** Needs internal product building

### 12. Severity: High

**Location:** Section XII; README UX QA package
**Issue:** The paper correctly composes Storybook, Playwright screenshots, ARIA snapshots, accessibility automation, visual regression, MSW, design tokens, and AI/CV triage, but the tool currently seems limited to DOM geometry checks. Playwright supports screenshot comparisons and ARIA snapshots, and Storybook supports visual testing and network mocking workflows, so `jankurai` should integrate rather than merely mention them. ([Playwright][6])
**Why it matters:** Without integration, jankurai is a good idea paper plus a small geometry library, not a UX QA standard.
**Proposed fix:** Add adapters: Storybook story enumerator, Playwright project matrix reader, ARIA snapshot writer, screenshot baseline manifest, MSW/mock evidence check, and a baseline-approval receipt schema.
**Type:** Needs internal product building

### 13. Severity: High

**Location:** `packages/ux-qa/src/collector.ts`; `rules.ts`
**Issue:** Several geometry checks will have false positives or false negatives. Focus visibility only checks outline/box-shadow; accessible labeling relies on limited form-control checks; overlap ignores z-index/visibility stacking; sticky obstruction may flag legitimate fixed headers; selector stability falls back to `nth-of-type`. ([GitHub][7])
**Why it matters:** A UX gate that blocks legitimate UI will be disabled. A UX gate that misses obvious UI defects will not be trusted.
**Proposed fix:** Add rule metadata with confidence, exemptions, and artifact crops. Use Playwright locators/test IDs where possible, record computed stacking context, distinguish warning/review/block, and build a false-positive label loop.
**Type:** Needs internal product building

### 14. Severity: High

**Location:** Section XII; WCAG/Web Vitals claims
**Issue:** The paper cites strong standards-backed checks such as WCAG target size and CLS, but it should be clearer which checks are normative, which are heuristic, and which are project policy. WCAG 2.2 target-size minimum uses 24×24 CSS pixels with exceptions, and Google’s Core Web Vitals guidance treats CLS ≤ 0.1 as good. ([W3C][8])
**Why it matters:** UX teams will push back if policy heuristics are presented as accessibility law.
**Proposed fix:** Add a “Normative vs heuristic UX detectors” table: WCAG-backed, Web Vitals-backed, design-token-backed, project-policy, and AI/CV-review-only.
**Type:** Needs writing-editing

### 15. Severity: High

**Location:** Appendix A; audit output
**Issue:** The paper defines stable rule IDs and evidence fields, but the project lacks a visible JSON Schema, sample reports, SARIF output, and conformance fixtures.
**Why it matters:** Standards become real when tools, schemas, and examples are stable. Otherwise every adopter has to reverse-engineer output.
**Proposed fix:** Add `schemas/repo-score.schema.json`, `schemas/ux-qa.schema.json`, `examples/reports/`, `fixtures/bad-repo-*`, and golden tests that validate exact findings and `agent_fix_queue` ordering.
**Type:** Needs internal product building

### 16. Severity: High

**Location:** README Quick Start; repo release state
**Issue:** The README is clear enough to start, but trust blockers remain: no release, no package distribution beyond installing from GitHub, no changelog, no security policy, no contributing guide, no badges, and no sample output linked near the quick start. The repo currently shows no releases published. ([GitHub][9])
**Why it matters:** Serious engineers often will not install an unreleased audit tool into CI, especially one that proposes security and conformance gates.
**Proposed fix:** Ship a signed/tagged `v0.2.0`, publish PyPI and npm packages if intended, add `SECURITY.md`, `CONTRIBUTING.md`, changelog, release notes, checksums, CI badge, and sample reports.
**Type:** Needs internal product building

### 17. Severity: High

**Location:** README; Section VI conformance
**Issue:** The conformance path is conceptually good, but it needs a “minimum viable adoption” path that does not require adopting the default stack.
**Why it matters:** The most likely early users are existing messy repos, not greenfield Rust/TS/Postgres repos.
**Proposed fix:** Add three tracks: “Audit-only,” “Agent-control conformance,” and “Target-stack profile.” For non-target stacks, show how Go/.NET/JVM repos can pass control conformance while failing or opting out of target-stack profile.
**Type:** Needs writing-editing

### 18. Severity: High

**Location:** Section XVI limitations; release plan benchmark pack
**Issue:** The limitations section is honest but too late and too brief. The release plan says benchmarks are needed, but the paper should pull that into the main argument. 
**Why it matters:** Reviewers will accept policy proposals more readily when the evaluation plan is concrete and falsifiable.
**Proposed fix:** Add an “Evaluation Plan” section before the conclusion: baseline repos, controlled jankurai repos, agents tested, task types, metrics, raw artifact policy, and failure analysis.
**Type:** Needs writing-editing

### 19. Severity: Medium

**Location:** Section I evidence chain
**Issue:** The paper cites GitHub Octoverse, DORA, Veracode, GitGuardian, OWASP, SetupBench, and ContextBench, but it does not fully connect each source to a specific control. 
**Why it matters:** Strong citations become stronger when each one justifies a standard requirement.
**Proposed fix:** Add a “Evidence → Required control” table: DORA → system controls; Veracode → security lane; GitGuardian → secret scanning; OWASP → source hierarchy/agency limits; SetupBench → one-command setup; ContextBench → owner/test maps.
**Type:** Needs writing-editing

### 20. Severity: Medium

**Location:** Section III “Languages as Bottleneck Compression”
**Issue:** This section is intellectually useful but currently slows the paper before the standard/control-plane contribution.
**Why it matters:** Standards readers want the defect class, controls, conformance, and tool quickly.
**Proposed fix:** Move language-history material after the control-plane standard, or compress it into a short “why stack still matters after controls” subsection.
**Type:** Needs writing-editing

### 21. Severity: Medium

**Location:** Section X tool adapter matrix
**Issue:** Tool-specific instruction behavior changes quickly, and the paper cites many agent tools. The standard doc also includes Codex, Cursor, Claude, Gemini, Antigravity, and Copilot guidance. ([GitHub][10])
**Why it matters:** Tool-specific claims are likely to go stale faster than architectural claims.
**Proposed fix:** Keep the paper high-level and move tool-specific loading behavior to versioned rule packs in the repo. Add “last verified” dates and tests that confirm generated rule packs match the canonical standard.
**Type:** Needs online research

### 22. Severity: Medium

**Location:** Section XIII security; Table XV
**Issue:** Memory safety support is directionally good, but the paper should avoid implying that language choice substitutes for security controls. NSA/CISA guidance supports memory-safe languages as a vulnerability-reduction strategy, not as a complete security posture. ([U.S. Department of War][11])
**Why it matters:** Security reviewers will object if Rust is treated as a broad security proof.
**Proposed fix:** Add one paragraph: “Memory safety reduces an important vulnerability class; it does not replace authz tests, secret scanning, SCA, sandboxing, output validation, threat modeling, or incident evidence.”
**Type:** Needs writing-editing

### 23. Severity: Medium

**Location:** References; Section III/VI product-speed claims
**Issue:** Some technology-version citations are timely but brittle: TypeScript 7.0 is currently beta, while Vite 8 is stable. ([Microsoft for Developers][12])
**Why it matters:** A paper with fast-moving tool references can look stale quickly.
**Proposed fix:** Phrase unstable references as examples of direction rather than foundation: “native/compiler-speed work in TypeScript” and “modern Vite feedback loop,” then pin exact versions in an online appendix.
**Type:** Needs online research

### 24. Severity: Medium

**Location:** Agent maps and proof lanes
**Issue:** The repo has owner maps, test maps, generated-zone manifests, and proof-lane files, which is good; but the maps are project-specific examples, not reusable templates with validation and failure fixtures. ([GitHub][13])
**Why it matters:** Adoption depends on copying a working pattern, not inventing JSON/TOML structure from prose.
**Proposed fix:** Add `templates/agent/owner-map.json`, `templates/agent/test-map.json`, `templates/agent/generated-zones.toml`, and a command `jankurai init --profile controls|target-stack`.
**Type:** Needs internal product building

### 25. Severity: Medium

**Location:** README “Vibe-Coding Risk Model”
**Issue:** The README sells the risk model well, but it should show one actual before/after finding and repair queue entry immediately after quick start.
**Why it matters:** Engineers trust tools when they see exact output and can imagine the first repair.
**Proposed fix:** Add a small sample: `HLT-007-HANDWRITTEN-CONTRACT`, path, evidence, repair command, proof lane, and receipt. Link to a full sample report.
**Type:** Needs writing-editing

### 26. Severity: Medium

**Location:** Appendix C repair templates
**Issue:** Repair receipts are an excellent idea, but the paper should distinguish “receipt required for every fix” from “receipt required for high-risk or non-trivial fixes.”
**Why it matters:** Too much ceremony will make teams ignore the process.
**Proposed fix:** Define receipt thresholds: critical/high findings, generated zones, DB migrations, security changes, UX baseline changes, incident regressions, model/eval changes. Low-risk copy changes can rely on PR metadata.
**Type:** Needs writing-editing

### 27. Severity: Medium

**Location:** Audit soft rules: dead language, weak names, mega files/functions
**Issue:** Some soft signals are valuable but currently too broad. Terms like `legacy`, `fallback`, `temp`, `manager`, `handler`, and `utils` can be legitimate in context.
**Why it matters:** False positives are the fastest way to lose adoption.
**Proposed fix:** Require compound evidence before high severity: runtime path + reachable code + no exception + no typed policy + risky context. Otherwise emit medium/low advisory with examples.
**Type:** Needs internal product building

### 28. Severity: Medium

**Location:** Section XII AI/CV triage
**Issue:** The paper correctly says AI/VLM opinions should not be sole merge gates, but it should specify how AI/CV review is allowed to help.
**Why it matters:** Otherwise teams may either overtrust VLM review or ignore it entirely.
**Proposed fix:** Define AI/CV as triage-only: summarize crop, classify likely category, suggest selector/rule, propose baseline label, never approve alone. Store model/version/prompt and human label for calibration.
**Type:** Needs writing-editing

### 29. Severity: Low

**Location:** README and docs navigation
**Issue:** The docs structure is promising, but several docs are terse and some Markdown is hard to read in raw form.
**Why it matters:** A standard needs excellent documentation ergonomics, especially if it criticizes context bloat and unclear instructions.
**Proposed fix:** Add a docs index with “For maintainers,” “For adopters,” “For agents,” “For reviewers,” and “For researchers.” Ensure all Markdown has normal headings, tables, and code fences.
**Type:** Needs writing-editing

### 30. Severity: Low

**Location:** Paper captions and tables
**Issue:** Many tables are dense. The paper is already at risk of feeling like a catalog rather than an argument.
**Why it matters:** Dense tables help standards users but can fatigue paper reviewers.
**Proposed fix:** Keep core tables in the body, move expansion tables to appendices, and add 2–3 narrative walkthroughs: “agent changes API contract,” “agent changes payment invariant,” “agent changes checkout UI.”
**Type:** Needs writing-editing

## Bottom line

The paper is hard to dismiss when it argues “trustworthy merge is the bottleneck.” It is easier to dismiss when it argues “this exact stack wins” with precise scores before benchmark evidence exists. The strongest path is to make `jankurai` first a serious, stack-independent agent-native control standard, then offer Rust/TypeScript/Postgres as a well-argued target profile.

The most urgent product work is: real security lane, real UX artifacts, schemas/fixtures/sample reports, release packaging, and a non-target-stack conformance mode. Those five changes would make the project feel substantially more like an open-source standard and less like a paper-backed prototype.
