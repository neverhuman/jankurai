I reviewed the uploaded PDF and the public repo materials. The strongest idea is clear: **AI coding failure is often a verification-interface failure, not merely a model-quality failure**. The paper already defines vibe coding operationally, requires proof lanes, owner maps, generated-zone policy, repair receipts, and explicit conformance levels, which is the right foundation.  The public README also does a good job positioning `jankurai` as an “audit now, ratchet toward conformance” project with JSON/Markdown reports and a concrete UX-QA package. ([GitHub][1])

Below is the highest-leverage feedback.

---

### 1.

**Severity:** Critical
**Location:** Sections VII–IX, Tables VI–IX, Fig. 2, conclusion
**Issue:** The paper’s “standard” argument and “default winner stack” argument are too tightly fused. The standard is broadly defensible; the Rust/TS/Postgres stack ranking is more contestable.
**Why it matters:** A skeptical reviewer can dismiss the whole paper as a language-preference essay even though the control-plane standard is much stronger than the stack ranking. The sentence-level framing sometimes implies that not choosing the preferred stack is “vibe coding,” which invites avoidable pushback.
**Proposed fix:** Split the claim into: “jankurai conformance is stack-independent in principle” and “this paper provides one reference profile for durable greenfield product systems.” Add a short “Other conformant profiles” table: Go services, .NET enterprise, JVM brownfield, Rails CRUD, Python AI/data, Elixir realtime. Each profile should show which controls must compensate for weaker points.
**Type:** Needs writing/editing

### 2.

**Severity:** Critical
**Location:** Fig. 1, Table IV, Section V
**Issue:** The TLR/RPN pie chart looks quantitative, but the paper says RPN is a policy priority, not empirical frequency. The percentages make it feel more statistically grounded than it is.
**Why it matters:** Reviewers will ask where the likelihood and detection-difficulty numbers came from. “32% Security” and “20% Business truth” are useful as policy priors, but the visual presentation risks pseudo-precision.
**Proposed fix:** Add separate `Impact`, `Likelihood prior`, and `Detection difficulty prior` columns, plus a note on who assigned the values and how future calibration will work. Consider caption language like: “Policy-prior risk budget for v0.2, awaiting empirical calibration.” Add a calibration plan: run jankurai across 20–50 repos, compare findings to escaped defects, review time, security issues, and agent repair success.
**Type:** Needs writing/editing

### 3.

**Severity:** Critical
**Location:** `jankurai/cli.py`, project credibility
**Issue:** The project appears to violate its own code-shape standard. The GitHub file page reports `jankurai/cli.py` as 3,239 lines / 2,813 LOC, while the standard says non-generated files over hard LOC limits should trigger refactor or exception. ([GitHub][2])
**Why it matters:** This is the easiest dismissal: “The standard says no mega-files, but the auditor is a mega-file.” For an open-source standard, self-conformance is a credibility gate.
**Proposed fix:** Make the repo its first case study. Split the CLI into modules such as `inventory.py`, `rules/`, `scoring.py`, `report_json.py`, `report_md.py`, `schemas.py`, and `queue.py`. Publish the current self-audit score and either fix or explicitly exception any violation with an expiration date.
**Type:** Needs internal product building

### 4.

**Severity:** Critical
**Location:** README, release/adoption path
**Issue:** The README asks users to install from GitHub `main`, but the repo shows no published releases. ([GitHub][1])
**Why it matters:** Serious engineers will hesitate to wire an unreleased, moving-branch auditor into CI, especially one intended to become a conformance standard.
**Proposed fix:** Publish tagged releases, checksums, release notes, and versioned install commands. Add `pip install jankurai==0.2.0`, an npm package path for `@jankurai/ux-qa`, a pinned GitHub Action example, and a “do not run unpinned main in blocking CI” warning.
**Type:** Needs internal product building

### 5.

**Severity:** Critical
**Location:** Sections VI, XI, Appendix A, README “Quick Start”
**Issue:** The standard promises machine-readable conformance, but the project needs stronger schemas and examples. The docs require owner maps, test maps, generated-zone manifests, audit JSON, and repair queues, but adopters need exact JSON Schema/TOML schemas and fixture repos. ([GitHub][3])
**Why it matters:** Without schemas and examples, the “standard” feels like prose plus a linter, not a durable open standard. Agents also need schema-stable outputs to repair findings reliably.
**Proposed fix:** Add `schemas/owner-map.schema.json`, `test-map.schema.json`, `generated-zones.schema.json`, `repo-score.schema.json`, and `repair-receipt.schema.json`. Add `examples/minimal-hl1`, `examples/hl3-rust-ts-postgres`, and `examples/nonconformant-vibe-repo` with expected audit outputs.
**Type:** Needs internal product building

### 6.

**Severity:** Critical
**Location:** Section XVI, whole paper
**Issue:** The limitations section is honest but too late and too light. The paper says weights need empirical calibration, but the top-level claims and charts arrive much earlier with strong certainty.
**Why it matters:** A reviewer will accept “standards proposal with early implementation” more readily than “benchmark-like scoring without benchmark data.”
**Proposed fix:** Move a short limitation paragraph into the introduction: “This is a standards proposal and policy-prior model, not a large-scale benchmark.” Then add a concrete evaluation roadmap table: repo sample size, metrics, expected artifacts, and what would falsify or revise the rubric.
**Type:** Needs writing/editing

### 7.

**Severity:** High
**Location:** Abstract, Section I, Section XVII
**Issue:** The rhetoric sometimes overpowers the precise claim. The title is fixed, but sentences like “anything less is just faster vibe coding” and “once the standard identifies a default, continuing to optimize for every runner-up becomes noise” overstate the case.
**Why it matters:** The paper says it is not anti-AI and not generic rhetoric, but these lines give skeptics a reason to read it as a manifesto rather than a standards paper.
**Proposed fix:** Replace absolute lines with standard-oriented language. Example: “Without explicit ownership, proof routing, and repair evidence, higher coding throughput increases ambiguity faster than the repository can reject it.”
**Type:** Needs writing/editing

### 8.

**Severity:** High
**Location:** Section II, Table I, Table IV, Appendix A
**Issue:** The taxonomy is broad, but several AI-specific data risks need sharper treatment: prompt/transcript retention, vector-store leakage, provider logging, tool-output-to-shell execution, untrusted MCP/tool connectors, and cross-repo context contamination.
**Why it matters:** The current taxonomy includes PII, prompt injection, excessive agency, and improper output handling, but modern agent systems leak through context pipelines as much as through source code. OWASP explicitly treats prompt injection, insecure output handling, supply-chain vulnerabilities, sensitive information disclosure, and excessive agency as first-order LLM risks. ([OWASP][4])
**Proposed fix:** Add a rule family such as `HLT-019-CONTEXT-DATA-LEAKAGE`: flags prompt logs, transcripts, vector-store indexes, MCP/tool configs, copied customer data, and provider-retention gaps. Require evidence for redaction, retention, and source-trust classification.
**Type:** Needs internal product building

### 9.

**Severity:** High
**Location:** Section XIII, Table XV
**Issue:** Security controls need a standards mapping table.
**Why it matters:** The paper cites OWASP and supply-chain tooling, but a standards-oriented engineering paper should show how `jankurai` aligns with established security frameworks, not just tool names.
**Proposed fix:** Add “HLT rule → OWASP LLM Top 10 → NIST SSDF → SLSA/OSV/GitHub secret scanning evidence.” NIST SSDF is explicitly a secure software development practice framework, SLSA is a framework for supply-chain integrity, OSV-Scanner supports vulnerability scanning, and GitHub push protection is designed to block hardcoded credentials before they reach a repository. ([NIST Computer Security Resource Center][5])
**Type:** Needs online research

### 10.

**Severity:** High
**Location:** Section XII, Tables XIII–XIV, README “Rendered UX QA”
**Issue:** The rendered UX section is strong, but it needs to become more algorithmic and less aspirational.
**Why it matters:** This is one of the paper’s most differentiated contributions. Many tools already do screenshots, Storybook, and axe. The paper’s unique claim is deterministic geometry and artifact policy. That needs precise rules.
**Proposed fix:** Define the geometry oracle formally: inputs, viewport matrix, DOM box model, computed-style fields, accessible names, scroll containers, occlusion checks, sticky/fixed obstruction, target spacing, clipping, overflow, focus visibility, and output schema. The README already says reports include rule IDs, selectors, viewport data, severity, merge decision, and artifact paths; turn that into a normative schema. ([GitHub][1])
**Type:** Needs internal product building

### 11.

**Severity:** High
**Location:** Section XII
**Issue:** The paper should explicitly distinguish four rendered-UX evidence types: semantic assertions, accessibility-tree proof, deterministic geometry, and visual-baseline review.
**Why it matters:** Otherwise readers may think “pixel QA” means screenshot diffs only. Playwright guidance supports user-visible behavior, resilient locators, web-first assertions, and screenshot comparison, but also warns that screenshots vary by OS/browser/rendering environment. ([Playwright][6])
**Proposed fix:** Add a table:
`Semantic proof = Playwright role/text/test-id assertions`;
`ARIA proof = accessible tree snapshots`;
`Geometry proof = boxes/styles/overlap/target size`;
`Visual proof = baseline diff with owner approval`.
Make deterministic geometry the hard gate, visual diff a review workflow, and AI/VLM triage advisory.
**Type:** Needs writing/editing

### 12.

**Severity:** High
**Location:** Section XII, Table XIII
**Issue:** Storybook and mocks are mentioned, but story-state coverage should be converted into a conformance rule.
**Why it matters:** Agents often only see the happy path. Storybook’s visual testing turns stories into baseline snapshots, Storybook accessibility checks audit rendered DOM as a first-line signal, and MSW lets stories mock network responses deterministically. ([Storybook][7])
**Proposed fix:** Add `HLT-020-STORY-STATE-GAP`: reusable UI components and critical screens must have normal/loading/empty/error/long-text/locale/theme/mobile/role states. Add a `storybook-state-manifest.json` generated from stories, linked to owner/test maps.
**Type:** Needs internal product building

### 13.

**Severity:** High
**Location:** Section XII, Table XIII
**Issue:** Accessibility automation is framed well, but the limits of automation need stronger caveats.
**Why it matters:** Playwright’s accessibility docs explicitly warn that automated tests catch only some common accessibility problems and recommend combining automation with manual assessment and inclusive testing. ([Playwright][8])
**Proposed fix:** Add a sentence: “A11y automation is a merge gate for known detectable failures, not a complete accessibility claim.” Require expert/manual review for semantic ambiguity, keyboard journey completeness, screen-reader usability, and content comprehension.
**Type:** Needs writing/editing

### 14.

**Severity:** High
**Location:** Section XII, WCAG/Web Vitals paragraph
**Issue:** The UX section cites good thresholds, but `jankurai` should expose them as configurable policy, not fixed taste.
**Why it matters:** WCAG 2.2 target-size minimum is 24×24 CSS pixels with exceptions, and Web Vitals recommends CLS ≤ 0.1 for good user experience. Those are solid anchors, but product contexts vary. ([W3C][9])
**Proposed fix:** Add `ux-policy.toml` with defaults: target-size rule, spacing exceptions, CLS threshold, overflow policy, sticky obstruction policy, token enforcement, and viewport matrix. Report which threshold triggered each finding.
**Type:** Needs internal product building

### 15.

**Severity:** High
**Location:** Section X, Table XI
**Issue:** Agent-tool adapter guidance will age quickly.
**Why it matters:** Codex, Claude Code, Cursor, Copilot, and other agent surfaces change faster than academic papers. The paper already acknowledges tool-specific instruction behavior changes quickly, but the adapter matrix reads normative.
**Proposed fix:** Move fast-changing adapter details into a versioned online companion doc. In the paper, define the invariant: “single canonical policy → generated/mirrored tool-specific instructions.” Add an update cadence and compatibility matrix in the repo.
**Type:** Needs online research

### 16.

**Severity:** High
**Location:** Tables VI–IX
**Issue:** The stack rubric lacks a reproducible scoring workbook.
**Why it matters:** The ranking will be challenged as author-driven. Table IX compresses many criteria into “Ops,” while Table VI has ten criteria. Reviewers need to see exact scoring definitions and why Go is 90, .NET 89, JVM 87, etc.
**Proposed fix:** Add `docs/research/stack-scoring-workbook.csv` or `.xlsx`, with rows for each criterion, cap, evidence, score, sensitivity band, and reviewer notes. Include an ablation: without memory safety weight, without ecosystem weight, with brownfield enterprise weight, with startup velocity weight.
**Type:** Needs internal product building

### 17.

**Severity:** High
**Location:** Section VIII, Table IX
**Issue:** Brownfield exceptions are acknowledged but underspecified.
**Why it matters:** Most serious adopters are not greenfield. If they run Java/.NET/Go/Python monorepos, they need a path that does not start with “migrate to Rust.”
**Proposed fix:** Add “conformant brownfield profile” examples: JVM + Postgres + generated contracts; .NET + SQL Server/Postgres + generated clients; Go services + Postgres; Rails + Postgres with stricter boundary rules. For each, define required compensating controls.
**Type:** Needs writing/editing

### 18.

**Severity:** High
**Location:** Appendix A, Table XIX
**Issue:** Several “hard” rules are actually heuristic detectors with high false-positive risk.
**Why it matters:** Terms like `legacy`, `fallback`, `helper`, `common`, TODOs, `unwrap`, snapshots, and `catch` can be legitimate. If too many early findings are noisy, teams will distrust the tool before they experience the standard.
**Proposed fix:** Add confidence levels: `hard-block`, `needs-evidence`, `warning`, `style-signal`. For regex-based findings, require contextual evidence before blocking: production path, changed file, no owner exception, no test lane, and risk category.
**Type:** Needs internal product building

### 19.

**Severity:** High
**Location:** Table IV, HLT-008
**Issue:** “False-green tests” is one of the most important ideas, but the detection strategy is underdeveloped.
**Why it matters:** False-green risk is central to the thesis: agents can pass the wrong tests. A simple skipped-test/snapshot/no-assertion detector will not catch most business-logic false greens.
**Proposed fix:** Add a roadmap for mutation testing, fault seeding, invariant coverage, changed-path-to-assertion mapping, incident replay tests, contract negative cases, and “red evidence” requirements for repaired bugs. Make HLT-008 a composite rule rather than one detector.
**Type:** Needs internal product building

### 20.

**Severity:** High
**Location:** Table IV, Fig. 1, Section XIV
**Issue:** Observability and repair opacity are underweighted at 3% in the TLR pie, despite “trustworthy merge” and repair evidence being central to the thesis.
**Why it matters:** If post-merge repair is part of the new bottleneck, observability cannot feel like a minor tail category.
**Proposed fix:** Either raise repair/observability weight or fold it into every high-risk category as a required secondary control. Add explicit “incident replay,” “trace-to-owner,” and “repair receipt closure rate” metrics.
**Type:** Needs writing/editing

### 21.

**Severity:** High
**Location:** Section VI, Table V, Appendix C
**Issue:** Conformance levels need precise claim language.
**Why it matters:** “HL3 Standard” and “HL5 Release contract” sound official, but adopters need to know exactly what they may claim and what artifacts prove it.
**Proposed fix:** Add a “Conformance Claims” section: allowed badge text, required artifacts, required CI mode, exception rules, audit version, schema version, and expiry. Example: “HL3-jankurai-0.2.0, audited on commit SHA, with 2 approved exceptions expiring by date.”
**Type:** Needs writing/editing

### 22.

**Severity:** High
**Location:** README, docs, project root
**Issue:** The project needs open-source trust scaffolding.
**Why it matters:** Engineers evaluating a standard look for governance, license clarity, contribution path, security reporting, and release discipline. The `pyproject.toml` declares MIT license text, but the root listing I saw does not show the usual `LICENSE`, `CONTRIBUTING.md`, `SECURITY.md`, `CODE_OF_CONDUCT.md`, or release artifacts. ([GitHub][10])
**Proposed fix:** Add `LICENSE`, `CONTRIBUTING.md`, `SECURITY.md`, `CODE_OF_CONDUCT.md`, `CHANGELOG.md`, `GOVERNANCE.md`, issue templates, and a compatibility policy for rule changes.
**Type:** Needs internal product building

### 23.

**Severity:** Medium
**Location:** README “Quick Start”
**Issue:** The README sells the concept well but does not show enough concrete output.
**Why it matters:** A serious engineer wants to see what the tool will emit before installing it.
**Proposed fix:** Add a short sample `repo-score.md` excerpt, a sample JSON `finding`, and a sample `agent_fix_queue` entry. Show one before/after repair receipt.
**Type:** Needs writing/editing

### 24.

**Severity:** Medium
**Location:** README “Rendered UX QA” and `packages/ux-qa`
**Issue:** UX-QA package trust needs package-level docs, not only root README snippets.
**Why it matters:** Frontend engineers will look for rule docs, examples, artifact samples, and flake guidance before adding browser QA to CI.
**Proposed fix:** Add `packages/ux-qa/README.md` with rule IDs, viewport defaults, screenshots/crops/ARIA artifacts, Playwright fixture examples, CI examples, known false positives, and baseline-governance guidance.
**Type:** Needs internal product building

### 25.

**Severity:** Medium
**Location:** Section XII, design tokens paragraph
**Issue:** Design-token coverage is promising but should become a concrete enforcement path.
**Why it matters:** The W3C Design Tokens Community Group announced a stable 2025.10 spec for vendor-neutral design decisions across tools; this is a strong standards citation, but `jankurai` should translate it into checks. ([W3C][11])
**Proposed fix:** Add a `design-system-manifest.json` or `tokens.json` policy: approved tokens, component imports, forbidden raw colors/spacing/z-index, allowed exceptions, and links to Storybook stories.
**Type:** Needs internal product building

### 26.

**Severity:** Medium
**Location:** Section XI, Table XII
**Issue:** Test-lane routing is right, but “smallest mapped lane” needs more operational detail.
**Why it matters:** Agents need to know whether a lane is required, optional, skipped, flaky, unavailable, or deferred.
**Proposed fix:** Extend `test-map.json` with `lane`, `command`, `timeout`, `requires_services`, `artifacts`, `risk_level`, `blocking_mode`, `skip_policy`, and `owner`. Generate a route explanation in audit output: “Changed X, therefore lanes A+B required.”
**Type:** Needs internal product building

### 27.

**Severity:** Medium
**Location:** Appendix B, versioned artifact manifest
**Issue:** Version binding is good, but artifact integrity should include hashes.
**Why it matters:** If the standard cares about generated artifacts and auditability, the manifest should prove that paper, standard, schemas, and generated outputs match expected content.
**Proposed fix:** Add `sha256`, generator version, generated timestamp, and source path for each generated artifact. Make `tools/check_versions.py` validate hashes or emit a deterministic manifest diff.
**Type:** Needs internal product building

### 28.

**Severity:** Medium
**Location:** Section XIII, Appendix A HLT-012
**Issue:** “Excessive agency” needs a concrete permission-profile schema.
**Why it matters:** The paper correctly names the risk, but adopters need to encode allowed shell, network, browser, migration, secret, and filesystem access.
**Proposed fix:** Add `agent/permission-profiles.toml`: profiles such as `read-only`, `fast-lane`, `web-qa`, `migration-review`, `release`. Include allowed commands, write scopes, network policy, secrets policy, approval requirements, and receipt fields.
**Type:** Needs internal product building

### 29.

**Severity:** Medium
**Location:** Section XV
**Issue:** Migration guidance is conceptually sound but should become a staged checklist.
**Why it matters:** Teams need a low-friction path from HL0 to HL3 without reorganizing the whole repo immediately.
**Proposed fix:** Add a 30/60/90-day adoption plan: week 1 advisory audit; week 2 AGENTS + owner/test maps; week 3 generated zones; week 4 CI artifacts; month 2 contracts/security lanes; month 3 rendered UX and ratchet mode.
**Type:** Needs writing/editing

### 30.

**Severity:** Medium
**Location:** References and Section XVI
**Issue:** The paper should separate enduring citations from fast-moving tool citations.
**Why it matters:** Official docs for Codex, Claude Code, Cursor, Copilot, Playwright, Storybook, Vite, TypeScript, and package scanners change quickly. A paper with many “accessed 2026” tool citations will age fast.
**Proposed fix:** Put stable standards and research in the paper; put fast-moving tool compatibility in `docs/research/tool-matrix.md` with last-verified dates.
**Type:** Needs online research

### 31.

**Severity:** Low
**Location:** Section III–IV
**Issue:** The “languages as bottleneck compression” argument is insightful but long relative to the standard contribution.
**Why it matters:** Readers who came for `jankurai` may feel delayed before they reach the actionable standard.
**Proposed fix:** Compress Sections III–IV by 20–30% and move some language-history material to an appendix or companion essay. Keep the compression thesis, but get to rules and conformance faster.
**Type:** Needs writing/editing

### 32.

**Severity:** Low
**Location:** Appendix D, file tree diagrams
**Issue:** The canonical tree is useful but may look too prescriptive.
**Why it matters:** Monorepo teams with existing layouts may assume they cannot adopt the standard.
**Proposed fix:** Add: “Names may vary; these ownership cells are normative, not directory names.” Then provide an example mapping for a non-default repo layout.
**Type:** Needs writing/editing

---

## Highest-impact build tasks

The fastest path to credibility is:

1. **Self-audit and refactor the repo** so the project visibly follows its own standard.
2. **Publish schemas, examples, and versioned releases** so `jankurai` feels like a standard, not just a paper.
3. **Make UX-QA concrete** with a rule schema, fixture examples, artifacts, and CI workflow.
4. **Add a scoring workbook and pilot evaluation** so the stack ranking and RPN model become harder to dismiss.
5. **Separate conformance from the preferred stack** so skeptics can adopt the control plane even if they reject the default architecture.
