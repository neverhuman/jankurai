I reviewed the paper PDF and the current public repo/README. The core idea is strong and differentiated: **agent-native engineering as a repository control plane** is much harder to dismiss than another “AI coding is good/bad” essay. The highest-leverage improvements are: make the empirical/policy boundary sharper, reduce the stack-ranking defensiveness, implement the security and conformance gates that are currently described more strongly than they exist, and turn the rendered UX section into a demonstrable product asset rather than mostly a persuasive section. 

## Prioritized feedback

### 1. Evidence boundary is still the paper’s biggest credibility risk

* **Severity:** Critical
* **Location:** Abstract; Sections I, II, V, VII–VIII; Tables IV, VI–IX; Figures 1–2
* **Issue:** The paper is honest in places that the RPN, stack scores, and caps are policy weights rather than empirical measurements, but the visual presentation still gives them the feel of measured facts. Figure 1’s risk-share pie and Figure 2’s close stack scores are especially vulnerable.
* **Why it matters:** A skeptical reviewer will accept “here is a proposed standard” faster than “Rust wins 94–90–89–88–87.” Small score gaps look precise even though the paper says the uncertainty bands are policy uncertainty, not statistical confidence.
* **Proposed fix:** Add a short “Policy vs. evidence” box before Table IV and Table VI. Use language like: “The numeric values are normative defaults for v0.2.0. They are intended to make governance explicit, not to estimate population incident rates.” Add a scoring workbook or appendix showing each criterion, the reason for each weight, all caps triggered, and a sensitivity/tornado chart. For Figure 1, keep the model but add an adjacent heat-map or matrix that shows impact, likelihood, and detection difficulty separately.
* **Type:** writing-editing

### 2. The central thesis is strong, but “not mainly a model problem” needs a narrower claim

* **Severity:** Critical
* **Location:** Abstract; Section I “New Bottleneck”; Section II threat model
* **Issue:** The paper argues that AI coding failure is a repository, architecture, verification, and process problem. That is defensible. The weaker phrasing is any implication that model quality is not a meaningful cause of failure.
* **Why it matters:** Reviewers can easily object: models hallucinate APIs, miss security invariants, and overgeneralize. The paper does not need to deny that. Its stronger claim is that **production adoption fails when repositories cannot reject, route, and repair model output**. DORA’s 2025 report supports the “AI amplifies existing systems” framing, and GitHub’s Octoverse supports the claim that AI/TypeScript are mainstream, but neither proves model behavior is secondary in all cases. ([Dora][1])
* **Proposed fix:** Rewrite the thesis as: “Model capability is necessary but not sufficient. For production engineering, the marginal bottleneck has moved from generating plausible code to proving trustworthy merge.” Then explicitly say model failures remain inside the threat model as false plausibility, hostile context, and eval drift.
* **Type:** writing-editing

### 3. The stack ranking feels more author-driven than the rest of the standard

* **Severity:** Critical
* **Location:** Sections VII–IX; Tables VI–IX; Figure 2
* **Issue:** The stack recommendation is plausible, but the paper’s strongest contribution is the control-plane standard, not the exact rank order. The ranking section risks making the paper look like a Rust/TypeScript/Postgres advocacy piece with a standard wrapped around it.
* **Why it matters:** A Go, .NET, JVM, Rails, or Python-heavy team may dismiss the whole standard if they think conformance is a disguised stack war. The paper says exceptions are allowed, but the scoring table still makes the default feel like a foregone conclusion.
* **Proposed fix:** Add a “stack-neutral minimum viable conformance” table before the ranking. Show how Go, .NET, JVM, Rails, and Python/FastAPI can reach HL3 with compensating controls. Move “default winner” after a paragraph that says: “The standard is stack-portable; the recommended stack is one instantiation.” Then publish a sample scoring workbook so readers can rerun the weights.
* **Type:** writing-editing

### 4. The project currently overstates security implementation

* **Severity:** Critical
* **Location:** README “CI”; Justfile `security`; `.github/workflows/jankurai-audit.yml`; Sections XIII and Table XV
* **Issue:** The paper and README position security as a hard gate, but the current repo’s security lane is mostly placeholders: the Justfile security target echoes tool names, and the workflow has a “Security lane placeholders” step that echoes scanner commands rather than running them. ([GitHub][2])
* **Why it matters:** A serious engineer will run the project and conclude that the standard says “hard gate” while the reference implementation says “echo.” That undercuts trust more than having no security lane yet.
* **Proposed fix:** Rename current behavior to “security readiness markers” until real scanners run. Then implement a minimal real lane: `gitleaks` or `trufflehog` for secrets, OSV-Scanner for dependencies, `zizmor` for Actions, SBOM generation with Syft, and optional SLSA provenance checks. OSV-Scanner and SLSA are directly relevant because they provide dependency vulnerability scanning and supply-chain integrity/provenance frameworks. ([Google GitHub][3])
* **Type:** internal product building

### 5. The tool violates its own “small files / repair locality” doctrine

* **Severity:** Critical
* **Location:** `jankurai/cli.py`; docs hard LOC limits; Section VI; Appendix A
* **Issue:** The main audit implementation is a single 3,239-line file, 2,813 LOC, 105 KB. The standard itself sets hard maxes for files/functions and argues that large files harm agent repairability. ([GitHub][4])
* **Why it matters:** This is the easiest “humans were the bug” counterpunch: the standard does not yet dogfood its own code-shape claims.
* **Proposed fix:** Split `cli.py` into `inventory.py`, `rules/security.py`, `rules/contracts.py`, `rules/ux.py`, `rules/code_shape.py`, `scoring.py`, `schema.py`, `reporters/json.py`, `reporters/markdown.py`, and `main.py`. Add a self-audit exception only if you need one during migration, with an expiration date and refactor plan.
* **Type:** internal product building

### 6. “Vibe coding” is defined well, but it is too broad without a contrast table

* **Severity:** High
* **Location:** Section II definitions; Section V taxonomy
* **Issue:** The operational definition is good: accepting plausible code without ownership, proof routing, generated-boundary checks, security gates, and repair evidence. But because it captures so many ordinary engineering failures, it risks sounding like “vibe coding means anything bad.”
* **Why it matters:** Practitioners need to see that the paper is not anti-AI and not just relabeling tech debt.
* **Proposed fix:** Add a two-column table: “Safe AI-assisted engineering” vs. “Unsafe vibe coding.” Example rows: “agent edits generated client” vs. “agent edits OpenAPI source and regenerates”; “passes unrelated unit test” vs. “runs mapped proof lane”; “human says looks fine” vs. “repair receipt contains artifact and proof command.”
* **Type:** writing-editing

### 7. The risk taxonomy is broad, but several risks need more explicit treatment

* **Severity:** High
* **Location:** Table IV; Appendix A; Sections V, XIII, XIV
* **Issue:** Many important risks are present, but some are buried or under-specified: RAG/vector-store poisoning, prompt transcript retention, model/provider drift, cost/DoS and unbounded token/tool consumption, license/IP risk, agent identity/session audit, dependency license policy, rollout/rollback safety, and migration rehearsal. OWASP’s LLM Top 10 explicitly includes prompt injection, insecure output handling, sensitive information disclosure, supply-chain vulnerabilities, excessive agency, vector/embedding weaknesses, misinformation, and unbounded consumption, so the taxonomy should map to those terms more directly. ([OWASP][5])
* **Proposed fix:** Add a “coverage against external risk frameworks” table mapping HLT rules to OWASP LLM Top 10, NIST SSDF, SLSA, and CISA/NSA memory-safety guidance. Add missing HLT candidates or subrules for `VECTOR-STORE-POISONING`, `UNBOUNDED-CONSUMPTION`, `LICENSE-DRIFT`, and `AGENT-IDENTITY-AUDIT`.
* **Type:** online research

### 8. The RPN/TLR pie chart is useful only if demoted from “risk truth” to “policy backlog”

* **Severity:** High
* **Location:** Figure 1 and Table IV
* **Issue:** The chart’s caption says the shares are policy-priority, not incident frequency, which is good. But a pie chart still implies additive, measured proportions.
* **Why it matters:** Reviewers will ask why security is 32%, business truth 20%, repair 3%, and whether those percentages came from incidents.
* **Proposed fix:** Add an equation and source-of-weight table: `impact`, `likelihood`, `detection difficulty`, and the reason each row’s score was chosen. Add a sensitivity note: “If detection difficulty is doubled, these buckets change as follows.” Keep the pie only as a v0.2.0 backlog-prioritization model.
* **Type:** writing-editing

### 9. The hard/soft detector distinction needs more rigor

* **Severity:** High
* **Location:** Table IV “Hardness”; Appendix Table XIX
* **Issue:** “Hard” and “Soft” are overloaded. Sometimes hard means deterministic detector; sometimes hard means required control; sometimes it means severe risk.
* **Why it matters:** Agents and CI need different behavior for “detected objective violation,” “missing evidence,” “heuristic suspicion,” and “policy recommendation.”
* **Proposed fix:** Replace “Hardness” with three fields: `detector_type` = deterministic / heuristic / evidence-required; `merge_behavior` = block / warn / route-review; `false_positive_risk` = low / medium / high.
* **Type:** writing-editing

### 10. Current false-green detection is too regex-based

* **Severity:** High
* **Location:** HLT-008; `jankurai/cli.py`; Section XI
* **Issue:** The current implementation catches obvious weak-test markers like skipped tests, `.only`, `assert true`, and snapshot-only tests. That is useful, but it will miss the more important false-green class: the wrong test lane ran, the assertion does not cover changed behavior, mocks hide contract drift, or a test passes because the fixture is stale. ([GitHub][4])
* **Why it matters:** False-green tests are one of the paper’s central claims. If the tool only finds textual smells, the implementation lags the argument.
* **Proposed fix:** Add changed-path to proof-lane validation, test-to-source ownership mapping, required semantic assertion markers for high-risk routes, snapshot-pairing rules, mutation-test hooks, and “red evidence” support for bug fixes. For example: a repair receipt should show the failing test before the fix or explain why red evidence is unavailable.
* **Type:** internal product building

### 11. Rendered UX section is one of the paper’s best ideas, but the implementation needs to catch up

* **Severity:** High
* **Location:** Section XII; Tables XIII–XIV; README “Rendered UX QA”; `packages/ux-qa`
* **Issue:** The paper makes a strong case for rendered UX QA and cites the right ecosystem: Storybook, Playwright screenshots, ARIA snapshots, visual regression, accessibility automation, design tokens, MSW mocks, and human feedback. The package currently implements a good seed geometry runtime, but not the full paper vision. The README itself describes existing tools and says jankurai adds a deterministic geometry layer. ([GitHub][6])
* **Why it matters:** This can become the project’s most distinctive product contribution, but only if it is visibly more than a paragraph in the paper.
* **Proposed fix:** Build `@jankurai/ux-qa` into a real integration package: Storybook story discovery, Playwright fixture, ARIA snapshot comparison, axe-core integration, visual baseline metadata, design-token checks, CLS collection, artifact manifest, and PR review output.
* **Type:** internal product building

### 12. UX geometry rules need false-positive controls before they block serious teams

* **Severity:** High
* **Location:** `packages/ux-qa/src/rules.ts`; `collector.ts`; Table XIII
* **Issue:** The package already checks target size, edge clearance, overlap, clipping, wrapping, overflow, sticky obstruction, z-index, focus visibility, form labels, and nested scrollbars. That is the right direction. But some rules are too coarse for blocking use: sticky obstruction does not appear to account for z-index/pointer-events/visibility; focus detection relies on outline or shadow; overlap can flag legitimate layered controls; target-size exceptions need WCAG-compatible spacing semantics. ([GitHub][7])
* **Why it matters:** Teams will abandon a UX gate if it blocks on noisy layout heuristics.
* **Proposed fix:** Add rule-level config, suppressions with owner/expiration, severity overrides by route/story risk, and artifact-backed false-positive labels. For WCAG target size, align the rule text and exceptions with the 24×24 CSS pixel minimum and spacing exceptions. ([W3C][8])
* **Type:** internal product building

### 13. Accessibility coverage should explicitly separate “automation” from “accessibility assurance”

* **Severity:** High
* **Location:** Section XII; Table XIII; HLT-014
* **Issue:** The paper says automation is first-line, which is correct, but the HLT-014 wording could still imply that axe/ARIA/keyboard checks are enough. Storybook’s own docs frame automated accessibility checks as a first-line way to catch blatant violations, not a full substitute for expert review. ([Storybook][9])
* **Why it matters:** Accessibility experts will distrust a standard that appears to equate automated scans with full accessibility conformance.
* **Proposed fix:** Rewrite HLT-014 as: “Automated checks are required minimum evidence; semantic, cognitive, screen-reader, and task-flow accessibility require human or specialist review for protected surfaces.” Add examples where automation passes but UX/accessibility still fails.
* **Type:** writing-editing

### 14. ARIA snapshots are cited well, but jankurai should define what it does with them

* **Severity:** High
* **Location:** Section XII; README UX QA CLI; `packages/ux-qa/src/cli.ts`
* **Issue:** The CLI can emit ARIA snapshot artifacts, and Playwright supports ARIA snapshots as a YAML representation of the accessibility tree. But the standard does not yet say whether jankurai compares them, baselines them, reviews them, or simply stores them. ([GitHub][10])
* **Why it matters:** “Captured artifact” is weaker than “proof.”
* **Proposed fix:** Add `aria_baseline_path`, `aria_diff`, `approved_by`, and `semantic_change_reason` fields to the UX report. Make ARIA snapshot comparison optional at first, but define the contract now.
* **Type:** internal product building

### 15. The paper should cite visual-regression environment stability more directly

* **Severity:** Medium
* **Location:** Section XII; Playwright/Storybook discussion
* **Issue:** The paper mentions stable rendering environments, but it should make the operational checklist explicit: pinned browser, OS/font set, viewport, timezone, locale, animations disabled, deterministic data, and baseline governance. Playwright’s visual comparison docs support screenshot baselines, and Storybook’s visual testing docs support baselines across stories. ([Playwright][11])
* **Why it matters:** Pixel QA will look flaky unless the paper teaches teams how to make it boring.
* **Proposed fix:** Add a “Minimum stable visual lane” checklist and sample Playwright config.
* **Type:** writing-editing

### 16. The human feedback loop is excellent conceptually but absent as a product artifact

* **Severity:** High
* **Location:** Section XII; Table XIII; README UX QA
* **Issue:** The paper describes labels like true layout bug, intentional change, false positive, design-system violation, and product/design review item. That is a strong idea, but there is no visible schema or workflow in the README.
* **Why it matters:** This is where jankurai can become a learning system rather than another linter. Without a schema, teams cannot store or reuse feedback.
* **Proposed fix:** Add `ux-feedback.schema.json` with fields for route/story, viewport, rule ID, selector, screenshot crop, label, reviewer, owner, expiration, and linked issue. Add a PR comment template that lets reviewers label diffs.
* **Type:** internal product building

### 17. README is clear, but adoption will be blocked by packaging and trust signals

* **Severity:** High
* **Location:** README Quick Start; GitHub releases/packages
* **Issue:** The README explains what jankurai does, gives install commands, describes CI, and gives UX QA usage. That is good. But there are no releases published, and the quick start installs directly from GitHub. ([GitHub][6])
* **Why it matters:** Serious engineers will ask: what version am I installing, what changed, is there a signed release, and can I pin this in CI?
* **Proposed fix:** Publish a tagged release, PyPI package for `jankurai`, npm package for `@jankurai/ux-qa`, changelog, release notes, SHA/checksum or provenance notes, and a “known limitations in v0.2.0” section.
* **Type:** internal product building

### 18. README needs an expected-output example

* **Severity:** Medium

* **Location:** README Quick Start and “Add The Agent Standard”

* **Issue:** The README tells users to run `jankurai` and then read the Markdown report, but it does not show a realistic sample finding, score, hard cap, or repair queue item.

* **Why it matters:** Engineers decide whether to try a tool by seeing the first useful output.

* **Proposed fix:** Add a compact example:

  ```json
  {
    "final_score": 76,
    "hard_caps": ["missing-rendered-ux-qa-lane"],
    "agent_fix_queue": [
      {
        "rule_id": "HLT-013-RENDERED-UX-GAP",
        "path": "apps/web/src/checkout.tsx",
        "lane": "web",
        "agent_fix": "add Playwright screenshot and geometry evidence for checkout mobile viewport"
      }
    ]
  }
  ```

* **Type:** writing-editing

### 19. Conformance needs a formal JSON Schema

* **Severity:** High
* **Location:** Section VI; Appendix A; README; `repo-score.json` contract
* **Issue:** The paper defines required report fields, but the repo should ship a formal schema for audit output, repair receipts, owner maps, test maps, generated zones, exceptions, and UX QA reports.
* **Why it matters:** A standard becomes real when other tools can validate against it.
* **Proposed fix:** Add `schemas/jankurai-report.schema.json`, `schemas/repair-receipt.schema.json`, `schemas/owner-map.schema.json`, `schemas/test-map.schema.json`, and `schemas/ux-qa-report.schema.json`. Validate them in CI.
* **Type:** internal product building

### 20. Owner/test maps are present, but too shallow for serious conformance

* **Severity:** Medium
* **Location:** `agent/owner-map.json`; `agent/test-map.json`; Section VI
* **Issue:** The repo has owner and test maps, which is good. But the owner map is mostly top-level path-to-owner, and the test map does not yet express allowed dependencies, risk class, required lanes, CI job name, timeout, artifacts, or changed-file matching strategy. ([GitHub][12])
* **Why it matters:** Agents need machine-readable routing, not just labels.
* **Proposed fix:** Expand map entries to include `owner`, `allowed_imports`, `forbidden_imports`, `required_lanes`, `risk_tags`, `primary_artifacts`, and `escalation_owner`.
* **Type:** internal product building

### 21. Generated-zone policy is promising but needs enforcement examples

* **Severity:** Medium
* **Location:** `agent/generated-zones.toml`; HLT-002; Appendix B
* **Issue:** The generated-zone manifest exists, but there should be a concrete stale-generation check and a failing example. ([GitHub][13])
* **Why it matters:** Generated-zone mutation is one of the easiest high-signal wins. It should be demonstrably enforceable.
* **Proposed fix:** Add a fixture repo where a generated client is hand-edited; show jankurai producing `HLT-002-GENERATED-MUTATION`; add a regeneration command; rerun to green.
* **Type:** internal product building

### 22. The audit rules are actionable, but some soft signals need evidence controls

* **Severity:** Medium
* **Location:** HLT-001, HLT-008, HLT-018; `cli.py` pattern rules
* **Issue:** “fallback,” “temporary,” “legacy,” “utils,” “snapshot,” or “retry” can be legitimate. The detector should distinguish high-risk runtime code from docs, test fixtures, generated files, i18n copy, and explicit exceptions. The implementation already has some allowlist logic, but the standard should specify suppression shape and evidence. ([GitHub][4])
* **Why it matters:** False positives will make teams disable the tool.
* **Proposed fix:** Add an exception schema with `rule_id`, `path`, `matched_term`, `reason`, `owner`, `expires`, `proof_lane`, and `reviewer`. Make soft-rule suppression auditable rather than silent.
* **Type:** internal product building

### 23. The paper needs a small empirical pilot, even if it is not a benchmark paper

* **Severity:** High
* **Location:** Section XVI limitations; after Appendix A or before conclusion
* **Issue:** The paper says it is not yet a full empirical benchmark paper, which is fine. But it would be much harder to dismiss with even a small pilot.
* **Why it matters:** A standards proposal without any sample application can feel aspirational.
* **Proposed fix:** Add a 3–5 repo pilot: run jankurai on one clean reference repo, one intentionally vibe-coded fixture, one typical TypeScript app, one backend API, and this repo. Report finding counts, false positives, repair time, and before/after score.
* **Type:** internal product building

### 24. Rendered UX should have its own benchmark fixture suite

* **Severity:** High
* **Location:** Section XII; `packages/ux-qa/tests`
* **Issue:** The current UX QA tests are a useful start: they cover target size, edge clearance, clipping, wrapping, overflow, overlap, focus, labels, nested scrollbars, and artifact-backed CLI receipts. ([GitHub][14]) But a world-class UX QA claim needs a fixture gallery.
* **Why it matters:** UX QA credibility depends on visual examples.
* **Proposed fix:** Add `packages/ux-qa/fixtures/gallery` with known-bad and known-good pages for mobile overflow, sticky obstruction, German long-label wrapping, RTL layout, focus rings, modals, menus, tables, dashboards, canvas/nonblank, and safe-area issues. Publish screenshots in docs.
* **Type:** internal product building

### 25. The paper needs stronger security-standard grounding beyond vendor reports

* **Severity:** High
* **Location:** Sections V, XIII; Table XV; references
* **Issue:** Veracode and GitGuardian are useful but vendor evidence alone will be challenged. The paper should also cite standards and public guidance: OWASP LLM Top 10, NIST SSDF, SLSA, OSV, GitHub secret scanning docs, and CISA/NSA memory-safe language guidance. The CISA/NSA memory-safe-language guidance supports the security rationale for Rust/other memory-safe defaults. ([U.S. Department of War][15])
* **Why it matters:** Standards-oriented papers need standards citations.
* **Proposed fix:** Add a “Controls mapped to public standards” table: HLT-009/010/011/012/016 → OWASP LLM Top 10, SLSA/OSV, secret scanning, memory safety, provenance.
* **Type:** online research

### 26. The paper’s citations are current, but citation hygiene needs tightening

* **Severity:** Medium
* **Location:** References pages 18–19
* **Issue:** The reference list is strong but noisy: many titles are lowercased, access dates dominate, and tool docs are mixed with research papers and vendor reports.
* **Why it matters:** Reviewers may read citation style as a proxy for rigor.
* **Proposed fix:** Split references or annotate them by class: empirical/survey, standards/security guidance, official tool docs, benchmark papers, project docs. Standardize title casing. Add arXiv links/versions for benchmarks.
* **Type:** writing-editing

### 27. The “agent tool adapter matrix” is timely but should emphasize non-enforcement

* **Severity:** Medium
* **Location:** Section X; Table XI
* **Issue:** The table is useful, but agent instruction files are context, not enforcement. Claude Code docs explicitly say project memory is context, not enforced configuration; Codex reads AGENTS.md and builds an instruction chain, but repo policy still needs CI/tool gates. ([Claude][16])
* **Why it matters:** This distinction is exactly the paper’s argument: instructions help, but deterministic control surfaces matter more.
* **Proposed fix:** Add a sentence before Table XI: “Every instruction surface listed here is advisory unless backed by generated maps, permissions, and CI gates.”
* **Type:** writing-editing

### 28. The open-source standard would benefit from a conformance path that is less stack-specific

* **Severity:** Medium
* **Location:** README; docs/agent-native-standard.md; Sections VI–IX
* **Issue:** The README says the audit works on any repo, while the default target profile is Rust/TypeScript/Postgres/bounded Python. That is clear, but teams outside the target stack need a more explicit “how to adopt without migrating stacks” path. ([GitHub][6])
* **Why it matters:** Adoption grows if Go/.NET/JVM/Rails/Python teams can get value before accepting the default stack argument.
* **Proposed fix:** Add “Profiles”: `jankurai-web-ts`, `jankurai-go-postgres`, `jankurai-dotnet-postgres`, `jankurai-jvm-postgres`, `jankurai-python-ai-boundary`, each with compensating controls and caps.
* **Type:** internal product building

### 29. The paper needs two concrete narrative examples before the tables

* **Severity:** Medium
* **Location:** Before Section V or inside Section II
* **Issue:** The paper gets table-heavy quickly. The definitions are clear, but a reader would benefit from one backend and one frontend failure story before the taxonomy.
* **Why it matters:** Concrete examples make the standard feel useful rather than ceremonial.
* **Proposed fix:** Add two half-page examples: “Agent hand-edits generated DTO and passes unit tests but breaks API contract” and “Agent changes checkout button; typecheck passes but mobile viewport clips the CTA.” Then show the jankurai finding and repair receipt.
* **Type:** writing-editing

### 30. “Repair receipts” are a major differentiator; make them more central

* **Severity:** Medium
* **Location:** Sections VI, XIV; Appendix C; README
* **Issue:** Repair receipts are one of the most original parts of the standard, but they appear after many other concepts.
* **Why it matters:** “Leave evidence for the next agent” is a memorable practitioner value proposition.
* **Proposed fix:** Add a repair-receipt example in the README right after the quick start, and add one paper figure showing failure → queue item → proof lane → receipt.
* **Type:** writing-editing

### 31. The limitations section is good but should name the evaluation plan more concretely

* **Severity:** Medium
* **Location:** Section XVI
* **Issue:** The section correctly says weights need calibration and benchmarks miss long-running maintenance, security regressions, UI correctness, and production repair quality. But it does not define the next evaluation artifacts.
* **Why it matters:** Skeptical readers want to see that the project knows how it could be falsified.
* **Proposed fix:** Add a research agenda table with metrics: escaped defects, repair locality, review time, false-positive rate, agent edit surface, proof-lane selection accuracy, UX artifact usefulness, security finding closure, and score-to-outcome correlation.
* **Type:** writing-editing

### 32. Add a “what jankurai is not” section to the README

* **Severity:** Low
* **Location:** README
* **Issue:** The user-facing repo should mirror the paper’s scope boundaries: not anti-AI, not a prompt guide, not a benchmark, not a guarantee that one agent is better than another.
* **Why it matters:** It prevents misreads and reduces drive-by criticism.
* **Proposed fix:** Add a concise “Not a replacement for review/security testing; not a universal stack mandate; not a model benchmark” block.
* **Type:** writing-editing

## Highest-leverage next moves

The top three moves I would prioritize are: implement real security/conformance gates instead of placeholders, publish a scoring workbook plus sample fixture repo, and turn `@jankurai/ux-qa` into a demonstrable artifact workflow with Storybook/Playwright/ARIA/axe/baseline outputs. Those three changes would make the paper much harder to dismiss because they convert the strongest claims into reproducible evidence.
