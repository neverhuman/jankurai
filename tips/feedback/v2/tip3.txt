Reviewed scope: uploaded paper PDF plus the public `jankurai` GitHub repo, README, standard docs, audit workflow, Python audit package, and `@jankurai/ux-qa` package. The central idea is strong: repos should become verification interfaces for generated code. The biggest credibility risk is not the thesis; it is the gap between a broad standards claim and the current evidence/tooling maturity. 

## Highest-leverage feedback

### 1.

* **Severity:** Critical
* **Location:** Abstract; Sections I–II; Conclusion
* **Issue:** The thesis is clear, but “AI coding failure is not mainly a model problem” still reads like a false dichotomy. The paper’s stronger claim is: model errors become production failures when repositories lack ownership, proof routing, boundary checks, and repair evidence.
* **Why it matters:** A skeptical reviewer can agree with the repo-control-plane thesis while objecting that model quality, context retrieval, tool design, and agent scaffolding still matter. DORA’s framing that AI amplifies existing organizational strengths/weaknesses supports your argument, but not a pure “not model” framing. ([Dora][1])
* **Proposed fix:** Add a guardrail paragraph early: “The claim is not that model capability is irrelevant. The claim is that production safety depends on repository controls that make model output rejectable, localizable, and repairable. Jankurai targets those controllable repository conditions.”
* **Type:** Needs writing/editing

### 2.

* **Severity:** Critical
* **Location:** Sections VII–IX; Tables VI–IX; Figure 2
* **Issue:** The stack ranking is the most dismissible part. The Rust + TypeScript + PostgreSQL recommendation is plausible, but the exact ANSS scores and “default winner” framing feel more author-driven than empirically derived.
* **Why it matters:** A reviewer will challenge why Rust beats Go/.NET/JVM by exactly 4–7 points, why product velocity is only 2 points, why ecosystem corpus is 8, and why Python/Rails/Next.js penalties are calibrated as shown. The README also foregrounds the same target profile, so this is not just a paper issue; it shapes the project identity. ([GitHub][2])
* **Proposed fix:** Keep the recommendation, but demote the numeric ranking to a “policy profile.” Add a public scoring workbook with raw criterion scores, rationale notes, alternative weight presets, and sensitivity plots. Add one paragraph: “These are versioned policy scores, not measured defect rates.”
* **Type:** Needs internal product building

### 3.

* **Severity:** Critical
* **Location:** Figure 1; Table IV; Section V
* **Issue:** The TLR/RPN pie chart is useful as a governance artifact, but the percentages look empirical despite the caption saying they are policy-weighted.
* **Why it matters:** Security evidence supports prioritizing generated-code flaws and secrets, and OWASP supports prompt injection, supply chain, improper output handling, and excessive agency as first-order risks. But those sources do not justify “Security = 32%” or “Repair = 3%.” ([Veracode][3])
* **Proposed fix:** Replace the pie chart with a “policy weight heatmap” or keep the pie but add a calibration table: evidence strength, auditability, blast radius, detection difficulty, and current detector maturity. Move the exact RPN math into a `policy-weights.yaml` or workbook.
* **Type:** Needs writing/editing

### 4.

* **Severity:** Critical
* **Location:** Section VI; Appendix A; repo `.github/workflows/jankurai-audit.yml`; `jankurai/cli.py`
* **Issue:** The standard promises hard gates and conformance, but the implementation is still mostly heuristic. The GitHub Actions “Security lane placeholders” currently echo commands like `gitleaks`, `syft`, `grype`, and `zizmor` instead of running them. ([GitHub][4])
* **Why it matters:** This is the fastest way for a serious engineer to lose trust: the paper says “hard gates,” while the repo demonstrates placeholders. The standard requires audit JSON/Markdown, owner maps, proof lanes, generated-zone manifests, CI outputs, and agent fix queues, so the implementation needs to visibly match the conformance story. ([GitHub][5])
* **Proposed fix:** Ship v0.3 around one promise: real hard gates. Implement actual scanner invocation or evidence ingestion for secrets/SCA/SBOM/workflow risk; return nonzero exit codes by mode; add SARIF; add JSON Schema; add changed-path owner/test-map coverage; and publish a sample failed/green report.
* **Type:** Needs internal product building

### 5.

* **Severity:** Critical
* **Location:** Section XII; Tables XIII–XIV; repo `packages/ux-qa`
* **Issue:** The rendered UX section is one of the paper’s best ideas, but the current tool does not yet justify a “world-class” claim. The package has a real seed runtime: it collects page metrics, elements, violations, artifacts, and rules such as edge clearance, target size, overlap, clipping, button wrapping, overflow, sticky obstruction, z-index, focus visibility, form labels, and nested scrollbars. But it does not yet integrate Storybook state discovery, axe, visual baseline review, CLS measurement, design-token parsing, or human label feedback. ([GitHub][6])
* **Why it matters:** The paper correctly argues that typechecks and component tests miss rendered failures. But adopters will expect the product to do more than geometry linting if the section discusses Storybook, Playwright screenshots, ARIA snapshots, visual regression, accessibility automation, design tokens, mocks, AI/CV triage, and feedback loops. Playwright and WCAG already provide strong external anchors for screenshots, ARIA snapshots, and 24×24 CSS-pixel target-size checks. ([Playwright][7])
* **Proposed fix:** Split the claim into layers: “implemented now,” “integrates externally,” and “planned.” Build `@jankurai/ux-qa` v0.3 with Storybook story enumeration, axe integration, CLS capture, artifact crops per violation, baseline-review metadata, and a label schema for human review decisions.
* **Type:** Needs internal product building

### 6.

* **Severity:** High
* **Location:** Sections II and V
* **Issue:** The distinction between AI-assisted engineering and unsafe vibe coding is present, but it needs a crisp comparative table.
* **Why it matters:** The paper says it is not anti-AI, but the title and rhetoric create a high burden to prove that. A practitioner should immediately see that “AI-assisted” is acceptable when routed through owner maps, proof lanes, security gates, generated zones, and repair receipts.
* **Proposed fix:** Add a table near Section II: “AI-assisted engineering” vs. “vibe coding.” Rows: ownership, edit scope, tests, generated files, security, UX, migrations, dependencies, observability, repair evidence.
* **Type:** Needs writing/editing

### 7.

* **Severity:** High
* **Location:** Table IV; Appendix A rule list
* **Issue:** The taxonomy is broad, but several important agent-era failure modes need first-class rows or clearer subrows: malicious generated tests, prompt injection into policy/test files, MCP/tool configuration risk, credential brokering/non-human identity sprawl, artifact/transcript PII leakage, eval dataset poisoning, model/provider version drift, license/IP provenance, and untrusted PR workflow escalation.
* **Why it matters:** OWASP’s LLM and AI Agent guidance treats prompt injection, tool abuse, privilege escalation, data exfiltration, supply chain, and excessive agency as core risks; the paper covers many but not all with enough specificity. ([OWASP][8])
* **Proposed fix:** Add 6–8 rows to Table IV or an “expanded security taxonomy” table. Give each row a rule ID, lane, deterministic evidence, and current detector status.
* **Type:** Needs online research

### 8.

* **Severity:** High
* **Location:** Table IV; Section V narrative
* **Issue:** The “fault, impact, why it happens, mitigation” explanations are not explicit enough. Table IV has fault, rule/lane, required control, and hardness, but it does not consistently explain mechanism.
* **Why it matters:** Practitioners learn from causal explanations. “False-green business logic” is important, but the table should show why it happens: generated tests mirror generated assumptions; mocks hide boundary drift; assertions check presence, not invariant; acceptance criteria are absent.
* **Proposed fix:** Add columns: `Agent failure mechanism`, `Escaped-defect example`, `Deterministic proof`. Keep RPN separate so the causal story is not buried under scoring.
* **Type:** Needs writing/editing

### 9.

* **Severity:** High
* **Location:** Appendix A; `jankurai/cli.py`; audit rules
* **Issue:** Missing hard catches should be added before broadening the standard further: GitHub Actions `permissions:` defaults, `pull_request_target` misuse, unpinned actions/images, curl-to-shell installers, conflicting lockfiles, missing lockfile for package manifests, dependency added without license/provenance, generated zone stale output, CODEOWNERS/owner-map mismatch, test `.only`, unreviewed skipped tests, destructive SQL without transaction/rollback/lock-timeout, and artifact/transcript secret leakage.
* **Why it matters:** These are concrete, auditable, high-signal checks. They make the tool feel serious faster than more rhetoric.
* **Proposed fix:** Add these as HLT subrules with stable IDs, e.g. `HLT-016A-UNPINNED-ACTION`, `HLT-010A-ARTIFACT-SECRET`, `HLT-008A-TEST-ONLY`, `HLT-006A-MIGRATION-NO-ROLLBACK`.
* **Type:** Needs internal product building

### 10.

* **Severity:** High
* **Location:** `jankurai/cli.py`; hard/soft rule semantics
* **Issue:** Several current soft signals are likely to create false positives: “dead language” terms, “fallback,” “legacy,” “utils,” “helpers,” LOC caps, “non-optimal language,” and Python ratio.
* **Why it matters:** Engineers will tolerate noisy advisory findings, but not noisy blocking findings. False positives are especially risky for a standard trying to influence architecture.
* **Proposed fix:** Split each noisy heuristic into: `evidence-only`, `warning`, and `hard only with context`. For example, `fallback` in product runtime plus no reason code is a finding; `fallback` in docs or product copy is not. Add precision/recall tests over 10–20 fixture repos.
* **Type:** Needs internal product building

### 11.

* **Severity:** High
* **Location:** README; Quick Start; CI section
* **Issue:** The README is clear about what the project is and how to run the audit, but it does not yet show enough “trust artifacts”: no sample `repo-score.json`, no sample Markdown report screenshot, no known-false-positive list, no conformance badge criteria, and no demo repo walking from red to green. The public repo also currently shows no GitHub releases. ([GitHub][2])
* **Why it matters:** Serious engineers will not adopt a standard from prose alone. They need to see output quality, failure modes, and maintenance posture before trying it in CI.
* **Proposed fix:** Add a “What you get after 5 minutes” section with a real report excerpt, top 5 findings, exact agent-fix queue, and before/after repair. Publish a tagged release and attach sample artifacts.
* **Type:** Needs internal product building

### 12.

* **Severity:** High
* **Location:** README score guidance
* **Issue:** The README says scores below `70` “usually” mean a repo is not agent-operable and `85+` is the target floor, but there is no calibration data. ([GitHub][2])
* **Why it matters:** Numeric thresholds invite skepticism. Without data, the score can look like style preference disguised as measurement.
* **Proposed fix:** Rewrite as: “In v0.2 these thresholds are policy defaults, not empirical pass/fail boundaries.” Then create a calibration page with 10 repos, their scores, manual review notes, and known false positives.
* **Type:** Needs writing/editing

### 13.

* **Severity:** High
* **Location:** Section VI; Appendix A; standard docs
* **Issue:** Conformance needs machine-readable schemas, not just prose. The standard defines required output fields and conformance levels, but the repo should ship schemas for `repo-score.json`, `owner-map.json`, `test-map.json`, `generated-zones.toml`, `repair receipt`, and `exception`. ([GitHub][5])
* **Why it matters:** Standards become credible when third parties can validate conformance independently. Schemas also make findings more actionable for agents.
* **Proposed fix:** Add `/schema/v1/*.schema.json`, a `jankurai validate-config` command, fixture tests, and a “conformance test suite” that other repos can run.
* **Type:** Needs internal product building

### 14.

* **Severity:** High
* **Location:** Section XVI limitations and research agenda
* **Issue:** The limitations section is honest but too general. It needs a concrete evaluation plan.
* **Why it matters:** You explicitly say this is not yet a full empirical benchmark paper. That is fine, but reviewers will want to know exactly how you would validate the standard. SetupBench and ContextBench show that setup reliability and context retrieval are measurable bottlenecks, and SWE-agent/SWE-bench show that agent interfaces and repository tasks can be benchmarked. ([arXiv][9])
* **Proposed fix:** Add a “Planned Evaluation” subsection with metrics: audit precision/recall, repair success rate, escaped defect rate, review time, proof-lane runtime, agent context tokens, UX defect capture, secret/dependency detection, and migration-safety detection. Include a 3-phase study: fixture repos, historical PR replay, live adoption pilots.
* **Type:** Needs internal product building

### 15.

* **Severity:** High
* **Location:** Section XII; Table XIII; README Rendered UX QA
* **Issue:** The UX QA section is conceptually strong but should classify evidence by determinism. Objective geometry failures, accessibility rule failures, visual diffs, and AI/CV opinions are different kinds of evidence.
* **Why it matters:** A merge gate that blocks on deterministic overlap is credible. A merge gate that blocks because a VLM says the page “looks wrong” is not. The README already says deterministic geometry failures should block and AI/VLM opinions should stay review-only unless backed by deterministic evidence; the paper should make that taxonomy central. ([GitHub][2])
* **Proposed fix:** Add a table: `Block automatically`, `Require owner review`, `Warn only`, `Collect label`. Put geometry, target size, overflow, missing labels, and severe CLS in block/review categories; put AI/CV in warning/review-only.
* **Type:** Needs writing/editing

### 16.

* **Severity:** High
* **Location:** Section XII; `@jankurai/ux-qa`
* **Issue:** The paper cites strong UX tooling concepts, but `jankurai` should build the missing integration layer, not only name the ecosystem.
* **Why it matters:** Storybook is widely used for isolated UI states, Playwright supports screenshot comparison and ARIA snapshots, and WCAG target-size rules give objective accessibility thresholds. Jankurai’s opportunity is to connect these into one evidence contract. ([Storybook][10])
* **Proposed fix:** Build `jankurai ux plan` to discover changed UI paths, map them to Storybook stories/routes, run Playwright screenshots, run axe, capture ARIA snapshots, run geometry rules, and emit one `ux-qa.json` artifact with merge decision and repair hints.
* **Type:** Needs internal product building

### 17.

* **Severity:** High
* **Location:** Section XIII security; Table XV
* **Issue:** Security should treat agents as non-human identities with scoped credentials, not only as code generators.
* **Why it matters:** Agentic tools increasingly hold repo, terminal, browser, cloud, and API access. The paper covers excessive agency, secrets, and tool overreach, but it should explicitly discuss credential scope, ephemeral tokens, local secret access, artifact leakage, and approval boundaries. GitGuardian’s 2026 report page highlights AI-service leak growth and public GitHub secret volume, which directly supports this direction. ([GitGuardian][11])
* **Proposed fix:** Add a “Non-human identity controls” subsection: scoped tokens, no long-lived credentials in agent context, secret redaction in logs/artifacts, local pre-commit secret scan, CI artifact scan, permission receipts, and deny-by-default tool profiles.
* **Type:** Needs online research

### 18.

* **Severity:** High
* **Location:** Section XIV; Appendix C; repair receipts
* **Issue:** Repair receipts are a great concept, but they need a state machine and concrete examples.
* **Why it matters:** “Repair receipt” could become ceremonial unless agents know what to produce and CI knows what to verify.
* **Proposed fix:** Define statuses: `opened`, `triaged`, `repair-attempted`, `proof-passed`, `proof-failed`, `exception-filed`, `closed`. Add one full receipt for a secret, one for a generated-client drift, one for UX regression, and one for destructive migration.
* **Type:** Needs internal product building

### 19.

* **Severity:** Medium
* **Location:** Sections X and XV; `agent/standard-version.toml`
* **Issue:** Versioning is a strength, but citation/version hygiene needs the same discipline. The manifest binds paper, standard, audit, schema, and UX package artifacts, which is good; the research citations should get a similar ledger. ([GitHub][12])
* **Why it matters:** Tool docs and AI-agent behaviors change quickly. A reviewer may not know which claims are stable, which are version-specific, and which need revalidation.
* **Proposed fix:** Add `docs/research/citation-ledger.md` with columns: claim, source, access date, source type, confidence, archived/permalink, and whether it supports an empirical claim or policy choice.
* **Type:** Needs online research

### 20.

* **Severity:** Medium
* **Location:** Figure 2; Tables VIII–IX
* **Issue:** The stack-ranking chart crops the axis to `x >= 85`. The caption discloses this, but the figure still visually magnifies close differences.
* **Why it matters:** Skeptical readers are sensitive to chart framing when the scores are policy-weighted.
* **Proposed fix:** Keep the zoomed chart, but add a small full-scale inset or an adjacent table with uncertainty bands sorted by overlapping ranges. Emphasize that 94 vs. 90 is not a measured effect size.
* **Type:** Needs writing/editing

### 21.

* **Severity:** Medium
* **Location:** Open-source repo root
* **Issue:** The project needs governance artifacts to feel like a serious standard: `CONTRIBUTING.md`, `SECURITY.md`, release policy, schema SemVer policy, RFC process, issue templates, changelog, and maintainer/conformance roles.
* **Why it matters:** A standard needs a path for disagreement. Without governance, weights and rules feel like one author’s preferences rather than an evolving open standard.
* **Proposed fix:** Add a lightweight governance folder: `docs/governance/rfc-process.md`, `docs/governance/rule-change-policy.md`, `docs/governance/conformance.md`, plus issue templates for “rule proposal,” “false positive,” “schema change,” and “adopter report.”
* **Type:** Needs internal product building

### 22.

* **Severity:** Medium
* **Location:** Section XI; testing/proof lanes
* **Issue:** Proof lanes need more guidance on flakiness, budgets, determinism, and environment pinning.
* **Why it matters:** “Run the mapped proof lane” is only meaningful if the lane is reproducible, fast enough, and not full of flakes.
* **Proposed fix:** Add proof-lane acceptance criteria: max runtime, required command determinism, pinned toolchain, artifact capture, retry policy, flake quarantine format, and rule that retries cannot convert unknown failures into green without receipt.
* **Type:** Needs writing/editing

### 23.

* **Severity:** Medium
* **Location:** Section XII; human feedback loop
* **Issue:** The human feedback loop is well described but not productized.
* **Why it matters:** Feedback labels are where UX QA becomes better over time. Without a schema, labels will remain subjective comments in PRs.
* **Proposed fix:** Add `ux-labels.schema.json` with labels such as `true_layout_bug`, `intentional_change`, `acceptable_variance`, `false_positive`, `design_system_violation`, `a11y_defect`, `responsive_only`, `browser_only`, and `needs_product_review`. Store labels with route/story, viewport, selector, crop, owner, and expiration.
* **Type:** Needs internal product building

### 24.

* **Severity:** Medium
* **Location:** README; adoption path
* **Issue:** The README sells the project reasonably, but the conformance path may feel abstract after the first audit command.
* **Why it matters:** Engineers need a guided next step: “I ran it; now what?”
* **Proposed fix:** Add three adoption tracks: `Solo repo advisory`, `Team CI guarded`, `Org conformance`. For each, show required files, time estimate, blocking rules, sample CI config, and success criteria.
* **Type:** Needs writing/editing

### 25.

* **Severity:** Medium
* **Location:** Section V; Table IV; Section XIV
* **Issue:** Observability and repair opacity are underweighted at 3% in the TLR chart.
* **Why it matters:** Poor observability is not just a repair problem; it amplifies security, UX, data, and business-truth failures by making them harder to detect and localize.
* **Proposed fix:** Treat observability as a cross-cutting multiplier rather than a small bucket. Example: any high-risk finding without trace/error/receipt evidence gets a severity bump.
* **Type:** Needs writing/editing

## Net assessment

The paper is hardest to dismiss when it talks about agent-native repositories as verification interfaces: owner maps, proof lanes, generated zones, CI modes, repair receipts, and rendered UX artifacts. The paper is easiest to dismiss when it presents exact stack scores, exact RPN shares, and broad conformance claims before the tooling has enough calibration and real-world artifacts.

The next best move is not more rhetoric. It is a small, excellent v0.3: real hard gates, schemas, sample reports, sample repo, calibrated score caveats, and a visibly useful UX QA artifact workflow.
