# Executive diagnosis

The strongest paper identity is now:

> **Jankurai is the anti-vibe coding standard for evidence-carrying AI pull requests.**

That is powerful. It names a new defect class, gives the industry a memorable phrase, and proposes a versioned artifact model for reducing it. The current title package is close:

> **Anti-Vibe Coding Standard**
> **Continuous Repository Alignment for Proof-Carrying AI Pull Requests**
> **Find the vibe. Prove the merge. Repair the repo.**

This is much stronger than earlier titles. But for tier-1 acceptance, I would slightly adjust the formal/public balance:

## Best final title package

**Jankurai: The Anti-Vibe Coding Standard**
**Continuous Repository Alignment for Evidence-Carrying AI Pull Requests**
**Find the vibe. Prove the merge. Repair the repo.**

I would use **“evidence-carrying”** rather than **“proof-carrying”** in the subtitle unless the paper explicitly distinguishes itself from the formal “proof-carrying code” tradition. “Proof-carrying” is punchy, but in academic/security contexts it can imply formal proofs. The paper’s actual claim is evidence-backed merge, not formal semantic proof. You can still use **proof receipts** because those are operational command receipts.

Alternative strong formal title:

> **Evidence-Carrying Pull Requests: The Jankurai Standard for Anti-Vibe Coding and Continuous Repository Alignment**

Alternative public title:

> **Proof, Not Vibes: Jankurai and the Future of AI Code Review**

Best tier-1 compromise:

> **Jankurai: The Anti-Vibe Coding Standard for Evidence-Carrying Pull Requests**

# Top 12 fixes before wider release

## 1. Fix the section flow so the standard is visibly primary

The source now inputs the Core Standard early, then taxonomy, evaluation, agent controls, proof, UX, security, repair, governance, and only later language/stack sections. That is the right direction. The wrapper confirms this improved order. 

However, the language and stack sections still take up too much of the paper’s perceived identity. A skeptical reader could still walk away thinking: “This is a Rust/TS/Postgres manifesto with a standard attached.” The paper must make the opposite impossible.

## 2. Add a formal conformance predicate

Section III is strong, but it still needs the “standard in one sentence.” Add this near the beginning of Section III:

```text
A repository R at commit C conforms to Jankurai Core level L
under standard version S, auditor version A, and schema version K
iff the repository can emit schema-valid, current artifacts proving
that every required ownership, generated-zone, proof-lane,
security, waiver, repair, and witness predicate for L is satisfied at C.
```

This turns the paper from a persuasive essay into a real standard proposal.

## 3. Add a one-page running PR example that travels through the whole paper

The checkout-flow PR is a good opener, but it should become a recurring **worked example**:


Show this as:

That would make the entire standard intuitive.

## 4. Strengthen “vibe artifact” as the named defect class

The paper defines vibe artifacts well in Table I, but this term should be emphasized even more. It is the industry hook.

Suggested language:

> A vibe artifact is not merely bad code. It is repository residue that makes future human or agent work less reliable: an ownerless path, an unmapped proof lane, an unreceipted review claim, an edited generated file, a false-green test, an unproven UI change, a stale waiver, or a security-sensitive change without evidence.

Then connect it to metrics:

```text
vibe artifact density = open HLT findings / changed path, PR, KLOC, or ownership cell
```

This gives Jankurai a measurable adoption story.

## Avoid `table*` in appendices

Appendix A uses `table*` immediately after the section heading source, but the rendered PDF places the tables before the heading.  Use non-floating `tabularx` with `\captionof{table}`.

## Fix first page style

The frontmatter uses a custom title page, image placement, and abstract block.  For a tier-1 paper, choose one:

1. **Formal paper mode:** no titlepage, IEEE standard top matter, content begins page 1.
2. **Standard whitepaper mode:** keep titlepage, but add document metadata, canonical source, conformance status, and maybe a small diagram.

Right now it is halfway between.

## Add `\raggedbottom`

IEEE two-column layout with floats can create ugly vertical stretching. Add:

```latex
\raggedbottom
```

after `\begin{document}` or in frontmatter. This may reduce strange vertical gaps.

## Use non-floating listings for small examples

For code snippets that must stay near prose, use:

```latex
\begin{minipage}{\columnwidth}
\begin{lstlisting}[style=codebox]
...
\end{lstlisting}
\end{minipage}
```

For examples that can float, use `figure*`.
