I treated the uploaded **“Anti‑Vibe Coding Standard / Continuous Repository Alignment for Proof‑Carrying AI Pull Requests”** PDF as the latest paper: **public_draft, Standard 0.7.0, Auditor 0.7.0, Schema 1.5.0, Paper edition 2026.05-ed7**. I also reviewed the public `/paper/` source structure on GitHub, including the current TeX wrapper, front matter, outline, standard section, taxonomy section, evaluation section, related-work section, artifact-manifest appendix, and Markdown companion. # Executive verdict

This version is the strongest so far. The paper now has a real hook, a clearer standard boundary, a better title direction, better stack neutrality, a seed conformance-suite claim, a compact coverage summary instead of a giant appendix dump, and a much stronger sense that **Jankurai is not “an AI coding tool” but a versioned repository alignment standard**. The core industry idea is now visible:

> **The repository is the alignment layer.**

That idea is profound. It reframes the AI-coding problem away from “which model writes better code?” and toward “what repository interface makes every human or agent change auditable before merge?” The paper’s strongest standard primitive remains the **merge witness**: a binding among changed paths, owner routes, required proof receipts, missing evidence, artifact digests, commit identity, tool identity, and decision. That should remain the center of the paper. 

The main remaining problem is not the idea. The main problem is **tier‑1 presentation discipline**. Right now, the paper still has float/layout problems, some overly dense tables, a few terminology inconsistencies, a weak empirical-evidence section relative to the ambition, and several places where the argument briefly drifts from “standard” back toward “reference profile.” The next version should make the flow feel inevitable:

**vibe artifacts → owner route → proof lane → receipt → merge witness → repair queue → reduced baseline.**

That loop is the paper.

# Highest-priority fixes before public release

## P0. Fix the page 4 and page 7 float/layout failures

Page 4 has a severe layout problem. The section heading appears at the top left, but the actual body text begins far down the left column while JSON listings occupy the lower left and top right. The reader experiences the most important section—**Jankurai Core Standard and Conformance**—as visually broken. This is especially damaging because this section is the formal contribution. 

Page 7 is worse: it is almost entirely blank except for a small TLR/RPN chart placed near the vertical center. This looks accidental and will reduce confidence in the paper’s polish. It is likely caused by the combination of `figure*`, `table*`, and `\FloatBarrier` usage in the TeX wrapper and sections. The current wrapper uses multiple `\FloatBarrier` calls between major section inputs, which is sensible for controlling drift but can force isolated float pages when full-width floats cannot fit naturally. ([GitHub][1])

Recommended fix:

* Convert the page 7 TLR chart into a normal one-column table/figure inside Section IV, or place it directly under Table VI as a compact non-floating block.
* For page 4, replace the two JSON listings with one compact full-width figure titled **“Example HL3 Merge Witness”**.
* Avoid stacked `lstlisting` environments at the top of Section III.
* Use `\begin{figure*}[!t]` sparingly, and do not put small figures in `figure*`.
* Consider replacing some `\FloatBarrier` calls with section-level float discipline only after major table clusters.

The current TeX source defines the alignment-loop as a full-width `figure*`, the taxonomy chart as another full-width float, and the standard section uses listings and full-width checklist tables. Those are exactly the kinds of objects that can create the current blank-page behavior. ([GitHub][2])

## P0. Make the visible title start with “Jankurai”

The PDF metadata title is good—**“Jankurai: Anti‑Vibe Coding Standard”**—but the visible page-1 title begins with **“Anti‑Vibe Coding Standard”** and relies on the logo/wordmark for “Jankurai.” For citations, screenshots, search snippets, and academic review, the visible title should start with the project name. The current front matter source has `\title{Jankurai: Anti-Vibe Coding Standard}`, but the custom title page prints “Anti‑Vibe Coding Standard” as the main visible title.  ([GitHub][3])

Recommended visible title:

**Jankurai: The Anti‑Vibe Coding Standard**
**Continuous Repository Alignment for Proof‑Carrying AI Pull Requests**
**Find the vibe. Prove the merge. Repair the repo.**

This keeps the broad hook while making the standard citable.

## P0. Fix the command rendering bug on page 22

On page 22, the line appears visually/textually as:

`cargo run -p jankurai - versions`

It should be:

`cargo run -p jankurai -- versions`

The raw TeX source has `--`, but in normal text mode LaTeX converts double hyphens into an en dash unless protected. Put commands in `\verb|...|`, `\texttt{\detokenize{...}}`, or a `lstlisting` block. The raw appendix source includes the version verification command and says it verifies `VERSION`, Cargo metadata, CLI constants, documents, companion Markdown, artifact bindings, and generated-zone registration, so preserving the command exactly matters.  ([GitHub][4])

## P0. Add page numbers and version footer

The PDF currently lacks visible page numbers. For a draft standard, reviewers need stable page references. Add a footer such as:

`Jankurai 0.7.0 public_draft • schema 1.5.0 • paper 2026.05-ed7 • p. X`

This also reinforces the version-controlled standard identity.

## P0. Fix bibliography capitalization

The references still show capitalization issues such as **W3c**, **Openssf**, **rFC**, **nSA/CISA**, **mDN**, **rust**, and **postgreSQL**. This is a credibility issue in a standards paper. Protect proper nouns in BibTeX:

```bibtex
title = "{{W3C}} Process Document"
title = "{{OpenSSF}} Scorecard"
title = "{{RFC}} 9457"
title = "{{NSA}}/{{CISA}} ..."
title = "{{MDN}} Web Docs"
title = "{{Rust}} By Example"
title = "{{PostgreSQL}} Constraints"
```

This should be fixed before any broad release. 

## P0. Reduce visual density in the conformance tables

Table III on page 5 is important, but the “Requirement” column is too narrow, producing tall, awkward wrapped text. This makes the table look less authoritative than its content. Use a wider full-width table with shorter requirement phrases, or split it into two tables:

1. **Required artifacts by level**
2. **Meaning of each artifact**

The content is strong; the layout is weakening it. 

## P0. Make “seed conformance suite” more concrete

The evaluation section says the seed suite has **10 fixture directories and 12 expected JSON files**, with one passing HL3 fixture and nine blocking fixtures. That is good, but a tier‑1 paper needs one compact evidence table showing:

* fixture name,
* expected decision,
* expected rule ID,
* expected missing evidence,
* expected witness field,
* whether the current reference auditor passes it.

The paper currently gives fixture classes and a miniature walkthrough, but not a results matrix. Add a table titled:

**Current Seed Conformance Results**

Example:

| Fixture                        | Expected decision |    Rule | Current reference auditor |
| ------------------------------ | ----------------: | ------: | ------------------------: |
| `hl3-pass-minimal`             |            `pass` |    none |                      pass |
| `generated-zone-mutation-fail` |           `block` | HLT-002 |                      pass |
| `ownerless-path-fail`          |           `block` | HLT-003 |                      pass |

The current evaluation section already contains the necessary ingredients in prose and source. It just needs to look like evidence, not only a promise.  ([GitHub][5])

# LaTeX/source-level feedback

## Reformat TeX source for reviewability

Many TeX files are effectively long single-line or low-line-count files. The current wrapper is a one-line chain of `\input{...}` calls, and the front matter packs many macros and content into very long lines. This makes diffs and reviews harder, which is ironic for a paper about auditable structure. ([GitHub][1])

Recommended style:

```tex
\input{paper/tex/frontmatter}

\input{paper/tex/sections/01_new_bottleneck}
\input{paper/tex/sections/02_definitions_threat_model}
\FloatBarrier

\input{paper/tex/sections/06_jankurai_standard}
\FloatBarrier
```

Also reflow prose to one sentence or one clause per line. This makes Git diffs meaningful.

## Split large tables into separate files

For example:

```text
paper/tex/tables/artifact_checklist.tex
paper/tex/tables/fault_taxonomy.tex
paper/tex/tables/conformance_suite.tex
paper/tex/tables/related_work.tex
paper/tex/figures/alignment_loop.tex
```

Then the section files become readable.

## Use generated files only for generated content

Generated registry summaries should live under:

```text
paper/tex/generated/
```

Human-authored standards tables should live under:

```text
paper/tex/tables/
```

This separation matters because Jankurai is about generated-zone policy.

## Fix float policy

Current source uses `\FloatBarrier` between major sections, which helps prevent drift but also contributes to isolated float pages when combined with `table*` and `figure*`. ([GitHub][1])

Recommendations:

* Avoid `figure*` for small figures.
* Use `table*` only for essential wide tables.
* Use `\afterpage{\clearpage}` sparingly.
* Consider `stfloats` or `dblfloatfix` if IEEE constraints permit.
* Convert some wide tables into one-column compact tables.
* For tiny charts, use inline `center` + `tabular`, not floating `figure*`.

## Use `\verb` or `lstlisting` for commands

Any command with `--` must be protected. The page 22 `cargo run -p jankurai -- versions` issue is the clearest example. Use:

```tex
\verb|cargo run -p jankurai -- versions|
```

or:

```tex
\begin{lstlisting}[style=codebox]
cargo run -p jankurai -- versions
\end{lstlisting}
```

## Add page footer and headers

Use `fancyhdr` carefully with IEEE, or a simple footer macro. At minimum, visible page numbers are needed.

## Add PDF accessibility work

The PDF metadata is now populated, which is good, but the PDF is not tagged. A future release should add:

* tagged PDF if feasible,
* alt text for figures,
* tables readable in grayscale,
* no reliance on color alone,
* bookmarks.

# The strongest framing to preserve

The paper should repeatedly reinforce these five claims:

1. **The repository is the alignment layer.**
2. **Vibe artifacts are repository states, not feelings.**
3. **Every changed path needs ownership and proof routing.**
4. **Every proof needs a receipt.**
5. **Every merge needs a witness.**

That is the tier‑1 idea.

The paper’s most important public sentence should be:

> Jankurai turns vibe coding into repairable evidence.

The paper’s most important standard sentence should be:

> A Jankurai-conformant repository emits current, schema-valid artifacts sufficient to audit the merge decision at the current commit.

The paper’s most important engineering sentence should be:

> No proof, no merge; no receipt, no trust.
