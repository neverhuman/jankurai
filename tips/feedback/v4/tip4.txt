# Abstract feedback

## What works

The abstract is now focused and compelling. It names the exact artifacts the standard controls: ownerless paths, unmapped proof, hand-edited generated zones, stale contracts, false-green tests, missing security evidence, overbroad permissions, unproven UI changes, and unreceipted review claims. It defines the merge witness as the central artifact and keeps the claim bounded. 

## What is missing

The abstract should explicitly say **what is new** compared with existing CI/security/provenance systems.

Right now, it says Jankurai is a standard for turning vibe coding into repairable evidence. Good. But the reader may ask:

> How is this different from CI, CODEOWNERS, SLSA, OpenSSF Scorecard, SARIF, or policy-as-code?

Add one sentence:

> Unlike CI dashboards or supply-chain attestations alone, Jankurai binds path ownership, proof requirements, generated-zone authority, missing evidence, repair queues, and merge decision into one repository-local witness.

That sentence would clarify the unique integration.

## Suggested revised abstract

> AI-assisted coding makes plausible code cheap and vibe-code drift expensive. Jankurai is a versioned repository conformance standard for finding and reducing vibe artifacts: ownerless paths, unmapped proof, hand-edited generated zones, stale contracts, false-green tests, missing security evidence, overbroad agent permissions, unproven UI changes, and unreceipted review claims. Its central artifact is the merge witness: a versioned binding among changed paths, owner routes, required proof receipts, observed evidence, score deltas, missing-evidence decisions, tool identity, artifact digests, and commit identity.
>
> Jankurai treats the repository as the alignment layer for human- and agent-authored code. A local, pull-request, and CI loop maps each changed path to bounded authority, proof lanes, evidence receipts, repair queues, or expiring waivers. Unlike CI dashboards or provenance attestations alone, Jankurai binds ownership, proof obligation, generated-zone authority, missing evidence, repair action, and merge decision into one repository-local witness.
>
> The paper contributes a stack-neutral repository model, a closed conformance decision interface, stable HLT rule IDs, proof-receipt validity rules, merge-witness schemas, repair queues, and a seed conformance suite. The claim is bounded: Jankurai does not prove full program semantics, does not require a Rust/TypeScript/PostgreSQL stack, and does not treat scores as merge approval. It proposes a version-controlled standard for turning vibe coding into repairable evidence.

# LaTeX and production recommendations

The GitHub paper directory is organized with `figures`, `sections`, `tex`, `citation-index.md`, `jankurai.md`, `jankurai.pdf`, `jankurai.tex`, `outline.md`, and bibliography files. That modular direction is good. ([GitHub][9])

The outline confirms the intended section order: frontmatter, From Vibe Coding to Verified Merge, Definitions/Threat Model, Jankurai Core, Taxonomy, Evaluation, Agent Controls, Continuous Proof, Rendered UX, Security, Waivers, Migration/Governance, then language/profile material, Related Work, Limitations/Conclusion, and appendices. That flow is now much better than previous drafts. ([GitHub][10])

## Fix float behavior

Current problem pages: 4, 7, 18, 24.

Likely causes:

* Too many `table*` and `figure*` floats.
* `[t]` placement on full-width floats.
* `\FloatBarrier` after major sections.
* Large code listings inside two-column flow.
* Long `lstlisting` blocks splitting badly.

Practical fixes:

```latex
\usepackage{stfloats}
\usepackage{dblfloatfix}
```

But the better fix is structural:

* Use fewer `table*` floats.
* Move long tables to appendix.
* Keep figures close to first mention.
* Convert Figure 2 into a one-column `table` or `minipage`.
* Use `\begin{figure}[!t]` not `figure*` unless truly needed.
* For code examples, use compact excerpts in the body and full examples in appendix.

## Add page numbers

IEEE conference mode often hides or minimizes page numbering. For draft/public standard review, add visible page numbers. A standards draft needs stable references.

## Add line numbers for review drafts

For public draft review, consider:

```latex
\usepackage[switch]{lineno}
\linenumbers
```

Use only in review builds.

## Fix code listing splits

Avoid listings that split across pages/columns. For examples:

```latex
\begin{lstlisting}[style=codebox,float=!t]
...
\end{lstlisting}
```

Or better, make code examples shorter.

## Improve source maintainability

The current raw `frontmatter.tex` is effectively minified into extremely long lines. Split it into readable LaTeX blocks. This will make peer review, diffs, and future standard updates much easier. ([GitHub][2])

Recommended file layout:

```text
paper/tex/
  preamble.tex
  metadata.tex
  titlepage.tex
  macros.tex
  listings.tex
  tables.tex
  frontmatter.tex
```

## Add draft/release build modes

Use a boolean:

```latex
\newif\ifdraftmode
\draftmodetrue
```

Draft mode:

* page numbers
* line numbers
* wider spacing
* warning boxes for unstable sections

Release mode:

* compact
* no line numbers
* polished figures

# Content that feels off or risky

## “Anti-Vibe Coding Standard” is strong but may sound informal

Keep it, but pair it with “Continuous Repository Alignment” so serious readers understand the formal category.

## “Proof-carrying AI pull requests” is powerful but underdefined

Define it early.

## The reference profile still takes too much attention

The current version is much better because profile sections come later and are clearly non-normative. Still, pages 16–19 spend substantial space on language history, standard gravity, scoring, and score decomposition. For a tier-1 standard paper, that space may be better spent on evaluation, conformance fixtures, and worked examples.

## “0 unmapped rows” may sound stronger than it is

Appendix A says 260 mapped, 17 detector-backed, 243 partial. That is good, but “0 unmapped” can sound like broad coverage. Always pair it with the detector-backed count. 

Recommended wording:

> v0.7.0 maps all 260 source rows to a control family, but only 17 are detector-backed. The remaining 243 are governed mappings that require fixtures, CI evidence, or organizational receipts.

## “Real-time repository conformance” needs precision

The definition says it is a repeatable local/PR/CI loop, not an omniscient daemon. Good. But the title/abstract should avoid overpromising real-time continuous monitoring unless a daemon exists. 

## “Score” can distract from conformance

The paper says scores are posture and prioritization, not merge approval. Good. Repeat this in every scoring/reference-profile section.

# Bottom-line recommendation

This version is very close to having the right identity. The paper should now emphasize:

> **Jankurai is the anti-vibe coding standard for proof-carrying pull requests. It continuously aligns the repository by converting vibe artifacts into owner-routed, proof-backed, repairable evidence before merge.**

To make it tier-1, the next version needs:

1. A polished worked example.
2. Observed conformance-suite results.
3. A formal Core public API box.
4. A conformance-claim/badge rule.
5. Cleaner float/layout handling.
6. Less main-body stack scoring.
7. Expanded related work.
8. More precise evidence security and receipt validity.
9. Page numbers and accessible output.
10. A stronger empirical roadmap with measurable outcomes.

The profound idea is already there. The next step is to make the paper read like the founding document of a standard: **clear claims, runnable evidence, stable artifacts, reproducible decisions, and no ambiguity about what is normative.**
