I reviewed the latest 26-page v0.7.0 PDF and the current GitHub paper source layout. This version is a major improvement over the earlier drafts: the hook is stronger, the standard is now moved earlier, “vibe artifact” is formalized, the old giant coverage appendix is replaced by a compact registry summary, the merge witness is now central, and the reference profile is explicitly non-normative. The paper now looks like it can become a real standard, not just a strong opinion piece. 

The main thing still holding it back from a tier-1 paper is **not the core idea**. The core idea is strong. The gap is that the paper still needs to feel less like “a compelling standard proposal with implementation notes” and more like **the first serious specification of a new industry category**:

> **Continuous Repository Alignment:** a versioned standard for detecting vibe artifacts, routing them to owners, requiring proof receipts, emitting merge witnesses, and turning missing evidence into repair work.

That is the profound industry claim. The paper now says this, but it should be made even more explicit and structurally dominant.

# 1. Highest-impact recommendations

## 1.1 The title is much better, but the title stack should include “Jankurai” visibly

The current first page uses:

> **Anti-Vibe Coding Standard**
> **Continuous Repository Alignment for Proof-Carrying AI Pull Requests**
> **Find the vibe. Prove the merge. Repair the repo.**

This is a much stronger hook than earlier versions. It finally names the problem in a way a broad audience can feel. But the visible title does not prominently say **Jankurai** except in the logo and project line. For a standard that you want people to cite, adopt, and talk about, the title itself should carry the brand.

Recommended title stack:

> **Jankurai: The Anti-Vibe Coding Standard**
> **Continuous Repository Alignment for Proof-Carrying AI Pull Requests**
> **Find the vibe. Prove the merge. Repair the repo.**

Even stronger and more precise:

> **Jankurai: The Anti-Vibe Coding Standard**
> **Continuous Repository Alignment for Evidence-Carrying AI Pull Requests**
> **Find the vibe. Prove the merge. Repair the repo.**

I would seriously consider **“Evidence-Carrying”** instead of **“Proof-Carrying.”** “Proof-carrying” has a formal-methods association with proof-carrying code. If you keep “proof-carrying,” add one sentence in the abstract or definitions:

> In this paper, “proof” means repository-local evidence receipts, not a formal proof of full program semantics.

That prevents reviewers from objecting that the title overclaims.

## 1.2 Add a boxed “standard thesis” on page 1

The abstract is good, but page 1 needs one unforgettable thesis box. Add it between the metadata and the abstract:

> **Thesis.** AI makes plausible code cheap. Jankurai makes merge evidence explicit: every changed path must route to an owner, a proof lane, a receipt, a witness, and a repair queue when evidence is missing.

Or:

> **Jankurai turns vibe coding from an invisible habit into an auditable repository state.**

This should become the paper’s signature line.

## 1.3 Add one concrete end-to-end PR walkthrough in the body

The running checkout-flow example is strong, but it remains mostly conceptual. A tier-1 paper needs one **worked PR** that shows the system failing and then passing.

Add a compact two-column figure or table:

**Initial PR:**

```text
Changed:
  apps/web/src/checkout/Checkout.tsx
  apps/web/src/generated/client.ts
  contracts/openapi/api.yaml

Findings:
  HLT-002-GENERATED-MUTATION
  HLT-013-RENDERED-UX-GAP

Witness:
  decision = block
  missing_evidence = ["contract_receipt", "rendered_ux_receipt"]
```

**After repair:**

```text
Repair:
  edit source contract
  regenerate client
  run contract lane
  run UX lane
  attach screenshot/ARIA/geometry evidence

Witness:
  decision = pass
  missing_evidence = []
```

This would make the entire standard click instantly.

## 1.4 Add observed conformance-suite results, not just fixture descriptions

The Evaluation section is much improved. It says the current seed suite contains 10 fixture directories and 12 expected JSON files, including `hl3-pass-minimal` and nine fail fixtures, and that validation runs through `rtk just conformance`. That is exactly the right direction. ([GitHub][1])

But for tier-1 credibility, add an observed-results table:

| Fixture                      | Expected decision | Observed decision | Expected rule | Observed rule | Schema valid? | Status |
| ---------------------------- | ----------------: | ----------------: | ------------- | ------------- | ------------: | ------ |
| hl3-pass-minimal             |              pass |              pass | none          | none          |           yes | pass   |
| generated-zone-mutation-fail |             block |             block | HLT-002       | HLT-002       |           yes | pass   |
| ownerless-path-fail          |             block |             block | HLT-003       | HLT-003       |           yes | pass   |

This would turn “seed suite exists” into “seed suite demonstrates conformance behavior.”

## 1.5 Fix the page-4 layout urgently

Page 4 is the most visibly broken page. The section title appears at the top left, but the left column has enormous blank space. The right column begins with the second JSON block. The first JSON block continues at the bottom of the left column, and the second block starts at the top of the right column, which makes the page feel out of order. 

The LaTeX source for the Core Standard section places the formula and two consecutive `lstlisting` blocks directly after the introductory paragraph. That source structure is causing the bad pagination. ([GitHub][2])

Fix by doing one of these:

1. Replace the two JSON blocks with one compact “merge witness excerpt.”
2. Move the full JSON example to Appendix E.
3. Put the example in a `figure*` spanning both columns.
4. Use a short table instead of JSON in the body.

Best body version:

```json
{
  "standard_version": "0.7.0",
  "commit": "9f3a1c4",
  "changed_paths": 3,
  "required_receipts": ["web", "contract", "security"],
  "present_receipts": ["contract"],
  "missing_evidence": ["rendered_ux_receipt"],
  "decision": "block"
}
```

Then put the full witness schema in an appendix.

## 1.6 Reduce stack-profile visual dominance

The reference profile is now clearly non-normative, which is excellent. The main TeX source also now places the standard and evaluation before the language/reference-profile sections, which is the right order. ([GitHub][3])

However, pages 15–18 still give a lot of visual weight to language evolution, standard gravity, scoring rubrics, and stack comparison. The paper says the durable contribution is not stack ranking, but the middle of the document still invites readers to debate the stack instead of the standard.

Move the largest scoring tables to an appendix or compress them into one summary page. Keep the reference profile, but make the standard visually dominant.

# 3. Section-by-section content review

## Abstract

**Strong:**
The abstract is now much better. It names vibe artifacts, merge witness, proof receipts, repair queues, and the bounded claim. It correctly says the result is not a larger prompt or hosted trust service, but a reproducible merge decision auditable at the current commit. 

**Needs improvement:**
Add “Jankurai” to the first sentence or title stack more visibly. Also clarify “proof-carrying” if the title keeps that term.

Suggested replacement first sentence:

> AI-assisted coding makes plausible code cheap, but it also leaves vibe artifacts: repository states that look acceptable until they are asked to prove ownership, security, UX, generated-boundary, and merge evidence.

This is more concrete and memorable.

---

## Section I: From Vibe Coding to Verified Merge

**Strong:**
Excellent opening example. The four operating claims are clear and compelling. The section says the repository, not prompt or reviewer memory, must encode ownership, allowed edit scope, proof lanes, generated boundaries, permission limits, evidence receipts, repair queues, and versioned conformance. 

**Needs improvement:**
This section should end with a more formal “new category” statement:

> We call this continuous repository alignment: keeping repository policy, proof, evidence, repair, and merge decisions synchronized as code changes.

The phrase appears elsewhere, but it should be defined as the field-level category right here.

---

## Section II: Definitions and Threat Model

**Strong:**
Much improved. The definitions are operational. The threat model includes false plausibility, fabricated/stale receipts, adapter drift, flaky proof lanes, emergency bypass, evidence leakage, CI downgrade, malicious MCP/tool schema, generated test self-confirmation, human rubber-stamp review, and screenshot/log leakage. 

**Needs improvement:**
Add a short “Claim Boundaries” box:

**Jankurai claims:**

```text
changed paths can be routed
required proof can be identified
receipts can be validated for freshness
missing evidence can become repair work
merge decisions can be reproduced
```

**Jankurai does not claim:**

```text
full semantic correctness
formal verification
malicious compiler/runtime defense
security certification without project evidence
universal stack superiority
```

The limitations section says some of this, but putting it early will increase trust.

## Section III: Jankurai Core Standard and Conformance

**Strong:**
This is now in the right place. The section is the heart of the paper. The closed decision enum, required HL3 artifacts, receipt invalidation rules, and conformance/score separation are all important. The source section also clearly says alternate stacks can conform if they provide equivalent owner routing, generated boundaries, proof lanes, security evidence, UX evidence, repair artifacts, and merge witnesses. ([GitHub][2])

**Needs improvement:**
Move the merge-witness JSON example to a cleaner format. Add a named normative requirement system:

```text
JNK-CORE-001: A conformant implementation MUST emit a merge witness.
JNK-CORE-002: A merge witness MUST include changed_paths, commit identity, required_receipts, present_receipts, missing_evidence, and decision.
JNK-CORE-003: A proof receipt MUST be invalidated when commit, command, lane, artifact digest, or path scope changes.
```

This makes the standard citable.

# 5. LaTeX and source-structure feedback

The GitHub `paper/` directory is well organized at the top level: it contains `figures`, `sections`, `tex`, `citation-index.md`, `jankurai.md`, `jankurai.pdf`, `jankurai.tex`, `outline.md`, and bibliography files. ([GitHub][9])

The main `jankurai.tex` is a single-line wrapper with many `\input{...}` calls and several `\FloatBarrier` / `\clearpage` controls. The modular input structure is good, but the file should be formatted one input per line so diffs are readable. ([GitHub][3])

## Recommended source restructure

```text
paper/
  jankurai.tex
  tex/
    preamble/
      packages.tex
      listings.tex
      colors.tex
      tables.tex
      metadata.tex
    frontmatter/
      titlepage.tex
      abstract.tex
    sections/
      01_intro.tex
      02_definitions_threat_model.tex
      03_core_standard.tex
      ...
    appendices/
      a_rule_registry.tex
      b_artifact_manifest.tex
```

## Specific LaTeX fixes

Use `microtype` for better justification:

```latex
\usepackage{microtype}
```

Use `balance` for references:

```latex
\usepackage{balance}
...
\balance
\bibliography{paper/references}
```

Make code blocks more stable:

```latex
\usepackage{tcolorbox}
\tcbuselibrary{listings,breakable}
```

For the merge witness JSON, use a full-width float:

```latex
\begin{figure*}[t]
\begin{lstlisting}[style=codebox]
...
\end{lstlisting}
\caption{Minimal merge witness excerpt.}
\end{figure*}
```

Add clear page breaks before appendices:

```latex
\clearpage
\appendices
\onecolumn
```

Then each appendix:

```latex
\clearpage
\section{Versioned Artifact Manifest}
```

Avoid starting an appendix at the bottom of a page.

# 8. Final assessment

This version is the strongest yet. The new framing around **vibe artifacts**, **continuous repository alignment**, and **proof-carrying AI pull requests** is the right direction. The standard is now much clearer: Jankurai is not a hosted AI trust service, not a stack mandate, and not a score-as-merge-approval system. It is a repository-local conformance model that makes merge evidence reproducible. 

The paper’s most important idea is:

> **The repository is the alignment layer.**

Everything should serve that.

The paper becomes tier-1 when a reader can say, after three pages:

> “I understand the new category. I understand the artifact. I understand how a PR fails. I understand how it gets repaired. I understand what conformance means. I understand why this should become a standard.”

You are very close. The remaining work is mostly presentation discipline, formal conformance clarity, and stronger evidence packaging.
