I studied the latest uploaded 26-page v0.7.0 PDF and the current LaTeX structure in `paper/`. This draft is a major improvement: the paper now has a stronger hook, a cleaner stack-neutral claim, a real conformance/evaluation section, a compact rule registry, and a much clearer thesis around **vibe artifacts**, **continuous repository alignment**, **proof receipts**, **merge witnesses**, and **repair queues**. The current version is far closer to a publishable standards paper than the earlier drafts. 

The biggest remaining issue is not the idea. The idea is strong. The biggest issue is **presentation discipline**: the paper still sometimes reads like a manifesto, standards spec, implementation manual, stack argument, and generated appendix compressed into one IEEE two-column article. For tier-1 impact, the next draft should make the core claim unavoidable:

> **Jankurai is a versioned anti-vibe coding standard that turns every risky pull request into a proof-carrying merge decision: changed paths, owners, required lanes, valid receipts, missing evidence, artifact digests, repair queue, and witness decision.**

That is the industry-shifting part. Everything else should support it.

# Executive verdict

The latest paper’s strongest move is the title shift to **“Anti-Vibe Coding Standard”** and the subtitle **“Continuous Repository Alignment for Proof-Carrying AI Pull Requests.”** This is much more powerful than the earlier “trustworthy merge” framing. It names a problem people recognize, then proposes a standard for reducing it. The abstract now says the key thing: Jankurai finds and reduces **vibe artifacts** such as ownerless paths, unmapped proof, hand-edited generated zones, stale contracts, false-green tests, missing security evidence, overbroad permissions, unproven UI changes, and unreceipted review claims. That is a strong category definition. 

The core paper should now be sharpened around five artifacts:

| Artifact             | Why it matters                                   |
| -------------------- | ------------------------------------------------ |
| **Vibe artifact**    | Names the failure mode in repository terms.      |
| **Proof receipt**    | Turns “tests passed” into replayable evidence.   |
| **Merge witness**    | Turns a PR decision into an auditable object.    |
| **Repair queue**     | Turns missing evidence into bounded next action. |
| **Version manifest** | Makes the standard updatable and auditable.      |

The paper’s greatest remaining weakness is that it still does not give the reader a single clean “Jankurai in one page” object model early enough. Page 1 has a strong title and abstract. Page 2 has a good running example. Page 3 has a good loop and definitions. But the reader should see, by page 2 or 3, a compact artifact diagram that says:

```text
changed paths
  -> owner route
  -> proof lanes
  -> proof receipts
  -> evidence index
  -> merge witness
  -> pass / review / block / ratchet_fail / release_fail
  -> repair queue or waiver
```

The current Figure 1 says this textually, but visually it looks like a boxed sentence. It should become the signature diagram of the standard.

# Highest-priority fixes

## 1. Make the title include “Jankurai”

The title page currently visually shows the Jankurai logo, but the actual visible title text starts with **“Anti-Vibe Coding Standard.”** The PDF metadata and LaTeX title say `Jankurai: Anti-Vibe Coding Standard`, but the rendered page title itself does not clearly begin with “Jankurai.” The frontmatter source sets `\title{Jankurai: Anti-Vibe Coding Standard}`, but the custom title page renders only “Anti-Vibe Coding Standard” as the large title. Use:

> **Jankurai: The Anti‑Vibe Coding Standard**
> **Continuous Repository Alignment for Proof‑Carrying AI Pull Requests**
> **Find vibe artifacts. Prove the merge. Repair the repo.**

I would replace **“Find the vibe”** with **“Find vibe artifacts”**. “Find the vibe” is catchy but slightly vague. “Find vibe artifacts” is category-defining.

## 3. Fix the page 4 layout immediately

Page 4 is the most serious layout problem. Section III starts on the left column, then the merge-witness equation appears low in the left column, the first JSON listing starts near the bottom, and the second listing appears at the top of the right column before the explanatory text. It visually feels broken and makes the paper look less polished than the idea deserves. 

The cause is that the merge witness example is too large for a normal two-column flow. In the TeX source, the merge equation and two `lstlisting` blocks are embedded directly in the section body. ([GitHub][2])

Fix it by making the witness example a full-width `figure*` or `table*`:

```latex
\begin{figure*}[t]
\centering
\begin{minipage}{0.96\textwidth}
\textbf{Example merge witness for the running checkout PR.}
\begin{lstlisting}[style=codebox]
...
\end{lstlisting}
\end{minipage}
\caption{A merge witness binds changed paths, owner routes, proof receipts, missing evidence, artifact digests, and decision.}
\label{fig:merge-witness-example}
\end{figure*}
```

Better yet, show **one** compact JSON object, not two.

## 4. Do not let floats split the argument

Several large tables float away from their sections. Page 5 starts with the normative artifact checklist, then the text continues from the prior standards paragraph. Page 6 starts with control-plane surfaces, then Section IV begins mid-page. Page 7 is almost entirely an RPN chart. Page 19 starts with the spine diagram, then ownership cells, then Related Work and Limitations begin on the same page. This causes the reader to feel like the paper is assembled from strong pieces rather than flowing as a tier-1 argument. 

You are already using `\FloatBarrier` in the wrapper `paper/jankurai.tex`, but the source has many full-width floats in two-column IEEE mode. The wrapper file is a one-line sequence of `\input{...}` calls with several barriers, appendices, and bibliography included at the end. ([GitHub][3])

For the next draft, either:

1. Produce a **single-column whitepaper/spec edition** for broad industry adoption, and a separate shorter IEEE-style academic version; or
2. Keep IEEE two-column, but reduce body tables by 40-50% and move most large matrices to appendices.

A standard paper with code blocks, schemas, and policy tables fights the two-column layout. The idea may land better in a clean one-column spec-paper format.

## 5. Add numbered normative requirements

The standards section currently uses strong MUST/SHOULD language, but it does not have stable requirement IDs. For a version-controlled standard, requirement IDs matter as much as rule IDs.

Add:

```text
JKR-001 Version Identity
A conformant repository MUST bind standard_version, auditor_version, schema_version, and commit identity in emitted reports.

JKR-002 Changed-Path Inventory
A conformant merge decision MUST include changed paths and generated-zone touches.

JKR-003 Owner Routing
A conformant repository MUST map changed paths to owner routes or emit HLT-003.

JKR-004 Proof Routing
A conformant repository MUST map changed paths to proof lanes or emit HLT-004.

JKR-005 Proof Receipts
A conformant HL3+ merge MUST include current valid receipts for required lanes.

JKR-006 Merge Witness
A conformant HL3+ merge decision MUST emit a merge witness with a closed decision enum.

JKR-007 Receipt Invalidation
A receipt MUST be invalid when commit, command, tool identity, path scope, digest, or lane authorization does not match.

JKR-008 Waivers
A standards deviation MUST be named, scoped, owned, justified, testable, and expiring.

JKR-009 Scores Are Not Approval
A score MUST NOT be represented as merge approval without required current evidence.

JKR-010 Generated Zones
Generated outputs MUST be repaired by source contract or generator change, not hand edit.
```

This would make Jankurai feel like a true standard rather than a persuasive framework.

## 8. Rename remaining “exceptions” in commands and paths

The paper now correctly reserves **exceptions** for runtime/program errors and uses **waivers** for standard deviations. That is a strong improvement. The source section explicitly says standards deviations are waivers and runtime exceptions are program errors. ([GitHub][5])

But the control-plane table still lists `exceptions expire` as a command, and older paths may still use `docs/exceptions/`. The paper should either explain that this is a legacy command name scheduled for deprecation or rename it consistently:

```text
jankurai waivers expire
docs/waivers/
```

Use:

```text
runtime exceptions
standards waivers
repair receipts
```

Do not use “exception” for standards deviation anywhere in the final text.

## 9. Make “proof-carrying pull requests” precise

The subtitle **“Proof-Carrying AI Pull Requests”** is strong, but it risks sounding like formal proof or proof-carrying code. The abstract correctly bounds the claim by saying Jankurai does not prove full program semantics. Still, add one sentence early:

> In this paper, “proof-carrying pull request” means evidence-carrying, not formally verified: the PR carries valid receipts, artifacts, digests, and a witness sufficient for the repository’s declared conformance level.

This prevents formal-methods readers from rejecting the phrase.

## 10. Make the LaTeX source diff-friendly

This is important because the project is about a **version-controlled standard**. The source itself should model that discipline.

Right now, `paper/jankurai.tex` is a single line of many `\input{...}` statements. ([GitHub][3]) The frontmatter file is also extremely compressed: it contains document class, package imports, metadata, custom macros, abstract text, title page, and index terms in only a handful of very long lines. ([GitHub][1]) Most section files are similarly minified into long lines. This makes reviews, diffs, blame, and standard governance harder.

Reformat the TeX source so every sentence or logical clause is diffable:

```latex
\documentclass[10pt,conference]{IEEEtran}

\usepackage[T1]{fontenc}
\usepackage{cite}
\usepackage{graphicx}
...
```

And:

```latex
\input{paper/tex/frontmatter}

\input{paper/tex/sections/01_new_bottleneck}
\input{paper/tex/sections/02_definitions_threat_model}
\FloatBarrier

\input{paper/tex/sections/06_jankurai_standard}
\FloatBarrier
...
```

Add `latexindent` as a paper CI step. A standards repo should not have minified normative source.

# LaTeX and source-level recommendations

The paper source structure is conceptually good: `paper/jankurai.tex` includes frontmatter, numbered sections, appendices, and bibliography. ([GitHub][3]) But the source is not yet standard-grade for collaborative versioning.

## Fix source readability

Current raw files are too compressed. Reformat with one sentence or logical unit per line. This matters because:

* pull request diffs become meaningful,
* reviewers can comment on specific sentences,
* governance discussions can cite exact lines,
* standards updates become auditable,
* merge conflicts become less likely.

## Split frontmatter

`frontmatter.tex` currently includes packages, metadata, title page, logo macros, abstract, and index terms in very long lines. ([GitHub][1]) Split into:

```text
paper/tex/preamble/packages.tex
paper/tex/preamble/listings.tex
paper/tex/preamble/metadata.tex
paper/tex/frontmatter/title_page.tex
paper/tex/frontmatter/abstract.tex
```

## Create macros for version values

Do not hard-code versions in many places. Use:

```latex
\newcommand{\JankuraiStandardVersion}{0.7.0}
\newcommand{\JankuraiAuditorVersion}{0.7.0}
\newcommand{\JankuraiSchemaVersion}{1.5.0}
\newcommand{\JankuraiPaperEdition}{2026.05-ed7}
```

Then use those in title page, examples, appendix manifests, and captions. This prevents version drift.

## Use a generated version check

Add CI:

```bash
jankurai paper verify-versions
```

It should fail if the paper contains stale strings such as `0.6.1`, `0.4.0`, `v0.6.1`, or mismatched `schema_version`.

## Make code blocks smaller and cleaner

For two-column mode, use:

```latex
\lstset{
  basicstyle=\ttfamily\footnotesize,
  breaklines=true,
  columns=fullflexible
}
```

But for large JSON/TOML examples, prefer full-width floats or appendix listings.

## Improve accessibility

The PDF is not tagged, and text extraction shows odd line-break/control artifacts. For a standards project, the paper should be accessible and machine-readable.

Add or verify:

```latex
\input{glyphtounicode}
\pdfgentounicode=1
\usepackage{microtype}
```

Also consider publishing:

```text
paper/jankurai.pdf
paper/jankurai.html
paper/jankurai.md
spec/jankurai-standard.md
schemas/*.json
```

The PDF should not be the only canonical reading surface.

# The main message to protect

The final paper should make this impossible to miss:

> **Jankurai does not ask teams to trust AI less. It asks repositories to trust evidence more.**

That is the industry-level idea.

The public hook is:

> **Find vibe artifacts. Prove the merge. Repair the repo.**

The standard object is:

> **merge witness = changed paths + owner route + proof receipts + missing evidence + artifact digests + commit/tool identity + repair queue**

The adoption promise is:

> **A repository can become continuously aligned without a rewrite: start advisory, route ownership, map proof, emit receipts, block high-risk missing evidence, ratchet posture, and publish conformance.**

The tier-1 version should look less like a collection of good tables and more like the first credible standard for evidence-carrying AI pull requests. The pieces are now present. The next step is ruthless flow, cleaner layout, numbered requirements, visible conformance results, and a front-page artifact model that people can remember.
