# Executive assessment

The paper’s strongest current positioning is:

> **Jankurai is the anti-vibe coding standard: it finds vibe artifacts, routes them to proof, records the merge witness, and turns missing evidence into repair.**

That is a genuinely strong industry thesis. It is larger and more memorable than “trustworthy merge.” The current title page now says:

> **Anti-Vibe Coding Standard**
> **Continuous Repository Alignment for Proof-Carrying AI Pull Requests**
> **Find the vibe. Prove the merge. Repair the repo.**

This is directionally excellent. It names the pain: vibe coding. It names the mechanism: continuous repository alignment. It names the artifact: proof-carrying pull requests. It names the loop: find, prove, repair. 

However, I would still refine the title because **“Proof-Carrying AI Pull Requests” risks overclaiming formal proof**. In academic and standards settings, “proof-carrying” can evoke proof-carrying code or formal verification. Jankurai’s actual claim is more pragmatic and defensible: evidence receipts, proof lanes, merge witnesses, and repair queues. I would use “evidence-carrying” unless the paper wants to claim formal proof semantics.

Best title option:

> **Jankurai: The Anti-Vibe Coding Standard**
> **Continuous Repository Alignment for Evidence-Carrying AI Pull Requests**
> **Detect drift. Prove the merge. Repair the repo.**

Alternative, slightly sharper:

> **Jankurai: The Anti-Vibe Coding Standard**
> **A Versioned Repository Conformance Standard for Evidence-Carrying AI Pull Requests**

The title should include **Jankurai** as text, not only as a logo. On the current title page, “Jankurai” is visually present in the wordmark, but the textual title is only “Anti-Vibe Coding Standard.” For citation, search, indexing, and serious paper identity, the title should begin with “Jankurai.”

# Biggest remaining tier-1 blockers

## 1. The PDF layout still looks unstable in several places

The content is improving faster than the typesetting. Several pages still have float-placement artifacts, awkward whitespace, tables appearing before their section heading, code blocks split across pages or columns, and very large tables that look machine-generated rather than designed.

The most serious layout issues are:

| Location    | Issue                                                                                                                                                                                    | Why it matters                                                         |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| Page 4      | The merge-witness code listing is split awkwardly across columns; one code block begins at the top-right while another starts at the bottom-left, and a stray `{` appears at the bottom. | This is the paper’s hero artifact. It must look clean and intentional. |
| Page 5      | The normative artifact checklist table is too tall and sparse; the rightmost requirement column is narrow and heavily wrapped.                                                           | This is a core standard table. It should be compact and authoritative. |
| Page 7      | Figure 2 occupies almost the whole page with huge whitespace.                                                                                                                            | It makes the PDF feel under-controlled.                                |
| Page 18     | The score-decomposition table has enormous row height and a cramped final column.                                                                                                        | This looks like a draft artifact, not a tier-1 table.                  |
| Page 21     | Appendix tables appear before the Appendix A heading.                                                                                                                                    | This is a serious float-ordering problem.                              |
| Pages 22-24 | Code/listing blocks continue across page boundaries without repeated titles or “continued” labels.                                                                                       | Readers lose context.                                                  |
| Whole PDF   | No page headers/footers with standard/schema/paper edition.                                                                                                                              | A versioned standard should carry version identity on every page.      |

## 2. The evaluation section still lacks actual observed results

The evaluation section says the seed suite exists and describes expected fixture outcomes. That is good, but a tier-1 paper needs an actual result table:

```text
Command: rtk just conformance
Fixtures: 10
Expected JSON files: 12
Passed: ?
Failed: ?
Skipped: ?
Auditor version: 0.7.0
Commit: ?
```

Right now the evaluation claim is mostly: “these fixtures exist and are expected to behave this way.” The next version should say: **“we ran them, here is the observed output, here is the commit, here are the artifact digests.”**

## 3. “Proof” terminology still risks overclaiming

The paper says “proof receipts,” “proof-carrying AI pull requests,” and “prove the merge.” These are memorable, but they need a precise definition early:

> In Jankurai, “proof” means repository-local evidence sufficient for the claimed conformance level. It does not mean formal proof of full program semantics.

The paper says it does not prove full semantics, but that disclaimer should appear next to the first “proof-carrying” use.

## 4. The normative core still needs a formal artifact schema section

The standard section includes examples and checklists, but a tier-1 standard paper should have a section called something like:

> **Core Artifact Model**

It should define the minimum required fields for:

* conformance claim,
* proof receipt,
* merge witness,
* evidence index,
* audit report,
* repair queue item,
* waiver,
* release gate receipt.

Right now those are distributed across the paper and appendices. The paper needs one place where readers can see the standard’s core object model.

## 5. The reference profile still takes too much main-body space

This version is much better because stack scoring appears later and is labeled non-normative. But pages 16-18 still devote significant body space to language history, standard gravity, stack scoring, score bands, and reference profile comparison. That material is interesting, but it is not the breakthrough.

The breakthrough is:

> **Continuous repository alignment turns vibe artifacts into merge evidence and repair queues.**

The language/profile material should either be shorter or moved to an appendix so the standard remains the main event.

# LaTeX/source-level recommendations

The current GitHub source is modular, but many TeX files appear minified into extremely long single lines in the raw view. `paper/jankurai.tex` is a single-line wrapper of `\input{...}` files, and `frontmatter.tex` packs many package imports, macros, metadata, titlepage logic, abstract text, and title construction into very long lines. This makes diffs, code review, and line-level comments much harder than necessary. ([GitHub][1])

## 1. Reformat all TeX files for reviewability

Use one semantic unit per line:

```latex
\documentclass[10pt,conference]{IEEEtran}

\usepackage[T1]{fontenc}
\usepackage{cite}
\usepackage{graphicx}
...
```

For prose, use one sentence per line. This makes Git diffs meaningful.

## 2. Fix float ordering

The paper currently suffers from float placement problems. Use:

```latex
\usepackage{float}
\usepackage{placeins}
\usepackage{dblfloatfix}
```

Then:

```latex
\FloatBarrier
\clearpage
\appendices
```

For appendix tables that must appear after headings, use non-floating tables:

```latex
\begin{center}
\captionof{table}{...}
\begin{tabularx}{...}
...
\end{tabularx}
\end{center}
```

or `[H]` floats.

## 3. Do not let hero code listings split across columns

The merge witness should be a `figure*` or non-floating full-width box. Do not use a raw `lstlisting` where TeX can split it across columns or pages.

## 4. Use breakable titled boxes for appendix templates

For waiver and repair templates, use `tcolorbox`:

```latex
\usepackage[most]{tcolorbox}

\newtcolorbox{specbox}[1]{
  breakable,
  title=#1,
  colback=black!2,
  colframe=black!35,
  fonttitle=\bfseries
}
```

This would make multi-page code blocks readable.

## 5. Add `microtype`

```latex
\usepackage{microtype}
```

This will improve justification and reduce awkward spacing.

## 6. Balance final columns

Use:

```latex
\usepackage{balance}
```

and place `\balance` before references if needed.

## 7. Add version footer

For a versioned standard, every page should carry version identity. Example:

```latex
\usepackage{fancyhdr}
\pagestyle{fancy}
\fancyhf{}
\fancyfoot[C]{Jankurai Standard 0.7.0 | Schema 1.5.0 | Paper 2026.05-ed7 | \thepage}
```

IEEE styles may need care, but the standards edition should definitely have this.

## 8. Add an explicit “paper mode” and “standard mode”

Use conditionals:

```latex
\newif\ifstandardedition
\standardeditiontrue
```

Then:

* standard edition: title page, full appendices, extensive command maps;
* paper edition: compact title, key tables, shorter appendices.

This project needs both.
