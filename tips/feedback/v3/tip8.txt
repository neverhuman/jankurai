I studied the paper, the repository README, and the core Rust code paths for the CLI, audit engine, caps, proof execution, context packs, repair plans, and repair application. The core idea is genuinely powerful: **Jankurai should not be positioned as “another AI coding tool.” It should be positioned as a merge-governance standard for the agent era.** The world-shocking version is: **AI made code cheap; Jankurai makes merge trust scarce, visible, auditable, and ratchetable.** The repo already describes itself this way: a local Rust “agent-native repository control-plane and audit CLI” that turns ownership maps, proof lanes, generated zones, security boundaries, and repair queues into files both humans and agents can read. ([GitHub][1])

## Overall verdict

This project has a real shot at becoming a standard, but the paper needs to make one strategic shift:

**The standard audit and rolling score must become the protagonist.**

Right now, the paper has three big ideas competing for the spotlight: vibe coding as a fault class, a Jankurai conformance standard, and a preferred stack ranking. All three are valuable, but for a tier-1-level story, the strongest and most defensible claim is this:

> Modern repositories need a machine-readable merge-control plane. Jankurai defines that control plane, audits it continuously, gives every repo a rolling trust score, and routes every failure into an agent-repair queue.

That is bigger than Rust versus Go, bigger than prompt engineering, and bigger than “AI coding.” That is a new discipline: **agent-native software assurance**. The paper already says the scarce act is no longer typing plausible code but “trustworthy merge,” and that repositories must reject wrong generated code, localize defects, prove repairs, preserve security posture, and teach the next agent what happened. That should become the paper’s spine.

# 1. Paper feedback, section by section

## Title and abstract

**What works:** “Humans Were the Bug” is unforgettable. It has manifesto energy. The abstract’s “trustworthy merge” framing is excellent, and the phrase “repositories designed as verification interfaces for generated code” is the cleanest academic contribution in the paper. 

**What to improve:** For a tier-1 venue, the title may read as hostile or rhetorical before reviewers reach the precise claim. Keep it as the public-facing title, but use a more research-friendly subtitle or alternate title in formal settings:

> **Agent-Native Engineering: Repository Verification Interfaces for AI-Generated Code**
> Public title: **Humans Were the Bug**

The abstract should explicitly list contributions:

1. An operational definition of vibe coding as an auditable fault class.
2. A versioned repository conformance model, HL0–HL5.
3. A machine-readable audit artifact with findings, caps, proof lanes, repair queues, and receipts.
4. A rolling-score governance loop for catching drift.
5. A reference implementation in the Jankurai CLI.

Right now, the abstract includes all of these pieces, but the contribution list should be impossible to miss.

## I. New Bottleneck: Trustworthy Merge

**This is the best opening idea.** The paper should start with a concrete failure vignette:

> An agent edits a checkout flow. The code compiles. Unit tests pass. The PR looks clean. But it hand-edits a generated client, weakens an authorization invariant, and never runs the browser path. Old review calls this “almost right.” Jankurai calls it three routed findings, one blocked proof lane, and a repair queue.

That would make “trustworthy merge” visceral. Then introduce the central thesis:

> The repo must become the reviewer of first resort.

The current four claims are good, but claim four—the default stack—should be softened or delayed. The core paper should win even if a reader prefers Go, .NET, JVM, or Rails. Make the standard universal; make the stack a **reference profile**.

## II. Definitions and Threat Model

This section is strong because it turns fuzzy terms into operational objects: vibe coding, proof lane, generated zone, repair receipt, and centerline drift. The threat model table is also a major asset because it names concrete failure modes: false plausibility, hostile context, tool overreach, secret exposure, dependency poisoning, and false-green tests. 

The biggest upgrade is to formalize **centerline drift**. Suggested definition:

> `centerline_drift = claimed_conformance_score - observed_audit_score`, capped by hard-rule violations.

Then show examples:

| Claim        | Observed behavior                | Drift            |
| ------------ | -------------------------------- | ---------------- |
| HL3 Standard | No generated-zone manifest       | High             |
| HL4 Ratchet  | Score dropped from 91 to 83      | Regression       |
| HL5 Release  | Missing DB/e2e/security evidence | Release-blocking |

This is where the rolling score should first appear.

## III. Languages as Bottleneck Compression

The idea is excellent: languages are not syntax preferences; they compress burdens away from human memory. That fits the paper beautifully.

The issue is length. The language-history section risks distracting from the standard. I would compress it by 30–40% and make the agent-era point sharper:

> In the agent era, syntax compression matters less than **rejection compression**: how quickly a stack can reject invalid states, invalid boundaries, invalid contracts, invalid permissions, and invalid migrations.

That phrase—**rejection compression**—could become a signature Jankurai concept.

## IV. Technical Promise Versus Standard Gravity

This section is valuable, especially the Julia discussion. It protects the paper from being read as “new languages are bad.” The core distinction is strong: technical elegance is not the same thing as default-standard status. 

To improve it, make “standard gravity” more measurable. A stack has standard gravity when it has:

* deterministic local proof lanes,
* mature CI/security/dependency tooling,
* generated contract workflows,
* deploy/observe/rollback conventions,
* enough public corpus for agents to route repairs reliably.

That turns the section from taste into an audit rubric.

## V. Vibe-Coding Fault Taxonomy

This is one of the strongest sections in the paper. The table that ranks vibe-coding faults by risk-priority number is exactly the kind of move that turns a meme into a discipline. The page 4 chart grouping risk into top-level buckets—security, business truth, contracts/data, verification, entropy, context/setup, and repair—is especially important because it says: “vibe mistakes are not random; they cluster.” 

The upgrade: every RPN number should be labeled visually as a **policy prior**, not an empirical frequency. The text says this, but tables and charts have more authority than disclaimers. Add a subtitle under the table:

> Policy-priority weights for Jankurai 0.4.0. Calibration against escaped defects and repair outcomes is future work.

Then add a second table: **“How a rolling score catches each fault.”**

Example:

| Vibe fault            | Old review symptom | Jankurai signal                             |
| --------------------- | ------------------ | ------------------------------------------- |
| Handwritten DTO drift | “Looks fine”       | HLT-007 + contract lane missing             |
| False-green tests     | “CI passed”        | HLT-008 + weak assertion detector           |
| Generated mutation    | “Tiny client fix”  | HLT-002 + generated-zone block              |
| UI layout break       | “No compile error” | HLT-013 + missing screenshot/geometry proof |
| Secret exposure       | “Fixture only”     | HLT-010 + secret scan evidence required     |

This would make the audit/score story obvious.

## VI. Jankurai Standard and Conformance

This should move earlier. It is the heart of the paper.

The progressive install ladder and HL0–HL5 levels are excellent. The paper’s conformance levels—Unscored, Advisory, Guarded, Standard, Ratchet, Release Contract—are exactly how a standard becomes adoptable rather than dogmatic. 

The improvement is to define conformance as a **public protocol**:

> A repository claiming Jankurai conformance must publish a standard version, policy, owner map, proof map, generated-zone manifest, audit artifact, score, caps, findings, and repair queue.

Then add pseudocode:

```text
raw_score = weighted_sum(dimensions)
caps = hard_rule_caps(repo)
final_score = min(raw_score, caps)
merge_decision = pass iff final_score >= floor and no blocking findings
ratchet_decision = pass iff final_score >= previous_accepted_score - allowed_drop
```

That one block would make the standard easier to implement independently.

## VII. A 100-Point Stack Rubric

The rubric is smart and well-weighted. I agree with the premise that first-draft velocity should be weighted low because agents already made plausible drafts cheap. Correctness, security, contracts, and repairability should dominate. 

The danger: readers may argue with the stack scores and miss the deeper standard. To avoid that, rename this section:

> **Reference Profile Rubric for Durable Product Systems**

Then state:

> The Jankurai standard is stack-independent. This rubric defines the default reference profile used by the current Jankurai policy.

That makes the Rust/TypeScript/PostgreSQL recommendation less vulnerable to “but my stack…” objections.

## VIII. Stack Ranking, Sensitivity, and Exceptions

The stack ranking is exciting, but it should be framed as a falsifiable recommendation, not a final law. The page 8 bar chart showing Rust at 94, Go at 90, .NET at 89, TypeScript/Rust/Go cells at 88, and JVM at 87 is persuasive visually, but because the axis is cropped to scores above 85, it may look more dramatic than the underlying spread. 

Add a sensitivity table that says:

> If ecosystem weight rises, Go/.NET/JVM improve. If memory safety/type-state rises, Rust separates. If product velocity rises, full-stack TypeScript improves but hits durable-truth caps.

You already say this in prose. Put it in a table.

The Kafka subsection is very good. The phrase “Kafka is a brownfield exception, not stack identity” is strong. Keep it.

## IX. Default Winner Architecture

The filesystem-as-routing-table idea is excellent. The page 8 ownership spine and agent-control spine diagrams are among the paper’s clearest visuals. 

The upgrade is to add a “bad patch / good patch” example.

Bad patch:

```text
apps/web/src/api/client.ts       handwritten DTO change
apps/web/src/checkout.tsx        business-rule workaround
```

Good patch:

```text
contracts/openapi/api.yaml       contract source
apps/web/src/generated/client.ts regenerated
crates/domain/src/payment.rs     invariant
crates/application/tests/...     workflow proof
```

This would show how the architecture prevents vibe coding.

## X. Agent Repository Controls and Tool Adapters

This is timely and important. The repository already includes agent-facing files and adapters such as AGENTS.md, CLAUDE.md, Cursor rules, and Gemini-related guidance in the tree, and the README frames Jankurai as a way to make AI agents follow the same local rules. ([GitHub][1])

The paper should add one more invariant:

> All agent instructions must be generated from or checked against the canonical standard manifest.

Call this **adapter drift**. If AGENTS.md says one thing and Cursor rules say another, the repo has already lost alignment.

Suggested new rule:

> `HLT-022-ADAPTER-DRIFT`: provider-specific agent instructions differ from the canonical standard version, owner map, generated-zone policy, or proof commands.

## XI. Continuous Proof, CI, and Test Explosion

This section is right: tests are useful only if routed. The code reflects this idea: `jankurai prove` builds a proof plan from changed paths, owner routes, test routes, required lanes, expected artifacts, residual risks, and human approval requirements. It also refuses to run commands that are not listed in proof lanes or the test map unless an explicit unsafe override is provided. 

The paper should emphasize a slogan:

> **No proof, no merge. Wrong proof, no merge. Unmapped proof, no merge.**

Also add a figure showing:

```text
diff -> owner map -> test map -> proof lane -> receipt -> score delta -> merge decision
```

That is the Jankurai loop in one line.

## XII. Rendered UX and Browser-Step QA

This is a hidden gem. Many AI-code failures are not compile failures; they are browser-runtime failures. The paper’s layered UX proof model—design-system constraints, Storybook states, Playwright screenshots, DOM geometry, accessibility, visual regression, AI/CV triage, and human labels—is excellent. 

The code already has UX readiness analysis looking for Storybook, Playwright visual proof, visual review, accessibility tooling, layout stability, API mocks, design tokens, geometry runtime, and artifact-backed proof. 

This should be elevated as one of Jankurai’s “world-shock” differentiators:

> Jankurai treats the browser as a proof lane, not a screenshot afterthought.

The next paper version should include three screenshots or diagrams of UI defects Jankurai would catch: overlap, clipped text, missing focus/ARIA, and responsive breakage.

## XIII. Security, Supply Chain, and Permissions

This is solid. The code already includes caps and detectors for missing security lanes, missing secret/dependency scanning, prompt-injection risk, overbroad agency, secret-like content, CI hardening gaps, and supply-chain drift.  

The paper should add a clear permission model:

| Profile          | Allowed                           | Blocked                              |
| ---------------- | --------------------------------- | ------------------------------------ |
| read-only        | inspect files, build context pack | writes, network, shell mutation      |
| scoped edit      | allowed paths only                | generated zones, secrets, migrations |
| proof execution  | signed proof commands only        | arbitrary shell                      |
| release/security | named human approval              | broad credentials, destructive ops   |

This would make “bounded agents” concrete.

## XIV. Exceptions, Observability, and Repair Receipts

This is one of the most important sections. The repair receipt is not just a log; it is how the repo teaches the next agent.

The code supports this direction strongly. The report model includes findings with stable rule IDs, evidence, rerun commands, fingerprints, owners, lanes, and repair fields. Proof receipts include command, exit code, artifacts, changed paths, owner, skipped reason, residual risk, log paths, digests, rules covered, and retryability. 

The paper should make this more profound:

> A repair receipt is the memory unit of agent-native engineering.

Then define the “repair learning loop”:

```text
failure -> routed finding -> repair packet -> proof receipt -> score recovery -> future context
```

That is how Jankurai becomes more than a linter.

## XV. Migration, Versioning, and Governance

This section is practical and strong. The paper correctly avoids rewrite fantasy and says teams should first add advisory audit, root instructions, owner maps, test maps, generated zones, version manifests, and proof lanes. 

The repo README matches this adoption posture: it starts with read-only `adopt` and `audit` commands that write named reports under `target/jankurai`, then moves to agents, full scaffold, and ratchet only after review. ([GitHub][1])

One issue: the paper, README, and CLI should be perfectly aligned on adoption levels. The paper discusses `agents → score → ci → full → ratchet`; the README presents Observe, Agents, Full, Ratchet; the CLI `init` levels are `agents`, `score`, `ci`, and `full`, while ratchet is a CI install mode. ([GitHub][1]) 

For a standard, these names must be exact everywhere.

## XVI. Limitations and Research Agenda

Good section, but it should become more empirical. Add a table:

| Claim                               | Metric                               | Study design                  |
| ----------------------------------- | ------------------------------------ | ----------------------------- |
| Jankurai reduces unsafe merges      | escaped defects / high-risk findings | compare before/after adoption |
| Jankurai speeds repair              | time from failing CI to green proof  | PR telemetry                  |
| Jankurai reduces agent over-editing | changed files per task               | benchmark tasks               |
| Jankurai catches vibe mistakes      | seeded fault detection rate          | synthetic fault suite         |
| Jankurai controls false positives   | findings dismissed / total findings  | review labels                 |

That would make the project feel like a research program, not just a manifesto.

## XVII. Conclusion

The conclusion is strong. The line “Anything less is just faster vibe coding” is memorable. Keep it.

I would add one final sentence:

> The future standard is not the model that writes the most code; it is the repository that knows which code is safe to merge.

That lands the thesis.

## Appendices

Appendix A is essential, but it needs to match the code. The paper lists HLT-001 through HLT-019, while the code already emits `HLT-020-CI-HARDENING-GAP` and `HLT-021-DESTRUCTIVE-MIGRATION`.  

Appendix B’s artifact manifest is great. Appendix C’s exception and repair templates are highly agent-friendly. Appendix D’s file tree diagrams are useful. Appendix E’s command examples are valuable, but the body should show only the canonical loop; the long command catalog belongs in docs and appendices.

# 2. The story upgrade: make the audit and rolling score unforgettable

The paper should repeatedly return to one central object:

> **The Jankurai rolling score is the repo’s trust heartbeat.**

A normal test suite says “green.” Jankurai says:

* green for which owner,
* green for which proof lane,
* green against which generated-zone policy,
* green with which security evidence,
* green with which repair receipts,
* green compared to which baseline,
* green with what residual risk.

That is the cure to vibe coding.

## Add a “rolling score catches vibe mistakes” narrative

Suggested new section title:

> **Rolling Trust: Why Green CI Is Not Enough**

Core argument:

1. Vibe mistakes are often not single catastrophic failures. They are small plausibility leaks.
2. A hand-edited generated client drops contract integrity.
3. A skipped test drops verification trust.
4. A broad agent permission drops security posture.
5. Missing screenshots drop rendered UX confidence.
6. Each individual change may look “reasonable.”
7. The rolling score makes the drift visible before it becomes an incident.

Suggested visual:

Commit      Score   New cap/finding                         Decision

The audit should answer:

That is the world-standard framing.

# 3. Code feedback

## What is already impressive

The implementation is much more than a paper prototype.

The CLI surface already maps closely to the paper: audit, adopt, init, update, doctor, context-pack, repair-plan, lane/proof/prove/proof-verify, registry, cell, migrate, bench, certify, govern, publish, repair, optimize, Rust analysis commands, exceptions, adapters, agent verification, CI install, hooks, issues export, UX passthrough, and security run. 

The audit engine already produces the right shape of artifact: fingerprints, policy, manifest fingerprints, dirty-worktree state, standard/auditor/schema/paper versions, target stack, score, raw score, decision, proof receipts, caps, dimensions, UX/security/boundary readiness, findings, and an `agent_fix_queue`.  

The hard caps are conceptually excellent. A repo cannot hide behind a high weighted score if it lacks a fast lane, security lane, generated contract discipline, secret/dependency scanning, rendered UX proof, or safe agent permissions. 

The proof system is one of the strongest parts of the code. It builds proof plans from changed paths, owner maps, test maps, proof lanes, route decisions, residual risk, and human approval requirements; it writes receipts and evidence indexes with digests; and it verifies proof evidence against plan and manifest fingerprints. 

The repair workflow is correctly cautious. Repair plans convert findings into packets with permission profiles, allowed paths, forbidden paths, expected patch shape, required proof, stop conditions, risk level, and human-review requirements. The fixture apply path blocks path traversal, generated zones, forbidden areas, high-risk edits, and human-required repairs.  

## Code gaps that matter for “global standard” credibility

The biggest gap is that many detectors are currently lexical or heuristic. That is fine for 0.4.0, but the paper should be transparent about it, and the code should move toward semantic detectors. For example, TODO/fallback/legacy markers, prompt-injection strings, false-green tests, streaming clients, and secret-like values are useful signals, but they need confidence, false-positive labeling, and eventually AST/dataflow support. 

The second gap is **alignment drift between paper, README, and code**. The paper’s rule appendix stops at HLT-019, but the code already has HLT-020 and HLT-021. The adoption ladder also needs one canonical naming scheme across paper, README, CLI help, docs, and examples.   ([GitHub][1])

The third gap is token minimization. `context_pack` is directionally right, but it currently exposes a hardcoded token budget and heuristic routing. For the “cure to token minimization” claim, context packs should actually compute token cost, rank files, and produce a bounded “read this, not that” bundle. 

The fourth gap is that the current `test-map` schema is relatively thin: path prefix to command and purpose. The proof-lane schema is richer, with fields such as command ID, kind, cost, rules covered, required artifacts, timeout, network requirement, and destructive flag. The standard would be stronger if test routes inherited or referenced rich proof-lane metadata rather than only commands.

# 4. The profound upgrade: Jankurai Merge Witness + Trust Ledger

The next-level feature should be:

> **Jankurai Merge Witness: a cryptographic, machine-readable proof object for every PR.**
> **Jankurai Trust Ledger: an append-only rolling score history that shows whether repo trust is improving or decaying.**

This would turn Jankurai from an audit CLI into a standard for agent-native merge governance.

## What the Merge Witness does

A merge witness answers:

```text
This diff touched these paths.
These owners were responsible.
These generated zones were protected.
These proof lanes were required.
These commands were allowed.
These receipts were produced.
These artifacts were hashed.
These findings remain.
This score changed from X to Y.
This merge is pass/block/ratchet-fail/release-fail.
```

Proposed command:

```bash
jankurai witness . \
  --changed-from origin/main \
  --out target/jankurai/merge-witness.json \
  --md target/jankurai/merge-witness.md \
  --sarif target/jankurai/jankurai.sarif
```

Then:

```bash
jankurai ledger append . \
  --witness target/jankurai/merge-witness.json \
  --history agent/trust-ledger.jsonl
```

And CI:

```bash
jankurai ratchet . \
  --ledger agent/trust-ledger.jsonl \
  --witness target/jankurai/merge-witness.json \
  --max-drop 0 \
  --fail-on high,critical
```

## Why this is profound

Today, AI coding agents can produce plausible code fast. The missing artifact is a **merge witness** that proves the repo did not accept plausibility alone.

This would give Jankurai a standard artifact that other tools could adopt. GitHub Actions, Cursor, Claude Code, Codex, Copilot, local terminal agents, and future IDEs could all point to the same witness.

## Suggested witness schema

```json
{
  "schema_version": "1.0.0",
  "standard_version": "0.4.0",
  "auditor_version": "0.4.0",
  "repo": "...",
  "git": {
    "base": "origin/main",
    "head": "abc123",
    "dirty_worktree": false
  },
  "changed_paths": [],
  "route_decisions": [],
  "generated_zone_decisions": [],
  "required_lanes": [],
  "proof_plan_digest": "sha256:...",
  "proof_receipts": [],
  "artifact_digests": [],
  "score": {
    "before": 91,
    "after": 88,
    "delta": -3,
    "raw_after": 93,
    "caps_applied": []
  },
  "findings_delta": {
    "new": [],
    "resolved": [],
    "carried": []
  },
  "exceptions": [],
  "residual_risk": [],
  "merge_decision": "pass|block|ratchet_fail|release_fail"
}
```

The code already has many ingredients: report fingerprints, input fingerprints, manifest fingerprints, proof receipts, proof evidence indexes, command digests, log digests, artifact digests, and score history options. The upgrade is to bind them into one canonical PR-level artifact.

# 5. Concrete code upgrades to implement next

## A. Add `jankurai witness`

Build it as a wrapper over existing audit and proof machinery:

1. Run changed-path discovery.
2. Build proof plan.
3. Run or load proof receipts.
4. Run audit.
5. Compare against baseline or previous ledger row.
6. Emit merge witness JSON/Markdown.
7. Optionally emit SARIF annotations.

This reuses existing architecture instead of creating a parallel system.

## B. Add score delta explanations

A rolling score is only useful if humans can understand why it moved.

Add:

```bash
jankurai explain-score . \
  --before agent/trust-ledger.jsonl \
  --after target/jankurai/merge-witness.json \
  --md target/jankurai/score-delta.md
```

Output:

```text
Score dropped 91 -> 84.

Main causes:
- HLT-013: web change lacks rendered UX proof (-4)
- HLT-004: apps/web/src/checkout.tsx has no proof route (-3)
- HLT-016: new dependency lacks rationale (-2)

Fastest repair:
1. Add test-map route for apps/web.
2. Run Playwright screenshot lane.
3. Add dependency rationale and SCA receipt.
```

That is how Jankurai helps coders go faster.

## C. Upgrade context packs into real token minimization

Current context packs are useful, but the next version should be budgeted and ranked.

Add:

```bash
jankurai context-pack . \
  --task "fix checkout validation" \
  --changed apps/web/src/checkout.tsx \
  --max-tokens 16000 \
  --rank graph \
  --out target/jankurai/context-pack.json
```

The output should include:

* exact files to read,
* exact files not to read,
* estimated tokens per file,
* why each file was included,
* stop conditions,
* required proof lane,
* generated-zone warnings,
* command-output filters.

This would make “token minimization” real repository infrastructure rather than a prompting trick.

## D. Build a Boundary Graph

Add:

```bash
jankurai graph build . \
  --out target/jankurai/boundary-graph.json
```

Then use AST-level detectors:

* Rust: `syn` for module/function/import/error-boundary mapping.
* TypeScript: tree-sitter or TypeScript compiler API for imports, API clients, generated-client usage, component boundaries.
* SQL: migration parser or structured SQL scan.
* Python: AST for production DB access, authz/product-truth ownership, unsafe subprocess/network use.

The audit already has a Rust dependency stack that can support deeper analysis, including `syn`, `quote`, `cargo_metadata`, `regex`, `globset`, and `walkdir`. 

## E. Low-risk auto-repair synthesis

The current repair system is safely conservative. Keep that. But add rule-specific auto-repairs for low-risk cases:

Safe candidates:

* create missing `agent/owner-map.json` entry,
* create missing `agent/test-map.json` entry from known proof lane,
* add missing generated-zone metadata when source/command can be inferred,
* scaffold missing docs/repair receipt template,
* add missing `just score` or `just fast` recipe,
* create CI observe-mode Jankurai job,
* add missing standard-version manifest field.

Never auto-repair:

* secrets,
* authz,
* destructive migrations,
* generated contract semantics,
* production DB behavior,
* broad permissions,
* security policy weakening.

This gives speed without creating a new vibe-coding machine.

## F. Public scorecard and certification badge

The README already has badges and says the project is early but usable as a local Rust CLI and standard workspace. ([GitHub][1]) The next step is a public scorecard:

```bash
jankurai publish . \
  --witness target/jankurai/merge-witness.json \
  --ledger agent/trust-ledger.jsonl \
  --badge-svg target/jankurai/jankurai-score.svg
```

Badge fields:

```text
Jankurai HL3
Score 91
No hard findings
Proof coverage 97%
Last audit 2026-05-04
Standard 0.4.0
```

That is how it becomes a global standard: other repos can display the same conformance artifact.

# 6. Highest-priority changes before the next public push

1. **Make the standard/audit/rolling score the main story.** The stack ranking should become a reference profile, not the headline.

2. **Add the Merge Witness concept to the paper.** This is the missing artifact that makes “trustworthy merge” concrete.

3. **Unify paper, README, CLI, and rule IDs.** Add HLT-020 and HLT-021 to the paper appendix, and standardize adoption levels across all docs and CLI help.

4. **Add a score-over-time figure.** The page 4 risk chart and page 8 stack chart are useful, but the paper needs one visual that shows Jankurai catching trust decay over commits.

5. **Add a seeded vibe-mistake demo.** Create a small example repo or fixture with intentional faults: generated mutation, false-green test, missing owner route, secret-like fixture, UI proof gap, wrong-layer SQL. Then show Jankurai catching and repairing them.

6. **Move toward semantic detectors.** Keep lexical detectors, but label them as heuristic and start building the AST/boundary graph.

7. **Make context packs truly token-bounded.** This directly supports the “token minimization” ambition.

8. **Turn repair receipts into learning memory.** Every repair should become future agent guidance: what failed, why, how fixed, what proof accepted it.

## The sharper thesis

The paper’s final form should make this claim:

> **Jankurai is a repository trust protocol for the agent era. It replaces vibe coding with audited ownership, routed proof, generated-boundary discipline, repair receipts, and a rolling conformance score.**
