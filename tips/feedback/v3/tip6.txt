## High-level verdict

This is already much bigger than a “coding tool.” The strongest version of Jankurai is a **conformance standard for trustworthy merge**: every generated change gets routed to an owner, checked against generated boundaries, assigned the smallest proof lane, scored, repaired, receipted, and ratcheted so the repo gets harder to corrupt over time. The paper’s thesis is excellent: AI has made plausible code cheap, so the scarce act is no longer writing code but proving that a change is safe to merge. That framing is the real lightning strike. 

The project already has the bones of that standard in code: the README positions Jankurai as an “agent-native repository control-plane and audit CLI,” with adoption modes, owner maps, generated-zone policy, proof lanes, repair planning, migration analysis, security evidence, UX QA, and publication evidence. ([GitHub][1]) The CLI surface is impressively ambitious, with commands for audit, adopt, init, doctor, context packs, repair plans, lanes, proof execution, proof verification, registry, cells, migration, benchmark, certification, governance, publishing, exceptions, adapters, hooks, issues, explanations, versions, UX, and security. ([GitHub][2])

My bluntest strategic feedback: **make the rolling audit score the product.** The stack recommendation is useful. The paper is provocative. The rules are valuable. But the thing that can become a global standard is the recurring public artifact: **“This repository changed; here is the Jankurai score delta; here are the failed HLT rules; here is the proof ledger; here are the repair receipts; here is the ratchet decision.”** That is the cure to vibe coding.

---

## The world-shocking story to sharpen

The current paper says, correctly, that “vibe coding” is accepting plausible code without ownership, proof routing, generated-boundary checks, security gates, and repair evidence. The next version should make this even more visceral:

**The old unit of engineering trust was the senior reviewer.
The new unit of engineering trust is the rolling proof ledger.**

That phrase, or something like it, should become the backbone. A repo should not merely “use AI well.” It should be able to prove, on every PR, that the invalid path was rejected. The rolling score matters because vibe mistakes are not one-off mistakes. They are **drift**: hidden contract drift, proof drift, UI drift, security drift, generated-zone drift, dependency drift, exception drift, and context drift. A static audit catches a snapshot. A rolling score catches the slope.

The strongest Jankurai claim is not “Rust plus TypeScript is best.” That will start arguments. The strongest claim is:

> **Any AI-assisted repository that cannot produce a change-scoped proof ledger is still vibe coding.**

That is tier-1. That is standard-worthy. That is how Jankurai can become a shared alignment layer across agents, humans, CI systems, IDEs, security teams, and production repair loops.

## Paper feedback, section by section

### Abstract

The abstract is strong, but it currently has three big ideas competing for attention: agent-native engineering, the Jankurai conformance standard, and the default stack. I would make the **audit/rolling-score artifact** the centerpiece. The abstract should tell the reader that Jankurai converts AI-generated code from a plausibility game into a measured merge-control process.

Suggested elevation:

> AI coding makes plausible code abundant and trustworthy merge scarce. Jankurai treats every generated change as an untrusted hypothesis until the repository can route ownership, select proof lanes, protect generated boundaries, verify security posture, emit a rolling score delta, and store repair receipts. The result is not a prompt style, but a repository conformance standard: a machine-readable interface for rejecting vibe-coded changes before they become production drift.

Then mention the reference stack later. The current stack sentence is useful, but in the abstract it risks making the paper sound like a stack manifesto. The true breakthrough is the **standard audit loop**. The stack should be framed as the reference implementation of the standard, not the standard itself. The project README already supports this broader framing by presenting Jankurai as a repo control-plane and audit CLI rather than only as a stack template. ([GitHub][1])

### I. New Bottleneck: Trustworthy Merge

This is one of the best sections. “Humans were the bug” is polarizing, memorable, and useful as a provocation. The section correctly narrows the claim: humans were not the bug because they wrote software; the bug was that repositories depended on human memory, patience, taste, and slow review. 

To make it tier-1, add a named metric here: **Trustworthy Merge Latency**, or TML. Define it as the time between “agent proposes code” and “repo has enough evidence to merge or reject.” This metric would let Jankurai say: “We do not merely increase coding velocity. We reduce trustworthy merge latency.”

I would also add one concrete failure vignette at the opening. For example:

> An agent edits a generated client instead of the OpenAPI source. The app compiles. The UI renders. The reviewer sees plausible TypeScript. Two releases later, regeneration deletes the fix. This is not an AI hallucination; it is a repository without generated-zone enforcement.

That example makes the reader feel why the old review model breaks.

### II. Definitions and Threat Model

The definitions are excellent. “Vibe coding,” “proof lane,” “generated zone,” “repair receipt,” and “centerline drift” are exactly the kind of vocabulary a standard needs. Table I’s threat model is also strong because it maps each threat to a Jankurai control rather than leaving the threat as rhetoric. 

I would add a few score-specific definitions here:

**Baseline score:** the accepted score at adoption time.

**Score delta:** the change in score for a PR, branch, release, or time window.

**Proof coverage ratio:** percentage of changed paths with mapped proof lanes that actually ran on the current diff.

**Receipt freshness:** whether proof evidence was produced after the relevant file hashes changed.

**Exception half-life:** how quickly exceptions expire, shrink, or get repaired.

**Agent action budget:** the maximum scope of writes, network calls, dependency changes, and destructive commands allowed for a lane.

These definitions turn the paper from “good audit idea” into “measurable governance system.”

I would also explicitly distinguish **proof** from **test pass**. A test pass is one kind of evidence. A proof receipt is a structured claim that the correct lane ran against the correct change with the correct artifacts and versions.

### III. Languages as Bottleneck Compression

This section is thoughtful and gives the paper intellectual depth. The “language as bottleneck-compression system” frame is much better than a normal language-war argument. The table connecting language families to human bottlenecks and agent-era interpretations is useful. 

The enhancement: add one more column or paragraph asking, **“Where does the language reject bad generated code?”** For Rust, many invalid states get rejected before runtime. For TypeScript, frontend shape errors can be rejected before browser interaction, but not durable truth. For Python, exploration is fast, but ownership and runtime errors need tighter boundaries. That would directly connect the language-history section back to Jankurai’s audit thesis.

I would also make this section shorter. It is good, but the paper’s unique contribution is not language history. It is the standard audit loop. Keep the language story as support, then move quickly back to proof.

### IV. Technical Promise Versus Standard Gravity

This is persuasive, especially the Julia example. The idea that technical elegance and standard gravity are different things is important. 

The risk is tone. Communities around Julia, Elm, Haskell, Scala, Elixir, and other ecosystems may read this as dismissal. To maximize adoption, make the section sound less like “these lost” and more like “default standards require operational gravity.” The phrase “the shelf is not a graveyard of bad ideas” is excellent and should stay.

I would add an **exception acceptance test**:

> A non-default stack can claim Jankurai compatibility when it provides equivalent ownership maps, proof lanes, generated-boundary enforcement, security evidence, repair receipts, and score ratcheting.

That makes the standard inclusive while preserving rigor.

### V. Vibe-Coding Fault Taxonomy

This is the heart of the paper. Table IV and the Top-Level Risk bucket chart are the closest thing to a publishable Jankurai “artifact.” The policy-priority model is exactly how you convert an insult into a governance surface. 

This section should be elevated even further. I would rename it from “Fault Taxonomy” to something more standard-like:

**The Vibe Fault Model**
or
**The Jankurai Risk Model for Generated Code**

Then introduce the central insight:

> Vibe mistakes are not mysterious. They cluster into repeatable fault families that can be routed, scored, repaired, and tracked.

The current RPN explanation is good, but it needs a calibration roadmap. Add a paragraph that says Jankurai’s initial RPN values are policy priors, and future releases will calibrate them against real repositories using escaped defects, time-to-repair, review delay, false-positive rate, proof coverage, and security regressions.

I would also add a **rolling risk ledger** concept: every HLT finding should be counted not only by current score, but also by recurrence. A repo that repeatedly reintroduces `HLT-007-HANDWRITTEN-CONTRACT` has a cultural/process defect, not just a local code defect.

### VI. Jankurai Standard and Conformance

This is the most important section and should probably move earlier, immediately after the threat model. The paper should establish the standard before the stack discussion. The current conformance ladder and adoption levels are very good: agents, score, CI, full, ratchet; HL0 through HL5; stable HLT rule IDs; repair queues; audit JSON; audit Markdown. 

The repo’s standard document already mirrors this nicely: it describes a mission around rejecting wrong code quickly, making ownership and commands machine-obvious, and treating vibe coding as a defect class; it also lists the HL0–HL5 conformance levels and HLT rule families. ([GitHub][3])

To make this section world-class, add a diagram called **The Jankurai Merge Loop**:

`diff → changed paths → owner map → generated-zone check → proof lane selection → security/UX/contract gates → findings → repair queue → receipts → score delta → ratchet decision`

This diagram should become the paper’s iconic figure. The current stack-ranking chart is visually clear, but the merge loop is the real revolution.

Also add badge semantics:

**Jankurai HL3 / Score 87 / Delta +2 / Proof Coverage 96% / Exceptions 3 expiring**

That kind of badge could be shown in README files, PR comments, GitHub Checks, enterprise dashboards, and release notes.

### VII. 100-Point Stack Rubric

The rubric is smart, especially the weighting that puts correctness, security, contracts, and repairability above first-draft velocity. That is philosophically aligned with the whole project. 

The risk is that readers may experience the scores as subjective. That is not fatal, but it needs transparency. I would publish the rubric as machine-readable policy:

`agent/stack-rubric.toml`
`schemas/jankurai-stack-rubric.schema.json`
`docs/rubric-calibration.md`

Then the paper can say: “These are policy weights, not laws of nature. They are versioned, sensitivity-tested, and open to calibration.” That makes disagreement productive.

I would also reduce the apparent precision of some stack scores. A score of 94 versus 90 looks crisp; the uncertainty bands later help, but many readers will remember only the ranking. Consider presenting the main result as bands first: “Rust reference stack: top band; Go/.NET/JVM: strong compatible bands; Python/FastAPI and full-stack TypeScript: require additional controls for durable truth.”

### VIII. Stack Ranking, Sensitivity, and Exceptions

This section is useful but strategically dangerous. It may attract debate away from the standard. A Go team, .NET team, JVM team, or Rails team might reject Jankurai because they think the paper is trying to replace their stack. That would be a shame, because the standard itself is much bigger than the reference stack.

My recommendation: rename “Default Winner Architecture” to **Reference Architecture** and consistently say:

> Jankurai is stack-compatible but reference-stack-opinionated.

The default stack can remain Rust core, TypeScript/React/Vite, PostgreSQL, generated contracts, and bounded Python. But the global standard should say: “Bring your stack. Prove your boundaries.”

The Kafka/Rust streaming exception is interesting, but it is probably too detailed for the main path. It may belong in an appendix, companion post, or “reference architecture exceptions” document. The broader rule is more important: brokers belong behind adapters and generated event contracts.

### IX. Default Winner Architecture

“The filesystem should be a routing table” is one of the paper’s strongest lines. It should be repeated, bolded, and turned into a slogan. 

The ownership-cell table is excellent. To strengthen it, add a “bad patch examples” table:

**Bad:** `apps/web` imports SQL client.
**Finding:** `HLT-006-DIRECT-DB-WRONG-LAYER`.
**Repair:** move query into adapter, expose generated contract, run DB lane.

**Bad:** agent edits `apps/web/src/generated/client.ts`.
**Finding:** `HLT-002-GENERATED-MUTATION`.
**Repair:** edit OpenAPI source and regenerate.

**Bad:** Python AI service writes billing status.
**Finding:** `HLT-005-PYTHON-PRODUCT-TRUTH`.
**Repair:** move durable truth to Rust/PostgreSQL.

This makes the architecture feel enforceable rather than aspirational.

### X. Agent Repository Controls and Tool Adapters

This section is very strong because it identifies a real emerging problem: agent tools each have their own instruction surfaces, but the repo needs one canonical policy. The codebase already reflects this ambition with adapter-oriented structures such as `.agents`, `.cursor/rules`, `.claude/skills/jankurai`, `agent/`, and related control-plane files. ([GitHub][1])

The phrase “token minimization is not a prompting trick; it is repository design” should become one of the project’s flagship claims. 

To enhance it, add a concrete **token budget contract**:

Root router: target under 800 tokens.
Lane context pack: target under 3,000 tokens.
Repair packet: target under 1,500 tokens.
Full standard: available by reference, not pasted into every agent prompt.

Then Jankurai can claim it minimizes tokens by minimizing ambiguity, not by starving the model.

### XI. Continuous Proof, CI, and Test Explosion

This section is conceptually right: tests must be routed by proof question, not author convenience.  The project’s audit implementation already appears to normalize changed-path scope, select scope files, build an audit context, and run analyzers across dimensions. ([GitHub][4]) The audit report also includes useful metadata such as fingerprints, policy/manifest fingerprints, dirty-worktree status, version fields, target stack, run ID, scope, score, raw score, and decision state. ([GitHub][4])

The next step is to distinguish **lane existence** from **lane proof**. A mapped proof lane is necessary, but the merge gate should ask:

Did the lane run after the relevant files changed?
Did it run against the current commit or diff?
Did it produce machine-readable evidence?
Did the evidence cover the changed ownership cell?
Was the result attached to the repair receipt?

This is where Jankurai can beat normal CI. Normal CI says “some tests passed.” Jankurai should say “the right proof ran for this exact change.”

### XII. Rendered UX and Browser-Step QA

This section is a major differentiator. Many AI coding tools focus on backend tests, type checks, and lint. Jankurai is right to treat the browser as a real runtime. The paper’s list of agent-created UI failures—overlapping text, broken responsive layout, invisible controls, stale selectors, missing focus states, blank canvases, cramped buttons, clipped labels, sticky footer collisions, layout shift—is exactly right. 

The repo already has a concrete `@jankurai/ux-qa` package described as Playwright DOM geometry checks for rendered UX quality, with Playwright as a peer dependency and axe-core included. ([GitHub][5]) The current UX rule runtime checks edge clearance, target size, interactive overlap, text clipping, button wrapping, horizontal overflow, sticky obstruction, z-index token drift, focus visibility, form labels, and nested scrollbars. ([GitHub][6]) The analyzer builds a report with schema/tool version, URL, viewport, metrics, collected elements, violations, artifacts, artifact coverage, summary, state coverage, and a decision. ([GitHub][7])

This deserves more spotlight in the paper. It is one of the clearest ways to show that Jankurai catches vibe mistakes that ordinary tests miss. Add before/after screenshots or simulated examples in the paper or companion site: “compiled, unit-tested, visually broken, caught by Jankurai.”

### XIII. Security, Supply Chain, and Permissions

This section is solid. The paper correctly treats agentic coding as an expansion of the write surface, not merely an autocomplete improvement.  The codebase’s rule set and standard docs already include stable security-relevant HLT families for generated security risk, secret sprawl, prompt injection, overbroad agency, supply-chain drift, and related controls. ([GitHub][3])

The upgrade I would add is **permission receipts**. When an agent installs a dependency, runs a migration, edits generated output, accesses browser automation, or touches a secret-adjacent path, the receipt should say:

`who/what requested it → which rule allowed it → which scope was granted → what changed → what proof closed it`

This turns “agent permission” into auditable infrastructure.

### XIV. Exceptions, Observability, and Repair Receipts

This is one of the most original sections. The paper’s claim that exceptions should teach repair systems what invariant was protected, why it failed, and what repairs are usually legal is excellent. 

I would move repair receipts earlier in the paper. They are not just a cleanup detail; they are the learning substrate. Every serious adoption story should show:

`finding → repair queue item → patch → proof lane → receipt → score delta`

Also add a command-centered story:

`jankurai explain HLT-010 --for-agent`
`jankurai repair-plan --from agent/repo-score.json`
`jankurai proof-verify --plan ...`

The CLI already has explanation, repair, and proof-verification commands in its command surface, so the paper can confidently make this operational rather than speculative. ([GitHub][2])

### XV. Migration, Versioning, and Governance

The migration section is wise because it rejects rewrite fantasy. Starting with advisory audit, then owner maps, proof lanes, generated zones, version manifests, contracts, database boundaries, bounded Python, and UX proof is the right adoption arc. 

I would add a simple adoption calendar:

**Day 1:** advisory audit, root router, no blocking.
**Week 1:** owner map, test map, generated zones, score baseline.
**Month 1:** ratchet mode for critical rules, proof receipts for changed paths.
**Quarter 1:** HL3 target, release gates, exception expiry, score trend dashboard.

This would make the standard feel adoptable by real teams, not only ideal repositories.

### XVI. Limitations and Research Agenda

Good section. Keep the humility. The paper correctly says weights need empirical calibration, file-size limits should be tested against repair accuracy, and benchmarks still struggle to measure long-running maintenance, security regressions, UI correctness, and production repair quality. 

To strengthen it, add a named benchmark plan:

**Jankurai Bench-10:** ten representative repos, each seeded with known vibe faults.
**Jankurai Bench-100:** broader calibration across stack families.
**Metrics:** detection rate, false-positive rate, time-to-repair, proof coverage, escaped defects, token budget, review latency, and score stability.

The repo already includes benchmark and publication commands in the CLI surface, so this direction matches the implementation trajectory. ([GitHub][2])

### XVII. Conclusion

The conclusion is strong, especially the line that anything less is just faster vibe coding.  I would make the final paragraph even more standard-centered:

> The next generation of software engineering will not be won by the team that generates the most code. It will be won by the team whose repository rejects the most wrong code, proves the most right code, and teaches the next repair agent exactly what happened. Jankurai is that interface.

That lands the paper on the audit/repair thesis, not merely the stack thesis.

### Appendices

The appendices are useful but dense. Appendix A’s stable rule IDs are essential and should remain. Appendix B’s versioned artifact manifest is important because it makes the standard self-auditing. Appendix C’s exception and repair templates are practical and valuable. Appendix D’s canonical file trees are useful but could be shortened in the paper and expanded in docs. Appendix E’s command examples are operationally helpful, but a reader may experience them as CLI overload. 

I would move many command examples into docs and keep only the core adoption loop in the paper:

`init → audit → lane → prove → repair-plan → proof-verify → ratchet`

## Code and project feedback

### What is already genuinely strong

The codebase is not just a paper demo. It has a real CLI, a Rust crate, a UX QA package, schemas, docs, contracts, examples, agent files, GitHub workflow surfaces, and tests. The repository layout shows a serious attempt to make the paper executable. ([GitHub][1])

The Rust crate structure includes audit, boundaries, commands, init, reporting, score history, UI, validation, and version modules. ([GitHub][8]) The crate is published internally as version `0.4.0` and uses practical dependencies for CLI parsing, walking repositories, glob matching, serialization, hashing, TOML, semver, Rust parsing, and temporary test work. ([GitHub][9])

The test surface is also broader than I expected. The repository includes smoke and fixture tests around adoption, audit, boundaries, command surface, generated zones, init/doctor flows, migration safety, proof verification, schema contracts, security evidence, UX policy, reports, and version checks. ([GitHub][10]) The UX QA package also has Playwright specs around artifacts, config, geometry, hit testing, matrices, selectors, Storybook, and visual baselines. ([GitHub][11])

That is a strong foundation.

### The biggest implementation risk

The project currently feels like it has **many powerful commands but not yet one unavoidable daily loop**. The README’s quick-start/adoption path is helpful, and the CLI command surface is rich, but a global standard needs a default behavior that is obvious to a tired engineer and to an autonomous coding agent. ([GitHub][1])

The daily experience should become:

`jankurai pulse`
or
`jankurai next`

One command should tell the coder or agent:

what changed,
what risk increased,
which proof must run,
which evidence is stale,
which repair is highest priority,
whether the score regressed,
and what exact command closes the loop.

Right now, the project has the pieces. The next level is orchestration.

### The code should obey the paper more aggressively

One thing stands out: GitHub shows `main.rs` as a large file, around 1,509 lines and 45.5 KB. ([GitHub][2]) That is not a disaster for an early CLI, but it cuts against the paper’s own philosophy of small files, narrow ownership, and agent-repairable surfaces. Refactoring command definitions and dispatch into smaller modules would make the project itself a better demonstration of Jankurai.

A possible split:

`commands/cli.rs` for top-level CLI shape
`commands/dispatch.rs` for command routing
`commands/proof.rs` for proof commands
`commands/audit.rs` for audit commands
`commands/adoption.rs` for init/adopt/update/doctor
`commands/governance.rs` for certify/govern/publish
`commands/agent.rs` for adapters/context/hooks/issues/explain

That refactor is not just cleanliness. It would let Jankurai dogfood `HLT-008`, ownership routing, and repair locality on its own CLI.

### Rename or harden the `--yolo` path

The code has an init path where `args.yolo` routes into `run_init_yolo(args)`. ([GitHub][2]) Even if the behavior is safe in practice, the word “yolo” is brand-hostile for a project trying to cure vibe coding. This is easy to fix and symbolically important.

Rename it to something like:

`--apply-full-scaffold`
`--adopt-and-commit`
`--dangerously-apply-all`
`--full-send-with-receipt`

Require a receipt. Require a dry-run plan. Require an explicit environment variable. Then the tool’s vocabulary matches its philosophy.

### Make stale proof impossible to miss

The audit report already includes rich fingerprint and version metadata. ([GitHub][4]) The next step is to use that metadata to reject stale evidence. A proof receipt should be invalid when:

the command ran before the changed file hash,
the lane did not cover the changed ownership cell,
the artifact is missing,
the artifact was generated by a different tool version,
the worktree was dirty in a way the receipt does not describe,
or the proof lane was selected by the agent instead of the test map.

This is how Jankurai becomes stronger than ordinary CI.

## The profound upgrade: Jankurai Pulse + ProofGraph

The upgrade I would build next is **Jankurai Pulse**, powered by a repo-local **ProofGraph**.

### What it is

**ProofGraph** is a machine-readable graph of the repository:

`path → owner → rule family → generated zone → source contract → proof lane → required evidence → current receipts → open findings → score impact`

**Jankurai Pulse** is the live command that uses that graph to guide every coding session.

A coder or agent runs:

```bash
jankurai pulse --watch
```

or on a PR:

```bash
jankurai pulse --changed-from origin/main \
  --out agent/pulse.json \
  --md agent/pulse.md
```

The output should say:

Current score: 89
Baseline score: 91
Delta: -2
Merge decision: blocked
Reason: `HLT-002-GENERATED-MUTATION`, `HLT-013-RENDERED-UX-GAP`
Required proof: `contract`, `web`
Stale evidence: Playwright screenshot predates changed route file
Next safest repair: edit OpenAPI source, regenerate client, rerun contract lane
Minimal agent context pack: 2,400 tokens

That is the daily miracle.

### Why this is the next level

It helps coders go faster because they stop guessing which tests to run, which owner owns the change, which generated files are illegal, and which evidence reviewers need.

It reduces mistakes because it catches the exact vibe-coding failure mode: plausible code with missing or wrong proof.

It minimizes tokens because the agent receives only the owner-map slice, rule snippets, changed paths, current findings, prior receipts, and exact rerun commands relevant to the current diff.

It aligns everyone because the human reviewer, the coding agent, CI, security, and release process all read the same artifact.

### The key output files

Add these as first-class artifacts:

`agent/proofgraph.json`
The compiled ownership/proof/generated-zone/rule graph.

`agent/pulse.json`
The current change-scoped score delta, findings, required lanes, and decision.

`agent/pulse.md`
The human-readable PR/review summary.

`agent/evidence-index.json`
All known proof receipts, artifacts, versions, hashes, and freshness status.

`agent/context-pack.md`
The minimal agent context for the next repair.

### New commands

```bash
jankurai pulse . --changed-from origin/main
```

Computes score delta, proof plan, stale evidence, and merge decision.

```bash
jankurai next .
```

Prints the single safest next action: repair, prove, regenerate, add exception, or ask for human review.

```bash
jankurai pack . --changed-from origin/main --max-tokens 3000
```

Generates a minimal context pack for Codex, Claude Code, Cursor, Copilot, terminal agents, or any other adapter.

```bash
jankurai receipts verify .
```

Rejects stale, missing, wrong-commit, wrong-lane, or wrong-version proof receipts.

```bash
jankurai scoreboard . --since 30d
```

Shows rolling score trend, recurring HLT findings, exception debt, mean time to repair, and proof coverage.

```bash
jankurai chaos . --seed generated-mutation
```

Injects safe synthetic vibe faults to prove that the repo catches them.

### The killer feature: vibe-fault seeding

To prove Jankurai works, add a mutation-style suite that deliberately creates common vibe mistakes in a fixture repo:

hand-edit generated client,
add direct SQL in UI,
add secret-like token,
skip a critical test,
add stale handwritten DTO,
add destructive migration,
change UI without UX artifact,
add dependency without rationale,
add overbroad agent permission,
move durable truth into Python.

Then Jankurai should catch them and emit the expected HLT rules. This becomes a benchmark, a demo, and a trust signal.

## How to make the rolling score central

The rolling score should not be a vanity number. It should be a **merge-control instrument**.

I would define five public score fields:

**Current score:** repository score at this commit.

**Baseline score:** accepted score at adoption or release baseline.

**Delta:** score change from baseline or target branch.

**Proof coverage:** percentage of changed paths with fresh, valid proof receipts.

**Exception debt:** count and severity-weighted cost of active exceptions.

Then every PR comment can say:

```text
Jankurai HL3
Score: 88 / 100
Delta: -3 from main
Proof coverage: 74%
Exception debt: 5 active, 2 expired
Decision: Blocked

Top repairs:
1. HLT-004-UNMAPPED-PROOF: apps/web/src/routes/checkout.tsx
2. HLT-013-RENDERED-UX-GAP: checkout route changed without screenshot/geometry proof
3. HLT-016-SUPPLY-CHAIN-DRIFT: new dependency lacks rationale/SCA receipt
```

That is how you make “vibe mistakes” visible before merge.

## Suggested roadmap

### v0.5: Make the loop undeniable

Ship `jankurai pulse`, `jankurai next`, and `jankurai pack`. Refactor the large CLI entry file into smaller ownership-oriented modules. Rename or harden `--yolo`. Add proof receipt freshness checks. Make the README revolve around the daily loop rather than the full command universe.

### v0.6: Make the score credible

Add score history, score delta, exception debt, proof coverage, and recurring-rule tracking. Publish example reports from fixture repos. Add vibe-fault seeding so users can see Jankurai catch real mistake classes.

### v0.7: Make UX QA famous

Turn `@jankurai/ux-qa` into a flagship demo. Show compiled UI that passes ordinary tests but fails geometry, accessibility, visual, or responsive checks. The package already has the right primitives; it needs a “wow” demo path. ([GitHub][5])

### v1.0: Make conformance portable

Separate the **Jankurai Standard** from the **Jankurai Reference Stack**. Keep Rust/TypeScript/PostgreSQL as the reference architecture, but let Go, .NET, JVM, Rails, and Python-heavy shops adopt the audit standard without feeling rejected. The standard wins when everyone can align around proof receipts, HLT IDs, generated-zone policy, and rolling score—even if they disagree on stack.
