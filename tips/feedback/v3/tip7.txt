Jankurai is **much closer to a category-defining project than a normal repo-audit tool**. The paper has a big, memorable thesis; the repo already has a real Rust CLI, audit modules, conformance artifacts, proof planning, repair planning, security evidence, UX QA, publishing/certification commands, and a standard workspace rather than just a manifesto. The README positions Jankurai as an “agent-native repository control-plane and audit CLI,” explicitly focused on ownership maps, proof lanes, generated zones, security boundaries, repair queues, local receipts, and non-hosted repo infrastructure. ([GitHub][1]) The source tree backs that up: the Rust CLI is split into audit, boundaries, commands, init, reporting, validation, versioning, and scoring modules, and there is also a TypeScript `packages/ux-qa` runtime for rendered UX evidence. ([GitHub][2])

I did not execute the full test suite locally, so treat the code feedback as a repository/paper/design review rather than a build verification report. The paper review below is based on the uploaded PDF and its figures, tables, appendices, and references. 

## The biggest upgrade: make Jankurai the “trustworthy merge standard,” not a stack argument

The world-shocking message is this:

**AI made code cheap. Jankurai makes merge trust measurable.**

That should become the center of the paper, README, CLI, website, badges, demo, and public benchmark. The Rust/TypeScript/Postgres default stack is useful, but it is not the most universal part. The universal part is this:

> Every agent-authored change should carry ownership, permission, proof, provenance, and repair evidence. If it does not, the rolling trust score drops and the merge is blocked or routed.

That is the cure to vibe coding. Not “better prompts.” Not “ask the model to be careful.” Not even “use Rust.” The cure is a **standard audit loop** that continuously asks:

1. What changed?
2. Who owns it?
3. Which rule was implicated?
4. Which proof lane covers it?
5. Which artifacts prove it?
6. Did the score regress?
7. Is the exception scoped, owned, and expiring?
8. Can the next agent repair it without guessing?

The repo already contains many of the ingredients: conformance docs define HL levels and HLT rule IDs, the testing docs describe proof lanes and receipts, and the architecture docs state the target stack and ownership model. ([GitHub][3]) The next leap is to turn those ingredients into a **PR-native rolling trust system**.

I would name that core concept something like:

**Jankurai Rolling Trust Score**, **Jankurai Merge Trust Index**, or **Jankurai Proof Score**.

The current paper has “centerline drift” and score/rubric ideas, but the “rolling score catches vibe mistakes over time” story should become a main plotline, not a secondary governance detail. A static repo score says, “How healthy is the repo?” A rolling score says, “Did this PR make the repo more trustworthy or more vibey?”

A strong formula could be:

```text
PR Trust Score =
  min(hard_caps,
      0.30 * changed_path_ownership
    + 0.25 * proof_receipt_coverage
    + 0.15 * security_and_permission_evidence
    + 0.10 * contract_and_generated_zone_integrity
    + 0.10 * rendered_ux_or_runtime_artifacts
    + 0.10 * repairability_and_observability)
  - regression_penalty
  - stale_proof_penalty
  - exception_debt_penalty
```

That gives the paper a new, powerful sentence:

> Jankurai does not merely score a repository. It scores the trust delta introduced by each change.

That is the missing “global standard” hook.

# Part 1 — Detailed paper feedback, section by section

## Title and abstract

**What works:**
“Humans Were the Bug” is unforgettable. It has manifesto energy. The subtitle, “From Vibe Coding to Agent-Native Engineering,” correctly names the transition. The abstract’s first paragraph is excellent: AI changes the scarce act from writing plausible code to achieving trustworthy merge. That should stay.

**What to improve:**
The title is emotionally strong but potentially easy to misunderstand. You already soften it in Section I by saying humans were the bug because repositories were organized around human memory, patience, taste, and slow review. I would move that clarification into the abstract or first page margin.

A tier-1 abstract should more explicitly state the contribution. Right now it says the paper “defines agent-native engineering” and “turns the manifesto into an enforceable standard.” Good, but make it sharper:

> This paper contributes: (1) a threat model for agent-generated code, (2) a fault taxonomy for vibe coding, (3) a conformance standard with stable rule IDs, (4) a rolling audit score for merge trust, and (5) a reference implementation that routes changed paths to proof lanes and repair queues.

The abstract should also say that Jankurai is **profile-compatible**, with Rust/TypeScript/Postgres as the reference profile. Otherwise, readers may think the paper is “another stack manifesto” instead of a standard.

**Suggested abstract opening:**

> AI coding has made plausible code abundant and trustworthy merge scarce. The central engineering question is no longer “can code be generated?” but “can the repository reject wrong generated code, localize the defect, prove the repair, and teach the next agent what happened?”

That sentence is the whole movement.

## I. New Bottleneck: Trustworthy Merge

**What works:**
This section is one of the strongest. “The correct response is not to slow agents down until the old review process survives” is a great thesis. The four claims are clear and useful.

**What to add:**
Open with a concrete failure story before the conceptual claims. The paper needs one memorable “vibe mistake” scenario:

> An agent changes checkout logic. TypeScript compiles. Unit tests pass. The reviewer sees plausible code. But the patch duplicated a payment side effect, bypassed an idempotency key, and edited a generated client by hand. In a human-native repo, this mistake looks reviewable. In an agent-native repo, the changed path triggers payment ownership, idempotency proof, generated-zone checks, security review, DB constraints, and a repair queue. The merge is blocked before production.

That story makes “trustworthy merge” visceral.

**Story upgrade:**
Position Jankurai as the next historical control layer:

* Version control made changes traceable.
* CI made builds repeatable.
* Linters made style enforceable.
* Type systems made some invalid states unrepresentable.
* SBOM/SCA made dependencies auditable.
* **Jankurai makes AI-generated change trust auditable.**

That framing makes Jankurai feel inevitable rather than optional.

---

## II. Definitions and Threat Model

**What works:**
The definitions are excellent. “Vibe coding is any development mode where plausible code is accepted without explicit ownership, proof routing, generated-boundary checks, security gates, and repair evidence” is strong and operational. The threat model table on page 3 is one of the paper’s best assets because it turns vague fear into specific failure modes: false plausibility, hostile context, tool overreach, secrets, dependency poisoning, and false-green tests.

**What to improve:**
Add a trust-boundary diagram. The section should show five zones:

```text
trusted standard
  ↓
repo manifests
  ↓
agent instructions
  ↓
tool outputs / CI evidence
  ↓
untrusted issue text / web / generated suggestions
```

This will make hostile context and prompt injection feel like a software architecture problem, not an AI ethics footnote.

**Add non-goals:**
A standard becomes more credible when it states what it does not claim. Add:

* Jankurai does not prove all program correctness.
* Jankurai does not replace domain review.
* Jankurai does not require all repos to use one stack.
* Jankurai does not make generated code inherently safe.
* Jankurai standardizes the evidence needed to trust a change.

That will make the claims harder to attack.

---

## III. Languages as Bottleneck Compression

**What works:**
The “language as compression package” idea is genuinely good. It explains why syntax matters less in the agent era and why auditability, proof locality, memory safety, and explicit boundaries matter more.

**What to improve:**
This section currently reads partly like a historical essay. Tie every paragraph back to the audit standard. The key point should be:

> In the agent era, a language is valuable to the extent that it moves failure detection earlier, localizes proof, and reduces hidden context.

Add a small table with three columns:

```text
Language feature → Human-era benefit → Agent-era audit benefit
```

Example:

* Rust ownership → fewer memory bugs → compile-time rejection of many unsafe generated edits.
* TypeScript types → maintainable JS → fast frontend contract drift detection.
* SQL constraints → durable data rules → invariant survives model hallucination.
* Python flexibility → fast exploration → must be bounded away from durable truth.

**Potential risk:**
The paper should avoid sounding like “agents can learn syntax, therefore language skill no longer matters.” A better nuance:

> Agents can imitate syntax cheaply; they still benefit from languages that expose invalid states early and produce precise diagnostics.

That is more defensible.

## IV. Technical Promise Versus Standard Gravity

**What works:**
This section is important because it distinguishes good technology from a default standard. The Julia discussion is respectful and useful. Table III gives a useful frame: real promise, adoption limiter, agent-era penalty.

**What to improve:**
This section could be shorter in the main paper. The paper’s tier-1 contribution is not “why Rust beats Julia/Nim/Crystal/Elm.” It is “why repositories need machine-auditable merge trust.” Move some language-by-language detail to an appendix and keep the main text focused on standard gravity.

**Add this sentence:**

> Standard gravity is not popularity worship. It is the amount of shared operational surface an agent can rely on without hallucinating local convention.

That makes the section much more aligned with the paper’s main claim.

## V. Vibe-Coding Fault Taxonomy

**What works:**
This is probably the paper’s most important technical contribution. Table IV gives the project a real ontology. Fig. 1 on page 4 is strong because it turns “vibe coding” into Top-Level Risk buckets: security, business truth, contracts/data, verification, entropy, context/setup, and repair. The paper already states that RPN is a Jankurai policy weight, not an empirical frequency claim, which is important.

**What to improve:**
Make this the emotional and scientific center of the paper. The phrase “vibe coding is not a mood; it is a fault class” is excellent. Build around it.

Add three things:

### 1. A “vibe mistake lifecycle”

```text
plausible generation
→ hidden boundary violation
→ wrong or missing proof lane
→ false-green result
→ human approval under uncertainty
→ production defect or repair debt
```

Then show how Jankurai interrupts each step.

### 2. A detector maturity column

You already have detector-status material in Appendix A, Table XXI. Move a simplified version into Section V:

* **Hard deterministic:** generated-zone mutation, secret-like content, missing owner map, destructive migration patterns.
* **Evidence-driven:** business invariants, idempotency, authorization drift.
* **Soft/experimental:** context retrieval quality, model/prompt drift, visual triage.

This makes the taxonomy honest and impressive.

### 3. Calibration plan

A tier-1 paper needs an empirical path:

> We will calibrate RPN weights using injected faults, real PR histories, repair time, escaped defects, false-positive rates, and proof-lane coverage.

That lets the standard be bold without sounding arbitrary.

## VI. Jankurai Standard and Conformance

**What works:**
This section is the core of the “global standard” idea. The adoption ladder — `agents → score → ci → full → ratchet` — is excellent. The HL0-HL5 conformance ladder is also exactly the right move: progressive adoption beats binary purity.

The hosted standard docs already reflect this: HL3 requires a root agent router, owner map, test map, generated-zone manifest, proof lanes, audit output, score policy, and repair queue, and the docs list rule families and hard gates. ([GitHub][3])

**What to improve:**
Separate three concepts more explicitly:

1. **The Jankurai Standard** — schemas, conformance levels, rule IDs, governance.
2. **The Jankurai CLI** — reference auditor and installer.
3. **The Jankurai Reference Profile** — Rust/TypeScript/Postgres/bounded Python.

That is crucial for global adoption. If the standard appears inseparable from one stack, many teams will reject it before understanding the audit model.

**Important consistency fix:**
The paper appendix lists initial HLT rules through HLT-019, but the code already includes newer rule IDs like `HLT-020-CI-HARDENING-GAP` and `HLT-021-DESTRUCTIVE-MIGRATION` in audit/cap logic. The rule catalog in the paper, docs, schemas, and CLI should be made perfectly consistent before public launch. This is exactly the kind of “standard drift” Jankurai itself is trying to prevent. 

**Big addition:**
Add a **Conformance Badge Contract**:

```text
HL3 / score 91 / no high findings
standard: 0.4.0
schema: 1.2.0
ruleset: 2026.05-ed3
commit: <sha>
report: agent/repo-score.json
proof receipts: target/jankurai/proof-receipts/
exceptions: 2 open, 0 expired
```

That turns Jankurai into something people can display, compare, and enforce.

## VII. A 100-Point Stack Rubric

**What works:**
The rubric weights the right things: correctness loop, security, runtime, concurrency, boundaries, simplicity, ecosystem, observability, durable data, and product velocity. The decision to weight velocity only 2 points is provocative and defensible because the paper argues that AI already made first-draft velocity cheap.

**What to improve:**
Rename the stack score so it cannot be confused with the repo audit score. The paper uses ANSS for stack ranking, while the code has repo score outputs and score history. I would make the distinction explicit:

* **ANSS**: Agent-Native Stack Score, a reference architecture score.
* **JRTS**: Jankurai Rolling Trust Score, a repository/PR trust score.

Then state:

> ANSS chooses a sane default stack. JRTS decides whether a change deserves merge trust.

That is a huge clarity boost.

**Risk:**
The stack rubric may attract arguments about Go/.NET/JVM before readers absorb the main standard. Keep it, but frame it as a reference profile, not the standard itself.

## X. Agent Repository Controls and Tool Adapters

**What works:**
The tool adapter matrix is excellent. You correctly identify that AGENTS.md, CLAUDE.md, Cursor rules, Copilot instructions, and terminal-agent prompts should not become separate sources of truth. They should be generated or mirrored from one canonical standard.

**What to improve:**
Make **token minimization** a first-class metric. The paper has the right claim — token minimization is repository design, not a prompting trick — but it should define measurable controls:

* Root agent router under a byte/token budget.
* No duplicate policy across AGENTS.md, CLAUDE.md, Cursor rules, and Copilot instructions.
* Context packs must list why each file was included.
* Command output must be filtered by default, with raw output available.
* Stale docs and contradictory instructions lower the score.
* Every context pack labels sources as trusted policy, local code, generated artifact, test evidence, or untrusted issue/web text.

This can become a killer CLI feature:

```bash
jankurai context-pack . \
  --task "fix checkout idempotency" \
  --changed apps/web/src/checkout.ts crates/application/src/payment.rs \
  --agent codex \
  --budget 16000 \
  --out target/jankurai/context-pack.json \
  --md target/jankurai/context-pack.md
```

The repo already exposes a `ContextPack` command in the CLI, so this is an evolution, not a new direction.

## What is already impressive

The CLI surface is already broad: audit, adopt, init, update, doctor, context pack, repair planning, proof/lane/prove/proof verification, registry, cells, migration, benchmarking, certification, governance, publishing, repair, optimization, Rust mapping/witness/diagnostics, exceptions, adapters, agent verification, CI installation, hooks, issue export, UX passthrough, and security evidence.

The dimension weights are also already coherent: ownership, contracts, proof, security, shape, data, observability, context, Python containment, and build speed. 

## Highest-priority code improvements

### 1. Make rolling score first-class

You already have score history concepts in the repo, but the product should make score deltas central:

```bash
jankurai score trend .
jankurai score diff . --base main --head HEAD
jankurai score explain . --rule HLT-008-FALSE-GREEN-RISK
jankurai score badge . --out target/jankurai/badge.svg
```

Every PR should show:

```text
Base score:      89
Current score:   84
Delta:           -5
Hard cap:        76 due to HLT-006-DIRECT-DB-WRONG-LAYER
Changed-path trust: fail
Proof coverage:  63%
Merge decision:  block
```

This is the “catches vibe mistakes” mechanism.

### 2. Build the Jankurai Proof Graph

This is the profound upgrade I would prioritize.

Create a graph where nodes are:

* changed files
* owners
* proof lanes
* commands
* receipts
* artifacts
* contracts
* generated zones
* dependencies
* exceptions
* rule IDs
* findings

Edges are:

* file → owner
* file → proof lane
* proof lane → command
* command → receipt
* receipt → artifact
* contract source → generated zone
* finding → rule
* rule → required evidence
* exception → rule suppression
* dependency → security lane

Command:

```bash
jankurai proof-graph . \
  --changed-from origin/main \
  --out target/jankurai/proof-graph.json \
  --md target/jankurai/proof-graph.md \
  --svg target/jankurai/proof-graph.svg
```

This would make Jankurai visually and operationally different from every linter. It becomes a **merge control tower**.

### 3. Upgrade from pattern scanning to semantic detectors

The current pattern scanners are a good MVP, but tier-1 credibility requires more semantic analysis. Keep the current detectors, but classify them as heuristic unless backed by AST or artifact evidence.

Add semantic analyzers:

* Rust: `syn`/cargo metadata call graph, crate ownership, domain I/O detection, unsafe ledger, typed error coverage.
* TypeScript: ts-morph or TypeScript compiler API for raw fetch detection, generated client imports, DTO mirrors, test coverage, Storybook state coverage.
* SQL: tree-sitter-sql or sqloxide for destructive migration parsing instead of line heuristics.
* CI: YAML parser for permissions, pinned actions, secret exposure, pull_request_target risk.
* Dependency graph: package additions mapped to security receipt requirements.

The current dependency list already includes `syn`, `quote`, `cargo_metadata`, regex/glob scanning, and serialization tools, so the Rust semantic path is natural. 

### 4. Rename or hide `--yolo`

The CLI has an init `--yolo` option for full-send adoption: apply scaffold, install observe CI, score, append history, stage, and commit. That is fun internally, but it conflicts with the anti-vibe message. Rename it before launch.

Better names:

```text
--guided-commit
--adopt-and-commit
--bootstrap-commit
--launch
--control-plane-commit
```

For a project fighting vibe coding, `--yolo` is a meme liability.

### 5. Move rules and caps into versioned ruleset files

Right now, many cap values are compiled into Rust constants. That is fine early, but a standard needs a visible ruleset:

```text
rulesets/jankurai-0.4.0.toml
rulesets/jankurai-0.5.0.toml
schemas/ruleset.schema.json
```

Then the CLI can report:

```json
{
  "ruleset": "jankurai-0.4.0",
  "ruleset_fingerprint": "sha256:..."
}
```

This makes governance, forks, experiments, and compatibility much cleaner.

### 6. Treat profile compatibility as a product feature

Keep the reference profile strong, but allow:

```bash
jankurai init . --profile rust-ts-postgres
jankurai init . --profile go-ts-postgres
jankurai init . --profile dotnet-ts-postgres
jankurai init . --profile rails-postgres
jankurai init . --profile python-ai-boundary
jankurai init . --profile custom --profile-file profile.toml
```

A global standard wins if teams can adopt the audit model without immediately accepting the full stack doctrine. The paper can still argue that Rust/TS/Postgres is the strongest default.

### 8. Make UX QA a standalone star

`@jankurai/ux-qa` could become the entry point for frontend teams even before they adopt the whole standard. Package it as:

```bash
npx @jankurai/ux-qa analyze \
  --url http://localhost:5173/checkout \
  --viewport 375x812 \
```


Rendered UX failures are a universal AI-coding pain. Lead with that in demos.

The fastest adoption path is:

# The profound upgrade: Jankurai Control Tower

My strongest recommendation is to build a new layer called something like:

**Jankurai Control Tower**
or
**Jankurai Flight Recorder**
or
**Jankurai Proof Graph**

It would combine:

1. **Changed-path analysis**
2. **Owner routing**
3. **Proof-lane selection**
4. **Permission receipts**
5. **Evidence receipts**
6. **UX/security/contract artifacts**
7. **Score delta**
8. **Repair queue**
9. **Context pack for the next agent**

The user experience:

```bash
jankurai control-tower . --changed-from origin/main
```

Output:

```text
Jankurai Control Tower

Merge decision: BLOCK
Rolling score: 89 → 76
Primary cap: HLT-006-DIRECT-DB-WRONG-LAYER

Changed surfaces:
- apps/web/src/checkout/CheckoutForm.tsx

- db/migrations/20260504_add_payment_status.sql

Required proof:
✓ fast
✓ contract
✗ db-idempotency
✗ web-mobile-rendered-ux
✓ security
✗ migration-safety

target/jankurai/context-pack.md
```

That is the next level. It directly helps coders go faster because it removes the biggest time sink in AI coding: figuring out what broke, what proof matters, and what the agent should read next.

# The standard audit and rolling score story to add to the paper

Here is the narrative I would add almost verbatim:

> Vibe coding fails because plausibility is evaluated at a single moment by an overloaded reviewer. Jankurai replaces momentary plausibility with a rolling trust ledger. Each change is compared against the repository’s claimed centerline: ownership maps, proof lanes, generated zones, permissions, security gates, contracts, UX artifacts, and repair receipts. A score regression is not merely a number; it is a localized trust delta. The system can say: “This PR lowered trust because it edited a generated client, skipped the mapped contract lane, added a dependency without rationale, and changed checkout UI without rendered proof.” That is the difference between code review and standard audit.

And this:

> The score is not a vanity metric. It is a merge-control protocol. A high score without receipts is invalid. A low score without routed findings is invalid. A finding without a repair command is incomplete. A repair without proof evidence is a claim. Jankurai turns all four into machine-readable artifacts.

That would elevate the project’s story dramatically.

# Final verdict

The project already has the bones of something major. The paper’s strongest idea is not merely “Rust/TypeScript/Postgres is the best stack.” The stronger, world-scale idea is:

**AI coding needs a standard audit layer, and every repository needs a rolling trust score that catches vibe mistakes before merge.**

Make that the center. Treat the stack as the reference implementation. Treat the CLI as the first auditor. Treat the rule IDs, proof receipts, generated-zone policy, UX artifacts, and score history as the actual standard.

The killer phrase is:

**No proof, no merge.**

That is how Jankurai becomes the cure to vibe coding.
