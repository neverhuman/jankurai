This is genuinely exciting. The project already has the seed of a category-defining idea: **the bottleneck is no longer code generation; the bottleneck is trustworthy merge**. That phrase should become the center of gravity. The paper’s strongest move is that it turns “vibe coding” from an insult into an auditable failure mode: missing ownership, missing proof lanes, generated-zone drift, secret exposure, false-green tests, and repair without receipts. The repository is also much more real than a manifesto: it already contains a Rust CLI, audit/report machinery, schemas, proof commands, repair planning, security evidence, UX QA hooks, score history, agent maps, and standard-version artifacts. 

My highest-level recommendation: **make Jankurai less about “the best stack” and more about “the public proof interface for AI-assisted software.”** The stack recommendation can stay, but the world-shocking standard is bigger: a repo should be able to prove, for every change, **who owns it, what proof lane applies, what evidence was produced, what risk changed, what repair happened, and how the rolling score moved**.

## Executive take

Jankurai’s most powerful thesis should be:

> **Vibe coding is not writing code with AI. Vibe coding is merging code without receipts.**

That framing is sharper, harder to dismiss, and easier to standardize. It also avoids blaming humans in a way that may alienate the exact senior engineers, security teams, platform teams, and standards bodies you need. Humans are not the defect; **repositories optimized around human memory are the defect**.

The project should aim to own a new primitive:

> **Mergeworthiness**: the machine-checkable claim that a change is owned, bounded, proven, secure enough, and repairable.

The paper already points there through proof lanes, generated zones, conformance levels, repair receipts, and centerline drift. The code already has a broad command surface around audit, adoption, proof, repair, security, UX, certification, governance, publishing, and versions. ([GitHub][1])

The next leap is to make this longitudinal and PR-native:

> **A rolling Jankurai score should not merely say “this repo scored 88.” It should say “this PR moved the repo from 86 to 88, resolved HLT-004, introduced no new HLT-010 risk, ran the required proof lanes, stayed within its permission budget, and left receipts.”**

That is the “shock the world” moment: not another linter, not another agent prompt, not another stack manifesto, but a **merge-time audit standard for agent-generated code**.

## I. New Bottleneck: Trustworthy Merge

This section is one of the paper’s strongest. The “typing plausible code is no longer scarce” thesis is correct and important. The section argues that agentic coding removes the old throttle of slow human authorship, so repositories must become control surfaces for generated code. That is the right opening move. 

What to strengthen:

The section should introduce **merge as the unit of trust** earlier and more forcefully. The old software lifecycle treated authoring as the center. The agent-native lifecycle treats merge as the center.

Suggested framing:

> In human-native engineering, the scarce event was authoring. In agent-native engineering, the scarce event is merge. A generated patch has almost no meaning until the repository can prove whether it is owned, bounded, secure, and behavior-preserving.

Add a tiny formal loop:

```text
draft → route → constrain → prove → receipt → merge
```

Then make Jankurai the standard for that loop.

Also, be careful with “humans were the bug.” Use it as a doorway, then immediately redeem humans:

> Humans remain responsible for goals, standards, judgment, exceptions, and calibration. What failed was not humanity; it was repository architecture that required humans to remember invisible rules at machine speed.

That will make the paper feel serious rather than merely provocative.

## II. Definitions and Threat Model

This is excellent. The definitions of vibe coding, agent-native engineering, proof lane, generated zone, repair receipt, and centerline drift are exactly the kind of vocabulary a standard needs. The paper’s threat model correctly identifies false plausibility, hostile context, tool overreach, secret exposure, dependency poisoning, and false-green tests. 

The best upgrade is to make one definition even sharper:

> **Vibe coding is an acceptance failure, not an authorship style.**

This is important because people will argue, “But vibe coding is just a fun way to prototype.” Your answer should be: prototyping is fine; **unproven merge is the fault**.

I would also add a short threat model boundary:

```text
In scope:
- generated or agent-assisted source changes
- repository-local policy drift
- proof-lane misrouting
- generated-zone mutation
- dependency and secret changes
- UI/runtime behavior not caught by compilation

Out of scope:
- malicious compiler/toolchain compromise beyond normal SCA/provenance controls
- formal verification of all program semantics
- fully automated product judgment
```

Then add an “attacker model” table:

| Actor               | Goal                                     | Jankurai response                     |
| ------------------- | ---------------------------------------- | ------------------------------------- |
| Plausible agent     | Merge code that looks right but is wrong | proof lanes, owner maps, receipts     |
| Hostile context     | Convince agent to bypass policy          | source hierarchy, permission receipts |
| Rushed human        | Approve because patch looks good         | score delta, required evidence        |
| Dependency attacker | Enter through generated package addition | SCA, SBOM, dependency rationale       |
| Future maintainer   | Misread old exception as normal policy   | scoped exceptions with expiration     |

The section introduces centerline drift, but the paper should turn it into the bridge to rolling score. Define it here as a time-series concept:

> **Centerline drift is not only the distance between claimed and observed conformance at one audit. It is the slope of that distance over time.**

That sentence sets up the rolling-score story.

## III. Languages as Bottleneck Compression

The language-history argument is compelling and memorable. It reframes languages as systems that move burdens from human memory into syntax, types, runtimes, libraries, and tooling. That is much better than the usual “language wars” framing.

A language or stack is agent-native when it rejects bad generated code quickly through:

| Property              | Agent-native value                             |
| --------------------- | ---------------------------------------------- |
| Static types          | reject wrong shapes before runtime             |
| Memory safety         | reject or prevent entire vulnerability classes |
| Explicit errors       | make failure repairable                        |

| Deterministic tooling | let agents verify locally                      |

This lets you say:

That phrase, **closest to the edit**, is important. It gives the paper a concrete evaluation lens.

## XI. Continuous Proof, CI, and Test Explosion

This section is strong and practical. The paper correctly says test volume is only useful if routed. A repo with many tests but no `test-map.json` still asks agents to guess. 

The key upgrade is to formalize **proof lane calculus**:

```text
required_lanes(change) =
  owner_lane(path)
  ∪ risk_lane(diff)
  ∪ boundary_lane(contract/db/generated/security/ux)
  ∪ release_lane(if protected surface)
```

Then require each PR report to include:

```json
{
  "required_lanes": ["fast", "contract", "web"],
  "executed_lanes": ["fast", "contract"],
  "missing_lanes": ["web"],
  "merge_decision": "blocked"
}
```

This is also where false-green tests become auditable. Add a requirement:

> If a changed path maps to a proof lane but the lane has no semantic assertion touching the changed behavior, the audit SHOULD emit HLT-008-FALSE-GREEN-RISK.

That is hard to detect perfectly, but even a heuristic detector is valuable.

Add “red evidence” to repair receipts:

```text
repair receipt = failing proof before + changed paths + passing proof after
```

Without the failing proof before, an agent might merely have run a green test that never covered the bug.

## Most important code improvements

### 1. Dogfood the small-file doctrine

The CLI appears to have a very large `main.rs` that mixes argument definitions, routing, audit writing, UX passthrough, yolo init behavior, and command execution. The repository has good module structure elsewhere, but the entrypoint itself should embody the paper’s “small files, routed ownership, narrow surfaces” philosophy. ([GitHub][5])

Recommended refactor:

```text
src/main.rs                  only parse + dispatch
src/cli/mod.rs               clap command definitions
src/cli/args/*.rs            per-command args
src/commands/*               execution
src/output/*                 JSON/MD/SARIF/JUnit/GitHub summary
src/safety/*                 env gates, git mutation gates, permission checks
```

This matters symbolically. A standard about agent repair locality should make its own repair locality obvious.

### 2. Replace stringly typed TLR priority with an enum

The repair queue currently sorts findings using TLR string matching. The priority function recognizes values such as `"Security"` and `"Business truth"`, while the rule metadata uses richer TLR labels such as `"Security, secrets, agency"`. That creates a risk that security findings may not receive the intended top priority unless normalized elsewhere. ([GitHub][11])

Fix:

```rust
enum TopLevelRisk {
    Security,
    BusinessTruth,
    ContractsData,
    VerificationUx,
    Entropy,
    ContextSetup,
    Repair,
    Streaming,
}
```

Then serialize display names separately. Never sort critical repair work by free-form prose.

### 3. Make changed-path coverage first-class now

The paper’s appendix says changed-path coverage is planned for context/setup rules. That should move to the top of the roadmap. HLT-003 ownerless path and HLT-004 unmapped proof are the essence of Jankurai. 

A PR audit should always answer:

```text
For every changed path:
- Which owner matched?
- Which proof lane matched?
- Which generated-zone policy matched?
- Which risk classifier matched?
- Which receipts satisfy the lane?
```

If this is not complete, the PR is not truly agent-native.

### 4. Upgrade score history into rolling score

The README already describes score history and local reports, and the agent folder contains score-history artifacts. ([GitHub][6]) But score history should become a first-class model, not just a log.

Add:

```text
jankurai score trend
jankurai score explain
jankurai score regressions
jankurai score recurrent
```

The score should penalize unresolved age and recurrence:

```text
recurrence_penalty(rule, window) =
  min(max_penalty, repeat_count(rule, window) * rule_weight)
```

This catches teams that repeatedly “fix” the same class of mistake without changing the repo shape.

### 5. Make proof receipts tamper-evident

The report already includes fingerprints and manifest fingerprints. ([GitHub][9]) Push that into receipts:

```json
{
  "lane": "contract",
  "command": "just contract-check",
  "started_at": "2026-05-04T12:32:10Z",
  "duration_ms": 18422,
  "exit_code": 0,
  "stdout_sha256": "...",
  "stderr_sha256": "...",
  "artifact_sha256": "...",
  "tool_versions": {
    "rustc": "...",
    "node": "...",
    "postgres": "..."
  }
}
```

This makes Jankurai more credible for regulated environments.

### 6. Rename or hide `--yolo`

The `--yolo` path is funny and developer-friendly, but it fights the seriousness of the standard. The code path appears to commit broad scaffold changes, observe CI, hooks, score history, and score trailers under a “yolo” mode. ([GitHub][5])

For public standards adoption, rename it:

```text
--adopt-and-commit
--full-adoption
--bootstrap-commit
```

Keep `--yolo` as an alias if you want the personality, but do not make it the visible enterprise path.

### 7. Make UX QA self-healing to install

The UX passthrough shells to the package CLI. ([GitHub][5]) That is okay, but the command should produce a friendly, agent-actionable repair if the package is not built.

Instead of:

```text
node packages/ux-qa/dist/cli.js failed
```

Emit:

```text
HLT-013-RENDERED-UX-GAP
Reason: UX QA runtime is not built.
Repair: run pnpm --dir packages/ux-qa install && pnpm --dir packages/ux-qa build
Proof: jankurai ux run --out target/jankurai/ux-qa.json
```

That makes setup failure repairable.

### 8. Treat rule repairs as executable recipes

The rules already have repair eligibility and risk levels. ([GitHub][11]) Use that to create machine-runnable repair recipes.

Example:

```toml
[HLT-003-OWNERLESS-PATH]
eligibility = "agent_assisted"
risk = "medium"
steps = [
  "find nearest owner-map entry",
  "suggest owner-map addition",
  "run jankurai audit --changed <path>"
]
requires_human = true
```

Then:

```text
jankurai repair --rule HLT-003 --prepare
```

This would convert Jankurai from an auditor into a repair accelerator.

# 4. The profound next-level upgrade: Jankurai Flight Recorder + Boundary Oracle

The biggest upgrade I would build is a combined system:

## **Jankurai Flight Recorder**

Every PR gets a complete, signed mergeworthiness record.

It answers:

```text
What changed?
Who owns it?
What was the blast radius?
What proof was required?
What proof actually ran?
What evidence was produced?
What permissions were used?
What score changed?
What findings were introduced, resolved, repeated, or aged?
Is this mergeworthy?
```

Example artifact:

```json
{

  "head": "abc123",

Commands:

jankurai flight . \

  --window 30d \
  --out target/jankurai/score-trend.json

jankurai score explain . \
  --from target/jankurai/previous-score.json \
  --to target/jankurai/repo-score.json \
  --md target/jankurai/score-delta.md
```

This is the standard audit story the world will understand. It becomes the “black box recorder” for AI-assisted software changes.

## **Jankurai Boundary Oracle**

This is the token-minimization breakthrough.

Build a repo graph:

```text
nodes:
- files
- symbols
- crates/packages
- owners
- contracts
- generated zones
- DB tables
- migrations
- API routes
- UI routes/stories
- proof lanes
- rules
- exceptions
- repair receipts

edges:
- imports
- owns
- generated_from
- proves
- reads_table
- writes_table
- calls_api
- emits_event
- consumes_event
- violates_rule
```

Then an agent can ask:

```bash
jankurai oracle ask \
  --task "fix checkout tax calculation" \
  --max-tokens 6000 \
  --out target/jankurai/context-pack.json
```

And receive:

```json
{
  "task": "fix checkout tax calculation",
  "likely_owner": "crates/application",
  "legal_edit_paths": [
    "crates/domain/**",
    "crates/application/**",
    "db/migrations/**"
  ],
  "forbidden_paths": [
    "apps/web/src/generated/**",
    "contracts/generated/**"
  ],
  "relevant_files": [
    "crates/domain/src/tax.rs",
    "crates/application/src/checkout.rs",
    "db/migrations/2026_checkout_tax.sql"
  ],
  "required_lanes": ["fast", "db", "release"],
  "recent_repairs": [
    "docs/repair/2026-04-12-tax-rounding.md"
  ],
  "known_risks": [
    "HLT-008-FALSE-GREEN-RISK",
    "HLT-006-DIRECT-DB-WRONG-LAYER"
  ]
}
```

This is where Jankurai becomes the cure for token bloat. The agent does not need 80,000 tokens of repository soup. It needs the **minimum sufficient proof context**.

Add a metric:

```text
Blast Radius Index = changed ownership cells + boundary crossings + proof lanes + generated-zone touches + permission elevation
```

Then block risky PRs:

```text
If blast_radius_index > lane_budget and no exception exists, block merge.
```

This gives teams a concrete anti-vibe control: agents can move fast, but they cannot silently widen the blast radius.
