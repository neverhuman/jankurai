I read the uploaded paper end-to-end and inspected the public repository/source through GitHub. The project is genuinely exciting: the strongest idea is that **the scarce resource in AI-era software is no longer code generation; it is trustworthy merge**. The paper already names the right primitives: ownership maps, proof lanes, generated zones, conformance levels, stable rule IDs, repair queues, versioned artifacts, rendered UX proof, and audit receipts. The repo also has far more implementation surface than a manifesto: the README frames Jankurai as a local Rust CLI/control plane, not a hosted service; it starts read-only, can install guidance/CI/hooks/ratchets, and produces repo-score/adoption-plan artifacts.  ([GitHub][1])

My top recommendation is this: **make “rolling score” the protagonist.** A static audit says “this repo is okay right now.” A rolling score says “this agentic workflow is getting safer or drifting into vibes.” That is the world-shocking move: not another linter, not another AI wrapper, but a **standardized proof ledger** that shows whether every AI-generated change made the repo more verifiable or less verifiable.

## Executive assessment

Jankurai has the bones of a serious standard. It is trying to give teams a shared language for agent-era engineering: “HLT-010 secret sprawl,” “generated-zone mutation,” “missing proof lane,” “ownerless path,” “rendered UX gap,” “repair receipt.” That shared vocabulary is exactly what can align humans, agents, CI, security, product, and leadership around the same facts.

The paper’s strongest thesis is already tier-1 caliber: **repositories must become verification interfaces for generated code.** That aligns with public evidence: DORA’s 2025 AI report frames AI as an amplifier of existing software-delivery capability rather than an automatic productivity win; Veracode reports risky security flaws in 45% of tested AI-generated code samples; GitGuardian reports 28.65 million new hardcoded secrets detected in public GitHub in 2025, with AI-service secret leaks rising sharply; and GitHub’s Octoverse describes AI and TypeScript as central forces in the current developer ecosystem. ([Dora][2])

The main risk is positioning. The stack recommendation is powerful, but it can distract from the more universal breakthrough. If Jankurai wants to become a global standard, the standard should feel bigger than one stack. I would frame Rust/TypeScript/Vite/Postgres/bounded Python as the **reference architecture profile**, not the definition of the standard. The standard is the audit/proof/control-plane protocol. The stack is one excellent conforming target.

## What already works extremely well

The paper’s title, **“Humans Were the Bug,”** is memorable and emotionally strong. The abstract immediately locates the real bottleneck: agents can create plausible code, but repositories must reject wrong code, localize defects, prove repairs, preserve security posture, and teach the next agent what happened. That is the correct battlefield.

The codebase supports this ambition. The repo has agent surfaces for Codex/Claude/Cursor-style workflows, `agent/` manifests, contracts, DB placeholders, docs, examples, schemas, UX tooling, and a Rust CLI under `crates/jankurai`. The repository language mix is mostly Rust, with TeX for the paper and TypeScript/JavaScript for UX tooling, which matches the paper’s implementation story. ([GitHub][1])

The CLI is broad in a good way. The command modules include adoption, benchmarking, cell registry/proof, certification, CI, context packs, doctor, exceptions, hooks, migration, optimization, proof, publication, repair, Rust diagnostics, security, and update flows. That means the project is already moving toward a complete “repo control plane,” not just a scanner. ([GitHub][3])

The audit engine also has the right shape: it inventories the repo, builds an audit context, computes dimensions through analyzers, applies hard caps and policy, attaches UX/security/boundary evidence, builds findings, produces an ordered agent fix queue, and emits report metadata including standard version, auditor version, schema version, paper edition, target stack, fingerprints, proof receipts, and score history. That is exactly the architecture a standard needs. ([GitHub][4])

The rule system is one of the best parts of the implementation. `RuleSpec` includes stable IDs, categories, lanes, owner hints, evidence kind, severity, repairability, repair risk, standard section, cap key, and confidence policy. That is the beginning of a real compliance language. ([GitHub][5])

## The strategic shift: from “manifesto plus tool” to “protocol plus evidence”

To become tier-1, Jankurai should make three claims sharper:

**First:** vibe coding is not “using AI casually.” Vibe coding is **accepting unproven change**. That definition is already in the paper, but it should become the central recurring phrase.

**Second:** Jankurai is not primarily a stack preference. It is a **proof protocol for repositories**. A repo either exposes ownership, proof, generated-zone, security, repair, and version evidence, or it does not.

**Third:** the audit score must become a **time-series governance signal**. The world-changing idea is not “this repo scores 87.” It is “this PR dropped the repo from 91 to 84, introduced two high-risk findings, bypassed a proof lane, and increased repair half-life; therefore this is a vibe-coded regression even though tests are green.”

That last point should be the emotional and technical center of the paper.

## Detailed paper feedback by section

### Title and abstract

The title is electric, but it should be paired with a more formal subtitle for standards audiences. I would keep “Humans Were the Bug” as the provocative banner, but add a standards-friendly phrase such as:

**“Jankurai: A Conformance Standard for Trustworthy Merge in Agent-Native Repositories.”**

The abstract is already strong. It should add a crisp contribution list:

1. A threat model for generated-code failure.
2. A conformance standard with stable rule IDs and proof lanes.
3. A rolling audit score that detects centerline drift over time.
4. A reference implementation and reference architecture profile.

Right now the abstract reaches the stack recommendation quickly. I would delay that slightly. Lead with the universal protocol, then introduce the Rust/TypeScript/Postgres stack as the highest-scoring default profile under the rubric. That will make the paper feel less like an opinionated stack essay and more like a standard that can win broad adoption.

Add one unforgettable line:

> A green test run proves a command passed; a Jankurai audit asks whether the right command ran, for the right owner, against the right boundary, with durable repair evidence.

That sentence captures the difference between CI and Jankurai.

### I. New Bottleneck: Trustworthy Merge

This is one of the strongest sections. The claim that old repositories were organized around human memory, patience, taste, and slow review is compelling. The section should sharpen the contrast:

Old bottleneck: “Can we produce code?”
New bottleneck: “Can we safely merge code we did not manually author?”
Final bottleneck: “Can the repo remember enough to constrain the next agent?”

I would add a short “trustworthy merge equation”:

```
Trustworthy merge =
  owned change
+ legal edit surface
+ routed proof lane
+ security evidence
+ generated-boundary integrity
+ repair receipt
+ score non-regression
```

This section should also introduce the rolling score early. Say that every PR should answer:

“Did this change improve, preserve, or degrade the repository’s ability to verify future changes?”

That question turns Jankurai from a scanner into a governance model.

### II. Definitions and threat model

The definitions are excellent. “Vibe coding,” “proof lane,” “generated zone,” “repair receipt,” and “centerline drift” are the vocabulary that can make this project spread.

To elevate the section, make the threat model more formal. Add:

**Assets:** source code, contracts, secrets, customer data, database truth, release artifacts, agent instructions, CI credentials, generated outputs.

**Trust boundaries:** root policy, local path policy, untrusted issues/docs/web content, generated code, external packages, CI runners, browser automation, model/tool output.

**Adversaries and failure sources:** malicious prompt injection, confused agents, careless humans, stale docs, dependency compromise, weak tests, overbroad tools, and accidental generated-zone edits.

**Non-goals:** Jankurai does not prove full program correctness, replace human design judgment, or guarantee that a secure dependency stays secure forever. It proves that the repository exposes enough structure and evidence for controlled merge.

The threat table on page 3 is valuable. I would add a column called **“rolling-score signal.”** For example:

| Threat             | Rolling-score signal                                           |
| ------------------ | -------------------------------------------------------------- |
| False plausibility | new high HLT-008 findings, proof gap, semantic test absence    |
| Tool overreach     | changed-path scope expansion, generated-zone touch             |
| Secret exposure    | HLT-010 count spike, missing scanner artifact                  |
| False-green tests  | proof lane mismatch, no negative assertion, low proof coverage |

That addition directly ties the threat model to the project’s unique audit value.

### III. Languages as bottleneck compression

This is intellectually interesting and gives the paper ambition. The phrase “languages are bottleneck-compression systems” is memorable.

The risk is that this section can feel like a separate essay. To make it tier-1, tie every language-history point back to auditability. For example:

“Assembly compressed opcodes but not ownership. Python compressed exploration but not durable truth. TypeScript compressed browser-scale maintenance but not backend invariants. Rust compresses memory and ownership failures into compiler-visible structure.”

That keeps the section pointed at the paper’s thesis.

I would also reduce the historical tour slightly and move some detail to an appendix. The main body should relentlessly serve the standard-audit story.

### IV. Technical promise versus standard gravity

This section is very useful because it explains why technically elegant languages do not automatically become global defaults. The Julia example is fair and nuanced.

To strengthen it, define **standard gravity** more concretely:

```
Standard gravity =
  public corpus
+ boring deployment
+ security tooling
+ migration norms
+ hiring availability
+ observability examples
+ contract-generation support
+ CI reproducibility
+ agent familiarity
```

Then use that formula to explain why Jankurai is not anti-Julia, anti-Elixir, anti-Haskell, or anti-Rails. It is pro-auditability.

This matters for adoption. A global standard cannot sound like it is dunking on communities. It should say: “Every stack can compete, but the evidence must be machine-readable.”

### V. Vibe-Coding Fault Taxonomy

This is one of the paper’s most important sections. Table IV and Figure 1 are excellent because they turn a cultural insult into a structured audit surface. The TLR distribution on page 4 is especially useful: security, business truth, contracts/data, verification, entropy, context/setup, and repair become a risk map instead of a vibe. 

The section needs one big addition: **Rolling Score and Vibe Drift.**

A static RPN table tells readers what matters. A rolling score tells them when an agent is making the repo worse. I would add a subsection:

### V-B. Rolling Score as Vibe Drift Detection

Define something like:

```
Jankurai Rolling Score Event =
  repo_id
  commit_sha
  base_sha
  actor_type: human | agent | mixed | ci
  standard_version
  score_before
  score_after
  score_delta
  new_findings_by_rule
  resolved_findings_by_rule
  proof_lanes_required
  proof_lanes_satisfied
  generated_zone_touches
  security_artifacts_present
  repair_receipts_opened
  repair_receipts_closed
  exceptions_added
  exceptions_expired
```

Then define a “vibe mistake” operationally:

> A vibe mistake is a change that appears locally plausible while reducing verifiability, ownership clarity, proof adequacy, security posture, generated-boundary integrity, or repair locality.

This is huge. It means Jankurai can catch mistakes that ordinary tests miss.

I would also add trend metrics:

```
score_slope_7d
score_volatility_30d
new_high_findings_per_pr
repair_half_life
proof_lane_coverage
owner_map_coverage
exception_debt
recurring_rule_rate
```

A team can then say:

“Velocity is up, but HLT-008 false-green risk is recurring and repair half-life doubled. We are not becoming more productive; we are accumulating vibe debt.”

That is a profound management and engineering signal.

### VI. Jankurai Standard and Conformance

This is probably the core of the paper. The progressive ladder—agents, score, CI, full, ratchet—is smart because it avoids demanding a full rewrite. The HL0–HL5 conformance levels are also strong.

To make this globally credible, separate the **Jankurai Standard** from the **Jankurai CLI**:

* The standard defines schemas, rule IDs, conformance levels, evidence requirements, and semantics.
* The CLI is the reference implementation.
* Other implementations may conform if they pass the conformance suite.

That move is essential. Standards win when they are bigger than one implementation.

Also, define conformance artifacts with stable schema names:

```
jankurai.audit.v1.json
jankurai.finding.v1.json
jankurai.proof_receipt.v1.json
jankurai.repair_queue.v1.json
jankurai.exception.v1.json
jankurai.score_event.v1.json
jankurai.cap.v1.json
```

The paper says proof beats claims. The standard should embody that by requiring that every score below a floor produces a finding with owner, rule ID, lane, rerun command, and repair path.

One issue to fix: the paper’s Appendix A lists HLT-001 through HLT-019, but the code’s rule registry includes additional rules such as HLT-020-CI-HARDENING-GAP and HLT-021-DESTRUCTIVE-MIGRATION. That mismatch is exactly the kind of centerline drift Jankurai exists to prevent. Either add those rules to the paper/standard or mark them experimental in the registry. `jankurai versions` should fail if the paper, standard docs, schema, and rule registry disagree. ([GitHub][5])

### VII. A 100-Point Stack Rubric

The rubric is useful, but it is also where reviewers may start arguing with the paper instead of absorbing the standard. The scoring categories are thoughtful: correctness loop, security/memory safety, runtime, concurrency, contracts, simplicity, ecosystem, observability, data integrity, and velocity.

To make this tier-1, I would reframe the stack rubric as:

**“Reference Architecture Scoring Under Jankurai 0.4.”**

Not “the universal winner,” but “the highest-scoring default profile under this policy version.”

The difference matters. A global standard should invite communities to prove equivalent control planes. For example:

* Go can score highly if proof lanes, contracts, DB ownership, and security evidence are strong.
* .NET can score highly in regulated Microsoft environments.
* Rails can score strongly for constrained CRUD if generated contracts, DB constraints, and proof lanes are rigorous.
* Python can be excellent inside bounded AI/data cells but dangerous as unbounded durable product truth.

This framing keeps the Rust/TypeScript/Postgres recommendation sharp without making the standard feel exclusionary.

### VIII. Stack Ranking, Sensitivity, and Exceptions

The sensitivity discussion is good. I would make it more explicit and quantitative. Add a small table:

| Weight change       | Winner changes?       | Interpretation                         |
| ------------------- | --------------------- | -------------------------------------- |
| Ecosystem +5        | Go/.NET/JVM close gap | Brownfield/platform fit matters        |
| Memory safety +5    | Rust separates        | Security-critical systems favor Rust   |
| Product velocity +5 | TypeScript rises      | Needs stronger durable-truth partition |
| Realtime +5         | Elixir/Phoenix rises  | Specialist profile can beat default    |

This would show intellectual honesty and reduce “stack-war” pushback.

The Kafka/Rust streaming subsection is interesting, but it may be too detailed for the main argument. Consider moving much of it to an appendix or a “reference architecture exceptions” section. Keep the main body focused on the standard.

### IX. Default Winner Architecture

The ownership-cell table is excellent. This is where the paper becomes operational. The strongest idea is:

> The filesystem should be a routing table.

That line deserves more prominence. It is beautiful and practical.

I would add a “before/after” example:

**Before Jankurai:**
Agent edits `apps/web/src/api.ts`, invents a DTO, adds SQL in a route handler, updates a generated client, and passes a mock test.

**After Jankurai:**
Agent is routed to `contracts/`, regenerates the client, updates Rust application logic, runs the contract lane, DB lane, and web lane, then emits repair receipts.

That example would make the architecture click instantly.

### X. Agent Repository Controls and Tool Adapters

This section has a major insight: **token minimization is repository design, not prompt magic.** That should become a headline concept.

Add a concrete metric:

```
Agent Context Budget =
  root_router_tokens
+ owner_context_tokens
+ proof_lane_tokens
+ relevant_contract_tokens
+ recent_repair_receipt_tokens
+ filtered_error_output_tokens
```

Then define success:

* The agent receives the smallest sufficient context.
* It does not read duplicate architecture docs.
* It does not see stale policy mirrors.
* It receives exact proof commands.
* Command output is filtered to actionable evidence, with a raw-output escape hatch.

This can become one of Jankurai’s most compelling differentiators. Most AI coding tools try to stuff more context into the model. Jankurai can say: **the right repo shape makes less context more correct.**

### XI. Continuous Proof, CI, and Test Explosion

This section correctly says tests must be routed by proof question, not author convenience. The paper should go one step further: define **proof adequacy**.

A test passing is not enough. Jankurai should ask:

* Was the proof lane mapped to the changed path?
* Did the lane actually run?
* Did it produce parseable evidence?
* Did it cover the changed contract, migration, UI surface, or security boundary?
* Did it include a negative case where appropriate?
* Did the repair receipt close the original finding fingerprint?

This is how Jankurai catches false-green tests.

I would also add “failure seeding” or “mutation packets” as a future direction. For a changed authorization path, Jankurai should be able to require at least one negative authorization test. For a generated contract change, it should require a drift check. For a UI change, it should require rendered artifacts. For a DB migration, it should require migration safety proof.

### XII. Rendered UX and Browser-Step QA

This section is a gem. It makes Jankurai broader than backend correctness. The browser is absolutely a runtime, and AI-generated UI often fails visually while passing typecheck.

The paper already identifies the right layers: design-system constraints, Storybook states, Playwright screenshots, geometry rules, accessibility rules, visual regression, AI/CV triage, and human labels. The repo also contains a `packages/ux-qa` package with Playwright/TypeScript tooling, so this is not just theory. ([GitHub][6])

I would make the UX proof model crisp:

```
Rendered UX Proof =
  semantic proof: roles, labels, keyboard paths, ARIA snapshot
+ geometric proof: bounding boxes, clipping, overlap, target size
+ visual proof: screenshot, diff, baseline governance
+ human label: intentional, defect, false positive, acceptable variance
```

Also add a “UX repair receipt” example. For example:

```
Rule: HLT-013-RENDERED-UX-GAP
Surface: checkout payment form
Viewport: 375x812
Failure: submit button overlapped by sticky footer
Artifacts: screenshot, DOM geometry JSON, Playwright trace
Repair: adjusted layout token and safe-area padding
Proof lane: just ux-qa checkout
Result: accepted
```

This will make the UX section concrete and memorable.

### XIII. Security, Supply Chain, and Permissions

The section is right that agentic coding expands the write surface. I would make the permission model more explicit.

Add a permission state machine:

```
read_repo -> propose_diff -> edit_owned_paths -> run_lane -> request_privilege -> privileged_action -> receipt
```

High-risk actions should require explicit leases:

* adding dependencies
* editing generated zones
* touching secrets/env files
* running migrations
* invoking browser automation against privileged surfaces
* writing release artifacts
* network access
* destructive shell commands

The key artifact should be a **denied-action receipt** as well as an approval receipt. If an agent attempts to edit a generated client directly, Jankurai should record:

“Denied HLT-002. Legal repair path: edit `contracts/openapi/api.yaml`, run regeneration command, attach drift proof.”

That turns blocked behavior into training data.

### XIV. Exceptions, Observability, and Repair Receipts

This section should move earlier or be promoted. Repair receipts are not an implementation detail. They are the memory system of the repo.

The paper should define the repair receipt as a first-class protocol object:

```
RepairReceipt {
  receipt_id
  rule_id
  finding_fingerprint
  owner
  changed_paths
  allowed_scope
  proof_lane
  command_fingerprint
  exit_code
  artifact_digests
  score_before
  score_after
  standard_version
  auditor_version
  schema_version
  result
  next_repair
}
```

The most important field is `finding_fingerprint`. It lets Jankurai distinguish between “same issue reappeared” and “similar rule, different location.” That enables repair half-life, recurrence rate, and trend analysis.

### XV. Migration, Versioning, and Governance

This section is practical and avoids rewrite fantasy. Good. To make it standard-grade, add governance machinery:

* Jankurai Enhancement Proposals, perhaps `JEP-0001`, `JEP-0002`, etc.
* A conformance test suite with intentionally flawed fixture repos.
* A public schema registry.
* Compatibility rules for breaking schema changes.
* A badge format.
* A score-report signing/provenance story.
* A vendor-neutral standard committee path if adoption grows.

A global standard needs more than a repo. It needs a way for other tools to implement the standard without asking permission.

### XVI. Limitations and Research Agenda

This section is appropriately humble. I would make it sharper with falsifiable hypotheses:

* H1: Repos with owner/proof maps reduce mean time to repair agent-introduced defects.
* H2: Rolling score regression predicts escaped defects better than raw test failure rate.
* H3: Generated-zone enforcement reduces contract drift incidents.
* H4: Rendered UX proof reduces post-merge visual regressions.
* H5: Context packs reduce token use while improving first-pass repair success.

Then define metrics:

* escaped defects per 100 PRs
* review time
* repair time
* proof-lane coverage
* recurring rule rate
* repair half-life
* score volatility
* score regression frequency
* agent context tokens per accepted repair

That would make the research agenda feel fundable and publishable.

### XVII. Conclusion

The conclusion is strong, but I would end with one sentence that can become the project’s rallying cry:

> The repo is the reviewer of first resort.

Or:

> No proof, no merge; no receipt, no learning.

Those are memorable enough to carry the standard.

## Code feedback

The implementation is already much deeper than I expected. The CLI crate declares a broad command surface and dependencies for parsing, walking, scanning, serialization, hashing, semver, TOML, Rust syntax parsing, and HTTP calls. ([GitHub][7])

The source layout is sensible: audit, boundaries, command modules, init, reporting, rendering, validation, versions, UI helpers, and score history are separated at the module level. ([GitHub][8])

The analyzer system is promising. There are analyzers for ownership, contracts, proof, security, shape, data, observability, context, Python, speed, and UX readiness. The UX analyzer checks for Storybook, Playwright visual proof, visual review, accessibility, layout stability, mocks, design tokens, geometry runtime, and artifact-backed UX receipts. ([GitHub][9])

The proof analyzer rewards one-command setup, fast lanes, test runners, CI workflows, proof/test routing maps, Jankurai CI lanes, e2e tooling, rendered UX QA, geometry runtime, Rust property tests, and integration tests. That aligns well with the paper’s proof-lane doctrine. ([GitHub][10])

The security analyzer looks for lockfiles, secret/dependency scan tooling, SBOM/provenance, workflow linting, security lanes, CI audit lanes, unsafe tracking, cargo/npm audits, and gitleaks-style evidence. That is a good foundation. ([GitHub][11])

The shape analyzer is practical: it penalizes huge files, duplicate blocks, TODO/fallback/future-hostile terms, weak names, and domain I/O patterns. That supports the paper’s claim that agents over-edit when boundaries are implicit. ([GitHub][12])

### Code improvements I would prioritize

**1. Make rule metadata external and schema-validated.**
Right now, the rule registry lives in Rust code. That is fine for a reference implementation, but a standard needs a machine-readable rule registry:

```
schemas/jankurai-rule.schema.json
rules/hlt-001-dead-marker.toml
rules/hlt-002-generated-mutation.toml
...
```

Rust can compile these into the binary, but the source of truth should be inspectable, versioned, and testable against the paper.

**2. Add a “standard drift” self-test.**
The project should dogfood itself ruthlessly. If the paper, docs, schemas, rule registry, CLI constants, and generated artifacts disagree, `jankurai versions` should fail hard. The HLT-020/021 mismatch is the perfect example to turn into a showcase.

**3. Separate “tool exists” from “proof ran.”**
Some analyzers currently appear to score the presence of tooling or files. That is useful for advisory mode, but standard/ratchet/release modes should require receipts. For example:

* `playwright` exists is readiness.
* `playwright ran on this changed surface and produced trace/screenshot artifacts` is proof.

Same for security scanners, SBOMs, generated contracts, DB migrations, and accessibility checks.

**4. Strengthen unsafe-code handling.**
Rust `unsafe` should not earn credit merely because it is present or “appears tracked.” It should require a specific unsafe ledger entry, owner, invariant, justification, and proof lane. Otherwise it should be a finding.

**5. Make changed-path impact analysis the heart of audit.**
The audit already accepts changed paths and scopes inventory, but the next leap is impact analysis:

```
changed path -> owner -> generated zone? -> public boundary? -> durable truth? -> security-sensitive? -> required proof lanes -> required artifacts
```

This should produce an `ImpactReport` before scoring.

**6. Make `main.rs` thinner over time.**
The CLI surface is ambitious. To keep agent repair easy, move toward a declarative command registry:

```
CommandSpec {
  name
  risk
  required_inputs
  output_schemas
  allowed_modes
  proof_artifacts
}
```

Then CLI docs, shell completions, agent instructions, and tests can be generated from the same registry.

**7. Promote repair eligibility into the UX.**
The rule registry already distinguishes repairability and repair risk. That should drive the agent queue directly:

* safe autofix
* suggested patch
* human approval required
* never auto-repair
* security rotation required
* destructive migration requires human release approval

This will make agents faster without letting them become reckless.

## The big upgrade: Jankurai ProofMesh, a “vibe firewall” for PRs

The profound next-level upgrade I would build is **Jankurai ProofMesh**.

It would unify audit, changed-path impact, proof planning, evidence capture, context minimization, repair queues, and rolling score into one local control loop.

The product promise:

> Before an agent edits, Jankurai gives it the smallest correct context.
> While it edits, Jankurai detects illegal drift.
> Before merge, Jankurai proves the right lanes ran.
> After repair, Jankurai records what happened so the next agent learns.

### What makes this world-class

The current project already has pieces of this: audit, proof, proof verification, context packs, repair planning, score history, CI, security, UX QA, and publication evidence. ProofMesh would make them feel like one coherent machine. ([GitHub][13])

The killer feature is that Jankurai would stop asking, “Did tests pass?” and start asking:

“Given what changed, did the repo receive the minimum sufficient proof to merge safely?”

That is the cure to vibe coding.

### Example: catching a vibe-coded PR

Suppose an agent changes checkout behavior. The ordinary CI run passes because unit tests are green. Jankurai sees:

* `apps/web` changed: requires rendered UX proof.
* `contracts` generated client changed directly: HLT-002.
* payment flow changed: requires business invariant proof.
* no negative authorization/idempotency test added: HLT-008.
* score drops from 92 to 84.
* repair queue opens three items.

The merge is blocked even though the patch “looks good.”

Then the agent receives a context pack:

* edit `contracts/openapi/api.yaml`, not generated client
* run `just contract`
* add idempotency replay test
* run `just web-ux checkout`
* attach screenshots, trace, and proof receipt

After repair, Jankurai records:

* HLT-002 closed
* HLT-008 closed
* HLT-013 closed
* score returns to 92
* repair half-life: 41 minutes
* proof receipts stored

That is the demo that can shock people: **green tests are not enough; proof must match the risk.**

## Rolling score: the standard audit story to emphasize

A rolling score should be more than `repo-score.json` over time. It should become the main dashboard.

### The score should answer five questions

1. **Is the repo conformant today?**
2. **Did this PR improve or degrade conformance?**
3. **Which rule IDs are recurring?**
4. **How long do repairs stay open?**
5. **Are agents becoming more precise or creating more exception debt?**

### Suggested top-level metrics

```
current_score
score_delta_from_base
score_slope_7d
score_volatility_30d
new_high_findings
resolved_high_findings
proof_lane_coverage
owner_map_coverage
generated_zone_integrity
security_artifact_coverage
ux_artifact_coverage
repair_half_life
exception_debt
recurring_rule_rate
```

### Suggested merge gates

Block when:

* score drops below conformance floor
* score regression exceeds threshold
* any new critical finding appears
* any new high finding appears in standard mode
* changed path has no owner
* changed path has no proof lane
* generated zone is hand-edited
* required proof artifact is absent
* repair receipt is malformed
* exception is expired
* high-risk action lacks permission receipt

Warn when:

* score volatility rises
* medium findings recur
* repair half-life grows
* context pack exceeds token budget
* proof output is too noisy
* exception debt increases

This creates a new operating model: teams no longer debate “was this vibe coded?” They look at the score trend, findings, receipts, and proof coverage.

## Token minimization upgrade

Jankurai can own a very important idea: **less context, more correctness**.

Most AI tooling treats token budget as a model limitation. Jankurai should treat token budget as a repository-quality signal.

A new `jankurai context-pack` should produce task-specific packs:

```
ContextPack:
  root router excerpt
  relevant owner-map entries
  relevant proof lanes
  generated-zone rules
  touched contracts
  recent repair receipts for same rule/path
  allowed paths
  forbidden paths
  exact commands
  max output budget
```

Add metrics:

```
context_tokens_to_first_patch
context_tokens_to_accepted_repair
irrelevant_context_ratio
command_output_lines_to_fix
raw_output_escape_count
```

A repo that needs 80,000 tokens of context for a small safe change is not agent-native. A repo that can route the agent with 2,000 precise tokens is.

That is a massive story.

## Global standard path

To become a global standard, Jankurai should deliberately decouple **protocol** from **implementation**.

I would create:

* `JANKURAI_SPEC.md`
* `schemas/`
* `rules/`
* `conformance/fixtures/`
* `reference-implementation/`
* `JEPs/`
* `profiles/`

Profiles could include:

```
profiles/rust-ts-postgres.toml
profiles/go-ts-postgres.toml
profiles/dotnet-ts-postgres.toml
profiles/rails-postgres.toml
profiles/python-ai-service.toml
```

Each profile defines expected controls. That lets communities adopt Jankurai without feeling forced into one architecture.

The badge should be precise:

```
Jankurai HL3
Score: 91/100
30-day trend: +3
New high findings: 0
Proof coverage: 96%
Repair half-life: 1.4 days
Standard: 0.4.0
Schema: 1.2.0
```

That badge would be far more meaningful than “AI-ready.”

## Final recommendation

Jankurai should position itself as:

> **The standard audit and rolling proof ledger for agent-native repositories.**

The stack recommendation is valuable, but the universal invention is bigger: **stable rule IDs, proof lanes, repair receipts, generated-zone enforcement, context minimization, and rolling score non-regression.**

The strongest version of the story is:

> Vibe coding is unproven change.
> Jankurai makes unproven change visible, routable, repairable, and eventually unmergeable.
> The repo becomes the reviewer of first resort.
