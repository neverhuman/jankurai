This is genuinely exciting. The strongest version of Jankurai is **not** “an AI coding linter.” It is a **merge protocol for the agent era**: every repository declares its ownership map, proof lanes, generated zones, score policy, repair queue, exceptions, and evidence receipts, so generated code is never accepted merely because it looks right.

The paper already has the raw material for a world-class idea: **trustworthy merge** as the new bottleneck, **vibe coding** as an auditable fault class, **stable HLT rule IDs**, **conformance levels**, **repair receipts**, **rendered UX proof**, and a default architecture that tries to make invalid changes hard to express. The README and code support this direction: the project describes itself as a local, read-only-first “agent-native repository control-plane and audit CLI,” with owner maps, proof lanes, generated zones, security boundaries, repair queues, adoption plans, audit scores, and score history artifacts. ([GitHub][1])

The next leap is to make the **rolling standard audit** the hero. The shocking claim should become:

> **No proof, no merge. No route, no edit. No receipt, no trust.**

Jankurai should become the thing every serious team runs before accepting AI-generated code.

# 1. High-level verdict

The paper’s central frame is excellent: AI has made plausible code cheap, so the bottleneck moves from writing code to **proving code is safe to merge**. The abstract and opening sections already say this well: the repository must reject wrong generated code, localize the defect, prove the repair, preserve security posture, and teach the next agent what happened. 

The project becomes tier-1 when it shifts from **“here is a bold standard and stack recommendation”** to **“here is a measurable protocol that catches vibe mistakes across commits, pull requests, and time.”**

The paper should make three ideas impossible to miss:

1. **Vibe coding is not a style. It is an unmeasured merge risk.**
2. **The cure is not better prompting. The cure is a repository-level proof interface.**
3. **The rolling score is the control loop. It catches drift when agents make the repo more plausible but less provable.**

The code is already closer to this than the paper fully advertises. The data model has report fingerprints, input and policy fingerprints, manifest fingerprints, dirty worktree state, schema/version fields, proof receipts, UX/security/boundary artifacts, findings, caps, dimensions, and an agent fix queue. That is the skeleton of a serious evidence system, not just a scanner.

# 2. The single biggest narrative upgrade

Right now, the paper’s story is:

> AI coding changes the bottleneck. Jankurai defines agent-native engineering. Rust + TypeScript + PostgreSQL is the best default stack.

The world-shocking version is:

> AI coding created a new failure mode: **plausible but unproven software**. Jankurai turns every repository into a **verification interface**. Every PR receives a **rolling merge passport**: ownership, proof lane, generated-zone status, security posture, rendered UX evidence, score delta, exceptions, and repair receipts. The score is not a vanity grade; it is the repository’s immune system.

That is more universal, more defensible, and more standard-like.

The stack recommendation should remain, but the global-standard story should be stack-neutral first. Rust + TypeScript + PostgreSQL can be the **reference profile** or **gold profile**, not the thing that makes Jankurai feel exclusionary. A global standard needs Go, .NET, JVM, Python, Rails, Elixir, and legacy organizations to say: “This still applies to us.”

# 3. Paper feedback, section by section

## Title and front page

**What works:**
“Humans Were the Bug” is memorable. The cover image and samurai identity create a strong brand. The subtitle, “From Vibe Coding to Agent-Native Engineering,” is exactly the right conceptual move.

**Risk:**
The title can sound anti-human. Tier-1 reviewers and enterprise buyers may react defensively unless the paper clarifies immediately that the bug is **human-memory-first repository design**, not humans.

**Suggested opening clarification:**

> Humans were not the bug because humans wrote code. Humans were the bug because software repositories were designed around human memory, patience, taste, and tribal review. Agentic coding removes that throttle. The fix is not to remove humans; the fix is to make repositories legible, enforceable, and repairable by humans and agents together.

That idea appears in Section I, but it should be even more prominent near the abstract.

## Abstract

**What works:**
The abstract is strong. “The scarce act is trustworthy merge” is the paper’s best phrase. It correctly reframes the AI coding era around verification rather than generation.

**Upgrade:**
Add one sentence that makes the standard audit and rolling score central:

> Jankurai operationalizes this through a rolling repository score and merge passport: every change is routed to an owner, proof lane, generated-zone policy, security posture, evidence artifact, and repair queue before it can claim conformance.

Also consider adding a sentence that separates standard from stack:

> The standard is stack-profiled; the paper evaluates a default reference profile but the conformance model is independent of any single language.

That one sentence would dramatically increase adoption potential.

## I. New Bottleneck: Trustworthy Merge

**What works:**
This is the best conceptual section. The paper identifies the right bottleneck: agents can produce plausible code, but repos cannot reliably accept or reject it.

**Upgrade:**
Add a vivid failure story. For example:

> An agent edits checkout logic. The patch compiles. Unit tests pass. But the payment path lost idempotency, the generated client was hand-edited, the mobile checkout button is clipped, and the security lane never ran. A human reviewer sees plausible code; Jankurai sees HLT-008, HLT-002, HLT-013, and HLT-009.

That miniature story makes the standard feel inevitable.

**Add a “trustworthy merge equation”:**

> Trustworthy merge = owned path + legal edit surface + mapped proof lane + security posture + artifact evidence + repair receipt + score delta.

This should become the paper’s memorable formula.

## II. Definitions and Threat Model

**What works:**
The operational definitions are strong: vibe coding, agent-native engineering, proof lane, generated zone, repair receipt, and centerline drift. These are excellent primitives for a real standard. 

**Upgrade:**
Add a formal threat model table with:

| Element          | Definition                                                                                          |
| ---------------- | --------------------------------------------------------------------------------------------------- |
| Assets           | source code, secrets, contracts, migrations, generated artifacts, CI authority, user data           |
| Actors           | human developer, coding agent, malicious issue author, compromised package, overbroad tool          |
| Trust boundaries | root policy, local policy, docs, issue text, web content, generated output                          |
| Non-goals        | proving semantic correctness of all software, replacing human product judgment, mandating one stack |

The current threat table is good, but a tier-1 paper will benefit from this more formal framing.

## III. Languages as Bottleneck Compression

**What works:**
The concept is interesting and memorable. “Languages as bottleneck-compression systems” is a fresh way to discuss programming language history.

**Risk:**
It can distract from the standard. Some readers may think the paper is a stack-war essay rather than a standards proposal.

**Upgrade:**
Tie every language observation back to auditability:

> In the agent era, the primary language question is not “Can an agent write it?” but “Can the repository reject bad edits quickly and locally?”

Add a small table with audit-facing language properties:

| Property            | Why agents need it                |
| ------------------- | --------------------------------- |
| Rejection speed     | bad edits fail before review      |
| Proof locality      | changed path maps to narrow lane  |
| Contract generation | boundary drift becomes detectable |
| Error structure     | repair agents get stable signals  |
| Ownership clarity   | agent does not infer tribal rules |

## IV. Technical Promise Versus Standard Gravity

**What works:**
This section correctly separates technical elegance from default-standard readiness.

**Risk:**
The examples may feel like language criticism. Julia, Elm, Haskell, Scala, Elixir, and others have strong communities. The paper should avoid sounding dismissive.

**Upgrade:**
Make the point about **standard gravity**, not language ranking:

> The standard does not ask whether a language is beautiful. It asks whether ordinary repositories can expose ownership, proof, contracts, security, repair, and examples with low ceremony.

That lets you keep the argument without alienating niche ecosystems.

## V. Vibe-Coding Fault Taxonomy

This is one of the strongest sections. The RPN table and TLR buckets are the heart of the paper. The figure on page 4, where Security takes 32% of policy RPN share, Business truth takes 20%, and Contracts/data plus Verification each take 13%, gives the standard a visual risk model. 

**Upgrade:**
Make this section more empirical and more productized.

Add three columns to the taxonomy:

| Field                  | Why it matters                                    |
| ---------------------- | ------------------------------------------------- |
| Detector maturity      | implemented, partial, evidence-driven, research   |
| False-positive risk    | low, medium, high                                 |

The paper already says RPN is policy weight, not empirical frequency. That honesty is good. The next step is to define how it will be calibrated:

> Future Jankurai releases calibrate RPN using escaped defects, repair time, false-positive rate, score regression, proof-lane misses, and reviewer override labels.

This makes the standard feel scientifically alive.

The progressive adoption levels, conformance levels, stable rule IDs, and agent fix queue are excellent. The paper’s “proof beats claims” sentence is exactly right.

Add a normative conformance matrix. For example:

# 4. Make the rolling score the emotional center

The project’s future depends on people understanding that the score is not a grade. It is a **repository control loop**.

The paper should introduce a phrase like:

> **The rolling score catches vibe mistakes at the moment they become merge risk.**

A PR should show:

```text
Jankurai Rolling Score

Base score:      91
PR score:        84
Delta:           -7
Decision:        BLOCKED

New hard caps:
- HLT-013-RENDERED-UX-GAP: checkout UI changed without rendered proof
- HLT-004-UNMAPPED-PROOF: changed path has no proof lane
```

The rolling score should track at least:

| Field              | Meaning                               |
| ------------------ | ------------------------------------- |
| score_before       | accepted baseline                     |

| raw_score_delta    | un-capped scoring movement            |
| cap_delta          | new or removed score ceilings         |
| new_findings       | findings introduced by change         |
| resolved_findings  | findings repaired by change           |
| proof_lane_delta   | proof coverage gained/lost            |
| exception_delta    | new/expired/widened exceptions        |
| evidence_delta     | new/missing proof artifacts           |
| token_delta        | context needed by agents before/after |

# 5. Code feedback

## What is already excellent

The implementation is much more serious than a prototype manifesto.

The audit pipeline inventories the repo, scopes changed files, computes dimensions, applies caps, loads policy, attaches UX artifacts, builds findings, builds the agent fix queue, computes report decisions, captures git state, loads proof receipts, and fingerprints the report. 

The cap system is explicit and readable. It encodes maximum scores for missing root instructions, missing setup/fast lanes, missing security lanes, contract drift, Python product-truth spill, missing scans, no audit CI, generated-zone risk, wrong-layer DB access, rendered UX gaps, prompt injection, overbroad agency, secrets, false-green tests, destructive migrations, missing Rust proof, missing docs, and streaming runtime drift. 

The rule metadata is strong. Repair eligibility and repair risk are especially important because they let the future repair engine distinguish auto-safe cleanup from human-required security and data changes. 

The proof command design is also strong. `jankurai prove` routes changed paths through owner maps and test maps, plans runs, writes receipts, records digests, and rejects unsigned proof commands unless explicitly env-gated. That is exactly the direction needed for agent-safe automation.

## Main code gaps to close

### 1. Move from heuristic scanning to evidence-grade analysis

The current detector layer uses many string and regex patterns. That is appropriate for early coverage, but a global standard needs deeper analyzers.

Examples:

* TODO/fallback/future-hostile terms can create noisy findings.
* Prompt-injection patterns should be case-insensitive and context-aware.
* Wrong-layer DB access should use the boundary manifest more than hardcoded path assumptions.
* Dependency drift should compare manifest and lockfile diffs, not only scan for tooling presence.
* CI hardening should parse workflow YAML structurally, not line-match only.

The string scanners are useful as “smoke alarms.” The next tier is **typed evidence**.

### 2. Make score deltas first-class

The report model is rich, but the central user experience should be:

```bash
jankurai score diff . --base origin/main --head HEAD
```

Output:

```text
Score: 91 -> 84 (-7)
Decision: blocked
New caps: missing-rendered-ux-qa-lane
New findings: HLT-013, HLT-004
Fixed findings: HLT-001
Required lanes: web, fast
```

This is more actionable than a static score.

### 3. Fix rule/paper synchronization

The code has HLT-020 and HLT-021. The paper appendix currently presents the initial rule identifiers through HLT-019. Align them before the next release. This is exactly the kind of version drift Jankurai itself should catch.

Add a self-check:

```bash
jankurai versions --strict
jankurai rules export --json target/jankurai/rules.json
jankurai paper verify-rules
```

### 4. Improve secret scanning semantics

The internal secret detector skips large path groups, including docs, paper, reference, tips, and `crates/jankurai/`. That avoids false positives, but it could also hide real exposure. The better policy is:

* scan everything,
* classify examples/test fixtures separately,
* require explicit allowlist entries with owner and expiry,
* rely on external scanners like gitleaks where available,
* never auto-repair secrets.

### 5. Make “unsafe” scoring stricter

Do not award security credit for detecting `unsafe` in Rust. Award credit only when unsafe usage is tied to an explicit ledger:

```toml
[[unsafe]]
path = "crates/adapters/src/ffi.rs"
owner = "runtime"
reason = "FFI boundary to vendor library"
invariants = ["pointer lifetime checked", "no shared mutation"]
proof_lane = "ffi-safety"
expires = "2026-09-01"
```

### 6. Add calibration data to every rule

Each rule should eventually carry:

```json
{
  "rule_id": "HLT-013-RENDERED-UX-GAP",
  "detector_maturity": "partial",
  "false_positive_risk": "medium",
  "confidence_policy": "high",
  "calibration_source": "jankurai-bench-2026.05",
  "last_weight_review": "2026-05-02"
}
```

The existing `confidence_policy` field is a great start.

# 6. The profound upgrade: Jankurai Merge Passport

The next-level feature should be a **Jankurai Merge Passport**.

This would make Jankurai feel inevitable.

A merge passport is a single signed artifact that answers:

> Is this change owned, legal, proven, secure, observable, generated correctly, UI-verified, and score-safe?

Command:

```bash
jankurai passport . \
  --changed-from origin/main \
  --out target/jankurai/merge-passport.json \
  --md target/jankurai/merge-passport.md \
  --sarif target/jankurai/merge-passport.sarif
```

It should combine:

| Component               | Source                                          |
| ----------------------- | ----------------------------------------------- |
| changed paths           | git diff                                        |
| owners                  | `agent/owner-map.json`                          |
| proof lanes             | `agent/test-map.json`, `agent/proof-lanes.toml` |
| generated-zone legality | `agent/generated-zones.toml`                    |
| security posture        | security evidence artifact                      |
| UX proof                | UX QA artifact                                  |
| score delta             | base report vs head report                      |
| repair queue            | findings ordered by TLR/RPN                     |
| exceptions              | active exception manifests                      |
| receipts                | proof receipts and digests                      |
| decision                | pass, block, review, release-gate               |

Example Markdown:

```text
# Jankurai Merge Passport

Decision: BLOCKED
Score: 91 -> 84 (-7)
Conformance: HL3 claimed, HL2 observed

Changed paths:
- apps/web/src/checkout/CheckoutForm.tsx
  owner: web-checkout
  proof lane: web
  status: missing rendered UX artifact

Blocking findings:
- HLT-013-RENDERED-UX-GAP
  lane: web
  required proof: Playwright screenshot + ARIA snapshot + geometry report

Repair queue:
1. Add Storybook long-label/mobile state.
2. Run jankurai prove --changed apps/web/src/checkout/CheckoutForm.tsx
3. Attach target/jankurai/ux-qa.json
```

This is the artifact a CTO, reviewer, GitHub Action, coding agent, and compliance team can all understand.

## Why this takes the project to the next level

It changes Jankurai from “scanner” to **merge infrastructure**.

Scanners produce findings.
Jankurai should produce **merge eligibility**.

Linters say “this line is bad.”
Jankurai should say “this PR is not yet provable.”

Test runners say “tests passed.”
Jankurai should say “the right proof lanes ran for the changed paths.”

Agents say “I fixed it.”
Jankurai should say “the repair receipt proves what failed, what changed, and what evidence accepted the repair.”

That is a global-standard shape.

# 8. The third profound upgrade: Vibe Mistake Firewall

Add a PR-native firewall that blocks the most common AI-generated mistakes before review:

```bash
jankurai firewall . --changed-from origin/main
```

It should block when:

| Vibe mistake           | Firewall condition                                         |
| ---------------------- | ---------------------------------------------------------- |
| plausible unowned edit | changed path lacks owner                                   |
| false-green proof      | changed path lacks mapped proof lane                       |
| generated drift        | generated zone edited directly                             |
| hidden product truth   | durable logic moved into wrong layer                       |
| security regression    | auth/input/crypto/dependency changed without security lane |
| secret exposure        | secret-like value or transcript leak                       |
| UI plausibility        | web change lacks rendered artifact                         |
| migration danger       | destructive SQL lacks rollback/backfill/lock evidence      |
| agent overreach        | tool permissions exceed lane                               |
| prompt injection       | trusted policy changed by untrusted context                |

The code already has many of these caps and findings. The upgrade is to present them as a single memorable product surface: **the vibe mistake firewall**.
