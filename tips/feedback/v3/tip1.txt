I studied the uploaded paper and the GitHub project surface in depth. The project is already far beyond a manifesto: the repo presents Jankurai as a local, agent-native repository control plane that turns ownership maps, proof lanes, generated zones, security boundaries, and repair queues into readable files and machine-checkable artifacts; it explicitly says it is not a model, hosted AI service, or open-source AI replacement.  ([GitHub][1])

My highest-level verdict: **the idea is powerful enough to become a real standard, but the paper should stop framing Jankurai mainly as a stack recommendation and instead frame it as the “trustworthy merge protocol” for the agent era.** The Rust/TypeScript/PostgreSQL stack should remain the flagship reference profile, but the global standard should be the audit interface: owner maps, proof maps, generated-zone policy, evidence receipts, repair queues, and a rolling score that catches drift before a team vibes itself into production debt.

## The core story to elevate

The current paper’s strongest sentence is essentially this: **the scarce act is no longer writing plausible code; the scarce act is trustworthy merge.** That should become the center of gravity. Everything else should orbit it.

The world does not need another “AI coding best practices” guide. It needs a shared answer to this question:

> When an agent changes a repository, what machine-readable evidence proves that the change touched the right owner, respected generated boundaries, ran the right proof lane, preserved security posture, and left repair knowledge behind?

That is the global-standard opening. Jankurai should be positioned as:

**Jankurai = a conformance protocol for trustworthy merge.**

The “cure to vibe coding” is not merely “better prompts.” Your own paper already says token minimization is repository design, not a prompting trick, and the repo has the artifacts to support that idea: `owner-map.json`, `test-map.json`, `proof-lanes.toml`, `generated-zones.toml`, `standard-version.toml`, audit reports, CI modes, and repair queues. ([GitHub][2])

The next leap is to make the **rolling score** the iconic product object. A single audit tells you “what is wrong now.” A rolling score tells you “whether this repo is becoming safer or more vibey over time.” That is the thing teams, agents, reviewers, auditors, vendors, and future standards bodies can align around.

# 1. Paper feedback, section by section

## Title and front matter

**Current strength:** “Humans Were the Bug” is unforgettable. It has punch, attitude, and viral energy. The abstract is also technically strong: it says the bottleneck has moved from code production to trustworthy merge, then names concrete controls: conformance levels, rule IDs, generated-zone policy, agent control planes, audit output, CI modes, repair queues, versioned artifacts, and a threat model. 

**Risk:** For a tier-1 audience, the title can sound like rhetoric before the reader discovers the precision. Some reviewers will bounce off the provocation even though the paper is much more rigorous than the title suggests.

**Upgrade:** Use a title/subtitle split that gives both academic credibility and viral force:

> **Trustworthy Merge: A Standard Interface for Agent-Native Engineering**
> **Humans Were the Bug: From Vibe Coding to Verifiable Repositories**

That keeps the memorable phrase but leads with the new technical primitive. “Trustworthy Merge” deserves to become the branded idea.

**Abstract upgrade:** Add one sentence that explicitly introduces rolling score:

> “Jankurai treats repository health as a continuously measured control variable: every change updates a rolling score, routed findings, proof receipts, and repair memory, allowing teams to detect recurring vibe mistakes rather than merely pass a one-time check.”

That one sentence would make the paper feel less like a static audit rubric and more like a living standard.

---

## Section I — New Bottleneck: Trustworthy Merge

This is one of the best sections. The claim that humans were the bug because repositories were organized around human memory, patience, taste, and slow review is sharp. The paper correctly argues that the answer is not to slow agents down until old review processes survive, but to redesign the repo as the control surface that constrains, proves, and repairs generated code. 

**What to strengthen:** Add a concrete opening failure story before the abstract claims. Something like:

> “An agent edits a checkout flow. The code compiles. Unit tests pass. But it hand-edits a generated client, skips the payment idempotency lane, weakens an authorization boundary, and no reviewer notices because the diff looks clean. This is not a model failure alone. It is a repository-interface failure.”

That would make “vibe coding” visceral. Then Jankurai becomes the answer to a real failure mode, not just a philosophy.

**Tier-1 framing:** Define the new systems boundary:

> “In agent-native engineering, the repository is no longer passive storage. It is the executable specification of permissible change.”

That sentence can anchor the whole paper.

---

## Section II — Definitions and Threat Model

The definitions are strong and operational: vibe coding, agent-native engineering, proof lane, generated zone, repair receipt, and centerline drift. Table I is especially useful because it turns AI coding risks into control surfaces: false plausibility, hostile context, tool overreach, secret exposure, dependency poisoning, and false-green tests. 

**What to strengthen:** Make **centerline drift** more important. Right now it is a good term, but it can become one of the paper’s signature concepts.

Suggested definition upgrade:

> “Centerline drift is the measured divergence between the repository’s claimed engineering standard and its observed merge behavior over time.”

Then immediately connect it to rolling score:

> “A static score detects nonconformance. A rolling score detects centerline drift.”

That is a huge story improvement. The paper should make “rolling score catches vibe mistakes” the operational sequel to the threat model.

**Add a source-trust hierarchy diagram.** Table I mentions hostile context, but the standard would become more concrete with a tiny diagram:

1. Standard/version manifest
2. Root agent router
3. Owner/test/generated-zone maps
4. Local owner rules
5. Trusted repo docs
6. Issue text, comments, webpages, pasted prompts, tool output
7. Model suggestions

Then state: lower layers may inform work, but must not modify policy.

---

## Section III — Languages as Bottleneck Compression

This is a memorable and genuinely useful lens. The paper explains language history as a sequence of bottleneck-compression packages rather than just abstraction history. That is a great way to make stack choice feel less like taste and more like systems design. 

**What to strengthen:** Tie the compression argument back to the audit interface earlier and more explicitly.

The killer claim is:

> “In the agent era, syntax compression matters less than rejection compression.”

That is the phrase I would add. Rust, TypeScript, PostgreSQL, generated contracts, and bounded Python are not chosen because they are fashionable; they compress different classes of rejection:

| Layer               | What it compresses                                      |
| ------------------- | ------------------------------------------------------- |
| Rust                | memory, ownership, invalid state, error locality        |
| TypeScript          | browser-product feedback and UI contract shape          |
| PostgreSQL          | durable truth, constraints, transactional invariants    |
| Generated contracts | boundary drift                                          |
| Bounded Python      | model/data experimentation without product-truth sprawl |
| Jankurai            | ownership, proof routing, repair memory, audit evidence |

That table would make the stack argument feel more inevitable.

---

## Section IV — Technical Promise Versus Standard Gravity

This section is wise because it separates technical excellence from standard-default status. The Julia discussion is respectful and useful. Table III makes the adoption-gravity argument clear across Julia, D, Nim, Crystal, Elm, Reason/OCaml, F#, Clojure, Haskell, Scala, and Elixir/Phoenix. 

**Risk:** This section can be misread as “Jankurai is a Rust-stack manifesto that ranks languages.” That may narrow adoption.

**Upgrade:** Introduce the phrase **reference profile** here.

> “Jankurai Core is stack-neutral. The Rust/TypeScript/PostgreSQL architecture is the flagship reference profile for durable product systems.”

This allows Go, .NET, JVM, Python-heavy data platforms, and legacy teams to adopt the audit standard without feeling they must first accept the entire stack thesis. The standard becomes globally alignable; the stack becomes one high-scoring profile.

---

## Section V — Vibe-Coding Fault Taxonomy

This is one of the paper’s strongest contributions. It turns “vibe coding” from an insult into an auditable fault class. The RPN framing, TLR buckets, and ranked Table IV are excellent. The page 4 chart is also strong because it gives a visual policy model: security receives 32% of RPN share, business truth 20%, contracts/data and verification 13% each, entropy 10%, context/setup 9%, and repair 3%. 

**What to strengthen:** Make the taxonomy dynamic. Right now it is a ranked table. It should become a **learning loop**:

1. Agent creates plausible patch.
2. Jankurai detects rule violation.
3. Finding gets stable fingerprint.
4. Proof lane or repair fails.
5. Repair receipt records accepted fix.
6. Future recurrence is recognized and escalated.
7. Rolling score reflects whether the repo is learning.

That is the cure to vibe coding: not just detecting mistakes, but **remembering classes of mistakes**.

**Add one new figure:** “The Vibe Mistake Lifecycle.”

```
plausible diff
   ↓
owner/proof/generated-zone audit
   ↓
finding fingerprint
   ↓
repair queue
   ↓
proof receipt
   ↓
rolling score + recurrence memory
   ↓
next agent gets narrower context
```

That figure would make the paper’s contribution emotionally obvious.

**Also:** Put the RPN model into a versioned artifact, for example `agent/risk-model.toml`. The paper says RPN is policy rather than empirical incident frequency. Good. Now make that reproducible in the repo.

## Section VI — Jankurai Standard and Conformance

This section is the heart of the standard. The install ladder — `agents → score → ci → full → ratchet` — is exactly the right adoption path. The HL0–HL5 conformance levels are also strong because they make adoption progressive rather than binary. 

The repo already aligns with this idea: it has agent manifests, proof lanes, generated zones, standard version files, and command surfaces for audit, init, doctor, proof, repair, migration, security, certification, governance, publishing, and UX. ([GitHub][3])

**What to strengthen:** Split the standard into three layers:

1. **Jankurai Core Spec**
   Required artifacts, schemas, finding fields, proof receipts, conformance levels, rolling score semantics.

2. **Rule Packs**
   HLT rules, severity defaults, repair recipes, detector status, evidence requirements.

3. **Reference Profiles**
   `rust-ts-postgres`, `go-service-postgres`, `dotnet-enterprise`, `legacy-jvm`, `python-ai-boundary`, etc.

This is essential for global adoption. A standard that starts by telling every org to pick one stack may be admired. A standard that lets every org measure trustworthy merge can become infrastructure.

**Add anti-gaming language.** A score standard will be gamed unless the paper directly addresses it. Add:

> “A repository MUST NOT receive credit for a proof lane unless the lane produced a receipt tied to the changed paths, command identity, exit status, artifact hashes, and timestamp.”

This turns conformance from “files exist” into “evidence exists.”

## Section VII — A 100-Point Stack Rubric

The rubric is well-structured. It correctly weights proof, security, runtime behavior, concurrency, boundaries, and review surface above raw velocity. The “hard filters before points” idea is excellent: no deterministic proof lane, no security lane, no generated contract discipline, or no credible data boundary means the repo cannot claim agent-native maturity. 

**Risk:** A 100-point stack rubric may compete with the more important 100-point repository health score. Readers may ask, “Is Jankurai about stack ranking or repository conformance?”

**Upgrade:** Rename this to **Reference Profile Rubric** and clearly separate it from the **Repository Audit Score**.

Suggested distinction:

| Score                           | Measures                                        | Changes how often    |
| ------------------------------- | ----------------------------------------------- | -------------------- |
| Agent-Native Stack Score        | Suitability of a stack/profile                  | Rarely               |
| Jankurai Repository Score       | Current repo conformance                        | Every audit          |
| Rolling Trustworthy-Merge Score | Whether merge behavior is improving or drifting | Every PR / over time |

This would prevent confusion and make rolling score the operational star.

## Section VIII — Stack Ranking, Sensitivity, and Exceptions

The sensitivity discussion is important. The top-five stack ranking is clear, and the paper already says the result is a default recommendation, not metaphysical law. The chart on page 8 is visually useful: Rust leads at 94, then Go at 90, .NET at 89, TypeScript product plane plus Rust/Go compute cells at 88, and JVM at 87. 

**What to strengthen:** Move some of the ranking detail later or into a profile appendix. The global-standard pitch should come first:

> “Any stack may claim Jankurai Core conformance if it exposes machine-readable ownership, proof routing, generated boundaries, security evidence, repair receipts, and rolling score history.”

Then say:

> “The paper’s flagship profile scores Rust/TypeScript/PostgreSQL highest for durable product engineering.”

That ordering makes Jankurai bigger than one architecture.

**Kafka/Rust streaming section:** The “Kafka today, Rust tomorrow” treatment is thoughtful. Keep it. But frame it as an example of exception governance:

> “Jankurai does not ban brownfield gravity. It forces brownfield gravity to become explicit, adapted, observable, and measured.”

That sentence helps avoid dogma.

## Section IX — Default Winner Architecture

The filesystem-as-routing-table idea is excellent. The page 8 ownership spine and agent-control spine are among the most important visuals in the paper. They show the project’s real novelty: an agent should not infer where to edit; the repo should route it.

Before:

After:

| Changed path               | Owner          | Required lane   | Forbidden move                      |
| -------------------------- | -------------- | --------------- | ----------------------------------- |
| `crates/domain/payment.rs` | domain         | fast + property | direct DB                           |
| `db/migrations/*`          | db             | db lane         | destructive migration without proof |

## Section XI — Continuous Proof, CI, and Test Explosion

This section lands the central insight: more tests are not enough if agents cannot route them. A repo with thousands of tests and no `test-map.json` still asks agents to guess. That is a beautiful line of argument. 

The current CLI already supports audit output in JSON/Markdown and additional integration surfaces such as SARIF, JUnit, GitHub summaries, repair queue JSONL, proof receipts, baselines, changed paths, and CI modes. ([GitHub][5])

**What to strengthen:** Introduce **proof freshness**.

A proof lane should not only exist; it should be fresh relative to the change. Add fields:

```json
{
  "proof_freshness": {
    "changed_paths": 14,
    "paths_with_valid_receipts": 12,
    "stale_receipts": 2,
    "freshness_score": 0.86
  }
}
```

This catches a common vibe mistake: “the right tests exist, but they were not run for this patch.”

**Add proof receipts as first-class standard objects.** The paper already has repair receipts. Add symmetrical **proof receipts**:

| Field                     | Purpose                    |
| ------------------------- | -------------------------- |
| `lane_id`                 | Which proof lane ran       |
| `command_id`              | Stable command identity    |
| `changed_paths`           | What the proof covered     |
| `exit_code`               | Accepted/rejected          |
| `artifact_hashes`         | Evidence integrity         |
| `started_at` / `ended_at` | Freshness                  |
| `runner`                  | local/CI/agent             |
| `stdout_excerpt`          | Useful filtered output     |
| `raw_output_ref`          | Escape hatch for full logs |

## Section XIII — Security, Supply Chain, and Permissions

The section is right: agentic coding expands the write surface, so security has to be structural. Table XVII is practical: prompt injection, secrets, dependency sprawl, sandbox escape/tool overreach, unsafe code/FFI, and false-green security all map to repair evidence. 

The repo already has a security lane script that checks for tools such as gitleaks, cargo audit, npm audit, syft, and zizmor, with required/advisory behavior. ([GitHub][8])

**What to strengthen:** Add **capability leases**.

An agent should not merely have “permission” or “no permission.” It should receive a short-lived, path-scoped, command-scoped lease:

```toml

task_id = "JKR-2026-05-0042"
allowed_paths = ["contracts/openapi/**", "apps/web/src/generated/**"]
allowed_commands = ["just contract", "just web"]
forbidden = ["db/migrations/**", "secrets/**", "release/**"]
expires_at = "2026-05-04T18:00:00Z"
requires_receipt = true
```

This is the right abstraction for “agents go faster, with fewer mistakes.”

# 2. The rolling score should become the product

Right now, the audit score is powerful. The next-level version is a **rolling trustworthy-merge score**.

A one-time score can be gamed. A rolling score reveals behavior.

Suggested model:

```json
{
  "latest_score": 88,
  "rolling_score_7d": 91,
  "rolling_score_30d": 86,
  "score_delta_pr": -3,
  "centerline_drift": 7,
  "new_high_findings": 2,
  "reopened_fingerprints": 1,
  "stale_exceptions": 3,
  "proof_freshness": 0.82,
  "context_budget_score": 0.76,
  "vibe_recurrence_index": 0.18
}
```

The important shift is this:

| Static audit asks           | Rolling audit asks                            |
| --------------------------- | --------------------------------------------- |
| Is the repo conformant now? | Is the repo getting safer or drifting?        |
| What findings exist?        | Which findings keep coming back?              |
| Did tests pass?             | Did the right proof lane run for this change? |
| Is the score above floor?   | Is score volatility rising?                   |
| Is there an exception?      | Is exception debt growing?                    |
| Did the agent fix it?       | Did the repair teach future agents?           |

This turns Jankurai into a repository immune system. It does not just block a single mistake. It detects patterns of weakness.

## New terms to add to the standard

**Proof freshness**
The percentage of changed paths covered by proof receipts newer than the last edit.

**Vibe recurrence**
The rate at which the same finding fingerprint or rule/path pattern reappears after being repaired.

**Exception debt**
The weighted count of active exceptions by severity, age, owner, and expiration status.

**Context drag**
How much token/context budget is spent before the agent can identify owner, legal edit surface, and proof lane.

**Centerline drift velocity**
How quickly the repo is moving away from the standard it claims.

These concepts make the rolling score feel profound rather than cosmetic.

# 3. Code and tooling review

## What is already excellent

The codebase has the right skeleton for a serious control plane. The CLI is not just `audit`; it exposes commands for audit, init, doctor, context packs, repair plans, lane selection, proof, proof verification, registry, cells, migration, benchmarking, certification, governance, publishing, repair, optimization, exception expiry, adapters, CI install, issue export, explanation, version checks, UX, and security. ([GitHub][5])

The audit pipeline already implements many of the paper’s ideas: it inventories the repo, computes scope from changed paths, runs dimensions, applies caps, loads policy, attaches UX reports, builds findings, generates an agent fix queue, handles decisions, proof receipts, fingerprints, and emits structured reports. ([GitHub][11])

The finding model is also strong. Findings include severity, category, path, problem, agent fix, evidence, check ID, hardness, confidence, evidence kind, rerun command, fingerprint, rule ID, TLR, lane, docs URL, owner, line, matched term, and reason. That is exactly the shape repair agents need. ([GitHub][12])

The repo also already has a serious policy nucleus: hard caps for missing root instructions, missing fast lane, missing security lane, generated contract drift, Python product truth, missing scanning, missing CI audit lane, generated-zone mutation, direct DB wrong layer, rendered UX gaps, prompt injection, overbroad agency, secret-like content, false-green tests, destructive migration, missing Rust property/integration tests, missing agent-readable docs, and streaming runtime drift. ([GitHub][13])

## The most important code gaps to fix

### 1. Make the source itself maximally agent-readable

Some raw GitHub-rendered source files appear flattened into extremely long lines. If that reflects the actual repository formatting rather than a web-rendering artifact, fix it immediately. For a project whose thesis is repair locality, source line-addressability is part of the product.

Add hard gates:

```bash
cargo fmt --check
cargo clippy --workspace --all-targets -- -D warnings
npm run format:check
npm run lint
```

Then add a Jankurai self-rule:

> Source files used by repair agents MUST be line-addressable, formatted, and diff-local.

This is not cosmetic. Agents repair faster when code has normal structure.

### 2. Reconcile CLI/docs command drift

The CLI exposes migration output through `--out` / `--md`, while at least one repo mapping appears to use `--json` for migration-related commands. That is exactly the kind of command drift that will confuse agents. The fix is simple: either add `--json` as an alias everywhere or update all docs/maps to use `--out`. ([GitHub][5])

Add a command:

```bash
jankurai commands verify .
```

It should parse every command in:

* `agent/test-map.json`
* `agent/proof-lanes.toml`
* `README.md`
* paper Appendix E
* docs command examples
* `.github/workflows/*`

Then it should run `--help` validation or a dry parse to prove the commands still exist.

### 3. Align CI defaults with the adoption story

The README/adoption story says observe first, baseline second, ratchet later. That is the right story. But the visible workflow and CLI defaults should make that impossible to accidentally violate. The generated CI should default to observe/advisory unless a reviewed baseline exists. ([GitHub][1])

Recommended rule:

> `ratchet` mode MUST require an explicit accepted baseline file.

No implicit baseline. No hidden score floor. The whole point is progressive adoption.

### 4. Harden CI supply chain

The workflow uses common GitHub Actions and toolchain setup. For a security-forward standard, add a “strict CI profile” that pins actions by SHA, sets least-privilege permissions, runs actionlint/zizmor, and uploads SARIF/JUnit/artifacts consistently. The repo already has the right security direction; now make its own CI a showcase. ([GitHub][14])

Suggested workflow principles:

```yaml
permissions:
  contents: read
  security-events: write
  actions: read
```

And add:

```bash
jankurai ci verify . --strict
```

### 5. Replace key regex heuristics with AST-backed detectors

The repo already has a dependency-graph pilot for Rust and TypeScript imports, but it appears intentionally lightweight. That is fine for the seed version. The next standard-grade upgrade is a true boundary graph. ([GitHub][15])

Use tree-sitter or language-native parsers for:

* Rust imports, modules, function calls, unsafe blocks, SQLx usage
* TypeScript imports, generated-client boundaries, fetch wrappers, React component patterns
* Python imports, DB clients, framework route ownership, notebook/product-truth leakage
* SQL migration hazards
* YAML/JSON schema drift

Then report forbidden dependency edges:

```json
{
  "rule_id": "HLT-006-DIRECT-DB-WRONG-LAYER",
  "edge": {
    "from": "apps/web/src/checkout/usePayment.ts",
    "to": "postgres",
    "kind": "import"
  },
  "owner": "web",
  "expected_owner": "adapters/postgres",
  "repair": "Move DB access behind generated API boundary"
}
```

That would massively improve confidence.

### 6. Make proof receipts non-optional

The audit model already supports proof receipts, but the standard should move from “lane exists” to “lane produced evidence for this change.” ([GitHub][5])

A proof lane should only count if it emits a receipt with:

* command ID
* command hash
* changed path coverage
* exit code
* artifact hashes
* elapsed time
* runner identity
* filtered output
* raw log reference
* schema version

This closes a major gaming hole.

### 7. Upgrade UX visual diff

The UX QA package is one of the project’s most exciting pieces. It already captures route/story/state metadata, browser/viewport data, geometry violations, screenshots, crops, ARIA snapshots, accessibility artifacts, and baseline metadata. ([GitHub][6])

The next upgrade is real image comparison:

* Pixel diff with tolerance
* Crop localization
* Changed-region bounding boxes
* Anti-aliasing tolerance
* Per-component baselines
* Human label storage
* “Accepted visual change” receipts
* VLM triage only after deterministic gates

This would make Jankurai not just a code auditor, but a UI safety net for AI-generated product work.

### 8. Add schema contracts for every artifact

The repo already has rich JSON outputs. Now formalize them:

```
schemas/
  repo-score.schema.json
  finding.schema.json
  proof-receipt.schema.json
  repair-receipt.schema.json
  rolling-score.schema.json
  context-pack.schema.json
  ux-report.schema.json
  exception.schema.json
```

Then require:

```bash
jankurai schemas verify .
```

The standard cannot become global unless artifacts are stable across tools and vendors.

# 4. The profound upgrade: Jankurai Flight Recorder

The project’s next-level tooling should be a **Flight Recorder for agentic coding**.

This would be the leap from “audit CLI” to “global engineering substrate.”

## What it does

Every AI-assisted change gets a durable task ledger:

1. What was the intent?
2. Which files did the agent read?
3. Which owner owned the change?
4. Which paths were legal to edit?
5. Which generated zones were protected?
6. Which proof lanes were required?
7. Which commands ran?
8. What evidence was produced?
9. What findings appeared?
10. What repair fixed them?
11. Did the repository score improve or drift?
12. Did this mistake happen before?

This is the missing memory layer for coding agents.

## New artifacts

```text
agent/
  score-history.jsonl
  rolling-score.json
  context-budget.json
  task-ledger/
    <task-id>.json
  proof-receipts/
    <task-id>/<lane-id>.json
  repair-receipts/
    <task-id>/<rule-id>.json
  fingerprints/
    active.json
    reopened.json
```

## New commands

```bash
jankurai task start . \
  --intent "Fix checkout retry handling" \
  --changed-from origin/main \
  --out target/jankurai/task.json

jankurai context pack . \
  --task target/jankurai/task.json \
  --max-tokens 8000 \
  --out target/jankurai/context-pack.json \
  --md target/jankurai/context-pack.md

jankurai prove . \
  --task target/jankurai/task.json \
  --receipts target/jankurai/proof-receipts \
  --evidence-index target/jankurai/evidence-index.json

jankurai score trend . \
  --history agent/score-history.jsonl \
  --latest target/jankurai/repo-score.json \
  --out target/jankurai/rolling-score.json \
  --md target/jankurai/rolling-score.md

jankurai pr summary . \
  --task target/jankurai/task.json \
  --score target/jankurai/rolling-score.json \
  --out target/jankurai/pr-summary.md
```

## Why this would shock people

Because it reframes AI coding from “prompt and hope” to **instrumented engineering**.

The current agent workflow is mostly invisible. A model edits code, maybe runs tests, maybe summarizes what happened. The Flight Recorder makes every step auditable. It turns ephemeral agent behavior into durable engineering evidence.

That is the global-standard move.

# 5. The “Rolling Score” design I would implement

A great rolling score should not be a simple average. It should detect risk, recurrence, and decay.

Suggested formula components:

```text
rolling_score =
  latest_score
  - new_high_risk_penalty
  - reopened_fingerprint_penalty
  - stale_exception_penalty
  - stale_proof_penalty
  - context_drag_penalty
  + repaired_finding_credit
  + proof_freshness_credit
```

## Required fields

```json
{
  "schema_version": "1.0.0",
  "standard_version": "0.4.0",
  "latest": {
    "score": 88,
    "decision": "warn",
    "high_findings": 2,
    "critical_findings": 0
  },
  "trend": {
    "ewma_7d": 91,
    "ewma_30d": 86,
    "score_delta_since_baseline": -4,
    "centerline_drift": 7,
    "volatility": 3.2
  },
  "recurrence": {
    "new_fingerprints": 8,
    "reopened_fingerprints": 2,
    "most_repeated_rules": [
      "HLT-007-HANDWRITTEN-CONTRACT",
      "HLT-013-RENDERED-UX-GAP"
    ]
  },
  "proof": {
    "changed_paths": 23,
    "paths_with_fresh_receipts": 20,
    "proof_freshness": 0.87,
    "missing_lanes": ["web"]
  },
  "exceptions": {
    "active": 6,
    "expired": 1,
    "debt_score": 14
  },
  "context": {
    "context_budget_score": 0.78,
    "largest_instruction_file": "docs/agent-native-standard.md",
    "recommended_pack": "target/jankurai/context-pack.md"
  }
}
```

## The magic behavior

When a team repeatedly reintroduces handwritten DTOs, skips UX proof, or edits generated files, the rolling score should say:

> “This is not an isolated finding. This is a recurring repository behavior.”

That is the difference between a linter and an immune system.

# 6. Global-standard strategy

To become a global standard, Jankurai should stop asking the world to agree on one stack first. Ask the world to agree on the **evidence contract** first.

## The standard should define

* Required artifact names
* JSON schemas
* Rule ID format
* Finding fields
* Proof receipt fields
* Repair receipt fields
* Conformance levels
* Rolling score semantics
* Exception lifecycle
* Generated-zone semantics
* Agent instruction hierarchy
* Minimum CI modes
* Profile extension mechanism

## Profiles should define

* Stack-specific ownership cells
* Forbidden imports
* Proof lanes
* Generated-zone defaults
* Security tools
* UX tools
* Migration expectations
* Language-specific repair recipes

That lets the standard become universal while the Rust/TypeScript/PostgreSQL architecture remains the flagship profile.

Suggested naming:

```text
Jankurai Core
Jankurai Web SaaS Profile
Jankurai Rust/TS/Postgres Gold Profile
Jankurai Go Service Profile
Jankurai .NET Enterprise Profile
Jankurai JVM Brownfield Profile
Jankurai Python AI Boundary Profile
```

This is how everyone can align without everyone first rewriting.

# Final positioning

The project should not be pitched as “a better way to organize repos for AI.”

It should be pitched as:

> **The trustworthy-merge standard for agentic software engineering.**

The enemy is not humans. The enemy is unmeasured plausibility.

The cure is not slower review. The cure is a repository that can say, for every agent-made change:

**who owned it, what it was allowed to touch, which proof accepted it, what evidence remains, what score changed, and whether this mistake has happened before.**
