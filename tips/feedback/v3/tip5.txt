I studied the uploaded paper and the public repository as a static review. I did **not** execute the full Rust/npm/LaTeX test suite locally, so the code comments below are based on repository inspection, not runtime validation. The big picture: **this is already much more than a manifesto.** The repo has a real CLI, real audit artifacts, owner/test/proof maps, generated-zone policy, proof receipts, repair planning, UX QA, security evidence, CI, and score history. The paper’s strongest move is naming the new bottleneck: not “can AI write code?” but **can the repo reject, route, prove, and repair generated code?** That is the world-shocking idea. 

## Core verdict

Jankurai can become a global standard if you make one thing unmistakable:

**Jankurai is not a stack manifesto. It is a merge-governance standard for the AI era.**

The paper currently has three powerful threads fighting for center stage: the anti-vibe-coding manifesto, the conformance/audit standard, and the Rust/TypeScript/Postgres stack recommendation. The first creates heat. The second creates a standard. The third creates debate. For a tier-1 topic, I would make the **standard audit + rolling score + proof receipts** the protagonist, and make the stack ranking a “default profile” rather than the central claim.

The repo already supports that story. The README defines Jankurai as an agent-native repository control plane and audit CLI that turns owner maps, proof lanes, generated zones, security boundaries, and repair queues into files humans and agents can read; it also emphasizes read-only adoption first, then CI/hooks/ratchets when teams choose. ([GitHub][1]) The code model already has stable versions, report fingerprints, input/policy/manifest fingerprints, caps, dimensions, findings, proof receipts, UX/security/boundary summaries, and an `agent_fix_queue`, which is exactly the substrate for a “merge black box.” 

The phrase to build the whole project around:

> **No proof lane, no merge. No receipt, no trust. No rolling score, no alignment.**

## The story upgrade: make “rolling audit” the cure to vibe coding

Right now the paper argues that vibe coding is plausible-code acceptance without explicit proof routing. That is excellent. The next elevation is to make the **rolling score** the thing that catches vibe mistakes before they become culture.

A one-time score is a snapshot. A **rolling score** is a memory system. It answers:

“Did this PR make the repository more agent-native or less?”

That framing is huge. It means every AI-generated change can be judged against the repo’s own baseline:

| Vibe mistake                                  | Rolling audit signal                       |
| --------------------------------------------- | ------------------------------------------ |
| Agent edits an unmapped file                  | owner-map coverage drops; HLT-003 appears  |
| Agent adds code but no proof lane             | test-map coverage drops; HLT-004 appears   |
| Agent hand-edits generated client             | generated-zone violation; HLT-002 appears  |
| Agent adds skipped/tautological test          | false-green risk; HLT-008 appears          |
| Agent changes checkout UI with no screenshot  | rendered UX proof missing; HLT-013 appears |
| Agent changes auth/data logic with weak tests | business truth risk rises                  |
| Agent adds dependency casually                | supply-chain drift appears                 |
| Agent adds fallback/TODO soup                 | entropy/vibe score drops                   |
| Agent touches DB migration destructively      | migration cap blocks or ratchets down      |

This is where the paper can become electric: **Jankurai is not merely scoring repo quality; it is detecting the delta between intended engineering and actual agent behavior.**

The code is close. The Justfile writes score history during `check` and `score`, including `agent/score-history.jsonl` and `agent/score-history.csv`.  The `score_history` module appends branch, commit, dirty-worktree state, score, raw score, caps, finding counts, hard/soft findings, decision, scope, changed paths, and report fingerprint.  That is the seed of the “rolling score catches vibe mistakes” story. I would make it a headline feature, not an implementation detail.

## Paper feedback by section

### Title, subtitle, and abstract

**What works:** The title is memorable. The abstract’s bottleneck shift is superb: agents can produce plausible code, so the scarce act becomes trustworthy merge. That is the paper’s north star. 

**Elevate it:** The title “Humans Were the Bug” is powerful as a movement slogan, but it risks alienating serious engineering readers. I would keep it, but make the first sentence soften and sharpen the claim:

> “Humans were not the bug because humans wrote code. Human-memory-centered repositories were the bug.”

The abstract should reduce stack emphasis and foreground the standard. Suggested abstract arc:

1. AI collapses typing cost.
2. This creates a merge-trust crisis.
3. Existing repos encode ownership, proof, security, and repair in human memory.
4. Jankurai defines a machine-auditable control plane.
5. The contribution is conformance levels, stable rule IDs, proof lanes, generated-zone policy, rolling score, repair queues, and receipts.
6. Stack guidance is one profile, not the main theorem.

Add one brutally clear sentence:

> “A Jankurai-conformant repository makes an AI patch either provable, repairable, or unmergeable.”

### I. New Bottleneck: Trustworthy Merge

**What works:** This is the strongest conceptual section. The “do not slow agents down; redesign the repo as the control surface” point is exactly right. 

**Elevate it:** Add a concrete opening incident. Tier-1 papers benefit from a motivating example that readers can remember. For example:

> “An agent changes a checkout component. TypeScript passes. Unit tests pass. The payment route now double-submits on retry, the mobile button overlaps the footer, and the generated API client was hand-edited. The reviewer sees plausible code. Jankurai sees four regressions: idempotency proof missing, rendered UX proof missing, generated-zone mutation, and proof-lane mismatch.”

Then show how a rolling score catches it:

```text
PR score delta:
- score: 91 -> 76
- new caps: generated-zone-mutation-risk, missing-rendered-ux-qa-lane
- new high findings: HLT-002, HLT-013
- required proof lanes: contract, web
- merge decision: blocked
```

That makes “vibe mistakes” real.

### II. Definitions and Threat Model

**What works:** The operational definitions are excellent: vibe coding, agent-native engineering, proof lane, generated zone, repair receipt, centerline drift. The threat model table on page 3 is one of the paper’s best artifacts because it converts vibes into controls. 

**Elevate it:** Add a formal lifecycle diagram:

```text
Change request
  -> context pack
  -> allowed edit surface
  -> proof plan
  -> proof receipts
  -> audit report
  -> rolling score delta
  -> repair queue or merge
```

Also add adversary assumptions:

* The agent may be competent but context-limited.
* The agent may follow hostile issue text unless trusted policy is explicit.
* The agent may over-edit if ownership is implicit.
* The human reviewer may be overloaded.
* CI may be green for the wrong reason.

This makes the threat model feel like security engineering, not just AI commentary.

### III. Languages as Bottleneck Compression

**What works:** This is a beautiful lens. “Languages are bottleneck-compression systems” is memorable and useful. It explains why syntax matters less once agents can imitate syntax. 

**Elevate it:** Make this section serve the audit standard. Right now it can sound like the paper is drifting into language philosophy. Add this bridge:

> “The agent-era question is not ‘Which language is expressive?’ It is ‘Which language and repository structure fail fastest when an agent violates reality?’”

Then define four agent-era language/repo properties:

1. **Rejection speed**: how fast invalid changes fail.
2. **Proof locality**: how narrowly tests can be routed.
3. **Boundary visibility**: whether ownership is machine-readable.
4. **Repair explainability**: whether errors produce actionable receipts.

That frames the stack discussion as a consequence of verification, not preference.

### IV. Technical Promise versus Standard Gravity

**What works:** The distinction between technical elegance and default-standard status is important. The Julia example is fair when framed as “real promise, narrower default gravity.” 

**Elevate it:** This section risks starting language wars. I would rename it:

> **Ecosystem Gravity and Agent Repairability**

Then score ecosystems by agent-repair substrate:

* public corpus density,
* standard deployment path,
* security tooling,
* contract generation,
* observability conventions,
* dependency governance,
* CI examples,
* migration norms.

This makes the argument less “these languages lose” and more “agent-native engineering rewards ecosystems with abundant enforceable examples.”

### V. Vibe-Coding Fault Taxonomy

**What works:** This is a major contribution. The RPN taxonomy, top-level risk buckets, and stable rule mapping are the heart of the paper. Table IV is exactly the kind of artifact people can adopt. 

**Elevate it:** Add the missing scoring mechanics. The paper says RPN = impact × likelihood × detection difficulty, but it should define each scale. Add a small table:

| Score | Impact                   | Likelihood               | Detection difficulty       |
| ----: | ------------------------ | ------------------------ | -------------------------- |
|     1 | cosmetic                 | rare                     | obvious locally            |
|     3 | user-visible or costly   | plausible                | requires mapped proof      |
|     5 | security/money/data loss | common under agent churn | hard without standard lane |

Then the RPN becomes auditable policy, not vibes about vibes.

Also add a “rolling score catches this” subsection. Example:

> “A vibe mistake is usually not a single catastrophic event. It is a regression pattern: one unmapped file, one skipped test, one generated client edit, one TODO, one hidden dependency. The rolling score turns these small leaks into visible deltas before they become architecture.”

### VI. Jankurai Standard and Conformance

**What works:** The install ladder and HL0–HL5 levels are strong. This is where the paper becomes a standard. The repo also matches this direction: the public README describes observe, agents, full, and ratchet adoption levels, and emphasizes safe read-only first passes. ([GitHub][1])

**Elevate it:** Make conformance more normative and less prose-driven. Add a table like:

| Artifact          |      HL1 |    HL2 |      HL3 |      HL4 |  HL5 |
| ----------------- | -------: | -----: | -------: | -------: | ---: |
| root agent router |   should |   must |     must |     must | must |
| owner map         |   should |   must |     must |     must | must |
| test map          |   should |   must |     must |     must | must |
| proof receipts    | optional | should |     must |     must | must |
| score history     | optional | should |   should |     must | must |
| ratchet baseline  |       no |     no | optional |     must | must |
| release evidence  |       no |     no |       no | optional | must |

Also, the paper Appendix A currently lists initial rule IDs through HLT-019, but the code already emits **HLT-020-CI-HARDENING-GAP** and **HLT-021-DESTRUCTIVE-MIGRATION** in the audit path. That mismatch should be fixed before a serious release, because stable rule identifiers are only stable if the paper, schema, and CLI agree.  

### VII. 100-Point Stack Rubric

**What works:** The rubric is useful and opinionated. The weighting toward correctness, security, contracts, and repairability is right because agents already reduce first-draft cost. 

**Elevate it:** The scoring needs reproducibility. Add a machine-readable example:

```json
{
  "profile": "rust-ts-vite-react-postgres-bounded-python",
  "weights": {
    "proof_loop": 18,
    "security": 16,
    "runtime": 13
  },
  "hard_caps": [
    {
      "id": "no_deterministic_fast_lane",
      "max_score": 65
    }
  ]
}
```

Then state that stack scores are **profile priors**, while repo scores are **observed conformance**. That distinction matters.

### VIII. Stack Ranking, Sensitivity, and Exceptions

**What works:** The sensitivity discussion is mature. It acknowledges that Go, .NET, JVM, and other stacks can win under different constraints. 

**Elevate it:** Move the detailed stack ranking later or into an appendix if the goal is global standard adoption. The default stack is interesting, but it can distract from the universal contribution. A Java team, a Go team, and a .NET team should all read the paper and think, “I can adopt this standard,” not “this paper is arguing against me.”

Recommended reframing:

> “Jankurai Core is stack-neutral. Jankurai Profiles are stack-specific. This paper ships one recommended default profile.”

That gives you both sharpness and adoption.

### IX. Default Winner Architecture

**What works:** The ownership cells are clear. The filesystem-as-routing-table idea is excellent. 

**Elevate it:** Rename the section to:

> **Reference Profile: Rust/TypeScript/PostgreSQL**

Then include two smaller alternate profile examples:

* Go + TypeScript + Postgres profile,
* .NET + TypeScript + Postgres profile.

This prevents readers from rejecting the whole standard because they disagree with the default stack. The standard wins if it can audit any stack; the default profile wins when teams want the strongest greenfield path.

### X. Agent Repository Controls and Tool Adapters

**What works:** This section maps directly to the real repo. The project includes adapter surfaces such as `.agents`, `.claude/skills/jankurai`, `.cursor/rules`, `AGENTS.md`, `CLAUDE.md`, and `GEMINI.md` in the root tree. ([GitHub][1])

**Elevate it:** Add a “single-source adapter generation” rule:

> “Provider instructions may mirror policy, but must not originate policy.”

In code terms, every agent adapter should include:

* standard version,
* source manifest hash,
* generated timestamp,
* “do not edit here; edit canonical policy” pointer.

This turns “alignment” into checksum-controlled repository infrastructure.

### XI. Continuous Proof, CI, and Test Explosion

**What works:** The proof-lane doctrine is crucial. The repo already has `agent/test-map.json` mapping paths to commands, and `agent/proof-lanes.toml` defining lanes such as fast, audit, paper, security, full, adapters, doctor, migration analysis, schema build, UX QA, and versions.  

**Elevate it:** Add a cost model. Test explosion becomes acceptable only if routed by cost:

```text
Changed path -> minimum proof lane -> receipt
             -> optional broader lane if public boundary changed
             -> nightly/release matrix if expensive
```

Also, the generated CI installer supports observe/advisory/ratchet modes, but the generated ratchet workflow currently gates primarily on score floor with `jq`, while the richer audit decision logic exists elsewhere.  I would make ratchet enforce the full decision object: score floor, no new hard caps, no new high/critical findings, no expired exceptions, and no unverified proof plan.

### XII. Rendered UX and Browser-Step QA

**What works:** This may be the most differentiated part of the project. Many AI coding tools still treat UI correctness as “it compiled.” Jankurai’s rendered UX section says the browser is a runtime and requires screenshots, geometry, ARIA/accessibility, state coverage, and artifacts. 

The code backs this up. `@jankurai/ux-qa` is a TypeScript package with Playwright peer dependency and `axe-core`; its rules check edge clearance, target size, interactive overlap, text clipping, button wrapping, horizontal overflow, sticky obstruction, z-index ceilings, focus visibility, form labels, and nested scrollbars.   Its page analyzer returns schema/tool versions, URL, viewport, metrics, elements, violations, artifacts, state coverage, and a pass/warn/block decision. 

**Elevate it:** Make this the public demo. A video or GIF of Jankurai catching a mobile layout overlap that unit tests miss would land hard. Suggested demo:

1. Agent changes checkout page.
2. TypeScript passes.
3. Component test passes.
4. `jankurai ux` blocks because the primary button is clipped or overlapped at 375px.
5. Report shows selector, bounding box, screenshot crop, ARIA snapshot, and repair hint.
6. Agent fixes it.
7. Score returns to baseline.

That is “shock the world” material because it is visceral.

### XIII. Security, Supply Chain, and Permissions

**What works:** The security posture is practical. The README lists controls for broad writes, weak proof, prompt injection, generated drift, security regressions, and lost context. ([GitHub][1]) The CLI has `security run`, which executes a security lane script, emits normalized evidence, captures logs, and parses `jankurai-security-step=<json>` lines from the script output. 

**Elevate it:** The security evidence should become first-class normalized findings, not just wrapper evidence. Add adapters for:

* secret scanner output,
* OSV/dependency scan output,
* SBOM/provenance output,
* GitHub Actions hardening checks,
* Semgrep/SAST output,
* SARIF ingestion and emission.

The audit already has lexical detectors for secret-like material, prompt-injection phrases, overbroad agency, false-green tests, destructive SQL, generated-zone issues, wrong-layer DB access, duplicate blocks, CI hardening, and streaming runtime drift.  The next step is to make those detectors pluggable and semantic.

### XIV. Exceptions, Observability, and Repair Receipts

**What works:** This section is very strong philosophically. “Agent-friendly exceptions” is a powerful idea: errors should teach repair systems what invariant was protected and what fixes are legal. 

The model supports rich proof receipts: lane, command, exit code, artifacts, changed paths, owner, residual risk, logs, plan digest, command digest, log SHA, artifact digests, rules covered, retryability, and captured stdout/stderr bytes. 

**Elevate it:** Add one concrete error example in Rust and one in TypeScript. The paper should show what “agent-friendly” means:

```json
{
  "code": "PAYMENT_IDEMPOTENCY_REQUIRED",
  "safe_user_message": "The payment could not be completed safely.",
  "repair_hint": "Add or pass an idempotency key before calling PaymentService::charge.",
  "proof_lane": "payment-idempotency",
  "docs": "docs/runbooks/payments.md#idempotency"
}
```

That turns observability into repair acceleration.

### XV. Migration, Versioning, and Governance

**What works:** The versioned artifact manifest is exactly right. The paper says the standard, auditor, schema, paper edition, target stack, generated artifacts, and CI policy should share a manifest.  The code model has constants for `standard_version`, `auditor_version`, `schema_version`, `paper_edition`, and `target_stack_id`. 

**Elevate it:** Add a governance mechanism:

> **Jankurai Improvement Proposals (JIPs)**

Each JIP should define:

* rule ID changes,
* schema migrations,
* scoring weight changes,
* new hard caps,
* conformance-level changes,
* detector false-positive policy,
* deprecation timeline.

That is how this becomes a global standard instead of one repo’s opinion.

### XVI. Limitations and Research Agenda

**What works:** The limitations section is honest and valuable. It says weights need empirical calibration and benchmarks still miss long-running maintenance, security regressions, UI correctness, and production repair quality. 

**Elevate it:** Make the research agenda measurable:

| Research question                      | Metric                                            |
| -------------------------------------- | ------------------------------------------------- |
| Does Jankurai reduce unsafe merges?    | seeded-defect catch rate, escaped-defect proxy    |
| Does rolling score help?               | score regression catch rate, mean time to repair  |
| Does context packing reduce tokens?    | context bytes/tokens per successful repair        |
| Do proof lanes improve speed?          | proof time, rerun count, repair success rate      |
| Are findings too noisy?                | false-positive rate by HLT rule                   |
| Does UX QA catch real issues?          | visual/layout defects caught before review        |
| Do repair receipts help future agents? | second-agent repair success with/without receipts |

This converts the project into a research platform.

### XVII. Conclusion

**What works:** The closing line “Anything less is just faster vibe coding” is strong. 

**Elevate it:** Make the conclusion less stack-centered and more civilization-scale:

> “The shared language is not taste. It is receipts.”

End with the global-standard idea:

> “Agent-native engineering gives teams a common merge constitution: every change has an owner, every risky path has a proof lane, every failure has a receipt, every exception has an expiry, and every repository can show whether it is drifting toward or away from trust.”

### Appendices

**What works:** Appendix A’s rule IDs are one of the best assets in the paper. Appendix B’s versioned artifacts, Appendix C’s exception templates, Appendix D’s canonical file trees, and Appendix E’s commands make the standard operational rather than abstract. 

**Elevate it:** Add three things:

1. **Schema references** for every artifact: report, finding, proof plan, proof receipt, evidence index, UX QA report, repair plan, exception.
2. **Rule maturity levels**: lexical, AST, semantic, external-tool-backed, empirically calibrated.
3. **Paper/code rule parity**: add HLT-020 and HLT-021 if the code keeps emitting them.

## Code feedback: what is already excellent

The CLI surface is bigger and more serious than I expected. The main command set includes audit, adopt, init, update, doctor, context pack, repair plan, proof planning, proof execution, proof verification, registry, cell, migrate, bench, certify, govern, publish, repair, optimize, Rust mapping/witness/diagnose, adapters, agent verification, CI install, hooks install, issues export, explain, versions, UX, and security. 

The architecture is coherent. The Rust crate exposes modules for audit, boundaries, commands, init, model, render, report, score history, validation, and versions.  The audit layer is decomposed into analyzers, caps, evidence, finding building, fix queues, filesystem inventory, rules, scanning, security artifacts, UX artifacts, and boundary artifacts. 

The owner/test/proof infrastructure is real. `agent/owner-map.json` assigns owners to root files and major directories, while `agent/test-map.json` maps changed paths to proof commands.   The proof engine builds proof plans from changed paths, routes through owner/test maps, creates planned runs, enforces that proof commands are signed by the test map or proof-lane manifest unless explicitly overridden, writes receipts, and verifies plan/manifest/receipt digests. 

The repair system has the right safety instincts. Repair planning creates packets with rule IDs, severity, owner, lane, permission profile, allowed/forbidden paths, expected patch shape, required proof, stop conditions, eligibility, risk, human-review requirement, and rollback guidance.  Real repair is gated by environment variables for file mutation, git mutation, and GitHub PR creation; it applies edits, runs proof, rolls back on failure, and can prepare/push a draft PR only when explicitly allowed.  

The UX QA package is a standout. It collects DOM state with Playwright, builds stable selectors, checks visibility/focus/labels/hit testing, captures viewport/page metrics, and runs geometry/accessibility-adjacent rules.   This is the piece I would use to make the project emotionally obvious to developers.

## Code gaps to close before the “global standard” push

### 1. Turn ratchet into true score-delta governance

The score-history module is promising, but ratchet should become more than “score must be above floor.” The CI installer’s ratchet workflow currently adds a shell gate checking `.score >= min_score`, while the richer decision object and findings exist elsewhere. 

Upgrade ratchet to compare current report against a reviewed baseline:

```text
block if:
- score decreases without active exception
- any new high/critical finding appears
- any new hard cap appears
- any protected dimension drops
- proof coverage decreases
- exception count increases or expiry passes
- generated-zone policy weakens
```

This is the feature that makes “rolling score catches vibe mistakes” undeniable.

### 2. Make `AuditMode` drive behavior everywhere

The code has `AuditMode` values for advisory, guarded, standard, ratchet, and release.  But the audit policy summary currently defaults mode to `standard` in the core report load path, and `report_decision` currently sets ratchet baseline to the current score rather than a real accepted baseline.  Make mode semantics explicit:

* advisory: never fail;
* guarded: fail only critical caps/malformed report;
* standard: fail high/critical and score floor;
* ratchet: fail regression against baseline;
* release: require full proof matrix and artifact versions.

### 3. Graduate detectors from lexical to semantic

The current scanning layer is fast and useful, but many detectors are lexical pattern matches: TODOs, fallback phrases, prompt-injection phrases, agency markers, false-green patterns, streaming client names, future-hostile terms, secret-like regexes, and SQL heuristics.  This is fine for 0.4.0, but global-standard confidence needs detector classes:

```text
detector.kind = "regex" | "ast" | "schema" | "external" | "receipt" | "semantic"
detector.confidence = 0.0..1.0
detector.false_positive_policy = "allowlist" | "exception" | "soft" | "block"
```

Add Tree-sitter/Semgrep-style detectors for TypeScript/Python/Go and expand the existing Rust `syn` infrastructure. The Rust map/witness code already uses `cargo_metadata` and `syn` to compute package maps, public items, interface hashes, implementation hashes, and witness diffs.  That should become the model for all languages.

### 4. Make low-risk auto-repair actually useful

The repair apply layer supports `append-text`, `replace-exact`, and `create-file`, but repair-plan currently emits planned edits with `apply_strategy: "none"` by default.   That is safe, but it undersells the project.

Add first-party repair recipes for low-risk, high-confidence issues:

| Rule                        | Auto-repair candidate                                         |
| --------------------------- | ------------------------------------------------------------- |
| HLT-003 ownerless path      | propose owner-map entry using nearest parent owner            |
| HLT-004 unmapped proof      | propose test-map entry using nearest lane                     |
| HLT-015 context/setup gap   | create minimal setup/fast lane template                       |
| HLT-020 CI hardening        | replace `write-all`, flag `@master`, avoid echoing secrets    |
| generated-zone metadata gap | add missing `source`/`command` placeholders with human review |
| missing adapter mirror      | regenerate thin provider adapter from canonical standard      |
| missing exception template  | create scoped exception draft with expiry blank-blocking      |

Make auto-repair conservative, but real. The magic experience should be:

```bash
jankurai audit .
jankurai repair-plan . --from agent/repo-score.json --out target/jankurai/repair-plan.json
jankurai repair . --plan target/jankurai/repair-plan.json --apply --git-commit --auto-pr
```

And the output should be a draft PR with proof receipts.

### 5. Normalize all proof-lane metadata

Some lanes in `agent/proof-lanes.toml` have rich metadata such as `command_id`, `kind`, `cost`, `rules_covered`, `required_artifacts`, `timeout_seconds`, `requires_network`, and `destructive`; other lanes only have name, command, and purpose.  For a standard, every lane should have the same schema. Add:

```toml
risk = "low|medium|high|critical"
parallel_safe = true
requires_secrets = false
writes_paths = []
reads_paths = []
rules_covered = []
artifacts_required = []
artifacts_optional = []
timeout_seconds = 900
```

That will let agents choose proof intelligently.

### 6. Clarify generated artifacts that are lane-writable but agent-read-only

`agent/generated-zones.toml` marks paper artifacts and `package-lock.json` as read-only, but `agent/repo-score.json` and `agent/repo-score.md` are generated zones with `read_only = false`.  That may be technically convenient, but conceptually confusing.

Use a stronger policy shape:

```toml
read_only_for_agents = true
writable_by_lanes = ["audit", "score"]
source = "crates/jankurai"
command = "cargo run -p jankurai -- audit ..."
```

This preserves audit output generation without implying agents may hand-edit score files.

## The profound upgrade: Jankurai Flight Recorder + Proof Autopilot

The next-level feature should be a **Jankurai Flight Recorder**.

Every PR gets a tamper-evident black box:

```text
.jankurai/flight-records/<commit>.json
```

It records:

* before/after score,
* score delta by dimension,
* caps added/removed,
* new/resolved findings,
* changed paths,
* owner routes,
* proof routes,
* commands allowed,
* commands run,
* proof receipts,
* log/artifact digests,
* generated-zone diffs,
* dependency diffs,
* security evidence,
* UX evidence,
* DB/migration evidence,
* exceptions added/expired,
* repair queue,
* agent/tool identity when available,
* context pack used,
* final merge decision.

Then add **Proof Autopilot**:

```bash
jankurai flight start --changed-from origin/main
jankurai context-pack --task "fix checkout bug"
jankurai prove --changed-from origin/main
jankurai audit --mode ratchet --baseline agent/baseline.json
jankurai flight close
```

The output:

```text
Jankurai Flight Record
Score: 91 -> 89
Decision: blocked
Why:
- HLT-013 rendered UX proof missing for apps/web/checkout
- HLT-004 proof route missing for apps/web/checkout/retry.ts
Required:
- web lane
- idempotency lane
Next:
- add Playwright screenshot receipt
- add retry/idempotency test
```

This upgrade makes Jankurai the **merge black box for AI coding**. That is the global-standard idea. Teams may disagree on languages, models, and IDEs, but everyone can understand a flight record.

```bash
jankurai score diff --from baseline --to agent/repo-score.json
```

### Phase 3: Semantic detector framework

## Final positioning

The world does not need another “AI coding best practices” doc. It needs a **merge constitution**.

Jankurai should become the standard that says:

* every path has an owner,
* every owner has proof,
* every generated file has a source,
* every risky change has a lane,
* every proof has a receipt,
* every exception expires,
* every PR changes the rolling score,
* every agent works inside the same local constitution.

That is the project that can shock people: not because it says AI coding is dangerous, but because it gives teams a concrete way to make AI coding **auditable, repairable, faster, and aligned**.
