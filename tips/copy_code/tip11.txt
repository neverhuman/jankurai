The smartest answer is **not one detector**. The best-known low-noise strategy is a **staged clone pipeline**:

**fast token/fingerprint scan → strict candidate filtering → AST/structural verification → CI baseline only for new serious clones.**

That is the pattern most likely to be **really fast** while keeping false positives low.

## Best methods, ranked

### 1. Exact and normalized hashing: fastest, lowest false positives

Start with hashes before using any “AI” or heavy clone detector. Hash at multiple granularities:

```text
file hash
function / method / React component hash
statement-block hash
sliding token-window hash
```

Normalize only safe things: whitespace, comments, formatting, path separators, line endings. For low false positives, **do not blindly erase string literals or numbers** in React, Dockerfiles, API clients, config, or tests; those values often carry meaning.

This catches Type-1 clones and near-exact duplicates almost instantly. It also gives you a clean baseline for CI.

**Best for:** “same code copied,” duplicate utility files, copied React components, repeated Python/Rust functions, duplicated Dockerfile command blocks.

---

### 2. Token-based clone detection with long minimum windows

For broad, fast discovery, token-based detectors are still the workhorse. Research consistently shows text/token approaches scale better, while AST/graph methods improve accuracy at higher cost; TACC’s study specifically concludes that low-threshold token filtering followed by AST verification is the optimal practical combination for large-scale clone detection. In that work, the combined approach completed a 100MLOC benchmark in under four hours and outperformed several prior detectors in the authors’ comparison. 

Use token detection with conservative settings:

```text
minimum tokens: 80–160+
minimum lines: 8–20+
same language only
exclude generated/vendor/build/test-fixture noise
keep literals significant unless deliberately searching Type-2 clones
```

**Best tools:**

**jscpd** is the fastest broad first pass for your stack. It supports 150+ formats and uses Rabin-Karp-style copy/paste detection; its supported formats include JavaScript, JSX, TypeScript, TSX, Python, Rust, Docker, JSON, YAML, and more. ([GitHub][1])

**PMD CPD** is excellent for stricter CI-style reporting. PMD’s Copy/Paste Detector supports many languages, requires a minimum-token threshold, and supports Python, TypeScript, JavaScript, and Rust for CPD. ([PMD][2])

**SourcererCC** is a strong research-grade token detector for Type-1, Type-2, and Type-3 clones. Its paper describes an optimized inverted index and filtering heuristics, and reports scaling to 250MLOC on a standard workstation. ([arXiv][3])

**My practical take:** run `jscpd` first across the whole repo, then use PMD CPD per language for stricter gates.

---

### 3. Token candidate generation + AST verification: the best low-false-positive method

This is the “smartest” technique overall.

The idea:

```text
1. Token scanner finds possible duplicate regions.
2. AST parser verifies that candidates have the same structure.
3. Reject candidates that only share imports, declarations, prop lists, schemas, or boilerplate.
4. Keep only clones with meaningful executable logic.
```

For your stack, AST verification can use Tree-sitter-based tooling, because Tree-sitter supports mainstream languages including JavaScript, Python, and Rust, handles syntax errors reasonably well, and enables structural matching instead of pure text matching. ([Ast Grep][4])

Use checks like:

```text
same language
same syntactic unit: function vs function, component vs component, method vs method
similar AST shape
same control-flow skeleton
consistent identifier renaming only
literals preserved by default
ignore imports, type-only declarations, generated schemas, snapshots
```

This approach avoids the worst false positives from pure token matching while staying much faster than full graph/semantic clone detection.

**Best for:** React components with similar logic, copied Python functions with renamed variables, Rust handlers with small edits, duplicated API client methods.

---

### 4. NiCad-style normalized pretty-printing for high-precision near-miss clones

NiCad is a classic high-precision near-miss clone detector. It supports clone detection at file, function, and block granularity, and supports languages including Python, Java, C, C#, PHP, Ruby, and others. It lets you tune difference thresholds: `0` for exact clones, `0.10` for at most 10% different pretty-printed lines, and `0.30` as a more permissive default. ([GitHub][5])

NiCad is especially valuable when you want:

```text
near-miss clones
renamed identifiers
small edits
high precision
human-readable reports
```

For your stack, NiCad is most useful for **Python**. It is less of an all-stack answer because TypeScript, TSX, Rust, and Dockerfile support are not its main strength.

**Best use:** run NiCad on Python service code after the broad jscpd/CPD pass.

---

### 5. Structural search for known duplicate patterns

After broad discovery, use structural pattern tools to catch specific recurring clone families.

**ast-grep** is built for AST-based structural search, linting, and rewriting; it uses Tree-sitter and supports ordinary-code patterns with wildcards. ([GitHub][6])

**Semgrep** is also useful for structural patterns; its pattern language supports ellipses and metavariables, and it supports languages including Python, TypeScript, JSX, Rust, and Dockerfile. ([Semgrep][7])

This is not the best first-pass clone detector, but it is excellent for **known duplicate idioms**, for example:

```text
duplicated React hook logic
duplicated fetch / axios wrappers
duplicated form validation
duplicated FastAPI / Flask handlers
duplicated Rust match arms
duplicated Docker RUN install blocks
duplicated permission checks
```

This can get very close to zero false positives because you define the exact structural pattern that matters.

---

### 6. Avoid broad semantic or ML clone detectors unless you verify results

Semantic clone detection sounds attractive, but it is where false positives explode. TACC’s survey notes that learning-based clone detectors can have precision problems, may be time-consuming, and may not be competent for large-scale real-world clone detection without careful handling. 

Use semantic/ML methods only for research or exploratory review, not as a CI blocker, unless you add verification such as tests, symbolic equivalence, or strict AST/control-flow validation.

**Rule:** Type-4 “same behavior, different implementation” clones are interesting, but they are not safe to treat as duplicates automatically.

## Best tool stack for Python, Rust, Vite, TypeScript, Docker, React

### Fastest practical stack

```text
1. jscpd
   Broad all-repo scan: TS, TSX, JS, JSX, Python, Rust, Dockerfile, JSON, YAML.

2. PMD CPD
   Stricter per-language duplicate reports for Python, TypeScript, JavaScript, Rust.

3. ast-grep / Semgrep
   AST verification and custom duplicate-pattern checks.

4. NiCad
   Deep near-miss clone detection for Python and other supported languages.

5. Optional SonarQube
   Dashboard / trend reporting, not the lowest-false-positive detector.
```

SonarQube can be useful for dashboards, but I would not rely on it as the lowest-noise clone engine for TS/React. Sonar’s non-Java duplication logic uses token/line thresholds, and its own community discussions show JavaScript/TypeScript duplication can produce surprising results, including noise from ignored string literals and limited ways to mark duplicate blocks as false positives. ([Sonar Documentation][8])

## The low-false-positive recipe

Use this as the default policy:

```text
Exclude:
  node_modules/
  dist/
  build/
  .vite/
  coverage/
  target/
  vendor/
  .venv/
  __pycache__/
  generated/
  migrations generated by tools
  snapshots
  lockfiles
  minified files
  *.d.ts unless you explicitly care about type duplication

Ignore:
  import-only clones
  type/interface-only clones
  enum-only clones
  schema-only clones
  test fixture clones
  trivial getters/setters
  tiny components
  repeated CSS/styling-only structures

Require:
  same language
  same syntactic unit
  minimum 80–160 tokens
  minimum 8–20 lines
  executable logic present
  literals preserved by default
  consistent identifier renaming only
  AST shape similarity
```

The biggest false-positive killers are:

```text
Do not ignore string literals by default.
Do not report tiny clones.
Do not scan generated/build/vendor code.
Do not mix languages.
Do not treat type declarations as real duplicate logic.
Do not fail CI on old baseline duplicates.
```

## Best pipeline for “really really fast”

### Phase 1: cache fingerprints

For every source file, compute:

```text
raw file hash
normalized file hash
function/component/block hashes
token-window hashes
AST structural hashes
```

Store those in a local cache keyed by:

```text
file path
git blob hash
language
parser version
normalization version
```

On each run, only rescan changed files and compare them against cached fingerprints from the rest of the repo.

### Phase 2: broad token scan

Run jscpd or PMD CPD with high minimum thresholds.

Example jscpd-style scope:

```bash
npx jscpd "src/**/*.{ts,tsx,js,jsx,py,rs}" "**/Dockerfile"
```

The exact flags should be checked against your installed jscpd version, but the important settings are:

```text
respect .gitignore
high min tokens
high min lines
JSON/HTML report
ignore generated/build/vendor paths
```

jscpd’s docs show CLI usage through `npx jscpd /path/to/source` and pattern-based scanning. ([GitHub][1])

### Phase 3: verify candidates structurally

For every candidate clone pair:

```text
parse both sides
extract nearest enclosing function/component/block
compare AST shape
compare control-flow skeleton
check identifier-renaming consistency
keep literals significant
reject boilerplate categories
```

### Phase 4: group clone classes

Do not report 200 pairwise matches. Group them:

```text
clone class A:
  file1:function
  file2:function
  file3:function
```

Rank by:

```text
number of copies
lines duplicated
cyclomatic complexity
recently changed files
production code over tests
bug-prone areas
```

### Phase 5: CI only fails on new serious clones

Keep a baseline file:

```text
.dup-baseline.json
```

Then fail only when a new clone class appears above your threshold:

```text
new production duplicate
same AST category
>= 120 tokens
>= 15 lines
not generated
not test fixture
not already baselined
```

This avoids the classic disaster where a detector finds 900 old duplicates and everyone disables it.

## Recommended thresholds

Use separate thresholds by language/context:

| Area                 |         Suggested threshold | Why                                                      |
| -------------------- | --------------------------: | -------------------------------------------------------- |
| TypeScript / React   | 100–160 tokens, 10–20 lines | Avoid false positives from props, JSX boilerplate, forms |
| Python               |  80–140 tokens, 10–20 lines | Good balance for functions and service logic             |
| Rust                 | 120–200 tokens, 10–25 lines | Avoid noise from match arms, derives, macros             |
| Dockerfile           |    5–10 meaningful commands | Token/line thresholds are less reliable                  |
| Config / JSON / YAML |             Usually exclude | Often structurally repetitive by design                  |
| Tests                | Separate report, no CI fail | Test duplication is sometimes acceptable                 |
| Generated code       |            Exclude entirely | Never useful for refactoring decisions                   |

For near-miss detection, start strict:

```text
exact / normalized exact first
then >= 90% similar
then >= 85% similar for review only
```

Avoid going below 80–85% similarity unless you want a noisy research report.

## What I would actually run

### For a modern Python/Rust/React/TypeScript/Vite/Docker repo

```text
Primary discovery:
  jscpd

Strict CI report:
  PMD CPD per supported language

Deep Python near-miss:
  NiCad

AST verification:
  ast-grep / Tree-sitter custom script

Known clone families:
  Semgrep or ast-grep rules

Dashboard only:
  SonarQube, carefully configured
```

Reddit field reports line up with this split: developers mention PMD CPD as purpose-built for duplication, but also warn that configuration matters and false positives happen; Python users mention Pylint’s duplicate-code rule, but describe it as slower and mainly useful for identical-line similarity rather than robust logic-clone discovery. ([Reddit][9])

## The “fewest false positives” decision table

| Goal                              | Best method                                      |
| --------------------------------- | ------------------------------------------------ |
| Zero/near-zero false positives    | Exact normalized hashing                         |
| Fast broad scan                   | jscpd                                            |
| CI-grade token clone detection    | PMD CPD                                          |
| Python near-miss clones           | NiCad                                            |
| Huge codebase research-scale scan | SourcererCC / TACC-style token → AST             |
| React/TS structural duplicates    | token scan + Tree-sitter / ast-grep verification |
| Known duplicate bug pattern       | Semgrep / ast-grep custom rule                   |
| Dashboard/trend tracking          | SonarQube, not as source of truth                |
| Semantic “same behavior” clones   | Only with verification; do not auto-fail         |

## Bottom line

The best-known practical method is:

```text
1. Exclude noise aggressively.
2. Hash exact normalized blocks.
3. Run fast token detection with high thresholds.
4. Verify candidates with AST structure.
5. Keep literals significant by default.
6. Group clone classes.
7. Fail CI only on new, large, production-code clones.
```

For your stack, the most effective setup is:

```text
jscpd + PMD CPD + AST verification with Tree-sitter/ast-grep
```

Add **NiCad** for deeper Python analysis, use **Semgrep/ast-grep** for known duplicate patterns, and treat **semantic/ML clone detection** as exploratory unless you can verify equivalence.
