## Proposal: **Tuiwright** — a Playwright-grade Rust TUI testing framework

I would build this as a **headless, real-terminal, Rust-native TUI automation system** rather than as a thin snapshot helper. The key idea is:

> Spawn the application inside a real pseudo-terminal, emulate the terminal exactly, drive input the way a user would, build a queryable “screen model,” render that screen model into high-resolution pixels, and record every action/frame into a trace artifact that can be replayed, inspected, diffed, and exported as compressed GIF.

This would be closer to **Playwright for terminals** than to today’s typical TUI tests. Existing Rust/Ratatui testing mostly covers render-buffer assertions or snapshot testing; Ratatui’s `TestBackend`, for example, exposes buffers, scrollback, resize, and buffer assertions, and Ratatui’s own docs recommend it for testing render output, but that still does not fully exercise PTY behavior, raw-mode input, terminal escape handling, mouse protocols, process lifecycle, screenshots, video, trace debugging, or user-flow semantics. ([Docs.rs][1]) There is also an early `ratatui-testlib` that already moves toward PTY-based testing, async support, snapshots, and Sixel testing, but the proposal below is intentionally much broader: locators, actionability, semantic probes, high-resolution rendering, GIF recording, trace viewer, codegen, and Playwright-style auto-waiting. ([Docs.rs][2])

---

# 1. Product vision

The framework should let a Rust TUI author write tests like this:

```rust
use tuiwright::{expect, key, mouse, App, ScreenshotOptions};

#[tuiwright::test]
async fn create_project_flow() -> tuiwright::Result<()> {
    let app = App::new("cargo")
        .args(["run", "--bin", "my-tui"])
        .env("RUST_LOG", "debug")
        .size(120, 36)
        .term("xterm-256color")
        .theme("wezterm-dark")
        .font_family("JetBrains Mono")
        .font_size(16.0)
        .record_video("artifacts/create-project.gif")
        .trace("artifacts/create-project.trace.zip")
        .spawn()
        .await?;

    await_screen!(app.get_by_text("Welcome")).visible();

    app.get_by_role("button", "New project").click().await?;
    app.get_by_role("textbox", "Project name").fill("tuiwright-demo").await?;
    app.get_by_role("button", "Create").click().await?;

    expect(app.get_by_text("Project created")).to_be_visible().await?;
    expect(app.get_by_test_id("project-list"))
        .to_contain_text("tuiwright-demo")
        .await?;

    app.screenshot("artifacts/create-project.png", ScreenshotOptions::default())
        .await?;

    Ok(())
}
```

And also test a black-box binary with no instrumentation:

```rust
#[tuiwright::test]
async fn black_box_vim_like_flow() -> tuiwright::Result<()> {
    let app = App::command("./target/debug/my-editor")
        .size(100, 30)
        .spawn()
        .await?;

    app.keyboard().type_text("hello world").await?;
    app.keyboard().press(key!("Ctrl+S")).await?;

    expect(app.screen()).to_contain_text("Saved").await?;
    expect(app.screen()).to_match_screenshot("editor-saved").await?;

    Ok(())
}
```

The framework should have three modes:

**Black-box mode:** works against any executable. It sees the same terminal bytes a real terminal would see and can drive keyboard, mouse, paste, resize, signals, and process lifecycle.

**Rust-enhanced mode:** Rust apps can add an optional `tuiwright-probe` feature that emits a semantic widget tree over a side channel. This unlocks Playwright-like `get_by_role`, `get_by_label`, `get_by_test_id`, stable element screenshots, and much better actionability checks.

**Adapter mode:** first-class adapters for Ratatui, Cursive, Termion/Crossterm apps, and Bevy/Ratatui apps, without forcing every project into one architecture.

# 2. Why Playwright is the right mental model

Playwright’s power is not only “click button, take screenshot.” The important pieces are:

Playwright has locators with auto-waiting and retryability; locators represent a way to find elements at any moment. ([Playwright][3]) Before actions such as `click`, Playwright performs actionability checks such as unique resolution, visibility, stability, receiving events, and enabled state, then times out if those checks do not pass. ([Playwright][4]) Playwright assertions auto-retry until the expected condition is met or timeout, which is essential for avoiding flaky tests. ([Playwright][5]) It can also capture screenshots as files or buffers, capture element-level screenshots, record videos under configurable policies, and produce traces with action timelines, screenshots, snapshots, console/log information, and timing. ([Playwright][6]) Its code generator records user interactions and generates resilient locators, preferring role, text, and test-id style locators where possible. ([Playwright][7])

Tuiwright should copy those concepts, but reinterpret them for terminals:

| Playwright concept | Tuiwright equivalent                                                                                    |
| ------------------ | ------------------------------------------------------------------------------------------------------- |
| Browser context    | Isolated PTY session plus process, env, cwd, terminal profile                                           |
| Page               | Terminal screen/session                                                                                 |
| DOM                | Screen cell tree plus optional semantic widget tree                                                     |
| Locator            | Text/style/region/role/test-id locator over terminal screen                                             |
| Actionability      | Visible cells, stable bounding box, focusable/enabled semantic state, mouse mode, not masked by overlay |
| Screenshot         | Pixel-rendered terminal viewport/region/scrollback                                                      |
| Video              | Frame-captured terminal recording exported as optimized GIF                                             |
| Trace              | Raw terminal bytes, actions, assertions, screen snapshots, screenshots, logs, process events            |
| Codegen            | Interactive recorder that watches user actions and emits Rust tests                                     |

# 3. Non-negotiable design goals

## 3.1 100% Rust implementation

The core should not shell out to `tmux`, `script`, `asciinema`, ImageMagick, ffmpeg, Node, Python, or browser drivers.

The implementation can still use OS primitives through Rust crates: PTYs on Unix, ConPTY on Windows, process APIs, signals, and terminal IO. `portable-pty` is a good starting point because it provides a cross-platform PTY API and supports terminal dimensions in rows, columns, and pixel dimensions. ([Docs.rs][8]) Crossterm is useful as an interoperability reference because it is a pure-Rust cross-platform terminal manipulation library. ([GitHub][9])

For rendering and artifacts, use Rust libraries: `image` for native Rust image encoding/processing, `gif` for GIF encoding/decoding, `color_quant` or a custom Rust quantizer for palettes, `oxipng` for PNG optimization, and `swash`/`cosmic-text` for pure-Rust shaping/rasterization/font handling. ([Docs.rs][10])

## 3.2 Real user behavior, not fake render calls

A TUI test must drive the actual binary in a PTY. That means testing:

Raw mode behavior.

Alternate screen behavior.

Resize handling and `SIGWINCH`.

Keyboard escape sequences.

Mouse protocols.

Bracketed paste.

Terminal queries and responses.

Color support and `TERM`/`COLORTERM` differences.

Async rendering, spinners, delays, progress bars, and output bursts.

The test must answer: **“What would a user actually see and what would the app actually do?”**

## 3.3 Deterministic visual artifacts

Screenshots and recordings should not depend on a human’s machine, GPU compositor, desktop environment, or installed terminal emulator. Tuiwright should render the emulated terminal itself into pixels using a declared terminal profile.

That gives repeatable CI screenshots and videos.

## 3.4 Graceful black-box operation, excellent white-box operation

A terminal has no DOM. Black-box tests can still be powerful, but true Playwright-style locators need app-provided semantics. Tuiwright should therefore work in layers:

No app changes: text, regex, style, cursor, cell-region, and coordinate locators.

Minimal app changes: `test_id`, role/name, label, enabled/focused/value metadata.

Full app integration: widget tree, state tree, accessibility-like descriptions, stable logical IDs, and action hints.

# 4. Crate layout

I would structure the project as a workspace with narrowly scoped crates:

```text
tuiwright/

    tuiwright-input/           # keyboard/mouse/paste/resize encoders
    tuiwright-locator/         # locator engine over screen + semantic tree
    tuiwright-action/          # click/fill/type/drag/wheel/actionability
    tuiwright-expect/          # assertions, auto-retry, diffs
    tuiwright-render/          # terminal-to-pixels renderer
    tuiwright-snapshot/        # golden snapshots, masks, pixel/cell diffs
    tuiwright-record/          # video frame capture, GIF encoder, compression
    tuiwright-trace/           # trace format, serializer, replay

    tuiwright-cursive/         # Cursive adapter
    tuiwright-cli/             # `cargo tuiwright`, `tuiwright show-trace`
```

The public crate should feel simple:

```rust
use tuiwright::{App, expect, key};

let app = App::command("./my-tui").spawn().await?;
app.get_by_text("Login").click().await?;
expect(app.get_by_text("Success")).to_be_visible().await?;
```

Everything else should be modular so serious users can use the emulator, renderer, recorder, or assertion layer independently.

# 6. PTY and process layer

This layer owns the app process and terminal IO.

Responsibilities:

Spawn commands without shell injection.

Set working directory, environment, terminal size, `TERM`, `COLORTERM`, locale, timezone, and feature flags such as `TUIWRIGHT=1`.

Open a PTY with rows, columns, pixel width, and pixel height.

Stream raw output bytes into the terminal emulator.

Write encoded keyboard/mouse/paste input into the PTY.

Resize the PTY and notify the app correctly.

Handle exit status, signals, kill-on-drop, graceful shutdown, and leaked child processes.

Record raw bytes and process events in the trace.

`portable-pty` can be used initially, with an internal abstraction so the project can later implement direct `rustix`/ConPTY backends if needed. `expectrl` proves the usefulness of a Rust “expect-like” automation model for spawning and interacting with terminal apps, but Tuiwright should go further by adding a full terminal screen model, locator layer, screenshot renderer, and recorder. ([Docs.rs][8])

Example internal API:

```rust
pub trait PtySession: Send {
    async fn read(&mut self, buf: &mut [u8]) -> io::Result<usize>;
    async fn write_all(&mut self, bytes: &[u8]) -> io::Result<()>;
    async fn resize(&mut self, size: TerminalSize) -> io::Result<()>;
    async fn terminate(&mut self, signal: Signal) -> io::Result<()>;
    fn process_id(&self) -> Option<u32>;
}
```

The PTY reader should never throw away raw bytes. Raw bytes are the source of truth and make traces replayable.

# 7. Terminal emulator layer

This is the heart of the project. It must consume terminal output and maintain a faithful state model.

Options:

Use `alacritty_terminal` as the first production-grade core, because it is a Rust library for writing terminal emulators and exposes terminal/grid abstractions. ([Docs.rs][11])

Use `termwiz` concepts for surfaces, cells, truecolor, hyperlinks, and screen deltas; its docs explicitly describe it as support for applications interested in displaying data to a terminal or building a terminal emulator. ([Docs.rs][12])

Use `vte` only as a lower-level parser if building a custom emulator. The `vte` crate parses ANSI/VT state-machine events but explicitly requires a `Perform` implementation to assign meaning to parsed data, so it is not a complete emulator by itself. ([Docs.rs][13])

My recommendation: **start by wrapping/forking `alacritty_terminal` for correctness**, then normalize its grid into Tuiwright’s own stable `Screen` model. Keep a `tuiwright-vt` abstraction so future engines can be swapped.

## Screen model

```rust
pub struct Screen {
    pub size: Size,
    pub cells: Vec<Cell>,
    pub cursor: Cursor,
    pub title: Option<String>,
    pub modes: TerminalModes,
    pub main_scrollback: Scrollback,
    pub alt_screen: Option<AltScreen>,
    pub images: Vec<TerminalImage>,
    pub version: u64,
    pub timestamp: Instant,
}

pub struct Cell {
    pub grapheme: SmolStr,
    pub width: CellWidth,       // Zero, One, Two
    pub fg: Color,
    pub bg: Color,
    pub underline: Option<Color>,
    pub attrs: CellAttrs,       // bold, dim, italic, inverse, hidden, crossed
    pub hyperlink: Option<HyperlinkId>,
    pub image_ref: Option<ImageId>,
    pub dirty: bool,
}
```

The emulator needs to support:

Main and alternate screens.

Scrollback.

SGR styles including truecolor and 256-color palettes.

Cursor position, shape, visibility, blinking.

Line wrapping and wide characters.

Grapheme clusters and combining marks.

Tabs, clears, insert/delete line, insert/delete char.

Bracketed paste mode.

Application cursor/keypad modes.

Focus events.

Mouse tracking modes.

OSC title and hyperlinks.

Terminal queries and replies.

At minimum, it should answer these terminal queries because real TUIs often depend on them:

Cursor position report.

Primary/secondary device attributes.

Terminal size.

Color queries where feasible.

Mode reports.

Kitty keyboard protocol negotiation, eventually.

The emulator must also track terminal modes so the input encoder knows how to encode keys and mouse events.

# 8. Input/action layer

A Playwright-like TUI framework must drive behavior through the same paths as a user:

```rust
app.keyboard().press(key!("Ctrl+C")).await?;
app.keyboard().type_text("hello").await?;
app.keyboard().paste("large text\nwith newlines").await?;
app.mouse().click(42, 10).await?;
app.mouse().drag((10, 5), (60, 5)).await?;
app.mouse().wheel_down(3).await?;
app.resize(140, 40).await?;
```

## Keyboard encoding

The encoder should support:

Printable UTF-8 text.

Control keys: Enter, Escape, Tab, Backspace.

Arrows, Home, End, PageUp, PageDown, Insert, Delete.

Function keys.

Modifiers: Ctrl, Alt, Shift, Super where terminal protocols allow it.

Application cursor mode.

Bracketed paste.

Kitty keyboard protocol as an advanced feature.

A user-facing key macro would help:

```rust
key!("Ctrl+S")
key!("Alt+Enter")
key!("Shift+Tab")
key!("F12")
```

## Mouse encoding

Mouse support must reflect active terminal modes:

X10 mouse.

Normal tracking.

Button-event tracking.

Any-event tracking.

SGR 1006 mouse mode.

Wheel up/down.

Drag with press/move/release.

If the app has not enabled mouse mode, `click()` should either fail with a useful actionability error or allow `force: true`.

```rust
app.get_by_text("Save").click().await?;
// Internally:
// 1. Resolve locator to bounding box.
// 2. Wait until visible and stable.
// 3. Compute center cell.
// 4. Check mouse mode.
// 5. Encode terminal mouse press/release.
// 6. Wait for resulting screen update or idle.
```

# 9. Locator system

The hardest product problem is that a terminal has no DOM. Tuiwright should solve this with a layered locator model.

## 9.1 Black-box locators

These require no app changes:

```rust
app.get_by_text("Save")
app.get_by_regex(r"Project \d+")
app.locator("cell=10,4")
app.locator("row=2")
app.locator("region=10,5,30,8")
app.get_by_style(|s| s.fg == Color::Red && s.bold)
app.get_by_cursor()
```

Black-box locators operate over the visible screen and scrollback. The locator engine builds text runs from cells, handles wide characters, ignores hidden cells by default, and maps matches back to cell bounding boxes.

Text locator behavior should be configurable:

Exact vs substring.

Regex.

Case sensitivity.

Normalize whitespace.

Include/exclude scrollback.

Include/exclude hidden text.

Prefer focused/selected match.

Require unique match or allow `.nth()`.

Example:

```rust
app.get_by_text("Delete")
    .nth(1)
    .click()
    .await?;
```

## 9.2 Geometric locators

Terminals often encode UI structure visually. Tuiwright should support:

```rust
app.region((0, 0), (120, 5))
app.box_with_title("Settings")
app.line_below(app.get_by_text("Username"))
app.near(app.get_by_text("Password")).get_by_text("[ ]")
```

A useful black-box parser can detect:

Box-drawing borders.

Tables.

Focused rows by style contrast.

Buttons such as `[ OK ]`, `<Cancel>`, `Save` with inverted style.

Input fields by cursor and border.

Menus and tabs by separators.

These heuristics are helpful but should never be the only path.

## 9.3 Semantic locators

For real Playwright-level reliability, apps should expose a semantic tree:

```rust
app.get_by_role("button", "Create")
app.get_by_label("Project name")
app.get_by_test_id("submit-project")
app.get_by_role("listitem").filter_text("prod").click().await?;
```

The semantic tree should look like this:

```rust
pub struct SemanticNode {
    pub id: NodeId,
    pub parent: Option<NodeId>,
    pub role: Role,
    pub name: String,
    pub test_id: Option<String>,
    pub bbox: Rect,
    pub state: NodeState,
    pub value: Option<String>,
    pub description: Option<String>,
    pub children: Vec<NodeId>,
}

pub struct NodeState {
    pub visible: bool,
    pub enabled: bool,
    pub focused: bool,
    pub selected: bool,
    pub checked: Option<bool>,
    pub expanded: Option<bool>,
    pub readonly: bool,
}
```

This gives terminals an ARIA-like testing layer without requiring a browser.

## 9.4 Ratatui instrumentation

For Ratatui, I would add an optional adapter:

```rust
use tuiwright_ratatui::{FrameExt, Role};

frame
    .semantic()
    .test_id("create-button")
    .role(Role::Button)
    .name("Create")
    .enabled(app.can_create)
    .render_widget(button, area);
```

Or a lower-friction wrapper:

```rust
Semantic::button("Create")
    .test_id("create-button")
    .render(frame, area, |frame, area| {
        frame.render_widget(button, area)
    });
```

The semantic metadata should not be printed into the terminal because that would pollute output. Instead, when `TUIWRIGHT_SEMANTIC_SOCKET` or `TUIWRIGHT_SEMANTIC_FD` is set, the app streams semantic frames to a side channel. In normal user runs, this code is inert.

This is analogous in spirit to accessibility metadata: visible output remains the source of truth, while semantic metadata makes tests stable and user-centered.

# 10. Actionability and auto-waiting

This is what prevents flaky tests.

Every locator action should auto-wait by default:

```rust
app.get_by_role("button", "Create").click().await?;
```

Internal checks:

Resolve to exactly one target unless `.nth()` or `.first()` is used.

Target has a non-empty bounding box.

Target is visible in the viewport.

Target is stable across at least two screen versions or a quiet interval.

Target is enabled if semantic metadata is available.

Target is not covered by a modal/overlay if semantic tree indicates layering.

If clicking, the app has enabled a mouse protocol or `force` is set.

If filling, the target is editable/focusable or black-box heuristics identify an input region.

If keyboard action, terminal is still alive and process has not exited.

If the check fails, produce a Playwright-like error:

```text
Timeout 5000ms exceeded while clicking get_by_role("button", "Create")

Resolved 1 element:
  role=button name="Create" bbox=(42,12 10x1)

Actionability checks:
  ✓ visible
  ✓ stable
  ✗ enabled: semantic state says disabled
  ✓ receives mouse events

Last screen:
  ...
Trace: artifacts/create-project.trace.zip
Screenshot: artifacts/create-project-failure.png
```

Assertions should also auto-retry:

```rust
expect(app.get_by_text("Saved")).to_be_visible().await?;
expect(app.get_by_test_id("status")).to_have_text("Ready").await?;
expect(app.screen()).to_match_screenshot("dashboard").await?;
```

Timeout categories:

`action_timeout`.

`expect_timeout`.

`spawn_timeout`.

`idle_timeout`.

`shutdown_timeout`.

`artifact_timeout`.

No fixed sleeps should be necessary for normal tests.

# 11. Screenshot system

Tuiwright should support four screenshot types:

## 11.1 Cell snapshot

A lossless logical snapshot:

```json
{
  "size": { "cols": 120, "rows": 36 },
  "cursor": { "x": 14, "y": 22, "visible": true },
  "cells": [
    { "x": 0, "y": 0, "ch": "┌", "fg": "#c0c0c0", "bg": "#000000" }
  ]
}
```

This is best for deterministic diffs.

## 11.2 ANSI snapshot

A compact terminal representation that can be replayed or viewed in a terminal.

## 11.3 SVG/HTML snapshot

A web-friendly, selectable-text artifact that renders cells as positioned spans. Useful for reports and trace viewer.

## 11.4 Pixel screenshot

A high-resolution PNG that looks like a real terminal viewport.

```rust
app.screenshot("dashboard.png", ScreenshotOptions {
    scale: 2.0,
    cursor: CursorCapture::Visible,
    region: None,
    theme: Some("wezterm-dark"),
    ..Default::default()
}).await?;
```

Pixel screenshot options:

Full viewport.

Region.

Locator-level screenshot.

Full scrollback screenshot, analogous to a “full page” screenshot.

Transparent background.

Cursor included/excluded.

Action highlight overlay.

Scale factor.

Terminal profile.

Masks for dynamic regions.

Animations disabled or captured at a chosen frame.

Playwright supports full-page, buffer, and element screenshots; Tuiwright should mirror that vocabulary as viewport, buffer, scrollback, and locator screenshots. ([Playwright][6])

## Rendering pipeline

```text
Screen cells
  → line layout
  → glyph shaping/rasterization
  → style/background/cursor drawing
  → terminal images/sixel compositing
  → scale factor
  → RGBA buffer
  → PNG encoding
  → optional oxipng optimization
```

A renderer profile:

```toml
[profiles.wezterm-dark]
font_family = "JetBrains Mono"
font_size = 16
cell_width = 9
cell_height = 20
line_height = 1.2
theme = "dark"
cursor_shape = "bar"
cursor_blink = false
palette = "wezterm-default"
scale = 2
```

For pure Rust rendering, `swash` is attractive because it is a pure Rust crate for font introspection, shaping, and glyph rendering; `cosmic-text` provides higher-level shaping, font discovery, fallback, layout, and rendering abstractions. ([Docs.rs][14])

The renderer must handle:

Bold, italic, underline, double underline, strikethrough.

Reverse video.

Dim.

Hidden text.

Wide characters.

Combining marks.

Emoji and fallback fonts.

Box drawing.

Powerline glyphs.

Cursor shapes.

Terminal images: Sixel, Kitty graphics, iTerm graphics.

Truecolor and 256-color palettes.

Ligature policy. Many terminal users disable ligatures; the profile should control that.

# 12. GIF screen recording

The recorder should work like Playwright videos, but optimized for terminals. Playwright can record video per test and supports policies like always, retain-on-failure, and first retry; Tuiwright should have the same artifact policies. ([Playwright][15])

Example:

```rust
let app = App::command("./my-tui")
    .recording(Recording {
        path: "artifacts/login.gif".into(),
        mode: RecordingMode::RetainOnFailure,
        fps: 30,
        scale: 2.0,
        highlight_actions: true,
        web_compress: true,
        max_colors: 256,
    })
    .spawn()
    .await?;
```

## Recorder architecture

```text
PTY output bytes
  → emulator state versions
  → frame scheduler
  → changed frame detection
  → pixel renderer
  → frame optimizer
  → GIF encoder
```

Do not render at a naive fixed 60 FPS. Terminal UIs are mostly static. Instead:

Record a new logical frame only when the terminal state changes.

Coalesce bursts of updates.

Preserve real frame durations.

Drop duplicate frames.

Cap frame rate for spinners/animations.

Use dirty rectangles.

Render only changed regions when possible.

## GIF compression strategy

GIF is palette-limited and can get huge at high resolution. The “web compressed” path should include:

Duplicate-frame elimination.

Duration coalescing.

Dirty rectangle frames.

Transparent unchanged regions.

Scene-based global palettes.

Optional per-frame local palettes for color-heavy terminal images.

Palette reuse across adjacent frames.

Temporal dithering tuned to avoid shimmer.

Action overlays drawn only when needed.

Configurable max dimensions.

Configurable color budget.

Configurable lossy tolerance.

The `gif` crate provides Rust GIF encoding/decoding primitives, while `color_quant` provides a NeuQuant-based color quantizer; Tuiwright can start with those and later add a custom cross-frame quantizer optimized for terminal recordings. ([Docs.rs][16]) Gifski is a useful reference point for quality goals because it uses cross-frame palette and temporal dithering ideas to produce high-quality GIFs, but the strict 100%-Rust core should avoid depending on non-Rust native tooling. ([Docs.rs][17])

## Recording modes

```rust
enum RecordingMode {
    Off,
    On,
    RetainOnFailure,
    OnFirstRetry,
    Manual,
}
```

## Output artifacts

For every test:

`test.gif` — optimized high-res recording.

`test.frames/` — optional debug PNG frames.

`test.timeline.json` — frame timestamps, actions, screen hashes.

`test.trace.zip` — full replayable trace.

The GIF can include optional annotations:

Action name subtitle: `click get_by_role("button", "Create")`.

Mouse cursor/click halo.

Keyboard overlay: `Ctrl+S`.

Focused locator outline.

Test step title.

Playwright supports action/test annotations in videos; Tuiwright should do the same for terminal recordings. ([Playwright][15])

# 15. Snapshot and visual regression testing

The framework should support multiple snapshot types:

```rust
expect(app.screen()).to_match_cells("dashboard").await?;
expect(app.screen()).to_match_ansi("dashboard").await?;
expect(app.screen()).to_match_screenshot("dashboard").await?;
expect(app.get_by_test_id("table")).to_match_screenshot("table").await?;
```

Snapshot files:

```text
tests/
  __tuiwright__/
    dashboard.cells.snap.json
    dashboard.ansi
    dashboard.ci-dark.120x36@2x.png
    dashboard.diff.png
```

Important features:

Update snapshots intentionally.

Mask dynamic areas like timestamps, random IDs, progress meters.

Ignore cursor.

Ignore specific style attributes.

Normalize color themes.

Compare only a locator region.

Compare semantic tree.

Generate side-by-side diff.

Generate terminal-readable cell diffs.

Example:

```rust
expect(app.screen())
    .to_match_screenshot_with("dashboard", |s| {
        s.mask(app.get_by_test_id("clock"));
        s.mask(app.get_by_regex(r"\d+ms"));
        s.threshold(0.002);
        s.cursor(CursorCapture::Hidden);
    })
    .await?;
```

The key is to store both a **pixel snapshot** and a **cell snapshot**. Pixel snapshots catch user-visible regressions; cell snapshots explain exactly which character/style changed.

# 16. Recorder/codegen

Tuiwright should include a terminal equivalent of Playwright codegen.

Command:

```bash
cargo tuiwright codegen -- cargo run --bin my-tui
```

It opens a native inspector with an embedded terminal renderer. The user interacts with the TUI. Tuiwright records:

Keys.

Text entry.

Mouse clicks/drags.

Window resizes.

Wait points.

Assertions selected by the user.

It then generates Rust:

```rust
#[tuiwright::test]
async fn recorded_flow() -> tuiwright::Result<()> {
    let app = App::command("cargo")
        .args(["run", "--bin", "my-tui"])
        .size(120, 36)
        .spawn()
        .await?;

    app.get_by_role("button", "New project").click().await?;
    app.get_by_label("Project name").fill("demo").await?;
    app.get_by_role("button", "Create").click().await?;
    expect(app.get_by_text("Project created")).to_be_visible().await?;

    Ok(())
}
```

Locator ranking should mirror Playwright’s philosophy:

1. `get_by_test_id`, if stable and unique.
2. `get_by_role(role, name)`, if semantic metadata exists.
3. `get_by_label`, for inputs.
4. `get_by_text`, if stable and unique.
5. Structural locator: within panel/table/row.
6. Coordinates only as a last resort.

The generated test should include comments when it had to fall back to coordinates:

```rust
// Fallback locator: semantic metadata unavailable.
// Prefer adding `.test_id("delete-button")` in the app.
app.cell(82, 14).click().await?;
```

# 17. Semantic probe design

The semantic probe is the feature that can make Tuiwright truly excellent.

## In-app API

```rust
use tuiwright_probe::{semantic_frame, Role};

semantic_frame(|root| {
    root.window("Main")
        .child(Role::Button, "Create")
        .test_id("create-button")
        .bbox(area)
        .enabled(app.can_create);

    root.child(Role::Textbox, "Project name")
        .test_id("project-name")
        .bbox(input_area)
        .value(&app.project_name)
        .focused(app.focus == Focus::ProjectName);
});
```

## Ratatui adapter

```rust
use tuiwright_ratatui::{SemanticFrameExt, Role};

frame.semantic_widget(area)
    .role(Role::Button)
    .name("Create")
    .test_id("create-button")
    .enabled(app.can_create)
    .render(button);
```

## Transport

When tests run, Tuiwright sets:

```text
TUIWRIGHT=1
TUIWRIGHT_SEMANTIC_FD=4
TUIWRIGHT_RUN_ID=...
```

The app writes compact semantic frames to that FD/socket. Outside tests, the probe does nothing.

Data format:

CBOR or MessagePack for speed.

Schema-versioned.

Includes screen version or render timestamp.

No dependency on Ratatui internals.

No visible terminal pollution.

## Why this matters

Without semantics, a terminal button is just text and color. With semantics, the framework can answer:

Is this button disabled?

Which textbox has focus?

Which list item is selected?

What is the accessible name?

What is the value of this input?

Which region belongs to this widget?

That enables robust locators and actionability.

# 25. Technical risks and solutions

## Risk: Terminal emulation is hard

Solution: do not build the first version from raw `vte` events unless necessary. Wrap a real Rust terminal emulator core, normalize its output, then add Tuiwright-specific metadata. Keep raw byte traces for replay and regression tests.

## Risk: No DOM means weak locators

Solution: black-box locators for zero setup, semantic probe for robust tests, adapters for popular Rust TUI frameworks.

## Risk: Pixel screenshots differ by machine

Solution: deterministic renderer profiles. Do not capture the desktop compositor. Render from the emulator state with declared font/theme/metrics.

## Risk: High-resolution GIFs get large

Solution: terminal-specific recording: frame coalescing, dirty rects, duplicate removal, palette reuse, and action overlays only when needed.

## Risk: Dynamic UIs produce flaky snapshots

Solution: masks, semantic masks, cursor suppression, stable-screen waits, deterministic env, and cell-level diffs.

## Risk: Mouse behavior varies

Solution: track terminal modes and encode mouse events according to the app’s active protocol. Fail clearly if mouse is not enabled unless `force` is set.

## Risk: Rust apps differ architecturally

Solution: make the probe transport framework-agnostic and provide thin adapters, not a giant required runtime.

# 26. What would make this “incredible,” not merely useful

The killer features would be:

**Trace-first debugging:** every failure gives a replayable timeline, not just “expected text missing.”

**High-resolution visual artifacts:** deterministic PNG and compressed GIF that look like a real terminal.

**Semantic TUI testing:** roles, labels, test IDs, states, values, and bounding boxes for terminal widgets.

**Auto-waiting actions:** no more sleeps sprinkled through TUI tests.

**Codegen:** record a real terminal interaction and generate a Rust test.

**Cell + pixel diffs:** explain exactly what changed visually and logically.

**Black-box compatibility:** test any binary, even without source access.

**First-class Ratatui integration:** make the common Rust TUI path effortless.

**CI-native:** no X server, no browser, no Node, no ffmpeg, no tmux.

# 28. Final recommendation

Build Tuiwright around five foundations:

1. **Real PTY execution** so tests exercise the actual binary.
2. **Faithful terminal emulation** so the framework knows what the user sees.
3. **Playwright-style locators/actions/assertions** so tests describe user intent.
4. **Deterministic rendering and recording** so screenshots and GIFs are CI-grade.
5. **Optional semantic probes** so Rust TUIs can become as testable as accessible web apps.

The end state is not “snapshot testing for Ratatui.” It is a full terminal automation platform: **drive flows, wait intelligently, inspect failures, capture screenshots, record GIFs, replay traces, and validate real TUI behavior with user-level confidence.**
