According to a document from 2026-05-04, I’m treating the uploaded **Jankurai v0.7.0 / auditor v0.7.0 / schema bundle v1.5.0 / paper edition 2026.05-ed7** PDF as the latest paper. This draft is much stronger than the prior one: it is shorter, more stack-neutral, has an evaluation section, adds related work, compresses the vibe-coverage registry, and makes the **merge witness** the public primitive. The next leap is positioning: the paper should stop sounding mainly like a careful conformance proposal and become the paper that names a new industry category—**real-time repository conformance for agentic code**. 

# The main diagnosis

The current title is accurate, but it does not yet carry the emotional or market force of the idea. The paper is not only about “trustworthy AI-assisted merge.” It is about **finding, measuring, reducing, and repairing vibe-coding artifacts continuously** so human and agent changes stay aligned with repository truth.

That is a much bigger hook.

The paper’s profound industry claim should be:

> **The repository is the alignment layer.**

Not the prompt. Not the model. Not the IDE. Not the reviewer’s memory. The repository itself should continuously encode ownership, allowed edit scope, proof lanes, generated boundaries, permission limits, evidence receipts, repair queues, and versioned conformance.

The current paper already contains this idea. It says Jankurai turns the pull-request merge decision into a reproducible, versioned audit artifact, with the merge witness binding changed paths, owner routing, proof receipts, evidence, score deltas, missing-evidence decisions, tool version, and commit identity. That is the paper’s most important object. 

But the paper should now make an even sharper public claim:

> **Jankurai finds vibe artifacts and turns them into repairable proof work.**

That phrase is more broadly understandable than “repository conformance standard,” while still being technically serious.

# Revised abstract recommendation

The current abstract is solid and careful. It says AI-assisted coding lowers the cost of plausible code, Jankurai turns merge into a reproducible audit artifact, and conformance is pass/fail while scores are posture signals. That is good. But it still underplays the “find and reduce vibe artifacts” promise. 

A stronger abstract would be:

> AI-assisted coding makes plausible code cheap and ambiguity expensive. Jankurai is a versioned repository conformance standard for finding and reducing **vibe artifacts**: ownerless paths, unmapped proof, hand-edited generated zones, stale contracts, false-green tests, missing security evidence, overbroad agent permissions, unproven UI changes, and unreceipted review claims. Its central artifact is the **merge witness**, a versioned binding among changed paths, owner routing, proof receipts, observed evidence, score deltas, missing-evidence decisions, tool identity, and commit identity.
>
> Jankurai treats the repository as the alignment layer for human- and agent-authored code. A local/PR/CI loop maps every changed path to bounded authority, proof lanes, evidence receipts, repair queues, or expiring exceptions. The result is not a larger prompt or a hosted trust service; it is a reproducible merge decision that can be audited at the current commit.
>
> The paper contributes a stack-neutral repository model with owner maps, test maps, generated-zone policy, stable rule identifiers, version manifests, bounded authority, and proof receipts; a conformance mechanism based on audit reports, repair queues, merge witnesses, and ratchets; and a non-normative reference implementation and reference profile. The claim is bounded: Jankurai does not prove full program semantics, does not require a Rust/TypeScript/PostgreSQL stack, and does not treat scores as merge approval. It proposes a version-controlled standard for turning vibe coding into repairable evidence.

This version gives the paper a sharper public identity while preserving the careful standards posture.

# The strongest conceptual upgrade

Add a formal term:

## Vibe artifact

A **vibe artifact** is any repository residue that makes a change plausible but not auditable.

Examples:

| Vibe artifact                            | Jankurai representation             |
| ---------------------------------------- | ----------------------------------- |
| Changed path has no owner                | `HLT-003-OWNERLESS-PATH`            |
| Changed path has no proof lane           | `HLT-004-UNMAPPED-PROOF`            |
| Generated file edited by hand            | `HLT-002-GENERATED-MUTATION`        |
| DTO/client mirrored manually             | `HLT-007-HANDWRITTEN-CONTRACT`      |
| Business behavior accepted by weak tests | `HLT-008-FALSE-GREEN-RISK`          |
| UI change lacks rendered proof           | `HLT-013-RENDERED-UX-GAP`           |
| Agent has broad tool permission          | `HLT-012-OVERBROAD-AGENCY`          |
| Security claim lacks receipt             | `HLT-027-HUMAN-REVIEW-EVIDENCE-GAP` |

The paper already says vibe coding is a fault class and maps 260 source rows into a registry, with 17 detector-backed and 243 partial rows in the v0.7.0 compact summary. That is exactly the basis for the “vibe artifact” concept. Make it explicit and make it central. 

This would let the paper say:

> Jankurai does not merely criticize vibe coding. It serializes vibe artifacts, assigns stable rule IDs, routes them to proof lanes, and reduces them through repair receipts.

That is the hook.

## 1. Fix remaining version drift inside the paper

The paper front matter says v0.7.0, auditor v0.7.0, schema bundle v1.5.0, and paper edition ed7. But inside the paper, some examples still appear to reference older versions.

Fix these immediately:

| Location                                 | Issue                                                              | Fix                                                                         |
| ---------------------------------------- | ------------------------------------------------------------------ | --------------------------------------------------------------------------- |
| Merge witness JSON example in Section VI | `tool_version` says `jankurai 0.6.1`                               | Change to `jankurai 0.7.0`                                                  |
| Figure 1 caption                         | says policy model for `jankurai 0.4.0`                             | Change to `jankurai 0.7.0`                                                  |

| Front matter vs Appendix B               | front says `schema_bundle_version`; appendix says `schema_version` | Choose one canonical field                                                  |
| Body vs Appendix B                       | body requires `target_stack_id`; appendix uses `target_stack`      | Choose `target_stack_id`                                                    |
| Appendix B command                       | shows `cargo run -p jankurai - versions`                           | Likely should be `cargo run -p jankurai -- versions` or `jankurai versions` |

For a version-controlled standard, internal version consistency is not cosmetic. It is the product.

## 2. Remove the remaining stack-neutrality contradiction

Section V still says a Jankurai-conformant repo should block when “durable truth moves outside Rust/PostgreSQL ownership.” That undercuts the paper’s new stack-neutral positioning.

Replace with:

> durable truth moves outside the declared durable-truth owner, database constraint layer, or approved domain/application boundary

Then, in the reference profile section, say:

> In the Rust/TypeScript/PostgreSQL reference profile, that durable-truth owner is Rust domain/application code plus PostgreSQL constraints.

This preserves the reference profile without making Rust/PostgreSQL normative.

## 3. Put the vibe-artifact loop on page 1 or page 2

The current page 2 has a useful **Agent-Native Merge Loop** table. It should become a visual signature diagram:

```text
Changed path
  ↓
Vibe artifact scan
  ↓
Owner route
  ↓
Proof lane
  ↓
Proof receipt
  ↓
Merge witness
  ↓
Pass / review / block / ratchet fail / release fail
  ↓
Repair queue or expiring exception
  ↓
Reduced vibe-artifact baseline
```

This is the paper’s real-time alignment story.

## 4. Make “real-time” precise

The paper wisely says Jankurai is not an omniscient daemon, but a repeatable local/PR/CI loop that keeps merge evidence current. That should move into the abstract and introduction. 

Define:

> **Real-time repository conformance** means that every changed path can be evaluated against current owner, proof, generated-zone, security, permission, and evidence requirements before merge.

This avoids overclaiming while preserving the power of the idea.

## 5. Add quantitative standard metrics

The paper already has score deltas, RPN, conformance levels, score history, and future research metrics. Add a small metrics table that makes Jankurai feel measurable from day one.

Recommended metrics:

| Metric                         | Meaning                                                                    |
| ------------------------------ | -------------------------------------------------------------------------- |
| **Vibe Artifact Density**      | open HLT findings per changed path, PR, KLOC, or module                    |
| **Proof Coverage Ratio**       | required proof lanes with current valid receipts divided by required lanes |
| **Owner Route Coverage**       | changed paths matched to owner cells divided by changed paths              |
| **Generated-Zone Drift Count** | hand-edited or stale generated outputs                                     |
| **Repair Half-Life**           | median time from finding to accepted repair receipt                        |
| **Context Efficiency**         | useful proof/retrieval success per context-pack token budget               |
| **Witness Completeness**       | required witness fields present and current                                |
| **Alignment Drift**            | delta from accepted conformance baseline                                   |

These metrics make the claim “reduce vibe coding artifacts” auditable.

# The most important reframing

Do not frame Jankurai mainly as:

* an AI coding tool,
* a Rust/TypeScript/PostgreSQL recommendation,
* a CI scanner,
* a vibe-coding critique,
* a merge checklist.

Frame it as:

> **A version-controlled alignment standard for repositories where humans and agents both write code.**

The public story:

> AI made plausible code cheap.
> Jankurai makes ambiguity expensive.
> It finds vibe artifacts, routes proof, emits receipts, witnesses merge, and turns missing evidence into repair.

The standard story:

> A Jankurai-conformant repository emits current, machine-readable artifacts sufficient to audit the merge decision at the current commit.

The developer story:

> Every PR gets a witness.

The security story:

> No authority without scope. No proof without receipt. No merge without evidence.

The agent-efficiency story:

> Stop feeding agents the whole repo. Give them trusted policy, bounded context, mapped proof, and repairable findings.
