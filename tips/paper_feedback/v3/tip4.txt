# The core reframing the paper needs

The current paper says:

> Jankurai turns a pull request merge decision into a reproducible, versioned audit artifact.

That is true and strong. But the larger industry message should be:

> Jankurai continuously detects, routes, proves, and repairs repository drift so human and AI contributors can move faster without accumulating invisible vibe-coding debt.

That phrase, **vibe-coding debt**, may be useful. The paper already defines vibe coding as plausible code accepted without ownership, proof routing, generated-boundary checks, security gates, and repair evidence. It also maps vibe-coding source rows into `agent/vibe-coverage.toml` and reports 260 source rows, 17 detector-backed rows, 243 partial rows, and 0 unmapped rows in the compact v0.7.0 summary. That should become a headline contribution, not just an appendix detail. 

I would introduce a first-class term:

> **Vibe artifact:** a repository artifact, change, test, instruction, generated file, configuration, or review claim that appears plausible but lacks explicit ownership, proof routing, boundary validation, or repair evidence.

Then the standard’s promise becomes concrete:

> Jankurai reduces vibe artifacts by converting them into stable rule IDs, proof lanes, receipts, merge-witness decisions, and repair queues.

That is more industry-grabbing than “a repository conformance standard.”

“Trustworthy merge” is still vital, but it is a checkpoint. The broader standard is about keeping the repo aligned continuously so agents can act efficiently.

## 2. Make “vibe artifact” a formal object

* time to first valid proof
