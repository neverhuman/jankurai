# Biggest remaining issues

## 1. The title is accurate but not powerful enough

The current title is precise but not viral, memorable, or category-defining. It does not say “vibe coding,” “continuous alignment,” “agent efficiency,” or “proof-backed software.” It reads like a standards appendix instead of a new industry movement.

The paper should lead with the thing people feel:

Then name the new standard:

> Jankurai is the anti-vibe coding standard.

Then explain the mechanism:

> continuous alignment through proof receipts, merge witnesses, and repair queues.

## 2. “Trustworthy merge” is too narrow for the full idea

Merge is the strongest enforcement point, but the paper also covers context packs, adapters, hooks, CI modes, UX proof, security evidence, repair receipts, score history, generated zones, and future public registries. That is more than merge. It is **continuous repository alignment**.

Keep merge as the hard gate, but make the larger category:

> **Continuous Code Alignment**
> local → PR → CI → release

## 3. “Real-time” needs a precise definition

You are right that Jankurai can support real-time or near-real-time alignment. But the paper should avoid implying an omniscient daemon if the implementation is local/PR/CI oriented.

Suggested definition:

> **Real-time alignment** means that changed paths can be routed to owners, proof lanes, evidence requirements, and repair actions at edit time, pre-commit time, pull-request time, CI time, and release time. Jankurai does not require a hosted daemon; it requires the repository to make alignment evidence current at the decision point.

This lets you claim real-time usefulness without overclaiming architecture.

## 4. “Agent efficiency” should become measurable

The paper implies agent efficiency through context packs, owner maps, test maps, filtered command output, repair queues, and proof lanes. It should explicitly define an efficiency metric.

Add a subsection called:

> **Agent Alignment Efficiency**

Possible metrics:

| Metric                     | Meaning                                                            |
| -------------------------- | ------------------------------------------------------------------ |
| Route hit rate             | Percentage of changed paths mapped to owners and proof lanes.      |
| Proof selection precision  | Required proof lanes selected without broad unnecessary test runs. |
| Context-pack reduction     | Tokens avoided by path-scoped policy and evidence packs.           |
| Repair locality            | Number of files touched to repair a finding.                       |
| First-repair acceptance    | Percentage of repairs accepted by proof on first rerun.            |
| Missing-evidence detection | Percentage of proof gaps caught before merge.                      |
| Stale-evidence rejection   | Percentage of old receipts rejected against current commit/policy. |
| Agent overreach rate       | Attempts to edit outside authority or generated zones.             |

This turns “maximum agent efficiency” into an auditable research program.

# The industry-impact version of the paper

The profound claim is not:

> AI code needs better review.

Everyone knows that.

The profound claim is:

> **Repositories need a versioned conformance standard that continuously aligns code, agents, humans, tests, generated artifacts, security controls, UI behavior, and merge decisions around auditable evidence.**

That is new. That is standards-level. That is what the paper should make impossible to unsee.

The paper should leave readers with this mental model:

```text
Vibe coding artifact found
        ↓
Owner route assigned
        ↓
Proof lane selected
        ↓
Receipt produced or missing
        ↓
Merge witness decides
        ↓
Repair queue teaches future agents
        ↓
Repository alignment improves
```

And the public message should be:

> **Find vibe coding. Prove the fix. Trust the merge.**

That is the hook. That is the standard. That is the category.
