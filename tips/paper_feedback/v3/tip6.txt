# Highest-priority changes before the next version

## 1. Lead with vibe-coding artifacts, not only merge

The current paper defines vibe coding as accepting plausible code without explicit ownership, proof routing, generated-boundary checks, security gates, and repair evidence. That definition is excellent. 

Now go one step further and define **vibe-coding artifacts**.

Suggested definition:

> A **vibe-coding artifact** is any repository state that lets plausible code evade ownership, proof, boundary, security, UX, or repair evidence. Examples include ownerless paths, unmapped proof, handwritten contract mirrors, generated-zone hand edits, false-green tests, stale exceptions, prompt-policy drift, missing security receipts, and UI changes without rendered evidence.

Then state:

> Jankurai’s purpose is to continuously detect, route, reduce, and prevent vibe-coding artifacts.

That phrase should appear in the abstract, introduction, conclusion, README, and launch material.

## 2. Move the core standard before the language sections

The paper currently places **Languages as Bottleneck Compression** and **Technical Promise Versus Standard Gravity** before the Jankurai standard section. These sections are interesting, but they still delay the main idea.

For broader impact, put the standard first:

1. New Bottleneck: Trustworthy Merge
2. Vibe-Coding Artifacts and Threat Model
3. Continuous Repository Alignment
4. Jankurai Standard and Conformance
5. Merge Witness, Proof Receipts, Repair Queues
6. Evaluation and Conformance Evidence
7. Reference Profile and Stack Discussion

Language and stack rationale should come after the standard is already understood.

## 3. Make the merge witness visual and unavoidable

The merge witness is the paper’s best standard object. It should be on page 1 or page 2 as a figure.

Current formula:

```text

+ required proof receipts + score delta
```

This is excellent. Make it visually central.

```text
     ↓
Score delta + missing evidence
     ↓
Merge witness
     ↓
pass | review | block | ratchet_fail | release_fail
```

This should replace or accompany Table I near the beginning.

## 4. Add an end-to-end demo PR

A standard becomes real when people can see one failing PR become a passing PR.

Add a two-page motivating example:

**PR: agent updates checkout flow**

Changed files:


Agent repairs:

```text
regenerate client
```

Final witness:

```json
  "present_receipts": ["contract", "web", "security"],

```

This would make the entire paper click.

The latest paper does a much better job saying the reference profile is non-normative. That is good. But the stack-ranking chart still visually dominates the middle of the paper.

# The paper’s best possible positioning

The paper should not primarily be positioned as:

> “A versioned repository conformance standard for trustworthy AI-assisted merge.”

That is accurate, but not explosive.

It should be positioned as:

> **A new standard for ending vibe coding by making every AI-assisted merge auditable in real time.**

Or more formally:

> **Jankurai is a continuous repository alignment standard: it detects vibe-coding artifacts, routes them to owners, requires proof receipts, emits merge witnesses, and turns missing evidence into repair queues.**

That is the profound industry claim.

The core idea nobody else has fully standardized is not “AI code needs tests.” The profound idea is:

> **The repository itself must become the alignment layer.**

Not the model. Not the prompt. Not the reviewer’s memory. The repository.

That is the sentence I would build the paper around.
