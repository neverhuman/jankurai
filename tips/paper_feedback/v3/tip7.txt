# Suggested revised abstract

The current abstract is good, but it can be sharper and more audience-facing. I would rewrite it like this:

> AI-assisted coding makes plausible code cheap and vibe-code drift expensive. The scarce engineering act is no longer producing a patch; it is proving that a human- or agent-authored change is aligned with repository ownership, generated-code boundaries, security policy, proof lanes, and release requirements at merge time. Jankurai is a versioned repository conformance standard for turning that merge decision into a reproducible audit artifact.
>
> The standard’s central artifact is the merge witness: a binding among changed paths, owner routes, required proof receipts, observed evidence, missing-evidence decisions, score deltas, commit identity, and tool version. Jankurai defines owner maps, test maps, generated-zone policy, stable rule IDs, version manifests, bounded authority, repair queues, and local/PR/CI proof lanes. Conformance is pass/fail against required artifacts; scores are posture and prioritization signals, not merge approval.
>
> Jankurai is stack-neutral. Its Rust/TypeScript/PostgreSQL profile and CLI are reference implementations, not requirements. The claim is deliberately bounded: Jankurai does not prove full program semantics and does not claim final empirical superiority. Its operational thesis is narrower and testable: repositories claiming conformance must emit current machine-readable evidence strong enough to audit why a change was allowed, blocked, reviewed, or repaired at merge time.

This abstract brings in the hook: **vibe-code drift**.
