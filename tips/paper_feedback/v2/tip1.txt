I reviewed the latest uploaded paper and the public GitHub repository. I could not run the repository locally in this environment because cloning from GitHub failed on DNS resolution, so the code review below is based on the GitHub source tree, README, raw files, schemas, docs, and visible Rust source rather than executed tests. The repository is already much more than a concept: it contains a Rust CLI, a large schema surface, tests/fixtures, docs, adapter surfaces, generated-zone policy, proof/witness/security/repair commands, and a `packages/ux-qa` seed package. The strongest launch risk is not lack of implementation. It is **version drift and standards-readiness polish**.

The latest paper is meaningfully stronger than the prior draft. The center is now much clearer: **Jankurai is a versioned repository standard for auditable merge decisions**. The paper’s strongest line remains “no proof, no merge; no receipt, no trust,” and the strongest formal object is the merge witness: `changed paths + owner route + proof receipts + score delta + missing evidence decision`. That should become the public identity of the standard. The paper’s current front matter says `standard_version=0.6.1`, `schema_version=1.4.1`, and `paper_edition=2026.05-ed6`, and the body now more explicitly separates the standard from the Rust/TypeScript/PostgreSQL reference profile.

The core recommendation is:

**Make Jankurai first a standard, second a conformance suite, third a reference implementation, fourth a stack profile.**

Right now, the paper and repo contain all four, but they are still intertwined. Tier-1 acceptance will come when a skeptical reader can say: “I may not use Rust, and I may not use this CLI, but I understand the Jankurai conformance contract and can implement it independently.”

# Executive assessment

## What is genuinely strong

Jankurai has a real standards-shaped insight: **AI-era software needs auditable merge boundaries, not just better code generation**. The paper correctly argues that plausible code is cheap, but safe merge remains scarce. It also correctly applies that idea to both agents and humans: a vague human PR and a vague agent patch fail at the same boundary when ownership, proof routing, evidence, and repair are missing.

The repository reinforces that the project is not vaporware. The README describes Jankurai as “a trustworthy-merge standard and local audit CLI” for making human/agent changes reproducible at merge time, with no hosted service required. It also documents first-hour adoption through `adopt`, advisory scoring through `audit`, daily flow through `context-pack`, `prove`, `audit`, and `witness`, and explicit adoption levels from observe/agents/full/ratchet. ([GitHub][1])

The codebase also has a broad control-plane surface: the repo contains `agent`, `schemas`, `docs`, `paper`, `crates/jankurai`, `packages/ux-qa`, `.agents`, `.claude`, `.cursor`, `.github`, `contracts`, `db`, examples, references, tips, and tools. That structure matches the paper’s claim that the repository itself becomes the verification interface. ([GitHub][1])

The CLI surface is impressively broad. The Rust CLI exposes commands for audit, adoption, init/update/doctor, context packs, proof plans and proof verification, registry/cell/bench/certify/govern/publish, migration, repair, optimization, exceptions, adapters, agent verification, CI, hooks, issue export, rule explanation, version checks, UX, and security. It also includes safety gates for unsigned proof commands and repair application through explicit environment variables, which is exactly the kind of “bounded authority” the paper argues for. ([GitHub][2])

The repository already contains a large schema catalog: schemas exist for adoption plans, audit policies, boundaries, certification badges, context packs, doctor reports, evidence indexes, exceptions, findings, generated zones, governance policy, merge witnesses, migration, optimization, owner maps, permission profiles, proof lanes, proof plans, proof receipts, proof verification, repair plans/runs/PRs, repo scores, rules, score diffs/trends, security evidence, standard versions, test maps, update plans, and more. That is a major positive for a versioned standard. ([GitHub][3])

The tests directory is also substantial, covering adoption UX, audit smoke tests, boundaries, generated zones, init/doctor golden tests, migration, proof surface, report compatibility, schemas, security evidence, UX QA, version checks, and other surfaces. That gives the project a credible path toward a public conformance suite. ([GitHub][4])

## The main launch blocker

The paper says `0.6.1 / schema 1.4.1 / ed6`, but several repository artifacts still say `0.4.0 / schema 1.2.0 / ed3`. The raw `agent/standard-version.toml` visible from the repo says `standard_version = "0.4.0"`, `paper_edition = "2026.05-ed3"`, `auditor_version = "0.4.0"`, and `schema_version = "1.2.0"`, while the latest paper and some docs say `0.6.1 / 1.4.1 / ed6`. The raw `VERSION` file also says `0.4.0`, and `packages/ux-qa/package.json` says version `0.4.0`. ([GitHub][5])

This matters because the repo already has a `versions` command intended to verify that `VERSION`, Cargo package version, CLI constants, standard documents, companion Markdown, artifact bindings, generated-zone registration, and package versions agree. That is exactly the right mechanism, but it currently exposes the most urgent standard-readiness problem: the public version story must become singular before broad release. ([GitHub][6])

A standard whose first public promise is versioned auditability cannot launch with visible version drift.

## P0. Fix rule-ID mapping errors before publication

The rule-ID system is one of Jankurai’s most important inventions. It makes audit findings durable, routable, and repairable. That means mapping errors are not cosmetic; they attack the standard’s credibility.

Visible issues in the paper:

| Location                                                                       | Current mapping | Problem                                                                              |                                            Suggested mapping |
| ------------------------------------------------------------------------------ | --------------: | ------------------------------------------------------------------------------------ | -----------------------------------------------------------: |
| Table IV, “Authorization or data-isolation drift”                              |  `HLT-005 / db` | Appendix defines HLT-005 as Python product truth. Authz/data isolation is HLT-022.   |                                               `HLT-022 / db` |
| Table IV, “Destructive migration or data-loss hazard”                          |  `HLT-006 / db` | Appendix defines HLT-006 as wrong-layer DB access. Destructive migration is HLT-021. |                                               `HLT-021 / db` |
| Appendix coverage, `tip2:12 Cross-site scripting / unsafe rendering`           |       `HLT-016` | XSS is not supply-chain drift.                                                       | Usually `HLT-023`; sometimes `HLT-013/014` if rendered/a11y. |
| Appendix coverage, `tip2:14 Server-side request forgery`                       |       `HLT-024` | SSRF is usually an input/network boundary issue, not agent-tool supply.              |                                                    `HLT-023` |
| Appendix coverage, `tip5:43 Broken business logic / skipped state transitions` |       `HLT-016` | Business logic/state transition failure is not dependency supply-chain.              |   `HLT-008`, `HLT-022`, or a new business-invariant subrule. |

The paper’s Appendix A defines HLT-021 as destructive migration and HLT-022 as authorization isolation, so the Table IV mismatches are easy to repair.

Add a CI check:

```text
every taxonomy row rule_id MUST exist in rule registry
every rule_id MUST match allowed category group
every top-level taxonomy table row MUST use canonical rule_id
every appendix source-row mapping MUST pass group validation
```

The repo already has rule metadata structures and rule verification surfaces, so this should become enforceable rather than editorial. The Rust rule registry uses rich `RuleSpec` metadata including ID, name, category, TLR, lane, docs URL, owner hint, evidence kind, severity, status, standard section, cap key, and confidence policy. ([GitHub][8])

## P0. Replace “absolute coverage” with “detector-backed coverage”

Appendix A still says:

> “Green = absolute, yellow = partial, red = none.”

That wording should change. Security and audit communities will punish the word “absolute.” Even a deterministic detector has false positives, false negatives, scope limits, and fixture limits. The body’s better phrase is “detector-backed Jankurai control plus proof/report evidence.” Use that everywhere.

Recommended labels:

| Current     | Better                                               |
| ----------- | ---------------------------------------------------- |
| absolute    | detector-backed                                      |
| partial     | policy-backed / fixture-needed                       |
| none        | unmapped                                             |
| no red rows | every source row is mapped to a reviewed rule family |

Suggested replacement:

> Green means detector-backed evidence exists in the current auditor for a narrowed issue family. Yellow means a reviewed rule family exists but still needs fixture, CI, governance, or semantic evidence before it can be treated as detector-backed. Red means no reviewed rule family exists.

This is more credible and more standards-ready.

## P0. Move the 13-page source-row coverage table out of the main paper

Pages 18–30 of the PDF are dominated by Table XXIII. The table is useful as generated evidence, but it is too large, too repetitive, and too visually generated for the main paper. It also makes the paper feel like a data dump rather than a tier-1 standards proposal.

Keep a one-page summary in the paper:

| Coverage class  | Count | Meaning                                      |
| --------------- | ----: | -------------------------------------------- |
| detector-backed |     N | current auditor emits evidence               |
| policy-backed   |     N | rule family exists, fixture/evidence missing |
| unmapped        |     0 | no reviewed rule family                      |

Then show 8–12 representative rows. Move the full table to:

```text

target/jankurai/vibe-coverage.csv
docs/generated/vibe-coverage.html
```

The paper should cite the generated registry by version, commit, schema, and digest.

## P0. Fix formatting artifacts

There is a visible stray `{` at the bottom of page 30 before Appendix B. It is especially damaging because the paper is about machine-readable correctness.

Other formatting fixes:

* Remove the stray purple mark near the subtitle on page 1.

The repo currently shows no GitHub releases. That is acceptable for internal development, but not for a versioned standard asking for trust. # What the repository changes about the paper feedback

The repository makes the paper more credible in several ways, but it also creates new responsibilities.

## The paper can claim more implementation reality

The paper already says Jankurai is a standard plus reference implementation. The repo supports that claim. It has:

* a Rust CLI under `crates/jankurai`;
* schemas for core artifacts;
* docs for architecture, boundaries, install, testing, merge witness, security tools, rolling score, source hierarchy, and migration;
* examples and tests;

That means the paper should include a short “Implementation Status” section:

| Component               | Status                              | Evidence |
| ----------------------- | ----------------------------------- | -------- |
| CLI                     | implemented                         |          |
| schemas                 | implemented                         |          |
| tests                   | partial but substantial             |          |

| UX QA                   | seed implementation                 |          |

## The code exposes version drift more sharply

The paper’s versioned-standard claim is now strong enough that the repo must live up to it. The visible mismatch between `0.6.1` paper/docs and `0.4.0` manifest/version/package files is the largest credibility issue. ([GitHub][5])

## The code suggests the standard can be more schema-first

The schema directory is broad enough that the paper should stop merely describing artifacts and start naming their schemas. For example:

```text
schemas/repo-score.schema.json
schemas/merge-witness.schema.json
schemas/proof-receipt.schema.json
schemas/evidence-index.schema.json
schemas/owner-map.schema.json
schemas/test-map.schema.json
schemas/generated-zones.schema.json
schemas/security-evidence.schema.json
```

The merge-witness schema already requires fields such as schema version, command, generated time, repo, git state, changed paths, route decisions, generated-zone touches, required lanes, available receipts, missing evidence, scores, caps, findings, decision, and next repair. That is very strong. ([GitHub][9])

But the proof receipt schema is currently too permissive for higher conformance. It requires only lane, command, exit code, elapsed time, and artifacts, while allowing additional properties. For HL3 and above, require git head, plan digest, command digest, artifact digests, generated time, run ID, tool versions, and environment/provenance fields. ([GitHub][10])

# Paper-level restructuring recommendation

The current paper still reads partly like a manifesto, partly like a standard, partly like a stack ranking, partly like a CLI manual, and partly like generated appendix output. The tier-1 version should separate those roles.

Recommended artifact family:

1. **Paper**
   12–18 pages. Persuasive, citable, conceptually sharp.

2. **Normative standard**
   Versioned spec with MUST/SHOULD/MAY, conformance levels, schemas, artifact contracts, rule lifecycle, compatibility rules.

3. **Conformance suite**
   Fixtures, expected reports, pass/fail criteria, certification process.

4. **Reference implementation**
   Rust CLI, UX QA package, GitHub Actions, docs.

5. **Reference profiles**
   Rust/TypeScript/PostgreSQL default profile plus Go/.NET/JVM/TypeScript/Python/Rails conformant alternatives.

The paper should not carry the full registry or CLI catalog. It should point to them.

Suggested revised paper structure:

1. **Introduction: Trustworthy Merge Is the Bottleneck**
2. **Definitions, Threat Model, and Non-Goals**
3. **Jankurai Standard Model**
4. **Conformance Levels, Artifacts, and Merge Witnesses**
5. **Fault Taxonomy and Stable Rule IDs**
6. **Implementation Status and Reference CLI**
7. **Agent Adapters and Bounded Authority**
8. **Continuous Proof, Security, and Rendered UX**
9. **Migration, Versioning, and Governance**
10. **Reference Profiles and Stack Sensitivity**
11. **Evaluation Plan**
12. **Limitations**
13. **Conclusion**

This order puts the standard before the stack.

# Abstract

## What works

The abstract is much improved. It clearly states the bottleneck shift: typing plausible code is not scarce; trustworthy merge is scarce. It also states three contributions: repository standard, conformance/audit mechanism, and separation of standard from reference profile. The bounded-claim paragraph is very good and should stay.

## What to improve

The abstract still lists too many artifact names in one breath. It should be easier for an outside reader to remember the contribution.

Recommended abstract shape:

1. Problem.
2. Standard object.
3. Conformance mechanism.
4. Reference implementation.
5. Bounded claim.

Suggested revision:

> AI-assisted coding shifts the bottleneck in software engineering from producing plausible code to deciding whether a change is safe to merge. Jankurai is a versioned repository standard for making that merge decision auditable. It treats every human- or agent-authored change as a bounded request that must resolve to ownership, proof lanes, evidence receipts, and either repair or an expiring exception.
>
> This paper contributes: (1) an agent-native repository model with owner maps, test maps, generated-zone policy, stable rule identifiers, version manifests, and bounded authority; (2) a conformance mechanism based on local, pull-request, and CI proof lanes that emit audit reports, proof receipts, evidence indexes, repair queues, and merge witnesses; and (3) a reference implementation and profile for Rust, TypeScript/React/Vite, PostgreSQL, generated contracts, and bounded Python AI/data service work.

It borrows proven standards mechanics—normative terms, version compatibility, maturity states, runnable conformance, current/previous artifact handling, and explicit supersession—and applies them to repository structure.

# Section I — New Bottleneck: Trustworthy Merge

## What works

This is the conceptual heart of the paper. The clarification that humans are not the bug because they wrote software, but because repositories depended on human memory and folklore, is important and effective. The operating loop is excellent:

```text
intent -> bounded authority -> proof lanes
-> evidence -> repair or exception expiry
-> reusable primitives
```

That loop should become a figure and probably the standard’s first diagram.

## What to improve

The section currently introduces the stack too early. The first section should sell the standard, not the reference profile. The fourth claim about Rust/TS/Postgres should move later or be framed as “this paper includes one reference profile.”

Suggested rewrite of the four claims:

1. AI-assisted development makes plausible code cheap.
2. The scarce act becomes auditable merge.
3. Auditable merge requires repository artifacts: ownership, proof routing, generated-zone policy, evidence, repair, and versioned conformance.
4. Jankurai defines those artifacts and demonstrates them through a reference implementation and profile.

Then later:

> Section VIII evaluates a reference profile. That profile is not a condition of standard conformance.

## Add a concrete PR example immediately

Example:

> A PR touches `apps/web/src/generated/client.ts`. In a conventional repository, a reviewer decides whether the diff “looks right.” In a Jankurai repository, the generated-zone manifest rejects the hand edit unless the source contract changed, the contract lane ran, the regenerated digest matches, and the merge witness records the receipt.

That small example would make the whole paper click.

## Recommended figure

**Figure 1: Trustworthy Merge Loop**

```text
Intent
  ↓
Bounded authority
```

# Section VI — Jankurai Standard and Conformance

## What works

This is the most important section. It defines the standard, progressive install levels, HL0–HL5, required HL3 artifacts, stable rule IDs, audit output, repair queues, and merge witness.

The paper’s HL3 requirement list is strong:

* root agent router,
* `agent/standard-version.toml`,
* `agent/owner-map.json`,
* `agent/test-map.json`,
* `agent/generated-zones.toml`,
* one-command fast lane,
* audit JSON,
* audit Markdown,
* proof receipts or evidence index,
* ordered `agent_fix_queue`,
* version bindings for standard, auditor, schema, paper, and stack.

The merge witness equation is excellent and should be the centerpiece.

## What to improve

### Move this earlier

This should be Section III or IV. Readers should see the formal contribution before language/stack ranking.

### Add a conformance matrix

Create a table like:

| Artifact          |    HL1 |     HL2 |                HL3 |         HL4 |     HL5 |
| ----------------- | -----: | ------: | -----------------: | ----------: | ------: |
| root router       | SHOULD |    MUST |               MUST |        MUST |    MUST |
| owner map         | SHOULD |    MUST |               MUST |        MUST |    MUST |
| test map          | SHOULD |    MUST |               MUST |        MUST |    MUST |
| generated zones   | SHOULD |    MUST |               MUST |        MUST |    MUST |
| audit JSON/MD     |   MUST |    MUST |               MUST |        MUST |    MUST |
| proof receipts    |    MAY |  SHOULD |               MUST |        MUST |    MUST |
| merge witness     |    MAY |  SHOULD | SHOULD/MUST for PR |        MUST |    MUST |
| score floor       |     no | limited |                yes |         yes | release |
| ratchet           |     no |      no |           optional |        MUST |    MUST |
| signed receipts   |     no |      no |             SHOULD | SHOULD/MUST |    MUST |
| release artifacts |     no |      no |                 no |      SHOULD |    MUST |

### Define “conformant” more sharply

Current text says repositories below HL3 should not market themselves as conformant. Good. Make this a formal rule:

> Only HL3 or higher MAY use “Jankurai conformant.” HL1 and HL2 MAY say “Jankurai advisory” or “Jankurai guarded.”

### Separate score from conformance

Add:

> A score is a posture measurement. Conformance is a pass/fail claim against required artifacts, schemas, and gates. A repository can score highly and still fail conformance if a required artifact is missing.

This should be repeated because the paper contains both score and conformance language.

### Make witness mandatory for PR conformance

At HL3, I would make merge witness mandatory for PR-gated repos. It is the paper’s most important primitive.

Suggested:

* HL1: audit report sufficient.
* HL2: audit report + generated-zone/security caps.
* HL3: audit report + proof receipts/evidence index + merge witness.
* HL4: merge witness + ratchet.
* HL5: release witness + signed receipts/provenance.

### Add receipt freshness rules

A proof receipt is invalid if:

* it was generated for a different git head,
* it lacks artifact digests,
* it references a command not in `proof-lanes.toml`,
* it omits changed paths,
* it is older than a changed file or baseline,
* it was generated in local mode but used as release proof,
* it lacks required security/UX/DB evidence for the changed path,
* it references a generated zone without source regeneration.

This should become a normative paragraph.

# Section VII — 100-Point Stack Rubric

# Section XVII — Conclusion

## What works

The conclusion is strong and memorable. “Anything less is just faster vibe coding” is a good public ending.

## What to improve

For a standard, end with collaboration and independent implementation.

Suggested conclusion:

> Jankurai’s claim is not that one model, tool, or stack can make software safe. Its claim is that AI-era repositories need a versioned merge interface: ownership, bounded authority, proof lanes, evidence receipts, repair queues, and conformance tests. The next step is independent implementation, empirical calibration, and public governance. No proof, no merge; no receipt, no trust.

This keeps the punch but invites adoption.

# Appendix A — Rule IDs and Conformance Evidence

## What works

Stable HLT IDs are one of the strongest parts of the standard. The rule list HLT-001 through HLT-027 gives future agents and humans durable repair handles.

The repo code also appears to represent rules with structured metadata rather than only prose, which is the right direction. ([GitHub][8])

## What to improve

### Add rule metadata fields

Each rule should have:

```text
id
title
family
status
introduced_in
required_from_hl
default_lane
default_severity
detector_status
required_evidence
repair_template
false_positive_guidance
false_negative_guidance
supersedes
superseded_by
schema
fixtures
```

### Split large rule families

Some rules are too broad. HLT-008 covers false-green risk, weak tests, business logic, model/eval drift, and skipped transitions. Consider subcodes:

```text
HLT-008A-FALSE-GREEN-TEST
HLT-008B-BUSINESS-INVARIANT-GAP
HLT-008C-MOCK-ONLY-PROOF
HLT-008D-MODEL-EVAL-DRIFT
```

Or keep HLT-008 as a family and define stable child findings.

### Move full source-row coverage out of the PDF

The giant table overwhelms the paper and repeats the same yellow gap text. Keep:

* rule list,
* summary counts,
* top 10 examples,
* link/digest to generated registry.

### Replace repeated gap prose

Instead of repeating:

> “needs issue-specific fixture, CI artifact, or governance receipt evidence…”

Use coded gap reasons:

```text
missing-fixture
missing-ci-artifact
needs-negative-test
needs-governance-receipt
org-level-only
needs-schema
needs-detector
```

Then explain them once.

# Appendix B — Versioned Artifact Manifest

## What works

The manifest concept is exactly right. A standard must bind paper source, rendered PDF, Markdown companion, standard docs, schemas, auditor version, and generated artifacts.

The repo also has a `versions` verifier that compares version sources, which is the right technical mechanism. ([GitHub][6])

## What to improve

Add:

```toml
digest = "sha256:..."
schema = "schemas/standard-version.schema.json"
canonical = true
generated = true
supersedes = "0.6.0"
previous = "0.6.0"
compatibility = "patch|minor|major"
license = "..."
verification_command = "cargo run -p jankurai -- versions"
```

Also normalize `target_stack` versus `target_stack_id`. The paper uses both forms in different places. Use one canonical name. I recommend `target_stack_id`.

# Appendix E — Command Examples

## What works

The command examples are useful, especially for agents.

## What to improve

Move most of the command catalog to docs. In the paper, keep only:

1. first-hour adoption,
2. advisory audit,
3. proof plan/prove/verify,
4. merge witness,
5. security strict lane,
6. repair-plan dry run.

Recommended body example:

```bash
jankurai adopt . --profile auto --mode observe
jankurai init . --level agents --dry-run
jankurai audit . --mode advisory
jankurai lane . --changed apps/web/src/checkout.tsx
jankurai prove . --changed apps/web/src/checkout.tsx
jankurai proof-verify .
jankurai witness . --changed-from origin/main
```

Then Appendix E or docs can hold the full catalog.

Also add a generated command verification test:

```bash
jankurai docs verify-commands
```

It should parse README, paper, docs, `test-map.json`, and examples to ensure commands match the CLI.

# Suggested public positioning

Do not lead with “Rust wins” or “humans were the bug.” Lead with:

> AI made plausible code cheap. Jankurai makes merge evidence auditable.

Then:

> Jankurai is a versioned repository standard for proof-carrying pull requests.

Then:

> No proof, no merge. No receipt, no trust.

Best adoption-friendly phrasing:

> Jankurai is not anti-human. It is anti-tribal-memory.
> Jankurai is not anti-agent. It is anti-unbounded-agent.
> Jankurai is not anti-stack-diversity. It is anti-unowned-boundary.
> Jankurai is not a proof of all correctness. It is a standard for reproducible merge evidence.

# Final verdict

The latest paper has the right core and the repo is much stronger than a concept draft. The project already contains many of the ingredients of a real standard: version manifests, schemas, CLI commands, docs, tests, rule IDs, proof receipts, evidence indexes, merge witnesses, repair queues, UX QA, security evidence, and adapter verification surfaces. ([GitHub][1])

The next step is discipline:

1. **Fix version truth.**
2. **Fix rule mappings.**
3. **Separate standard from reference profile.**
4. **Move generated registry material out of the main paper.**
5. **Harden proof/witness schemas.**
6. **Publish release artifacts and conformance fixtures.**
7. **Invite independent implementations.**

The most powerful version of Jankurai is not “the best AI coding stack.” It is:

**A versioned, auditable merge standard for agent-native repositories.**

That is the category-defining idea. Keep the stack profile, but make the merge witness the star.
