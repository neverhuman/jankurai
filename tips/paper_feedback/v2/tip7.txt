# Executive verdict

Jankurai has a strong and timely thesis:

> **AI coding makes plausible code cheap; trustworthy merge becomes the scarce engineering act.**

That is still the central insight. The paper’s most important standardizable contribution is the claim that a repository can expose enough versioned, machine-readable structure for a merge decision to become reproducible: owner maps, test maps, generated-zone policy, proof lanes, proof receipts, evidence indexes, audit reports, merge witnesses, repair queues, exceptions, score history, and release gates.

The repo substantially strengthens that claim. It contains many schemas under `schemas/`, including merge-witness, proof-receipt, repo-score, evidence-index, repair, security, UX QA, governance, certification, public-evidence, context-pack, owner-map, test-map, generated-zones, and more. That schema surface is exactly what a serious standard needs. ([GitHub][2])

But the paper is **not yet tier-1** because it still has four blockers:

1. **It does not yet present Jankurai as a crisp conformance standard first.** The stack profile, language argument, streaming discussion, and vibe-coding appendix still compete with the core standard.

2. **It lacks an Evaluation / Early Evidence section.** A tier-1 standards paper needs case studies, fixtures, conformance tests, or measured outcomes.

3. **The PDF still has serious formatting and credibility issues.** The title page remains too mascot-heavy, Figure 1 still says “jankurai 0.4.0” while the paper is v0.6.1, Appendix A is visually overwhelming, and page 30 still has the stray `{` artifact.

4. **The repo needs release/governance hardening before the phrase “versioned standard” will feel fully earned.** The GitHub repo currently shows **no published releases**, despite the paper’s heavy emphasis on versioned standards. ([GitHub][1])

The project is very promising. The next phase should be: **turn Jankurai from a powerful repo + paper into a standard distribution.**

# Top recommendations to make this a tier-1 version-controlled standard

## 1. Reframe the paper around the standard artifact suite

Right now, the paper’s center of gravity is split between:

* AI-era engineering thesis,
* vibe-coding taxonomy,
* standards proposal,
* stack ranking,
* Rust/TS/Postgres reference architecture,
* streaming/Kafka positioning,
* implementation command guide.

The next revision should make **the artifact suite** the unavoidable center:

```text
Jankurai Core Standard
  ├── Normative prose
  ├── Stable rule IDs
  ├── JSON schemas
  ├── Conformance levels
  ├── Conformance fixtures
  ├── Reference auditor
  ├── Merge witness format
  ├── Proof receipt format
  ├── Repair queue format
  ├── Evidence index format
  └── Version/governance process
```

The paper should say, early and repeatedly:

> **A Jankurai-conformant repository is one whose merge decision can be reproduced from versioned artifacts.**

That is the standard.

## 2. Separate “Jankurai Core” from “Jankurai Reference Profiles”

The paper says it separates the standard from the reference profile, but structurally it still spends a lot of its main body arguing for Rust + TypeScript/Vite/React + PostgreSQL + bounded Python. That stack argument is valuable, but it should be explicitly named as a profile, not as the standard.

Recommended naming:

```text
Jankurai Core v0.7
Jankurai Profile: rust-ts-vite-react-postgres-bounded-python v0.7
Jankurai Profile: go-ts-postgres v0.7
Jankurai Profile: dotnet-ts-postgres v0.7
Jankurai Profile: jvm-ts-postgres v0.7
```

The core rule should be:

> A profile may differ in language or stack, but it must provide equivalent owner routing, generated-boundary enforcement, proof lanes, security evidence, repair receipts, and merge witnesses.

This would dramatically improve acceptance among Go, JVM, .NET, Rails, Python, and enterprise teams.

## 3. Publish versioned GitHub releases

This is critical. The paper claims Jankurai is versioned, but the GitHub repo currently shows **no published releases**. That is a credibility problem for a version-controlled standard. ([GitHub][1])

For v0.7, publish a release with:

```text
jankurai-standard-v0.7.0/
  SPEC.md
  CHANGELOG.md
  COMPATIBILITY.md
  GOVERNANCE.md
  CONFORMANCE.md
  SECURITY_MODEL.md
  schemas/
  fixtures/
  examples/
  reference-auditor/
  paper.pdf
  paper.md
  release-manifest.json
  checksums.txt
```

Each release should include:

* Git tag.
* Release notes.
* Schema versions.
* Auditor version.
* Compatibility statement.
* Migration notes.
* Known limitations.
* Checksums.
* Public conformance fixture results.
* Badge JSON schema.

## 4. Add a real conformance suite

The repo has many schemas, but a standard becomes serious when people can run a conformance suite. Add a top-level conformance area:

```text
conformance/
  README.md
  runner/
  fixtures/
    generated-zone-mutation/
    unmapped-proof/
    ownerless-path/
    secret-sprawl/
    destructive-migration/
    handwritten-contract/
    missing-ux-evidence/
    prompt-injection-policy/
    overbroad-agency/
    ci-unpinned-action/
  expected/
    generated-zone-mutation.repo-score.json
    generated-zone-mutation.merge-witness.json
```

The paper should include one table:

| Fixture                   | Expected finding                | Expected decision              |
| ------------------------- | ------------------------------- | ------------------------------ |
| `generated-zone-mutation` | `HLT-002-GENERATED-MUTATION`    | block                          |
| `ownerless-path`          | `HLT-003-OWNERLESS-PATH`        | block at HL3                   |
| `unmapped-proof`          | `HLT-004-UNMAPPED-PROOF`        | block at HL3                   |
| `secret-sprawl`           | `HLT-010-SECRET-SPRAWL`         | block                          |
| `destructive-migration`   | `HLT-021-DESTRUCTIVE-MIGRATION` | block                          |
| `missing-ux-evidence`     | `HLT-013-RENDERED-UX-GAP`       | review/block depending profile |

This is the difference between “interesting project” and “standard candidate.”

## 5. Make the merge witness the central object

The repo’s merge-witness schema is already a strong artifact. It requires fields such as schema version, command, generated time, repo, git, changed paths, route decisions, generated-zone touches, required lanes, available proof receipts, missing evidence, current score, caps, findings, decision, and next repair. ([GitHub][3])

The next paper should open with this exact object:

```json
{
  "schema_version": "1.4.1",
  "command": "jankurai witness",
  "changed_paths": ["apps/web/src/generated/client.ts"],
  "route_decisions": [
    {
      "path": "apps/web/src/generated/client.ts",
      "owner": "contracts",
      "lane": "contract"
    }
  ],
  "generated_zone_touches": ["apps/web/src/generated/client.ts"],
  "required_lanes": ["contract"],
  "available_proof_receipts": [],
  "missing_evidence": ["regeneration receipt"],
  "new_findings": ["HLT-002-GENERATED-MUTATION"],
  "decision": "block",
  "next_repair": "change contracts/openapi/api.yaml and regenerate"
}
```

Then say:

> **This is the new merge primitive.**

Everything else flows from that.

## 6. Harden proof receipts

The current proof-receipt schema is a good start, but it is too permissive for a standard-grade artifact. It requires only `lane`, `command`, `exit_code`, `elapsed_ms`, and `artifacts`, while fields like `generated_at`, `git_head`, `run_id`, `plan_digest`, `command_digest`, `log_sha256`, and `artifact_digests` are optional. It also allows `additionalProperties: true`. ([GitHub][4])

For HL3+, make these required:

```json
{
  "schema_version": "1.4.1",
  "standard_version": "0.7.0",
  "auditor_version": "0.7.0",
  "receipt_id": "...",
  "run_id": "...",
  "repo": "...",
  "git_head": "...",
  "dirty_worktree": false,
  "lane": "contract",
  "command": "just contract-check",
  "command_digest": "sha256:...",
  "plan_digest": "sha256:...",
  "started_at": "...",
  "finished_at": "...",
  "elapsed_ms": 1234,
  "exit_code": 0,
  "changed_paths": ["..."],
  "artifacts": [
    {
      "path": "target/jankurai/contract.log",
      "sha256": "..."
    }
  ],
  "rules_covered": [
    {
      "rule_id": "HLT-002-GENERATED-MUTATION",
      "status": "passed"
    }
  ]
}
```

Use `additionalProperties: false` for the base schema and allow extensions only under a namespaced field such as:

```json
"extensions": {
  "com.example.extra": {}
}
```

That gives vendors room to innovate without breaking the standard.

## 7. Add `conformance_level` to report schemas

The paper’s HL0-HL5 conformance model is central, but I did not find `conformance_level` in the repo-score schema. The schema requires many important fields, including standard version, auditor version, schema version, paper edition, target stack, repo, run ID, score, decision, git, policy, proof receipts, findings, and repair queue, but conformance level appears absent. ([GitHub][5])

Add this to every top-level audit report, merge witness, public evidence bundle, and badge:

```json
{
  "claimed_conformance_level": "HL3",
  "observed_conformance_level": "HL2",
  "conformance_decision": "fail",
  "conformance_blockers": [
    "missing proof receipts",
    "unmapped changed path"
  ]
}
```

This is essential because a score is not the same thing as conformance.

## 8. Remove or rename hidden `--yolo`

The CLI contains a hidden `--yolo` flag tied to init bootstrap behavior. Even hidden, this is a rhetorical liability for a standard that is explicitly trying to replace vibe coding with auditable merge. ([GitHub][6])

Replace it with something boring and auditable:

```text
--bootstrap-commit
--unsafe-bootstrap-commit
--allow-bootstrap-commit
```

Then require:

```text
JANKURAI_ALLOW_BOOTSTRAP_COMMIT=1
```

and emit a bootstrap receipt.

A serious standard should not contain a hidden “yolo” path, even as a convenience alias.

## 9. Dogfood CI hardening

The paper includes `HLT-020-CI-HARDENING-GAP`, including unpinned actions as a concern. The repo workflow currently uses version tags such as `actions/checkout@v4`, `dtolnay/rust-toolchain@stable`, and `actions/setup-node@v4`. It also runs audit in ratchet mode writing to `agent/repo-score.json` without an obvious `--baseline` argument in the workflow line. ([GitHub][7])

For a standard project, dogfood the strictest CI posture:

```yaml
permissions:
  contents: read
  security-events: write
  actions: read

steps:
  - uses: actions/checkout@<commit-sha>
  - uses: dtolnay/rust-toolchain@<commit-sha>
```

Also make ratchet baseline explicit:

```bash
cargo run -p jankurai -- audit . \
  --mode ratchet \
  --baseline agent/repo-score.json \
  --json target/jankurai/repo-score.json \
  --md target/jankurai/repo-score.md
```

Do not write generated CI candidate output to tracked `agent/repo-score.json` unless the workflow is intentionally updating a baseline.

## 10. Split large implementation files to match the small-file/repair-locality doctrine

The repo’s `main.rs` is 1,764 lines / 52.7 KB, and `audit/mod.rs` is 1,293 lines / 49.1 KB. ([GitHub][6])

That conflicts with the paper’s own doctrine that agent-native repos should reduce review surface and make repair locality explicit. It is not fatal, but it is an easy criticism.

Recommended split:

```text
crates/jankurai/src/cli/
  mod.rs
  args.rs
  dispatch.rs
  audit_args.rs
  proof_args.rs
  init_args.rs
  security_args.rs

crates/jankurai/src/audit/
  mod.rs
  engine.rs
  decision.rs
  policy.rs
  score.rs
  fingerprints.rs
  report_builder.rs
  findings.rs
  repair_queue.rs
```

Then add a Jankurai self-rule:

```text
HLT-028-IMPLEMENTATION-SPRAWL
The reference auditor must route large files through owner/test maps
and split command dispatch from policy logic when file-size thresholds
are exceeded.
```

Or keep it internal, but dogfood the principle.

## Abstract

**What works:**
The abstract is much stronger than a typical manifesto. It clearly says the bottleneck is no longer typing plausible code but making merge decisions reproducible. It also explicitly separates standard and reference profile, which is exactly the right direction.

**What to improve:**
It still lists many artifacts in one dense sentence. That is accurate, but the abstract should sharpen the one contribution reviewers will remember:

> **Jankurai makes pull-request merge decisions reproducible by requiring versioned proof artifacts.**

Suggested rewrite:

> AI coding shifts software engineering’s bottleneck from producing plausible code to deciding whether a change can be trusted at merge time. Jankurai is a versioned repository standard that makes that decision reproducible. A conformant repository exposes machine-readable ownership, proof routing, generated-zone policy, version manifests, proof receipts, evidence indexes, repair queues, and merge witnesses. These artifacts bind human- and agent-authored changes to the same rule: no proof, no merge; no receipt, no trust. The standard defines conformance levels, stable rule identifiers, local/PR/CI proof lanes, audit reports, ratchets, and release gates. A separate reference profile uses Rust, TypeScript/React/Vite, PostgreSQL, generated contracts, and bounded Python, but alternate stacks can conform when they provide equivalent evidence. Jankurai does not prove all program semantics and does not claim institutional equivalence to established standards bodies. Its narrower claim is testable: a repository claiming conformance must emit current artifacts sufficient to audit the merge decision.

## Index Terms

Current index terms are reasonable but too stack-heavy. Add terms that name the actual novelty.

Recommended index terms:

> AI-assisted software engineering, agentic coding, trustworthy merge, repository conformance, proof receipts, merge witnesses, continuous audit, generated contracts, software supply chain, repository control plane, repair queues, Rust, TypeScript, PostgreSQL.

## I. New Bottleneck: Trustworthy Merge

**What works:**

intent -> bounded authority -> proof lanes -> evidence

Start with the technical contribution, then introduce the provocative thesis. The paper currently leads with “Humans were the bug,” then explains the nuance. For broader acceptance, invert it:

Then use the blunt slogan.

Generated-zone policy: hand edits forbidden

Evidence found: none

## II. Definitions and Threat Model

**What works:**
The definitions are operational and useful. “Vibe coding,” “agent-native engineering,” “proof lane,” “proof ledger,” “repair queue,” “generated zone,” “repair receipt,” and “centerline drift” are all valuable terms.

**What to improve:**
Turn the definitions into a compact glossary table. Reviewers and implementers need to scan terms quickly.

Suggested table:

| Term                    | Normative meaning                                                                                                                            |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| Agent-native repository | A repository whose ownership, edit surface, proof lanes, and repair evidence are discoverable by humans and agents without tribal knowledge. |
| Proof lane              | A deterministic command or CI job mapped to a path or change class.                                                                          |
| Proof receipt           | A durable record of command, commit, lane, artifacts, digests, and result.                                                                   |
| Merge witness           | A reproducible merge-decision artifact combining changed paths, owner route, receipts, score delta, missing evidence, and decision.          |
| Generated zone          | An output path repaired only through its source contract and regeneration command.                                                           |
| Centerline drift        | Distance between claimed standard and observed repository behavior.                                                                          |

**Add explicit out-of-scope threats:**
The threat model should say what Jankurai does not solve:

* compromised CI root,
* malicious maintainers with admin rights,
* fake third-party attestations,
* full semantic proof of arbitrary code,
* all UI taste judgments,
* all organization-level culture issues,
* model-provider correctness,
* production incident detection after merge.

That honesty will make the standard more credible.

**What works:**
The “language as bottleneck compression” concept is insightful. It explains why language choice changes when syntax is no longer scarce.

**Risk:**

**Recommended change:**
Shorten the language-history table in the main paper and move detail to an appendix. The body should keep only the agent-era conclusion:

> Syntax imitation is cheap. Rejection speed, proof locality, safety, and repairability are expensive.

That one sentence prevents the stack debate from swallowing the standard.

## XVI. Limitations and Research Agenda

**What works:**
This section is appropriately humble and acknowledges calibration, file-size limits, tool-specific behavior, benchmark gaps, and the risk of ceremony/dogma.

**What to add:**
Add “failure modes of Jankurai itself”:

| Failure mode          | Mitigation                                          |
| --------------------- | --------------------------------------------------- |
| Artifact theater      | require fixture-backed evidence and receipt digests |
| Fake receipts         | sign receipts or verify command/artifact digests    |
| Stale owner/test maps | drift audit and changed-path coverage               |
| Score gaming          | separate conformance blockers from score            |
| Stack dogma           | core/profile split and alternate profile tests      |
| Ceremony overload     | progressive levels and advisory adoption            |
| Flaky proof lanes     | quarantine and explicit non-merge status            |
| Hidden policy forks   | adapter verification                                |

This will preempt criticism and make the standard feel mature.

## XVII. Conclusion

**What works:**
The conclusion lands emotionally and has a memorable final line.

**What to improve:**
End on the merge witness and standard, not the stack.

Suggested ending:

> Jankurai’s claim is simple: a repository that asks humans or agents to change production software must expose enough structure for the merge decision to be reproduced. Changed paths route to owners. Owners route to proof lanes. Proof lanes leave receipts. Receipts feed an audit report and merge witness. Findings become repair queues or expiring exceptions. That loop is the agent-native repository standard. Without it, AI does not give software engineering a new discipline; it gives old ambiguity a faster keyboard.

The repo source exposes this through the Rust CLI. ([GitHub][6])

The schema directory is a major asset. It is the seed of a serious standard. ([GitHub][2])

## Critical code/repo callouts

### 1. No published releases

For a versioned standard, this is the biggest repo-level gap. GitHub shows “No releases published.” ([GitHub][1])

Publish releases before presenting this as a versioned standard to a broad audience.

### 2. `proof-receipt.schema.json` is too permissive

As noted above, required fields are too minimal and `additionalProperties` is true. ([GitHub][4])

For a trust standard, receipts must be stricter.

### 3. `repo-score.schema.json` lacks explicit conformance level

The repo-score schema is rich and requires many important top-level fields, but `conformance_level` was not found. ([GitHub][5])

Add claimed and observed conformance level.

### 4. Ratchet semantics need review

The audit decision code appears to create a ratchet object with `baseline_score: score`, which means the baseline inside that decision is the current score, not necessarily an accepted external baseline. ([GitHub][8])

That may be fine for a summary object in advisory mode, but for true ratchet semantics, make baseline source explicit:

```json
"ratchet": {
  "baseline_source": "agent/repo-score.json",
  "baseline_report_fingerprint": "sha256:...",
  "baseline_score": 87,
  "candidate_score": 84,
  "allowed_drop": 0,
  "passed": false
}
```

### 5. GitHub workflow should not undermine HLT-020

The workflow uses major-version actions and `stable` toolchain references. That is common, but this project specifically warns about CI hardening and unpinned actions. Either pin them or document an exception. ([GitHub][7])

### 6. Hidden `--yolo` should be removed

A hidden `--yolo` flag is off-message for a project fighting “vibe coding.” ([GitHub][6])

Remove it before any serious publication push.

### 7. Large source files conflict with agent-native repair locality

`main.rs` and `audit/mod.rs` are both large. ([GitHub][6])

This is a good self-dogfooding opportunity.

### 8. Workflow should include conformance fixture validation

The workflow currently runs versions, audit, security, vibe coverage, bench/certify/govern/publish, UX QA, and paper build. ([GitHub][7])

Add:

```bash
cargo test -p jankurai
cargo fmt --all --check
cargo clippy --workspace --all-targets -- -D warnings
jankurai conformance run conformance/fixtures
jankurai schemas validate examples/**/expected/*.json
```

The README mentions `cargo fmt`, `cargo test`, `just fast`, `just score`, and `git diff --check` for contributors; put the same discipline visibly in CI. ([GitHub][1])

# Final assessment

The core idea is still profound: **Jankurai turns merge trust from a social guess into a versioned artifact problem.** That is the right standard-sized idea.

The paper already has many of the right concepts, but the repo now proves that the idea is more than prose. The next revision should lean into that: schemas, conformance fixtures, proof receipts, merge witnesses, and public evidence bundles. The stack profile can remain, but the standard must stand above it.

The strongest version of Jankurai is not “Rust wins.” It is:

> **A repository claiming AI-era engineering maturity must be able to reproduce why a change was allowed to merge.**

That is the standard worth making unavoidable.
