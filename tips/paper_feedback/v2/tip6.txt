I reviewed the latest 37-page edition of the Jankurai paper and inspected the public Jankurai repository/code structure. The latest version is materially stronger than the earlier one: it now has a more explicit conformance model, a more concrete merge-witness primitive, better UX/browser QA treatment, a stronger future-work table, and real implementation evidence in the repository. The repo is not just a paper companion; it contains a Rust CLI, schemas, proof/witness/security/audit commands, standard docs, adapters, paper sources, and a TypeScript UX-QA package. The project README also now frames Jankurai clearly as a **trustworthy-merge standard and local audit CLI**, not as a hosted AI/model service.   ([GitHub][1])

The paper’s central idea remains profound: **AI makes plausible code cheap, so merge must become a reproducible evidence decision.** That is the standard-setting idea. The implementation is beginning to support it: the Rust model binds standard/auditor/schema/paper versions, report fingerprints, manifest fingerprints, dirty-worktree state, proof receipts, security evidence, UX evidence, findings, and an ordered `agent_fix_queue`; the proof system creates plans, runs allowlisted commands, emits receipts, records artifact/log/receipt digests, and verifies plans against manifest fingerprints; and the witness command turns changed paths, route decisions, receipts, generated-zone touches, score deltas, caps, and findings into a merge decision.

The biggest remaining issue is not lack of ambition. It is **separation and credibility packaging**. The paper still blends four artifacts that should be distinct: the Jankurai Standard, the `jankurai` reference implementation, the Rust/TypeScript/PostgreSQL reference profile, and the manifesto/thesis. To become a tier-one, version-controlled standard, Jankurai should make the standard profile-neutral, publish conformance artifacts and test fixtures as versioned releases, move generated coverage tables out of the paper body, fix rule-mapping inconsistencies, and dogfood its own rules more visibly.

# Executive recommendations

## 1. Make the normative standard impossible to miss

The latest Section VI is much stronger, but the paper still needs a one-page normative core very early, ideally before language/stack discussion. A reader should know by page 3 exactly what a conformant repository must emit.

Recommended title:

**Normative Core of the Jankurai Standard**

Suggested content:

> A repository claiming Jankurai HL3 conformance MUST expose current, schema-valid artifacts for standard version, owner routing, proof routing, generated-zone policy, audit policy, audit report, proof receipts or evidence index, ordered repair queue, and merge witness. These artifacts MUST be bound to the current commit, auditor version, schema version, standard version, paper/spec edition, changed-path set, and artifact digests.

Then list the minimum artifact set:

```text
agent/standard-version.toml
agent/owner-map.json
agent/test-map.json
agent/generated-zones.toml
agent/proof-lanes.toml
agent/audit-policy.toml
agent/repo-score.json
agent/repo-score.md
target/jankurai/proof-plan.json
target/jankurai/evidence-index.json
target/jankurai/proof-receipts/*.json
target/jankurai/merge-witness.json
target/jankurai/merge-witness.md
```

The code already supports much of this. `RepoCatalog` loads owner maps, test maps, generated zones, and proof lanes; proof planning uses those maps to route changed paths; evidence indexes include digests and manifest fingerprints; and merge witnesses include changed paths, route decisions, missing evidence, scores, findings, and decisions.

## 2. Split the paper’s identity into four layers

Right now the paper says “the standard is the primary claim; the stack is a default profile,” which is a good improvement. But structurally the stack still occupies enough space that some readers will interpret Jankurai as “Rust/TS/Postgres standardization” rather than “trustworthy merge conformance.” The repo itself already supports the more important framing: it presents Jankurai as a local audit CLI and standard for ownership maps, proof lanes, generated zones, security boundaries, rolling scores, merge witnesses, and repair queues. ([GitHub][1])

Use four named layers everywhere:

**Jankurai Standard**
Normative, profile-neutral, versioned conformance model.

**Jankurai Reference Implementation**
The `jankurai` Rust CLI and schemas.

**Jankurai Reference Profile**
Rust core, TypeScript/React/Vite product surface, PostgreSQL durable truth, generated contracts, bounded Python.

**Jankurai Public Thesis**
“No proof, no merge; no receipt, no trust.”

This would make adoption much easier for Go, JVM, .NET, Python, Rails, Elixir, or enterprise teams.

## 3. Publish versioned releases, not only a versioned repo

The paper is about a version-controlled standard, but the repository currently shows **no GitHub Releases**. For standard credibility, this is a major gap. The repo page shows no published releases, even though the paper and code identify version `0.6.1`. ([GitHub][1])

For every public standard version, publish:

```text
v0.6.1/
  jankurai-standard.md
  jankurai-standard.pdf
  rule-registry.json
  schemas/*.schema.json
  conformance-suite.zip
  jankurai-cli-checksums.txt
  sbom.spdx.json or cyclonedx.json
  changelog.md
  migration-guide.md
  public-evidence-bundle.json
```

Also create immutable Git tags such as `standard/v0.6.1`, `schema/v1.4.1`, and `cli/v0.6.1`, or clearly document why a single tag covers all three.

## 4. Make the conformance suite the next flagship artifact

The future-work table correctly lists a public conformance suite as the first future-work lane. That should become the main project priority. A standard becomes real when outside tools can test themselves against it.

Build a **Jankurai Conformance Suite** with fixture repositories:

```text
fixtures/
  hl3-pass-minimal/
  ownerless-path-fail/
  unmapped-proof-fail/
  generated-mutation-fail/
  secret-sprawl-fail/
  destructive-migration-fail/
  xss-input-boundary-fail/
  authz-isolation-fail/
  overbroad-agency-fail/
  rendered-ux-gap-fail/
  stale-proof-receipt-fail/
  ratchet-regression-fail/
expected/
  *.repo-score.json
  *.merge-witness.json
  *.proof-verify.json
```

This lets other implementations say:

```bash
jankurai-conformance test --profile core-hl3
```

or

```bash
my-auditor --emit jankurai --suite core-hl3
```

That is how Jankurai becomes a standard rather than only a CLI.

## 5. Fix the Appendix A coverage table before any serious publication

The latest paper still includes 13 pages of source-row coverage tables. This was one of the biggest issues in the previous version, and it remains. The full table proves the project has machinery, but it weakens the paper’s presentation. It looks generated, repetitive, and visually exhausting. Pages 18–30 are dominated by repeated yellow rows saying “needs issue-specific fixture, CI artifact, or governance receipt evidence…”

Move the complete table to the repository as a machine-readable and HTML artifact. In the paper, replace it with:

```text
Total parsed rows: N
Absolute coverage: X
Partial coverage: Y
No coverage: Z
Top partial categories:
  - authz/data isolation
  - input boundary
  - agent-tool supply chain
  - human-review evidence
  - release readiness
Representative rows: 10–15
Known mapping caveats: yes
```

This change alone would make the paper feel far more tier-one.

## 6. Correct taxonomy-rule mappings

Some coverage mappings are still visibly wrong or at least surprising. This was flagged before, and it was not fixed. Examples:

* Page 21: **“Cross-site scripting / unsafe rendering”** maps to `HLT-016 dependency-supply-chain`; it should map to `HLT-023-INPUT-BOUNDARY-GAP` and possibly `HLT-013/014` if the failure is rendered UI/accessibility.
* Page 22: **“No rollback / weak backup discipline”** maps to `HLT-016`; it should map to `HLT-025-RELEASE-READINESS-GAP` or `HLT-021-DESTRUCTIVE-MIGRATION`.
* Page 28: **“Client-side-only authorization”** maps to `HLT-024-AGENT-TOOL-SUPPLY-GAP`; it should map to `HLT-022-AUTHZ-ISOLATION-GAP`.
* Page 28: **“No server-side validation”** maps to `HLT-024`; it should map to `HLT-023-INPUT-BOUNDARY-GAP`, and sometimes `HLT-022` if it creates authz bypass.
* Page 30: **“Broken business logic / skipped state transitions”** maps to `HLT-016`; it should map to `HLT-008-FALSE-GREEN-RISK` or to a new business-invariant rule.
* Page 18: **“Database Row-Level Security Misconfiguration”** maps to `HLT-010`; it should usually map to `HLT-022-AUTHZ-ISOLATION-GAP`.

The root cause is the “exactly one canonical issue” constraint. That is too rigid. Real issues often need a primary rule plus secondary rules.

Recommended schema:

```toml
primary_rule = "HLT-023-INPUT-BOUNDARY-GAP"
secondary_rules = ["HLT-013-RENDERED-UX-GAP", "HLT-009-GENERATED-SECURITY"]
mapping_confidence = "reviewed"
coverage = "partial"
```

This would preserve canonical routing while allowing accurate security taxonomy.

## 7. Dogfood the standard more visibly

The implementation is strong, but there is one credibility problem: the CLI’s `main.rs` is very large. The repo page shows `crates/jankurai/src/main.rs` as a 52.7 KB file, and GitHub renders it as 1,764 lines. That conflicts with the standard’s own hard-LOC philosophy, even if CLI dispatch code is not “product runtime.” ([GitHub][2])

Refactor `main.rs` into:

```text
crates/jankurai/src/cli/
  mod.rs
  args.rs
  dispatch.rs
  commands/
    audit.rs
    proof.rs
    witness.rs
    security.rs
    ux.rs
    repair.rs
    governance.rs
```

Then publish Jankurai’s own `agent/repo-score.json`, merge witness, and repair queue as public dogfood artifacts.

## 8. Clarify schema versioning

The paper front matter says `schema_version=1.4.1`, while implementation artifacts use several artifact-local schema versions: proof plan `1.0.0`, evidence index `1.2.0`, merge witness `1.0.0`, security evidence `1.0.0`, and the global model constant `SCHEMA_VERSION = 1.4.1`. This may be intentional, but the paper should clarify the difference between:

```text
schema_bundle_version = 1.4.1
artifact_schema_version.proof_plan = 1.0.0
artifact_schema_version.evidence_index = 1.2.0
artifact_schema_version.merge_witness = 1.0.0
artifact_schema_version.security_evidence = 1.0.0
```

Without that distinction, readers may think the implementation is inconsistent. The model constants and artifact fields are visible in the code.

## 9. Declare detector maturity honestly

The paper says the v0.6.1 registry has no red rows, but most rows are yellow/partial. The code confirms that many detectors are pattern-based heuristics: TODO/fallback/prompt-injection/agency/false-green/secret/authz/input-boundary/release/cost/human-review patterns are scanned through string and regex matching. This is useful, but it is not semantic proof.

Introduce a detector maturity scale:

```text
M0 — no detector; evidence requirement only
M1 — pattern detector
M2 — structured artifact detector
M3 — semantic/static detector
M4 — executable negative fixture
M5 — multi-repo calibrated detector
```

Then label every rule and coverage row with maturity. This will make the project more credible, not less.

# What improved since the earlier version

The latest paper followed several important recommendations from the previous review.

First, the standard/reference-profile distinction is now more explicit. The abstract and Section I now say the standard is the primary claim and the Rust/TS/Postgres stack is the reference/default profile. That is a substantial improvement.

Second, the merge witness is now much more central. Section VI calls it the v0.5 public primitive and gives the trustworthy-merge equation:

```text
merge = changed paths + owner route
+ proof receipts + score delta
+ missing evidence decision
```

The repository also implements `witness.rs` with route decisions, generated-zone touches, required lanes, available receipts, missing evidence, score delta, caps, findings, and pass/review/block/ratchet decisions.

Third, the UX/browser QA section is far stronger. It now frames the browser as a runtime, defines layered rendered proof, names DOM geometry checks, ARIA snapshots, target-size evidence, CLS, visual baseline governance, and human feedback labels. The repo also includes `@jankurai/ux-qa`, a package described as “Playwright DOM geometry checks for rendered UX quality.”

Fourth, the future-work table is concrete and useful. It includes conformance suites, evidence registries, signed proof receipts, repair-locality benchmarks, adapter tests, UI oracle hardening, semantic detector expansion, exception libraries, cost-budget tooling, and reusable cell certification. That is exactly the right direction.

Fifth, the code now backs many of the paper’s claims. The repository has a Rust CLI, schema validation, audit reports, proof plans, proof receipts, evidence indexes, merge witnesses, security evidence envelopes, UX QA artifacts, repair planning, migration analysis, certification/public evidence commands, and version constants matching the paper. ([GitHub][3])

# Feedback from before that was not followed

Several earlier recommendations remain unresolved.

## 1. “Humans Were the Bug” is still in the main title stack

The line is memorable, but it remains risky as a formal subtitle. It can alienate exactly the people needed for adoption: maintainers, reviewers, enterprise engineering leaders, standards people, and safety/security teams.

Better:

**Title:**
**Jankurai: A Versioned Repository Standard for Trustworthy Agentic Merge**

**Epigraph or opening hook:**
**Humans were not the bug. Repositories that depended on human memory were the bug.**

That preserves the punch without sounding anti-human.

## 2. Abstract says three contributions; Section I says four claims

This mismatch remains. The abstract has three contributions, while Section I says the paper makes four claims. Align them.

Either make both “three contributions” or both “four claims.” I recommend:

1. repository conformance standard;
2. audit/proof/merge-witness mechanism;
3. rule registry and repair queue;
4. non-normative reference profile.

## 3. The front page is still visually imbalanced

The logo/art is large, the lower page has a lot of white space, and there is still a small blue stray mark near the subtitle. Page 1 looks branded but not yet publication-polished.

## 4. The stack still has too much rhetorical gravity

The sentence in Section IX remains:

> “Once the standard identifies a default, continuing to optimize for every runner-up becomes noise.”

This was flagged before. It still risks making the paper sound dogmatic.

Replace with:

> “A reference profile gives agents a default map. Alternate stacks remain conformant when they provide equivalent ownership, proof, boundary, security, and repair artifacts.”

## 5. Appendix A’s giant coverage table remains in the paper

This was one of the clearest previous layout recommendations. It was not followed. The full table still overwhelms pages 18–30.

## 6. Table XI still looks broken

Page 10’s representative score decomposition table still has extreme vertical whitespace and a cramped rightmost column. It visually looks like a layout failure rather than a deliberate table.

## 7. The stray `{` and split JSON block remain

Page 30 still ends with a stray `{`, and page 31 begins with a JSON fragment whose opening brace is separated from the rest of the block. This was previously called out and remains unresolved.

## 8. Exceptions are still mostly Markdown templates

Appendix C gives useful Markdown exception and repair templates, but the earlier recommendation was to add machine-readable TOML/JSON forms. This still needs to be done.

## 9. No formal claim-boundaries box

The paper says the claim is bounded, but it still lacks a crisp “Jankurai claims / does not claim” box.

Add it near the abstract or Section II.

## 10. No case study or evaluation section

The paper still lacks a concrete end-to-end demonstration. For tier-one acceptance, add at least one worked example PR:

1. agent edits generated file by hand;
2. audit flags generated-zone mutation;
3. repair changes source contract;
4. proof receipts are emitted;
5. merge witness moves from block to pass.

# Abstract

## What works

The abstract is strong. The opening claim that the scarce act is no longer typing plausible code but trustworthy merge is excellent. The phrase **“no proof, no merge; no receipt, no trust”** remains the paper’s strongest public slogan.

The abstract also correctly separates the standard from the reference profile, which is a major improvement.

## What to improve

The abstract should mention **merge witnesses** explicitly, because that is the artifact that makes the idea tangible.

It should also stop sounding defensive about not being IETF/W3C/CNCF/etc. Keep the humility, but phrase it as status:

> Jankurai is proposed as an implementation-backed standard rather than an institutional specification.

## Revised abstract suggestion

> AI coding shifts the bottleneck in software engineering from producing plausible code to deciding whether a change is safe to merge. Jankurai proposes a versioned, auditable repository conformance standard for making that merge decision reproducible. A conformant repository exposes machine-readable ownership, proof-lane routing, generated-zone policy, security policy, audit evidence, repair queues, and merge witnesses bound to standard, schema, auditor, artifact, and commit versions. The core rule is simple: no proof, no merge; no receipt, no trust.
>
> This paper contributes: (1) a repository conformance model with stable rule identifiers, owner maps, test maps, generated-zone controls, version manifests, and bounded authority for human- and agent-authored changes; (2) an audit mechanism spanning local, pull-request, CI, ratchet, and release lanes using JSON/Markdown reports, proof receipts, evidence indexes, repair queues, and merge witnesses; (3) a versioned rule registry and repair model for making findings durable and agent-actionable; and (4) a non-normative reference profile using Rust, TypeScript/React/Vite, PostgreSQL, generated contracts, and bounded Python.
>
> Jankurai does not prove all program semantics and is not yet an institutional standard. It is an implementation-backed proposal for turning repository structure into a verifiable control plane for agent-native software engineering.

# Section I — New Bottleneck: Trustworthy Merge

## What works

This section remains one of the strongest. It reframes the “Humans Were the Bug” slogan in a more defensible way: humans are not the bug because they write software; the bug is that repositories have depended on human memory, patience, taste, and slow review.

The operating loop is excellent:

```text
intent -> bounded authority -> proof lanes ->
evidence -> repair or exception expiry ->
reusable primitives
```

This should become a polished figure.

## What to improve

The section should introduce merge as a decision function:

```text
D = f(C, O, P, G, S, R, X, V)
```

Where:

```text
C = changed paths
O = owner map
P = proof-lane map
G = generated-zone policy
S = security policy
R = receipts/evidence
X = exceptions/ratchet state
V = standard/schema/auditor versions
```

Then:

```text
D ∈ {pass, review, block, ratchet_fail, release_fail}
```

This formalizes the paper’s core idea without making it too abstract.

## Specific recommendation

Move the four claims into a numbered contribution list that matches the abstract. Right now the “three contributions” versus “four claims” mismatch is distracting.

# Section II — Definitions and Threat Model

## What works

The operational definitions are useful. “Vibe coding,” “agent-native engineering,” “conformance artifact,” “proof lane,” “proof ledger,” “repair queue,” “generated zone,” “repair receipt,” and “centerline drift” are all valuable vocabulary.

## What to improve

The threat model is still too thin for a tier-one standard paper. It starts with false plausibility, which is correct, but it does not yet fully define assets, actors, trust boundaries, attacker capabilities, assumptions, and out-of-scope threats.

Add a structured threat model table:

```text
Assets:
  source integrity
  production data
  credentials/secrets

  generated contracts
  audit evidence
  release artifacts
  user-facing behavior
  organization trust

Actors:
  human contributor
  coding agent
  reviewer
  malicious contributor
  compromised dependency
  hostile issue/comment/doc/web page
  CI runner
  release operator

Trust boundaries:
  root policy
  local path policy
  repo code
  generated artifacts
  issue/PR text
  web content
  model output
  tool output
  package registries
  CI artifacts

Attacker capabilities:
  prompt injection
  code contribution
  dependency poisoning
  artifact forgery
  CI manipulation

Out of scope:
  full formal verification
  malicious maintainers with total admin control
```

Also add **evidence poisoning** as a first-class threat. Once receipts become valuable, attackers will try to forge, stale, or misroute them.

# Section VI — Jankurai Standard and Conformance

## What works

This is the heart of the paper and the strongest section. It defines progressive install levels, HL0–HL5 conformance, stable rule families, current artifacts, hard caps, centerline drift, and merge witnesses.

The code supports the paper’s claims here. The audit report model includes version fields, fingerprints, scores, decisions, proof receipts, caps, hard rules, dimensions, UX/security/boundary evidence, findings, and repair queue.

## What to improve

### Separate standard from CLI

The section says “the adoption loop is itself part of the standard: `jankurai init`, `jankurai doctor`, `jankurai audit`…” This is dangerous for standard adoption. The standard should define required behaviors and artifacts. The CLI is the reference implementation.

Better:

> A conformant implementation MUST be able to initialize, validate, audit, route proof, emit evidence, and produce merge witnesses. The reference implementation exposes those behaviors through `jankurai init`, `jankurai doctor`, `jankurai audit`, `jankurai prove`, and `jankurai witness`.

### Add conformance claim types

Define:

```text
Implementation-conformant:
  A tool emits schema-valid Jankurai artifacts.

Repository-conformant:
  A repository satisfies a declared HL level at a commit.

Release-conformant:
  A release artifact is gated by HL5 evidence.

Profile-conformant:
  A repository satisfies a named profile, such as Core, Security, Web UX, Data, or Release.

Certified:
  A third-party or public conformance suite validates the claim.
```

### Define artifact freshness

A proof receipt should be invalid if:

```text
commit changed after receipt was generated
changed paths differ from the plan
owner/test/generated/proof manifests changed
standard/schema/auditor versions changed incompatibly
command digest differs
log/artifact digest differs
receipt lacks exit code or lane
dirty worktree status invalidates the run
```

The code already verifies important parts of this, including plan digest, changed paths, manifest fingerprints, receipt validation, and artifact digests. The paper should surface those details.

# Section X — Agent Repository Controls and Tool Adapters

## What works

# Section XIII — Security, Supply Chain, and Permissions

## What works

The section is practical. It correctly says agentic coding expands the write surface and that permission policy should be boring. The code’s `security run` command creates a normalized security evidence envelope, records policy snapshots, strict mode, wrapper command, per-step status when available, logs, and exit codes.

## What to improve

Make the section more tool-agnostic. Instead of saying the current workspace treats `gitleaks` as required and dependency scanners as advisory, say:

> A conformant security lane MUST distinguish required, advisory, unavailable, failed, and blocking controls. The reference implementation may use tools such as secret scanners, SCA scanners, SBOM generators, workflow linters, and provenance checkers.

## Code-level call-out

The security wrapper bails when the overall script exits nonzero, and it models parsed steps as required/advisory/blocking. However, the paper should make clear that a security lane must fail if any required parsed step is blocking, even if a wrapper script accidentally exits zero. The implementation may already handle this through the script’s strict mode, but the standard should state it explicitly.

Add permission profiles:

```text
read-only
workspace-write
generated-zone-regenerate
dependency-change
migration-change
browser-automation
network-enabled
secret-adjacent
release-capable
```

Each profile should require a receipt.

# Repository and code feedback

# Overall repository assessment

The repository is substantially more credible than the paper alone would suggest. It is public, has a Rust CLI, schemas, paper source, standard docs, agent adapters, example materials, a UX-QA package, and a broad command surface. The repo tree includes `agent`, `crates/jankurai`, `schemas`, `packages/ux-qa`, `paper`, `docs`, `examples`, `contracts`, `db`, `tools`, and provider-adapter files. ([GitHub][1])

The README’s positioning is strong:

> Jankurai is a trustworthy-merge standard and local audit CLI.

That is exactly the right public message. ([GitHub][1])

# Code strength: audit model

The audit implementation is aligned with the paper. `run_audit_with_options` inventories files, scopes to changed paths, computes dimensions, applies caps, loads policy, attaches UX/security/boundary/vibe coverage summaries, builds findings, creates an `agent_fix_queue`, and emits a report with fingerprints and version fields.

This is strong. It means the paper can honestly claim implementation-backed audit artifacts.

## Recommended improvement

Publish a sample `repo-score.json` in the paper appendix or docs, with sensitive paths sanitized. Readers need to see the artifact shape.

# Code strength: proof plans and receipts

The proof implementation is one of the most important parts of the repo. It routes changed paths, creates proof plans, executes allowlisted commands, writes receipts, writes evidence indexes, records command/log/artifact/receipt digests, and verifies evidence against plan digests and manifest fingerprints.

The allowlist design is especially good: proof commands must come from `agent/proof-lanes.toml` or `agent/test-map.json`, unless the user passes an explicit flag and environment variable.

## Recommended improvement

Add:

```text
timeout enforcement from proof-lanes.toml
argv-form commands, not only shell strings
environment allowlist
network/destructive permission gates
tool version capture
OS/container image capture
redacted log mode
```

The paper should state which are implemented now and which are future work.

# Code strength: merge witness

The merge witness implementation is a major asset. It encodes the paper’s core claim into a concrete artifact. It checks route decisions, generated-zone touches, required lanes, proof receipts, missing evidence, baseline score, score delta, caps added, new/resolved/carried findings, and emits pass/review/block/ratchet decisions.

## Recommended improvement

Add an example witness in the paper body. Do not bury the idea in prose. A compact JSON witness will make the standard understandable.

# Code strength: schema validation

The repo has a custom schema validator and many artifact schemas. This is excellent for a standard project because schema-valid artifacts are central to conformance.

## Caution

The validator supports a limited subset of JSON Schema. That is fine, but the paper and docs should explicitly define it as the **Jankurai Schema Subset** or switch to a full JSON Schema implementation.

Current supported keywords include `$ref`, `$defs`, `type`, `const`, `enum`, `required`, `properties`, `additionalProperties`, `items`, `minItems`, `maxItems`, `uniqueItems`, `minimum`, `maximum`, `minLength`, `maxLength`, `pattern`, `allOf`, `anyOf`, and `oneOf`; unsupported keywords are rejected.

That means schemas using `format`, `patternProperties`, `if/then/else`, `not`, `dependentRequired`, or newer draft features will not work unless support is added.

# Code concern: detector maturity

Many detectors are string/regex heuristics. This is appropriate for an early standard, but the paper should not let “no red rows” sound like full protection. `scan.rs` uses pattern lists for TODOs, fallbacks, prompt injection, agency, false-green tests, streaming clients, authz markers, input-boundary markers, agent-tool supply markers, release readiness, cost budgets, and human-review evidence.

Recommended language:

> Jankurai v0.6.1 distinguishes evidence requirements from detector maturity. Some controls are detector-backed; others are policy-backed and require fixtures, receipts, or governance evidence before absolute coverage is claimed.

# Code concern: `main.rs` size

The CLI’s `main.rs` is very large. Since the standard itself says large files create repair risk, the reference implementation should dogfood that principle. GitHub shows `main.rs` as 52.7 KB and 1,764 lines. ([GitHub][2])

Refactor it before trying to gain broad acceptance.

# Code concern: public releases

The repo has no GitHub Releases. That conflicts with the paper’s standard-versioning ambition. ([GitHub][1])

Create releases immediately, even if they are pre-1.0.

# Code concern: README/paper adoption mismatch

The paper describes install levels `agents → score → ci → full → ratchet`, but README shows Observe, Agents, Full, Ratchet. Align terminology.  ([GitHub][1])

# Code concern: schema bundle packaging

The validator locates schemas via `CARGO_MANIFEST_DIR/../../schemas`. That works inside the workspace, but the project should document how schemas are bundled when installing the CLI or publishing a crate/binary.

# The core message to preserve

The paper’s deepest contribution is not the stack ranking. It is this:

> In the agentic era, merge must become an auditable, reproducible, versioned evidence decision.

That is the idea with standard-setting potential.

Everything else should support that idea:

```text
ownership maps
proof lanes
generated-zone policy
version manifests
proof receipts
evidence indexes
merge witnesses
repair queues
exception expiry
ratchets
release gates
conformance suites
```

The strongest public framing is:

> Jankurai does not ask teams to trust agents. It asks repositories to distrust every merge claim until ownership, proof, receipts, and repair evidence are current.

That sentence should guide the next revision.
