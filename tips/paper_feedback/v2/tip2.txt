I read the latest 37-page paper and reviewed the public GitHub repository source, schemas, command surface, tests, manifests, agent maps, proof-lane files, and the UX package. The paper is substantially stronger than the prior draft because the repository now makes the thesis tangible: Jankurai is no longer only an argument for safer AI coding; it is becoming a **versioned repository control plane for auditable merge evidence**. The paper’s strongest concept remains the merge rule: **“no proof, no merge; no receipt, no trust.”** The repo README reinforces that same framing by describing Jankurai as a local audit CLI and standard that turns owner maps, proof lanes, generated zones, security boundaries, rolling scores, merge witnesses, and repair queues into files, rather than a hosted AI service.  ([GitHub][1])

The most important thing to fix before seeking wide acceptance is this: **the version-control story is powerful in the paper, but the repo currently contradicts it.** The paper claims `standard_version=0.6.1`, `schema_version=1.4.1`, and `paper_edition=2026.05-ed6`; the repo’s `VERSION` and Cargo package version are `0.6.1`, but `agent/standard-version.toml` still declares `standard_version="0.4.0"`, `schema_version="1.2.0"`, and `paper_edition="2026.05-ed3"`. For a standard whose central promise is auditable version identity, that mismatch is the highest-priority credibility blocker.  ([GitHub][2])

My repo review is based on the public GitHub files I inspected, not on a local execution run. That matters because some recommendations below are about code structure, schemas, manifests, tests, and release discipline, rather than runtime behavior.

# Executive verdict

Jankurai’s tier-1 potential is real. The paper has a rare and valuable insight: **AI-era software engineering should not be governed primarily by prompts, policies, or human review rituals. It should be governed by repository artifacts that can prove merge readiness.**

That is the right abstraction. The repository becomes the interface through which humans and agents discover:

* what changed,
* who owns it,
* which proof lanes apply,
* which evidence exists,
* which evidence is missing,
* what must be repaired,
* which waivers expire,
* and whether the merge is allowed.

The codebase supports this direction. It already contains a Rust CLI, a broad command surface, schemas for major evidence artifacts, owner/test maps, generated-zone manifests, proof lanes, security policy, UX QA package, and many tests. The repository is therefore much more than a paper prototype. The README lists a serious control-plane surface: `adopt`, `init`, `doctor`, `context-pack`, `adapters verify/sync`, `lane`, `proof`, `prove`, `proof-verify`, `audit`, `witness`, `score`, `security`, `ux`, `repair`, `exceptions expire`, `registry`, `cell`, `bench`, `certify`, `govern`, and `publish`. ([GitHub][1])

The current risk is not that the idea is too small. The risk is that the paper tries to be a manifesto, standard, stack manifesto, implementation manual, and rule registry all at once. For tier-1 acceptance, the project should be separated into:

| Artifact                   | Purpose                                                                 |
| -------------------------- | ----------------------------------------------------------------------- |
| **Paper**                  | Citable argument and architecture rationale.                            |
| **Normative spec**         | MUST/SHOULD/MAY requirements, conformance levels, artifact obligations. |
| **Schemas**                | Machine-readable contract for evidence artifacts.                       |
| **Conformance suite**      | Fixtures, expected outputs, pass/fail criteria.                         |
| **Rule registry**          | Versioned HLT rule lifecycle and detector status.                       |
| **Reference profile**      | Rust/TypeScript/PostgreSQL/bounded-Python recommendation.               |
| **Implementation**         | Rust CLI, UX package, GitHub workflows, commands.                       |
| **Public evidence bundle** | Checksums, release metadata, badges, signed receipts.                   |

The paper should become the clearest public explanation of the standard. The repo should become the evidence that the standard can be tested.

# Highest-priority corrections before public push

## 1. Fix the version identity mismatch immediately

This is the most urgent issue. The paper’s core promise is versioned auditability, but the repo’s canonical manifest is stale. The paper says the canonical manifest is `agent/standard-version.toml` and that the repo binds paper source, rendered paper, Markdown companion, standard docs, and agent brief under one versioned artifact manifest. But the actual GitHub manifest still shows `0.4.0` / `1.2.0` / `ed3`, while the paper, `VERSION`, and Cargo crate show `0.6.1` / `1.4.1` / `ed6`.  ([GitHub][2])

This should block publication. Add a release gate:

```bash
jankurai versions --strict --release-block
```

That command should fail if any of these disagree:

```text
VERSION
crates/jankurai/Cargo.toml
agent/standard-version.toml
paper metadata
paper/jankurai.tex
paper/jankurai.md
docs/agent-native-standard.md
agent/JANKURAI_STANDARD.md
schemas/* declared schema version
package.json for @jankurai/ux-qa, if included in same release
Git tag
GitHub release metadata
```

For a tier-1 standard, **the version manifest must be boringly correct every time**.

## 2. Publish a real GitHub release before claiming release-level standard maturity

The README currently says there are no published releases. That is acceptable for an early working standard, but the paper should not sound like a formal public standard release until the repo has tags, release notes, checksums, schema bundle, conformance fixtures, and a migration note. ([GitHub][1])

Minimum release package for `v0.6.1`:

```text
Git tag: v0.6.1
GitHub release: Jankurai 0.6.1
checksums.txt
schemas-1.4.1.tar.gz
jankurai-standard-0.6.1.md
jankurai-paper-2026.05-ed6.pdf
rule-registry-0.6.1.json
conformance-fixtures-0.6.1.tar.gz
migration-0.4.x-to-0.6.1.md
CHANGELOG.md
```

A badge should not simply say “Jankurai conformant.” It should say:

```text
Jankurai HL3
standard 0.6.1
schema 1.4.1
auditor 0.6.1
checked <date>
scope: repository
```

## 3. Separate “standard” from “reference profile” more aggressively

The paper repeatedly says the standard is primary and the Rust/TypeScript/PostgreSQL/bounded-Python stack is a reference profile. That is good. But sections VII–IX still read as though the stack ranking is part of the standard itself. This will reduce acceptance among Go, .NET, JVM, Rails, Python, Elixir, and TypeScript-heavy teams.

The standard should say:

> A repository can conform if it emits equivalent owner routing, proof lanes, generated-boundary controls, security evidence, repair receipts, and versioned merge witnesses.

The reference profile should say:

> This paper’s default profile is one recommended implementation path.

Rename:

* **“A 100-Point Stack Rubric”** → **“Reference Profile Scoring Method”**
* **“Stack Ranking, Sensitivity, and Exceptions”** → **“Reference Profile Sensitivity and Alternate Conformance”**
* **“Default Winner Architecture”** → **“Reference Profile Architecture”**

Avoid “winner” language in the standard paper. It sounds combative. Standards win by being adoptable.

## 4. Add a current implementation-status matrix

The paper describes a broad standard: fast, contract, DB, web, e2e, security, release, proof receipts, UX evidence, public evidence, certification, and governance. The repo has impressive scaffolding, but the live `proof-lanes.toml` calls itself a “live minimal lane set” and currently lists fast, audit, paper, security, and full lanes. ([GitHub][3])

Add a table to the paper:

| Capability           |      Spec status |  CLI status |         Test status | Conformance status |
| -------------------- | ---------------: | ----------: | ------------------: | -----------------: |
| Owner map            |     Required HL3 | Implemented |              Tested |          Candidate |
| Test map             |     Required HL3 | Implemented |              Tested |          Candidate |
| Generated zones      |     Required HL3 | Implemented |              Tested |          Candidate |
| Merge witness        | Required HL3/HL4 | Implemented | Needs fixture suite |          Candidate |
| Proof receipts       |     Required HL3 | Implemented |  Needs replay tests |          Candidate |
| Contract lane        | Required profile |     Partial |      Needs fixtures |              Draft |
| DB lane              | Required profile |     Partial |      Needs fixtures |              Draft |
| Web/UX lane          | Required profile |     Partial |        Needs matrix |              Draft |
| Release gate         |              HL5 |     Partial |      Needs fixtures |              Draft |
| Public certification |           Future |    Skeleton |          Not mature |              Draft |

This will increase trust because it shows exactly what exists and what is still maturing.

## 5. Move the massive coverage table out of the main paper

Appendix Table XXIII spans pages 18–30. It is valuable as generated evidence, but it overwhelms the paper. A tier-1 paper should include:

* one summary table by top-level risk,
* one sample mapping row,
* one detector-status table,
* one link/reference to the full generated registry.

Keep the full table as:

```text
agent/vibe-coverage.toml
target/jankurai/vibe-coverage.html
target/jankurai/vibe-coverage.json
```

The current table also uses “absolute” coverage. That word is too strong. Replace with:

```text
implemented
detector_backed
partial
policy_only
not_covered
not_applicable
```

“Absolute” invites criticism.

# Paper-wide structural recommendations

The paper should be reorganized around the standard first, then the profile.

Recommended structure:

1. **Introduction: Trustworthy Merge Is the Bottleneck**
   Concrete PR example, thesis, contributions.

2. **Definitions and Threat Model**
   Assets, actors, trust boundaries, in-scope/out-of-scope.

3. **The Jankurai Standard**
   Conformance levels, required artifacts, normative requirements.

4. **Merge Witnesses, Proof Receipts, and Evidence Indexes**
   Schemas and examples.

5. **Rule Registry and Repair Queue**
   HLT IDs, detector maturity, repair receipts, waivers.

6. **Continuous Audit Implementation**
   Local/PR/CI proof lanes, ratchets, release gates.

7. **Security and Permission Model**
   Agent authority, hostile context, redaction, provenance.

8. **Rendered UX as Runtime Proof**
   Browser-step QA, geometry oracle, accessibility, visual evidence.

9. **Evaluation**
   Seeded fixtures, brownfield case study, false positives, CI cost.

10. **Reference Profile**
    Rust/TypeScript/PostgreSQL/bounded Python.

11. **Migration and Governance**
    Versioning, release process, Jankurai Enhancement Proposals.

12. **Limitations and Future Work**

13. **Conclusion**

Appendices should contain schemas, command examples, file trees, and generated coverage excerpts.

# Section-by-section feedback

## Title page

**Current strength:**
The title “A Versioned Agent-Native Repository Standard for Trustworthy Merge” is excellent. It says exactly what the work wants to become. The page also clearly displays version metadata, which is essential to the standard’s identity.

**Main issue:**
The title page still feels like a branded manifesto rather than a standards paper. The logo is visually large, and the subtitle “Humans Were the Bug” is memorable but polarizing.

**Recommendation:**
Use two title variants:

Academic/spec edition:

> **Jankurai: A Versioned Agent-Native Repository Standard for Trustworthy Merge**
> **No Proof, No Merge; No Receipt, No Trust**

Website/blog edition:

> **Humans Were the Bug: From Vibe Coding to Agent-Native Engineering**

A standards audience will respond better to the first. The second is excellent for launch essays, talks, and landing pages.

Also fix the small stray purple mark visible near the subtitle on page 1.

## Abstract

**Current strength:**
The abstract correctly identifies the bottleneck: not code generation, but trustworthy merge. It also clearly states the three contributions: repository standard, conformance/audit mechanism, and reference-profile separation.

**Main issue:**
It is list-heavy. The strongest sentence should be more operational:

> Jankurai makes merge decisions reproducible by requiring repositories to emit versioned evidence that binds changed paths, owners, proof lanes, receipts, missing evidence, repair queues, and merge decisions.

**Add one sentence about implementation status:**
Because the repo exists and has substantial schemas/tests, say:

> A working Rust CLI and schema set implement the current draft surfaces; conformance remains pre-release pending public fixtures and release artifacts.

That would be more credible than implying institutional maturity.

## Index terms

Add:

```text
repository conformance, merge witness, proof receipts,
software supply chain, policy-as-code, generated-code governance,
agentic software engineering, CI/CD evidence, repair automation,
browser-step QA, UX verification
```

“Rust” and “TypeScript” are useful but should not dominate the keyword identity. The novel standard is bigger than the stack.

## I. New Bottleneck: Trustworthy Merge

**What works:**
This is still one of the best sections. The loop—

```text
intent -> bounded authority -> proof lanes -> evidence
-> repair or exception expiry -> reusable primitives
```

—is the entire standard in one line.

**Make it visual:**
Turn the loop into a figure with six nodes and artifacts under each node:

| Loop step          | Artifact                            |
| ------------------ | ----------------------------------- |
| Intent             | task description, issue, PR summary |
| Bounded authority  | owner map, permission profile       |
| Proof lanes        | test map, proof plan                |
| Evidence           | proof receipts, evidence index      |
| Repair/waiver      | repair queue, waiver expiry         |
| Reusable primitive | cell registry, rule update          |

**Add a concrete PR story:**
Before the abstract claims, add a scenario:

> A PR changes a checkout form and a generated API client. Traditional review sees plausible code. Jankurai sees a generated-zone touch, a web surface change, a missing contract regeneration receipt, and no mobile Playwright trace. The merge witness blocks the PR and emits a repair queue.

That will make the standard intuitive in 30 seconds.

**Tone suggestion:**
Replace:

> Humans were the bug.

With:

> Human-memory-centered repositories were the bug.

That keeps the force but removes unnecessary defensiveness.

## II. Definitions and Threat Model

**What works:**
The definitions are strong. “Conformance is a public interface, not a vibe” is one of the paper’s best lines. The threat table is also useful.

**What to add:**
A standards paper needs a more formal threat model:

| Threat-model element | Additions                                                                                                                     |
| -------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| Assets               | Source code, secrets, schemas, generated files, CI tokens, audit artifacts, production data.                                  |
| Actors               | Human developer, coding agent, reviewer, malicious contributor, compromised dependency, malicious tool/MCP server, CI runner. |
| Trust boundaries     | Trusted policy, repo code, generated artifacts, proof evidence, untrusted PR/issue/web text, external packages.               |
| In scope             | False plausibility, generated-zone mutation, secret leakage, prompt injection, excessive agency, stale evidence replay.       |
| Out of scope         | Full formal verification, malicious maintainer collusion, all semantic correctness.                                           |
| Assumptions          | VCS exists, CI/local runner exists, artifacts include commit IDs and digests.                                                 |

**Important addition:**
Add **evidence replay** and **policy tampering**. A malicious or careless actor could attach an old proof receipt to a new commit, or change `owner-map.json` to avoid a gate. The standard should require proof receipts to include:

```text
repo_revision
base_revision
head_revision
input_path_hashes
policy_hash
manifest_hash
tool_versions
timestamp
```

## III. Languages as Bottleneck Compression

**What works:**
The idea that languages compress human bottlenecks is good and distinctive. It helps explain why syntax is no longer the scarce resource.

**Main issue:**
This section appears too early. It pulls attention away from the standard. Move it after the core Jankurai section or compress it to a short subsection.

**Better framing:**
Say:

> In agent-native engineering, language choice matters only insofar as it changes rejection speed, proof locality, runtime safety, and repair evidence.

Avoid sounding like the paper is primarily a language-ranking argument.

## IV. Technical Promise Versus Standard Gravity

**What works:**
The section makes an important distinction: technical elegance does not automatically become default-standard adoption.

**Main issue:**
It still reads like a language-culture survey. Tie it directly to standards adoption.

**Revised purpose:**
Use this section to explain why Jankurai must provide:

```text
clear conformance levels
fixtures
schemas
version compatibility
migration guides
release artifacts
reference implementations
```

This would make the section less about Julia/Elm/Haskell/Elixir and more about why Jankurai itself needs boring standards machinery.

## V. Vibe-Coding Fault Taxonomy

**What works:**
The taxonomy is a major asset. The paper correctly turns “vibe coding” from an insult into an auditable surface: rule IDs, lanes, required controls, hardness, and repair evidence.

**Main improvements:**

First, define RPN scoring more explicitly:

| Score | Impact                      | Likelihood                 | Detection difficulty                    |
| ----: | --------------------------- | -------------------------- | --------------------------------------- |
|     1 | Local annoyance             | Rare                       | Obvious                                 |
|     3 | User-visible defect         | Plausible                  | Requires targeted proof                 |
|     5 | Security/data/business loss | Common in AI-assisted work | Often invisible without mapped evidence |

Second, rename “Hardness” to **“Evidence hardness.”** “Hard” and “Soft” are useful, but readers need to know whether they mean detection difficulty, repair difficulty, or proof determinism.

Third, fix the rule mappings in Appendix Table XXIII. Several mappings currently weaken confidence:

| Table row                                         | Current mapping           | Better mapping                                       |
| ------------------------------------------------- | ------------------------- | ---------------------------------------------------- |
| Cross-site scripting / unsafe rendering           | HLT-016 supply chain      | HLT-023 input boundary, possibly HLT-013 rendered UX |
| SSRF and unsafe fetching                          | HLT-024 agent-tool supply | HLT-023 input boundary/security                      |
| No rollback / weak backup discipline              | HLT-016 supply chain      | HLT-025 release readiness                            |
| Prototype architecture shipped to production      | HLT-016 supply chain      | HLT-025 release readiness or HLT-001 entropy         |
| Prompt-level rules treated as security controls   | HLT-022 authz isolation   | HLT-011 prompt injection / HLT-024 agent-tool supply |
| No server-side validation                         | HLT-024 agent-tool supply | HLT-023 input boundary                               |
| Broken business logic / skipped state transitions | HLT-016 supply chain      | HLT-008 false-green/business truth                   |

Fourth, remove the 13-page generated table from the body. It should be a public registry artifact.

## VI. Jankurai Standard and Conformance

**This is the core of the paper. Move it earlier.**

The section has the strongest standard mechanics: install levels, HL0-HL5 conformance, HLT rule IDs, audit loop, merge witness equation, and repair queue.

**What to strengthen:**
Add numbered normative requirements:

```text
JKR-001: A conformant repository MUST declare standard_version, schema_version, auditor_version, and claim_scope.
JKR-002: A conformant repository MUST provide owner routing for changed paths.
JKR-003: A conformant repository MUST provide proof-lane routing for changed paths.
JKR-004: A conformant repository MUST identify generated zones.
JKR-005: A conformant repository MUST emit a merge witness for gated merges.
JKR-006: A conformant merge witness MUST identify missing evidence explicitly.
JKR-007: A conformant proof receipt MUST bind command, commit, input paths, policy hash, and artifact digests.
JKR-008: A conformant waiver MUST be scoped, owned, justified, and expiring.
```

**Make the merge witness the hero artifact:**
The repo has a `merge-witness.schema.json` with required fields such as schema version, command, generated timestamp, repo, git, changed paths, route decisions, generated-zone touches, required lanes, proof receipts, missing evidence, scores, caps, findings, decision, and next repair. That is exactly the right primitive. ([GitHub][4])

But the merge-witness schema should require the same version identity that `repo-score.schema.json` already requires: standard version, auditor version, paper edition, target stack ID, report fingerprints, policy fingerprints, and manifest fingerprints. The repo-score schema already contains many of these stronger fields, which is excellent; the merge witness should be brought up to the same standard. ([GitHub][5])

Recommended required merge-witness fields:

```json
{
  "standard": "jankurai",
  "standard_version": "0.6.1",
  "schema_version": "1.4.1",
  "auditor_version": "0.6.1",
  "paper_edition": "2026.05-ed6",
  "claim_scope": "pull_request",
  "conformance_level_claim": "HL3",
  "target_stack_id": "rust-ts-vite-react-postgres-bounded-python",
  "repo_revision": "...",
  "base_revision": "...",
  "head_revision": "...",
  "policy_fingerprint": "...",
  "manifest_fingerprints": {},
  "changed_paths": [],
  "required_lanes": [],
  "proof_receipts": [],
  "missing_evidence": [],
  "decision": "block"
}
```

**What works:**
The rubric is thoughtful. It correctly weights correctness, security, contracts, and repairability above first-draft velocity.

**Main issue:**
It can distract from the standard. Some readers will reject the paper because they disagree with the stack ranking.

**Recommendation:**
Keep the rubric but label it explicitly:

> This is a reference-profile policy score, not a conformance requirement.

Also publish the scoring file in the repo:

```text
profiles/rust-ts-vite-postgres-bounded-python/profile.toml
profiles/rust-ts-vite-postgres-bounded-python/scoring.toml
```

Then the paper can say:

> We expect future profiles to define different weights while preserving the same evidence contract.

## VIII. Stack Ranking, Sensitivity, and Exceptions

**What works:**
The sensitivity caveats are good. They make the stack argument less dogmatic.

**What to improve:**
Move Kafka/Rust streaming material to a reference-profile appendix. It is interesting, but it is highly time-sensitive and not central to the standard.

The standard-level point is:

> Streaming infrastructure must live behind declared adapters, generated event contracts, ownership cells, and observable proof lanes.

Kafka, Tansu, Iggy, Fluvio, RobustMQ, NATS, and Redis Streams are examples, not normative claims.

**Layout issue:**
Figure 2 uses a cropped x-axis starting at 85. The caption discloses this, but it still visually exaggerates small score differences. Use a full 0–100 axis or make the “zoomed axis” label much more prominent.

## IX. Default Winner Architecture

**What works:**
“The filesystem should be a routing table” is a superb principle. This should be one of Jankurai’s signature ideas.

**What to change:**
Rename the section to **Reference Profile Architecture**.

The first sentence currently says:

> Once the standard identifies a default, continuing to optimize for every runner-up becomes noise.

That is too aggressive for a standard. Replace with:

> Once a repository selects a profile, agents need one default routing map, one proof policy, and one set of gates. Exceptions remain available, but they must be named, owned, and versioned.

This preserves the operational point without alienating alternate-stack adopters.

## X. Agent Repository Controls and Tool Adapters

**What works:**
This is one of the strongest sections. Treating AGENTS.md, CLAUDE.md, Cursor rules, Copilot instructions, and terminal-agent prompts as **thin routers over canonical policy** is exactly right.

The repo supports this with actual owner maps, test maps, generated zones, and adapter/agent verification commands. The owner map and test map are already concrete, which gives the paper credibility. ([GitHub][6])

**Add a source-trust hierarchy:**

```text
1. Jankurai standard and version manifest
2. Root repository policy
3. Owner map and test map
4. Local path policy
5. Repository source code
6. Generated artifacts
7. Proof evidence
8. Documentation
9. Untrusted issue/PR/web/tool text
```

**Add one normative rule:**

> Adapter files MAY summarize canonical policy but MUST NOT grant broader authority than the canonical Jankurai artifacts.

## XI. Continuous Proof, CI, and Test Explosion

**What works:**
The proof loop is concrete and practical. The repo has schemas for proof plans, proof receipts, evidence indexes, proof verification, repo score, merge witness, repair plans, and more. The schema directory is one of the strongest parts of the implementation. ([GitHub][7])

**What to improve:**
Add stale-evidence protection. Every proof receipt should include:

```text
repo_revision
base_revision
head_revision
input_paths
input_path_hashes
policy_fingerprint
manifest_fingerprint
command
exit_code
started_at
ended_at
tool_versions
artifact_digests
environment_summary
```

**Conformance rule:**
A proof receipt is invalid if:

```text
receipt.repo_revision != candidate.head_revision
receipt.policy_fingerprint != current_policy_fingerprint
receipt.input_path_hashes != current_input_hashes
required_artifacts_missing == true
```

This matters because stale proof is one of the easiest ways to fake trust.

## XII. Rendered UX and Browser-Step QA

**This is a major differentiator.**

Most AI coding safety work under-discusses UI. Jankurai’s claim that “the browser is a real runtime” is important. The repo backs this up with `@jankurai/ux-qa`, a Playwright DOM geometry package with modules for accessibility, collection, hit testing, rules, receipts, selectors, Storybook, and visual baselines.  ([GitHub][8])

**Important current-status caveat:**
The package is promising, but the Playwright config I inspected uses a single mobile-ish viewport of `390x844`. That is fine as a seed, but the paper should not imply full browser-step QA maturity from one viewport. ([GitHub][9])

Add an implementation-status sentence:

> The current package seeds the geometry-oracle model; full conformance requires viewport matrices, stable rendering environments, baseline governance, and fixture-backed false-positive measurement.

Add a sample UX report:

```json
{
  "rule_id": "UX-OVERLAP-001",
  "route": "/checkout",
  "viewport": "390x844",
  "selector": "[data-testid='pay-button']",
  "severity": "block",
  "evidence": {
    "screenshot": "target/jankurai/ux/checkout-mobile.png",
    "aria_snapshot": "target/jankurai/ux/checkout.aria.yml",
    "trace": "target/jankurai/ux/checkout-trace.zip"
  },
  "decision": "block"
}
```

Also distinguish hard and soft gates:

| Hard gate             | Soft/advisory gate              |
| --------------------- | ------------------------------- |
| Clipped required text | Aesthetic preference            |
| Horizontal overflow   | Minor anti-aliasing diff        |
| Missing focus state   | Subjective spacing disagreement |
| Inaccessible label    | Design taste concern            |
| Severe layout shift   | Brand review item               |

## XIII. Security, Supply Chain, and Permissions

**What works:**
The security section is practical. The repo includes `agent/security-policy.toml` and `jankurai security run`, with required/advisory tool handling and strict mode.

**Repo/code callout:**
The CLI has good safety gates for repair and proof behavior. The raw command surface shows environment-variable gates for unsigned proof commands and repair/git/PR mutation: `JANKURAI_ALLOW_UNSIGNED_PROOF_COMMANDS`, `JANKURAI_ALLOW_REPAIR_APPLY`, `JANKURAI_ALLOW_GIT_MUTATION`, and `JANKURAI_ALLOW_GITHUB_PR`. That is directionally correct. ([GitHub][10])

**But remove or rename hidden `--yolo`:**
The CLI args include a hidden `yolo` flag in init arguments. Even if harmless or internal, the word undermines the seriousness of a standard about bounded authority. ([GitHub][10])

Rename it to something boring and explicit:

```text
--accept-risk
--apply-unreviewed-plan
--unsafe-allow-write
```

Or remove it entirely.

**Add security requirements per level:**

| Level | Security requirement                                                                        |
| ----- | ------------------------------------------------------------------------------------------- |
| HL1   | Advisory secret scan and dependency posture report.                                         |
| HL2   | Blocking generated-zone mutation and critical secret findings.                              |
| HL3   | Required secret scan, dependency scan, CI permission check, and security evidence envelope. |
| HL4   | Ratcheted security score and no regression without waiver.                                  |
| HL5   | SBOM/provenance/release security receipts required.                                         |

Add a public/private evidence distinction. Public evidence bundles must not leak secrets, private file paths, internal architecture secrets, customer data, or incident details.

## XIV. Exceptions, Observability, and Repair Receipts

**What works:**
The repair-receipt idea is excellent. It turns failures into reusable agent memory.

**Terminology issue:**
The paper uses “exceptions” in two ways:

1. temporary standard waivers,
2. runtime/program exceptions.

Separate them.

Use:

```text
waiver = temporary scoped deviation from the standard
exception = runtime/program error
repair receipt = record of failed proof, repair, and accepted evidence
```

Rename:

```text
docs/exceptions/
```

to:

```text
docs/waivers/
```

and keep runtime exceptions in the observability section.

Add waiver lifecycle:

```text
proposed -> accepted -> active -> expiring -> expired -> removed
```

A waiver should require:

```yaml
id: JKR-WAIVER-2026-001
owner: apps/web
scope:
  - apps/web/src/legacy/**
reason: "legacy generated client pending contract migration"
expires: 2026-07-01
allowed_edits:
  - "copy-only"
forbidden_edits:
  - "authz"
  - "billing"
required_proof:
  - fast
  - web
```

## XV. Migration, Versioning, and Governance

**This section should be elevated.**

For a version-controlled standard, governance is not a back matter topic. It is the trust foundation.

**What works:**
No-write adoption is exactly right. The README also emphasizes safe first steps: `adopt` and `audit` read source files and write outputs under `target/jankurai`, which is the right adoption posture.  ([GitHub][1])

**What to add:**
A formal governance model:

| Governance item      | Recommendation                                                  |
| -------------------- | --------------------------------------------------------------- |
| Proposal process     | JEP: Jankurai Enhancement Proposal                              |
| Rule lifecycle       | draft → candidate → stable → superseded → obsolete              |
| Schema versioning    | Major for breaking, minor for additive, patch for clarification |
| Release evidence     | tag, checksums, generated docs, schemas, conformance fixtures   |
| Compatibility        | previous/current schema support policy                          |
| Extension namespaces | `HLT-*`, `ORG-*`, `PROFILE-*`, `EXPER-*`                        |
| Deprecation          | every superseded rule gets migration notes                      |
| Certification        | only from runnable conformance suite, not branding              |
| Badge policy         | level, version, date, auditor, scope required                   |

The `standard-version.schema.json` is currently too permissive for conformance. It only requires `standard` and `standard_version`, while other fields are optional. For real conformance, require at least:

```text
standard
standard_version
schema_version
auditor_version
paper_edition
published
target_stack or target_profile
artifacts[]
```

and require each artifact to include:

```text
id
path
kind
version_field
version
sha256
source
command
generated
canonical
```

The schema already has a good start; it just needs to become stricter for conformant claims. ([GitHub][11])

## XVI. Limitations and Research Agenda

**What works:**
This section is appropriately humble. It says weights need calibration, alternate stacks can conform, and audit cannot prove all semantic properties. That humility makes the paper more credible.

**What to add:**
Turn the research agenda into testable questions:

| Research question                    | Measurement                                 |
| ------------------------------------ | ------------------------------------------- |
| Does Jankurai reduce unsafe merges?  | Seeded-fault detection rate                 |
| Does it improve repair locality?     | Files touched, context size, time to repair |
| Does it reduce review time?          | Reviewer minutes per PR                     |
| Does it reduce escaped defects?      | Escaped defect rate before/after adoption   |
| Which rules are noisy?               | False-positive/false-negative report        |
| Which proof lanes are most valuable? | Bugs caught per CI minute                   |
| Does context-pack help?              | Token cost vs route accuracy                |
| Does UX QA catch real defects?       | Labeled browser-step corpus                 |

The repo’s tests directory already includes many promising test areas: adoption UX, audit smoke, generated zones, schema contracts, proof surface, security evidence, UX QA, versions, vibe coverage, detector fixtures, and more. That should be presented as implementation evidence, but the next step is a public conformance suite with expected outputs. ([GitHub][12])

## XVII. Conclusion

**Current strength:**
The conclusion is memorable and returns to verification.

**Main improvement:**
End on the standard, not the stack.

Suggested final paragraph:

> The repository must become a verification interface. A merge should not be trusted because the code looks plausible, the author sounds confident, or the agent completed a task. It should be trusted because changed paths route to owners, required lanes produce receipts, missing evidence is explicit, waivers expire, and repairs become reusable evidence. Jankurai’s core claim is therefore simple: no proof, no merge; no receipt, no trust.

```text
agent/standard-version.toml

VERSION
Cargo.toml
```

This is the single most important repo correction.

### 2. No published release

The README currently reports no published releases. That is fine for development, but a standard needs tagged releases and immutable artifacts. ([GitHub][1])

Add:

```text
GitHub Release

schema bundle

conformance fixtures
```

### 3. `main.rs` is too large

The CLI entrypoint is large. GitHub shows `crates/jankurai/src/main.rs` at 1,764 lines. ([GitHub][15])

Refactor into:

```text
src/cli/mod.rs
```

The command modules already exist in `src/commands`, which is good. The next step is making the top-level CLI mostly dispatch and metadata.

### 4. Add command metadata

Every command should declare machine-readable behavior:

```json
{
  "command": "repair",
  "writes": true,
  "network": false,
  "destructive": "gated",
  "requires_env": ["JANKURAI_ALLOW_REPAIR_APPLY"],
  "outputs": ["repair-run.schema.json"],
  "conformance_role": "repair_receipt"
}
```

This would let Jankurai audit itself.

### 5. Tighten schemas

The schemas are already a major strength. Now make them conformance-grade.

Specifically:

* Merge witness should require version identity and fingerprints.
* Standard-version schema should require more fields for conformant claims.
* Rule registry should require maturity and detector status.
* Proof receipts should bind commit, policy hash, input hashes, and artifact digests.
* Public evidence schemas should include redaction status.

### 6. Add conformance fixtures

Create:

```text
conformance/
  0.6.1/
    fixtures/
      minimal-hl1/

      handwritten-contract/

      prompt-injection-policy-tamper/

      *.json
      *.md
```

Then add:

```bash
jankurai conformance run conformance/0.6.1 --level HL3
```

A standard becomes real when outsiders can run its tests.

### 7. Make the repo self-auditing

The repo should claim its own level honestly:

```bash
jankurai audit . --mode standard
jankurai witness . --changed-from origin/main
jankurai versions --strict
```

Then publish the output under:


or a generated evidence branch/site.

# The strongest positioning

Do not position Jankurai as “better AI coding rules.” Position it as:

> **A versioned conformance standard for auditable merge evidence in human- and agent-authored software.**

The slogan remains:

> **No proof, no merge; no receipt, no trust.**

The thing to make famous is the merge witness:

```text
merge =
  changed paths
+ owner route
+ proof receipts
+ score delta
+ missing evidence decision
```

That equation is the paper’s breakthrough. The repo is already close enough to make it believable. The next leap is release discipline: version identity, schemas, conformance fixtures, public evidence, and honest implementation status.
