I reviewed the latest PDF as **Jankurai v0.6.1 / schema v1.4.1 / paper edition 2026.05-ed6**, and I also reviewed the public repository structure, README, schemas, docs, agent manifests, CLI surface, and key Rust source files. The paper’s center is still the right one: **Jankurai is not merely an AI coding opinion; it is an attempt to define an auditable merge standard**. The strongest form of the idea is:

> **A repository that claims trust must prove trust at merge time, with current, versioned, machine-readable evidence.**

That is profound because it moves software review away from memory, charisma, and “LGTM” toward a reproducible transaction: changed paths, owners, generated zones, proof lanes, receipts, evidence, exceptions, and a merge witness. The codebase shows that this is not just prose. The README describes Jankurai as a local audit CLI and trustworthy-merge standard that turns owner maps, proof lanes, generated zones, security evidence, rolling scores, merge witnesses, and repair queues into files, while starting read-only and leaving JSON/Markdown reports and proof artifacts behind. ([GitHub][1]) The uploaded paper’s abstract and opening sections make the same claim in standard form: “no proof, no merge; no receipt, no trust.”

My highest-level recommendation is unchanged but sharper after reading the code:

> **Split Jankurai into Jankurai Core, the Jankurai Auditor, and Jankurai Reference Profiles.**

The current paper is strongest when it presents **Jankurai Core** as a versioned conformance standard. It is weakest when the standard, the Rust/TypeScript/PostgreSQL default, streaming-roadmap opinions, language-stack scoring, paper manifesto, and CLI manual are all competing for the same attention. The codebase already has enough concrete machinery to support a serious standard: schemas, owner maps, test maps, generated zones, proof lanes, audit reports, proof receipts, evidence indexes, merge witnesses, rule registries, security evidence, UX QA artifacts, and version manifests. The paper should now reorganize around those concrete artifacts.

# Executive assessment after reading the paper and repository

The latest paper is materially stronger than the previous draft because it now aligns more closely with implemented code. The repository contains a Rust CLI package at version `0.6.1`, with a broad command surface for audit, adoption, initialization, update, doctor, context packs, witness, score diff/trend, rule export/verify, repair planning, proof planning/execution/verification, registry/cells, migration, benchmark/certification/governance/publication, adapters, CI, hooks, issues export, UX, security, and vibe coverage.  The package metadata also declares `jankurai` version `0.6.1`, MIT license, and the expected Rust CLI dependencies.

The repository structure supports the paper’s claim that this is becoming a real control plane. It includes `agent/`, `docs/`, `schemas/`, `crates/jankurai/`, `packages/ux-qa/`, `paper/`, `contracts/`, `db/`, examples, and tool-specific adapter directories such as `.agents`, `.claude`, `.cursor`, and `.github`. ([GitHub][1]) The schema directory is especially important: it already includes schemas for repo scores, proof plans, proof receipts, evidence indexes, merge witnesses, security evidence, owner maps, test maps, generated zones, standard versions, rule registries, score diffs/trends, repair queues, public evidence bundles, UX QA, vibe coverage, and more. ([GitHub][2])

The biggest opportunity is to stop presenting Jankurai as a brilliant argument and start presenting it as a **testable standard**. The codebase is now strong enough to support that move. The paper should say: “Here is the standard. Here are the artifacts. Here are the schemas. Here is the conformance suite. Here is a reference auditor. Here is one reference profile.”

The biggest risk is overclaiming. The paper currently uses language such as “absolute coverage” in Appendix A, includes stack rankings with exact scores, and contains a huge generated coverage table that visually implies more maturity than the detector status supports. The code is real and promising, but the repository page currently describes the project as “early but usable,” and no release has been published yet. ([GitHub][1]) The paper should embrace that honestly: **Jankurai is an early, runnable, versioned standard proposal with a reference auditor**, not yet an institutional standard or broadly validated benchmark.

# The most important strategic change

## Make the paper’s main object a conformance standard, not a manifesto or stack ranking

The central object should be:

```text
Jankurai Core Standard
  -> normative conformance model
  -> required artifacts
  -> schema versions
  -> rule registry
  -> proof-lane semantics
  -> evidence freshness
  -> merge witness decision
  -> governance and compatibility policy
```

Then separate:

```text
Jankurai Auditor
  -> reference Rust CLI
  -> schemas
  -> reports
  -> fixtures
  -> conformance suite
```

And separate again:

```text
Reference Profile RP-2026.05-RUST-TS-PG-PY
  -> Rust core
  -> TypeScript/React/Vite UI
  -> PostgreSQL durable truth
  -> generated contracts
  -> bounded Python AI/data service
```

This lets Go, Java, .NET, Rails, Elixir, Python, and enterprise teams adopt the **standard** without feeling they must accept the **stack opinion** first.

The standard’s most powerful formal predicate should look like this:

```text
A repository R at commit C conforms to Jankurai Core level L
under standard version S and schema version K iff an auditor A
can produce a valid audit report, evidence index, and merge witness
showing that all required artifacts, proof lanes, generated-zone
rules, security gates, exception rules, and repair receipts for L
are present, current, and schema-valid at C.
```

That one sentence should appear in Section VI, and all later sections should support it.

# What the code changes about the paper review

After reviewing the repository, I would **upgrade** the paper’s credibility in several places and **tighten** its claims in several others.

## What the code proves is real

The paper claims Jankurai emits version-bound artifacts. The code supports that. `model.rs` defines `STANDARD_VERSION = "0.6.1"`, `AUDITOR_VERSION = "0.6.1"`, `SCHEMA_VERSION = "1.4.1"`, `PAPER_EDITION = "2026.05-ed6"`, and the reference target stack ID. It also defines the main report model, proof receipts, artifact digests, UX QA summaries, tool-adoption readiness, security evidence readiness, boundaries readiness, vibe coverage, findings, manifest fingerprints, and agent-fix queue structures.

The paper claims proof receipts and evidence indexes matter. The code supports that strongly. `proof.rs` builds proof plans from changed paths, owner/test maps, required lanes, commands, expected artifacts, route decisions, planned runs, skipped lanes, and human approval requirements; it then executes commands, writes receipts, logs, artifact digests, receipt digests, and an evidence index. It also implements proof verification against plan digests and manifest fingerprints.

The paper claims merge witnesses are a public primitive. The code supports this too. `witness.rs` builds a merge witness containing changed paths, route decisions, generated-zone touches, required lanes, proof receipts, missing evidence, score deltas, caps, new/resolved/carried findings, decision, and next repair tasks. It exits nonzero when the witness decision is `block`, `ratchet_fail`, or `release_fail`.

The paper claims adapters should be thin mirrors over canonical policy. The repository has actual agent and tool surfaces: root instructions, agent manifests, owner maps, test maps, proof lanes, generated zones, security policy, UX policy, vibe coverage, and tool-specific adapter directories. ([GitHub][3])

## What the code reveals should be corrected

The code also reveals several concrete corrections.

First, **merge witness freshness needs to be stronger**. `witness.rs` currently checks that required lanes have successful proof receipts by lane name, but the witness should also verify receipt freshness against the current commit, changed paths, plan digest, command digest, manifest fingerprints, and evidence index. The existing `proof-verify` machinery already performs much of this work, so the witness should depend on a verified evidence index rather than merely loading successful receipts.

Second, **security strict mode should enforce parsed blocking steps in Rust**, not only the wrapper script exit code. `security.rs` parses structured `jankurai-security-step=...` rows and computes whether steps are required or blocking, but the final failure path still depends on the wrapper script’s exit code. A required tool marked failed or skipped should fail strict mode even if the shell wrapper exits `0`.

Third, **schema versioning needs clarification**. The global report says schema version `1.4.1`, but proof plan, evidence index, witness, and security evidence structures use their own hardcoded schema versions such as `1.0.0`, `1.2.0`, and `1.0.0`. That may be intentional, but the paper must distinguish:

```text
core_schema_version = version of the repo-score/report compatibility line
artifact_schema_version = version of a specific artifact schema
standard_version = normative standard version
auditor_version = implementation version
paper_edition = explanatory document edition
```

Right now, readers may assume `schema_version=1.4.1` applies uniformly to every artifact.

Fourth, **the owner map is flatter than the paper implies**. The paper and docs say owner maps should help define ownership and allowed dependency boundaries. The current `agent/owner-map.json` maps path prefixes to string owners, which is useful but not enough for full conformance.  A future schema should support owner IDs, risk tier, allowed dependencies, default proof lanes, reviewers, contact, docs, and exception policy.

Fifth, **the proof-lane manifest is promising but uneven**. The live `agent/proof-lanes.toml` contains named lanes such as `fast`, `audit`, `security`, `full`, `adapters`, `versions-check`, `db-migration-analyze`, `ux-qa`, and `observability`, but some lanes have rich metadata while others contain only name/command/purpose.  For HL3+ conformance, every lane should have a full metadata record: ID, command, kind, timeout, rules covered, required artifacts, network/destructive flags, environment requirements, and evidence freshness rules.

Sixth, **generated-zone semantics need refinement**. The current generated-zone manifest includes generated artifacts such as `paper/jankurai.pdf`, figure files, the generated vibe coverage table, `agent/repo-score.json`, `agent/repo-score.md`, and `package-lock.json`; some are `read_only=true`, others are `read_only=false`.  That is reasonable in practice, but the standard should not treat generated zones as only “read-only or not.” It should introduce a clearer field:

```toml
write_policy = "generator_only" | "auditor_output" | "lockfile" | "reviewed_manual"
```

Seventh, **there appears to be command drift worth auditing**. In `agent/test-map.json`, one legacy migration fixture command uses `cargo run -p jankurai -- migrate examples/legacy-node-api --json ...`, while the CLI arguments shown in `main.rs` use output fields named `out`/`md` for migration rather than `json`.   This may be harmless if aliases exist elsewhere, but it is exactly the kind of drift Jankurai should catch with `rules verify` or `proof-lanes verify`.

Eighth, **remove or rename hidden “yolo” affordances**. `main.rs` includes hidden `--yolo` initialization arguments.  Even if hidden and harmless, the name undercuts a standard whose central claim is disciplined, evidence-backed mutation. Rename it internally to something like `--bootstrap-commit-unsafe` or remove it before v1.0.

Ninth, **the schema validator should be positioned carefully**. The repository has a homegrown JSON Schema-like validator that supports a useful subset of JSON Schema and rejects unsupported keywords.  That is fine for a bootstrapping implementation, but tier-1 standard acceptance will require either a full JSON Schema validator or a formally named “Jankurai Schema Subset” with its own conformance tests.

# Highest-priority recommendations to make this a tier-1 standard

## 1. Define “Jankurai Core” as the only normative standard

The paper should repeatedly say:

```text
Jankurai Core is stack-neutral.
The Rust/TypeScript/PostgreSQL/Python architecture is a reference profile.
```

Then make the standard’s required artifacts independent of stack:

```text
agent/standard-version.toml
agent/owner-map.json
agent/test-map.json
agent/generated-zones.toml
agent/proof-lanes.toml
agent/audit-policy.toml
agent/security-policy.toml
target/jankurai/proof-plan.json
target/jankurai/evidence-index.json
target/jankurai/merge-witness.json
target/jankurai/repair-queue.jsonl
agent/repo-score.json
```

A Go or .NET repository should be able to pass Jankurai Core if it produces equivalent evidence.

## 2. Make the merge witness the hero artifact

The paper currently introduces the merge witness well, but too late. It should appear in the abstract or Section I.

A compact example would make the paper instantly clearer:

```json
{
  "decision": "block",
  "changed_paths": [
    "apps/web/src/generated/client.ts",
    "apps/web/src/pages/checkout.tsx"
  ],
  "generated_zone_touches": [
    {
      "path": "apps/web/src/generated/client.ts",
      "source": "contracts/openapi/api.yaml",
      "command": "just contract"
    }
  ],
  "required_lanes": ["contract", "web"],
  "missing_evidence": [
    "contract regeneration receipt",
    "Playwright checkout screenshot"
  ],
  "next_repair": [
    "change contracts/openapi/api.yaml and regenerate",
    "run web proof lane and attach screenshot/trace"
  ]
}
```

This shows what makes Jankurai different from a checklist.

## 3. Publish a conformance suite before making adoption claims

The paper should define a runnable conformance suite:

```text
conformance/
  fixtures/
    hlt-002-generated-mutation/
    hlt-004-unmapped-proof/
    hlt-010-secret-sprawl/
    hlt-013-rendered-ux-gap/
    hlt-021-destructive-migration/
    hlt-022-authz-isolation-gap/
    hlt-023-input-boundary-gap/
    hlt-024-agent-tool-supply-gap/
  expected/
    repo-score.json
    merge-witness.json
    repair-queue.jsonl
    evidence-index.json
```

The tier-1 claim should be: “A conforming auditor passes the suite.” That is much stronger than “the standard seems useful.”

## 4. Replace “absolute coverage” everywhere

The coverage table currently says “Green = absolute” and repeats “before claiming absolute coverage” across Appendix A. That wording is too brittle for security, governance, and academic review.

Use:

```text
implemented = detector-backed and emits evidence
governed = policy-backed but human/governance evidence required
partial = detector family exists but issue-specific fixture is missing
research = no deterministic detector yet
unmapped = no current Jankurai control
out_of_scope = intentionally not a repository-audit control
```

Then show a summary:

```text
implemented: N
partial: N
governed: N
research: N
unmapped: N
out_of_scope: N
```

A transparent status ledger will earn more trust than “no red rows.”

## 5. Move the huge vibe-coverage table out of the PDF body

Appendix A pages 18–30 are valuable as generated data, but not as a paper appendix. The table is too long, too repetitive, and too small to be useful in PDF form. It also visually overwhelms the core standard.

In the paper, keep:

```text
Table: coverage summary by TLR/rule/status
Table: representative 10–15 rows
Figure: detector-backed vs partial/governed/research counts
Artifact: agent/vibe-coverage.toml
Artifact digest: sha256:...
```

Move the full table to HTML, CSV, TOML, and JSON artifacts.

## 6. Add an empirical evaluation section

To reach tier-1 acceptance, add even a small evaluation:

```text
RQ1: Can Jankurai route changed paths to the expected owner and proof lane?
RQ2: Does the auditor detect seeded HLT faults?
RQ3: Does merge witness reduce review ambiguity?
RQ4: What is the false-positive rate by HLT family?
RQ5: How much CI/runtime overhead does Jankurai add?
RQ6: Can multiple coding agents obey the same canonical policy through adapters?
```

Use 5–10 repositories or fixtures:

```text
perfect-web-api-db
legacy-node-api
rust-api
react-web
python-ai-service
mixed-monorepo
```

Even a small result table would transform the paper from “proposal” to “standard plus evidence.”

## 7. Define governance like a real standard

Add a governance section with:

```text
JEP: Jankurai Enhancement Proposal
Rule registry maintainers
Schema compatibility policy
Conformance suite release process
Deprecation and supersession policy
Security disclosure process
Reference profile registry
Badge/trademark policy
Current/previous supported versions
```

The paper already borrows good standards mechanics. Now it should implement them.

## 8. Separate normative, informative, reference-profile, and implementation text

Use labels throughout:

```text
Normative: Required for Jankurai Core conformance.
Informative: Explanation and examples.
Reference profile: Required only for RP-2026.05-RUST-TS-PG-PY.
Implementation note: Behavior of the current Rust auditor.
```

This will prevent the paper from being rejected as “too opinionated” while preserving the strong reference profile.

## Abstract

### What works

The abstract correctly identifies the shift from plausible code generation to trustworthy merge. It also correctly lists the standard artifacts: conformance levels, stable rule IDs, generated zones, owner maps, test maps, version manifests, proof lanes, audit reports, proof receipts, evidence indexes, merge witnesses, repair queues, ratchets, and release gates.

### What weakens it

The abstract spends too much time defending what Jankurai is not. The sentence saying Jankurai does not have institutional standing with IETF/W3C/CNCF/SPDX/OpenAPI/OpenSSF is honest, but it belongs in limitations or governance. In the abstract it makes the paper sound defensive.

### Suggested revised abstract

> AI coding shifts the bottleneck in software engineering from plausible code generation to trustworthy merge. Human and agent-authored patches can compile, pass shallow tests, and still violate ownership, generated boundaries, security, data integrity, contract, or runtime assumptions. This paper presents **Jankurai Core**, a versioned repository conformance standard that makes merge decisions reproducible at a commit. A conformant repository binds changed paths to owner maps, generated-zone policy, proof lanes, security gates, evidence indexes, repair queues, exception expiry, and a machine-readable merge witness. The standard is stack-independent; a Rust, TypeScript/React/Vite, PostgreSQL, generated-contract, and bounded-Python profile is provided as one reference profile. Jankurai does not prove arbitrary program semantics. Its narrower claim is that repositories claiming agent-native conformance must expose current, machine-readable artifacts strong enough for local, pull-request, and CI auditors to test the claim before merge. The paper defines the conformance model, rule identifiers, audit artifacts, versioning policy, and research agenda needed to turn “vibe coding” into auditable engineering.

## I. New Bottleneck: Trustworthy Merge

### What works

This section is compelling. The loop is excellent:

```text
intent -> bounded authority -> proof lanes ->
evidence -> repair or exception expiry ->
reusable primitives
```

That loop is the standard’s soul. It should become **Figure 1**, not only a text block.

The section also correctly states that the same machinery must govern human-authored code. That is crucial. Jankurai should not become “AI code is suspicious, human code is trusted.” Its deeper claim is that **merge trust is independent of author identity**.

### What to improve

Define “trustworthy merge” immediately:

```text
A trustworthy merge is a merge decision that can be reconstructed
from changed paths, owner routes, generated-zone policy, proof
requirements, proof receipts, unresolved findings, exceptions,
versioned policy, and the merge witness at the commit being merged.
```

Add a concrete opening example:

```text
A pull request changes:
- apps/web/src/generated/client.ts
- apps/web/src/pages/checkout.tsx
- db/migrations/20260504_add_coupon.sql

A normal review asks:
- Does it look right?
- Did tests pass?

A Jankurai review asks:
- Why was a generated file edited?
- Which source contract produced it?
- Which owner owns checkout?
- Which proof lanes are required?
- Did the DB migration prove rollback/backfill/lock safety?
- Are rendered UX artifacts attached?
- What does the merge witness decide?
```

That example will make the paper’s significance land immediately.

### Fix a claim-count mismatch

The abstract says the paper makes **three contributions**, while Section I says it makes **four claims**. Harmonize this:

```text
The paper makes three standard contributions and one reference-profile argument.
```

```text
Rejection speed = the time and locality with which invalid changes are rejected
```

Keep this section’s core thesis:

## VI. Jankurai Standard and Conformance

### What works

This is the heart of the paper. It should be earlier and more formal. The install ladder, conformance levels, rule IDs, audit loop, and merge witness are the actual standard.

The repository supports this section well. The CLI exposes the claimed surfaces, including audit, adoption, initialization, update, doctor, context pack, witness, score diff/trend, rules verify/export, repair plan, proof plan/execution/verification, registry, cells, migration, benchmark, certification, governance, publication, adapters, CI, hooks, issues export, UX, security, and vibe coverage.

### Formalize conformance

Add this:

```text
A repository R at commit C conforms to Jankurai Core level L under
standard version S, schema line K, and auditor A iff A can emit
schema-valid artifacts proving every required predicate for L at C.
```

Then define required predicates per level.

### Rename HL0–HL5

`HL` is not self-explanatory. Consider:

```text
JCL0–JCL5 = Jankurai Conformance Level
```

or:

```text
JK0–JK5
```

If you keep `HL`, define it.

### Separate conformance claim types

Add:

```text
Self-attested conformance:
  The repository emits valid artifacts using a declared auditor.

Audited conformance:
  A third party verifies artifacts and conformance-suite results.

Certified conformance:
  A recognized conformance program verifies the repo/auditor/profile.
```

This prevents early repositories from overstating maturity.

### Make the merge witness stronger

The code already implements a merge witness, but the paper should require freshness semantics:

A witness is stale if any of the following changed after proof execution:

```text
changed paths
owner-map.json
test-map.json
generated-zones.toml
proof-lanes.toml
audit-policy.toml
security-policy.toml
standard-version.toml
auditor version
schema version
command string
command digest
toolchain lockfile
generated source contract
evidence artifact digest
git head
```

The existing proof verification code already compares plan digests, changed paths, manifest fingerprints, receipt digests, log digests, and artifact digests, so the standard can build on the implementation.

## VII. A 100-Point Stack Rubric

### What works

The rubric is a useful reference-profile scoring tool. Its weights are reasonable for agent-maintained product systems: correctness loop, security, runtime, concurrency, contracts, simplicity, ecosystem, observability, data integrity, and product velocity.

### Main risk

This section is politically risky. It can cause readers to reject the standard because they disagree with the stack ranking.

### Fix

Rename it:

> **A Reference-Profile Scoring Rubric**

Then say:

```text
This rubric is not required for Jankurai Core conformance.
It is a versioned policy rubric for selecting and justifying reference profiles.
```

The hard caps are more important than the exact scores. Emphasize the stack-neutral caps:

```text
No deterministic local proof lane.
No security lane for high-risk changes.
No generated contract discipline.
Wrong-layer DB access.
No secret/dependency scanning.
Overbroad agent authority.
No rendered UX proof.
False-green tests.
Destructive migration without evidence.
```

These are broadly acceptable across ecosystems.

## XVII. Conclusion

### What works

The conclusion is memorable and clear. The final line has energy.

### What to adjust

“Anything less is just faster vibe coding” is punchy, but it may sound dismissive in a standards paper. A more durable final paragraph:

> Agent-native engineering should not be judged by how much code it generates, but by whether a later human or agent can reconstruct why a change was safe to merge. Jankurai’s claim is deliberately narrow: a repository that claims conformance must produce current, versioned, machine-readable evidence for ownership, proof, exceptions, and repair. No proof, no merge. No receipt, no trust.

That preserves the slogan while making the paper sound more standard-grade.

# Code-level recommendations

## P0: Make witness depend on verified evidence, not just receipts

Current witness logic checks available proof receipts by lane and blocks on missing lane evidence.  That is good, but tier-1 conformance needs stronger freshness.

Require:

```text
witness consumes evidence-index.json
evidence index has been proof-verified
proof verification is pass
receipt git_head matches witness head or declared base/head range
changed_paths match
manifest fingerprints match
command digests match
artifact digests match
```

This turns the witness into an actual merge contract.

## P0: Enforce security strict mode in Rust

`security.rs` parses structured step rows and computes blocking status, but should fail if any required blocking step failed or was skipped, regardless of shell script exit.

Add:

```rust
let has_blocking_step = commands.iter().any(|step| step.blocking);
if args.strict && has_blocking_step {
    anyhow::bail!("security lane has blocking required-tool failures");
}
```

Also emit a normalized decision:

```json
"decision": "pass" | "warn" | "block"
```

## P0: Clarify schema version semantics

The code has global schema version constants and artifact-specific schema versions.    Make this explicit in schemas and paper:

```json
{
  "standard_version": "0.6.1",

    "name": "merge-witness",

}
```

# Final assessment

The paper has a real chance to become important because it names the right bottleneck: **merge trust**, not code generation. The repository makes the idea much stronger because many of the claimed artifacts already exist: a Rust CLI, schemas, maps, proof lanes, audit reports, proof receipts, evidence indexes, merge witnesses, security evidence, UX QA surfaces, generated-zone manifests, and versioned standard docs. ([GitHub][5])

The next version should become less like a manifesto and more like a standard package:

```text
Core standard
Reference auditor
Artifact schemas
Conformance suite
Reference profiles
Governance process
Evidence registry
```

The strongest version of Jankurai is not “Rust wins” and not “humans were bad.” The strongest version is:

> **No proof, no merge. No receipt, no trust. A repository that claims conformance must prove it with current, versioned, machine-readable evidence at the commit being merged.**
