I reviewed the latest 37-page paper and the GitHub repository as a static source review. I did not execute the repository locally, so the code feedback below is based on the GitHub-visible source, schemas, docs, workflow files, and repository structure rather than a test run.

The latest paper is **materially stronger** than the prior version. It now looks much more like an early standard with a reference implementation, not just a manifesto. The biggest improvement is that the paper now explicitly separates the **standard** from the **reference profile**, adds a better bounded claim, introduces concrete HL0–HL5 conformance language, includes a threat-model table, strengthens proof receipts/merge witnesses, expands UX QA, adds migration/versioning/governance detail, and admits key limitations. The central idea remains powerful: **Jankurai turns the repository into a versioned evidence interface for trustworthy merge.** The paper’s abstract now states this more clearly by naming conformance levels, stable rule identifiers, generated-zone policy, owner/test maps, proof lanes, evidence indexes, merge witnesses, repair queues, ratchets, release gates, and a separate Rust/TypeScript/PostgreSQL reference profile.

The next leap is to make the work feel less like “a paper plus a broad CLI” and more like **a credible, testable, version-controlled conformance standard**. The repository already supports that direction: the README describes Jankurai as a trustworthy-merge standard and local audit CLI, says it starts read-only, emits JSON/Markdown evidence, and turns owner maps, proof lanes, generated zones, security boundaries, rolling scores, merge witnesses, and repair queues into repository files. ([GitHub][1]) The repo also has a large schema surface for audit reports, proof receipts, proof verification, merge witnesses, security evidence, generated zones, test maps, public evidence bundles, repairs, rule registries, and related artifacts. ([GitHub][2]) That is exactly the right direction. Now the paper should promote those artifacts as the **public API of the standard**.

# Executive verdict

This version has moved from “provocative and promising” to **plausibly standardizable**.

The strongest paper identity is now:

> **Jankurai is a versioned repository conformance standard for trustworthy AI-assisted merge.**

That sentence should become the organizing principle. Everything else should be sorted into one of four buckets:

1. **Normative standard**: required artifacts, schemas, rule IDs, conformance levels, proof receipt validity, merge witness semantics, exit codes, compatibility rules.
2. **Reference implementation**: the `jankurai` Rust CLI.
3. **Reference profile**: Rust core, TypeScript/React/Vite product surface, PostgreSQL durable truth, generated contracts, bounded Python.
4. **Rationale and research**: vibe-coding taxonomy, stack scoring, language history, empirical agenda, streaming discussion.

Right now, the paper still blends those four layers. That is the main remaining weakness.

# What clearly improved since the earlier version

Several prior concerns were addressed well.

The abstract is much more standard-like. It now gives a bounded claim, says Jankurai does not prove all program semantics, and admits it lacks the institutional standing of IETF, W3C, CNCF, SPDX, OpenAPI, or OpenSSF. That humility is important. It makes the paper more credible, not less.

The opening thesis was softened in exactly the right direction. The new Section I says the public line is intentionally blunt, then clarifies that humans were not the bug because humans wrote software; the problem was repository organization around human memory, patience, taste, and slow review. That is a major tonal improvement.

The threat model is now real. Table I covers false plausibility, hostile context, tool overreach, secret exposure, dependency poisoning, and false-green tests. That is much better than treating “vibe coding” as only a cultural issue.

The RPN framing is more honest. Section V now says RPN is a Jankurai policy weight, not a universal incident statistic, and future releases should calibrate it against repair success, escaped defects, security regressions, and review time. That is exactly the kind of caveat the standard needs.

The conformance section is substantially stronger. Section VI now says HL3+ repositories MUST provide root agent routing, `agent/standard-version.toml`, `agent/owner-map.json`, `agent/test-map.json`, `agent/generated-zones.toml`, a one-command fast lane, audit JSON/Markdown, proof receipts or evidence index, and an ordered `agent_fix_queue`. That is one of the most important improvements in the paper.

The merge witness is now a centerpiece. The equation

```text
merge = changed paths + owner route
      + proof receipts + score delta
      + missing evidence decision
```

is still one of the paper’s best contributions. It should move earlier, but it is now more operationally supported by both paper and code.

The repository backs many paper claims. The docs define an adoption contract for HL3+ repositories, including root instructions, owner/test maps, generated-zone policy, standard version, audit output, fast/full commands, and CI audit. ([GitHub][3]) The code includes audit/report structures with version metadata, findings, fix queues, proof receipts, score history, generated-zone policy, security evidence, and fingerprints. ([GitHub][4])

# Prior feedback that was not fully followed

Several important recommendations were only partially followed.

First, the paper still over-centers the stack ranking. The abstract says the standard is separate from the reference profile, but Sections VII–IX still give the Rust/TypeScript/PostgreSQL stack a “default winner” framing. Section IX is still titled **“Default Winner Architecture.”** That should become **“Reference Architecture Profile.”** The word “winner” will trigger unnecessary resistance from Go, JVM, .NET, Python, Rails, Elixir, and TypeScript communities.

Second, the huge Appendix A coverage table still dominates the paper. The prior recommendation was to move the full row-by-row vibe-coding registry out of the paper and keep only a summary plus sample rows. This version still spends pages 18–30 on the generated coverage table. It is too repetitive and visually exhausting. It also still uses **“absolute”** as a coverage label on page 18, even though the changelog says broad “absolute” claims were downgraded in 0.6.1.  The repo changelog says v0.6.1 hardens the coverage registry, downgrades broad absolute claims, and regenerates the coverage table, but the paper still has “Green = absolute.” ([GitHub][5]) Use **“detector-backed”** instead.

Third, the page 30 formatting bug was not fixed. The paper still has `a) Rule legend.:` and a stray `{` at the bottom of the page. That was previously called out and remains visible.

Fourth, there is still no empirical evaluation section. Section XVI says the weights need calibration and the future-work table calls for a benchmark corpus, but the paper still does not show a pilot study, seeded defect set, false-positive/false-negative analysis, repair-time comparison, review-time reduction, or real repository conformance trial.  For tier-1 acceptance, this is now the largest substantive gap.

Fifth, the merge witness equation still appears too late. It belongs in Section I as a figure or boxed definition. Section VI is a good implementation section, but the equation is the paper’s sharpest conceptual artifact and should be introduced before the language/stack discussion.

Sixth, the title page is still too branded for a serious standards paper. The mascot/logo remains visually dominant, there is a stray purple mark near the subtitle, and the paper still lacks visible authorship, affiliation, status, license, and page numbers.

Seventh, the appendix still reads partly like CLI documentation. Appendix E is operationally useful, but it is too long for the paper. Keep the adoption/proof/witness commands in the paper and move the full command catalog to docs.

# The most important strategic recommendation

Make the next version a **standard package**, not just a paper.

Publish these as separate but version-bound artifacts:

```text
Jankurai Standard
  normative spec
  rule registry
  JSON Schemas
  conformance levels
  compatibility policy

Jankurai Conformance Suite
  fixtures
  expected JSON reports
  expected Markdown reports
  expected SARIF/JUnit output
  proof receipt validity tests
  merge witness validity tests

Jankurai Reference Implementation
  Rust CLI
  schemas
  adapters
  CI integration

Jankurai Reference Profile
  Rust + TypeScript/React/Vite + PostgreSQL + bounded Python

Jankurai Paper
  rationale, design, evaluation, limitations, roadmap
```

The paper should explicitly define the **Jankurai public API**:

> The Jankurai public API consists of conformance levels, rule identifiers, required artifact names, JSON Schema fields, proof-receipt validity rules, merge-witness decision semantics, CLI command/exit semantics, compatibility rules, and the conformance fixture suite.

That sentence would immediately make the work feel standard-grade.

## Abstract

The abstract is much improved. It now names the three key contributions and explicitly bounds the claim. That is good.

It can still be sharper. The abstract should define **trustworthy merge** directly rather than relying on the slogan.

Suggested revised abstract:

> AI-assisted coding shifts the scarce act in software engineering from drafting plausible code to proving that a change is safe to merge. This paper introduces Jankurai, a versioned repository conformance standard for trustworthy AI-assisted merge. Jankurai defines machine-readable owner maps, test maps, generated-zone policies, proof lanes, version manifests, stable rule identifiers, proof receipts, evidence indexes, repair queues, and merge witnesses. A conforming repository can reproduce why a change was accepted, rejected, routed for review, or blocked: which paths changed, who owned them, which proof lanes were required, which commands ran, what evidence was produced, which exceptions applied, and which release or ratchet gates were satisfied. The standard separates normative conformance from a non-normative reference profile based on Rust, TypeScript/React/Vite, PostgreSQL, generated contracts, and bounded Python. Jankurai does not prove all program semantics; it defines a versioned, testable evidence interface for repository governance in human- and agent-authored software.

That version is shorter, more precise, and more standard-like.

The next improvement is to separate **definitions** from **threat model**.

The threat model table should gain two columns:

```text
Out of scope
```

Table I is currently useful but slightly too optimistic. For example, “Owner map, test map, small files, generated contracts, typed errors” reduce false plausibility, but they do not eliminate hidden business-rule errors. Add residual-risk language.

The section is intellectually interesting. The language-history table gives philosophical depth. The risk is that it pulls the paper away from the standard.

The phrase “A model can imitate syntax” is memorable, but slightly too broad. A better version:

That is more precise.

Consider shortening Table II or moving it to the reference-profile rationale appendix. For a standards audience, the language-history framing is useful but should not compete with conformance.

The idea of “standard gravity” is excellent. It explains why technically elegant ecosystems do not automatically become default enterprise stacks. The Julia discussion is fair and not dismissive.

Still, this section should be shorter in the main paper. A tier-1 standard paper should not invite language-community side debates too early.

Keep the central sentence:

> Agent-native engineering rewards stacks where examples, tools, contracts, tests, and deployment conventions are abundant enough that agents can be constrained rather than merely prompted.

Then move most of Table III to a companion document called:

```text
Reference Profile Rationale
```

This would prevent critics from attacking language rankings instead of engaging the standard.

## Section V — Vibe-Coding Fault Taxonomy

This section is distinctive and valuable. It turns “vibe coding” into a real audit surface. The paper now correctly says RPN is a policy weight, not empirical truth. That is a major improvement.

However, the taxonomy still needs a formal scoring rubric. Define:

```text
Impact: 1–5
Likelihood: 1–5
Detection difficulty: 1–5
RPN = impact × likelihood × detection difficulty
```

Then define who assigns those values and how they are updated:

```text
RPN weights are standard-policy metadata.
Repos MAY adopt stricter local weights.
Repos MUST NOT claim global empirical frequency from RPN.
Jankurai releases SHOULD publish migration notes when RPN changes alter default gates.
```

Fix the version inconsistency in Figure 1. The caption says the chart is a policy-priority model for **jankurai 0.4.0**, while the paper version is 0.6.1. That looks like a stale caption.

Replace “absolute” coverage with “detector-backed.” The appendix still says “Green = absolute,” which is too strong and contradicts the more careful language elsewhere.

Recommended coverage language:

```text
Green = detector-backed
Yellow = governed / evidence-incomplete
Red = uncovered
```

Better still:

```text
✓ Detector-backed
△ Governed but fixture/evidence incomplete
✗ Uncovered
```

That also helps colorblind and grayscale readers.

## Section VI — Jankurai Standard and Conformance

This is the paper’s core. It is now much stronger. The adoption ladder and HL0–HL5 levels are practical and credible. The MUST language for HL3+ is one of the best additions.

The repository also supports this section well. The standard document in the repo defines HL3+ adoption requirements including root `AGENTS.md`, `agent/JANKURAI_STANDARD.md`, owner/test maps, generated zones, standard-version, audit output, fast/full commands, and CI audit. ([GitHub][3])

The next improvement is to define **claim language**.

Add a table:

| Claim                       | Allowed wording                        | Requirements                                             |
| --------------------------- | -------------------------------------- | -------------------------------------------------------- |
| Uses Jankurai tooling       | “Runs Jankurai audit in advisory mode” | HL1 artifacts                                            |
| Jankurai-guarded            | “Jankurai HL2 guarded”                 | critical caps and generated-zone blocking                |
| Jankurai-conformant         | “Jankurai HL3 conformant”              | current required artifacts and passing conformance suite |
| Jankurai-ratcheted          | “Jankurai HL4 ratcheted”               | no score regression without exception                    |
| Jankurai release-conformant | “Jankurai HL5 release contract”        | release evidence bundle                                  |

Add a normative rule:

> A repository MUST NOT claim Jankurai HL3 or higher conformance unless it publishes a current version manifest, owner map, test map, generated-zone policy, proof-lane map, audit report, proof receipts or evidence index, repair queue, and merge witness for the relevant merge or release.

Also define what is **normative** and **informative**. For example:

```text
Normative:
  conformance levels
  rule IDs
  required artifact names
  JSON Schemas
  proof receipt validity
  merge witness decision semantics
  compatibility policy

Informative:
  stack scoring rationale
  reference architecture
```

This is critical. Without it, readers may think the Rust stack is required for conformance.

The rubric is useful, but it is still the paper’s most attackable section.

It should be explicitly labeled:

> **Non-normative reference-profile scoring model**

The first paragraph should say:

> The ANSS rubric is not part of Jankurai conformance. It is the scoring model used to justify the first reference profile.

```text
Who can propose changes
What evidence can change weights
```


## Section VIII — Stack Ranking, Sensitivity, and Exceptions

The sensitivity discussion is useful. It acknowledges that Go, .NET, JVM, and TypeScript rise under different weights. Keep that.

The Kafka/Rust streaming subsection is too detailed and too time-sensitive for the main paper. It belongs in a companion profile note. The paper’s core standard does not need to evaluate Tansu, Iggy, Fluvio, and RobustMQ in the main body.

Use this shorter principle instead:

> Streaming infrastructure is workload-specific. Kafka or a Rust-native alternative may conform when broker clients are isolated behind owned adapters, event schemas live in contracts, generated event types live in generated zones, and migration claims are backed by replay, compatibility, security, observability, and recovery evidence.

That keeps the architectural insight without turning the standard paper into a streaming-platform comparison.

## Section IX — Default Winner Architecture

This section should be renamed.

Current:

> **Default Winner Architecture**

Recommended:

> **Reference Architecture Profile**

## Section XI — Continuous Proof, CI, and Test Explosion

This section is strong and practical. It explains the local/PR/CI loop, proof planning, proof execution, evidence indexing, proof verification, audit, witness, score diff, score trend, and repair queue.

The code strongly supports this section. The proof command includes controls for unsigned commands, proof plans, command receipts, evidence indexes, artifact digests, command/log digests, plan digests, and proof verification against the current repository state. ([GitHub][6])

The key improvement is to formalize **proof receipt validity** in the paper.

A proof receipt should be invalid if:

```text
repo SHA does not match candidate SHA
changed_paths do not cover the changed path set
command is not authorized by proof-lanes.toml
owner/test/generated-zone manifest fingerprints changed after proof
artifact digest is missing or mismatched
exit code is nonzero
receipt is stale beyond freshness window
runner identity is missing at HL4+
log digest is missing at HL4+
receipt is unsigned when signed receipts are required
```

The current proof receipt JSON Schema is too permissive for a conformance standard. It requires only `lane`, `command`, `exit_code`, `elapsed_ms`, and `artifacts`, and it allows additional properties. ([GitHub][7]) That is fine for early experimentation, but HL3/HL4/HL5 need stricter receipt profiles.

Recommendation:

```text
proof-receipt.schema.json        permissive base schema
proof-receipt.hl3.schema.json    required repo/head/changed_paths/owner/lane/command/artifacts
proof-receipt.hl4.schema.json    adds manifest fingerprints, log digest, command digest, runner identity
proof-receipt.hl5.schema.json    adds signed attestation/provenance requirements
```

The repository has a `packages/ux-qa` surface and the standard docs include TypeScript/web evidence expectations, including typecheck, lint, tests, Storybook, Playwright, and `@jankurai/ux-qa` output for UI changes. ([GitHub][3])

```text
Objective hard gates

  severe CLS
```

Add this rule:

## Section XIV — Exceptions, Observability, and Repair Receipts

This section is strong. The paper correctly treats repair as a governed loop, not permission for unattended mutation. It also says real application is gated behind explicit environment variables and reviewed flags, while high-risk autonomous mutation and automerge remain outside the standard.

The repository supports this with `repair-plan`, `repair`, `exceptions expire`, and related commands in the source tree. ([GitHub][9])

The improvement is to distinguish three concepts:

```text
Exception: temporary waiver from a rule.
Suppression: detector false positive.
Compensating control: alternate long-term evidence.
```

Each should have a different schema. A suppression should not look like a risk exception. A compensating control should not silently expire like a temporary waiver.

Add machine-readable templates in addition to Markdown:

```toml
id = "EX-2026-004"
kind = "exception"
rule_id = "HLT-021-DESTRUCTIVE-MIGRATION"
owner = "db"
scope = ["db/migrations/20260504_add_customer_archive.sql"]
reason = "legacy archival migration"
expires_on = "2026-06-01"
allowed_edits = ["migration and rollback only"]
forbidden_edits = ["application authz", "billing logic"]
required_lanes = ["db", "release"]
compensating_controls = ["staging restore proof", "lock-timeout proof"]
```

Table XIX’s minimum repair receipt fields are good but too minimal for HL4/HL5. Add:

```text
repo_sha
baseline_sha
started_at / finished_at
receipt_id
artifact digests
command digest
log digest
proof verifier result
exception/suppression references
human reviewer, if required
```

## Section XV — Migration, Versioning, and Governance

This section improved a lot. The no-write adoption path is exactly right. `adopt --mode observe`, `init --dry-run`, migration analysis, public evidence, version manifests, and TeX-as-canonical source all help the project feel more serious.

## Section XVII — Conclusion

The conclusion is strong and memorable. The final line is still too combative for a standards venue:

# Repository and code review

The repository is much more substantial than a paper-only project. It has a real Rust CLI, extensive schemas, docs, commands, workflows, and standard artifacts. The README positions it clearly as a local audit CLI and standard workspace, not a hosted AI service or model. ([GitHub][1]) The Cargo package is versioned 0.6.1, MIT-licensed, and uses Rust dependencies appropriate for a CLI/audit tool: `clap`, `serde`, `globset`, `ignore`, `rayon`, `regex`, `semver`, `toml`, `syn`, and others. ([GitHub][10])

## What is strong in the repo

The schema surface is excellent. The presence of schemas for audit policy, evidence index, generated zones, merge witness, proof plan, proof receipt, proof verification, repair, repo score, rule registry, security evidence, standard version, test map, tool adoption, UX, and vibe coverage is exactly what a version-controlled standard needs. ([GitHub][2])

The audit implementation appears aligned with the paper. It inventories the repo, loads policy, evaluates dimensions, applies caps, builds findings and repair queues, includes git summary and proof receipt data, computes fingerprints, includes version metadata, and writes structured reports. ([GitHub][4])

The proof system is one of the strongest implementation areas. The `prove` command creates proof plans, command receipts, evidence indexes, artifact digests, plan digests, command/log digests, and proof verification output. It also requires an explicit environment variable for unsigned proof commands. ([GitHub][6])

The proof verifier checks plan digest, changed paths, owner/test/generated/boundary/proof/standard manifest fingerprints, receipt digest, command digest, log digest, and failed exit codes. That is very much in the spirit of “no receipt, no trust.” ([GitHub][6])

The repo’s contributing guide reinforces the standard culture: read policy files before editing, do not hand-edit generated zones, add owner/test map entries when creating new paths, avoid fallback/TODO/stub language without exceptions, and run appropriate proof lanes. ([GitHub][11])

## Major code correction: the repo violates its own file-size doctrine

This is the clearest repo-level callout.

The standard document sets hard LOC limits for Rust files, including target and hard limits for application/domain/adapter code. ([GitHub][3]) But the repo currently has very large files:

```text
crates/jankurai/src/main.rs
  1764 lines, 1684 LOC, 52.7 KB

crates/jankurai/src/commands/proof.rs
  1469 lines, 51.1 KB
```

Those line counts are visible in the GitHub file views. ([GitHub][12])

This is not just style. For a project whose thesis is agent repair locality, mega-files weaken credibility. The tool should self-demonstrate the standard.

Recommended refactor:

```text
crates/jankurai/src/main.rs
  only CLI root and dispatch

crates/jankurai/src/cli/
  mod.rs
  args.rs
  dispatch.rs

crates/jankurai/src/commands/proof/
  mod.rs
  plan.rs
  execute.rs
  verify.rs
  receipt.rs
  evidence.rs
  command_policy.rs
  digest.rs
  render.rs
```

The paper should mention this as future work or, better, the repo should fix it before the next public push.

## Proof receipt schema is too permissive

The implementation verifies many fields, but the schema does not yet require enough for conformance. The proof receipt schema requires only a small set of fields and allows extra fields. ([GitHub][7]) That is good for iteration, but weak for HL3+ claims.

Add HL-specific stricter schemas, as noted above.

At minimum, HL3 proof receipts should require:

```text
standard_version
schema_version
auditor_version
repo_root or repo_id
git_head
changed_paths
owner
lane
command
exit_code
started_at
finished_at
artifacts with sha256
```

HL4 should require manifest fingerprints and log/command digests.

HL5 should require signatures or attestations.

## Merge witness freshness needs to be stricter

The merge witness schema is strong: it includes schema version, command, generated time, repo, git, changed paths, route decisions, generated-zone touches, required lanes, available receipts, missing evidence, score deltas, caps, finding diffs, decision, and next repair. It also defines the decision enum: `pass`, `review`, `block`, `ratchet_fail`, and `release_fail`. ([GitHub][13])

The next improvement is to ensure the witness does not merely find a receipt for a lane, but verifies that the receipt is fresh for the exact candidate state.

The witness should reject receipts when:

```text
receipt.git_head != current candidate head
receipt.changed_paths does not cover the changed path
receipt.plan_digest mismatches current proof plan
receipt.manifest fingerprints are stale
receipt.artifacts are missing
receipt.generated_at is older than freshness window
```

The paper already conceptually wants this. The code should make it unavoidable.

## CI workflow should model HL4/HL5 hardening

The workflow runs audit, versions, security, UX, paper build, and uploads artifacts. That is good. ([GitHub][8])

But for a standards project, the CI should itself be exemplary.

Recommended changes:

```text
Add top-level permissions.
Pin actions by SHA for strict/release workflows.
Upload SARIF when produced.
Run cargo fmt, clippy, and tests explicitly.
Run schema validation.
Run conformance fixtures.
Run proof-verify on generated proof receipts.
Use an explicit ratchet baseline if mode is ratchet.
Separate PR CI from release CI.
```

The workflow currently runs `audit --mode ratchet` in the visible command, but the excerpt does not show a baseline argument. ([GitHub][8]) Since the paper says ratchet compares against an accepted baseline, make this explicit in CI.

## The command surface is broad

The CLI surface is impressive but large. The paper’s Table VI lists many surfaces: adoption, context/adapters, proof ledger, audit/witness/score, security/UX, repair, cells/public evidence, Rust proof aids.  The source tree confirms many command modules. ([GitHub][9])

Expose it in:

The paper should state that only stable/candidate commands are part of the standard’s public API.

# Final assessment

This version is close to something important. The paper now has the right primitives and a real implementation behind them. The most profound idea is not the stack ranking. It is the auditable merge interface:

```text
changed paths
+ owner route
+ proof receipts
+ score delta
+ missing evidence decision
= reproducible merge decision
```

That is the part that can become widely adopted.

The paper’s next job is to become narrower, stricter, and more testable. Reduce the stack-war surface. Move giant registries out of the main paper. Add empirical evidence. Make conformance claims machine-verifiable. Make the repo self-conformant. Define the public API. Ship the conformance suite.

The project is strongest when it says:

> **No proof, no merge. No receipt, no trust. No conformance claim without current, versioned, machine-checkable evidence.**
