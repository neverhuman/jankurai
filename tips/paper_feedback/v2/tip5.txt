I studied the latest 37-page PDF and reviewed the public GitHub repository through GitHub and raw source views. I was not able to run a local clone/tests from this environment, so the code comments below are source-review recommendations, not execution results. The repository review still adds a lot: this is **not vapor**. The project already has a Rust CLI, schemas, agent manifests, docs, a TypeScript UX QA package, proof/evidence concepts, and a real local-control-plane shape. The paper should use that implementation more aggressively as evidence.

# Executive verdict

The latest paper is better than the previous version in several important ways. It now more clearly states that Jankurai is a **standard first** and a **reference profile second**. It adds stronger treatment of merge witnesses, browser-step QA, security evidence, exception receipts, migration, governance, and future work. The GitHub repository strengthens the case: the README defines Jankurai as a “trustworthy-merge standard and local audit CLI,” explains the no-proof/no-receipt rule, and documents read-only first adoption, proof lanes, merge witnesses, repair queues, and local artifacts. ([GitHub][1])

The core remains profound: **Jankurai’s real contribution is an auditable merge interface**. The standard’s essence is:

```text
changed paths
  -> owner route
  -> required proof lanes
  -> proof receipts / evidence index
  -> score and hard caps
  -> missing-evidence decision
  -> merge witness
  -> repair queue / exception expiry
```

That is the concept that can become a tier-1 standard. The paper should make that impossible to miss. It should not let readers leave thinking, “This is an opinionated Rust/TypeScript/PostgreSQL stack paper.” They should leave thinking, “Of course every serious repository should be able to prove why this merged.”

The biggest remaining problems are:

1. **The paper still mixes four artifacts:** standard, implementation report, reference profile, and manifesto.
2. **The title/subtitle still risks alienating adopters.**
3. **The standard is still too tightly bound to the Rust/TypeScript/PostgreSQL reference profile.**
4. **The paper still lacks a real evaluation section.**
5. **The repository has no published releases yet, despite claiming precise standard/auditor/schema versions.**
6. **Appendix A still overclaims with “absolute” coverage and “no red rows.”**
7. **The code appears to violate the standard’s own small-file/agent-repairability doctrine in several central files.**
8. **The PDF still has formatting, metadata, and appendix issues from the previous review.**

The next leap is not more rhetoric. The next leap is **standardization discipline**: stable schemas, conformance fixtures, signed evidence bundles, release tags, governance process, reference profiles, and self-conformance reports.

# What was not followed from the previous feedback

Several prior recommendations were not followed or were only partially addressed.

## 1. The title still includes “Humans Were the Bug”

The paper still uses:

> Humans Were the Bug: From Vibe Coding to Agent-Native Engineering

This is memorable, but still risky. It will attract attention, but it can also repel senior engineers, reviewers, open-source maintainers, standards people, and enterprise security teams who need to feel respected rather than blamed. The paper itself later clarifies that humans were not the bug because they wrote software; the bug was reliance on memory, taste, patience, and folklore. That clarification is good, but many readers will react before they reach it.

Best fix:

```text
Jankurai:
A Versioned Agent-Native Repository Standard for Trustworthy Merge

Auditable Merge Evidence for Human and AI-Authored Code
```

Keep “Humans Were the Bug” as a blog post, keynote tagline, or manifesto subtitle, not the formal standards-paper subtitle.

## 2. The large mascot/logo still dominates page 1

The visual identity is memorable, but the formal PDF still looks more like a project launch page than a tier-1 standards or research paper. The mascot should remain in the website, README, and community material. The formal PDF should have a restrained title page with author names, affiliations, version tuple, artifact links, license, and reproducibility statement.

## 3. The version mismatch in Figure 1 remains

Figure 1 still says the chart is a policy-priority model for **jankurai 0.4.0**, while the title page and body use **standard_version=0.6.1**, **schema_version=1.4.1**, and **paper_edition=2026.05-ed6**. This was called out previously and still needs to be fixed.

Use one of these:

```text
Introduced in Jankurai 0.4.0 and retained in 0.6.1.
```

or:

```text
The chart is a policy-priority model for Jankurai 0.6.1.
```

## 4. Appendix A still uses “absolute” coverage

The source-row coverage table still says:

> Green = absolute

and many yellow rows still say:

> before claiming absolute coverage

This remains a credibility problem. “Absolute” is the wrong word for a detector-backed software-audit standard. It invites critics to disprove one edge case and then dismiss the whole taxonomy.

Replace:

```text
absolute
```

with:

```text
detector-backed
```

or:

```text
implemented for narrowed fixture family
```

Then define:

```text
green = detector-backed for a narrowed issue family
yellow = mapped to a reviewed control family but pending issue-specific fixture, CI artifact, or governance receipt
red = unmapped or unsupported
```

## 5. Appendix A still says there are no red rows

The paper still says the v0.6.1 source has no red rows. That sounds good, but because most rows are partial, it can read like an overclaim. The safer and more credible statement is:

```text
The v0.6.1 registry maps every parsed source row to a reviewed rule family. Most rows remain partial pending narrowed fixtures, CI artifacts, or governance receipts.
```

That is still impressive and much harder to attack.

## 6. Appendix A is still too large for the main paper

Pages 18-30 remain dominated by the huge source-row coverage table. It is repetitive, visually overwhelming, and still contains questionable mappings. This should become supplemental material or generated registry documentation. The main paper should show a summary table, a count matrix, and 10 representative rows.

## 7. The orphaned `{` formatting issue remains

At the bottom of page 30, there is still a stray `{` before the JSON block continues on page 31. This was previously called out and should be fixed before any serious release.

## 8. Appendix D still says “canonical” and “normative target shape”

Appendix D is still titled “Canonical File Tree Diagrams,” and it still says the diagrams are the “normative target shape for a default jankurai repository.” This continues to weaken the paper’s claim that the standard is stack-neutral.

Rename it:

```text
Appendix D: Rust/TypeScript/PostgreSQL Reference Profile File Tree
```

Then say:

```text
This tree is normative for the 2026.05 Rust/TypeScript/PostgreSQL reference profile, not for core Jankurai conformance. Other profiles may conform if they provide equivalent ownership, proof-lane, generated-zone, evidence, and repair surfaces.
```

## 9. The paper still lacks a real evaluation section

The Limitations section now says scoring weights need calibration and calls for research, but the paper still has no actual evaluation: no fixture repos, no pilot repos, no seeded faults, no false-positive analysis, no ablations, no review-time measurements, no conformance results.

For tier-1 acceptance, this is the largest missing piece.

## 10. The repository still has no published release

The GitHub repository currently shows **no releases published**, while the paper and source constants claim standard/auditor/schema/paper versions. That is a mismatch for a version-controlled standard. A versioned standard needs tags, release notes, checksums, schemas, generated artifacts, and reproducible evidence bundles. ([GitHub][1])

# Repository and code-informed findings

The repository is stronger than the paper makes it sound. The README says Jankurai turns ownership maps, proof lanes, generated zones, security boundaries, rolling scores, merge witnesses, and repair queues into files that agents and humans can read, and it emphasizes local artifacts rather than a hosted service. That is exactly the right positioning for trust. ([GitHub][1])

The repository structure already reflects a serious control plane. It includes `.agents`, `.claude`, `.cursor`, `.github`, `agent`, `contracts`, `crates/jankurai`, `db`, `docs`, `examples`, `packages/ux-qa`, `paper`, `reference`, `schemas`, `tips`, and `tools`. That is a real standards-and-reference-auditor workspace, not just a paper. ([GitHub][1])

The CLI surface is broad. The README lists commands for adoption, context, adapters, proof/evidence, audit/routing, security/UX, repair/expiry, reusable cells, certification, governance, and publishing. The `main.rs` parser also exposes many of those command families directly: audit, adopt, init, update, doctor, context-pack, witness, score, rules, repair-plan, lane/proof/prove/proof-verify, registry, cell, migrate, bench, certify, govern, publish, repair, optimize, Rust aids, exceptions, adapters, agent verify, CI install, hooks install, issues export, explain, versions, UX, security, and vibe coverage/validation. ([GitHub][1])

The source model already binds the key version constants: `STANDARD_VERSION=0.6.1`, `AUDITOR_VERSION=0.6.1`, `SCHEMA_VERSION=1.4.1`, `PAPER_EDITION=2026.05-ed6`, and `TARGET_STACK_ID=rust-ts-vite-react-postgres-bounded-python`. It also defines serialized report fields for fingerprints, dirty-worktree state, schema URL, versions, target stack, scope, score, decisions, Git summary, policy, proof receipts, caps, dimensions, UX QA, tool adoption, security evidence, boundaries, vibe coverage, findings, and the agent fix queue. ([GitHub][4])

This is both good and revealing. Good: the evidence model is concrete. Revealing: the current model still hard-binds the **paper edition** and **target stack** into core reports. For a stack-neutral standard, those should be separated into optional provenance/profile fields.

The docs already define an HL3 adoption contract requiring root AGENTS guidance, `agent/JANKURAI_STANDARD.md`, owner map, test map, generated-zone manifest, standard-version manifest, CI-produced repo score, fast/full validation, and PR audit. They also define stable HLT-001 through HLT-027 rule IDs and hard gates for root instructions, validation, ownership, contract drift, generated files, file/function size, fallbacks, duplicate behavior, DB misuse, Python sprawl, security lanes, and disabled tests. ([GitHub][5])

That means the paper should stop apologizing quite so much and instead show the real artifact surface: schemas, CLI, docs, manifests, expected reports, and conformance fixtures.

# Most important code corrections and future work

## 1. Add a self-conformance report for Jankurai itself

This is the biggest code-related recommendation. Jankurai should publish its own current self-audit:

```bash
jankurai audit . \
  --mode standard \
  --json target/jankurai/self/repo-score.json \
  --md target/jankurai/self/repo-score.md \
  --sarif target/jankurai/self/jankurai.sarif \
  --junit target/jankurai/self/jankurai.junit.xml

jankurai witness . \
  --changed-from origin/main \
  --baseline agent/repo-score.json \
  --out target/jankurai/self/merge-witness.json \
  --md target/jankurai/self/merge-witness.md
```

Then publish those artifacts with every release.

This matters because the project’s own standard emphasizes small files, explicit ownership, proof lanes, generated zones, security gates, and repair queues. If the Jankurai repo itself does not visibly pass Jankurai, critics will use that against the standard.

## 2. Split oversized central files

By inspection, the CLI entry point centralizes a very large amount of parser and dispatch structure, and the audit module appears to carry many responsibilities. The standard itself says large files and large functions harm agent repairability, and the docs define too-large files/functions as hard gates. ([GitHub][6])

Recommended split:

```text
crates/jankurai/src/
  cli/
    mod.rs
    args.rs
    dispatch.rs
    output.rs
    commands/
      audit.rs
      init.rs
      proof.rs
      witness.rs
      repair.rs
      security.rs
      ux.rs
      governance.rs
  audit/
    run.rs
    decision.rs
    fingerprints.rs
    evidence.rs
    findings.rs
    report.rs
    ratchet.rs
    release.rs
  model/
    mod.rs
    report.rs
    finding.rs
    proof.rs
    witness.rs
    security.rs
    ux.rs
    governance.rs
    profile.rs
```

This is not aesthetic. It is self-conformance. The standard argues that agents repair better with small files, narrow ownership, and explicit routing. Jankurai should be the exemplar.

## 3. Separate core standard fields from profile fields

The model currently includes `paper_edition`, `target_stack_id`, and `target_stack` as report fields. That matches the paper, but it weakens stack neutrality. ([GitHub][4])

Use this instead:

```json
{
  "standard": {
    "id": "jankurai",
    "version": "0.6.1",
    "schema_version": "1.4.1"
  },
  "auditor": {
    "id": "jankurai-reference-auditor",
    "version": "0.6.1"
  },
  "claim": {
    "level": "HL3"
  },
  "profile": {
    "id": "rust-ts-vite-react-postgres-bounded-python",
    "version": "2026.05",
    "optional": true
  },
  "publication": {
    "paper_edition": "2026.05-ed6",
    "optional": true
  }
}
```

Core conformance should not require a Rust/TypeScript/PostgreSQL target stack.

## 4. Remove the hidden `--yolo` alias

The CLI still exposes a hidden deprecated `yolo` flag in the init arguments. Even if hidden, the word contradicts the standard’s seriousness. ([GitHub][6])

Remove it completely before v1.0. The replacement should be boring and auditable:

```bash
jankurai init . --level full --apply --yes --receipt target/jankurai/init-receipt.json
```

## 5. Align report decisions with the paper’s merge decisions

The paper says merge witnesses should decide among:

```text
pass, review, block, ratchet fail, release fail
```

The model’s `ReportDecision` currently appears centered around status, minimum score, passed boolean, hard/soft finding counts, and optional ratchet metadata. ([GitHub][4])

Align the schema:

```json
{
  "decision": "block",
  "reason": "missing_required_proof_receipt",
  "mode": "standard",
  "blocking_findings": [],
  "review_findings": [],
  "ratchet": null,
  "release": null
}
```

This will make the implementation match the paper’s strongest primitive.

## 6. Publish release artifacts immediately

The repo currently has no published releases, despite the standard and code claiming exact versions. ([GitHub][1])

For v0.6.1 or v0.7.0, publish:

```text
jankurai-standard-0.7.0.md
jankurai-reference-auditor-0.7.0.tar.gz
jankurai-schemas-1.5.0.zip
jankurai-conformance-fixtures-0.1.0.zip
jankurai-paper-2026.05-ed7.pdf
checksums.txt
SLSA/provenance attestation
self-audit.json
self-audit.md
self-merge-witness.json
```

A versioned standard without versioned release artifacts will not gain standards trust.

## 7. Make the schema directory the center of the paper

The repository’s `schemas/` directory is a major strength. It includes schemas for evidence index, merge witness, owner map, proof lanes, proof receipt, public evidence bundle, repair queue, repo score, rule registry, security evidence, standard version, test map, UX QA, and vibe coverage. ([GitHub][3])

The paper should include a figure:

```text
Jankurai Standard
  -> normative prose
  -> JSON Schemas
  -> Reference Auditor
  -> Conformance Fixtures
  -> Evidence Bundles
  -> Reference Profiles
```

Right now, the paper undersells how much machine-readable standard infrastructure already exists.

# Tier-1 standard strategy

To become a real standard rather than a strong project, Jankurai should split into six public artifacts.

## 1. Jankurai Core Standard

This is the normative, stack-neutral standard.

It should define:

```text
MUST owner-map
MUST test-map
MUST generated-zone policy
MUST proof lanes
MUST audit report
MUST proof receipts or evidence index
MUST merge witness at HL3+
MUST repair queue for blocking findings
MUST version manifest
MUST exception expiry semantics
SHOULD SARIF/JUnit integration
MAY reference profile
```

It should not require Rust, TypeScript, PostgreSQL, Python, `paper_edition`, or `target_stack_id`.

## 2. Jankurai Evidence Schemas

This should be an independent schema release.

Version it separately:

```text
schemas/
  repo-score.schema.json
  finding.schema.json
  proof-receipt.schema.json
  evidence-index.schema.json
  merge-witness.schema.json
  owner-map.schema.json
  test-map.schema.json
  generated-zones.schema.json
  exception.schema.json
  repair-queue.schema.json
  public-evidence-bundle.schema.json
```

The repo already has much of this. Promote it. ([GitHub][3])

## 3. Jankurai Reference Auditor

This is the Rust CLI.

It should be clearly named:

```text
jankurai-reference-auditor
```

The current CLI can remain `jankurai`, but the paper should call it the reference auditor, not the standard itself.

## 4. Jankurai Conformance Suite

This is essential.

Suggested tree:

```text
conformance/
  fixtures/
    ownerless-path/
    unmapped-proof/
    generated-zone-mutation/
    handwritten-contract/
    missing-secret-scan/
    destructive-migration/
    overbroad-agent-permission/
    rendered-ux-gap/
    stale-proof-receipt/
    tampered-evidence-index/
  expected/
    ownerless-path.repo-score.json
    ownerless-path.merge-witness.json
```

Suggested command:

```bash
jankurai conformance run \
  --standard-version 0.7.0 \
  --fixtures conformance/fixtures \
  --expected conformance/expected
```

Without this, “conformant” is still more aspirational than testable.

## 5. Reference Profiles

Create named profiles:

```text
profile/rust-ts-postgres/2026.05
profile/go-ts-postgres/2026.05
profile/dotnet-ts-postgres/2026.05
profile/jvm-ts-postgres/2026.05
profile/fullstack-ts-postgres/2026.05
profile/python-ai-boundary/2026.05
```

The Rust/TypeScript/PostgreSQL stack can remain the flagship profile, but it should not be the core standard.

## 6. Public Evidence Registry

The repo already has commands such as `bench`, `certify`, `govern`, and `publish` documented as public-evidence surfaces. ([GitHub][1])

Make this real:

```text
evidence/
  releases/
    v0.7.0/
      self-audit.json
      self-witness.json
      schema-validation.json
      conformance-results.json
      checksums.txt
```

A badge should point to immutable current evidence, not just a marketing claim.

## Abstract

The abstract is now much stronger. It correctly states that the scarce act is trustworthy merge, not plausible code generation. It names the standard artifacts, conformance/audit mechanisms, and reference profile split.

Still improve it in three ways.

First, mention the merge witness directly in the first paragraph:

```text
Jankurai makes the merge decision reproducible by emitting a merge witness: a versioned artifact that binds changed paths, owner routing, required proof lanes, current receipts, score deltas, missing-evidence decisions, and repair obligations.
```

Second, reduce the institutional disclaimer. It is honest, but the abstract spends too much prime space saying what Jankurai is not. Move some of that to Limitations.

Third, add one sentence about implementation artifacts:

```text
The reference implementation includes a local Rust auditor, JSON schemas, repository manifests, proof/evidence commands, and UX/security evidence adapters.
```

The GitHub repo supports that claim. ([GitHub][1])

## Section I — New Bottleneck: Trustworthy Merge

This remains the heart of the paper. The operating loop is excellent:

```text
intent -> bounded authority -> proof lanes ->
evidence -> repair or exception expiry ->
reusable primitives
```

That loop should become Figure 1, replacing or preceding the RPN pie chart. The pie chart is secondary; the operating loop is the standard.

The opening still leans too hard on the provocative human-bug frame. Keep the clarification, but shift the lead:

```text
The failure was not human authorship. The failure was unauditable merge.
```

Add one concrete PR story:

```text
A PR modifies a generated client, adds a dependency, and changes a payment route. A normal reviewer sees plausible code. Jankurai sees three merge questions: generated-zone mutation, supply-chain drift, and payment invariant proof. The merge witness blocks until the source contract changes, the dependency rationale and scan exist, and the payment lane emits a receipt.
```

That example would make the paper instantly easier to understand.

## Section II — Definitions and Threat Model

The definitions are strong but too dense. Convert them into a glossary table.

Suggested table:

| Term             | Normative meaning                                                    | Artifact                                        |
| ---------------- | -------------------------------------------------------------------- | ----------------------------------------------- |
| Proof lane       | Deterministic command or CI job tied to path/change class            | `agent/test-map.json`, `agent/proof-lanes.toml` |
| Merge witness    | Versioned merge-decision artifact                                    | `target/jankurai/merge-witness.json`            |
| Repair queue     | Ordered findings with owner, lane, evidence, and bounded next action | `agent_fix_queue`                               |
| Generated zone   | Output path repaired by changing source and regenerating             | `agent/generated-zones.toml`                    |
| Centerline drift | Distance between claimed standard and observed repo state            | `repo-score.json`                               |

Add threats to Jankurai itself:

```text
forged receipts
stale receipts
tampered evidence index
owner-map manipulation
test-map manipulation
disabled CI
malicious adapter sync
hidden Unicode in policy files
secret leakage in artifacts
tampered baselines
overbroad repair automation
```

The repo’s model already includes fingerprints and manifest fingerprints, which can support this section. ([GitHub][4])

## Section III — Languages as Bottleneck Compression

This section is intellectually interesting, but it still competes with the main standard. It should be shortened in the main paper and moved mostly to an appendix or reference-profile document.

Keep only the standard-relevant message:

```text
In the agent era, language choice matters because it affects rejection speed, proof locality, boundary clarity, and repairability.
```

Then move quickly back to repository artifacts.

## Section IV — Technical Promise Versus Standard Gravity

The Julia discussion is fair and respectful, but it still feels like a language-adoption essay inside a repository-standard paper. Rename this section:

```text
Ecosystem Gravity and Agent Repairability
```


## Section V — Vibe-Coding Fault Taxonomy

This section is valuable. It turns “vibe coding” from a vibe insult into an audit surface. That is one of the paper’s strongest moves.

But it still needs four fixes.

First, define the RPN scale. The paper says:

```text
RPN = impact × likelihood × detection difficulty
```

but does not define the numbers. Add:

```text
Impact: 1-5
Likelihood: 1-5
Detection difficulty: 1-5
```

Then include one worked example.

Second, replace “Hard/Soft” with more precise detector maturity:

```text
deterministic
heuristic
governance-backed
research
```

Third, stop saying “no red rows” in the main argument. Say “all rows are mapped to a reviewed control family.”

Fourth, fix the taxonomy mappings in Appendix A. More on that below.

## Section XI — Continuous Proof, CI, and Test Explosion

This section is strong and code-backed. The README documents a daily loop using `context-pack`, `prove`, `audit`, and `witness`, and the command surface includes proof and evidence commands. ([GitHub][1])

Add freshness semantics:

```text
A receipt is current only if:
  git_head matches the candidate commit
  plan_digest matches the proof plan
  command_digest matches the allowed command
```

Current ending emphasizes the default target: Rust core, TypeScript/React/Vite, PostgreSQL, generated contracts, bounded Python.

Better ending:

```text
Jankurai’s central claim is author-neutral: neither human confidence nor model confidence is merge evidence. A conformant repository must expose current ownership, proof, receipt, witness, and repair artifacts sufficient to audit why a change merged. Reference profiles show one way to do this. The standard’s larger purpose is to make trustworthy merge portable, testable, and improvable across stacks.

No proof, no merge. No receipt, no trust.
```

# The strongest final positioning

The best version of the paper is not:

> AI coding is dangerous, so use Rust/TypeScript/PostgreSQL.

The best version is:

> AI-assisted development makes unauditable merge decisions the central engineering bottleneck. Jankurai defines a versioned, testable, stack-neutral repository standard that requires ownership, proof lanes, receipts, evidence indexes, merge witnesses, and repair queues before high-risk changes merge.

That framing can travel across languages, companies, security teams, open-source projects, and agent tools.

The concept is strong enough to become a real standard. The next step is to make the standard impossible to confuse with a stack manifesto, impossible to claim without evidence, and impossible to update without versioned governance.
