## Overall read: this paper has a real, important center

I read the full 37-page PDF, including the rendered pages, tables, figures, appendices, command examples, and references. The core insight is strong: **AI coding makes code generation cheap, so the new scarce resource is trustworthy merge**. The most valuable contribution is not “use Rust” or “use this stack”; it is the proposed repository control plane: owner maps, generated-zone rules, proof lanes, receipts, evidence indexes, merge witnesses, repair queues, versioned manifests, and ratcheting conformance. That is the part that could become a widely used standard.

The paper is already much more substantial than a manifesto. It has definitions, a threat model, stable rule identifiers, conformance levels, command examples, appendices, audit artifacts, and an explicit version manifest. The key opportunity is to **separate the standard from the persuasion**. Right now the draft reads like four things at once: a provocative AI-era manifesto, a standards proposal, a reference architecture argument, and an implementation manual. That energy is valuable, but tier-1 acceptance will require a clearer hierarchy:

**The standard:** “Here is the testable repository interface for auditable AI/human merge.”

**The proof:** “Here is the conformance suite, schema, witness, fixtures, and evidence that make it testable.”

**The reference profile:** “Here is the default Rust/TypeScript/PostgreSQL/bounded-Python profile that satisfies the standard well.”

**The movement language:** “No proof, no merge; no receipt, no trust.”

The paper should keep the shock, but make the shock come from the reproducible merge witness, not from rhetoric alone.

I also spot-checked several load-bearing external claims. The environment the paper describes is real: GitHub’s 2025 Octoverse says TypeScript reached #1 on GitHub and frames AI as a major force in development; DORA’s 2025 report frames AI as an amplifier of existing organizational strengths and weaknesses; GitGuardian’s 2026 report page says AI-service leaks surged and reports 28.65 million new secrets in public GitHub commits in 2025. ([The GitHub Blog][1]) Current tool references are also plausible: Microsoft has an official TypeScript 7 beta announcement, Vite has an official Vite 8 announcement, and OpenAI’s Codex docs describe `AGENTS.md` project guidance. ([Microsoft for Developers][2]) That external support gives the paper a credible moment; now it needs standard-grade polish.

# Highest-leverage changes before publication

## 1. Make the paper’s one-sentence contribution impossible to miss

The current abstract has many strong pieces, but the central novelty should be stated even more sharply:

> **Jankurai turns a pull request merge decision into a reproducible, versioned audit artifact.**

That sentence, or something close to it, should appear in the title page, abstract, introduction, and conclusion. The current “no proof, no merge; no receipt, no trust” slogan is excellent, but the paper needs a formal sentence that reviewers, CTOs, standards people, and tool vendors can quote.

## 2. Split “standard” from “reference profile”

This is the biggest adoption risk. The standard is broadly useful. The Rust/TypeScript/PostgreSQL/bounded-Python stack recommendation is more controversial. If the paper makes readers feel that adopting Jankurai means accepting the entire stack ranking, many Go, JVM, .NET, Python, Rails, and enterprise teams will bounce before they understand the standard.

Recommended framing:

> **Jankurai Core**: stack-neutral conformance requirements.
> **Jankurai Reference Profile RUST-TS-PG-01**: one recommended profile satisfying the core standard.
> **Jankurai Alternate Profiles**: Go, .NET, JVM, Rails, Python/FastAPI, Elixir, etc., allowed when they provide equivalent proof artifacts.

Keep the stack ranking, but move it from “main claim” to “reference profile rationale.” This preserves the argument while making the standard adoptable.

## 3. Add an Evaluation section

A tier-1 paper needs evidence beyond plausible architecture. The current draft admits the weights need calibration, which is good, but it needs at least one concrete evaluation path. Add a section before Limitations:

**XV. Evaluation and Early Evidence** could include:

| Evaluation question                   | Minimum evidence to include                                      |
| ------------------------------------- | ---------------------------------------------------------------- |
| Does Jankurai reduce owner ambiguity? | Before/after changed-path routing accuracy on sample repos.      |
| Does it improve proof selection?      | Percentage of changed paths mapped to deterministic proof lanes. |
| Does it reduce false-green merges?    | Seeded defects caught by rule families and proof lanes.          |
| Does it help agents repair?           | Time/steps to repair with and without `agent_fix_queue`.         |
| Does it reduce review burden?         | PR review time or reviewer comments before/after.                |
| Does it preserve velocity?            | Median local/PR lane runtime.                                    |
| Does it improve security posture?     | Secret/dependency/permission findings caught before merge.       |

Even a small case study would transform the paper from “compelling proposal” into “credible standard candidate.”

## 4. Make the merge witness the hero artifact

The merge witness is probably the most important invention in the paper. The equation on page 7 is excellent:

> merge = changed paths + owner route + proof receipts + score delta + missing evidence decision

This should become Figure 1, not a later section detail. Put a simplified merge-witness JSON and Markdown example in the first third of the paper. Show the exact before/after PR decision:

```json
{
  "decision": "block",
  "changed_paths": ["apps/web/src/generated/client.ts"],
  "owner_route": "contracts",
  "required_lanes": ["contract"],
  "missing_evidence": ["regeneration_receipt"],
  "finding": "HLT-002-GENERATED-MUTATION",
  "repair": "change contracts/openapi/api.yaml and regenerate"
}
```

This is the “shocking” part: the merge decision becomes an auditable object.

## 5. Add a formal conformance table

The conformance levels HL0-HL5 are useful, but the paper needs a crisp table that says exactly what is required at each level. The current table defines the level meaning, but not the artifact checklist.

Add:

| Artifact / behavior       |    HL1 |     HL2 |     HL3 |     HL4 |  HL5 |
| ------------------------- | -----: | ------: | ------: | ------: | ---: |
| Root agent router         | SHOULD |    MUST |    MUST |    MUST | MUST |
| Owner map                 | SHOULD |    MUST |    MUST |    MUST | MUST |
| Test map                  | SHOULD |    MUST |    MUST |    MUST | MUST |
| Generated-zone manifest   | SHOULD |    MUST |    MUST |    MUST | MUST |
| Audit JSON                |   MUST |    MUST |    MUST |    MUST | MUST |
| Audit Markdown            |   MUST |    MUST |    MUST |    MUST | MUST |
| Proof receipts            |    MAY |  SHOULD |    MUST |    MUST | MUST |
| Merge witness             |    MAY |  SHOULD |    MUST |    MUST | MUST |
| Score floor blocks merge  |     no | partial |     yes |     yes |  yes |
| Ratchet blocks regression |     no |      no |      no |     yes |  yes |
| Release gates             |     no |      no | partial | partial |  yes |

This will make adoption and certification much clearer.

## 6. Replace repeated appendix filler with dense evidence

Appendix A’s vibe-coding coverage table spans pages 18-30 and repeats the phrase “needs issue-specific fixture, CI artifact, or governance receipt evidence…” many times. The existence of the registry is useful, but the repetition weakens the paper visually and rhetorically. It looks generated rather than curated.

Better structure:

1. Keep a one-page summary table in the paper.
2. Move the full row-level registry to a supplemental artifact or repository.
3. Include counts by coverage status: green/yellow/red.
4. Include only the most interesting 12-20 representative rows in the PDF.
5. For yellow rows, use concise gap categories instead of full repeated sentences.

Example:

| Gap code | Meaning                                                    |
| -------- | ---------------------------------------------------------- |
| FIXTURE  | Needs detector fixture.                                    |
| CI       | Needs CI proof artifact.                                   |
| GOV      | Needs governance receipt.                                  |
| ORG      | Requires organization-level evidence beyond repo scanning. |

Then rows can say `FIXTURE + CI`, not a paragraph.

## 7. Fix rule-mapping anomalies

Several vibe-coverage rows appear misclassified. This matters because the appendix is supposed to prove coverage discipline.

Examples to review:

| Page / row                                                           | Current mapping              | Concern                                                      | Better likely mapping                                                                         |
| -------------------------------------------------------------------- | ---------------------------- | ------------------------------------------------------------ | --------------------------------------------------------------------------------------------- |
| Page 21, `tip2:12 Cross-site scripting / unsafe rendering`           | HLT-016 supply chain         | XSS is not primarily supply chain.                           | HLT-023 input boundary, HLT-014 a11y/UX if rendering-specific, or HLT-009 generated security. |
| Page 22, `tip2:28 No rollback / weak backup discipline`              | HLT-016 supply chain         | This is release/ops readiness, not dependency drift.         | HLT-025 release readiness.                                                                    |
| Page 25, `tip3:47 Prompt-Level Rules Treated as Security Controls`   | HLT-022 authz/data isolation | This is agent policy/security boundary.                      | HLT-011 prompt injection or HLT-024 agent-tool supply.                                        |
| Page 30, `tip5:43 Broken business logic / skipped state transitions` | HLT-016 supply chain         | This is business logic / false-green risk, not dependencies. | HLT-008 false-green risk or HLT-022 authz/data isolation depending on case.                   |

The appendix should be run through a “mapping audit” before publication.

## 8. Add “Related Work” as a real section

The references are rich, but the paper lacks a dedicated Related Work section. Tier-1 readers will expect it. It should not merely list sources; it should position Jankurai against them.

Suggested subsections:

* AI coding benchmarks: SWE-bench, SWE-agent, OpenHands, Aider, SetupBench, ContextBench.
* Secure software supply chain: SLSA, SBOM, SPDX, OSV, OpenSSF Scorecard.
* API/schema standards: OpenAPI, Protobuf/Buf, JSON Schema.
* Conformance programs: Kubernetes conformance, W3C maturity process.
* Agent instruction files: AGENTS.md, Claude memory, Cursor rules, Copilot instructions.
* UI/a11y testing: Playwright, Storybook, axe, WCAG, visual regression tools.

The key positioning sentence:

> Existing tools test code, scan dependencies, guide agents, or standardize APIs. Jankurai’s contribution is to bind these controls into a versioned repository-level merge standard with auditable proof receipts and conformance levels.

## 9. Replace some numerical precision with calibration language

The RPNs, 100-point stack scores, and exact rankings create authority, but they also invite criticism. Since the paper admits these are policy weights, avoid implying empirical precision.

For example, instead of:

> Rust + TS/Vite + Postgres = 94

Use:

> Rust + TS/Vite + Postgres: Tier A, 91-96 under the v0.6.1 policy rubric.

Or keep the number but add a “Rubric Calibration” subsection:

* Who assigned weights?
* Are weights derived from incidents, expert judgment, or policy?
* How should future versions recalibrate?
* What would change the ranking?
* What hard filters override score?

This makes the scoring feel honest rather than arbitrary.

## 10. Improve the title page’s seriousness

The mascot and branding are memorable, but the current first page looks more like a project brochure than a standards paper. The purple squiggle near the subtitle and floating icons should be removed. For a tier-1 PDF, use a cleaner title page and move the mascot/wordmark to a small footer, project page, or final “project identity” page.

The current title is energetic, but “Humans Were the Bug” may alienate the very people needed for adoption. Keep it as a marketing subtitle or epigraph, not the main academic subtitle.

## Abstract

**Current strength:**
The abstract does a good job naming the bottleneck shift: code generation is no longer scarce; trustworthy merge is scarce. It clearly lists three contributions: standard, conformance/audit mechanism, and reference profile.

**Main issue:**
It is slightly too dense and undersells the simplest breakthrough: a merge decision can become a reproducible artifact.

**Recommended restructuring:**

1. Sentence 1: AI coding changed the scarce resource.
2. Sentence 2: Jankurai’s core contribution.
3. Sentence 3: What artifacts make it auditable.
4. Sentence 4: What conformance means.
5. Sentence 5: What it does not claim.
6. Sentence 6: Why it matters.

**Possible revised abstract skeleton:**

> AI coding shifts the bottleneck in software engineering from producing plausible code to deciding whether a change can be trusted at merge time. Jankurai is a versioned, agent-native repository standard that makes that decision reproducible. It requires repositories to expose machine-readable ownership, proof routing, generated-zone policy, version manifests, audit reports, proof receipts, repair queues, and merge witnesses. These artifacts bind human- and agent-authored changes to the same rule set: no proof, no merge; no receipt, no trust. The standard defines conformance levels, stable rule identifiers, local/PR/CI proof lanes, and release gates, while separating the core standard from a reference profile based on Rust, TypeScript/React/Vite, PostgreSQL, generated contracts, and bounded Python. Jankurai does not prove all program semantics and does not claim institutional equivalence to established standards bodies. Its claim is narrower: a repository claiming conformance must provide current, machine-readable evidence sufficient to audit the merge decision.

**Layout feedback:**
The abstract is readable, but on page 1 it sits too low because of the large branding. Consider a standard two-column IEEE-style first page or a clean single-column standards-paper front matter.

* Define whether exceptions can lower conformance level.
* Add “exception debt” to score/rachet.
* Add example of an expired exception blocking merge.

**Suggested phrase:**

> An exception is not permission to ignore the standard; it is a versioned promise to return to it.

**Layout improvements:**

* Table XIX is concise and useful.
* Appendix C’s templates are good, but they should include machine-readable YAML front matter so agents can parse them reliably.

## XV. Migration, Versioning, and Governance

**Current strength:**
This section is very important. The “no-write adoption first” principle is excellent. The standard’s strongest adoption path is progressive installation: observe, score, CI, full, ratchet.

**Content improvements:**

* Add a governance model:

  * who can propose rules
  * how rules become candidate/stable
  * how breaking changes are announced
  * how deprecated rules remain interpretable
  * how alternate auditors can be certified
* Define version compatibility:

  * standard version
  * schema version
  * auditor version
  * profile version
  * paper edition
* Add “current and previous” policy, like real standards.
* Define how repositories upgrade from v0.6.1 to v0.7 or v1.0.

**Standards lesson:**
The paper correctly looks toward successful standards mechanics: normative keywords, version compatibility, visible spec history, machine-readable artifacts, maturity states, and conformance tests. The next step is to turn that inspiration into process documents: `PROCESS.md`, `CONFORMANCE.md`, `COMPATIBILITY.md`, and `GOVERNANCE.md`.

**Layout improvements:**

* Table XX is strong but basic. Add “who decides” and “required artifact.”
* The manifest in Appendix B is valuable. It should be introduced earlier as a core artifact.

## XVI. Limitations and Research Agenda

**Current strength:**
This section is appropriately humble. It admits scoring weights need empirical calibration, file-size limits need testing, tool-specific behavior changes, and audit cannot prove every semantic property.

**Content improvements:**

* Move some limitations earlier to build trust.
* Add “known non-goals.”
* Add “failure modes of Jankurai itself”:

  * ceremony without enforcement
  * stale maps
  * fake receipts
  * overfitting to one stack
  * false confidence from score
  * gaming the audit
* Add a direct plan for empirical calibration.

**Suggested additional limitation:**

> A repository can satisfy artifact presence while still having weak proofs. Jankurai must therefore distinguish evidence existence from evidence quality.

That distinction is essential.

**Layout improvements:**

* Table XXI on future work is strong, but page 16 is mostly a narrow table with lots of whitespace. Make it full-width or move it after the conclusion as a roadmap.

## XVII. Conclusion

**Current strength:**
The conclusion lands the core message: teams that optimize for human comfort alone will create ambiguity faster; teams that optimize for verification will reject bad code, repair failures, and change architecture without guessing. The final “Anything less is just faster vibe coding” is memorable.

**Content improvements:**

* End on the standard, not the stack.
* Mention the merge witness again.
* Reiterate that humans and agents are governed equally.
* Make the call to action concrete.

**Possible revised ending:**

> Jankurai’s claim is simple: a repository that asks agents or humans to change production software must expose enough structure for the merge decision to be reproduced. Changed paths must route to owners. Owners must route to proof lanes. Proof lanes must leave receipts. Receipts must feed an audit report and merge witness. Findings must become repair queues or expiring exceptions. That loop is the agent-native repository standard. Without it, AI does not give software engineering a new discipline; it gives old ambiguity a faster keyboard.

# Content architecture to maximize wide acceptance

## Make Jankurai a family of artifacts, not only a paper

For adoption, the paper should point to a clean public structure:

```text
jankurai.org/
  standard/
    core/
      v0.6.1/
      current/
      previous/
    profiles/
      rust-ts-postgres/
      go-ts-postgres/
      dotnet-ts-postgres/
  schemas/
    repo-score.schema.json
    merge-witness.schema.json
    proof-receipt.schema.json
    repair-queue.schema.json
  conformance/
    fixtures/
    expected/
    runner/
  examples/
    minimal-hl3-repo/
    generated-zone-failure/
    secret-sprawl-failure/
    rendered-ux-failure/
  governance/
    process.md
    compatibility.md
    maturity.md
```

The paper should become the flagship explanation, while the standard lives as testable documents and schemas.

## Define adoption personas

The paper will gain attention faster if it speaks to distinct audiences:

| Audience               | What they care about          | Message                                                            |
| ---------------------- | ----------------------------- | ------------------------------------------------------------------ |
| CTO / VP Engineering   | AI velocity without chaos     | Jankurai turns AI coding into auditable merge governance.          |
| Security leader        | Secrets, supply chain, agency | Jankurai makes risky agent actions visible and gated.              |
| Staff engineer         | Review burden                 | Jankurai routes proof and repair instead of relying on memory.     |
| Open-source maintainer | Contributor quality           | Jankurai gives contributors and agents deterministic expectations. |
| Tool vendor            | Integration surface           | Jankurai provides schemas, receipts, and conformance fixtures.     |
| Researcher             | Evaluation                    | Jankurai creates measurable artifacts for studying agent repair.   |

## Create a “minimal HL3 demo”

A public demo will do more than another paragraph. Show a tiny repo where:

1. Agent edits generated file directly.
2. Jankurai detects HLT-002.
3. Merge witness blocks.
4. Agent changes source OpenAPI.
5. Regeneration receipt appears.
6. Merge witness passes.

That single demo would communicate the standard instantly.

## Add a badge, but make it serious

Example:

```text
Jankurai HL3
Standard: 0.6.1
Schema: 1.4.1
Auditor: 0.6.1
Profile: rust-ts-postgres-0.6
Witness: passing
Last verified: 2026-05-04
```

Make the badge link to immutable evidence, not just a logo.
