I studied the source package, the Rust auditor, the UX QA package, the docs, the checked-in score reports, and the paper source closely. My strongest read: **the thesis is excellent, but the project needs to become an installer + verifier + evidence router, not just a paper and audit CLI.** The standard should feel like `rustup` or `cargo nextest`: one command, obvious files, fast feedback, deterministic reports, and agent-readable repair instructions.

The best direction is: **Jankurai should become the control plane for agent-native repos.** It should install the standard into every agent’s expected instruction location, detect repo shape, propose structure fixes, run fast hard/soft checks, generate proof receipts, and export issues that humans and agents can both act on.

## Critical findings before the 3 upgrades

These are important because they show where adoption will fail today:

1. **The current checked-in score appears stale or false-green.** `repo-score.json` / `repo-score.md` say there are no findings and no caps, but the archive is missing `Cargo.lock`, missing `package-lock.json`, includes 100+ AppleDouble `._*` files, references missing `assets/`, `tips/`, and `reference/` paths, and has a CI security lane that mostly echoes scanner names instead of running them.

2. **The install path is not yet easy enough.** README currently asks users to `cargo install` and manually copy/append agent files. That is too much friction for a global standard. The standard needs `jankurai init`, `jankurai doctor`, `jankurai ci`, and ideally prebuilt binaries.

3. **The audit has several false-green risks.** It often checks for marker text rather than executed proof. Examples: “generated contracts” can pass because the word “generated” appears; UX QA can pass because docs mention Storybook or Playwright; security can pass because workflows mention scanner names; owner mapping is hardcoded instead of loaded from `agent/owner-map.json`; policy thresholds in `agent/audit-policy.toml` are not enforced by the CLI.

4. **The project does not yet install files into the real agent-helping locations.** Codex uses `AGENTS.md` and can layer instructions by directory; its own docs recommend repo layout, build/test commands, conventions, constraints, and done/verify guidance in that file. ([OpenAI Developers][1]) Claude Code reads `CLAUDE.md`, supports hooks, and supports skills under `.claude/skills/*/SKILL.md`. ([Claude][2]) GitHub Copilot uses `.github/copilot-instructions.md` and path-specific `.github/instructions/*.instructions.md`. ([GitHub Docs][3]) Jankurai should generate all of these automatically.

5. **The project is very close to something powerful.** The paper’s core idea—stop trusting vibes, require ownership, proof lanes, generated-zone discipline, stack boundaries, and rendered UX evidence—is the right one. The implementation needs to become stricter, faster, and much easier to install.

Here are the **best 3 upgrades** I would make.

# Upgrade 1 — **HL-Init: One-Command Adoption + Agent Rulepack Compiler**

## Goal

Make Jankurai installable in any repo with one command:

```bash
jankurai init
```

or, for CI/agent-safe automation:

```bash
jankurai init \
  --profile rust-ts-vite-react-postgres \
  --ides codex,claude,cursor,copilot,antigravity,aider \
  --ci github \
  --issue beads \
  --dry-run
```

Then:

```bash
jankurai init --apply
jankurai doctor
jankurai audit --changed
```

This upgrade turns Jankurai from “a standard someone reads” into **a standard the repo installs**.

## What it should do

`jankurai init` should inspect the repo and produce an **Init Plan**:

```text
Jankurai Init Plan

Detected:
  Rust workspace: yes
  TypeScript package: yes
  Vite/React: yes
  PostgreSQL markers: partial
  Python product code: no
  Existing agent instructions: AGENTS.md
  CI provider: GitHub Actions

Will add:
  agent/JANKURAI_STANDARD.md
  agent/audit-policy.toml
  agent/boundaries.toml
  agent/owner-map.json
  agent/test-map.json
  agent/generated-zones.toml
  agent/proof-lanes.toml
  agent/ux-qa.toml
  .github/workflows/jankurai.yml
  .github/copilot-instructions.md
  .github/instructions/jankurai-boundaries.instructions.md
  CLAUDE.md
  .claude/skills/jankurai/SKILL.md
  .cursor/rules/jankurai.mdc
  AGENTS.md patch

Will warn:
  No Cargo.lock
  No package-lock.json / pnpm-lock.yaml / yarn.lock
  Existing AGENTS.md contains local absolute path
  Existing repo-score.json appears stale
```

The installer should not destructively reorganize a repo by default. It should produce a plan, then apply only when the user passes `--apply`.

# Upgrade 2 — **Proof Ledger: CI Gate, Ratchet, SARIF, and Issue Memory**

## Goal

Make every audit produce **machine-readable proof**, not just a score.

The current audit report is useful, but it needs to become a CI contract that agents can consume:

```bash
jankurai ci \
  --policy agent/audit-policy.toml \
  --json repo-score.json \
  --md repo-score.md \
  --sarif jankurai.sarif \
  --github-step-summary \
  --annotations github \
  --issues beads \
  --fail
```

This is the upgrade that makes Jankurai usable as the repo’s automatic scorekeeper.

## The new report contract

`repo-score.json` should include:

## Hard checks vs soft checks

### Hard checks

These should block CI unless explicitly excepted:

```text
HLT-ROOT-001   Missing root AGENTS.md / agent standard
HLT-LOCK-001   Missing package lockfile
HLT-CI-001     CI security lane is missing or no-op
HLT-CI-002     CI proof lane only echoes commands
HLT-SEC-001    Secret-like token in tracked source
HLT-GEN-001    Generated-zone manual edit
HLT-DB-001     Direct DB access from frontend/domain/wrong layer
HLT-DTO-001    Handwritten DTO where generated contract is expected
HLT-PY-001     Python owns product truth outside allowed roots
HLT-TEST-001   False-green tests: skip/only/no asserts/tautologies
HLT-MIG-001    Destructive migration without safety metadata
HLT-PROMPT-001 Prompt-injection or excessive agency in trusted agent files
HLT-UX-001     Critical rendered UX blocker on required route/story
HLT-POLICY-001 audit-policy.toml exists but is not enforced
```

### Soft checks

These should produce warnings and repair guidance:

```text
HLT-NAME-001   Weak names: manager, helper, util, misc, common
HLT-SIZE-001   Oversized file/function
HLT-FB-001     Broad fallback / best-effort behavior
HLT-DOC-001    Agent instructions too long or vague
HLT-OWNER-001  Owner map incomplete for non-critical path
HLT-PERF-001   No performance budget for hot path
HLT-UX-LOW     Non-critical UI polish issue
HLT-PY-LOW     Python present but bounded and justified
```

## Fix the false-green audit problem

The Rust audit should stop accepting “mentions” as proof.

Examples of changes:

### Current weak behavior

```text
Workflow contains "gitleaks" in an echo line
=> audit thinks secret scanning exists
```

### Required stronger behavior

Jankurai should parse CI workflows and require one of:

```text
uses: gitleaks/gitleaks-action@...
run: gitleaks detect ...
run: cargo deny check ...
run: cargo audit ...
run: zizmor ...
run: syft ...
run: grype ...
run: osv-scanner ...
```

But this should not pass:

```yaml
run: echo "Run gitleaks later"
```

Same for UX QA:

```text
Docs mention Playwright
=> capability documented, not proof lane passed
```

Required proof:

```text
package.json script exists
CI runs script
or jankurai UX receipt exists
or repo has explicit exception
```

Same for generated contracts:

```text
Word "generated" appears
=> not enough
```

Required proof:

```text
agent/generated-zones.toml declares zones
declared generation command exists
generated artifact exists
source contract exists
checksum or timestamp receipt exists
```

## CI and release packaging

The current README install path requires compiling from source. That is fine for Rust users, but a global standard needs prebuilt binary installs too. `cargo-binstall` is specifically intended to install Rust binaries without building from source, while `cargo-nextest` can speed Rust test execution, and `cargo-deny` / `cargo-vet` cover supply-chain policy and dependency audit workflows. ([GitHub][6])

Recommended install paths:

```bash
cargo binstall jankurai
```

Fallback:

```bash
cargo install jankurai --locked
```

Node-friendly bootstrap:

```bash
npx @jankurai/init
```

That wrapper should only download/execute the official binary after showing the version and checksum.

# Upgrade 3 — **Boundary Oracle: Stack Enforcement + Rendered UX Proof Runtime**

## Goal

Pick the blessed stack and make its boundaries machine-checkable.

The paper already says the ideal standard is:

```text
Rust core
TypeScript / React / Vite product surface
PostgreSQL durable truth
Generated contracts
Bounded Python for AI/data/eval only
```

That should become a manifest-driven “Boundary Oracle” that agents can read and the Rust audit can enforce.

## Make the boundaries painfully clear

The paper and installed standard should say this directly:

```text
React/Vite/TypeScript:
  Owns what the user sees and touches.
  Does not own durable truth, authorization, secrets, or server DTO truth.
  Talks through generated clients.

Rust domain:
  Owns pure product rules.
  No IO. No DB. No HTTP. No env. No time. No random.
  Easy to test, easy to reason about.

Rust application:
  Owns use cases, authz, idempotency, transactions, and orchestration.
  Calls ports/traits.
  Does not render UI or hand-roll transport details.

Rust adapters:
  Own DB, queues, provider clients, external APIs, and persistence mapping.
  May use SQL and provider SDKs.
  Must not fork domain policy.

API transport:
  Owns HTTP/gRPC request/response mapping.
  Thin layer.
  No business policy except calling application services.

PostgreSQL:
  Owns durable constraints, migrations, indexes, RLS, and data integrity.
  Destructive changes require safety metadata.

Python:
  Allowed for AI, ML, data, experiments, evals, and scripts.
  Not allowed to become hidden product truth.
```

This is the heart of the standard. Agents need this more than prose about “clean architecture.”

## Boundary checks to implement

### TypeScript / React / Vite

Hard findings:

```text
Frontend imports direct DB package
Frontend imports server-only package
Frontend defines handwritten DTO matching known generated contract
Frontend stores durable product truth in local storage without explicit exception
Frontend has raw fetch wrappers outside generated client boundary
Frontend contains secrets or server env access
```

Soft findings:

```text
Component over size budget
Route has no rendered UX coverage
Form has local validation not mirrored by contract/server
```

### Rust domain

Hard findings:

```text
crates/domain imports sqlx/diesel/reqwest
crates/domain reads env
crates/domain performs filesystem IO
crates/domain depends on web framework
crates/domain uses wall-clock time without injected clock
```

Soft findings:

```text
Domain type named Manager/Helper/Util
Domain function has too many branches
Domain test coverage weak for invariant-heavy module
```

### Rust application

Hard findings:

```text
Use case lacks authz check where owner map requires it
Idempotent command has no idempotency key path
Application layer contains raw SQL
```

### Rust adapters

Hard findings:

```text
Adapter duplicates domain policy
Adapter maps provider errors to strings instead of typed errors
Adapter has unbounded retry/backoff
```

### Database

Hard findings:

```text
DROP TABLE / DROP COLUMN without migration safety metadata
ALTER TYPE destructive operation without plan
No down/safety note where policy requires one
Missing index for declared hot query
```

### Python

Hard findings:

```text
Python outside allowed roots owns product model/entity
Python writes product DB without explicit exception
Python bypasses Rust application/API layer
Python service lacks typed contract with Rust boundary
```

Soft findings:

```text
Notebook-like code in product path
Unpinned Python dependencies
No eval receipt for model behavior
```

## Turn UX QA into a first-class proof lane

The `packages/ux-qa` seed is promising. It already collects DOM geometry and checks target size, overlap, clipping, horizontal overflow, focus, labels, sticky obstruction, and nested scrollbars. The gap is that it is not yet integrated as a normal Jankurai proof lane, and some rules need false-positive protection.

Add:

```bash
jankurai ux audit --config agent/ux-qa.toml
jankurai audit --include-ux .jankurai/ux-qa.json
```

Example `agent/ux-qa.toml`:

[runner]

## UX QA rule improvements

The current `edge-clearance` rule is likely too aggressive as a hard rule. Buttons near a viewport edge can be intentional if target size, safe area, and spacing are okay. Make it:

```text
hard: target is clipped, unreachable, or collides with safe area
soft: target is visually cramped near edge
```

Improve these rules:

```text
target-size:
  Hard fail below 24px unless exception applies.
  Soft warn below 44px for touch-heavy surfaces.

interactive-overlap:
  Use hit testing with document.elementFromPoint.
  Ignore pointer-events:none, inert, disabled, aria-hidden.

sticky-obstruction:
  Check whether the interactive center/critical points are actually blocked.
  Do not fail merely because rectangles overlap.

focus-visible:
  Compare visual delta before/after focus.
  Accept outline, box-shadow, border, background, transform, or tokenized focus ring.

selectors:
  Prefer data-testid, role/name, stable id.
  Avoid global nth-of-type where possible.
  Include parent-scoped selector fallback.

storybook:
  Parse Storybook index.
  Visit each required story.
  Produce route/story identity in receipt.

artifacts:
  Always include screenshot path, crop path, ARIA snapshot, DOM snapshot hash, viewport, browser, and rule ids.
```

# The deepest strategic recommendation

Do **not** frame Jankurai as another linter. Frame it as:

```text
The installable standard for agent-native software engineering.
```

The paper should argue:

```text
Jankurai removes human scatter by converting repo expectations into files agents already read, checks CI can enforce, and proof receipts humans can trust.
```

The project should implement that by making this true:

```bash
jankurai init
jankurai ci
```

After that, every agent knows:

```text
Where instructions live.
What stack is allowed.
Which layer owns what.
Which files are generated.
Which command proves the change.
Which findings block merge.
Which issue to fix next.
```

That is the path from “vibe coding critique” to **global standard**.
