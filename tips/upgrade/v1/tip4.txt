## Verdict

The project already has the right nucleus: **a standard + paper + Rust audit + UX QA seed + machine-readable agent files**. The big opportunity is not more theory. It is turning this into a **one-command adoption product** with hard evidence, real CI gates, and generated agent-IDE adapters.

The main risk I found is that the repo currently **looks more conformant than it actually is**. The current `repo-score.md` reports score `72`, no caps, no findings, and no queued fixes, even though the repo still has low-scoring areas like data truth, observability, build speed, and contract maturity. That “green-ish but unactionable” state is the exact kind of false confidence jankurai is supposed to eliminate.

I could not run the Rust test suite in this container because `cargo` is not installed here, so this review is based on a close static/source read plus current ecosystem docs. I did inspect the archive structure, README, paper sections, audit code, agent manifests, CI workflow, Justfile, and UX QA package.

# The best 3 upgrades

## 1. **Jankurai Bootstrap Doctor**

**Name:** `jankurai init` / `jankurai doctor`
**Goal:** Make adoption one command, not a README ritual.

Right now the README asks users to install the CLI, manually `curl` `agent/JANKURAI_STANDARD.md`, manually create or edit `AGENTS.md`, then manually add maps. That is too much friction for something you want to become a global standard. The project should ship a **repo standardizer** that detects the existing project, proposes the optimal layout, writes all agent helper files, installs UX QA where applicable, and creates CI.

This aligns with the paper’s own claim that tool-specific instructions should be generated or mirrored from one canonical standard, not hand-maintained separately; see `paper/tex/sections/10_agent_controls.tex:3-19`. It also matches the current reality of agent IDEs: Codex reads `AGENTS.md`, Claude Code uses persistent `CLAUDE.md`, GitHub Copilot supports repo and path-specific instruction files, Cursor supports project rules, and Antigravity-style environments use agent/workflow customization surfaces. ([OpenAI Developers][1])

### What it should do

Add:

```bash
jankurai init
jankurai init --yes
jankurai init --dry-run
jankurai init --ide all --ci github --ux-qa
jankurai doctor .
jankurai adapters --verify
```

The install flow should detect the repo and produce an **adoption plan**:

```text
Detected:
- Rust workspace: yes
- Vite/React app: yes/no
- TypeScript app: yes/no
- PostgreSQL migrations: yes/no
- Python service: yes/no
- Existing AGENTS.md: yes/no
- Existing CLAUDE.md: yes/no
- Existing .cursor/rules: yes/no
- Existing GitHub Actions: yes/no

Recommended:
- Add canonical agent standard files
- Generate IDE adapters from canonical standard
- Add owner/test/generated-zone maps
- Add jankurai CI
- Add UX QA lane for web surface
- Add real security lane
```

For an unorganized project, the doctor should not immediately rewrite everything. It should produce a clear plan:

```text
Your repo has web UI files at ./frontend, but the standard expects apps/web.
Options:
1. Keep ./frontend and map it as owner=web in agent/owner-map.json.
2. Move ./frontend -> apps/web and update imports/package scripts.
3. Defer with docs/exceptions/2026-05-02-nonstandard-web-layout.md.
```

Default behavior: **non-destructive**.
`--yes` applies safe file additions.
`--force` overwrites generated adapter files only.
`--dry-run` prints patch-style diffs.

### The canonical stack boundary should be explicit

The standard should pick one best stack and make it boring:


## 2. **Evidence-First Audit Engine**

**Name:** `jankurai audit --evidence`
**Goal:** Stop false-green scoring and make every score actionable.

The audit should become stricter, more precise, and more useful to agents. The current audit has several important breakpoints:

| Current issue                                                                                                                | Why it matters                                               |
| ---------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------ |
| `repo-score.md` says score `72`, but “No findings” and “No queued fixes”                                                     | Agents cannot repair a low score without a queue.            |
| `.github/workflows/jankurai-audit.yml:18-23` only echoes security commands                                                  | The audit currently rewards marker strings instead of proof. |
| `Justfile:31-32` security lane only echoes tool names                                                                        | Same false-green problem.                                    |
| `crates/jankurai/src/audit/mod.rs:1467-1469` treats any file containing `"generated"` as generated-contract evidence        | Very high false-positive risk.                               |
| `crates/jankurai/src/audit/mod.rs:1518-1528` detects web surface from any text containing `react`, `vite`, or `storybook`   | Docs/package text can accidentally trigger this.             |
| `crates/jankurai/src/audit/mod.rs:1536-1545` excludes `crates/jankurai/` and `packages/ux-qa/` from runtime product checks | The tool does not fully audit itself.                        |
| `crates/jankurai/src/audit/mod.rs:1904-1906` has `weak_name_hits` as a stub                                                 | A declared vibe-risk category is not implemented.            |
| `agent/owner-map.json` and `agent/test-map.json` exist, but the audit logic is mostly hardcoded                              | The maps are not yet authoritative.                          |
| `Justfile:5-6` emits JSON and Markdown both to stdout for `fast`                                                             | Bad for agents and CI parsers.                               |

The standard says a green score without receipts is not conformance; see `paper/tex/sections/06_jankurai_standard.tex:24-28`. The implementation should enforce that principle.

### Add audit modes

```bash
jankurai audit .
jankurai audit . --mode advisory
jankurai audit . --mode standard
jankurai audit . --mode ratchet --baseline agent/repo-score.json
jankurai audit . --changed-from origin/main
jankurai audit . --format json,md,sarif,junit,github
jankurai audit . --fail-under 85 --fail-on high,critical
jankurai audit . --explain HLT-010-SECRET-SPRAWL
```

Modes:

```text
observe   = never fails, just inventories
advisory  = emits findings and queue, no CI fail
guarded   = fails critical caps
standard  = fails high/critical and score floor
ratchet   = fails regression from baseline
release   = requires receipts across audit/security/contracts/db/e2e/versions
```

### Make maps authoritative

The audit should parse and enforce:

```text
agent/owner-map.json
agent/test-map.json
agent/generated-zones.toml
agent/proof-lanes.toml
agent/audit-policy.toml
agent/standard-version.toml
```

Then every changed file should answer:

```text
Who owns this?
What layer is allowed to import/use it?
Which proof lane proves it?
Is it generated?
If generated, what command regenerates it?
Is there a receipt?
```

For example, a change to `apps/web/src/Checkout.tsx` should route to:

```json
{
  "owner": "web",
  "forbidden": ["direct-db", "durable-truth", "secrets"],
  "required_lanes": ["typecheck", "web-unit", "ux-qa"],
  "required_artifacts": ["screenshot", "aria-snapshot", "ux-qa-report"]
}
```

### Replace string scoring with proof scoring

Some string checks are fine for fast triage, but hard gates need evidence. For example:

Bad:

```text
command_surface_text contains "gitleaks"
```

Better:

```text
proof-lanes.toml has lane security
security lane command is not echo-only
CI runs security lane
receipt contains tool result
tool result is current for this commit or PR
```

For supply chain, do not reinvent mature Rust tools. Integrate and verify outputs from `cargo-audit`, `cargo-deny`, `cargo-vet`, and `cargo-semver-checks`. RustSec’s own docs describe `cargo-audit` for vulnerable Rust crates and `cargo-deny` for vulnerabilities, license/source controls, and duplicate-version policy; `cargo-vet` addresses third-party dependency auditing; `cargo-semver-checks` is useful for public API break detection with very low false-positive goals. ([RustSec][3])

Recommended real security lane:

```bash

cargo deny check advisories bans licenses sources
cargo vet
gitleaks detect --source .
zizmor .github/workflows
actionlint
```

For npm/web repos, add:

```bash
npm audit --audit-level=high
npx playwright install --with-deps chromium
npm run typecheck
npm run test
```

## 3. **Proof Receipts + UX Oracle + Issue Loop**

**Name:** `jankurai prove`
**Goal:** Make agents periodically audit, run targeted proof lanes, leave evidence, and create/close work items.

This is the upgrade that turns jankurai from an audit report into an agent operating system.

The paper already points here: HL3+ requires an ordered `agent_fix_queue`, and rendered UX proof needs route/story identity, browser, viewport, screenshot/crop paths, ARIA snapshot path, rule IDs, selectors, severity, reproduction command, and decision. See `paper/tex/sections/06_jankurai_standard.tex:24-28` and `paper/tex/sections/12_pixel_qa.tex:9-11`.

### New command

```bash
jankurai prove .
jankurai prove --changed
jankurai prove --changed-from origin/main
jankurai prove apps/web/src/Checkout.tsx
jankurai prove --next-fix
jankurai prove --receipt
```

What it does:

```text
1. Reads changed paths.
2. Resolves owner-map/test-map/generated-zones.
3. Runs only mapped proof lanes.
4. Runs jankurai audit on that scope.
5. Writes receipt JSON + Markdown.
6. Updates repo-score.
7. Emits next actionable fix.
8. Optionally exports issues to GitHub or Beads.
```

Receipt location:

```text
agent/receipts/
  2026-05-02T18-31-22Z-abc123.audit.json
  2026-05-02T18-31-22Z-abc123.audit.md
  2026-05-02T18-31-22Z-abc123.ux.json
  2026-05-02T18-31-22Z-abc123.security.json
```

Receipt schema:

```json
{
  "schema": "jankurai.receipt.v1",
  "commit": "abc123",
  "scope": ["apps/web/src/Checkout.tsx"],
  "owner": "web",
  "lanes": [
    {
      "name": "ux-qa",
      "command": "npm run ux:qa -- --route checkout",
      "status": "pass",
      "duration_ms": 1840,
      "artifacts": [
        "artifacts/checkout.390x844.png",
        "artifacts/checkout.390x844.aria.yml"

    }
  ],
  "audit": {
    "score": 88,
    "findings": []

}
```

### Upgrade UX QA into a real “oracle”

The existing `@jankurai/ux-qa` is a good seed. It checks geometry, clipping, target size, overlap, focus, labels, nested scrollbars, z-index ceiling, and artifacts. But it needs stronger repo integration and fewer false positives.

Current code issues:

| File                                      | Issue                                                                                                                                    |
| ----------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `packages/ux-qa/src/cli.ts:26-47`         | Requires a live `--url`; no server-start helper, route config, or Storybook discovery.                                                   |
| `packages/ux-qa/src/collector.ts:156-160` | Collects boxes and states in separate browser evaluations, then merges by index. Focus/visibility changes can misalign state to element. |
| `packages/ux-qa/src/collector.ts:125-138` | Focus probing mutates the page. Good idea, but should be isolated and stable.                                                            |
| `packages/ux-qa/src/rules.ts:44-55`       | Overlap check is O(n²); this can get slow on dense pages.                                                                                |
| `packages/ux-qa/src/cli.ts:119-123`       | ARIA snapshot support exists; make it part of standard receipts.                                                                         |

# Immediate “before public launch” fixes

I would fix these before positioning jankurai as the new standard:

1. **Remove false-green placeholders.**
   Replace `Justfile:31-32` and `.github/workflows/jankurai-audit.yml:18-23` with real security commands or mark the lane explicitly “not implemented” so it cannot earn score.

2. **Make low scores produce queue items.**
   A report with score `72` and no findings is not useful. Any dimension below the configured target should create at least one finding.

3. **Do not mix JSON and Markdown on stdout.**
   Change `just fast` from:

   ```bash
   cargo run -p jankurai -- . --json - --md -
   ```

   to something parseable:

   ```bash
   cargo run -p jankurai -- audit . --json repo-score.json --md repo-score.md
   ```

   Or:

   ```bash
   cargo run -p jankurai -- audit . --format json
   ```

4. **Parse the actual maps.**
   `agent/owner-map.json`, `agent/test-map.json`, and `agent/proof-lanes.toml` should control routing. Hardcoded prefix logic should only be fallback.

5. **Fix broad detectors.**
   `has_generated_contracts()` should not pass because any file contains the word `generated`. `has_web_surface()` should not trigger on prose alone.

6. **Make CI actually runnable.**
   Add Playwright browser install and TeX setup, or remove those jobs until setup is complete.

7. **Clean release archive.**
   The uploaded tar includes many `._*` AppleDouble metadata files and test-result artifacts. A standard-setting project should ship clean source archives.

# The sharpened thesis

The strongest version of this project is:

> **jankurai is the installable control plane for agent-native repositories. It standardizes layout, agent instructions, proof lanes, generated zones, stack boundaries, rendered UX evidence, and repair queues so coding agents can work faster with fewer human-shaped mistakes.**

Keep the stack opinionated:

```text
Rust + TypeScript/React/Vite + PostgreSQL + generated contracts + bounded Python.
```

Do not expand beyond that as the default. Support other repos in advisory mode, but make the “gold standard” one clear target. The standard wins by being **boring, installable, fast, evidence-based, and agent-readable**.
