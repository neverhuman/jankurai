I read the archive as a **standard + installer + auditor + proof-pack** project, not just a paper. The core thesis is strong: reduce “jankurai” by forcing repos into a structure agents can understand, verify, repair, and ratchet. The best next move is to make `jankurai` feel like `cargo init`, `npx create`, and `cargo audit` combined: one command installs the standard, agent IDEs immediately understand the repo, CI enforces it, and reports produce a repair queue an agent can act on.

Validation note: I could not run the Rust tests in this container because `cargo`/`rustc` are unavailable here. I inspected the source, configs, current reports, CI, docs, and package layout. I also did not run `npm ci`; the archive does not include a root `package-lock.json`, while the Justfile uses `npm ci`, so a clean install is not currently self-contained.

## The most important gaps I found

The project is already pointed in the right direction, but a few things will block adoption or create false confidence:

1. **Install is still too manual.** The README tells users to manually create `agent/`, `curl` a standard file, edit `AGENTS.md`, and wire CI themselves. That is too much for the standard you want. It needs `jankurai init`.

2. **The current score can be non-actionable.** The included `repo-score.json` shows score `72`, `caps_applied: []`, and `findings: []`. That is a major reporting bug. A repo below the target should always produce a repair queue.

3. **The security lane has false-green risk.** `.github/workflows/jankurai-audit.yml` and `Justfile security` currently echo security tool names such as `gitleaks`, `syft`, `grype`, `zizmor`, `cargo-audit`, and `dependency-review`; the audit appears to treat these markers as evidence. A standard meant to fight vibe-coding mess cannot give credit for echoed placeholders.

4. **The auditor does not appear to truly enforce repo-owned maps yet.** `agent/owner-map.json`, `agent/test-map.json`, and `agent/audit-policy.toml` exist, but the Rust audit code mostly uses compiled defaults/static prefixes rather than parsing and enforcing those files as the repo contract.

5. **Web/UX detection is too broad.** The audit can classify the repo as having a web surface because docs or audit code mention React/Vite/Storybook. UX evidence should come from actual app/package manifests, tests, routes, or UX QA reports, not from documentation text.

6. **Agent setup files need adapter shims.** `AGENTS.md` is the right universal root, but Claude, Cursor, Codex, and Antigravity each have their own conventions. Codex explicitly reads `AGENTS.md` through global/project/nested scopes; Claude Code uses `CLAUDE.md` memory files and recommends keeping them concise; Cursor supports persistent project/team/user rules plus `AGENTS.md`; Antigravity exposes Skills via `SKILL.md` and Workflows. The installer should generate thin shims for each, all pointing back to the same canonical `agent/JANKURAI_STANDARD.md`. ([OpenAI Developers][1])

7. **Lockfiles are missing.** The README uses `cargo install --locked`, and the Justfile uses `npm ci`, but the archive does not include `Cargo.lock` or root `package-lock.json`. That undermines “easy install” and deterministic CI.

8. **The root `AGENTS.md` contains a local absolute path.** `Keep all new files under /Users/bentaylor/Code/jankurai` should never appear in a reusable standard. This is exactly the kind of local human scatter the standard should prevent.

9. **The paper has two source paths.** `paper/sections/*.md` and `paper/tex/sections/*.tex` both exist. The README says TeX is canonical, but agents may patch the wrong source. Mark old sections archived, remove them, or add a hard generated/legacy warning.

10. **The project’s self-audit excludes much of its own authored code.** Excluding `crates/jankurai` and `packages/ux-qa` from product-shape checks makes sense for scoring adopters, but for this repo it hides the most important code from structural pressure.

The three upgrades below are the highest-leverage path to making this a standard people actually adopt.

# Upgrade 1 — **`jankurai init`: the Agent-Native Installer**

## Goal

Make adoption one command:

```bash
cargo install jankurai --locked
jankurai init . --profile rust-ts-vite-react-postgres --agent-ides codex,claude,cursor,antigravity --with-ux-qa --with-ci
jankurai audit . --json repo-score.json --md repo-score.md
jankurai gate --report repo-score.json
```

The installer should not merely copy a Markdown file. It should inspect the repo, detect its shape, write the canonical agent files, create adapter shims for major coding agents, add CI, configure proof lanes, and offer safe repairs when the repo is organized poorly.

## Why this is the top upgrade

The standard only becomes global if adoption is nearly frictionless. `AGENTS.md` is the right cross-agent anchor because it is already positioned as a simple open format for agent instructions and supports nested files for monorepos; Codex also has explicit `AGENTS.md` discovery behavior. Claude and Cursor need thin adapter files, not competing standards. ([Agents][2])

## Commands to add

Add these CLI surfaces:

```bash
jankurai init [PATH]
  --profile rust-ts-vite-react-postgres-bounded-python
  --level HL1|HL2|HL3|HL4|HL5
  --agent-ides all|codex,claude,cursor,antigravity
  --ci github|none
  --with-ux-qa
  --with-security
  --yes
  --dry-run
  --force
  --plan jankurai-init-plan.md

jankurai doctor [PATH]
  --json doctor.json
  --md doctor.md

jankurai emit ci --github
jankurai emit agents --ides codex,claude,cursor,antigravity
jankurai explain HLT-007
```

`init` should be interactive by default, but deterministic under `--yes`. It should never move or delete code without explicit confirmation. The safe default is to **write maps, shims, CI, and warnings**, not restructure the repo destructively.

`jankurai init` should classify the repo before writing files.

Detect:


# Upgrade 2 — **Evidence-Gated Auditor & Ratchet Report**

## Goal

Make the audit impossible to game with placeholders and easy for agents to repair.

The audit should move from “scan and score” to:

```text
detect repo shape
load policy
load owner/test/generated maps
collect evidence
classify hard/soft findings
redact sensitive data
produce repair queue
gate CI
ratchet against baseline
emit SARIF/JUnit/Markdown/JSON
```

The auditor should be especially strong against the known failure mode of agent-generated projects: they often pass shallow functional checks while accumulating duplication, large methods, framework mistakes, exception-handling problems, and accessibility issues. A 2026 study of Cursor-generated projects found high average functional correctness but many CodeScene/SonarQube issues, including duplication, complexity, large methods, exception handling, framework best-practice violations, and accessibility problems. ([arXiv][4])

## Why this is the second upgrade

Right now the audit has the right instincts but too many checks are marker-based. The standard will lose trust if a repo can pass because it mentions `gitleaks` in an echo command or says “React” in a doc. A global agent standard needs evidence-quality checks.

## Eliminate false-green security evidence

The audit must reject this:

```yaml
- name: Security lane placeholders
  run: echo "gitleaks syft grype zizmor cargo-audit"
```

It should only accept real commands/actions:

```yaml
- name: Rust advisories
  run: cargo audit --deny warnings

- name: Rust supply-chain policy
  run: cargo deny check advisories bans licenses sources

- name: GitHub Actions security
  uses: zizmorcore/zizmor-action@<pinned-sha>
```

`cargo-audit` checks `Cargo.lock` against RustSec advisories, and `cargo-deny` can check advisories, licenses, sources, and duplicate versions; `zizmor` is a static analyzer for GitHub Actions security issues. These should be real proof lanes, not strings in comments or echoes. ([RustSec][5])

Add rule:

```text
HLT-SEC-PLACEHOLDER
Critical hard check.
Security tool names inside echo/comment/doc text do not count as security evidence.
```

# Upgrade 3 — **Proof Pack: Rust Security + UX QA + Stack Boundary Contract**

## Goal

Make Jankurai not just a standard, but the default proof system for one best stack:

```text
Rust durable core
TypeScript/React/Vite product surface
PostgreSQL truth
Generated contracts
Bounded Python for AI/data/tooling only
```

The proof pack should install the commands, configs, CI jobs, and reports needed to verify that each layer stays inside its boundary.

## Why this is the third upgrade

The paper says “pick one best-known stack and align behind it.” That is right. But the repo needs a concrete proof pack that makes the boundary enforceable. Otherwise people will read the standard, agree with it, and then drift back into scatter.

## Boundary contract to make explicit

Add this to `docs/boundaries.md` and `agent/JANKURAI_STANDARD.md`.

### Rust owns

```text
- Durable domain logic
- Authorization decisions
- Invariants
- API edge
- Application commands
- Idempotency
- Transactions
- Database adapters
- Contract generation
- Security-sensitive parsing
```

### TypeScript/React/Vite owns

```text
- UI rendering
- Routes
- Forms
- Client-side state
- Browser-only interaction
- Generated API clients
- Storybook/component previews
- Playwright tests
```

It must not own:

```text
- Durable truth
- Authorization source of truth
- Direct database access
- Duplicated backend business rules
- Hidden production side effects
```

### PostgreSQL owns

```text
- Durable data
- Constraints
- Indexes
- Migrations
- Optional RLS where appropriate
```

### Python owns only

```text
- AI/ML/data experiments
- Offline analytics
- Bounded AI service surfaces
- Tooling where Rust replacement is not practical yet
```

Python must not own product truth, authorization, or direct production DB writes without an exception.

```yaml
      - name: Rust proof
        run: |
          cargo check --workspace --all-targets --locked
          cargo test --workspace --locked

      - name: Security proof
        run: |
          cargo audit --deny warnings
          cargo deny check advisories bans licenses sources

      - name: Install Playwright browser
        run: npx playwright install --with-deps chromium

      - name: UX QA
        run: npm run ux-qa --if-present

      - name: Jankurai audit
        run: |
          jankurai audit . \
            --json repo-score.json \
            --md repo-score.md \
            --sarif jankurai.sarif \

      - name: Jankurai gate
        run: jankurai gate --report repo-score.json --policy agent/audit-policy.toml

      - name: Upload Jankurai artifacts
        uses: actions/upload-artifact@v4
        with:
          name: jankurai-report

            repo-score.json
            repo-score.md
            jankurai.sarif

        run: cat repo-score.md >> "$GITHUB_STEP_SUMMARY"
```

### Fix collector robustness

Current `collectUxElements` appears to collect geometry and then collect element state separately. That risks index mismatch if focus or layout changes between evaluations. Replace it with a single-page snapshot:

```ts
const elements = await page.evaluate(() => {
  const nodes = Array.from(document.querySelectorAll(INTERACTIVE_SELECTOR));
  return nodes.map((node, index) => {
    const rect = node.getBoundingClientRect();
    const style = window.getComputedStyle(node);
    return {
      index,
      selector: stableSelector(node),
      role: node.getAttribute("role"),
      tag: node.tagName.toLowerCase(),
      text: visibleText(node),
      rect: {
        x: rect.x,
        y: rect.y,
        width: rect.width,
        height: rect.height
      },
      visible: rect.width > 0 && rect.height > 0 && style.visibility !== "hidden",
      disabled: isDisabled(node),
      ariaLabel: node.getAttribute("aria-label"),
      labelledBy: node.getAttribute("aria-labelledby"),
      hasAccessibleName: hasAccessibleName(node)
    };
  });
});
```

# The best 3-upgrade implementation order

1. **Ship `jankurai init` first.** Adoption beats everything. Generate canonical files, IDE shims, CI, lockfile warnings, proof-lane configs, and no local paths.

2. **Ship evidence-gated audit/gate second.** Parse policy/maps, kill placeholder evidence, add ratchet, SARIF/JUnit, issue export, and repair prompts.

3. **Ship proof pack third.** Make Rust security and UX QA real, installable, configurable, and CI-enforced; harden the stack boundary contract.

This keeps scope sharp: one installer, one auditor/gate, one proof pack. That is enough to make Jankurai feel like the default way agents should understand and improve a repo.
