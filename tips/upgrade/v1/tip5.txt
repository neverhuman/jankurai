I would **not ship this as “the new standard” yet**. The idea is strong, and the repo has the right thesis: make repositories easier for agents to understand, score, repair, and prove. But the current project still has a gap between **standard-as-paper** and **standard-as-installed-operating-system**.

The three upgrades below are the highest-leverage path to make this adoptable, hard to game, fast, and agent-friendly.

I’d frame the product as:

> **Jankurai = agent-native repo standard + one-command installer + proof-carrying audit gate.**

A key compatibility point: make `AGENTS.md` the canonical front door, then generate tool-specific adapters from it. `AGENTS.md` is now an open, predictable agent-instruction format with broad ecosystem support and nested-file behavior for monorepos; Codex reads `AGENTS.md` before work and layers global/project/current-directory instructions; Cursor supports `.cursor/rules` plus `AGENTS.md`; Claude Code uses `CLAUDE.md`, `.claude/settings.json`, and `SKILL.md`-based skills; Antigravity/Gemini-style tooling exposes rules/workflows and skills, so Jankurai should generate adapters rather than hand-maintain divergent docs. ([Agents][1])

## The 3 best upgrades

## 1. **HL Launchpad: one-command install, repo doctor, and IDE adapter generator**

### Why this matters

Right now, adoption is too manual. The README says to install the Rust CLI, then separately `curl` an agent standard file. CI examples use `cargo run -p jankurai`, which only works from this repository, not from an adopting repo. The project also has no `init`, no `doctor`, no interactive repair flow, no tool-specific adapter generator, and no first-class UX QA install path.

This should become the product’s front door:

```bash
jankurai init
jankurai doctor
jankurai audit .
jankurai gate .
```

The goal is that a messy repo can run:

```bash
jankurai init --profile rust-ts-vite-react-postgres --agents all --ci github --ux-qa
```

…and get a clear plan, safe file creation, IDE helper files, CI gate, audit policy, owner/test maps, generated-zone rules, and optional UX QA.

### What to build

Add these CLI subcommands:

```bash
jankurai init [PATH]
  --profile rust-ts-vite-react-postgres
  --level HL1|HL2|HL3
  --agents codex,cursor,claude,antigravity,copilot,all
  --ci github|none
  --ux-qa
  --issue-backend none|github|beads|jsonl
  --dry-run
  --apply
  --yes

jankurai doctor [PATH]
  --apply
  --json
  --md

jankurai adapters generate [PATH]
  --agents all
  --force

jankurai ux init [PATH]
  --package-manager npm|pnpm|yarn|bun
```

`init` should be idempotent. Second run should produce “no changes” unless the standard version changed.

## 2. **Evidence-Grade Auditor: fast, hard-to-game, policy-driven proof engine**

### Why this matters

The current audit is a useful prototype, but it is too easy to fool with strings. For example, the CI workflow can contain:

```yaml
run: echo "gitleaks syft grype zizmor cargo-audit"
```

…and the audit currently treats that as security evidence because it scans for words, not executable proof.

That is the biggest trust problem. Jankurai cannot become a standard if a repo can pass by mentioning tools in prose or `echo`.

The audit needs to become a **proof-carrying gate**:

```bash
jankurai audit .
jankurai audit . --changed-from origin/main
jankurai gate . --policy agent/audit-policy.toml --baseline agent/baseline.json
```

### Hard checks vs soft checks

Hard checks should be things that make a merge unsafe or misleading:

```text
Hard:
- real secret pattern in source
- generated file edited by hand
- changed path has no owner
- changed path has no test lane
- security lane is placeholder-only
- lockfile missing for detected package manager
- direct DB access from UI or wrong layer
- Rust domain layer does IO/env/time/random/database work
- Python owns product truth outside declared AI/eval exception
- destructive SQL migration lacks receipt/rollback/backfill/lock evidence
- skipped/focused tests in mergeable code
- prompt-injection text in trusted agent instruction files
- agent permission settings grant broad destructive power without receipt

Soft:
- missing Cursor/Claude/Gemini/Copilot adapters
- missing Storybook or route-level UX QA
- large files over threshold
- weak names like misc/helpers/common without ownership
- missing CODEOWNERS
- missing architecture decision record
- missing design-token contract
- missing error-shape guidance
```

### Fix current audit weaknesses

These are the main issues I found in the current Rust audit:

```text
Security evidence:
- Echoed command names count as security tooling.
- Placeholder CI lanes pass.
- No SARIF or annotations.

Maps:
- owner-map/test-map files are checked for existence but not actually parsed as the source of truth.
- Owner routing is hard-coded in Rust.

Generated zones:
- Generated path detection is shallow.
- Header requirements are too weak.
- Generated-zone manifest is not enforced deeply.
- No changed-source-vs-generated diff logic.

Stack boundaries:
- Rust/TS/Python boundaries are mostly text heuristics.
- Direct DB checks can false-positive on prose and false-negative on real imports.
- packages/ux-qa and crates/jankurai are excluded from “product code,” so the repo’s own code escapes some checks.

Future-hostile terms:
- Terms like “old,” “compat,” “fallback,” and “temp” are too broad.
- These need whole-word matching, context, and exception receipts.

False-green tests:
- Good start, but needs better skip/only/snapshot-only/no-assertion detection.

Rendering:
- `--json - --md -` can concatenate JSON and Markdown on stdout.
- Machine-readable output should never share stdout with Markdown.

Performance:
- `rayon` is declared but not meaningfully used.
- File scanning is simple and likely fast on small repos, but the design needs parallel inventory, mmap/bstr handling, and changed-path mode for large repos.
```

### Stronger analyzers

#### CI analyzer

Parse workflow YAML. Do not count words in prose.

Rules:

```text
- echo "gitleaks" does not count.
- comments do not count.
- README mentions do not count.
- `just security` counts only if Justfile security recipe executes real tools.
- commands must be non-placeholder and fail the job on findings.
- pinned actions are preferred; mutable @main is warning or hard depending policy.
```

Test fixture:

```yaml
jobs:
  security:
    steps:
      - run: echo "gitleaks syft grype zizmor"
```

Expected:

```text

```

#### Owner/test-map analyzer

Actually parse:

```json
{
  "entries": [
    {
      "glob": "apps/web/**",
      "owner": "frontend",

    },

      "glob": "crates/domain/**",
      "owner": "domain",
```

#### Generated-zone analyzer

Manifest:

```toml
[[zones]]
glob = "apps/web/src/generated/**"
source = "contracts/openapi/app.yaml"
generator = "openapi-typescript"
```

```text

Source:
Command:
DO NOT EDIT:
```

Hard fail when:

```text
- generated file changed but source contract did not
- generated file lacks header
- generated file outside manifest
- source contract changed but generated files stale
```

#### Vibe-coding risk analyzer

Add checks for the actual “vibe coding mess”:

```text
- junk drawers: utils/, helpers/, common/, misc/ with no owner README
- duplicate behavior: normalized token shingles across functions
- silent fallbacks: catch-all catch blocks, unwrap_or_default in business paths, return null fallback, empty error handling

- false certainty: docs or comments claiming “production ready” while gates fail
- generated drift: generated code changed without source
- hand-written API clients where generated clients exist
- prompt-injection in trusted agent files
- excessive root agent context over size budget
- tool permission creep in .claude/settings.json or other agent configs
- “temporary” exceptions with no expiry
- direct mutable global state in shared modules
- test theater: snapshot-only tests, no assertions, skipped tests
```

### Conformance levels

Make adoption gradual:

```text
HL0 — Observed
- audit runs
- no fail gate
- report only

HL1 — Agent-readable
- AGENTS.md
- JANKURAI_STANDARD.md
- owner-map/test-map
- basic CI audit

HL2 — Agent-actionable
- generated zones
- layer map
- tool adapters
- JSON/Markdown/SARIF reports
- fix queue
- baseline/ratchet

HL3 — Merge-gated
- hard checks enforced
- CI gate
- real security lane

HL4 — Proof-carrying

- dependency rationale

```

This prevents the standard from feeling impossible on day one.

### UX QA hardening

The `packages/ux-qa` package is promising, but it needs to become deterministic and less flaky before it is positioned as a standard lane.

Current issues to fix:

```text
1. CLI should close the browser in try/finally.
2. networkidle can be flaky for apps with sockets, analytics, polling, or streaming.
```

### CI integration

The adopting repo should get this:

```yaml
name: jankurai

on:
  pull_request:
  push:
    branches: [main]

    runs-on: ubuntu-latest

  ux-qa:
    runs-on: ubuntu-latest

    steps:
      - uses: actions/checkout@v4

      - run: npm run build
```

# Paper and standard improvements

The paper has the right energy, but it should be sharpened into an adoption spec. I’d add these sections:

```text
1. Conformance levels: HL0 through HL4
2. The adoption loop: init → audit → fix queue → proof receipts → gate → ratchet
3. Agent adapter matrix: AGENTS.md, Cursor, Claude, Codex, Gemini/Antigravity, Copilot
4. Threat model: agent with file, shell, browser, and IDE access
5. Hard vs soft checks
6. Generated-code contract
7. Layer-boundary contract
8. CI proof contract
9. UX proof contract
10. Exception and expiry contract
```

The most important conceptual change:

> Jankurai should not say “humans are wrong, AI is right.”
> It should say: “Unstructured repositories are hostile to both humans and agents. Jankurai makes the repo legible, bounded, testable, and repairable.”

That phrasing will get broader adoption without losing the ambition.

# The final product shape I’d aim for

```bash
# Install
cargo install jankurai --locked

# Adopt
jankurai init --profile rust-ts-vite-react-postgres --agents all --ci github --ux-qa --apply

# Check
jankurai audit . --json jankurai.json --md jankurai.md --sarif jankurai.sarif

# Gate
jankurai gate . --policy agent/audit-policy.toml

# Fix queue
jankurai issues export --backend github
jankurai issues export --backend beads

# Agent loop
jankurai audit . --changed-from origin/main
jankurai doctor --dry-run
```

The winning standard is not just the paper. It is the loop:

```text
Agent reads canonical instructions.
Agent identifies layer/owner/test lane.
Agent edits bounded files.
Agent runs targeted audit.
Agent receives actionable findings.
Agent repairs.
CI enforces hard checks.
Repo ratchets upward over time.
```

That is the path to making Jankurai feel like the default operating system for agentic coding rather than another style guide.
