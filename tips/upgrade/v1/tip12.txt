I studied the archive closely: paper, docs, installer guidance, Rust audit CLI, CI workflow, agent files, and the TypeScript Playwright UX QA package. The core idea is excellent: **Jankurai should not be “another linter”; it should be the repo control plane that makes agent work legible, scoped, auditable, and fast.** The project is already aiming at the right target.

The biggest risk is that the current repo can produce **false confidence**. The bundled score report says “no findings,” but the source-only archive has missing lockfiles, placeholder security checks, weak string-based evidence, and the audit excludes much of its own authored code from scope. Before this becomes a standard, the project needs to make one promise: **Jankurai never says “passed” unless something real was parsed, executed, or artifact-backed.**

I could not execute Rust tests in this environment because `cargo`/`rustc` are not installed here, so the notes below are from source/config inspection rather than runtime verification.

## Best 3 upgrades

These are the three upgrades I would make first. They are deliberately big enough to make Jankurai a standard, but bounded enough to avoid product creep.

---

# 1. **Jankurai Init Pack: one-command adoption for every agent IDE**

## Goal

Make Jankurai installable with one obvious command that:

1. detects the repo shape,
2. writes the correct agent-helping files,
3. installs or configures the Rust audit and UX QA tester,
4. creates CI,
5. asks the user only when a repo is non-canonical or risky to modify.

This is the adoption unlock. Right now the project asks users to manually install the CLI, copy markdown, create `AGENTS.md`, and wire CI. That is too much friction for a global standard.

The installer should treat `AGENTS.md` as the cross-agent root, then generate adapter files for major IDE/agent systems. This matches the current ecosystem: `AGENTS.md` is explicitly positioned as an open format for agent instructions, Codex reads layered `AGENTS.md` files with a size cap, Claude Code uses `CLAUDE.md` memory files, GitHub Copilot supports repository and path-specific instruction files plus `AGENTS.md`, and Antigravity examples use `.agents` folders with skills/workflows. ([Agents][1])

## New command surface

```bash
jankurai init
jankurai init --yes
jankurai init --dry-run
jankurai init --ide all
jankurai init --ide codex,cursor,claude,copilot,antigravity
jankurai init --mode advisory
jankurai init --mode standard
jankurai init --mode ratchet
jankurai init --issue-tracker beads
jankurai doctor
jankurai upgrade
jankurai explain HLT-004
```

Recommended first-run behavior:

```bash
jankurai init --ide all --mode advisory --issue-tracker auto
```

The installer should default to **advisory mode**, because global adoption depends on being safe in messy repos. Then CI can ratchet over time.

## Boundary standard: what each layer owns

Add this as a clear table in the paper and `docs/rust-ts-vite-react-boundaries.md`.

| Layer                | Owns                                                                                    | Must not own                                                                 |
| -------------------- | --------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| Rust core/domain     | business rules, invariants, validation, permissions, durable product logic              | browser rendering, CSS, local UI state                                       |
| Rust app/API/workers | orchestration, DB access, jobs, auth enforcement, observability                         | React component behavior, generated UI code                                  |
| PostgreSQL           | durable constraints, indexes, migrations, RLS where applicable                          | unreviewed business logic hidden from tests                                  |
| TypeScript           | browser interop, generated clients, UI state, form wiring, Playwright/Storybook tooling | product truth, authorization truth, DB mutation rules                        |
| React                | visual composition, input flows, accessibility surfaces                                 | domain decisions that should be tested in Rust                               |
| Vite                 | browser build/dev server                                                                | backend routing or product logic                                             |
| Python               | ML/model/eval/data-science islands only                                                 | core product backend, auth, DB write logic, general scripts that can be Rust |

This will stop a lot of “vibe coding scatter” because agents will know where a change belongs before writing code.

# 2. **Evidence-Grade Audit: no more false green**

## Goal

Turn the audit from a useful heuristic scanner into a standard-grade proof engine.

The current scanner can be fooled by strings in docs, placeholder commands, or echoed tool names. That is dangerous for a standard whose job is to reduce agent mistakes. The next version needs an evidence model:

```text
not checked ≠ passed
declared ≠ executed
mentioned in docs ≠ configured
configured ≠ artifact-backed
artifact-backed ≠ reviewed
```

## Current audit issues to fix

These are the most important issues I found in the archive:

1. **Security lane placeholders can pass.**
   The workflow contains placeholder `echo` commands for security tools. The audit can still detect tool names from command surfaces.

2. **The audit excludes its own authored code.**
   `crates/jankurai/` and `packages/ux-qa/` are effectively excluded from runtime/product code scope, so the repo can score itself while skipping its actual implementation.

3. **Docs can create fake evidence.**
   `react`, `vite`, `storybook`, or `generated` appearing in docs can affect detected repo capabilities.

4. **Policy files exist but are not authoritative.**
   `agent/audit-policy.toml`, `agent/owner-map.json`, `agent/test-map.json`, and `agent/proof-lanes.toml` should drive checks. Today too much is hardcoded.

5. **Changed-path mode is too blunt.**
   It scopes some files, but many checks still look at the whole repo. Agents need a report that separates “changed-path blockers” from “baseline debt.”

6. **No lockfiles in the source-only archive.**
   The archive has no `Cargo.lock` or `package-lock.json`, while the workflow expects `npm ci`. That is a release-quality issue.

7. **Pattern scanning is too case-sensitive and shallow.**
   `fallback`, `TODO`, `stub`, `panic`, `unwrap`, etc. need more robust scanners and waiver handling.

8. **Weak-name detection is stubbed.**
   The standard talks about vague naming, but the scanner currently does not enforce it meaningfully.

## New evidence model

Add:

```text
crates/jankurai/src/audit/evidence.rs
```

Core model:

```rust
pub enum EvidenceKind {
    FileExists,
    ParsedManifest,
    ParsedPolicy,
    ParsedCiStep,
    CommandDeclared,
    CommandExecuted,
    ArtifactPresent,
    Finding,
    Waiver,
}

pub enum EvidenceConfidence {
    Low,
    Medium,
    High,
}

pub struct Evidence {
    pub kind: EvidenceKind,
    pub confidence: EvidenceConfidence,
    pub path: Option<Utf8PathBuf>,
    pub line: Option<usize>,
    pub command: Option<String>,
    pub artifact: Option<Utf8PathBuf>,
    pub note: String,
}
```

Scoring rule:

```text
A rule may only award credit from EvidenceKind values explicitly allowed by that rule.
```

Examples:

```text
Security lane present:
- FileExists: no credit
- CommandDeclared: partial credit only
- ParsedCiStep with real non-echo command: partial credit
- ArtifactPresent from a completed scan: full credit

UX lane present:
- package mentions Playwright: no credit
- script exists: partial credit
- ux-qa artifact consumed: full credit
```

## Hard checks vs soft checks

Add this distinction to `agent/audit-policy.toml`.

### Hard checks

Hard checks should block standard-mode CI:

```text
HLT-001 root AGENTS.md or equivalent missing
HLT-002 Jankurai standard file missing or stale
HLT-003 owner-map missing changed authored path
HLT-004 test-map/proof lane missing changed authored path
HLT-005 generated zone modified without source contract
HLT-006 secret or token-like material detected
HLT-007 prompt injection in trusted agent instructions
HLT-008 security lane declared but not executable
HLT-009 Python product logic outside allowed ML/eval roots
HLT-010 direct DB/auth/network boundary violation
HLT-011 destructive migration without rollback/proof
HLT-012 TypeScript browser code owns backend/domain truth
HLT-013 web surface changed without rendered UX proof
HLT-014 CI missing Jankurai audit
HLT-015 lockfile missing for package manager in use
HLT-016 score below configured floor
```

### Soft checks

Soft checks should produce issues but not always block:

```text
vague variable names
large file approaching cap
large function approaching cap
TODO/FIXME without issue link
unused dependency suspicion
non-canonical layout but mapped
missing optional IDE adapters
weak docs-to-code synchronization
duplicated utility logic
excessive test snapshots
overuse of any/unknown/ts-ignore
```

## Real Rust safety/security/performance lanes

Jankurai should not rebuild every mature Rust tool. It should orchestrate and consume them.

Recommended optional lanes:

```text
cargo audit
cargo deny check
cargo nextest run
cargo geiger
cargo machete
cargo mutants
```

`cargo-audit` uses the RustSec advisory database against `Cargo.lock`; `cargo-deny` can check advisories, licenses, bans, sources, and duplicate versions; `cargo-nextest` is built for faster Rust test execution in CI; `cargo-geiger` reports unsafe Rust usage; `cargo-machete` detects unused dependencies but notes it can be imprecise; `cargo-mutants` finds areas where tests do not fail after code mutations. ([RustSec][3])

The standard should treat these as **lanes**, not as mandatory dependencies for every repo.

Example:

```toml
[lanes.rust_fast]
commands = [
  "cargo fmt --check",
  "cargo clippy --workspace --all-targets -- -D warnings",
  "cargo nextest run --workspace"
]

[lanes.rust_security]
commands = [
  "cargo audit --json > target/jankurai/cargo-audit.json",
  "cargo deny check"
]

[lanes.rust_deep]
mode = "nightly"
commands = [
  "cargo geiger",
  "cargo machete",
  "cargo mutants --in-place --timeout 30"
]
```

## Speed improvements

The audit should be fast enough for agents to call repeatedly.

Implementation guidance:

```text
- Build one repo inventory.
- Hash file contents once.
- Classify files once.
- Parse manifests once.
- Parse policy/maps once.
- Precompile Aho-Corasick matchers once.
- Use changed-path mode by default for PRs.
- Cache expensive results in target/jankurai/cache.json.
- Run deep lanes only nightly or when touched paths require them.
- Treat UX QA as route-specific, not whole-app by default.
```

Current repeated whole-repo string scanning should become:


# 3. **Proof Router + UX Receipts: make agents faster, not just safer**

## Goal

Jankurai should become the thing agents call before, during, and after coding:

```bash
jankurai route --changed apps/web/src/components/BillingForm.tsx
jankurai audit . --changed-from origin/main
jankurai ux audit --config agent/ux-qa.toml
jankurai receipt append --lane ux --artifact target/jankurai/ux-qa.json
```

This is where Jankurai becomes more than a compliance tool. It becomes an **agent efficiency multiplier**.

## Why this matters

Agents fail when they do not know:

```text
- where code belongs,
- what tests prove the change,
- what files are generated,
- what UI states must be rendered,
- what repo-specific rules matter,
- what issue or proof artifact to produce.
```

The project already has the ingredients: proof lanes, owner maps, test maps, generated zones, audit reports, and a Playwright UX QA package. The next step is to connect them into a router.

## New command surface

```bash
jankurai route --changed <path>
jankurai route --changed-from origin/main
jankurai audit . --changed-from origin/main --with-route
jankurai receipt append --lane rust_fast --cmd "cargo nextest run" --artifact target/nextest/default/junit.xml
jankurai ux audit --url http://localhost:5173 --config agent/ux-qa.toml --out target/jankurai/ux-qa.json
jankurai ux from-storybook --storybook-url http://localhost:6006 --config agent/ux-qa.toml
```

## UX QA package improvements

The current `packages/ux-qa` is a genuinely strong seed. It checks edge clearance, target size, overlaps, clipping, wrapping, horizontal overflow, sticky obstruction, z-index, focus visibility, form labels, nested scrollbars, screenshots, crops, and ARIA snapshots.

But it needs correctness hardening before being a standard.

### Fix selector stability

Current selector generation appears to risk using a global `nth-of-type` index. That can generate selectors that do not resolve to the original element.

Add:

```text
packages/ux-qa/src/selector.ts
```

Selector strategy:

```text
1. data-testid / data-jankurai-id
2. accessible role + name where stable
3. id if unique and not generated-looking
4. class only if stable-looking and unique enough
5. DOM path with tag-specific nth-of-type among siblings
6. fallback internal element hash
```

Test:

```text
packages/ux-qa/tests/selector.spec.ts
```

Acceptance:

```text
- Every emitted selector resolves back to the original element in fixture pages.
- No global NodeList index is used for nth-of-type.
```

### Fix focus visibility

Current focus detection mainly looks at outline and box shadow. Real focus can be visible through border, background, underline, transform, or component-specific state.

Add:

```text
packages/ux-qa/src/focus.ts
```

Better approach:

```text
- capture computed style before focus,
- focus element,
- capture computed style after focus,
- compare outline, box-shadow, border, background, text-decoration, transform,
- optionally crop screenshot before/after for artifact-backed proof.
```

Test:

```text
packages/ux-qa/tests/focus.spec.ts
```

### Fix sticky obstruction

Sticky/fixed obstruction should not only use bounding-box overlap. It should use hit-testing.

Add:

```text
packages/ux-qa/src/hit-test.ts
```

Algorithm:

```text
- sample important points on target element,
```

# Paper improvements

The paper is pointed in the right direction, but it should be sharpened around evidence and adoption.

## Add an “Evidence Maturity Model”

This should become a central concept.

```text
Level 0: Mentioned
A tool, test, or standard is mentioned in docs.

Level 1: Declared
A config file or script declares it.

Level 2: Parsed
Jankurai can parse and validate the declaration.

Level 3: Executed
CI or local audit executed the proof command.

Level 4: Artifact-backed
The proof produced a machine-readable artifact.

Level 5: Routed
The artifact is tied to changed paths, owners, and proof lanes.

Level 6: Ratcheted
The repo tracks baseline debt and requires non-regression.
```

This resolves the current weakness where documentation can masquerade as proof.

## Reframe the “jankurai” tone

The “humans are the bug” energy is memorable, but global adoption will be easier if the standard says:

> Jankurai removes human scatter from repositories. Humans still set intent, approve tradeoffs, and own product judgment; agents get a stable operating surface.

That keeps the edge without alienating maintainers.

## Add a canonical-stack commitment

The paper should explicitly say:

```text
Jankurai standardizes on one preferred stack:
Rust for product truth,
PostgreSQL for durable data truth,
TypeScript/React/Vite for browser UI,
Playwright/UX-QA for rendered UI proof,
Python only for ML/eval islands.
```

Then say:

```text
Other stacks are allowed through declared exceptions, but Jankurai optimizes for this stack.
```

That is how you prevent endless human scatter.

## Add a “standard, not framework” section

Jankurai should not become an app framework. It should remain:

```text
- repo layout standard,
- agent instruction standard,
- audit standard,
- proof routing standard,
- report/receipt standard,
- IDE adapter pack.
```

Do not build:

```text
- a full task manager,
- a replacement for Beads/GitHub Issues,
- a replacement for Playwright,
- a replacement for RustSec/cargo-deny,
- a full web framework.
```

This keeps it powerful without creep.

# The single most important change

The project must stop awarding confidence for weak evidence.

The next version should treat every check as one of:

```text
passed
failed
not checked
not applicable
waived
baseline debt
```

Never collapse `not checked` into `passed`.

That one rule will make Jankurai dramatically more trustworthy.
