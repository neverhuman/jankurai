Here’s the blunt read: **jankurai has the right core idea, but it is not yet “new global standard” ready because installation, evidence trust, and repo-shape guidance need to become product-grade.** The project already has the right ingredients: a Rust scorer, versioned agent standard, CI starter, and Playwright rendered-UX package. The next step is to make adoption feel like running one command, not reading a paper and hand-copying conventions.

I would make **exactly three upgrades** and treat everything else as backlog.

## Immediate release blockers I found

Before the three upgrades, fix these first because they directly weaken trust:

| Blocker                                                                                     | Why it matters                                                                                     | Fix                                                                                             |
| ------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------- |
| Release archive includes many `._*` AppleDouble files                                       | Agents may inventory junk files and produce noise.                                                 | Add release packaging script that excludes `._*`, `.DS_Store`, `test-results`, local artifacts. |
| `Cargo.lock` is missing while docs use `--locked`                                           | A locked install cannot be reproducible without a lockfile.                                        | Commit `Cargo.lock` for the CLI workspace.                                                      |
| `package-lock.json` is missing while CI uses `npm ci`                                       | `npm ci` requires a lockfile. Current CI would fail in a clean clone.                              | Commit `package-lock.json` or change CI, preferably commit the lockfile.                        |
| Root `AGENTS.md` contains `/Users/bentaylor/Code/jankurai`                                 | This ships a local path into the standard.                                                         | Replace with repo-relative guidance.                                                            |
| CI security lane only echoes `gitleaks`, `syft`, `grype`, `zizmor`                          | Current audit can see those strings and treat them as proof. This is the biggest false-green risk. | Require real commands/actions, not echoed placeholders.                                         |
| `repo-score.md/json` says lockfile present even though lockfiles are absent                 | Audit output can become stale or untrustworthy.                                                    | Add report freshness fields: commit SHA, dirty flag, input fingerprint, policy fingerprint.     |
| README/docs reference missing `assets/`, `tips/`, `reference/`, and generated PDF artifacts | Agents follow broken paths and waste context.                                                      | Either include those artifacts or mark the archive as source-only and suppress those links.     |
| UX QA committed `test-results` and absolute local artifact paths                            | Agent outputs become environment-specific.                                                         | Move artifacts under `.jankurai/artifacts/ux/` and gitignore them.                             |

I could inspect the source archive closely, but I could not complete dependency-backed tests in this environment because external crate/package fetching was unavailable. That said, the lockfile issues above are independently visible and should be treated as real release blockers.

# Upgrade 1 — **Jankurai Adopt Kit**

**Goal:** make jankurai installable and adoptable with one command across Codex, Claude Code, Cursor, GitHub Copilot, Gemini CLI, Antigravity, and generic agent IDEs.

Right now adoption is too manual. The standard says the right things, but a global standard needs a **guided installer** that writes the expected files, explains detected repo problems, and can repair obvious layout gaps.

## What to build

Add these CLI commands:

```bash
jankurai init \
  --level HL3 \
  --stack rust-ts-vite-react-postgres-bounded-python \
  --agents all \
  --ci github \
  --ux-qa \
  --dry-run

jankurai init --apply --yes

jankurai doctor --fix

jankurai audit \
  --policy agent/audit-policy.toml \
  --min-score 85 \
  --fail-on high,critical
```

The install/adopt flow should be:

1. Detect repo root, language surfaces, package manager, CI, web surface, lockfiles, existing agent docs.
2. Generate a **plan** first.
3. In an interactive terminal, ask before overwriting or restructuring.
4. In CI or noninteractive mode, fail with a machine-readable repair plan unless `--yes` is passed.
5. Write the standard files and agent-IDE adapters.
6. Run `jankurai audit --changed` immediately after setup.

This should feel like `rustup`, `cargo init`, or `npx create-*`, not a research repo.

## Generated `AGENTS.md` should be tiny

Template:

```md
# Agent Instructions

Read `agent/JANKURAI_STANDARD.md` first.

Jankurai target stack:
Rust core + TypeScript/React/Vite UI + PostgreSQL durable truth
+ generated contracts + bounded Python AI/data service only.

Before edits:
- inspect `agent/owner-map.json`
- inspect `agent/test-map.json`
- inspect `agent/generated-zones.toml`
- choose the smallest proof lane
- do not edit generated files by hand
- do not broaden agent/tool permissions

After meaningful edits:
- run `jankurai audit --changed`
- run the mapped proof lane
- leave a repair receipt when fixing non-trivial drift
```

The tool-specific files should mostly point back to this. For Claude:

```md
@AGENTS.md

## Claude Code

Use plan mode for broad refactors, migrations, generated-zone changes, and security-sensitive edits.
Use `.claude/rules/` for path-scoped details instead of expanding this file.
```

This also lets jankurai enforce **context economy**: short root, local detail, no contradictory agent files.

# Upgrade 2 — **Proof-Bearing Audit Engine**

**Goal:** make the audit impossible to confuse with string-grep vibes. Every score needs evidence, confidence, rule type, proof lane, and repair command.

This is the most important upgrade. The current audit already has many good rule IDs, but several checks are too string-based. The worst example is the CI security lane: the workflow echoes security tools, and the scanner can still treat those words as evidence. That breaks the core promise: **trustworthy merge**.

## What to build

Refactor the audit into a proof engine with real parsed inputs:

```text
crates/jankurai/src/audit/context.rs
crates/jankurai/src/audit/evidence.rs
crates/jankurai/src/audit/maps.rs
crates/jankurai/src/audit/policy.rs
crates/jankurai/src/audit/git.rs
crates/jankurai/src/audit/rules/
  context.rs

  proof_lanes.rs
  generated.rs
  contracts.rs
  security.rs
  boundaries.rs
  python.rs
  ux.rs
  ci.rs
  shape.rs
  reports.rs
```

The scanner should parse and use:

```text

agent/owner-map.json
agent/test-map.json
agent/generated-zones.toml
agent/proof-lanes.toml
.github/workflows/*.yml
```

Do not hard-code the owner map if `agent/owner-map.json` exists. Do not treat a path as mapped unless the config says it is.

## Evidence model

Add a first-class evidence object:

```rust
#[derive(Debug, Clone, Serialize)]
pub struct Evidence {
    pub kind: EvidenceKind,
    pub path: String,
    pub line: Option<u32>,
    pub command: Option<String>,
    pub expected: Option<String>,
    pub actual: Option<String>,
    pub confidence: EvidenceConfidence,
    pub hard: bool,
}

#[derive(Debug, Clone, Serialize)]
pub enum EvidenceKind {
    FileExists,
    FileMissing,
    ConfigParsed,
    CommandDeclared,
    CommandRan,
    LockfilePresent,
    ToolPinned,
    WorkflowAction,
    AstViolation,
    ImportGraph,
    GeneratedMarker,
    Heuristic,
}
```

Every finding should carry:

```json
{
  "rule_id": "HLT-016-SUPPLY-CHAIN-DRIFT",
  "severity": "high",
  "check_type": "hard",
  "confidence": "high",
  "path": ".github/workflows/jankurai-audit.yml",
  "line": 17,
  "problem": "security lane echoes gitleaks instead of running it",
  "agent_fix": "replace placeholder echo with a real secret scan step",
  "verify_command": "jankurai audit --rule security-ci",
  "owner": "ops/security",
  "lane": "security",
  "autofixable": false,
  "issue_title": "Replace echoed security lane with real scanner",
  "issue_body": "..."
}
```

## Hard checks vs soft checks

Hard checks should cap or fail CI. Soft checks should guide repair without blocking early adopters.

| Hard check                                                              | Why                                       |
| ----------------------------------------------------------------------- | ----------------------------------------- |
| `npm ci` without lockfile                                               | CI cannot be deterministic.               |
| `cargo install --locked` / `cargo test --locked` without `Cargo.lock`   | Rust dependency resolution is not locked. |
| CI security step only echoes tool names                                 | False-green security lane.                |
| Root agent file missing or contradictory                                | Agents start with bad context.            |
| Ownerless changed path                                                  | No accountable repair route.              |
| Unmapped proof lane                                                     | Agent cannot know what to run.            |
| Generated file changed without source contract change                   | Contract drift.                           |
| Direct DB access from UI/domain/Python                                  | Durable truth leak.                       |
| Python owns authz/product truth                                         | Stack boundary violation.                 |
| Prompt-injection or overbroad tool permissions in trusted agent docs    | Agent safety boundary violation.          |
| Skipped/focused/tautological test                                       | False-green proof.                        |
| Secret-like material in source, fixtures, transcripts, or agent configs | Merge-blocking leak.                      |
| Stale `repo-score.*`                                                    | Audit report cannot be trusted.           |

Soft checks:

| Soft check                                                    | Why                                |
| ------------------------------------------------------------- | ---------------------------------- |
| Future-hostile words like `todo`, `fallback`, `legacy`, `old` | Useful smell, but needs context.   |
| Weak names like `utils`, `helpers`, `common`, `misc`          | Often ownership failure.           |
| Large files/functions                                         | Refactor pressure.                 |
| Duplicate behavior                                            | Agent drift amplifier.             |
| Docs mention React/Vite but no actual web app                 | Avoid false web-surface detection. |
| Too much Python outside allowed AI/data/tooling               | Product-truth risk.                |
| Missing UX route manifest                                     | UI proof incomplete.               |
| Missing issue/receipt export                                  | Harder multi-agent repair.         |

## Fix the current audit gaps

The current engine should be changed in these exact ways:

```text
1. Stop treating docs mentions of React/Vite/Storybook as web surface.
   Require actual package deps, Vite config, TSX app files, or web app path.

2. Stop treating any file containing "generated" as generated contract evidence.
   Require generated-zones.toml plus marker block:
   Generated by / Source / Command / DO NOT EDIT BY HAND.

3. Stop treating `rstest` as property testing.
   Count proptest/quickcheck as property testing; count rstest as table/fixture testing.

4. Stop giving positive security evidence for `unsafe`.
   `unsafe` should require an allowlist, comment, owner, and security lane.

5. Stop swallowing unreadable text as empty content.
   Silent `unwrap_or_default` in file reading should become low/medium finding or inventory warning.

6. Stop excluding the audit crate from code-shape checks.
   This repo is itself a Rust tool; `crates/jankurai` must be auditable.

7. Make `--changed` truly changed-aware.
   Global invariants still run, but local findings should use changed path closure from test-map/owner-map.

8. Parse workflow YAML.
   Distinguish `echo "gitleaks"` from an actual command or action.

9. Parse policy TOML.
   The scanner should not have hidden score caps that differ from `agent/audit-policy.toml`.

10. Detect stale score files.
    Require commit SHA, dirty flag, generated timestamp, input fingerprint, and policy fingerprint.
```


# Upgrade 3 — **Winner Stack Boundary + UX Proof Pack**

**Goal:** make the standard sharper: one recommended stack, one canonical layout, one exception path, and rendered UX proof that agents can run without guessing.

Do **not** dilute the standard by trying to support every stack equally. The project’s core power is that it chooses:

```text
Rust core
+ TypeScript/React/Vite UI
+ PostgreSQL durable truth
+ generated contracts
+ bounded Python AI/data service only
```

That should remain the default. Other stacks can exist by exception, but the standard should not become a polite checklist for everyone’s current mess.

## Make boundaries impossible to misunderstand

Add or expand:

```text
docs/winner-stack.md
docs/boundaries.md
docs/layout.md
docs/exceptions/template.md
templates/winner-stack/
agent/stack-policy.toml
```

```text

  CLAUDE.md
  GEMINI.md
  agent/
    JANKURAI_STANDARD.md
    audit-policy.toml
    owner-map.json
```

## Boundary table to enforce

| Layer             | Owns                                                         | Must never own                                                              |
| ----------------- | ------------------------------------------------------------ | --------------------------------------------------------------------------- |
| Vite              | Build/dev server, frontend bundling, HMR, preview            | Product truth, API behavior, authz, DB access                               |
| React             | Components, view state, forms, generated client usage        | Durable rules, direct SQL, secrets, business invariants                     |
| TypeScript        | UI types, generated API types, client-side validation, tests | Canonical domain model, production truth, handwritten server contract drift |
| Rust API edge     | HTTP/RPC extraction, auth context loading, response mapping  | Hidden domain decisions, scattered SQL, UI rules                            |
| Rust domain       | IDs, invariants, state machines, pure decisions              | IO, env, time, random, framework types, DB, logging side effects            |
| Rust application  | Commands, authorization, transactions, idempotency           | UI concerns, provider-specific protocols                                    |
| Rust adapters     | PostgreSQL, queues, external APIs, filesystem, env           | Business rules                                                              |
| PostgreSQL        | Constraints, migrations, indexes, RLS, transactional truth   | App orchestration or UI policy                                              |
| Python AI service | Models, embeddings, evals, transforms, ML/data service       | Product authz, production DB writes, durable product truth                  |
| Contracts         | OpenAPI/protobuf/JSON Schema source and generated clients    | Handwritten duplicated DTOs                                                 |

This is the anti-scatter core. Every audit rule should point back to this table.

## Fix UX QA correctness gaps

The current package is a strong start, but I would change these:

```text
1. Replace `networkidle` as the only readiness signal.
   Use route-specific `wait_for` selectors plus optional networkidle.

2. Stop writing absolute artifact paths.
   Reports should use repo-relative paths.

3. Do not commit `packages/ux-qa/test-results`.
   Move runtime artifacts to `.jankurai/artifacts/ux/`.

4. Collect element boxes and states in one browser evaluation.
   Current separate evaluations can drift by index if focus/visibility changes.

5. Key states by stable selector or internal element id, not array index.

6. Make selector generation collision-aware.
   Prefer data-testid, role/name, id, then full path only as fallback.

7. Add owner/route/viewport/action trace to every violation.

8. Add config thresholds by route.
   Critical mobile flows can use stricter target-size/edge rules.

9. Support Storybook mode.
   Route IDs can map to `iframe.html?id=<story>`.

10. Emit repair receipts:
    screenshot, crop, ARIA snapshot, DOM summary, route manifest entry, viewport.
```

# The final standard shape

After these three upgrades, the project becomes:

```text
jankurai
  = one-command adopter
  + agent-IDE instruction router
  + proof-bearing Rust audit
  + rendered UX QA lane
  + winner-stack boundary standard
  + CI/action integration
  + issue/beads repair queue
```

The clean adoption story should be:

```bash
cargo install jankurai --locked

jankurai init \
  --level HL3 \
  --agents all \
  --ci github \
  --ux-qa \
  --yes

jankurai audit --changed
jankurai audit --full --min-score 85 --fail-on high,critical
```

And the agent behavior should become:

```text
Start task:
  read AGENTS.md -> agent/JANKURAI_STANDARD.md
  inspect owner-map/test-map/generated-zones
  run changed audit if repo already has drift

During task:
  edit only mapped owner scope
  avoid generated hand edits
  run smallest proof lane

Before final:
  run jankurai audit --changed
  run mapped tests
  leave repair receipt
```

My strongest recommendation: **do not expand the scope horizontally.** Do not add ten more stacks. Do not turn UX QA into a general browser framework. Do not turn jankurai into a full issue tracker. Instead, make the default stack and proof path so clear that agents stop guessing. The winning move is not “support everything”; it is **reduce ambiguity until wrong code has nowhere to hide.**
