I studied the source closely. The project is already unusually strong in thesis and direction: it has a paper, a root `AGENTS.md`, a short `agent/JANKURAI_STANDARD.md`, a Rust audit CLI, stable rule IDs, a score report, a target stack, CI starter, and a separate Playwright UX geometry package. The missing piece is not vision. The missing piece is **operational certainty**: one command must install the standard, adapt it to each agent IDE, repair repo structure, run the right proof lanes, and fail CI for the right reasons.

A recent 2026 study of AI IDE-generated large projects is almost a direct argument for jankurai: generated projects were functionally strong on average, but still had large volumes of design issues, especially duplication, high complexity, large methods, framework best-practice violations, exception-handling problems, and accessibility issues. That is exactly the gap jankurai should own: **not “can the code run?” but “can agents keep repairing it safely?”** ([arXiv][1])

## The three highest-leverage upgrades

### 1. **Jankurai Init Doctor**

**Goal:** make adoption one command, self-repairing, and tool-aware.

Right now the project says the right things, but installation still expects humans or agents to manually place files. That is the wrong standard for a project whose thesis is “remove jankurai.” The standard should install itself, explain mismatches, generate a patch, and optionally apply it.

The new command should be:

```bash
jankurai init
jankurai doctor
jankurai fix
jankurai ci install --github
jankurai agents sync
```

The key idea: **`AGENTS.md` stays canonical**, while tool-specific files are thin generated adapters. This matches the real agent-IDE landscape. Codex reads `AGENTS.md` before doing work and supports nested instruction precedence; Claude Code uses `CLAUDE.md` and can import `@AGENTS.md`; Gemini uses `GEMINI.md`/memory files and supports imports; Copilot supports `.github/copilot-instructions.md`, path-specific `.github/instructions/*.instructions.md`, and `AGENTS.md`; Antigravity’s official codelab uses `.agents/agents.md`, `.agents/skills`, and `.agents/workflows`; Cursor’s official rules documentation describes persistent rules plus `AGENTS.md`. ([OpenAI Developers][2])

#### What this upgrade fixes

The current repo has these adoption blockers:

* No `init` or `doctor` subcommand.
* `agent/audit-policy.toml` exists but the audit engine does not actually consume it.
* Manual curl/install steps are too fragile for broad adoption.
* Tool-specific agent files are discussed in docs, but not generated.
* The CI starter has placeholder security commands that are only `echo` statements.
* The uploaded source has `package.json` but no `package-lock.json`, while CI runs `npm ci`.
* The uploaded source has no `Cargo.lock`, but install/docs use `--locked`.
* `packages/ux-qa/test-results/**` is present in source and should be ignored/removed.
* Paper CI runs `latexmk` without installing a TeX environment.
* The audit is advisory by default; CI does not enforce a score floor or severity floor.

Those are exactly the kind of “vibe coding mess” this project should eliminate.

#### Distribution strategy

Do not make users compile from source as the main path. Keep `cargo install`, but add prebuilt binaries and wrapper installs:

```bash
# best default
curl -LsSf https://jankurai.dev/install.sh | sh

# Rust users
cargo binstall jankurai

# fallback
cargo install jankurai --locked

# JS/agent users
npx jankurai init
```

`cargo-binstall` is a good fit because it installs Rust CLIs from prebuilt binaries instead of compiling from source and supports unattended CI usage with `--no-confirm`; `cargo-dist`/`dist` can generate release binaries, installers, manifests, and release CI. ([GitHub][3])

### 2. **ProofGraph Audit Engine**

**Goal:** make the audit fast, precise, configurable, hard/soft aware, CI-native, and agent-actionable.

The current audit is impressive for `0.2.0`, but it is too monolithic and too heuristic to become the global standard. `crates/jankurai/src/audit/mod.rs` is doing inventory, scoring, policy, rule checks, finding construction, UX readiness, and route inference in one large file. Several expensive signals are recomputed repeatedly: `todo_hits`, `fallback_hits`, `future_hostile_hits`, `duplicate_blocks`, `wrong_layer_db_hits`, `ux_qa_status`, `has_web_surface`, and `has_generated_contracts`.

This should become a cached evidence graph.

Replace the current monolith with:

#### Hard checks to implement first

These should block, cap, or fail CI.

```text
HLT-003-OWNERLESS-PATH
  Parse agent/owner-map.json.
  Every changed path must match an owner rule.
  Finding includes expected owner-map location to add.

HLT-004-UNMAPPED-PROOF
  Parse agent/test-map.json.
  Every changed path must map to at least one proof lane.
  Finding includes the exact lane command.

HLT-002-GENERATED-MUTATION
  Parse agent/generated-zones.toml.
  Check generated header.
  If generated output changed without source contract change, block.

HLT-006-DIRECT-DB-WRONG-LAYER
  Detect SQLx/Diesel/Postgres imports and DATABASE_URL reads.
  Allow only db/, crates/adapters/, and explicit exceptions.

HLT-005-PYTHON-PRODUCT-TRUTH
  Python may exist for AI/data/eval/tooling.
  Python may not own authz, product write paths, direct prod DB, or API contracts.

HLT-008-FALSE-GREEN-RISK
  Detect .skip, .only, xtest, xit, tautology asserts, snapshot-only changed behavior.

HLT-010-SECRET-SPRAWL
  Keep internal regex as a cheap pass, but also require gitleaks or equivalent in CI.

HLT-016-SUPPLY-CHAIN-DRIFT
  Dependency changes require lockfile, vulnerability scan, license/source policy, and review rationale.

HLT-013/014-RENDERED-UX/A11Y-GAP
  Web surfaces require rendered UX evidence for changed critical routes.
```

#### Soft checks to implement next

These should produce warnings, repair suggestions, and score drag without blocking unless policy says so.

```text
weak names:
  utils, helpers, misc, common, shared, legacy, old, temp, scratch, stuff

fallback soup:
  broad catch, silent default, retry loops without policy, stale cache fallback

orphan code:
  module has no incoming imports and no test path

oversized functions:
  parse Rust/TS/Python functions, not just file LOC

framework drift:
  React components doing server truth, direct fetch wrappers where generated client exists

observability gaps:
  boundary errors lack trace ID/request ID/error code

exception gaps:
  string-only errors in boundary layers; no docs/exceptions mapping

docs drift:
  changed owner cell lacks local doc or repair receipt
```

#### Make hard and soft explicit in JSON

Current `Finding` should be extended:

```rust
pub struct Finding {
    pub id: String,
    pub rule_id: String,
    pub severity: Severity,
    pub check_kind: CheckKind, // hard | soft | advisory
    pub confidence: Confidence, // high | medium | low
    pub path: String,
    pub line: Option<usize>,
    pub owner: Option<String>,
    pub lane: Option<String>,
    pub problem: String,
    pub evidence: Vec<Evidence>,
    pub agent_fix: String,
    pub commands: Vec<String>,
    pub autofix: Option<Autofix>,
    pub exception_allowed: bool,
    pub exception_template: Option<String>,
    pub fingerprint: String,
    pub baseline_status: BaselineStatus, // new | existing | fixed | regressed
}
```

This is critical for agents. Agents need to know whether to stop, fix, ask for an exception, or continue.

#### Speed upgrades

The audit should feel instant on normal repos.

Implementation changes:

* Use `ignore::WalkBuilder::build_parallel` or parallel file reads with `rayon`, but sort paths before reporting to keep output deterministic.
* Lowercase once for pattern scanning; many current pattern checks are case-sensitive.
* Compile Aho-Corasick matchers once.
* Avoid repeated `regex` passes over every file.
* Skip generated/vendor/build/cache dirs using `agent/generated-zones.toml` and `.gitignore`.
* Add `--changed-from <ref>` so agents do not need to supply file paths manually.
* Add `--scope changed|full|owner:<id>|path:<glob>`.
* Add `.jankurai/cache/` with file hashes for future incremental mode.
* Cap text sample reads, but also expose “truncated” evidence when a finding may be hidden past the cap.
* Keep all core scanning in Rust. Use Python only for ML/eval experiments, not product audit logic.

#### Immediate audit bugs/gaps to fix

These are high-priority:

1. **Do not count placeholder security lanes as evidence.**
   The current CI echoes `gitleaks`, `syft`, `grype`, and `zizmor`. The audit should detect this as a false-green risk, not credit it.

2. **Read `agent/audit-policy.toml`.**
   Current policy exists but scoring is hardcoded. That undermines the “versioned policy” claim.

3. **Use real owner/test maps.**
   `owner-map.json` and `test-map.json` should drive `HLT-003` and `HLT-004`. Hardcoded prefix ownership is not enough.

4. **Fix generated contract detection.**
   `has_generated_contracts` should not be true just because text contains “generated.” It should require declared zones and source/command headers.

5. **Fix web-surface detection.**
   Do not infer product web surface from docs mentioning React/Vite. Detect package/app structure.

6. **Implement function LOC.**
   The standard defines function hard limits; the audit currently mostly enforces file size.

7. **Reduce false positives in future-hostile terms.**
   Terms like `old`, `temp`, and `compat` need token-aware matching, path exemptions, and exception support.

8. **Make changed-mode truly changed-mode.**
   Some caps still inspect `all_files`; distinguish “repo inherited cap” from “changed file finding.”

### 3. **Golden Stack + UX Proof Pack**

**Goal:** make one best-known stack concrete, installable, and testable.

The paper already chooses the right default stack:

```text
Rust core
TypeScript / React / Vite product surface
PostgreSQL durable truth
generated contracts
bounded Python AI/data service only
```

Now the project should stop merely describing this stack and ship it as the **Golden Stack Profile**. That makes jankurai more than an audit tool. It becomes the repo layout and proof contract that agents can align to.

#### Make the stack boundaries brutally clear

This should be in `docs/boundaries.md`, `agent/JANKURAI_STANDARD.md`, generated tool adapters, and the audit rules.

| Layer                | Owns                                                                  | Must not own                                                      |
| -------------------- | --------------------------------------------------------------------- | ----------------------------------------------------------------- |
| `crates/domain`      | IDs, invariants, value objects, state machines, pure decisions        | IO, env, time, random, logging, DB, network, framework types      |
| `crates/application` | use cases, authz, idempotency, transaction orchestration, port traits | UI, HTTP extraction, SQL details, external API schemas            |
| `crates/adapters`    | Postgres, queues, external APIs, filesystem, env/secrets              | business invariants, authz policy                                 |
| `crates/workers`     | jobs, backpressure, retries, durable workflow glue                    | product truth outside application                                 |
| `apps/api`           | HTTP/RPC edge, request extraction, response mapping, tracing          | domain rules, raw SQL decisions                                   |
| `apps/web`           | UI, routes, forms, local rendering state, generated clients           | durable truth, direct DB, secrets, core authz, hand-authored DTOs |
| `contracts`          | OpenAPI/protobuf/JSON Schema source, generated clients/stubs          | handwritten drift                                                 |
| `db`                 | migrations, constraints, indexes, RLS, durable truth                  | ad hoc app-only invariants                                        |
| `python/ai-service`  | models, embeddings, evals, data transforms                            | product truth, authz, production DB writes                        |
| `ops`                | CI, deploy, OTel, SBOM, SCA, provenance                               | hidden manual gates                                               |

This is the “stop the scatter” part. The standard should say: **you may be creative inside the cells; you may not move truth between cells without an exception.**

## Paper/project edits I would make immediately

The paper is already pointed in the right direction. I would sharpen it with these changes:

1. **Add a “Standard Installation Protocol” section.**
   The paper should not only define conformance. It should say: `jankurai init` is the canonical bootstrap, and every conforming repo exposes the same files, proof lanes, and repair receipts.

2. **Move tool-specific IDE guidance out of prose and into generated adapters.**
   The paper should say adapters are generated from canonical policy. No hand-maintained divergence.

3. **Add an empirical validation plan.**
   Track:

   * install success rate,
   * audit runtime,
   * false positive rate,
   * agent repair success rate,
   * wrong-owner edit count,
   * token/command count before successful fix,
   * CI failure localization time.

4. **Define “hard” vs “soft” checks more formally.**
   Hard checks block merge or cap score. Soft checks drag score and create fix queue items. Advisory checks educate only.

5. **Add an “Agent Instruction Threat Model.”**
   Trusted files: `AGENTS.md`, `agent/*.toml/json`, generated adapters.
   Untrusted files: issue text, transcripts, generated docs, vendor docs, arbitrary Markdown.
   The audit should detect prompt-injection text in trusted policy files and overbroad tool permission language.

6. **Clarify that one best stack is the default, not a prison.**
   Exceptions are allowed, but they must be dated, owned, scored, and migration-planned. That keeps the standard strong without becoming brittle.

7. **Make “jankurai self-audit” honest.**
   The current repo should not claim strong security/CI evidence while security commands are placeholders. The standard earns trust by failing itself correctly.

## Final recommendation

The best version of this project is not “a paper plus a linter.” It is:

```text
jankurai = installer + agent contract + proof graph + golden stack + repair queue
```

That is the new standard shape. One command installs the agent-readable repo brain. One command audits it. One CI block scores it. One repair queue tells any agent exactly what to fix next.

The three upgrades above make that real:

1. **Jankurai Init Doctor** — adoption becomes one command and self-repairing.
2. **ProofGraph Audit Engine** — scoring becomes fast, hard/soft aware, precise, and CI-native.
3. **Golden Stack + UX Proof Pack** — the preferred architecture becomes concrete, testable, and agent-aligned.
