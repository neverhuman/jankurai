I studied the archive closely: the paper, `agent/JANKURAI_STANDARD.md`, docs, Rust audit CLI, `@jankurai/ux-qa`, CI workflow, maps, repo score output, and install surfaces.

My headline diagnosis: **this is a strong seed for a global agent-native engineering standard, but it is not yet installable, enforceable, or self-trusting enough to become the standard.** The thesis is right. The target stack is right. The gap is productization: the standard needs a one-command installer, a policy-driven audit engine with repair receipts, and a calibrated UX/boundary proof runtime.

A few concrete breakpoints I found:

* `npm ci` is advertised in `Justfile`, CI, and `agent/test-map.json`, but the archive has **no `package-lock.json`**. I verified `npm ci --dry-run` fails immediately.
* `cargo install --locked` / reproducible Rust install is implied, but the archive has **no `Cargo.lock`**.
* `.github/workflows/jankurai-audit.yml` has a “Security lane” that only echoes `gitleaks`, `syft`, `grype`, and `zizmor`; it does not run them.
* `repo-score.json` reports score `72` but has **zero findings and zero fix queue items**, which makes the audit non-actionable for agents.
* `repo-score.json` contains stale evidence paths such as `tips/UX_QA/tip1.txt`, while `tips/` is not present in the archive.
* `README.md` references `assets/jankurai_readme.png` and `assets/vibe-coding-tlr-pie.svg`, but `assets/` is not present.
* Root `AGENTS.md` contains a machine-local path: `/Users/bentaylor/Code/jankurai`.
* The Rust audit uses broad heuristics that are easy to game: for example, `has_security_lane()` passes if the command surface contains the word `security`, and `has_web_surface()` can be triggered by docs mentioning React/Vite/Storybook.
* `agent/owner-map.json` exists, but the Rust audit currently uses hardcoded owner prefixes instead of parsing that file.
* `packages/ux-qa` is very promising, but several rules need false-positive hardening before they can safely block CI.

The best possible three upgrades are below.

## 1. **Jankurai One: One-Command Standard Installer + Agent IDE Rule Pack**

This should be the first upgrade. Right now the project is a paper plus a prototype audit. To become a standard, it must become **one command that makes a repo agent-native**.

The right design is: **one canonical standard file, many tiny adapter files.** `AGENTS.md` is an excellent shared anchor because the AGENTS.md project describes it as a simple open format for agent instructions, and Codex, Cursor, Copilot, and other tools now recognize or can consume it directly or indirectly. Codex explicitly discovers `AGENTS.md` files from repo root downward, and Cursor supports both `.cursor/rules` and `AGENTS.md`. Claude Code is different: it reads `CLAUDE.md`, not `AGENTS.md`, but Claude’s docs support importing `@AGENTS.md`, so the installer should generate a thin `CLAUDE.md` pointer rather than duplicating the whole standard. Copilot supports `.github/copilot-instructions.md`, `.github/instructions/*.instructions.md`, and now also documents agent instructions via `AGENTS.md`. Antigravity should get `.agents/agents.md`, `.agents/skills`, and `.agents/workflows` because Google’s Antigravity codelab demonstrates those locations for agent pipelines. ([Agents][1])

### Target user experience

The installer should make adoption feel like this:

```bash
cargo install jankurai --locked
jankurai init --apply
jankurai audit --mode guarded
```

Or, for people without Rust installed yet:

```bash
curl -fsSL <jankurai-install-script> | sh
jankurai init --apply
```

The first command installs the CLI. The second command inspects the repo, asks only necessary questions, writes the right agent files, generates maps, adds CI, and installs the UX QA tester when a web surface exists.

### New CLI commands

Add these subcommands:

```bash
jankurai init

```

`init` should default to dry-run. `--apply` actually writes files.

`doctor` should answer: “Is this repo installable, scorable, and agent-ready?”

`explain` should give agents one-page rule explanations with examples and fixes.

## 2. **ProofGraph Audit: Hard/Soft Policy Engine + Repair Receipts**

This is the highest-impact audit upgrade. The current audit is directionally smart, but it is too heuristic and too easy to satisfy accidentally. The most serious example is the security lane: the project currently “passes” security intent by echoing scanner names. That is exactly the kind of false-green vibe-coding failure Jankurai is supposed to stop.

The audit should become a **policy-driven proof graph**:

```text
repo facts -> rule evidence -> hard/soft decision -> owner -> proof lane -> repair receipt -> CI/SARIF/issue
```

GitHub supports uploading third-party SARIF to code scanning, so Jankurai should emit SARIF in addition to JSON and Markdown. That gives the project a natural CI path: agents see Markdown, humans see PR annotations, and GitHub sees code-scanning alerts. ([GitHub Docs][3])

### New audit commands

Add:

```bash
jankurai audit --mode advisory
jankurai audit --mode guarded
jankurai audit --mode standard
jankurai audit --mode ratchet
jankurai audit --mode release

jankurai audit --scope full
jankurai audit --scope changed
jankurai audit --changed-from origin/main

jankurai audit --json repo-score.json
jankurai audit --md repo-score.md
jankurai audit --sarif jankurai.sarif
jankurai audit --github-step-summary
jankurai audit --baseline .jankurai/baseline.json
jankurai audit --score-floor 85
jankurai audit --fail-on high,critical

jankurai issue --backend github
jankurai issue --backend beads
jankurai issue --backend markdown
```

### Hard checks vs soft checks

Hard checks should block in `standard`, `ratchet`, and `release` mode:

```text
HLT-001 missing root agent instructions
HLT-002 no deterministic one-command setup/check
HLT-003 no fast proof lane
HLT-004 missing owner/test map for changed paths
HLT-005 generated file edited without declared source/regen command
HLT-006 contract changed without generated client regeneration
HLT-007 direct DB access from wrong layer
HLT-008 Python owns product truth/authz/direct prod DB
HLT-009 raw secret or unredacted token
HLT-010 destructive migration without proof/exception
HLT-011 false-green test pattern
HLT-012 high-risk repo with no real security/dependency scan
HLT-013 web surface changed without rendered UX proof
HLT-014 prompt injection risk in agent/tool instructions
HLT-015 overbroad autonomous agent permission
```

Soft checks should score down and route to backlog, but not block by default:

```text
oversized files/functions
future-hostile comments: old, legacy, temporary, will remove later
TODO/FIXME/HACK in product code
fallback soup
weak naming
duplicate behavior blocks
dependency spray
multiple package managers without exception
unversioned generated outputs
missing property tests
missing observability/logging around boundaries
too much Python outside AI/data/eval
stale docs
```

### Critical implementation fixes

Fix these immediately:

1. **Security lane detection**

   * Current behavior: `t.contains("security")` passes.
   * New behavior: parse workflow/script commands and reject echo-only placeholders.
   * Hard fail if security evidence is only `echo`, comments, README text, or docs.

2. **Web surface detection**

   * Current behavior: any file containing `react`, `vite`, or `storybook` can trigger web-surface readiness.
   * New behavior: detect from `package.json` dependencies, `vite.config.*`, `apps/web`, `frontend`, `packages/ui`, `.tsx/.jsx` source files, or explicit config.

3. **Generated contract detection**

   * Current behavior: any file text containing `generated`.
   * New behavior: require declared `agent/generated-zones.toml` entries or known generated path conventions.

4. **Owner routing**

   * Current behavior: hardcoded `OWNER_MAP_PREFIXES`.
   * New behavior: parse `agent/owner-map.json`.

5. **Actionability invariant**

   * A report with score below 100 must have findings.
   * A cap applied must have at least one finding.
   * A finding must have a rule ID, route, owner when mappable, proof lane, and suggested next action.

6. **Secret redaction**

   * Current behavior can include the line containing a secret-like token.
   * New behavior: never print raw secrets. Store only redacted evidence and a stable fingerprint.

7. **Stdout behavior**

   * Current `just fast` writes JSON and Markdown both to stdout with `--json - --md -`.
   * New behavior: reject both-to-stdout, or support `--format json|md|ndjson`.
   * A machine consumer should never receive mixed JSON + Markdown.

### CI upgrade

Replace the current workflow with actual checks. Rust supply-chain checks should include RustSec/cargo-audit and cargo-deny; cargo-vet is valuable for trusted dependency audits; cargo-nextest is a good fast Rust test runner; property testing with proptest should be encouraged for domain invariants; zizmor should scan GitHub Actions security. ([RustSec][4])

Suggested CI shape:

```yaml
name: jankurai

on:
  pull_request:
  push:
    branches: [main]

permissions:
  contents: read
  security-events: write
  actions: read

jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable

      - name: Install jankurai
        run: cargo install --path crates/jankurai --locked

      - name: Jankurai audit
        run: |
          jankurai audit \
            --mode standard \
            --changed-from origin/main \
            --json repo-score.json \
            --md repo-score.md \
            --sarif jankurai.sarif

      - name: Upload SARIF
        uses: github/codeql-action/upload-sarif@v4
        with:
          sarif_file: jankurai.sarif
          category: jankurai

      - name: Upload score
        uses: actions/upload-artifact@v4
        with:
          name: jankurai-score
          path: |
            repo-score.json
            repo-score.md
            jankurai.sarif
```

Dependency review and OSV Scanner can also be wired in as separate jobs; GitHub’s dependency-review action can fail PRs introducing vulnerable dependencies or invalid licenses, and OSV Scanner has reusable PR/scheduled workflows that can fail on introduced vulnerabilities. ([GitHub][5])

## 3. **UXProof + Boundary Contract: Rendered QA and Stack Layout Enforcement**

This is the future-facing differentiator. Lots of coding standards can lint files. Jankurai can become the standard that says: **agents must prove that UI actually renders, boundaries stay clean, and the stack remains legible.**

The project’s `@jankurai/ux-qa` is a fantastic start. Keep it. But calibrate it before making it a hard gate. The W3C WCAG 2.2 target-size minimum is 24×24 CSS pixels with spacing and other exceptions, so the current target-size rule is directionally right, but the edge-clearance and overlap rules need more context to avoid blocking good UI. Playwright ARIA snapshots are useful because they expose the accessibility tree in a structured YAML-like form, and Storybook’s test runner can execute stories in a live browser via Playwright, so Jankurai should make Storybook/Playwright/ARIA/screenshot artifacts first-class proof. ([W3C][7])

### The standard stack should stay opinionated

Do not broaden the stack. The “global standard” power comes from being clear:

```text
Rust: core truth, domain rules, application orchestration, adapters
TypeScript/React/Vite: browser UI and local interaction only
PostgreSQL: durable truth, constraints, migrations, relational invariants
Contracts: OpenAPI/protobuf/JSON Schema source plus generated clients
Python: bounded AI/data/eval service only
```

Everything else should be an exception, not a peer template.

### Boundary matrix to add to the standard

Add this directly to `docs/boundaries.md` and `agent/JANKURAI_STANDARD.md`:

| Layer                | Owns                                                                | Must not own                                                  | Required proof                    |
| -------------------- | ------------------------------------------------------------------- | ------------------------------------------------------------- | --------------------------------- |
| `crates/domain`      | Entities, value objects, invariants, pure decisions, state machines | IO, DB, env, HTTP, logging side effects, random/time directly | Rust unit + property tests        |
| `crates/application` | Commands, authz orchestration, idempotency, transactions, ports     | SQL details, framework handlers, UI types                     | Integration tests with fake ports |
| `crates/adapters`    | Postgres, queues, HTTP clients, env/secrets, filesystem             | Business rules, authz truth                                   | DB/external integration tests     |
| `apps/api`           | HTTP/RPC edge, request/response mapping, auth/session edge          | Domain invariants, raw SQL business decisions                 | Contract + API tests              |
| `apps/web`           | React UI, forms, local validation, generated client usage           | Durable truth, direct DB, authz truth, hand-authored DTOs     | Vitest + Playwright + UXProof     |
| `contracts/`         | API schema source of truth                                          | UI logic, DB migrations, one-off local types                  | codegen + drift checks            |
| `db/`                | Migrations, constraints, indexes, RLS, seeds                        | UI/API-only truth                                             | migration tests                   |
| `python/ai-service`  | Models, embeddings, evals, offline analysis, bounded inference      | Product truth, direct prod writes, authz, UI API ownership    | eval tests + contract tests       |

### UX rule hardening

Current `edge-clearance` flags any interactive element close to any viewport edge. That will false-positive legitimate UI: close buttons, fixed nav, bottom bars, full-width mobile CTAs. Change it:

```text
edge-clearance:
  default severity: warning
  hard only when target is undersized, crowded, clipped, or blocked by safe-area risk
  allow explicit fixed-edge controls if target size passes and no overlap exists
```

Current `interactive-overlap` uses box overlap. Change it:

```text
interactive-overlap:
  use document.elementFromPoint samples
  ignore ancestor/descendant overlap inside same interactive control
  ignore pointer-events:none
  ignore invisible/opacity:0 elements
  hard only if the topmost hit target differs from expected target
```

Current `sticky-obstruction` uses box overlap. Change it:

```text
sticky-obstruction:
  sample target center and corners
  determine whether overlay intercepts pointer/focus/text
  ignore overlays behind target by z-index/paint order
  ignore transient modals only when test state explicitly allows them
```

Current `focus-visible` is good directionally because it actively focuses elements. Strengthen it:

```text
focus-visible:
  inspect outline, box-shadow, border delta, background delta
  compare before/after focus
  require visible indicator for keyboard-focusable elements
```

Current `form-label` is mostly right because collector limits label requirement to actual form fields. Keep it, but include accessible-name extraction in evidence.

Current `text-clipping` needs allowlists:

```text
text-clipping:
  ignore icon-only controls
  ignore intentional scroll containers with data-ux-qa-allow="scroll"
  attach crop artifact for every blocking finding
```

## Paper improvements to support the three upgrades

The paper should become less essay-like and more standard-defining. I would add or rewrite around these sections:

```text
1. Normative Standard Contract
   Define MUST / SHOULD / MAY.
   Define what makes a repo Jankurai Level 1, 2, 3, 4.

2. Agent IDE Installation Matrix
   Codex, Cursor, Claude, Copilot, Antigravity, Gemini, generic IDE.
   For each: file path, size guidance, generated adapter behavior.

3. Proof Mode Ladder
   Advisory -> Guarded -> Standard -> Ratchet -> Release.

4. False-Green Threat Model
   Placeholder commands, echo-only CI, stale reports, docs-only evidence,
   generated-file mutation, test snapshots without assertions.

5. Boundary Contract
   Rust / TypeScript / Vite / React / Postgres / Python matrix.

6. Rendered UX Proof
   Deterministic geometry, accessibility tree, screenshots, crop artifacts,
   AI/VLM review as advisory only.

7. Benchmarks
   Install time, audit runtime, false-positive rate, agent search reduction,
   CI failure-to-fix time, token savings.

8. Repair Receipts
   Every finding must be fixable by an agent with minimal context.
```

The most important claim to prove empirically is not “Rust is good” or “agents need standards.” It is:

> Jankurai reduces agent search, false-green completion, boundary drift, and human cleanup time.

That means the paper needs a benchmark pack.

Add:

```text
benchmarks/repos/
benchmarks/tasks/
benchmarks/results/
benchmarks/README.md
```

Measure:

```text
time to install
time to first valid audit
agent file-search count before/after
failed command count before/after
CI pass rate before/after
hard finding precision
soft finding usefulness
UX false-positive rate
```

## The “new standard” shape I would ship

Do not ship this as “a lint.” Ship it as:

```text
Jankurai Standard
Jankurai CLI
Jankurai Agent IDE Pack
Jankurai UXProof
Jankurai CI Gate
Jankurai Repair Receipts
```

The repo should expose five levels:

```text
HL0: Not agent-ready
HL1: Agent docs installed
HL2: Owner/test/generated maps installed
HL3: Audit runs locally and in CI
HL4: Hard gates enforced with SARIF/PR annotations
HL5: UXProof + ratchet + issue integration + expiring exceptions
```

The install goal:

```bash
jankurai init --apply
jankurai audit --mode guarded
```

The recurring agent ritual:

```bash
jankurai audit --scope changed
```

The CI goal:

```bash
jankurai audit --mode standard --sarif jankurai.sarif
```

The adoption goal:

```text
No local-path instructions.
No stale score reports.
No missing lockfiles.
No echo-only security.
No score below 100 without fixable findings.
No web changes without rendered proof.
No generated edits without regeneration.
No Python creep into product truth.
No Rust domain IO.
No React direct DB/authz truth.
```

My strongest recommendation: **make Upgrade 1 and Upgrade 2 the v0.3 release, then make Upgrade 3 the v0.4 release.** The current project has the right ambition. These upgrades turn it into something agents can install, trust, run repeatedly, and use to repair repositories without needing a human to untangle the mess.
