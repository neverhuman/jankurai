I studied the archive closely. My read: **the thesis is strong, and the Rust audit + UX QA package are a very good nucleus, but the project is not yet “standard-ready” because the install path, evidence model, and CI proof loop are not hard enough.** The biggest opportunity is to turn Jankurai from a paper + repo scanner into an **agent passport**: one command installs the standard, writes the right agent-help files, configures CI, creates actionable repair queues, and gives every agent IDE the same repo contract.

A few concrete breakpoints I found in the provided archive:

| Area             | Issue                                                                                                                                                                                                        | Why it matters                                                                                                                              |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------- |
| Installability   | No `Cargo.lock` or `package-lock.json` in the source archive. `npm ci` fails immediately; `cargo install --locked`/locked install story is inconsistent.                                                     | The first-run experience must never fail on missing lockfiles.                                                                              |
| Paper build      | `latexmk` fails because `paper/tex/frontmatter.tex` references missing `assets/jankurai_header.png`. README also references missing assets.                                                                 | A “standard” cannot ship with broken reproducible docs.                                                                                     |
| Release hygiene  | Archive contains many AppleDouble `._*` files.                                                                                                                                                               | Signals weak packaging; can pollute audits and confuse agents.                                                                              |
| CI security lane | Workflow currently echoes tools like `gitleaks`, `syft`, `grype`, `zizmor` rather than running them.                                                                                                         | The audit can mistake “tool name mentioned” for proof. This is exactly the kind of vibe-coding false-green the project is meant to prevent. |
| Audit evidence   | Several checks scan all text, so docs or the audit source itself can satisfy operational requirements. Example: UX/web evidence can be satisfied by mentioning React/Vite/Storybook in docs or scanner code. | The audit needs scoped evidence: docs are not proof, real commands/artifacts are proof.                                                     |
| Reporting        | JSON/Markdown reports exist, but they do not yet produce a stable repair queue with rule IDs, exact commands, owner/lane, exception template, SARIF, ratchet, or issue sync.                                 | Agents need the report to be directly executable.                                                                                           |
| Productization   | There is no `jankurai init` or installer that writes Codex/Cursor/Claude/Gemini/Copilot/Antigravity adapter files.                                                                                          | Adoption dies unless setup is one-command.                                                                                                  |

Also, I could not run Cargo tests in this environment because `cargo`/`rustc` are not installed here. I did run the available Node/paper checks: `npm ci` failed due missing lockfile, and `latexmk` failed due the missing header asset.

Below are the **best 3 upgrades** I would make. These are intentionally large enough to change the project’s trajectory, but bounded enough that an agent could implement them.

# Upgrade 1 — **Agent Passport Installer**

## Goal

Make Jankurai installable in one command:

```bash
jankurai init . --yes --tools all --ci github --mode advisory
jankurai audit . --md repo-score.md --json repo-score.json
```

This should create the repo’s **Agent Passport**: the canonical contract that every coding agent and IDE reads before touching code.

This aligns with where agent tooling is going. Codex reads `AGENTS.md` files globally and per project; Gemini uses hierarchical `GEMINI.md`; GitHub Copilot supports repo/path instructions plus `AGENTS.md`, `CLAUDE.md`, and `GEMINI.md`; and SKILL-style bundles are becoming a portable primitive for agent capabilities. ([OpenAI Developers][1])

## What it should do

Add a new CLI subcommand:

```bash
jankurai init [repo]
  --yes
  --dry-run
  --force
  --mode advisory|standard|strict
  --stack rust-ts-vite-react-postgres
  --tools codex,cursor,claude,gemini,copilot,antigravity,all
  --ci github|none
  --issue-tracker beads|github|none
  --ux-qa
  --apply-moves
```

Default behavior should be safe:

```bash
jankurai init .
```

should inspect the repo, produce a plan, and ask only when ambiguity matters. In noninteractive mode, it should create the minimum safe files and mark unresolved items as TODOs.

```md
Before final response:
- Run `jankurai audit . --changed`.
```

## Best-known stack boundary file

Add:

```text

```

With this canonical table:

| Layer                     | Owns                                                                                                            | Must not own                                                                           |
| ------------------------- | --------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| Rust domain crates        | Durable product behavior, invariants, authorization policy, typed errors, idempotency, command/query boundaries | Browser UI, handwritten generated DTOs, direct framework leakage into domain           |
| Rust application/adapters | HTTP/RPC handlers, persistence adapters, workers, external service clients, transaction boundaries              | Business rules hidden in handlers, direct DB access from UI, untyped JSON soup         |
| PostgreSQL                | Durable state, constraints, indexes, migrations, referential integrity, RLS where applicable                    | App-only invariants that should be constrained durably                                 |
| TypeScript/React/Vite     | Browser UI, local interaction, rendering, accessibility, form state, generated API clients                      | Durable product truth, authorization, direct DB access, handwritten backend DTOs       |
| Contracts                 | API schemas, generated clients, generated bindings                                                              | Manual edits in generated outputs                                                      |
| Python                    | ML/data/eval/model-serving islands only                                                                         | Product truth, authorization, primary business workflows, direct production DB control |

This is the anti-scatter rule. The standard should say: **Rust and PostgreSQL own truth; TypeScript owns presentation; Python owns ML/data islands only; contracts connect them.**

# Upgrade 2 — **Proof Graph Auditor**

## Goal

Replace broad string scanning with a scoped, graph-backed evidence model.

Right now, the audit is promising but too easy to fool. It can treat documentation mentions as operational evidence. The new rule should be:

> **Docs can explain proof. Only executed commands, configs, lockfiles, generated reports, source topology, and declared maps can satisfy proof.**

This is the core anti-vibe-coding upgrade.

## New audit model

Add four evidence scopes:

```rust
pub enum EvidenceScope {
    RuntimeSource,
    OperationalConfig,
    GeneratedArtifact,
    Documentation,
}
```

Then make every rule declare which scopes can satisfy it.

Examples:

| Rule                         | Valid evidence                                                                           | Invalid evidence                        |
| ---------------------------- | ---------------------------------------------------------------------------------------- | --------------------------------------- |
| UX QA exists                 | `ux-qa.config.*`, Playwright report, CI step that runs UX QA                             | README saying “we use UX QA”            |
| Security lane exists         | Workflow step actually invoking `gitleaks`, `cargo audit`, `osv-scanner`, `zizmor`, etc. | `echo "run gitleaks"`                   |
| Web surface exists           | `package.json` deps, `vite.config.*`, `src/**/*.tsx`                                     | Paper mentioning React                  |
| Rust proof exists            | `Cargo.lock`, `cargo nextest`, Rust tests, CI step                                       | Docs saying “run cargo test”            |
| Generated contract protected | declared generated zone + file hashes or generation commands                             | Folder name containing `generated` only |

## Hard checks vs soft checks

Make policy explicit:

```toml
# agent/audit-policy.toml

mode = "standard"
fail_below = 85
fail_on_hard_findings = true
ratchet = true

[hard]
secrets = true
generated_mutation = true
missing_agent_passport = true
missing_fast_lane = true
missing_owner_map = true
missing_test_map = true
missing_lockfile = true
python_product_truth = true
db_wrong_layer = true
destructive_migration_without_safety = true
prompt_injection_tool_policy = true
security_lane_missing = true

[soft]
naming_fog = true
large_file_warning = true
todo_density = true
fallback_soup = true
weak_docs = true
missing_storybook_for_noncritical_ui = true
```

Hard checks should fail `standard` and `release` mode. Soft checks should affect score, ratchet, and repair queue, but not necessarily block early adoption.

## New CLI modes

```bash
jankurai audit .
jankurai audit . --changed
jankurai audit . --changed-from origin/main
jankurai audit . --mode advisory
jankurai audit . --mode standard
jankurai audit . --mode ratchet
jankurai audit . --mode release
jankurai audit . --sarif jankurai.sarif
jankurai audit . --github-step-summary
jankurai audit . --beads
jankurai audit . --github-issues
```

Modes:

| Mode       | Behavior                                                                    |
| ---------- | --------------------------------------------------------------------------- |
| `advisory` | Never fails, writes report and queue. Best for first adoption.              |
| `standard` | Fails on hard findings or score below threshold.                            |
| `ratchet`  | Allows existing debt but blocks regressions.                                |
| `release`  | Strictest mode; requires clean proof lanes and no unresolved hard findings. |

## External Rust/security proof tools

The standard should recommend these tools as optional lanes, not as required dependencies for the Jankurai binary:

| Tool                           | Why                                                                                                  |
| ------------------------------ | ---------------------------------------------------------------------------------------------------- |
| `cargo-nextest`                | Faster Rust test runner with test selection and retry/slow-test controls. ([Nexte][4])               |
| `cargo-audit`                  | Checks `Cargo.lock` against RustSec advisories. ([Rust Blog][5])                                     |
| `cargo-deny`                   | Policy checks for advisories, licenses, sources, and dependency policy. ([Crates.io][6])             |
| `cargo-vet`                    | Tracks audited dependency trust and reusable audits. ([Mozilla GitHub][7])                           |
| `cargo-machete`                | Fast unused dependency detection, with explicit allowance for false-positive metadata. ([GitHub][8]) |
| `cargo-geiger`                 | Reports unsafe Rust usage statistics. ([GitHub][9])                                                  |
| `cargo-semver-checks`          | Public API drift for libraries.                                                                      |
| `zizmor` + `actionlint`        | GitHub Actions security/linting.                                                                     |
| `gitleaks` or `trufflehog`     | Secret scanning.                                                                                     |
| `osv-scanner`, `syft`, `grype` | Vulnerability/SBOM/container/dependency lanes.                                                       |

The audit should verify either:

1. the tool was actually run in CI, or
2. a current artifact exists, or
3. an explicit exception exists.

It should never accept a string mention as proof.

## UX hard/soft rules

Hard UX checks for critical routes:

```text
No invisible focused element
No unlabeled form controls
No critical button below minimum target size
No horizontal overflow at mobile viewport
No sticky obstruction of primary action
No clipped primary text
No overlapping interactive targets
```

Soft UX checks:

```text
Noncritical target-size warnings
Minor wrapping warnings
Nested scroll warnings
Z-index token warnings
Low-confidence overlap warnings
```

# Paper upgrades to match the product

The paper is compelling, but it should become more install-first and less manifesto-only.

I would add or rewrite these sections:

## 1. “The Jankurai Contract in 90 Seconds”

Put this near the front:

```md
Install:

cargo install jankurai
jankurai init . --tools all --ci github --ux-qa
jankurai audit . --mode advisory

This creates:
- AGENTS.md
- agent/JANKURAI_STANDARD.md
- owner/test/proof maps
- IDE-specific agent adapters
- CI audit workflow
- optional UX QA config
- repair queue
```

## 2. “Strings Are Not Evidence”

Make this a named principle.

```md
A repository does not pass because it says it has tests.
It passes because the proof lane exists, runs, and emits artifacts.
Documentation is guidance, not proof.
```

This directly addresses the current scanner weakness.

## 3. “Default Stack, Not Universal Dogma”

Keep the strong recommendation:

```text
Rust + TypeScript/React/Vite + PostgreSQL + generated contracts + bounded Python
```

But define it as:

> the best-known default for agent-optimized product repositories, not a universal mandate.

Then say alternate stacks can pass if they satisfy the same control-plane requirements:

```text
agent passport
owner/test map
proof lanes
generated-zone protection
security lane
UX lane when UI exists
ratchet
issue queue
```

That will improve adoption because teams with unusual constraints will not feel rejected immediately.

## 4. “Agent IDE Support Matrix”

Add a table:

| Tool        | Primary file                        | Scoped/project files                     | Skill/workflow support      |
| ----------- | ----------------------------------- | ---------------------------------------- | --------------------------- |
| Codex       | `AGENTS.md`                         | nested `AGENTS.md`                       | `.agents/skills/*/SKILL.md` |
| Cursor      | `.cursor/rules/*.mdc` + `AGENTS.md` | rule-specific files                      | generated Jankurai rules   |
| Claude      | `CLAUDE.md`                         | `.claude/*`                              | `.claude/skills/*/SKILL.md` |
| Gemini      | `GEMINI.md`                         | hierarchical context files               | generated Jankurai context |
| Copilot     | `.github/copilot-instructions.md`   | `.github/instructions/*.instructions.md` | `.github/skills/*/SKILL.md` |
| Antigravity | rules/workflows adapter             | project rules                            | skills/workflows adapter    |

## 5. “Empirical Calibration Plan”

The paper currently has a strong rubric. To become a global standard, add a benchmark plan:

```text
Run Jankurai on 100 repos:
- 25 Rust/TS web apps
- 25 Python-heavy apps
- 25 JS/TS apps
- 25 mixed enterprise repos

Measure:
- false positive rate
- false negative rate
- audit duration
- agent repair success rate
- PR cycle time
- number of repeated agent mistakes
- score delta after one repair loop
```

Then publish:

```text
docs/benchmarks/
docs/calibration/
```

This turns the paper from “opinionated” into “ratcheting standard.”

# Minimal implementation order for an agent

Hand this to an implementation agent:

```md
Implement Jankurai v0.3 around three deliverables:

1. Agent Passport Installer
   - Add `jankurai init`.
   - Generate root AGENTS.md, agent/*.toml/json, tool adapters, GitHub workflow, UX QA config.
   - Must be idempotent and support --dry-run.

2. Proof Graph Auditor
   - Add evidence scopes: RuntimeSource, OperationalConfig, GeneratedArtifact, Documentation.
   - Refactor rules so docs cannot satisfy operational proof.
   - Add hard/soft policy modes, SARIF, repair_queue.jsonl, and ratchet baseline.
   - Add regression fixtures for docs-only false positives and echo-security false positives.

3. Proofpack CI + UX QA Runtime
   - Add action.yml.
   - Add ux-qa config/discovery/reporters.
   - Fix release packaging, lockfiles, paper asset fallback.
   - Add release archive smoke test.

Acceptance:
- `npm ci` works.
- paper builds.
- no `._*` in release archive.
- `jankurai init --yes --tools all --ci github --ux-qa` works in temp repo.
- `jankurai audit --mode advisory` produces JSON, Markdown, SARIF, and repair queue.
- docs-only claims do not satisfy proof.
- echo commands do not satisfy proof.
```

My strongest recommendation: **make Upgrade 1 the headline, Upgrade 2 the trust engine, and Upgrade 3 the adoption engine.** The paper can stay ambitious, but the project wins when an agent can enter any repo, run `jankurai init`, read the generated passport, run `jankurai audit --changed`, and know exactly what to fix next.
