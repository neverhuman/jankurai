I studied the bundle closely. My read: **the paper has the right thesis and the standard is powerful**, but the project currently needs a sharper adoption product, a harder-to-fool audit engine, and a continuous proof/CI loop. Right now the standard says “agent-native,” but the shipped repo still has friction points that agents and humans will trip over.

The best path is not to broaden the project. It is to make it **much more installable, much more evidence-based, and much more actionable**.

## What I found that matters most

The core doctrine is strong: **Rust core + TypeScript/React/Vite product surface + PostgreSQL durable truth + generated contracts + bounded Python only for AI/data work**. That is the right kind of narrow standard. The paper and docs are already clear that Jankurai is trying to remove “human scatter” by giving agents stable boundaries, proof lanes, generated contracts, ownership maps, and fast feedback.

But the current bundle has several adoption blockers:

1. **Install is not yet “one command.”**
   The README points users toward manual `curl` setup for `agent/JANKURAI_STANDARD.md`, and the repo lacks the lockfiles needed for clean deterministic install. `npm ci` fails because `package-lock.json` is missing. `cargo test` could not complete cleanly in this environment because dependencies had to be fetched and no `Cargo.lock`/vendor state was present. That directly weakens the “easy global standard” goal.

2. **The source package contains 103 AppleDouble `._*` metadata files.**
   This is a release-quality problem. Agents may scan them, humans may wonder what they are, and audits may count them incorrectly. Add packaging hygiene immediately.

3. **The current CI security lane is placeholder-shaped.**
   The workflow echoes tools like `gitleaks`, `syft`, `grype`, and `zizmor`, but does not actually run them. The current Rust audit appears to give credit based on string presence, so a workflow can look safe while doing no real security work. That is the most important “false green” risk.

4. **The auditor under-checks the paper’s own standard.**
   The docs require owner maps, test maps, generated zones, repo scores, agent files, proof lanes, CI checks, and stack boundaries. The current audit catches some vibe-coding smells, but it does not yet enforce many of the strongest parts: ownerless paths, unmapped proof, real import boundaries, function LOC, generated-zone drift, real CI command execution, package-lock/Cargo.lock consistency, or actionable issue export.

5. **There is an internal path leak.**
   Root `AGENTS.md` includes an absolute local path: `/Users/bentaylor/Code/jankurai`. That should be removed before public standardization.

6. **The `agent/repo-score.json` expectation conflicts with the actual root `repo-score.json`.**
   The standard says `agent/repo-score.json`; the repo has `repo-score.json`. Pick one canonical location. I recommend `agent/repo-score.json` and `agent/repo-score.md`, because agents naturally look in `agent/`.

7. **Tool-specific IDE support is documented but not generated.**
   The standard mentions Codex, Cursor, Claude, Gemini, Antigravity, and Copilot, but install does not scaffold those files. This is where Jankurai can become sticky: it should generate the correct agent-helping files for each environment.

8. **UX QA is promising but not yet product-grade.**
   The Playwright geometry checker is a great start. It needs route config, app-start support, axe integration, screenshots/traces/ARIA snapshots, baseline handling, and CI output formats. Playwright officially supports ARIA snapshots, and its accessibility docs recommend axe-style scans, so Jankurai can make UX proof first-class rather than optional. ([Playwright][1])

The agent IDE landscape is favorable for this standard. Codex officially reads `AGENTS.md`; Cursor supports project rules and also documents `AGENTS.md`; Claude Code officially supports `CLAUDE.md` locations and project settings; Antigravity is officially an agentic IDE with editor/terminal/browser surfaces and a Skills mechanism. Jankurai should exploit that by generating tool-specific adapters from one canonical standard file. ([OpenAI Developers][2])

# The best possible 3 upgrades

## Upgrade 1 — **Jankurai Launchpad**

**Thesis:** Jankurai should not be a paper plus an audit. It should be a one-command project alignment system.

This is the adoption unlock. The current setup asks humans to understand too much. Jankurai should instead inspect the repo, propose a minimal migration plan, write the expected agent files, install the audit/UX tooling, and produce the first score.

### What it should do

Add a new command:

```bash
jankurai init
```

With modes:

```bash
jankurai init --profile standard --agents all --install-ux-qa --apply
jankurai init --profile audit-only --apply
jankurai init --profile migrate --apply
jankurai init --dry-run
jankurai doctor
```

The command should detect:

```text
Rust: Cargo.toml, workspace layout, crates, lockfile
Node: package.json, npm/pnpm/yarn/bun, Vite, React, TypeScript, lockfile
Python: pyproject.toml, uv, poetry, requirements, notebooks, AI/data roots
DB: db/migrations, Prisma, SQLx, Diesel, Drizzle, Postgres hints
CI: GitHub Actions, GitLab, Buildkite, CircleCI
Agent IDE files: AGENTS.md, CLAUDE.md, GEMINI.md, .cursor/rules, .github/copilot-instructions.md
Generated contracts: openapi, protobuf, graphql, typed clients, bindings
```

Then it should generate a patch plan:

```text
Jankurai found:
- Rust workspace: yes
- TypeScript/Vite/React: yes
- PostgreSQL migrations: likely
- Python: docs/scripts only
- Existing agent instructions: AGENTS.md
- Missing: agent/JANKURAI_STANDARD.md
- Missing: agent/owner-map.json
- Missing: agent/test-map.json
- Missing: agent/generated-zones.toml
- Missing: .cursor/rules/jankurai.mdc
- Missing: CLAUDE.md
- Missing: .github/copilot-instructions.md
- Missing: jankurai CI workflow
- Missing: UX QA route config

Recommended changes:
1. Add canonical Jankurai agent standard.
2. Add IDE adapter files pointing to canonical standard.
3. Add owner/test/generated-zone maps.
4. Add CI audit workflow.
5. Add UX QA package config for Vite app.
```

It should ask only when ambiguity matters:

```text
Detected both apps/web and frontend/. Which is the primary web app?
Detected Python under scripts/ and services/ml/. Is services/ml/ the bounded AI service?
Detected DB migrations under migrations/ and db/migrations/. Which is canonical?
```

No overwrite without showing a diff. Running twice should produce zero changes.

## Upgrade 2 — **Evidence-First Audit Engine**

**Thesis:** The audit must stop rewarding words and start requiring receipts.

This is the trust unlock. Jankurai cannot become the global agent standard if a repo can pass by mentioning tools in docs or echoing them in CI. The audit needs to distinguish:

```text
Claim: "we run gitleaks"
Evidence: actual CI step or local command that executes gitleaks and fails on findings
```

### Replace string scoring with evidence scoring

Every rule result should include:

```json
{
  "rule_id": "HLT-SEC-001",
  "severity": "hard",
  "status": "fail",
  "title": "Secret scanning is mentioned but not executed",
  "evidence": [
    {
      "path": ".github/workflows/jankurai-audit.yml",
      "line": 42,
      "kind": "ci_run_step",
      "observed": "echo gitleaks"
    }
  ],
  "why_it_matters": "Echoed tools create false-green security lanes.",
  "fix": {
    "summary": "Run gitleaks detect in CI.",
    "command": "gitleaks detect --source . --redact --exit-code 1"
  },
  "agent_hint": "Patch the workflow. Do not satisfy this rule by adding text mentions."
}
```

The report should become an agent repair queue, not just a score.

### Hard checks vs soft checks

Use two buckets.

#### Hard checks

These should cap or fail CI:

```text
Missing AGENTS.md or canonical agent standard
Missing lockfile for detected package manager
CI security lane is absent or placeholder-only
Jankurai audit not run in CI
Generated files modified outside generated-zone rules
Changed file has no owner-map match
Changed file has no test-map/proof-lane match
Direct DB/auth/secrets code in React/Vite layer
Network/SQL/env/filesystem code in Rust domain layer
Python product-truth ownership outside bounded AI/data service
Secret-like content in source
Test skip/only/fake assertion in active tests
Destructive migration without expand/contract or rollback note
Overbroad agent permission instructions
Contract drift between generated client/server artifacts
```

#### Soft checks

These should warn or reduce score:

```text

Function near limit
Weak names: util, helper, misc, manager, temp, new2
Missing Storybook or rendered UX evidence
No cargo-vet
No SLSA provenance
No SBOM
No issue export integration
No baseline/ratchet policy
Too much Python relative to non-AI product code
Docs missing local decision record
Overlong agent instructions
No observability proof
No benchmark/performance receipt
```

### Most important detector upgrades

#### 1. CI command execution detector

Current risk: echoing `gitleaks` looks like a security lane.

Required behavior:

```text
PASS:
run: gitleaks detect --source . --redact --exit-code 1

FAIL:
run: echo "gitleaks"
run: echo "TODO: add gitleaks"
comment: gitleaks
docs mention: gitleaks
```

Parse GitHub workflow YAML. Inspect `jobs.*.steps[*].run` and `jobs.*.steps[*].uses`.

For shell `run` blocks:

```text
- Split commands.
- Ignore comments.
- Ignore echo/printf/cat/heredoc-only mentions.
- Require executable command position or known action.
```

Tools worth real support:

```text
cargo audit
cargo deny check
cargo vet
gitleaks detect
zizmor
syft
grype
slsa-github-generator
cargo nextest
```

`cargo-audit` audits Rust lockfiles for known vulnerabilities via RustSec, and `cargo-deny` can audit vulnerabilities, licenses, sources, and duplicate packages. `cargo-vet` adds third-party dependency audit evidence. `cargo-nextest` is a faster Rust test runner and is often a better default proof lane than raw `cargo test`. ([RustSec][4])

For broader supply-chain/security lanes, Gitleaks detects secrets, zizmor statically analyzes GitHub Actions, Syft generates SBOMs, Grype scans images/filesystems/SBOMs, and SLSA’s GitHub generator supports provenance for GitHub Actions builds. ([GitHub][5])

### Important implementation fix

The current CLI appears to manually special-case `versions` and `audit` and then parse args after skipping the first two argv values. That is brittle because `clap::Parser::parse_from` expects an argv-like sequence including a binary name. Replace this with real clap subcommands.

Target shape:

```rust
#[derive(clap::Parser)]
struct Cli {
```

## Upgrade 3 — **Continuous Proof Pack**

**Thesis:** Jankurai should not only score repos. It should continuously turn audit findings into repairable agent tasks.

This is the workflow unlock. The project should become the default loop for agents:

```text
edit → targeted audit → fix queue → proof lane → full audit → CI report → issue graph
```

### Fix the current UX QA collector risk

The current collector merges element boxes and states by index after separate browser evaluations. That can mismatch if focus/visibility changes alter node order. Collect everything in a single browser evaluation or assign stable IDs before collecting.

### Add agent periodic audit behavior

The generated standard should explicitly teach agents:

```markdown
## Agent audit loop

Before editing:
1. Identify touched owner/layer.
2. Read the matching proof lane in agent/test-map.json.

After a meaningful edit:
1. Run targeted audit:
   jankurai audit . --changed <files> --json -
2. Run the mapped fast proof command.
3. Fix hard failures before continuing.

Before final response or PR:
1. Run full audit:
   jankurai audit . --json agent/repo-score.json --md agent/repo-score.md
2. Report remaining hard failures, soft findings, and skipped proof.
```

This is one of the highest-value additions because it changes Jankurai from a CI-only score into an agent operating loop.

# The sharpened standard: what Jankurai should become

Jankurai should be framed as:

> **The agent-native engineering standard that gives every repo a canonical stack, canonical agent instructions, executable proof lanes, and evidence-backed audit reports.**

The best-known-stack should stay narrow:

```text
Rust      → correctness, APIs, domain, adapters, durable product behavior
TypeScript/React/Vite → UI, product surface, generated API clients, rendered UX proof
PostgreSQL → durable truth, schema constraints, migrations, indexes
Python    → bounded AI/data/eval/prompt tooling only
```

The standard should not say “all languages are fine.” It should say:

```text
Jankurai standard stack:
- Rust for product logic and services.
- TypeScript/React/Vite for web UI.
- PostgreSQL for durable truth.
- Python only where Python is clearly superior: AI, data, evals, notebooks.

Other stacks are allowed only by explicit exception.
```

That gives agents something priceless: a default answer.
