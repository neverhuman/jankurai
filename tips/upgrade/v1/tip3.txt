I studied the uploaded source archive closely. My strongest diagnosis: **the paper has the right thesis, but the project is not yet installable or enforceable enough to become a global standard.** The missing piece is not more prose; it is a **one-command bootstrap**, a **proof-grade audit that cannot be fooled by vibes**, and a **single executable stack template with boundary/UX proof built in**.

I’d make these the **top 3 upgrades**.

---

## 1. **Zero‑Scatter Bootstrap**

**Goal:** make Jankurai adoptable in one command, across all agent IDEs, without humans deciding where instructions belong.

Right now the repo has strong standard files, but adoption is still manual. `README.md` tells users to copy `agent/JANKURAI_STANDARD.md`, create an `AGENTS.md` pointer, and run the audit. That is not enough for broad adoption. The standard should install itself into the known agent-helping locations and keep those files synchronized.

This matters because different coding agents now have different native memory/instruction locations. Codex uses `AGENTS.md`, discovers files from the repo root down to the working directory, and gives closer files precedence; OpenAI also recommends keeping these files concise and pointing to task-specific markdown when needed. Claude Code uses `CLAUDE.md`/`.claude/CLAUDE.md` and supports path-specific memory. GitHub Copilot supports `.github/copilot-instructions.md` plus path-specific `.github/instructions/*.instructions.md` files. Gemini CLI uses `GEMINI.md`. Antigravity-style workflows use `.agents` directories for workflows and skills. The open `AGENTS.md` convention is also explicitly meant to be a simple “README for agents.” ([OpenAI Developers][1])

### New command

Add:

```bash
jankurai init
```

With useful variants:

```bash
jankurai init --yes
jankurai init --dry-run
jankurai init --interactive
jankurai init --profile rust-ts-vite-react-postgres
jankurai init --agents codex,claude,cursor,copilot,gemini,antigravity
jankurai init --ci github
jankurai init --install-ux-qa
jankurai doctor
jankurai adapters write
```

The command should do three things:

1. **Detect the repo shape.**
   It should inspect `Cargo.toml`, `package.json`, Vite config, TS config, Python files, `db/`, `migrations/`, `contracts/`, `.github/workflows`, existing agent files, and monorepo layout.

2. **Generate the Jankurai standard files.**
   It should create or merge these:

   ```text
   AGENTS.md
   agent/JANKURAI_STANDARD.md
   agent/audit-policy.toml
   agent/owner-map.json
   agent/test-map.json
   agent/proof-lanes.toml
   agent/generated-zones.toml
   agent/standard-version.toml
   docs/architecture.md
   docs/boundaries.md
   docs/testing.md
   docs/exceptions/README.md
   ```

3. **Generate IDE/agent adapter files from one canonical source.**
   These files should not contain divergent advice. They should point back to the canonical Jankurai standard.

   ```text
   AGENTS.md
   CLAUDE.md
   .claude/CLAUDE.md
   .cursor/rules/jankurai.mdc
   .cursor/rules/rust-core.mdc
   .cursor/rules/web-ui.mdc
   .github/copilot-instructions.md
   .github/instructions/jankurai-rust.instructions.md
   .github/instructions/jankurai-web.instructions.md
   GEMINI.md
   .agents/agents.md
   .agents/skills/jankurai/SKILL.md
   .agents/workflows/jankurai-audit.md
   ```

### The installer should be interactive, but not chaotic

When the repo layout is messy, `jankurai init --interactive` should produce a patch plan instead of silently imposing structure.

Example:

```text
Jankurai found:
- Vite app in ./src
- Rust API code in ./server
- SQL migrations in ./migrations
- No owner map
- No proof lane map
- Existing CLAUDE.md with conflicting test command

Recommended alignment:
- Treat ./src as apps/web
- Treat ./server as apps/api
- Treat ./migrations as db/migrations
- Add local AGENTS.md files for web/api/db boundaries
- Preserve existing paths with aliases instead of moving files

Apply?
[ ] Write agent files
[ ] Write audit policy
[ ] Write owner/test/proof maps
[ ] Add CI workflow
[ ] Move directories to canonical layout
```

Default behavior should be conservative:

```bash
jankurai init --dry-run
```

should write nothing.

```bash
jankurai init --yes
```

should create config, docs, CI, and adapters, but **not move application code** unless explicitly told:

```bash
jankurai init --migrate-layout
```

## 2. **Evidence‑Grade Audit Core**

**Goal:** make the audit fast, hard to fool, policy-aware, and directly actionable.

This is the most important technical upgrade.

The current audit is promising, but it has several release-blocking weaknesses in the uploaded archive:

* `repo-score.md` shows **score 72** but also says **“No findings”** and an empty Agent Fix Queue. A score below the policy floor should never produce zero actions.
* `agent/audit-policy.toml` defines a minimum score and fail conditions, but the Rust audit does not appear to enforce it.
* `.github/workflows/jankurai-audit.yml` only echoes several security tool names. The audit treats those strings as evidence, so placeholder text can falsely satisfy security checks.
* `Justfile` has a `security` recipe that only echoes tools instead of running them.
* The archive lacks `Cargo.lock` and `package-lock.json`, while the docs/CI use `--locked` and `npm ci`.
* `crates/jankurai/src/audit/mod.rs` is very large, but the audit excludes `crates/jankurai/` from product-code checks, so the project avoids its own shape rules.
* Generated-contract detection is too loose; any file containing “generated” can help satisfy the check.
* Owner/test/generated-zone manifests exist, but the audit mostly uses hardcoded heuristics instead of validating those manifests.
* Report output is not yet agent-optimal: findings need stable fingerprints, next commands, exception templates, affected owners, CI annotations, and issue export.

### New command set

Add:

```bash
jankurai audit . --policy agent/audit-policy.toml --fail
jankurai audit . --changed --json agent/repo-score.json --md agent/repo-score.md
jankurai ci . --policy agent/audit-policy.toml --sarif jankurai.sarif
jankurai ci . --baseline agent/jankurai-baseline.json --ratchet
jankurai issues export --format ndjson --out agent/fix-queue.ndjson
jankurai issues export --format beads --out .beads/jankurai.jsonl
jankurai issues export --format github --out .github/jankurai-issues.md
```

### Hard checks vs soft checks

Make this explicit in the standard.

**Hard checks** should block CI:

```text
No root AGENTS.md or equivalent agent entrypoint
Missing Jankurai standard file
Audit policy not enforced in CI
Score below policy floor
Security lane is placeholder/echo-only
Missing lockfile when install commands require one
Generated contract/code changed without manifest or regeneration command
Secret-like material in repo
Prompt-injection or overbroad agent-agency language in trusted files
Direct DB access in UI/domain/Python product code
Python used for product truth/authz/billing instead of ML/eval boundary
Destructive migration without rollback/backfill/lock evidence
Focused/skipped/tautological tests
Critical web UI surface without rendered UX proof
```

**Soft checks** should create warnings and fix queue items:

```text
Files approaching size limits
Weak names
Duplicated logic below hard threshold
Unmapped ownership
Unmapped tests
Long context files
Stale docs
TODOs without owner/expiration
Non-critical UX geometry issues
Dependency sprawl
Missing observability receipt
```

Each rule should produce:

```rust
pub struct Finding {
    pub id: String,
    pub title: String,
    pub severity: Severity,
    pub hard: bool,
    pub confidence: Confidence,
    pub path: Option<PathBuf>,
    pub line: Option<u32>,
    pub owner: Option<String>,
    pub lane: Option<String>,
    pub evidence: Vec<Evidence>,
    pub why_it_matters: String,
    pub suggested_fix: String,
    pub suggested_commands: Vec<String>,
    pub exception_template: Option<String>,

}
```

Add confidence levels:

```text
exact       parsed config / AST / lockfile / CI command
strong      direct file/path evidence
heuristic   text-risk signal
```

A rule that only has heuristic evidence should usually be soft unless the pattern is dangerous enough.

### Make the audit hard to spoof

The audit should not count this as a security lane:

```yaml
- run: echo "cargo audit"
- run: echo "gitleaks detect"
```

It should require either:

```yaml
- run: cargo audit
```

or:

```yaml
- run: just security
```

where `just security` actually invokes tools, not echoes them.

For Rust dependency risk, add actual support for the Rust security tooling ecosystem. `cargo-audit` audits `Cargo.lock` against the RustSec advisory database; `cargo-deny` can check advisories, license policy, crate bans, sources, and duplicate versions; `cargo-vet` helps teams record third-party dependency audits; `cargo-nextest` is a faster Rust test runner with per-test isolation and CI support. ([RustSec][2])

Recommended security lane:

```bash
cargo audit
cargo deny check advisories bans licenses sources
cargo vet
cargo nextest run --workspace
cargo llvm-cov --workspace --fail-under-lines 70
cargo machete
cargo semver-checks
gitleaks detect --source .
syft . -o spdx-json=sbom.spdx.json
grype sbom:sbom.spdx.json
zizmor .github/workflows
actionlint
```

Do not require every repo to run every tool on day one. But the audit should distinguish:

```text
missing
declared-only
runs-locally
runs-in-ci
runs-and-blocks
```

### Speed improvements

The audit should be extremely fast. Right now it repeatedly scans file text for different dimensions. Move to one inventory pass:

```text
1. Walk repo once with ignore.
2. Classify paths once.
3. Read text files once.
4. Store lowercase text once.
5. Run pattern packs with Aho-Corasick or compiled regex sets.
6. Run rules in parallel with rayon.
7. Cache file hashes optionally under target/jankurai/cache.json.
```

Add performance targets:

```text
small repo: < 250 ms
medium repo: < 2 s
large monorepo changed-mode: < 2 s
large monorepo full scan: < 15 s
```

**Goal:** stop stack scatter by shipping one blessed executable layout: Rust core/API, Vite/TypeScript/React UI, Postgres truth, Python only for ML/eval boundaries, plus deterministic UX QA.

### Boundary rules

Make this table normative and generate local `AGENTS.md` files from it.

| Layer                | Owns                                                                                                      | Must not own                                                             |
| -------------------- | --------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| `apps/web`           | React UI, route state, forms, local validation, generated API clients, component tests, Playwright, UX QA | Secrets, durable truth, authz, billing truth, direct DB access           |
| `apps/api`           | HTTP/RPC edge, request parsing, identity extraction, response mapping                                     | Domain invariants hidden in handlers, raw SQL sprawl                     |
| `crates/domain`      | IDs, invariants, state machines, pure decisions, typed domain errors                                      | IO, DB, env vars, framework code, time/random without injected interface |
| `crates/application` | Use cases, authz, idempotency, transactions, workflow orchestration                                       | UI details, SQL driver details                                           |
| `crates/adapters`    | Postgres, queues, external APIs, filesystem, env                                                          | Domain rules                                                             |
| `crates/workers`     | Background jobs, retries, replay, backoff                                                                 | Request/response UI concerns                                             |
| `contracts`          | OpenAPI/Protobuf/JSON schemas, generated clients                                                          | Hand-edited generated code                                               |
| `db`                 | Migrations, constraints, indexes, RLS, transactional truth                                                | Business workflows                                                       |
| `python/ai-service`  | ML inference, evals, embeddings, offline transforms                                                       | Product truth, authz, billing, direct production DB writes               |

The audit should enforce this with both path rules and import rules.

Examples:

```text
apps/web must not import pg, postgres, sqlx, diesel, Prisma DB clients, or server secrets.
crates/domain must not import sqlx, tokio-postgres, reqwest, std::env, std::fs.
python/ai-service must not import production DB clients unless marked as offline/eval and read-only.
contracts/generated must not be hand-edited.
```

# The biggest current gaps to fix before release

These are the concrete issues I would fix before polishing the paper further.

1. **Do not ship a stale or misleading score.**
   The archive’s `repo-score.md` says score 72 with no findings. That undermines trust. A score below 85 should always produce actionable findings.

2. **Do not count echoed commands as proof.**
   The current security lane placeholders create a false-green risk.

3. **Add lockfiles or stop using lockfile-dependent commands.**
   The archive does not include `Cargo.lock` or `package-lock.json`, but docs/CI use `--locked` and `npm ci`.

4. **Split `crates/jankurai/src/audit/mod.rs`.**
   A flagship anti-scatter project should not hide a 2,000+ line audit module outside its own shape rules.

5. **Make `agent/audit-policy.toml` real.**
   The policy file should control score floors, caps, hard checks, changed-mode behavior, and CI failure.

6. **Unify report location.**
   Some docs imply `agent/repo-score.json`; the archive has root `repo-score.json`. Pick one. I recommend:

   ```text
   agent/repo-score.json
   agent/repo-score.md
   agent/fix-queue.ndjson
   ```

7. **Make UX QA installable.**
   The UX package is promising, but it needs `jankurai ux init`, route config, CI integration, and a stable receipt schema.

8. **Add adapter files now, not later.**
   The paper names the right ecosystem. The project should generate those files immediately.

# Minimal implementation order for an agent

If an implementation agent starts with minimal context, give it this order.

## Phase 1: bootstrap

```text
1. Add jankurai init/doctor/adapters subcommands.
2. Embed templates.
3. Generate AGENTS.md, agent/*.toml/json, docs/*.md.
4. Generate CLAUDE.md, Cursor rules, Copilot instructions, GEMINI.md, .agents files.
5. Add dry-run and --yes modes.
6. Add tests for generated files and merge behavior.
```

## Phase 2: audit correctness

```text
1. Parse agent/audit-policy.toml.
2. Add policy decision to JSON/Markdown report.
3. Make score below floor produce blocking finding.
4. Make low dimensions produce fix queue items.
5. Reject echo-only proof commands.
6. Parse owner/test/generated-zone manifests.
7. Add SARIF output.
8. Add issue export.
9. Split audit/mod.rs into rule modules.
10. Add bad/good repo fixtures.
```

## Phase 3: stack + UX

```text
1. Add rust-ts-vite-react-postgres stack template.
2. Add boundary AGENTS.md files.
3. Add import/path boundary rules.
4. Add jankurai ux init/audit.
5. Add UX route matrix and receipt schema.
6. Connect UX receipt to jankurai audit.
7. Add CI template with fast/security/ux/audit lanes.
```

# My recommended final positioning

The project should not present itself as “another lint tool.” It should be positioned as:

> **Jankurai is the executable standard for agent-native repositories. It installs the agent memory files, defines the canonical stack boundaries, runs fast proof lanes, scores the repo, and gives agents a stable repair queue.**

The paper should sharpen around this sentence:

> **Humans scatter context; agents need stable routing. Jankurai turns a repo into a routable, auditable, repairable workspace.**

The best version of this project is not bigger. It is stricter, easier to install, and harder to fake.
