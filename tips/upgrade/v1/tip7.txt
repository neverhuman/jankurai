I studied the archive as a paper, a standard, an installable product, an auditor, a CI gate, and an agent-IDE setup kit. My verdict: **the thesis is strong, but the project needs to pivot from “paper + scanner” to “installer + policy engine + proof pack.”** The current artifact already has the bones of a serious standard, but it is not yet easy enough to install, not strict enough to trust, and not agent-adapter-aware enough to become the default across Cursor, Claude, Codex, Copilot, Antigravity, and terminal agents.

One verification note: I could not run Rust tests because `cargo` was not available in the container. I did inspect the Rust, TypeScript, docs, CI, paper, and config files directly. Node/npm were available, and `npm ci` failed because the archive has no package lockfile, despite the README, Justfile, and GitHub Actions workflow all using `npm ci`.

The **best possible 3 upgrades** are:

1. **Jankurai Init — Zero-Scatter Adoption Kit**
2. **Jankurai Sentinel — Policy-as-Code Audit + Issue Export**
3. **Stackline Pack — Canonical Boundaries + UX Proof Runtime**

These three keep the scope sharp: install the standard everywhere agents look, make the audit impossible to “false green,” and enforce one best-known stack with clear boundaries and proof.

---

## What I found first

The current project has a compelling standard:

* `paper/jankurai.md` frames the central thesis well: the bottleneck is no longer code generation, but trustworthy merge.
* `docs/agent-native-standard.md` already names a strong default stack: **Rust core + TypeScript/React/Vite UI + PostgreSQL durable truth + generated contracts + bounded Python**.
* `agent/` already contains useful standard assets: `JANKURAI_STANDARD.md`, `audit-policy.toml`, `owner-map.json`, `test-map.json`, `generated-zones.toml`, `proof-lanes.toml`, and `standard-version.toml`.
* `packages/ux-qa` is a good start for UI proof, with deterministic rule IDs like target size, overlap, clipping, wrapping, obstruction, focus visibility, labels, and nested scrollbars.
* The paper already understands that agent adapters must be mirrored from one canonical source rather than becoming divergent manuals.

But the install/product layer currently breaks trust:

* `README.md` recommends `cargo install --git ... --locked`, but the archive has **no `Cargo.lock`**.
* `README.md`, `Justfile`, and `.github/workflows/jankurai-audit.yml` use `npm ci`, but the archive has **no `package-lock.json`, `pnpm-lock.yaml`, or `yarn.lock`**.
* `README.md` references missing assets: `assets/jankurai_readme.png` and `assets/vibe-coding-tlr-pie.svg`.
* The source-only archive references `paper/jankurai.pdf`, but the PDF is not present. That may be intentional, but the README and version manifest should explain “source-only mode” rather than producing broken paths.

The current audit also gives a dangerous false green:

* `repo-score.md` reports score `72`, below the documented target floor of `85`, but says **“No findings”** and **“No queued fixes.”**
* The auditor excludes its own core code from runtime-stack surface checks. `crates/jankurai/src/audit/mod.rs` is about **2,114 lines**, while `agent/audit-policy.toml` sets a Rust hard max of `350`, but the audit misses this because `crates/jankurai/` is excluded.
* `agent/audit-policy.toml`, `owner-map.json`, and `test-map.json` exist, but the Rust audit currently hardcodes important weights, caps, and owner prefixes instead of treating those files as the real policy source.
* The security lane can be satisfied by placeholder text. The GitHub workflow and Justfile currently echo security tool names, and the audit only checks whether strings like `gitleaks`, `syft`, `grype`, `zizmor`, or `security` appear in command surfaces.

Those are not just bugs. They are exactly the kind of “vibe coding mess” the project is supposed to eliminate.

# Upgrade 1: **Jankurai Init — Zero-Scatter Adoption Kit**

### Goal

Make Jankurai installable with one command, then have it place the correct standard files exactly where modern coding agents expect them.

The current project has a standard, but it does not yet have a strong enough adoption path. The new product center should be:

```bash
jankurai init --profile rust-ts-vite-react-postgres --ide all --ci github --ux-qa --apply
jankurai doctor
jankurai check .
```

Also support a safe planning mode:

```bash
jankurai init --plan --json jankurai-init-plan.json
```

This upgrade makes the installer the standard. The paper can still matter, but adoption happens through `jankurai init`.

### Why this matters

Different agent IDEs read different instruction files. Codex explicitly reads `AGENTS.md` before starting work and has discovery/instruction precedence behavior; OpenAI’s docs also note a default `32 KiB` instruction read cap, which means the root agent file must stay short and pointer-based. ([OpenAI Developers][1]) Claude Code does **not** directly use `AGENTS.md` as its primary file; Anthropic recommends `CLAUDE.md`, and it can import `@AGENTS.md`; Claude also supports modular `.claude/rules/` files. ([Claude][2]) GitHub Copilot supports `.github/copilot-instructions.md` and `.github/instructions/*.instructions.md`, and GitHub’s coding agent also supports root and nested `AGENTS.md` plus other common instruction files. ([GitHub Docs][3]) Cursor has project/team/user rules and supports `AGENTS.md`; Antigravity has rules/workflows stored as Markdown-like customization surfaces, so these should be generated from the same source rather than hand-maintained. ([Cursor][4])

### The product behavior

`jankurai init` should inspect the repo, create a change plan, and either apply it or stop with precise instructions.

It should detect:

* Git repo root.
* Existing `AGENTS.md`, `CLAUDE.md`, `.cursor/rules`, `.github/copilot-instructions.md`, `.github/instructions`, Antigravity rule/workflow locations.
* Rust workspace and `Cargo.lock`.
* Node package manager and lockfile.
* Vite/React/TypeScript app location.
* PostgreSQL migrations.
* Existing Python usage.
* Existing GitHub Actions workflows.
* Existing issue tooling, especially `.beads`, GitHub Issues, or plain `.git`.
* Existing docs: `docs/architecture.md`, ADRs, runbooks, testing docs.
* Existing generated-code zones.
* Current package manager mismatch, for example `npm ci` without a lockfile.

It should not blindly rewrite user repos. It should generate an explicit plan:

```text
Jankurai Init Plan

Will add:
  AGENTS.md
  CLAUDE.md
  .claude/rules/jankurai.md
  .cursor/rules/jankurai.mdc
  .github/copilot-instructions.md
  .github/instructions/jankurai-boundaries.instructions.md
  agent/JANKURAI_STANDARD.md
  agent/audit-policy.toml
  agent/owner-map.json
  agent/test-map.json
  agent/generated-zones.toml
  agent/proof-lanes.toml
  agent/ux-qa.toml
  .github/workflows/jankurai.yml

Will modify:
  package.json
  README.md

Will not modify automatically:
  src/
  db/migrations/

Warnings:
  npm ci is configured but no package-lock.json exists.
  Existing CLAUDE.md has no import of AGENTS.md.
  Python files found outside python/ai-service.
```

For interactive use, it can ask targeted questions only when needed:

```text
I found Python in scripts/, services/ml/, and tools/.
Which location is allowed to contain model/eval/data-transform Python?

[1] python/ai-service
[2] services/ml
[3] Keep existing layout but create documented exception
```

For CI/noninteractive mode, it must never block on a question. It should write a plan, fail clearly, or create an exception file:

```bash
jankurai init --noninteractive --plan
jankurai init --noninteractive --apply --prefer-safe-exceptions
```

# Upgrade 2: **Jankurai Sentinel — Policy-as-Code Audit + Issue Export**

### Goal

Turn the auditor from a helpful scanner into a trusted policy engine that cannot report “no findings” when the repo is clearly below standard.

The current self-score is the best evidence: score `72`, no findings, no queued fixes. That is the exact failure mode to eliminate.

### Required behavior

Add a new top-level command:

```bash
jankurai check . --min-score 85 --fail-on high,critical
```

Keep `audit` as the report generator:

```bash
jankurai audit . \
  --json repo-score.json \
  --md repo-score.md \
  --sarif jankurai.sarif \
  --junit jankurai.junit.xml \
  --issue-export beads \
  --github-annotations
```

GitHub supports uploading SARIF from third-party tools into code scanning, so Jankurai should emit SARIF for repo-shape, boundary, generated-code, and security findings. ([GitHub Docs][5]) Beads is especially well aligned with this project because it is a git-backed issue tracker with structured JSON/JSONL state, dependency-aware tasks, and agent-friendly CLI output. ([GitHub][6])

### Immediate fixes

These should happen before adding more features:

1. **Parse the actual policy files.**
   `agent/audit-policy.toml`, `owner-map.json`, `test-map.json`, `generated-zones.toml`, and `proof-lanes.toml` must be the source of truth. The Rust code should not hardcode owner prefixes, weights, caps, or limits that already exist in policy.

2. **Self-audit the Jankurai tool.**
   Do not exclude `crates/jankurai/` or `packages/ux-qa/` from code-shape rules. The current `crates/jankurai/src/audit/mod.rs` is over 2,000 lines and should be flagged hard against the Rust file cap.

3. **No below-floor empty queue.**
   When score is below the configured floor, the audit must emit at least one finding and one fix-queue item:

   ```text
   HLT-019-SCORE-UNDER-FLOOR
   Severity: high
   Why: repo score 72 is below required floor 85.
   Fix: address the lowest dimensions first: observability, speed, data truth, contract.
   ```

4. **Security lanes require proof, not words.**
   The audit must distinguish “tool name appears in an echo command” from “tool actually ran and produced an artifact.” The current workflow echoing `gitleaks`, `syft`, `grype`, and `zizmor` should fail.

5. **Implement naming and junk-drawer checks.**
   `weak_name_hits()` is currently a stub. It should catch common vibe-coded fog:

   ```text
   utils/
   helpers/
   common/
   misc/
   stuff/
   temp/
   new/
   final/
   final2/
   manager/
   service/
   processor/
   handler/
   data.ts
   types.ts with unrelated types
   index.ts barrel dumping unrelated exports
   ```

6. **Validate manifest artifact paths.**
   `agent/standard-version.toml` currently lists artifacts such as `paper/jankurai.pdf`. The versions check should validate paths or explicitly support `source_only = true`.

### Hard checks vs soft checks

The report should label every finding as hard or soft.

Hard checks should fail `jankurai check` by default:

```text
HLT-001 missing root AGENTS.md
HLT-002 missing IDE adapter for configured agent
HLT-003 install command references missing lockfile
HLT-004 source file exceeds hard LOC cap
HLT-005 generated file edited by hand
HLT-006 contract changed without regenerated clients
HLT-007 generated client changed without source contract
HLT-008 UI code imports database/client secrets
HLT-009 Rust domain layer performs IO/env/time/random/db access
HLT-010 Python outside approved ML/eval/data-transform zone
HLT-011 security lane is placeholder or missing proof artifact
HLT-012 direct production DB write from unapproved layer
HLT-013 migration lacks rollback/backfill/safety note
HLT-014 changed UI route lacks UX QA receipt
HLT-015 owner map missing changed path
HLT-016 test map missing changed path
HLT-017 score below floor
HLT-018 critical dependency/security finding
HLT-019 public API/contract drift
HLT-020 CI workflow references missing command
```

Soft checks should warn and feed the fix queue:

```text
HLT-S001 root AGENTS.md too long
HLT-S002 file approaching LOC cap
HLT-S003 naming fog / junk drawer
HLT-S004 duplicate logic block
HLT-S005 TODO without owner or expiry
HLT-S006 package manager multiplicity
HLT-S007 too much Python outside bounded purpose
HLT-S008 weak observability
HLT-S009 slow CI lane lacks split/changed-only mode
HLT-S010 unpinned GitHub Action
HLT-S011 broad dependency import
HLT-S012 missing ADR for architectural exception
```

### Report shape

Make the report agent-first. The current Markdown report is too passive.

Recommended report order:

```text
1. Final score and gate result
2. Top 3 fixes to pass CI
3. Hard failures
4. Soft warnings
5. Changed-file findings
6. Boundary violations
7. Security/proof receipts
8. UX QA receipts
9. Agent fix queue
10. Exact commands to rerun
11. Timing profile
```

Each finding should include:

```json
{
  "id": "HLT-004",
  "kind": "hard",
  "severity": "high",
  "confidence": 0.98,
  "path": "crates/jankurai/src/audit/mod.rs",
  "line": 1,
  "message": "Rust file exceeds hard LOC cap.",
  "evidence": {
    "actual_lines": 2114,
    "max_lines": 350,
    "policy": "agent/audit-policy.toml"
  },
  "why_it_matters": "Large files reduce agent locality and increase hidden coupling.",
  "fix": {
    "summary": "Split audit/mod.rs into policy, context, rules, report, and fix_queue modules.",
    "commands": [
      "jankurai explain HLT-004"
    ]
  },
  "fingerprint": "stable-hash-here"
}
```

The `agent_fix_queue` should not be optional. It is the core product.

# Upgrade 3: **Stackline Pack — Canonical Boundaries + UX Proof Runtime**

### Goal

Package one best-known stack and make its boundaries impossible to misunderstand.

Do not start by supporting every stack. That causes the exact scatter the project is fighting. Start with one canonical profile:

```text
rust-ts-vite-react-postgres
```

Meaning:

```text
Rust        = domain, application, adapters, workers, API core
TypeScript  = generated clients, UI types, React app, browser-side validation
React/Vite  = product surface, interaction, route-level UX
PostgreSQL  = durable truth, constraints, migrations, indexes
Python      = bounded ML/eval/data-transform only
```

### Boundary rules to write into `docs/boundaries.md`

| Layer                | Owns                                                                        | Must not own                                                      |
| -------------------- | --------------------------------------------------------------------------- | ----------------------------------------------------------------- |
| Rust domain          | Invariants, IDs, state machines, pure validation                            | IO, env vars, clocks, randomness, DB, HTTP, framework types       |
| Rust application     | Commands, authz orchestration, idempotency, transactions, ports             | UI rendering, raw external API types, SQL details                 |
| Rust adapters        | DB, queues, external APIs, filesystem, environment, telemetry wiring        | Domain rules, authz policy decisions                              |
| Rust API             | Protocol boundary, request/response mapping, generated contract conformance | Business invariants that belong in domain                         |
| TypeScript generated | API clients, generated DTOs, generated validators                           | Manual edits                                                      |
| React/Vite           | UI rendering, forms, routes, browser interactions, local UI state           | Durable truth, secrets, direct DB access, authz source of truth   |
| PostgreSQL           | Durable constraints, indexes, RLS, migrations, data integrity               | UI behavior, temporary app-only rules                             |
| Python               | ML inference, embeddings, evals, offline transforms                         | Product truth, authz, production write paths, core business rules |

### Jankurai boundary checks

Add specific checks:

```text
React/TS:
  - fail if UI imports pg, postgres, sqlx, Prisma client, raw SQL, or server secrets
  - fail if handwritten API types duplicate generated contract types
  - warn on broad barrel exports that hide ownership
  - fail if generated client is edited manually

Rust domain:
  - fail on std::fs, std::env, SystemTime, rand, sqlx, reqwest, tokio::spawn
  - fail on framework request/response types
  - warn on large service objects or vague names

Rust application:
  - fail if transaction boundaries are unclear
  - fail if authz checks are absent on command handlers
  - warn if command names are generic

Rust adapters:
  - fail if they contain business invariants
  - fail if SQL result shape bypasses mapped domain types

Postgres:
  - fail destructive migration without safety note
  - fail table without primary key
  - fail missing index on obvious foreign key
  - warn on nullable fields without reason

Python:
  - fail outside allowed path unless exception exists
  - fail imports that imply production DB writes
  - fail if Python owns product truth
```

### CI generated by the stack pack

Generated GitHub workflow should actually gate:

```yaml
name: jankurai

on:
  pull_request:
  push:
    branches: [main]

permissions:
  contents: read
  security-events: write
  actions: read

jobs:
  jankurai:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Jankurai
        run: cargo install --path crates/jankurai --locked

      - name: Jankurai check
        run: |
          jankurai check . \
            --min-score 85 \
            --fail-on high,critical \
            --json repo-score.json \
            --md repo-score.md \
            --sarif jankurai.sarif

      - name: Upload Jankurai SARIF
        uses: github/codeql-action/upload-sarif@v3
        with:
          sarif_file: jankurai.sarif

      - name: Upload Jankurai artifacts
        uses: actions/upload-artifact@v4
        with:
          name: jankurai-report
          path: |
            repo-score.json
            repo-score.md
            jankurai.sarif
            artifacts/ux-qa/**
```

Security should not be placeholder text. The generated workflow should run real lanes, for example:

```bash
gitleaks detect --no-git --redact --report-format sarif --report-path gitleaks.sarif
osv-scanner --lockfile Cargo.lock --lockfile package-lock.json --format json --output osv.json
cargo deny check advisories bans licenses sources
zizmor .github/workflows --format sarif > zizmor.sarif
syft dir:. -o spdx-json=sbom.spdx.json
```

# Paper improvements to make the standard land harder

The paper is ambitious and mostly right, but it needs more “adoption mechanics” and less reliance on thesis alone.

## 1. Add “The Installer Is the Standard”

Add a section after the agent-controls section:

```text
A standard that requires manual copying is not a standard. Jankurai is defined by a reproducible installer that writes canonical policy, generates agent adapters, configures CI, and validates itself. The paper is explanatory; the installer is normative.
```

Then include the exact adapter matrix:

```text
AGENTS.md
CLAUDE.md
.claude/rules/jankurai.md
.cursor/rules/jankurai.mdc
.github/copilot-instructions.md
.github/instructions/*.instructions.md
.antigravity/rules/jankurai.md
.antigravity/workflows/jankurai-audit.md
```

## 2. Add “No False Green” as a core design law

The strongest critique of the current project is that its own repo can score below floor and still emit no fix queue.

Add this law:

```text
A failed or below-floor audit without actionable repair output is itself a product defect.
```

Then define:

```text
Every below-floor score must name:
1. the lowest dimensions,
2. the top three repairs,
3. the owner/test/proof lanes affected,
4. the exact command to rerun,
5. the agent_fix_queue item IDs.
```

## 3. Add hard vs soft checks to the paper

Right now the standard talks about scoring, but adoption will be easier if users understand what is non-negotiable.

Hard examples:

```text
missing lockfile for install command
manual generated-code edit
direct DB import from UI
Python owning product truth
security placeholder lane
changed UI route without UX proof
```

Soft examples:

```text
naming fog
large-but-not-yet-over-cap file
weak observability
too many package managers
long AGENTS.md
TODO without owner
```

## 4. Add “agent context economy” as a measurable value

The standard is not just about cleaner code. It is about reducing the number of tokens and decisions an agent needs before making a safe change.

Add metrics:

```text
time_to_first_safe_edit
files_opened_before_correct_patch
fix_queue_completion_rate
false_green_rate
rerun_success_rate
agent_instruction_bytes
boundary_violation_rate
changed_files_without_owner
changed_files_without_test_lane
```

This helps make the standard empirical rather than ideological.

## 5. Adjust public tone for adoption

The “humans were the bug” frame is provocative and memorable, but adoption will be broader if the public README leads with:

```text
Stop vibe drift. Ship proof.
```

Then the sharper jankurai framing can remain in the paper.

For maintainers, the pitch should be:

```text
Jankurai helps agents make smaller, safer, faster changes by giving every repo a standard map, boundary policy, proof lanes, and actionable audit queue.
```

# The strongest final standard

The project should define itself this way:

> **Jankurai is an agent-native repo standard and audit gate. It installs canonical agent instructions, enforces stack boundaries, generates proof lanes, and emits an actionable fix queue so humans and agents can ship faster without vibe drift.**

The three upgrades above make that real:

1. **Jankurai Init** makes it easy enough to adopt.
2. **Jankurai Sentinel** makes it strict enough to trust.
3. **Stackline Pack** makes it clear enough for all agents to follow.

The highest-impact change is philosophical but concrete: **the audit must stop being a score report and become the repo’s operating system for agent repair.** Every failing score should become a precise queue. Every queue item should name the files, the rule, the proof lane, and the next command. Every agent IDE should read the same short standard through its native surface. Every boundary should be machine-checkable. Every exception should be explicit.

That is how this becomes a standard rather than another repo checklist.
