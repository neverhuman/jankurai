I dug through the source archive, the paper, the Rust auditor, the UX QA package, CI, agent files, and current agent-IDE conventions. The idea is genuinely strong: **jankurai can become the “repo nervous system” for AI coding**—not just a linter, but the standard that tells every agent where truth lives, what boundaries are non-negotiable, and what proof must exist before changes count.

My conclusion: the next version should not primarily add more prose. It should become **installable, evidence-backed, and self-enforcing**. The paper already has the thesis. The project now needs to make adoption feel like:

```bash
jankurai init --profile rust-vite-react-postgres --toolpacks all
jankurai audit --ci
```

…and then every agent, every IDE, and CI immediately know what “good” means.

There are also a few release blockers worth fixing before this becomes public-standard quality:

| Breakpoint                                                                                                                                 | Why it matters                                                                                              |
| ------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------- |
| `AGENTS.md` contains `/Users/bentaylor/Code/jankurai`                                                                                     | This leaks a local path into the public standard and teaches agents the wrong repo root.                    |
| No `Cargo.lock`, but README uses `cargo install ... --locked`; no `package-lock.json`, but CI/Justfile use `npm ci`                        | The install path is currently not reproducible and likely fails for fresh users.                            |
| `.github/workflows/jankurai-audit.yml` has `echo "gitleaks..."`, `echo "syft..."`, etc.                                                   | The current audit detects those strings as security evidence, so placeholder text can fake a security lane. |
| `repo-score.json` has score `72`, no findings, no timestamp, no caps, and says “lockfile present” even though the archive has no lockfiles | This is the most important trust gap: the report can be stale or misleading without any attestation.        |
| `crates/jankurai/src/audit/mod.rs` is ~2,114 lines, but the auditor excludes `crates/jankurai/` from runtime stack checks                | Jankurai is not fully auditing itself. That undermines the “new standard” claim.                           |
| `weak_name_hits()` is a stub                                                                                                               | The paper promises naming/entropy detection that does not yet exist.                                        |
| `owner-map.json`, `test-map.json`, and `generated-zones.toml` are present but not really enforced as first-class audit inputs              | The standard has the right machine maps, but the CLI does not yet fully trust/use them.                     |
| The source archive includes many AppleDouble `._*` files                                                                                   | Release packaging needs cleanup, or agents/audits will inspect junk.                                        |
| `packages/ux-qa/dist/` and `test-results/` are committed but not declared as generated zones                                               | Generated/runtime artifacts need consistent treatment.                                                      |
| Docs disagree on `repo-score.json` location and LOC limits                                                                                 | The standard needs one canonical answer.                                                                    |

Also: I could not run the full Rust test suite in this environment because the sandbox has no internet and the archive has no lockfile/vendor cache; `cargo test` attempted to fetch crates. That failure itself is useful evidence: the project needs deterministic install/CI artifacts before it can be the standard.

## Best possible 3 upgrades

### 1. **StandardKit Init: one-command adoption across every agent IDE**

This is the highest-leverage upgrade. Today the project has a good standard, but adoption still requires humans to copy files, understand every agent IDE, and wire CI manually. That is the exact “human scatter” this project wants to eliminate.

Build a real initializer:

```bash
jankurai init
jankurai init --profile rust-vite-react-postgres
jankurai init --toolpacks codex,claude,cursor,copilot,gemini,antigravity
jankurai init --ci github --ux-qa --beads --dry-run
jankurai doctor --fix
```

The installer should detect the repo, propose changes, ask only when mutation is risky, then write the minimum set of files. `AGENTS.md` is a strong universal root because Codex explicitly reads `AGENTS.md` before work and layers root/nested files with closer files overriding earlier guidance. ([OpenAI Developers][1]) But it should not be the only file: Claude Code currently reads `CLAUDE.md`, not `AGENTS.md`, and recommends creating `CLAUDE.md` that imports `@AGENTS.md`; Claude also recommends keeping instructions concise, under roughly 200 lines per `CLAUDE.md`. ([Claude][2]) GitHub Copilot expects `.github/copilot-instructions.md` plus optional path-specific `.github/instructions/*.instructions.md`, and Copilot code review only reads the first 4,000 characters of any custom instruction file. ([GitHub Docs][3]) Gemini CLI uses `GEMINI.md` as its default context file and loads workspace, parent, and just-in-time context files. ([Gemini CLI][4]) Antigravity now has enough public signal to support a first-class adapter too: Google’s codelab uses `.agents/agents.md`, `.agents/skills/`, and `.agents/workflows/`, and a March 2026 Antigravity update says it added support for reading `AGENTS.md` in addition to `GEMINI.md`. ([Google Codelabs][5])

**The new standard should have one source of truth and many thin adapters.**

### 2. **EvidenceGraph Audit: make the score impossible to fake**

The current auditor has the right ambition but is too string-search driven. It can be fooled by comments, docs, placeholders, or stale generated reports. This is the central risk if jankurai is going to become a global standard: **agents will optimize for the score unless the score is evidence-backed.**

The new audit should build a single repo model first, then score from evidence. I would call this internal layer `EvidenceGraph`.

```text
RepoInventory
  -> ManifestGraph
  -> AgentInstructionGraph
  -> CiCommandGraph
  -> OwnershipGraph
  -> TestRouteGraph
  -> GeneratedZoneGraph
  -> StackBoundaryGraph
  -> FindingGraph
  -> Report
```

Concrete current false-positive risks to fix:

```text
has_security_lane() looks for "gitleaks", "syft", "grype", "zizmor", "security"
has_secret_or_dependency_scans() also uses raw string search
has_generated_contracts() returns true if any file text contains "generated"
has_web_surface() returns true if any file text contains "react", "vite", or "storybook"
has_fast_lane() treats "jankurai" as a fast lane
weak_name_hits() returns vec![]
owner_for_path() ignores agent/owner-map.json and uses hard-coded prefixes
```

That means docs, echo commands, or README text can become proof. This has to change.

The evidence model should distinguish:

```rust
enum EvidenceKind {
    ExecutableCommand,
    PlaceholderCommand,
    CommentOnly,
    DocumentationMention,
    ManifestDeclaration,
    GeneratedHeader,
    GitDiff,
    Lockfile,
    ToolOutput,
    Missing,
}

struct Evidence {
    kind: EvidenceKind,
    path: Utf8PathBuf,
    line: Option<usize>,
    command: Option<String>,
    confidence: Confidence,
}
```

Example: this workflow step should be classified as `PlaceholderCommand`, not security proof:

```yaml
- name: Security lane placeholders
  run: |
    echo "gitleaks detect --source ."
    echo "syft . -o spdx-json"
```

A real lane should require actual executable commands, for example:

```yaml
- name: Secret scan
  run: gitleaks detect --source . --redact --exit-code 1

- name: Rust dependency advisories
  run: cargo audit --deny warnings

- name: Rust dependency policy
  run: cargo deny check

- name: GitHub Actions security
  run: zizmor .github/workflows
```

Those tools are good fits for the proof pack: RustSec’s `cargo-audit` audits `Cargo.lock` for vulnerable crates, `cargo-deny` supports advisories, licenses, bans, sources, and duplicate dependency detection, and `cargo-vet` helps projects verify third-party Rust dependencies have been audited by trusted entities. ([RustSec][7]) For speed, `cargo-nextest` is the obvious default Rust test runner because it advertises faster execution, per-test isolation, and CI support. ([Nexte][8]) For broader supply chain, OSV-Scanner, Gitleaks, Dependency Review, Syft/Grype, actionlint, and zizmor are all appropriate external receipts rather than strings in a YAML file. ([GitHub][9])

The report should add provenance:

```json
{
  "schema_version": "2.0.0",
  "standard_version": "0.3.0",
  "auditor_version": "0.3.0",
  "generated_at": "2026-05-02T00:00:00Z",
  "repo": {
    "git_sha": "abc123",
    "dirty": false,
    "tracked_file_count": 412,
    "input_digest": "sha256:..."
  },
  "command": "jankurai audit --ci --json agent/repo-score.json --md agent/repo-score.md",
  "score": 87,
  "raw_score": 91,
  "hard_findings": [],
  "soft_findings": [],
  "caps_applied": [],
  "evidence_summary": {
    "security_lane": "executable",
    "ux_qa": "artifact-backed",
    "generated_zones": "verified",
    "owner_map": "verified",
    "test_map": "verified"
  }
}
```

The audit should also fail stale score files. If `repo-score.json` exists but has no `git_sha`, no `generated_at`, or an input digest not matching the current inventory, that should be a high-severity finding:

```text
HLT-020-STALE-SCORE
Severity: high
Path: agent/repo-score.json
Why: score report has no provenance or does not match current repository state.
Agent fix: rerun `jankurai audit --ci --json agent/repo-score.json --md agent/repo-score.md`.
```

Hard checks should be rare, deterministic, and difficult to dispute:

```text
HARD:
- missing AGENTS.md / agent standard
- stale or missing audit report in CI
- security lane placeholder counted as proof
- secret-like token in non-test path
- generated file edited without source regeneration
- changed path has no owner-map route
- changed path has no test-map proof route
- UI app exists but no rendered UX lane
- Python owns product truth outside approved AI/data boundary
- direct DB access from UI/domain/frontend/Python
- destructive migration without lock/backfill/rollback evidence
- no lockfile for package manager used by CI
```

Soft checks should guide agents without blocking unnecessarily:

```text
SOFT:
- large files/functions approaching thresholds
- weak names: util, misc, helper, manager, common, old, temp
- TODO/fallback markers in product code
- duplicated blocks
- too many generated-client touchpoints
- missing observability receipts
- unpinned GitHub Actions
- TypeScript `any` sprawl
- Rust `unwrap()`/`expect()` outside tests without ledger
- snapshot-only tests without semantic assertions
```

### 3. **ProofPack: the blessed stack, UX QA, and repair loop**

The paper’s strongest move is picking one best-known stack instead of letting every repo “vibe” into a unique snowflake. Keep that. Do not expand into ten blessed stacks. The standard should say:

> Default product stack: **Rust core + TypeScript/React/Vite UI + PostgreSQL truth + generated contracts + bounded Python only for AI/data service work.**

Then ship a proof pack that makes this layout easier than improvising.

Canonical layout:

```text
apps/
  web/                         # Vite React UI only
    src/
      app/
      routes/
      components/
      generated/               # generated client only
      test/
crates/
  domain/                      # pure business invariants, no IO, no DB, no HTTP
  application/                 # use cases, transaction boundaries, ports
  api/                         # HTTP/RPC surface, generated contract bindings
  adapters/                    # Postgres, queues, email, object storage, external APIs
  workers/                     # async jobs
contracts/
  openapi/
  schemas/
db/
  migrations/
  seeds/
python/
  ai-service/                  # only model/data/AI boundary, never product truth
agent/
  JANKURAI_STANDARD.md
  owner-map.json
  test-map.json
  generated-zones.toml
  proof-lanes.toml
  audit-policy.toml
.github/
  workflows/
```

Boundary table to put in the paper, standard, and generated `AGENTS.md`:

| Layer                | Owns                                                           | Must not own                                              |
| -------------------- | -------------------------------------------------------------- | --------------------------------------------------------- |
| `crates/domain`      | invariants, entities, value objects, pure decisions            | SQL, HTTP, filesystem, env vars, timers                   |
| `crates/application` | use cases, ports, transaction orchestration                    | concrete DB drivers, React/UI, generated UI code          |
| `crates/adapters`    | Postgres/sqlx, queues, email, external APIs                    | business decisions                                        |
| `crates/api`         | protocol boundary, request/response mapping, auth extraction   | product invariants not delegated to domain/application    |
| `apps/web`           | React views, routing, client state, generated API client usage | handwritten API shapes, DB access, business truth         |
| `contracts`          | OpenAPI/JSON Schema/proto source of truth                      | hand-edited generated clients                             |
| `db/migrations`      | schema, constraints, safe migrations                           | product behavior hidden in SQL without domain mirror      |
| `python/ai-service`  | model calls, embeddings, evals, feature extraction             | durable product truth, direct ownership of core DB tables |

The UX QA package is a great differentiator. Keep it, but make it installable and less brittle by default.

Current UX QA strengths:

```text
- deterministic Playwright geometry scan
- artifact-backed screenshot/crop/ARIA outputs
- checks for target size, edge clearance, overlap, clipping, wrapping, overflow, sticky obstruction, focus, labels, nested scrollbars, z-index
```

Current UX QA risks:

```text
- CLI uses `networkidle`, which can be bad for apps with websockets/long polling
- no auth/setup/action script support
- no config file
- no route/story discovery
- no `try/finally` around browser close
- edge-clearance may be noisy for valid nav/full-bleed controls
- sticky-obstruction flags fixed/sticky elements even when they are legitimate containing nav
- focus-visible only checks outline/box-shadow and may miss valid focus styles
- screenshots/crops can explode artifact count on large failure pages
```

## Paper improvements to make the standard land

The paper is compelling, but to maximize adoption, I would sharpen it in three ways:

1. **Add “Install contract: first 60 seconds.”**
   Show exactly what command to run, exactly what files appear, and exactly how Codex, Claude, Cursor, Copilot, Gemini, and Antigravity consume them.

2. **Add “Minimal Viable Standard” before the full standard.**
   Many teams will not adopt HL5 immediately. Define HL1 as five files plus one CI lane:

   ```text
   AGENTS.md
   agent/JANKURAI_STANDARD.md
   agent/owner-map.json
   agent/test-map.json
   agent/generated-zones.toml
   .github/workflows/jankurai.yml
   ```

3. **Make the stack rubric reproducible as data.**
   The Rust/TS/React/Vite/Postgres result is persuasive, but it will be more defensible if the paper’s ranking table is generated from:

   ```text
   paper/rubric/stack-rubric.toml
   paper/rubric/scores/*.toml
   ```

   Then `just paper` regenerates the table/figure. That turns the stack choice from “manifesto” into “auditable rubric.”

## Final priority order

Do these in this order:

1. **StandardKit Init** — because adoption must be one command.
2. **EvidenceGraph Audit** — because the score must be trustworthy and impossible to fake with placeholders.
3. **ProofPack** — because the standard needs a blessed stack, real UX QA, CI lanes, and issue repair loops.

That is the clean path to making jankurai the default standard for agent-optimized codebases: **one install, one stack, one proof graph, many IDE adapters, zero ambiguous human scatter.**
