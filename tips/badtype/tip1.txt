Absolutely ‚Äî here is the production-grade ‚Äúnot excusable‚Äù TypeScript bad-behavior list I would use for vibe-coded or AI-assisted TypeScript reviews.

One important framing point: in TypeScript, the unforgivable sin is usually **not** ‚Äúthe type is imperfect.‚Äù It is **hiding uncertainty, disabling checks, or pretending runtime data is safe when it has not been proven safe**. TypeScript‚Äôs own docs are very explicit that `strict` gives stronger correctness guarantees, `strictNullChecks` prevents runtime surprises around `null`/`undefined`, and type assertions/non-null assertions do not add runtime checks. ([TypeScript][1])

## The core rule

**Never ship TypeScript that gets green by lying.**

That includes lying to the compiler, lying to reviewers, lying to CI, lying to runtime validation, lying to tests, or letting an AI agent make code ‚Äúpass‚Äù by weakening the evidence.

# The highest-signal ‚Äúreject immediately‚Äù list

If I had to turn the exhaustive list into a PR-review kill switch, I would block immediately on these:

1. `any`, `as any`, or `unknown as T`.
2. `@ts-ignore` or `@ts-nocheck`.
3. TypeScript/lint config weakened in a feature PR.
4. `strict` disabled for new code.
5. `strictNullChecks` disabled.
6. Runtime input cast instead of validated.
7. Floating promise.
8. Non-null assertion on external or optional data.
9. Public API returning/accepting `any`.
10. Tests removed or weakened.
11. AI-generated code accepted without typecheck/lint/test evidence.
12. Security-sensitive code with casts, unchecked input, or swallowed errors.

GitLab‚Äôs own TypeScript guide is delightfully blunt here: configure strict type-safety rules for ESLint and TypeScript, use `"strict": true`, avoid `any` ‚Äúat all costs,‚Äù prefer `unknown` plus narrowing, and avoid casting with `<>` or `as` because it circumvents type safety. ([GitLab Docs][12])

# What is not automatically ‚Äúbad behavior‚Äù

These can be justified when deliberate and documented:

* `unknown` at real boundaries.
* `@ts-expect-error` in type tests.
* Narrow, documented suppressions during a controlled migration.
* Generated code in a clearly marked generated directory.
* `skipLibCheck` for dependency declaration performance, when your own source is still checked.
* Very small, reviewed interop wrappers around bad third-party types.
* Type assertions after a runtime check that TypeScript cannot express.
* A temporary migration bridge with owner, issue, and removal deadline.

The distinction is simple: **a boundary is acceptable; a lie is not.**
