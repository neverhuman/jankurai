Absolutely. Here is the production-grade “never excuse this in TypeScript” list I would use as a PR rejection checklist, especially for vibe-coded / AI-assisted TypeScript.

I’m interpreting **“never justified”** as: **never acceptable as a permanent state in production/shared code**. A tiny, isolated migration shim or test double might exist briefly, but only with a clear comment, owner, ticket, and removal plan. The bad behavior is making unsafe TypeScript normal.

TypeScript’s whole value is the feedback loop: static checking, editor tooling, refactoring confidence, and catching bugs before runtime. But TypeScript types are erased from emitted JavaScript, and type annotations do not change runtime behavior, so pretending a value is safe is not safety. ([TypeScript][1]) The official docs, typescript-eslint rules, Google’s TypeScript style guide, GitLab’s TypeScript guidance, Reddit/X community pain points, and AI-code security research all converge on the same theme: **bad TypeScript is usually code that lies, silences, widens, skips validation, or ships AI output without understanding it**. ([TypeScript ESLint][2])

## 1. Silencing TypeScript instead of fixing the problem

These are the clearest “no excuses” behaviors.

* Committing `// @ts-nocheck` in source files.
* Using `// @ts-ignore` to make a red squiggle disappear.
* Using `// @ts-expect-error` without a precise explanation.
* Leaving `@ts-expect-error` in place after the underlying error is gone.
* Disabling a lint rule at file level because one line is hard.
* Disabling type-aware linting because it exposes too many real problems.
* Changing `.ts` / `.tsx` files back to `.js` just to escape type errors.
* Excluding broken source folders from `tsconfig.json`.
* Making CI run tests but not `tsc --noEmit`.
* Allowing deployments when `tsc` reports errors.
* Using `noEmitOnError: false` in a deploy/build path so broken code still emits JavaScript.
* Turning off strict compiler options after the project has already adopted TypeScript.
* Changing types until the compiler shuts up without understanding why.
* Asking an AI agent to “fix TypeScript errors” and accepting a diff full of casts, `any`, or suppressions.
* Hiding errors inside wrapper functions named things like `unsafeCast`, `forceType`, `fixType`, or `asWhatever`.
* Treating TypeScript errors as “compiler drama” rather than design feedback.

typescript-eslint explicitly says suppressing TypeScript compiler errors reduces TypeScript’s effectiveness and that correcting the types is generally better; its default strict handling allows `@ts-expect-error` only with descriptions and reports `@ts-ignore`/`@ts-nocheck` depending on configuration. ([TypeScript ESLint][3]) Reddit TypeScript threads show the same community norm: prefer `@ts-expect-error` over `@ts-ignore` when a suppression is genuinely unavoidable, because stale suppressions can be detected. ([Reddit][4])

## 2. `any` pollution

`any` is the classic vibe-coding smell. Used lazily, it converts TypeScript back into JavaScript with extra ceremony.

Never accept:

* `let x: any`.
* Function parameters typed as `any`.
* Function return values typed as `any`.
* `Promise<any>`.
* `Array<any>` / `any[]`.
* `Record<string, any>`.
* `Map<string, any>`.
* `Set<any>`.
* `unknown as any`.
* `as any`.
* `as any as T`.
* `JSON.parse(...) as any`.
* API responses typed as `any`.
* Event payloads typed as `any`.
* Redux/Zustand/React state typed as `any`.
* Express/Fastify/Koa request bodies typed as `any`.
* Database rows typed as `any`.
* Config/env objects typed as `any`.
* Test mocks typed as `any` when `Partial<T>`, a builder, or a narrow mock type would work.
* Public package APIs exposing `any`.
* Generic defaults like `<T = any>`.
* Utility types that leak `any` through the whole codebase.
* “Temporary” `any` with no issue, owner, or removal path.
* `any` introduced by AI because it could not infer the real type.

typescript-eslint calls `any` a dangerous escape hatch that disables many type-checking rules, and recommends describing known types or using `unknown` when a type is not known. ([TypeScript ESLint][2]) GitLab’s TypeScript guide is even blunter: “Avoid `any` at all costs,” especially at domain boundaries, and use `unknown` plus narrowing/predicates instead. ([GitLab Docs][5]) A Reddit thread specifically about hitting a “vibe coding wall” in TypeScript describes AI-generated TypeScript with `ANY sprinkled everywhere`, which is exactly the failure mode this rule is meant to stop. ([Reddit][6])

## 3. Letting `any` leak through operations

Even when `any` enters from a library or old file, the unexcusable part is **letting it spread**.

Never accept:

* Assigning an `any` value to a typed variable.
* Calling an `any` value as a function.
* Accessing properties on an `any` value.
* Returning `any` from a function.
* Passing `any` as an argument to a typed function.
* Spreading `any[]` into function calls.
* Putting `any` inside generic positions like `Set<any>` where `Set<string>` is expected.
* Optional chaining on `any` and pretending it is now safe.
* Using `any` from external libraries without quarantining it at the boundary.
* Allowing `any` in reducers, middleware, RPC routers, API clients, or shared helpers.
* Letting `any` cross module boundaries.

The `no-unsafe-*` typescript-eslint rules exist specifically for these cases: unsafe assignment, unsafe member access, unsafe calls, unsafe arguments, and unsafe returns are all called out as type-safety holes and bug sources. ([TypeScript ESLint][7])

## 4. Misusing `unknown`

`unknown` is good only when it forces validation. It is bad when used as a laundering step.

Never accept:

* `const x = value as unknown as RealType` without proof.
* `unknown` immediately cast to the desired type.
* `unknown` stored in app state and used everywhere.
* `unknown` returned from public APIs when the function could validate and return a real type.
* `unknown` in error handlers with no narrowing.
* `unknown` used to hide incomplete modeling.
* `unknown` used as “safe any” while still skipping validation.
* `Record<string, unknown>` used for real domain objects that deserve a schema.
* Treating `unknown` as validated because it came from “our backend.”
* Treating generated OpenAPI/GraphQL types as a runtime guarantee.

Good `unknown` is temporary: it enters at a boundary, gets checked, then becomes a real type.

## 5. Lying with type assertions

Type assertions do not validate data. They just tell TypeScript to stop objecting.

Never accept:

* `(value as User)` for unvalidated API data.
* `(req.body as CreateUserInput)` without schema validation.
* `(JSON.parse(raw) as Config)` without validation.
* `as Foo` on object literals to dodge excess-property checks.
* `as Foo` on return objects instead of annotating the return type.
* `as unknown as Foo` without a documented invariant.
* `as never` to bypass exhaustiveness errors.
* `as const` used to freeze a wrong or over-specific structure instead of modeling it.
* Casting away `readonly`.
* Casting nullable values to non-nullable values.
* Casting partial mocks to full services in production code.
* Casting one domain ID to another, such as `userId as OrgId`.
* Casting external library results instead of writing a narrow adapter.
* Casting because “the tests pass.”
* Letting AI add casts in bulk.

Google’s TypeScript style guide says type assertions and non-null assertions are unsafe because they only silence the compiler and do not insert runtime checks; it recommends runtime checks when asserting type or non-nullability. It also warns that object-literal assertions can hide refactoring bugs that annotations would catch. ([Google GitHub][8]) GitLab similarly says casting explicitly circumvents type safety and recommends type predicates. ([GitLab Docs][5])

## 6. Abusing non-null assertions

The `!` operator is often just `any` wearing a smaller hat.

Never accept:

* `user!.id`.
* `ref.current!` without a lifecycle guarantee.
* `array.find(...)!.value`.
* `map.get(key)!`.
* `process.env.SECRET!`.
* `document.querySelector(...)!`.
* `params.id!`.
* `props.foo!`.
* `config.whatever!`.
* `promiseResult.data!`.
* Class fields declared with `!` to avoid initialization.
* Non-null assertions repeated across a file instead of one validated boundary.
* `!` added by AI to fix “possibly undefined” errors.
* `!` in code paths reachable from user input.
* `!` in async code where timing can change the invariant.

`strictNullChecks` exists because when null/undefined are ignored, TypeScript lets you write code that can fail at runtime; the docs show `Array.find` as a concrete example of a value that may be undefined. ([TypeScript][9])

## 7. Disabling strictness

A TypeScript project without strictness is often just “JavaScript with type cosplay.”

Never accept turning these off to make code pass:

* `strict`.
* `noImplicitAny`.
* `strictNullChecks`.
* `strictFunctionTypes`.
* `strictBindCallApply`.
* `strictPropertyInitialization`.
* `noImplicitThis`.
* `useUnknownInCatchVariables`.
* `noUncheckedIndexedAccess`.
* `exactOptionalPropertyTypes`.
* `noImplicitReturns`.
* `noFallthroughCasesInSwitch`.
* `noImplicitOverride`.
* `noPropertyAccessFromIndexSignature`.
* `forceConsistentCasingInFileNames`.
* `noUncheckedSideEffectImports`, where supported and applicable.
* `allowUnreachableCode: true`.
* `allowUnusedLabels: true`.
* `suppressImplicitAnyIndexErrors`.
* `suppressExcessPropertyErrors`.

`noImplicitAny` exists because TypeScript otherwise falls back to `any` when it cannot infer a type, which can miss errors. ([TypeScript][10]) `noUncheckedIndexedAccess` exists because unknown indexed fields otherwise look definitely present; enabling it adds `undefined` to undeclared/indexed access. ([TypeScript][11]) GitLab recommends `strict: true` for new TypeScript projects because it enforces the strongest type-checking capabilities and prohibits overriding type safety. ([GitLab Docs][5])

## 8. Misusing `skipLibCheck`

`skipLibCheck` is not always bad. It can be a practical compile-time setting. The unexcusable behavior is using it to hide real problems indefinitely.

Never accept:

* Using `skipLibCheck` to ignore broken hand-written `.d.ts` files in your own repo.
* Using it to avoid resolving duplicate incompatible versions of the same types.
* Using it to mask dependency tree corruption.
* Using it as the only fix after a TypeScript upgrade breaks declarations.
* Using it without knowing whether bad declarations affect your app.
* Using it to ignore generated types you publish to others.
* Treating `skipLibCheck` as a substitute for dependency hygiene.

The official docs say `skipLibCheck` saves time at the expense of type-system accuracy and recommend resolving duplicate library types rather than relying on the flag for that case. ([TypeScript][12]) GitLab recommends it for compile-time reasons in its own setup guidance, so the key distinction is whether it is a conscious build tradeoff or a garbage chute for broken types. ([GitLab Docs][5])

## 9. Skipping runtime validation at trust boundaries

This is one of the biggest TypeScript-specific traps: **types are not runtime validation**.

Never accept unvalidated data from:

* HTTP request bodies.
* Query strings.
* Route params.
* Headers.
* Cookies.
* JWT/session claims.
* Webhooks.
* Third-party APIs.
* Your own API if independently deployed/versioned.
* `JSON.parse`.
* `localStorage` / `sessionStorage`.
* `postMessage`.
* WebSocket messages.
* Message queues.
* IPC.
* CLI args.
* Environment variables.
* Config files.
* Feature flags.
* Database rows after schema drift.
* Cache values.
* URL params.
* Form data.
* Uploaded files.
* LLM/tool outputs.
* Generated code or generated JSON.

Never accept:

* `fetch(...).then(r => r.json() as User)`.
* `const env = process.env as Env`.
* `const body: CreateUserInput = req.body`.
* `JSON.parse(raw) as Settings`.
* Trusting client-side validation only.
* Assuming TypeScript protects API boundaries.
* Assuming OpenAPI/GraphQL generated types validate runtime data.
* Assuming Prisma/ORM types validate external input.
* Trusting `z.infer<T>` without actually calling `parse`/`safeParse`.
* Validating only happy-path fields.
* Ignoring unknown extra fields when they matter for security.
* Converting invalid data into partial objects and continuing.

The TypeScript docs are explicit that type annotations are erased and never change runtime behavior. ([TypeScript][1]) OWASP’s REST guidance says not to trust input parameters/objects and to validate length, range, format, and type. ([OWASP Cheat Sheet Series][13]) Zod’s docs frame schema parsing as the step that turns untrusted data into a validated, type-safe result. ([Zod][14])

## 10. Unsafe async and promise handling

Async mistakes are extremely common in AI-generated TypeScript because the code “looks right” but is semantically wrong.

Never accept:

* Floating promises.
* Missing `await`.
* `if (somePromise)` instead of `if (await somePromise)`.
* Unhandled promise rejections.
* `array.forEach(async item => ...)`.
* Async callbacks passed where return value is ignored.
* Async React event/effect code with unhandled errors.
* Fire-and-forget promises without explicit `void`, logging, cancellation, and error handling.
* `Promise.all` used when partial failure must be handled.
* `Promise.all` over unbounded user-sized arrays.
* Swallowed `.catch(() => {})`.
* `try/catch` around code that starts promises but does not await them.
* Tests that forget to `await expect(...).resolves/rejects`.
* API handlers that start background work and return success before work is durably queued.
* Mixing callbacks and promises in the same code path.
* Async constructors or initialization hidden behind half-initialized objects.
* Using sleeps/timeouts instead of awaiting deterministic signals.
* Ignoring `AbortSignal`.
* Not cleaning timers, subscriptions, sockets, or event listeners.

typescript-eslint’s `no-floating-promises` rule says floating promises lack error handling and can cause improperly sequenced operations and ignored rejections. ([TypeScript ESLint][15]) Its `no-misused-promises` rule forbids promises in logical locations where they are not handled properly, commonly due to a missing `await`. ([TypeScript ESLint][16])

## 11. Bad error handling

Never accept:

* `catch (e: any)`.
* Empty `catch {}` blocks.
* Logging and continuing after unrecoverable errors.
* Throwing strings.
* Throwing plain objects with no stack.
* Replacing specific errors with `throw new Error("failed")` and losing context.
* Ignoring `cause`.
* Returning `null` for errors in one function, throwing in another, and returning `{ error }` elsewhere with no convention.
* Treating validation errors as impossible.
* Swallowing parse errors and using defaults silently.
* Catching errors only to satisfy TypeScript.
* Logging secrets/tokens/PII in error objects.
* Sending raw internal errors to clients.
* Hiding async errors in detached promises.
* Converting all errors to `any`.
* Assuming `catch` values are `Error`.

## 12. Fake domain modeling

Bad TypeScript often compiles because the model is too vague.

Never accept:

* `type Status = string` when only specific statuses are allowed.
* `type Role = string`.
* `type UserId = string` interchangeable with `OrgId`, `PostId`, `SessionId`.
* Boolean flag explosions like `{ isAdmin, isOwner, isEditor, isViewer }`.
* Optional-field blobs representing multiple states.
* `interface Thing { type: string }` instead of discriminated unions.
* State machines represented as loose objects.
* Magic strings copied across files.
* Dates represented inconsistently as `Date`, ISO string, timestamp number, and nullable string.
* Money represented as floating-point numbers without currency/minor-unit rules.
* Permissions represented only in UI components.
* Domain invariants only in comments.
* `Partial<T>` used as a substitute for proper create/update DTOs.
* `Record<string, T>` used when keys are a known union.
* `object` / `{}` used instead of real shapes.
* Optional properties used because the AI does not know which state owns which fields.
* One “MegaType” shared across frontend, backend, DB, and API even though each layer has different invariants.

The TypeScript docs show discriminated unions and `never` exhaustiveness as the right way to model variant states and catch missing cases. ([TypeScript][17]) GitLab’s guide likewise recommends union types to improve inference and avoid casts. ([GitLab Docs][5])

## 13. Non-exhaustive logic

Never accept:

* `switch` over a union without exhaustiveness checking.
* `default: return null` hiding unhandled cases.
* Adding a new union member without updating all handlers.
* `if/else` ladders over literal states with no final `never` check.
* Reducers that ignore unknown actions.
* State machines that silently fall back to initial state.
* Permission checks with catch-all allow behavior.
* Feature flag variants with no default-deny path.
* Enum/string union conversions with unchecked casts.
* `as never` to make exhaustive checks pass.

## 14. Bad public API types

Never accept:

* Exported functions with inferred accidental return types when the return shape is part of the contract.
* Public APIs returning `any`, `unknown`, `{}`, `object`, or `Record<string, unknown>` without a reason.
* Package `.d.ts` files that lie about runtime behavior.
* `declare module "foo";` to silence missing types.
* `declare module "*";`.
* Global augmentations that affect the whole app unnecessarily.
* Re-exporting unstable internal types as public API.
* Changing exported types without versioning.
* Publishing generated types that were not produced by CI/codegen.
* Mismatched runtime exports and type declarations.
* Type-only breaking changes treated as harmless.
* Exporting broad DTOs that let callers construct invalid states.
* Public generics that accept impossible combinations.
* `Function`, `Object`, `String`, `Number`, or `Boolean` in public types.

Google’s style guide says wrapper types like `String`, `Boolean`, and `Number` should not be used and recommends more descriptive alternatives to `{}` because `{}` accepts any non-nullish primitive or object. ([Google GitHub][8]) ESLint also highlights `{}` as a TypeScript-specific blunder because it allows values such as `0` and `""`, and recommends `object` or `unknown` depending on intent. ([ESLint][18])

## 15. Generic type abuse

Never accept:

* Generics that do not relate input to output.
* `<T>(x: unknown) => x as T`.
* `<T = any>`.
* `<T extends object>` when primitives are valid or when the shape is known.
* Fake type-safe fetch helpers: `api.get<T>(url): Promise<T>` with no decoder.
* Generic repositories/services that erase domain constraints.
* Deep conditional types no one can debug.
* Recursive mapped types that destroy compiler performance.
* Over-abstracted utility types instead of a simple interface.
* Type-level programming to avoid writing runtime logic.
* `keyof any` where `PropertyKey` or a narrower key type is intended.
* `T as unknown as U` inside generic helpers.
* Generic “form builders” that allow impossible field/value pairs.
* Using `Partial<T>`, `Pick<T>`, and `Omit<T>` mechanically instead of designing boundary types.
* Overusing `ReturnType<typeof fn>` across module boundaries where an explicit contract is clearer.
* Hiding breaking changes behind utility types.

## 16. Unsafe indexed access

Never accept:

* `arr[0].id` without checking length.
* `record[key].value` when `key` may be absent.
* `map.get(key).value`.
* `object[someString]` without an index signature or key guard.
* Treating `process.env.X` as definitely present.
* Treating URL params as definitely present.
* Using `!` after every indexed access.
* Turning off `noUncheckedIndexedAccess` because it is “annoying.”
* Using `Record<string, T>` for maps where absence is possible but not modeled.
* Assuming non-empty arrays without a `NonEmptyArray` type or runtime check.
* Indexing tuple-like arrays as if every position exists.
* Relying on array order from external APIs.

## 17. Truthiness bugs

Never accept:

* `if (count)` when `0` is valid.
* `if (name)` when empty string is valid.
* `if (value)` for `number | undefined`.
* `if (obj.prop)` when `false`, `0`, or `""` are legitimate.
* `foo || default` when `foo` may validly be `0`, `false`, or `""`.
* `Boolean(value)` as validation.
* Truthy checks on promises.
* Truthy checks on arrays to mean “non-empty.”
* Truthy checks on objects to mean “has keys.”
* Optional chaining that masks missing required data.
* Nullish and falsy checks mixed casually.

Use explicit comparisons: `value !== undefined`, `value !== null`, `array.length > 0`, `typeof value === "number"`, and so on.

## 18. Unsafe object/dictionary patterns

Never accept:

* Using plain objects as maps with untrusted keys.
* Prototype pollution risks via `Object.assign(target, userInput)`.
* Spreading unvalidated input into domain objects.
* Accepting unexpected extra keys in security-sensitive objects.
* Using `{ [key: string]: any }`.
* Mutating shared config objects.
* Returning internal mutable objects from APIs.
* Casting `readonly` objects to mutable.
* Using `delete` to change object shapes instead of modeling variants.
* Relying on property order.
* Deep cloning with `JSON.parse(JSON.stringify(...))` when Dates, Maps, Sets, `undefined`, BigInt, or class instances matter.
* Comparing objects by `JSON.stringify`.
* Confusing absent property with property set to `undefined`.
* Ignoring `exactOptionalPropertyTypes`.

## 19. Unsafe module/import/build behavior

Never accept:

* Runtime path aliases that only TypeScript understands.
* Build succeeds but runtime cannot resolve imports.
* Type-only imports emitted as runtime imports by mistake.
* Value imports used only as types.
* Side-effect imports nobody checks.
* Circular dependencies that produce partially initialized modules.
* Mixing ESM/CJS randomly.
* Relying on `esModuleInterop` to hide incorrect import semantics without understanding runtime output.
* Import casing that works on macOS but fails on Linux.
* `require` inside typed ESM code without a boundary.
* Dynamic imports from user input.
* Barrel files that accidentally create cycles or huge import graphs.
* Multiple versions of the same type package producing incompatible nominal-ish types.
* Committing generated JavaScript from TypeScript source when it is not meant to be source-controlled.
* Source maps exposed with secrets or private source paths in production.

Never accept:

* `children: any`.
* Event handlers typed as `any`.
* `useState<any>`.
* `useRef<any>`.
* Context values typed as `any`.
* Reducer actions typed as `{ type: string; payload?: any }`.

## 21. Bad Node/API TypeScript

Never accept:

* `req.body as SomeType` without validation.
* `req.params.id as string` without checking.
* `process.env as Env`.
* Hardcoded secrets.
* Secrets in client bundles.
* Logging authorization headers, cookies, tokens, or PII.
* Raw SQL string interpolation.
* NoSQL query objects built directly from user input.
* Shell commands built from user input.
* File paths built from user input without normalization/allow-listing.
* `eval`, `new Function`, or dynamic code execution on untrusted input.

* Authorization checks only in frontend code.
* CORS `*` with credentials.
* Rate limits omitted from sensitive endpoints.
* Server-side validation omitted because “the form already validates.”
* Accepting webhook payloads without signature verification.
* Treating TypeScript interfaces as API security.

OWASP’s SQL guidance recommends parameterized queries for SQL injection prevention, its REST guidance says to validate input type/length/range/format, and its secrets guidance focuses on proper centralized management rather than leaking secrets into code/logs. ([OWASP Cheat Sheet Series][19])

## 23. Security-critical TypeScript lies

Never accept:

* Types that imply a user is authenticated before auth is checked.
* `user as AuthenticatedUser`.
* Client-provided `role`, `isAdmin`, `ownerId`, `price`, `credits`, or `plan`.
* Trusting hidden form fields.
* Trusting localStorage for permissions.
* Trusting decoded JWT payload without verifying signature and claims.
* Client-side feature gates as security.
* Stringly typed permissions.
* Permission checks duplicated inconsistently.
* Missing default-deny behavior.
* Authorization logic hidden in UI components.
* Audit logs typed as optional and silently skipped.
* Payment/subscription state controlled client-side.
* “Admin mode” flags in frontend-only state.

This is a frequent AI/vibe-coding failure: the app appears to work, but authority is on the wrong side of the network boundary.

## 28. Vibe-coding-specific bad behavior

This is the AI-assisted section. These are all PR rejection reasons.

Never accept:

* Merging code you cannot explain.
* Merging code the AI cannot explain coherently.
* Merging code without reading the diff.
* Asking AI to “continue” repeatedly until the app works.
* Letting AI rewrite unrelated files.
* Letting AI change architecture to fix a local bug.
* Letting AI add `any`, casts, suppressions, or optional chaining as a “fix.”
* Letting AI duplicate existing helpers/types instead of importing them.
* Letting AI invent APIs, config flags, package functions, or framework behavior.
* Letting AI add dependencies without review.
* Letting AI generate security-sensitive code without human threat modeling.
* Letting AI write auth/payment/crypto code unsupervised.
* Letting AI write migrations without review.
* Letting AI update schemas without runtime compatibility checks.
* Letting AI create giant files because “it works.”
* Letting AI leave TODOs as design.
* Letting AI produce over-general abstractions.
* Letting AI produce under-modeled blobs.
* Letting AI generate tests that only pass the generated code.
* Letting AI “fix” failing tests by weakening assertions.
* Letting AI remove tests.
* Letting AI change tsconfig/eslint to pass.
* Letting AI ignore TypeScript errors.
* Letting AI flatten all errors into `try/catch`.
* Letting AI create fake fallback behavior instead of handling the real case.
* Letting AI use client-side state for server-side truth.
* Letting AI hide data-shape uncertainty under `as`.
* Letting AI turn every nullable thing into optional chaining.
* Letting AI turn every hard edge into `any`.
* Treating “compiled” as “correct.”
* Treating “the UI rendered” as “the system is safe.”

A GitHub community discussion draws the same line between wise AI use and vibe coding: wise use requires understanding, reviewing, and validating the generated code; vibe coding skips that scrutiny and can introduce fragile, unmaintainable, insecure code. ([GitHub][20]) Recent repository-mining research on AI-generated code found thousands of CWE instances across AI-attributed public GitHub files and recommends language/context-aware security practices for responsible integration. ([arXiv][21])

## 29. Giant, unmaintainable files

Never accept:

* 700-line components.
* 1,000-line service files.
* Files with unrelated responsibilities.
* One file containing types, validation, database, UI, and business logic.
* Deeply nested conditionals generated by AI.
* Copy-pasted functions with tiny variations.
* Multiple sources of truth for the same type.
* Repeated inline object types instead of named domain concepts.
* Repeated validation logic instead of shared schemas.
* “Utils” files that become junk drawers.
* Circular imports caused by poor decomposition.
* No clear ownership boundary between modules.
* No meaningful naming.
* Code that only the original prompt can explain.

## 30. Optional chaining as duct tape

Never accept optional chaining used to suppress real invariants.

Bad:

* `user?.profile?.email` where email is required after login.
* `order?.items?.[0]?.price` in payment logic.
* `permissions?.includes("admin")` in auth logic without default-deny clarity.
* `config?.apiKey` where startup should fail if missing.
* `data?.foo?.bar` repeated everywhere instead of validating `data`.
* Optional chaining in reducers to hide invalid state.
* Optional chaining in tests to avoid crashes.
* Optional chaining added by AI to eliminate null errors.

Optional chaining is fine for genuinely optional data. It is bad when it hides a contract violation.

## 31. Bad naming and misleading types

Never accept:

* `data`, `payload`, `obj`, `thing`, `stuff` for domain values.
* `IUser` meaning API user in one place and DB user in another.
* `User` reused for public profile, auth session, database row, and admin view.
* `Config` for both raw env and validated config.
* `FooType`, `FooInterface`, `FooModel` with no semantic distinction.
* Types named after implementation rather than domain.
* `Response` shadowing DOM/Fetch response concepts.
* `Error` types that are not errors.
* `Nullable<T>` used everywhere instead of modeling why null occurs.
* Comments contradicting types.
* Types that describe what the code wishes were true.

## 32. Mutability hazards

Never accept:

* Mutating function inputs without naming/documenting it.
* Mutating objects from React state.
* Mutating cached/shared objects.
* Exporting mutable singletons.
* Casting `readonly` away.
* Returning mutable internal arrays/objects.
* Sorting arrays in place when callers expect original order.
* Reusing `Date` objects and mutating them.
* Global mutable test state.
* Mutable default config shared across requests.
* Shared request-scoped state in module globals.
* Hidden mutation inside validators/parsers.
* Mutating objects after validation so their type no longer reflects reality.

## 33. Bad date/time/number handling

Never accept:

* Money as float without rounding/minor-unit policy.
* `Date.parse` on ambiguous strings.
* Timezone assumptions hidden in code.
* Comparing date strings with mixed formats.
* Storing local time when UTC is required.
* Treating timestamps as seconds in one place and milliseconds in another.
* Using `number` for branded quantities that must not mix.
* Ignoring `NaN`.
* Ignoring `Infinity`.
* Parsing numbers with `Number(value)` and accepting `"" -> 0`.
* Type aliasing `type Dollars = number` but mixing cents/dollars.
* AI-generated date logic without tests around DST/timezones.

## 34. Bad forms and user input

Never accept:

* Form values typed as final domain objects before parsing.
* Treating all form inputs as strings and casting later.
* Client-only validation.
* Missing server validation.
* Not trimming/canonicalizing when domain requires it.
* Trimming when domain must preserve input exactly.
* Treating empty string, null, and undefined interchangeably.
* Ignoring multi-select/file edge cases.
* Type assertions on `FormData.get`.
* No validation for file type, size, and content.
* Trusting browser-provided MIME types.
* Generated forms that submit fields not allowed by the backend.

## 35. Bad configuration/env handling

Never accept:

* Reading `process.env.X!` throughout the app.
* No startup validation.
* No distinction between raw env and validated config.
* Config values parsed at every use site.
* Boolean env parsed by truthiness.
* Number env parsed without range checks.
* Defaults that hide missing production secrets.
* Client-exposed env vars with secrets.
* Config drift between local, CI, and production.
* `as Config` on env/config files.
* Feature flags typed as plain strings.
* Unused config silently ignored.
* Required config optional in types.

## 36. Bad comments and TODOs

Never accept:

* `TODO fix types`.
* `// @ts-ignore TODO`.
* `// temporary` with no date/owner/ticket.
* Comments explaining what the type system should express.
* Comments claiming validation happened elsewhere when it did not.
* Comments that rationalize unsafe casts without proof.
* Dead comments from AI output.
* Large blocks of generated explanation instead of clear code.
* Misleading JSDoc on public APIs.
* TODOs in security-sensitive paths.
* TODOs that survive review.

## 37. Bad CI/repo policy

Never accept:

* No `tsc --noEmit` in CI.
* No lint in CI.
* No type-aware linting for TypeScript.
* No tests for changed code.
* CI that runs on different tsconfig than local.
* CI that ignores generated type drift.
* CI that ignores schema/codegen drift.
* Warnings allowed forever.
* TypeScript version differs across packages without reason.
* Multiple tsconfigs with inconsistent strictness.
* Monorepo packages importing source across boundaries without declared dependencies.
* Package builds that pass independently but fail as consumers.
* No code owner for shared types/schemas.
* No dependency update strategy.
* No lockfile integrity.

typescript-eslint strongly recommends type-aware linting despite the performance cost because the rules are more powerful, and ESLint’s own guidance says ESLint and TypeScript catch different classes of defects and should be used together in TypeScript projects. ([TypeScript ESLint][22])

## 38. Bad monorepo TypeScript

Never accept:

* One root tsconfig that accidentally includes everything.
* Packages importing each other via relative `../../` paths.
* Runtime dependency not listed in `package.json`.
* Type dependency only working because of hoisting.
* Package boundary violations.
* Circular workspace dependencies.
* Different packages using incompatible `strict` settings.
* Generated types committed in one package but regenerated from another.
* Path aliases that do not match package exports.
* Package exports missing `types`.
* Source imports from packages that should consume built declarations.
* No project references or equivalent build graph in large repos.
* Typecheck too slow, so people stop running it.

TypeScript project references exist to split large TypeScript programs into smaller pieces, improve build times, and enforce logical separation between components. ([TypeScript][23])

## 39. Bad “fixes” to TypeScript errors

These are especially common from AI agents.

Never accept a “fix” that:

* Replaces a real type with `any`.
* Replaces a real type with `unknown` and never narrows it.
* Adds `!`.
* Adds `as SomeType`.
* Adds `as never`.
* Adds optional properties everywhere.
* Makes required fields optional.
* Makes union discriminants plain `string`.
* Deletes a branch.
* Deletes an assertion.
* Deletes a test.
* Changes tsconfig.
* Changes ESLint config.
* Adds `@ts-ignore`.
* Wraps code in `try/catch` and swallows the error.
* Adds `|| ""`, `|| []`, or `|| {}` without domain justification.
* Converts compile-time failure into runtime fallback.
* Moves the error somewhere else.
* Creates a duplicate type instead of importing the canonical one.

## 40. Bad review behavior

Never accept:

* “Looks fine” review on a large AI-generated diff.
* Reviewing only the UI behavior.
* Not checking type changes.
* Not checking generated files.
* Not checking dependency additions.
* Not checking security boundaries.
* Not checking runtime validation.
* Not checking error paths.
* Not checking whether removed code was intentional.
* Not checking whether types match runtime behavior.
* Approving suppressions without explanation.
* Approving `any` because “we can fix it later.”
* Approving code the author cannot explain.
* Approving code that only passes because strictness was weakened.

## 41. Security smells that TypeScript does not forgive

Never accept:

* Hardcoded credentials.
* Secrets committed to repo.
* Secrets logged.
* Secrets sent to client.
* `eval`.
* `new Function`.
* Unsafe `innerHTML`.
* Unsafe `dangerouslySetInnerHTML`.
* Raw SQL interpolation.
* Shell command interpolation.
* Path traversal risks.
* SSRF-prone URL fetching.
* Prototype pollution via untrusted objects.
* Insecure random for tokens.
* Weak crypto because AI suggested it.
* Password handling without a vetted library.
* JWTs decoded but not verified.
* Missing CSRF protection where relevant.
* Missing authz on server routes.
* Trusting TypeScript types as validation/security.

## 42. The “vibe-coded TypeScript smell list”

When scanning a repo, these are the fingerprints that code was vibe-coded badly:

* `any` appears in normal app code.
* `@ts-ignore` appears.
* `as unknown as` appears.
* `!` appears repeatedly.
* `TODO fix types` appears.
* `JSON.parse(...) as X`.
* `fetchJson<T>()` with no schema.
* `process.env.X!`.
* `Record<string, any>`.
* Giant components/services.
* Duplicate interfaces with similar names.
* Types defined far away from the runtime parser.
* Optional properties everywhere.
* Discriminants typed as `string`.
* Catch blocks that swallow.
* Tests removed or weakened.
* New dependencies for trivial tasks.
* Server/client boundary confusion.
* Auth logic in UI only.
* Generated tests that only test happy path.
* Lint/tsconfig loosened in the same PR.
* Code compiles but nobody can explain the invariants.

## Minimum enforcement baseline

At minimum, a serious TypeScript repo should have:

* `strict: true`.
* `noImplicitAny: true`.
* `strictNullChecks: true`.
* `noUncheckedIndexedAccess: true`.
* `exactOptionalPropertyTypes: true`.
* `noImplicitReturns: true`.
* `noFallthroughCasesInSwitch: true`.
* `noImplicitOverride: true`.
* `noPropertyAccessFromIndexSignature: true`.
* `forceConsistentCasingInFileNames: true`.
* `noEmitOnError: true` for build/deploy paths.
* `tsc --noEmit` in CI.
* `@typescript-eslint/recommended-type-checked` or stricter.
* `no-explicit-any`.
* `no-unsafe-assignment`.
* `no-unsafe-member-access`.
* `no-unsafe-call`.
* `no-unsafe-argument`.
* `no-unsafe-return`.
* `ban-ts-comment`.
* `no-floating-promises`.
* `no-misused-promises`.
* Runtime schemas at every trust boundary.
* Human review of all AI-generated diffs.
* Tests for success, failure, null/undefined, invalid input, and authz.

The shortest possible rule is this: **do not let TypeScript become decorative**. If the code lies to the compiler, disables the compiler, skips runtime validation, weakens the model, or ships AI output nobody understands, it is not excusable TypeScript.
