# Phase 08: Agent Context And Repair

Status: complete
Owner: agent
Last reviewed: 2026-05-02
Parallel MCP candidate: yes

## Objective

Make Jankurai the best control plane for coding agents. Agents should receive bounded context, bounded edit authority, exact proof requirements, and repair packets that prevent broad guessing.

The exit state is a context and repair system that turns audit findings into executable agent tasks while preserving human control over risky actions.

## Current State

Existing surfaces:

- Root `AGENTS.md`, `CLAUDE.md`, `GEMINI.md`, Cursor rules, Copilot instructions, `.agents/` and `.claude/skills/` adapters exist or are generated by init.
- `jankurai adapters verify` and `adapters sync` exist.
- `jankurai agent verify`, `jankurai context-pack`, and `jankurai repair-plan` exist as control-plane commands; JSON outputs for context-pack and repair-plan are **schema-validated on write** (and before stdout when `--out` is omitted).
- Findings include agent repair text.
- `agent_fix_queue` exists in JSON and Markdown reports.
- `issues export` can emit Markdown, JSONL, Beads-ish, and GitHub CLI commands.
- Shared schemas exist for context packs, repair packets, repair plans (v1 envelope matches CLI output), and permission profiles.
- `jankurai doctor` validates conventional paths `target/jankurai/context-pack.json` and `target/jankurai/repair-plan.json` when those files are present.

Gaps:

- Repair queue items still need richer owner-aware scoping and stop conditions.
- Permission profiles exist as schema, but lane enforcement is still advisory only.
- Prompt-injection and overbroad agency checks need sharper source hierarchy.
- No MCP/server tool surface yet.
- Context pack schema does not yet carry explicit token budget fields; repair plan schema does not yet include deferred planner-only fields (`plan_mode`, `planned_edits`, etc.).

## Workstreams

### 1. Adapter Canonicalization

Implementation tasks:

- Ensure all agent adapters are thin pointers to canonical standard docs.
- Add adapter verification tests for Codex, Claude, Cursor, Copilot, Gemini, and generic agents where files exist.
- Keep root and adapter files short.
- Add generated marker policy so adapters can be refreshed safely.

Acceptance:

- `jankurai adapters verify` catches adapter drift.
- No adapter contains contradictory policy.
- Durable policy lives in `agent/` or `docs/`.

### 2. Context Pack

Implementation tasks:

- Build context pack from owner map, test map, generated zones, proof lanes, rule metadata, and optional changed paths.
- Start with deterministic path/task matching, not embeddings.
- Include read-first docs and files.
- Include forbidden generated zones and read-only paths.
- Include token budget hints.
- Output JSON and Markdown.

Acceptance:

- A changed Rust CLI task gets Rust files, relevant rules, and cargo tests.
- A UX task gets UX package files and UX validation.
- A docs task avoids loading large generated artifacts.
- Pack output stays token-minimized and honors forbidden/generated paths.

### 3. Repair Plan

Implementation tasks:

- Expand `agent_fix_queue` into repair packets.
- Include safe edit scope and proof commands.
- Include "ask human" triggers for destructive migrations, security secrets, ambiguous product intent, generated code source uncertainty, and broad architecture changes.
- Add JSONL export for parallel agents.

Acceptance:

- Each repair packet is executable by a fresh coding session.
- High-risk packets clearly require human review.
- Repair plans preserve fingerprints for dedupe.
- Repair packets carry scoped proof and rollback guidance.

### 4. Permission Profiles

Implementation tasks:

- Define permission profile schema for agent lanes:
  - read-only
  - docs-only
  - code-edit
  - generated-regeneration
  - security-investigation
  - release
- Define forbidden destructive commands by default.
- Map rule/lane to profile.
- Add audit findings for overbroad agency language.

Acceptance:

- No task asks an agent for broader authority than needed.
- Permission profiles are explicit enough for future MCP tooling.

### 5. Prompt Injection Defense

Implementation tasks:

- Define trusted source hierarchy.
- Mark untrusted content zones: issue text, PR comments, logs, webpages, model outputs, copied prompts, fixtures with hostile text.
- Add docs and rule checks for bypass language in trusted files.
- Add repair guidance for isolating untrusted content.

Acceptance:

- Trusted policy files cannot be overridden by untrusted content.
- Prompt-injection findings are specific and repairable.

### 6. MCP/Server Surface

Implementation tasks:

- Design, but do not overbuild, a future MCP server contract.
- Candidate resources: report, findings, owner map, test map, generated zones, context pack, proof plan.
- Candidate tools: audit, lane plan, context pack, repair-plan, explain rule.
- Keep CLI source of truth.

Acceptance:

- MCP design doc exists.
- No server is required for local CLI use.

## Risks

- Context packs can become too broad and defeat token savings.
- Auto-repair framing can imply unsafe autonomy if permission profiles are vague.
- Agent adapters can drift if hand-maintained.
