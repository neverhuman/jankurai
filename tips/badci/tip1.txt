Below is a comprehensive CI “bad behavior” catalog you can use for testing, reviews, CI-policy scanners, red-team exercises, or agent evaluation. I’m treating **BAD behavior** as anything that should cause a CI design, CI runner/agent configuration, or `.git*.yml` change to be blocked, flagged, quarantined, or explicitly risk-accepted.

A useful framing: modern CI/CD is part of the software supply chain, not “just automation.” OWASP calls out CI/CD pipelines as a major attack surface because they connect source code, build automation, runners/nodes, deployment systems, credentials, and release procedures; NIST SP 800-204D similarly focuses on integrating supply-chain security measures directly into CI/CD pipelines. ([OWASP Cheat Sheet Series][1])

One source note: I weighted **official docs, security publications, incident reports, and academic/industry research** more heavily than social posts. Reddit was useful for community pain points around CI ergonomics and scaling; Quora and X posts are less reliable as citable primary evidence, so I would not use them as authoritative references for policy.

# 1. Executive taxonomy: the biggest classes of bad CI behavior

## A. Trust-boundary failures

These are usually the most dangerous CI mistakes.

### Bad behavior

1. **Running untrusted pull-request code with secrets available.**
   This includes fork PRs, external contributors, generated branches, bot-authored branches, or branches from compromised developer accounts.

2. **Using GitHub `pull_request_target` to build, install, test, or execute code from the pull request.**
   GitHub Security Lab’s “pwn requests” guidance is very explicit: `pull_request_target` has access to write permissions and secrets, and is intended for limited privileged tasks such as labeling/commenting, not for running untrusted PR code. ([GitHub Security Lab][2])

3. **Using `workflow_run` as a privilege bridge without validating artifacts.**
   A common pattern is: unprivileged PR workflow uploads an artifact, privileged `workflow_run` downloads it, then executes a script or binary from that artifact. That recreates the original problem in two steps.

4. **Checking out attacker-controlled code in a privileged workflow.**
   Example: `pull_request_target` plus `actions/checkout` with `ref: ${{ github.event.pull_request.head.sha }}` and then `npm install`, `make test`, `pytest`, `gradle`, or `docker build`.

5. **Treating “internal PR” as trusted by default.**
   A compromised employee account, stale bot token, or malicious insider can abuse CI. “Internal” is not the same as “safe.”

6. **Allowing untrusted code to influence privileged jobs through files.**
   Examples: PR modifies `.npmrc`, `package.json` scripts, `Makefile`, `tox.ini`, `setup.py`, Gradle/Maven scripts, Dockerfile, test harnesses, generated shell scripts, or test fixtures consumed by later privileged jobs.

7. **Running default-branch scheduled jobs that execute repository code after unreviewed changes.**
   Scheduled workflows often have stronger credentials and are easy to overlook.

8. **Allowing PR titles, branch names, issue comments, commit messages, labels, or usernames to flow directly into shell scripts.**
   GitHub explicitly warns that contexts such as titles, bodies, refs, emails, and other event fields should be treated as untrusted input and not inserted directly into code execution contexts. ([GitHub Docs][3])

9. **Using comment-triggered workflows such as `/test`, `/deploy`, or `/rerun` without authorization checks.**
   Bad variants include accepting commands from anyone, from anyone with prior comment history, or from a compromised bot.

10. **Letting CI jobs from low-trust branches write to high-trust locations.**
    Examples: pushing to `main`, updating release tags, publishing packages, uploading “latest” artifacts, modifying GitHub releases, writing to package registries, or updating deployment manifests.

11. **Using a single pipeline for both untrusted validation and privileged release work.**
    A single monolithic pipeline tends to accumulate secrets, write permissions, deployment logic, and attacker-controlled input in one place.

12. **Trusting CI configuration generated by code from the same untrusted branch.**
    Dynamic pipelines, included YAML, or generated workflow files can become a policy bypass if the branch being tested controls the policy that tests it.

13. **Allowing pull requests to modify CI policy without mandatory review from CI/security owners.**
    `.github/workflows/**`, `.gitlab-ci.yml`, `bitbucket-pipelines.yml`, reusable workflow files, runner configuration, dependency lockfiles, Dockerfiles, and release scripts should be treated as sensitive code.

14. **Assuming “tests only” jobs are safe.**
    Tests execute arbitrary code. In most languages, dependency installation and test discovery can run arbitrary code before any test actually starts.

15. **Using untrusted test data as executable configuration.**
    Examples: test fixtures interpreted as shell, YAML, Jinja, Groovy, Gradle, Maven, npm scripts, Docker Compose, Terraform, Helm, or Kubernetes manifests.

OpenSSF’s GitHub workflow attack-vector guidance specifically highlights privileged workflow misuse, injections, malicious or vulnerable actions, unsafe caches, and tag-renaming risks as recurring CI threats. ([OpenSSF][4])

## B. Secrets, credentials, and token misuse

### Bad behavior

1. **Hardcoding secrets in CI YAML.**
   Examples: cloud keys, deploy tokens, npm tokens, PyPI tokens, Docker registry passwords, SSH private keys, kubeconfigs, signing keys, database passwords, Slack webhooks, Sentry tokens, license keys, and internal API tokens.

2. **Setting secrets at workflow/global scope.**
   Bad: putting secrets in top-level `env:` so every job, step, tool, dependency, and test can read them.

3. **Providing secrets to jobs that only need to run tests.**
   Unit tests should almost never need production credentials.

4. **Using long-lived cloud credentials instead of short-lived identity federation.**
   Static AWS/GCP/Azure keys in CI are high-value targets.

5. **Using personal access tokens where a scoped automation token would work.**
   PATs are often overbroad, long-lived, and tied to a human account.

6. **Using admin/org-owner tokens in CI.**
   A test pipeline should not have organization administration capability.

7. **Using deploy keys with write access when read-only is sufficient.**

8. **Sharing one credential across projects, environments, or runners.**
   Shared credentials make blast radius huge and incident response difficult.

9. **Failing to mask, hide, protect, or scope CI variables.**
   GitLab warns that CI/CD variables are convenient but less secure than a secrets manager, can be overridden, and can be exposed if not handled properly. ([GitLab Docs][5])

10. **Assuming masked secrets cannot leak.**
    Secrets can leak via encoding, chunking, command traces, artifacts, screenshots, test reports, crash dumps, dependency logs, package-manager debug output, or memory scraping.

11. **Printing environment variables.**
    Bad commands include `env`, `printenv`, `set`, `export`, `docker inspect`, `kubectl describe secret`, verbose package-manager logs, or test frameworks that dump process environments.

12. **Using `set -x` or shell debug mode in jobs with secrets.**

13. **Writing secrets into artifacts.**
    Examples: `.env`, `.npmrc`, `.pypirc`, `.docker/config.json`, kubeconfigs, Terraform state, coverage reports with env dumps, browser traces, screenshots, crash dumps, and JUnit attachments.

14. **Writing secrets into caches.**
    Cache poisoning and cache leakage become much worse when caches include credentials.

15. **Passing secrets through child-pipeline inputs or shared variables that are logged in plaintext.**
    Bitbucket documents that certain child-pipeline input variables are logged in plain text and do not support secured variables. ([Atlassian Support][6])

16. **Using structured secrets that masking systems cannot reliably redact.**
    Multi-line JSON, YAML, certificates, and PEM files often leak partially.

17. **Failing to rotate secrets after CI compromise.**
    CI incidents should trigger credential rotation, log/artifact review, cache invalidation, runner reprovisioning, and package-token revocation.

18. **Keeping logs and artifacts forever.**
    Long retention increases the window for secret discovery.

19. **Letting any user with write access read or indirectly exfiltrate repository/workspace variables.**
    This is especially dangerous in Bitbucket/GitLab-style project variable models.

20. **Using production secrets for test environments.**

21. **Using real customer data in tests.**

22. **Giving CI direct access to production databases, production message queues, production object stores, or production Kubernetes clusters.**

23. **Allowing test jobs to mint deployment credentials.**

24. **Allowing secrets in local runner host files.**
    Examples: `/home/runner/.aws/credentials`, `/root/.docker/config.json`, SSH agent sockets, cloud metadata credentials, kubeconfigs, or mounted host directories.

GitHub’s secure-use guidance recommends least privilege for `GITHUB_TOKEN`, secret masking/rotation/auditing, and protected environments with approvals for sensitive secrets. ([GitHub Docs][7])

## C. Permission and authorization failures

### Bad behavior

1. **Using broad default token permissions.**
   Bad GitHub example: omitting `permissions:` and allowing unintended write access.

2. **Using `permissions: write-all`.**
   This is almost never appropriate for tests.

3. **Granting `contents: write` to jobs that only read code.**

4. **Granting `pull-requests: write` to jobs that run untrusted code.**

5. **Granting package publish permissions to test jobs.**

6. **Allowing Actions or bots to create and approve pull requests.**
   GitHub specifically recommends preventing GitHub Actions from creating or approving pull requests where that could bypass review. ([GitHub Docs][7])

7. **Using one CI service account for everything.**
   Build, test, publish, deploy, release, infrastructure, and admin tasks should not share one principal.

8. **No environment-level approvals for production secrets or deploys.**

9. **No protected branches or protected tags.**

10. **Deployment jobs runnable from arbitrary branches.**

11. **Release jobs runnable from unprotected tags.**
    Mutable or attacker-created tags are a common supply-chain footgun.

12. **No CODEOWNERS or required reviewers for CI files.**

13. **Letting bot-authored PRs bypass CI/security review.**

14. **Allowing workflow dispatch inputs to select arbitrary refs, scripts, images, environments, or commands.**

15. **Allowing user-controlled paths in upload/download artifact steps.**

16. **Using overly broad cloud trust policies.**
    Bad OIDC pattern: cloud role trusts all repos in an org, all branches, all workflows, or all environments.

17. **Missing separation between staging and production deployment credentials.**

18. **Letting one compromised test job affect all later jobs.**
    GitHub notes that individual jobs and third-party actions can compromise other jobs in the same workflow depending on permissions and shared state. ([GitHub Docs][7])

## D. Runner, executor, and agent-host compromise behavior

### Bad behavior

1. **Running untrusted code on self-hosted runners.**
   GitHub-hosted runners are ephemeral, while self-hosted runners can be persistently compromised if untrusted code runs on them. ([GitHub Docs][7])

2. **Using persistent self-hosted runners without complete cleanup.**
   Leftover workspaces, caches, tool directories, Docker layers, home directories, and temp files become cross-job attack paths.

3. **Using the same runner for trusted and untrusted jobs.**

4. **Using the same runner for different tenants, organizations, business units, or sensitivity levels.**

5. **Using GitLab shell executor for untrusted jobs.**
   GitLab says shell executor is generally unsafe because jobs can steal code from other projects or run arbitrary commands as a privileged user. ([GitLab Docs][8])

6. **Leaving SSH keys or deployment credentials on the runner host.**
   GitLab runner security guidance warns to secure the runner host, remove SSH keys, and treat malicious CI jobs as capable of compromising the host. ([GitLab Docs][9])

7. **Mounting the Docker socket into jobs.**
   Docker socket access is effectively host-root in many configurations.

8. **Adding CI users to the `docker` group.**
   This often allows host compromise through containers.

9. **Using privileged Docker-in-Docker for untrusted builds.**
   GitLab documents that privileged mode disables container security mechanisms and can enable privilege escalation or container breakout. ([GitLab Docs][10])

10. **Running CI containers as root unnecessarily.**

11. **Allowing hostPath mounts in Kubernetes runners.**

12. **Using cluster-admin Kubernetes service accounts for CI pods.**

13. **Running CI pods in the same Kubernetes namespace as production workloads.**

14. **No Kubernetes NetworkPolicy or egress control for runners.**

15. **Allowing CI jobs to reach internal networks by default.**

16. **Allowing CI jobs to reach cloud metadata endpoints.**

17. **Allowing arbitrary outbound internet from privileged jobs.**

18. **No CPU, memory, disk, process, or runtime limits.**

19. **No job timeout.**

20. **No cleanup of Docker images, volumes, buildx builders, or service containers.**

21. **No runner patching process.**

22. **No runner version pinning or upgrade tracking.**

23. **No runner attestation or inventory.**

24. **No logging of which runner executed which job.**

25. **No quarantine or reprovisioning after suspicious jobs.**

26. **Using public shared runners for sensitive closed-source or regulated workloads without understanding data exposure and compliance constraints.**

27. **Using internal self-hosted runners for public pull requests.**
    This is one of the highest-risk patterns.

28. **Using generic runner labels such as `self-hosted` without more restrictive labels.**
    Jobs may land on more privileged machines than intended.

29. **Letting jobs choose arbitrary runner labels from user-controlled input.**

30. **Running CI directly on the Git hosting server.**
    The GitBucket CI plugin explicitly warns not to use it in public because it allows arbitrary commands to run on the GitBucket instance and is not flexible/scalable compared with a dedicated CI system. ([GitHub][11])

## E. Third-party actions, components, pipes, images, and remote scripts

### Bad behavior

1. **Using third-party GitHub Actions without pinning to a full commit SHA.**
   GitHub recommends pinning actions to a full-length commit SHA because tags can be moved or changed. ([GitHub Docs][7])

2. **Using `@main`, `@master`, `@latest`, or mutable major-version tags for critical actions.**

3. **Using third-party actions with write tokens or secrets unless absolutely necessary.**

4. **Using actions from unverified, abandoned, newly created, or typo-squatted maintainers.**

5. **Not reviewing action source code.**

6. **Not reviewing transitive dependencies of actions.**

7. **Not using Dependabot/Renovate or equivalent to manage pinned action updates.**

8. **Using public Docker images without digest pinning.**

9. **Using `image: latest` in GitLab or Bitbucket pipelines.**
   GitLab explicitly recommends pinning Docker images by version instead of using `latest`. ([GitLab Docs][10])

10. **Pulling remote scripts with `curl | bash`, `wget | sh`, PowerShell `iwr | iex`, or similar.**

11. **Downloading installers without checksum or signature verification.**

12. **Using GitLab `include: remote` without pinning to an immutable ref.**

13. **Using GitLab CI/CD components with `@latest`.**
    GitLab recommends pinning components to a SHA or release tag, and avoiding partial versions such as `1` or `1.1`. ([GitLab Docs][12])

14. **Using Bitbucket Pipes without version pinning or source review.**

15. **Trusting package-manager lifecycle scripts from untrusted branches.**
    Examples: `npm install`, `pip install .`, `setup.py`, Maven/Gradle plugins, Composer scripts, Ruby gems, Cargo build scripts.

16. **Letting test dependencies run arbitrary postinstall/build scripts before security controls are applied.**

17. **Using untrusted GitHub Actions in privileged `pull_request_target` or release workflows.**

18. **Using actions that read all environment variables when they only need one token.**

19. **Allowing third-party uploaders to access all CI environment variables.**
    The Codecov Bash Uploader incident showed how a compromised CI uploader could collect environment variables from CI environments. ([Codecov][13])

20. **Ignoring known CI supply-chain incidents.**
    The 2025 `tj-actions/changed-files` compromise modified action tags and exposed secrets in logs for many repositories; this is a concrete example of why mutable tags and broad secret exposure are dangerous. ([GitHub][14])

## F. Cache, artifact, and workspace poisoning

### Bad behavior

1. **Restoring caches created by untrusted branches into trusted jobs.**

2. **Using cache keys that allow cross-branch or cross-PR poisoning.**

3. **Using cache keys based only on branch name.**

4. **Using cache keys based on attacker-controlled files without isolation.**

5. **Caching directories that contain secrets.**
   Examples: home directories, `.aws`, `.ssh`, `.docker`, `.kube`, `.npmrc`, `.pypirc`, `.gradle/gradle.properties`.

6. **Caching build outputs that later privileged jobs execute.**

7. **Downloading artifacts from untrusted jobs and executing them.**

8. **Publishing artifacts from untrusted branches as release artifacts.**

9. **Uploading entire workspaces as artifacts.**
   Bad: `path: .`, `paths: ["."]`, or uploading the full repository with generated files and hidden config.

10. **Uploading `.git` directories.**

11. **Uploading test reports that include secrets, request headers, cookies, or screenshots of sensitive systems.**

12. **No artifact integrity verification.**
    Trusted jobs should know exactly what they consume.

13. **No artifact expiration.**

14. **Artifact promotion replaced by environment rebuilds.**
    Bitbucket warns that rebuilding a branch for each environment can produce different artifacts because dependency versions can change; promoting a tested artifact is safer. ([Atlassian Support][15])

15. **Using caches as a deployment channel.**

16. **Using cache fallback keys that make trusted jobs fall back to untrusted caches.**

17. **Storing dependency lockfiles in caches instead of version control.**

18. **Not invalidating caches after dependency, compiler, image, or toolchain changes.**

19. **Cache keys too specific, causing constant misses and slow builds.**

20. **Cache keys too broad, causing stale or contaminated builds.**

21. **Changing GitLab hashed cache-key settings inconsistently across runners.**
    GitLab notes that inconsistent cache-key hashing settings can cause cache misses and longer first pipeline runs. ([GitLab Docs][16])

## G. Test correctness and reliability failures

### Bad behavior

1. **Letting tests pass when the test command failed.**
   Examples: `pytest || true`, `npm test || true`, `mvn test || exit 0`, `continue-on-error: true`, GitLab `allow_failure: true`.

2. **Marking required test jobs as optional.**

3. **Continuing deployment after failed tests.**

4. **Running tests but not collecting reports.**
   Without JUnit/XML or equivalent reports, teams cannot reliably track failure rates, duration, or slow/flaky tests. Bitbucket’s test reporting feature, for example, expects JUnit-style XML reports to show test failure and duration details. ([Atlassian Support][17])

5. **Treating “no tests found” as success.**

6. **Using broad `continue-on-error` to hide flakiness.**

7. **Rerunning failing tests until they pass without tracking flake rate.**
   Google’s testing guidance discusses rerunning failing tests as a mitigation, but also warns that flaky tests need identification and tracking rather than silent normalization. ([Google Testing Blog][18])

8. **Quarantining flaky tests forever.**

9. **Disabling flaky tests with no owner, ticket, expiry date, or re-enable policy.**

10. **Overusing end-to-end tests for basic validation.**
    Larger tests are more likely to be flaky, slower, and harder to debug than smaller isolated tests. ([Google Testing Blog][19])

11. **No deterministic seed handling.**

12. **No isolation between tests.**
    Examples: shared databases, shared queues, shared files, shared ports, shared browser sessions, shared global state.

13. **Tests depend on order.**

14. **Tests depend on wall-clock time, timezone, locale, or current date without control.**

15. **Tests depend on external network services without mocks, fixtures, retries, or contract boundaries.**

16. **Tests use production APIs.**

17. **Tests mutate shared staging environments without cleanup.**

18. **Tests race because CI parallelism was added without resource isolation.**

19. **No timeout for tests.**

20. **No per-test timeout.**

21. **No slow-test reporting.**

22. **No failure classification.**
    Infrastructure failure, product bug, flaky test, dependency outage, and security scan failure should not all look identical.

23. **No test ownership.**

24. **No test-impact analysis in large monorepos.**

25. **Path filters skip tests that are actually impacted by shared code changes.**

26. **Unit, integration, E2E, performance, and security tests all run in one opaque job.**

27. **Coverage is uploaded from untrusted PRs using privileged tokens.**

28. **Coverage or quality gates are advisory only but branch protection treats them as required.**

29. **Snapshot tests are auto-updated in CI.**

30. **Tests commit generated changes back to the branch without human review.**

31. **CI agent deletes, rewrites, or weakens tests to make the pipeline green.**

32. **CI agent changes assertions into smoke checks.**

33. **CI agent marks failing tests as skipped instead of fixing the product or test.**

34. **No negative/security tests in pipeline.**

35. **No regression tests for previously exploited CI weaknesses.**

Flaky tests are not a minor nuisance; research and industry reports consistently treat them as a major CI cost and reliability problem. ([ScienceDirect][20])

## H. Performance, cost, and scalability bad behavior

### Bad behavior

1. **Running every possible job on every push.**
   Full unit, integration, E2E, fuzzing, SAST, DAST, dependency scanning, container scanning, license scanning, performance tests, mobile tests, and release builds on every tiny documentation change is expensive and slow.

2. **No path filters or change detection.**

3. **No test selection or test-impact analysis.**

4. **No pipeline stages by feedback speed.**
   Fast lint/unit checks should fail early before expensive jobs.

5. **Monolithic pipelines with long sequential stage waterfalls.**

6. **Not using DAG/`needs` dependencies where supported.**

7. **Excessive matrix expansion.**
   Example: OS × language version × database × browser × architecture × feature flag × region across every PR.

8. **No cap on matrix size.**

9. **No cancellation of superseded runs.**
   GitHub supports concurrency controls and canceling in-progress runs, which is important for fast-moving branches. ([GitHub Docs][21])

10. **No `interruptible` jobs in GitLab for superseded pipelines.**

11. **No queue visibility.**

12. **Over-parallelizing beyond runner capacity.**
    This can increase queue time, cost, and external-service contention.

13. **Under-parallelizing expensive independent tests.**

14. **Serializing jobs that could safely run independently.**

15. **Running expensive security scans on generated/vendor directories.**

16. **Running full historical secret scans on every commit.**
    GitLab supports scanning current/future commits and separately running a historic scan for repository history; repeatedly scanning all history can be unnecessarily expensive. ([GitLab Docs][22])

17. **No dependency cache.**
    GitHub and Bitbucket both document dependency caches as a way to reduce repeated dependency download/build time. ([GitHub Docs][23])

18. **Bad dependency cache keys.**
    Example: key does not include OS, lockfile hash, language version, architecture, or package-manager version.

19. **Caching too much.**
    Huge caches can cost more to upload/download than rebuilding.

20. **Caching unstable build outputs.**

21. **No Docker layer caching or image reuse where appropriate.**

22. **Building the same image in multiple jobs.**

23. **Installing the same system packages in every job.**

24. **Downloading the same toolchain in every job.**

25. **Not using prebuilt CI images for stable toolchains.**

26. **Using giant CI images for small jobs.**

27. **Always doing full Git clone.**
    Bitbucket defaults to a shallow clone depth and documents full clone only when needed, such as operations requiring full history. ([Atlassian Support][24])

28. **Always fetching submodules, LFS files, or full history when not needed.**

29. **No artifact reuse between build and test.**

30. **Rebuilding per environment instead of promoting an artifact.**

31. **No build-system incremental cache.**
    Examples: Gradle, Bazel, Pants, Turborepo, Nx, Maven, Cargo, ccache, sccache.

32. **Using package-manager commands that are non-deterministic or slow.**
    Examples: `npm install` instead of `npm ci` for lockfile-based CI; no frozen lockfile mode for Yarn/pnpm/Bundler/Poetry/Cargo where available.

33. **Using floating dependency versions.**

34. **No timeout on jobs that can hang.**

35. **No retry policy for known transient infrastructure failures.**

36. **Retrying product failures as if they were infrastructure failures.**

37. **No separation between smoke tests and full regression tests.**

38. **E2E tests blocked by shared staging environment contention.**

39. **Performance tests run on noisy shared runners and produce meaningless results.**

40. **No baseline, variance tracking, or trend analysis for performance tests.**

41. **Uploading huge artifacts on every run.**

42. **Excessive artifact retention.**

43. **No cleanup of old artifacts, caches, and container layers.**

44. **No observability for critical path.**

45. **No visibility into slow tests.**

46. **No visibility into runner utilization.**

47. **No cost allocation by repository, team, branch, or workflow.**

48. **Pipelines duplicate because push and pull-request triggers overlap.**
    Bitbucket documents that overlapping default and pull-request pipeline definitions can cause two pipelines to run for the same push. ([Atlassian Support][25])

49. **Using one CI YAML file so large that teams cannot reason about it.**
    Community feedback from Reddit’s DevOps discussions calls out painful monolithic pipeline files, runner scaling, and parallelized Docker-build instability as real operational issues. ([Reddit][26])

GitLab’s pipeline-efficiency guidance specifically frames slow pipelines as a cost and feedback-loop problem, and recommends analyzing repository size, job/stage count, dependencies, critical path, and runner capacity. ([GitLab Docs][27])

## I. Deployment and release bad behavior

### Bad behavior

1. **Deploying from untrusted branches.**

2. **Deploying from unprotected tags.**

3. **Deploying automatically from pull requests.**

4. **Running deployment jobs in the same trust context as PR validation.**

5. **No environment approvals for production.**

6. **No separation between test, staging, and production credentials.**

7. **No artifact promotion.**
   Rebuilding for every environment can produce different artifacts from the ones tested.

8. **No release provenance, SBOM, signature, or attestation.**
   SLSA’s threat model is built around protecting source and build integrity so consumers can trust what was built and released. ([SLSA][28])

9. **Publishing packages from test jobs.**

10. **Publishing on every branch.**

11. **Publishing with mutable version tags.**

12. **Overwriting previously published artifacts.**

13. **Using `latest` as the primary deploy mechanism.**

14. **Allowing CI to force-push release branches.**

15. **Allowing CI to move release tags.**

16. **No deploy concurrency lock.**
    Two deployments can race and corrupt an environment.

17. **No rollback mechanism.**

18. **No deployment audit trail.**

19. **No manual break-glass path with logging.**

20. **No post-deploy validation.**

21. **No change window or freeze enforcement where required.**

22. **No separation of duties for regulated environments.**

23. **Tests against production after deployment use privileged credentials and leak data into logs.**

24. **Infrastructure-as-code plans are generated in one job and applied in another without verifying the plan artifact.**

25. **Terraform or cloud credentials are available to jobs that only run tests.**

## J. Logging, monitoring, and audit failures

### Bad behavior

1. **No alert when CI config changes.**

2. **No required review for CI config changes.**

3. **No audit trail for who changed secrets, variables, runners, deploy keys, webhooks, environments, or branch protections.**

4. **No monitoring for newly registered self-hosted runners.**

5. **No monitoring for runner label changes.**

6. **No detection for unusual outbound traffic from CI.**

7. **No detection for crypto-mining behavior.**

8. **No detection for mass secret access.**

9. **No detection for new package-publishing jobs.**

10. **No detection for new `pull_request_target`, `workflow_run`, scheduled, or manual-dispatch workflows.**

11. **No detection for `curl | bash`, mutable action refs, `latest` images, or `write-all` permissions.**

12. **No security scan on CI YAML.**

13. **No periodic review of CI tokens and variables.**

14. **No inventory of third-party actions/components/pipes.**

15. **No review of workflow run logs after suspicious runs.**

16. **No incident-response playbook for CI compromise.**

17. **No ability to revoke all CI credentials quickly.**

18. **No cache purge procedure.**

19. **No artifact purge procedure.**

20. **No runner reimage/reprovision procedure.**

21. **No branch protection verification after CI changes.**

GitHub points to tools such as OpenSSF Scorecards for checks including dangerous token permissions, script injection, and pinned dependencies; these are exactly the kinds of behaviors that should become automated policy checks. ([GitHub Docs][7])

# 2. Bad behavior specific to CI `.git*.yml` files

This section is written as a review checklist for `.github/workflows/*.yml`, `.gitlab-ci.yml`, `bitbucket-pipelines.yml`, GitBucket CI plugin YAML/config, and reusable CI templates.

---

## A. Bad GitHub Actions YAML behavior: `.github/workflows/*.yml`

### Dangerous triggers and privilege boundaries

1. `on: pull_request_target` combined with dependency install, build, test, lint, Docker build, package scripts, or arbitrary `run`.

2. `pull_request_target` combined with checkout of the PR head SHA or branch.

3. `workflow_run` that downloads an artifact from an untrusted workflow and executes files from it.

4. `issue_comment`, `pull_request_review`, or `repository_dispatch` workflows that execute commands from comments, labels, titles, or user input.

5. `workflow_dispatch` inputs used as shell commands, refs, image names, artifact names, environment names, or paths without allowlists.

6. `schedule` workflows that run privileged tasks and execute mutable repository scripts.

7. `push` triggers on all branches for jobs that publish or deploy.

8. `tags: ["*"]` release workflows without protected tag enforcement.

9. Missing `paths` or `paths-ignore` on expensive workflows.

10. Overlapping `push` and `pull_request` workflows that duplicate work.

GitHub Security Lab recommends using unprivileged `pull_request` workflows for processing untrusted code and separate privileged workflows only for narrow tasks such as commenting, after carefully transferring passive results. ([GitHub Security Lab][2])

### Dangerous permissions

1. Missing top-level `permissions:`.

2. `permissions: write-all`.

3. `contents: write` in test jobs.

4. `pull-requests: write` in jobs that run PR code.

5. `actions: write` except in tightly controlled administrative workflows.

6. `id-token: write` on jobs that do not need cloud federation.

7. `packages: write` in test jobs.

8. `issues: write` or `pull-requests: write` just to report status when safer mechanisms exist.

9. Using `GITHUB_TOKEN` to push changes from a test job.

10. Allowing Actions to approve PRs or create self-approving change loops.

GitHub recommends setting the default token permission to read-only and escalating job permissions only as needed. ([GitHub Docs][7])

# 5. Best practices: secure, fast, testable CI design

## A. Start with a trust model

Every CI design should classify jobs into trust zones:

1. **Untrusted validation**
   Runs code from PRs, forks, generated branches, bots, and external contributors. No secrets. Read-only token. Ephemeral hosted runner. No internal network access.

2. **Trusted build**
   Runs after merge to protected branches. Limited secrets only when required. Produces signed/attested artifacts.

3. **Release/publish**
   Runs only from protected branches or protected tags. Uses short-lived credentials. Publishes immutable artifacts.

4. **Deploy**
   Uses environment approvals, protected environments, concurrency locks, and deployment-specific credentials.

5. **Administrative automation**
   Labels PRs, comments, updates dashboards, or opens issues. Should not checkout and run untrusted code.

This model follows the same principle behind GitHub’s recommendation to process untrusted code in `pull_request`, then use separate privileged workflows only for narrowly scoped follow-up actions. ([GitHub Security Lab][2])

---

## B. Use least privilege everywhere

### GitHub Actions

Recommended baseline:

```yaml
permissions:
  contents: read
```

Then grant extra permissions only at the job level:

```yaml
jobs:
  test:
    permissions:
      contents: read

  comment:
    permissions:
      contents: read
      pull-requests: write
```

Best practices:

1. Set organization/repository default `GITHUB_TOKEN` to read-only.

2. Use job-level `permissions`.

3. Never use `write-all` for test jobs.

4. Do not allow Actions to approve pull requests.

5. Keep `id-token: write` only on jobs that actually exchange OIDC tokens.

6. Put production secrets behind protected environments with required reviewers.

7. Use reusable workflows carefully; avoid broad `secrets: inherit`.

8. Require CODEOWNERS review for workflow changes.

GitHub’s hardening docs explicitly recommend least privilege, read-only defaults, secret review/rotation, and environment protections. ([GitHub Docs][7])

### GitLab

Best practices:

1. Use **protected branches** for trusted code.

2. Use **protected tags** for releases.

3. Use **protected variables** for deployment credentials.

4. Use **protected runners** for deployment jobs.

5. Mask and hide sensitive variables.

6. Prefer external secrets managers over CI variables for high-value secrets.

7. Use protected environments and approvals for production.

GitLab documents protected branches, protected variables, and protected runners as mechanisms to prevent untrusted code from accessing deployment credentials. ([GitLab Docs][29])

### Bitbucket

Best practices:

1. Use deployment environments and deployment variables for environment-specific credentials.

2. Use secured variables only for true secrets.

3. Do not pass secrets through child-pipeline input variables.

4. Use OIDC/federated identity for cloud access where available.

5. Keep repository/workspace variables scoped and periodically reviewed.

Bitbucket’s variable documentation distinguishes secured variables from other pipeline variable mechanisms and notes contexts where secure masking does not apply. ([Atlassian Support][6])

---

## C. Split untrusted and privileged workflows

A safe GitHub PR pattern:

```yaml
name: PR validation

on:
  pull_request:

permissions:
  contents: read

concurrency:
  group: pr-${{ github.event.pull_request.number }}
  cancel-in-progress: true

jobs:
  test:
    runs-on: ubuntu-latest
    timeout-minutes: 20
    steps:
      - uses: actions/checkout@<full-commit-sha>
      - name: Install
        run: npm ci
      - name: Test
        run: npm test
```

A privileged workflow should not checkout PR code. It should only consume a narrow, validated result such as a JSON status file, coverage summary, or check result. Never execute artifacts from the untrusted workflow.

Bad privileged pattern:

```yaml
on:
  pull_request_target:

jobs:
  dangerous:
    steps:
      - uses: actions/checkout@v4
        with:
          ref: ${{ github.event.pull_request.head.sha }}
      - run: npm install && npm test
```

Good principle: **privileged workflows may comment, label, or update metadata; they should not execute attacker-controlled code.** GitHub Security Lab’s guidance is very clear on this boundary. ([GitHub Security Lab][2])

---

## D. Use ephemeral, isolated runners

Best practices:

1. Use hosted ephemeral runners for public PRs.

2. Use self-hosted runners only for trusted code, or use ephemeral just-in-time self-hosted runners that are destroyed after one job.

3. Separate runner pools by trust level:

   * public PR validation
   * internal PR validation
   * protected branch build
   * release/publish
   * production deploy

4. Never let untrusted jobs run on deployment runners.

5. Avoid shell executors for untrusted code.

6. Avoid privileged Docker-in-Docker for untrusted code.

7. Avoid Docker socket mounts.

8. Remove SSH keys and persistent cloud credentials from runner hosts.

9. Deny access to production/internal networks by default.

10. Restrict outbound egress for high-risk jobs.

11. Apply CPU, memory, disk, and runtime limits.

12. Clean workspaces, containers, volumes, and temp files after every job.

13. Patch runner hosts and runner binaries.

14. Keep runner inventory and job-to-runner audit logs.

15. Reimage runners after suspicious jobs.

These practices are supported by GitHub’s distinction between ephemeral hosted runners and persistently compromisable self-hosted runners, plus GitLab’s warnings about shell executors, privileged containers, and runner host compromise. ([GitHub Docs][7])

---

## E. Pin and verify the supply chain

Best practices:

1. Pin GitHub Actions to full commit SHA.

2. Pin GitLab components to SHA or release tag.

3. Pin Docker images by digest or immutable version.

4. Avoid `latest`.

5. Avoid `@main` and `@master`.

6. Avoid `curl | bash`.

7. Verify checksums/signatures for downloaded tools.

8. Review third-party action/component/pipe source before use.

9. Maintain an inventory of CI dependencies.

10. Use Dependabot/Renovate to update pinned references through reviewable PRs.

11. Require CODEOWNERS review for CI dependency changes.

12. Prefer official/vendor-maintained actions where possible.

13. Minimize third-party actions in jobs with secrets.

14. Use SLSA-style provenance/attestations for build outputs.

15. Generate SBOMs for release artifacts.

16. Sign release artifacts and container images.

17. Verify artifacts before deployment.

18. Promote the same artifact across environments instead of rebuilding.

GitHub recommends full-SHA pinning because tags can move; GitLab recommends pinning CI/CD components; SLSA and NIST both emphasize build/source integrity as part of supply-chain security. ([GitHub Docs][7])

---

## F. Design secrets so CI compromise has limited blast radius

Best practices:

1. Prefer OIDC/federated identity over static cloud keys.

2. Scope cloud trust policies to:

   * specific repository
   * specific branch/tag
   * specific workflow
   * specific environment
   * specific audience
   * specific subject claim

3. Use short-lived credentials.

4. Use different credentials for build, publish, staging, and production.

5. Put production secrets behind approvals.

6. Never expose secrets to PR/fork jobs.

7. Never put secrets in global `env`.

8. Never print environment variables.

9. Disable shell tracing in secret-bearing jobs.

10. Avoid structured multi-line secrets where possible.

11. Rotate secrets regularly.

12. Rotate immediately after suspicious CI activity.

13. Keep artifact/log retention minimal for secret-bearing jobs.

14. Scan for secrets in code, logs, and artifacts.

15. Do one-time historic secret scans, then incremental scanning for new commits.

16. Keep package-publishing tokens narrowly scoped.

17. Use read-only registry tokens for dependency installation.

18. Use separate signing infrastructure for high-value release signing.

19. Keep production data out of CI tests.

20. Mock or synthesize sensitive data.

GitLab’s secret-detection docs distinguish ongoing scanning from historic scans, and GitHub/GitLab both emphasize scoping and protecting secrets. ([GitLab Docs][22])

---

## G. Make tests fast without making them unsafe

A strong testing pipeline usually has tiers:

1. **Preflight**
   YAML lint, dependency lockfile check, formatting, static config policy.

2. **Fast correctness**
   Unit tests, type checks, lint, small security checks.

3. **Build verification**
   Build artifact/container once.

4. **Integration**
   Service tests with isolated databases/queues.

5. **End-to-end**
   Minimal critical paths, not every combination.

6. **Security**
   SAST, dependency scan, secrets scan, IaC scan, container scan.

7. **Release qualification**
   Provenance, SBOM, signing, smoke tests.

8. **Deployment validation**
   Post-deploy smoke checks and monitoring gates.

Best practices:

1. Keep fast PR feedback fast.

2. Fail early on cheap checks.

3. Use path-aware workflows.

4. Use dependency caches keyed by lockfile, OS, architecture, and toolchain.

5. Use shallow clones unless full history is required.

6. Use prebuilt CI images for stable toolchains.

7. Build once, test the same artifact, and promote that artifact.

8. Parallelize independent tests.

9. Do not parallelize tests that share mutable state.

10. Use service containers with unique per-job credentials/databases.

11. Track slow tests.

12. Track flaky tests.

13. Quarantine flaky tests only with owner, ticket, and expiry.

14. Rerun suspected flaky failures for classification, not silent success.

15. Collect JUnit or equivalent reports.

16. Fail when no tests are discovered.

17. Fail when reports are missing.

18. Enforce timeouts.

19. Use concurrency cancellation for obsolete PR runs.

20. Measure queue time, setup time, execution time, cache hit rate, and artifact upload/download time.

Martin Fowler’s CI guidance emphasizes a self-testing build, frequent integration, immediate broken-build repair, visibility, and keeping the build fast. ([martinfowler.com][32])

---

## H. Use caching carefully

Good cache keys should include the things that change dependency output:

```yaml
key: dependency-cache-${{ runner.os }}-${{ hashFiles('**/package-lock.json') }}
```

Best practices:

1. Cache dependencies, not secrets.

2. Cache package-manager directories, not entire home directories.

3. Include lockfile hash in cache keys.

4. Include OS and architecture where relevant.

5. Include language/runtime/toolchain version where relevant.

6. Separate caches by trust zone.

7. Avoid restoring untrusted caches in privileged jobs.

8. Keep caches small.

9. Track cache hit rate.

10. Invalidate caches after dependency/toolchain changes.

11. Do not use caches as artifact promotion.

12. Do not execute build outputs restored from untrusted caches.

GitHub’s cache examples include OS plus lockfile hash in cache keys, which is the right general pattern. ([GitHub Docs][23])

---

## I. Use artifacts as immutable promotion units

Best practices:

1. Build once.

2. Test that exact artifact.

3. Sign/attest that artifact.

4. Promote that artifact to staging.

5. Promote the same artifact to production.

6. Do not rebuild per environment.

7. Store checksums.

8. Verify checksums before deployment.

9. Keep artifact retention appropriate.

10. Do not include secrets in artifacts.

11. Do not execute untrusted artifacts in privileged jobs.

12. Use release versions that are immutable.

Bitbucket’s docs directly warn that rebuilding for each environment can produce different outputs because dependencies can change, which is a strong argument for artifact promotion. ([Atlassian Support][15])

---

## J. Add CI governance and review controls

Best practices:

1. Require CODEOWNERS review for:

   * `.github/workflows/**`
   * `.gitlab-ci.yml`
   * `bitbucket-pipelines.yml`
   * Dockerfiles
   * build scripts
   * release scripts
   * dependency lockfiles
   * package-manager config
   * IaC/deployment manifests

2. Block direct pushes to protected branches.

3. Require signed commits/tags where appropriate.

4. Require status checks.

5. Prevent skipped CI from satisfying branch protection.

6. Require review for workflow changes from bots.

7. Run CI policy scanners on every CI-file change.

8. Run OpenSSF Scorecards or equivalent.

9. Use centralized reusable workflows/components.

10. Pin centralized components.

11. Version reusable workflow contracts.

12. Keep an exception registry.

13. Expire exceptions.

14. Review exceptions periodically.

15. Audit third-party CI dependencies.

16. Track CI incidents and near misses.

17. Train reviewers to treat CI config as privileged code.

GitLab’s component and project-hardening guidance includes protected branches, merge request approvals, two-factor authentication, signed commits, and careful cache/artifact handling for CI/CD components. ([GitLab Docs][12])

# 6. Reference hardening snippets

## A. GitHub Actions: safer PR test workflow

```yaml
name: pr-test

on:
  pull_request:
    branches: [main]

permissions:
  contents: read

concurrency:
  group: pr-test-${{ github.event.pull_request.number }}
  cancel-in-progress: true

jobs:
  test:
    runs-on: ubuntu-latest
    timeout-minutes: 20

    steps:
      - name: Checkout
        uses: actions/checkout@<full-commit-sha>

      - name: Setup Node
        uses: actions/setup-node@<full-commit-sha>
        with:
          node-version-file: .nvmrc
          cache: npm

      - name: Install
        run: npm ci

      - name: Test
        run: npm test -- --ci

      - name: Upload test report
        if: always()
        uses: actions/upload-artifact@<full-commit-sha>
        with:
          name: junit-report
          path: reports/junit.xml
          retention-days: 7
```

Important properties: read-only token, no secrets, no `pull_request_target`, pinned actions, timeout, concurrency cancellation, deterministic install, narrow artifact.

## B. GitLab CI: safer test and protected deploy shape

```yaml
workflow:
  rules:
    - if: $CI_MERGE_REQUEST_IID
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH
    - if: $CI_COMMIT_TAG

default:
  image: registry.example.com/ci/node@sha256:<digest>
  interruptible: true
  timeout: 20m

stages:
  - test
  - build
  - deploy

cache:
  key:
    files:
      - package-lock.json
  paths:
    - .npm/

test:
  stage: test
  script:
    - npm ci --cache .npm --prefer-offline
    - npm test -- --ci
  artifacts:
    when: always
    reports:
      junit: reports/junit.xml
    expire_in: 7 days

build:
  stage: build
  needs: [test]
  script:
    - npm run build
  artifacts:
    paths:
      - dist/
    expire_in: 7 days

deploy_prod:
  stage: deploy
  needs: [build]
  environment: production
  resource_group: production
  rules:
    - if: $CI_COMMIT_TAG
      when: manual
  script:
    - ./deploy.sh dist/
```

Important properties: controlled workflow creation, pinned image digest, interruptible jobs, lockfile cache, test reports, artifact promotion, protected/manual deployment shape.

## C. Bitbucket Pipelines: safer artifact promotion shape

```yaml
image: node:20.11.1

clone:
  depth: 50

pipelines:
  pull-requests:
    "**":
      - step:
          name: Test
          caches:
            - node
          script:
            - npm ci
            - npm test -- --ci
          artifacts:
            - reports/junit.xml

  branches:
    main:
      - step:
          name: Build
          caches:
            - node
          script:
            - npm ci
            - npm run build
          artifacts:
            - dist/**

      - step:
          name: Deploy to staging
          deployment: staging
          script:
            - ./deploy.sh dist

      - step:
          name: Deploy to production
          deployment: production
          trigger: manual
          script:
            - ./deploy.sh dist
```

Important properties: PRs test without deployment, main builds an artifact, staging/prod deploy the artifact, production is manual, clone depth is explicit, caches are used, and deployment-specific variables can be scoped by environment.

# 7. “Bad behavior” scoring rubric for CI/agent evaluation

Use this to grade a human or AI agent’s CI design.

## Critical fail

Any of these should block immediately:

1. Untrusted PR code runs with secrets.

2. `pull_request_target` runs PR code.

3. Self-hosted internal runner executes public PR code.

4. CI YAML contains hardcoded credentials.

5. Test job has production deploy credentials.

6. Third-party action with secrets is unpinned.

7. Docker socket or privileged runner is exposed to untrusted jobs.

8. Deployment from unprotected branches/tags.

9. Release artifacts are created from unreviewed code.

10. CI logs print secrets.

## High-risk fail

1. Missing explicit token permissions.

2. Overbroad cloud role trust.

3. Global secrets in `env`.

4. Mutable image/action/component refs.

5. Cache shared across trust boundaries.

6. Artifact from untrusted job executed by trusted job.

7. `allow_failure` or `continue-on-error` on required tests.

8. No timeout on expensive jobs.

9. No CODEOWNERS review for CI changes.

10. No protected variables/runners/environments.

## Performance fail

1. Full expensive pipeline on every change.

2. No concurrency cancellation.

3. No dependency cache or bad cache key.

4. Full clone where shallow clone works.

5. Huge matrix with no cap.

6. Rebuilding same artifact repeatedly.

7. No test reports or slow-test tracking.

8. Unbounded artifact/cache retention.

9. No critical-path analysis.

10. Flaky tests normalized through reruns.

# 8. Condensed best-practice checklist

A secure, high-performance CI design should satisfy all of these:

1. **Untrusted PRs run with no secrets and read-only permissions.**

2. **Privileged workflows never checkout or execute untrusted code.**

3. **All workflow tokens use least privilege.**

4. **Production secrets require protected environments and approvals.**

5. **Self-hosted runners are isolated by trust zone.**

6. **Public PRs never run on internal self-hosted runners.**

7. **Shell executors, Docker socket mounts, and privileged containers are avoided for untrusted jobs.**

8. **Third-party actions/components/pipes/images are pinned and reviewed.**

9. **Mutable refs like `latest`, `main`, and `master` are prohibited in sensitive jobs.**

10. **Remote installer scripts are verified or avoided.**

11. **Secrets are never stored in YAML, logs, caches, or artifacts.**

12. **OIDC/short-lived credentials replace static cloud keys where possible.**

13. **Caches are scoped by trust zone and keyed by lockfile/toolchain.**

14. **Artifacts are narrow, expire, and are never used to execute untrusted code in privileged contexts.**

15. **Build once, test once, promote the same artifact.**

16. **Fast checks run first.**

17. **Expensive checks use path filters, test selection, schedules, or release gates.**

18. **Pipelines use timeouts and cancellation of obsolete runs.**

19. **Flaky tests are tracked, owned, and fixed.**

20. **Security gates fail closed unless there is an explicit, expiring exception.**

21. **CI config changes require expert review.**

22. **CI policy is automatically scanned.**

23. **Runner hosts are patched, ephemeral where possible, and fully cleaned.**

24. **Network access from CI is denied by default and allowed by need.**

25. **Audit logs cover secrets, runners, variables, deploy keys, environments, webhooks, and workflow changes.**

26. **Incident response includes secret rotation, cache/artifact purge, runner reprovisioning, and package-token revocation.**

27. **Test reports, coverage, duration, queue time, flake rate, and cache hit rate are observable.**

28. **Release artifacts have provenance, SBOMs, signatures, and immutable versions.**

29. **Deployment jobs use protected branches/tags, manual gates where appropriate, and concurrency locks.**

30. **The CI system is treated as production infrastructure, not developer convenience glue.**
