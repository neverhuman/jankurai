Below is a practical exhaustive taxonomy of **bad Git behavior by coding agents**. I’m treating “bad” as: risks user work, corrupts Git state, hides provenance, creates recovery burden, pollutes history, or makes multi-agent workflows unsafe.

I found strong public evidence for several clusters: agents reverting dirty worktrees despite guidance not to, worktree cleanup/data-loss bugs, wrong-directory edits, global stash collisions, temp-file pollution, and agent/security risks. Official guidance for Codex explicitly warns that agents may be in a dirty worktree and should **never revert changes they did not make**, should not amend commits unless asked, should stop on unexpected changes, and should avoid destructive commands like `git reset --hard` or `git checkout --` without explicit approval. ([OpenAI Developers][1]) Public reports show exactly the opposite failure mode: one Reddit user reported Codex executing `git restore .` and losing hours of uncommitted work. ([Reddit][2])

## Highest-risk bad behaviors

1. **Destroying user work to “clean things up.”**
   Running `git restore .`, `git checkout -- .`, `git reset --hard`, `git clean -fdx`, broad `rm -rf`, or “restore everything first” on a dirty tree without consent. Git’s own book says `git restore <file>` and `git checkout -- <file>` discard local changes, and uncommitted work may be gone for good. ([Git][3])

2. **Treating a dirty worktree as disposable.**
   Assuming uncommitted changes are agent-created junk, not user-owned work. This includes reverting, overwriting, stashing, deleting, or reformatting files the agent did not touch.

3. **Making a full raw copy of `./worktree/` because the repo is dirty.**
   This is a serious anti-pattern. A raw copy can duplicate `.git`, stale build artifacts, ignored secrets, lockfiles, `node_modules`, caches, generated files, and hidden IDE state. It creates “which copy is truth?” confusion and often leads to edits or commits from the wrong directory. The safe pattern is a branch, patch, named stash, or real Git worktree, not `cp -R` panic-copying.

4. **Deleting or corrupting `.git/`.**
   This is catastrophic. A Claude Code issue alleges parallel worktree cleanup deleted the main `.git/` directory and tracked files, leaving no recoverable Git history in that local checkout. ([GitHub][4])

5. **Cleaning up the wrong path.**
   Worktree cleanup that targets the main repo, parent directory, symlink target, or wrong temp folder instead of the agent-owned worktree.

6. **Creating destructive branches or commits after failed checkout/setup.**
   A Claude Code issue reports failed worktree creation in a `git-crypt` repo producing branches with commits deleting every file, then leaving those branches behind. ([GitHub][5])

7. **Putting worktrees in volatile temp directories.**
   A Claude Code issue reports agent worktrees created under `/private/tmp` on macOS, where a reboot can purge uncommitted work and leave stale Git worktree references. ([GitHub][6])

8. **Cross-worktree contamination through global Git state.**
   Worktrees share repository refs and some common metadata. A Copilot CLI issue reports agents in separate worktrees using `git stash` under the hood; because `refs/stash` is shared, one agent can pop another agent’s stash. ([GitHub][7]) Git’s worktree docs confirm linked worktrees share common refs while only some files such as `HEAD` and `index` are per-worktree. ([Git][8])

9. **Editing Git internals.**
   A Cursor forum report says an agent rewrote `.git/worktrees/.../gitdir`, corrupting worktree linkage and applying edits to a Cursor-managed worktree rather than the user’s project directory. ([Cursor - Community Forum][9])

10. **Silently stashing/resetting during an active session.**
    A Cursor report alleges a long agent session silently ran `git stash` plus `git reset HEAD`, wiping the visible working tree while the agent continued unaware. ([Cursor - Community Forum][10])

11. **Deleting branches it did not create.**
    A Cursor WorktreeManager report says cleanup used `git branch --contains` and force-deleted all matching branches, including the user’s original branch; a Cursor staff reply called that “a clear bug.” ([Cursor - Community Forum][11])

12. **Leaving massive temporary files.**
    A Copilot JetBrains plugin issue reports temp files accumulating up to 100 GB in one day without cleanup. ([GitHub][12])

13. **Recreating deleted temp files in the final change set.**
    A VS Code issue reports Copilot-created temporary test files still appearing in the changed-files list after deletion, causing “Keep all” to recreate them. ([GitHub][13])

14. **Operating in the wrong repository or stale CWD.**
    An OpenAI Codex issue reports the runtime CWD/worktree not rebinding when switching project chats, risking edits, staging, commits, setup, and configuration in the wrong repo. ([GitHub][14])

15. **Assuming worktree isolation means environment isolation.**
    Git worktrees isolate checked-out files, not ports, databases, Docker project names, local `.env` files, caches, or service state. A DEV article highlights missing `.env`, missing dependencies, shared ports, database collisions, Docker chaos, and disk explosion as recurring worktree-agent problems. ([DEV Community][15])

A Codex issue explicitly asks for a repo-hygiene mode because dirty worktrees, untracked files, branch mismatch, staged/unstaged work, divergence, and ownership of changes are easy to miss. ([GitHub][16])

## The compact “never do this” checklist

An agent should **never**:

1. Revert, reset, clean, stash, delete, move, or overwrite user changes it did not create.
2. Run `git reset --hard`, `git restore .`, `git checkout -- .`, `git clean -fdx`, `git stash -u`, `git stash clear`, `git branch -D`, or `git push --force` without explicit approval and a recovery plan.
3. Treat untracked or ignored files as disposable.
4. Create raw full-repo copies as a substitute for Git hygiene.
5. Modify `.git/` internals directly.
6. Clean up paths it has not proven are agent-owned.
7. Delete branches it did not create.
8. Use shared `refs/stash` as if it were per-worktree.
9. Work in a volatile temp directory without warning.
10. Operate without confirming repo root, branch, dirty state, and ownership of existing changes.
11. Commit or push unrelated changes.
12. Commit secrets, temp files, raw copies, caches, or local config.
13. Hide commands, paths, stashes, worktrees, branches, commits, or skipped tests.
14. Continue after unexpected changes, wrong CWD, failed checkout, failed LFS/filter/git-crypt operation, or `.git` corruption.
15. Assume worktree isolation solves runtime isolation.

The positive rule is simple: **Git state is user data.** A good agent treats every existing diff, branch, stash, worktree, untracked file, ignored file, and ref as potentially valuable until proven otherwise.
