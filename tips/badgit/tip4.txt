Absolutely — here’s a **broad, failure-mode–oriented catalog** of bad Git behavior by coding agents. I’m treating “exhaustive” as: *everything I would want in a red-team checklist, product spec, guardrail policy, or postmortem taxonomy for Git-aware agents.*

A caveat: “all of X/Reddit/open source” is not literally enumerable from search, but I did survey representative public docs, GitHub/GitLab docs, GitHub issues, Reddit reports, indexed X posts, agent docs, open-source guard projects, and recent papers. The same themes repeat very strongly: agents lose track of dirty state, mutate the wrong tree, create stale worktrees, run destructive Git commands, stage unrelated files, and produce large opaque diffs.

Git itself already has the safe primitive for many parallel-agent workflows: `git worktree`, which lets one repository support multiple working trees and recommends removing linked worktrees when done. Git also documents that `git clean` recursively removes untracked files, and with `-x` it removes ignored files too, which makes it a dangerous “cleanup” tool in a human workspace. ([Git][1]) Official agent products are also converging on branch/worktree/PR isolation: GitHub’s cloud agent works on a branch in an ephemeral environment, while local IDE agent mode edits directly in the user’s local environment; OpenAI’s Codex app has built-in worktree support, and its sandbox docs explicitly distinguish bounded workspace-write behavior from “danger-full-access.” ([GitHub Docs][2]) Community and open-source reports show the messier reality: stale agent worktrees leaving full repo copies and undeletable branches, worktree isolation being silently ignored so concurrent agents clobber each other, Cursor cleanup force-deleting a user branch, Reddit users losing work after `git reset --hard`, and command chains hiding destructive operations off-screen. ([GitHub][3]) Recent research also backs the broader safety concern: one filesystem-safety paper analyzed 290 public reports of agents corrupting data, deleting files, or leaking secrets, and a 2026 empirical study of 304,362 AI-authored commits across 6,275 GitHub repositories found AI-authored commits introducing persistent bugs, smells, and security issues. ([arXiv][4])

## Exhaustive bad-agent Git behavior catalog

### 1. Dirty-worktree negligence

1. **Does not run `git status --short --branch` before changing files.**
2. **Does not inspect staged vs. unstaged vs. untracked files before acting.**
3. **Treats a dirty tree as “probably mine” instead of “unknown ownership.”**
4. **Edits on top of a user’s uncommitted work without first identifying what belongs to whom.**
5. **Stages or commits the user’s existing changes along with the agent’s changes.**
6. **Overwrites user edits because it assumes the last-read file is still current.**
7. **Ignores untracked files because `git diff` is empty.**
8. **Assumes `git reset --hard` restores “everything,” while untracked files remain unless separately cleaned.**
9. **Runs cleanup based on `git status -uno`, thereby hiding untracked agent output.** A Claude Code report describes stale cleanup using `git status --porcelain -uno`, missing untracked files and allowing deletion of worktrees containing unadded output. ([GitHub][5])
10. **Fails to detect local commits not pushed.**
11. **Fails to detect remote commits not pulled.**
12. **Fails to detect branch divergence.**
13. **Fails to detect an in-progress merge, rebase, cherry-pick, bisect, or revert.**
14. **Continues after Git reports conflicts or lock failures.**
15. **Misreports “clean” when ignored files, untracked generated files, or staged changes exist.**
16. **Does not snapshot the starting commit, branch, and working-tree state.**
17. **Does not provide a “before” diff before making broad changes.**
18. **Does not ask the user when the worktree is dirty and ownership is unclear.**
19. **Interprets “fix it” as permission to reorganize the whole repo.**
20. **Proceeds in a dirty tree after explicitly being told to avoid dirty trees.**

### 2. Bad worktree and isolation behavior

21. **Works directly in the main user worktree when isolation was requested.**
22. **Claims to be using a worktree but actually runs in the shared main checkout.** A Claude Code issue reports the `isolation: "worktree"` flag being ignored, with multiple agents operating in the shared main working tree, switching branches for each other, and losing or reverting files. ([GitHub][6])
23. **Creates a full `./worktree/` or `./repo-copy/` copy inside the repository instead of using `git worktree`.**
24. **Copies the entire repo because the current tree is dirty, rather than stopping, asking, or using a proper linked worktree.**
25. **Creates a nested clone or nested `.git` directory inside the project.**
26. **Creates a full copy without adding it to `.gitignore`, so `git add .` may stage the entire duplicate.**
27. **Creates a full copy inside paths searched by tests, linters, formatters, ripgrep, TypeScript, Python discovery, or build tools.**
28. **Creates full copies that duplicate `node_modules`, virtualenvs, build artifacts, caches, or generated files.**
29. **Creates worktrees under `/tmp` or another volatile directory where uncommitted work can disappear on reboot.** A Claude Code issue reports agent-spawned worktrees created in `/private/tmp`, with uncommitted work lost on reboot and stale worktree refs left behind. ([GitHub][7])
30. **Creates worktrees in a global hidden directory without telling the user where the work is.**
31. **Creates worktrees in a path the repo’s tools do not expect.**
32. **Creates worktrees but does not record which agent/session owns them.**
33. **Creates worktrees but does not record base commit, target branch, task ID, or cleanup policy.**
34. **Leaves stale worktree directories after success, crash, cancellation, or timeout.**
35. **Leaves stale `.git/worktrees/*` metadata.**
36. **Leaves stale branches checked out by dead worktrees, making `git branch -D` fail.** A public Claude Code issue says stale worktrees left behind full repo copies, branches Git refused to delete because they were checked out in a worktree, and persistent `.git/worktrees` refs. ([GitHub][3])
37. **Leaves stale `extensions.worktreeConfig` or modified repo format in `.git/config`.** One Claude Code issue reports cleanup leaving `repositoryformatversion = 1`, `extensions.worktreeConfig = true`, and stale prunable worktrees, breaking other IDE agents. ([GitHub][8])
38. **Deletes a worktree directory with `rm -rf` instead of `git worktree remove`, leaving Git metadata stale.**
39. **Runs `git worktree remove --force` without checking uncommitted, untracked, or unpushed work.**
40. **Runs `git worktree prune` blindly while sessions are still active.**
41. **Deletes active agent worktrees because cleanup uses age/mtime rather than liveness.**
42. **Deletes the parent session’s current worktree due to ID/path collision.** A Claude Code report describes a sub-agent cleanup deleting the parent session’s worktree when an agent ID prefix collided with the existing worktree name. ([GitHub][9])
43. **Reuses stale worktree branches without resetting to the intended base.**
44. **Reuses stale stashes from an earlier agent session.**
45. **Lets stale changes contaminate a new agent’s task.** A Claude Code issue reports stale branch reuse after an 8-character agent-ID collision, inheriting old stashes, uncommitted modifications, and an outdated base; the contaminated changes passed tests and required manual diff audit. ([GitHub][10])
46. **Uses short deterministic branch names that collide across sessions.**
47. **Uses the same worktree path for multiple agents.**
48. **Creates many worktrees but has no `doctor`, `cleanup`, or garbage-collection command.**
49. **Fails to distinguish disposable worktrees from rescue-worthy worktrees containing unmerged commits.**
50. **Deletes worktree branches even when commits are not merged or pushed.**
51. **Force-deletes user branches during worktree cleanup.** Cursor’s forum includes a confirmed bug report where cleanup ran `git branch --contains <commit>` and force-deleted all matching branches, including the user’s pre-existing source branch. ([Cursor - Community Forum][11])
52. **Does not serialize `git worktree add/remove` operations across agents, causing `.git/config.lock` or `index.lock` contention.**
53. **Treats lockfile errors as permission problems rather than concurrency problems.**
54. **Deletes stale lockfiles without verifying no Git process is running.**
55. **Creates worktrees from stale local `main` rather than `origin/main` or the requested base.**
56. **Creates worktrees from a user’s private unpushed branch without preserving that dependency.**
57. **Does not tell the user where to find worktree output.**
58. **Does not produce a patch before removing a worktree.**
59. **Does not clean up branches after removing worktrees.**
60. **Does not clean up worktrees after deleting branches.**

### 3. Destructive Git command behavior

70. **Creates a stash with a vague name like “WIP” and later cannot identify it.**
71. **Deletes branches with `git branch -D` without checking merge status.**
72. **Deletes remote branches with `git push origin --delete` without approval.**
73. **Force-pushes with `git push --force` instead of avoiding history rewrite or using `--force-with-lease`.**
74. **Force-pushes to shared, protected, release, or default branches.**
75. **Rebases public/shared branches without permission.**
76. **Amends commits already pushed to a shared branch.**
77. **Runs `git filter-repo`, BFG, or history-rewriting cleanup without explicit approval and backup.**
78. **Deletes tags, especially release tags.**
79. **Moves tags to new commits.**
80. **Runs `git update-ref -d` or other plumbing commands without expert-level context.**
81. **Runs `git reflog expire --expire=now --all` or aggressive `git gc --prune=now`, destroying recovery paths.**
82. **Deletes `.git`, `.git/refs`, `.git/index`, or `.git/worktrees` manually.**
83. **Edits `.git/config` manually and leaves it corrupt or nonstandard.**
84. **Runs destructive shell commands from inside Git workflows, such as `rm -rf src`, `find . -delete`, `docker volume rm`, or database resets.** A Reddit report described a hidden `docker volume rm` at the far right of a long AI-suggested command chain, deleting unsaved work; commenters also warned about reset and force-push hazards. ([Reddit][12])
85. **Chains destructive commands behind harmless-looking commands with `&&`, hiding risk off-screen.**
86. **Ignores user fatigue/approval risk and asks for confirmation on giant command blobs rather than one command at a time.**
87. **Runs commands copied from old logs, comments, README snippets, or Stack Overflow without adapting them.**
88. **Treats “revert” as `reset --hard` instead of `git revert` or a targeted patch.**
89. **Does not create a patch or backup before irreversible operations.**
90. **Does not explain how to recover using reflog, stash, local history, or backups after damage.**

Open-source guard projects exist almost entirely because of these classes of failures. One guard’s README explicitly lists catastrophic agent commands like `git reset --hard`, `rm -rf ./src`, and destructive database operations, and blocks them before execution. ([GitHub][13]) Indexed X posts show the same concern, including public discussion of blocking misguided agent `git reset --hard HEAD` commands and “check the worktree first; if dirty, stop” norms. ([X (formerly Twitter)][14])

### 26. Especially bad variants of your two examples

**Not cleaning temporary files is bad, but the worst versions are:**

457. Leaving temp files **tracked** in the final commit.
458. Leaving temp files **untracked**, so the next agent sees a dirty tree and misbehaves.
459. Leaving temp files **ignored**, so humans do not see them but tools do.
460. Leaving temp files that tests import or execute accidentally.
461. Leaving temp scripts containing destructive commands.
462. Leaving temp files with secrets.
463. Leaving temp worktrees with unmerged commits.
464. Leaving temp branches that block deletion.
465. Leaving temp `.git` metadata that breaks IDEs.
466. Leaving temp copies that double search/build scope.

**Making an entire `./worktree/` copy inside the repo is bad because:**

467. It is not a Git worktree; it is just a duplicate source tree.
468. It can be accidentally staged by `git add .`.
469. It can be traversed by tests, linters, formatters, type checkers, and search tools.
470. It can create duplicate symbols, duplicate packages, duplicate migrations, or duplicate test suites.
471. It can contain stale code that future agents treat as authoritative.
472. It bloats disk and CI.
473. It may copy secrets or local config.
474. It may copy build artifacts and caches.
475. It may contain a nested `.git`, confusing Git operations.
476. It may hide the fact that the agent refused to reason about dirty state.
477. It is often a symptom of the agent optimizing for its own convenience over repository hygiene.
478. The correct behavior is usually: stop, inspect dirty state, ask for ownership confirmation, create a proper `git worktree` outside the tracked tree or under a clearly ignored/managed directory, record ownership/base, and clean it up with `git worktree remove`.

## The shortest “ban list” for dangerous autonomous Git actions

These should generally require explicit human confirmation, a shown diff, and a backup/patch first:

```bash
git reset --hard
git clean -f
git clean -fd
git clean -fdx
git checkout -- .
git restore .
git stash clear
git stash drop
git branch -D
git push --force
git push --mirror
git push --delete
git tag -d
git rebase
git filter-repo
git gc --prune=now
rm -rf .git
rm -rf <worktree-or-source-dir>
git worktree remove --force
git worktree prune
```

## The positive standard that this list implies

A well-behaved Git agent should do the opposite: inspect state first, preserve user work, isolate each task in a real branch/worktree, never mutate shared history without consent, stage only intentional files, leave no temp artifacts, produce small auditable diffs, run the documented checks, and give an honest final report with exact commands, changed files, test results, and unresolved risks.
