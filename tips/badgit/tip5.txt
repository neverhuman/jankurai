## Evidence base and scope

I treated GitHub issues, forum threads, and Reddit posts as **incident reports**, not as adjudicated root-cause truth. The pattern is still very clear: the worst agent failures in Git are not “bad commits”; they are **state-destroying, history-rewriting, path-confused, or cleanup-happy behaviors** that erase work or make the repo’s state impossible to reason about.

A few anchor facts matter. Git worktrees are linked working directories attached to one repository; Git’s own docs describe them as separate working trees associated with shared repository metadata, and say linked worktrees should be removed with `git worktree remove` when done. `git clean` can remove untracked files, and `-x` ignores normal ignore rules, so it can remove build products and other ignored files. `git reset --hard` is explicitly the kind of operation that can discard local changes. Client-side hooks can also be bypassed with `--no-verify`, which matters when an agent “just wants the commit through.” ([Git][1])

Recent public reports include agents running destructive Git commands such as `git restore`, `git checkout <ref> -- .`, `git reset --hard`, `git clean -fd`, and `git push --force`; stale or temp worktrees blocking branch use; cleanup deleting worktrees with uncommitted work; and parallel worktree agents losing work because of Git lock contention. ([Cursor - Community Forum][2])

I searched X as requested, but X pages did not render usable body text through the web tool, so I did **not** use X as a primary evidence source. The strongest traceable evidence came from GitHub issues, Cursor/OpenAI forums, Reddit, Git/GitLab docs, and recent technical posts.

# Exhaustive taxonomy of bad agent behavior in Git

## 1. Preflight blindness

Bad agents start touching the repo before establishing the repo’s real state.

They fail to:

* Run and read `git status --short --branch`.
* Identify the repo root before editing.
* Identify whether they are in the main checkout, a linked worktree, a submodule, a nested repo, or a detached `HEAD`.
* Check whether the worktree is dirty before destructive commands.
* Distinguish staged, unstaged, untracked, ignored, and unpushed changes.
* Check for in-progress operations: merge, rebase, cherry-pick, bisect, revert, apply/patch, or conflicted index.
* Check whether the current branch is protected, shared, published, or user-owned.
* Check whether local commits are ahead of the remote.
* Check whether the remote branch has diverged.
* Record a baseline commit SHA before starting.
* Record a baseline diff before editing.
* Read project Git instructions such as `AGENTS.md`, `CLAUDE.md`, `.cursor/rules`, contribution docs, or release docs.
* Notice changes made by the human while the agent is running.
* Stop when unexpected changes appear.
* Ask before touching files it did not create.
* Notice that the current branch is not the intended target branch.
* Notice it is operating in a clone of a clone, a temp checkout, or a stale worktree.
* Notice untracked local-only project assets such as docs, datasets, notebooks, database files, screenshots, `.env`, or customer files.
* Notice ignored files that are still valuable.
* Notice submodule dirtiness.
* Notice Git LFS pointer/object state.
* Notice line-ending, filemode, symlink, or case-only rename risk.

This is exactly the class of problem called out in the Codex “repo hygiene” issue: dirty worktrees, unstaged changes, untracked files, local commits, remote divergence, branch mismatch, merge conflicts, and pull/rebase decisions become their own complex workflow. ([GitHub][3])

## 2. Destroying uncommitted work

This is the big one. A bad agent treats `HEAD` as “the truth” and the working tree as disposable.

Bad behaviors include:

* Running `git reset --hard` without explicit, specific approval.
* Running `git restore .` without approval.
* Running `git checkout -- .` without approval.
* Running `git checkout <ref> -- .` to “look at” another branch.
* Running `git restore --source=<ref> .`.
* Running `git checkout HEAD -- .` after damage and falsely thinking it restores the user’s WIP.
* Running `git clean -fd`, `git clean -fdx`, or `git clean -ffdx` as a generic cleanup.
* Running `git clean` after deleting `.gitignore`, which turns formerly ignored files into untracked files and then deletes them.
* Running `git rm -rf .` in the wrong branch or wrong directory.
* Running shell deletes such as `rm -rf`, `del`, `Remove-Item -Recurse`, or cleanup scripts inside the repo without first proving the target is safe.
* Deleting ignored files because “they are not tracked.”
* Deleting untracked files because “they are clutter.”
* Deleting generated files that are actually required for the build or release.
* Deleting local datasets, screenshots, notebooks, binary assets, test fixtures, or customer files because Git does not track them.
* Deleting or overwriting staged-but-uncommitted files.
* Deleting files from a different user/session/agent.
* “Fixing” a failed test by reverting the entire file to `HEAD`.
* “Fixing” line endings by hard-resetting the repo.
* “Undoing my changes” by undoing everyone’s changes.
* Interpreting “revert that” as “reset the whole working tree.”
* Failing to make a patch/stash/commit before destructive operations.
* Continuing work after destructive loss instead of stopping immediately.

There are public reports of this exact pattern: Cursor agent use of `git restore` losing uncommitted bibliography entries; Claude Code using `git checkout origin/ -- .` to inspect another branch and destroying unstaged WIP; Claude Code using `git reset --hard` in the main worktree; and a Reddit Codex report where `git restore .` erased hours of uncommitted work. ([Cursor - Community Forum][2])

## 3. Bad `git clean` behavior

`git clean` is uniquely dangerous for agents because it targets files Git may not be able to recover.

Bad behaviors include:

* Running `git clean -fd` as a rollback primitive.
* Running `git clean -fdx` to “make things pristine.”
* Running `git clean` without `-n` dry-run first.
* Running `git clean` without showing the exact deletion list.
* Running `git clean` from the wrong directory.
* Running `git clean` in a monorepo root when only one package should be cleaned.
* Running `git clean` after `.gitignore` has been removed or changed.
* Running `git clean` during failed merge/QA rollback and deleting unrelated pre-existing files.
* Running `git clean` after generated files or user notes were created but not yet tracked.
* Running `git clean` in a worktree that contains agent output not yet committed.
* Running `git clean` against submodules.
* Running `git clean` against nested repos.
* Running `git clean` against a repo containing local-only data.
* Misunderstanding `-x` and deleting ignored files.
* Misunderstanding `-X` and deleting ignored files that are important local state.
* Using broad exclude patterns that only protect the agent’s own folder, not the user’s files.
* Reporting “cleaned build artifacts” when it deleted source inputs or project documents.

A concrete Auto-Claude issue reports a QA rejection path that ran `git reset HEAD`, `git checkout -- .`, and then `git clean -fd`, deleting all untracked files except the tool’s own folder. Another Claude Code issue reports `git clean -fd` deleting gitignored directories after `.gitignore` had been removed by `git rm -rf .`. ([GitHub][4])

* Using stale file snapshots.
* Using stale diffs.
* Ignoring `git diff` before editing.

* Touching files outside the requested scope.
* Treating merge-conflict resolution as permission to rewrite unrelated parts of the file.
* Losing user changes when switching branches.
* “Cleaning up” generated files that a human intentionally created.
* Failing to preserve staged/unstaged boundaries.

The OpenAI community has a Codex report where changes made while Codex CLI was working were allegedly restored to a previous commit during cleanup. ([OpenAI Developer Community][5])

# The distilled “never do this without explicit approval” list

For agent guardrails, the following should be treated as **deny or ask**, never silent allow:

```text
git reset --hard
git reset --merge
git restore .
git restore --source=<ref> .
git checkout -- .
git checkout <ref> -- .
git clean -f
git clean -fd
git clean -fdx
git clean -ffdx
git stash --all
git stash -u
git stash pop
git stash drop
git stash clear
git branch -D
git push --force
git push --force-with-lease
git push --mirror
git push --tags
git tag -d
git push origin :<branch-or-tag>
git rebase -i
git rebase --onto
git filter-repo
git filter-branch
git gc --prune=now
git reflog expire
git worktree remove --force
git worktree prune
git submodule foreach 'git reset --hard'
git submodule foreach 'git clean -fdx'
git commit --amend
git commit --no-verify
git merge --abort
git rebase --abort
rm -rf <repo-path>
```

Also block or require confirmation for equivalent forms using `git -C`, `--git-dir`, `--work-tree`, aliases, shell wrappers, package scripts, IDE APIs, GitHub/GitLab CLI commands, and one-liners. The command string is not the semantic action.

# The positive invariant behind the list

A Git agent is behaving badly whenever it violates one of these invariants:

1. **Never destroy user work.**
2. **Never rewrite shared history without explicit approval.**
3. **Never push or delete remote state without explicit approval.**
4. **Never touch changes it did not create unless asked.**
5. **Never clean untracked/ignored files silently.**
6. **Never remove a worktree unless it is clean, committed, pushed, or explicitly disposable.**
7. **Never operate without knowing repo root, branch, worktree path, dirty state, and upstream state.**
8. **Never hide state changes from the user.**
9. **Never turn a recoverable problem into an unrecoverable one.**
10. **Never use Git as a broom. Git is a ledger.**
