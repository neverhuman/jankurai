Below is a **stop-ship checklist** for SQL/Postgres/database-design “vibe coding.” I’m treating **“not excusable”** as: production-bound work, valuable data, multi-user systems, security/privacy expectations, and no explicit documented waiver with compensating controls.

A note on “exhaustive”: no list can be mathematically complete because new failure modes appear with new systems, extensions, and deployment patterns. This is the most comprehensive practical list: security, correctness, integrity, migrations, operations, performance, and AI-assisted coding. I weighted official PostgreSQL docs, OWASP, and GitLab’s database guidance above Reddit/X/GitHub anecdotes; GitHub anti-pattern repositories and SQLCheck-style linters are useful signal, while Reddit/X are best treated as practitioner pain reports rather than authority. OWASP explicitly calls out string-concatenated dynamic SQL as a SQL injection root cause, PostgreSQL’s own docs warn about unsafe dynamic SQL construction, and PostgreSQL’s constraint docs are clear that primary keys, foreign keys, unique constraints, `NOT NULL`, and checks are core integrity tools rather than optional decoration. ([OWASP Cheat Sheet Series][1])

# Non-excusable SQL/Postgres bad behavior

## 1. SQL injection and unsafe dynamic SQL

These are never acceptable because safer primitives exist.

1. Concatenating user input into SQL strings.
2. Interpolating request parameters into `WHERE`, `ORDER BY`, `LIMIT`, `OFFSET`, `JOIN`, or `RETURNING` clauses.
3. Treating “we escaped quotes” as a SQL injection defense.
4. Building raw SQL from JSON/API payloads without strict allowlists.
5. Letting users supply column names, table names, sort directions, operators, or function names without allowlisting.
6. Using bind parameters for values but forgetting that identifiers still need allowlisting and safe quoting.
7. Building PL/pgSQL dynamic SQL with raw variables instead of `format()`, `%I`, `%L`, `quote_ident`, `quote_literal`, `quote_nullable`, or `USING`.
8. Dollar-quoting dynamic variable content in PL/pgSQL and pretending it is safe.
9. Using ORM “raw SQL” fragments with string interpolation.
10. Passing unreviewed GraphQL/filter-builder input directly into SQL.
11. Letting frontend state define arbitrary SQL predicates.
12. Accepting AI-generated SQL that contains interpolated strings.
13. Using “internal admin tool” as an excuse for injectable SQL.
14. Relying on WAFs, API gateways, or input validation instead of parameterization.
15. Exposing SQL error details to end users.
16. Logging full SQL with secrets, tokens, passwords, session IDs, or PII.
17. Using `EXECUTE` in stored procedures without proving every interpolated part is safe.
18. Building `IN (...)` lists by joining raw strings instead of binding values safely.
19. Using “trusted users” as the security model for SQL construction.
20. Copy-pasting SQL from dashboards into code without parameter review.

OWASP recommends parameterized queries/prepared statements as a primary defense, and PostgreSQL’s PL/pgSQL docs show safe dynamic SQL patterns using `format()` and `USING`; they specifically warn that dollar-quoting variable values in dynamic SQL is a bad idea. ([OWASP Cheat Sheet Series][1])

## 2. Authentication, connection security, and credentials

These are never acceptable in production-like environments.

21. Using `trust` authentication for non-local or shared environments.
22. Allowing network access to Postgres without strong authentication.
23. Using cleartext password authentication where avoidable.
24. Continuing to use weak/deprecated password methods when stronger options are available.
25. Disabling TLS for sensitive data paths.
26. Using TLS encryption without server identity verification when MITM matters.
27. Sharing one database credential across developers, services, CI, and production.
28. Hardcoding database passwords in source code.
29. Committing `.env` files, dumps, connection strings, or service-account keys.
30. Reusing production DB credentials in staging, local development, notebooks, or BI tools.
31. Giving long-lived credentials to scripts with no rotation plan.
32. Having no emergency credential revocation path.
33. Letting developers connect to production as the application role.
34. Letting the application connect as a superuser.
35. Letting the application connect as the schema owner.
36. Using passwordless admin access because “it’s behind VPN.”
37. Not auditing production DB logins.
38. Not separating human, app, migration, analytics, and replication roles.
39. Storing user passwords in plaintext.
40. Storing user passwords with reversible encryption instead of password hashing.
41. Storing API keys or secrets in normal app tables without encryption/access policy.
42. Backing up secrets into less secure environments.
43. Printing database URLs in logs, test failures, CI output, or error pages.

PostgreSQL documents that `trust` authentication allows anyone who can connect to log in as any database user, even a superuser; its password-authentication docs describe SCRAM-SHA-256 as the strongest built-in password method and warn that cleartext password auth should be avoided when possible. PostgreSQL’s SSL docs also distinguish mere encryption from certificate verification against man-in-the-middle attacks. ([PostgreSQL][2])

## 3. Privileges, roles, RLS, and tenant isolation

These are never acceptable because least privilege is basic hygiene.

44. Granting `SUPERUSER` to application roles.
45. Granting `CREATEDB`, `CREATEROLE`, `BYPASSRLS`, or replication privileges casually.
46. Granting `pg_read_all_data` or `pg_write_all_data` to app roles.
47. Granting file/server-program roles such as `pg_execute_server_program` without extreme justification.
48. Using `GRANT ALL` because permissions are annoying.
49. Granting broad privileges to `PUBLIC`.
50. Failing to revoke default privileges where needed.
51. Letting analytics/BI users write to OLTP tables.
52. Letting migration roles run during normal request handling.
53. Letting support tooling bypass tenant boundaries.
54. Relying only on application code for tenant isolation in a high-risk multi-tenant system.
55. Missing `tenant_id` from tenant-owned tables.
56. Having `tenant_id` columns but no tenant-scoped foreign keys or unique constraints.
57. Joining tenant-owned tables without tenant predicates.
58. Using globally unique IDs as an excuse to skip tenant constraints.
59. Disabling row-level security “temporarily” and forgetting to restore it.
60. Marking RLS policies as “later.”
61. Granting `BYPASSRLS` to roles used by apps, dashboards, or workers.
62. SECURITY DEFINER functions without a locked-down `search_path`.
63. SECURITY DEFINER functions owned by overly powerful roles.
64. SECURITY DEFINER functions that call unqualified object names.
65. Letting untrusted users create objects in schemas on another role’s `search_path`.
66. Using `public` schema as a dumping ground for everything.
67. Not schema-qualifying security-sensitive functions, operators, and tables.
68. Treating “internal admin” as authorization.
69. Hiding tenant filters in app helper functions instead of enforcing boundaries where possible.
70. No tests that prove cross-tenant reads/writes fail.

PostgreSQL documents the breadth of predefined powerful roles like `pg_read_all_data`, `pg_write_all_data`, and server-file/program roles, and warns that functions, triggers, and row-security policies can act as Trojan horses if search paths and privileges are unsafe. `GRANT ... PUBLIC` means every role, not “public-facing users.” ([PostgreSQL][3])

## 4. Primary keys, foreign keys, uniqueness, and constraints

These are database-design sins because they discard the database’s strongest correctness features.

71. Creating tables without primary keys.
72. Using heap tables with no stable row identity for mutable business data.
73. Treating primary keys as optional because “the ORM works.”
74. Allowing duplicate business records instead of enforcing uniqueness.
75. Deduplicating in application code instead of a unique constraint.
76. Using `LIMIT 1` to hide duplicates.
77. Using `DISTINCT` to hide a broken join or broken model.
78. Omitting foreign keys for real relationships.
79. Saying “the app enforces referential integrity.”
80. Removing foreign keys to make tests or imports easier.
81. Creating orphan rows as normal behavior.
82. Using nullable foreign keys where the relationship is mandatory.
83. Having foreign keys but no thought about `ON DELETE` / `ON UPDATE`.
84. Using `ON DELETE CASCADE` on important business data without explicit domain approval.
85. Using `ON DELETE SET NULL` where null would create invalid state.
86. Using `CASCADE` to make deletion errors go away.
87. Creating polymorphic references like `(object_type, object_id)` with no enforceable foreign key.
88. Having one column that sometimes references table A and sometimes table B.
89. Storing comma-separated foreign keys in text.
90. Storing arrays of foreign keys when relationships need referential integrity.
91. Storing relational facts inside `jsonb` to avoid join tables.
92. Modeling many-to-many relationships without a junction table.
93. Missing composite primary keys or unique constraints on junction tables.
94. Allowing multiple “active” rows when only one should exist.
95. Not using partial unique indexes for conditional uniqueness.
96. Not using exclusion constraints for non-overlapping reservations, bookings, ranges, or schedules where overlap is invalid.
97. Missing `CHECK` constraints for valid ranges, positive amounts, non-empty strings, or state rules.
98. Using unconstrained text for statuses, categories, and modes.
99. Letting invalid enum-like values accumulate.
100. Making every column nullable by default.
101. Refusing `NOT NULL` because “the app validates it.”
102. Using null, empty string, zero, and “unknown” interchangeably.
103. Using magic sentinel values like `-1`, `9999-12-31`, or `'N/A'` instead of a modeled state.
104. Having no invariant documented for each important table.
105. Having no database tests for constraints.
106. Disabling constraints during imports and not revalidating them.
107. Creating constraints as `NOT VALID` and never validating them.
108. Trusting a migration succeeded without checking constraint validity.

PostgreSQL describes primary keys as unique plus not-null identifiers, foreign keys as referential-integrity enforcement, unique constraints as duplicate prevention, and `NOT NULL` as common for most database designs. It also supports exclusion constraints for more general invariants, such as non-overlap-style rules. ([PostgreSQL][4])

113. Tables named `data`, `items`, `records`, or `objects` with no precise domain meaning.
114. Columns named `value`, `data`, `json`, `payload`, `misc`, or `extra` as a substitute for modeling.
115. Encoding multiple facts in one column.
116. Encoding hierarchy paths in strings when proper hierarchy querying/integrity is needed.
117. Storing lists in delimited strings.
118. Storing units in the same text field as values.
119. Storing numbers as text.
120. Storing booleans as text.
121. Storing dates as text.
122. Storing money as floating point.
123. Storing timestamps without a timezone policy.
124. Storing local wall-clock time where an instant is required.
125. Storing only an instant where a local civil time and timezone are required.
126. Using email addresses, names, or mutable PII as primary keys without a migration strategy.
127. Using natural keys that can change, then pretending they cannot.
128. Using surrogate keys but failing to enforce the real business key.
129. Having “soft delete” columns everywhere with no uniqueness/FK/query strategy.
130. Having `deleted_at` but forgetting every unique index now needs the correct partial condition.
131. Having `deleted_at` but letting foreign keys point to logically deleted parents.
132. Having multiple lifecycle flags that contradict each other, such as `is_active`, `is_deleted`, `archived_at`, `disabled_at`.
133. Modeling state machines with random booleans instead of a constrained state.
134. Allowing impossible states because “the UI won’t send that.”
135. Denormalizing without source-of-truth ownership.
136. Denormalizing without invalidation logic.
137. Denormalizing without reconciliation checks.
138. Using materialized/cached tables without refresh semantics.
139. Using JSONB as a junk drawer for core relational data.
140. Using arrays as a workaround for missing schema design.
141. Using generated IDs but no domain constraints.
142. Creating history/audit tables that cannot reconstruct who changed what and when.
143. Mutating audit records.
144. Deleting audit records without retention policy.
145. Storing files/blobs in hot OLTP rows without a deliberate storage, backup, and performance plan.

The classic “comma-separated IDs in a text column” problem is repeatedly called out in SQL anti-pattern literature and community repositories because it harms validation, querying, and referential integrity; the usual repair is a dependent table with one row per relationship and foreign keys. ([GitHub][5])

## 6. Data-type negligence

153. Assuming sequence values prove chronological order.
154. Using UUIDs without considering index locality/write amplification.
155. Using random UUID primary keys on very hot tables without understanding index impact.
156. Using time zone types without a policy.
157. Using `timestamp without time zone` for global event instants by accident.
158. Using `timestamp with time zone` but misunderstanding display/session timezone behavior.
159. Comparing dates/timestamps by formatting strings.
160. Storing durations as ambiguous strings.
161. Storing measurements without units.
162. Mixing units in one column.
163. Using unconstrained `jsonb` for data that requires shape guarantees.
164. Using enums for values that product changes every week without migration planning.
165. Avoiding enums/domains/checks entirely and letting arbitrary text spread.
166. Using case-sensitive text for emails/usernames without a deliberate normalization/collation policy.
167. Ignoring collations and case-sensitivity in uniqueness.
168. Using `citext`, lowercased columns, or expression indexes inconsistently.
169. Storing IP addresses as text instead of appropriate network types when querying/ranging matters.

171. Storing encrypted blobs for fields that must be queried, sorted, or constrained without a design.

PostgreSQL’s docs distinguish exact numeric types from floating point and note that exact storage/calculation is appropriate for monetary amounts; they also provide native date/time, JSON/JSONB, UUID, enum, and domain facilities, which means “stringly typed everything” is usually a choice, not a necessity. ([PostgreSQL][6])

## 7. Query correctness failures

These are never excusable in reviewed production SQL.

173. `DELETE` without a `WHERE` clause unless intentionally full-table and protected.

189. Joining on non-key columns by accident.
190. Joining text IDs to integer IDs through implicit casts.
191. Using `DISTINCT` because the join produced too many rows.
192. Using `GROUP BY` to mask duplicates.
193. Aggregating at the wrong grain.
194. Selecting non-aggregated values that do not belong to the grouping grain.
195. Using window functions without deterministic ordering.
196. Using `ORDER BY random()` on large production tables.
197. Using leading-wildcard `LIKE '%term%'` on large tables with no search/index strategy.
198. Applying functions to indexed columns in predicates without expression indexes.

200. Comparing dates by wrapping columns in `DATE(column)` instead of using ranges.
201. Using local-time date boundaries for global users without timezone handling.
202. Using `BETWEEN` incorrectly for half-open time intervals.
203. Writing `OR` predicates that defeat indexes without checking plans.
204. Writing huge `IN` lists instead of staging data/joining when appropriate.
205. Using correlated subqueries with accidental quadratic behavior.
206. Repeating identical subqueries instead of factoring or indexing.
207. Using CTEs as optimization fences based on outdated assumptions without checking current behavior.
208. Assuming the ORM-generated SQL is good because the code is pretty.
209. Not testing query behavior on duplicate, null, missing, and concurrent data.
210. Not testing query plans on representative cardinality.

PostgreSQL’s planner chooses execution plans from statistics, and `EXPLAIN`/`EXPLAIN ANALYZE` are the tools for seeing planned and actual execution; treating query correctness/performance as invisible is therefore not defensible. ([PostgreSQL][7])

PostgreSQL’s concurrency documentation emphasizes that database applications must understand concurrent access to maintain integrity, and its docs note that explicit locking can create deadlocks; it also provides `idle_in_transaction_session_timeout` specifically because dangling idle transactions are harmful. ([PostgreSQL][8])

## 20. “Not automatically bad” tools that become inexcusable when cargo-culted

Some things are not categorically evil. They become inexcusable when used to dodge modeling, integrity, testing, or operations.

590. JSONB is fine for flexible/semi-structured data; it is bad when used to avoid schema design for core relational facts.
591. Soft deletes are fine with a full lifecycle, uniqueness, FK, retention, and query strategy; they are bad as a random `deleted_at` column slapped everywhere.
592. Denormalization is fine for measured read performance; it is bad without source-of-truth, invalidation, and reconciliation.
593. Materialized views are fine; they are bad without refresh semantics and staleness guarantees.
594. Triggers are fine for database-owned invariants/auditing; they are bad as hidden, undocumented business logic.
595. Stored procedures are fine; they are bad when they hide privilege escalation, unsafe dynamic SQL, or untestable behavior.
596. Raw SQL is fine; it is bad when unparameterized, unreviewed, or dialect-misunderstood.
597. ORMs are fine; they are bad when treated as a substitute for understanding SQL.
598. UUIDs are fine; they are bad when write locality/index/storage consequences are ignored.
599. Natural keys are fine for truly stable identifiers; they are bad when mutable/PII values become primary keys by accident.
600. Enums are fine for stable finite domains; they are bad for fast-changing product taxonomy without migration planning.
601. Partitioning is fine; it is bad when added before query/retention/access patterns are understood.
602. Read replicas are fine; they are bad when used where fresh reads are required.
603. Caching is fine; it is bad when it becomes the source of truth.
604. `SELECT *` is fine for ad hoc exploration; it is bad in durable application contracts.
605. Temporary tables are fine; they are bad when used to smuggle unbounded production work into request paths.
606. Index hints/workarounds are sometimes useful in other databases; in Postgres, fighting the planner without statistics/query evidence is bad engineering.
607. “NoSQL-style” flexible attributes are fine for genuinely unknowable sparse metadata; they are bad for known business entities.
608. Application validation is fine; it is bad when it replaces database constraints for durable facts.
609. Background jobs are fine; they are bad when they create race conditions or inconsistent invariants.
610. Manual admin scripts are fine; they are bad without review, dry run, transaction strategy, and audit trail.

Practitioner discussions on Reddit/X repeatedly complain about missing indexes, polymorphic references, soft deletes, JSON misuse, and raw SQL caching, but those are best read as signals: the truly inexcusable behavior is not “ever using JSON” or “ever using soft deletes,” it is using such techniques without constraints, tests, documented semantics, and operational safeguards. ([Reddit][15])

# The simplest non-negotiable review rule

For every SQL/schema change, the author must be able to answer:

**What invariant is protected? What can go wrong under concurrency? What locks does it take? What data can it corrupt or expose? How is it backed up/restored? How is it monitored? How is it rolled forward if rollback is impossible?**

When the answer is “the app handles it,” “the ORM handles it,” “the AI said so,” “it worked locally,” “we’ll fix it later,” or “there probably won’t be much data,” that is the bad behavior.
