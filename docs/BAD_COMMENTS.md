# Bad Comments: Comprehensive Guide

This document defines the comment hygiene standard enforced by `HLT-041-COMMENT-HYGIENE`.

## Philosophy

Single words like `TODO`, `HACK`, or `FIXME` are **not flagged on their own**.
They are common, often legitimate, and flagging them creates unacceptable false-positive noise.

Instead, the comment hygiene detector targets two categories:

1. **Tier 1 — Hard Block**: Compound phrases that *inherently confess* a dangerous, incomplete, or deceptive state. These are blocking findings.
2. **Tier 2 — Advisory**: Marker words (`TODO`, `FIXME`, etc.) that co-occur with security/safety nouns in the same comment. These are non-blocking advisories.

## Scope

The detector **only scans comments in source code files**.

**Scanned**: `.rs`, `.ts`, `.tsx`, `.js`, `.jsx`, `.py`, `.sh`, `.go`, `.sql`, `.c`, `.cpp`, `.java`, `.kt`, `.rb`, `.swift`, `.scala`, `.dart`, `.lua`, `.php`

**Never scanned**: `.md`, `.txt`, `.tex`, `.toml`, `.json`, `.yaml`, `.yml`, `.lock`, `.env`, `.cfg`

**Excluded paths**: tests, fixtures, examples, docs, reference, generated zones, vendor, paper, and Jankurai's own crates.

## Tier 1 — Hard Block Phrases

These compound phrases should **never** appear in production source comments.
Each phrase inherently confesses a dangerous state.

### Security Bypass

| Phrase | Why it's dangerous |
|---|---|
| `skip auth` / `bypass auth` / `disable auth` | Authentication is disabled or circumvented |
| `skip authentication` / `bypass authentication` | Same, full word form |
| `skip authorization` / `bypass authorization` | Authorization checks removed |
| `no permission check` / `skip permission` | Permission enforcement absent |
| `disable ssl` / `disable tls` / `skip ssl` / `skip tls` | Transport security disabled |
| `skip certificate` / `trust all certificates` | Certificate validation bypassed |
| `skip validation` / `disable validation` | Input validation removed |
| `allow all origins` / `disable cors` | CORS policy disabled |
| `disable csrf` / `skip csrf` | CSRF protection removed |
| `skip rate limit` / `disable rate limit` / `no rate limit` | Rate limiting removed |
| `disable security` / `skip security` / `security disabled` | Blanket security bypass |

### Production Confessions

| Phrase | Why it's dangerous |
|---|---|
| `remove before production` / `delete before release` | Code that admits it shouldn't ship |
| `not production ready` / `not prod ready` | Explicit admission of unreadiness |
| `for testing only` / `dev only` / `debug only` | Development-only code in production path |
| `will fix later` / `fix later` | Acknowledged debt with no timeline |
| `known vulnerability` / `known security issue` | Documented but unpatched vulnerability |
| `temporary workaround` / `temporary fix` / `temporary hack` | Temporary code that became permanent |
| `do not ship` / `do not deploy` / `not for production` | Explicit deployment blockers |

### Hardcoded Secrets

| Phrase | Why it's dangerous |
|---|---|
| `hardcoded password` / `hardcoded secret` / `hardcoded token` | Admits credential is embedded |
| `hardcoded key` / `hardcoded credential` / `hardcoded api key` | Same, alternate forms |
| `default password` / `default secret` / `default token` | Default credentials in code |
| `password goes here` / `secret goes here` / `token goes here` | Placeholder scaffold for credentials |

### AI Residue

| Phrase | Why it's dangerous |
|---|---|
| `generated by ai` / `ai generated` / `llm generated` | Unreviewed AI output in production |
| `copilot suggested` / `chatgpt wrote` / `claude generated` | Specific tool attribution without review |
| `ai placeholder` / `agent placeholder` / `llm placeholder` | AI-generated stub left in place |
| `ai wrote this` / `auto-generated stub` / `gpt generated` | Unverified generated code |

### Fake Implementation

| Phrase | Why it's dangerous |
|---|---|
| `fake implementation` / `stub implementation` | Code that doesn't do what it claims |
| `mock in production` / `dummy data in production` | Test doubles in production |
| `placeholder logic` / `not actually implemented` | Logic that's missing |
| `noop implementation` / `no-op implementation` / `empty implementation` | Code that does nothing |
| `returns hardcoded` / `returns dummy` | Admissions of fake behavior |
| `pretend to` | Code that fakes behavior |

### Error Suppression

| Phrase | Why it's dangerous |
|---|---|
| `swallow exception` / `swallow error` | Error handling that hides failures |
| `ignore error` / `ignore exception` | Explicit error dismissal |
| `silence warning` / `silence error` | Diagnostic suppression |
| `suppress error` / `suppress exception` | Error suppression |
| `catch all exceptions` / `empty catch` | Overly broad error handling |
| `intentionally swallowed` / `deliberately ignored error` | Documented but dangerous suppression |
| `eat the error` / `discard error` | Colloquial error suppression |

## Tier 2 — Advisory Context Pairs

These fire only when a marker word co-occurs with a security/safety noun.

**Markers**: `TODO`, `FIXME`, `HACK`, `XXX`, `WORKAROUND`, `KLUDGE`

**Context nouns**: `auth`, `authentication`, `authorization`, `permission`, `encrypt`, `encryption`, `decrypt`, `certificate`, `tls`, `ssl`, `credential`, `password`, `secret`, `token`, `injection`, `sanitize`, `sanitization`, `validate`, `validation`, `xss`, `csrf`, `cors`, `rate limit`, `timeout`, `session`, `privilege`, `escalation`, `rollback`, `migration`, `backup`, `vulnerability`, `exploit`, `security`

### Examples

```rust
// TODO: fix authentication bypass  → ADVISORY (todo + authentication)
// HACK: skip rate limit for now    → ADVISORY (hack + rate limit)
// TODO: refactor this loop         → NOT flagged (no security context)
// FIXME: rename variable           → NOT flagged (no security context)
```

## Suppression

Use `jankurai:allow` to suppress legitimate findings:

```rust
// jankurai:allow comment-hygiene reason="documented security tradeoff in threat model" expires=2026-06-01
// skip auth check for internal admin endpoint
fn admin_handler() {}
```

## What Is NOT Flagged

The following are explicitly **not flagged** to avoid false positives:

- **Word Boundaries**: The scanner enforces word boundaries to prevent substring matching. `author` will not match `auth`; `flawlessly` will not match `ssl`; `prefix later` will not match `fix later`.
- **Negations**: The scanner ignores phrases immediately preceded by negations (`not`, `don't`, `dont`, `never`). For example, `// do not skip auth` will NOT be flagged.
- `TODO` alone (handled by HLT-001-DEAD-MARKER)
- `FIXME` alone
- `HACK` alone
- Single words like `auth`, `security`, `password` in comments
- Documentation comments explaining security concepts
- Comments in markdown, text, or config files
- Comments in test files, fixtures, or examples
- String literals containing flagged phrases
- Code (non-comment) lines containing flagged phrases

## Remediation

For Tier 1 findings:
1. **Resolve the underlying issue** described in the comment
2. **Remove the comment** once the issue is fixed
3. If the comment documents a necessary and reviewed tradeoff, use `jankurai:allow`

For Tier 2 advisories:
1. **Review the security-sensitive TODO** for urgency
2. **Create a tracking issue** if the TODO represents real work
3. **Resolve or suppress** with documented reasoning
