# 6. Shared blind spots across all three

All three are good, but these are the gaps I would correct.

## 1. They still overweight “language” and underweight “verification loop”

The future stack is not just Rust vs Go vs C#. It is:

**language + compiler + tests + generated contracts + SAST/SCA + SBOM + observability + provenance + runtime isolation + rollback path.**

OpenTelemetry matters because logs, metrics, and traces are how you recover from incorrect generated code in production. OpenTelemetry describes itself as a vendor-neutral framework for generating, collecting, and exporting telemetry such as traces, metrics, and logs. ([OpenTelemetry][13])

## 2. They under-discuss data ownership

Everyone says PostgreSQL, but the real question is:

Who owns schema migrations?
Which service can write which tables?
Are there generated query types?
Are constraints enforced in the database or only in app code?
Is vector search in Postgres enough, or do you need a dedicated vector DB?

For many AI products, PostgreSQL plus pgvector should be the first default because pgvector lets teams store vectors with normal data while retaining ACID compliance, point-in-time recovery, and joins. Dedicated vector databases should be added only when scale or retrieval requirements justify another system. ([GitHub][14])

## 3. They underweight secrets and non-human identity risk

AI agents touch configs, env files, cloud SDKs, MCP tools, credentials, and local developer machines. GitGuardian’s 2026 report coverage says secrets sprawl worsened in 2025 and AI-assisted commits leaked secrets at about twice the baseline. That means stack design must include local secret scanning, vaulting, scoped credentials, short-lived tokens, and CI rejection of secret leaks. ([TechRadar][15])

## 4. They do not separate BFF/product logic from core domain logic enough

A **BFF** can be TypeScript, even when the core backend is Rust/Go/.NET/Kotlin. But the BFF should not slowly become the true system of record. The boundary must be explicit:

TypeScript BFF: session, UI aggregation, feature flags, low-risk product glue.
Core backend: authz, invariants, transactions, durable workflows, compute.
Database: source of truth and constraints.

## 5. They do not say enough about “AI corpus advantage”

Languages with huge public examples, simple idioms, deterministic formatters, and strong compiler feedback tend to be easier for agents. This is a real reason Go and TypeScript are unusually strong for AI-assisted development, even when Rust is technically safer. Community discussions repeatedly point to Go’s stable corpus, formatting, build system, static typing, and concurrency as LLM-friendly traits; I treat that as sentiment, not proof, but it matches the broader evidence. ([Hacker News][11])
